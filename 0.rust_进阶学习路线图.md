### 1.2 标准 Trait 图谱 & Trait 系统进阶

- [ ] **转换类：From/Into、AsRef/AsMut、Borrow/ToOwned** → [[1.5-conversion-traits|📖]]
  - `From<T> for U` ⇒ 自动获得 `Into<U> for T`
  - `AsRef` 用于轻量级引用转换，`From` 用于所有权转换
  - `Borrow` vs `AsRef`：Borrow 要求 `Hash` / `Eq` / `Ord` 结果一致
  - `ToOwned`：`&str` → `String`，`&[T]` → `Vec<T>`

- [ ] **比较类：PartialEq/Eq/Ord/PartialOrd/Hash** → [[1.6-comparison-traits|📖]]
  - `PartialEq` 的对称性与传递性
  - `Eq` 是 marker trait（无方法），表示 `a == a` 恒成立
  - `f64` / `f32` 只实现 `PartialEq` / `PartialOrd`（NaN ≠ NaN）
  - `Hash` 与 `Eq` 的一致性契约

- [ ] **实用标记类：Clone/Copy/Display/Debug/Send/Sync** → [[1.7-marker-utility-traits|📖]]
  - `Copy` 的隐含约束：必须也是 `Clone`，类型中所有字段都是 `Copy`
  - `Display` 给用户看，`Debug` 给开发者看
  - `Send` / `Sync` 是 auto trait，编译器自动推导

- [ ] **Dispatch 决策：静态分发 vs 动态分发** → [[1.15-dispatch-static-vs-dynamic|📖]]
  - 泛型（静态分发）= 单态化：每个具体类型生成独立副本，零运行时开销
  - trait object（动态分发）= vtable 间接调用：抹除具体类型，允许异构集合
  - 决策框架：性能 vs 灵活性、编译时间 vs 运行时、类型信息是否需保留
  - `impl Trait` 的中间道路：静态分发 + 类型隐藏（RPIT 是 existential type）
  - 何时该用 `Box<dyn Trait>` 而不是泛型？——异构集合、减小二进制体积、递归类型
  - 对 API 设计的影响：泛型暴露类型、trait object 隐藏类型

- [ ] **Supertrait 系统** → [[1.16-supertrait-system|📖]]
  - `trait Foo: Bar` 的真实语义——是对**实现者的约束**，不是继承
  - 与 Java/C++ 继承的根本区别：无字段继承、无构造链、不引入子类型关系
  - Blanket impl 与 supertrait 的交互：`impl<T: Bar> Foo for T {}`
  - Supertrait 的方法可在 trait 定义内直接调用
  - 真实案例：`trait AsyncWrite: AsyncRead` 的设计为何被废弃

- [ ] **Fully Qualified Syntax（完全限定语法）** → [[1.17-fully-qualified-syntax|📖]]

  - 当类型实现多个 trait 的同名方法时的选择

## Phase 2: 内存控制线

> 理解"这块内存到底发生了什么"是专家的分水岭。

### 2.1 堆分配与智能指针内部

- [ ] **Box 内存布局 & Box::leak** → [[2.1-box-layout-leak|📖]]
  - `Box<T>` 是唯一的"不可复制但拥有堆分配"的指针
  - `Box::leak` 泄漏为 `&'static mut T`，什么场景用？
  - `Box<dyn Trait>` 是胖指针：ptr + vtable
  - `Box<[T]>` 是胖指针：ptr + len

- [ ] **Rc/Arc 内部实现** → [[2.2-rc-arc-internals|📖]]
  - 引用计数存在堆上：`heap { strong_count, weak_count, data }`
  - `Rc::clone` 只是增加计数，不拷贝数据
  - `Weak` 指针：如何检测 "dangling"？strong_count == 0 时 weak 仍可用
  - Arc 的原子操作开销 vs Rc 的非原子开销
  - `make_mut` / `get_mut` 的写时复制模式

- [ ] **Deref & DerefMut 强制解引用** → [[2.3-deref-coercion|📖]]
  - 编译器自动插入 `*` 或 `&**`
  - Deref 链：`&String` → `&str`、`&Vec<T>` → `&[T]`、`&Box<T>` → `&T`
  - 方法解析与 Deref：编译器沿着 Deref 链查找方法
  - 什么时候**不要**实现 Deref：语义不匹配时

### 2.2 析构与内存控制

- [ ] **Drop 析构顺序 & ManuallyDrop** → [[2.4-drop-manuallydrop|📖]]
  - 析构顺序：字段按**定义**顺序（不是按字母序）
  - 结构体析构顺序：字段先析构 → 结构体自身 Drop
  - `ManuallyDrop<T>`：阻止自动析构，用于 unsafe 场景
  - `std::mem::forget` vs `ManuallyDrop`：前者泄漏后不可访问，后者仍可访问内部
  - Pin 析构保证

- [ ] **内存布局与对齐** → [[2.5-memory-layout-alignment|📖]]
  - `#[repr(C)]` — C 兼容布局
  - `#[repr(align(N))]` — 自定义对齐
  - `#[repr(packed)]` — 紧凑布局（可能产生未对齐引用！）
  - `std::alloc::Layout` — 动态计算布局
  - 编译器自动重排字段以减小内存占用（默认 repr(Rust)）
  - `std::mem::size_of` / `align_of`

- [ ] **PhantomData & 型变** → [[2.6-phantom-data-variance|📖]]
  - 协变 (Covariant)：`&'a T` — 生命周期可以缩短
  - 逆变 (Contravariant)：`fn(T)` — 参数生命周期可以拉长
  - 不变 (Invariant)：`&'a mut T`、`UnsafeCell<T>`、`PhantomData<fn(T) -> T>`
  - `PhantomData<T>`：标记所有权（type-system only, zero runtime）
  - 为什么 `Cell<T>` / `RefCell<T>` 是不变的？

- [ ] **Cow (Copy on Write)** → [[2.7-cow-copy-on-write|📖]]
  - `Cow<'a, B>`：`Borrowed(&'a B)` | `Owned(B::Owned)`
  - `Cow<str>` / `Cow<[T]>` 的典型应用
  - 何时执行实际克隆：`to_mut()` 时
  - 性能权衡

- [ ] **Union（联合体）** → [[2.8-union|📖]]
  - union 与 enum 的本质区别：untagged vs tagged，无运行时判别位
  - union 的内存布局：大小 = 最大字段的大小，对齐 = 最大字段的对齐
  - 安全访问：`unsafe { union.field }` —— 编译器不检查类型正确性
  - 与 C union 的 FFI 兼容：`#[repr(C)]` union
  - 安全封装模式：用 enum 标记当前活跃变体、`MaybeUninit` + union 的高性能模式
  - 与 `std::mem::transmute` 的关系——union 是更安全（仍 unsafe）的 transmute

- [ ] **ZST（Zero-Sized Types / 零大小类型）** → [[2.9-zst-zero-sized-types|📖]]
  - 哪些类型是 ZST：`()`、空 struct、空 enum、`PhantomData<T>`、`[T; 0]`
  - ZST 的内存行为：不占空间、操作被编译期消除
  - ZST 在集合中的特殊表现：`Vec<()>` 不分配堆内存、`HashMap<Key, ()>` 等价于 `HashSet<Key>`
  - ZST 的 niche 价值：`Option<ZST>` 可以利用 niche 优化
  - ZST 作为类型级状态标记：`PhantomData`、类型状态模式中的状态标签

### 2.4 内部可变性（Interior Mutability）

- [ ] **UnsafeCell —— 内部可变性的唯一底层原语** → [[2.15-cell-refcell-interior-mutability|📖]]
  - `UnsafeCell<T>` 是编译器识别的"例外"类型——告诉借用检查器"这个位置允许内部可变"
  - 所有内部可变性类型（Cell、RefCell、Mutex、Atomic*）底层都用 `UnsafeCell`
  - `UnsafeCell<T>` 对 `T` 的型变是不变（Invariant）——为什么？
  - `UnsafeCell<T>` 本身 `!Sync` —— 有多线程需求的类型需要自己保证
  - 极少直接使用，但理解它是理解整个内部可变性模型的前提

- [ ] **`Cell<T>` —— Copy 类型的零开销内部可变性** → [[2.15-cell-refcell-interior-mutability|📖]]
  - `Cell::get()` / `Cell::set()` —— 整体替换，无需借用
  - 只适用于 `T: Copy` 类型，因为 get 返回的是值拷贝而非引用
  - `Cell::replace()` / `Cell::take()` / `Cell::swap()`
  - 零运行时开销——没有借用计数的维护，与直接字段访问一样快
  - 典型场景：计数器、标志位、配置项

- [ ] **`RefCell<T>` —— 运行时借用检查的内部可变性** → [[2.15-cell-refcell-interior-mutability|📖]]
  - `RefCell::borrow()` → `Ref<T>`（不可变借用）、`RefCell::borrow_mut()` → `RefMut<T>`（可变借用）
  - 借用规则在**运行时**检查（而非编译期）：违反则 `panic!`
  - `RefCell<T>: Send` 但 `!Sync` —— 多个线程不能同时持有 `&RefCell<T>`
  - `RefCell::try_borrow()` / `try_borrow_mut()` —— 返回 `Result` 而非 panic
  - 典型场景：`Rc<RefCell<T>>` 共享可变状态

- [ ] **Cell / RefCell / Mutex / Atomic 选择指南** → [[2.15-cell-refcell-interior-mutability|📖]]
  - Cell：Copy 类型 + 单线程 → 零开销首选
  - RefCell：非 Copy 类型 + 单线程 → 运行时检查
  - Mutex：多线程 + 阻塞等待 → 跨线程安全
  - RwLock：多线程 + 读多写少 → 更细粒度的并发
  - Atomic*：整数/布尔/指针 + 多线程 → 无锁，硬件原子指令
  - 选择框架：从 Copy → 单线程 → 多线程 → 性能需求逐级升级

- [ ] **内部可变性与借用检查器的哲学关系** → [[2.15-cell-refcell-interior-mutability|📖]]
  - 为什么需要内部可变性？——共享引用（`&T`）默认不可变，但有些场景需要"通过共享引用修改"
  - 内部可变性不是绕过借用检查器——是把检查从编译期推迟到运行时（RefCell）或换了检查方式（Cell）
  - 与 C++ `mutable` 关键字的对比：`mutable` 是编译器层面绕过 const，Cell/RefCell 是类型系统层面明确定义

## 🏆 专家级学习原则

1. **"为什么"比"怎么用"更重要**。读 RFC、读源码、读设计讨论。
2. **读标准库源码**。标准库是 Rust 风格的教科书。
3. **追踪 nightly features**。了解语言演进方向，即使不用 nightly 写生产代码。
4. **贡献开源**。提交 PR 是最好的学习方式。
5. **写技术博客**。把学到的东西写出来，会发现自己的盲区。
6. **阅读 Rust 编译器源码**。哪怕只看懂借用检查器的一小部分。
7. **跨语言比较**。理解 Rust 的设计取舍 vs C++ / Haskell / Go / Zig。
8. **参与社区**。Discord / Zulip / Reddit / internals.rust-lang.org。
