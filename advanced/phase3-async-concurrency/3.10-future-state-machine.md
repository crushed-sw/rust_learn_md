# 3.10 Future 本质 & 状态机 —— 从 async/await 到编译器生成的枚举

> 每一个 `.await` 都是一个断点。当你写下 `async fn example() { step_a().await; step_b().await; }` 时，编译器为你生成了一个匿名的枚举类型，枚举的每个变体对应一个"等待后恢复执行"的位置。`Future` trait 的 `poll` 方法被运行时反复调用——每次调用，状态机推进一个状态，直到最终返回 `Poll::Ready`。`Pin` 则保证了这个自引用的状态机在内存中永远不会被移动——因为移动一个自引用结构等于破坏了内部指针。理解了这个模型，你就理解了 Rust 的 async 不是魔法——它是零成本抽象的极致实践：编译器生成你手写也很难更好的状态机代码。

## 1. Future Trait —— 三行代码定义一个范式

### 1.1 核心定义

Rust 的整个异步编程模型建立在 `Future` trait 之上——它只有三行，却定义了所有异步计算的抽象接口。

```rust
use std::pin::Pin;
use std::task::{Context, Poll};

// 这是 Rust 标准库中最重要也最简洁的 trait 之一
pub trait Future {
    type Output;

    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}
```

这个 trait 的三个组成部分各自承载了关键语义，我们需要逐一拆解。

### 1.2 `type Output` —— 异步计算的最终结果

`Output` 是一个关联类型，代表这个 Future 最终完成时产生的值的类型。每个 Future 有且只有一个 Output 类型——它贯穿整个状态机生命周期。

当你写 `async fn foo() -> i32` 时，编译器生成的匿名 Future 类型的 `Output = i32`。一个 Future 要么还没完成（返回 `Poll::Pending`），要么已经完成并带着 `Output` 类型的值返回（`Poll::Ready(Output)`），没有第三种状态。这种二值性（pending/ready）是设计上的选择——Future 没有"取消"或"错误"状态，取消通过丢弃 Future 实现，错误通过 `Output = Result<T, E>` 表达。

### 1.3 `self: Pin<&mut Self>` —— 固定住的自身引用

注意 `poll` 的 `self` 不是普通的 `&mut Self`，而是 `Pin<&mut Self>`。这是一个精妙的设计决策。

`Pin<&mut Self>` 表达了一个关键约束：**一旦这个 Future 开始被轮询，它在内存中的位置就不能再改变**。为什么需要这个约束？因为 async 代码块中跨 `.await` 的局部变量引用会形成自引用结构——如果 Future 被移动（比如被推进一个 `Vec`），那些引用会变成悬垂指针。`Pin` 通过类型系统使"移动 `Pin<P>` 指向的 `T`"成为 unsafe 操作，从而在 safe Rust 层面阻止了这类错误。

我们在第 4 节会用一个具体例子展示为什么移动 Future 会导致灾难。

### 1.4 `cx: &mut Context<'_>` —— 与运行时的通信通道

`Context` 是 Future 和运行时（executor）之间的唯一通信接口。在当前的标准库定义中，`Context` 的唯一职责是提供一个 `Waker`：

```rust
// 标准库中 Context 的简化定义
pub struct Context<'a> {
    waker: &'a Waker,
}

impl<'a> Context<'a> {
    /// 获取一个 Waker，可以用来唤醒关联的任务
    pub fn waker(&self) -> &'a Waker {
        self.waker
    }
    
    /// 从给定的 Waker 构建 Context
    pub fn from_waker(waker: &'a Waker) -> Context<'a> {
        Context { waker }
    }
}
```

当一个 Future 在 `poll` 中发现自己尚未准备好（例如 I/O 数据还没到达），它会把自己注册到某个事件源上，然后把 `cx.waker()` 的克隆交给事件源——"等我准备好了，调用这个 waker 通知你"。我们将在第 3 节深入 Waker 的内部机制。

### 1.5 `Poll<T>` —— 异步世界的 Option

```rust
pub enum Poll<T> {
    Ready(T),
    Pending,
}
```

`Poll` 是异步 Rust 中最频繁出现的类型之一。它的语义类似于 `Option<T>`，但有方向性的含义：

- `Ready(T)` 表示计算结果已就绪，Future 的任务完成。运行时拿到这个值后不会再调用 `poll`。
- `Pending` 表示还需要等待。运行时会安排稍后再次 `poll`。

关键设计要点：当 Future 返回 `Pending` 时，它**必须**确保 `cx.waker()` 已经被安排好——要么通过 I/O 事件触发，要么通过其他机制通知。如果 Future 返回 `Pending` 但没有安排 waker，这个 Future 就永远不会被再次轮询，形成"丢失的唤醒"（lost wakeup）bug。

## 2. 状态机——编译器如何转换 async fn

### 2.1 最简单的 async 函数及其生成的代码

考虑一个含有一个 `.await` 的 async 函数：

```rust
async fn simple_async() -> u32 {
    let a = async { 42 }.await;
    a + 1
}
```

编译器将这个函数展开成一个匿名 Future 类型。我们可以手动写出编译器大致生成的代码来理解这个过程：

```rust
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll};

// 编译器为 simple_async 生成的匿名类型（概念上的等价代码）
enum SimpleAsyncFuture {
    // 状态 0：初始状态，尚未开始执行
    Start,
    // 状态 1：正在等待内部的 future（async { 42 }）
    WaitingOnInner {
        inner: std::future::Ready<u32>,  // async { 42 } 被编译器优化为 Ready(42)
    },
    // 状态 2：已完成（不需要存储更多状态）
    Done,
}

impl Future for SimpleAsyncFuture {
    type Output = u32;

    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        loop {
            // 根据当前状态执行对应的代码段
            match &mut *self {
                SimpleAsyncFuture::Start => {
                    // 相当于 async fn 体中的第一条语句之前的代码
                    let inner = std::future::ready(42);
                    // 状态转换：Start → WaitingOnInner
                    *self = SimpleAsyncFuture::WaitingOnInner { inner };
                    // 不返回——立即继续执行 loop 的下一轮迭代
                    // （因为 ready(42) 是立即就绪的，不需要真正等待）
                }
                SimpleAsyncFuture::WaitingOnInner { inner } => {
                    // 相当于 .await 点——询问内部 future 是否完成
                    let inner_pin = unsafe { Pin::new_unchecked(inner) };
                    match inner_pin.poll(cx) {
                        Poll::Ready(value) => {
                            // inner future 完成了，拿到值 a = 42
                            let a = value;
                            let result = a + 1;
                            *self = SimpleAsyncFuture::Done;
                            return Poll::Ready(result);
                        }
                        Poll::Pending => {
                            // inner future 还没完成，返回 Pending
                            // 此时 self 保持为 WaitingOnInner 状态
                            // inner future 的 poll 内部已经设置了 waker
                            return Poll::Pending;
                        }
                    }
                }
                SimpleAsyncFuture::Done => {
                    // Future 已经完成——不应该再被 poll
                    panic!("Future polled after completion");
                }
            }
        }
    }
}
```

这个手动实现展示了编译器自动生成的代码的核心结构：每个 `.await` 点对应枚举的一个变体，`poll` 方法使用 `loop + match` 推进状态机，直至遇到真正的等待点。

### 2.2 多 await 点的状态机——变体如何累积

当 async 函数中有多个 `.await` 点时，状态机的变体会更多：

```rust
// 一个包含两个 .await 的 async 函数
async fn two_awaits() -> String {
    let a = fetch_id().await;   // 第一个 .await —— 获取用户 ID
    let b = fetch_name(a).await; // 第二个 .await —— 根据 ID 获取名称
    format!("User {}: {}", a, b)
}

// 假设这两个 async 函数
async fn fetch_id() -> u64 { /* ... */ }
async fn fetch_name(id: u64) -> String { /* ... */ }
```

编译器为 `two_awaits` 生成的枚举大致如下：

```rust
// 为了说明状态转换，这里用伪代码展示生成的状态枚举
// 实际生成的名字是匿名的（existential type），你无法在代码中引用
enum TwoAwaitsFuture {
    // 状态 0：开始——尚未执行任何代码
    Start,
    // 状态 1：正在等待 fetch_id() 完成
    // 需要存储 fetch_id() 生成的 Future 实例
    AwaitingId {
        id_future: <fetch_id() as Future>::Output,  // 概念上的，实际存储 Future
    },
    // 状态 2：fetch_id 已完成，正在等待 fetch_name
    // 此时 a（用户 ID）已经在 hand 中，需要保存在状态机中
    AwaitingName {
        a: u64,  // ← 跨 .await 存活的局部变量！
        name_future: <fetch_name() as Future>::Output,
    },
    // 状态 3：已完成
    Done,
}
```

注意 `AwaitingName` 变体同时保存了 `a: u64`（跨 `.await` 存活的局部变量）和正在等待的 `name_future`。这正是状态机需要 `Pin` 的核心原因——如果 `name_future` 内部持有指向 `a` 的引用，移动状态机就会破坏这个引用。

### 2.3 从源代码到枚举的完整转换路径

整个转换过程可以分步骤理解。以下是一个更完整的示例，展示了编译器对 async fn 的处理流水线：

```rust
// 源代码——你写的：
async fn compute(value: i32) -> i32 {
    let doubled = async { value * 2 }.await;  // await 点 1
    let tripled = async { doubled * 3 }.await; // await 点 2
    tripled + value
}

// 编译器处理步骤：
//
// 步骤 1：降低 async fn 为一个返回 impl Future 的普通函数
// fn compute(value: i32) -> impl Future<Output = i32> {
//     async move {  // ← 函数参数被 move 进 async block
//         let doubled = async { value * 2 }.await;
//         let tripled = async { doubled * 3 }.await;
//         tripled + value
//     }
// }
//
// 步骤 2：将 async block 转换为一个状态机
// （编译器生成类似下面结构的匿名枚举 + Future impl）
//
// 步骤 3：MIR（中级中间表示）优化——内联、常量传播等
//
// 步骤 4：LLVM 后端生成机器码
```

这就是 async fn 从"看起来像同步代码"到"实际的轮询驱动状态机"的完整旅程。

## 3. Context 与 Waker —— 唤醒机制的内部原理

### 3.1 Waker 不是简单的回调函数

`Waker` 看起来像是一个简单的"通知回调"，但它的设计要复杂得多——它必须是 `Send + Sync + Clone` 的，并且不持有任务本身的生命周期。

```rust
// Waker 的核心 API（简化版）
pub struct Waker {
    // 内部持有一个 RawWaker —— 一个 vtable + data pointer 的组合
}

impl Waker {
    /// 唤醒关联的任务——poll 应该被再次调用
    pub fn wake(self) {
        // 实际调用内部 RawWaker 的 vtable.wake 函数指针
        // 这个操作会把任务重新排入调度队列
        let wake_fn = /* vtable.wake */;
        wake_fn(/* data pointer */);
    }

    /// 克隆一个 Waker（Clone trait 的实现）
    /// 克隆的 Waker 和原始 Waker 唤醒同一个任务
    pub fn wake_by_ref(&self) {
        // 不消耗 self，通过引用唤醒
    }
}
```

`Waker` 设计的一个关键洞察是：**它不持有对任务数据的引用**。`Waker` 内部是一个数据指针（`*const ()`）和一个虚函数表（vtable）。这意味着 Waker 可以在任务被取消后仍然存活而不会导致悬垂引用——这正是 `Send + Sync` 得以实现的原因。

### 3.2 RawWaker 和 RawWakerVTable —— vtable 机制

`Waker` 的底层实现依赖两样东西：一个类型擦除的数据指针和一个函数指针表。

```rust
// 标准库中的定义（概念版本）
pub struct RawWaker {
    data: *const (),      // 类型擦除的任务指针
    vtable: &'static RawWakerVTable,
}

pub struct RawWakerVTable {
    // 每个函数接收 data 指针作为第一个参数
    clone: unsafe fn(*const ()) -> RawWaker,
    wake: unsafe fn(*const ()),
    wake_by_ref: unsafe fn(*const ()),
    drop: unsafe fn(*const ()),
}
```

不同的运行时（tokio / async-std / smol / embassy）提供不同的 vtable 实现。比如 tokio 的 vtable 中的 `wake` 函数会把它自己任务结构体的指针重新放入 tokio 的任务队列中。

下面是构建 Waker 的概念性示意——展示 vtable 机制如何连接运行时和 Future：

```rust
// 概念上，tokio 的 Waker 创建流程是这样的
// （实际代码更复杂，这里展示核心思想）

use std::task::{RawWaker, RawWakerVTable, Waker};

// tokio 内部的任务结构体
struct Task {
    // ... 任务状态、Future、调度信息 ...
}

// vtable 的具体实现 —— 这些函数实际上是由 tokio 运行时提供的
unsafe fn tokio_waker_clone(data: *const ()) -> RawWaker {
    // 增加任务的引用计数
    let task = data as *const Task;
    // refcount_inc(task);
    RawWaker::new(data, &TOKIO_VTABLE)
}

unsafe fn tokio_waker_wake(data: *const ()) {
    let task = data as *const Task;
    // 把 task 重新放回 tokio 的调度队列
    // tokio::runtime::scheduler::enqueue(task);
}

unsafe fn tokio_waker_wake_by_ref(data: *const ()) {
    // 类似 wake，但不消耗引用计数
    let task = data as *const Task;
    // tokio::runtime::scheduler::enqueue(task);
}

unsafe fn tokio_waker_drop(data: *const ()) {
    let task = data as *const Task;
    // refcount_dec(task);
}

// 静态的 vtable —— 整个进程只有这一份
static TOKIO_VTABLE: RawWakerVTable = RawWakerVTable::new(
    tokio_waker_clone,
    tokio_waker_wake,
    tokio_waker_wake_by_ref,
    tokio_waker_drop,
);

// 从 Task 构建 Waker
fn build_waker(task: *const Task) -> Waker {
    let raw = RawWaker::new(task as *const (), &TOKIO_VTABLE);
    unsafe { Waker::from_raw(raw) }
}
```

这个 vtable 设计使得 Waker 成为运行时无关的——同一个 `Waker` 类型可以适配任何运行时，无需泛型参数。这是 Rust 异步生态系统的关键设计决策。

### 3.3 唤醒的生命周期——从事件到 poll

完整的事件循环可以描述为以下流程：

```
1. Future::poll() 被运行时调用
2. Future 发现自己尚未就绪（例如 socket 上没有数据）
3. Future 调用 cx.waker().clone() 获取一个 Waker
4. Future 把 Waker 注册到事件源上（例如 epoll/kqueue/IOCP）
5. Future 返回 Poll::Pending
6. 运行时记录：此任务等待中，去处理其他任务
7. ... 时间流逝 ...
8. 事件发生（socket 可读了）
9. 事件循环检测到事件
10. 事件循环调用 Waker::wake()  ← 之前注册的那个
11. Waker 将任务重新放入调度队列
12. 运行时再次调用 Future::poll()
13. 这一次，I/O 就绪了，Future 继续执行
14. Future 最终返回 Poll::Ready(T)
```

关键要点：`Waker::wake()` 本身**不是**直接调用 `poll`——它只是将任务标记为"就绪"，放入调度队列。实际的 `poll` 调用发生在上层运行时的事件循环中。

## 4. Pin 的必要性——自引用状态机与移动语义的冲突

### 4.1 为什么 async 状态机必然是自引用的

考虑以下 async 函数：

```rust
async fn self_referential_example() {
    let x = String::from("hello");
    let y = &x;           // y 持有对 x 的引用
    some_async_fn().await; // ← 跨越 .await 的引用！
    println!("{}", y);
}
```

编译器生成的状态机需要在 `AwaitingSomeAsync` 状态下同时存储：
- `x`: `String` —— 字符串的所有权在此
- `y`: `&String` —— 指向 x 的引用
- 正在等待的 future

如果这个状态机被移动（内存位置改变），`y` 中的指针仍然指向旧的 `x` 位置——变成了悬垂指针。

下面是展示该问题的一个具体示例：

```rust
// 这个例子用普通结构体模拟 async 状态机的自引用困境
struct SelfReferential {
    value: String,
    // 危险！这个字段指向 self.value
    // 如果 SelfReferential 被移动，此指针变悬垂
    reference: *const String,
}

impl SelfReferential {
    fn new(value: String) -> Self {
        let mut s = SelfReferential {
            value,
            reference: std::ptr::null(),
        };
        s.reference = &s.value as *const String;
        s
    }

    fn check(&self) {
        unsafe {
            // 这始终成立——只要 SelfReferential 没被移动
            assert_eq!(&*self.reference as *const String, 
                       &self.value as *const String);
        }
    }
}

let s = SelfReferential::new("hello".into());
s.check(); // ✅ 成立

// 移动！value 字段被复制到新位置，但 reference 仍然指向旧位置
let s_moved = s;

// s_moved.check(); // ❌ 现在 reference 指向的是 s 的 value（已被 drop）
// 断言失败——reference 和 &self.value 不相等了
```

这就是编译器强制要求 `Pin` 的原因。`Pin<&mut Self>` 保证了你无法通过 safe Rust 代码移动 Future——而 `poll` 接收 `Pin<&mut Self>` 正是为了利用这个保证。

### 4.2 Pin 如何保护 async 状态机

`Pin<P>` 的运作方式是：如果 `P` 的解引用目标（`P::Target`）不是 `Unpin`，那么 `Pin<P>` 不提供获取 `&mut P::Target` 的 safe API。

```rust
use std::marker::PhantomPinned;
use std::pin::Pin;

// 编译器为 async block 生成的匿名类型一定不是 Unpin
// 这意味着它被 Pin<&mut Self> 包裹后，无法通过 safe 方式被移动

// 手动模拟——一个 !Unpin 的结构体
struct NotUnpin {
    _marker: PhantomPinned,  // PhantomPinned 是 !Unpin 的
    data: String,
}

// Pin 保护的语义：
let mut value = NotUnpin {
    _marker: PhantomPinned,
    data: String::from("hello"),
};

// 把 value 固定到栈上
let pinned = unsafe { Pin::new_unchecked(&mut value) };

// 现在无法通过 safe Rust 获取 &mut NotUnpin 来移动它
// 只能通过 pinned 进行受限的操作

// 但你可以通过 Pin::as_mut() 获取 Pin<&mut NotUnpin>
// 或通过 Pin::get_mut() 获取 &mut T（如果 T: Unpin——但 NotUnpin 不是）
```

这就是编译器利用 `Pin` 保证 async 状态机内存安全的完整图景：Future 的类型不是 `Unpin`，`poll` 接收 `Pin<&mut Self>`，因此运行时无法在 safe Rust 中移动正在被轮询的 Future。

### 4.3 什么情况下 async 的 Future 是 Unpin

并非所有 async 生成的状态机都需要 Pin。如果编译器检测到状态机中**没有跨 .await 的引用**，它可以标记生成的 Future 为 `Unpin`，从而消除 Pin 的约束。

```rust
// 这个 async fn 不需要 Pin——没有跨 .await 的引用
async fn no_cross_await_ref() -> i32 {
    let x = 42;          // 没有引用
    tokio::time::sleep(std::time::Duration::from_millis(1)).await;
    // sleep 的 future 不需要引用 x
    x + 1
}

// 这个 async fn 需要 Pin——有跨 .await 的引用
async fn has_cross_await_ref() -> String {
    let s = String::from("hello");
    let r = &s;           // r 引用了 s
    some_io().await;       // ← r 跨过了 .await
    r.to_uppercase()
}
```

但你不能依赖编译器"总是"标记为 Unpin——除非你在编写对 Pin 敏感的 unsafe 代码，否则你应该始终假设 async Future 是 `!Unpin` 的。

## 5. 手动实现 Future——从零构建一个异步原语

### 5.1 最简单的 Future：立即就绪的值

理解 Future 最好的方式就是手写一个：

```rust
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll};

/// 一个永远立即就绪的 Future——最简单的手动实现
struct AlwaysReady<T> {
    value: Option<T>,
}

impl<T> Future for AlwaysReady<T> {
    type Output = T;

    fn poll(mut self: Pin<&mut Self>, _cx: &mut Context<'_>) -> Poll<Self::Output> {
        // take() 取出值——因为 poll 可能被多次调用，需要确保只返回一次
        match self.value.take() {
            Some(v) => Poll::Ready(v),
            None => panic!("AlwaysReady polled after returning Ready"),
        }
    }
}
```

这个实现展示了 Future 的三个基本规则：第一，`poll` 返回 `Ready` 后不应再被调用；第二，使用 `Option` + `take()` 是消费 Future 内部值的标准模式；第三，`AlwaysReady` 不需要 waker——因为它永远不会返回 `Pending`。

### 5.2 一个需要等待的 Future：手动实现定时器

下面实现一个"等到某个时刻"的 Future，展示 waker 注册、pending 返回和唤醒的完整流程：

```rust
use std::future::Future;
use std::pin::Pin;
use std::sync::{Arc, Mutex};
use std::task::{Context, Poll, Waker};
use std::time::Instant;

/// 一个朴素的定时器 Future——到指定时刻才就绪
struct TimerFuture {
    deadline: Instant,
    // 使用 Arc<Mutex<Option<Waker>>> 来存储 waker
    // 真实实现会用原子操作或更高效的结构
    waker: Arc<Mutex<Option<Waker>>>,
}

impl TimerFuture {
    fn new(deadline: Instant) -> Self {
        TimerFuture {
            deadline,
            waker: Arc::new(Mutex::new(None)),
        }
    }
}

impl Future for TimerFuture {
    type Output = ();

    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        // 检查是否已经到了截止时间
        if Instant::now() >= self.deadline {
            // 时间到了——立即就绪
            return Poll::Ready(());
        }

        // 还没到时间——需要等待
        // 保存 waker，以便稍后被唤醒
        let mut saved_waker = self.waker.lock().unwrap();
        *saved_waker = Some(cx.waker().clone());

        // 在真实的运行时中，这里会：
        // 1. 把 waker 和 deadline 注册到定时器轮（timer wheel）中
        // 2. 当时刻到达时，定时器轮调用 waker.wake()
        //
        // 但在这个简化版中，我们用另一个线程来模拟定时器：
        let waker_clone = self.waker.clone();
        let deadline = self.deadline;
        std::thread::spawn(move || {
            let now = Instant::now();
            if now < deadline {
                std::thread::sleep(deadline - now);
            }
            // 唤醒任务
            if let Some(waker) = waker_clone.lock().unwrap().take() {
                waker.wake();
            }
        });

        Poll::Pending
    }
}
```

这个实现的价值在于展示了完整的"挂起-注册-唤醒"循环。注意几个关键设计点：用 `Arc<Mutex<Option<Waker>>>` 存储 waker 以便从外部线程访问；`take()` 确保 wake 只被调用一次；在 `poll` 中检查条件——如果条件已满足就直接返回 `Ready`，避免不必要的延迟。

### 5.3 一个多状态转换的 Future

当 Future 需要经历多个异步步骤时，状态机的模式变得清晰：

```rust
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll};

/// 一个经历两个阶段才完成的 Future
/// 阶段 1：等待初始化
/// 阶段 2：等待最终完成
enum TwoStepFuture {
    Init,
    Working { steps_remaining: u32 },
    Done,
}

impl Future for TwoStepFuture {
    type Output = String;

    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        loop {
            match &mut *self {
                TwoStepFuture::Init => {
                    println!("TwoStepFuture: 初始化阶段");
                    // 立刻进入 Working 状态
                    *self = TwoStepFuture::Working { steps_remaining: 3 };
                    // 不返回——loop 会立即处理新状态
                }
                TwoStepFuture::Working { steps_remaining } => {
                    if *steps_remaining == 0 {
                        println!("TwoStepFuture: 所有步骤完成");
                        *self = TwoStepFuture::Done;
                        // 继续 loop——下一轮迭代进入 Done
                    } else {
                        println!("TwoStepFuture: 还剩 {} 步", steps_remaining);
                        // 模拟一次异步操作——唤醒自己
                        cx.waker().wake_by_ref();
                        *steps_remaining -= 1;
                        return Poll::Pending;
                    }
                }
                TwoStepFuture::Done => {
                    return Poll::Ready(String::from("two steps complete"));
                }
            }
        }
    }
}
```

这段代码展示了 `loop + match` 模式的力量：当状态转换后不需要等待（比如 `Init → Working`），`loop` 让代码立即进入下一个状态的匹配分支，而无需返回到 executor 等待下一次 `poll`。这种"批量推进"减少了不必要的挂起和唤醒，是高效的异步状态机设计的核心技巧。

## 6. 叶子 Future 与组合子 Future —— 两种角色的分工

### 6.1 叶子 Future（Leaf Future）

叶子 Future 是异步世界的"原子操作"——它们不依赖其他 Future，直接与 I/O、定时器或同步原语交互。

```rust
// 叶子 Future 的特征：
// 1. 它不会在内部 .await 其他 Future
// 2. 它在 poll() 中直接检查资源状态（socket、文件、定时器等）
// 3. 它是"真正的"异步原语——一切 async 的起点
// 4. 通常由运行时库（tokio、async-std 等）提供

// 例如：
// - tokio::net::TcpStream 的 read —— 直接操作 epoll/kqueue/IOCP
// - tokio::time::Sleep —— 直接操作定时器轮
// - tokio::sync::Mutex::lock —— 直接操作内部等待队列
// - 你手动实现的 TimerFuture（上一节）也是一个叶子 Future
```

叶子 Future 是运行时与操作系统之间的桥梁。它们知道如何与 epoll（Linux）、kqueue（macOS）或 IOCP（Windows）交互，将操作系统的异步通知转换为 Rust 的 `Poll::Ready`。

### 6.2 组合子 Future（Combinator Future）

组合子 Future 组合或转换其他 Future——它们是 async/await 的"胶水代码"。

```rust
// async fn 和 async block 生成的都是组合子 Future
// 它们在 poll() 中调用内部 Future 的 poll()，在所有内部 Future 都 Ready 后才 Ready

// async fn 生成的 Future —— 最典型的组合子
async fn my_combinator() -> String {
    let a = leaf_future_a().await;  // ← 在 poll 中调用 leaf_future_a 的 poll
    let b = leaf_future_b(a).await; // ← 在 poll 中调用 leaf_future_b 的 poll
    format!("result: {}-{}", a, b)
}

// 手动组合子示例：一个"轮流执行两个 Future"的组合子
struct Race<A: Future, B: Future> {
    a: A,
    b: B,
}

impl<A: Future, B: Future> Future for Race<A, B> {
    type Output = ();

    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        // unsafe 是必需的，因为我们需要把 Pin<&mut Self> 投影到 Pin<&mut A> 和 Pin<&mut B>
        let this = unsafe { self.get_unchecked_mut() };
        
        // 检查 a 是否完成
        // （此处简化——实际需要 Pin 投影）
        // 如果两个都还没 Ready，就返回 Pending
        
        Poll::Pending
    }
}
```

组合子 Future 和叶子 Future 的关系类似于"业务代码"与"基础设施代码"：你写的每个 async fn 都是组合子 Future，它最终依赖的是 tokio 等运行时提供的叶子 Future。

## 7. Executor 的轮询循环——运行时如何驱动状态机

### 7.1 一个极简 Executor 的实现

要理解整个异步系统是如何运作的，最好的方式是亲手写一个 Executor。以下是单线程上最简单的实现：

```rust
use std::cell::RefCell;
use std::collections::VecDeque;
use std::future::Future;
use std::pin::Pin;
use std::rc::Rc;
use std::task::{Context, Poll, RawWaker, RawWakerVTable, Waker};

/// 一个极度简化的单线程 executor
/// 展示"任务队列 + 轮询循环"的基本结构
struct MiniExecutor {
    // 用 Rc<RefCell<>> 是因为单线程 + 教学目的
    tasks: Rc<RefCell<VecDeque<Pin<Box<dyn Future<Output = ()>>>>>>,
}

impl MiniExecutor {
    fn new() -> Self {
        MiniExecutor {
            tasks: Rc::new(RefCell::new(VecDeque::new())),
        }
    }

    /// 将一个 Future 提交到 executor
    fn spawn<F: Future<Output = ()> + 'static>(&self, future: F) {
        self.tasks.borrow_mut().push_back(Box::pin(future));
    }

    /// 核心轮询循环——运行所有任务直到没有任务剩余
    fn run(&self) {
        // 构建一个总是唤醒自己的 Waker
        let waker = Self::create_waker();

        while let Some(mut task) = self.tasks.borrow_mut().pop_front() {
            // 每次 poll 时创建 Context
            let mut cx = Context::from_waker(&waker);

            match task.as_mut().poll(&mut cx) {
                Poll::Ready(()) => {
                    // 任务完成——丢弃它
                    println!("Task completed!");
                }
                Poll::Pending => {
                    // 尚未完成——放回队尾继续等待
                    self.tasks.borrow_mut().push_back(task);
                }
            }
        }
    }

    /// 构建 Waker——当被调用时，它不直接做任何复杂事情
    /// 在真实运行时中，wake 会把任务放回队列
    fn create_waker() -> Waker {
        // 构建最小可行的 vtable + Waker
        // 这个 waker 的 wake 不做什么——因为我们在 busy-loop 中不断 poll
        fn noop_clone(_data: *const ()) -> RawWaker {
            RawWaker::new(std::ptr::null(), &NOOP_VTABLE)
        }
        fn noop_wake(_data: *const ()) {
            // 在这个极简实现中，wake 是 noop
            // 因为 run() 会不断轮询队列中的所有任务
        }
        fn noop_drop(_data: *const ()) {}

        static NOOP_VTABLE: RawWakerVTable = RawWakerVTable::new(
            noop_clone,
            noop_wake,
            noop_wake,  // wake_by_ref 和 wake 一样
            noop_drop,
        );

        let raw = RawWaker::new(std::ptr::null(), &NOOP_VTABLE);
        unsafe { Waker::from_raw(raw) }
    }
}
```

这个极简 Executor 展示了核心的轮询循环模式：从队列中取出任务 → poll → 如果 Pending 就放回队尾 → 继续下一个。真实的 tokio Executor 比这复杂得多（多线程、工作窃取、I/O 事件驱动），但核心循环逻辑是一致的。

### 7.2 真实的 Executor 循环逻辑

真实 Executor 的关键区别在于：它不会 busy-loop（空转消耗 CPU）。它使用 epoll/kqueue/IOCP 等待事件，只在有事件时轮询对应任务：

```
真实 Executor 的主循环（简化版）：

loop {
    // 1. 轮询所有"就绪"的任务（刚从等待队列中出来的）
    while let Some(task) = ready_queue.pop() {
        match task.poll() {
            Ready    → 任务完成，drop
            Pending  → 任务注册了 waker，等待事件
        }
    }
    
    // 2. 如果没有就绪任务，阻塞等待 I/O 事件
    //    epoll_wait(epoll_fd, events, timeout);
    //    或者如果没有定时器，可以无限期等待
    
    // 3. 处理 I/O 事件
    for event in io_events {
        // 找到事件关联的 waker，放入 ready_queue
        waker_by_event_id[event.id].wake();
    }
    
    // 4. 处理到期的定时器
    for expired_timer in timer_wheel.expired() {
        expired_timer.waker.wake();
    }
    
    // 5. 回到步骤 1
}
```

这个循环是 Rust 异步运行时的核心——它驱动着从操作系统事件到你的 async fn 之间的整个链条。

## 8. Async 的零成本承诺——为什么不需要分配

### 8.1 零成本的含义

Rust 的 async 被宣传为"零成本抽象"。零成本的具体含义是：如果你不使用它，你不会为它付费；如果你使用它，你支付的开销不比你手写状态机更大。

```rust
// 使用 async/await：
async fn compute() -> i32 {
    let x = read_number().await;
    x * 2
}

// 编译器生成的代码在功能上等同于手动写枚举状态机
// ——不会凭空多出：
//   - 堆分配（除非你显式 Box::pin）
//   - 虚函数调用（状态转换通过 match，静态分发）
//   - 引用计数（除非你的运行时用了 Arc）
//   - GC（Rust 没有 GC）
```

### 8.2 什么时候需要堆分配

唯一强制需要堆分配的时机是：你需要将 Future 类型擦除为 `dyn Future` 或将 Future 的所有权传递给运行时的任务队列。

```rust
// 场景 1：动态分发——需要 trait object
// 这需要 Box<dyn Future>，堆分配
fn return_different_futures(flag: bool) -> Pin<Box<dyn Future<Output = i32>>> {
    if flag {
        Box::pin(async { 42 })
    } else {
        Box::pin(async { 24 })
    }
}

// 场景 2：递归 async fn——需要 Box 来打破无限大小的类型
// async fn recurse(n: u32) {
//     if n == 0 { return; }
//     recurse(n - 1).await;  // ← 编译错误：递归 async fn
// }
// 必须 Box::pin

// 场景 3：spawn 到运行时——需要 'static + Send + 类型擦除
// tokio::spawn(async { ... });  // 内部使用 Box
```

除了这三种情况，Rust 的 async 代码可以在栈上运行，完全不需要堆分配。这是 Rust async 相比于 Go goroutine（每个 goroutine 需要自己的栈）、C# async（通过堆分配的状态机）等模型的一个关键性能优势。

## 9. 生命周期的完整图景——从 Pending 到 Ready

### 9.1 完整生命周期图（文字版）

下面用 Rust 伪代码展示一个 Future 从创建到完成的完整旅程：

```rust
// 这是一个概念性的演示——展示完整的时间线
// 不是可运行的代码，而是对生命周期各阶段的标注

// === 阶段 1：创建 ===
let future = async {
    read_socket(&mut buf).await  // 需要等待 I/O
};

// === 阶段 2：提交给 executor ===
// executor.spawn(future);  // future 被固定到堆上或栈上

// === 阶段 3：第一次 poll ===
// executor 调用 future.poll(cx)
//  → Future 进入 poll()
//  → 执行到 read_socket().await
//  → read_socket 的 poll() 被调用
//  → socket 还没数据——注册 waker 到 epoll
//  → 返回 Poll::Pending

// === 阶段 4：等待 ===
// executor 记录：此任务挂起
// executor 调用 epoll_wait() 阻塞等待
// ... 数据到达 socket ...
// epoll 返回：socket fd 可读

// === 阶段 5：唤醒 ===
// executor 找到该 fd 关联的 waker
// 调用 waker.wake() → 把 future 放回就绪队列

// === 阶段 6：第二次 poll ===
// executor 再次调用 future.poll(cx)
//  → Future 从上次 .await 之后恢复执行
//  → read_socket 的 poll() 被调用
//  → socket 有数据了——读入 buf
//  → 返回 Poll::Ready(data)

// === 阶段 7：完成 ===
// executor 收到 Poll::Ready
// future 被 drop
// 如果有 join!() 或 select!()，其他分支继续执行
```

### 9.2 关键保证和边界条件

在理解上述生命周期时，以下保证至关重要：

```rust
// 保证 1：Future 在返回 Ready 后不会再被 poll
// （违反此保证的行为是未定义的）

// 保证 2：Future 在返回 Pending 时必须已设置 waker
// （否则 executor 永远不会再次 poll——"lost wakeup" bug）

// 保证 3：poll 的 Pin<&mut Self> 意味着 Future 在 poll 期间不会被移动
// （编译器 + Pin API 强制执行）

// 保证 4：同一个 Future 的两次 poll 调用之间可能间隔任意长的时间
// （不能对时序做任何假设）
```

### 练习 1 答案

```rust
use std::future::Future;
use std::pin::Pin;
use std::sync::{Arc, Mutex};
use std::task::{Context, Poll, Waker};
use std::time::{Duration, Instant};

struct Delay {
    deadline: Instant,
    waker: Arc<Mutex<Option<Waker>>>,
    started: bool,
}

impl Delay {
    fn new(duration: Duration) -> Self {
        Delay {
            deadline: Instant::now() + duration,
            waker: Arc::new(Mutex::new(None)),
            started: false,
        }
    }
}

impl Future for Delay {
    type Output = ();

    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        if !self.started {
            // 第一次 poll：启动后台定时线程
            self.started = true;
            let waker = self.waker.clone();
            let deadline = self.deadline;
            std::thread::spawn(move || {
                let now = Instant::now();
                if now < deadline {
                    std::thread::sleep(deadline - now);
                }
                if let Some(w) = waker.lock().unwrap().take() {
                    w.wake();
                }
            });
        }

        if Instant::now() >= self.deadline {
            Poll::Ready(())
        } else {
            // 关键：在检查时间之后、返回 Pending 之前注册 waker
            // 此时定时线程可能已经完成，但 waker 尚未注册
            // 所以注册后需要再检查一次时间（double check）
            *self.waker.lock().unwrap() = Some(cx.waker().clone());

            // Double check——防止在注册 waker 之前定时器已经触发
            if Instant::now() >= self.deadline {
                Poll::Ready(())
            } else {
                Poll::Pending
            }
        }
    }
}
```

```rust
use std::future::Future;
use std::pin::Pin;
use std::sync::{Arc, Mutex};

    ready: Arc<Mutex<bool>>,
    waker: Mutex<Option<Waker>>,
}

impl Future for FixedFuture {
    type Output = ();

    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        // 步骤 1：先注册 waker
        *self.waker.lock().unwrap() = Some(cx.waker().clone());

        // 步骤 2：再检查 ready（顺序关键！）
        let ready = *self.ready.lock().unwrap();

        if ready {
            Poll::Ready(())
        } else {
            // 此时即使另一个线程在步骤 1 和 2 之间将 ready 设为 true，
            // 并且调用了 waker.wake()，也没关系：
            // - 如果 wake 在步骤 1 之后（waker 已注册）——wake 成功
            // - 如果 wake 在步骤 2 之前（ready 还没读）——步骤 2 读到 true，返回 Ready
            // - 如果 wake 在步骤 2 之后（ready 是 false）——waker 已注册，正常线程唤醒
            Poll::Pending
        }
    }
}

// 关键修复：先注册 waker，再检查条件——这是所有异步 API 实现的标准模式
// 这被称为"注册后检查"（register-then-check）模式
```
