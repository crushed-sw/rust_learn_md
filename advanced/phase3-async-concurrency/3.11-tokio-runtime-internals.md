# 3.11 Tokio 运行时内部 —— 工作窃取、I/O 驱动与定时器轮

> Tokio 不仅仅是一个"帮你 poll Future"的循环——它是一个拥有多线程工作窃取调度器、epoll/kqueue/IOCP 抽象层、高精度定时器轮、以及双线程池架构的生产级异步运行时。当你写下 `#[tokio::main]` 时，这个宏为你生成了 `Runtime::new().block_on()` 的完整启动代码。理解 Tokio 的内部结构，能让你合理配置运行时参数、正确选择 `spawn` 与 `spawn_blocking`、避免"饿死"其他任务的长计算——并且意识到 Rust 的 async 运行时并非语言内置，而是一个可以用标准库和 mio 自实现的库。

---

## 1. Tokio 的整体架构——四个核心层次

### 1.1 从上到下的分层视图

Tokio 的运行时可以从上到下分为四层。理解这四层各自的职责和交互方式，是理解整个 Tokio 架构的关键。

```
┌─────────────────────────────────────────────────────────────┐
│ 第一层：用户代码层                                            │
│   async fn、select!、join!、tokio::spawn、spawn_blocking     │
├─────────────────────────────────────────────────────────────┤
│ 第二层：调度器层（Scheduler）                                 │
│   多线程工作窃取调度器、任务队列、worker 线程管理              │
├─────────────────────────────────────────────────────────────┤
│ 第三层：驱动层（Driver）                                      │
│   I/O Driver（epoll/kqueue/IOCP 封装）                       │
│   Timer Driver（hashed timer wheel）                         │
│   Signal Driver（Unix 信号处理）                              │
├─────────────────────────────────────────────────────────────┤
│ 第四层：操作系统接口层                                        │
│   mio（跨平台事件循环库）→ epoll / kqueue / IOCP             │
└─────────────────────────────────────────────────────────────┘
```

用户代码（async fn）跑在调度器之上。调度器管理着 worker 线程的生命周期和任务分发。驱动层是调度器与操作系统的桥梁——I/O 驱动负责网络和文件 I/O，定时器驱动负责 `sleep` / `interval` / `timeout`。最底层是 mio——它提供了跨平台的事件循环抽象。

### 1.2 Tokio 的核心 Crate 组成

Tokio 不是一个单独的 crate，而是一个精心设计的 crate 集合。以下是核心 crate 及其职责：

```rust
// tokio 的 crate 结构（概念上的）：

// tokio        —— 用户面对的"主 crate"，重新导出所有子 crate
// tokio-macros —— proc macro 实现（#[tokio::main], #[tokio::test]）
// tokio-rt     —— 运行时核心：调度器、驱动、block_on
// tokio-net    —— 异步网络类型：TcpStream, TcpListener, UdpSocket
// tokio-io     —— 异步 I/O trait：AsyncRead, AsyncWrite
// tokio-sync   —— 异步同步原语：Mutex, RwLock, Semaphore, Notify, watch, broadcast, mpsc
// tokio-time   —— 定时器：sleep, timeout, interval
// tokio-fs     —— 异步文件操作（内部用 spawn_blocking 实现）
// tokio-util   —— 额外的工具类型：StreamExt, SinkExt, codec
// tokio-stream —— Stream trait 实现和组合子

// 底层依赖：
// mio         —— 跨平台事件循环库
// bytes       —— 高效的字节缓冲区（用于网络）
// tracing     —— 结构化日志（用于 tokio 内部的诊断信息）
```

## 2. 多线程工作窃取调度器——核心调度机制

### 2.1 为什么是工作窃取

Tokio 的多线程运行时采用"工作窃取"（work-stealing）调度模型。每个 worker 线程有自己的本地任务队列，当本地队列为空时，该 worker 会从其他 worker 的队列中"窃取"任务：

```
       Worker 1          Worker 2          Worker 3          Worker 4
    ┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
    │ 本地队列    │    │ 本地队列    │    │ 本地队列    │    │ 本地队列    │
    │ [T1][T2]   │    │ [T3]       │    │ (空)        │    │ [T4][T5]   │
    └────────────┘    └────────────┘    └────────────┘    └────────────┘
         ↓ 窃取?           ↑ Worker 3 从 Worker 1 或 Worker 4 窃取
    ┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
    │ 全局队列    │    │            │    │            │    │            │
    │ [T6]       │    │  ← spawn 的新任务先放这里       │            │
    └────────────┘    └────────────┘    └────────────┘    └────────────┘
```

工作窃取的优势在于：负载自动均衡——忙的 worker 不需要主动分发任务，闲的 worker 主动去找任务。这比"全局队列 + 互斥锁"（所有线程从同一个队列取任务，竞争一把锁）有更好的扩展性。

### 2.2 任务数据结构的内部表示

每个 `tokio::spawn` 调用创建的任务，在 tokio 内部有复杂的结构。下面是概念性的简化展示：

```rust
// tokio 内部任务结构的概念模型（大大简化版）
// 实际实现在 tokio-rt 的 task 模块中

use std::future::Future;
use std::pin::Pin;
use std::task::Waker;

// 每个任务的核心数据（简化）
struct Task {
    // 任务的状态——原子变量存储
    // 状态包括：是否已调度、是否已完成、是否有 waker 等待等
    state: std::sync::atomic::AtomicUsize,
    
    // 任务的 Future——存储在堆上（Box + Pin）
    // 类型擦除为 dyn Future
    future: UnsafeCell<Pin<Box<dyn Future<Output = ()> + Send>>>,
    
    // 该任务的调度信息——指向所属运行时的引用
    // scheduler: *const Scheduler,
}

// 实际的实现还包含：
// - 输出结果的存储（JoinHandle 返回的位置）
// - 引用计数（多个 JoinHandle + waker 共享所有权）
// - 任务的 ID（用于 tracing）
// - 预算（budget）计数器（用于协作调度——见第 6 节）
```

每个 spawn 的 Future 都必须被放在堆上（Box），因为 tokio 需要在不同线程之间传递任务的所有权，而 Future 的类型是大小未知的（编译器生成的匿名类型）。

### 2.3 Worker 线程的生命周期

每个 worker 线程运行以下核心循环：

```rust
// worker 线程主循环的伪代码（概念）
// 实际代码在 tokio-rt 中分散在多个模块里

fn worker_loop(worker: &Worker) {
    loop {
        // 步骤 1：尝试从本地队列获取任务
        let task = worker.local_queue.pop()
            .or_else(|| {
                // 步骤 2：本地队列为空，尝试从全局队列获取
                worker.inject_queue.pop()
            })
            .or_else(|| {
                // 步骤 3：全局也为空，尝试从其他 worker 窃取
                worker.steal_from_others()
            });
        
        let Some(task) = task else {
            // 步骤 4：所有队列都是空的——park（休眠等待事件）
            // 此时阻塞在 epoll_wait / kqueue / IOCP 上
            // 当 I/O 事件唤醒此线程时，任务被自动放入本地队列
            worker.park();
            continue;
        };
        
        // 步骤 5：执行任务——调用 Future::poll
        // 重置该任务的协作调度预算
        task.reset_budget();
        
        let result = task.poll();
        
        match result {
            Poll::Ready(output) => {
                // 任务完成——通知 JoinHandle
                task.complete(output);
            }
            Poll::Pending => {
                // 任务未完成——它已在 I/O 驱动或定时器上注册了 waker
                // 不需要重新入队（waker 会在事件发生时自动将其入队）
            }
        }
    }
}
```

这个循环的关键思想是：poll 不是 busy-loop——当所有任务都在等待 I/O 时，worker 线程通过 `park()` 休眠，由操作系统在 I/O 事件到达时将其唤醒。这就是"事件驱动"的含义。

## 3. I/O 驱动层——操作系统异步的抽象

### 3.1 epoll / kqueue / IOCP 的共通抽象

Tokio 的 I/O 驱动通过 mio 库抽象了一组跨平台的事件通知机制。三个主流平台的机制完全不同，但它们提供了相似的语义。

```rust
// 平台差异：
// Linux:   epoll (epoll_create, epoll_ctl, epoll_wait)
// macOS:   kqueue (kqueue, kevent)
// Windows: IOCP (CreateIoCompletionPort, GetQueuedCompletionStatus)
//
// mio 提供的统一抽象：
// - Registry: 注册文件描述符/句柄的兴趣事件
// - Poll: 等待已注册的事件发生

// mio 的核心使用模式（简化版）：
use mio::{Events, Poll, Interest, Token};
use mio::net::TcpListener;

// 1. 创建 Poll 实例——底层创建 epoll fd / kqueue / IOCP port
let mut poll = Poll::new().unwrap();
let mut events = Events::with_capacity(1024);

// 2. 注册一个 socket 和对它的兴趣（可读）
let mut listener = TcpListener::bind("127.0.0.1:8080".parse().unwrap()).unwrap();
poll.registry().register(
    &mut listener,
    Token(0),         // 用户自定义的标识——用于事件回查
    Interest::READABLE,
).unwrap();

// 3. 事件循环
loop {
    // 阻塞等待事件——超时参数控制等待时间
    poll.poll(&mut events, None).unwrap();
    
    for event in events.iter() {
        match event.token() {
            Token(0) => {
                // listener 上有新连接
                // accept 并注册新的 socket
            }
            // ... 其他 token 的处理
        }
    }
}
```

Tokio 在这个 mio 基础之上构建了更高层的抽象——将 `Token` 映射到 `Waker`，使得当 socket 就绪时自动唤醒对应的任务。

### 3.2 Tokio 的 I/O 驱动如何与 Future 集成

当你在 tokio 中执行 `socket.read(&mut buf).await` 时，在底层发生了一系列操作：

```rust
// tokio::net::TcpStream::read() 内部流程（概念上的简化版）

// 步骤 1：用户调用 read().await
// 步骤 2：TcpStream 的 read() 返回一个 ReadFuture
// 步骤 3：ReadFuture::poll() 被调用
//         ↓
// 步骤 4：尝试非阻塞读取（read(fd, buf, len) 在非阻塞模式下）
//         如果读到了数据 → 返回 Poll::Ready(n)
//         如果返回 EWOULDBLOCK（没数据） → 继续步骤 5
// 步骤 5：向 I/O 驱动注册此 socket 的可读事件
//         提供一个 Waker（来自 cx）
// 步骤 6：返回 Poll::Pending
//         任务退出调度循环，worker 继续执行其他任务
//
// ... 时间流逝，数据到达 ...
//
// 步骤 7：I/O 驱动的 poll 循环检测到此 socket 可读
// 步骤 8：I/O 驱动调用关联的 Waker::wake()
// 步骤 9：任务重新进入调度队列
// 步骤 10：ReadFuture::poll() 再次被调用
//         这次非阻塞 read 读到了数据 → 返回 Poll::Ready(n)
```

注意步骤 4 中的"非阻塞尝试"——这是 `try_read` 优化：在向 epoll 注册之前先尝试读取，因为数据可能已经在内核缓冲区中了。这避免了不必要的 epoll 注册和上下文切换。

## 4. 定时器——Hashed Timer Wheel

### 4.1 为什么不用简单的排序列表

`tokio::time::sleep` 和 `tokio::time::interval` 使用了一种称为"哈希定时器轮"（hashed timer wheel）的数据结构。为什么不简单地按过期时间排序、用最小堆呢？

```rust
// 朴素方案：最小堆（BinaryHeap）
// - 插入定时器：O(log N)
// - 取出最近过期的定时器：O(log N)
// - 对于拥有百万级定时器的场景（比如每个连接一个 timeout），log N 仍然很大

// 哈希定时器轮：O(1) 插入、O(1) 取出过期项
// - 类似钟表的齿轮组，将时间范围分层
// - 适用于"插入频率高、过期时间分布在一定范围内"的场景
// - 是网络服务器中定时器管理的事实标准（Linux 内核也使用此结构）
```

### 4.2 哈希定时器轮的工作原理

```
层级结构（3 层为例）：
  
  第 0 层（毫秒轮）：256 个槽，每个槽 = 1ms，覆盖 0~255ms
  第 1 层（秒轮）：  64 个槽， 每个槽 = 256ms，覆盖 256ms~16s  
  第 2 层（分钟轮）：64 个槽， 每个槽 = 16s，覆盖 16s~17min

插入一个 500ms 的定时器：
  - 500ms > 256ms → 放到第 1 层
  - 500 / 256 = 1（槽 1），偏差 = 500 - 256 = 244ms
  - 当秒轮指针移动到槽 1 时，这个槽中的所有定时器被"降级"到毫秒轮
  - 降级后：在毫秒轮的槽 244 中

当前时间在定时器轮上的表示像一个多级指针：
  [第2层的某槽] → [第1层的某槽] → [第0层的某槽]
  指针每走完一圈，上一层的指针移动一步
```

下面是定时器轮在 tokio 中概念上的工作流程：

```rust
// tokio 定时器的工作流程（概念简化版）

// 1. 你调用 sleep(duration).await
//    内部创建一个 Sleep future，插入定时器轮

// 2. Sleep::poll() 被调用
//    检查当前时间是否 >= 过期时间
//    如果没到 → 向定时器轮注册 waker → 返回 Pending

// 3. 定时器轮在每次 I/O 驱动的事件循环中检查：
//    - 计算"下一个最近过期的定时器还有多久"
//    - 将此时间作为 epoll_wait/poll 的超时参数
//    - 这样 worker 线程不会永久休眠——最多等到最近的定时器到期

// 4. 当 epoll_wait 因超时而返回（定时器到期了）
//    定时器轮触发所有过期槽中的 waker
//    这些 waker 将 Sleep future 放回调度队列

// 5. Sleep::poll() 再次被调用
//    这次时间到了 → 返回 Poll::Ready
```

定时器轮与 I/O 驱动的配合是 Tokio 高效的关键：在没有 I/O 事件时，worker 最多等到下一个定时器到期，而不是永远休眠或不断轮询。

## 5. `#[tokio::main]` 宏——展开后的真实代码

### 5.1 宏展开

`#[tokio::main]` 是 Tokio 中最常用但也最"魔法"的部分。理解它的展开有助于理解运行时的启动过程。

```rust
// 你写的代码：
#[tokio::main]
async fn main() {
    println!("Hello from tokio!");
}

// 宏展开后大致等价于：
fn main() {
    // 步骤 1：构建运行时
    let rt = tokio::runtime::Runtime::new().unwrap();
    
    // 步骤 2：在运行时上阻塞执行 async main
    rt.block_on(async {
        println!("Hello from tokio!");
    });
}

// 带配置的版本：
#[tokio::main(worker_threads = 4, flavor = "multi_thread")]
async fn main() {
    // ...
}

// 展开为：
fn main() {
    let rt = tokio::runtime::Builder::new_multi_thread()
        .worker_threads(4)
        .build()
        .unwrap();
    rt.block_on(async {
        // ...
    });
}
```

### 5.2 Runtime::block_on 的内部

`block_on` 是同步世界与异步世界的桥接点。它的核心实现相对简洁：

```rust
// block_on 的简化实现（展示核心逻辑）
impl Runtime {
    pub fn block_on<F: Future>(&self, future: F) -> F::Output {
        // 步骤 1：把传入的 future 设为"根任务"
        // 将它固定在栈上，并创建一个"单次 poll"的特殊 Context
        
        // 步骤 2：在当前线程上进入事件循环
        // 在 multi_thread 运行时中，block_on 所在的线程
        // 会临时成为一个 worker 线程
        let mut enter = self.enter();  // 设置线程局部运行时上下文
        
        // 步骤 3：pin 住 future 并反复 poll
        // 同时处理 I/O 事件和定时器
        let result = loop {
            // ... poll 逻辑 ...
            // 当 root future 返回 Ready 时退出
        };
        
        // 步骤 4：运行结束后清理
        result
    }
}
```

关键点：`block_on` 不只是等待一个 Future——它在等待过程中还**驱动了整个运行时的 I/O 和定时器**。也就是说，你在 `block_on` 中 spawn 的其他任务也会被执行。

## 6. Worker 线程与 Blocking 线程——双线程池架构

### 6.1 两个线程池的区别

Tokio 维护着两个独立的线程池：

```
┌──────────────────────────────────────┐
│        WORKER THREAD POOL            │
│  (tokio::spawn 的任务在这里执行)       │
│  线程数：默认 CPU 核心数              │
│  职责：执行 async 任务（poll Future）│
│  特点：工作窃取、事件驱动、协作调度    │
│  不应该执行阻塞操作！                 │
├──────────────────────────────────────┤
│      BLOCKING THREAD POOL            │
│  (spawn_blocking 的任务在这里执行)    │
│  线程数：默认 512（上限）             │
│  职责：执行同步阻塞操作               │
│  特点：传统的线程模型、可以阻塞        │
│  适合：CPU 密集型计算、同步文件 I/O    │
└──────────────────────────────────────┘
```

### 6.2 spawn vs spawn_blocking——何时用哪个

```rust
// ✅ tokio::spawn —— 用于 async 任务
tokio::spawn(async {
    // 这里执行异步代码
    // 可以 .await，会暂停和恢复
    // 不应该调用 std::thread::sleep 或其他阻塞操作
    let data = read_from_tcp().await;
    process_data(data).await;
});

// ✅ tokio::task::spawn_blocking —— 用于同步/阻塞操作
tokio::task::spawn_blocking(move || {
    // 这里执行同步代码
    // 可以安全地使用 std::thread::sleep
    // 可以执行 CPU 密集计算
    // 可以调用同步文件 I/O（std::fs::read_to_string 等）
    let result = heavy_computation();
    result
}).await.unwrap();

// ❌ 错误：在 spawn 中执行阻塞操作
// tokio::spawn(async {
//     std::thread::sleep(Duration::from_secs(10));  // 阻塞了 worker 线程！
//     // 在此期间该 worker 无法执行其他任务
//     // 如果所有 worker 都被阻塞 → 死锁
// });
```

关键原则：`spawn` 的任务跑在 worker 线程上，如果 worker 线程被阻塞（比如调用 `std::thread::sleep` 或同步文件 I/O），那么该 worker 上的所有其他任务都无法被 poll——这是 Tokio 中常见的性能问题根源。

### 6.3 为什么阻塞线程池默认有 512 个上限

```rust
// 阻塞线程池的设计考虑：
// 1. 阻塞操作通常比 async 操作耗时更长
// 2. 需要足够的线程来处理并发的阻塞请求
// 3. 但不能无限增长（线程有内存开销——每个线程约 2MB-8MB 栈空间）
// 4. 512 是经验上的合理上限

// 运行时配置示例：
let rt = tokio::runtime::Builder::new_multi_thread()
    .worker_threads(8)          // worker 线程数 = CPU 核心数
    .max_blocking_threads(256)  // 阻塞线程池上限
    .thread_name("my-worker")   // 线程名称前缀（便于调试）
    .thread_stack_size(3 * 1024 * 1024) // 每个线程的栈大小（默认 2MB）
    .build()
    .unwrap();
```

## 7. 协作调度——你写的代码如何让出 CPU

### 7.1 协作调度的含义

Tokio 使用**协作调度**（cooperative scheduling）而非**抢占式调度**（preemptive scheduling）。这意味着任务通过 `.await` 主动让出控制权，而不是被运行时强制暂停。

```rust
// 协作调度的含义：
// - 每个 .await 是任务的"yield point"
// - 在 .await 之间，任务独占了 worker 线程
// - 如果两个 .await 之间有长计算，该 worker 上的其他任务会被"饿死"

// ❌ 危险：长计算阻塞了 worker
// tokio::spawn(async {
//     let mut sum = 0u64;
//     for i in 0..1_000_000_000u64 {
//         sum = sum.wrapping_add(i);
//     }
//     println!("sum = {}", sum);
//     // 这个任务在计算过程中不 .await
//     // 它所在的 worker 上的其他任务在这段时间都无法执行
// });

// ✅ 修复方案 1：用 spawn_blocking 把计算移到阻塞线程池
tokio::task::spawn_blocking(move || {
    let mut sum = 0u64;
    for i in 0..1_000_000_000u64 {
        sum = sum.wrapping_add(i);
    }
    sum
});

// ✅ 修复方案 2：在计算中插入 yield point
// tokio::spawn(async {
//     let mut sum = 0u64;
//     for chunk in (0..1_000_000_000u64).collect::<Vec<_>>().chunks(100_000) {
//         for &i in chunk {
//             sum = sum.wrapping_add(i);
//         }
//         tokio::task::yield_now().await;  // 每 10 万次迭代让出一次
//     }
// });
```

### 7.2 预算（Budget）机制

Tokio 内部为每个任务维护一个"预算"计数器，防止单个任务无限期占用 worker：

```rust
// 预算机制的概念模型（简化版）
// 每个任务在 poll 之前被赋予 N 个"预算单元"
// 每次 .await（yield point）消耗一个预算单元
// 预算耗尽时，任务必须返回 Pending 重新调度

// tokio::task::yield_now() —— 显式让出
// 等价于：消耗一个预算单元，如果预算耗尽则重新调度

// 在长循环中使用 yield_now：
async fn long_running_task() {
    for i in 0..1000 {
        do_some_work(i);
        if i % 100 == 0 {
            tokio::task::yield_now().await;  // 定期让出
        }
    }
}
```

## 8. 用 mio 自己实现一个"迷你 Tokio"——理解架构的本质

### 8.1 极简版架构

为了真正理解 Tokio 是如何工作的，我们来实现一个单线程、只能处理 TCP 连接的极简运行时。这个实现大约 150 行，但它包含了 Tokio 的核心架构要素。

```rust
use mio::{Events, Interest, Poll, Token};
use mio::net::TcpListener;
use std::collections::VecDeque;
use std::future::Future;
use std::pin::Pin;
use std::sync::{Arc, Mutex};
use std::task::{Context, Poll, RawWaker, RawWakerVTable, Waker};

// ============================================================
// 第一部分：任务管理
// ============================================================

/// 一个可执行的任务——把 Future 包装在 Box 中
type Task = Pin<Box<dyn Future<Output = ()>>>;

/// 迷你运行时——单线程、mio 驱动
struct MiniTokio {
    /// 就绪任务队列——等待被 poll 的任务
    ready_queue: VecDeque<Task>,
    /// mio 的事件轮询器——底层是 epoll/kqueue/IOCP
    poll: Poll,
    /// 当前正在等待的事件集合（可复用）
    events: Events,
}

    /// 在这个极简实现中，我们使用一个共享的 Arc<Mutex<VecDeque<Task>>> 
    /// 作为就绪队列，waker 的 wake 操作就是将任务重新入队。
    /// 真实的 Tokio 使用更高效的原子操作和侵入式链表。
    fn make_waker() -> Waker {

    /// 事件循环——整个运行时的核心
    fn run(&mut self) -> std::io::Result<()> {
        loop {
            // 步骤 1：处理所有就绪任务
            while let Some(mut task) = self.ready_queue.pop_front() {
                let waker = Self::make_waker();
                let mut cx = Context::from_waker(&waker);
                
                match task.as_mut().poll(&mut cx) {
                    Poll::Ready(()) => {
                        // 任务完成——丢弃

                    Poll::Pending => {
                        // 任务未完成——它应已在 I/O 驱动上注册了 waker
                        // 当 I/O 就绪时，对应的 waker 会将它重新入队
                    }
                }
            }

            // 步骤 2：没有就绪任务时，等待 I/O 事件
            // 超时设置为 0 意味着如果也没有 I/O 事件就立即返回
            // 真实 Tokio 这里会使用"下一个定时器的过期时间"作为超时
            self.poll.poll(&mut self.events, None)?;
            
            // 步骤 3：处理 I/O 事件

        }
    }
```

### 8.2 完整的迷你 Tokio——可直接运行

```rust
// 一个更加完整的迷你 Tokio 实现
// 支持：spawn(async task) + TCP accept + TCP read/write

use mio::{Events, Interest, Poll, Token};
use mio::net::{TcpListener, TcpStream};
use std::collections::{HashMap, VecDeque};
use std::future::Future;
use std::io::{self, Read, Write};
use std::pin::Pin;
use std::sync::{Arc, Mutex};
use std::task::{Context, Poll, RawWaker, RawWakerVTable, Waker};

// --- Waker 的辅助结构 ---
// 每个任务的 Waker 唤醒时，会将 token 放入 shared 的 VecDeque
// 注意：真实代码需要一个全局静态或 Arc 共享的任务调度器引用

// --- 概念性的 TcpStream 异步包装 ---
struct AsyncTcpStream {
    stream: TcpStream,
    token: Token,
    // 读取缓冲区
    read_buf: Vec<u8>,
    // waker——在数据到达时用来唤醒任务
    waker: Option<Waker>,
}

impl AsyncTcpStream {
    fn read(&mut self, cx: &mut Context<'_>) -> Poll<io::Result<Vec<u8>>> {
        let mut buf = [0u8; 1024];
        match self.stream.read(&mut buf) {
            Ok(n) if n > 0 => {
                Poll::Ready(Ok(buf[..n].to_vec()))
            }
            Ok(_) => {
                // 读到 0 字节——对端关闭连接
                Poll::Ready(Ok(vec![]))
            }
            Err(ref e) if e.kind() == io::ErrorKind::WouldBlock => {
                // 没有数据可读——注册 waker，返回 Pending
                self.waker = Some(cx.waker().clone());
                Poll::Pending
            }
            Err(e) => Poll::Ready(Err(e)),
        }
    }
}

// --- 主循环骨架 ---
// 完整的实现需要：
// 1. 用 Arc<Mutex<...>> 或全局变量共享就绪队列
// 2. 构建 RawWaker，其 vtable 中的 wake 函数将任务入队
// 3. 处理 I/O 事件时找到对应的任务并唤醒

// 由于完整实现较长，这里仅展示核心架构。
// 理解了这个骨架，再去读 tokio 源码就会容易得多。
```

关键收获：即使是最简化的实现，你也能清楚地看到 Tokio 不是魔法——它就是任务队列 + mio 事件循环 + Waker 回调的组合。复杂的部分在于让它高效（工作窃取、无锁数据结构、精细的唤醒优化）、稳定（正确处理所有边界条件）和跨平台。

### 9.1 常用配置选项

```rust
use tokio::runtime::Builder;

let rt = Builder::new_multi_thread()
    // worker 线程数——默认 = CPU 核心数
    // 对于 I/O 密集型应用，可以调大
    // 对于 CPU 密集型应用，保持默认
    .worker_threads(8)
    
    // 阻塞线程池的最大线程数——默认 512
    // 如果你的 spawn_blocking 并发量很高，可能需要调节
    .max_blocking_threads(256)
    
    // 线程名称前缀——便于在调试器/top/htop 中识别
    .thread_name("my-app-worker")
    
    // 每个线程的栈大小（字节）——默认 2MB
    // 如果任务很深（深度递归），可能需要增加
    // 如果内存紧张，可以减少
    .thread_stack_size(3 * 1024 * 1024)
    
    // 全局队列的间隔——影响新 spawn 任务是先入全局队列还是本地队列
    // 默认 61：每 61 次 spawn 有 1 次放入全局队列
    // 这控制了"工作窃取"的频率——全局队列人人都能看到
    .global_queue_interval(31)
    
    // 事件间隔——worker 在没有本地任务时检查全局队列和 I/O 事件的频率
    // 默认 61
    .event_interval(61)
    
    // 是否启用 I/O 驱动（默认 true）
    // 如果设为 false，tokio 只做任务调度，不做 I/O
    // .enable_io() 和 .enable_time() 默认都启用
    
    .build()
    .unwrap();
```

## 11. 练习答案

### 练习 1 答案

```
完整的 TCP 连接处理生命周期追踪：

1. TcpListener::bind("127.0.0.1:8080").await
   - 创建 std::net::TcpListener（操作系统 socket）
   - 设置为非阻塞模式
   - 包装为 mio::net::TcpListener
   - 向 mio::Poll 注册此 socket，兴趣事件为 READABLE，Token=LISTENER_TOKEN
   - 返回就绪（bind 不需要等待 I/O）

2. listener.accept().await
   - 第一次 poll()：调用 listener.accept()（非阻塞）
     → 如果没有新连接，返回 WouldBlock
     → 向 mio 重新注册 READABLE 兴趣 + 保存 waker → 返回 Pending
   - 当连接到达时：
     → epoll_wait 返回 LISTENER_TOKEN 可读
     → I/O 驱动调用关联的 Waker::wake()
     → 任务重新入队
     → 第二次 poll()：accept 成功 → 返回 Ready((socket, addr))

3. tokio::spawn(async move { ... })
   - 将 async block 的 Future 包装为 Task
   - 放入 worker 的本地队列（或全局队列）
   - 返回 JoinHandle

4. reader.read(&mut buf).await
   - poll()：尝试非阻塞 read
   - 如果数据未到达 → 注册可读 + waker → Pending
   - 数据到达时 epoll 通知 → waker.wake() → re-poll → Ready(n)

5. writer.write_all(&buf).await
   - 类似读取，但注册的是可写事件
   - 如果写缓冲区已满 → 注册 WRITABLE + waker → Pending
   - 缓冲区可用时 → wake → re-poll → Ready
```

### 练习 3 答案

```rust
use tokio::sync::{mpsc, oneshot};
use tokio::time::{self, Duration};

type Request = i32;
type Response = i32;

struct BatchProcessor {
    request_tx: mpsc::Sender<(Request, oneshot::Sender<Response>)>,
}

impl BatchProcessor {
    fn new() -> Self {
        let (tx, mut rx) = mpsc::channel::<(Request, oneshot::Sender<Response>)>(1024);

        tokio::spawn(async move {
            loop {
                let mut batch: Vec<(Request, oneshot::Sender<Response>)> = Vec::new();

                // 收集第一批数据——至少等 1 个请求到达
                match rx.recv().await {
                    Some(first) => batch.push(first),
                    None => break,  // channel 关闭
                }

                // 继续收集：每 100ms 或满 10 个就处理
                let deadline = time::sleep(Duration::from_millis(100));
                tokio::pin!(deadline);

                loop {
                    if batch.len() >= 10 {
                        break;  // 够了，处理
                    }

                    tokio::select! {
                        maybe_item = rx.recv() => {
                            match maybe_item {
                                Some(item) => batch.push(item),
                                None => break,
                            }
                        }
                        _ = &mut deadline => {
                            break;  // 超时了，处理
                        }
                    }
                }

                // 批量处理
                if !batch.is_empty() {
                    println!("Processing batch of {} requests", batch.len());
                    for (req, tx) in batch {
                        let result = req * 2;  // 模拟处理
                        let _ = tx.send(result);  // 忽略 receiver 已 drop 的错误
                    }
                }
            }
        });

        BatchProcessor { request_tx: tx }
    }

    async fn process(&self, req: Request) -> Response {
        let (tx, rx) = oneshot::channel();
        self.request_tx.send((req, tx)).await.unwrap();
        rx.await.unwrap()
    }
}

#[tokio::main]
async fn main() {
    let processor = BatchProcessor::new();

    // 发送 25 个请求——将被分组为 3 批（10 + 10 + 5）
    let mut handles = vec![];
    for i in 0..25 {
        let result = processor.process(i).await;
        handles.push(result);
    }
    // 注意：这里顺序发送 + 同步等待，实际 batch 效果有限
    // 真实场景中你可能并发发送，让 batch 更满
}
```
