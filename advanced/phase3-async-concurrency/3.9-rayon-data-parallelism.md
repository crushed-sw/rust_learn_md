# 3.9 Rayon 数据并行 —— 把数组拆成 N 份，N 个核心一起算

> 假设你有一个包含 1000 万条记录的 `Vec<Transaction>`，需要对每条记录做同样的处理——解析字段、计算佣金、更新余额。你用 `iter().map().collect()` 写了很漂亮的函数式代码，一切正确。但你注意到它只用了 1 个 CPU 核心——你的 16 核机器上有一个核在满负荷运转，另外 15 个在围观。这就是"数据并行"要解决的问题：**当逻辑本身是"对每个元素独立做一件事"时，你可以把这 1000 万条记录分成 16 份，16 个核心各自处理自己的那份，最后合并结果。** Rayon 做到了让这个过程几乎不改变你的代码风格——把 `iter()` 换成 `par_iter()`，然后一切都并行发生了。但 Rayon 不是魔法，它内置了一个精巧的**工作窃取调度器**来保证负载均衡，它有一组**顺序语义保证**来防止数据竞争，它还有明确的**性能边界**——不合适的数据规模上，并行的开销可能超过收益。这一课的目标是：让你能用 Rayon 的直觉写出正确的并行代码，同时理解"为什么它正确"和"什么时候它反而不快"。

## 1. 从 `iter()` 到 `par_iter()` —— 最小的改动，最大的变化

### 1.1 并行迭代器是什么

Rayon 的核心思想是：**并行迭代器**（`ParallelIterator`）。它的接口和标准库的 `Iterator` 故意设计得非常相似——这是为了让程序员能以最小的成本从串行切换到并行。

```rust
use rayon::prelude::*;

// 串行版本——标准库 Iterator
fn sum_sequential(v: &[i32]) -> i32 {
    v.iter()
        .map(|&x| x * 2)
        .sum()
}

// 并行版本——Rayon ParallelIterator
fn sum_parallel(v: &[i32]) -> i32 {
    v.par_iter()           // 并行迭代
        .map(|&x| x * 2)   // 每个元素 * 2（分布在不同核心上）
        .sum()              // 并行归约——自动合并部分和
}

// 区别只有 iter() → par_iter()
// 其他的 map / filter / sum 保持不变
// 但执行模型完全不同——从单核变成了所有核心
```

这段代码展示的是 Rayon 最核心的设计哲学：**API 兼容**。你不需要学习一套全新的概念——如果你会用 `Iterator`，就基本会用 `ParallelIterator`。这个设计决策使得 Rayon 的入门成本极低。

### 1.2 Rayon 不是"什么都并行"

一个重要的澄清：`par_iter()` 保证的是**并行执行**，但不保证**执行顺序**。这意味着以下事情：

```rust
use rayon::prelude::*;

// ❌ 不要依赖执行顺序
let results: Vec<i32> = (0..100).into_par_iter()
    .map(|i| {
        println!("处理: {}", i);  // 打印顺序不是 0, 1, 2, ..., 99
        i * 2
    })
    .collect();
// results 的顺序仍然是 0, 2, 4, ..., 198（collect 保证顺序）
// 但 println 的输出顺序是乱的——因为不同核心上的任务交错执行

// ✅ Rayon 的顺序保证：
// - collect() 出来的 Vec 和输入的顺序一致
// - 但每个元素的 map 闭包的执行顺序不确定
// - 你不应该在闭包中依赖"前面元素已经处理完了"的假设
```

## 2. 两个核心 trait —— ParallelIterator 和 IntoParallelIterator

### 2.1 IntoParallelIterator —— 进入并行世界的入口

`IntoParallelIterator` 是连接"普通数据"和"并行迭代"的桥梁。它定义了如何把各种 Rust 数据结构**转换**成并行迭代器。

```rust
use rayon::prelude::*;

// IntoParallelIterator 为常见类型实现了：
// - 切片：      &[T]     → par_iter() 返回并行迭代器
// - 可变切片：   &mut [T] → par_iter_mut() 返回可变并行迭代器
// - Vec<T>：    Vec<T>   → into_par_iter() 消费 Vec
// - 范围：      0..100   → into_par_iter()
// - 数组：      [T; N]   → into_par_iter()
// - HashMap / HashSet / BTreeMap / BTreeSet
// - 大多数实现了 IntoIterator 的集合

let v = vec![1, 2, 3, 4, 5];

// 三种入口方式：
let _: Vec<i32> = v.par_iter()        // &T   —— 不可变引用，不消费原始数据
    .map(|x| x * 2)
    .collect();

let _: Vec<i32> = v.clone().into_par_iter()  // T   —— 消费数据，拿走所有权
    .map(|x| x * 2)
    .collect();

// par_iter_mut 需要 mut 变量
let mut v2 = vec![1, 2, 3, 4, 5];
v2.par_iter_mut()                 // &mut T —— 可变引用，原地修改
    .for_each(|x| *x *= 2);
assert_eq!(v2, vec![2, 4, 6, 8, 10]);
```

选择哪种入口取决于你的所有权需求：
- **只需要读** → `par_iter()`，开销最小（不需要复制数据）
- **需要改每个元素** → `par_iter_mut()`，原地修改
- **需要将数据转移到线程中** → `into_par_iter()`，避免了引用计数的开销

### 2.2 ParallelIterator 的方法全景

`ParallelIterator` 大部分方法对标 `Iterator`，但存在一些重要的差异——特别是涉及到"归约"（reduction）的操作。

```rust
use rayon::prelude::*;

let v: Vec<i32> = (0..10).collect();

// —— 对标 Iterator 的操作（并行化）——
// map / filter / filter_map / flat_map / flatten
// cloned / copied
// for_each / count / find_any / find_first / find_last
// any / all
// sum / product
// min / max / min_by / max_by
// reduce / fold
// collect

// 示例：find_any —— 找到任意一个满足条件的元素（并行查找）
let result = v.par_iter()
    .find_any(|&&x| x > 5);
// 结果可能是 6、7、8、9 中的任何一个——取决于哪个线程先找到
println!("第一个找到的 > 5 的数: {:?}", result);

// 示例：find_first —— 保证找到"按顺序第一个"（较慢）
let result = v.par_iter()
    .find_first(|&&x| x > 5);
// 结果一定是 6——和串行迭代的结果一致
assert_eq!(result, Some(&6));
// 但 find_first 有额外开销：需要协调所有线程确保顺序
```

`find_any` 和 `find_first` 的对比揭示了一个重要的设计权衡：**越强的顺序保证，越大的并行开销**。如果你不关心顺序（大多数数据处理场景），用 `find_any` 等宽松方法能获得更好的并行性能。

### 2.3 并行版特有的操作——par_sort, par_chunks 等

有些操作只在并行上下文中有意义，Rayon 提供了专门的方法。

```rust
use rayon::prelude::*;
use rayon::slice::ParallelSliceMut;

let mut v = vec![3, 1, 4, 1, 5, 9, 2, 6, 5, 3, 5];

// par_sort —— 并行排序（使用多线程归并排序）
v.par_sort();
assert_eq!(v, vec![1, 1, 2, 3, 3, 4, 5, 5, 5, 6, 9]);

// par_sort_unstable —— 并行不稳定排序（更快，但不保证相等元素的相对顺序）
let mut v2 = vec![4, 2, 3, 1];
v2.par_sort_unstable();
assert_eq!(v2, vec![1, 2, 3, 4]);
```

`par_sort` 的实现使用了并行的归并排序（merge sort）——把数组分段、每段分别排序、再并行合并。对大数据集效果显著。

```rust
use rayon::prelude::*;

// par_chunks —— 将切片分成固定大小的块，并行处理每个块
let v: Vec<i32> = (0..100).collect();

// 每 10 个元素一个 chunk，每个 chunk 在一个线程中处理
let results: Vec<i32> = v.par_chunks(10)
    .map(|chunk| {
        // chunk: &[i32] —— 这段切片属于当前线程
        chunk.iter().sum::<i32>()  // 每个 chunk 的局部和
    })
    .collect();
// results 包含 10 个局部和

// par_chunks 和 par_chunks_mut 是"任务级并行"的便捷工具
// 它们不需要 Rayon 帮你分解任务——你已经手动分好粒度了
```

## 3. fold / reduce —— 正确理解并行归约

### 3.1 串行 fold 和并行 fold 的本质差异

在标准库中，`Iterator::fold()` 是**串行累积**：从左到右，一个接一个。在 Rayon 中，`ParallelIterator::fold()` 的语义有重要差异。

```rust
use rayon::prelude::*;

// 标准库 fold（串行）：
// fold(init, |acc, elem| acc + elem)
// 执行顺序：((((init + elem0) + elem1) + elem2) + ...)
// 这要求操作是结合的，但实际上 Rust 标准库不要求结合律——
// 因为它总是从左到右执行

// Rayon fold（并行）：
// fold(|| identity_value, |acc, elem| acc + elem)
//       ↑ 注意：这里是闭包，不是直接传值！
// 
// 执行过程：
//   1. 把数据分成 N 块
//   2. 每个线程对自己的那一块执行串行 fold：
//      线程 0: identity = || 0; fold: 0 + elem0 + elem1 + ...
//      线程 1: identity = || 0; fold: 0 + elem5 + elem6 + ...
//   3. 合并所有线程的结果（使用 reduce 操作）：
//      result = thread0_result + thread1_result + ...

let sum: i32 = (0..1000).into_par_iter()
    .fold(
        || 0,           // identity 工厂——每个线程调用一次
        |acc, x| acc + x, // 每个线程内的累积操作
    )
    .reduce(
        || 0,            // 合并的 identity
        |a, b| a + b,    // 合并各线程的结果
    );

// 为什么 fold 的参数是闭包而不是值？
// 因为每个线程需要独立的累积器实例。
// 如果直接传一个值，多个线程就要共享同一个累积器→需要同步→失去并行性。
```

这个设计揭示了并行归约的核心权衡：**fold 要求操作是结合的（associative）**，因为合并的顺序不确定。对于加法、乘法这类数学操作没有问题，但对于字符串拼接或列表追加就不一定了——虽然 `String::push_str` 是结合的，但你可能关心最终字符串的顺序，而并行 fold 不保证顺序。如果顺序重要，用 `collect()`。

### 3.2 reduce vs fold —— 什么时候用哪个

```rust
use rayon::prelude::*;

// reduce —— 不需要初始值，直接用元素
// 适用于：元素本身就能累积的情况
let v = vec![1, 2, 3, 4, 5];
let sum: i32 = v.par_iter()
    .cloned()
    .reduce(
        || 0,          // 没有元素时的返回值
        |a, b| a + b,  // 合并两个部分结果
    );

// fold —— 需要一个"与元素类型不同"的累积器
// 适用于：累积器类型和元素类型不同的情况
let word_count = vec!["hello", "world", "rust", "rayon"];
let total_chars: usize = word_count.par_iter()
    .fold(
        || 0usize,
        |acc, word| acc + word.len(),  // 累积器是 usize，元素是 &str
    )
    .reduce(
        || 0,
        |a, b| a + b,
    );

// 总结：reduce 适合"累积器和元素类型相同"
//       fold 适合"累积器和元素类型不同"
```

## 4. join 和 scope —— 细粒度的并行控制

### 4.1 join —— 同时做两件独立的事

`rayon::join(a, b)` 接收两个闭包，**可能并行执行**它们。当两个任务完全独立时，这是最简单直接的方式。

```rust
use rayon::prelude::*;

// join 的典型用法：两个独立的、耗时差不多的子任务
fn process_files() {
    let (data1, data2) = rayon::join(
        || load_and_parse("file1.csv"),   // 第一个任务
        || load_and_parse("file2.csv"),   // 第二个任务——同时执行
    );
    // 两个任务可能在不同的线程上并行运行
    // 当两个都完成后，继续执行
    println!("data1: {:?}, data2: {:?}", data1, data2);
}

fn load_and_parse(path: &str) -> Vec<String> {
    // 模拟耗时操作
    std::thread::sleep(std::time::Duration::from_millis(100));
    vec![format!("{}:data", path)]
}

// join 的语义保证：
// - 两个闭包都可能被立即执行（不一定在同一个线程）
// - 如果 Rayon 全局线程池有空闲线程，两个任务真正并行
// - 如果没有空闲线程，它们可能串行执行（在调用线程上）
// - 不管怎样，join 返回时两个闭包都已经执行完毕
```

`join` 不需要分割数据——它只是把两段独立代码标记为"可以同时做"。这比 `par_iter` 更通用，因为两个闭包可以是完全不同的计算。

### 4.2 递归并行——join 的经典应用

当问题本身是递归的（如分治算法），`join` 可以自然地表达并行分解。

```rust
use rayon::prelude::*;

// 并行快速排序（概念性实现——实际应该用 par_sort）
fn parallel_quicksort<T: Ord + Send>(data: &mut [T]) {
    if data.len() <= 1 {
        return;
    }

    // 分区：选 pivot，把小于 pivot 的放左边，大于的放右边
    let pivot_index = partition(data);

    // 递归地对两部分排序——同时进行
    let (left, right) = data.split_at_mut(pivot_index);
    rayon::join(
        || parallel_quicksort(left),
        || parallel_quicksort(&mut right[1..]),  // 跳过 pivot
    );
}

fn partition<T: Ord>(data: &mut [T]) -> usize {
    // 简化版——真实实现用 Lomuto 或 Hoare 分区
    let len = data.len();
    let pivot = len - 1;
    let mut i = 0;
    for j in 0..pivot {
        if data[j] <= data[pivot] {
            data.swap(i, j);
            i += 1;
        }
    }
    data.swap(i, pivot);
    i
}

// 这个递归并行的模式展示了 Rayon 的强大表达力：
// - 你写的是"递归分解"的自然逻辑
// - Rayon 在后台自动把任务分配给线程
// - 不需要手动管理线程池、任务队列、同步
```

### 4.3 scope —— 创建一组并行任务并等待全部完成

`rayon::scope(|s| { ... })` 允许你在一段代码中动态地派生（spawn）多个任务，`scope` 会等所有任务完成后才返回。

```rust
use rayon::prelude::*;

fn process_multiple_files(paths: &[&str]) -> Vec<(String, Vec<String>)> {
    // 使用 scope 处理多个文件
    // 结果需要在线程间共享——使用 Mutex
    let results = std::sync::Mutex::new(Vec::new());

    rayon::scope(|s| {
        for path in paths {
            let results = &results;  // 借用
            s.spawn(move |_s| {
                let data = load_and_parse(path);
                results.lock().unwrap().push((path.to_string(), data));
            });
        }
    });
    // 所有 spawnd 任务在这里才全部完成

    results.into_inner().unwrap()
}

// scope 和 std::thread::scope 的对比：
// - std::thread::scope  → 每个任务分配一个 OS 线程
// - rayon::scope        → 任务在 Rayon 的线程池上执行
//   - 如果有 1000 个任务，不会创建 1000 个线程
//   - 任务在固定数量的工作线程上轮流执行
// scope 内部可以再调用 scope，形成嵌套的并行结构
```

`scope` 的关键特性是它保证了"所有子任务完成前不会返回"——这让你可以安全地借用栈上的数据。Rayon 的 scope 实现了 `Send` 但非 `'static` 的借用模式。

## 5. 工作窃取调度器 —— Rayon 的心脏

### 5.1 问题：如何保证负载均衡

并行化的核心挑战是**负载均衡**。假设你有 1000 个任务和 8 个线程，某些任务耗时 10ms，某些耗时 1ms。如果你在开始时平均分配，耗时长的线程还在忙碌时，短任务线程已经闲着了。

```rust
// 问题演示：
// 8 个线程，1000 个任务，计算量不均匀
use rayon::prelude::*;

let data: Vec<u64> = (0..1000).collect();

// 假设：偶数序列的计算量是奇数的 10 倍
let results: Vec<u64> = data.par_iter()
    .map(|&n| {
        if n % 2 == 0 {
            // 模拟重计算
            (0..1_000_000).fold(0, |a, b| a + b)
        } else {
            n
        }
    })
    .collect();

// 如果 Rayon 静态分配——给每个线程 125 个元素，
// 拿到奇数多的线程早就完成了，拿到偶数多的还在吭哧干活
// → 一半的 CPU 在浪费

// 工作窃取（Work Stealing）的解决方案：
// 1. 空闲线程去其他线程的"任务队列"偷任务
// 2. 被偷的线程不知道（无锁的设计）
// 3. 结果是：繁忙线程自动获得帮助，空闲线程自动找到活干
```

### 5.2 工作窃取的数据结构——双端队列（deque）

Rayon 的每个工作线程都有自己的**双端任务队列**（work-stealing deque）。

```
每个 Worker 线程：
┌─────────────────────────────────────────────┐
│  任务双端队列（deque）                        │
│  [任务A] ← [任务B] ← [任务C] ← ... ← [任务Z]  │
│   ↑ 自己 pop_front            push_back ↑    │
│                                            │
│  如果队列空了：                               │
│  → 随机选一个其他 Worker                      │
│  → 从它的队尾 steal 一个或多个任务              │
│  → 继续干活                                   │
└─────────────────────────────────────────────┘

关键特性：
- 自己的操作在队首（pop/push_front）——无锁，效率高
- 窃取操作在队尾（steal from back）——可能和主人的 push 冲突
  → 用原子操作处理冲突
- 窃取是"批量"的——一次偷多个任务，减少 steal 频率
```

这种设计的精妙之处在于：**大部分时间，线程只操作自己的队首，因此不需要锁**。只有在被窃取时才涉及原子操作。这保证了极低的开销。

### 5.3 任务粒度与 work splitting

Rayon 的迭代器在内部会自动做**任务分割**（job splitting）。它不是一开始就把所有任务分好——而是使用一种"递归二分"策略。

```
初始任务：处理 [0..1000]
              │
      ┌───────┴────────┐
    [0..500]        [500..1000]     ← 二分
      │                  │
  ┌───┴───┐          ┌───┴───┐
[0..250] [250..500] [500..750] [750..1000]  ← 再二分
  ...       ...        ...        ...
  直到每个子任务足够小（达到序列阈值）

当一个线程的队列被偷空时：
  → 它去偷另一半（大任务）
  → 那一半再二分
  → 这样大任务被逐步分解成小任务，自动实现负载均衡
```

**序列阈值**（sequential threshold）是一个关键参数：当子任务足够小时，Rayon 不再继续分解，而是**在一个线程中串行处理**。这避免了"任务管理开销超过实际计算开销"的问题。默认情况下，Rayon 的目标是让每个子任务的工作量约为 1000-10000 个元素。

## 6. 全局线程池与配置

### 6.1 Rayon 的全局线程池

Rayon 维护一个**全局线程池**，默认线程数等于 `std::thread::available_parallelism()`（即 CPU 逻辑核心数）。

```rust
use rayon::prelude::*;

// 默认行为：使用全局线程池（CPU 核心数个线程）
let result: i32 = (0..1000).into_par_iter().sum();

// 查询当前全局线程池的大小
let pool = rayon::current_thread_pool();
println!("全局线程池线程数: {}", pool.current_num_threads());

// 可以通过环境变量控制：
// RAYON_NUM_THREADS=4 cargo run
// 这不需要修改代码，对于不同部署环境很方便
```

### 6.2 创建独立线程池

有时你想让某些并行任务在独立的线程池上运行——比如隔离计算密集任务和 I/O 任务。

```rust
use rayon::ThreadPoolBuilder;

// 创建独立的线程池
let pool = ThreadPoolBuilder::new()
    .num_threads(4)              // 4 个工作线程
    .thread_name(|i| format!("my-pool-{}", i))
    .build()
    .unwrap();

// 在这个线程池上执行并行任务
let result = pool.install(|| {
    // 这里面的 par_iter 使用 my-pool 而非全局线程池
    (0..1000).into_par_iter().sum::<i32>()
});

println!("独立线程池计算结果: {}", result);
// 全局线程池不受影响

// ThreadPoolBuilder 的其他配置选项：
// .stack_size(2 * 1024 * 1024)  // 栈大小
// .start_handler(|i| println!("线程 {} 启动", i))
// .exit_handler(|i| println!("线程 {} 退出", i))
// .panic_handler(|err| eprintln!("线程内 panic: {:?}", err))
// .build_scoped(...)  // 创建 scope 级别的线程池
```

独立线程池适合：
- 需要隔离不同优先级的并行任务
- 某个计算密集任务会长时间占用线程
- 测试中控制并行度

## 7. 性能考量 —— 并行不是免费的

### 7.1 并行化的成本

从串行切换到并行需要付出三种代价：

```
1. 线程调度开销
   - 操作系统在多个线程之间切换的成本
   - Rayon 通过线程池重用降低了这个开销，但不能完全消除

2. 任务管理开销
   - 将 work 拆分成 task，维护任务队列，work stealing
   - Rayon 通过递归二分和批量窃取最小化了这个开销

3. 内存开销
   - 每个线程需要独立的栈（通常 2MB）
   - 并行操作可能分配额外的缓冲区（如 collect 前的中间结果）
   - 多个线程同时访问可能降低 CPU 缓存效率（缓存乒乓）
```

### 7.2 什么时候并行反而更慢

并行化有收益的前提是：**每个元素的计算量足够大，以至于并行带来的加速超过了上述开销**。这个问题没有固定答案，但有一些经验法则：

```rust
use rayon::prelude::*;
use std::time::Instant;

// 场景 1：元素极多但每元素计算极轻 → 并行可能比串行慢
fn too_light_workload() {
    let v: Vec<i32> = (0..10_000_000).collect();

    let start = Instant::now();
    let _sum: i32 = v.par_iter().sum();  // 每个元素只做一次加法
    // 并行时间可能比串行长——因为任务管理开销 > 实际计算
}

// 场景 2：元素中等但每元素计算重 → 并行的最佳场景
fn ideal_workload() {
    let v: Vec<u64> = (0..1000).collect();

    let start = Instant::now();
    let results: Vec<bool> = v.par_iter()
        .map(|&n| {
            // 每个元素做 100 万次计算
            (2..100_000).any(|d| n % d == 0)
        })
        .collect();
    // 并行效果显著——实际计算主导了时间
}

// 场景 3：数据量小 → 不需要并行
// 处理 100 个元素的数组就没必要上 par_iter
// 经验法则：元素数 < 1000 时，par_iter 的开销通常不值得
```

### 7.3 测量，不要猜测

并行性能是高度环境相关的——取决于 CPU 核心数、缓存大小、内存带宽、数据布局。唯一可靠的方法是**实际测量**。

```rust
use rayon::prelude::*;
use std::time::Instant;

// 一个简单的性能对比框架
fn benchmark<F1, F2, R>(name: &str, f_seq: F1, f_par: F2)
where
    F1: Fn() -> R,
    F2: Fn() -> R,
{
    // 预热——让 CPU 缓存热起来
    for _ in 0..5 {
        f_seq();
        f_par();
    }

    let n = 20;  // 跑 20 轮取平均
    let start = Instant::now();
    for _ in 0..n { f_seq(); }
    let seq_time = start.elapsed() / n;

    let start = Instant::now();
    for _ in 0..n { f_par(); }
    let par_time = start.elapsed() / n;

    println!(
        "{}: 串行 {:?}, 并行 {:?}, 加速比 {:.2}x",
        name,
        seq_time,
        par_time,
        seq_time.as_secs_f64() / par_time.as_secs_f64(),
    );
}

// 使用：
// benchmark("简单求和",
//     || (0..1_000_000).sum::<i64>(),
//     || (0..1_000_000).into_par_iter().sum::<i64>(),
// );
```

测量中要注意预热（warmup）——第一次运行通常更慢（冷缓存、JIT 等）。取多轮平均、排除异常值。

## 8. Rayon 的安全保证 —— 为什么并行不会引入数据竞争

### 8.1 核心保证：顺序语义

Rayon 的核心安全承诺是：**并行执行的程序在语义上等同于某个顺序执行**。这意味着，如果你的代码在串行执行时没有数据竞争，那么 Rayon 的并行执行就不会引入数据竞争。

```rust
use rayon::prelude::*;

// Rayon 通过以下机制保证顺序语义：

// 1. 闭包是纯函数式的（映射、过滤）
//    .map(|x| x * 2) —— 每个元素的处理不依赖其他元素
//    → 无论并行执行顺序如何，结果都等价于某种串行执行

// 2. collect 保证最终顺序
let v = vec![3, 1, 4, 1, 5];
let doubled: Vec<i32> = v.par_iter()
    .map(|x| x * 2)
    .collect();
assert_eq!(doubled, vec![6, 2, 8, 2, 10]);
// collect 保证了结果顺序和输入顺序一致

// 3. 归约操作（sum, reduce 等）要求操作是结合的
//    结合律保证了无论分组和合并顺序如何，结果一致
```

### 8.2 闭包捕获的可变性

Rayon 并行迭代器中的闭包要求是 `Fn + Send + Sync`。因为是并行执行的，每个闭包实例可能运行在不同线程上。

```rust
use rayon::prelude::*;

// ❌ 在闭包中修改共享状态（数据竞争）
fn bad_example() {
    let mut counter = 0;  // 普通的 i32
    let v = vec![1, 2, 3, 4, 5];

    // v.par_iter().for_each(|x| {
    //     counter += x;  // ❌ 编译错误！
    //     // 多个线程同时修改 counter → 数据竞争
    //     // 编译器拒绝：闭包需要 Fn，不能捕获 &mut counter
    // });
}

// ✅ 用原子类型替代
fn good_example() {
    use std::sync::atomic::{AtomicI32, Ordering};
    let counter = AtomicI32::new(0);
    let v = vec![1, 2, 3, 4, 5];

    v.par_iter().for_each(|x| {
        counter.fetch_add(*x, Ordering::Relaxed);
    });

    assert_eq!(counter.load(Ordering::Relaxed), 15);
}

// ✅ 或者用 Rayon 的 fold/reduce 替代手动累加
fn better_example() {
    let v = vec![1, 2, 3, 4, 5];
    let sum: i32 = v.par_iter()
        .fold(|| 0, |acc, x| acc + x)
        .reduce(|| 0, |a, b| a + b);
    assert_eq!(sum, 15);
}
```

Rayon 在编译期阻止大多数数据竞争：闭包不能捕获 `&mut` 到非 Sync 类型。如果你需要在并行闭包中访问可变状态，必须使用 `Mutex`、`Atomic*`，或者使用 `fold`/`reduce` 提供的"线程局部分配 + 合并"模式。

## 9. 实战案例

### 9.1 案例 1：并行图像处理

图像处理是数据并行的经典案例——每个像素独立处理，天然适合并行化。

```rust
use rayon::prelude::*;

// 一个简化的图像类型（灰度图）
struct Image {
    width: usize,
    height: usize,
    pixels: Vec<u8>,  // 每像素一个字节（0-255）
}

impl Image {
    // 并行亮度调整：每个像素 * brightness_factor
    fn adjust_brightness(&mut self, factor: f32) {
        self.pixels.par_iter_mut()
            .for_each(|pixel| {
                let new_val = (*pixel as f32 * factor) as u8;
                *pixel = new_val.min(255);
            });
    }

    // 并行高斯模糊（简化版——3x3 均值模糊）
    fn blur_simple(&self) -> Image {
        // 每个输出像素 = 周围 3x3 区域的平均值
        let blurred: Vec<u8> = (0..self.height)
            .into_par_iter()
            .flat_map(|y| {
                (0..self.width).map(move |x| {
                    self.get_blur_pixel(x, y)
                }).collect::<Vec<u8>>()
            })
            .collect();

        Image {
            width: self.width,
            height: self.height,
            pixels: blurred,
        }
    }

    fn get_blur_pixel(&self, cx: usize, cy: usize) -> u8 {
        let mut sum: u32 = 0;
        let mut count: u32 = 0;
        for dy in 0..3 {
            for dx in 0..3 {
                let y = cy + dy;
                let x = cx + dx;
                if y < self.height && x < self.width && y > 0 && x > 0 {
                    sum += self.pixels[y * self.width + x - 1] as u32;
                    count += 1;
                }
            }
        }
        if count == 0 { 0 } else { (sum / count) as u8 }
    }
}

// 这个例子展示了：
// 1. par_iter_mut 用于原地修改（像素不需要重新分配数组）
// 2. into_par_iter + flat_map 用于"每行一个线程"的二维处理
// 3. 图像大小 > 1000x1000 时并行效果显著
```

### 9.2 案例 2：并行 map-reduce 风格的数据分析

Rayon 特别适合"对大量独立记录做同样的聚合"的场景。

```rust
use rayon::prelude::*;
use std::collections::HashMap;

#[derive(Debug, Clone)]
struct Sale {
    region: String,
    product: String,
    amount: f64,
}

fn analyze_sales(sales: &[Sale]) -> HashMap<String, f64> {
    // 目标：按 region 汇总 amount
    // 方法：每个线程维护局部 HashMap，最后合并

    sales.par_iter()
        .fold(
            || HashMap::new(),  // 每个线程创建自己的 HashMap
            |mut acc, sale| {
                // 局部累积（线程内串行）
                *acc.entry(sale.region.clone()).or_insert(0.0) += sale.amount;
                acc
            },
        )
        .reduce(
            || HashMap::new(),
            |mut map1, map2| {
                // 合并多个线程的 HashMap（串行——但 HashMap 数量 = 线程数）
                for (key, val) in map2 {
                    *map1.entry(key).or_insert(0.0) += val;
                }
                map1
            },
        )
}

// 这个模式的精髓：
// 1. 每个线程维护自己独立的 HashMap（fold 阶段）
//    → 没有锁竞争
// 2. 最后合并时只有 N_thread 个 HashMap 需要合并（reduce 阶段）
//    → 合并开销与线程数相关，与数据量无关
// 3. 比"共享一个 Mutex<HashMap>"快得多
//    → 因为 fold 阶段完全没有同步开销
```

### 9.3 案例 3：并行全文搜索

在大量文件中搜索关键词是既 I/O 密集又计算密集的任务。I/O 部分通常主线程做，搜索部分可以并行。

```rust
use rayon::prelude::*;

struct SearchResult {
    file_path: String,
    line_number: usize,
    content: String,
}

fn search_in_files(files: &[String], keyword: &str) -> Vec<SearchResult> {
    // 先串行读取所有文件（I/O——不适合并行化）
    let file_contents: Vec<(String, String)> = files.iter()
        .map(|path| {
            let content = std::fs::read_to_string(path)
                .unwrap_or_default();
            (path.clone(), content)
        })
        .collect();

    // 并行搜索（计算密集——适合并行化）
    file_contents.par_iter()
        .flat_map(|(path, content)| {
            content.lines()
                .enumerate()
                .filter(|(_, line)| line.contains(keyword))
                .map(|(line_num, line)| SearchResult {
                    file_path: path.clone(),
                    line_number: line_num + 1,
                    content: line.to_string(),
                })
                .collect::<Vec<SearchResult>>()
        })
        .collect()
}

// 设计决策：
// - I/O 部分保持串行——多个线程同时读磁盘通常不会更快（受磁盘带宽限制）
// - 搜索部分并行——纯 CPU 计算，可以充分利用多核
// - flat_map 把每个文件的搜索展开成结果流
// - 最终的 collect 收集所有匹配结果
```

## 10. Rayon vs 手动线程 —— 选择指南

### 10.1 何时用 Rayon

```
适合 Rayon 的场景：
├── 数据并行——对大量元素做相同操作
│   - map、filter、sum、for_each...
│   - 特点：逻辑统一，数据独立，适合分裂
│
├── 任务并行——多个独立任务需要同时执行
│   - join(a, b)、scope(|s| { s.spawn(...) })
│   - 特点：任务数量远大于核心数
│
└── 不需要精细控制线程生命周期
    - 任务之间没有复杂的依赖关系
    - 不需要线程优先级、亲和性等底层控制
```

### 10.2 何时用手动线程

```
适合手动线程的场景：
├── 需要长期存在、有特定角色的线程
│   - 例如：后台监控线程、网络 I/O 线程
│   - Rayon 的线程是"临时工"——用完即回收
│
├── 线程之间有复杂的同步关系
│   - 例如：生产者-消费者管道，精确控制缓冲区
│   - 用 std::thread + channels
│
├── 需要操作系统级别的控制
│   - 线程优先级（nice value）
│   - CPU 亲和性（affinity）
│   - 实时调度（SCHED_FIFO）
│   - Rayon 抽象掉了这些细节
│
└── 阻塞操作
    - 如果任务中包含可能长时间阻塞的操作（I/O、锁等待）
    - Rayon 的线程池会被阻塞的任务堵住
    - 应该用 async/await 或独立的专用线程
```

### 10.3 混合使用的场景

实际项目中，Rayon 和手动线程/异步通常是配合使用的：

```
┌────────────────────────────────────────────┐
│              主线程 / 异步运行时             │
│                                             │
│   收到一批数据（如 100 万条记录）              │
│       │                                     │
│       └→ tokio::task::spawn_blocking(move || │
│             数据.par_iter()   ← Rayon       │
│                 .map(heavy_compute)          │
│                 .collect()                   │
│          )                                   │
│                                             │
│   heavy_compute 中不要有 .await              │
│   ── Rayon 的任务应该是纯计算的              │
└────────────────────────────────────────────┘
```

这是"异步 + Rayon"的经典组合：异步处理 I/O，Rayon 处理计算。`spawn_blocking` 把 CPU 密集型任务放在 tokio 的阻塞线程池上执行，避免阻塞异步 worker 线程。

## 11. 常见陷阱

### 陷阱 1：在 par_iter 闭包中使用非线程安全的类型

```rust
use rayon::prelude::*;
use std::cell::RefCell;

// ❌ RefCell 不是 Sync——不能在并行闭包中共享
fn bad_parallel_state() {
    let state = RefCell::new(0);  // RefCell: !Sync

    // let v: Vec<i32> = (0..10).collect();
    // v.par_iter().for_each(|x| {
    //     *state.borrow_mut() += x;  // ❌ 编译错误
    // });
    // error: RefCell<i32> cannot be shared between threads safely
}

// ✅ 使用线程安全的替代品
fn good_parallel_state() {
    use std::sync::atomic::{AtomicI32, Ordering};
    let state = AtomicI32::new(0);  // AtomicI32: Sync

    let v: Vec<i32> = (0..10).collect();
    v.par_iter().for_each(|x| {
        state.fetch_add(*x, Ordering::Relaxed);
    });

    assert_eq!(state.load(Ordering::Relaxed), 45);
}
```

### 陷阱 2：忽略 par_iter 的顺序不确定性

```rust
use rayon::prelude::*;

// ❌ 依赖执行顺序的代码
fn bad_order_assumption() {
    let mut last_seen = 0;
    // (0..100).into_par_iter().for_each(|i| {
    //     assert!(i >= last_seen);  // ❌ 这个断言会失败！
    //     last_seen = i;
    // });
    // 并行执行中 i 的出现顺序不是 0, 1, 2, ..., 99
}

// ✅ 正确处理顺序不确定的代码
fn good_order_handling() {
    let results: Vec<i32> = (0..100).into_par_iter()
        .map(|i| i * 2)
        .collect();
    // collect 保证顺序——results 是 [0, 2, 4, ..., 198]
    // 即使 map 的执行顺序不确定
}
```

### 陷阱 3：小数据集过度并行化

```rust
use rayon::prelude::*;

// ❌ 100 个元素不值得并行
// let v: Vec<i32> = (0..100).collect();
// v.par_iter().for_each(|x| {
//     // 任何轻量操作——任务管理开销远超实际计算
// });

// ✅ 经验法则：
// - 元素数 < 1000  → 通常不值得并行
// - 元素计算量 < 1µs → 通常不值得并行
// - 例外：每个元素的计算量极重（> 10ms）→ 即使只有 10 个元素也值得

// ✅ 可以使用 with_min_len 控制任务粒度
fn controlled_parallelism(v: &[i32]) -> i32 {
    v.par_iter()
        .with_min_len(1024)  // 每份至少 1024 个元素才拆分
                             // 数据量小时自动退化为串行
        .sum()
}
```

`with_min_len` 是一个很好的实践——它告诉 Rayon "每个子任务至少要处理这么多元素"。如果数据总量小于这个阈值，Rayon 自动退化为串行执行，避免了不必要的并行开销。

### 陷阱 4：在 par_iter 中调用阻塞 I/O

```rust
use rayon::prelude::*;

// ❌ 在 Rayon 任务中做阻塞操作
// let urls = vec!["http://a.com", "http://b.com"];
// urls.par_iter().for_each(|url| {
//     let response = reqwest::blocking::get(url).unwrap();  // ❌ 阻塞 I/O
//     // 问题：这会阻塞 Rayon 线程池的线程
//     // 如果所有线程都被阻塞 → 线程池饿死
// });

// ✅ 正确做法：
// 1. 用异步 I/O（tokio + reqwest）做 HTTP 请求
// 2. Rayon 只用于纯计算
// 3. 或者用 std::thread::scope 创建专用线程（而非 Rayon 线程池）
```

## 11. 练习

### 练习 1：并行计算 π 的近似值

使用 Rayon 的 `par_iter` 实现蒙特卡洛方法计算 π：在边长为 1 的正方形中随机撒点，统计落在四分之一圆内的比例。用 `(1..N).into_par_iter()` 并行化。比较并行版本和串行版本的性能差异。

### 练习 2：用 join 实现并行归并排序

use rand::Rng;
use rayon::prelude::*;

fn monte_carlo_pi(n: u64) -> f64 {
    let count_in_circle = (0..n)
        .into_par_iter()
        .map(|_| {
            let mut rng = rand::thread_rng();
            let x: f64 = rng.gen();
            let y: f64 = rng.gen();
            if x * x + y * y <= 1.0 { 1u64 } else { 0u64 }
        })
        .sum::<u64>();

    4.0 * count_in_circle as f64 / n as f64
}

fn main() {
    let n = 100_000_000;

    // 并行版本
    let start = Instant::now();
    let pi_parallel = monte_carlo_pi(n);
    let parallel_time = start.elapsed();
    println!("Rayon:   π ≈ {:.10} ({:?})", pi_parallel, parallel_time);

    // 串行版本（对比）
    let start = Instant::now();
    let mut count = 0u64;
    let mut rng = rand::thread_rng();
    for _ in 0..n {
        let x: f64 = rng.gen();
        let y: f64 = rng.gen();
        if x * x + y * y <= 1.0 { count += 1; }
    }
    let pi_serial = 4.0 * count as f64 / n as f64;
    let serial_time = start.elapsed();
    println!("Serial:  π ≈ {:.10} ({:?})", pi_serial, serial_time);

    // 预期：Rayon 版本在多核上应该快 3-6 倍
    println!("Speedup: {:.1}x", serial_time.as_secs_f64() / parallel_time.as_secs_f64());

### 练习 2 答案

```rust
use rayon::prelude::*;

/// 并行归并排序
fn par_merge_sort<T: Ord + Send + Sync + Copy>(slice: &mut [T]) {
    let len = slice.len();

    // 小数组直接串行排序——避免任务管理开销超过实际计算
    if len <= 4096 {
        slice.sort();
        return;
    }

    let mid = len / 2;
    let (left, right) = slice.split_at_mut(mid);

    // 用 join 并行排序左右两半
    rayon::join(
        || par_merge_sort(left),
        || par_merge_sort(right),
    );

    // 合并两个已排序的切片
    merge(left, right);
}

/// 合并两个已排序的切片（结果写回原数组）
fn merge<T: Ord + Copy>(left: &mut [T], right: &mut [T]) {
    let mut result = Vec::with_capacity(left.len() + right.len());
    let mut i = 0;
    let mut j = 0;

    while i < left.len() && j < right.len() {
        if left[i] <= right[j] {
            result.push(left[i]);
            i += 1;
        } else {
            result.push(right[j]);
            j += 1;
        }
    }
    result.extend_from_slice(&left[i..]);
    result.extend_from_slice(&right[j..]);

    // 写回 left 和 right 的连续内存区域
    let total = left.len() + right.len();
    left.iter_mut()
        .chain(right.iter_mut())
        .zip(result.into_iter())
        .for_each(|(dst, src)| *dst = src);
}

// 关键设计决策：阈值 4096
// 当数组小于 4096 时，join 的任务调度开销（~100ns）
// 超过了并行化节省的时间，所以切换到串行 sort()
```

### 练习 3 答案

```rust
use rayon::prelude::*;
use std::collections::BinaryHeap;
use std::cmp::Reverse;

/// 用 Rayon 找出 vec 中最大的 k 个值
fn top_k_parallel(data: &[u64], k: usize) -> Vec<u64> {
    let chunk_size = (data.len() / rayon::current_num_threads()).max(1024);

    // 每个 chunk 内部找 top-k（用最小堆）
    let partial_tops: Vec<BinaryHeap<Reverse<u64>>> = data
        .par_chunks(chunk_size)
        .map(|chunk| {
            let mut heap = BinaryHeap::with_capacity(k + 1);
            for &val in chunk {
                heap.push(Reverse(val));
                if heap.len() > k {
                    heap.pop(); // 弹出最小的，保持堆大小为 k
                }
            }
            heap
        })
        .collect();

    // 合并所有 chunk 的 top-k 结果
    let mut final_heap = BinaryHeap::with_capacity(k + 1);
    for heap in partial_tops {
        for Reverse(val) in heap {
            final_heap.push(Reverse(val));
            if final_heap.len() > k {
                final_heap.pop();
            }
        }
    }

    // 转换成降序排列
    let mut result: Vec<u64> = final_heap
        .into_iter()
        .map(|Reverse(v)| v)
        .collect();
    result.sort_unstable_by(|a, b| b.cmp(a));
    result
}

// 复杂度分析：
// 串行：O(n log k)
// 并行：O((n/p) log k + p*k log k)，其中 p = 线程数
// 当 n 很大时，接近线性加速比
```

### 练习 4 答案

当 `v.len() == 100` 时，Rayon 版本**比串行慢**——每个元素只做一次加法（~1ns），而 Rayon 的任务调度、work-stealing 队列操作、线程同步的开销（~数十纳秒到微秒）远超计算本身。`with_min_len` 的默认阈值约 1000-10000，100 个元素根本不值得并行化。

当 `v.len() == 100,000,000` 时，Rayon 版本比串行**快 3-6 倍**（在多核上）。计算总量足够大，任务调度开销被摊薄到可忽略。

经验法则：**每个元素的计算量 × 元素数量 >> 调度开销时，才值得并行化。** 轻量操作（加法、比较）需要更大的数据量；重量操作（hash 计算、字符串处理）在较小的数据量上就能受益。
