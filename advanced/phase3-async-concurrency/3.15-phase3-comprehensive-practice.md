## 2. Demo 1: 同步并发全栈 —— 从 Mutex 到 Rayon

### 2.1 设计哲学

> "并发不难——难的是**选对工具**。同一个'并行搜索'任务，初学者用 `Mutex<Vec<T>>`，老手用 scoped threads + TLS，专家用 Rayon 一行代码。本 Demo 带你走完这条进化链，让你在每次选择时都有清晰的成本模型。"

### 2.2 场景：并行搜索大数组

```rust
// 目标：在 100_000_000 个 i32 中找所有能被 17 整除的数
// 我们分别用四种方案实现，对比性能和安全边界

const DATA_SIZE: usize = 100_000_000;
const NUM_THREADS: usize = 8;
```

### 2.3 方案 A：Mutex + Arc（最直观，但有锁竞争）

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn search_with_mutex(data: &[i32]) -> Vec<i32> {
    let chunk_size = data.len() / NUM_THREADS;
    let shared_data = Arc::new(data);  // ━━━ 3.1 Send/Sync ━━━
    let results = Arc::new(Mutex::new(Vec::new())); // ━━━ 3.3 Mutex ━━━

    let mut handles = vec![];
    for i in 0..NUM_THREADS {
        let data = Arc::clone(&shared_data);
        let results = Arc::clone(&results);

        let handle = thread::spawn(move || {
            // ━━━ Send/Sync 检查点（3.1）━━━
            // Arc<[i32]>: Send + Sync (因为 [i32]: Send + Sync)
            // Arc<Mutex<Vec<i32>>>: Send + Sync (因为 Mutex 让任何 T: Send 变 Sync)
            let start = i * chunk_size;
            let end = ((i + 1) * chunk_size).min(data.len());
            let mut local = Vec::new();

            for &val in &data[start..end] {
                if val % 17 == 0 {
                    local.push(val);
                }
            }

            // ━━━ 锁竞争点（3.3 Mutex 性能瓶颈）━━━
            let mut guard = results.lock().unwrap();
            guard.extend(&local);
            // guard 在这里自动 drop——RAII 解锁
        });
        handles.push(handle);
    }

    for h in handles { h.join().unwrap(); }

    // ━━━ Mutex 中毒检查 ━━━
    let results = Arc::try_unwrap(results)
        .unwrap_or_else(|_| panic!("Arc still has references"))
        .into_inner()
        .unwrap();  // Mutex::into_inner 获取内部值
    results
}
```

### 2.4 方案 B：Scoped Threads + 分片借用（零 Arc，零锁）

```rust
use std::thread;

fn search_with_scoped(data: &mut [i32]) -> Vec<i32> {
    // ━━━ 3.6 Scoped Threads ━━━
    // scope 闭包可以直接借用栈上数据——不需要 Arc！
    let chunk_size = data.len() / NUM_THREADS;

    // ━━━ 3.7 Once: 初始化全局配置（只跑一次）━━━
    use std::sync::Once;
    static INIT: Once = Once::new();
    static mut THREAD_INIT_COUNT: usize = 0;
    INIT.call_once(|| {
        // 初始化只执行一次——即使多线程同时调用
        println!("[Once] 首次初始化完成");
    });

    // ━━━ 3.4 Barrier: 让所有线程在同一阶段同步 ━━━
    use std::sync::Barrier;
    let barrier = Barrier::new(NUM_THREADS + 1); // +1 给主线程

    // 分片给每个线程：`&mut [i32]` 切分成互不重叠的切片
    let mut results: Vec<Vec<i32>> = (0..NUM_THREADS).map(|_| Vec::new()).collect();

    thread::scope(|s| {
        for (i, result_slot) in results.iter_mut().enumerate() {
            let start = i * chunk_size;
            let end = ((i + 1) * chunk_size).min(data.len());
            let slice = &data[start..end];

            s.spawn(move || {
                // ★ 关键：scope 保证所有线程在 scope 结束前 join
                // 所以 slice 的借用是安全的——data 在栈上没有被移动
                for &val in slice {
                    if val % 17 == 0 {
                        result_slot.push(val);
                    }
                }

                // ━━━ 3.4 Barrier: 阶段同步 ━━━
                barrier.wait();  // 所有线程在这里等待其他线程
                // 所有线程 collect 完成后才继续
            });
        }

        // 主线程也等待
        barrier.wait();
    });

    // 合并结果——不需要 Mutex，因为每个线程写入独立槽位
    results.into_iter().flatten().collect()
}
```

### 2.5 方案 C：TLS（Thread Local Storage）—— 无锁聚合

```rust
use std::cell::RefCell;
use std::thread;

fn search_with_tls(data: &[i32]) -> Vec<i32> {
    // ━━━ 3.5 TLS: 每个线程独立的结果缓冲区 ━━━
    thread_local! {
        // RefCell 是 thread_local 的经典搭档——单线程内部可变（回顾 Phase 2 2.15）
        static LOCAL_RESULTS: RefCell<Vec<i32>> = RefCell::new(Vec::new());
    }

    let chunk_size = data.len() / NUM_THREADS;

    thread::scope(|s| {
        for i in 0..NUM_THREADS {
            let start = i * chunk_size;
            let end = ((i + 1) * chunk_size).min(data.len());
            let slice = &data[start..end];

            s.spawn(move || {
                for &val in slice {
                    if val % 17 == 0 {
                        LOCAL_RESULTS.with(|cell| {
                            cell.borrow_mut().push(val);
                            // borrow_mut 在单线程内总是安全的——不存在竞争
                        });
                    }
                }
            });
        }
    });

    // 收集所有线程的 TLS 结果
    // 注意：TLS 结果在 scope 结束后无法访问——需要在 scope 内收集
    // 所以 TLS 更适合"每个线程只处理自己的工作"的场景
    Vec::new()  // 示意——实际收集需要额外机制
}
```

### 2.6 方案 D：Rayon（一行搞定）

```rust
use rayon::prelude::*;

fn search_with_rayon(data: &[i32]) -> Vec<i32> {
    // ━━━ 3.9 Rayon: 数据并行，零样板代码 ━━━
    data.par_iter()              // 并行迭代器
        .filter(|&&x| x % 17 == 0)
        .copied()
        .collect()               // 自动合并——Rayon 自己处理聚合

    // 对比：方案 A 需要 ~30 行，方案 B 需要 ~25 行，Rayon 只需 4 行
    // Rayon 内部使用 Work Stealing + 自适应分片
}
```

## 3. Demo 2: Lock-Free 编程 — 从 Atomics 到 Channel 内部

### 3.1 设计哲学

> "锁是并发编程的安全网——也是最慢的安全网。当你需要极致性能时，`Atomic` 是你的武器。但 Memory Ordering 不是神秘咒语——每个 Ordering 都有精确的语义。学会 Relaxed/Acquire/Release/SeqCst 的选择原则，你就从'用锁的开发者'变成了'理解硬件的工程师'。"

### 3.2 Treiber Stack —— 无锁栈的经典实现

```rust
use std::ptr::NonNull;
use std::sync::atomic::{AtomicPtr, Ordering};

/// ━━━ Treiber Stack: CAS 循环的教科书案例 ━━━
///
/// push: CAS(old_head, new_node) — 原子地更新栈顶
/// pop:  CAS(old_head, old_head.next) — 原子地移除栈顶
struct Node<T> {
    value: T,
    next: Option<NonNull<Node<T>>>,
}

pub struct TreiberStack<T> {
    // ━━━ 3.2 Atomic: AtomicPtr 是 lock-free 的基础 ━━━
    head: AtomicPtr<Node<T>>,
}

impl<T> TreiberStack<T> {
    pub fn new() -> Self {
        TreiberStack { head: AtomicPtr::new(std::ptr::null_mut()) }
    }

    pub fn push(&self, value: T) {
        let new_node = Box::into_raw(Box::new(Node {
            value,
            next: None,
        }));

        // ━━━ CAS 循环（3.2 Atomic）━━━
        loop {
            let old_head = self.head.load(Ordering::Acquire);  // ← Acquire
            // SAFETY: old_head is null or a valid pointer
            unsafe { (*new_node).next = NonNull::new(old_head).map(|p| p.as_ptr().cast()); }

            // ━━━ CAS: 如果 head == old_head，原子地设置为 new_node ━━━
            match self.head.compare_exchange_weak(
                old_head,
                new_node,
                Ordering::Release,   // ← Release: 保证 new_node 的写入对后续 pop 可见
                Ordering::Relaxed,   // ← Relaxed: 失败时不需要同步保证
            ) {
                Ok(_) => break,      // 成功——new_node 现在是栈顶
                Err(_) => continue,  // 失败——另一个线程抢先了，重试
            }
        }
    }

    pub fn pop(&self) -> Option<T> {
        loop {
            let old_head = self.head.load(Ordering::Acquire);
            if old_head.is_null() {
                return None;  // 栈空
            }

            let next = unsafe { (*old_head).next };

            match self.head.compare_exchange_weak(
                old_head,
                next.map_or(std::ptr::null_mut(), |p| p.as_ptr()),
                Ordering::Release,
                Ordering::Relaxed,
            ) {
                Ok(_) => {
                    // 成功取出——把旧节点转回 Box 以正确释放
                    let boxed = unsafe { Box::from_raw(old_head) };
                    return Some(boxed.value);
                }
                Err(_) => continue,  // 被抢先，重试
            }
        }
    }
}

// ━━━ Memory Ordering 选择理由 ━━━
//
// Acquire (load): "我之后的所有读写不能被重排到这个 load 之前"
//                 用于读取共享状态——保证看到最新值
//
// Release (store): "我之前的所有读写不能被重排到这个 store 之后"
//                  用于发布共享状态——保证发布前所有写入对其他线程可见
//
// Relaxed: 只保证原子性，不保证顺序
//          用于计数器——不需要 happens-before 关系
//
// SeqCst: 全局统一顺序——最重，只在不确定时使用

// ━━━ ABA 问题 ━━━
// Treiber Stack 的 CAS 循环有一个经典问题：ABA
// 线程 A 读到 head=0x1000，线程 B pop 了 0x1000 又 push 了新的 0x1000
// 线程 A 的 CAS 会成功——但 0x1000 的内容已经变了
// 解决方案：tagged pointers（指针+版本号）、epoch-based reclamation
```

### 3.3 多线程测试（Scoped Threads + TreiberStack）

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use std::thread;

    #[test]
    fn test_treiber_stack_concurrent() {
        let stack = TreiberStack::new();

        // ━━━ 3.6 Scoped Threads: 借用栈上的 stack ━━━
        thread::scope(|s| {
            for t in 0..4 {
                s.spawn(|| {
                    for i in 0..1000 {
                        stack.push(t * 1000 + i);
                    }
                });
            }
        });

        // 并行 push 完成——现在 pop 出来
        let mut count = 0;
        while let Some(_) = stack.pop() {
            count += 1;
        }
        assert_eq!(count, 4000);  // 4 threads × 1000
    }
}
```

### 3.4 标准库 mpsc 的内部解剖

```rust
/// ━━━ 3.8 mpsc 内部：Mutex + Condvar + VecDeque ━━━
///
/// std::sync::mpsc::channel 的内部大致等价于：

use std::collections::VecDeque;
use std::sync::{Arc, Condvar, Mutex};

struct Inner<T> {
    queue: Mutex<VecDeque<T>>,     // ━━━ 3.3 Mutex ━━━
    // Condvar 用于阻塞等待（3.4 Condvar）
    // receiver 在队列空时 wait，sender 在 send 后 notify
    condvar: Condvar,
}

struct MySender<T> {
    inner: Arc<Inner<T>>,           // ━━━ 3.1 Send/Sync ━━━
    // Arc 允许 Sender clone——实现 multi-producer
}

struct MyReceiver<T> {
    inner: Arc<Inner<T>>,
}

impl<T> MySender<T> {
    fn send(&self, value: T) {
        let mut queue = self.inner.queue.lock().unwrap();
        queue.push_back(value);
        // ━━━ 3.4 Condvar: 通知等待的 receiver ━━━
        self.inner.condvar.notify_one();
        // lock 在这里自动释放
    }
}

impl<T> MyReceiver<T> {
    fn recv(&self) -> Option<T> {
        let mut queue = self.inner.queue.lock().unwrap();

        // ━━━ Condvar 的经典模式: while + wait ━━━
        while queue.is_empty() {
            // 检查 Sender 是否已全部 drop
            if Arc::strong_count(&self.inner) == 1 {
                // 只剩下 receiver 自己——Sender 已全部释放
                return None;
            }
            // wait 原子地释放锁并阻塞，收到 notify 后重新获取锁
            queue = self.inner.condvar.wait(queue).unwrap();
        }

        queue.pop_front()
    }
}

// ━━━ oneshot 的内部 ━━━
// oneshot::channel 更简单：AtomicBool 标记是否已发送 + Mutex<Option<T>>
// 因为只有一次 send，不需要 VecDeque

// ━━━ broadcast 的内部 ━━━
// broadcast 用环形缓冲区 + RwLock + 每个 receiver 的独立游标
// lagged 检测：如果 receiver 落后太多，缓冲区覆盖了它还没读的数据
```

## 4. Demo 3: Actor 系统 —— 同步版 vs 异步版

### 4.1 设计哲学

> "Actor 模型是并发编程的'面向对象'——每个 Actor 是独立的状态 + 消息处理循环。同一个 Actor 模式，用线程+mpsc 实现是'每个 Actor 一个线程'（重量级），用 `tokio::spawn` 实现是'每个 Actor 一个 task'（轻量级）。两者的核心逻辑一模一样——差异全在 Send/Sync 约束、async trait 和取消安全上。"

### 4.2 公用定义

```rust
/// Actor 可以处理的消息
trait Message: Send + 'static {
    type Response: Send + 'static;
}

/// 一个简单的计数器 Actor——支持 Increment 和 GetCount
enum CounterMsg {
    Increment(i64),
    GetCount,
}

impl Message for CounterMsg {
    type Response = i64;
}
```

### 4.3 同步版 Actor（threads + mpsc）

```rust
use std::sync::mpsc;
use std::thread;

struct SyncActor<M: Message> {
    sender: mpsc::SyncSender<(M, mpsc::Sender<M::Response>)>,
    // oneshot 用于请求-响应模式（3.8）
}

impl<M: Message> SyncActor<M> {
    fn spawn<F>(initial_state: F) -> Self
    where
        F: FnOnce() -> M::Response,
    {
        // SyncSender 带 backpressure——队列满时阻塞
        let (tx, rx) = mpsc::sync_channel::<(M, mpsc::Sender<M::Response>)>(32);

        thread::spawn(move || {
            // Actor 的"状态"——闭包捕获的局部变量
            let mut state = initial_state();

            for (msg, reply_to) in rx {
                // 处理消息...
                // reply_to.send(response) 把结果发回调用者
                let _ = reply_to;  // 简化
                let _ = msg;
                state = initial_state();  // 示意
            }
        });

        SyncActor { sender: tx }
    }

    fn send(&self, msg: M) -> M::Response {
        let (reply_tx, reply_rx) = mpsc::channel();
        self.sender.send((msg, reply_tx)).unwrap();
        reply_rx.recv().unwrap()
    }
}

// ━━━ 3.4 Condvar 在同步 Actor 中的角色 ━━━
// mpsc::sync_channel 内部使用 Mutex + Condvar
// 当队列满时，send 调用 condvar.wait 阻塞
// 当队列有空位时，receiver 调用 condvar.notify 唤醒 sender
```

### 4.4 异步版 Actor（tokio + async mpsc + async trait）

```rust
use tokio::sync::{mpsc, oneshot};

/// ━━━ 3.12 async trait ━━━
/// Actor 的 handle 方法——异步处理消息
trait AsyncActor: Send + 'static {
    type Msg: Send + 'static;

    /// 处理单条消息，返回响应
    async fn handle(&mut self, msg: Self::Msg) -> String;
}

/// ━━━ 3.10 Future: Actor 的消息循环是一个 Future ━━━
async fn run_actor<A: AsyncActor>(mut actor: A, mut rx: mpsc::Receiver<(A::Msg, oneshot::Sender<String>)>) {
    // ━━━ 3.13 Stream: 将 channel 转为 Stream ━━━
    // 等价于：while let Some((msg, reply)) = rx.recv().await
    while let Some((msg, reply)) = rx.recv().await {
        let response = actor.handle(msg).await;
        let _ = reply.send(response);
    }
}

/// 异步 Actor 句柄
struct AsyncActorHandle<M: Send + 'static> {
    sender: mpsc::Sender<(M, oneshot::Sender<String>)>,
}

impl<M: Send + 'static> AsyncActorHandle<M> {
    async fn send(&self, msg: M) -> Result<String, String> {
        let (reply_tx, reply_rx) = oneshot::channel();
        self.sender.send((msg, reply_tx)).await
            .map_err(|_| "Actor已关闭".to_string())?;
        reply_rx.await.map_err(|_| "Actor在回复前崩溃".to_string())
    }
}

fn spawn_async_actor<A: AsyncActor<Msg = M>, M: Send + 'static>(
    actor: A,
) -> AsyncActorHandle<M> {
    let (tx, rx) = mpsc::channel(32);
    tokio::spawn(run_actor(actor, rx));  // 轻量级 task——不是线程！
    AsyncActorHandle { sender: tx }
}
```

### 4.5 取消安全对比

```rust
/// ━━━ 3.14 取消安全 ━━━
///
/// 同步 Actor：取消意味着 drop handle + thread join
/// - 简单粗暴：线程被强制终止
/// - 数据一致性由 RAII + Drop 保证
///
/// 异步 Actor：取消意味着 future 不再被 poll
/// - 微妙！可能在任意 .await 点被取消
/// - 需要用 CancellationToken 做优雅关闭

use tokio_util::sync::CancellationToken;

async fn cancellable_actor(
    mut rx: mpsc::Receiver<(CounterMsg, oneshot::Sender<i64>)>,
    cancel: CancellationToken,
) {
    let mut count: i64 = 0;

    loop {
        tokio::select! {
            // ━━━ select! 分支（3.14）━━━
            Some((msg, reply)) = rx.recv() => {
                match msg {
                    CounterMsg::Increment(n) => { count += n; let _ = reply.send(count); }
                    CounterMsg::GetCount => { let _ = reply.send(count); }
                }
                // ★ 取消安全：如果在处理消息中被取消，
                // reply 可能永远不会被 send——
                // 调用者的 oneshot::Receiver 会返回 RecvError
            }
            _ = cancel.cancelled() => {
                // 优雅关闭：清理资源
                println!("Actor 收到取消信号，当前 count={}", count);
                break;
            }
        }
    }
}

// ━━━ 哪些操作是取消安全的？━━━
// ✓ 纯内存操作（读写局部变量）
// ✓ 原子操作
// ✗ 多步事务——中间状态可能被外部观察到
// ✗ 持有 MutexGuard 跨 .await（标准 Mutex）
// 经验法则：如果操作是可逆的或原子的——取消安全
```

```rust
/// ━━━ 3.10 Future: 手写 Future trait（简化版）━━━
/// 标准库的 Future 使用 Pin<&mut Self> + Context + Waker
/// 这里我们直接使用标准库的——因为自定义 Future trait 需要 nightly
```

### 5.3 Mini Executor — 任务调度

```rust
/// ━━━ 3.11 Tokio: 迷你版任务调度器 ━━━
///
/// 核心组件：
/// 1. 任务队列（VecDeque<Task>）
/// 2. poll 循环（block_on）
/// 3. Waker 唤醒机制

struct Task {
    // Box<dyn Future> 的类型擦除——类似 Tokio 的 RawTask
    future: Pin<Box<dyn Future<Output = ()> + Send>>,
}

struct MiniExecutor {
    // ━━━ 3.3 Mutex + 3.1 Send/Sync ━━━
    ready_queue: Arc<Mutex<VecDeque<Task>>>,
}

impl MiniExecutor {
    fn new() -> Self {
        MiniExecutor { ready_queue: Arc::new(Mutex::new(VecDeque::new())) }
    }

    /// 提交一个 future
    fn spawn<F>(&self, future: F)
    where
        F: Future<Output = ()> + Send + 'static,
    {
        let task = Task { future: Box::pin(future) };
        self.ready_queue.lock().unwrap().push_back(task);
    }

    /// 事件循环——一直 poll 直到所有 future 完成
    fn block_on_all(&self) {
        let waker = make_dummy_waker();
        let mut cx = Context::from_waker(&waker);

        loop {
            let mut queue = self.ready_queue.lock().unwrap();

            if queue.is_empty() {
                break;
            }

            // 取一个 task
            let mut task = queue.pop_front().unwrap();
            drop(queue);  // 释放锁——poll 期间不持有 Mutex

            // ━━━ poll: Future 状态机的核心 ━━━
            if task.future.as_mut().poll(&mut cx).is_pending() {
                // 未完成——放回队列
                self.ready_queue.lock().unwrap().push_back(task);
            }
            // 如果 Ready，task 被 drop——future 也一起 drop
        }
    }
}
```

### 5.4 Mini Timer — 异步定时器

```rust
use std::time::{Duration, Instant};

/// ━━━ 3.10 Future: Timer 是一个 Future ━━━
struct MiniTimer {
    deadline: Instant,
    // ━━━ 3.2 Atomic: 标记是否已唤醒 ━━━
    woken: AtomicBool,
}

impl MiniTimer {
    fn new(duration: Duration) -> Self {
        MiniTimer {
            deadline: Instant::now() + duration,
            woken: AtomicBool::new(false),
        }
    }
}

impl Future for MiniTimer {
    type Output = ();

    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<()> {
        if Instant::now() >= self.deadline {
            Poll::Ready(())
        } else {
            if !self.woken.swap(false, Ordering::AcqRel) {
                // 安排唤醒——真实运行时用 epoll timerfd
                let waker = cx.waker().clone();
                let deadline = self.deadline;
                std::thread::spawn(move || {
                    let now = Instant::now();
                    if now < deadline {
                        std::thread::sleep(deadline - now);
                    }
                    waker.wake();  // 唤醒 executor 重新 poll
                });
            }
            Poll::Pending
        }
    }
}
```

```rust
/// 将 mpsc receiver 包装为 Stream
struct ChannelStream<T> {
    // 真实实现用 tokio::sync::mpsc::Receiver
    // 这里用 channel 模拟
    buffer: Arc<Mutex<VecDeque<T>>>,
    // ━━━ 3.7 Once/OnceCell: 惰性初始化 stream 状态 ━━━
    initialized: std::sync::OnceLock<()>,
}

impl<T> MyStream for ChannelStream<T> {
    type Item = T;

    fn poll_next(self: Pin<&mut Self>, _cx: &mut Context<'_>) -> Poll<Option<Self::Item>> {
        let mut buf = self.buffer.lock().unwrap();
        Poll::Ready(buf.pop_front())
    }
}

// ━━━ Stream 适配器：map ━━━
struct MapStream<S, F> {
    inner: S,
    f: F,
}

impl<S: MyStream, F, B> MyStream for MapStream<S, F>
where
    F: FnMut(S::Item) -> B,
{
    type Item = B;

    fn poll_next(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Option<Self::Item>> {
        // SAFETY: 结构投影——只访问 inner
        let inner = unsafe { self.as_mut().map_unchecked_mut(|s| &mut s.inner) };
        match inner.poll_next(cx) {
            Poll::Ready(Some(item)) => Poll::Ready(Some((self.f)(item))),
            Poll::Ready(None) => Poll::Ready(None),
            Poll::Pending => Poll::Pending,
        }
    }
}
```

### 6.3 参考答案

```rust
use std::sync::{Arc, Mutex, Barrier};
use std::thread;
use std::collections::VecDeque;
use std::sync::atomic::{AtomicBool, AtomicUsize, Ordering};
use std::cell::RefCell;

type Task = Box<dyn FnOnce() + Send + 'static>;

pub struct WorkStealingPool {
    workers: Vec<thread::JoinHandle<()>>,
    global_queue: Arc<Mutex<VecDeque<Task>>>,
    running: Arc<AtomicBool>,
    active_workers: Arc<AtomicUsize>,
}

impl WorkStealingPool {
    pub fn new(num_workers: usize) -> Self {
        let global_queue = Arc::new(Mutex::new(VecDeque::new()));
        let running = Arc::new(AtomicBool::new(true));
        let active_workers = Arc::new(AtomicUsize::new(0));

        // ━━━ 3.5 TLS: 每个 worker 的本地队列 ━━━
        thread_local! {
            static LOCAL_QUEUE: RefCell<VecDeque<Task>> = RefCell::new(VecDeque::new());
        }

        let mut workers = Vec::with_capacity(num_workers);
        for id in 0..num_workers {
            let gq = Arc::clone(&global_queue);
            let run = Arc::clone(&running);
            let aw = Arc::clone(&active_workers);

            let handle = thread::spawn(move || {
                aw.fetch_add(1, Ordering::SeqCst);  // ━━━ 3.2 Atomic ━━━

                while run.load(Ordering::Acquire) {
                    // 策略1: 本地队列
                    let task = LOCAL_QUEUE.with(|q| q.borrow_mut().pop_front());
                    let task = task.or_else(|| {
                        // 策略2: 全局队列
                        gq.lock().unwrap().pop_front()
                    });
                    // 策略3: 偷其他 worker（简化——直接去全局队列）

                    match task {
                        Some(task) => task(),
                        None => thread::yield_now(),
                    }
                }

                aw.fetch_sub(1, Ordering::SeqCst);
            });
            workers.push(handle);
        }

        WorkStealingPool { workers, global_queue, running, active_workers }
    }

    /// 提交任务到全局队列
    pub fn spawn<F>(&self, f: F)
    where
        F: FnOnce() + Send + 'static,
    {
        self.global_queue.lock().unwrap().push_back(Box::new(f));
    }

    /// 等待所有提交任务完成
    pub fn wait_idle(&self) {
        while !self.global_queue.lock().unwrap().is_empty()
            || self.active_workers.load(Ordering::Acquire) > 0
        {
            thread::yield_now();
        }
    }
}

impl Drop for WorkStealingPool {
    fn drop(&mut self) {
        self.running.store(false, Ordering::Release);
        // workers 在这里 join（当 Vec drop 时，JoinHandle drop 不 join）
        // 实际实现需要显式 join
    }
}
```
