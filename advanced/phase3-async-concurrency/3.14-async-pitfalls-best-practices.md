# 3.14 async 陷阱与最佳实践 —— Async Rust 的暗礁与航标

> Async Rust 是 Rust 中最锋利也最危险的工具。它的威力在于用看似"同步"的语法处理海量并发——但 `.await` 点的每一次暂停，都是一次"交出控制权"的隐式操作，对代码的正确性来说每一个 `.await` 都是一颗地雷：它可能是**取消点**（你的 future 可能再也不被 poll），它可能是**死锁点**（你持有的 MutexGuard 跨越了 .await 而被其他 task 等待），它可能是**性能悬崖**（你在 async 上下文中执行 CPU 密集型计算阻塞了整个运行时）。这一课不是教你"怎么写 async"——Phase 3 前 13 课已经覆盖了——而是教你"怎么写**对** async"。每一个陷阱都来自真实项目的血泪教训，每一个最佳实践都是为了让你在 `.await` 的刀尖上安全起舞。

---

## 1. 取消安全（Cancellation Safety）——Async Rust 的第一陷阱

### 1.1 什么是取消安全——问题的根源

```rust
// 核心问题：在 Rust 的异步模型中，一个 Future 可以在任何 .await 点被取消
// "取消"意味着：Future 被 drop，不再被 poll
// 如果这个 Future 在被取消时处于"中间状态"，数据一致性就可能被破坏

// 举例：一个简单的转账操作
async fn transfer(
    db: &mut Database,
    from: u64,
    to: u64,
    amount: i64,
) -> Result<(), DbError> {
    // 第一步：扣款
    db.debit(from, amount).await?;  // ← .await 点！如果在这里被取消...
    // 第二步：收款
    db.credit(to, amount).await?;   // ...钱已经从 from 扣了，但还没加到 to
    // → 钱凭空消失了！这就是取消不安全的典型后果
    Ok(())
}

// transfer() 的问题在于：它有两个 .await 点，中间的状态是"不一致的"
// 第一个 .await 之后，系统的状态是"from 扣了钱但 to 还没收到"
// 如果此时 Future 被取消（drop），这个不一致状态就永久固化了
```

取消安全的本质要求是：**一个 Future 在任何 `.await` 点被取消（drop）时，其已经完成的副作用不能破坏数据一致性**。这不是 Rust 的 bug——这是异步模型的核心特征。任何基于轮询的异步系统（包括 JavaScript 的 Promise、C# 的 Task、Python 的 asyncio）都面临同样的问题——只是在 Rust 中，因为所有权和 Drop 语义的显式性，这个问题更加尖锐。

### 1.2 典型案例：select! 中的取消

`tokio::select!` 是取消安全问题的"高发地带"——当多个分支竞速时，第一个完成的分支获胜，其他分支的 future 被 **drop**（取消）：

```rust
use tokio::sync::mpsc;

// ❌ 经典的取消不安全场景
async fn cancel_unsafe_example() {
    let (tx, mut rx) = mpsc::channel::<String>(32);
    let (result_tx, mut result_rx) = mpsc::channel::<String>(32);

    let tx_clone = tx.clone();

    // 任务 A：接收消息并写入数据库（写入中间可能被取消）
    let task_a = tokio::spawn(async move {
        tokio::select! {
            msg = rx.recv() => {
                let msg = msg.unwrap();
                // 假设这里有一个异步的数据库写入操作
                write_to_db(&msg).await;  // ← 如果 select! 的另一个分支先完成...
                result_tx.send(msg).await.unwrap(); // 这行可能不会执行！
            }
            _ = tokio::time::sleep(std::time::Duration::from_secs(5)) => {
                println!("timeout!");
                // 此时 rx.recv() 分支的 future 被取消
                // 但如果 write_to_db 已经执行了，result_tx.send 还没执行
                // → 数据被写入数据库但没有通知到位——不一致状态
            }
        }
    });

    // 如果超时发生在 write_to_db 之后、result_tx.send 之前
    // 数据就被写入了但没有发送确认——问题！
}
```

### 1.3 为什么有些 I/O 操作是取消安全的，有些不是

```rust
use tokio::io::{self, AsyncReadExt, AsyncWriteExt};

// 取消安全性的关键在于：操作是"原子的"还是"有状态的"

// ✅ 取消安全的操作：tokio::io::AsyncWriteExt::write_all()
// 原因：write_all 要么完成写入，要么不写入（从调用方视角看）
// 如果在 write_all 内部被取消，它不会让缓冲区处于"半写入"状态
// 调用方可以安全地重试整个写入
async fn safe_write(stream: &mut (impl AsyncWriteExt + Unpin), data: &[u8]) -> io::Result<()> {
    stream.write_all(data).await?; // ✅ 取消安全——要么写完，要么没写
    Ok(())
}

// ❌ 取消不安全的操作：tokio::io::AsyncReadExt::read_exact()
// read_exact(buf) 承诺"填满 buf"
// 如果 read_exact 读了一半被取消，buf 的前半部分已经被修改了
// 你不知道 buf 中哪些数据是"真正的新数据"——无法安全重试
async fn unsafe_read(stream: &mut (impl AsyncReadExt + Unpin)) -> io::Result<()> {
    let mut buf = vec![0u8; 1024];
    // ❌ 如果在 read_exact 中间被取消，buf 的前半部分已经变了
    // 你无法区分"读了一半"和"读了全部"——重试会丢失数据
    stream.read_exact(&mut buf).await?;
    Ok(())
}

// 同理：tokio::io::AsyncReadExt::read() 是取消安全的
// 因为它返回实际读取的字节数——你可以知道读了多少
// 而 read_exact 只告诉你"成功或失败"，不透露中间状态
```

取消安全性的判断公式很简单：**一个异步函数是取消安全的，当且仅当它在任何 `.await` 点被 drop 后，外部可观察状态与"该异步函数从未被调用过"完全一致**。`write_all` 满足这个条件（要么写完，要么写零字节——重试无害），`read_exact` 不满足（已经修改了缓冲区）。

### 1.4 如何写出取消安全的代码

```rust
// 原则 1：把"有副作用的 .await"和"非异步的关键区域"分离
// 先做非异步的准备工作，把不可取消的逻辑放前面

async fn cancel_safe_transfer(
    db: &Database,
    from: u64,
    to: u64,
    amount: i64,
) -> Result<(), DbError> {
    // 第一步：先在"非异步"的代码中验证+记录意图
    // （这里假设 validate 和 begin_transaction 是同步操作）
    db.validate_account(from)?;
    db.validate_account(to)?;
    let txn_id = db.begin_transaction()?;  // 同步——不可取消

    // 第二步：原子性地执行异步操作
    // 把整个"扣款+收款"作为一个原子数据库事务
    db.atomic_transfer(txn_id, from, to, amount).await?;
    // → 如果这里被取消，事务回滚，不会有部分修改残留

    Ok(())
}

// 原则 2：使用 tokio::pin! 确保 Future 在取消时被正确 drop
// pin! 不仅用于自引用——它让 Future 在 select! 中地址稳定

async fn pin_in_select() {
    let fut1 = async { /* ... */ };
    let fut2 = async { /* ... */ };

    // ✅ 用 tokio::pin! 确保 futures 在 select 期间地址稳定
    tokio::pin!(fut1, fut2);

    tokio::select! {
        _ = &mut fut1 => { println!("fut1 done"); }
        _ = &mut fut2 => { println!("fut2 done"); }
    }
    // 未被选中的 future 在这里被 drop——因为是 pin 的，drop 行为正确
}
```

### 1.5 实际分析——常见标准库/生态系统的取消安全状况

```rust
// Tokio 中取消安全的操作：
// ✅ TcpStream::connect()   — 要么连上，要么没连（不会处于"半连接"状态）
// ✅ TcpStream::write_all() — 要么写完，要么没写
// ✅ TcpStream::read()      — 返回读取的字节数，可以区分
// ✅ mpsc::Sender::send()   — 要么发送成功，要么数据还在你手里（返回 Err）
// ✅ mpsc::Receiver::recv() — 要么收到，要么还在等
// ✅ oneshot::Sender::send()— 要么发送，要么拿到 Err（拥有数据）

// ❌ 取消不安全的操作：
// ❌ TcpStream::read_exact() — 缓冲区被部分修改
// ❌ AsyncBufReadExt::read_until()/read_line() — 缓冲区被部分修改
// ❌ 任何"读取到固定大小的缓冲区"且不透露读取量的操作

// 通用规则：
// 如果异步函数的参数中有一个 &mut 引用（如 &mut [u8]），
// 且该引用指向的数据在函数执行完成后才"有意义"，
// 那么这个函数大概率是取消不安全的——因为取消时 &mut 引用已被部分修改。
```

## 2. select! 分支安全深度分析

### 2.1 tokio::select! vs futures::select!——关键差异

```rust
// tokio::select! 和 futures::select! 的主要区别：

// 1. 分支选择策略：
//    tokio::select!     → 偏向策略（biased）：按声明顺序检查分支
//    futures::select!   → 随机策略（random）：公平随机选择就绪分支
//
// tokio 的 biased 策略意味着：
// 如果两个分支同时就绪，第一个声明的分支胜出
// → 你可以通过"优先声明关键路径"来控制优先级
// → 但也要注意不要饿死后面的分支

// 2. tokio::select! 会在每次被 poll 时重新评估所有分支
//    futures::select! 在第一个分支完成后就会 fuse 整个 select

// 例子：tokio::select! 的偏向性
async fn tokio_select_biased() {
    let fast = async { "fast" };
    let slow = async {
        tokio::time::sleep(std::time::Duration::from_millis(10)).await;
        "slow"
    };

    // 在 tokio 中，如果两个同时就绪（fast 立即 Ready，slow 需要等 10ms）
    // fast 被声明在前面 → fast 会先被检查 → fast 获胜
    tokio::select! {
        result = fast => {
            println!("fast won: {}", result);
        }
        result = slow => {
            println!("slow won: {}", result);
        }
    }
}

// 相比之下，futures::select! 在同时就绪时会随机选择
// 这在测试中更难预测——但也防止了"偏向导致的饥饿 bug"
```

### 2.2 select! 的借用在循环中的陷阱

```rust
use tokio::sync::mpsc;

// ❌ 经典错误：在 loop 中使用 select!，分支中的引用跨循环存活
// async fn loop_select_bad() {
//     let (tx, mut rx) = mpsc::channel::<i32>(32);
//     let mut state = 0;
//
//     loop {
//         tokio::select! {
//             val = rx.recv() => {
//                 let val = val.unwrap();
//                 state += val;  // ❌ 编译错误：state 被 borrow了多次
//                 // 每次循环迭代，rx.recv() 持有 rx 的引用
//                 // 同时上一轮的 state 引用可能还未释放
//             }
//             _ = tokio::time::sleep(std::time::Duration::from_secs(1)) => {
//                 println!("state: {}", state);
//             }
//         }
//     }
// }

// ✅ 修复：把 select! 中的可变借用限制在每次迭代内
async fn loop_select_good() {
    let (tx, mut rx) = mpsc::channel::<i32>(32);
    let mut state = 0;

    loop {
        let val = tokio::select! {
            val = rx.recv() => {
                val.unwrap_or(0)
            }
            _ = tokio::time::sleep(std::time::Duration::from_secs(1)) => {
                println!("current state: {}", state);
                continue; // 跳过 state 更新
            }
        };
        // select! 结束后才修改 state——没有借用冲突
        state += val;
    }
}
```

### 2.3 select! 中的分支"消耗"问题

```rust
// tokio::select! 会"消耗"（move）分支中使用的变量
// 如果分支式引用了一个变量，那个变量的所有权在该分支被选中时就转移了

// 对于一些需要"重复使用"的资源（如 channel receiver），
// 变量需要在每次循环中重新借用

async fn select_consumption() {
    let (tx, mut rx) = mpsc::channel::<String>(32);

    // rx 在 select! 的 recv() 分支中以 &mut 形式被借用
    // 如果 rx 在另一个分支中也被使用（例如作为 &rx 被引用），就有冲突

    // ✅ 正确模式：用 &mut rx 在每个分支中显式重新借用
    tokio::select! {
        msg = rx.recv() => {
            println!("got: {:?}", msg);
        }
        _ = tokio::time::sleep(std::time::Duration::from_secs(1)) => {
            println!("timeout");
            // rx 没有被消耗——它只在 recv 分支中被使用
        }
    }

    // 如果你需要"引用检查"rx 是否关闭：
    // 使用 rx.is_closed() 或检查 recv() 返回 None
}
```

## 3. MutexGuard 跨越 .await——死锁高速公路

### 3.1 为什么 MutexGuard 不能跨越 .await

```rust
use std::sync::Arc;
use tokio::sync::Mutex;

// ❌ 最致命的异步陷阱：在持有锁的情况下 .await

async fn deadlock_waiting_to_happen() {
    let mutex = Arc::new(Mutex::new(0));
    let m1 = mutex.clone();
    let m2 = mutex.clone();

    // 任务 1：获取锁，在锁内做异步操作
    let task1 = tokio::spawn(async move {
        let mut guard = m1.lock().await;  // 获取锁
        *guard += 1;

        // ❌ 在持有锁时 .await——致命错误！
        // 当前任务在此暂停，锁仍然被持有
        tokio::time::sleep(std::time::Duration::from_secs(10)).await;

        // 在这 10 秒内，任何其他任务尝试获取这个锁都会阻塞
        // 如果任务 2 在当前线程（同一个 worker）上运行
        // 且当前线程是单线程运行时 → 死锁（任务 2 等任务 1 释放锁，但任务 1 在等 .await 完成）
        *guard += 1;
        // guard 在此自动释放
    });

    // 任务 2：尝试获取同一个锁
    let task2 = tokio::spawn(async move {
        let mut guard = m2.lock().await; // 等待任务 1 释放锁...
        *guard += 1;
    });

    // 在多线程运行时中，任务 2 可能在另一个线程上运行——不会死锁
    // 但在当前线程的运行时中（或者当所有 worker 都被锁住的协程占据时）
    // 这会导致性能问题甚至死锁
}
```

核心问题是：`tokio::sync::Mutex::lock().await` 是异步的——获取锁可能需要等待。如果你在持有锁的期间 `.await`，其他等待同一个锁的任务会被阻塞。在单线程运行时中，这会直接导致死锁（因为没有其他线程来驱动那个等待锁的任务）。在多线程运行时中，这可能导致严重的性能退化。

### 3.2 正确的模式——锁的作用域最小化

```rust
use std::sync::Arc;
use tokio::sync::Mutex;

// ✅ 正确：在 .await 之前释放锁

async fn proper_lock_usage() {
    let state = Arc::new(Mutex::new(Vec::<i32>::new()));
    let s = state.clone();

    tokio::spawn(async move {
        // 第一步：持有锁，做同步操作
        let data_to_process = {
            let mut guard = s.lock().await;
            guard.push(42);
            guard.clone() // 复制数据，然后释放锁
        }; // ← guard 在这里被 drop——锁释放！

        // 第二步：不持有锁，做异步操作
        let result = process_data(data_to_process).await;

        // 第三步：再次获取锁，写回结果
        let mut guard = s.lock().await;
        guard.clear();
        guard.extend(result);
    });
}

async fn process_data(data: Vec<i32>) -> Vec<i32> {
    // 模拟异步处理
    tokio::time::sleep(std::time::Duration::from_millis(100)).await;
    data.into_iter().map(|x| x * 2).collect()
}

// 遵循这个三步模式：
// 1. 获取锁 → 读取/复制需要的数据 → 释放锁
// 2. 在无锁状态下执行异步操作
// 3. 获取锁 → 写入结果 → 释放锁
// 每一步中锁的作用域都限制在同步代码块内
```

### 3.3 使用 std::sync::Mutex 在 async 中的陷阱

```rust
use std::sync::{Arc, Mutex as StdMutex};

// ❌ 用 std::sync::Mutex 替代 tokio::sync::Mutex 并不能解决问题
// 实际上更危险：

async fn blocking_mutex_danger() {
    let mutex = Arc::new(StdMutex::new(0));
    let m = mutex.clone();

    // ❌ std::sync::Mutex::lock() 是同步阻塞的！
    // 如果锁被另一个任务持有，当前线程会被阻塞
    // 在 tokio 的 worker 线程中阻塞 = 阻塞整个运行时
    tokio::spawn(async move {
        let guard = m.lock().unwrap();      // ← 如果等锁——阻塞当前 worker 线程！
        tokio::time::sleep(std::time::Duration::from_secs(10)).await;
        drop(guard);
    });

    // 即使锁没有争用，在持有 std::sync::Mutex 时跨越 .await 同样是危险的
}

// 规则总结：
// - tokio::sync::Mutex  → 适合需要跨越 .await 的场景（异步锁）
// - std::sync::Mutex    → 适合锁的持有时间很短（同步操作），
//                          且不需要跨越 .await 的场景
//                          
// 如果误用 std::sync::Mutex 跨越 .await：
//   → 锁被长时间持有，其他任务阻塞
//   → 阻塞的是 OS 线程（tokio worker 线程）——可能饿死运行时
```

## 4. Async Drop 问题——为什么还没有 async fn drop

### 4.1 问题的根源——Drop 必须是同步的

```rust
// Drop trait 的签名是：
// fn drop(&mut self)

// 它不是 async fn——这是 Rust 的设计决策，原因深远：
// 1. Drop 在多种上下文中被调用——panic 展开时、作用域结束时、显式 drop 时
// 2. 在 panic 展开时，你无法运行异步代码（没有 executor，没有 waker）
// 3. 类型系统需要保证 Drop 总能被调用——如果 Drop 是 async 的，
//    那么在"谁驱动 Drop 的 future"上就没有明确答案

// 这个限制导致了一个实际的问题：
// 很多异步资源需要异步清理

struct AsyncConnection {
    // ...
}

// impl Drop for AsyncConnection {
//     fn drop(&mut self) {
//         // ❌ 我想在这里 send 一个"关闭连接"的消息
//         // 但 send 是 async 的——不能在 Drop 中 await
//         // self.shutdown_tx.send(());  // 即使 send 返回了，future 也无法被 poll
//     }
// }

// 这就是 "async drop 问题"——异步资源无法在 Drop 中执行清理
// 目前 Rust 社区在讨论 AsyncDrop trait（RFC 中有相关讨论）
// 但短期内（2024-2026）不会有语言级别的解决方案
```

### 4.2 当前的解决方案 1——显式 shutdown() 方法

```rust
// 方案 A：要求调用方显式调用 shutdown()
// 这是最常用的模式——Tokio 自身也使用这个模式

struct MyService {
    shutdown_tx: tokio::sync::mpsc::Sender<()>,
    // ... 其他字段
}

impl MyService {
    async fn new() -> Self {
        // 初始化...
        let (tx, rx) = tokio::sync::mpsc::channel::<()>(1);
        // 启动后台清理任务...
        MyService { shutdown_tx: tx }
    }

    // 显式清理方法——调用者必须调用
    async fn shutdown(self) {
        // 发送关闭信号
        let _ = self.shutdown_tx.send(()).await;
        // 等待子任务完成...
        // 释放资源...
    }
}

// 使用模式：
async fn use_service() {
    let service = MyService::new().await;
    // ... 使用 service ...
    service.shutdown().await; // 必须显式调用
    // 如果忘记调用 shutdown()，资源泄漏——但不安全（在 Rust 中只是"泄漏"）
}
```

### 4.3 当前的解决方案 2——CancellationToken（结构化取消）

```rust
use tokio_util::sync::CancellationToken;

// 方案 B：使用 CancellationToken 实现优雅关闭
// CancellationToken 是 tokio_util crate 提供的一个同步原语
// 它允许你发出"取消"信号，所有持有 clone 的组件都能收到通知

struct GracefulService {
    cancel: CancellationToken,
    join_handle: tokio::task::JoinHandle<()>,
}

impl GracefulService {
    fn new() -> Self {
        let cancel = CancellationToken::new();
        let child_cancel = cancel.clone();

        let join_handle = tokio::spawn(async move {
            loop {
                tokio::select! {
                    _ = child_cancel.cancelled() => {
                        // 收到取消信号——执行清理
                        println!("cleanup: closing connections, flushing buffers...");
                        break;
                    }
                    _ = do_work() => {
                        // 正常业务逻辑
                    }
                }
            }
        });

        GracefulService { cancel, join_handle }
    }

    async fn shutdown(self) {
        // 发出取消信号
        self.cancel.cancel();
        // 等待子任务结束
        let _ = self.join_handle.await;
        // 此时子任务已完成清理
    }
}

// CancellationToken 的优势：
// 1. 一次信号通知所有 clone——适合多个子任务同时取消
// 2. 非所有权模型——不需要 move 所有权
// 3. 可以组合使用（tree 模式：父 token 取消 → 所有子 token 被取消）
```

## 5. CancellationToken——结构化并发与优雅取消

### 5.1 CancellationToken 的基本用法

```rust
use tokio_util::sync::CancellationToken;

// CancellationToken 的核心 API：
// - new()         → 创建新的 token
// - clone()       → 创建一个新的引用（浅克隆，共享同一个取消状态）
// - cancel()      → 发出取消信号
// - is_cancelled()→ 检查是否已被取消（同步，不阻塞）
// - cancelled()   → 返回一个 Future，token 被取消时 resolve

async fn cancellation_basic() {
    let token = CancellationToken::new();
    let child_token = token.clone();

    // 后台任务监听取消信号
    let handle = tokio::spawn(async move {
        tokio::select! {
            _ = child_token.cancelled() => {
                println!("task was cancelled, cleaning up...");
                // 执行清理逻辑
            }
            _ = long_running_work() => {
                println!("work completed naturally");
            }
        }
    });

    // 在超时或条件满足时取消
    tokio::time::sleep(std::time::Duration::from_secs(5)).await;
    token.cancel();

    let _ = handle.await;
}

async fn long_running_work() {
    // 模拟长时间工作
    tokio::time::sleep(std::time::Duration::from_secs(100)).await;
}
```

### 5.2 CancellationToken 的树形结构——父取消传播到子

```rust
use tokio_util::sync::CancellationToken;

// CancellationToken 可以通过 child_token() 创建父子关系
// 父 token 被取消时，所有子 token 自动被取消

async fn cancellation_tree() {
    let parent = CancellationToken::new();

    // 创建子 token——当 parent 被取消时，子 token 自动取消
    let child1 = parent.child_token();
    let child2 = parent.child_token();

    let h1 = tokio::spawn(async move {
        child1.cancelled().await;
        println!("child1: cleaning up...");
    });

    let h2 = tokio::spawn(async move {
        child2.cancelled().await;
        println!("child2: cleaning up...");
    });

    // 取消父 token——所有子 token 自动被取消
    parent.cancel();

    let _ = tokio::join!(h1, h2);
    // 输出：
    // child1: cleaning up...
    // child2: cleaning up...

    // 注意：子 token 只能被父 token 取消，不能被"独立取消"
    // child1.cancel() 只影响 child1，不影响 parent
    // parent.cancel() 会影响所有 child_token
}

// 树形取消的实际场景：
// - 一个 HTTP 请求 → 多个数据库查询 + 多个微服务调用
// - 当请求被取消（客户端断开连接），所有下游操作都被取消
// - 用一棵 CancellationToken 树来表达这个关系
```

## 6. 背压（Backpressure）策略——生产消费不平衡的处理

### 6.1 背压的本质问题

```rust
// 在异步系统中，当生产者比消费者快时，需要"背压"来减速生产者
// 背压的核心问题：如何在"不丢数据"和"不 OOM"之间取得平衡

// 方案 1：有界 Channel（Bounded Channel）——最直接的背压
use tokio::sync::mpsc;

async fn bounded_channel_backpressure() {
    // capacity=16 意味着缓冲区最多装 16 个元素
    // 当缓冲区满时，send().await 会阻塞生产者（背压生效）
    let (tx, mut rx) = mpsc::channel::<i32>(16);

    // 生产者
    let producer = tokio::spawn(async move {
        for i in 0..1000 {
            // 如果消费者处理慢，这里会等待（背压）
            if tx.send(i).await.is_err() {
                break; // channel 关闭——停止生产
            }
        }
    });

    // 消费者——模拟慢处理
    let consumer = tokio::spawn(async move {
        while let Some(val) = rx.recv().await {
            // 模拟慢处理——每秒 10 个
            tokio::time::sleep(std::time::Duration::from_millis(100)).await;
            println!("processed: {}", val);
        }
    });

    let _ = tokio::join!(producer, consumer);
}
```

### 6.2 多种背压策略对比

```rust
// 策略 A：Bounded Channel —— "生产者阻塞等待"
// tokio::sync::mpsc::channel(capacity)
// 优点：最简单，不会丢数据
// 缺点：如果消费者太慢，生产者被阻塞——上游压力传递

// 策略 B：丢弃 (Drop) —— "满了就扔"
// 使用 mpsc::channel(capacity) + try_send()
// 如果缓冲区满了，直接丢弃最新的数据
// 适合：日志系统、指标收集（丢几条无所谓）

// 策略 C：覆盖 (Overwrite) —— "只要最新的"
// 使用 tokio::sync::watch
// 生产者只更新最新的值，消费者总是拿到最新的
// 适合：配置更新、状态通知

// 策略 D：采样 (Sampling) —— "随机丢弃"
// 生产者检查缓冲区使用率，按比例丢弃
// 适合：流量过载时的优雅降级

// 策略 E：优先级队列 —— "重要的先处理"
// 生产者根据优先级选择是否放入
// 高优先级的永远不会被丢弃
// 低优先级的在高水位时被丢弃

// 选择策略的关键问题：
// 1. 可以丢数据吗？（可以 → B/C/D，不可以 → A）
// 2. 需要最新数据吗？（需要 → C，不需要 → A/B）
// 3. 有优先级区分吗？（有 → E，没有 → A/B/C/D）
```

## 7. 缓冲策略——buffer_unordered vs buffer_ordered vs 手动

### 7.1 buffer_unordered——无序并发，最大化吞吐

```rust
use futures::stream::{self, StreamExt};

// buffer_unordered(n) 的核心语义：
// - 最多 n 个异步操作同时进行
// - 哪个先完成就先产出哪个
// - 排序后的结果是无序的

async fn buffer_unordered_example() {
    let items = stream::iter(0..20);

    let results: Vec<i32> = items
        .map(|i| async move {
            // 模拟不同耗时的操作
            let delay = (20 - i) as u64 * 10; // 越小越慢（故意的）
            tokio::time::sleep(std::time::Duration::from_millis(delay)).await;
            println!("item {} completed", i);
            i
        })
        .buffer_unordered(5) // 最多 5 个并发
        .collect()
        .await;

    // 结果顺序不可预测——取决于每个操作的完成时间
    println!("collection order: {:?}", results);
}

// 选择 buffer_unordered 的场景：
// - 每个操作的耗时高度可变
// - 不关心结果的顺序
// - 希望最大化吞吐量（尽快拿到可用的结果）
```

### 7.2 buffer_ordered——有序并发，保证输出顺序

```rust
use futures::stream::{self, StreamExt};

// buffer_ordered(n) 保持和 buffer_unordered(n) 相同的并发模型
// 但确保输出顺序和输入顺序一致

async fn buffer_ordered_example() {
    let items = stream::iter(0..20);

    let results: Vec<i32> = items
        .map(|i| async move {
            let delay = (20 - i) as u64 * 10;
            tokio::time::sleep(std::time::Duration::from_millis(delay)).await;
            i
        })
        .buffer_ordered(5)
        .collect()
        .await;

    // 输出始终是 [0, 1, 2, ..., 19]——严格有序
    println!("results are in order: {:?}", results);

    // 代价：内部可能需要缓冲已完成的"超前"结果
    // 如果第 5 个任务先完成了，它必须等待第 1-4 个任务完成后才能产出
    // → 额外的内存使用（缓冲 5-1=4 个结果）
}

// 选择 buffer_ordered 的场景：
// - 需要并发（提高吞吐）但结果必须有序
// - 下游消费者依赖顺序（如按序列号处理事件）
```

### 7.3 选择合适的并发限制

```rust
// buffer_unordered(n) 中 n 的选择是一个重要的工程决策：

// n 太小 → 并发度不够，吞吐量低
// n 太大 → 资源耗尽（连接数、内存、文件描述符）

// 经验法则：
// n 的值应该等于你的"瓶颈资源能同时支持的操作数"

// 示例场景：
// - HTTP 请求（瓶颈 = 网络连接池）：n = 连接池大小（如 50-200）
// - 数据库查询（瓶颈 = 连接池）：n = 连接池大小（如 20-100）
// - CPU 密集计算（瓶颈 = CPU 核心）：n = CPU 核心数或略多
// - 文件 I/O（瓶颈 = 磁盘带宽）：n = 通常 32-128

// 在生产中，n 通常需要经过压测来确定最优值
// 建议从较小值开始（如 10），逐步增加直到 QPS 不再提高或延迟开始恶化
```

## 8. 常见错误清单——速查表

### 8.1 错误 1：忘记 Pin 当需要时

```rust
// ❌ 当你需要在 select! 或某些异步操作中引用一个 Future 时
// 忘记 pin 会导致 Future 的地址不稳定

// async fn forget_to_pin() {
//     let fut = async { 42 };
//     
//     tokio::select! {
//         val = fut => {  // ❌ fut 不是 pinned！可能无效
//             println!("{}", val);
//         }
//         _ = tokio::time::sleep(Duration::from_secs(1)) => {}
//     }
// }

// ✅ 修正：
async fn remember_to_pin() {
    let fut = async { 42 };
    tokio::pin!(fut);  // 关键：pin

    tokio::select! {
        val = &mut fut => {  // ✅ fut 是 pinned 的
            println!("{}", val);
        }
        _ = tokio::time::sleep(std::time::Duration::from_secs(1)) => {}
    }
}
```

### 8.2 错误 2：在 async 中使用 std::sync::Mutex 跨越 .await

这个陷阱已在第 3 节详细分析。简单规则：**async 代码中用 tokio::sync::Mutex，且不要跨越 .await 持锁**。

### 8.3 错误 3：在 async 上下文中运行 CPU 密集型任务

```rust
// ❌ CPU 密集型任务在 async 上下文中会阻塞整个线程
async fn cpu_heavy_bad() {
    // 这个大矩阵乘法可能耗时几百毫秒
    let result: Vec<Vec<f64>> = (0..1000)
        .map(|i| {
            (0..1000)
                .map(|j| (i as f64).powf(j as f64))
                .collect()
        })
        .collect();
    // 在这几百毫秒内，当前 tokio worker 线程完全不处理其他任务
    // → 所有其他等待在这个 worker 上的 task 被饿死
    println!("done");
}

// ✅ 修复：使用 spawn_blocking 将 CPU 任务移到阻塞线程池
async fn cpu_heavy_good() {
    let result = tokio::task::spawn_blocking(move || {
        // 这段代码在专门的阻塞线程池中运行——不影响 async worker
        (0..1000)
            .map(|i| (0..1000).map(|j| (i as f64).powf(j as f64)).collect::<Vec<_>>())
            .collect::<Vec<_>>()
    })
    .await
    .unwrap();

    println!("computation done");
}

// 判断是否需要 spawn_blocking 的简单规则：
// - 操作耗时 > 1ms → 考虑 spawn_blocking
// - 操作耗时 > 10ms → 必须 spawn_blocking
// - 同步 I/O（如 std::fs::read）→ 必须 spawn_blocking
```

### 8.4 错误 4：不处理 tokio::spawn 的 JoinHandle

```rust
// ❌ 忽略 JoinHandle——错误被静默吞掉
async fn ignore_join_error() {
    tokio::spawn(async {
        panic!("oops!");
        // panic 在 spawn 的 task 中发生——但如果没有人 join
        // 这个错误就静默丢失了（在 tokio 的默认配置下 panic 被捕获）
    });

    tokio::time::sleep(std::time::Duration::from_secs(1)).await;
    // 程序继续运行——你完全不知道 panic 发生了
}

// ✅ 处理 JoinHandle——对关键任务显式 join 或 select
async fn handle_join_result() {
    let handle = tokio::spawn(async {
        do_critical_work().await;
    });

    // 方式 1：等待并检查结果
    match handle.await {
        Ok(result) => println!("task succeeded"),
        Err(e) => {
            // JoinError——任务被取消了或发生了 panic
            if e.is_panic() {
                eprintln!("task panicked! attempting recovery...");
            } else if e.is_cancelled() {
                eprintln!("task was cancelled");
            }
        }
    }
}
```

### 8.5 错误 5：tokio 运行时中的"活锁"——CPU-bound future 饿死 I/O

```rust
// ❌ 在 async 函数中做密集循环——即使没有"阻塞"，也可能饿死运行时
async fn tight_loop_bad() {
    loop {
        // 这个循环中没有 .await，当前 task 不会 yield 控制权
        // tokio 的协作式调度依赖于 task 在 .await 时主动让出
        // 如果没有 .await，当前 task 会独占 worker 线程
        calculate_next_value(); // 这个函数可能很快——但 10000 次循环就不一定了
    }
}

// ✅ 在长循环中显式 yield 控制权
async fn tight_loop_good() {
    for i in 0..10000 {
        calculate_next_value();
        if i % 100 == 0 {
            // 每 100 次迭代主动 yield，允许其他任务运行
            tokio::task::yield_now().await;
        }
    }
}

// tokio::task::yield_now() 的行为：
// - 将当前 task 放回调度队列的末尾
// - 允许其他 task 在当前 worker 上运行
// - 不保证其他 task 一定会运行（只是 yield，不是 block）
```

## 9. 最佳实践总结——Async Rust 生存指南

### 9.1 结构设计最佳实践

```
Async Rust 结构设计清单：

□ 每个异步组件都有明确的"启动"（spawn/init）和"关闭"（shutdown）路径
□ 使用 CancellationToken 实现优雅关闭——不要在 Drop 中做异步清理
□ 把共享状态放在 Arc<tokio::sync::Mutex<T>> 中（而非 std::sync::Mutex）
□ 最小化锁的作用域——不在持有锁时 .await
□ 用 mpsc::channel 做组件间通信——不要共享可变状态除非必要
□ 为每个 spawn 的 task 保存 JoinHandle，在 shutdown 时等待或取消它们
```

### 9.2 错误处理最佳实践

```
Async 错误处理清单：

□ 每个 spawn 的 task 的 JoinHandle 要么被 join 要么被 abort——不要"放飞"
□ select! 中每个分支都考虑"如果另一个分支赢了，我正在做的事会怎样"（取消安全）
□ 用 try_next() / try_collect() 处理 Result stream——在第一个错误处短路
□ spawn_blocking 的 JoinError 需要处理——可能 panic
□ 在 select! 中使用 biased 策略（tokio 默认）时，确保关键分支声明在前
```

### 9.3 性能最佳实践

```
Async 性能清单：

□ CPU 密集型操作 → spawn_blocking——永远不要在 async worker 上做重计算
□ 合理设置 buffer_unordered 的并发限制——不要"无限并发"
□ 在长循环中定期 yield_now——防止饿死其他 task
□ 选择合适的 channel 容量——太小→背压过度，太大→内存浪费
□ std::sync::Mutex 只在锁持有时间极短（<1μs）且不跨越 .await 时使用
□ 多读少写场景用 tokio::sync::RwLock 代替 Mutex
□ 使用 tokio::sync::Semaphore 限制并发资源使用（如数据库连接）
```

```
数据一致性：

  □ 有没有 CPU 密集操作在 async worker 上运行？（→ 用 spawn_blocking）

  □ 每个 spawn 的 task 是否有清理路径？（CancellationToken 或 shutdown）

错误处理：
  □ spawn 的 JoinHandle 是否被正确处理？（join 或 abort）

  □ Stream 的 try_next / try_collect 的错误是否被传播？
```

### 练习 1 答案

**问题**：如果在 `from.save_to_db().await` 之后、`to.save_to_db().await` 之前被取消，`from` 已经扣款，但 `to` 没有入账——钱凭空消失了。

**修复方案 A**：把两个写操作放在同一个非取消区间内

```rust
async fn transfer_money(
    from: &mut Account,
    to: &mut Account,
    amount: u64,
) -> Result<(), Error> {
    // 先用一个 tokio::task::spawn 包装——spawn 的 task 不会被 select! 取消
    let handle = tokio::spawn(async move {
        from.balance -= amount;
        from.save_to_db().await?;
        to.balance += amount;
        to.save_to_db().await?;
        Ok::<_, Error>(())
    });

    handle.await??;  // 等待完成或传播错误
    Ok(())
}
```

**修复方案 B**：先做所有校验和准备工作，再原子性地执行

```rust
async fn transfer_money(
    from: &mut Account,
    to: &mut Account,
    amount: u64,
) -> Result<(), Error> {
    // 校验：检查余额（不涉及 I/O，不会在这里被取消）
    if from.balance < amount {
        return Err(Error::InsufficientFunds);
    }

    // 准备数据：构造两个 update 语句
    // 使用数据库事务保证原子性
    let mut tx = db.begin().await?;

    // 在事务中执行——要么全部提交，要么全部回滚
    db.execute("UPDATE accounts SET balance = balance - ? WHERE id = ?", &[amount, from.id]).await?;
    db.execute("UPDATE accounts SET balance = balance + ? WHERE id = ?", &[amount, to.id]).await?;

    tx.commit().await?;  // 如果在这里被取消，tx 会被 drop → 自动回滚
    Ok(())
}
```

### 练习 2 答案

**问题**：`reader.read(&mut buf)` 在 select! 的两个分支中都借用了 `buf`。如果第一个分支被取消（timeout 分支胜出），`buf` 的 `&mut` 借用可能没有被释放——实际上 tokio 的 select! 会正确释放，但编译器保守地拒绝。另外，`buf` 在 loop 中需要在多次迭代间保持状态，但 select! 会"消耗"分支中的 future。

**修复**：

```rust
async fn handle_connection(stream: TcpStream) {
    let mut buf = vec![0u8; 1024];
    let mut reader = BufReader::new(stream);

    loop {
        // 修复方案：把 read future 先 pin 住，避免 select! 的借用冲突
        let read_future = reader.read(&mut buf);
        tokio::pin!(read_future);

        tokio::select! {
            result = &mut read_future => {
                let n = result.unwrap();
                if n == 0 { break; }
                println!("read {} bytes", n);
            }
            _ = tokio::time::sleep(Duration::from_secs(30)) => {
                println!("timeout——closing connection");
                break;
            }
        }
    }
}
```

关键：用 `tokio::pin!` 和 `&mut read_future` 让 select! 借用 future 而不是消耗它。

### 练习 3 答案

```rust
use tokio::io::{self, AsyncWrite, AsyncWriteExt};
use std::pin::Pin;
use std::task::{Context, Poll};

/// 一个取消安全的缓冲写入器
///
/// 核心思路：跟踪"已成功写入的字节数"
/// 即使 write_all 在中间被取消，下次调用时从上次的位置继续
pub struct CancelSafeWriter<W: AsyncWrite + Unpin> {
    inner: W,
    buf: Vec<u8>,
    written: usize,  // 已从 buf 成功写入到 inner 的字节数
}

impl<W: AsyncWrite + Unpin> CancelSafeWriter<W> {
    pub fn new(inner: W) -> Self {
        CancelSafeWriter {
            inner,
            buf: Vec::new(),
            written: 0,
        }
    }

    /// 将数据添加到缓冲区并尝试刷新
    pub async fn write_all(&mut self, data: &[u8]) -> io::Result<()> {
        // 把新数据追加到缓冲区（从上次中断的位置开始）
        self.buf.extend_from_slice(data);
        self.flush().await
    }

    /// 将缓冲区内容写入底层 writer
    pub async fn flush(&mut self) -> io::Result<()> {
        while self.written < self.buf.len() {
            // 只写入"还没写入"的部分
            let n = self.inner.write(&self.buf[self.written..]).await?;
            self.written += n;  // ✅ 即使这里之后被取消，written 已更新
        }
        // 全部写入完成，清空缓冲区
        self.buf.clear();
        self.written = 0;
        self.inner.flush().await?;
        Ok(())
    }
}

// 为什么这是取消安全的：
// 假设 write_all(b"hello world") 被调用
// 1. b"hello world" 被追加到 self.buf
// 2. flush() 开始写入
// 3. inner.write() 成功写入 5 字节 → self.written = 5
// 4. 此时被取消 → self.written 仍然是 5
// 5. 下次调用 write_all(b"!!!") 时：
//    - b"!!!" 追加到 self.buf
//    - flush() 从 self.written=5 开始 → 写入 " world!!!"
//    - 不会重复写入已完成的 5 字节 ✅
```

### 练习 4 答案

```rust
use tokio::net::TcpListener;
use tokio::signal;
use tokio_util::sync::CancellationToken;
use std::time::Duration;

async fn graceful_shutdown_server() -> Result<(), Box<dyn std::error::Error>> {
    let listener = TcpListener::bind("127.0.0.1:8080").await?;
    let token = CancellationToken::new();
    let active_connections = tokio::sync::Semaphore::new(1000); // 最多 1000 并发

    // 启动关闭信号监听
    let shutdown_token = token.clone();
    tokio::spawn(async move {
        // 等待 SIGTERM 或 Ctrl+C
        signal::ctrl_c().await.ok();
        eprintln!("Shutdown signal received. Draining connections...");
        shutdown_token.cancel();  // 通知所有 worker 停止
    });

    // 接受连接的主循环
    loop {
        let conn_token = token.clone();
        let permit = active_connections.clone().acquire_owned().await?;

        tokio::select! {
            result = listener.accept() => {
                let (stream, addr) = result?;
                tokio::spawn(async move {
                    let _permit = permit;  // 持有许可直到请求完成
                    handle_request(stream, conn_token).await;
                });
            }
            _ = token.cancelled() => {
                // 不再接受新连接
                eprintln!("No longer accepting new connections.");
                break;
            }
        }
    }

    // 等待所有进行中的请求完成（最多 30 秒）
    eprintln!("Waiting for {} active requests to complete...",
             1000 - active_connections.available_permits());

    match tokio::time::timeout(
        Duration::from_secs(30),
        active_connections.acquire_many(1000), // 等所有许可被归还
    ).await {
        Ok(_) => eprintln!("All requests completed. Shutting down."),
        Err(_) => eprintln!("Timeout! Force-closing {} requests.",
                            1000 - active_connections.available_permits()),
    }

    Ok(())
}

async fn handle_request(
    mut stream: tokio::net::TcpStream,
    token: CancellationToken,
) {
    tokio::select! {
        result = process(&mut stream) => {
            // 正常完成
            let _ = result;
        }
        _ = token.cancelled() => {
            // 收到关闭信号——尽快完成当前工作
            eprintln!("Request cancelled by shutdown signal");
        }
    }
}

async fn process(_stream: &mut tokio::net::TcpStream) -> Result<(), Box<dyn std::error::Error>> {
    // 模拟处理
    tokio::time::sleep(Duration::from_secs(2)).await;
    Ok(())
}
```

**关键设计要点**：
1. `CancellationToken` 作为全局"停止"信号
2. `Semaphore` 追踪活跃连接数
3. `select!` 在新连接和关闭信号之间选择
4. 每个请求也监听 `token.cancelled()` 以便快速终止
5. 超时机制防止无限等待
