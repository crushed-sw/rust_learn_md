# 3.0 并发与异步 —— 从"单线程思维"到"多任务世界"

> 如果你和我一样，之前写的 Rust 代码都是单线程的——一个 `fn main()`，从上到下顺序执行，最多用用迭代器和闭包——那么"并发"和"异步"这两个词可能让你感到不安。**这很正常。** 并发编程在任何语言中都是一个"台阶"，在 Rust 中尤其如此，因为 Rust 不仅要保证正确性，还要在编译期就保证并发安全。但这也正是 Rust 的优势：**一旦你的并发代码通过编译，它就已经避开了绝大多数其他语言中常见的并发 bug。**
>
> 这一章的目标不是让你马上写出一个 tokio 服务，而是帮你建立一个**心理模型**——当你在代码中看到 `Arc<Mutex<T>>`、`async fn`、`tokio::spawn` 时，你的大脑里应该浮现出什么样的画面。

---

## 1. 先搞清楚三个概念：并发、并行、异步

很多人混淆这三个词。先举一个生活中的例子：

```
你在厨房做饭：

并发 (Concurrency)： 你一个人，同时管三个锅——炒菜、煮汤、烤面包。
                     你在三个任务之间**切换**，但任一瞬间你只在做一件事。
                     关键：多个任务在**时间上重叠**，但不一定同时执行。

并行 (Parallelism)： 你和朋友一起做饭——你炒菜、他煮汤、另一个人烤面包。
                     三个任务**真正同时**进行。
                     关键：需要**多个执行单元**（多个人/多个 CPU 核心）。

异步 (Asynchrony)：  你把面包放进烤面包机，设好定时器，然后**不等它完成**就去炒菜。
                     等面包机"叮"一声，你**被通知**回来看面包。
                     关键：发起操作和获取结果在时间上是**分离**的。
```

在编程中：

| 概念 | 代码层面 | 典型 Rust 机制 |
|------|---------|---------------|
| 并发 | 多个任务交替执行 | `std::thread::spawn`、`tokio::spawn` |
| 并行 | 多个 CPU 核心同时执行 | 同上（在多核上天然并行） |
| 异步 | 发起 I/O 后不阻塞，等通知 | `async fn`、`Future`、`.await` |

**重要：并发 ≠ 并行。** 你可以在单核 CPU 上写出高度并发的程序（比如同时处理 10000 个网络连接），这是靠任务切换实现的。异步是一种**实现高效率并发**的手段——当一个任务在等 I/O 时，CPU 去处理另一个任务，而不是傻等。

## 2. 为什么并发编程这么难？——三个经典问题

在进入 Rust 的解决方案之前，先理解问题本身。

### 2.1 数据竞争 (Data Race)

两个线程同时访问同一块内存，至少一个是写操作，且没有同步机制。

```rust
// 想象这是 C 语言的伪代码（Rust 会阻止你写这种代码）：
// 线程 1: counter = counter + 1;
// 线程 2: counter = counter + 1;
// 
// counter 初始值为 0，我们期望结果是 2。
// 
// 但"counter = counter + 1"不是原子操作，它分为三步：
//   1. 从内存读 counter 到寄存器  →  线程 1 读到 0
//   2. 寄存器加 1                  →  线程 2 读到 0（还没等线程 1 写回）
//   3. 寄存器写回内存               →  两个线程都写回 1
// 
// 结果：counter = 1，不是 2。这就是数据竞争。
```

**数据竞争的可怕之处**：它不一定每次都出现。你的测试可能跑 1000 次都正确，第 1001 次才触发。这是并发 bug 最难调试的原因。

### 2.2 死锁 (Deadlock)

两个线程各自持有一个锁，同时等待对方释放另一个锁。

```
线程 A：持有锁 1，等待锁 2
线程 B：持有锁 2，等待锁 1

→ 两个线程永远等下去，程序卡死。
```

### 2.3 并发 ≠ 并行带来的心智负担

即使没有数据竞争和死锁，并发代码也天然比顺序代码更难理解——你必须考虑**所有可能的交错执行顺序**。这就像下棋时你要预判对手的所有可能走法，但"对手"是操作系统的调度器。

## 3. Rust 的并发哲学：编译期保证，而非运行时祈祷

### 3.1 从"约定"到"类型检查"

其他语言的并发安全通常依赖**约定和代码审查**：
- "这个函数是线程安全的吗？——看文档。"
- "这个变量可以在多线程中共享吗？——团队约定不能。"
- "这里加锁了吗？——Code review 时记得检查。"

Rust 的做法：**把这些约定写成类型系统的规则**。

```
C++/Java/Go:  并发安全 = 人的纪律 + 运行时检测工具的辅助
Rust:         并发安全 = 类型系统 + 借用检查器 —— 编译不过就是有 bug
```

这不是夸大。Rust 的 `Send` 和 `Sync` 两个 trait 就是为并发安全设计的类型级"标签"。编译器会自动检查：
- 你在新线程中使用的所有数据都必须实现 `Send`（所有权可以安全转移）
- 你在多线程中共享引用的所有数据都必须实现 `Sync`（引用可以安全共享）

**如果你试图把非线程安全的类型传到另一个线程，编译器会直接拒绝编译——不是运行时 panic，是编译不过。**

### 3.2 一个让你感受"编译器救我"的例子

```rust
use std::rc::Rc;
use std::thread;

fn main() {
    let data = Rc::new(42);
    
    thread::spawn(move || {
        println!("{}", data);  // ❌ 编译错误！
    }).join().unwrap();
}
```

编译错误：
```
error[E0277]: `Rc<i32>` cannot be sent between threads safely
   --> src/main.rs:6:5
    |
6   |     thread::spawn(move || {
    |     ------------- ^------ `Rc<i32>` cannot be sent between threads safely
    |     |
    |     required by a bound introduced by this call
    |
    = help: the trait `Send` is not implemented for `Rc<i32>`
    = note: use `Arc<i32>` instead, which is `Send`
```

**注意到编译器的建议了吗？** `use Arc<i32> instead`。Rust 编译器不仅告诉你"你错了"，还告诉你怎么改。

```rust
use std::sync::Arc;
use std::thread;

fn main() {
    let data = Arc::new(42);
    let data_clone = Arc::clone(&data);
    
    thread::spawn(move || {
        println!("{}", data_clone);  // ✅ 编译通过
    }).join().unwrap();
    
    println!("{}", data);  // ✅ 主线程也能继续用
}
```

### 3.3 Rust 并发安全的两层防护

```
第一层：编译期类型检查
  ├── Send trait：判断类型"所有权"能否跨线程转移
  ├── Sync trait：判断类型"引用"能否跨线程共享
  └── 借用检查器：保证没有同时存在的 &mut（即使在多线程中）

第二层：运行时原语
  ├── Mutex / RwLock：提供内部可变性 + 互斥访问
  ├── Atomic：无锁的原子操作
  └── Channel：所有权转移替代共享内存
```

两层防护的关系是：**第一层是"能不能"，第二层是"怎么做"。** 编译期检查保证你不会犯"把 Rc 发给另一个线程"这种错误；运行时原语是你"正确地做并发"的工具。

## 4. Rust 的两种并发模型：线程同步 vs 异步

Phase 3 的内容可以分为两条主线：

### 4.1 主线一：线程同步并发（3.1 - 3.9）

这是最传统的并发模型，也是理解异步的基础。

```
操作系统线程
    │
    ├── 锁与同步原语（Mutex、RwLock、Condvar、Barrier）
    │   └── 解决：多个线程如何协调访问共享数据
    │
    ├── 原子操作与内存排序（Atomic、Ordering）
    │   └── 解决：如何不用锁也能安全共享数据
    │
    ├── 消息传递（mpsc、oneshot、broadcast、watch）
    │   └── 解决："不要共享内存，用通信来传递数据"
    │
    ├── 数据并行（Rayon）
    │   └── 解决："我有一个大数组要处理，怎么自动拆到多核上？"
    │
    └── 线程管理（Scoped Threads、TLS、Once）
        └── 解决：如何高效安全地管理线程生命周期
```

**核心心智模型**：线程 = 独立的栈 + 共享的堆。你要解决的是"多个执行流如何不打架地访问同一块 heap 数据"。

### 4.2 主线二：异步并发（3.10 - 3.14）

异步是 Rust 并发模型的"上层建筑"——它建立在 Future trait 和 async/await 语法之上。

```
async/await 语法糖
    │
    └── Future trait（核心抽象）
        │
        ├── 状态机：每个 .await 点生成一个状态
        │   └── 为什么 async fn 返回值是不透明类型？
        │
        ├── Waker 机制：通知运行时"我准备好了"
        │   └── 为什么不需要回调函数？
        │
        ├── Pin：自引用结构的内存安全保证
        │   └── 为什么 async 块需要 Pin？
        │
        └── 运行时（Tokio）
            ├── 调度器：Work-stealing 多线程调度
            ├── I/O 驱动：epoll / kqueue / IOCP
            └── 定时器：tokio::time
```

**核心心智模型**：Future = 一个"还没完成的计算"。你需要一个**运行时**来驱动这些 Future 完成。这和我们习惯的"函数调用-返回"模型有本质不同。

## 5. 同步和异步，什么时候用哪个？

这是新手最常问的问题。下面是一个决策树：

```
你的任务主要在做什么？
    │
    ├── CPU 密集型（大量计算、图像处理、科学计算）
    │   └── 用同步线程（std::thread 或 Rayon）
    │       原因：CPU 不会"等"，线程切换反而浪费。用多核并行。
    │
    ├── I/O 密集型（网络请求、文件读写、数据库查询）
    │   └── 用异步（tokio + async/await）
    │       原因：大部分时间在等 I/O，异步可以在等待时处理其他请求。
    │       一个线程就能处理数万个并发连接。
    │
    └── 混合型（既要算又要等）
        └── 异步 + spawn_blocking
            CPU 密集部分放到 spawn_blocking 的专用线程池
```

**一个直观的数字**：假设每个请求的处理时间是 10ms（其中 9ms 等数据库，1ms 做计算）。

| 模型 | 需要的线程数（1000 并发请求） | 内存开销 |
|------|---------------------------|---------|
| 同步线程（每请求一线程） | 1000 个 OS 线程 | ~8MB × 1000 = 8GB |
| 异步（tokio，8 worker 线程） | 8 个 OS 线程 | ~8MB × 8 = 64MB |

差距是两个数量级。这就是为什么 Web 服务器都选异步。

## 6. 你需要转变的思维模式

### 6.1 从"函数调用"到"任务提交"

```rust
// 同步思维：我调用一个函数，它执行完返回结果
let result = compute_something(input);
println!("{}", result);

// 异步思维：我把任务"提交"给运行时，得到 Future，直到 .await 才真正等结果
let future = compute_something(input);  // 还没执行，只是一个"计划"
// ... 可以做别的事 ...
let result = future.await;  // 现在才真正等它完成
```

### 6.2 从"共享数据"到"传递消息"

Go 语言的名言同样适用于 Rust：

> "Don't communicate by sharing memory; share memory by communicating."

```rust
// 共享内存思维（需要 Mutex 保护）：
let data = Arc::new(Mutex::new(Vec::new()));
// ... 多个线程 lock().push() ...

// 消息传递思维（所有权转移，无需锁）：
let (tx, rx) = mpsc::channel();
// 线程 A 通过 tx.send() 发送数据
// 线程 B 通过 rx.recv() 接收数据
// 数据的所有权随消息转移，没有共享，没有锁
```

### 6.3 从"这个函数返回什么"到"这个 Future 的 poll 会被调用几次"

当你深入 Future 的内部机制时，你会发现异步编程的本质是**事件驱动**——运行时不是"执行你的函数"，而是"反复询问你的 Future 是否完成了"。这种思维转变是异步编程的核心挑战，也是 Phase 3 后半部分要帮你建立的。

## 8. 学习建议

1. **3.1-3.7 是基础中的基础。** 即使你最终主要写异步代码，理解 `Send`/`Sync`/`Mutex`/`Arc` 这些是必须的，因为异步代码里到处都在用它们。你不会在 tokio 里看到 `std::thread::spawn`，但你会看到大量的 `Arc<Mutex<T>>` 和 `Send` bound。

2. **3.10 是最重要的一课。** Future 的 poll 机制是整个异步体系的基石。如果你只理解"async/await 是语法糖"这个层面，遇到复杂问题时会很痛苦——你不知道为什么 `select!` 会有取消安全问题，不知道为什么 `Future` 需要 `Pin`，不知道为什么异步递归需要 `Box::pin`。**理解 poll 就理解了所有答案。**

3. **写代码比读代码重要。** 每学完一课，打开 `cargo new`，把书中的例子敲一遍、改一改，看看编译器报什么错。特别是 3.2 Atomic——内存排序是"感觉懂了但实际写就错"的典型。

4. **3.14 放在最后不是因为它不重要——恰恰相反。** 取消安全、select! 陷阱是实际写异步代码时最常踩的坑。但你需要先理解 Future 和 tokio 才能理解这些坑。

5. **异步部分不需要一次性完全理解。** Future 的 Pin 机制、Waker 的 vtable、tokio 的 work-stealing 调度器——这些细节可以在实际项目中慢慢消化。先把 3.10 的核心概念吃透，后续的可以逐步深入。

---

> 💡 **最重要的一个思维转换**：在同步世界里，代码的执行顺序就是你写的顺序。在并发世界里，**你写的顺序不一定是执行的顺序**。编译器、CPU 乱序执行、操作系统的线程调度、异步运行时的任务调度——每一层都可能重排你的代码。Phase 3 的全部内容，本质上就是在回答一个问题：**在这些重排面前，如何保证你的程序仍然正确。**
>
> Rust 的答案是：**把正确性条件编码到类型系统里。** `Send` 保证所有权转移安全，`Sync` 保证共享引用安全，`Mutex` 保证互斥访问，`Atomic` + `Ordering` 保证你允许的乱序程度。一旦你理解了这些类型不只是"注解"而是**编译器强制执行的安全契约**，你就真正理解了 Rust 并发模型的核心。
