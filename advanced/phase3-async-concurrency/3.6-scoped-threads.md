# 3.6 Scoped Threads（作用域线程）—— 让线程"借用"栈上的数据

> 在 `thread::spawn` 的世界里，所有传入线程的数据必须满足 `'static` 生命周期——这意味着你几乎必须用 `Arc` 来共享数据。这造成了两个问题：第一，每次用线程都要做堆分配（Arc），即使数据明明就在栈上；第二，你必须手动 `join()` 来回收线程，编译器不会帮你检查你是否忘了 join。Scoped Threads（`std::thread::scope`，Rust 1.63 稳定）一举解决了这两个问题：**scope 内部生成的线程可以直接借用栈上的数据，而且编译器保证所有线程在 scope 返回前都已 join。** 这不仅是语法糖——它从根本上改变了你使用线程的方式，让多线程编程的体验更接近单线程借用。

## 1. thread::spawn 的烦恼——'static 约束的代价

### 1.1 为什么 spawn 要求 'static

```rust
use std::thread;

let data = vec![1, 2, 3, 4, 5];

// ❌ 编译错误：data 的生命周期不够长
// thread::spawn(move || {
//     println!("{:?}", data);
// });

// 错误信息：
// error[E0373]: closure may outlive the current function,
//               but it borrows `data`, which is owned by the current function

// 原因：spawn 创建的线程可能比当前函数活得更久
// 编译器不知道你什么时候会 join（join 是可选的！）
// 如果函数返回了而线程还在跑 → data 被释放 → 线程访问悬垂指针 → UB
// 所以 spawn 强制要求闭包是 'static —— 线程不能借用任何局部数据

// "修复"：把 data 移到堆上（用 Arc）
// let data = std::sync::Arc::new(vec![1, 2, 3, 4, 5]);
// let d = data.clone();
// thread::spawn(move || {
//     println!("{:?}", d);
// }).join().unwrap();
//
// 代价：额外的堆分配（Arc 的堆分配 + Vec 的堆分配原本就有）
// 以及引用计数的原子操作开销
```

这段代码展示了 `thread::spawn` 的根本限制：它不能保证子线程在父线程的栈帧被销毁前结束。这是出于安全考虑——Rust 不允许悬垂引用。但这个限制意味着每次用线程处理局部数据时，你都必须支付 Arc 的堆分配和引用计数开销，即使你明知线程会在函数返回前 join。

### 1.2 spawn + join 的模式和其局限性

```rust
use std::thread;

// 一个常见的多线程模式：启动 N 个线程并行处理数据，然后 join 所有
// 这种模式中，所有线程的生命周期都被函数边界约束
// 但编译器不知道这一点——它只看签名字面

fn process_data(data: Vec<i32>) -> Vec<i32> {
    let mut handles = vec![];

    // 想把 data 分成几块给不同线程处理
    // ❌ 不能把 &data 传进去（不是 'static）
    // ❌ 不能把 data move 进去（每个线程都需要一份——但只能 move 一次）

    for _ in 0..4 {
        // 只能用 Arc：
        // let d = Arc::new(data);  // ❌ 也不能这样——每次循环都会创建新的 Arc
        // handles.push(thread::spawn(move || { /* 处理 d */ }));
    }

    // 或者提前创建 Arc，然后每个线程 clone：
    // let data = Arc::new(data);
    // for _ in 0..4 {
    //     let d = data.clone();
    //     handles.push(thread::spawn(move || { /* 处理 d */ }));
    // }

    // for h in handles {
    //     h.join().unwrap();
    // }

    // 哪怕你保证了在函数返回前 join 所有线程，"借用检查器不知道" → 必须用 Arc
    vec![]
}

// 这就是 scope threads 要解决的问题——
// 提供一个"编译器知道一定会 join"的线程创建方式
```

## 2. thread::scope —— 借用的线程

### 2.1 基本用法：scope 让借用成为可能

```rust
use std::thread;

let data = vec![1, 2, 3, 4, 5, 6, 7, 8];

// thread::scope 接受一个闭包，闭包参数是 &Scope
// scope 内部可以调用 scope.spawn 创建线程
thread::scope(|s| {
    // s: &Scope —— 可以在这个 scope 内创建"作用域线程"

    // ✅ 子线程可以直接借用 data！不需要 Arc！
    s.spawn(|_| {
        println!("thread 1 sees: {:?}", &data[..4]);
        // data 的生命周期由 scope 的闭包保证
        // 编译器知道：scope 返回前，所有子线程一定已经 join
    });

    s.spawn(|_| {
        println!("thread 2 sees: {:?}", &data[4..]);
    });

    // scope 闭包结束时，自动 join 所有子线程
    // 你不需要手动收集 JoinHandle 和 join
    println!("scope ending...");
});  // ← 所有子线程在这里被 join

println!("all threads done, data still alive: {:?}", data);
```

这就是 scope 的核心能力：**子线程可以安全地借用外部作用域的变量**。编译器能保证这一点，因为 `thread::scope` 的函数签名强制了"所有子线程在 scope 返回前 join"的约定——这不是运行时检查，而是类型系统层面的保证。

### 2.2 scope 的签名解析——编译器如何保证安全

```rust
use std::thread;

// std::thread::scope 的签名（简化）：
// pub fn scope<'env, F, T>(f: F) -> T
// where
//     F: for<'scope> FnOnce(&'scope Scope<'scope, 'env>) -> T,
//     T: Send + 'static,

// 关键的高阶生命周期：
// 'env  —— 外部环境（调用 scope 的函数中的局部变量）的生命周期
// 'scope —— scope 内部的生命周期
//
// Scope<'scope, 'env> 中的 spawn 签名：
// pub fn spawn<F, T>(&'scope self, f: F) -> ScopedJoinHandle<'scope, T>
// where
//     F: FnOnce() -> T + Send + 'env,  // ← 注意：'env 而不是 'static！
//     T: Send + 'env,
//
// 对比 thread::spawn：
// pub fn spawn<F, T>(f: F) -> JoinHandle<T>
// where
//     F: FnOnce() -> T + Send + 'static,  // ← 'static 约束
//     T: Send + 'static,
//
// scope.spawn 的闭包只需要 F: FnOnce() + 'env
// 'env 就是外部栈帧的生命周期——这正是我们需要的！

// 编译器推理过程：
// 1. data 的生命周期是 'env（在 scope(f) 调用之前定义）
// 2. scope.spawn 的闭包约束是 F: 'env
// 3. 闭包中借用了 data → 闭包的生命周期 ≤ data 的生命周期 → 合法
// 4. scope 返回时所有线程已 join → data 在 scope 返回后依然有效
```

这个类型签名是 scope threads 最精妙的设计。`'env` 这个生命周期参数精确地表达了"子线程的所有借用不能超过外部环境"的约束。因为 scope 本身在 `'env` 结束前返回，所以在 scope 闭包中启动的线程也不可能超过 `'env`。

### 2.3 scope.spawn 也返回 JoinHandle

```rust
use std::thread;

let mut results = vec![0; 4];

thread::scope(|s| {
    for (i, result) in results.iter_mut().enumerate() {
        // scope.spawn 返回 ScopedJoinHandle
        // 但你不一定要用它——scope 结束时自动 join
        s.spawn(move |_| {
            *result = i * i;  // 各线程写入不同的位置——安全！
            // 注意：&mut results 的各个元素被分配给了不同线程
            // Rust 的借用检查器允许这样做，因为没有重叠
        });
    }
    // ScopedJoinHandle 可以被手动 join，但不是必须的
    // 在 scope 结束时会自动 join 所有未 join 的线程
});

println!("results: {:?}", results);  // [0, 1, 4, 9]
```

这段代码展示了一个强大的模式：**把 `&mut [T]` 的不同部分借给不同的线程并行修改**。在 `thread::spawn` 中这是不可能的（因为 `&mut` 不是 `'static`），但在 scope 中这是完全安全的——编译器确保了没有数据竞争（每个线程写不同位置）而且所有线程在函数继续前已完成。

## 3. 实战：并行处理大数组

### 3.1 把切片分成 N 块，N 个线程并行处理

```rust
use std::thread;

// 经典的"分块并行处理"模式
fn parallel_map<T: Send + Sync>(data: &mut [T], f: fn(&mut T)) {
    const NUM_THREADS: usize = 4;
    let chunk_size = (data.len() + NUM_THREADS - 1) / NUM_THREADS;

    thread::scope(|s| {
        for chunk in data.chunks_mut(chunk_size) {
            // 把 &mut [T] 的每个 chunk 借给一个线程
            // 不同线程处理不同的 chunk → 没有重叠 → 安全
            s.spawn(move |_| {
                for item in chunk {
                    f(item);
                }
            });
        }
    });
    // scope 返回时，所有线程已完成，data 全部处理完毕
}

// 使用：
let mut numbers: Vec<i32> = (0..1000).collect();
parallel_map(&mut numbers, |x| *x = x.wrapping_mul(2));
// 所有元素都被乘以 2，由 4 个线程并行完成
// 没有 Arc，没有 Mutex，没有 unsafe —— 纯粹的 safe Rust
```

这个 `parallel_map` 函数展示了 scope threads 的核心价值：**零分配的并行处理**。数据直接在原切片上修改，不需要复制、不需要 Arc 包装、不需要锁——因为每个线程处理不重叠的切片区域，这是编译器和程序员都能确证的安全。

### 3.2 并行收集结果到 Vec

```rust
use std::thread;

// 场景：每个线程处理一部分数据，结果写入共享的 Vec 的不同位置
fn parallel_compute_sum_of_squares(input: &[i32]) -> Vec<i64> {
    let n = input.len();
    let mut results = vec![0i64; n];
    let num_threads = 4;
    let chunk_size = (n + num_threads - 1) / num_threads;

    thread::scope(|s| {
        for (thread_idx, chunk_range) in
            (0..n).step_by(chunk_size).enumerate()
        {
            let end = (chunk_range + chunk_size).min(n);
            // 把 results 的不同切片范围借给不同线程
            let result_slice = &mut results[chunk_range..end];

            s.spawn(move |_| {
                for (i, &val) in input[chunk_range..end].iter().enumerate() {
                    result_slice[i] = (val as i64) * (val as i64);
                }
            });
        }
    });

    results
}

// 测试：
let input = vec![1, 2, 3, 4, 5, 6, 7, 8];
let output = parallel_compute_sum_of_squares(&input);
assert_eq!(output, vec![1, 4, 9, 16, 25, 36, 49, 64]);
// input 是共享引用（所有线程只读），results 被安全地分区写入
```

这里有三个细节值得注意：第一，`input` 被所有线程共享只读（多个 `&[i32]` 共存）；第二，`results` 的不同区间被不同的线程独占写入（`&mut [i64]` 的不同子切片）；第三，`move` 关键字把 `result_slice` 的所有权转移到闭包中——这满足 borrow checker 的要求（闭包的生命周期不超出 scope）。

### 3.3 并行归约（reduce）：计算总和

```rust
use std::thread;
use std::sync::Mutex;

fn parallel_sum(data: &[i64]) -> i64 {
    let num_threads = 4;
    let chunk_size = (data.len() + num_threads - 1) / num_threads;

    // 每个线程计算部分和，存入数组
    let partial_sums = Mutex::new(vec![0i64; num_threads]);

    thread::scope(|s| {
        for (thread_idx, chunk) in
            data.chunks(chunk_size).enumerate()
        {
            let partial = &partial_sums;
            s.spawn(move |_| {
                let sum: i64 = chunk.iter().sum();
                partial.lock().unwrap()[thread_idx] = sum;
                // 这里用了 Mutex——但只在写入时加锁，计算过程无锁
            });
        }
    });

    partial_sums.lock().unwrap().iter().sum()
}

// 或者：更优雅的方式——每个线程返回自己的结果
fn parallel_sum_v2(data: &[i64]) -> i64 {
    let num_threads = 4;
    let chunk_size = (data.len() + num_threads - 1) / num_threads;
    let results = Mutex::new(Vec::new());

    thread::scope(|s| {
        for chunk in data.chunks(chunk_size) {
            let r = &results;
            s.spawn(move |_| {
                let sum: i64 = chunk.iter().sum();
                r.lock().unwrap().push(sum);
            });
        }
    });

    results.lock().unwrap().iter().sum()
}
```

## 4. scope 作为作用域——不只是线程，还有清理

### 4.1 scope 保证在返回前 join——这是安全根基

```rust
use std::thread;
use std::time::Duration;

// 有人可能问：如果 scope 闭包中发生了 panic 怎么办？
// 答案：scope 仍然会等待所有子线程完成，然后重新抛出 panic

let result = std::panic::catch_unwind(|| {
    thread::scope(|s| {
        s.spawn(|_| {
            thread::sleep(Duration::from_secs(1));
            println!("slow thread done");
        });

        s.spawn(|_| {
            panic!("boom!");  // 这个线程 panic 了
        });

        // scope 会：
        // 1. 等待 slow thread 完成（1 秒后）
        // 2. 收集 panic
        // 3. 如果任何子线程 panic → 重新 panic
    });
});

// result: Err(...) —— "boom!" 的 panic 被捕获
// 但在 panic 之前，slow thread 已经正常完成了
// 没有任何线程被"遗弃"——scope 保证了这个不变量
```

### 4.2 嵌套 scope

```rust
use std::thread;
use std::time::Duration;

thread::scope(|outer| {
    let shared_data = vec![1, 2, 3, 4];

    // 外层线程借用 shared_data
    outer.spawn(|_| {
        println!("outer thread sees: {:?}", &shared_data);

        // ✅ 在 scope 内部可以再嵌套 scope
        thread::scope(|inner| {
            let local_data = vec![5, 6, 7, 8];

            // 内层线程可以同时借用 outer 的数据和 local 数据
            // 只要 inner scope 在 outer scope 内部
            inner.spawn(|_| {
                println!("inner: {:?} + {:?}", &shared_data, &local_data);
                // shared_data 的生命周期 > inner scope > 内层线程 → 安全
            });
            // inner scope 结束时 join 内层线程
        });
        // inner scope 结束——内层线程已结束
    });

    // outer 线程和内层线程在 outer scope 结束时都会被 join
});
```

嵌套 scope 的能力意味着你可以把并行的粒度细化到任意层级——外层并行处理大块数据，内层对外层的每个结果再做并行处理。生命周期规则保证了每一层的借用都是安全的。

### 5.2 选择指南

```rust
// ┌──────────────────────┬─────────────────┬───────────────────────┐
// │ 场景                 │ 推荐方案         │ 原因                  │
// ├──────────────────────┼─────────────────┼───────────────────────┤
// │ 并行处理数组/切片     │ Rayon            │ 自动分块+work stealing │
// │ 每个线程做不同的事情  │ thread::scope    │ Rayon 不适合异构任务   │

// │ 学习并发原理           │ thread::scope    │ 能看到线程创建的原貌  │
// │ 生产代码的数据并行     │ Rayon            │ 久经考验、优化充分    │

// scope threads 适合：
// 1. 异构任务——每个线程做完全不同的事（非数据并行）
// 2. 需要借用共享栈数据的任务
// 3. 需要精细控制线程生命周期的场景
// 4. 教育和理解并发底层机制

// Rayon 适合：
// 1. 数据并行——对数组/切片的每个元素应用相同操作
// 2. 不需要关心线程调度细节
// 3. 生产环境（经过大量优化，包括 work stealing）
```

### 6.2 scope 与错误处理

```rust
use std::thread;

fn fallible_parallel_process(data: &[i32]) -> Result<Vec<i32>, String> {
    let num_threads = 4;
    let chunk_size = (data.len() + num_threads - 1) / num_threads;
    let mut results = vec![0i32; data.len()];

    // 收集各线程的结果状态
    let errors = std::sync::Mutex::new(Vec::new());

    thread::scope(|s| {
        for (thread_idx, start) in (0..data.len()).step_by(chunk_size).enumerate() {
            let end = (start + chunk_size).min(data.len());
            let err_ref = &errors;

            s.spawn(move |_| {
                for (j, &val) in data[start..end].iter().enumerate() {
                    let idx = start + j;
                    match val.checked_mul(2) {
                        Some(v) => results[idx] = v,
                        None => {
                            err_ref.lock().unwrap()
                                .push(format!("overflow at index {}", idx));
                            return;
                        }
                    }
                }
            });
        }
    });

    let errs = errors.into_inner().unwrap();
    if errs.is_empty() {
        Ok(results)
    } else {
        Err(errs.join("; "))
    }
}

// 核心思想：用 Mutex<Vec<String>> 收集各线程的错误
// 线程写不同位置的结果数组（安全） + 锁保护地收集错误信息
```

### 练习 1 答案

```rust
use std::thread;

fn par_quicksort(data: &mut [i32]) {
    const THRESHOLD: usize = 1000;

    if data.len() <= THRESHOLD {
        data.sort_unstable();  // 串行排序——更快（避免线程创建开销）
        return;
    }

    // 分区：选择 pivot 并重排
    let pivot = data[data.len() / 2];
    let mut left = 0;
    let mut right = data.len() - 1;

    while left <= right {
        while data[left] < pivot { left += 1; }
        while data[right] > pivot { right -= 1; }
        if left <= right {
            data.swap(left, right);
            if left < data.len() - 1 { left += 1; }
            if right > 0 { right -= 1; }
        }
    }

    let (left_part, right_part) = data.split_at_mut(left);

    thread::scope(|s| {
        // 左侧在新线程上排序
        s.spawn(|_| par_quicksort(left_part));
        // 当前线程排序右侧（不浪费当前线程）
        par_quicksort(right_part);
    });
}

// 测试：
// let mut data: Vec<i32> = (0..10000).rev().collect();
// par_quicksort(&mut data);
// assert!(data.windows(2).all(|w| w[0] <= w[1]));
//
// 关键设计决策：
// 1. 阈值 1000：线程创建有开销，小数组串行更快
// 2. 当前线程也做一部分工作（right_part），不浪费
// 3. 只创建一个新线程（不递归地在两个子数组上都创建新线程）
//    这样线程数约为 log2(N/THRESHOLD)，不会爆炸
```

### 练习 2 答案

```rust
use std::thread;

fn par_map_filter<T: Send + Sync, U: Send>(
    input: &[T],
    map_fn: fn(&T) -> U,
    filter_fn: fn(&U) -> bool,
) -> Vec<U> {
    let num_threads = 4;
    let chunk_size = (input.len() + num_threads - 1) / num_threads;

    let intermediate = std::sync::Mutex::new(
        vec![Vec::new(); num_threads]
    );

    thread::scope(|s| {
        for (thread_idx, chunk) in input.chunks(chunk_size).enumerate() {
            let inter = &intermediate;
            s.spawn(move |_| {
                let mapped: Vec<U> = chunk.iter()
                    .map(map_fn)
                    .filter(|u| filter_fn(u))
                    .collect();
                inter.lock().unwrap()[thread_idx] = mapped;
            });
        }
    });

    // 收集结果
    let mut result = Vec::new();
    for chunk in intermediate.into_inner().unwrap() {
        result.extend(chunk);
    }
    result
}

// 或者：更好的方式——避免 Mutex，线程直接写入共享 Vec 的不同位置
// 但这需要知道过滤后的元素数量——可以两阶段：
// 阶段 1：各线程计算过滤后的数量
// 阶段 2：分配空间并写入

fn par_map_filter_v2<T: Send + Sync, U: Send + Default + Clone>(
    input: &[T],
    map_fn: fn(&T) -> U,
    filter_fn: fn(&U) -> bool,
) -> Vec<U> {
    let num_threads = 4;
    let chunk_size = (input.len() + num_threads - 1) / num_threads;

    // 各线程映射结果（包含过滤信息）
    let thread_results = std::sync::Mutex::new(
        vec![Vec::new(); num_threads]
    );

    thread::scope(|s| {
        for (thread_idx, chunk) in input.chunks(chunk_size).enumerate() {
            let tr = &thread_results;
            s.spawn(move |_| {
                let results: Vec<U> = chunk.iter()
                    .map(map_fn)
                    .filter(|u| filter_fn(u))
                    .collect();
                tr.lock().unwrap()[thread_idx] = results;
            });
        }
    });

    let mut result = Vec::new();
    for chunk in thread_results.into_inner().unwrap() {
        result.extend(chunk);
    }
    result
}
```

```
Scoped Threads 学习清单：

□ 理解 thread::spawn 的 'static 约束及其代价
□ 掌握 thread::scope 的基本用法——创建借用栈数据的线程

□ 理解 scope 如何保证"返回前 all joined"——这是安全根基
□ 理解 scope 中的 panic 处理——仍然会 join 所有线程
□ 掌握嵌套 scope 的用法
□ 能区分 scope threads vs Rayon 的适用场景
□ 知道 ScopedJoinHandle 的生命周期约束——不能逃逸 scope

```

> **scope 改变了我们使用线程的心智模型**：线程不再是"独立运行、可能比你活得久的实体"，而是"在你的作用域内短暂存在的协作者"。这种模型更安全、更直观，而且编译器帮你检查。scope threads 是 Rust 1.63 最重要的并发相关特性之一——如果你之前因为 `Arc` 的摩擦而避免使用多线程，现在是重新审视的时候了。下一节 [[3.7-sync-once|std::sync::Once]] 将回到同步原语的世界，探索"一次性初始化"这个看似简单却影响深远的基础设施。
