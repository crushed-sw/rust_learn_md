# 3.4 Condvar & 信号量 & Barrier —— 线程间的"等待"与"集合"

> Mutex 解决的是"只有一个线程访问共享数据"的问题，但它解决不了"等待某个条件成立"的问题。如果你用 Mutex + 循环轮询来等待条件，你会浪费 CPU 做无意义的检查——这就是"忙等"（busy wait）。Condvar（条件变量）是操作系统的经典解决方案：它让线程**在等待条件时挂起**，等条件可能成立时被唤醒。信号量（Semaphore）从另一个角度解决并发控制：**限制同时访问资源的线程数量**。Barrier 则解决"阶段性同步"问题：**所有线程都到达同一点后再继续**。这三个同步原语加起来，构成了 Rust 并发工具箱中除 Mutex/RwLock 之外的"等待与协调"能力。

## 1. Condvar（条件变量）—— 不是等锁，而是等条件

### 1.1 问题场景：忙等为什么是坏的

```rust
use std::sync::{Arc, Mutex};
use std::thread;

// 场景：一个生产者线程往 Vec 里放数据，一个消费者线程等着取数据
// 如果没有 Condvar，消费者只能不断轮询：

let queue = Arc::new(Mutex::new(Vec::<i32>::new()));
let q = queue.clone();

let producer = thread::spawn(move || {
    thread::sleep(std::time::Duration::from_millis(100));
    q.lock().unwrap().push(42);
});

// ❌ 忙等——消费者不停检查"队列里有东西吗"
loop {
    let data = queue.lock().unwrap();
    if !data.is_empty() {
        println!("got: {}", data[0]);
        break;
    }
    drop(data);  // 释放锁——但马上又会重新获取
    // CPU 在这里空转，做无意义的"拿锁-检查-放锁"循环
    // 100ms 内可能执行了数百万次无用的检查
}

producer.join().unwrap();
```

上面这段代码虽然能工作，但在那 100ms 内，消费者线程占满了 CPU，不停获取和释放锁，做无意义的检查。这就是"忙等"（busy waiting / spin-waiting）。对于可能等待几秒钟乃至几分钟的条件来说，这种浪费是不可接受的。

### 1.2 Condvar 的基本用法：wait 和 notify

```rust
use std::sync::{Arc, Mutex, Condvar};
use std::thread;

// Condvar 必须和一个 Mutex 配对使用——这是操作系统的约定
let pair = Arc::new((Mutex::new(Vec::<i32>::new()), Condvar::new()));
let pair2 = pair.clone();

// 消费者：等队列非空
let consumer = thread::spawn(move || {
    let (lock, cvar) = &*pair2;

    let mut queue = lock.lock().unwrap();
    // wait() 返回时，Mutex 已经被重新锁定
    // 所以你可以安全地访问受保护的数据
    while queue.is_empty() {
        queue = cvar.wait(queue).unwrap();
        // wait 做了三件事：
        // 1. 释放 Mutex（让生产者能拿到锁）
        // 2. 阻塞当前线程（不消耗 CPU）
        // 3. 被 notify 唤醒后，重新获取 Mutex，返回 MutexGuard
    }
    println!("consumer got: {}", queue.remove(0));
});

// 生产者：放数据，然后通知
let producer = thread::spawn(move || {
    let (lock, cvar) = &*pair;
    thread::sleep(std::time::Duration::from_millis(100));

    let mut queue = lock.lock().unwrap();
    queue.push(42);
    cvar.notify_one();  // 通知一个等待的线程"条件可能变了"
    // MutexGuard 在这里自动释放
});

producer.join().unwrap();
consumer.join().unwrap();
// 输出：consumer got: 42
// 关键：消费者没有做任何忙等！它在 cvar.wait() 中挂起，CPU 使用为 0
```

Condvar 的 `wait` 方法的签名值得仔细看：`fn wait<'a, T>(&self, guard: MutexGuard<'a, T>) -> LockResult<MutexGuard<'a, T>>`。它消耗掉你传入的 MutexGuard（释放锁），返回时给你一个新的 MutexGuard（重新获取了锁）。这个"消耗-返回"的模式保证了锁的释放和重新获取是原子地结合在等待操作中的——不会出现在释放锁和被挂起之间有其他线程插入修改了条件的情况。

### 1.3 为什么必须用 while 循环而不是 if

```rust
use std::sync::{Arc, Mutex, Condvar};
use std::thread;

// ❌ 常见错误：用 if 代替 while
// if queue.is_empty() {
//     queue = cvar.wait(queue).unwrap();
// }
// // 假设队列一定非空了——但这可能不成立！

// 原因 1：虚假唤醒（spurious wakeup）
// 操作系统可能在"没人通知"的情况下唤醒等待线程。
// 这是 POSIX 标准允许的行为——某些信号或系统事件可能触发唤醒。

// 原因 2：被唤醒不等于条件成立
// 如果有多个消费者，producer 用 notify_all 唤醒了所有消费者
// 但只放了一个数据——只有一个消费者能拿到数据
// 其他消费者被唤醒后发现队列又空了。

// ✅ 正确写法：while 循环在唤醒后重新检查条件
let pair = Arc::new((Mutex::new(Vec::<i32>::new()), Condvar::new()));
let pair2 = pair.clone();

let consumer = thread::spawn(move || {
    let (lock, cvar) = &*pair2;
    let mut queue = lock.lock().unwrap();
    while queue.is_empty() {
        // while 保证：即使被虚假唤醒，也会重新检查条件
        // 条件不满足 → 继续 wait → 再次挂起
        queue = cvar.wait(queue).unwrap();
    }
    // 到这里，队列一定非空
    println!("got: {}", queue.remove(0));
});

let producer = thread::spawn(move || {
    let (lock, cvar) = &*pair;
    thread::sleep(std::time::Duration::from_millis(50));

    // 如果有多个消费者等待，用 notify_all 可以唤醒所有
    // 但每个被唤醒的消费者都必须重新检查条件
    let mut queue = lock.lock().unwrap();
    queue.push(99);
    cvar.notify_all();
});

producer.join().unwrap();
consumer.join().unwrap();
```

这是 Condvar 使用中最容易被忽视的规则：**永远用 `while` 而不是 `if` 检查条件**。即使你不预期有虚假唤醒，这也是防御性编程的基本要求——未来代码可能被修改为多消费者场景，而 `if` 版本在那种情况下会悄悄出错。

### 1.4 notify_one vs notify_all

```rust
use std::sync::{Arc, Mutex, Condvar};
use std::thread;

// notify_one：只唤醒一个等待线程
// notify_all：唤醒所有等待线程

// 何时用 notify_one？
// - 你知道只有一个线程在等待
// - 或者一次只应该有一个线程被服务（如任务队列中一次取一个任务）
// - 性能更好——只唤醒一个线程，其他继续睡

// 何时用 notify_all？
// - 多个线程在等待不同条件（同一个 Condvar 上可能等待不同的谓词）
// - 你放入了一个"批次"的数据，想让所有消费者重新检查
// - 条件变化可能对所有等待者都有意义

// 例子：notify_one 的典型场景——单生产者单消费者
let pair = Arc::new((Mutex::new(Option::<i32>::None), Condvar::new()));
let pair2 = pair.clone();

thread::spawn(move || {
    let (lock, cvar) = &*pair2;
    let mut val = lock.lock().unwrap();
    while val.is_none() {
        val = cvar.wait(val).unwrap();
    }
    println!("received: {}", val.unwrap());
});

let (lock, cvar) = &*pair;
let mut val = lock.lock().unwrap();
*val = Some(42);
cvar.notify_one();  // 只有一个消费者——notify_one 足够
drop(val);
```

### 1.5 Condvar 与 Mutex 的关系——为什么必须配对

```rust
use std::sync::{Arc, Mutex, Condvar};
use std::thread;

// 核心问题：如果不配对 Mutex，会发生什么？

// 场景：生产者在"消费者检查条件后、调用 wait 前"通知了 Condvar
// 如果没有 Mutex：
//   Consumer:                    Producer:
//   检查条件（队列为空）
//                               lock, push, notify  ← 通知发出！但消费者还没开始 wait
//   wait() ← 开始等待一个已经失效的通知！

// 有 Mutex 的版本避免了这个问题：
//   Consumer:                    Producer:
//   lock  ← 持有锁              lock  ← 阻塞，等待消费者释放锁
//   检查条件（队列为空）
//   wait() → 释放锁、挂起       ← 现在能拿到锁了
//                               push, notify
//                               释放锁
//   ← 被唤醒，重新获取锁

// 关键：wait() 内部"释放锁 + 挂起"是原子的
// 这意味着生产者不可能在"消费者释放锁后、挂起前"这个窗口拿到锁并通知
// 操作系统保证了这两个操作的原子性

// 这就是为什么 Condvar 的 API 设计成 wait() 接受 MutexGuard：
// 你在持有锁的前提下调用 wait，释放和挂起在操作系统层面是原子的
```

这段分析揭示了 Condvar 一个根本性的设计约束：条件检查和 wait 调用必须在同一个锁的保护下进行，否则会产生"丢失唤醒"（lost wakeup）的经典并发 bug。丢失唤醒是指通知在线程开始等待之前发出，导致线程永远等不到通知。

## 2. 信号量（Semaphore）—— 控制并发数量

### 2.1 信号量的核心概念

信号量解决的是另一种并发控制问题：**不是"只能一个线程访问"，而是"最多 N 个线程同时访问"**。典型的场景包括：数据库连接池（最多 10 个连接）、API 速率限制（每秒最多 100 个请求）、文件句柄限制（最多 256 个打开文件）。

Rust 标准库没有提供 Semaphore，但 `std::sync` 提供了构建信号量所需的基础设施。tokio 提供了 `tokio::sync::Semaphore` 用于异步场景。

信号量有两种操作：**acquire（P 操作，减计数）**和**release（V 操作，增计数）**。计数为 0 时，acquire 会阻塞直到有线程 release。

### 2.2 用标准库原语实现一个简单的 Semaphore

```rust
use std::sync::{Arc, Condvar, Mutex};
use std::thread;

// 一个用 Mutex + Condvar 实现的计数信号量
struct Semaphore {
    count: Mutex<usize>,       // 当前可用许可数
    max: usize,                // 最大许可数
    cvar: Condvar,             // 当 count == 0 时，acquire 的线程在这里等
}

impl Semaphore {
    fn new(max: usize) -> Self {
        Semaphore {
            count: Mutex::new(max),
            max,
            cvar: Condvar::new(),
        }
    }

    // acquire：尝试获取一个许可，如果 count == 0 就等待
    fn acquire(&self) {
        let mut count = self.count.lock().unwrap();
        while *count == 0 {
            // while 循环——处理虚假唤醒 + 多线程竞争
            count = self.cvar.wait(count).unwrap();
        }
        *count -= 1;
        // count 减 1，许可被取走
    }

    // release：归还一个许可，如果有人在等就唤醒它
    fn release(&self) {
        let mut count = self.count.lock().unwrap();
        *count += 1;
        // 可以加上限检查：if *count > self.max { panic!("too many releases"); }
        self.cvar.notify_one();  // 唤醒一个等待者
        // 注意：这里用 notify_one 就够，因为 release 只归还 1 个许可
    }
}

// 使用示例：限制最多 3 个线程同时访问某个资源
let sem = Arc::new(Semaphore::new(3));  // 最多 3 个并发

let mut handles = vec![];
for i in 0..10 {
    let s = sem.clone();
    handles.push(thread::spawn(move || {
        println!("thread {} waiting for permit...", i);
        s.acquire();
        println!("thread {} got permit, working...", i);
        thread::sleep(std::time::Duration::from_millis(200));
        // 模拟受保护的工作
        println!("thread {} releasing permit", i);
        s.release();
    }));
}

for h in handles {
    h.join().unwrap();
}
// 输出会显示：最多只有 3 个线程同时"got permit"
// 其他线程在"waiting for permit"
```

这个实现展示了一个重要的设计模式：**Condvar + Mutex = Semaphore**。事实上，Condvar 是更基础的同步原语，你可以用它构建出信号量、读写锁、屏障等更高级的同步机制。

### 2.3 tokio 的异步信号量

```rust
// tokio::sync::Semaphore 提供了异步版本的信号量
// 在异步场景中，"阻塞"意味着 .await 而不是阻塞操作系统线程

// use tokio::sync::Semaphore;
// use std::sync::Arc;
//
// async fn example() {
//     let sem = Arc::new(Semaphore::new(5));  // 最多 5 个并发任务
//
//     let mut handles = vec![];
//     for i in 0..20 {
//         let s = sem.clone();
//         handles.push(tokio::spawn(async move {
//             let _permit = s.acquire().await.unwrap();  // 异步等待
//             // ... 做需要被限流的工作 ...
//             // _permit 在 drop 时自动 release
//         }));
//     }
//
//     for h in handles {
//         h.await.unwrap();
//     }
// }

// tokio 的 Semaphore 有一个关键设计差异：acquire().await 返回 SemaphorePermit
// 这个 permit 实现了 Drop——drop 时自动释放许可
// 这比手动 release 更安全（不会忘记释放）
```

信号量是并发控制中"第二常用"的原语（仅次于 Mutex），因为它直接对应"资源池"这个普遍需求。理解信号量之后，你会发现很多看似复杂的问题（连接池、速率限制、并发度控制）其实都是同一个模式。

## 3. Barrier（屏障）—— 所有线程集合到同一点

### 3.1 Barrier 的核心概念

```rust
use std::sync::{Arc, Barrier};
use std::thread;

// Barrier：设置一个计数器 N，当 N 个线程都调用 wait() 后，所有线程同时继续执行
// 类比：远足时，先到的人等着，等所有人都到齐了再一起出发

let barrier = Arc::new(Barrier::new(3));  // 3 个线程到齐后一起出发
let mut handles = vec![];

for i in 0..3 {
    let b = barrier.clone();
    handles.push(thread::spawn(move || {
        println!("thread {} 完成第一阶段", i);
        thread::sleep(std::time::Duration::from_millis(i as u64 * 100));
        // 线程以不同速度到达屏障

        b.wait();  // 到达屏障，等待其他人
        // 当 3 个线程都调用了 wait() 后，所有线程同时从这里继续

        println!("thread {} 完成第二阶段", i);
    }));
}

for h in handles {
    h.join().unwrap();
}

// 可能的输出：
// thread 0 完成第一阶段
// thread 1 完成第一阶段
// thread 2 完成第一阶段
// ← 三个线程都到达屏障后，继续
// thread 0 完成第二阶段
// thread 1 完成第二阶段
// thread 2 完成第二阶段
```

### 3.2 Barrier 的返回值——谁是最后一个到达的？

```rust
use std::sync::{Arc, Barrier};
use std::thread;

let barrier = Arc::new(Barrier::new(4));
let mut handles = vec![];

for i in 0..4 {
    let b = barrier.clone();
    handles.push(thread::spawn(move || {
        let is_leader = b.wait().is_leader();
        // Barrier::wait() 返回 BarrierWaitResult
        // is_leader() 返回 true 当且仅当当前线程是最后一个到达的

        if is_leader {
            println!("thread {} 是最后一个到达的，负责汇总", i);
            // 常见用法：让 leader 做汇总/清理工作
        } else {
            println!("thread {} 到达屏障", i);
        }
    }));
}

for h in handles {
    h.join().unwrap();
}
// 只有一个线程会打印"是最后一个到达的"
```

`is_leader()` 在并行计算中非常有用：比如第一阶段各线程并行计算各自的部分，到达屏障后，leader 线程负责汇总结果，然后所有线程再并行执行下一阶段。

### 3.3 实战：多阶段并行计算

```rust
use std::sync::{Arc, Barrier};
use std::thread;

// 场景：并行处理一个大数组
// 阶段 1：每个线程独立处理自己的分片
// 阶段 2：需要汇总——但汇总前必须等所有线程完成阶段 1
// 阶段 3：基于汇总结果继续处理

struct WorkData {
    local_sum: f64,
}

let barrier = Arc::new(Barrier::new(4));
let shared_sum = Arc::new(std::sync::Mutex::new(0.0_f64));

let mut handles = vec![];
for i in 0..4 {
    let b = barrier.clone();
    let sum_ref = shared_sum.clone();
    handles.push(thread::spawn(move || {
        // 阶段 1：各线程独立计算
        let mut data = WorkData {
            local_sum: (i as f64 + 1.0) * 10.0,
        };
        println!("thread {} 阶段 1 完成, local_sum={}", i, data.local_sum);

        // 到达屏障——等所有线程完成阶段 1
        let is_leader = b.wait().is_leader();

        // 阶段 2：汇总（由 leader 执行）
        if is_leader {
            println!("leader 开始汇总...");
            // 这里是简化：实际每个线程的 local_sum 应该通过共享数据结构传递
            // 此处用 Mutex 累加作为示例
            let mut sum = sum_ref.lock().unwrap();
            *sum += data.local_sum;
        }

        // 再次到达屏障——等 leader 完成汇总
        b.wait();

        // 阶段 3：基于汇总结果继续（所有线程都能看到汇总结果）
        let total = *shared_sum.lock().unwrap();
        println!("thread {} 阶段 3: 看到汇总结果 total={}", i, total);
    }));
}

for h in handles {
    h.join().unwrap();
}
// 注意：这个简化版有设计缺陷——leader 只汇了自己的 local_sum
// 实际场景中需要更好的共享数据结构传递各线程的局部结果
```

Barrier 在科学计算、图像处理、数据流水线等场景中频繁出现。它的本质是"同步点"：标记出计算中"必须等所有并行任务完成"的关键边界。

## 4. 实战场景：生产者-消费者（有界缓冲区）

### 4.1 用 Mutex + 两个 Condvar 实现有界队列

```rust
use std::sync::{Arc, Condvar, Mutex};
use std::thread;
use std::collections::VecDeque;

// 有界缓冲区：capacity 限制最大元素数
// - 队列满时，生产者等待 not_full
// - 队列空时，消费者等待 not_empty
struct BoundedQueue<T> {
    queue: Mutex<VecDeque<T>>,
    capacity: usize,
    not_full: Condvar,   // 队列不满时通知生产者
    not_empty: Condvar,  // 队列不空时通知消费者
}

impl<T> BoundedQueue<T> {
    fn new(capacity: usize) -> Self {
        BoundedQueue {
            queue: Mutex::new(VecDeque::with_capacity(capacity)),
            capacity,
            not_full: Condvar::new(),
            not_empty: Condvar::new(),
        }
    }

    fn push(&self, item: T) {
        let mut queue = self.queue.lock().unwrap();
        while queue.len() >= self.capacity {
            // 队列满了——等消费者取走数据
            queue = self.not_full.wait(queue).unwrap();
        }
        queue.push_back(item);
        // 放入数据后通知消费者
        self.not_empty.notify_one();
    }

    fn pop(&self) -> T {
        let mut queue = self.queue.lock().unwrap();
        while queue.is_empty() {
            // 队列空了——等生产者放入数据
            queue = self.not_empty.wait(queue).unwrap();
        }
        let item = queue.pop_front().unwrap();
        // 取走数据后通知生产者
        self.not_full.notify_one();
        item
    }
}

// 使用示例
let bq = Arc::new(BoundedQueue::new(3));  // 容量为 3

let bq_prod = bq.clone();
let producer = thread::spawn(move || {
    for i in 0..10 {
        println!("producer pushing {}", i);
        bq_prod.push(i);
        println!("producer pushed {}", i);
    }
});

let bq_cons = bq.clone();
let consumer = thread::spawn(move || {
    for _ in 0..10 {
        let item = bq_cons.pop();
        println!("consumer popped {}", item);
    }
});

producer.join().unwrap();
consumer.join().unwrap();
// 你会看到 producer 在队列满时被阻塞，consumer 在队列空时被阻塞
// 两者交替推进——这就是有界缓冲区的经典行为
```

这个有界缓冲区是操作系统中生产者-消费者问题的标准解法。注意这里用了两个 Condvar（not_full 和 not_empty），这是关键——如果只用一个 Condvar，可能出现"生产者通知生产者"或"消费者通知消费者"这种无效唤醒。

### 4.2 有界缓冲区的真实意义

```rust
// 有界缓冲区不只是教学示例——它在真实系统中有重要的工程价值：

// 1. 背压（backpressure）：限制生产者不会无限堆积数据
//    如果没有容量限制，生产者可能 OOM（内存耗尽）
//    有界缓冲区让生产者在队列满时自然被阻塞

// 2. 流量整形：平滑突发流量
//    高速生产者 + 低速消费者 → 缓冲区吸收突发 → 消费者不会被冲垮

// 3. 解耦：生产者和消费者不需要知道彼此的存在
//    只需要通过缓冲区通信——这是管道/Channel 模式的基础

// 实际上 std::sync::mpsc::channel 内部就用到了类似的有界缓冲区机制
// （mpsc 的同步 channel——sync_channel——就是有界的）
```

## 5. Condvar 使用陷阱与最佳实践

### 5.1 陷阱 1：忘记用 while 循环

```rust
use std::sync::{Arc, Condvar, Mutex};
use std::thread;

// ❌ 错误：用 if 代替 while
// 这在多消费者场景中会导致 bug：
// 两个消费者线程分别在 while(空) 中等待
// 生产者 push 一个元素，notify_all 唤醒两个线程
// 第一个消费者拿走元素，第二个消费者醒来后发现队列又空了
// → if 版本：第二个消费者会试图取元素 → panic 或读到错误数据
// → while 版本：第二个消费者重新检查条件 → 发现队列空 → 继续 wait

// 即使是单消费者场景，虚假唤醒也可能导致问题
// 所以 always been: while !condition { cvar.wait(guard) }
```

### 5.2 陷阱 2：在持有锁的情况下通知

```rust
use std::sync::{Arc, Condvar, Mutex};
use std::thread;

let pair = Arc::new((Mutex::new(false), Condvar::new()));
let pair2 = pair.clone();

let t = thread::spawn(move || {
    let (lock, cvar) = &*pair2;
    let mut done = lock.lock().unwrap();
    while !*done {
        done = cvar.wait(done).unwrap();
    }
    println!("done!");
});

let (lock, cvar) = &*pair;
{
    let mut done = lock.lock().unwrap();
    *done = true;
    cvar.notify_one();
    // 这里 MutexGuard 还持有中
    // 被唤醒的线程将尝试获取锁 → 立即阻塞
    // 直到当前线程释放 done 后，被唤醒线程才能拿到锁
}
// 释放后消费者才能拿到锁

// 通常 notify 后立即释放锁是最佳实践，
// 因为你在持有锁时通知不会导致正确性问题，
// 但可能导致不必要的上下文切换：
// 被唤醒线程立即尝试拿锁 → 发现被占用 → 又 sleep → 浪费
```

### 5.3 陷阱 3：同一个 Condvar 上等待不同的条件

```rust
use std::sync::{Arc, Condvar, Mutex};

// ❌ 坏设计：一个队列用同一个 Condvar 来通知"非空"和"非满"
// 消费者在等"非空"，生产者通知"非满"时会唤醒消费者
// → 消费者醒来发现队列还是空的 → 无效唤醒 → 性能浪费

// ✅ 好设计：两个 Condvar 分别用于"非空"和"非满"（如 4.1 的示例）
// 或者更通用的做法：为每个"等待条件"使用单独的 Condvar
```

### 练习 1：实现一个简单的 Barrier

使用 `Mutex + Condvar` 从零实现一个 Barrier。要求：

- `Barrier::new(n)` —— 传入需要等待的线程数量

- wait 返回一个 `bool` 指示当前线程是否为 leader（最后一个到达的线程）
- 支持复用（第二轮 wait 也能正常工作）

## 7. 练习答案

### 练习 1 答案

```rust
use std::sync::{Arc, Condvar, Mutex};

struct SimpleBarrier {
    count: Mutex<usize>,
    generation: Mutex<usize>,
    total: usize,
    cvar: Condvar,
}

impl SimpleBarrier {
    fn new(n: usize) -> Self {
        assert!(n > 0, "barrier requires at least 1 thread");
        SimpleBarrier {
            count: Mutex::new(0),
            generation: Mutex::new(0),
            total: n,
            cvar: Condvar::new(),
        }
    }

    fn wait(&self) -> bool {
        let gen = *self.generation.lock().unwrap();
        let mut count = self.count.lock().unwrap();
        *count += 1;
        if *count == self.total {
            // 最后一个到达的线程：重置计数器 + 推进 generation + 唤醒所有
            *count = 0;
            *self.generation.lock().unwrap() = gen.wrapping_add(1);
            self.cvar.notify_all();
            true  // is_leader
        } else {
            while *count < self.total {
                count = self.cvar.wait(count).unwrap();
                // 被唤醒后检查是否进入下一轮
                let current_gen = *self.generation.lock().unwrap();
                if current_gen != gen {
                    // generation 变了 → 可以继续了
                    break;
                }
                // generation 没变 → 虚假唤醒 → 继续 wait
            }
            false  // not leader
        }
    }
}
```

### 练习 2 答案

```rust
use std::sync::{Arc, Condvar, Mutex};
use std::time::{Duration, Instant};

struct SemaphoreV2 {
    count: Mutex<usize>,
    max: usize,
    cvar: Condvar,
}

impl SemaphoreV2 {
    fn new(max: usize) -> Self {
        SemaphoreV2 { count: Mutex::new(max), max, cvar: Condvar::new() }
    }

    fn acquire(&self) {
        let mut count = self.count.lock().unwrap();
        while *count == 0 {
            count = self.cvar.wait(count).unwrap();
        }
        *count -= 1;
    }

    // try_acquire：不阻塞——立即返回
    fn try_acquire(&self) -> bool {
        let mut count = self.count.lock().unwrap();
        if *count > 0 {
            *count -= 1;
            true
        } else {
            false
        }
    }

    // acquire_timeout：等待最多 timeout 时间
    fn acquire_timeout(&self, timeout: Duration) -> bool {
        let deadline = Instant::now() + timeout;
        let mut count = self.count.lock().unwrap();
        while *count == 0 {
            let remaining = deadline.saturating_duration_since(Instant::now());
            if remaining.is_zero() {
                return false;  // 超时
            }
            let (new_count, wait_result) = self.cvar.wait_timeout(count, remaining).unwrap();
            count = new_count;
            if wait_result.timed_out() {
                return false;  // wait_timeout 超时
            }
        }
        *count -= 1;
        true
    }

    fn release(&self) {
        let mut count = self.count.lock().unwrap();
        *count += 1;
        self.cvar.notify_one();
    }
}
```

### 练习 3 答案

```rust
use std::sync::{Arc, Condvar, Mutex};
use std::collections::VecDeque;

struct CloseableBoundedQueue<T> {
    queue: Mutex<VecDeque<T>>,
    capacity: usize,
    closed: Mutex<bool>,
    not_full: Condvar,
    not_empty: Condvar,
}

impl<T> CloseableBoundedQueue<T> {
    fn new(capacity: usize) -> Self {
        CloseableBoundedQueue {
            queue: Mutex::new(VecDeque::with_capacity(capacity)),
            capacity,
            closed: Mutex::new(false),
            not_full: Condvar::new(),
            not_empty: Condvar::new(),
        }
    }

    fn push(&self, item: T) -> Result<(), T> {
        let mut queue = self.queue.lock().unwrap();
        while queue.len() >= self.capacity && !*self.closed.lock().unwrap() {
            queue = self.not_full.wait(queue).unwrap();
        }
        if *self.closed.lock().unwrap() {
            return Err(item);  // 队列已关闭，拒绝
        }
        queue.push_back(item);
        self.not_empty.notify_one();
        Ok(())
    }

    fn pop(&self) -> Result<T, ()> {
        let mut queue = self.queue.lock().unwrap();
        while queue.is_empty() && !*self.closed.lock().unwrap() {
            queue = self.not_empty.wait(queue).unwrap();
        }
        if queue.is_empty() {
            // 队列空且已关闭
            return Err(());
        }
        let item = queue.pop_front().unwrap();
        self.not_full.notify_one();
        Ok(item)
    }

    fn close(&self) {
        *self.closed.lock().unwrap() = true;
        // 唤醒所有等待者，让他们检查 closed 条件
        self.not_full.notify_all();
        self.not_empty.notify_all();
    }
}
```
