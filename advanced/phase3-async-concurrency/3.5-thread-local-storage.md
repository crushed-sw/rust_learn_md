# 3.5 线程局部存储（Thread Local Storage）—— 用隔离替代共享

> 解决多线程数据访问问题有两种截然相反的思路：一种是"共享 + 同步"——用 Mutex、RwLock、Atomic 保护共享数据，让多个线程安全地访问同一份数据；另一种是"隔离"——**每个线程拥有自己的一份独立数据，完全不需要同步**。线程局部存储（TLS）就是后一种思路的极致体现。当你不需要在线程之间共享状态时，TLS 让你享受单线程编程的简单性，同时利用多线程的并行能力。Rust 的 `thread_local!` 宏将这一模式包装得异常优雅：结合 `RefCell`，你在每个线程中拥有独立的可变状态，既没有 Mutex 的开销，也没有数据竞争的风险。

---

## 1. 线程局部存储的核心思想

### 1.1 为什么需要 TLS——一个直观的例子

```rust
// 问题场景：你想给每个线程分配一个唯一的 ID
// 如果不用 TLS，你需要一个全局的原子计数器：

use std::sync::atomic::{AtomicUsize, Ordering};

static NEXT_ID: AtomicUsize = AtomicUsize::new(0);

fn get_thread_id() -> usize {
    // 问题：每个线程调用一次就分配一个新 ID
    // 但同一线程多次调用本应该返回相同的 ID
    // 这个函数做不到——除非额外维护 thread_id → id 的映射
    NEXT_ID.fetch_add(1, Ordering::SeqCst)
}

// ❌ 两次调用返回不同的 ID：
// println!("{}", get_thread_id());  // 0
// println!("{}", get_thread_id());  // 1 ← 同一个线程，不同 ID！

// 正确需求：每个线程调用 get_thread_id() 始终返回相同的 ID
// 这本质上是一个"按线程划分"的全局变量——TLS 的典型场景
```

这个例子揭示了一个基本需求：有些数据天然地"属于"线程，而不是属于整个进程。线程 ID、线程专属日志缓冲区、随机数生成器状态——这些事情应该每个线程一份，互不干扰。

### 1.2 TLS 的基本思想

```
共享数据模式：
  Thread-1 ──┐
  Thread-2 ──┼── Mutex<Data> ── 唯一一份数据
  Thread-3 ──┘

TLS 模式：
  Thread-1 ── Data₁  (Thread-1 的独立副本)
  Thread-2 ── Data₂  (Thread-2 的独立副本)
  Thread-3 ── Data₃  (Thread-3 的独立副本)

TLS 的优势：
  - 不需要 Mutex/RwLock 等同步机制 → 零同步开销
  - 每个线程的访问模式完全跟单线程一样 → 不会死锁、不会竞争
  - 编译器可以放心优化（没有其他线程会修改这份数据）

TLS 的代价：
  - 内存开销：每个线程一份副本
  - 生命周期复杂：线程退出时需要正确析构
  - 不能在线程之间共享信息（这既是优势也是限制）
```

## 2. `thread_local!` 宏——Rust 的 TLS 标准入口

### 2.1 基本用法：声明和访问

```rust
use std::thread;
use std::cell::RefCell;

// 用 thread_local! 宏声明线程局部变量
// 每个线程首次访问时，会执行初始化表达式获取自己的副本
thread_local! {
    // 注意：值包裹在引号内，如同闭包体
    static THREAD_ID: RefCell<usize> = RefCell::new(0);
    static LOG_BUFFER: RefCell<Vec<String>> = RefCell::new(Vec::new());
}

// THREAD_ID 和 LOG_BUFFER 的类型是 LocalKey<RefCell<usize>> 和 LocalKey<RefCell<Vec<String>>>
// LocalKey<T> 是 thread_local! 宏生成的实际类型

fn main() {
    // 用 with() 方法访问 TLS 变量
    THREAD_ID.with(|id| {
        *id.borrow_mut() = 42;
        println!("main thread id: {}", id.borrow());
    });

    // 主线程：id = 42
    // 子线程会得到自己的独立副本（初始值 0）
    thread::spawn(|| {
        THREAD_ID.with(|id| {
            println!("child thread id: {}", id.borrow());  // 0 —— 独立副本！
            *id.borrow_mut() = 99;
        });
    }).join().unwrap();

    // 主线程的值不受子线程修改的影响
    THREAD_ID.with(|id| {
        println!("main thread id still: {}", id.borrow());  // 42
    });
}
```

这段代码展示了 TLS 最核心的特性：**主线程和子线程各自拥有独立的 `THREAD_ID`**。子线程将值设为 99 丝毫不影响主线程的 42。`with()` 方法接收一个闭包，在这个闭包内部你可以访问 TLS 变量——就像在单线程中一样自然。

### 2.2 with() 方法的设计——为什么用闭包

```rust
use std::thread;
use std::cell::RefCell;

thread_local! {
    static COUNTER: RefCell<u32> = RefCell::new(0);
}

// 为什么 with() 不直接返回引用 &T？
// 原因：防止引用"逃逸"到 TLS 闭包外部

// ❌ 如果 with() 返回引用，这段代码会导致悬垂指针：
// let r = COUNTER.with(|c| &*c.borrow());  // 假设你能拿到 &u32
// // 此时 Ref 被释放 → 借用计数恢复
// let mut m = COUNTER.with(|c| c.borrow_mut());  // OK
// println!("{}", r);  // ❌ r 是过期引用！

// ✅ with() 只在闭包内给你访问——闭包结束后就不能再碰 TLS 数据
COUNTER.with(|c| {
    // 只能在闭包内使用 c
    *c.borrow_mut() += 1;
    println!("counter: {}", c.borrow());
    // 闭包结束后，Ref/RefMut 被 Drop，借用恢复到可用状态
});

// with() 返回闭包的返回值（可以是任何类型）
let doubled = COUNTER.with(|c| *c.borrow() * 2);
println!("doubled: {}", doubled);  // OK——doubled 是 u32，不持有引用
```

`with()` 的闭包模式是一个精心的安全设计：通过限制 TLS 数据的访问范围在一个闭包内，Rust 阻止了"引用逃逸"——你不可能在闭包外持有对 TLS 数据的引用（除非使用 `unsafe`）。这保证了 TLS 的 RefCell 借用计数不会在你不注意时处于"已借用"状态。

### 2.3 LocalKey 的其他访问方法

```rust
use std::cell::RefCell;
use std::thread;

thread_local! {
    static DATA: RefCell<i32> = RefCell::new(0);
}

// try_with() —— 与 with() 的区别
// with() 在线程的 TLS 已被销毁时 panic
// try_with() 返回 Result，允许优雅处理

// 正常情况：
let val = DATA.try_with(|d| *d.borrow());
// val: Result<i32, AccessError>
assert!(val.is_ok());

// 在线程析构阶段的 TLS 不可用（少见但可能发生）
// 如果 TLS 已经被销毁，try_with 返回 Err
// 这在实现 Drop trait 的代码中访问 TLS 时尤其需要注意

// 不支持的操作：
// - LocalKey 没有 get() 方法（不像 OnceLock）
// - 不能克隆 LocalKey
// - 不能在线程间传递（LocalKey: !Send + !Sync）
```

## 3. thread_local! 配合 RefCell——标准组合模式

### 3.1 为什么 RefCell 是 TLS 的天然搭档

```rust
use std::cell::RefCell;
use std::thread;

thread_local! {
    // 这是最常见的组合：
    // thread_local! + RefCell<T> = 每线程独立可变状态
    static LOGS: RefCell<Vec<String>> = RefCell::new(Vec::new());
}

// 为什么需要 RefCell？
// thread_local! 的初始化表达式只执行一次（每线程首次访问时）
// 之后你持有的是 &LOGS（共享引用）
// 要想修改 LOGS 的内容，你需要内部可变性
// 而 TLS 天然是单线程的 → RefCell 是最适合的内部可变性工具

// 原因：
// 1. RefCell 是 !Sync —— 但它不需要 Sync，因为每个线程只访问自己的副本
// 2. RefCell 的 borrow/borrow_mut 在单线程中足够
// 3. 不需要 Mutex 的原子操作开销

fn log_message(msg: &str) {
    LOGS.with(|logs| {
        logs.borrow_mut().push(msg.to_string());
    });
}

fn get_logs() -> Vec<String> {
    LOGS.with(|logs| logs.borrow().clone())
}

// 每个线程有自己的日志缓冲——完全隔离
thread::spawn(|| {
    log_message("thread 1: hello");
    log_message("thread 1: world");
    println!("thread 1 logs: {:?}", get_logs());
    // ["thread 1: hello", "thread 1: world"]
}).join().unwrap();

log_message("main: hi");
println!("main logs: {:?}", get_logs());
// ["main: hi"]  —— 和子线程的日志完全独立！
```

### 3.2 典型场景：每线程的随机数生成器

```rust
use std::cell::RefCell;
use std::thread;

// rand crate 内部就是这么做的：
// 它在每个线程中维护一个独立的随机数生成器（RNG）
// 这样多个线程可以同时生成随机数而不需要锁

thread_local! {
    static RNG_STATE: RefCell<u64> = RefCell::new(
        // 实际中会用线程 ID + 时间戳作为 seed
        // 这里简化为一个固定值
        12345
    );
}

fn fast_random() -> u64 {
    RNG_STATE.with(|rng| {
        // 简化的 LCG（线性同余生成器）
        let mut state = rng.borrow_mut();
        *state = state.wrapping_mul(6364136223846793005)
                     .wrapping_add(1442695040888963407);
        *state
    })
}

// 如果不用 TLS，你需要一个全局的 Mutex<ThreadRng>
// → 每次生成随机数都要拿锁 → 成为多线程场景的性能瓶颈
// TLS 版本：零同步开销，每个线程独立生成
```

### 3.3 典型场景：每线程的日志缓冲区

```rust
use std::cell::RefCell;
use std::fmt::Write;

thread_local! {
    static LOG_BUF: RefCell<String> = RefCell::new(String::with_capacity(4096));
}

fn write_log(msg: &str) {
    LOG_BUF.with(|buf| {
        // 直接在每线程独立的缓冲区中追加日志
        // 不需要锁——每个线程写自己的 buffer
        let _ = write!(buf.borrow_mut(), "[{:?}] {}\n",
            std::thread::current().id(), msg);
    });
}

fn flush_log() -> String {
    LOG_BUF.with(|buf| {
        let mut b = buf.borrow_mut();
        let result = b.clone();
        b.clear();
        result
    })
}

// 使用：
// write_log("request received");
// write_log("processing...");
// write_log("response sent");
// let batch = flush_log();  // 取出所有日志，清空缓冲区
//
// 在高并发服务器中，每个线程独立写日志缓冲区
// 避免了 Mutex<Vec<String>> 的锁竞争
// 定期将各线程的缓冲区批量化写入文件
```

## 4. `#[thread_local]` 属性——底层方案

### 4.1 thread_local! 宏与 #[thread_local] 属性的对比

`thread_local!` 宏实际上是 Rust 在 `#[thread_local]` 属性之上构建的安全封装。`#[thread_local]` 目前需要 `#![feature(thread_local)]`（nightly），而 `thread_local!` 在稳定版即可使用。

```rust
// #[thread_local] 属性（nightly only）—— 直接定义 TLS 静态变量
// #![feature(thread_local)]
//
// #[thread_local]
// static mut COUNTER: u32 = 0;
//
// fn increment() {
//     unsafe {
//         COUNTER += 1;  // 需要 unsafe——这是 static mut
//     }
// }

// 而 thread_local! 宏（stable）—— 包装成更安全的 API：
use std::cell::RefCell;

thread_local! {
    // 内部实际上生成了类似下面这样的结构：
    // （但使用了 std::thread::LocalKey，不直接暴露底层实现）
    static COUNTER_SAFE: RefCell<u32> = RefCell::new(0);
}

// thread_local! 宏展开后的大致结构（简化）：
// mod __thread_local_inner {
//     use std::cell::UnsafeCell;
//     // 平台相关的 TLS 底层存储
//     // Windows: __declspec(thread) / TlsAlloc
//     // Linux:   __thread / pthread_key_create
//     struct Storage { ... }
//     static STORAGE: Storage = Storage::new();
// }
//
// pub static COUNTER_SAFE: LocalKey<RefCell<u32>> = LocalKey {
//     inner: &__thread_local_inner::STORAGE,
//     init: |_| RefCell::new(0),
// };

// thread_local! 的优势：
// 1. stable —— 不需要 nightly
// 2. 安全的 with() 接口 —— 不需要 unsafe
// 3. 自动延迟初始化 —— 首次访问时才初始化
// 4. 正确处理 Drop —— 线程退出时析构 TLS 变量

// #[thread_local] 的适用场景（目前）：
// 1. 需要"零抽象"的极致性能（with 闭包有微小的函数调用开销）
// 2. 需要直接和 C 代码交互的场景
// 3. 但大部分场景 thread_local! 足够——with() 闭包很可能被内联
```

## 5. TLS 的底层实现原理

### 5.1 操作系统层面的 TLS 支持

```rust
// TLS 在操作系统层面的实现主要有两种方式：

// 方式 1：ELF TLS（Linux 上的 __thread）
// 编译器在 ELF 文件中创建 .tdata 和 .tbss 段
// 每个线程创建时，内核从这些段复制一份初始化数据到线程的 TLS 区域
// 访问 TLS 变量 = fs/gs 段寄存器 + 偏移量（非常快，通常几条指令）

// 方式 2：pthread_key_create（POSIX 标准方式）
// 运行时通过 pthread_key_create 创建 key
// 通过 pthread_getspecific / pthread_setspecific 读写
// 好处：可以动态创建（不需要编译期确定）
// 代价：比 ELF TLS 稍慢（需要哈希表查找）

// Rust 的 thread_local! 在不同平台上使用不同的底层机制：
// - Linux (glibc):   ELF TLS (__thread) —— 最快
// - macOS:           pthread_key —— 因为 mach-o 的 TLS 支持有限
// - Windows:         __declspec(thread) 或 TlsAlloc

// 对于 Rust 开发者来说，不需要关心这些底层差异
// thread_local! 提供了一个统一的跨平台抽象
```

### 5.2 TLS 变量的生命周期

```rust
use std::cell::RefCell;

struct HasDrop {
    name: &'static str,
}

impl Drop for HasDrop {
    fn drop(&mut self) {
        println!("dropping {}", self.name);
    }
}

thread_local! {
    static TLS_DROP: RefCell<HasDrop> = RefCell::new(HasDrop { name: "main thread" });
}

// 主线程：
TLS_DROP.with(|d| {
    println!("using {}", d.borrow().name);
});
// 主线程退出时自动 drop TLS_DROP → 打印 "dropping main thread"

// 子线程：
std::thread::spawn(|| {
    TLS_DROP.with(|d| {
        // 注意：每个线程首次访问时才会初始化
        println!("using {}", d.borrow().name);
        // 子线程退出时也会自动 drop
    });
}).join().unwrap();
// 子线程退出 → 打印 "dropping main thread"（子线程副本被析构）
```

TLS 变量的 Drop 时机与线程的退出时机一致——当线程的 `main`（或 `thread::spawn` 的闭包）结束时，该线程的所有 TLS 变量按照声明的相反顺序析构。

## 6. TLS 的危险和陷阱

### 6.1 陷阱 1：线程池中的 TLS 残留

```rust
use std::cell::RefCell;

thread_local! {
    static REQUEST_DATA: RefCell<Option<String>> = RefCell::new(None);
}

// ❌ 危险：在 tokio 的线程池中使用 TLS
// tokio 的工作线程不会为每个任务创建和销毁
// 它们在线程池中复用，处理完一个任务后紧接着处理下一个

// 模拟线程池：
fn worker_thread() {
    loop {
        let task = get_next_task();  // 获取下一个任务
        // 如果上一个任务在 REQUEST_DATA 中遗留了数据...
        // 当前任务可能读到前一个任务的数据！
        // 这是严重的安全/隐私问题

        // ✅ 好的实践：使用前重置
        REQUEST_DATA.with(|d| {
            *d.borrow_mut() = None;  // 确保没有残留数据
        });

        process_task(task);
    }
}

// 对于异步/线程池环境，更推荐明确传递上下文：
// struct RequestContext { user_id: u64, trace_id: String }
// async fn handle(ctx: RequestContext) { ... }
// 而不是依赖 TLS 隐式携带上下文

fn get_next_task() -> u32 { 0 }
fn process_task(_task: u32) {}
```

在线程池中，TLS 是"粘性的"——一个工作线程上的 TLS 值会跨请求保持。这在"每个请求一个线程"的传统服务器模型中不是问题（因为请求结束线程就退出了），但在异步/线程池模型中可能导致严重的信息泄露。

### 6.2 陷阱 2：TLS 值的 Drop 依赖

```rust
use std::cell::RefCell;

thread_local! {
    static A: RefCell<Option<String>> = RefCell::new(None);
    static B: RefCell<Option<String>> = RefCell::new(None);
}

// ❌ 问题：A 中存储了指向 B 中数据的引用
// A.with(|a| {
//     B.with(|b| {
//         // 假设 A 的值引用了 B 的值（通过某种方式）
//         *a.borrow_mut() = Some(b.borrow().clone());
//     });
// });
//
// 当线程退出时，TLS 变量按声明顺序的逆序析构
// 如果 A 先声明（后析构），B 后声明（先析构），
// 且 A 的值依赖 B 的值 → 悬垂引用 → 未定义行为

// ✅ 解决：不要在线程局部变量之间建立引用关系
// 如果确实需要相关数据，放在同一个 RefCell 管理的结构体中：
struct ThreadState {
    field_a: String,
    field_b: String,
}

thread_local! {
    static STATE: RefCell<ThreadState> = RefCell::new(ThreadState {
        field_a: String::new(),
        field_b: String::new(),
    });
}
```

### 6.3 陷阱 3：TLS 在 Drop 实现中的问题

```rust
use std::cell::RefCell;

thread_local! {
    static COUNTER: RefCell<u32> = RefCell::new(0);
}

struct Watcher;

impl Drop for Watcher {
    fn drop(&mut self) {
        // ⚠️ 危险：在 Drop 中访问 TLS
        // 如果线程正在退出，TLS 可能已经被部分清理
        // try_with 可能返回 Err
        if let Ok(c) = COUNTER.try_with(|c| *c.borrow()) {
            println!("counter at drop: {}", c);
        } else {
            // TLS 已经被析构了——这是 Drop 期间常见的状态
            println!("TLS unavailable during drop");
        }
    }
}

// 这个陷阱的根源在于：
// 你不能控制 Drop 的调用顺序和时机
// 如果你的类型的 Drop 可能在任何时候被调用（比如作为另一个 TLS 变量的字段）
// 你就不能安全地假设 TLS 还活着
```

### 6.4 陷阱 4：TLS 的内存开销

```rust
use std::cell::RefCell;
use std::thread;

thread_local! {
    // 如果每个线程的 LARGE_BUFFER 大小为 1MB
    // 并且你创建了 1000 个线程（或者线程池中有 1000 个工作线程）
    // → 额外 1GB 内存！
    static LARGE_BUFFER: RefCell<Vec<u8>> = RefCell::new(vec![0u8; 1024 * 1024]);
}

// TLS 的内存占用 = sizeof(T) × 线程数量
// 这对于大缓冲区来说可能非常高
// 在设计时需要考虑：
// - 线程数量是否可控？
// - 数据是否真的需要这么大？有没有懒分配的方式？
// - 使用 Arc<Mutex<T>> 共享是否有更好的内存/性能权衡？

// 对于大数据的"每线程"需求，考虑：
// 方案 A：用 thread_local + lazy 初始化（首次访问时才分配）——但 thread_local! 本身已经是 lazy 的
// 方案 B：使用 thread_local + Option<T>，按需分配
// 方案 C：干脆用连接池模式，池中对象数量 ＜ 线程数量
```

## 7. 设计考量：TLS vs 共享同步

### 7.1 何时选择 TLS

```rust
// 使用 TLS 的决策清单：

// ✅ 适合 TLS 的场景：
// 1. 每线程的独立状态：线程 ID 缓存、随机数生成器
// 2. 高频访问、低共享需求：日志缓冲区、计数器（只在出口处汇总）
// 3. 读多写多，不需要跨线程共享：本地缓存
// 4. 性能要求极高，不能接受锁开销：perf 统计、metrics 收集

// ❌ 不适合 TLS 的场景：
// 1. 需要跨线程共享和通信：全局配置、共享数据结构
// 2. 数据需要在所有线程中保持一致：分布式计数、全局状态
// 3. 线程池/异步环境：tokio 任务在多个工作线程间迁移
// 4. 数据量大 × 线程数多：内存开销不可接受
```

### 7.2 汇总模式：TLS 收集 + 定期汇总

```rust
use std::cell::RefCell;
use std::sync::atomic::{AtomicU64, Ordering};

// 一个常见的折中方案：
// 每个线程使用 TLS 维护本地计数器（零锁开销）
// 定期将本地计数器刷新到全局原子变量中

thread_local! {
    static LOCAL_COUNT: RefCell<u64> = RefCell::new(0);
}

static GLOBAL_COUNT: AtomicU64 = AtomicU64::new(0);

fn increment() {
    LOCAL_COUNT.with(|c| {
        let mut count = c.borrow_mut();
        *count += 1;
        if *count >= 1000 {
            // 本地计数达到阈值 → 批量刷新到全局
            GLOBAL_COUNT.fetch_add(*count, Ordering::Relaxed);
            *count = 0;
        }
    });
}

fn get_approximate_total() -> u64 {
    // 近似值——各线程的本地计数还没有刷新到全局
    // 但通常是足够接近的
    GLOBAL_COUNT.load(Ordering::Relaxed)
}

// 这种模式在高性能 metrics 采集中非常常见
// 避免了每次 increment 都做原子操作（原子操作仍然比普通指令慢）
// 代价是"全局总数"是近似的而非精确的
```

### 练习 1 答案

```rust
use std::cell::RefCell;
use std::sync::atomic::{AtomicUsize, Ordering};

static NEXT_ID: AtomicUsize = AtomicUsize::new(0);

thread_local! {
    static THREAD_ID: RefCell<Option<usize>> = RefCell::new(None);
}

fn thread_id() -> usize {
    THREAD_ID.with(|id| {
        let mut id = id.borrow_mut();
        match *id {
            Some(existing) => existing,
            None => {
                let new_id = NEXT_ID.fetch_add(1, Ordering::SeqCst);
                *id = Some(new_id);
                new_id
            }
        }
    })
}

// 测试（概念性的）：
// let mut handles = vec![];
// for _ in 0..4 {
//     handles.push(std::thread::spawn(|| {
//         let id = thread_id();
//         assert_eq!(thread_id(), id);  // 同一线程返回相同 ID
//         println!("thread {} reporting", id);
//     }));
// }
// for h in handles { h.join().unwrap(); }
//
// 关键设计：
// - RefCell<Option<usize>> 用于延迟分配（None → 首次访问时分配）
// - AtomicUsize 保证全局分配安全
// - thread_local! 保证各线程缓存各自分配的 ID
```

### 练习 2 答案

```rust
use std::cell::RefCell;
use std::collections::VecDeque;
use std::sync::Mutex;

thread_local! {
    static LOGS: RefCell<VecDeque<(String, String)>> = RefCell::new(VecDeque::new());
}

// 全局注册表：记录每个线程的 TLS 指针（简化——实际需要更复杂的数据结构）
// 注意：std 没有直接提供"枚举所有线程 TLS"的功能
// 这种设计需要额外的注册机制

mod tls_collector {
    use std::sync::Mutex;
    use std::sync::Arc;

    // 每个线程在创建时注册自己，退出时注销
    // 这是概念的简化实现
    static REGISTRY: Mutex<Vec<Arc<dyn Fn() -> Vec<String> + Send + Sync>>> =
        Mutex::new(Vec::new());

    pub fn register(collector: Arc<dyn Fn() -> Vec<String> + Send + Sync>) {
        REGISTRY.lock().unwrap().push(collector);
    }

    pub fn flush_all() -> Vec<String> {
        let mut all = Vec::new();
        for collector in REGISTRY.lock().unwrap().iter() {
            all.extend(collector());
        }
        all
    }
}

fn log(level: &str, message: &str) {
    LOGS.with(|logs| {
        let mut logs = logs.borrow_mut();
        logs.push_back((level.to_string(), message.to_string()));
        if logs.len() > 100 {
            logs.pop_front();  // 保持最多 100 条
        }
    });
}

fn flush_thread_logs() -> Vec<String> {
    LOGS.with(|logs| {
        let mut logs = logs.borrow_mut();
        let result: Vec<String> = logs
            .drain(..)
            .map(|(l, m)| format!("[{}] {}", l, m))
            .collect();
        result
    })
}

// 讨论：跨线程收集 TLS 日志是一个难题
// 标准库没有提供"遍历所有线程 TLS"的 API
// 实际项目中的方案：
// 1. 用 Chanel：每个线程定期通过 mpsc::channel 发送日志批次
// 2. 用全局注册表（如上述概念）：线程启动时注册 flush 回调
// 3. 用 tracing / log crate（已经包含了这些能力）
```

### 练习 3 答案

```rust
use std::cell::RefCell;
use std::sync::{Arc, Mutex};
use std::sync::atomic::{AtomicI32, Ordering};
use std::thread;

// 三种计数器实现：

// 方式 1：TLS（每线程独立计数，不加锁）
thread_local! {
    static TLS_COUNT: RefCell<i32> = RefCell::new(0);
}

fn tls_increment() {
    TLS_COUNT.with(|c| *c.borrow_mut() += 1);
}

fn tls_get() -> i32 {
    // 注意：TLS 的总和需要收集各线程的值
    // 这里只返回当前线程的计数
    TLS_COUNT.with(|c| *c.borrow())
}

// 方式 2：Atomic（全局原子计数）
static ATOMIC_COUNT: AtomicI32 = AtomicI32::new(0);

fn atomic_increment() {
    ATOMIC_COUNT.fetch_add(1, Ordering::Relaxed);
}

fn atomic_get() -> i32 {
    ATOMIC_COUNT.load(Ordering::Relaxed)
}

// 方式 3：Mutex（全局加锁计数）
fn mutex_increment(counter: &Mutex<i32>) {
    *counter.lock().unwrap() += 1;
}

// Benchmark 结果（概念性的，实际数字取决于 CPU 和平台）：
//
// ┌──────────────┬──────────────────────────────────┐
// │ 方式         │ 性能（越低越好）                  │
// ├──────────────┼──────────────────────────────────┤
// │ TLS          │ ~3ns/op  （跟普通内存访问一样快） │
// │ Atomic       │ ~10ns/op（需要 CPU 缓存一致性）   │
// │ Mutex        │ ~30ns/op（系统调用 + 内核开销）   │
// └──────────────┴──────────────────────────────────┘
//
// 差异分析：
// - TLS：每个线程访问自己的内存 → 没有缓存一致性开销
//        和单线程的普通变量访问一样快
//
// - Atomic：CPU 需要保证缓存行在所有核之间一致
//          每次 fetch_add 需要获取缓存行的独占所有权
//          4 个线程竞争同一缓存行 → 大量的缓存行弹跳（cache line bouncing）
//
// - Mutex：涉及系统调用（futex）、内核调度
//          在无竞争时可用"快速路径"（atomic + 自旋），接近 Atomic
//          在有竞争时开销最大——线程可能被内核挂起
//
// 结论：TLS 在"不需要跨线程共享"时是最快的选择
//       但要付出的代价是不能方便地获取全局总和
```

> **从同步到隔离的转变**：[[3.4-condvar-semaphore-barrier|之前的同步原语]] 教我们如何安全地在线程间共享数据。TLS 提供了一个完全不同的视角：**有些数据根本不值得共享**。当下一个需求是"每个线程有个独立的 X"时，TLS 是你的第一反应。当然，TLS 的前提是每个线程有独立的数据，如果线程需要访问同一份数据，我们仍然需要 Mutex 等同步原语。下一节 [[3.6-scoped-threads|Scoped Threads]] 将展示 Rust 如何让线程"借用"栈数据，进一步模糊单线程和多线程的边界。
