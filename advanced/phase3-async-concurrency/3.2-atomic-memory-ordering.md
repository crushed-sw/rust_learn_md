# 3.2 Atomic & Memory Ordering —— 无锁编程的底层引擎

> 锁是对并发访问的"粗粒度排队"——它好用但代价高。如果你只需要保护一个计数器、一个标志位或一个指针的更新，用 `Mutex` 就像用大炮打蚊子。Rust 的 `Atomic*` 类型提供了**基于 CPU 原子指令的细粒度并发原语**：一条 `fetch_add` 指令在硬件层面保证"读-改-写"不可分割，不需要操作系统介入，不需要上下文切换。但原子操作的真正难点不在原子性——而在**内存顺序（memory ordering）**。`Relaxed`、`Acquire/Release`、`SeqCst` 是三种不同的"可见性保证级别"，选错了不会导致数据竞争（原子操作本身就阻止了竞争），但会导致你看到"不可能"的执行顺序——逻辑错误比崩溃更难排查。本课从 CPU 缓存一致性的角度解释为什么需要内存顺序，然后带你从 `Relaxed` 的"极简保证"走到 `SeqCst` 的"全局统一时钟"，最终构建一个无锁并发栈作为实战检验。

## 1. 为什么需要原子操作 —— 从数据竞争说起

### 1.1 非原子读-改-写的根本问题

在多线程环境中，最经典的并发问题是"两个线程同时对同一个变量做读-改-写"。这不是 Rust 特有的问题——所有语言在并发场景下都面临同样的挑战。

```rust
// 假设我们有一个并发计数器（错误的实现）
use std::thread;

fn bad_counter() {
    let mut counter: i32 = 0;  // 普通变量，非原子

    // ❌ 这段代码实际上无法通过 Rust 编译
    // 因为 Rust 的所有权系统不允许两个线程同时访问同一个可变变量
    // 但假设我们用了 unsafe 绕过了检查（或者在 C/C++ 中），问题就会暴露：

    // 在汇编层面，counter += 1 是这样的：
    //   load  counter → register   (1) 读取
    //   add   register, 1          (2) 修改
    //   store register → counter   (3) 写回

    // 两个线程并发执行：
    // 时间 →
    // 线程 A: load(42)  ----add(43)----store(43)
    // 线程 B:       load(42)----add(43)----store(43)
    //
    // 期望结果：counter = 44（两个线程各加一次）
    // 实际结果：counter = 43（两个线程都读到了 42，各自加 1 后写回 43）
    // → 丢失了一次更新！这就是"丢失更新（lost update）"问题
}
```

非原子操作的根因在于"读-改-写"不是一个 CPU 指令能完成的操作，而是被分解为多条指令，这些指令之间可以被其他线程的操作交错执行。即使只有一条 `store` 或 `load` 指令，在多核 CPU 的缓存一致性协议下，一个核心的写操作也可能不会立即被另一个核心看到——这就引出了内存顺序的概念。

### 1.2 原子操作如何解决这个问题

原子操作的核心承诺是：**"读-改-写"作为一个不可分割的整体被执行**。在 CPU 层面，现代处理器提供专门的原子指令（如 x86 的 `LOCK CMPXCHG`、ARM 的 `LDREX/STREX`），这些指令在执行期间锁住缓存行或总线，阻止其他核心访问同一内存位置。

```rust
use std::sync::atomic::{AtomicI32, Ordering};

// 原子计数器的正确实现
let counter = AtomicI32::new(0);

// AtomicI32::fetch_add 在汇编级别对应一条原子指令：
// x86:  lock xadd [addr], 1
// ARM:  ldrex + add + strex（LL/SC 循环）
//
// CPU 保证：在 fetch_add 执行期间，其他核心无法读写该内存位置
// 因此不存在"读-改-写"被交错执行的风险

counter.fetch_add(1, Ordering::SeqCst);
counter.fetch_add(1, Ordering::SeqCst);
// 结果一定是 2——不会丢失更新
```

原子操作保证的是"操作的不可分割性"（atomicity）。但仅仅是原子性还不够——在多核系统中，CPU 缓存会导致一个核心的写入对其他核心"延迟可见"。这就需要内存顺序来规定"写入何时对其他核心可见"和"操作之间的先后关系如何建立"。

## 2. 内存顺序的三层模型

### 2.1 Relaxed —— 只保证原子性，不保证顺序

`Ordering::Relaxed` 是最弱的内存顺序约束。它只承诺单次操作的原子性（不会被其他线程看到"撕裂"的中间状态），但对操作之间的顺序不做任何保证。

```rust
use std::sync::atomic::{AtomicI32, AtomicBool, Ordering};
use std::thread;

// Relaxed 示例：两个独立计数器，不关心彼此的顺序
fn relaxed_counters() {
    let counter_a = AtomicI32::new(0);
    let counter_b = AtomicI32::new(0);

    // 线程 1：递增 counter_a
    let a = &counter_a;
    let t1 = thread::spawn(move || {
        for _ in 0..1000 {
            a.fetch_add(1, Ordering::Relaxed);
        }
    });

    // 线程 2：递增 counter_b
    let b = &counter_b;
    let t2 = thread::spawn(move || {
        for _ in 0..1000 {
            b.fetch_add(1, Ordering::Relaxed);
        }
    });

    t1.join().unwrap();
    t2.join().unwrap();

    // 最终值——一定是 1000 和 1000
    // 因为每次 fetch_add 本身是原子的
    assert_eq!(counter_a.load(Ordering::Relaxed), 1000);
    assert_eq!(counter_b.load(Ordering::Relaxed), 1000);

    // 但 Relaxed 不保证"线程 1 的第 5 次递增"在"线程 2 的第 10 次递增"之前
    // ——如果你需要这种"跨变量的顺序"，Relaxed 不够
}
```

`Relaxed` 的典型使用场景是"统计计数器"——比如一个 HTTP 服务器的请求计数，你只关心最终总数的准确性（不能丢更新），但不关心"第 100 个请求发生在第 90 个请求之后的多久"。在 Relaxed 模式下，编译器可以做激进的重排优化（只要不影响单线程语义），CPU 也可以将写入暂存在 store buffer 中延迟提交。

### 2.2 为什么需要更强的顺序 —— 一个具体的反例

当我们不仅需要原子性，还需要操作之间的"先后关系"被其他线程观察到时，Relaxed 就不够了。

```rust
use std::sync::atomic::{AtomicBool, AtomicI32, Ordering};
use std::thread;

// ❌ 使用 Relaxed 的错误示范
fn broken_flag_data_pattern() {
    static DATA: AtomicI32 = AtomicI32::new(0);
    static READY: AtomicBool = AtomicBool::new(false);

    // 线程 1（写入者）：写入数据，然后设置 flag
    let t1 = thread::spawn(|| {
        DATA.store(42, Ordering::Relaxed);      // (1) "写入"数据
        READY.store(true, Ordering::Relaxed);    // (2) "通知"数据就绪
    });

    // 线程 2（读取者）：检查 flag，如果已设置则读取数据
    let t2 = thread::spawn(|| {
        if READY.load(Ordering::Relaxed) {       // (3) 检查 flag
            let val = DATA.load(Ordering::Relaxed); // (4) 读取数据
            // 这里可能打印 0！——为什么？
            println!("data: {}", val);
        }
    });

    t1.join().unwrap();
    t2.join().unwrap();
}

// 问题分析：
//
// CPU 或编译器可能重排线程 1 的操作：
//   store(READY, true) 先执行！
//   store(DATA, 42)    后执行！
//
// 或者即使线程 1 按序写了，线程 2 看到的顺序可能不同：
//   CPU 核心 2 的缓存在处理时可能先收到 READY 的更新
//   而 DATA 的更新还在 store buffer 中
//
// Relaxed 允许所有这些重排——
// 它只保证每个单独的 store/load 是原子的，
// 不保证跨变量的顺序关系
```

这个"flag-data"模式是并发编程中最经典的模式之一：一个线程写数据并设置标志，另一个线程看到标志后读取数据。它要求 `store(DATA)` 必须在 `store(READY)` **之前对所有线程可见**，`load(READY)` 看到 `true` 后 `load(DATA)` 必须看到最新的值。`Relaxed` 做不到这一点——它不建立跨操作的"之前发生"（happens-before）关系。

## 3. Acquire 和 Release —— 经典锁的"先序"语义

### 3.1 Acquire/Release 的核心语义

`Acquire` 和 `Release` 成对出现，在操作之间建立 happens-before 关系。

```rust
use std::sync::atomic::{AtomicBool, AtomicI32, Ordering};
use std::thread;

// Acquire/Release 正确实现 flag-data 模式
fn correct_flag_data_pattern() {
    static DATA: AtomicI32 = AtomicI32::new(0);
    static READY: AtomicBool = AtomicBool::new(false);

    let t1 = thread::spawn(|| {
        DATA.store(42, Ordering::Relaxed);        // (1) 写数据——可以用 Relaxed
        READY.store(true, Ordering::Release);     // (2) Release 写——"释放"之前的操作
    });
    // Release 语义：当前线程中所有在 Release 之前的写入（包括 Relaxed 的）
    // 在 Release 之后对所有做了 Acquire 的线程可见。

    let t2 = thread::spawn(|| {
        if READY.load(Ordering::Acquire) {        // (3) Acquire 读——"获取"之后的操作
            // Acquire 语义：在此之后的所有读取（包括 Relaxed 的）
            // 都会看到 Release 之前的所有写入。
            let val = DATA.load(Ordering::Relaxed); // (4) 保证看到 42
            assert_eq!(val, 42);  // ✅ 一定通过
        }
    });

    t1.join().unwrap();
    t2.join().unwrap();
}

// 核心保证：
//   Release 存储（(2)）与 Acquire 加载（(3)）配对时，
//   在 (2) 之前的全部写入对在 (3) 之后的全部读取可见。
//
// 等价于：
//   (1) happens-before (2)  （单线程内的程序顺序）
//   (2) happens-before (3)  （Release-Acquire 配对）
//   (3) happens-before (4)  （单线程内的程序顺序）
//   → (1) happens-before (4)  （传递性）
```

`Acquire/Release` 是实现"临界区"效果的最小必要约束。设想一个 `Mutex::lock()` 的内部实现：`lock()` 成功时相当于做了一个 `Acquire` 读，`unlock()` 相当于做了一个 `Release` 写。临界区内的代码在 `Acquire` 之后执行，`Release` 之前执行——因此临界区内的所有修改对其他后续获取同一锁的线程可见。

### 3.2 Release-Acquire 的成对机制 —— 可视化理解

理解 Release 和 Acquire 如何建立"同步点"是掌握内存顺序的关键。下面用时间线图来可视化这个过程。

```rust
// Release-Acquire 同步的可视化（概念图）
//
// 线程 A (writer)         线程 B (reader)
// ===================     ===================
// data = 42  ────┐
//                │        ┌──── if ready {
// ready = true ──┼───┐    │
//                │   │    │      val = data
//                │   └────┤
//                │        │    }
//                │        │
//  Release point ┘        └─── Acquire point
//  "我之前的写入           "我之后的读取
//   现在对外可见"           保证看到 Release 前的写入"
//
// Release store 建立一个"发布点"——
// 该线程在此刻之前的所有写入被"发布"给世界。
//
// Acquire load 建立一个"接收点"——
// 该线程在此刻之后的所有读取都能"接收"到发布点的数据。
//
// 当 Acquire load 看到 Release store 写入的值时，
// 这两个点之间建立了同步关系（synchronizes-with）。
```

注意一个关键细节：**Release 和 Acquire 的配对是通过"值"来关联的**——线程 B 的 Acquire 读必须"看到"线程 A 的 Release 写所写入的值，同步关系才建立。如果 Acquire 读到的值是另一个线程写的，那么同步关系在那些线程之间建立，而不是与线程 A。

### 3.3 一个更复杂的 Acquire/Release 例子 —— 消息传递

Acquire/Release 不仅可以用于单个 flag-data 模式，还可以用于更复杂的"消息传递"场景。

```rust
use std::sync::atomic::{AtomicBool, AtomicUsize, Ordering};

struct LockFreeQueue<T> {
    data: Vec<Option<T>>,        // 固定大小的环形缓冲区
    write_idx: AtomicUsize,       // 写入指针
    read_idx: AtomicUsize,        // 读取指针
    // 每个槽位有它自己的"ready"标志
    ready_flags: Vec<AtomicBool>,
}

impl<T: Clone> LockFreeQueue<T> {
    /// 尝试写入一个元素。如果队列满了返回 Err。
    fn try_push(&self, item: T) -> Result<(), T> {
        let w = self.write_idx.load(Ordering::Relaxed);
        let next_w = (w + 1) % self.data.len();
        let r = self.read_idx.load(Ordering::Acquire);  // (1) Acquire 读 read_idx
                                                          // 保证能看到之前的写入者释放的数据

        if next_w == r {
            return Err(item);  // 队列满
        }

        // 此时可以安全写入——因为 r 之前的槽位已经被读取者消费
        // 而 w 槽位不属于读取者的范围

        // 写入数据（Relaxed 足够——还没释放）
        unsafe {
            // 简化：实际上需要处理 Vec 中已有数据的情况
            // 这里仅作概念演示
        }

        self.write_idx.store(next_w, Ordering::Release);  // (2) Release 写 write_idx
                                                           // "发布"新数据给读取者
        Ok(())
    }

    fn try_pop(&self) -> Option<T> {
        let r = self.read_idx.load(Ordering::Relaxed);
        let w = self.write_idx.load(Ordering::Acquire);  // (3) Acquire 读 write_idx
                                                          // 保证能看到写入者的 Release 发布的数据

        if r == w {
            return None;  // 队列空
        }

        // 此时可以安全读取 r 槽位的数据
        // 因为写入者在 (2) 的 Release 保证了数据可见

        let item = self.data[r].clone();
        self.read_idx.store((r + 1) % self.data.len(), Ordering::Release);  // (4) Release 写
                                                                             // "发布"槽位已空
        item
    }
}
// 注意：这是一个高度简化的概念演示
// 实际的无锁队列（如 crossbeam 的实现）要复杂得多
// 需要处理 ABA 问题、内存回收等
```

## 4. SeqCst —— 全局统一时钟

### 4.1 SeqCst 的定义和代价

`Ordering::SeqCst`（Sequentially Consistent）是最强的内存顺序。它不仅是 Release-Acquire 的超集，还额外要求：**所有标记为 SeqCst 的操作在所有线程看来都以单一的全序执行**。

```rust
use std::sync::atomic::{AtomicBool, AtomicI32, Ordering};
use std::thread;

// SeqCst 的"全局统一时钟"保证
// 假设每个操作都是"瞬间完成"的
// 所有线程看到的操作顺序完全一致
//
// 例子：四个 SeqCst 存储操作
// 线程 A: X.store(1, SeqCst)
// 线程 B: Y.store(1, SeqCst)
// 线程 C: 读 X 和 Y
// 线程 D: 读 X 和 Y
//
// SeqCst 保证：线程 C 和线程 D 看到的 X 和 Y 的更新顺序一致
// ——即使 X 和 Y 是不同的变量
//
// Relaxed 或 Release/Acquire 不提供这个保证

fn seqcst_demo() {
    static X: AtomicI32 = AtomicI32::new(0);
    static Y: AtomicI32 = AtomicI32::new(0);

    let t1 = thread::spawn(|| {
        X.store(1, Ordering::SeqCst);  // SeqCst 写入
    });

    let t2 = thread::spawn(|| {
        Y.store(1, Ordering::SeqCst);  // SeqCst 写入
    });

    let t3 = thread::spawn(|| {
        let x = X.load(Ordering::SeqCst);
        let y = Y.load(Ordering::SeqCst);
        // 因为 SeqCst，所有线程看到 X=1 和 Y=1 的顺序一致
        println!("Thread 3: x={}, y={}", x, y);
    });

    let t4 = thread::spawn(|| {
        let y = Y.load(Ordering::SeqCst);
        let x = X.load(Ordering::SeqCst);
        // 这里的输出必须与 Thread 3 一致
        // ——不能用 Release/Acquire 保证这一点
        println!("Thread 4: y={}, x={}", y, x);
    });

    t1.join().unwrap();
    t2.join().unwrap();
    t3.join().unwrap();
    t4.join().unwrap();
}
```

`SeqCst` 的代价比 `Acquire/Release` 高：在 x86 平台上，`SeqCst` 存储通常需要一个 `MFENCE`（memory fence）指令，而 `Release` 存储只需要一个普通的 `MOV`（因为 x86 的 TSO 模型天然提供了 acquire-release 语义）。在 ARM 等弱内存模型平台上，差距更大——`SeqCst` 需要额外的 `DMB`（Data Memory Barrier）指令。

### 4.2 为什么你几乎永远不需要 SeqCst

在实际工程中，`SeqCst` 很少是必需的。`Acquire/Release` 已经涵盖了 99% 的并发模式——锁、无锁队列、消息传递、引用计数。

```rust
use std::sync::atomic::{AtomicI32, Ordering};

// 你真正需要 SeqCst 的场景：
// 1. 多个 SeqCst 原子变量之间的全局排序
//    ——比如你想要"所有线程同意 A 在 B 之前发生"
// 2. 调试阶段——用 SeqCst 排除内存顺序问题
//    然后逐个降级
// 3. 某些特定的 lock-free 算法可能要求 SeqCst

// 不需要 SeqCst 的场景（这些都可以用 Acquire/Release）：
// ❌ 经典的 lock/unlock 模式 → Release/Acquire 足够
// ❌ 生产者-消费者队列 → Release/Acquire 足够
// ❌ 引用计数加减 → Release/Acquire 足够
// ❌ flag + data 模式 → Release/Acquire 足够
// ❌ 线程安全计数器 → Relaxed 就够（如果你只需要最终一致）

// 经验法则：
// 1. 从 Relaxed 开始思考——"我真的需要顺序保证吗？"
// 2. 如果需要顺序，用 Acquire/Release——"这是经典的同步模式"
// 3. 只在绝对必要时才用 SeqCst——"我需要全局统一顺序"
```

## 5. 原子类型全景 —— AtomicI32, AtomicBool, AtomicPtr

### 5.1 AtomicI32 / AtomicUsize / AtomicI64 等

整数原子类型（`AtomicI32`、`AtomicU64`、`AtomicUsize` 等）是最常用的原子类型。它们提供了在整数上的原子操作全集。

```rust
use std::sync::atomic::{AtomicI32, AtomicU64, Ordering};
use std::sync::Arc;
use std::thread;

// 整数原子类型的常用操作
fn integer_atomics() {
    let counter = AtomicI32::new(0);

    // 基本读写
    counter.store(10, Ordering::Release);      // 原子写入
    let val = counter.load(Ordering::Acquire);  // 原子读取
    assert_eq!(val, 10);

    // fetch_add / fetch_sub —— 原子加减
    let old = counter.fetch_add(5, Ordering::Relaxed);   // 返回旧值
    assert_eq!(old, 10);
    assert_eq!(counter.load(Ordering::Relaxed), 15);

    let old = counter.fetch_sub(3, Ordering::Relaxed);
    assert_eq!(old, 15);
    assert_eq!(counter.load(Ordering::Relaxed), 12);

    // fetch_and / fetch_or / fetch_xor —— 按位原子操作
    counter.store(0b1100, Ordering::Relaxed);
    counter.fetch_and(0b1010, Ordering::Relaxed);  // 1100 & 1010 = 1000
    assert_eq!(counter.load(Ordering::Relaxed), 0b1000);

    counter.fetch_or(0b0011, Ordering::Relaxed);   // 1000 | 0011 = 1011
    assert_eq!(counter.load(Ordering::Relaxed), 0b1011);

    // fetch_max / fetch_min —— 原子最大值/最小值（nightly）
    // counter.fetch_max(100, Ordering::Relaxed);
    // counter.fetch_min(0, Ordering::Relaxed);

    // swap —— 原子替换，返回旧值
    let old = counter.swap(42, Ordering::AcqRel);
    assert_eq!(old, 0b1011);
    assert_eq!(counter.load(Ordering::Relaxed), 42);
}

// 实战：多线程并发计数器（Relaxed 足够）
fn concurrent_counter() {
    let counter = Arc::new(AtomicU64::new(0));
    let mut handles = vec![];

    for _ in 0..4 {
        let c = counter.clone();
        let h = thread::spawn(move || {
            for _ in 0..1_000_000 {
                c.fetch_add(1, Ordering::Relaxed);
                // Relaxed 足够因为：
                // 1. 单次 fetch_add 是原子的（不会丢更新）
                // 2. 我们不关心线程之间的递增顺序
            }
        });
        handles.push(h);
    }

    for h in handles {
        h.join().unwrap();
    }

    assert_eq!(counter.load(Ordering::Relaxed), 4_000_000);
    println!("Final count: {}", counter.load(Ordering::Relaxed));
}
```

### 5.2 AtomicBool —— 最轻量的线程安全标志位

`AtomicBool` 是一个字节大小的原子类型，专门用于线程间的布尔标志传递。它是最简单的原子类型，也是最常用的。

```rust
use std::sync::atomic::{AtomicBool, Ordering};
use std::sync::Arc;
use std::thread;
use std::time::Duration;

// AtomicBool 常用模式：停止标志
fn stop_flag_pattern() {
    let running = Arc::new(AtomicBool::new(true));
    let r = running.clone();

    // 工作线程：循环直到收到停止信号
    let worker = thread::spawn(move || {
        let mut count = 0u64;
        while r.load(Ordering::Acquire) {  // Acquire 保证能看到主线程的"清理数据"
            count += 1;
            // 做一些工作...
            if count % 10_000_000 == 0 {
                println!("Worker: {}", count);
            }
        }
        // Release-Acquire 配对保证：
        // 主线程在 store(false, Release) 之前的操作在这里可见
        println!("Worker stopping at count: {}", count);
    });

    // 主线程：一段时间后发出停止信号
    thread::sleep(Duration::from_millis(100));
    // 如果需要在停止前做清理工作，在 store 之前做
    running.store(false, Ordering::Release);  // Release 发布停止信号
    worker.join().unwrap();
}

// AtomicBool 对比 Mutex<bool> 的优势：
// - 零系统调用开销
// - 不会阻塞（load 和 store 都是纯 CPU 指令）
// - 没有死锁风险
// - 内存占用仅 1 字节
```

`AtomicBool` 的 `load` 和 `store` 方法是其最核心的操作。`swap`、`compare_exchange` 等方法也同样是原子的，适用于更复杂的并发控制逻辑。

### 5.3 AtomicPtr —— 无锁数据结构的基础

`AtomicPtr<T>` 提供了对裸指针的原子操作，是构建无锁数据结构（如链表、栈、队列）的基础积木。

```rust
use std::sync::atomic::{AtomicPtr, Ordering};
use std::ptr;

// AtomicPtr 的核心操作
fn atomic_ptr_operations() {
    let mut value = 42i32;
    let ptr = &mut value as *mut i32;

    let atomic_ptr = AtomicPtr::new(ptr);

    // load —— 原子读取指针
    let loaded = atomic_ptr.load(Ordering::Acquire);
    unsafe {
        assert_eq!(*loaded, 42);
    }

    // store —— 原子写入指针
    let mut new_value = 100i32;
    let new_ptr = &mut new_value as *mut i32;
    atomic_ptr.store(new_ptr, Ordering::Release);

    // swap —— 原子交换
    let dead_ptr: *mut i32 = unsafe { std::mem::transmute(0xDEAD_BEEF_usize) };
    let old = atomic_ptr.swap(dead_ptr, Ordering::AcqRel);
    // old 是之前的 new_ptr

    // compare_exchange —— CAS（Compare-And-Swap）
    // 如果当前值等于 expected，则替换为 new 并返回 Ok
    // 否则返回 Err（当前实际值）
    let mut new_data = 200i32;
    let new = &mut new_data as *mut i32;
    let expected = dead_ptr;

    match atomic_ptr.compare_exchange(
        expected,
        new,
        Ordering::AcqRel,
        Ordering::Acquire,
    ) {
        Ok(_old) => println!("CAS succeeded!"),
        Err(actual) => println!("CAS failed, actual value: {:?}", actual),
    }
}
```

`AtomicPtr` 是构建无锁数据结构的核心基础类型。它的 `compare_exchange` 方法允许你原子地"如果指针还是那个值，就更新为新的值"——这是实现无锁栈、无锁队列等数据结构的 CAS 循环（一种典型的无锁算法模式）的关键。

## 6. compare_exchange —— CAS 循环的核心

### 6.1 compare_exchange 和 compare_exchange_weak 的区别

`compare_exchange` 是 CAS（Compare-And-Swap）的 Rust 实现。它有两个版本：`compare_exchange`（强）和 `compare_exchange_weak`（弱）。两者的语义几乎相同，但弱版本允许"伪失败"（spurious failure）。

```rust
use std::sync::atomic::{AtomicUsize, Ordering};

fn cas_demo() {
    let atomic = AtomicUsize::new(0);

    // compare_exchange(
    //     current: usize,    // 期望的当前值
    //     new: usize,        // 要写入的新值
    //     success: Ordering, // CAS 成功时使用的内存顺序
    //     failure: Ordering, // CAS 失败时使用的内存顺序
    // ) -> Result<usize, usize>
    //
    // Ok(old_value)   → CAS 成功：当前值 == current，已更新为 new
    // Err(actual)     → CAS 失败：当前值 != current，返回实际值

    // 成功的情况：
    let result = atomic.compare_exchange(0, 42, Ordering::AcqRel, Ordering::Acquire);
    assert_eq!(result, Ok(0));  // 旧值是 0，现在变成了 42
    assert_eq!(atomic.load(Ordering::Relaxed), 42);

    // 失败的情况：
    let result = atomic.compare_exchange(0, 100, Ordering::AcqRel, Ordering::Acquire);
    assert_eq!(result, Err(42));  // 当前值是 42，不是 0

    // ===== compare_exchange_weak —— 弱版本 =====
    // 弱版本允许"伪失败"：
    // 即使 atomic == expected，也可能返回 Err(expected)
    // ——因为某些 CPU（如 ARM）的 LL/SC 指令很容易被中断
    //
    // 伪失败的本质：
    //   ARM 使用 LDREX（Load Exclusive）和 STREX（Store Exclusive）
    //   在 LDREX 和 STREX 之间，如果缓存行的独占状态被打破
    //   （例如发生了上下文切换或中断），STREX 就会失败
    //   ——这就是"伪失败"

    // 使用 compare_exchange_weak 的惯用模式：CAS 循环
    fn increment(atomic: &AtomicUsize) -> usize {
        loop {
            let current = atomic.load(Ordering::Relaxed);
            let new = current + 1;
            match atomic.compare_exchange_weak(current, new, Ordering::AcqRel, Ordering::Relaxed) {
                Ok(_) => return new,       // 成功
                Err(_) => continue,        // 失败（可能是伪失败）→ 重试
            }
        }
    }
    // weak 版本在 x86 上几乎不会伪失败（x86 使用硬件级 CAS）
    // 在 ARM 上需要循环——但 LL/SC 循环的 weak 模式效率更高
}

// 经验法则：
// - compare_exchange_weak → 你在 CAS 循环中使用（几乎总是如此）
//   因为循环本身就能处理伪失败（continue），所以 weak 版本效率更高
// - compare_exchange（强版本）→ 你只需要一次 CAS，不想处理伪失败
//   内部可能是一个包含 weak CAS 的循环——效率略低但更简洁
```

`compare_exchange_weak` 在实践中使用频率远高于强版本，因为 CAS 操作几乎总是出现在循环中——这也是所有无锁算法的基本结构："读取、计算新值、尝试 CAS，失败就重试。"

### 6.2 CAS 循环 —— 无锁编程的标准模板

CAS 循环是实现无锁原子更新的标准范式。它遵循一个固定的结构。

```rust
use std::sync::atomic::{AtomicUsize, Ordering};

// CAS 循环的标准模板：
// 1. load  —— 读取当前值
// 2. compute —— 基于当前值计算新值
// 3. cas —— 如果值没有变化，原子地替换为新值
// 4. retry —— 如果值被其他线程改了，回到步骤 1

fn cas_loop_template(atomic: &AtomicUsize) {
    let mut attempts = 0;

    loop {
        // 步骤 1：读取当前值
        let current = atomic.load(Ordering::Relaxed);

        // 步骤 2：计算新值
        // 这里可以是任意复杂的计算
        // 只要它仅依赖 current 和不变量
        let new = current * 2 + 1;

        // 步骤 3：尝试 CAS
        match atomic.compare_exchange_weak(
            current,
            new,
            Ordering::AcqRel,  // 成功时的顺序
            Ordering::Relaxed,  // 失败时的顺序
        ) {
            Ok(_) => break,     // 成功！
            Err(_) => {
                attempts += 1;
                // 继续循环——current 已经被其他线程修改
            }
        }
    }

    println!("CAS succeeded after {} attempts", attempts);
}

// 这个模板的威力在于：
// - 无需锁（lock-free）
// - 不会死锁
// - 一个线程的阻塞不会影响其他线程的进展
// - 相比 Mutex，在低竞争下延迟更低

// CAS 循环的劣势：
// - 高竞争下可能导致大量重试（性能退化）
// - ABA 问题（见下文）
// - 复杂逻辑容易写出 bug
```

## 7. 构建一个无锁并发栈 —— 从理论到实战

### 7.1 设计无锁栈

我们将使用 `AtomicPtr` 和 CAS 循环构建一个简单的无锁并发栈（Treiber stack），这是无锁数据结构的"Hello World"。

```rust
use std::sync::atomic::{AtomicPtr, Ordering};
use std::ptr;

/// 无锁栈（Treiber stack）的节点
struct Node<T> {
    value: T,
    next: *mut Node<T>,
}

/// 无锁并发栈
/// 
/// 线程安全：所有操作都通过 CAS 循环保证原子性
/// 不需要 Mutex 或 RwLock
pub struct LockFreeStack<T> {
    head: AtomicPtr<Node<T>>,
}

impl<T> LockFreeStack<T> {
    pub fn new() -> Self {
        LockFreeStack {
            head: AtomicPtr::new(ptr::null_mut()),
        }
    }

    /// 将值推入栈顶
    pub fn push(&self, value: T) {
        // 1. 分配新节点
        let new_node = Box::into_raw(Box::new(Node {
            value,
            next: ptr::null_mut(),
        }));

        // 2. CAS 循环：将新节点设置为新的栈顶
        loop {
            // 读取当前栈顶（old_head）
            let old_head = self.head.load(Ordering::Acquire);

            // 新节点的 next 指向旧栈顶
            unsafe {
                (*new_node).next = old_head;
            }

            // 尝试 CAS：如果 head 仍然是 old_head，更新为 new_node
            match self.head.compare_exchange_weak(
                old_head,
                new_node,
                Ordering::Release,  // 成功：Release 发布新节点
                Ordering::Relaxed,   // 失败：重新读取
            ) {
                Ok(_) => return,  // 推入成功！
                Err(_) => {
                    // head 被其他线程修改了，重试
                    // new_node 的 next 需要更新为新的 head
                    continue;
                }
            }
        }
    }

    /// 从栈顶弹出值
    pub fn pop(&self) -> Option<T> {
        loop {
            // 读取当前栈顶
            let old_head = self.head.load(Ordering::Acquire);

            if old_head.is_null() {
                return None;  // 栈空
            }

            // 读取下一个节点
            let next = unsafe { (*old_head).next };

            // 尝试 CAS：如果 head 仍然是 old_head，更新为 next
            match self.head.compare_exchange_weak(
                old_head,
                next,
                Ordering::Acquire,  // 成功：Acquire 获取数据
                Ordering::Relaxed,
            ) {
                Ok(_) => {
                    // 成功弹出！需要释放旧节点的内存
                    // 这里有一个重要的并发问题——见下文
                    let value = unsafe {
                        // 读取值并释放节点
                        let node = Box::from_raw(old_head);
                        node.value
                    };
                    return Some(value);
                }
                Err(_) => {
                    // head 被其他线程修改了，重试
                    continue;
                }
            }
        }
    }
}

// 使用示例：
// let stack = LockFreeStack::new();
//
// // 多线程推入
// stack.push(1);
// stack.push(2);
//
// // 多线程弹出
// assert_eq!(stack.pop(), Some(2));
// assert_eq!(stack.pop(), Some(1));
// assert_eq!(stack.pop(), None);
```

### 7.2 无锁栈的内存回收 —— ABA 问题与解决方案

上面无锁栈的实现有一个严重的内存回收问题：在 `pop()` 中，使用 `Box::from_raw` 释放节点时，其他线程可能仍然持有指向该节点的指针。这是 ABA 问题的变种。

```rust
use std::sync::atomic::{AtomicPtr, Ordering};
use std::ptr;

//
// pop 中我们做了：
//   1. 读 old_head = 节点 A
//   2. 读 next = A->next = 节点 B
//   3. CAS(old_head=A→next=B) 成功
//   4. 释放 A 的内存
//
// 在步骤 3 和 4 之间，另一个线程可能：
//   1. pop 得到节点 A（还没有释放）
//   2. pop 得到节点 B

//   此时 head 又指向了 A，但 A 的内容已经变了

// 这是无锁编程中最棘手的问题之一

// 解决方案通常依赖：
// 1. epoch-based reclamation（crossbeam-epoch）
//    —— 分代回收，延迟释放
// 2. hazard pointers
//    —— 显式标记"我正在用这个指针"
// 3. RCU（Read-Copy-Update）
//    —— 读取者无锁，写入者等待所有读取者完成后回收
// 4. 引用计数 + 原子操作（Arc 的内部机制）
//
// crossbeam 的 TreiberStack 使用 epoch-based reclamation
// —— 我们在此不展开，因为那是更高级的话题

// 概念性的安全 pop 实现（使用 crossbeam-epoch）：
// 在 Cargo.toml 中添加：crossbeam-epoch = "0.9"
//
// use crossbeam_epoch::{Atomic, Owned, Shared, Guard, pin};

// fn safe_pop<T>(head: &Atomic<Node<T>>) -> Option<T> {

//         let old_head = head.load(Ordering::Acquire, &guard);
```

## 8. CPU 内存模型可视化

### 8.1 x86 TSO（Total Store Order）模型

不同的 CPU 架构有不同的内存模型，这对理解内存顺序至关重要。x86 使用 TSO——在所有主流架构中属于相对强的模型。

```
x86 TSO 内存模型（Total Store Order）

┌─────────────────────────────────────────────────────────────┐
│                        CPU Core 1                           │
│  ┌─────────────────┐      ┌─────────────────┐              │
│  │ Store Buffer    │      │ Store Buffer    │              │
│  │ (先写到这，然后   │      │ (先写到这，然后   │              │
│  │  异步刷到L1)     │      │  异步刷到L1)     │              │
│  └────────┬────────┘      └────────┬────────┘              │
│           │                        │                        │
│  ┌────────▼────────────────────────▼────────┐              │
│  │              L1 Cache（本核私有）           │              │
│  └──────────────────┬───────────────────────┘              │
└─────────────────────┼──────────────────────────────────────┘
                      │
        ┌─────────────▼─────────────┐
        │    L2 / L3 Cache (共享)    │
        │    缓存一致性协议 (MESI)    │
        └─────────────┬─────────────┘
                      │
        ┌─────────────▼─────────────┐
        │       主内存 (RAM)         │
        └───────────────────────────┘

x86 TSO 有三个关键特性：

1. 写入 (Store) 不直接到 L1——
   而是先进入 Store Buffer，再异步刷到缓存
   这就是为什么一个核心写入的值，其他核心不一定立即能看到

2. Store-Load 重排被允许——
   store 还在 store buffer 中时，后续的 load 可以先执行
   编译器也可能重排无关的指令

3. Store-Store 和 Load-Load 在 x86 上天然有序——
   x86 的 TSO 模型已经提供了"acquire load"和"release store"语义
   因此 Acquire/Release 在 x86 上几乎是零成本（不需要额外屏障指令）
   但 SeqCst store 需要 MFENCE——有性能代价
```

### 8.2 ARM Weak Memory Model

ARM 处理器使用弱内存模型，比 x86 的 TSO 更弱。这意味着在 ARM 上几乎所有的重排都需要显式的屏障指令。

```
ARM Weak Memory Model

与 x86 TSO 的关键区别：

1. 几乎所有重排都可能发生：
   - Store-Store 重排 ✅ 可能（x86 不允许）
   - Load-Load 重排   ✅ 可能（x86 不允许）
   - Store-Load 重排  ✅ 可能（x86 也允许）
   - Load-Store 重排  ✅ 可能（x86 不允许）

2. 需要显式的屏障指令：
   - DMB (Data Memory Barrier) —— 内存屏障
   - DSB (Data Synchronization Barrier) —— 比 DMB 更强
   - ISB (Instruction Synchronization Barrier) —— 指令同步

3. 原子操作用 LL/SC (Load-Linked/Store-Conditional) 实现：
   - LDREX: Load Exclusive（标记"我关注这个地址"）
   - STREX: Store Exclusive（仅当"独占标记"还在时才写入）
   - 这导致 compare_exchange_weak 在 ARM 上容易伪失败

对 Rust 开发者的实际影响：
- 在 x86 上，Acquire load 和 Release store 几乎没有性能影响
- 在 ARM 上，Acquire/Release 需要 DMB 指令（有成本）
- 因此在 ARM 上选择正确 Ordering 对性能影响更大
- Relaxed 在 ARM 上确实比 Acquire/Release 快得多
```

理解 CPU 内存模型对正确使用原子操作非常重要。在 x86 上你可能"侥幸"地用宽松的顺序也能得到正确的结果（因为 TSO 已经提供了很多隐式保证），但在 ARM 上底层排序更激进，不正确的 Ordering 就会暴露出问题。这就是为什么在生产代码中正确选择 `Ordering` 很重要——你不能依赖特定平台的隐性保证。

## 9. 选择 Ordering 的实战指南

### 9.1 决策树 —— 我该用什么顺序

```
我需要原子操作 + 内存顺序保证

问：多个操作之间有"先后依赖"吗？
├── 没有 → Relaxed
│         例：统计计数器、唯一 ID 生成
│         只要计数不丢失，不关心"哪个线程先递增"
│
└── 有 → 继续往下判断

    问：是经典的"生产者-消费者"或"锁"模式吗？
    ├── 是 → Acquire/Release 配对
    │        - 生产者：写数据 Relaxed，最后一步 Release
    │        - 消费者：Acquire 读标志，然后 Relaxed 读数据
    │        - 锁：lock() = Acquire, unlock() = Release
    │
    └── 否 → 继续判断

        问：需要多个 SeqCst 原子操作在所有线程中保持全局统一顺序吗？
        ├── 是 → SeqCst
        │        极其罕见的场景
        │        例：分布式共识算法中的"统一时钟"
        │
        └── 否 → 重新审视设计
                可能你对"先后依赖"的理解有偏差
                99% 的情况 Relaxed 或 Acquire/Release 就够了
```

### 9.2 常见模式的 Ordering 速查

以下是一组在实际编程中最常见的并发模式及其对应的 `Ordering` 选择。

```rust
use std::sync::atomic::{AtomicBool, AtomicUsize, Ordering};

// 模式 1：统计计数器 → Relaxed
fn stats_counter() {
    static COUNTER: AtomicUsize = AtomicUsize::new(0);
    COUNTER.fetch_add(1, Ordering::Relaxed);  // 只需要原子性
    let _ = COUNTER.load(Ordering::Relaxed);   // 只需要读到一个"最终正确"的值
}

// 模式 2：flag + data → Acquire/Release
fn flag_data() {
    static DATA: AtomicUsize = AtomicUsize::new(0);
    static READY: AtomicBool = AtomicBool::new(false);

    // 写入者：
    DATA.store(42, Ordering::Relaxed);
    READY.store(true, Ordering::Release);  // ← Release 发布

    // 读取者：
    if READY.load(Ordering::Acquire) {     // ← Acquire 获取
        let val = DATA.load(Ordering::Relaxed);  // ← 保证看到 42
    }
}

// 模式 3：互斥锁（概念实现）→ Acquire/Release
fn spin_lock() {
    static LOCKED: AtomicBool = AtomicBool::new(false);

    // lock()  → Acquire
    while LOCKED.swap(true, Ordering::Acquire) {
        // spin...
        std::hint::spin_loop();
    }
    // 临界区代码...（在 Acquire 之后，能看到之前持有者的修改）

    // unlock() → Release
    LOCKED.store(false, Ordering::Release);  // 发布临界区的修改
}

// 模式 4：唯一 ID 生成器 → Relaxed
fn unique_id_generator() {
    static NEXT_ID: AtomicUsize = AtomicUsize::new(0);
    // 不关心 ID 分配的"顺序"
    // 只关心每个线程拿到不同的 ID
    let id = NEXT_ID.fetch_add(1, Ordering::Relaxed);
}

// 模式 5：懒初始化（单次写入）→ Acquire/Release
fn lazy_init() {
    use std::sync::atomic::AtomicPtr;

    static DATA: AtomicPtr<String> = AtomicPtr::new(std::ptr::null_mut());

    // get_or_init:
    let ptr = DATA.load(Ordering::Acquire);
    if !ptr.is_null() {
        return;  // 已经初始化，Acquire 保证能看到数据
    }

    // 未初始化，创建数据
    let new_data = Box::into_raw(Box::new(String::from("initialized")));

    match DATA.compare_exchange(
        std::ptr::null_mut(),
        new_data,
        Ordering::Release,  // 成功：Release 发布数据
        Ordering::Acquire,  // 失败：Acquire 读取其他线程的数据
    ) {
        Ok(_) => {}  // 初始化成功
        Err(_) => {
            // 其他线程抢先初始化了
            unsafe { drop(Box::from_raw(new_data)); }
        }
    }
}

// 模式 6：引用计数加减 → Acquire/Release
// Arc 内部实现中：
//   clone(): fetch_add(1, Relaxed)  —— 只增加计数
//   drop():  fetch_sub(1, Release)  —— 可能需要析构，Release 保证可见性
//   如果在 drop 中减到 0：需要 Acquire 来看到其他线程的修改
```

## 10. 常见陷阱

### 10.1 陷阱 1：所有原子用 SeqCst（性能杀手）

```rust
use std::sync::atomic::{AtomicUsize, Ordering};

// ❌ 懒人写法：所有操作都用 SeqCst
fn lazy_all_seqcst(counter: &AtomicUsize) {
    // 每次递增都触发完整的内存屏障
    for _ in 0..1_000_000 {
        counter.fetch_add(1, Ordering::SeqCst);  // 慢！
    }
}

// ✅ 分析需要什么顺序
fn optimized_ordering(counter: &AtomicUsize) {
    // 这只是个统计计数器——不需要顺序保证
    for _ in 0..1_000_000 {
        counter.fetch_add(1, Ordering::Relaxed);  // 快！
    }
}

// 在 x86 上，SeqCst store 的代价大约是 Relaxed store 的 2-10 倍
// 在 ARM 上差距更大
// 过度使用 SeqCst 会显著降低性能
```

### 10.2 陷阱 2：Relaxed 用在有依赖的操作上

```rust
use std::sync::atomic::{AtomicBool, AtomicUsize, Ordering};
use std::thread;

// ❌ 错误：Relaxed 用于 flag-data 模式
fn broken_flag_data() {
    static DATA: AtomicUsize = AtomicUsize::new(0);
    static READY: AtomicBool = AtomicBool::new(false);

    let t1 = thread::spawn(|| {
        DATA.store(42, Ordering::Relaxed);
        READY.store(true, Ordering::Relaxed);  // ❌ 应该是 Release
    });

    let t2 = thread::spawn(|| {
        if READY.load(Ordering::Relaxed) {  // ❌ 应该是 Acquire
            let val = DATA.load(Ordering::Relaxed);
            // 可能读取到 0！——在 ARM 上特别容易触发
            assert_eq!(val, 42);  // ❌ 可能失败
        }
    });

    t1.join().unwrap();
    t2.join().unwrap();
}
// 要点：只要操作之间有"先后依赖"，就需要 Release/Acquire 或更强的顺序
```

### 10.3 陷阱 3：CAS 循环中忘记重新读取

```rust
use std::sync::atomic::{AtomicUsize, Ordering};

// ❌ 错误：CAS 失败后使用旧的 current 值
fn broken_cas(atomic: &AtomicUsize) {
    let current = atomic.load(Ordering::Relaxed);  // 循环外读取

    // ...其他代码...

    loop {
        let new = current + 1;
        match atomic.compare_exchange_weak(current, new, Ordering::AcqRel, Ordering::Relaxed) {
            Ok(_) => return,  // 成功
            Err(actual) => {
                // ❌ 没有更新 current！
                // 下次循环会使用同样的 current 值，几乎必定再次失败
                // actual 包含了当前的实际值，应该用它更新 current
            }
        }
    }
}

// ✅ 正确：from Err 的 actual 值中获取最新值
fn correct_cas(atomic: &AtomicUsize) {
    let mut current = atomic.load(Ordering::Relaxed);

    loop {
        let new = current + 1;
        match atomic.compare_exchange_weak(current, new, Ordering::AcqRel, Ordering::Relaxed) {
            Ok(_) => return,        // 成功
            Err(actual) => {
                current = actual;   // ✅ 更新为最新值
                continue;
            }
        }
    }
}
```

### 练习 1 答案

```rust
use std::sync::atomic::{AtomicBool, Ordering};
use std::cell::UnsafeCell;
use std::ops::{Deref, DerefMut};

pub struct SpinLock<T> {
    locked: AtomicBool,
    data: UnsafeCell<T>,
}

// SpinLock<T>: Send 当 T: Send
// SpinLock<T>: Sync 当 T: Send
// —— 因为 lock() 提供互斥访问
unsafe impl<T: Send> Send for SpinLock<T> {}
unsafe impl<T: Send> Sync for SpinLock<T> {}

impl<T> SpinLock<T> {
    pub fn new(data: T) -> Self {
        SpinLock {
            locked: AtomicBool::new(false),
            data: UnsafeCell::new(data),
        }
    }

    /// 获取锁。如果锁已被占用，自旋等待。
    pub fn lock(&self) -> SpinLockGuard<'_, T> {
        // 尝试将 false 换为 true
        // Acquire 语义：lock 成功后，能看到上一个持有者的 Release 写入
        while self.locked.swap(true, Ordering::Acquire) {
            // 自旋等待——提示 CPU 我们在做自旋
            std::hint::spin_loop();
        }
        SpinLockGuard { lock: self }
    }

    /// 释放锁——由 SpinLockGuard 的 Drop 调用
    fn unlock(&self) {
        // Release 语义：临界区中的修改对所有后续 Acquire 可见
        self.locked.store(false, Ordering::Release);
    }
}

pub struct SpinLockGuard<'a, T> {
    lock: &'a SpinLock<T>,
}

impl<T> Deref for SpinLockGuard<'_, T> {
    type Target = T;
    fn deref(&self) -> &T {
        unsafe { &*self.lock.data.get() }
    }
}

impl<T> DerefMut for SpinLockGuard<'_, T> {
    fn deref_mut(&mut self) -> &mut T {
        unsafe { &mut *self.lock.data.get() }
    }
}

impl<T> Drop for SpinLockGuard<'_, T> {
    fn drop(&mut self) {
        self.lock.unlock();
    }
}
```

### 练习 3 答案

```rust
use std::sync::atomic::{AtomicI32, Ordering};

fn fetch_max(atomic: &AtomicI32, val: i32) -> i32 {
    let mut current = atomic.load(Ordering::Relaxed);

    loop {
        // 如果当前值已经 >= val，不需要更新
        if current >= val {
            return current;
        }

        // 尝试 CAS：如果 current 没变，更新为 val
        match atomic.compare_exchange_weak(
            current,
            val,
            Ordering::AcqRel,
            Ordering::Relaxed,
        ) {
            Ok(old) => return old,      // 成功更新
            Err(actual) => {
                current = actual;       // 被其他线程修改了，重试
                // 重新检查 actual 是否已经 >= val
            }
        }
    }
}

// 测试：
// let a = AtomicI32::new(0);
// fetch_max(&a, 10);  // → 0, a = 10
// fetch_max(&a, 5);   // → 10, a = 10 (因为 10 > 5)
// fetch_max(&a, 20);  // → 10, a = 20
```

### 练习 4 答案

```rust
use std::sync::atomic::{AtomicBool, AtomicUsize, Ordering};

// 原始代码的问题：
// FLAG.store(true, Relaxed) 和 FLAG.load(Relaxed)
// 没有 Release/Acquire 配对
// → DATA1 和 DATA2 的写入可能对 reader 不可见

// ✅ 修复方案：
static DATA1: AtomicUsize = AtomicUsize::new(0);
static DATA2: AtomicUsize = AtomicUsize::new(0);
static FLAG: AtomicBool = AtomicBool::new(false);

fn writer_fixed() {
    DATA1.store(10, Ordering::Relaxed);   // Relaxed 足够——FLAG 的 Release 会保证可见性
    DATA2.store(20, Ordering::Relaxed);   // 同理
    FLAG.store(true, Ordering::Release);  // ✅ Release——"发布"之前的全部写入
}

fn reader_fixed() {
    if FLAG.load(Ordering::Acquire) {     // ✅ Acquire——"获取"写入者的发布
        // 此时保证能看到 writer 在 Release 之前的所有写入
        let d1 = DATA1.load(Ordering::Relaxed);  // ✅ 保证读到 10
        let d2 = DATA2.load(Ordering::Relaxed);  // ✅ 保证读到 20
        assert_eq!(d1, 10);
        assert_eq!(d2, 20);
    }
}

// 原理：
// 在 writer 中：
//   (1) DATA1.store(10, Relaxed)  happens-before → (2) 程序顺序
//   (2) DATA2.store(20, Relaxed)  happens-before → (3) 程序顺序
//   (3) FLAG.store(true, Release)  —— 发布点
//
// 在 reader 中：
//   (4) FLAG.load(Acquire)  —— 获取点
//     如果看到了 (3) 写入的 true，则 (3) synchronizes-with (4)
//   → (1) 和 (2) happens-before (4) （传递性）
//   → (4) happens-before (5) 和 (6) （程序顺序）
//   → (1) 和 (2) happens-before (5) 和 (6)
//   → reader 必定看到 DATA1=10, DATA2=20
```
