# 3.1 Send & Sync 深度 —— 类型系统的线程安全基石

> Rust 不依赖运行时（GC 或全局锁）来保证线程安全，而是在**编译期通过两个自动 trait——`Send` 和 `Sync`——对整个类型系统做"线程安全分类"**。`Send` 回答"这个值能否被移动到另一个线程"，`Sync` 回答"这个值的共享引用能否被多个线程同时使用"。编译器在 `std::thread::spawn` 的闭包边界、`Arc` 的约束、`MutexGuard` 的析构等关键节点强制执行这两个 trait。理解 `Send`/`Sync` 的自动推导规则、负面实现（negative impl）的用途、以及它们如何与内部可变性类型（`Cell`/`RefCell`）和同步原语（`Mutex`/`RwLock`/`Arc`）交互，是编写正确多线程 Rust 代码的理论根基。

---

## 1. 什么是 Send 和 Sync —— 直觉与定义

### 1.1 两个 trait 的一句话定义

`Send` 和 `Sync` 是 Rust 标准库中定义在 `std::marker` 模块中的两个**自动 trait**（auto trait）。它们的定义极其简短，但影响极其深远。

```rust
// std::marker 中的定义（简化版）：
// pub unsafe auto trait Send { }
// pub unsafe auto trait Sync { }

// 两个 trait 都是 unsafe trait——
// 实现它们需要程序员保证特定的安全属性
// 但编译器会自动为大多数类型推导实现
```

这两个 trait 各自回答一个关于"跨线程传递"的问题。`Send` 关心的是"所有权转移"——一个 `T` 类型的值能否被安全地从线程 A 移动到线程 B。`Sync` 关心的是"共享引用"——一个 `&T` 类型的引用能否被多个线程同时持有和使用。两者共同构成了 Rust 编译期线程安全检查的全部基础。

### 1.2 用一句话记住 Send 和 Sync

理解这两个 trait 最简洁的方式是记住两句英文定义：

```rust
// Send:  "A type is Send if it is safe to send it to another thread."
//        类型可以安全地将所有权转移到另一个线程

// Sync:  "A type is Sync if it is safe to share a reference to it between threads."
//        类型的共享引用可以被安全地在多个线程之间共享

// 关键推论：
// T: Sync  等价于  &T: Send
// ——"能共享引用" ⇔ "共享引用本身能被发送到别的线程"
```

这个等价关系（`T: Sync` iff `&T: Send`）不仅是定义，也是编译器推导规则的一部分。当你看到某个类型是 `Sync`，可以理解为"这个类型的共享引用本身是 `Send` 的"。

## 2. Send 详解 —— 所有权跨线程转移的安全性

### 2.1 哪些类型天生是 Send

绝大多数 Rust 基本类型和标准库类型都是 `Send`。编译器会自动推导实现，因为它会检查类型的所有字段是否都是 `Send`。

```rust
use std::marker::Send;
use std::thread;

// 基本类型都是 Send
fn assert_send() {
    fn is_send<T: Send>() {}

    is_send::<i32>();        // ✅
    is_send::<bool>();       // ✅
    is_send::<String>();     // ✅
    is_send::<Vec<i32>>();   // ✅
    is_send::<Box<i32>>();   // ✅
    is_send::<&str>();       // ✅ —— &str 是 Send！（引用本身是值）
    is_send::<Option<i32>>();// ✅
    is_send::<Result<i32, String>>(); // ✅
    is_send::<HashMap<String, i32>>(); // ✅
}

// 编译器通过"结构体推导"来决定自定义类型是否为 Send：
// 一个 struct 是 Send 当且仅当其所有字段都是 Send
struct MyStruct {
    a: i32,
    b: String,
    c: Vec<f64>,
}
// MyStruct: Send ✅ —— 因为 i32、String、Vec<f64> 都是 Send
```

编译器自动推导 `Send` 时采用的是"积木原则"：一个复合类型是 `Send` 当且仅当它包含的每一个字段都是 `Send`。如果有一个字段不是 `Send`，整个复合类型就不再是 `Send`——传染性传播。

### 2.2 哪些类型不是 Send —— 典型反例

不是 `Send` 的类型通常是那些"依赖线程本地状态"或"设计上只允许单线程使用"的类型。

```rust
use std::rc::Rc;
use std::cell::Cell;
use std::marker::PhantomData;
use std::thread;

// Rc<T> 不是 Send —— 最经典的非 Send 类型
// Rc<T> 的引用计数使用非原子操作（普通的整数加减）
// 如果两个线程同时 clone/drop Rc，会产生数据竞争
fn rc_not_send() {
    fn is_send<T: Send>() {}
    // is_send::<Rc<i32>>();  // ❌ Rc<i32> 没有实现 Send
}

// 裸指针不是 Send —— 编译器无法判断所指向的内存是否线程安全
fn raw_pointer_not_send() {
    fn is_send<T: Send>() {}
    // is_send::<*const i32>();  // ❌ *const T 没有实现 Send
    // is_send::<*mut i32>();    // ❌ *mut T 没有实现 Send
    // 裸指针不实现 Send 是为了防止你无意中把
    // 一个指向线程本地数据的指针传到另一个线程
}

// Cell<T> 是 Send（当 T: Send 时）但不是 Sync
// 它允许移动到另一个线程，但不允许多个线程共享对它的引用
fn cell_send_but_not_sync() {
    // Cell<i32>: Send ✅ —— 可以移动到另一个线程
    // &Cell<i32>: !Send —— Cell 不是 Sync，因此 &Cell<i32> 不是 Send
}

// thread::spawn 会强制要求闭包捕获的值是 Send
fn spawn_requires_send() {
    let rc = Rc::new(42);
    // thread::spawn(move || {
    //     println!("{}", rc);  // ❌ Rc<i32> is not Send
    // });
    // 编译器错误信息非常清晰：
    // error[E0277]: `Rc<i32>` cannot be sent between threads safely
    //     = help: the trait `Send` is not implemented for `Rc<i32>`
}
```

理解"为什么某些类型不是 `Send`"比记住"哪些类型是 `Send`"更重要。核心原则：如果一个类型的设计假设了"只会被一个线程使用"（如 `Rc` 的非原子引用计数），或者它的安全性无法在编译期验证（如裸指针），那它就不应该实现 `Send`。

### 2.3 Send 和线程 spawn 的交互 —— 编译器如何强制执行

`std::thread::spawn` 的闭包要求 `Send + 'static`。编译器在闭包边界检查这些约束。

```rust
use std::thread;

// spawn 的签名（简化）：
// pub fn spawn<F, T>(f: F) -> JoinHandle<T>
// where
//     F: FnOnce() -> T,    // 闭包只能调用一次
//     F: Send + 'static,   // 闭包必须是 Send 且 'static
//     T: Send + 'static,   // 返回值也必须是 Send 且 'static
//
// 'static 约束要求：闭包中捕获的引用必须满足 'static 生命周期
// 这意味着不能引用局部变量（除非是 &'static 引用）

// 正确的 spawn 使用：
let data = vec![1, 2, 3];  // Vec<i32>: Send + 'static
let handle = thread::spawn(move || {
    let sum: i32 = data.iter().sum();
    sum  // 返回值 i32 也是 Send + 'static
});
let result = handle.join().unwrap();
// ✅ Vec<i32> 的所有权被移动到闭包 → 移动到新线程

// 错误的 spawn 使用：
fn bad_spawn() {
    let local = 42;
    // thread::spawn(|| {
    //     println!("{}", local);  // ❌ 引用生命周期不够 'static
    // });
    // 解决方案：move 闭包（复制/移动值而不是借用）
}
```

编译器在 `spawn` 的调用点自动检查闭包中所有捕获的变量是否实现 `Send`。这一检查完全在编译期完成，零运行时开销。

## 3. Sync 详解 —— 共享引用的跨线程安全性

### 3.1 Sync 的核心语义

`Sync` 回答的问题是：`&T` 能否被多个线程同时安全地访问？这里"安全"意味着不存在数据竞争——即不会有一个线程写而另一个线程同时读或写。

```rust
use std::marker::Sync;

// Sync 的定义（简化）：
// pub unsafe auto trait Sync { }
//
// 关键等价关系：T: Sync  iff  &T: Send
//
// Rust 编译器可以自动推导 Sync：
// 一个类型 T 是 Sync 当且仅当 &T 是 Send
// 即 T 的所有"通过引用可访问的"数据都支持线程安全共享

// 基本类型大多是 Sync：
fn assert_sync() {
    fn is_sync<T: Sync>() {}

    is_sync::<i32>();        // ✅
    is_sync::<bool>();       // ✅
    is_sync::<&str>();       // ✅
    is_sync::<String>();     // ✅ —— String 是不可变视角下线程安全的
    is_sync::<Vec<i32>>();   // ✅ —— 通过 &Vec<i32> 只能读，不能改
    is_sync::<Box<i32>>();   // ✅
    is_sync::<Option<i32>>();// ✅
}
```

`Vec<i32>` 是 `Sync` 的关键原因：通过 `&Vec<i32>` 你只能读取元素，不能添加、删除或修改。这就是 Rust 不可变引用天然线程安全的体现——如果没有内部可变性介入，共享即安全。

### 3.2 哪些类型不是 Sync —— 内部可变性类型

`Cell`、`RefCell`、`UnsafeCell` 这些内部可变性类型都不是 `Sync`。这是出于工程上的审慎判断。

```rust
use std::cell::{Cell, RefCell, UnsafeCell};

// Cell<T>:  !Sync —— 即使 T 是 Sync
// 原因：Cell::set() 是普通的内存写入（非原子操作）
// 两个线程同时 set 同一个 Cell → 数据竞争

fn cell_not_sync() {
    fn is_sync<T: Sync>() {}
    // is_sync::<Cell<i32>>();  // ❌ Cell<i32> 不是 Sync
    // is_sync::<RefCell<i32>>(); // ❌ RefCell<i32> 不是 Sync
    // is_sync::<UnsafeCell<i32>>(); // ❌ UnsafeCell<i32> 不是 Sync
}

// 这导致了重要的组合限制：
// Arc<Cell<T>> 不能通过 & 在多线程中共享 Cell
// ——因为 Cell 不是 Sync，Arc<T> 只在 T: Send + Sync 时才是 Sync
// 但 Thread Local 变量可以用 Cell/RefCell：
// thread_local! {
//     static COUNTER: Cell<i32> = Cell::new(0);  // ✅
// }
// 每个线程有自己的副本，不需要 Sync
```

`Mutex<T>` 通过使用原子操作来保证内部可变性的线程安全，所以 `Mutex<T>` 是 `Sync`（当 `T: Send` 时）。这是将内部可变性从"单线程安全"升级到"多线程安全"的标准方式。

### 3.3 Sync 与 Arc 的协作

`Arc<T>` 是多线程版 `Rc<T>`。它的 `Send`/`Sync` 实现取决于 `T`。

```rust
use std::sync::Arc;
use std::cell::Cell;

// Arc 的 Send/Sync 实现规则：
// Arc<T>: Send  当 T: Send + Sync   （可以把 Arc 移动到另一个线程）
// Arc<T>: Sync  当 T: Send + Sync   （可以通过 &Arc<T> 在多个线程间共享）

// 例子：
let a1: Arc<i32> = Arc::new(42);
// Arc<i32>: Send ✅  —— i32: Send + Sync
// Arc<i32>: Sync ✅  —— i32: Send + Sync

let a2: Arc<Cell<i32>> = Arc::new(Cell::new(0));
// Cell<i32>: Send ✅  （Cell 是 Send 当 T: Send）
// Cell<i32>: !Sync   （Cell 绝不是 Sync）
// Arc<Cell<i32>>: Send ✅  —— Cell<i32>: Send + !Sync → 不满足！
// 等等——检查条件：Arc<T>: Send 要求 T: Send + Sync
// Cell<i32>: Send 但不是 Sync → Arc<Cell<i32>>: !Send
// 这意味着你不能把 Arc<Cell<i32>> 移动到另一个线程！
// 这是正确的设计——Cell 不具备多线程安全能力

// 实践中的对应关系：
// 单线程共享可变：Rc<RefCell<T>>
// 多线程共享可变：Arc<Mutex<T>>
// 多线程只读：     Arc<T>
```

这是 Rust 线程安全设计中非常精妙的一环：`Arc` 通过对内部类型 `T` 施加 `Send + Sync` 约束，确保共享引用在跨线程场景中的安全性不会被绕过。即便你把一个单线程内部可变类型（如 `Cell`）放在 `Arc` 里，整个 `Arc` 就不再是 `Send`，从而根本无法被 `spawn` 的闭包捕获。

## 4. 自动 Trait 推导 —— 编译器如何决定 Send/Sync

### 4.1 自动 trait 的推导规则

`Send` 和 `Sync` 都是 **auto trait**（自动 trait）。编译器会在编译时为每个类型自动判断是否实现它们，而无需手动写出 `impl Send for MyType`。

```rust
// 自动推导规则（简化版）：

// 规则 1：基本类型由标准库显式实现
//   i32: Send + Sync（标准库有明确实现）
//   bool: Send + Sync
//   &T: Send（当 T: Sync）
//   &mut T: Send（当 T: Send）

// 规则 2：复合类型的自动推导
//   struct Foo<A, B> {
//       a: A,
//       b: B,
//   }
//   Foo<A, B>: Send  当 A: Send && B: Send
//   Foo<A, B>: Sync  当 A: Sync && B: Sync
//   ——所有字段都必须满足条件

// 规则 3：泛型类型——推导依赖于具体类型参数
//   Vec<T>: Send  当 T: Send
//   Vec<T>: Sync  当 T: Sync
//   Box<T>: Send  当 T: Send
//   Box<T>: Sync  当 T: Sync

// 规则 4：裸指针永远不自动实现 Send/Sync
//   *const T: !Send, !Sync
//   *mut T:   !Send, !Sync
//   但可以手动 unsafe impl（需确保所指数据线程安全）

// 规则 5：trait object 的处理
//   dyn Trait: Send  当 trait 本身要求 Send 且具体类型是 Send
//   dyn Trait + Send  —— 显式指定 trait object 的 Send 标记
```

自动推导的核心逻辑是"传染性"——一个复合类型是否 `Send`/`Sync` 取决于它包含的所有子类型。只要有一个字段不满足条件，整个类型就不满足。这保证了类型系统的"诚实"——你不能通过包装来隐藏非线程安全的内部结构。

### 4.2 PhantomData 对 Send/Sync 的影响

`PhantomData<T>` 是一块"幽灵内存"——它在运行时不存在，但在编译期的类型分析和 trait 推导中参与其中。

```rust
use std::marker::PhantomData;
use std::rc::Rc;

// PhantomData<T> 会被编译器当作一个 T 类型的字段来处理
// 对于 Send/Sync 的自动推导至关重要

struct Wrapper1<T> {
    _phantom: PhantomData<T>,
}
// Wrapper1<T>: Send  当 T: Send（受 phantom 影响）
// Wrapper1<T>: Sync  当 T: Sync（受 phantom 影响）

// 即使运行时没有 T 类型的值，编译器仍然根据 PhantomData<T>
// 推导 Wrapper 的 Send/Sync —— 这正是你想要的行为
// 因为 Wrapper 在逻辑上"拥有" T 类型的数据

// 对比：不含 PhantomData<T>
struct Wrapper2<T> {
    _marker: (),
}
// Wrapper2<T>: Send  ✅ —— 总是 Send（没有依赖 T 的字段）
// Wrapper2<T>: Sync  ✅ —— 总是 Sync
// 这可能不是你想要的行为——Wrapper2<*mut u8> 也是 Send/Sync
// 而裸指针不应该是

// 实践建议：如果你在逻辑上拥有 T 类型的数据，
// 务必加入 PhantomData<T> 来保持正确的 Send/Sync 推导
// 这在你创建"包含裸指针"的包装类型时尤其重要
```

`PhantomData<T>` 是控制自动 trait 推导的工具。当你的类型包含裸指针或通过 FFI 管理的外部资源时，正确使用 `PhantomData` 可以防止编译器错误地推导 `Send`/`Sync`。

## 5. 手动 unsafe impl —— 何时需要、如何正确使用

### 5.1 为什么需要手动 impl Send/Sync

绝大多数情况下，你不应该手动实现 `Send` 或 `Sync`——自动推导已经足够。但以下场景需要手动介入：

```rust
use std::marker::Send;

// 场景 1：包装裸指针（你的类型逻辑上是线程安全的）
// 编译器不会为包含 *const T / *mut T 的类型自动推导 Send
// 但如果你通过外部机制保证线程安全，可以手动 unsafe impl

// 场景 2：包装 FFI 类型（C 库的数据结构）
// C 库的很多类型实际上是线程安全的，但编译器不知道
// 你需要阅读 C 库文档，确认线程安全后手动实现

// 场景 3：你的类型需要负面实现（负面 trait impl）
// "我的类型绝不应该是 Send"——例如 FFI 绑定中的类型
// 显式的 impl !Send 告诉编译器和使用者这个保证

// 场景 4：Rc<T> 类自定义智能指针
// 如果你实现了自己的引用计数类型，需要手动决定 Send/Sync
```

手动 `unsafe impl Send` 本质上是向编译器承诺："我知道自动推导说不行，但我通过编程逻辑保证了线程安全，编译器你信我这一回。"

### 5.2 unsafe impl Send 的正确写法

手动实现 `Send`/`Sync` 时必须使用 `unsafe impl`，因为你要承担安全责任。

```rust
use std::marker::Send;
use std::sync::Mutex;

// 示例：一个包装裸指针的线程安全类型
struct ThreadSafeRawPointer<T> {
    ptr: *mut T,
    // 外部保证：这个指针指向的数据被 Mutex 保护
    _lock: Mutex<()>,  // 仅用于告知人类读者"有锁保护"
}

// 编译器自动推导：
// *mut T: !Send → ThreadSafeRawPointer<T>: !Send
// 但我们是设计上线程安全的（通过外部同步机制）
// → 需要手动 unsafe impl

unsafe impl<T: Send> Send for ThreadSafeRawPointer<T> {}
unsafe impl<T: Send> Sync for ThreadSafeRawPointer<T> {}
// 安全理由（程序员必须在注释或文档中说明）：
// ptr 指向的数据在 Mutex 的保护下，只有一个线程能在任何时刻访问
// 因此 Send 和 Sync 是安全的

// 不使用 unsafe impl 的错误示范：
// struct BadWrapper(*mut u8);
// unsafe impl Send for BadWrapper {}  // ⚠️ 如果没有外部同步保证，这很危险！
// 这样任意线程都可以拥有 BadWrapper 并随意读写裸指针——数据竞争
```

手动实现 `Send`/`Sync` 时，注释是**强制性的好习惯**——你必须记录"为什么这个类型实际上是线程安全的"，因为编译器无法帮你验证这个承诺。

### 5.3 负面 trait 实现 —— impl !Send for MyType

有时你需要**拒绝**自动推导——明确告诉编译器"我的类型绝不应该是 `Send` 或 `Sync`"。这在 FFI 场景中特别常见。

```rust
use std::marker::Send;
use std::marker::Sync;

// 假设一个 C 库提供了一个"不能跨线程使用"的数据结构
// 它在内部使用了线程本地存储（TLS）
mod ffi {
    pub struct CTlsHandle {
        raw: *mut std::ffi::c_void,
    }
}

// 这个类型绝不应该跨线程发送——C 库的设计决定了它只能在一个线程使用
// 但编译器看到 *mut c_void 只是不自动推导 Send
// 不会主动"拒绝"——我们想显式声明它是 !Send，以便在误用时提前报错
use ffi::CTlsHandle;

// 负面 impl（需要 nightly Rust 的 negative_impls feature，或使用标准模式）：
// 在 stable Rust 中，通过包含一个 !Send 的字段来阻止 Send 推导：
pub struct ThreadLocalHandle {
    inner: ffi::CTlsHandle,
    _not_send: std::marker::PhantomData<*const ()>,  // *const T: !Send
}

// 这样 ThreadLocalHandle 就绝不可能是 Send
// 任何尝试跨线程传递它的代码都会在编译期报错

// 在 nightly Rust 中可以直接写负面 impl：
// #![feature(negative_impls)]
// impl !Send for CTlsHandle {}
// impl !Sync for CTlsHandle {}

// 负面 impl 的典型使用场景：
// 1. FFI 类型（确定不能跨线程使用）
// 2. 包装了线程本地状态的类型
// 3. "守卫"类型——持有期间不能跨线程（如 MutexGuard）
```

负面 trait 实现是一种"防御性设计"——它不是让代码能编译，而是让错误用法尽早被编译器捕获。

## 6. Send/Sync 与内部可变性的交叉

### 6.1 各种内部可变性类型的线程安全标记

理解每种内部可变性类型的 `Send`/`Sync` 标记是驾驭 Rust 多线程编程的关键。

```rust
use std::cell::{Cell, RefCell, UnsafeCell};
use std::sync::{Mutex, RwLock, atomic::AtomicI32};
use std::rc::Rc;

// 完整的线程安全标记一览（简化）：

// Cell<T>:      Send if T: Send,  !Sync always
// RefCell<T>:   Send if T: Send,  !Sync always
// UnsafeCell<T>: Send if T: Send, !Sync always
//
// Mutex<T>:     Send if T: Send,  Sync if T: Send
// RwLock<T>:    Send if T: Send,  Sync if T: Send
// AtomicI32:    Send,             Sync
// AtomicBool:   Send,             Sync
// AtomicPtr<T>: Send,             Sync
//
// Rc<T>:        !Send,            !Sync
// Arc<T>:       Send if T: Send + Sync,  Sync if T: Send + Sync
//
// Box<T>:       Send if T: Send,  Sync if T: Sync
// Vec<T>:       Send if T: Send,  Sync if T: Sync
// String:       Send,             Sync
// &T:           Send when T: Sync
// &mut T:       Send when T: Send

// 核心规律：
// 1. 单线程内部可变性（Cell/RefCell/UnsafeCell）→ !Sync
//    —— 因为内部操作不是原子的，多线程共享引用不安全
// 2. 多线程同步原语（Mutex/RwLock/Atomic*）→ Sync
//    —— 因为它们通过原子操作或系统级同步保证了线程安全
// 3. Rc → !Send + !Sync
//    —— 非原子引用计数，单线程专用
// 4. Arc → 要求 T: Send + Sync 才能是 Send/Sync
//    —— 传播约束，确保内部数据也是线程安全的
```

这个规律表是你日常 Rust 编程中做"线程安全推理"的速查表。内部可变性类型被分为两类：单线程版（`Cell`/`RefCell`，标记为 `!Sync`）和多线程版（`Mutex`/`RwLock`/`Atomic*`，标记为 `Sync`）。这种分类不是随意的——它反映了底层同步机制的有无。

### 6.2 为什么 Cell/RefCell 是 !Sync —— 根因分析

`Cell` 和 `RefCell` 之所以被标记为 `!Sync`，不是类型系统保守，而是它们确实做不到线程安全。

```rust
use std::cell::Cell;

// Cell<T> 的内部操作本质上是"读-改-写"序列
// 在汇编层面就是普通的 load 和 store 指令（非原子）
//
// Cell::set(value) 等价于:
//   mov [ptr], value   ; 一次普通的 store
//
// Cell::get() 等价于:
//   mov eax, [ptr]     ; 一次普通的 load
//
// Cell::set(Cell::get() + 1) 等价于:
//   mov eax, [ptr]     ; load 当前值
//   add eax, 1         ; 加一
//   mov [ptr], eax     ; store 回内存
// ——这是经典的"读-改-写"数据竞争场景！

// 如果两个线程同时执行 Cell::set(Cell::get() + 1)：
//   线程 A: load [ptr] → 读取到 42
//   线程 B: load [ptr] → 读取到 42  （也在 A 写回之前读！）
//   线程 A: add 1, store 43
//   线程 B: add 1, store 43  ← 应该得到 44，但只得到 43！
// 这就是数据竞争：两个线程同时在不加同步的情况下访问同一内存
// 且至少一个是写操作。

// Mutex<T> 解决同样的问题但通过原子操作 + 系统同步：
// Mutex::lock() → 原子 compare_exchange 或 futex 系统调用
// 确保同一时刻只有一个线程持有锁
// 因此 Mutex<T> 是 Sync（当 T: Send 时）
```

这个根因分析说明：`!Sync` 不是类型系统在"惩罚"内部可变性，而是内部可变性类型在没有原子同步的情况下天然不具备多线程安全性。这是一个**反映真实物理限制**的类型标记，而非随意施加的限制。

### 6.3 什么让 Mutex<T> 成为 Sync —— 它比 Cell 多了什么

`Mutex<T>` 之所以能成为 `Sync`，是因为它用原子操作和操作系统原语在 `lock()` 和 `unlock()` 之间建立了一个"临界区"，保证同一时刻只有一个线程访问内部数据。

```rust
use std::sync::Mutex;

// Mutex<T> 的 Sync 条件：T: Send
// 为什么是 T: Send 而不是 T: Sync？
//
// 因为 Mutex::lock() 返回的是 MutexGuard<T>，
// 它提供了 &mut T 的访问——独占可变引用。
// 独占访问意味着同一时刻只有一个线程能访问 T，
// 所以 T 只需要是 Send（能从持有锁的线程转移到另一个线程），
// 不需要是 Sync（多个线程同时共享 &T）。
//
// 例子：Mutex<Cell<i32>>
// Cell<i32>: Send ✅（当 i32: Send）
// Cell<i32>: !Sync
// Mutex<Cell<i32>>: Sync ✅ —— 因为 Mutex 保证互斥访问
//
// 这展示了一个重要模式：
//   Mutex<T: !Sync> 可以是 Sync
//   Mutex 将"单线程内部可变性"升级为"多线程安全"
//   这是将 Rc<RefCell<T>> 模式迁移到多线程的关键原理

// 实践验证：
let m = Mutex::new(Cell::new(0));
// 可以通过 Arc<Mutex<Cell<i32>>> 在多个线程间共享
// 因为 Mutex 提供了互斥保证
```

这个设计模式是 Rust 多线程编程的核心："如果你想在多线程中共享可变状态，把它包在 `Mutex`（或 `RwLock`）里，再包在 `Arc` 里。"`Mutex` 是类型系统线程安全拼图的最后一块。

## 7. 常见模式与反模式

### 7.1 Send/Sync 推理实践的步骤

当你需要写多线程代码时，按以下步骤进行 `Send`/`Sync` 推理：

```rust
use std::thread;
use std::sync::{Arc, Mutex};
use std::rc::Rc;
use std::cell::RefCell;

// 步骤 1：找出所有被 spawn 闭包捕获的类型
// 步骤 2：对每个类型问两个问题
//         Q1: 它是 Send 吗？（能被移动到另一个线程）
//         Q2: 它是 Sync 吗？（共享引用能在多线程使用）
// 步骤 3：如果类型不满足要求
//         用 Arc 替代 Rc
//         用 Mutex<T> 包住 Cell<T> / RefCell<T> / 原始数据 T

// 示例：将单线程代码迁移到多线程

// === 单线程版本（正常工作）===
fn single_thread() {
    let data = Rc::new(RefCell::new(vec![1, 2, 3]));
    let d1 = data.clone();
    let d2 = data.clone();

    d1.borrow_mut().push(4);
    d2.borrow_mut().push(5);

    println!("{:?}", data.borrow());
}

// === 尝试直接迁移到多线程（失败）===
fn multi_thread_wrong() {
    let data = Rc::new(RefCell::new(vec![1, 2, 3]));
    // thread::spawn(move || {
    //     data.borrow_mut().push(4);  // ❌ Rc 不是 Send
    // });
    // 编译器会阻止你
}

// === 正确的多线程迁移 ===
fn multi_thread_correct() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));

    let d1 = data.clone();
    let t1 = thread::spawn(move || {
        let mut guard = d1.lock().unwrap();
        guard.push(4);
    });

    let d2 = data.clone();
    let t2 = thread::spawn(move || {
        let mut guard = d2.lock().unwrap();
        guard.push(5);
    });

    t1.join().unwrap();
    t2.join().unwrap();

    println!("{:?}", data.lock().unwrap());
}

// 迁移规则：
// Rc<T>       → Arc<T>
// RefCell<T>  → Mutex<T>（或 RwLock<T> 如果读多写少）
// Cell<T>     → AtomicI32/AtomicBool/... 或 Mutex<T>
// 裸指针       → 保持裸指针 + PhantomData（如果必须）
```

### 7.2 常见的 Send/Sync 编译错误及其修复

以下是在实际工程中最常遇到的 `Send`/`Sync` 相关编译错误，以及它们的标准修复方法。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

// 错误 1："Rc<T> cannot be sent between threads safely"
// 原因：Rc 不是 Send
// 修复：用 Arc 替换 Rc
fn fix_rc_to_arc() {
    // ❌: let rc = Rc::new(42);
    let arc = Arc::new(42);  // ✅

    let a = arc.clone();
    thread::spawn(move || {
        println!("{}", a);
    }).join().unwrap();
}

// 错误 2："Cell<T> cannot be shared between threads safely"
// 原因：Cell 不是 Sync，而 Arc 要求 T: Sync
// 修复：用 Mutex<T> 或 Atomic* 替换 Cell<T>
fn fix_cell() {
    use std::sync::atomic::{AtomicI32, Ordering};

    // ❌: let counter = Arc::new(Cell::new(0));
    let counter = Arc::new(AtomicI32::new(0));  // ✅

    let c = counter.clone();
    thread::spawn(move || {
        c.fetch_add(1, Ordering::SeqCst);
    }).join().unwrap();
}

// 错误 3："*const T cannot be sent between threads safely"
// 原因：闭包中捕获了裸指针
// 修复：确保指针指向线程安全的数据，然后 unsafe impl Send
//       或者（更常见的）使用安全的包装类型

// 错误 4："MutexGuard<'_, T> is not Send"
// 原因：试图把 MutexGuard 发送到另一个线程
// 修复：在发送前 drop MutexGuard
fn fix_mutex_guard() {
    let m = Arc::new(Mutex::new(42));

    let m2 = m.clone();
    let t = thread::spawn(move || {
        let guard = m2.lock().unwrap();
        // ❌: 不能在 guard 存活时把它移到别的线程
        // let t2 = thread::spawn(move || { drop(guard); });
        drop(guard);  // ✅ 先 drop
        let val = *m2.lock().unwrap();  // ✅ 再获取
        val
    });
    t.join().unwrap();
}
```

### 7.3 构建线程安全包装器的实战示例

下面的例子展示了如何从零构建一个线程安全的"写一次读多次"类型——我们将综合运用 `Send`/`Sync` 的知识。

```rust
use std::sync::Mutex;
use std::marker::{Send, Sync};

/// WriteOnce<T>: 一个只能写入一次的线程安全容器
/// 第一次 set 写入值，之后只能读取
/// 内部用 Mutex<Option<T>> 实现——Mutex 提供了互斥 + Send/Sync
pub struct WriteOnce<T> {
    // Mutex<Option<T>>: Sync 当 Option<T>: Send
    // Option<T>: Send 当 T: Send
    // → 要求 T: Send
    value: Mutex<Option<T>>,
}

impl<T: Send> WriteOnce<T> {
    pub fn new() -> Self {
        WriteOnce {
            value: Mutex::new(None),
        }
    }

    /// 尝试设置值。如果已经设置过，返回 Err(旧值)。
    /// 如果还没设置，设置并返回 Ok(())。
    pub fn set(&self, val: T) -> Result<(), T> {
        let mut guard = self.value.lock().unwrap();
        match &*guard {
            Some(_) => Err(val),  // 已经设置过
            None => {
                *guard = Some(val);
                Ok(())
            }
        }
    }

    /// 获取值。如果还没设置，返回 None。
    pub fn get(&self) -> Option<std::sync::MutexGuard<'_, Option<T>>> {
        // 注意：返回 MutexGuard 以保持锁定
        // 调用者可以通过 Deref 读取值
        // 但在 guard 存活期间不能调用 set
        Some(self.value.lock().unwrap())
    }
}

// Send/Sync 的自动推导
// WriteOnce<T>: 包含 Mutex<Option<T>>
// Mutex<Option<T>>: Send 当 Option<T>: Send
// Mutex<Option<T>>: Sync 当 Option<T>: Send
// Option<T>: Send 当 T: Send
// → WriteOnce<T>: Send 当 T: Send
// → WriteOnce<T>: Sync 当 T: Send
// ✅ 自动推导正确，无需手动 impl

// 使用示例：
// let wo = Arc::new(WriteOnce::new());
// let wo2 = wo.clone();
// thread::spawn(move || {
//     wo2.set(42).unwrap();
// }).join().unwrap();
// println!("{:?}", *wo.get().unwrap());  // Some(42)
// wo.set(100);  // Ok(()) ← 未设置所以成功...等等，已经设置了！
// 实际上 get() 返回 guard 期间 set() 会死锁
// 需要小心 MutexGuard 的生命周期
```

## 8. 深入理解 —— Send/Sync 与 Rust 编译器的内部视角

### 8.1 auto trait 与 negative reasoning

在编译器内部，`Send` 和 `Sync` 被标记为 `auto trait`——属于一类特殊的 trait，编译器在 trait 解析时对它们有特殊处理。

```rust
// auto trait 的三个关键特性：

// 1. 默认实现 —— 编译器自动为所有类型推导
//    不需要写 #[derive(Send)]，它自动发生
//
// 2. 负面推导 —— 编译器知道"什么时候不"
//    如果结构体任何一个字段不是 Send，整个结构体就不是 Send
//
// 3. 特殊语法 —— impl !Trait for Type
//    nightly Rust 中可以直接写 impl !Send for MyType
//    stable Rust 中通过包含 !Send 的字段来"传染"

// 示例：编译器实际的推导过程（概念性）
// struct Pair<A, B> { a: A, b: B }
//
// 编译器问：
//   A 是 Send 吗？
//     YES → 继续
//     NO  → Pair<A, B>: !Send（立即停止）
//   B 是 Send 吗？
//     YES → Pair<A, B>: Send ✅
//     NO  → Pair<A, B>: !Send

// auto trait 的这种"传染性"设计确保了：
// 你不可能通过"包装"来隐藏非线程安全的内部类型
// 这保证了线程安全检查的完整性
```

### 8.2 为什么 Send/Sync 是 unsafe trait

`Send` 和 `Sync` 是 `unsafe trait`——这意味着手动实现它们需要 `unsafe impl`。这不是为了给程序员增加负担，而是因为编译器无法验证你的"线程安全"承诺。

```rust
// Send 和 Sync 是 unsafe trait 的原因：

// 1. 它们代表了对编译器的"承诺"（而非可验证的属性）
//    编译器可以检查"所有字段都是 Send"，但不能检查
//    "裸指针指向的数据确实被外部同步机制保护"

// 2. 手动实现意味着你承诺了编译器无法验证的东西
//    如果你手动 impl Send for *mut UnsafeData
//    但实际没有同步机制——UB（未定义行为）就在那里

// 3. unsafe trait 和 unsafe fn 有相同的哲学：
//    实现者承担额外的安全责任
//    调用者可以安全使用（因为实现者保证了安全性）

// 对比：如果你在 C++ 中把数据标记为"线程安全"
// 但实际没有同步机制——编译器不会帮忙
// Rust 的 unsafe trait 至少标记了"这里需要人工审查"
```

### 练习 2 答案

```rust
use std::collections::HashMap;
use std::hash::Hash;
use std::sync::{Arc, RwLock};
use std::marker::{Send, Sync};

/// 多线程共享缓存，读多写少场景
pub struct SharedCache<K: Eq + Hash, V> {
    // RwLock<HashMap<K, V>>:
    //   - RwLock<T>: Sync 当 T: Send  → 要求 HashMap<K, V>: Send
    //   - HashMap<K, V>: Send 当 K: Send && V: Send
    inner: RwLock<HashMap<K, V>>,
}

impl<K: Eq + Hash + Send + Sync, V: Send + Sync + Clone> SharedCache<K, V> {
    pub fn new() -> Self {
        SharedCache {
            inner: RwLock::new(HashMap::new()),
        }
    }

    pub fn insert(&self, key: K, value: V) -> Option<V> {
        let mut map = self.inner.write().unwrap();
        map.insert(key, value)
    }

    pub fn get(&self, key: &K) -> Option<V> {
        let map = self.inner.read().unwrap();
        map.get(key).cloned()
    }

    pub fn get_or_insert_with(&self, key: K, f: impl FnOnce() -> V) -> V {
        // 先用读锁检查（乐观）
        {
            let map = self.inner.read().unwrap();
            if let Some(v) = map.get(&key) {
                return v.clone();
            }
        } // 读锁释放

        // 再用写锁插入（悲观——其他线程可能已经插入了）
        let mut map = self.inner.write().unwrap();
        if let Some(v) = map.get(&key) {
            return v.clone();
        }
        let val = f();
        map.insert(key, val.clone());
        val
    }
}

// Send/Sync 推导：
// SharedCache<K, V> 包含 RwLock<HashMap<K, V>>
// RwLock 自动为 Send/Sync（当内部 T: Send）
// → SharedCache<K, V>: Send 当 K+V: Send
// → SharedCache<K, V>: Sync 当 K+V: Send
// ✅ 可以在多线程中使用
```

### 练习 3 答案

```rust
use std::sync::{Arc, Mutex};
use std::thread;

// 原始代码的问题：
// - Rc<T> 不是 Send，不能跨线程传递
// - RefCell<T> 不是 Sync，不能通过引用在多线程中共享

// ✅ 修复版本：
struct AppState {
    counter: Arc<Mutex<i32>>,          // Rc → Arc, RefCell → Mutex
    data: Arc<Mutex<Vec<String>>>,     // 同样的迁移
}

fn process_in_thread(state: AppState) {
    let s1 = state.counter.clone();
    let t1 = thread::spawn(move || {
        // ✅ Arc<Mutex<i32>>: Send + Sync
        let mut guard = s1.lock().unwrap();
        *guard += 1;
    });

    let s2 = state.data.clone();
    let t2 = thread::spawn(move || {
        // ✅ Arc<Mutex<Vec<String>>>: Send + Sync
        let mut guard = s2.lock().unwrap();
        guard.push("hello".into());
    });

    t1.join().unwrap();
    t2.join().unwrap();
}

// 修复后的 Send/Sync 推导：
// Arc<Mutex<i32>>:
//   - Mutex<i32>: Send (i32: Send)
//   - Mutex<i32>: Sync (i32: Send)
//   - Arc<T>: Send 当 T: Send + Sync → ✅
//   - Arc<T>: Sync 当 T: Send + Sync → ✅
//
// AppState:
//   - 两个字段都是 Arc<Mutex<T>>, 都是 Send + Sync
//   - → AppState: Send ✅, Sync ✅
```

### 练习 4 答案

```rust
use std::marker::PhantomData;

// 假设 C 库头文件中有注释：
// "This struct uses thread-local storage internally.
//  Do NOT access from multiple threads."
mod c_library {
    // 模拟 C 库的类型
    #[repr(C)]
    pub struct UnsafeHandle {
        pub(crate) raw: *mut std::ffi::c_void,
    }
}

/// Safe Rust wrapper for a C handle that is NOT thread-safe.
/// The C library uses TLS internally, so we must prevent
/// cross-thread usage at compile time.
pub struct CHandle {
    raw: *mut std::ffi::c_void,
    // PhantomData<*const ()> 阻止自动推导 Send 和 Sync
    // *const T: !Send + !Sync
    _not_thread_safe: PhantomData<*const ()>,
}

impl CHandle {
    /// 仅从当前线程调用。包装 C 库的 create_handle()。
    pub unsafe fn from_raw(raw: *mut std::ffi::c_void) -> Self {
        CHandle {
            raw,
            _not_thread_safe: PhantomData,
        }
    }

    /// 使用 handle（仅在当前线程安全）
    pub fn do_something(&self) {
        // 调用 C 库函数 ...
    }
}

// CHandle: !Send ❌ —— *const T 不是 Send
// CHandle: !Sync ❌ —— *const T 不是 Sync
// 任何试图在 spawn 中使用 CHandle 的代码都会编译失败
// ✅ 编译器阻止了潜在的 UB
```

## 11. 回顾

```
Send & Sync 学习清单：

□ 理解 Send = "所有权可以跨线程转移"
□ 理解 Sync = "共享引用可以跨线程共享"
□ 记住关键等价关系：T: Sync ⇔ &T: Send
□ 知道基本类型（i32, String, Vec<T>, Box<T>）的 Send/Sync 状态
□ 理解 Rc<T> 不是 Send 的根因（非原子引用计数）
□ 理解 Cell/RefCell 是 Send 但 !Sync 的原因（内部可变性非原子）
□ 掌握 Arc<T> 的 Send/Sync 推导条件（T: Send + Sync）
□ 理解 Mutex<T> 是 Sync 当 T: Send 的原因（互斥保证）
□ 知道自动 trait 的推导规则（所有字段满足 → 整体满足）
□ 理解 PhantomData<T> 对 Send/Sync 推导的影响
□ 掌握 unsafe impl Send/Sync 的安全使用场景
□ 了解负面 trait 实现（impl !Send）的用途（FFI/防御性编程）
□ 能够将 Rc<RefCell<T>> 模式迁移到 Arc<Mutex<T>>
```

> `Send` 和 `Sync` 是 Rust 写在类型系统里的"线程安全契约"——编译器在编译期就帮你划定了数据跨线程的合法边界。理解了它们，你就掌握了 Rust 多线程编程的"准入规则"。
