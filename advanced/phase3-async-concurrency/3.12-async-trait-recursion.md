# 3.12 async trait & async fn 递归 —— 当异步遇上 trait 和自引用

> Trait 中的 async fn 看似只是多写 5 个字符（`async`），背后却是 `impl Future` 的类型不透明性、动态分发的根本限制、Send bound 的传染性传播三重挑战。Rust 1.75 的 RPITIT（Return Position Impl Trait In Trait）稳定化给出了"零成本"的 native 方案，但 `#[async_trait]` 宏仍然是动态分发和向后兼容的重要工具。递归的 async fn 则揭开了另一个伤口——async block 编译出的状态机类型包含自身，产生无限大小的类型级递归——解决方案是 `Box<Pin<...>>`，也就是引入一次堆分配来打断类型循环。这一课将这三个硬核话题一一拆解。

---

## 1. async fn in trait —— 从噩梦到 RPITIT

### 1.1 为什么"async fn in trait"曾经是问题

在 Rust 1.75 之前，在 trait 中写 async fn 是编译错误。原因是 async fn 本质上是一个返回 `impl Future` 的语法糖，而 `impl Trait` 在 trait 的返回位置（RPITIT）直到 1.75 才被稳定支持。

```rust
// ❌ 1.75 之前：编译错误
// trait DatabaseClient {
//     async fn query(&self, sql: &str) -> Vec<String>;
// }

// 编译器需要知道每个方法的完整类型签名
// async fn → fn(...) -> impl Future<Output = ...>
// impl Future 是"存在性类型"（existential type）
// 编译器说："我不知道返回的具体 Future 类型是什么，
//             所以无法为 trait object 生成 vtable"

// ✅ 1.75 之前唯一能用的方案：显式返回 Future trait object
trait DatabaseClient {
    fn query(&self, sql: &str) -> Pin<Box<dyn Future<Output = Vec<String>> + Send + '_>>;
}
```

这种写法有三个问题：第一，每次调用都会堆分配（`Box::pin`）；第二，类型签名冗长难读；第三，`dyn Future` 的动态分发损失了一部分性能。RPITIT 的设计目标就是在不引入堆分配的前提下支持这种模式。

### 1.2 1.75 之后的 native 写法

从 Rust 1.75 开始，你终于可以直接在 trait 中写 async fn：

```rust
// ✅ Rust 1.75+：直接在 trait 中写 async fn
trait DatabaseClient {
    async fn query(&self, sql: &str) -> Vec<String>;
    async fn insert(&self, table: &str, data: &str) -> u64;
    async fn delete(&self, table: &str, id: u64) -> bool;
}

// 编译器内部将其展开为：
// trait DatabaseClient {
//     fn query(&self, sql: &str) -> impl Future<Output = Vec<String>> + '_;
//     fn insert(&self, table: &str, data: &str) -> impl Future<Output = u64> + '_;
//     fn delete(&self, table: &str, id: u64) -> impl Future<Output = bool> + '_;
// }
```

注意编译器自动添加的 `+ '_` bound——它确保返回的 Future 不能活得比 `&self` 更长。这是"匿名生命周期"的捕获规则自动推导的。

### 1.3 为 trait 编写实现

实现 RPITIT 的 trait 就像实现普通 trait 一样——你可以在 impl 中用 async fn 语法，也可以用显式的 Future 类型。

```rust
struct PostgresClient {
    connection_string: String,
}

impl DatabaseClient for PostgresClient {
    // ✅ 直接用 async fn 实现——最常见的写法
    async fn query(&self, sql: &str) -> Vec<String> {
        println!("[Postgres] executing query: {}", sql);
        // 实际的异步数据库查询
        vec!["result_1".into(), "result_2".into()]
    }

    async fn insert(&self, table: &str, data: &str) -> u64 {
        println!("[Postgres] insert into {}: {}", table, data);
        42  // 返回插入的行 ID
    }

    async fn delete(&self, table: &str, id: u64) -> bool {
        println!("[Postgres] delete from {} where id={}", table, id);
        true
    }
}

// 也可以显式返回具体的 Future 类型（当你有特殊需求时）
struct InMemoryClient;

impl DatabaseClient for InMemoryClient {
    fn query(&self, sql: &str) -> impl Future<Output = Vec<String>> + '_ {
        // 显式构造一个 future
        async move {
            println!("[Memory] executing query: {}", sql);
            vec![]  // 内存数据库，返回空
        }
    }

    async fn insert(&self, _table: &str, _data: &str) -> u64 { 0 }
    async fn delete(&self, _table: &str, _id: u64) -> bool { false }
}
```

两种写法的效果完全相同——编译器生成的代码也相同。用 async fn 语法更简洁，用显式 `impl Future` 可以在需要时添加自定义 bound。

## 2. Send Bound 问题——async fn 默认不 Send

### 2.1 默认情况下 async fn 返回的 Future 不是 Send

这是一个常见的"坑"：trait 中的 async fn 生成的 Future 默认**不是** `Send`。这意味着你不能在多线程运行时中跨线程传递它。

```rust
// trait 定义——async fn 没有显式标注 Send
trait Service {
    async fn process(&self) -> String;
}

// ❌ 问题：无法在多线程 tokio 中 spawn
#[tokio::main]
async fn main() {
    let service = MyService;

    // ❌ 编译错误：future 不是 Send
    // tokio::spawn 要求 Future 是 Send 的
    // tokio::spawn(service.process());
    //
    // error: future cannot be sent between threads safely
    // the trait `Send` is not implemented for
    // `impl Future<Output = String>`
}
```

为什么会这样？因为 `impl Future` 是一个不透明类型——编译器只暴露 `Future` trait 的实现，不暴露 `Send`。在 RPITIT 中，`Send` 需要显式标注。

### 2.2 解决方案：显式添加 Send bound

Rust 1.75 的 RPITIT 支持在 trait 定义中使用 `use` 语法添加 Send bound（这个语法从 1.84 开始变得更加简洁）：

```rust
// ✅ 方案 1：在 trait 中显式要求 Send（Rust 1.84+ 语法)
trait Service {
    async fn process(&self) -> String
    // Rust 1.84 起支持在 async fn 上直接标注 Send bound
    where
        Self: Send;  // 或者使用更精确的 bound
}

// ✅ 方案 2：用显式 impl Future + Send 标注（Rust 1.75+ 可用）
trait ServiceV2 {
    fn process(&self) -> impl Future<Output = String> + Send + '_;
}

struct MyService;

impl ServiceV2 for MyService {
    fn process(&self) -> impl Future<Output = String> + Send + '_ {
        async move {
            // 这个 async block 内部的所有跨 .await 的类型都必须是 Send
            format!("processed")
        }
    }
}

// 验证：现在可以 spawn 了
#[tokio::main]
async fn main() {
    let service = MyService;
    let future = ServiceV2::process(&service);
    tokio::spawn(future);  // ✅ Send 了
}
```

关键要点：`Send` bound 会"传染"——如果 trait 的 async fn 返回 `Send` future，那么所有 impl 中的 async block 内部不能持有 `!Send` 的类型（如 `Rc`、`RefCell`、`*const T` 等）。

### 2.3 Send bound 的传染性示例

```rust
use std::rc::Rc;

trait SendService {
    fn process(&self) -> impl Future<Output = String> + Send + '_;
}

struct BrokenService {
    data: Rc<String>,  // Rc 不是 Send
}

impl SendService for BrokenService {
    fn process(&self) -> impl Future<Output = String> + Send + '_ {
        let data = self.data.clone();
        async move {
            // ❌ 编译错误：async block 捕获了 Rc<String>（!Send）
            // 所以这个 future 不是 Send
            // 但 trait 要求 future 是 Send → 冲突
            format!("data: {}", data)
        }
    }
}
// 修复：把 Rc 换成 Arc
// struct FixedService {
//     data: Arc<String>,  // Arc 是 Send + Sync
// }
```

这个例子说明了 async trait 背后最微妙的一点：Send bound 表面上是加在 trait 定义上的，但实际上约束了所有实现者的**内部实现细节**——你必须知道哪些类型是 Send，哪些不是，否则编译器会给你一个难以理解的错误。

## 3. RPITIT 与动态分发——为什么不能直接 dyn

### 3.1 RPITIT 与 dyn Trait 的互斥

使用 RPITIT 的 trait **不能**直接创建为 trait object：

```rust
trait AsyncRunner {
    async fn run(&self) -> String;
}

fn handle(runner: &dyn AsyncRunner) {
    // ❌ 编译错误：
    // error[E0038]: the trait `AsyncRunner` cannot be made into an object
    // `async fn` in trait returns `impl Future`, which is not object-safe
    //
    // 原因：编译器不知道 dyn AsyncRunner::run() 返回的
    // 具体 Future 类型是什么（不同类型有不同的大小和布局）
    // 所以无法构造 vtable
}
```

这就是 RPITIT 的根本限制：async fn 在 trait 中生成的是 **静态分发** 代码。如果你需要动态分发，必须返回 `Pin<Box<dyn Future>>`——这正是 `#[async_trait]` 宏所做的。

### 3.2 静态分发 vs 动态分发——两种设计

```rust
// 方案 A：RPITIT + 静态分发（零开销，但需要泛型）
// 适合：编译时已知具体类型的场景

trait StaticAsync {
    async fn work(&self) -> i32;
}

async fn use_static<T: StaticAsync>(worker: &T) -> i32 {
    worker.work().await  // 编译时单态化——零开销
}

// 方案 B：async_trait + 动态分发（有堆分配，但灵活）
// 适合：需要存储不同类型、运行时选择的场景

#[async_trait::async_trait]
trait DynamicAsync {
    async fn work(&self) -> i32;
}

async fn use_dynamic(worker: &dyn DynamicAsync) -> i32 {
    worker.work().await  // 通过 vtable 调用——有一次堆分配
}
```

选择标准：如果你可以接受泛型参数，用 RPITIT（零开销）；如果你需要把不同类型的实现者放进同一个容器（如 `Vec<Box<dyn Trait>>`），用 `#[async_trait]`。

## 4. `#[async_trait]` 宏——RPITIT 之前的标准方案

### 4.1 async_trait 做的事

`#[async_trait]` 宏（来自 async-trait crate）将 async fn 转换为返回 `Pin<Box<dyn Future + Send>>` 的普通方法：

```rust
use async_trait::async_trait;

// 你写的代码：
#[async_trait]
trait MyTrait {
    async fn do_something(&self) -> String;
}

// 宏展开后（概念上）：
// trait MyTrait {
//     fn do_something<'a>(&'a self) 
//         -> Pin<Box<dyn Future<Output = String> + Send + 'a>>;
// }

// 实现中：
#[async_trait]
impl MyTrait for MyType {
    async fn do_something(&self) -> String {
        String::from("done")
    }
}
// 展开后（概念上）：
// impl MyTrait for MyType {
//     fn do_something<'a>(&'a self) 
//         -> Pin<Box<dyn Future<Output = String> + Send + 'a>> 
//     {
//         Box::pin(async move { String::from("done") })
//     }
// }
```

关键观察：`#[async_trait]` 引入了**每次调用一次堆分配**（`Box::pin`）。这是它相对于 RPITIT 的主要性能代价。

### 4.2 async_trait vs RPITIT——选择指南

下面是两种方案的对比，用实际的 HTTP 客户端 trait 做例子：

```rust
// ========== RPITIT 方案（native, 1.75+） ==========
// 优势：零堆分配、零虚函数调用
// 劣势：不能 dyn trait

trait HttpClientRPITIT {
    async fn get(&self, url: &str) -> Result<String, String>;
    async fn post(&self, url: &str, body: &str) -> Result<String, String>;
}

// ========== async_trait 方案 ==========
// 优势：可以 dyn trait、编译器版本兼容性好
// 劣势：每次调用堆分配、动态分发开销

use async_trait::async_trait;

#[async_trait]
trait HttpClientAsyncTrait {
    async fn get(&self, url: &str) -> Result<String, String>;
    async fn post(&self, url: &str, body: &str) -> Result<String, String>;
}

// ========== 选择场景 ==========

// 场景 1：中间件的"构建时"配置——用 RPITIT
// 你在编译时就知道了所有实现者，不需要动态分发
fn build_app<C: HttpClientRPITIT>(client: C) -> App<C> {
    App { client }
}

// 场景 2：运行时的插件系统——用 async_trait
// 你在运行时加载不同的 HTTP 客户端实现
struct PluginManager {
    plugins: Vec<Box<dyn HttpClientAsyncTrait>>,
}
```

对于新项目，如果不需要动态分发，首选 RPITIT。对于需要 `dyn Trait` 的场景，`#[async_trait]` 仍然是必要的工具。

## 5. 递归 async fn——为什么不能直接写

### 5.1 问题的根源：无限大小的类型

递归的 async fn 不能编译，因为编译器需要为 async block 生成一个确定大小的类型——而递归会产生无限大小的类型：

```rust
// ❌ 编译错误：
// async fn recurse(n: u32) {
//     if n == 0 {
//         return;
//     }
//     recurse(n - 1).await;  // ← 递归调用
// }
//
// 错误信息（简化）：
// error: recursion in an `async fn` requires boxing
// = note: an `async fn` is not a fixed-size type,
//         and recursion would create a type of unbounded size

// 为什么会无限大小？
// 编译器需要为 recurse 的 async block 生成一个状态机类型：
//   enum RecursiveFuture {
//       Start,
//       AwaitingRecurseCall {
//           inner: RecursiveFuture,   // ← 包含自身！
//           n: u32,                   //    类型大小 = 自身大小 + sizeof(u32) + ...
//       },                            //    ↑ 这会无限展开
//       Done,
//   }
// 这就像 struct Foo { inner: Foo } ——大小无穷
```

### 5.2 解决方案：用堆分配打破类型级递归

解决方案是强制递归调用通过堆分配进行——`Box<Pin<...>>` 将无限大小的类型变成固定大小的指针。

```rust
// ✅ 方案 1：使用 async-recursion crate（最方便）
use async_recursion::async_recursion;

#[async_recursion]
async fn recurse(n: u32) {
    if n == 0 {
        return;
    }
    println!("level {}", n);
    recurse(n - 1).await;  // ✅ 编译通过
    // async_recursion 宏内部自动加了 Box::pin(...)
}

// ✅ 方案 2：手动 Box::pin（不依赖 macro）
fn recurse_manual(n: u32) -> Pin<Box<dyn Future<Output = ()> + Send>> {
    Box::pin(async move {
        if n == 0 {
            return;
        }
        println!("level {}", n);
        recurse_manual(n - 1).await;  // ✅ 递归调用返回 Box<Pin<...>>
    })
}

// 注意方案 2 中：
// - 函数签名返回 Pin<Box<dyn Future<Output = ()> + Send>>
// - 不再是 async fn，而是普通 fn 返回 future
// - 递归调用 recurse_manual(n-1).await 合法——因为返回的是固定大小的指针
```

### 5.3 深入理解 async-recursion 宏

`#[async_recursion]` 宏的实质是把 async fn 转换为返回 `Pin<Box<dyn Future>>` 的普通 fn。它的展开大致如下：

```rust
// 宏之前的代码：
#[async_recursion]
async fn factorial(n: u64) -> u64 {
    if n <= 1 {
        1
    } else {
        n * factorial(n - 1).await
    }
}

// 宏展开后的等价代码（概念版）：
fn factorial(n: u64) -> Pin<Box<dyn Future<Output = u64> + Send>> {
    Box::pin(async move {
        if n <= 1 {
            1
        } else {
            n * factorial(n - 1).await
        }
    })
}

// 这个设计的代价：
// 1. 每次递归调用一次堆分配（Box::pin）
//    —— N 层递归 = N 次堆分配
//    —— 这是性能代价，但通常可接受（递归层次一般不会很深）

// 2. 类型擦除了（dyn Future）
//    —— 这意味着轻微的动态分发开销
//    —— 对于递归场景，这不是主要瓶颈

// 3. Send bound
//    —— 可能有 Send/Sync 相关的编译问题
//    —— 可以用 #[async_recursion(?Send)] 去掉 Send bound
```

如果你的递归层次很深（>1000），考虑将递归改为迭代——这不仅避免了 N 次堆分配，也避免了潜在的栈溢出。

## 6. 完整实战——异步数据库 trait

### 6.1 定义数据库操作 trait

将前面各节的知识综合起来，我们来构建一个真实场景的异步数据库 trait：

```rust
use std::future::Future;

/// 一个通用的异步数据库操作接口
/// 使用 RPITIT（1.75+）——零堆分配，静态分发
pub trait AsyncDatabase {
    /// 执行查询，返回行集合
    async fn query(&self, sql: &str) -> Result<Vec<Vec<String>>, String>;

    /// 执行更新，返回受影响的行数
    async fn execute(&self, sql: &str) -> Result<u64, String>;

    /// 启动事务——返回一个实现了 Transaction trait 的具体类型
    async fn begin_transaction(&self) -> Result<Self::Transaction<'_>, String>
    where
        Self: Sized;

    /// 关联类型——事务的具体类型（由实现者决定）
    type Transaction<'a>: AsyncTransaction
    where
        Self: 'a;
}

/// 事务 trait
pub trait AsyncTransaction {
    async fn commit(self) -> Result<(), String>;
    async fn rollback(self) -> Result<(), String>;
}
```

这里使用了 RPITIT 的 async fn 和关联类型结合——对于事务，返回的具体类型由实现者决定（可能是 PostgresTransaction 或 MySQLTransaction）。

### 6.2 实现具体的数据库后端

```rust
// Postgres 实现——使用 tokio-postgres（概念版）
struct PostgresDb {
    // client: tokio_postgres::Client,
}

impl AsyncDatabase for PostgresDb {
    type Transaction<'a> = PostgresTransaction;

    async fn query(&self, sql: &str) -> Result<Vec<Vec<String>>, String> {
        // 实际的异步查询
        println!("[PG] query: {}", sql);
        Ok(vec![vec!["col1".into(), "col2".into()]])
    }

    async fn execute(&self, sql: &str) -> Result<u64, String> {
        println!("[PG] execute: {}", sql);
        Ok(1)
    }

    async fn begin_transaction(&self) -> Result<PostgresTransaction, String> {
        println!("[PG] BEGIN");
        Ok(PostgresTransaction { committed: false })
    }
}

struct PostgresTransaction {
    committed: bool,
}

impl AsyncTransaction for PostgresTransaction {
    async fn commit(mut self) -> Result<(), String> {
        self.committed = true;
        println!("[PG] COMMIT");
        Ok(())
    }

    async fn rollback(self) -> Result<(), String> {
        println!("[PG] ROLLBACK");
        Ok(())
    }
}
```

### 6.3 使用泛型参数——静态分发

调用代码使用泛型参数来接受任何实现 `AsyncDatabase` 的类型：

```rust
// 仓库层——接受任意 AsyncDatabase 实现
struct UserRepository<DB: AsyncDatabase> {
    db: DB,
}

impl<DB: AsyncDatabase> UserRepository<DB> {
    async fn find_by_id(&self, id: u64) -> Result<Option<String>, String> {
        let rows = self.db
            .query(&format!("SELECT name FROM users WHERE id = {}", id))
            .await?;
        Ok(rows.first().and_then(|r| r.first().cloned()))
    }

    async fn create_user(&self, name: &str) -> Result<u64, String> {
        self.db
            .execute(&format!("INSERT INTO users (name) VALUES ('{}')", name))
            .await
    }
}

// 使用——编译时单态化，零开销
// async fn app_main() {
//     let pg = PostgresDb {};
//     let repo = UserRepository { db: pg };
//     let user = repo.find_by_id(1).await.unwrap();
// }
```

这个架构展示了 RPITIT 的最佳用法：trait 中定义 async 接口，泛型参数传递实现，编译时单态化，零堆分配。

## 7. 手动 Future 类型——一种零分配的模式

### 7.1 什么时候需要手动定义 Future 类型

当 RPITIT 不够用（需要额外的 trait bound 或需要精细控制 Future 行为）且你不想用 `#[async_trait]` 的堆分配时，你可以手写一个 Future 类型。

```rust
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll};

/// 手动定义的 Future 类型——模拟一个"带重试"的异步操作
pub struct RetryFuture<F, Fut>
where
    F: Fn() -> Fut,
    Fut: Future<Output = Result<String, String>>,
{
    /// 构造 Future 的工厂函数
    factory: F,
    /// 正在等待的内部 Future（如果有的话）
    current: Option<Fut>,
    /// 剩余重试次数
    retries_left: u32,
    /// 重试之间的延迟（毫秒）
    delay_ms: u64,
}

impl<F, Fut> RetryFuture<F, Fut>
where
    F: Fn() -> Fut,
    Fut: Future<Output = Result<String, String>>,
{
    pub fn new(factory: F, max_retries: u32, delay_ms: u64) -> Self {
        RetryFuture {
            factory,
            current: None,
            retries_left: max_retries,
            delay_ms,
        }
    }
}

impl<F, Fut> Future for RetryFuture<F, Fut>
where
    F: Fn() -> Fut,
    Fut: Future<Output = Result<String, String>>,
{
    type Output = Result<String, String>;

    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        loop {
            // 如果还没有开始或上一个重试失败了，创建新的 Future
            if self.current.is_none() {
                self.current = Some((self.factory)());
            }

            let current = self.current.as_mut().unwrap();
            // 安全：我们确保 current 是被正确 pin 的
            let current_pin = unsafe { Pin::new_unchecked(current) };

            match current_pin.poll(cx) {
                Poll::Ready(Ok(value)) => {
                    return Poll::Ready(Ok(value));
                }
                Poll::Ready(Err(e)) => {
                    if self.retries_left > 0 {
                        self.retries_left -= 1;
                        self.current = None;  // 下次循环创建新的 Future
                        println!("重试中... 剩余 {} 次 ({})", self.retries_left, e);
                        // 注意：这里没有真实的延迟
                        // 在生产代码中，你需要结合 tokio::time
                        continue;
                    } else {
                        return Poll::Ready(Err(format!(
                            "所有重试已用尽。最后错误: {}", e
                        )));
                    }
                }
                Poll::Pending => {
                    return Poll::Pending;
                }
            }
        }
    }
}
```

这个手写的 `RetryFuture` 不需要堆分配（虽然它内部的 `factory()` 返回的 Future 可能有自己的分配），并且是完全零虚函数调用的——所有分发都在编译时通过泛型决议。

### 7.2 使用手动 Future 类型

```rust
// 使用 RetryFuture——通过 trait 的普通（非 async）方法暴露
trait ReliableService {
    /// 这里没有用 async fn——返回一个具名的 Future 类型
    fn fetch_with_retry(&self, url: &str) -> RetryFuture<
        impl Fn() -> impl Future<Output = Result<String, String>>,
        impl Future<Output = Result<String, String>>,
    >;
}

// 实现者构造 RetryFuture
struct MyService;

impl ReliableService for MyService {
    fn fetch_with_retry(&self, url: &str) -> RetryFuture<
        impl Fn() -> impl Future<Output = Result<String, String>>,
        impl Future<Output = Result<String, String>>,
    > {
        let url = url.to_owned();
        RetryFuture::new(
            move || {
                let url = url.clone();
                async move {
                    // 模拟 HTTP 请求——实际应用中使用 reqwest
                    if rand::random::<f32>() < 0.7 {
                        Err("临时错误".into())
                    } else {
                        Ok(format!("来自 {} 的响应", url))
                    }
                }
            },
            3,      // 最多重试 3 次
            100,    // 重试间隔 100ms
        )
    }
}
```

这种模式在需要精确控制异步生命周期、添加自定义 poll 行为、或零分配的共享代码库中非常有用。tokio 内部大量使用这种模式——它提供了比 RPITIT 更多控制力，比 async_trait 更少开销。

## 8. 综合对比——三种 async trait 方案的取舍

### 8.1 方案矩阵

```rust
// ============================================================
// 方案 1: RPITIT (Rust 1.75+, native)
// ============================================================
trait Trait1 {
    async fn method(&self) -> String;
}
// 堆分配：   无
// 虚函数调用：无
// 可以 dyn：  否
// Send 默认：否（需手动标注）
// 向后兼容：  Rust 1.75+

// ============================================================
// 方案 2: #[async_trait] (async-trait crate)
// ============================================================
// #[async_trait]
// trait Trait2 {
//     async fn method(&self) -> String;
// }
// 底层：Pin<Box<dyn Future<Output = String> + Send + '_>>
// 堆分配：   每次调用 = 1 次
// 虚函数调用：无（仅 Future 是 dyn，调用者不是）
// 可以 dyn：  是
// Send 默认：是（可去掉）
// 向后兼容：  Rust 1.39+

// ============================================================
// 方案 3: 手动 Future 类型
// ============================================================
trait Trait3 {
    fn method(&self) -> MyCustomFuture;
}
// 堆分配：   无可选
// 虚函数调用：无
// 可以 dyn：  取决于设计
// Send 默认：取决于设计
// 灵活度：   最高
```

### 8.2 选择决策流程

```
你的 trait 需要用作 trait object (dyn Trait) 吗？
    ├── 是 → 必须用 #[async_trait]
    │         （或手动返回 Pin<Box<dyn Future>>）
    │
    └── 否 → 使用方法可能是静态分发（泛型参数）
              │
              ├── 需要全面控制 Future 行为、
              │   避免任何堆分配、或支持 nightly 特性？
              │   └── 是 → 手动 Future 类型
              │
              └── 否 → 用 RPITIT (native async fn in trait)
                        简单、零开销、标准 Rust
```

## 10. 练习答案

### 练习 1 答案

```rust
// 迁移到 RPITIT——步骤：
// 1. 删除 #[async_trait] 宏
// 2. trait 中 async fn 保持不变
// 3. impl 中 async fn 保持不变
// 4. 必要时在 trait 中添加 Send bound

use std::future::Future;

// ✅ RPITIT 版本——Rust 1.75+
trait Fetch {
    // 默认不 Send，如果需要在多线程环境中使用，需要标注 Send
    fn fetch_data(&self, key: &str) -> impl Future<Output = Result<String, String>> + Send + '_;
    fn fetch_batch(&self, keys: &[String]) -> impl Future<Output = Result<Vec<String>, String>> + Send + '_;
}

struct CacheFetch;

impl Fetch for CacheFetch {
    fn fetch_data(&self, key: &str) -> impl Future<Output = Result<String, String>> + Send + '_ {
        let key = key.to_owned();
        async move {
            Ok(format!("cached({})", key))
        }
    }

    fn fetch_batch(&self, keys: &[String]) -> impl Future<Output = Result<Vec<String>, String>> + Send + '_ {
        // 这里需要把 keys clone 进 async block，因为返回的 future 需要 'static bound
        // 如果 keys 的引用被捕获，lifecycle 会很复杂
        // 一个让步方案是直接用 async fn 语法（编译器处理生命周期）：
        // 但这里显式写出 impl Future 来展示底层发生了什么
        let keys = keys.to_vec();
        async move {
            let mut results = Vec::new();
            // 注意：在 RPITIT 版本中，self.fetch_data 返回 impl Future
            // 在 async block 内部 .await 是合法的
            // 但如果直接在 fn 体中用循环，由于 &self 的生命周期问题，
            // 需要先把 self 借用到 async block 内部。
            // 这是 RPITIT 的一个已知不便——futures 不能借用 &self 太长。
            for key in &keys {
                // 简化：手动实现而不是调用 self.fetch_data
                results.push(format!("cached({})", key));
            }
            Ok(results)
        }
    }
}
```

### 练习 2 答案

```
depth(5) 的堆分配分析：

调用链：depth(5) → depth(4) → depth(3) → depth(2) → depth(1) → depth(0)
depth(0) 直接返回 0 ——没有进一步的递归调用

每次 #[async_recursion] 展开的函数调用 = 1 次 Box::pin
（macros 将 async fn 变为返回 Pin<Box<dyn Future>> 的普通 fn）

所以：
  depth(5): 1 次 Box::pin（创建 depth(5) 的 future）
  depth(4): 1 次 Box::pin（创建 depth(4) 的 future）
  depth(3): 1 次 Box::pin
  depth(2): 1 次 Box::pin
  depth(1): 1 次 Box::pin
  depth(0): 0 次 Box::pin（没有递归调用，直接返回）

总计：5 次堆分配

每层递归都有一次独立的堆分配。
在深度大的递归中，这个分配成本不可忽视。
对于 N > 1000 的递归，强烈建议改为迭代。
```

### 练习 3 答案

```rust
use std::future::Future;
use std::time::Duration;

// 选择 RPITIT——因为不需要 dyn Trait
// 静态分发的零开销优势在这里很重要
// 重试逻辑中的多次调用如果每次都有堆分配会有性能影响

trait ReliableHttpClient {
    /// 带重试和超时的异步 HTTP 请求
    /// 
    /// 参数：
    /// - url: 请求的 URL
    /// - max_retries: 最大重试次数（不含初始尝试）
    /// - timeout: 每次请求的超时时间
    async fn fetch_with_retry(
        &self,
        url: &str,
        max_retries: u32,
        timeout: Duration,
    ) -> Result<String, String>;

    /// 底层的单次请求——由实现者提供
    async fn fetch_once(&self, url: &str) -> Result<String, String>;
}

// 默认实现——利用 RPITIT，所有实现者自动获得
// （注意：这里用了一个技巧——在 trait 定义中直接提供默认实现）
// 
// 实际上对于 RPITIT，我们可以这样做：
// （简化——直接在另一个 fn 或组合子中实现）

struct RealHttpClient;

impl ReliableHttpClient for RealHttpClient {
    async fn fetch_with_retry(
        &self,
        url: &str,
        max_retries: u32,
        timeout: Duration,
    ) -> Result<String, String> {
        let mut attempts = 0;
        loop {
            match tokio::time::timeout(timeout, self.fetch_once(url)).await {
                Ok(Ok(result)) => return Ok(result),
                Ok(Err(e)) => {
                    if attempts >= max_retries {
                        return Err(format!("所有重试已用尽: {}", e));
                    }
                    attempts += 1;
                    // 指数退避延迟
                    tokio::time::sleep(Duration::from_millis(100 * 2u64.pow(attempts))).await;
                }
                Err(_elapsed) => {
                    if attempts >= max_retries {
                        return Err("请求超时，所有重试已用尽".into());
                    }
                    attempts += 1;
                    tokio::time::sleep(Duration::from_millis(100 * 2u64.pow(attempts))).await;
                }
            }
        }
    }

    async fn fetch_once(&self, url: &str) -> Result<String, String> {
        // 实际的 HTTP 请求——使用 reqwest 或其他 HTTP 客户端
        println!("fetching: {}", url);
        Ok(format!("response from {}", url))
    }
}
```
