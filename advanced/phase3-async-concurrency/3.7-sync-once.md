# 3.7 std::sync::Once —— 一次性初始化的底层原语

> 你有没有想过 `lazy_static!` 是依靠什么机制保证"多线程同时首次访问时只初始化一次"的？答案就藏在 `std::sync::Once` 里。`Once` 是 Rust 标准库中最精悍的同步原语之一——它**保证一段代码（闭包）在整个程序生命周期中只被执行一次**，即使被成千上万个线程同时调用。它不存储值，只存储"是否已执行"这一比特的信息。这个看似微小的能力实际上是 `lazy_static!`、`OnceCell`、`OnceLock` 和 `LazyLock` 的基石。理解 `Once`，你就理解了 Rust 懒初始化体系的"第一性原理"。

---

## 1. Once 解决的问题——全局资源的一次性初始化

### 1.1 问题场景：全局日志文件

```rust
use std::fs::File;
use std::io::Write;
use std::sync::Mutex;

// 问题：你需要一个全局的日志文件，所有线程都能往里面写
// 但这个文件只需要被打开一次（所有线程共享同一个文件句柄）

// ❌ 错误的尝试 1：在 main 中打开，通过参数传递
// 问题：深层函数也需要写日志 → 需要把 file 参数传到每个函数 → 传染性

// ❌ 错误的尝试 2：static mut + unsafe
// static mut LOG_FILE: Option<Mutex<File>> = None;
//
// fn log(msg: &str) {
//     unsafe {
//         if LOG_FILE.is_none() {
//             // ❌ 数据竞争！两个线程可能同时走到这里
//             LOG_FILE = Some(Mutex::new(
//                 File::create("app.log").unwrap()
//             ));
//         }
//         writeln!(LOG_FILE.as_ref().unwrap().lock().unwrap(), "{}", msg).unwrap();
//     }
// }

// ✅ 正确的方式：用 Once 保证文件只被打开一次
static LOG_FILE: Mutex<Option<File>> = Mutex::new(None);
static INIT: std::sync::Once = std::sync::Once::new();

fn log(msg: &str) {
    // call_once 保证：里面的闭包在整个程序生命周期中
    // 只被执行一次——即使被多个线程同时调用
    INIT.call_once(|| {
        let file = File::create("app.log")
            .expect("failed to open log file");
        *LOG_FILE.lock().unwrap() = Some(file);
    });

    // 初始化完成后，所有线程直接使用已打开的文件
    if let Some(ref mut file) = *LOG_FILE.lock().unwrap() {
        let _ = writeln!(file, "{}", msg);
    }
}
```

这个例子展示了 `Once` 最核心的价值：**解决全局资源的竞争初始化问题**。你不用在 `main` 函数中预先打开文件再传遍所有函数，也不用手动处理多线程同时初始化的数据竞争——`Once` 帮你处理了所有这些问题。

### 1.2 Once 的核心 API

```rust
use std::sync::Once;

static INIT: Once = Once::new();

// Once 只有两个操作：
// 1. Once::new()            —— 创建一个未执行过的 Once 实例
// 2. once.call_once(f)      —— 如果尚未执行过，执行闭包 f；否则什么都不做

// call_once 是线程安全的：
INIT.call_once(|| {
    println!("这行只打印一次");
});

INIT.call_once(|| {
    println!("这行永远不会打印");
});

// call_once 的闭包类型：FnOnce()
// 这意味着闭包可以消耗外部变量，但只能被执行一次
// （事实上也只能被执行一次——这正好匹配 FnOnce 的语义）
```

`Once` 的 API 表面极其简单，但这简单的接口隐藏了操作系统层面的精巧：`call_once` 内部使用原子操作 + 可能的系统调用（futex/WaitOnAddress）来保证多线程安全，同时无竞争时走快速路径（几乎零开销）。

## 2. Once 的行为细节

### 2.1 Once 的状态机——NOT called → CALLING → FINISHED

```rust
use std::sync::Once;
use std::thread;

// Once 内部有三个状态（概念性的）：
// State::Incomplete  (0)  —— 初始状态：尚未开始执行
// State::InProgress  (N)  —— 某个线程正在执行（其他线程等待）
// State::Complete    (1)  —— 执行完成（后续调用直接返回）

// 状态转换：
// Incomplete + call_once → InProgress（执行闭包）
// InProgress + 其他线程 call_once → 一直等待到 Complete
// InProgress + 闭包完成 → Complete
// Complete + call_once → 立即返回（不执行闭包）

static INIT: Once = Once::new();

// 模拟两个线程同时调用：
let handle1 = thread::spawn(|| {
    INIT.call_once(|| {
        println!("thread1 is initializing...");
        thread::sleep(std::time::Duration::from_millis(100));
        println!("thread1 done");
    });
    println!("thread1 after call_once");
});

let handle2 = thread::spawn(|| {
    INIT.call_once(|| {
        // 这段代码永远不会执行，因为 handle1 抢到了执行权
        println!("thread2 is initializing...");
    });
    // 但 handle2 的 call_once 会阻塞等待 handle1 完成
    println!("thread2 after call_once");
});

handle1.join().unwrap();
handle2.join().unwrap();

// 可能输出：
// thread1 is initializing...
// thread1 done
// thread1 after call_once
// thread2 after call_once  ← thread2 在 handle1 执行完后才返回
//
// 注意：thread2 的 call_once 调用会阻塞，直到 handle1 完成
// 它不会返回"已经初始化过了"然后跳过——它等待初始化完成
```

### 2.2 call_once 是阻塞的

```rust
use std::sync::Once;
use std::thread;
use std::time::Duration;

// call_once 的阻塞语义：如果闭包正在另一个线程中执行，
// 当前线程会阻塞等待直到闭包执行完成

static INIT: Once = Once::new();

let long_init = thread::spawn(|| {
    INIT.call_once(|| {
        println!("long init starts...");
        thread::sleep(Duration::from_secs(2));
        println!("long init finishes");
    });
});

// 让 long_init 先拿到执行权
thread::sleep(Duration::from_millis(50));

// 后续线程的 call_once 会阻塞
let waiter = thread::spawn(|| {
    println!("waiter: calling call_once (will block)...");
    INIT.call_once(|| {
        // 不会被执行
        println!("waiter: inside closure");
    });
    println!("waiter: call_once returned (init complete)");
});

long_init.join().unwrap();
waiter.join().unwrap();

// 输出：
// long init starts...
// waiter: calling call_once (will block)...
// long init finishes
// waiter: call_once returned (init complete)
//
// 注意：waiter 在 "calling call_once" 之后直到 "long init finishes"
// 之后才继续执行——它被阻塞了 2 秒
```

这个行为非常重要：`call_once` 不仅保证"只执行一次"，还保证"后续调用者看到初始化完成后的状态"。如果它不阻塞，后续的 `log()` 调用可能看到半初始化的日志文件——这是 bugs 的温床。

### 2.3 panic 处理：Once 的"中毒"机制

```rust
use std::sync::Once;

static INIT: Once = Once::new();

// ⚠️ 关键行为：如果 call_once 的闭包 panic 了
// Once 被认为是"中毒"（poisoned）
// 下一次 call_once 将 NOT 重试——它会立即再次 panic！

// INIT.call_once(|| {
//     panic!("init failed!");
// });
//
// // 下一次调用：
// // INIT.call_once(|| {
// //     println!("this will NOT execute");
// // });
// // 这次 call_once 会直接 panic，不会执行闭包

// 一旦 Once 中的闭包 panic，这个 Once 实例就永久失效了
// 这是设计决策——不是 bug：
//
// 理由 1：Once 的闭包是 FnOnce——本身就不能重复调用
//         如果闭包 panic 了，它的状态是不确定的（可能已经消耗了资源）
//
// 理由 2：如果允许重试，可能导致"部分初始化"的问题
//         第一次执行：打开文件成功，设置共享变量失败 → panic
//         第二次重试：文件已经打开了（但共享变量没更新）→ 状态不一致
//
// 理由 3：与 Mutex 的"中毒"一致——Panic = Unrecoverable

// 如果你需要"失败后可重试"的一次性初始化，应该用 OnceLock：
// use std::sync::OnceLock;
// static CONFIG: OnceLock<String> = OnceLock::new();
// CONFIG.get_or_init(|| load_config().unwrap_or_else(|e| { ... }));
// // OnceLock 的 get_or_init 可以多次调用（如果值尚未设置成功）
```

Once 在 panic 时的"永久中毒"行为是它的一个关键缺陷——这也是为什么在更高层的 API（OnceLock, LazyLock）中这个行为被改善了。

## 3. Once 的常见使用场景

### 3.1 场景 1：初始化全局日志文件（再论）

```rust
use std::fs::{File, OpenOptions};
use std::io::Write;
use std::sync::{Mutex, Once};

// 完整的全局日志实现
static LOG_FILE: Mutex<Option<File>> = Mutex::new(None);
static LOG_INIT: Once = Once::new();

fn init_log(path: &str) {
    LOG_INIT.call_once(|| {
        let file = OpenOptions::new()
            .create(true)
            .append(true)
            .open(path)
            .expect("failed to open log file");
        *LOG_FILE.lock().unwrap() = Some(file);
        println!("[log system initialized]");
    });
}

fn write_log(level: &str, msg: &str) {
    // 注意：每次写日志都要检查是否已初始化
    // 如果没初始化，这条日志会丢失
    // 因为这个原因，实际项目中通常用专门的日志库（如 log + env_logger）
    // 它们内部就使用了 Once 或类似机制
    if let Some(ref mut file) = *LOG_FILE.lock().unwrap() {
        let _ = writeln!(file, "[{}] {}", level, msg);
    }
}

// 使用：
// init_log("app.log");
// write_log("INFO", "application started");
// write_log("DEBUG", "configuration loaded");
```

### 3.2 场景 2：设置信号处理器

```rust
use std::sync::Once;

// 信号处理器应该只被设置一次
// （多次设置会覆盖之前的处理器，可能导致不可预期行为）
static SIGNAL_INIT: Once = Once::new();

fn setup_signal_handlers() {
    SIGNAL_INIT.call_once(|| {
        // 在真实代码中，这里调用操作系统的 signal() 或 sigaction()
        // 或者使用 signal-hook crate
        println!("signal handlers installed");

        // ctrl-c 处理器
        // ctrlc::set_handler(|| {
        //     println!("received Ctrl-C, shutting down...");
        //     std::process::exit(0);
        // }).expect("failed to set Ctrl-C handler");
    });
}

// 在程序入口处调用：
// setup_signal_handlers();
// 后续再次调用不会重复设置
// setup_signal_handlers();  // no-op
```

### 3.3 场景 3：延迟加载的全局配置

```rust
use std::collections::HashMap;
use std::sync::{Mutex, Once};

static CONFIG: Mutex<Option<HashMap<String, String>>> = Mutex::new(None);
static CONFIG_INIT: Once = Once::new();

fn load_config() -> HashMap<String, String> {
    // 模拟从文件加载配置
    let mut config = HashMap::new();
    config.insert("host".to_string(), "localhost".to_string());
    config.insert("port".to_string(), "8080".to_string());
    config
}

fn get_config(key: &str) -> Option<String> {
    CONFIG_INIT.call_once(|| {
        let config = load_config();
        *CONFIG.lock().unwrap() = Some(config);
    });
    CONFIG.lock().unwrap()
        .as_ref()
        .and_then(|c| c.get(key).cloned())
}

// 第一次调用 get_config 时，自动加载配置
// 后续调用直接使用缓存的配置
// 这就是 OnceCell / OnceLock 要统一解决的问题
```

## 4. Once 的进化：从 Once 到 OnceLock 到 LazyLock

### 4.1 进化图谱

```
Rust 的一次性初始化体系：

Once (1.0+)
  │
  ├── 保证：闭包被执行恰好一次（或 panic）
  ├── 不存储值——需要额外的 static mut / Mutex<Option<T>>
  ├── panic 后永久中毒
  └── 使用者需要手动管理两个全局变量（Once + 存储）
      │
      ▼
OnceCell<T> (1.70+, !Sync — 单线程)
  │
  ├── Once + 值的存储合二为一
  ├── OnceCell<T> 同时管理"是否初始化"和"值"
  ├── get_or_init() 如果闭包 panic，下次调用会重试
  └── 单线程使用（不实现 Sync）
      │
      ▼
OnceLock<T> (1.70+, Sync — 多线程)
  │
  ├── OnceCell 的线程安全版本
  ├── 内部用 Once 或等效机制
  ├── get_or_init() 线程安全
  └── 适合全局 static
      │
      ▼
LazyLock<T> (1.80+, Sync)
  │
  ├── OnceLock + 初始化闭包自动化
  ├── 构造时传入闭包，首次访问时自动调用
  ├── static LAZY: LazyLock<String> = LazyLock::new(|| "hello".into());
  └── 类似 lazy_static! 但比 lazy_static! 更符合 Rust 习惯
```

### 4.2 各类型的选择指南

```rust
// ┌──────────────┬──────────┬──────────┬──────────────────────┐
// │ 类型         │ 存值     │ Sync     │ panic 后              │
// ├──────────────┼──────────┼──────────┼──────────────────────┤
// │ Once         │ 否       │ 是       │ 永久中毒              │
// │ OnceCell<T>  │ 是       │ 否(!Sync)│ 下次重试              │
// │ OnceLock<T>  │ 是       │ 是(Sync) │ 下次重试              │
// │ LazyLock<T>  │ 是       │ 是(Sync) │ 下次重试              │
// └──────────────┴──────────┴──────────┴──────────────────────┘

// 选择逻辑：
// - 用 Once:  当你只需要"执行某操作一次"，不需要返回值
//             （如打开日志文件、设置信号处理器、初始化 FFI 库）
//
// - 用 OnceCell:  单线程中的懒初始化值
//                 （如在 fn 内部延迟创建昂贵对象）
//
// - 用 OnceLock:  多线程中的全局懒初始化值（替代 Once + static mut）
//                 （如全局配置、全局缓存）
//
// - 用 LazyLock:  最方便——构造时直接传入初始化闭包
//                 （如 static CONFIG: LazyLock<Config> = LazyLock::new(|| ...)）
```

### 4.3 OnceLock 的使用

```rust
use std::sync::OnceLock;
use std::collections::HashMap;

// OnceLock 替代 "Once + Mutex<Option<T>>" 的模式
static CONFIG: OnceLock<HashMap<String, String>> = OnceLock::new();

fn get_or_load_config() -> &'static HashMap<String, String> {
    CONFIG.get_or_init(|| {
        println!("loading config...");
        let mut config = HashMap::new();
        config.insert("host".to_string(), "localhost".to_string());
        config.insert("port".to_string(), "8080".to_string());
        config
    })
}

// 使用：
let config = get_or_load_config();
println!("host: {}", config.get("host").unwrap());

// 与 Once + Mutex<Option<T>> 的对比：
// ✅ 类型更简洁：一个 static 而不是两个
// ✅ 不阻塞读取：初始化完成后，get() 不需要锁
// ✅ panic 可重试：init 闭包 panic 后，下一次 get_or_init 会重试
// ✅ 返回 &T 而非 MutexGuard：访问更方便
```

### 4.4 LazyLock 的使用

```rust
// LazyLock (1.80+ stabilized) —— 最接近 lazy_static! 的标准库等价物
use std::sync::LazyLock;

// static 懒初始化——类似 lazy_static!
static LOG_PREFIX: LazyLock<String> = LazyLock::new(|| {
    let now = std::time::SystemTime::now();
    format!("[{:?}]", now)
});

// LazyLock 的初始化闭包在首次解引用时自动执行
// println!("{}", *LOG_PREFIX);  // 首次访问时执行闭包

// LazyLock 也适合非 static 场景：
let expensive = LazyLock::new(|| {
    println!("computing...");
    (0..10_000_000).sum::<u64>()
});

// 多次解引用只计算一次
let a = *expensive;  // "computing..."
let b = *expensive;  // 直接返回缓存值（不打印）
assert_eq!(a, b);
```

## 5. Once 的底层实现概览

### 5.1 内部机制

```rust
// Once 的内部实现是一个精巧的状态机（简化版概念）：
//
// use std::sync::atomic::{AtomicU32, Ordering};
// use std::thread;
//
// const INCOMPLETE: u32 = 0;
// const RUNNING: u32 = 1;
// const COMPLETE: u32 = 2;
//
// struct Once {
//     state: AtomicU32,
//     // 实际中还有用于等待/唤醒的 futex 或 WaitOnAddress
// }
//
// impl Once {
//     fn call_once(&self, f: impl FnOnce()) {
//         // 快速路径：如果已经完成了，直接返回
//         if self.state.load(Ordering::Acquire) == COMPLETE {
//             return;
//         }
//
//         // 慢路径：尝试获取执行权
//         match self.state.compare_exchange(
//             INCOMPLETE, RUNNING, Ordering::Acquire, Ordering::Acquire
//         ) {
//             Ok(_) => {
//                 // 当前线程负责执行闭包
//                 f();
//                 self.state.store(COMPLETE, Ordering::Release);
//                 // 唤醒所有等待的线程（futex wake）
//             }
//             Err(RUNNING) => {
//                 // 另一个线程正在执行——等待它完成
//                 while self.state.load(Ordering::Acquire) != COMPLETE {
//                     thread::yield_now();
//                     // 实际中使用 futex / WaitOnAddress 阻塞等待
//                 }
//             }
//             Err(COMPLETE) => {
//                 // 在 compare_exchange 期间另一个线程完成了
//                 return;
//             }
//             _ => unreachable!(),
//         }
//     }
// }

// 关键点：
// 1. 快速路径：一个 acquire load → 如果已 COMPLETE → 直接返回（最常见情况）
//    这解释了为什么后续调用几乎零开销
//
// 2. 慢路径：compare_exchange 获取执行权
//    只有一个线程能成功（CAS 的原子性保证）
//
// 3. 等待：其他线程循环等待或通过 futex 被挂起
//    被挂起的方式避免了忙等
```

### 5.2 Once 不需要 Mutex 的原因

```rust
// 有人会问：为什么要用 Once 而不是 Mutex<bool> 来实现一次性初始化？
//
// 答案：
//
// 1. Mutex 有锁定/解锁开销——即使初始化完成后，每次检查都要拿锁
//    Once 的 COMPLETE 状态检查是单条原子 load 指令——极快
//
// 2. Mutex<bool> 不能区分"正在初始化"和"已完成"
//    如果初始化需要 2 秒，第二个线程调用时：
//    - Once: 阻塞等待初始化完成（正确行为）
//    - Mutex<bool>: 看到 false，尝试拿锁，被阻塞... 锁释放后拿到锁，看到 true → 行了
//      但这需要初始化逻辑正确地设置 flag（容易出错）
//
// 3. 公平性：Once 内部可以高效地唤醒所有等待线程（futex wake all）
//    而 Mutex 每次只唤醒一个
```

## 6. 实战：从零构建 OnceLock

### 6.1 用 Once 实现 OnceLock

```rust
use std::sync::Once;
use std::cell::UnsafeCell;

// OnceLock 本质上就是 Once + 值的存储
// 这是概念性的简化实现——实际的 std::sync::OnceLock 更复杂

struct SimpleOnceLock<T> {
    once: Once,
    // UnsafeCell 是必须的——我们需要在共享引用下写入值
    value: UnsafeCell<Option<T>>,
    // 注意：Option<T> 需要额外追踪"是否有值"
    // 但 Once 已经提供了这个信息（COMPLETE = 有值）
    // 所以真实的 OnceLock 不需要额外的 Option
}

// 为了线程安全，SimpleOnceLock 需要 unsafe impl Sync
// 实际的 OnceLock 在标准库中的实现经过了严格的安全审查

impl<T> SimpleOnceLock<T> {
    const fn new() -> Self {
        SimpleOnceLock {
            once: Once::new(),
            value: UnsafeCell::new(None),
        }
    }

    fn get_or_init(&self, f: impl FnOnce() -> T) -> &T {
        self.once.call_once(|| {
            // 安全：Once 保证这段代码只执行一次
            // 此时没有其他线程访问 value
            unsafe {
                *self.value.get() = Some(f());
            }
        });
        // 初始化完成后，value 一定是 Some
        // 安全：Once 保证此时不会再有写入
        unsafe {
            (*self.value.get()).as_ref().unwrap()
        }
    }

    fn get(&self) -> Option<&T> {
        // 如果没有初始化，返回 None
        // 但标准库的 OnceLock::get() 实际上不检查 Once 状态
        // 而是通过内部标志位来判断
        if self.once.is_completed() {
            unsafe { (*self.value.get()).as_ref() }
        } else {
            None
        }
    }
}

// 这个简化版展示了 Once → OnceLock 的核心思路：
// Once 提供"执行一次"的保证，UnsafeCell 提供写入能力
// 两者结合 = 线程安全的懒初始化值
```

## 8. 练习答案

### 练习 1 答案

```rust
use std::sync::atomic::{AtomicBool, Ordering};
use std::sync::Once;

struct InitOnce {
    started: AtomicBool,
    done: AtomicBool,
}

impl InitOnce {
    fn new() -> Self {
        InitOnce {
            started: AtomicBool::new(false),
            done: AtomicBool::new(false),
        }
    }

    fn call_once_retry<F, T, E>(&self, f: F) -> Result<(), E>
    where
        F: FnOnce() -> Result<T, E>,
    {
        // 快速路径：已经完成 → 直接返回
        if self.done.load(Ordering::Acquire) {
            return Ok(());
        }

        // 尝试获取执行权
        if self.started.compare_exchange(
            false, true, Ordering::Acquire, Ordering::Relaxed
        ).is_ok() {
            // 当前线程负责执行
            let result = f();
            match result {
                Ok(_) => {
                    self.done.store(true, Ordering::Release);
                    self.started.store(false, Ordering::Release);
                    Ok(())
                }
                Err(e) => {
                    // 失败：允许重试
                    self.started.store(false, Ordering::Release);
                    Err(e)
                }
            }
        } else {
            // 另一个线程正在执行——等待
            while self.started.load(Ordering::Acquire) && !self.done.load(Ordering::Acquire) {
                std::thread::yield_now();
            }
            if self.done.load(Ordering::Acquire) {
                Ok(())
            } else {
                // 上一次执行失败了，但自己没抢到执行权
                // 简化：让调用者下次再试
                Err(panic!("TODO: complex retry logic"))
                // 实际中可以用更复杂的机制（如 Condvar）来高效等待和重试
            }
        }
    }
}

// 简化版的主要缺陷：
// - 没有处理"执行失败后多线程重试"的竞争（多线程可能同时决定重试）
// - 使用了 spin-wait（生产代码应该用 Condvar 或 futex）
// 但这个简化版展示了核心概念：分离"正在执行"和"已完成"两个状态
```

### 练习 2 答案

```rust
use std::sync::atomic::{AtomicU64, Ordering};
use std::sync::Once;

static COUNTER: AtomicU64 = AtomicU64::new(0);
static COUNTER_INIT: Once = Once::new();

fn next_id() -> u64 {
    COUNTER_INIT.call_once(|| {
        // 从文件读取上次保存的起始 ID
        // 如果文件不存在或读取失败，从 0 开始
        let start_id = std::fs::read_to_string("counter.save")
            .ok()
            .and_then(|s| s.trim().parse::<u64>().ok())
            .unwrap_or(0);
        COUNTER.store(start_id, Ordering::SeqCst);
        println!("counter initialized from file: start_id={}", start_id);
    });

    COUNTER.fetch_add(1, Ordering::SeqCst)
}

// 程序退出时保存（使用 atexit 或 Drop）
struct CounterGuard;

impl Drop for CounterGuard {
    fn drop(&mut self) {
        // 仅在初始化被执行后才保存
        // 注意：这里无法检查 Once 的状态
        // 简化处理：如果 COUNTER > 0 则保存
        let val = COUNTER.load(Ordering::SeqCst);
        if val > 0 {
            let _ = std::fs::write("counter.save", val.to_string());
            println!("counter saved: {}", val);
        }
    }
}

// 在 main 中：
// let _guard = CounterGuard;
// println!("id: {}", next_id());  // 1
// println!("id: {}", next_id());  // 2
// // 程序退出时自动保存 2 到文件
```

### 练习 3 答案

```rust
// 问题 1：为什么 lazy_static! 需要 static mut？
//
// lazy_static! 展开的核心结构：
// struct LazyWrapper<T> {
//     init: Once,
//     value: UnsafeCell<Option<T>>,
// }
//
// 实际上 lazy_static! 用 Once 管理"是否初始化"，
// 用 static mut Option<T> 存储值。
// static mut 是必需的，因为：
// - 值需要在 Once 的闭包中写入（需要可变性）
// - 但又必须是全局的（需要 static）
// - OnceLock 出现之前，没有 Sync 的"可写的 static"方案
//
// static mut 的代价：
// - 每次使用需要 unsafe（编译器不能验证安全性）
// - 程序员必须手动保证"只在 Once 内部写入"
//
// 问题 2：用 OnceLock 替代的优势：
// - 消除 unsafe：OnceLock 提供安全的 API
// - 更好的封装：一个 static 代替两个
// - panic 可重试：OnceLock 的初始化闭包 panic 后可以重试
// - 类型更清晰：OnceLock<T> 直接表达"懒初始化的 T"
//
// 这就是为什么 Rust 社区在推动用 OnceLock/LazyLock 替代 lazy_static!
```

Once / OnceLock / LazyLock 学习清单：

□ 理解 Once 的核心价值：多线程安全的一次性执行

□ 理解 Once → OnceCell → OnceLock → LazyLock 的进化路径

> **从 Once 开始，Rust 的一次性初始化体系清晰了起来**：`Once` 是"执行一次"的原子保证（没有存储），`OnceLock` 在此基础上加了存储，`LazyLock` 再加了构造时绑定闭包的便利。这三者组成了从底层到高层的完整链条。Phase 2 中我们已经学习了 [[2.14-once-lazy-init|OnceCell/LazyCell 的单线程版本]]，现在有了多线程版本的 OnceLock/LazyLock，你已经掌握了 Rust 懒初始化的完整图景。接下来，Phase 3 将进入异步的世界——`Future`、`async/await` 和 `tokio`——在那里，"等待"将不再是阻塞系统线程，而是通过状态机实现的任务切换。
