# 3.3 Mutex / RwLock 原理与实战 —— 临界区的艺术

> 并发编程中最古老的挑战是"如何让多个线程安全地共享可变状态"。Rust 的答案是将数据包裹在锁中，然后用 RAII 守卫（`MutexGuard`）管理对数据的访问——在你持有守卫的这段时间里，你拥有对数据的独占访问权，就像你持有一个 `&mut T`。这个模型如此优雅，以至于它成了 Rust 并发编程的标志性模式。但锁的使用远不止 `lock().unwrap()` 这么简单：`std::sync::Mutex` 和 `tokio::sync::Mutex` 有着本质的不同（阻塞 vs 异步），`RwLock` 在读写场景之间做取舍，而 `parking_lot` 库展示了标准库实现可以如何被超越。理解锁的内部实现——从原子操作到操作系统原语（futex）的降级路径——能帮助你在"锁争用"发生时做出正确的架构决策。

## 1. std::sync::Mutex —— 标准库的阻塞互斥锁

### 1.1 Mutex 的基本用法和 RAII 守卫

`std::sync::Mutex<T>` 是 Rust 中最基础的线程间互斥锁。它将数据 `T` 包裹起来，保护对数据的并发访问。

```rust
use std::sync::Mutex;
use std::thread;

// Mutex<T> 的使用模式
fn basic_mutex() {
    // 创建 Mutex——数据被包裹在锁内部
    let m = Mutex::new(42);

    // lock() 返回 LockResult<MutexGuard<T>>
    // MutexGuard 实现了 Deref<Target = T> 和 DerefMut
    {
        let mut guard = m.lock().unwrap();  // 获取锁
        *guard += 1;                        // 通过 DerefMut 修改内部数据
        println!("value: {}", *guard);       // 通过 Deref 读取内部数据
        // guard 在这里离开作用域——自动释放锁（Drop）
    }

    // 此时锁已经释放，可以再次获取
    let guard = m.lock().unwrap();
    assert_eq!(*guard, 43);

    // 关键设计：Mutex<T> 拥有数据 T
    // 不存在"锁和数据分离"的模式
    // 这在 Rust 中是编译期保证的——不持有锁就无法访问数据
}
```

Rust 的 `Mutex<T>` 设计有一个核心哲学：**数据被锁拥有，而不是锁和数据分离**。在 C/C++ 中，常见的模式是用一个独立的互斥锁变量来保护某个数据结构。这种分离导致了大量 bug——忘记加锁、锁了不同的 mutex、在锁释放后继续使用指针。Rust 通过将数据嵌入 `Mutex` 内部，并由编译器强制执行"必须通过锁才能访问数据"的规则，彻底消除了一整类并发 bug。

### 1.2 MutexGuard 的生命周期管理

`MutexGuard` 是一个 RAII 守卫——获取它时加锁，销毁它时自动解锁。这是 Rust 防止"忘记解锁"的关键机制。

```rust
use std::sync::Mutex;

fn mutex_guard_lifetime() {
    let m = Mutex::new(vec![1, 2, 3]);

    // MutexGuard<'_, Vec<i32>> —— 生命周期被约束在 Mutex 的生命周期内
    let mut guard = m.lock().unwrap();

    // 可以通过 guard 修改 Vec
    guard.push(4);

    // MutexGuard 实现了 Deref 和 DerefMut
    // 所以你可以调用 Vec 的方法
    let len = guard.len();  // 等价于 (&*guard).len()

    // ❌ 不能把 MutexGuard 发送到另一个线程
    // MutexGuard 不是 Send——设计如此
    // thread::spawn(move || { drop(guard); });  // 编译错误

    // ✅ 显式释放锁
    drop(guard);  // 提前解锁
    // 现在可以再次 lock() 了

    // ❌ 常见错误：在 guard 存活时尝试再次 lock
    // let guard = m.lock().unwrap();
    // let guard2 = m.lock().unwrap();  // 死锁！—— 同线程重复加锁
    // Mutex 不是可重入的（non-reentrant）
}
```

`MutexGuard` 的 `!Send` 特性是一个重要的安全设计：它确保一个线程持有的锁守卫不会被转移到另一个线程。如果允许发送，其他线程可能在不同时间解锁——违反了"谁加锁谁解锁"的原则。

## 2. Mutex vs tokio::sync::Mutex —— 阻塞和异步的分水岭

### 2.1 关键差异 —— 为什么不能跨 .await 持有 std Mutex

这是 Rust 异步编程中最容易犯的错误之一。`std::sync::Mutex` 的 `lock()` 会阻塞当前线程直到获取锁——在异步上下文中，这意味着阻塞整个 executor 线程。

```rust
use std::sync::Mutex;
use std::sync::Arc;

// ❌ 致命错误：在异步代码中跨 .await 持有 std::sync::Mutex 的锁
async fn bad_async_mutex_example() {
    let data = Arc::new(Mutex::new(0));

    // 在某个 async 函数中...
    async fn do_something(data: Arc<Mutex<i32>>) {
        let guard = data.lock().unwrap();  // 获取 std Mutex 的锁
        // ... 做一些同步操作 ...

        // ❌ 致命的错误！
        // tokio::time::sleep(Duration::from_secs(1)).await;
        // 在 .await 期间 guard 仍然存活！
        // → Mutex 一直被锁定
        // → 其他任务无法获取锁
        // → 如果其他任务被调度到同一线程 → 死锁
        // → 即使不在同一线程——该 Mutex 被锁定了整整 1 秒

        *guard += 1;
        // guard 在这里 drop——锁在 1 秒后才释放
    }
}

// 问题分析：
//
// std::sync::Mutex 基于操作系统原语（futex / pthread_mutex）
// lock() 阻塞调用线程——如果不是可重入锁，同一线程再次 lock → 死锁
//
// 在异步世界中：
// 一个线程运行多个任务（通过 executor 调度）
// 如果任务 A 持有 Mutex 并在 .await 处挂起
// 任务 B 被调度到同一线程
// 任务 B 尝试获取同一个 Mutex → 线程阻塞！
// executor 无法再调度任何任务 → 整个线程卡死
//
// 这就是为什么 tokio 有自己的 Mutex——
// tokio::sync::Mutex 的 lock() 返回 Future
// 在等待锁期间会 yield 给 executor
```

`std::sync::Mutex` 不应该跨 `.await` 持有的核心原因是：异步运行时通过协作式调度来执行多个任务（在同一个线程上），而 `std::sync::Mutex::lock()` 会阻塞操作系统线程——这意味着如果锁被持有，整个线程上的所有任务都会被阻塞。

### 2.2 tokio::sync::Mutex —— 为异步设计的互斥锁

`tokio::sync::Mutex` 解决了上述问题——它的 `lock()` 返回一个 `Future`，在等待锁的过程中可以 yield 给 executor。

```rust
// 需要在 Cargo.toml 中添加：
// tokio = { version = "1", features = ["full"] }

use tokio::sync::Mutex;
use std::sync::Arc;
use std::time::Duration;

async fn tokio_mutex_example() {
    let data = Arc::new(Mutex::new(0));

    let d1 = data.clone();
    let task1 = tokio::spawn(async move {
        // tokio::sync::Mutex::lock() 返回 Future
        // 等待锁时会 yield，不阻塞线程
        let mut guard = d1.lock().await;  // ← .await！不会阻塞线程
        *guard += 1;

        // ✅ 可以安全地跨 .await 持有 tokio::sync::Mutex 的锁
        tokio::time::sleep(Duration::from_millis(100)).await;
        // 注意：虽然不会死锁，但长时间持有异步锁
        // 仍会造成严重的竞争（其他任务一直在等）
        // 尽可能缩短持有锁的时间

        *guard += 1;
        // guard 在这里 drop——锁被释放
    });

    let d2 = data.clone();
    let task2 = tokio::spawn(async move {
        let mut guard = d2.lock().await;  // 等待 task1 释放锁
        *guard *= 2;
    });

    // 等待两个任务完成
    let _ = tokio::join!(task1, task2);
    assert_eq!(*data.lock().await, 4);
}
```

两个 Mutex 的使用场景有清晰的分界线：如果你在同步代码（`std::thread::spawn`）中，用 `std::sync::Mutex`。如果你在异步代码中，并且锁的持有期会跨越 `.await`，用 `tokio::sync::Mutex`。如果你在异步代码中，但锁的持有期非常短（在 `.await` 之前就会释放），`std::sync::Mutex` 也可以安全使用——因为线程被阻塞的时间可以忽略不计。

### 2.3 两个 Mutex 的实现差异 —— 从内部机制理解

理解两个 Mutex 的底层实现差异可以帮助你在正确的场景选择正确的锁。

```rust
// std::sync::Mutex 的内部实现路径（简化）：
//
// 1. lock() → 尝试原子地将锁状态从 0 改为 1
//    - 如果成功 → 立即返回（fast path，纯原子操作）
//    - 如果失败 → 进入 slow path
//
// 2. slow path → 调用操作系统原语
//    - Linux:  futex(FUTEX_WAIT)  → 线程被挂起，OS 调度其他线程
//    - macOS:  pthread_mutex_lock  → 同样的效果
//    - Windows: WaitOnAddress / SRWLock
//
// 3. unlock() → 同一线程将锁状态改回 0
//    - 如果有等待者：futex(FUTEX_WAKE) → 唤醒一个等待的线程
//
// 关键点：slow path 阻塞的是 OS 线程，不是 Rust 的 task

// tokio::sync::Mutex 的内部实现路径（简化）：
//
// 1. lock() → 返回 Future<Output = MutexGuard>
//    - try_lock() 快速路径：原子尝试，如果成功直接返回 Ready
//    - 否则：将当前 task 的 waker 注册到等待队列
//           然后返回 Pending → executor 调度其他 task
//
// 2. unlock() → 从等待队列中取出下一个 waker
//    - 调用 waker.wake() → executor 重新调度该 task
//
// 关键点：整个过程中没有任何 OS 线程被阻塞
// 代价：tokio Mutex 有额外的内存分配（等待队列）
//      lock() 返回 Future 而不是同步值

// 性能对比（概念性的）：
// ┌────────────────────┬──────────────────────┬──────────────────────┐
// │ 场景               │ std::sync::Mutex     │ tokio::sync::Mutex   │
// ├────────────────────┼──────────────────────┼──────────────────────┤
// │ 无竞争 lock        │ ~1 次原子 CAS        │ ~1 次原子 CAS        │
// │ lock + unlock      │ ~2 次原子操作        │ ~2 次原子操作        │
// │ 有竞争（低）       │ futex wait/wake      │ waker 注册/唤醒      │
// │ 有竞争（高）       │ OS 线程调度          │ executor 调度        │
// │ 持有期间 .await    │ ❌ 阻塞线程/死锁     │ ✅ 安全（yield）     │
// │ 持有期间 CPU 计算  │ ✅ 合适              │ ⚠️ 应避免长时间持有   │
// └────────────────────┴──────────────────────┴──────────────────────┘
//
// 选择原则：
// 1. 同步代码 → std::sync::Mutex
// 2. 异步代码 + 不跨 .await → std::sync::Mutex（更低开销）
// 3. 异步代码 + 跨 .await → tokio::sync::Mutex
// 4. 锁持有时间应尽可能短——无论哪种 Mutex
```

## 3. RwLock —— 读写锁的原理和权衡

### 3.1 RwLock 的基本使用

`RwLock<T>` 允许多个并发读取或独占写入。读取者之间不互斥，但写入者与读取者和其他写入者互斥。

```rust
use std::sync::RwLock;
use std::thread;

fn rwlock_basic() {
    let data = RwLock::new(42);

    // 多个读锁可以同时持有
    {
        let r1 = data.read().unwrap();
        let r2 = data.read().unwrap();  // ✅ 多个读者同时持有
        println!("r1: {}, r2: {}", *r1, *r2);
        // r1 和 r2 在这里被 drop
    }

    // 写锁是独占的
    {
        let mut w = data.write().unwrap();
        *w += 1;
        // 此时不能获取读锁或写锁
        // let r = data.read().unwrap();  // 会阻塞
    }  // w 被 drop，写锁释放

    // RwLockReadGuard 和 RwLockWriteGuard
    // 都是 RAII 守卫——离开作用域自动释放
}
```

`RwLock` 的核心权衡是：读取操作非常频繁而写入操作很少时，`RwLock` 能提供比 `Mutex` 更好的并发吞吐量——因为多个读锁可以同时持有，互不阻塞。

### 3.2 写入者饥饿问题和公平性策略

`RwLock` 在实际实现中面临一个经典问题：写入者饥饿。如果读取者持续到来，写入者可能永远等不到锁。

```rust
use std::sync::RwLock;

// 写入者饥饿的场景（概念演示）
//
// 时间 →
// 读锁持有者 1: [=============]
// 写锁等待者:                     [一直在等...]
// 读锁持有者 2:        [======]
// 读锁持有者 3:               [=========]
// 读锁持有者 4:                       [=======]
// ...
//
// 如果读请求持续不断，写请求可能永远得不到服务
// 这就是"写入者饥饿"

// std::sync::RwLock 的实现策略：
// 标准库的文档明确说明：
// "This RWLock uses a task-fair locking policy,
//  which avoids reader and writer starvation."
//
// 标准库的 RwLock 对公平性的处理策略因平台而异：
// - 在 Linux 上（pthread_rwlock）：通常偏向写入者
// - 在 Windows 上（SRWLock）：通常也是写入者优先
// - 但文档不保证任何特定的公平性策略

// parking_lot::RwLock 提供了显式的公平性控制：
// use parking_lot::RwLock;
//
// fn parking_lot_fairness() {
//     // 默认策略：偏向写入者（避免写入者饥饿）
//     let rw = parking_lot::RwLock::new(42);
//
//     // 也可以使用公平锁（严格 FIFO 顺序）
//     // let rw = parking_lot::FairRwLock::new(42);
//     // 公平锁保证按申请顺序服务——代价是吞吐量降低
// }
```

选择 `RwLock` 时的关键权衡：
- **读多写少** → `RwLock` 比 `Mutex` 更好（并发读）
- **读写比例接近** → `Mutex` 可能更好（`RwLock` 的写锁比 `Mutex` 的锁有更高的开销）
- **写操作很频繁** → `Mutex` 通常更好（避免读写锁的额外开销和潜在饥饿）

### 3.3 RwLock vs Mutex 的性能分析

用具体的场景来分析 `RwLock` 和 `Mutex` 的性能差异。

```rust
use std::sync::{Arc, Mutex, RwLock};
use std::thread;
use std::time::Instant;

// 场景：对共享数据进行"大量读取 + 少量写入"
fn benchmark_read_heavy() {
    const READERS: usize = 8;
    const WRITERS: usize = 1;
    const OPS_PER_THREAD: usize = 1_000_000;

    // --- Mutex 版本 ---
    {
        let data = Arc::new(Mutex::new(0u64));
        let start = Instant::now();
        let mut handles = vec![];

        // 读取者线程
        for _ in 0..READERS {
            let d = data.clone();
            handles.push(thread::spawn(move || {
                let mut local = 0u64;
                for _ in 0..OPS_PER_THREAD {
                    let val = *d.lock().unwrap();
                    local = local.wrapping_add(val);
                }
                local
            }));
        }

        // 写入者线程
        for _ in 0..WRITERS {
            let d = data.clone();
            handles.push(thread::spawn(move || {
                for i in 0..(OPS_PER_THREAD / 100) {
                    *d.lock().unwrap() = i as u64;
                }
            }));
        }

        for h in handles {
            h.join().unwrap();
        }
        println!("Mutex: {:?}", start.elapsed());
    }

    // --- RwLock 版本 ---
    {
        let data = Arc::new(RwLock::new(0u64));
        let start = Instant::now();
        let mut handles = vec![];

        // 读取者线程
        for _ in 0..READERS {
            let d = data.clone();
            handles.push(thread::spawn(move || {
                let mut local = 0u64;
                for _ in 0..OPS_PER_THREAD {
                    let val = *d.read().unwrap();  // 使用 read() 而不是 lock()
                    local = local.wrapping_add(val);
                }
                local
            }));
        }

        // 写入者线程
        for _ in 0..WRITERS {
            let d = data.clone();
            handles.push(thread::spawn(move || {
                for i in 0..(OPS_PER_THREAD / 100) {
                    *d.write().unwrap() = i as u64;
                }
            }));
        }

        for h in handles {
            h.join().unwrap();
        }
        println!("RwLock: {:?}", start.elapsed());
    }

    // 典型结果（多核 CPU）：
    // Mutex:  ~1200ms  （所有读被串行化）
    // RwLock: ~200ms   （多个读并发执行）
    // → RwLock 在"读多写少"场景下有约 6 倍的性能优势
}
```

## 4. 锁中毒 —— Mutex Poisoning 机制

### 4.1 什么是锁中毒

"锁中毒"是 Rust 的 `Mutex` 和 `RwLock` 特有的安全机制：当持有锁的线程 panic 时，锁会被标记为"中毒"（poisoned），之后任何线程在尝试 `lock()` 时都会收到一个 `PoisonError`。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn mutex_poisoning() {
    let data = Arc::new(Mutex::new(0));

    // 线程 1：持有锁并 panic
    let d1 = data.clone();
    let t1 = thread::spawn(move || {
        let _guard = d1.lock().unwrap();
        // 持有锁后 panic——锁被污染
        panic!("oops!");
        // guard 不会被正常 Drop（panic 展开时栈被清理，但锁状态不一致）
    });

    // 等待线程 1 完成（它 panic 了）
    let _ = t1.join();  // 返回 Err

    // 线程 2：尝试获取同一个锁
    match data.lock() {
        Ok(_guard) => {
            println!("锁是干净的");
        }
        Err(poisoned) => {
            // 锁中毒了！但我们依然可以获取 guard
            let mut guard = poisoned.into_inner();
            println!("锁中毒了，但数据还在: {}", *guard);
            // 可以选择修复数据、记录日志或直接 panic
            *guard = 0;  // 重置数据
        }
    }
}

// Poisoning 的设计哲学：
//
// 当持有 MutexGuard 的线程 panic 时，被保护的数据可能处于不一致状态
// 例如，线程在修改数据结构的"半路上"panic 了
// 后续获取锁的线程需要知道"数据可能已经损坏"
//
// Poisoning 不是强制性的——你可以：
// 1. 用 poisoned.into_inner() 获取 guard 并手动修复数据
// 2. 用 lock().unwrap() 忽略中毒（如果真的确定数据没问题）
// 3. 用 lock().unwrap_or_else(|e| e.into_inner()) 获取 guard 但标记为已处理
```

锁中毒是 Rust 在"防御性编程"和"实用性"之间的折中。它提醒你数据可能已经不一致，但不强制终止整个进程——给你一个机会来检查和修复。

### 4.2 RwLock 的中毒行为

`RwLock` 的中毒行为比 `Mutex` 稍微复杂——因为存在读锁和写锁两种守卫。

```rust
use std::sync::{Arc, RwLock};
use std::thread;

fn rwlock_poisoning() {
    let data = Arc::new(RwLock::new(vec![1, 2, 3]));

    // 线程 1：持有写锁并 panic
    let d1 = data.clone();
    let t1 = thread::spawn(move || {
        let mut w = d1.write().unwrap();
        w.push(4);
        panic!("写锁中 panic!");  // RwLock 被污染
    });

    let _ = t1.join();

    // 后续读取——也会返回 PoisonError
    match data.read() {
        Ok(r) => println!("读到: {:?}", *r),
        Err(poisoned) => {
            let r = poisoned.into_inner();
            println!("中毒后的数据（可能不完整）: {:?}", *r);
            // 此时 Vec 可能是 [1, 2, 3, 4]（push 完成后才 panic）
            // 也可能是不一致的状态（虽然 Vec 不太可能如此）
        }
    }

    // 对于只是读取的场景，也可以使用 get_mut（如果不需要并发）
    // 或者先"治愈"锁：清除中毒标记
    // （标准库没有直接的"治愈"API——需要用 into_inner 取出数据再重新构造）
}
```

## 5. parking_lot —— 标准库锁的替代品

### 5.1 parking_lot 对比 std 的优势

`parking_lot` 是一个高性能的同步原语库，提供了标准库的 `Mutex`、`RwLock` 等类型的替代实现，通常有更好的性能。

```rust
// 需要在 Cargo.toml 中添加：
// parking_lot = "0.12"

// parking_lot::Mutex vs std::sync::Mutex 的关键差异：
//
// 1. 性能：parking_lot 通常比 std 快
//    - 更优化的 futex 使用（减少系统调用）
//    - 更好的自旋 + yield 策略（减少上下文切换）
//    - 没有 poisoning 开销（不需要维护中毒状态）
//
// 2. 特性：
//    - parking_lot::Mutex 没有 poisoning
//      （认为 poisoning 的代价大于收益）
//    - 支持 try_lock_for / try_lock_until（带超时的锁尝试）
//    - 多种公平性策略可选
//
// 3. API 差异：
//    - parking_lot 的 lock() 直接返回 MutexGuard（不是 Result）
//    - 因为没有 poisoning，不需要 unwrap

use parking_lot::Mutex as ParkingLotMutex;
use std::sync::Mutex as StdMutex;

fn parking_lot_example() {
    let m = ParkingLotMutex::new(0);

    // lock() 直接返回 MutexGuard（不需要 unwrap——没有 poisoning）
    let mut guard = m.lock();
    *guard += 1;

    // try_lock() 返回 Option<MutexGuard>
    if let Some(mut g) = m.try_lock() {
        *g += 1;
    }

    // try_lock_for —— 在指定时间内尝试获取锁
    use std::time::Duration;
    if let Some(mut g) = m.try_lock_for(Duration::from_millis(10)) {
        *g += 1;
    }
    // parking_lot 的这些额外 API 在某些场景中非常实用
}
```

### 5.2 性能差异的根因

`parking_lot` 比标准库快的核心原因在于它更积极地使用自旋（spinning）来避免昂贵的系统调用。

```rust
// 标准库 Mutex 的锁获取路径（简化）：
//
// fast path:
//   尝试原子 CAS 获取锁
//   成功 → 返回
//
// slow path:
//   失败一次 → 立即调用 futex_wait（系统调用，开销大）
//   futex_wait 将线程挂起 → 上下文切换
//
//
// parking_lot Mutex 的锁获取路径（简化）：
//
// fast path:
//   尝试原子 CAS 获取锁
//   成功 → 返回
//
// medium path（自旋循环）:
//   失败 → 自旋 N 次（纯 CPU 循环，无系统调用）
//   每次自旋检查锁是否释放
//   如果释放 → CAS 获取 → 成功返回
//
// slow path:
//   自旋 N 次后仍然失败 → 调用 futex_wait
//   但同时注册了一个"线程 park 队列"（避免惊群效应）
//
//
// 在锁持有时间很短（几个 CPU 周期）的场景下：
// parking_lot 的"自旋等待"避免了系统调用的开销
// 性能可以比 std 高 2-5 倍

// 标准库实现使用"轻自旋"策略——只自旋很少的次数
// 然后立刻进入 futex——因为它优先考虑公平性
//
// parking_lot 使用"重自旋"策略——自旋较多
// 因为它优先考虑吞吐量

// 选择建议：
// - 锁持有时间极短（< 100ns） → parking_lot（自旋优于阻塞）
// - 锁持有时间较长（> 1us）    → 两者差异不大（都会进入 futex）
// - 需要 poisoning 保护        → std::sync::Mutex
// - 需要 try_lock_for          → parking_lot
// - 不希望引入外部依赖           → std::sync::Mutex
```

## 6. 死锁 —— 场景和预防

### 6.1 经典的死锁场景

死锁发生在两个或多个线程相互等待对方释放锁，从而永远无法继续前进。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

// 经典死锁模式：两个线程需要获取两个锁，但顺序不同
fn deadlock_example() {
    let lock_a = Arc::new(Mutex::new(0));
    let lock_b = Arc::new(Mutex::new(0));

    let a1 = lock_a.clone();
    let b1 = lock_b.clone();
    let t1 = thread::spawn(move || {
        let _ga = a1.lock().unwrap();  // 获取 A
        println!("线程 1: 获取了 A");
        std::thread::sleep(std::time::Duration::from_millis(10));
        let _gb = b1.lock().unwrap();  // 等待 B（但 B 被线程 2 持有）
        println!("线程 1: 获取了 B");
    });

    let a2 = lock_a.clone();
    let b2 = lock_b.clone();
    let t2 = thread::spawn(move || {
        let _gb = b2.lock().unwrap();  // 获取 B
        println!("线程 2: 获取了 B");
        std::thread::sleep(std::time::Duration::from_millis(10));
        let _ga = a2.lock().unwrap();  // 等待 A（但 A 被线程 1 持有）
        println!("线程 2: 获取了 A");
    });

    t1.join().unwrap();
    t2.join().unwrap();

    // 执行时间线：
    //
    // 时间 →
    // 线程 1: lock(A) ────┐              等待 B...
    // 线程 2:      lock(B) ────┐         等待 A...
    //
    // 线程 1 持有 A，等待 B
    // 线程 2 持有 B，等待 A
    // → 谁也无法继续 → 死锁

    // Rust 无法在编译期检测这种死锁
    // ——它发生在运行时，依赖代码逻辑
}
```

### 6.2 预防死锁的策略

以下是在 Rust 中预防死锁的实用策略。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

// 策略 1：锁的获取顺序一致（最有效）
fn lock_ordering() {
    let lock_a = Arc::new(Mutex::new(0));
    let lock_b = Arc::new(Mutex::new(0));

    // ✅ 两个线程都按 A → B 的顺序获取锁
    let a1 = lock_a.clone();
    let b1 = lock_b.clone();
    let t1 = thread::spawn(move || {
        let _ga = a1.lock().unwrap();
        let _gb = b1.lock().unwrap();
        // ... 使用两个锁 ...
    });

    let a2 = lock_a.clone();
    let b2 = lock_b.clone();
    let t2 = thread::spawn(move || {
        let _ga = a2.lock().unwrap();  // ← 同样是 A 先
        let _gb = b2.lock().unwrap();  // ← 然后是 B
        // ... 使用两个锁 ...
    });
    // 永远不会有死锁——因为获取顺序一致
}

// 策略 2：使用 try_lock 和重试
fn try_lock_approach() {
    let lock_a = Arc::new(Mutex::new(0));
    let lock_b = Arc::new(Mutex::new(0));

    let a = lock_a.clone();
    let b = lock_b.clone();
    thread::spawn(move || {
        loop {
            // 尝试获取 A
            if let Some(ga) = a.try_lock() {
                // 尝试获取 B
                match b.try_lock() {
                    Some(gb) => {
                        // ✅ 同时获取了两个锁
                        println!("a={}, b={}", *ga, *gb);
                        return;
                    }
                    None => {
                        // B 被持有——释放 A 并重试
                        drop(ga);
                        std::thread::yield_now();
                    }
                }
            }
            // 短暂的退避，避免忙等
            std::thread::yield_now();
        }
    });
    // 这种方案有 livelock 的风险（两个线程都在不停地 try_lock 和释放）
    // 但不会死锁（死锁要求线程永远阻塞，livelock 中线程仍在活动）
}

// 策略 3：减少锁的数量和粒度
fn reduce_lock_granularity() {
    // ❌ 粗粒度——一个大锁保护所有数据
    struct Bad {
        users: Mutex<Vec<String>>,
        counters: Mutex<HashMap<String, u64>>,
        config: Mutex<Config>,
    }
    // 如果每个操作需要多个锁 → 死锁风险

    // ✅ 细粒度——每个操作尽可能只用一个锁
    //    或者使用单一锁保护整个结构（如果操作足够简单）
    struct Better {
        state: Mutex<AppState>,  // 一个锁保护所有相关数据
    }
    struct AppState {
        users: Vec<String>,
        counters: HashMap<String, u64>,
        config: Config,
    }
    // 减少了锁的数量 → 降低了死锁的可能

    // 或者更细粒度但设计好获取顺序
    // 或者使用 RwLock 代替 Mutex（减少写锁竞争）
}

// 策略 4：避免嵌套锁
fn avoid_nested_locks() {
    let outer = Arc::new(Mutex::new(0));
    let inner = Arc::new(Mutex::new(0));

    // ❌ 嵌套锁
    fn bad_nested(outer: &Mutex<i32>, inner: &Mutex<i32>) {
        let _go = outer.lock().unwrap();
        let _gi = inner.lock().unwrap();
        // ... 做事情 ...
    }

    // ✅ 方法：重构代码，消除嵌套的需要
    // 例如：在获取内层锁之前先释放外层
    fn better_no_nesting(outer: &Mutex<i32>, inner: &Mutex<i32>) {
        let value = {
            let go = outer.lock().unwrap();
            *go  // 复制出需要的值
        };  // outer 锁释放

        // 现在可以安全获取 inner
        let mut gi = inner.lock().unwrap();
        *gi += value;
    }
}
```

## 7. 最佳实践

### 7.1 锁持有时间最小化

锁应该被持有的时间越短越好——这是并发性能优化的黄金法则。

```rust
use std::sync::Mutex;

// ❌ 在锁内做不必要的工作
fn bad_lock_scope(data: &Mutex<Vec<i32>>) {
    let mut guard = data.lock().unwrap();

    // 大量不需要锁的计算
    let mut result = 0u64;
    for i in 0..1_000_000 {
        result = result.wrapping_mul(13).wrapping_add(i as u64);
    }
    // 网络请求、文件 I/O 等都不应该在锁内进行

    guard.push(result as i32);
}

// ✅ 只锁住必要的最小范围
fn good_lock_scope(data: &Mutex<Vec<i32>>) {
    // 在锁外做计算
    let mut result = 0u64;
    for i in 0..1_000_000 {
        result = result.wrapping_mul(13).wrapping_add(i as u64);
    }

    // 只在需要修改共享数据时才加锁
    let mut guard = data.lock().unwrap();
    guard.push(result as i32);
    // guard 在这里被释放（离开作用域）
}

// ✅ 使用块表达式精确控制锁的范围
fn precise_lock_scope(data: &Mutex<Vec<String>>) -> String {
    // 在锁内读
    let last = {
        let guard = data.lock().unwrap();
        guard.last().cloned()
    };  // ← 锁在这里释放

    // 在锁外做复杂操作
    match last {
        Some(s) => s.to_uppercase(),
        None => String::new(),
    }
}
```

### 7.2 避免在持有锁时调用外部代码

在持有锁时调用你无法控制的外部代码（回调、trait 方法、闭包参数等）是并发 bug 的温床。

```rust
use std::sync::Mutex;

// ❌ 在锁内调用外部代码
fn bad_callback_in_lock<T>(data: &Mutex<Vec<T>>, f: impl FnOnce(&[T])) {
    let guard = data.lock().unwrap();
    f(&guard);  // ❌ f 可能做什么？
                // - 可能再次调用同一个锁 → 死锁
                // - 可能执行耗时操作 → 长时间持有锁
                // - 可能 panic → 锁中毒（如果使用 std Mutex）
}

// ✅ 先复制数据，再在锁外调用外部代码
fn good_callback_outside_lock<T: Clone>(data: &Mutex<Vec<T>>, f: impl FnOnce(&[T])) {
    let snapshot = {
        let guard = data.lock().unwrap();
        guard.clone()  // 快速复制
    };  // 锁释放

    f(&snapshot);  // ✅ 在锁外调用外部代码——安全
}
```

## 8. 内部实现深度解析

### 8.1 从原子操作到 futex —— std::sync::Mutex 的完整路径

理解 Mutex 从快速路径（原子操作）退化到慢速路径（系统调用）的完整过程。

```rust
// std::sync::Mutex 的内部实现结构（概念性，高度简化）

use std::sync::atomic::{AtomicU8, Ordering};

// 实际的标准库实现因平台而异
// Linux: 基于 futex
// macOS: 基于 pthread_mutex_t
// Windows: 基于 SRWLock

// 概念性的 Linux futex 实现：
struct ConceptualMutex {
    // 状态：
    //   0 = 未锁定
    //   1 = 已锁定，无等待者
    //   2 = 已锁定，有等待者（需要通过 futex 唤醒）
    state: AtomicU8,
}

impl ConceptualMutex {
    fn lock(&self) {
        // Fast path: 尝试从 0 → 1
        if self.state.compare_exchange(
            0, 1, Ordering::Acquire, Ordering::Relaxed
        ).is_ok() {
            return;  // ✅ 成功！无需系统调用
        }

        // Slow path: 锁已被持有
        loop {
            // 检查当前状态
            let current = self.state.load(Ordering::Relaxed);

            if current == 0 {
                // 尝试重新获取
                match self.state.compare_exchange_weak(
                    0, 1, Ordering::Acquire, Ordering::Relaxed
                ) {
                    Ok(_) => return,  // ✅ 成功！
                    Err(_) => continue,
                }
            }

            // 如果有等待者或需要标记有等待者
            if current == 1 {
                // 尝试将状态从 1 改为 2（标记"有等待者"）
                let _ = self.state.compare_exchange(
                    1, 2, Ordering::Relaxed, Ordering::Relaxed
                );
                continue;
            }

            // state == 2: 有等待者，进入 futex 等待
            // futex_wait 会将线程挂起直到有人调用 futex_wake
            unsafe {
                // linux_futex_wait(&self.state, 2);
                // 这是一个系统调用——开销大
                // 线程在这里被操作系统挂起
            }
            // 被唤醒后，循环重新开始
        }
    }

    fn unlock(&self) {
        // 将状态从 1 改为 0
        if self.state.swap(0, Ordering::Release) == 2 {
            // 有等待者——唤醒一个
            unsafe {
                // linux_futex_wake(&self.state, 1);
                // 唤醒一个在 futex_wait 中等待的线程
            }
        }
    }
}

// 关键洞察：
// 无竞争时：lock + unlock 仅需要 2 次原子操作（CAS + swap）
//   这是"零开销"的——不需要系统调用
// 有竞争时：退化到 futex 系统调用（线程挂起/唤醒）
//   这是"不可避免的开销"——OS 必须参与线程调度
//
// parking_lot 的优化：
// 在 CAS 失败后、调用 futex 之前，
// 加入一个"自旋阶段"（几十到几百次的 CPU 空转）
// 如果锁持有时间极短（几个 CPU 周期），
// 自旋可以在不进入 futex 的情况下获取到锁
// → 减少了系统调用的数量 → 提升了吞吐量
```

### 8.2 RwLock 的内部状态机

`RwLock` 的内部实现比 `Mutex` 更复杂，因为它需要跟踪读锁的数量。

```
RwLock 的状态机（概念性）

状态编码在一个原子整数中：

  ┌─────┬────────────────────────────────────┐
  │ bit │ 含义                               │
  ├─────┼────────────────────────────────────┤
  │  0  │ 写锁标志（1 = 已锁定写）             │
  │ 1-N │ 读锁计数器（N 位）                   │
  │ ... │ 可能包含等待者标志、排队信息等        │
  └─────┴────────────────────────────────────┘

  read() 的尝试：
    IF 写锁位 == 0 AND (无等待写者 OR 策略允许):
        增加读锁计数（原子操作）
        返回读锁守卫
    
    ELSE:
        等待（futex / parking / 自旋）

  write() 的尝试：
    IF 写锁位 == 0 AND 读锁计数 == 0:
        设置写锁位（原子操作）
        返回写锁守卫
    
    ELSE:
        等待（futex / parking / 自旋）

  read unlock:
    减少读锁计数（原子操作）
    IF 读锁计数变为 0 AND 有等待的写者:
        唤醒一个写者

  write unlock:
    清除写锁位（原子操作）
    唤醒等待者（按策略：可能唤醒一个写者或多个读者）

  关键设计决策：
  1. 读锁计数器是有限的——如果读锁数量超过最大值
     （通常是 2^(N-1) - 1），panic 或阻塞
  2. 公平性策略：
     - 偏向读者：只要有读者，写者就一直等 → 写入者饥饿
     - 偏向写者：有写者在等时，新读者进入等待 → 读者吞吐量下降
     - FIFO：按到达顺序服务 → 折中方案
```

```rust
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::Duration;

fn try_lock_deadlock_avoidance() {
    let a = Arc::new(Mutex::new(0));
    let b = Arc::new(Mutex::new(0));

    let a1 = a.clone();
    let b1 = b.clone();
    let t1 = thread::spawn(move || {
        // 使用 try_lock 尝试按 A → B 顺序获取
        loop {
            let ga = match a1.try_lock() {
                Some(g) => g,
                None => {
                    thread::sleep(Duration::from_micros(10));
                    continue;
                }
            };

            match b1.try_lock() {
                Some(gb) => {
                    println!("线程 1: 获取了 A 和 B: {} {}",
                             *ga, *gb);
                    return;
                }
                None => {
                    // B 被占用——释放 A 并重试
                    drop(ga);
                    thread::sleep(Duration::from_micros(10));
                }
            }
        }
    });

    let a2 = a.clone();
    let b2 = b.clone();
    let t2 = thread::spawn(move || {
        // 使用 try_lock 尝试按 B → A 顺序获取（故意反序来测试）
        loop {
            let gb = match b2.try_lock() {
                Some(g) => g,
                None => {
                    thread::sleep(Duration::from_micros(10));
                    continue;
                }
            };

            match a2.try_lock() {
                Some(ga) => {
                    println!("线程 2: 获取了 B 和 A: {} {}",
                             *gb, *ga);
                    return;
                }
                None => {
                    drop(gb);
                    thread::sleep(Duration::from_micros(10));
                }
            }
        }
    });

    t1.join().unwrap();
    t2.join().unwrap();
    // 两个线程都能最终完成——没有死锁
    // 但可能有短暂的 livelock（两个线程都在重试）
}
```

```rust
        let cache = Arc::new(Mutex::new(HashMap::<i32, String>::new()));
        // 预填充数据
        for i in 0..100 {
            cache.lock().unwrap().insert(i, format!("value_{}", i));
        }

        let start = Instant::now();
        let mut handles = vec![];

        for _ in 0..READ_THREADS {
            let c = cache.clone();
            handles.push(thread::spawn(move || {
                let mut local_count = 0usize;
                for i in 0..OPS_PER_READER {
                    let key = (i % 100) as i32;
                    let map = c.lock().unwrap();
                    if map.contains_key(&key) {
                        local_count += 1;
                    }
                }
                local_count
            }));
        }

        for _ in 0..WRITE_THREADS {
            let c = cache.clone();
            handles.push(thread::spawn(move || {
                for i in 0..OPS_PER_WRITER {
                    let key = (100 + i) as i32;
                    c.lock().unwrap()
                        .insert(key, format!("new_value_{}", i));
                }
            }));
        }

        for h in handles {
            h.join().unwrap();
        }
        println!("Mutex cache: {:?}", start.elapsed());
    }

    // --- RwLock 版本 ---
    {
        let cache = Arc::new(RwLock::new(HashMap::<i32, String>::new()));
        for i in 0..100 {
            cache.write().unwrap().insert(i, format!("value_{}", i));
        }

        let start = Instant::now();
        let mut handles = vec![];

        for _ in 0..READ_THREADS {
            let c = cache.clone();
            handles.push(thread::spawn(move || {
                let mut local_count = 0usize;
                for i in 0..OPS_PER_READER {
                    let key = (i % 100) as i32;
                    let map = c.read().unwrap();
                    if map.contains_key(&key) {
                        local_count += 1;
                    }
                }
                local_count
            }));
        }

        for _ in 0..WRITE_THREADS {
            let c = cache.clone();
            handles.push(thread::spawn(move || {
                for i in 0..OPS_PER_WRITER {
                    let key = (100 + i) as i32;
                    c.write().unwrap()
                        .insert(key, format!("new_value_{}", i));
                }
            }));
        }

        for h in handles {
            h.join().unwrap();
        }
        println!("RwLock cache: {:?}", start.elapsed());
    }

    // 典型输出（多核系统）：
    // Mutex cache:  ~800ms   （读者被串行化）
    // RwLock cache: ~120ms   （10 个读者并发读）
}
```

### 练习 3 答案

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn mutex_poisoning_recovery() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3, 4, 5]));

    let d1 = data.clone();
    let t1 = thread::spawn(move || {
        let mut guard = d1.lock().unwrap();
        // 模拟"写入到一半"后 panic
        guard.push(6);
        guard.push(7);
        // 在这里 panic——数据是 [1,2,3,4,5,6,7]
        // 但调用者期望的是"要么都没有添加 6 和 7，要么都添加了"
        panic!("写入中断！");
    });

    let _ = t1.join();  // 线程 panic 了

    // 检查并恢复
    match data.lock() {
        Ok(guard) => {
            println!("锁是干净的: {:?}", *guard);
        }
        Err(poisoned) => {
            println!("检测到锁中毒！");
            let mut guard = poisoned.into_inner();

            // 检查数据一致性
            let len = guard.len();
            println!("中毒后的数据（长度 {}）: {:?}", len, *guard);

            // 恢复策略：如果数据不符合不变条件，回滚到上一个已知的好状态
            // 这里我们简单地检查长度是否是 5 的倍数
            // （实际场景中，你应该检查业务相关的数据一致性）
            if len > 5 {
                println!("数据可能被部分写入，回滚到安全状态");
                guard.truncate(5);  // 恢复到写入前
            }

            println!("恢复后的数据: {:?}", *guard);
            assert_eq!(*guard, vec![1, 2, 3, 4, 5]);
        }
    }
}
```

```rust
// 概念性 benchmark —— 展示性能差异的根因
//
// 低竞争场景（每次锁定时间极短——几个 CPU 周期）：
//   - std::sync::Mutex: ~20ns per lock/unlock
//   - parking_lot::Mutex: ~12ns per lock/unlock
//   - parking_lot 更快：自旋避免系统调用
//
// 中竞争场景（4 线程，临界区 ~100ns）：
//   - std::sync::Mutex: ~200ns per lock/unlock
//   - parking_lot::Mutex: ~150ns per lock/unlock
//   - parking_lot 仍有优势但差距缩小
//
// 高竞争场景（8 线程，临界区 ~1us）：
//   - std::sync::Mutex: ~10us per lock/unlock
//   - parking_lot::Mutex: ~10us per lock/unlock
//   - 两者都退化到 futex 等待——差异可忽略
//
// 结论：
//   parking_lot 的主要优势在低到中竞争场景
//   高竞争场景下两者表现相似（瓶颈在系统调度，不在锁的实现）
//
// 实际选择建议：
//   - 如果能接受额外依赖 → parking_lot（更好的性能 + 更多特性）
//   - 如果追求零依赖 → std::sync::Mutex（完全够用）
//   - 如果需要 try_lock_for → parking_lot（std 没有此功能）
//   - 如果需要 poisoning → std::sync::Mutex（parking_lot 不支持）
```

> 锁不是并发编程的"敌人"——它是并发编程的"合同"。正确使用锁的关键不是避免锁，而是让每次锁定都尽可能短暂、明确和安全。Rust 的类型系统将这份合同写入了编译期——不持有 `MutexGuard` 就不能访问被保护的数据，持有 `MutexGuard` 就不能让它泄漏到 `.await` 之外。这是 Rust 在并发安全方面对 C/C++ 最根本的超越之一。
