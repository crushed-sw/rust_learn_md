# 8.5 axum：Web 框架的提取器模式

> axum 是 Rust web 框架中的后来者，但它用一种极简而强大的设计迅速获得了广泛采用。核心思想只有两个 trait：`FromRequest`（"把请求的一部分提取为参数"）和 `IntoResponse`（"任何东西都可以是 HTTP 响应"）。这种"类型驱动的 API 设计"让 handler 函数几乎不需要样板代码——参数类型决定了"从请求中提取什么"，返回类型决定了"响应是什么"。本节剖析 axum 的源码架构：Router 的层级结构、`tower::Service` 的异步处理模型、以及中间件的洋葱模型。

---

## 1. 两个核心 trait 驱动一切

### 1.1 FromRequest：类型即提取器

```rust
// axum 的核心提取器 trait（简化）
pub trait FromRequest<S>: Sized {
    type Rejection: IntoResponse;

    fn from_request(req: Request, state: &S) -> Result<Self, Self::Rejection>;
}

// ━━━ 各种提取器的实现 ━━━

// Json<T>: 从请求体提取 JSON
impl<S, T> FromRequest<S> for Json<T>
where T: DeserializeOwned
{
    type Rejection = JsonRejection;

    async fn from_request(req: Request, state: &S) -> Result<Self, Self::Rejection> {
        let body = req.into_body();
        let bytes = body.collect().await?.to_bytes();
        let value: T = serde_json::from_slice(&bytes)?;
        Ok(Json(value))
    }
}

// Path<T>: 从 URL 路径提取参数
// GET /users/:id → Path { id: String }
impl<S, T> FromRequest<S> for Path<T>
where T: DeserializeOwned
{ ... }

// Query<T>: 从查询字符串提取
// GET /search?q=rust → Query { q: "rust" }
impl<S, T> FromRequest<S> for Query<T>
where T: DeserializeOwned
{ ... }

// State<S>: 从应用状态提取
impl<S: Clone> FromRequest<S> for State<S> { ... }

// Extension<T>: 从请求扩展中提取
impl<S, T: Clone + Send + Sync + 'static> FromRequest<S> for Extension<T> { ... }
```

**Handler 函数中的使用**：

```rust
// axum 的 handler 可以声明它需要的任何提取器
// Router 在调用 handler 时，自动提取每个参数

async fn get_user(
    Path(user_id): Path<u32>,               // 从 URL 提取
    Query(params): Query<SearchParams>,      // 从查询字符串提取
    State(state): State<Arc<AppState>>,      // 从应用状态提取
    Json(body): Json<UpdateUserRequest>,     // 从请求体提取
) -> impl IntoResponse {
    // user_id, params, state, body 已经全部提取完毕
    // 如果有任何一个提取失败 → 自动返回 400/404/500
    // 你不需要写任何错误处理代码！
}
```

### 1.2 IntoResponse：一切皆响应

```rust
// axum 的核心响应 trait（简化）
pub trait IntoResponse {
    fn into_response(self) -> Response;
}

// ━━━ 标准库类型的实现 ━━━
impl IntoResponse for String {
    fn into_response(self) -> Response {
        Response::new(self.into())
    }
}

impl IntoResponse for &str {
    fn into_response(self) -> Response {
        Response::new(self.to_owned().into())
    }
}

// ━━━ axum 类型的实现 ━━━
impl<T: Serialize> IntoResponse for Json<T> {
    fn into_response(self) -> Response {
        let json = serde_json::to_string(&self.0).unwrap();
        Response::builder()
            .header("content-type", "application/json")
            .body(json.into())
            .unwrap()
    }
}

impl IntoResponse for StatusCode {
    fn into_response(self) -> Response {
        Response::builder()
            .status(self)
            .body(Body::empty())
            .unwrap()
    }
}

// ━━━ 元组的实现（组合多个 trait）━━━
impl<R1, R2> IntoResponse for (R1, R2)
where R1: IntoResponse, R2: IntoResponse
{
    fn into_response(self) -> Response {
        // 合并两个响应的 headers 和 body
        let mut response = self.0.into_response();
        // 合并 R2 的 headers...
        response
    }
}

// ━━━ Result 的实现 ━━━
impl<T, E> IntoResponse for Result<T, E>
where T: IntoResponse, E: IntoResponse
{
    fn into_response(self) -> Response {
        match self {
            Ok(t) => t.into_response(),
            Err(e) => e.into_response(),  // → 通常是 500 Internal Server Error
        }
    }
}
```

**设计的精妙之处**：

```rust
// handler 可以返回几乎任何东西！
async fn example() -> impl IntoResponse {
    // 返回纯文本
    // return "hello world";

    // 返回 JSON
    // return Json(serde_json::json!({"msg": "ok"}));

    // 返回状态码
    // return StatusCode::NOT_FOUND;

    // 返回带状态码的 JSON —— 用元组！
    (StatusCode::CREATED, Json(serde_json::json!({"id": 42})))
}

// ↑ 所有这些都工作，因为 IntoResponse 被广泛实现
```

## 2. Router 的层级结构

### 2.1 路由匹配

```rust
// axum Router 的层级结构（简化）

Router {
    // 匹配 /users/:id
    //   MethodRouter::GET  → get_user
    //   MethodRouter::POST → create_user
    //
    // 匹配 /posts（嵌套 Router）
    //   MethodRouter::GET      → list_posts
    //   /posts/:id
    //     MethodRouter::GET    → get_post
}

// 路由匹配的过程：
// 1. 按路径段逐段匹配
// 2. 静态段优先于动态段（/users/me 优先于 /users/:id）
// 3. 匹配成功后 → 按 HTTP 方法分发到对应的 handler
```

### 2.2 嵌套路由 + 状态共享

```rust
use axum::{Router, routing::get};

let app = Router::new()
    // 顶层路由
    .route("/", get(root))
    .route("/health", get(health_check))
    // 嵌套路由（/users 路径下）
    .nest("/users", Router::new()
        .route("/", get(list_users).post(create_user))
        .route("/:id", get(get_user).delete(delete_user))
    )
    // 共享状态
    .with_state(Arc::new(AppState::new()));
```

## 3. tower::Service 模型

### 3.1 axum 是 tower 生态的一部分

```rust
// axum 的底层是 tower——一个通用的"服务"抽象

// tower::Service trait（简化）
pub trait Service<Request> {
    type Response;
    type Error;
    type Future: Future<Output = Result<Self::Response, Self::Error>>;

    fn call(&self, req: Request) -> Self::Future;
}

// axum 把 HTTP 适配到 tower 的 Service 模型：
// Request = http::Request<Body>
// Response = http::Response<Body>
// Error = Infallible（axum 处理了所有错误）

// Handler 也是 Service：
// 你的 async fn get_user(...) 被 axum 适配为 Service trait
```

### 3.2 为什么用 tower？

```
使用 tower 意味着 axum 可以使用所有 tower 生态的中间件：
  - tower::limit::ConcurrencyLimit  → 限制并发请求数
  - tower::timeout::TimeoutLayer    → 请求超时
  - tower_http::cors::CorsLayer     → CORS 处理
  - tower_http::trace::TraceLayer   → 请求追踪
  - tower_http::compression::CompressionLayer  → 响应压缩

所有这些都通过 Layer 模式无缝集成到 axum Router 中。
```

## 4. 中间件的洋葱模型

### 4.1 Layer 模式

```rust
use axum::{middleware, Router};
use tower_http::cors::CorsLayer;

let app = Router::new()
    .route("/", get(handler))
    // ↓ 中间件按添加顺序执行（从外到内的洋葱层）
    .layer(CorsLayer::permissive())        // 第 3 层：CORS headers
    .layer(TraceLayer::new_for_http())     // 第 2 层：tracing
    .layer(CompressionLayer::new())        // 第 1 层：响应压缩

// 请求流程（洋葱从外到内）：
// Request → CorsLayer → TraceLayer → CompressionLayer → handler
// Response ← CorsLayer ← TraceLayer ← CompressionLayer ← handler
//                                  ↑ 压缩在 handler 之后执行
```

### 4.2 中间件的实现

```rust
// axum 中间件就是一个异步函数，接收 Request 和 next
async fn my_middleware<B>(
    request: Request<B>,
    next: Next<B>,
) -> impl IntoResponse {
    // ← 请求处理前
    let start = std::time::Instant::now();

    let response = next.run(request).await;

    // ← 响应处理后
    let duration = start.elapsed();
    tracing::info!("request took {:?}", duration);

    response
}

// 使用
let app = Router::new()
    .route("/", get(handler))
    .layer(middleware::from_fn(my_middleware));
```

## 5. 架构总结：axum 为什么"简单但强大"？

```
axum 的成功在于"做减法"：

1. 不做 ORM → 用 sqlx/tokio-postgres
2. 不做模板 → 用 askama/tera
3. 不做 session → 用 tower-sessions
4. 只做一件事：HTTP 路由 + 请求提取 + 响应生成

核心设计：
  ✓ FromRequest → 类型驱动的请求提取（零样板）
  ✓ IntoResponse → 一切皆响应（灵活组合）
  ✓ tower::Service → 通用中间件生态（复用而非重造）
  ✓ 洋葱模型 → 可组合的中间件

对比其他框架：
  - Actix-web: 基于 Actor 模型，更多"魔法"
  - Rocket: 依赖 proc macro 做路由，更"声明式"
  - axum: 最接近"纯 Rust trait 组合"的方式，最少魔法
```
