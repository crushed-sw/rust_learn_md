# 8.3 tracing：基础设施设计

> tracing 是 Rust 生态中"可观测性基础设施"的标杆——它不仅是一个日志库，更是一个**结构化追踪框架**：结构化字段而不是拼接字符串、Span 而不是行号、Layer 洋葱模型而不是硬编码的日志输出。本节深入 tracing 的源码架构：`Subscriber` trait 作为"日志消费端"的抽象、`Layer` trait 的装饰器模式（洋葱模型）、Span 的生命周期管理、以及最重要的：**tracing 如何做到"编译时不启用则零开销"**——这是基础设施 crate 设计的经典案例。

---

## 1. tracing 的核心抽象三角

### 1.1 Subscriber + Span + Event

```
tracing 的核心三角：

    Event (info!/warn!/error!/...)
    ↓ 发生时调用 subscriber.event()
    
    Span (span!(...), #[instrument])
    ↓ 进入时调用 subscriber.enter_span()
    ↓ 退出时调用 subscriber.exit_span()
    
    Subscriber (接收所有事件的"后端")
    
    tracing-subscriber 提供了默认 Subscriber 实现：
    - fmt: 人类可读格式
    - json: 机器可读格式
    - filter: 按级别/模块过滤
    - layer: 组合多个 Subscriber
```

### 1.2 关键设计：Subscriber 和 Layer 分离

```rust
// tracing_core 中只定义了 Subscriber trait
// tracing-subscriber 中定义了 Layer trait（装饰器模式）

// Subscriber: 接收事件的核心抽象
pub trait Subscriber: 'static {
    fn enabled(&self, metadata: &Metadata<'_>) -> bool;
    fn new_span(&self, span: &Attributes<'_>) -> Id;
    fn event(&self, event: &Event<'_>);
    fn enter(&self, span: &Id);
    fn exit(&self, span: &Id);
    // ...
}

// Layer: 可组合的中间件（类似 tower::Layer 的模式）
pub trait Layer<S: Subscriber> {
    fn on_event(&self, event: &Event<'_>, ctx: Context<'_, S>) { ... }
    fn on_new_span(&self, attrs: &Attributes<'_>, id: &Id, ctx: Context<'_, S>) { ... }
    fn on_enter(&self, id: &Id, ctx: Context<'_, S>) { ... }
    // ... 几乎所有方法都有默认空实现
}
```

## 2. Layer 的洋葱模型

### 2.1 层层包裹

```rust
use tracing_subscriber::{fmt, layer::SubscriberExt, util::SubscriberInitExt, EnvFilter};

// ━━━ 构建一个多层 Subscriber ━━━
tracing_subscriber::registry()
    // Layer 1: 按 RUST_LOG 过滤
    .with(EnvFilter::try_from_default_env().unwrap_or_else(|_| "info".into()))
    // Layer 2: 格式化为 JSON（输出到 stdout）
    .with(fmt::layer().json())
    // Layer 3: 同时写文件
    .with(fmt::layer().with_writer(std::fs::File::create("app.log").unwrap()))
    .init();

// 调用 info!("hello"); 时：
// → Layer 1 (EnvFilter) 检查: level >= info? → 通过
// → Layer 2 (JSON stdout) 处理: 写 {"level": "INFO", "message": "hello"}
// → Layer 3 (file writer) 处理: 写 INFO hello
// ↑ 每个 Layer 都可以独立处理同一事件，互不影响
```

### 2.2 洋葱模型的实现

```rust
// tracing-subscriber 如何实现 Layer 组合（简化）

// Registry 是所有 Layer 的容器
struct Registry {
    layers: Vec<Box<dyn Any>>,  // 存储所有 Layer
}

// 当事件发生时，Registry 遍历所有 Layer：
impl Subscriber for Registry {
    fn event(&self, event: &Event<'_>) {
        for layer in &self.layers {
            layer.on_event(event, ...);  // 每个 Layer 处理事件
        }
    }
}

// Layer trait 的每个方法有默认空实现
// → 你只需要覆盖你关心的方法
// → 不需要实现所有 15+ 个方法
```

## 3. Span 的生命周期

### 3.1 Span 的数据存储

```rust
// tracing 中，Span 的数据结构（概念层面）：

// tracing_core 中，Span 是轻量的——只有一个 ID
// 实际数据存储在 Subscriber 的"extensions"（类型安全映射）中

struct SpanData {
    name: &'static str,
    fields: Vec<FieldValue>,    // 结构化字段
    parent: Option<Id>,          // 父 Span 的 ID
    references: Vec<Id>,         // 引用的其他 Span
    extensions: Extensions,      // 任意类型的数据存储（类似 HashMap<TypeId, Box<dyn Any>>）
}
```

### 3.2 Span 的创建和销毁

```rust
use tracing::{span, Level};

// 创建 Span
let span = span!(Level::INFO, "my_span", user_id = 42);
// → subscriber.new_span() 被调用
// → Subscriber 创建 SpanData 并返回 Id

// 进入 Span（通常由 #[instrument] 自动完成）
let _enter = span.enter();
// → subscriber.enter(span_id) 被调用
// → Subscriber 把 span_id 设为"当前活跃 Span"（线程局部存储）

// info! 在 Span 内调用
info!("processing");
// → 事件自动关联到当前活跃 Span

// Span 离开作用域
drop(_enter);
// → subscriber.exit(span_id) 被调用

// Span 被 drop
drop(span);
// → Span 被关闭（但数据可能保留在 Subscriber 中用于完成后的记录）
```

## 4. 零开销设计

### 4.1 compile-time filtering

```rust
// tracing 的核心性能技巧：编译时过滤

// 当 level 低于 enabled 时，整个宏调用被优化为 no-op
info!("expensive computation: {}", compute_something());
// ↑ 如果 level 设置为 warn（info < warn），上述代码被展开为：

// 展开后（简化）：
if tracing::level_enabled!(Level::INFO) {  // ← 编译时常量检查
    // 这个块在编译时被消除！
    // compute_something() 根本不会被调用
    let result = compute_something();
    tracing::event!(Level::INFO, message = "expensive computation: {}", result);
}
```

### 4.2 静态分发 vs 动态分发

```rust
// tracing 在设计上做了聪明的取舍：

// 热路径（每次调用都经过）→ 静态分发
// - enabled() 检查：内联到调用点
// - event() 的 metadata：编译时已知

// 冷路径（初始化、配置）→ 动态分发
// - new_span() 创建
// - Layer 的注册和组合

// 这保证：在 release 模式下，禁用的 tracing 调用 = 零指令
```

### 4.3 Event 不是一个对象

```rust
// tracing 的 Event 不是堆分配的——它是栈上的临时结构
// Event 不拥有数据——它引用调用点的数据

// ❌ 如果是这样设计：
// pub struct Event { message: String, fields: HashMap<String, Value> }
// → 每次 info! 都分配 String + HashMap → 太贵

// ✅ tracing 的实际设计：
// Event 持有对调用点数据的引用 + metadata 指针
// 所有重活留给 Subscriber（如果不需要就不会发生）
```

## 5. `#[instrument]` 宏展开

```rust
// 你写的
#[tracing::instrument]
async fn process(id: u64, data: &str) -> Result<()> {
    info!("processing data");
    do_work(data).await
}

// 展开后（大幅简化）
async fn process(id: u64, data: &str) -> Result<()> {
    // 1. 创建 Span
    let span = tracing::span!(
        tracing::Level::INFO,
        "process",
        id = id,        // ← 自动记录参数
        data = data,
    );
    // 2. 进入 Span
    let _enter = span.enter();

    // 3. 你的原始代码
    info!("processing data");
    do_work(data).await
    // 4. Span 在函数返回时自动离开 + 关闭
}
// #[instrument(skip(data))] 可以排除敏感参数
// #[instrument(fields(result = ?result))] 可以在返回时记录结果
```

## 6. 架构总结：为什么 tracing 的设计是"基础设施级的"？

```
┌──────────────────────────────────────────────┐
│  tracing 的设计体现了基础设施 crate 的原则：    │
│                                              │
│  1. 最小化核心 (tracing-core)                 │
│     - 只定义 Subscriber + Span + Event       │
│     - 没有依赖（除了 tracing-core 自身）      │
│                                              │
│  2. 可组合性 (Layer 模式)                    │
│     - 多个 Layer 可以堆叠                    │
│     - 每个 Layer 可以独立演进                 │
│     - 不需要修改核心 trait                    │
│                                              │
│  3. 零开销的默认值                           │
│     - 编译时常量检查避免运行时开销            │
│     - 热路径上静态分发                        │
│     - 不启用的代码被编译器消除                │
│                                              │
│  4. 生态兼容性                               │
│     - OpenTelemetry 集成不修改 tracing-core   │
│     - metrics 集成通过独立的 Layer            │
│     - 第三方可以通过 subscriber 扩展          │
└──────────────────────────────────────────────┘
```
