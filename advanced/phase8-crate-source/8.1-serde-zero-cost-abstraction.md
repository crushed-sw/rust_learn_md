# 8.1 serde：零成本抽象的极致

> serde 是 Rust 生态中使用最广的 crate 之一——几乎每个处理 JSON、YAML、TOML、Bincode 的项目都依赖它。但 serde 真正的成就不在于"能序列化"——而在于**零成本**：编译期展开、没有反射、没有类型擦除、没有任何运行时开销。这节不是教你 `#[derive(Serialize)]` 怎么写——你肯定已经会了——而是拆解 serde 的**架构设计**：Serializer/Deserializer trait 的后端抽象、Visitor 模式怎么避免中间分配、Serde Data Model 的 29 种类型、以及 `#[derive(Serialize)]` 展开后到底生成了什么代码。

---

## 1. serde 的核心问题

### 1.1 一个 trait 支持所有格式

```rust
// serde 要解决的核心问题：
// 定义一种方式，让 "任何 Rust 类型" 可以被序列化为 "任何数据格式"

// Rust 类型                   数据格式
// ──────────                  ────────
// struct Point {              JSON:    {"x": 1.0, "y": 2.0}
//     x: f64,                 YAML:    x: 1.0\n  y: 2.0
//     y: f64,                 Bincode: [8 bytes of f64][8 bytes of f64]
// }                           TOML:    x = 1.0\n  y = 2.0

// 每种组合只需要实现一个 trait：
// Point: Serialize → 任何格式都可以序列化 Point
// JSON Serializer 实现了 Serializer trait → Point 可以序列化为 JSON
```

### 1.2 三层架构

```
┌──────────────────────────────────────────────┐
│          你的类型 (struct Point)             │
│    实现 Serialize / Deserialize trait         │
├──────────────────────────────────────────────┤
│         Serde Data Model (29 种类型)          │
│    bool, i8-i64, f32-f64, char, str,         │
│    bytes, option, unit, seq, map, struct,    │
│    tuple, ...                                │
├──────────────────────────────────────────────┤
│    后端 (Serializer / Deserializer impl)       │
│    JSON, YAML, Bincode, TOML, ...            │
└──────────────────────────────────────────────┘

关键洞察：中间层的 Data Model 是"最小公分母"
- 你的类型用 Data Model 的词汇描述自己
- 后端用 Data Model 的词汇构建格式特定的输出
- 你的类型和后端之间零耦合！
```

## 2. Serialize trait：把 Rust 类型"说"出来

### 2.1 核心定义

```rust
// serde/src/ser.rs (简化)
pub trait Serialize {
    fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
    where
        S: Serializer;
}

pub trait Serializer {
    type Ok;
    type Error;
    // 以下每个方法对应 Data Model 的一种类型

    fn serialize_bool(self, v: bool) -> Result<Self::Ok, Self::Error>;
    fn serialize_i32(self, v: i32) -> Result<Self::Ok, Self::Error>;
    fn serialize_str(self, v: &str) -> Result<Self::Ok, Self::Error>;
    fn serialize_seq(self, len: Option<usize>) -> Result<Self::SerializeSeq, Self::Error>;
    fn serialize_map(self, len: Option<usize>) -> Result<Self::SerializeMap, Self::Error>;
    fn serialize_struct(
        self, name: &'static str, len: usize
    ) -> Result<Self::SerializeStruct, Self::Error>;
    // ... 共约 29 个方法
}
```

**设计亮点**：
- `Serialize` trait 只有一个方法 `serialize`——调用 serializer 的对应方法来描述自己
- `Serializer` trait 有 ~29 个方法——每个对应 Data Model 的一种类型
- 这种设计让**你的类型决定"我是什么"**（调用 `serialize_struct`），**serializer 决定"怎么输出"**（JSON 写 `{}`，Bincode 写字节）

### 2.2 手动实现 Serialize

```rust
use serde::{Serialize, Serializer};

struct Duration {
    secs: u64,
    nanos: u32,
}

impl Serialize for Duration {
    fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
    where
        S: Serializer,
    {
        // "我是一个 struct，有 2 个字段"
        let mut s = serializer.serialize_struct("Duration", 2)?;
        s.serialize_field("secs", &self.secs)?;
        s.serialize_field("nanos", &self.nanos)?;
        s.end()
    }
}
// ↑ 无论目标是 JSON, YAML 还是 Bincode，这段代码都一样！
// serializer 决定 "secs: 5" 在 JSON 中是 "secs": 5，
// 在 Bincode 中是一个 u64 的 little-endian 字节序列
```

## 3. Deserialize + Visitor：反序列化的精妙设计

### 3.1 为什么需要 Visitor？

```rust
// ━━━ 朴素的思路（为什么不能用）━━━
// trait Deserialize {
//     fn deserialize(???) -> Self;
// }
// 问题：deserialize 需要接收的是什么？不同格式传递不同信息！
// JSON: 看到一个数字 → 直接给你 f64
// Bincode: 看到了 8 个字节 → 给你 u64
// → 无法用一个共同的参数类型

// ━━━ serde 的方案：Visitor 模式 ━━━
// 不是 "格式告诉类型它是什么"（push）
// 而是 "类型告诉格式它期待什么，格式回调确认"（pull）
```

### 3.2 Visitor 模式的核心

```rust
// serde/src/de.rs (简化)
pub trait Deserialize<'de>: Sized {
    fn deserialize<D>(deserializer: D) -> Result<Self, D::Error>
    where
        D: Deserializer<'de>;
}

pub trait Deserializer<'de> {
    type Error;

    // 每个方法接收一个 Visitor
    fn deserialize_bool<V>(self, visitor: V) -> Result<V::Value, Self::Error>
    where V: Visitor<'de>;

    fn deserialize_i32<V>(self, visitor: V) -> Result<V::Value, Self::Error>
    where V: Visitor<'de>;

    fn deserialize_any<V>(self, visitor: V) -> Result<V::Value, Self::Error>
    where V: Visitor<'de>;
    // ... 约 29 个方法
}

pub trait Visitor<'de>: Sized {
    type Value;

    // 每种预期类型有一个方法
    fn visit_bool(self, v: bool) -> Result<Self::Value, ...> { ... }
    fn visit_i32(self, v: i32) -> Result<Self::Value, ...> { ... }
    fn visit_string(self, v: String) -> Result<Self::Value, ...> { ... }
    // ... 约 20 个方法（每个都有默认实现：返回 "unexpected type" 错误）
}
```

**Visitor 的工作流**：

```
Deserializer (JSON) 说："我看到一个数字 42，你是 i32 类型吗？"
  → 调用 visitor.visit_i32(42)
     → Visitor (你的类型) 说："是的！我接受 i32"
       → 返回 Ok(your_type)

或者：
Deserializer (JSON) 说："我看到一个数字 42，你是 string 类型吗？"
  → 调用 visitor.visit_string("42")... 
     → Visitor 说："不，我是 Duration，不接受 string"
       → 返回 Err(invalid type: string, expected Duration)
```

### 3.3 为什么 Visitor 模式的默认方法返回错误？

```rust
// Visitor 有约 20 个方法，每个都有一个默认实现
// 默认实现返回 "unexpected type" 错误

// 你只需要覆盖你的类型能接受的几种类型
impl<'de> Visitor<'de> for DurationVisitor {
    type Value = Duration;

    fn visit_seq<A>(self, mut seq: A) -> Result<Duration, A::Error>
    where A: SeqAccess<'de>
    {
        // Duration 接受序列（比如 JSON 数组 [5, 0]）
        let secs = seq.next_element()?.unwrap();
        let nanos = seq.next_element()?.unwrap();
        Ok(Duration { secs, nanos })
    }

    fn visit_map<A>(self, mut map: A) -> Result<Duration, A::Error>
    where A: MapAccess<'de>
    {
        // Duration 也接受 map（比如 JSON 对象 {"secs": 5, "nanos": 0}）
        // ...
    }

    // visit_bool, visit_f64, visit_string... 都用默认实现
    // → 如果有人尝试从 bool 反序列化为 Duration → 自动报错
}
```

## 4. `#[derive(Serialize)]` 展开后

### 4.1 源码 vs 生成的代码

```rust
// 你写的
use serde::Serialize;

#[derive(Serialize)]
struct Point {
    x: f64,
    y: f64,
}

// 宏展开后（用 cargo expand 查看，简化版）
impl serde::Serialize for Point {
    fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
    where
        S: serde::Serializer,
    {
        let mut s = serializer.serialize_struct("Point", 2)?;
        s.serialize_field("x", &self.x)?;
        s.serialize_field("y", &self.y)?;
        s.end()
    }
}

// ━━━ 零成本的关键 ━━━
// 1. 宏在编译期展开 → 生成的代码就是你手写的样子
// 2. 没有反射（不像 Java 的 serialization）
// 3. 没有类型擦除（不像 serde_json::Value 的中间表示）
// 4. 序列化 Point → JSON：直接从 f64 字段写到 JSON 输出流，零中间分配
```

### 4.2 `#[serde(borrow)]` 的零拷贝反序列化

```rust
#[derive(Deserialize)]
struct Message<'a> {
    #[serde(borrow)]  // ← 关键！不分配 String，借用原始数据
    text: &'a str,    //    直接指向 JSON 缓冲区中的字符串
}

// 反序列化时：
// 标准方式：从 JSON 中读取字符串 → 分配 String → 存储
// 零拷贝方式：直接返回指向 JSON 缓冲区的 &str → 零分配！
// 代价：Message 的生命周期被绑定到 JSON 缓冲区的生命周期
```

## 5. Serializer trait 的"后端抽象"之美

### 5.1 添加对新格式的支持

```rust
// ━━━ 只需要实现 Serializer trait ━━━
// 实现一个"简化版 JSON Serializer"来展示原理

struct SimpleJsonSerializer {
    output: String,
}

impl Serializer for SimpleJsonSerializer {
    type Ok = ();
    type Error = std::fmt::Error;

    fn serialize_bool(self, v: bool) -> Result<(), Self::Error> {
        // 写 "true" 或 "false"
    }

    fn serialize_str(self, v: &str) -> Result<(), Self::Error> {
        // 写 "\"{v}\""
    }

    fn serialize_struct(
        self, _name: &'static str, _len: usize
    ) -> Result<Self::SerializeStruct, Self::Error> {
        // 返回一个 "正在写 JSON 对象" 的状态
        Ok(SimpleJsonStruct { output: self.output })
    }
    // ... 实现其他 26 个方法
}

// 一旦实现 Serializer trait：
// ✅ 所有实现 Serialize 的类型都能序列化为你的格式！
// 包括：std 类型（i32, String, Vec, HashMap...）
//      第三方类型（chrono::DateTime, uuid::Uuid...）
//      你的自定义类型（只要 #[derive(Serialize)]）
```

## 6. Serde Data Model 的 29 种类型

```
Serde Data Model:
  * 14 种基本类型 (primitive types)
    - bool
    - i8, i16, i32, i64, i128
    - u8, u16, u32, u64, u128
    - f32, f64
    - char
  * 4 种复合类型 (compound types)
    - str (&str / String)
    - bytes (&[u8] / Vec<u8>)
    - option (Option<T>)
    - unit (() / struct with no fields)
  * 3 种序列类型 (sequence types)
    - seq (Vec<T>, HashSet<T>, ...)
    - tuple ((T1, T2, ...), struct with unnamed fields)
    - tuple_struct
  * 2 种映射类型 (map types)
    - map (HashMap<K,V>, BTreeMap<K,V>)
    - struct (struct with named fields)
  * 2 种特殊类型
    - newtype_struct (struct Point(i32))
    - enum (Rust enum)
  * 2 种"自描述"类型
    - any (serde_json::Value, serde_yaml::Value)
    - ignored_any (跳过不认识的字段)
```

**关键原则**：这 29 种类型是 Rust 的类型系统和常见数据格式（JSON/XML/YAML/...）的"最大公约数"。任何新格式需要支持这 29 种类型就能工作。
