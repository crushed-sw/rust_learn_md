# 8.7 rayon：数据并行的工作窃取

> `v.iter().map(f).collect()` 是单线程的——4核心 CPU 上你只用了 1 个核心。`v.par_iter().map(f).collect()` 是并行的——简单到只有一个字母的区别。rayon 是 Rust 数据并行的事实标准，它的 API 几乎完全仿照 `Iterator` 的设计——`par_iter()`、`map`、`filter`、`fold`、`reduce`——让你用"迭代器"的心智模型写并行代码。本节拆解 rayon 的架构：work-stealing 调度器的内部实现（与 tokio 的对比）、`ParallelIterator` trait 的方法设计、以及全局 `ThreadPool` 的单例模式。

## 1. rayon 的 API 设计：ParallelIterator

### 1.1 与 Iterator 的对应关系

```rust
// 单线程 Iterator：
let sum: i32 = vec![1, 2, 3, 4, 5]
    .iter()           // Iterator<Item = &i32>
    .map(|x| x * 2)  // 每个元素 ×2
    .sum();           // 累加

// rayon 并行 ParallelIterator：
use rayon::prelude::*;

let sum: i32 = vec![1, 2, 3, 4, 5]
    .par_iter()       // ParallelIterator<Item = &i32>
    .map(|x| x * 2)  // 每个元素 ×2（分布在多个线程上）
    .sum();           // 并行累加

// 方法对应表：
//  Iterator           →  ParallelIterator
//  ─────────          →  ─────────────────
//  iter()             →  par_iter()
//  iter_mut()         →  par_iter_mut()
//  into_iter()        →  into_par_iter()
//  map()              →  map()
//  filter()           →  filter()
//  fold()             →  fold() / reduce()
//  for_each()         →  for_each()
//  find()             →  find_any() / find_first()
//  collect()          →  collect()
```

### 1.2 ParallelIterator trait 的核心方法

```rust
// rayon 的核心 trait（简化）
pub trait ParallelIterator: Send {
    type Item: Send;

    fn drive_unindexed<C>(self, consumer: C) -> C::Result
    where
        C: UnindexedConsumer<Self::Item>;

    // ━━━ 提供的默认方法 ━━━

    fn map<F, R>(self, map_op: F) -> Map<Self, F>
    where F: Fn(Self::Item) -> R + Sync + Send,
          R: Send
    { Map::new(self, map_op) }

    fn filter<F>(self, filter_op: F) -> Filter<Self, F>
    where F: Fn(&Self::Item) -> bool + Sync + Send
    { Filter::new(self, filter_op) }

    fn fold<T, ID, F>(self, identity: ID, fold_op: F) -> Fold<Self, ID, F> { ... }
    fn reduce<OP, ID>(self, identity: ID, reduce_op: OP) -> Reduce<Self, ID, OP> { ... }
    fn for_each<F>(self, op: F) where F: Fn(Self::Item) + Sync + Send { ... }
    // ...
}

// 关键设计：每个适配器方法返回一个新的类型（Map、Filter 等）
// → 与 Iterator 的适配器模式完全一致
// → 编译器可以内联所有适配器层，零运行时开销
```

### 1.3 fold vs reduce

```rust
// fold: 每个线程独立累积，最后合并
let sum: i32 = (0..1000)
    .into_par_iter()
    .fold(
        || 0,                    // 每个线程的初始值
        |acc, x| acc + x,        // 线程内的累积操作（必须满足结合律！）
    )
    .reduce(
        || 0,                    // 合并的初始值
        |a, b| a + b,            // 合并各线程的结果
    );

// reduce: 两两合并
let max: Option<i32> = (0..1000)
    .into_par_iter()
    .reduce(
        |a, b| a.max(b)          // 两两比较
    );
```

## 2. Work-Stealing 调度器

### 2.1 rayon vs tokio 的调度器对比

```
┌─────────────────────────────────────────────────────────┐
│  rayon 的 Work-Stealing                                  │
│  ─────────────────────                                   │
│  - 目标：CPU-bound 数据并行任务                          │
│  - 任务：无状态的纯计算（map、filter、fold）             │
│  - 偷取策略：LIFO（自己取）+ FIFO（别人偷）              │
│  - 全局线程池：单例，与 CPU 核心数相同                    │
│  - 特点：任务可以被分割（split）                          │
├─────────────────────────────────────────────────────────┤
│  tokio 的 Work-Stealing                                  │
│  ─────────────────────                                   │
│  - 目标：I/O-bound 异步任务                              │
│  - 任务：有状态的 future（跨 .await 恢复）               │
│  - 偷取策略：类似（也是 LIFO + FIFO）                    │
│  - 线程管理：worker threads + blocking threads           │
│  - 特点：任务可以暂停和恢复（.await）                     │
└─────────────────────────────────────────────────────────┘
```

### 2.2 rayon 的任务分割（Splitting）

```rust
// rayon 的核心：把工作"分割"成小块，分布到各线程

// 当 par_iter() 处理一个大 Vec 时：
// 1. 初始：1 个任务 [0..1000000]
// 2. 主线程开始处理
// 3. 其他线程空闲 → 从主线程"偷"一半：500000..1000000
// 4. 主线程继续处理 0..500000
// 5. 更多线程加入 → 继续分割 → 直到每个块足够小

// 这个分割过程是通过 ParallelIterator 的"内部 consumer 模式"实现的
// 每个 consumer 可以被 split_at() 分成两个
```

### 2.3 join / scope 的细粒度并行

```rust
use rayon::prelude::*;

// ━━━ join: 并行执行两个闭包 ━━━
let (a, b) = rayon::join(
    || expensive_calculation_1(),
    || expensive_calculation_2(),
);
// 两个计算在 rayon 线程池中并行执行

// ━━━ scope: 创建子任务的作用域 ━━━
rayon::scope(|s| {
    s.spawn(|_| {
        // 子任务 1（在 rayon 线程池中运行）
    });
    s.spawn(|_| {
        // 子任务 2
    });
});  // ← scope 结束时自动 join 所有子任务
```

## 3. 全局线程池的单例模式

### 3.1 rayon 的静态初始化

```rust
// rayon 维护一个全局线程池（单例）
// 第一次使用 par_iter() 时自动初始化
// 线程数默认 = CPU 逻辑核心数

// 也可以显式配置：
rayon::ThreadPoolBuilder::new()
    .num_threads(8)
    .build_global()
    .unwrap();

// ━━━ 如果需要独立的线程池 ━━━
let pool = rayon::ThreadPoolBuilder::new()
    .num_threads(4)
    .build()
    .unwrap();

pool.install(|| {
    // 在这个独立线程池中执行
    let result: Vec<_> = (0..1000)
        .into_par_iter()
        .map(|x| x * 2)
        .collect();
});
```

### 3.2 线程池的生命周期

```
rayon 的全局线程池：
  - 懒初始化（lazy init）：第一次使用时创建
  - 进程级生命周期：随进程退出而销毁
  - 在 main() 退出前保持活跃

与 std::thread::scope 的区别：
  - rayon: 全局线程池，适合"有很多并行任务"的场景
  - thread::scope: 临时线程，适合"一次性并行操作"
  - rayon 不会为每次 par_iter() 创建/销毁线程
```

## 4. rayon vs tokio：何时用什么？

```
┌─────────────────────┬──────────────────┬──────────────────┐
│ 场景                 │ rayon            │ tokio            │
├─────────────────────┼──────────────────┼──────────────────┤
│ CPU 密集型计算       │ ✅ 首选          │ ❌ 会阻塞 worker │
│ 图像处理、科学计算   │ ✅               │ ❌               │
│ I/O 密集型（网络）   │ ❌ 没有异步 I/O  │ ✅ 首选          │
│ Web 服务器           │ ❌               │ ✅               │
│ 混合场景             │ spawn_blocking   │ 配合 rayon      │
└─────────────────────┴──────────────────┴──────────────────┘

// 混合使用：tokio 做 I/O，rayon 做计算
async fn handle_request(data: Vec<u8>) -> Result<Vec<u8>> {
    // tokio 负责 I/O
    let parsed = parse_headers(&data)?;

    // rayon 负责纯计算
    let processed = tokio::task::spawn_blocking(move || {
        data.par_iter()
            .map(|b| transform(*b))
            .collect::<Vec<_>>()
    }).await?;

    Ok(processed)
}
```
