# 8.4 anyhow + thiserror：错误处理的双子星

> David Tolnay 写了两个最常用的 Rust 错误处理 crate——`thiserror` 和 `anyhow`。有趣的是：它们的定位恰恰相反——`thiserror` 是给库用的（定义结构化的错误类型），`anyhow` 是给应用用的（灵活地传播任何错误）。更妙的是，它们来自同一个作者，意味着它们在设计上互补而不重叠。本节不教你怎么用它们（Phase 6.3 已经覆盖了），而是剖析它们的**内部实现**：`anyhow::Error` 如何用一个 `Box<ErrorImpl>` 容纳一切？`thiserror` 的 derive 宏生成了什么？`anyhow::Context` trait 的设计妙在哪里？

---

## 1. 两个 crate，两种哲学

### 1.1 库 vs 应用

```
库 (Library) 的错误需求：
  ✓ 调用者需要能 match 不同的错误变体
  ✓ 错误类型要暴露在公共 API 中
  ✓ 每个变体携带可操作的信息
  → thiserror: #[derive(Error)] 自动生成 Display + Error + source()

应用 (Application) 的错误需求：
  ✓ 任何错误都可以传播（不需要为每个依赖定义变体）
  ✓ 给错误附加"我当时在做什么"的上下文
  ✓ main() 中统一处理所有错误
  → anyhow: anyhow::Error 可以包装任何 std::error::Error
```

### 1.2 分工

```rust
// thiserror 做的事：定义错误类型
#[derive(Error, Debug)]
pub enum DataStoreError {
    #[error("database connection failed")]
    ConnectionFailed(#[source] sqlx::Error),

    #[error("key `{0}` not found")]
    NotFound(String),

    #[error("serialization failed")]
    Serialization(#[from] serde_json::Error),
}

// anyhow 做的事：传播任何错误 + 附加上下文
use anyhow::Context;

fn load_config(path: &str) -> anyhow::Result<Config> {
    let data = std::fs::read_to_string(path)
        .with_context(|| format!("failed to read config from {}", path))?;
    let config: Config = serde_json::from_str(&data)
        .context("failed to parse config as JSON")?;
    Ok(config)
}
```

## 2. `anyhow::Error` 的内部布局

### 2.1 一个 Box 装下所有

```rust
// anyhow::Error 的内部结构（简化）

// 核心：一个 Box<ErrorImpl>
pub struct Error {
    inner: Box<ErrorImpl>,
}

// ErrorImpl 容纳了：
struct ErrorImpl {
    // 链式错误：每个 anyhow::Error 可以包装另一个错误
    // → 形成 source() 链
    // 实际的错误对象 + backtrace + 链
    // 内部用 vtable 实现动态分发
}

// 关键设计：Error 不暴露内部类型！
// 你不能 match anyhow::Error 的变体
// → 因为 anyhow 是"应用级"的——应用不需要 match 变体，只需要 Display + 传播
```

### 2.2 `anyhow!` 宏的展开

```rust
// 你写的
anyhow!("failed to process item {}", id)

// 展开后（简化）
{
    // 使用 format! 创建错误消息
    let message = format!("failed to process item {}", id);
    // 创建一个 anyhow::Error，内部存储为字符串错误
    anyhow::Error::msg(message)
}
// anyhow::Error::msg 内部：
// - 如果可能，存储 &'static str（避免分配）
// - 否则，存储 String
// - 确保实现了 Display + Debug + Error
```

### 2.3 `Context` trait 的设计

```rust
// anyhow::Context trait（简化）
pub trait Context<T, E> {
    fn context<C>(self, context: C) -> Result<T, anyhow::Error>
    where C: Display + Send + Sync + 'static;

    fn with_context<C, F>(self, f: F) -> Result<T, anyhow::Error>
    where C: Display + Send + Sync + 'static,
          F: FnOnce() -> C;
}

// 实现：为所有 Result<T, E> where E: std::error::Error + Send + Sync + 'static
impl<T, E> Context<T, E> for Result<T, E>
where E: std::error::Error + Send + Sync + 'static
{
    fn context<C>(self, context: C) -> Result<T, anyhow::Error>
    where C: Display + Send + Sync + 'static
    {
        self.map_err(|error| {
            // 创建一个新的 anyhow::Error
            // - 把原始 error 作为 source
            // - 把 context 作为消息
            anyhow::Error::new(error).context(context)
        })
    }
}

// 使用对比：
// ❌ 传统方式：
fs::read_to_string(path)
    .map_err(|e| format!("failed to read {}: {}", path, e))?;
// → 丢失了原始错误类型！只能得到 String

// ✅ anyhow 的方式：
fs::read_to_string(path)
    .with_context(|| format!("failed to read {}", path))?;
// → 保留原始 io::Error 作为 source，附加了格式化消息
// → 打印时：failed to read config.json: No such file or directory (os error 2)
```

## 3. `thiserror` derive 的代码生成

### 3.1 展开前 vs 展开后

```rust
// 你写的
use thiserror::Error;

#[derive(Error, Debug)]
pub enum MyError {
    #[error("IO error: {0}")]
    Io(#[from] std::io::Error),

    #[error("invalid input: {detail}")]
    InvalidInput { detail: String },
}

// thiserror 生成的代码（用 cargo expand 查看，简化版）：

impl std::fmt::Display for MyError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            MyError::Io(_0) => {
                write!(f, "IO error: {}", _0)
            }
            MyError::InvalidInput { detail } => {
                write!(f, "invalid input: {}", detail)
            }
        }
    }
}

impl std::error::Error for MyError {
    fn source(&self) -> Option<&(dyn std::error::Error + 'static)> {
        match self {
            MyError::Io(_0) => Some(_0),  // ← #[source] 或 #[from] 标注的字段
            MyError::InvalidInput { .. } => None,
        }
    }
}

impl From<std::io::Error> for MyError {
    fn from(source: std::io::Error) -> Self {
        MyError::Io(source)  // ← #[from] 自动生成 From impl
    }
}
```

### 3.2 `#[from]` vs `#[source]`

```rust
// #[from] 做了两件事：
// 1. 把字段标记为 error source（出现在 source() 链中）
// 2. 生成 From<E> for MyError 实现

// #[source] 只做第 1 件事：
// 把字段标记为 error source，但不生成 From impl
// 用于不能自动转换的场景

#[derive(Error, Debug)]
pub enum AppError {
    #[error("database error")]
    Database(#[from] sqlx::Error),     // ← 可以用 ? 自动转换

    #[error("config error: {0}")]
    Config(#[source] ConfigError),     // ← 出现在 source() 链中，但不用 ? 自动转换
}
```

## 4. 为什么分两个 crate？

### 4.1 库不应该依赖 anyhow

```rust
// 如果你的库用 anyhow::Error 作为公共 API 的错误类型：
pub fn parse_config(path: &str) -> anyhow::Result<Config> { ... }
// ↑ 调用者收到了 anyhow::Error，但他们：
// - 不能 match（因为是 opaque type）
// - 不能做程序化的错误恢复
// - 被强制依赖 anyhow（传递性依赖）

// 正确的方式：
pub fn parse_config(path: &str) -> Result<Config, ConfigError> { ... }
// ↑ ConfigError 用 thiserror 定义
// ↑ 调用者可以 match 变体、做决策、做恢复
```

### 4.2 应用不需要 thiserror 的严谨性

```rust
// 一个 CLI 工具可能有 20 个不同的错误来源：
// - 文件不存在 (io::Error)
// - JSON 解析失败 (serde_json::Error)
// - 网络请求失败 (reqwest::Error)
// - 配置无效 (ConfigError)
// - 环境变量未设置 (env::VarError)
// ...

// 如果为每个来源定义 enum 变体 → 20 个变体的 enum
// → 90% 的代码是错误转换（From impl）
// → 这是"库级别的严谨性"在应用中的浪费

// anyhow 的哲学：应用只需要知道"出了什么错"和"为什么"
// → 不需要程序化匹配 20 个变体（因为 main() 只打印错误就退出了）
```

## 5. `anyhow::Error` 与 Box<dyn Error> 的区别

```rust
// 为什么用 anyhow::Error 而不是 Box<dyn Error>？

// 1. anyhow::Error 自带 backtrace 捕获
use anyhow::anyhow;
let err = anyhow!("something went wrong");
// err.backtrace() → Some(Backtrace)（如果 RUST_BACKTRACE=1）
// Box<dyn Error> → 没有自带 backtrace

// 2. anyhow::Error 的 Display 更友好
// anyhow 自动打印错误链："{context}: {source}"
// Box<dyn Error> 只打印自己

// 3. anyhow::Error 有一套 Context trait 方法
// .context() / .with_context() 给 Result 附加信息
// Box<dyn Error> 没有这些方法

// 4. anyhow::Error 实现了 Send + Sync + 'static
// （Box<dyn Error> 不一定，取决于内部类型）
```
