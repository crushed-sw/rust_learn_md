# 8.6 regex：编译期正则引擎

> 大多数语言的 regex 库依赖 PCRE（Perl Compatible Regular Expressions）——一个 30 年历史的 C 库。Rust 的 regex crate 走了完全不同的路：纯 Rust 实现、编译期预处理、运行时 SIMD 加速——结果是在许多 benchmark 上比 PCRE 更快。本节拆解 regex 的架构：正则表达式如何从字符串编译为 NFA → DFA → 字节码；为什么纯 Rust 实现反而更快（DFA 缓存 + SIMD 加速）；以及 `regex-automata` 子 crate 如何提供底层的"正则引擎构建 API"。

---

## 1. regex 的编译管线

### 1.1 从字符串到匹配引擎

```rust
use regex::Regex;

// 这一行背后发生了什么？
let re = Regex::new(r"\d{4}-\d{2}-\d{2}").unwrap();
//                           ↑ 正则表达式（字符串）
//                           ↓
//                      编译过程开始
```

```
Regex::new() 的编译管线：

源代码字符串
  │ "\d{4}-\d{2}-\d{2}"
  ▼
Parser（解析器）
  │ 解析为正则 AST
  ▼
AST（抽象语法树）
  │ Digit(4), Literal('-'), Digit(2), Literal('-'), Digit(2)
  ▼
HIR（高级中间表示）
  │ 去糖：\d → [0-9]，简化表示
  ▼
NFA（非确定性有限自动机）
  │ 有 ε-transition 的状态机
  ▼
DFA（确定性有限自动机）
  │ 无 ε-transition，每个状态 + 输入 → 唯一下一个状态
  ▼
字节码（或直接使用 DFA 引擎）
  │ 最终的执行引擎
```

### 1.2 为什么需要多层？

```rust
// 每一层有不同目的：

// AST: 语法层面，保留所有结构
// \d{4} → DigitQuantifier { count: 4 }

// HIR: 语义层面，去糖和简化
// \d → [0-9]（展开字符类）
// {4} → 展开为四次重复

// NFA: 执行引擎的中间形式
// 引入 ε-transition（"免费"状态转移）
// 适合构建和合并（如 | 运算符、重复）

// DFA: 优化的执行形式
// 从 NFA 通过"子集构造法"（subset construction）计算出来
// 每个输入字符 → O(1) 状态转移
// 是正则引擎运行时的"快车道"
```

## 2. NFA → DFA：为什么 DFA 更快？

### 2.1 NFA 的问题

```
NFA (Nondeterministic Finite Automaton):

状态 A --[0-9]--> 状态 B
状态 A --[0-9]--> 状态 C  ← 同一个输入可以有多个可能的下一个状态！
状态 A --ε-----> 状态 D  ← "免费"的转移（不消耗输入）

运行时：需要同时追踪所有"可能的活动状态"
→ 每次状态转移可能需要遍历多个状态
→ 回溯（如果实现为回溯引擎）或并行模拟
```

### 2.2 DFA 的优势

```
DFA (Deterministic Finite Automaton):

每个（状态, 输入）组合 → 唯一的下一状态
→ O(1) 每次字符转移
→ 对输入字符串的每个字符，只做一次查表

代价：DFA 的状态数可能是 NFA 状态数的指数倍（最坏情况）
→ 实际模式中，状态数通常是可管理的
→ regex crate 使用 DFA 缓存：编译时计算，运行时查找
```

### 2.3 为什么纯 Rust 比 C 的 PCRE 快？

```
PCRE（C 实现的 Perl 兼容正则引擎）：
  - 基于回溯（backtracking）
  - 最坏情况 O(2^n)（对某些模式和输入）
  - 支持 backreference（如 \1），强制使用回溯

regex（Rust 实现）：
  - 基于有限自动机（NFA → DFA）
  - 严格 O(n)（n = 输入长度）
  - 不支持 backreference（故意的——保持线性复杂度）
  - 使用 SIMD 加速字符串搜索（memchr + packed_simd）

结论：regex 通过"限制功能"换取了"保证性能"
→ 不支持 backreference，但保证 O(n) 时间
→ 对大多数实际模式（没有 backreference），regex 更快
```

## 3. SIMD 加速的字符串搜索

### 3.1 `memchr`：找单个字节

```rust
// regex 内部大量使用 memchr crate
// memchr 用 SIMD 指令在字节数组中高速查找

// 原理：一次比较 32 个字节（AVX2）或 16 个字节（SSE2）
// 而不是传统的一次一个字节

// 在 regex 中的应用：
// 1. 先找模式的"前缀"（prefix literal）
//    例如 \d{4}-\d{2}-\d{2} 的前缀可能是 '2' '0' '2' '4'
// 2. 用 SIMD 快速跳过不匹配的输入区域
// 3. 只在候选位置运行完整的正则引擎

// 这是 regex 在"大量不匹配"场景下极快的原因：
// 用 SIMD 快速排除 99% 的位置，只在 1% 的位置运行完整引擎
```

### 3.2 Teddy 算法（多子串搜索）

```
Teddy algorithm（regex 内部使用的多模式搜索）：

问题：同时搜索多个子串（如 ["foo", "bar", "baz"]）
传统方法：对每个子串分别搜索 → O(m * n)
Teddy：用 SIMD 一次检查 16/32 个字节 → 接近 O(n)

原理：
1. 把每个子串的每个位置映射到 SIMD 向量的一个字节
2. 对输入做 SIMD 比较
3. 从 SIMD 结果中推导出候选匹配位置
4. 只在候选位置做精确验证
```

## 4. `regex::bytes::Regex` 的零拷贝

```rust
// regex 有两套 API：

// 1. regex::Regex —— 作用于 &str
let re = Regex::new(r"\d+").unwrap();
let matches: Vec<&str> = re.find_iter("a 42 b 99 c").collect();
// → ["42", "99"]（str 切片）

// 2. regex::bytes::Regex —— 作用于 &[u8]
use regex::bytes::Regex as BytesRegex;
let re = BytesRegex::new(r"\d+").unwrap();
let data = b"a 42 b 99 c";
let matches: Vec<&[u8]> = re.find_iter(data).collect();
// → [b"42", b"99"]（字节切片）
// 优势：不需要 UTF-8 验证！直接在字节上匹配
// → 可以搜索任意的二进制数据
// → 零拷贝（返回的是输入数据的引用）
```

## 5. `regex-automata` 子 crate

### 5.1 底层自动机 API

```rust
// regex-automata 提供了底层的 DFA/NFA 构建和执行 API
// 适合需要"自己构建正则引擎"的场景

use regex_automata::{dfa, nfa, Input, Match};

// 构建 DFA
let dfa = dfa::dense::DFA::new(r"\d{4}-\d{2}-\d{2}")?;

// 执行匹配
let input = Input::new(b"2024-01-15");
let result = dfa.try_search_fwd(&input)?;

// 构建 NFA
let nfa = nfa::thompson::NFA::new(r"foo|bar|baz")?;

// 从 NFA 构建 PikeVM（NFA 模拟器）
// ... 用于需要捕获组的场景（DFA 不直接支持捕获组）
```

### 5.2 架构分层

```
regex crate（高层 API）
  ├── 编译：regex → HIR → NFA → DFA
  ├── 执行：自动选择最优引擎
  └── 依赖 regex-automata

regex-automata crate（底层 API）
  ├── NFA 编译和执行
  ├── DFA 编译和执行
  ├── PikeVM（NFA 模拟器，支持捕获组）
  ├── BoundedBacktracker（有限回溯引擎）
  └── meta（自动引擎选择器）

regex-syntax crate（最底层）
  ├── 解析：正则字符串 → AST → HIR
  └── 被 regex 和 regex-automata 使用
```
