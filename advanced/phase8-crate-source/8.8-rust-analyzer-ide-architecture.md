# 8.8 rust-analyzer：IDE 架构的典范

> rust-analyzer 不是传统意义上的"crate"——它是一个完整的 IDE 后端（LSP 服务器），有超过 30 万行 Rust 代码。但它的架构设计是 Rust 项目工程的顶级范例：**salsa 增量计算框架**让你用声明式的方式定义"这个计算结果依赖什么"，自动获得增量重算和缓存；**rowan 语法树**（CST）用无生命周期的 `&self` 代替传统的 AST 引用，同时支持增量重解析；**LSP 异步事件循环**优雅地处理编辑器与语言服务器的通信。即使你永远不会写 IDE 代码，这些架构思想——增量计算、容错解析、抽象分离——可以直接应用到任何大型 Rust 项目中。

---

## 1. 为什么 rust-analyzer 不用 rustc？

### 1.1 rustc 是"批处理编译器"

```
rustc 的设计目标（批处理编译器）：
  ✓ 编译整个 crate，从源码到二进制
  ✓ 对正确性要求极高（不能有假阴性）
  ✓ 可以花几秒钟甚至几分钟处理一个 crate

rust-analyzer 需要（IDE 后端）：
  ✓ 实时响应——每次按键后 < 100ms 给出结果
  ✓ 增量计算——只重算改了的部分
  ✓ 容错——代码不完整时也要能给出补全、诊断
  ✓ 只分析单个文件（不编译整个 crate）
```

### 1.2 关键差异

| 维度 | rustc | rust-analyzer |
|------|-------|---------------|
| 输入 | 完整的 crate | 编辑中的文件（可能语法错误） |
| 响应时间 | 秒~分钟 | < 100ms |
| 计算方式 | 全量 | 增量（只重算改了的部分） |
| 容错 | 报错就停 | 继续分析，给出部分结果 |
| 生命周期 | 进程级 | 跨编辑会话（内存常驻） |

## 2. salsa：增量计算框架

### 2.1 问题：每次按键都要重新分析整个文件？

```rust
// 你改了第 1000 行的函数名
// → rust-analyzer 需要重新分析整个文件吗？
// 如果是，那每次按键都可能需要 500ms+ —— 不可接受

// salsa 的解决方案：
// 把每个计算任务定义为"带依赖的查询"
// → 只重算输入改变了的查询
```

### 2.2 salsa 的核心抽象

```rust
// salsa 是一个声明式的增量计算框架（类似于 rustc 的 query system）

// ━━━ 定义数据库 ━━━
#[salsa::database(Jar)]
struct AnalysisDatabase {
    storage: salsa::Storage<Self>,
}

// ━━━ 定义输入（改了就 invalidate）━━━
#[salsa::input]
struct SourceFile {
    text: String,
    file_path: String,
}

// ━━━ 定义查询（自动缓存、自动增量）━━━
#[salsa::tracked]
fn parse_file(db: &dyn AnalysisDatabase, file: SourceFile) -> SyntaxTree {
    // 解析 text 为语法树
    // salsa 自动：缓存结果 + 追踪依赖
    // 如果 file.text 没变 → 直接返回缓存结果
}

#[salsa::tracked]
fn type_check(db: &dyn AnalysisDatabase, file: SourceFile) -> Vec<Diagnostic> {
    let tree = parse_file(db, file);  // ← salsa 追踪这个依赖
    // 如果 parse_file 的结果没变 → 不需要重算 type_check
    // 如果 parse_file 的结果变了 → 自动重算
    // ...
}

// ━━━ 使用 ━━━
let file = SourceFile::new(db, source_code, "main.rs".into());
let diagnostics = type_check(&db, file);
// 第二次调用时，如果 source_code 没变 → 直接返回缓存结果（O(1)）
```

### 2.3 salsa 的设计思想

```
salsa 的核心：把"缓存失效"自动化

传统方式（手动缓存）：
  - 你需要自己决定"什么时候缓存失效"
  - 你需要自己写"如果 A 变了，B 和 C 也需要重新计算"

salsa 方式（声明式）：
  - 你声明"这个查询依赖那个查询"
  - salsa 自动追踪依赖 → 自动决定重算范围
  - 你只需要写"怎么算"（业务逻辑），salsa 负责"什么时候算"（缓存管理）

这种模式可以应用到任何"有依赖关系的计算密集型任务"——
不一定是编译器，也可以是：
  - 构建系统（类似 Bazel/Buck 的 action graph）
  - 金融计算（依赖链上的重新定价）
  - 游戏引擎（资源依赖的重新加载）
```

## 3. rowan：无生命周期的语法树

### 3.1 问题：传统的 AST 有生命周期问题

```rust
// 传统 Rust AST（带生命周期）：
enum Expr<'a> {
    Literal(i64),
    Add(&'a Expr<'a>, &'a Expr<'a>),  // ← 生命周期！
}

// 问题：
// 1. 所有的 Expr 必须属于同一个生命周期 'a
// 2. 不能在树的不同部分有不同的生命周期
// 3. 增量重解析时：旧节点和新节点有不同的生命周期 → 不能混用
// 4. 任何节点持有引用 → 不能在不重建树的情况下修改树

// → 这种 AST 不支持增量重解析
```

### 3.2 rowan 的解决方案：CST（Concrete Syntax Tree）

```rust
// rowan 的"红绿树"（Red-Green Tree）设计：

// "绿树"（Green Tree）：不可变的、无引用的、基于索引的树
// - 节点通过索引（而非指针）引用子节点
// - 可以在不销毁旧树的情况下构建新树
// - 旧树和新树共享不变的部分

// "红树"（Red Tree / SyntaxNode）：
// - 包装绿树 + 偏移量 → 提供树导航 API

// ━━━ 使用 ━━━
use rowan::SyntaxNode;

let root: SyntaxNode = parse(source_code);
let child = root.first_child().unwrap();

// 关键：所有方法返回的都是拥有所有权的值，不是引用！
// child 不借用 root！root 和 child 可以独立存在
// → 支持增量重解析：旧节点和新节点可以共存
```

### 3.3 红绿树如何支持增量重解析

```
将 source code "hello world" 解析为 CST：

完整解析：
  [hello world]
  → Node A (整个文件)
    ├── Node B ("hello")
    └── Node C ("world")

增量重解析（只改了 "world" → "rust"）：
  [hello rust]
  → 新的 Node A'
    ├── Node B  ← 不变！重用旧树中的 Node B！
    └── Node C' ("rust")  ← 新建

关键：Node B 在两个版本之间共享（因为 "hello" 没变）
→ 不需要重新解析 "hello"
→ 不需要为 Node B 分配新内存
→ 旧树中的 Node B 和新树中的 A' 可以共存
```

## 4. LSP 异步事件循环

### 4.1 LSP 协议

```
LSP (Language Server Protocol):
  编辑器 ←→ 语言服务器（rust-analyzer）

  编辑器发送请求（Request）：
    - textDocument/didChange (文档被修改)
    - textDocument/completion (请求自动补全)
    - textDocument/hover (鼠标悬停)
    - textDocument/definition (跳转到定义)
    - ...

  rust-analyzer 返回响应：
    - CompletionList (补全列表)
    - Hover (悬停信息)
    - Location (定义位置)
    - ...
```

### 4.2 rust-analyzer 的事件循环架构

```rust
// rust-analyzer 的主事件循环（极度简化）

// ━━━ 三个层次的抽象 ━━━

// 1. lsp-server crate: 处理 JSON-RPC 协议
//    → 序列化/反序列化 LSP 消息

// 2. rust-analyzer: 协调分析请求
//    → 接收 LSP 请求 → 调度分析任务 → 返回响应

// 3. Analysis (salsa 数据库):
//    → 实际的语义分析（类型检查、名称解析等）

// 关键设计：请求去抖动
// 用户快速打字 → 10 个 didChange 事件快速连续到来
// → rust-analyzer 不会为每个事件重新分析
// → 只处理最后一个（去抖动）
// → 中间的 didChange 只用于更新文件内容，不触发分析
```

### 4.3 请求调度

```rust
// rust-analyzer 的请求调度策略（简化）

// 高优先级（阻塞用户交互）：
//   - completion (自动补全)
//   - hover (悬停)
//   → 需要 < 100ms 响应

// 中优先级（后台分析）：
//   - semantic tokens (语义高亮)
//   - diagnostics (错误诊断)
//   → 可以在用户空闲时处理

// 低优先级（非关键）：
//   - inlay hints (类型标注提示)
//   - code lens (代码透镜)
//   → 最低优先级，不影响交互体验

// 调度器根据优先级和文件焦点分配 CPU 时间
```

## 5. 架构全景

```
rust-analyzer 的 crate 地图（简化）：

  rust-analyzer/              ← 二进制 + LSP 事件循环
  ├── ide/                    ← 高级 IDE 功能（补全、重构、诊断）
  ├── ide-db/                 ← IDE 层的数据聚合
  ├── hir/                    ← 高级中间表示（类型、名称解析）
  ├── hir-def/                ← HIR 定义
  ├── hir-ty/                 ← HIR 类型系统
  ├── hir-expand/             ← 宏展开
  ├── base-db/                ← salsa 数据库的基础 trait
  ├── syntax/                 ← 语法层（rowan 的包装）
  ├── parser/                 ← 手写递归下降解析器
  ├── tt/                     ← Token Tree（宏处理）
  ├── profile/                ← 性能分析
  ├── vfs/                    ← 虚拟文件系统
  ├── project-model/          ← Cargo 项目模型
  └── flycheck/               ← 调用 cargo check

架构层次：
  LSP 层  ──────  接收/发送 LSP 消息
  ide 层  ──────  补全、重构、诊断（高级功能）
  hir 层  ──────  类型检查、名称解析（语义分析）
  syntax 层 ────  解析、语法树（语法分析）
  salsa 层 ─────  增量计算框架（基础设施）
```

## 6. 值得迁移到其他项目的思想

```
从 rust-analyzer 可以学到的架构思想：

1. 增量计算（salsa）
   → 适用：任何"有依赖关系的计算密集型任务"
   → 关键：声明式依赖 → 自动缓存失效 → 自动增量重算

2. 无生命周期的树（rowan）
   → 适用：任何需要"增量更新 + 部分复用"的树结构
   → 关键：基于索引的引用 → 节点独立生命周期 → 新旧树共存

3. 分层架构
   → LSP 层 / ide 层 / hir 层 / syntax 层 / 基础设施层
   → 每层独立演进，上层不依赖下层的内部实现

4. 容错优先
   → 代码有语法错误？→ 继续解析，尽力给出补全
   → IDE 不是编译器——它必须在不完整的代码上工作

5. 请求优先级调度
   → 不是所有请求都一样重要
   → 用户可见的交互优先，后台分析其次
```
