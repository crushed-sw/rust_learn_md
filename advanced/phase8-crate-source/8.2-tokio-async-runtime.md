# 8.2 tokio：异步运行时的标杆

> tokio 是 Rust 异步生态的基石——从 web 服务器到数据库驱动，从 gRPC 到 WebSocket，几乎所有异步项目都运行在它之上。但你调用 `tokio::spawn` 的时候，背后发生了什么？本节深入 tokio 的源码架构：多线程 work-stealing 调度器如何分配任务、`mio` 如何抽象 epoll/kqueue/IOCP、定时器的哈希时间轮如何用 O(1) 时间管理百万个定时器。这不是 tokio 使用教程——Phase 3 已经覆盖了异步编程——这是**运行时内部设计**研读。

## 1. tokio 的架构全景

### 1.1 四层架构

```
┌─────────────────────────────────────────────────────┐
│  用户代码层                                          │
│  async fn、tokio::spawn、select!、#[tokio::main]    │
├─────────────────────────────────────────────────────┤

```

## 2. Work-Stealing 调度器

### 2.1 为什么是 Work-Stealing？

```
简单调度方案：
  全局任务队列 + N 个工作线程
  → 每个线程从全局队列取任务
  → 问题：全局队列成为瓶颈（锁竞争）

Work-Stealing：
  每个工作线程有自己的本地队列
  → 线程先处理自己的任务
  → 空闲时从其他线程的队列"偷"任务
  → 优点：几乎没有锁竞争（偷任务时才需要同步）
```

### 2.2 tokio 的实现结构

```rust
// tokio 多线程调度器的核心结构（简化）

// Worker（工作线程）
struct Worker {
    // 每个 Worker 有：
    // 1. 本地任务队列（单生产者、多消费者——只有自己和偷取者访问）
    // 2. 全局注入队列（多生产者、多消费者——任何线程都可以放入）
    // 3. 状态（搜索中、休眠中、运行中）
}

// 任务被 spawn 时的分发策略：
// - tokio::spawn 在运行时上下文中 → 放入当前 Worker 的本地队列
// - tokio::spawn 在非运行时上下文中 → 放入全局注入队列

// Worker 的主循环（简化）：
fn run(worker: &Worker) {
    loop {
        // 1. 先检查本地队列
        if let Some(task) = worker.local_queue.pop() {
            task.run();
            continue;
        }

        // 2. 再检查全局队列
        if let Some(task) = worker.global_queue.pop() {
            task.run();
            continue;
        }

        // 3. 从其他 Worker 偷任务
        if let Some(task) = steal_from_other_worker(worker) {
            task.run();
            continue;
        }

        // 4. 没有任务 → 进入休眠，等待 I/O 事件或新任务唤醒
        worker.park();
    }
}
```

### 2.3 偷取策略

```rust
// tokio 的偷取算法（简化工作逻辑）：

// 选择偷取目标：
// - 不是随机选择（避免缓存 ping-pong）
// - 使用轮询（round-robin）从其他 worker 偷取
// - 偷取时取对方队列的"另一半"（批量偷取，减少同步开销）

// 关键：每个 worker 的本地队列是一个"双向"队列
// - worker 自己从队列头部取任务（LIFO，缓存热）
// - 其他 worker 从队列尾部偷任务（FIFO，减少竞争）
// → 同一队列的两端，减少争用！
```

## 3. I/O 驱动：mio 的统一抽象

### 3.1 mio 做了什么

```rust
// mio 为三种平台提供了统一的异步 I/O 接口：
//
// Linux:   epoll
// macOS:   kqueue
// Windows: IOCP (I/O Completion Ports)
//
// tokio 的 I/O 驱动层在 mio 之上再封装

// mio 的核心 API：
use mio::{Events, Poll, Interest, Token};

let mut poll = Poll::new()?;
let mut events = Events::with_capacity(1024);

// 注册要监听的文件描述符
poll.registry().register(
    &mut socket,
    Token(0),
    Interest::READABLE,
)?;

// 阻塞等待事件
poll.poll(&mut events, None)?;

for event in events.iter() {
    match event.token() {
        Token(0) => { /* socket 可读了 */ }
        _ => {}
    }
}
```

### 3.2 tokio 如何在 I/O 和任务之间桥接

```rust
// tokio 的 I/O 驱动和调度器的协作（简化流程）：

// TcpListener::accept() 内部：
// 1. 向 I/O 驱动注册"这个 socket 可读了吗？"
// 2. 返回 Poll::Pending（I/O 还没就绪）
// 3. 调度器把这个任务挂起
// 4. I/O 事件到达 → epoll 通知 I/O 驱动
// 5. I/O 驱动唤醒对应的 waker → 任务被重新放入调度器队列
// 6. 调度器再次 poll 这个 future → 这次返回 Poll::Ready

// 关键：一切都是异步的，没有线程阻塞在 I/O 上
```

## 4. 定时器：哈希时间轮

### 4.1 为什么是哈希时间轮？

```
常规定时器实现：
  二叉堆（BinaryHeap）→ 插入 O(log n)，取最小 O(1)
  → 对于大量定时器（百万级），log n 仍然可观

哈希时间轮（Hashed Timing Wheel）：
  → 插入 O(1)，取到期的 O(1)
  → 适合：大量定时器、大多数在短时间内到期

tokio 的定时器就是哈希时间轮：
  多层时间轮，每层有不同的精度
  第 1 层: 1ms 精度，覆盖 ~1s
  第 2 层: ~1s 精度，覆盖 ~1min
  第 3 层: ~1min 精度，覆盖 ~1hour
  
  当第 1 层转完一圈 → 从第 2 层降级一批定时器到第 1 层
  → 摊销 O(1) 操作
```

### 4.2 时间轮的"降级"过程

```rust
// 简化版的哈希时间轮数据结构
struct TimingWheel {
    // 每层是一个"槽"的数组 + 当前指针
    levels: Vec<Level>,
}

struct Level {
    slots: Vec<Vec<TimerEntry>>,  // 每个槽存储到期的定时器列表
    current_slot: usize,          // 当前指针位置
    tick_duration: Duration,      // 每个 tick 的时间精度
}

// 插入定时器:
// 如果到期时间 < 第 1 层覆盖范围 → 插入第 1 层的对应槽
// 如果到期时间 > 第 1 层覆盖 → 计算在第 2 层的哪个槽
// ...

// 时间推进:
// 第 1 层指针前进 → 检查和触发当前槽中的定时器
// 第 1 层转完一圈 → 从第 2 层降级
//   第 2 层的当前槽中的定时器 → 根据剩余时间分配到第 1 层
```

## 5. `#[tokio::main]` 宏展开

```rust
// 你写的
#[tokio::main]
async fn main() {
    println!("Hello from async!");
}

// 宏展开后（简化版）
fn main() {
    // 1. 创建一个多线程 Runtime
    let runtime = tokio::runtime::Builder::new_multi_thread()
        .worker_threads(num_cpus)  // 默认 = CPU 核心数
        .enable_all()
        .build()
        .unwrap();

    // 2. 在这个 Runtime 上阻塞运行 async main
    runtime.block_on(async {
        println!("Hello from async!");
    });
}
```

## 6. 协作式调度的关键设计

### 6.1 为什么 tokio 不抢占

```rust
// tokio 是协作式调度（cooperative scheduling）
// → 任务必须主动让出 CPU（通过 .await）

// ❌ 这会阻塞整个 worker：
async fn bad_task() {
    loop {
        // 计算密集操作，没有 .await
        // → 这个 worker 的所有其他任务都被饿死
    }
}

// ✅ 应该用 spawn_blocking 或 tokio::task::yield_now：
async fn good_task() {
    loop {
        heavy_computation();
        tokio::task::yield_now().await;  // ← 主动让出 CPU
    }
}

// 或用 spawn_blocking（在独立线程池上运行）：
tokio::task::spawn_blocking(|| {
    heavy_computation();
}).await;
```

### 6.2 Budget 机制

```rust
// tokio 为每个任务分配一个"budget"（预算）
// 默认一个任务一次可以运行 128 次 poll 操作
// 超过预算 → 强制让出 → 放回队列尾部

// 这防止了一个任务"饿死"其他任务
// 但大多数正常任务不会触发这个限制
```
