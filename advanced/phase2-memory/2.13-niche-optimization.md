# 2.13 Niche 优化——编译器如何"偷"空间

> `Option<&T>` 和 `&T` 在内存中占同样的字节数。不是巧合——是编译器精心设计的**Niche 优化**。当一个类型有"不可能出现的位模式"时（比如引用不能是全零），编译器利用这些非法值来表示 `None`，从而省去了 `Option` 的判别位。`NonZeroU32`、`NonNull<T>`、`bool` 都有 niche——正是这些优化让 Rust 的 `Option<Box<T>>` 比 C++ 的 `std::optional<std::unique_ptr<T>>` 更省内存。理解 Niche 优化，是理解 Rust 为何能"零成本"的重要拼图。

---

## 1. Niche 优化的基本原理

### 1.1 什么是 Niche

```rust
// Niche（生态位）：一个类型中"不可用的"位模式
// 编译器利用这些位模式存储额外的信息（如 Option 的 None）

// 示例 1：引用的 Niche
// &T 永远不能是空（null）——全零的指针不是合法的引用
// → 全零是 &T 的 niche
// → Option<&T> 用全零表示 None，非零指针表示 Some(&T)
// → Option<&T> 大小 = &T 大小 = 8 字节

use std::mem::size_of;

assert_eq!(size_of::<Option<&i32>>(), size_of::<&i32>());  // 8 == 8
assert_eq!(size_of::<Option<&i32>>(), 8);

// 示例 2：NonZeroU32 的 Niche
// u32 的取值范围：0..=4294967295
// NonZeroU32 的取值范围：1..=4294967295
// → 0 是 NonZeroU32 的 niche
// → Option<NonZeroU32> 用 0 表示 None，非零值表示 Some
// → Option<NonZeroU32> 大小 = NonZeroU32 大小 = 4 字节

use std::num::NonZeroU32;

assert_eq!(size_of::<Option<NonZeroU32>>(), size_of::<NonZeroU32>());  // 4 == 4
assert_eq!(size_of::<Option<NonZeroU32>>(), 4);

// 对比：没有 niche 的 u32
assert_eq!(size_of::<Option<u32>>(), 8);  // 4 (value) + 4 (discriminant + padding)
// u32 的每个位模式都有效 → 没有 niche → Option 需要额外的空间存判别位
```

### 1.2 Niche 的工作原理

```rust
// 编译器的 Niche 检测过程：

// 1. 检查类型的"有效位模式"集合
//    - 引用：254 个可能值（全零被排除）
//    - bool：0 和 1（其他 254 个是 niche）
//    - NonZeroU32：1..=2^32-1（0 是 niche）
//    - char：0..=0x10FFFF（大于此范围的 u32 值是 niche）
//    - 枚举：已定义的 discriminant（未定义的 discriminant 是 niche）

// 2. 检查类型是否有足够多的 niche 来表示 Option 的 None
//    - 只需要 1 个 niche 就够

// 3. 将 niche 值映射到 None，将有效值映射到 Some
//    - 编译期完成——没有运行时开销

// 4. 布局结果：Option<T> 的大小 = T 的大小
//    （如果 T 有至少 1 个 niche）
```

## 2. 标准库中的 Niche 类型

### 2.1 `NonZero` 家族

```rust
use std::num::{
    NonZeroU8, NonZeroU16, NonZeroU32, NonZeroU64, NonZeroUsize,
    NonZeroI8, NonZeroI16, NonZeroI32, NonZeroI64, NonZeroIsize,
};
use std::mem::size_of;

// 每个 NonZero 类型都有一个 niche（值 0）
assert_eq!(size_of::<Option<NonZeroU32>>(), 4);
assert_eq!(size_of::<Option<NonZeroU8>>(), 1);
assert_eq!(size_of::<Option<NonZeroUsize>>(), 8);

// 用途：
// - Option<NonZeroU32> 表示"可能缺失的 ID" —— 比 Option<u32> 省 50%
// - 为"索引从 1 开始"的集合提供类型安全
// - FFI 中表示"非空句柄"
```

### 2.2 `NonNull<T>`

```rust
use std::ptr::NonNull;
use std::mem::size_of;

// NonNull<T>：保证非空的裸指针
// niche = 空指针（全零）
// → Option<NonNull<T>> 大小 = 指针大小 = 8 字节

assert_eq!(size_of::<Option<NonNull<i32>>>(), 8);
assert_eq!(size_of::<OptionalNonNull<i32>>(), size_of::<*mut i32>()); // 8 == 8

// NonNull<T> 是构建所有"非空"抽象的基础：
// - Box<T> 内部使用 NonNull
// - Vec<T> 内部使用 NonNull
// - Rc<T> / Arc<T> 内部使用 NonNull
// → Option<Box<T>> 大小 = 指针大小 = 8 字节！
assert_eq!(size_of::<Option<Box<i32>>>(), 8);
```

### 2.3 `bool` 的 Niche

```rust
use std::mem::size_of;

// bool 只有两个有效值：0 (false) 和 1 (true)
// u8 有 256 个值 → 254 个是 niche
// → Option<bool> 大小 = 1 字节（不是 2 字节）

assert_eq!(size_of::<Option<bool>>(), 1);          // 1 字节！
assert_eq!(size_of::<Option<Option<bool>>>(), 1);  // 还是 1 字节！

// 三重 Option 呢？
assert_eq!(size_of::<Option<Option<Option<bool>>>>(), 1); // 仍是 1 字节！

// 因为 254 个 niche 值 > 需要的 3 个状态——足够了

// 对比 C++：
// std::optional<bool> 至少 2 字节（bool + discriminant）
```

### 2.4 `char` 的 Niche

```rust
use std::mem::size_of;

// char 是 4 字节（Unicode 标量值：0..=0x10FFFF）
// 有效的 u32 值：0..=1114111（约 111 万个值）
// 无效的 u32 值：1114112..=4294967295（约 31.8 亿个！）
// → 巨大的 niche 空间

assert_eq!(size_of::<Option<char>>(), 4);           // 和 char 一样大
assert_eq!(size_of::<Option<Option<char>>>(), 4);   // 还是 4 字节
```

### 2.5 枚举的 Niche

```rust
use std::mem::size_of;

// 编译器自动为枚举检测 niche：

enum Color { Red, Green, Blue }
// 三个变体 → discriminant 需要 0, 1, 2
// 如果底层用 u8 存储：3..=255 是 niche
// → Option<Color> 可以是 1 字节

assert_eq!(size_of::<Option<Color>>(), 1);

// 更复杂的枚举：
enum Value {
    Int(i32),
    Float(f64),
}
// 两个变体 + 大数据 → discriminant + union
// i32 有 niche 吗？没有——所有 2^32 个值都有效
// f64 有 niche 吗？没有——所有 2^64 个位模式可能都有效（NaN 有多个表示）

// → Option<Value> 不能享受 niche 优化
assert!(size_of::<Option<Value>>() > size_of::<Value>());
```

## 3. Niche 优化的限制

### 3.1 没有 Niche 的类型

```rust
use std::mem::size_of;

// 以下类型没有 niche —— Option 会给它们增加额外的字节：
assert!(size_of::<Option<u8>>() > size_of::<u8>());
assert!(size_of::<Option<u16>>() > size_of::<u16>());
assert!(size_of::<Option<u32>>() > size_of::<u32>());
assert!(size_of::<Option<u64>>() > size_of::<u64>());
assert!(size_of::<Option<f64>>() > size_of::<f64>());
assert!(size_of::<Option<i32>>() > size_of::<i32>());

// 原因：这些类型的每个位模式都代表一个有效值
// → 没有"剩余的"位模式可以用来表示 None
```

### 3.2 Niche 只对 `Option` 类似的枚举有效

```rust
// Niche 优化只适用于"一个有效变体 + 一个/多个空变体"的模式
// 本质上是 Option-like 的枚举

// ✅ 享受 niche 优化：
enum MyOption<T> {
    MySome(T),
    MyNone,
}
// 如果 T 有 niche → MyOption<T> 大小 = T 大小

// ✅ 也享受（两个"空"变体 + 一个"有值"变体）：
enum Result<T, E> {
    Ok(T),
    Err(E),
}
// 如果 T 有 niche 且 E 没有 → Err 可能被放入 niche

// ❌ 不享受（两个"有值"变体）：
enum TwoValues<T, U> {
    First(T),
    Second(U),
}
// T 和 U 都有 niche——但编译器不知道该把 niche 给谁
```

### 3.3 repr(C) 禁用 Niche

```rust
// #[repr(C)] 禁用 niche 优化——因为 C 没有 niche 的概念
#[repr(C)]
enum CCompatible {
    A(i32),
    B,
}

// Option<CCompatible> 布局由 C ABI 决定——编译器不优化
// size 可能比 repr(Rust) 的版本更大
```

## 4. 常见陷阱

### 陷阱 1：假设类型有 Niche 但实际没有

```rust
// ❌ 错误假设：u8 有 niche
// u8 覆盖 0..=255——没有非法位模式
// Option<u8> 不享受 niche 优化（大小 = 2）

// ❌ 错误假设：f32 有 niche
// f32 的所有位模式都是"有效的"（包括 NaN 的所有编码）
// Option<f32> 不享受 niche 优化

// ❌ 错误假设：包含 NonZeroU8 的结构体自动享受 niche
struct Wrapper(NonZeroU8);
// Wrapper 是否有 niche？
// 可能！编译器可能把 NonZeroU8 的 niche 传播到 Wrapper
// 但这不是保证的——取决于编译器的实现
```

### 陷阱 2：依赖 Niche 做 transmute

```rust
// ❌ 不要假设 niche 的编码方式：
let opt: Option<&i32> = Some(&42);
// let bytes: [u8; 8] = unsafe { std::mem::transmute(opt) };
// ❌ bytes 的编码没有稳定保证！
// None 是全零——这只是当前实现的细节，不是保证
```

### 陷阱 3：Niche 在不同平台上的差异

```rust
// 引用在 64 位平台是 8 字节——niche 是全零的 8 字节
// 在 32 位平台是 4 字节——niche 是全零的 4 字节
// 在 WASM 上——引用可能有不同的表示
// → niche 的具体值取决于平台——不能跨平台假设
```

## 6. 练习答案

### 练习 1 答案

```
(a) T = &i32 → ✅ 享受 niche（全零指针是 niche）
     Option<&i32> 大小 = 8 字节 = &i32 大小

(b) T = u32 → ❌ 不享受 niche（所有 2^32 个值都有效）
     Option<u32> 大小 > u32 大小

(c) T = Box<String> → ✅ 享受 niche
     Box 内部是 NonNull——全零是 niche
     Option<Box<String>> 大小 = 8 字节

(d) T = NonZeroU8 → ✅ 享受 niche
     0 是 niche——Option<NonZeroU8> 大小 = 1 字节

(e) T = fn(i32) -> i32 → ✅ 享受 niche
     函数指针不能为空（全零的函数指针是无效的）

(f) T = Result<NonZeroU32, ()> → ✅ 可能享受 niche
     NonZeroU32 有 niche (0)
     Err(()) 可以放入 niche 中
     Option<Result<NonZeroU32, ()>> 大小可能 = 4 字节
```

### 练习 2 答案

```
Option<Option<Option<bool>>> 的状态空间：
- None                              (1 种)
- Some(None)                         (1 种)
- Some(Some(None))                   (1 种)
- Some(Some(Some(false))) (1 种)
- Some(Some(Some(true)))  (1 种)
总计：5 种状态

u8 有 256 个位模式：
- 256 - 5 = 251 个"空闲位模式"
- 编译器利用了其中非常多的 niche——1 字节就够
```

### 练习 3 答案

```rust
#[derive(Debug, PartialEq)]
struct EvenU32(u32);

impl EvenU32 {
    fn new(value: u32) -> Option<Self> {
        if value % 2 == 0 {
            Some(EvenU32(value))
        } else {
            None
        }
    }

    fn get(&self) -> u32 {
        self.0
    }
}

// EvenU32 的有效值：0, 2, 4, 6, ...（偶数）
// 奇数位模式是 niche → Option<EvenU32> 可以利用奇数表示 None

// 但是——编译器不会自动检测这个！
// EvenU32 包装了 u32——而 u32 没有 niche
// → Option<EvenU32> 不享受 niche 优化（在当前的 Rust 中）
// 要让编译器知道 niche，需要用 NonZeroU32 + 位移：
struct EvenU32Optimized(NonZeroU32);

impl EvenU32Optimized {
    fn new(value: u32) -> Option<Self> {
        NonZeroU32::new(value / 2).map(EvenU32Optimized)
    }
    fn get(&self) -> u32 { self.0.get() * 2 }
}
// 现在 Option<EvenU32Optimized> 大小 = 4（因为 NonZeroU32 有 niche）
```

### 练习 4 答案

```
不能直接使用 NonZeroU32 的 niche。

原因：
- NonZeroU32 的 niche 是 0——只有一个
- C API 返回 0 表示"成功"，1 表示"失败"，2 表示"超时"
- 但 NonZeroU32 的 0 niche 只能编码一个额外的状态

如何解决：
- 方案 A：用 enum CResult { Success(u32), Failure, Timeout }
  编译器自动处理 niche 优化
- 方案 B：用 NonZeroU32 + 自定义解读（如 0→Failure, 1→Timeout, 2..→Success(id-2)）
- 方案 C：不用 niche 优化——直接用 i32 / u32 加 match
  牺牲 4 字节的内存换取代码清晰度
```

## 7. 深度扩展

### 7.1 C++ 的 empty base optimization (EBO) vs niche——两种"白嫖"空间的方式

```cpp
// C++ 的空基类优化（EBO）：
class Empty {};  // sizeof = 1（C++ 保证）

class Derived : public Empty {
    int value;   // sizeof = 4
};
// sizeof(Derived) = 4（不是 8）——因为 Empty 基类不占空间
// 这就是 EBO——空基类的零字节占用

// C++20 [[no_unique_address]]：
struct Wrapper {
    [[no_unique_address]] Empty e;
    int value;
};
// sizeof(Wrapper) = 4（EBO 扩展到成员变量）

// C++ 的 niche-like 优化比较：
// - EBO/[[no_unique_address]] ≈ Rust 的 ZST 优化
// - std::optional 不能使用 niche——因为 C++ 不能保证"引用非空"
//   （但 GCC/Clang 对某些类型做了 ABI-breaking 的 niche 优化）

// C++ 不能像 Rust 那样广泛使用 niche 的原因：
// 1. 没有"引用必须非空"的语言保证
// 2. 没有 NonNull/NonZero 标准类型（直到 C++20 的部分）
// 3. ABI 稳定性顾虑——改变 niche 会破坏 ABI
// 4. 类型布局更多由 ABI 规范（Itanium ABI）决定——不是编译器自由发挥
```

### 7.2 如果没有 Niche 优化——Rust 的内存效率会下降多少

```rust
// 假设没有 niche 优化：

// 每个 Option<&T> 多 8 字节（共 16 字节）
// → 大量使用 Option<&T> 的代码内存翻倍

// Option<Box<T>> 多 8 字节
// Option<NonZeroU32> 多 4 字节
// Option<bool> 多 1 字节 → 内存翻倍

// 在典型的 Rust 程序中：
// - Option 在结果类型、迭代器、错误处理中无处不在
// - 没有 niche = 整个程序的内存占用可能增加 20-40%

// 这就是为什么 niche 优化不是"锦上添花"——它是 Rust "零成本抽象"承诺的基础
// C++ 的 std::optional 不能做类似优化——对某些类型来说，C++ 的"可选"比 Rust 贵
```

### 7.3 Niche 优化的实现——编译器的视角

```rust
// 编译器内部的 niche 检测流程：

// 1. 类型分析：T 有多少个"有效位模式"？
//    - 整数：2^N 个
//    - 引用：2^N - N（空引用被排除）
//    - NonZeroU32：2^32 - 1
//    - enum：已分配 discriminant 的数量

// 2. 如果有未使用的位模式 → 它们是 niche

// 3. 当包装成 Option<T> 时：
//    - 如果 niche >= 1 → Option<T> = T 的大小
//    - 用 1 个 niche 值表示 None
//    - 所有其他值表示 Some

// 4. 代码生成：
//    match opt {
//        None => ...,
//        Some(x) => ...,
//    }
//    → 编译器生成检查"niche 值"的代码（而不是检查判别位）
//    → 例如：if ptr == 0 { None } else { Some(ptr) }
//    → 和"检查判别位"一样快——甚至更快（因为 CPU 对零检查有优化）
```

### 7.4 niche 和 `#[rustc_layout_scalar_valid_range_start]` 属性

```rust
// 编译器内部属性（只用于标准库）：

// #[rustc_layout_scalar_valid_range_start(1)]
// #[rustc_layout_scalar_valid_range_end(MAX)]
// struct NonZeroU32(u32);

// 这些属性告诉编译器：
// - 有效值范围是 [1, MAX]（不包括 0）
// - → 0 是 niche

// 普通用户不能用这些属性——它们是不稳定的编译器细节
// 但理解它们有助于理解 niche 的底层机制
```

### 7.5 Niche 与 `Result<T, E>` 的布局

```rust
// Result<T, E> 也可以利用 niche：

use std::num::NonZeroU32;
use std::mem::size_of;

// 如果 Ok 的类型有 niche，Err 可以放入 niche：
type MyResult = Result<NonZeroU32, ()>;
assert_eq!(size_of::<MyResult>(), 4);  // Ok 的大小！

// 编译器会分析：
// 1. Ok(NonZeroU32) 的有效范围：1..=2^32-1
// 2. 0 是 niche
// 3. Err(()) 用 0 表示 ← 放入 niche

// 但反过来不行——如果 Err 有 niche：
type MyResult2 = Result<(), NonZeroU32>;
// Err 可能在 niche 中——但编译器优先考虑 Ok 的 niche
// 因为 Ok 是"正常路径"——优化更有价值

// ⚠️ 但 Result<T, E> 的 niche 优化是有限的
// 如果 T 和 E 都有值——niche 优化可能不适用
```

### 7.6 Niche 对 `Vec<Option<T>>` 的影响

```rust
// 假设存储 1,000,000 个 Option<u32>：
// 无 niche: 1M × 8 bytes = 8 MB
// 无 niche 的 Option<u32> 本身是 8 字节 (4 data + 4 padding)

// 如果有 niche（如改用 NonZeroU32+1）：
// 1M × 4 bytes = 4 MB
// 节省 50%！

// 这在大数据处理、游戏引擎、嵌入式系统中非常重要

// 实际案例：rustc 大量使用 Option<NonZeroU32> 表示"IDs"
// 因为编译器需要追踪数百万的 AST 节点——每字节都重要
```

### 7.7 Niche 对 `enum` 的布局影响

```rust
// 枚举的 niche 传播：

enum MyEnum {
    A(NonZeroU32),
    B,
    C,
}

// NonZeroU32 有 1 个 niche (0)
// MyEnum 有 3 个变体 → 需要 2 个 discriminant 位（至少）
// 但 NonZeroU32 的 niche 可以存储这些！→ size = 4

use std::mem::size_of;
assert_eq!(size_of::<MyEnum>(), 4);

// 编译器把 discriminant 编码进了 NonZeroU32 的 niche：
// 值 == 0 → 用于 B 或 C
// 值 != 0 → A(value)

// 这是 Rust 枚举比 C 枚举更紧凑的原因之一
// C 的 enum + union 总是需要显式的 tag 字段
```

### 7.8 Niche 的未来——更多的 niche 来源

```
未来可能的 niche 优化扩展：

1. 多个 niche 的更好利用：
   - 当前 Option<Option<bool>> 已经享受（多个 niche）
   - 未来可能有更智能的组合优化

2. 自定义 niche：
   - 让用户声明"我的类型有不用的位模式"
   - 类似于 #[rustc_layout_scalar_valid_range] 但稳定化

3. 浮点数的 niche：
   - f32/f64 的 NaN 有 2^23-1（或 2^52-1）种位模式
   - 理论上可以包含很多 niche
   - 但 NaN 有符号——处理很复杂——目前没有优化

4. B-tree / niche 的组合：
   - 将多个 niche 合并以编码更多状态

5. WASM 上的特殊布局：
   - WASM 的引用有不同的表示——可能有独特的 niche 机会
```

## 8. 回顾

```
Niche 优化学习清单：

□ 理解 Niche 概念：类型中不可用的位模式——被编译器用来编码额外状态
□ 知道标准 Niche 类型：引用、NonNull、NonZero*、bool、char、函数指针
□ 理解 Option<T> 享受 Niche 的条件：T 有至少 1 个不可用的位模式
□ 知道没有 Niche 的类型：u8/u16/u32、f32/f64（所有位模式都有效）
□ 理解 repr(C) 禁用 Niche 的原因（C ABI 兼容）
□ 知道 Niche 对内存效率的巨大影响（Option<Box<T>> 和 Box<T> 一样大）
□ 对比 C++ EBO / [[no_unique_address]] —— C++ 无法做等价的 niche 优化
□ 知道 Result<T, E> 也能利用 niche（但优先优化 Ok 变体）
```

> **一句话总结**：Niche 优化让 `Option<&T>` 和 `&T` 一样大——编译器利用"全零指针不是合法引用"这一事实，用零值表示 `None`。`NonZero*`、`NonNull<T>`、`bool`、`char`、函数指针都有 niche，`Box<T>`（内部用 `NonNull`）也因此受益。C++ 的 `std::optional<T>` 没法做等价优化——因为 C++ 不保证引用非空、也没有语言层面的 Niche 概念。这是 Rust "零成本抽象"的又一块关键拼图。
