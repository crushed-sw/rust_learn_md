# 2.9 Zero-Sized Types (ZST)——编译时存在的幽灵类型

> 你写了一个 `struct Config`，里面有一个 `_marker: PhantomData<MyState>` 字段。在运行时，这个字段**不占用任何内存空间**——它的大小是 0 字节。但你发现没有它，整个类型系统的状态建模就会崩溃。这是 Rust 的 ZST（Zero-Sized Types）：`()`、空结构体、空枚举、`PhantomData<T>`、`[T; 0]`——它们存在于编译时，在类型层面编码信息，但到了运行时，所有的操作都被编译为"什么都不做"。它们让 Rust 在不付出任何运行时代价的情况下拥有了强大的类型级抽象能力——这是"零成本抽象"最极致的体现。

---

## 1. 什么是 ZST——零大小类型的分类

### 1.1 ZST 的家族

```rust
use std::mem;
use std::marker::PhantomData;

// ZST 类型 1：单元类型 ()
assert_eq!(mem::size_of::<()>(), 0);
assert_eq!(mem::align_of::<()>(), 1);

// ZST 类型 2：空结构体
struct Empty;
assert_eq!(mem::size_of::<Empty>(), 0);

struct EmptyBraces {}
assert_eq!(mem::size_of::<EmptyBraces>(), 0);

// ZST 类型 3：空枚举（uninhabited type——不可构造）
enum Void {}
assert_eq!(mem::size_of::<Void>(), 0);
// ⚠️ Void 永远不能被实例化——没有任何变体

// ZST 类型 4：PhantomData<T>
struct MyPhantom {
    _marker: PhantomData<i32>,
}
assert_eq!(mem::size_of::<PhantomData<i32>>(), 0);
assert_eq!(mem::size_of::<MyPhantom>(), 0);  // 只包含 ZST 的结构体也是 ZST

// ZST 类型 5：零长度数组
assert_eq!(mem::size_of::<[u8; 0]>(), 0);
assert_eq!(mem::size_of::<[String; 0]>(), 0);  // 即使是非 ZST 元素的零长度数组

// ZST 类型 6：只有 ZST 字段的结构体
struct MultiZst {
    empty: Empty,
    unit: (),
    bits: [u8; 0],
}
assert_eq!(mem::size_of::<MultiZst>(), 0);

// 验证——这些类型真的不占空间：
println!("Size of ():            {}", mem::size_of::<()>());
println!("Size of PhantomData:    {}", mem::size_of::<PhantomData<String>>());
println!("Size of [i32; 0]:       {}", mem::size_of::<[i32; 0]>());
// 所有输出：0
```

### 1.2 ZST 的对齐

```rust
use std::mem;

// 大多数 ZST 的对齐是 1（最小可能的对齐）
assert_eq!(mem::align_of::<()>(), 1);
assert_eq!(mem::align_of::<PhantomData<i32>>(), 1);

// 但这并不意味着 ZST 总是对齐为 1
// 如果 ZST 在包含更大对齐字段的结构体中，编译器可能选择不同

// #[repr(align(N))] 可以手动控制 ZST 的对齐
#[repr(align(64))]
struct CacheLineAligned;  // 缓存行对齐的空结构体

assert_eq!(mem::size_of::<CacheLineAligned>(), 0);
assert_eq!(mem::align_of::<CacheLineAligned>(), 64);

// 这有什么用？
// 在无锁数据结构中，你需要在不同槽位之间加 padding 来避免 false sharing
// ZST 配合 align 可以达到零成本 padding 的效果
```

### 1.3 ZST 与非 ZST 混合时的布局

```rust
use std::mem;

#[derive(Debug)]
struct WithZst {
    data: u64,        // 8 字节
    marker: (),       // 0 字节——但不影响 data 的位置
    tag: PhantomData<String>,  // 0 字节
}

// ZST 字段不占用任何空间，不影响其他字段的偏移
assert_eq!(mem::size_of::<WithZst>(), 8);  // 只有 u64

struct ManyZsts {
    a: (),
    b: PhantomData<i32>,
    c: PhantomData<String>,
    d: [u8; 0],
}
assert_eq!(mem::size_of::<ManyZsts>(), 0);  // 全是 ZST——整体也是 ZST

// 但是 ZST 可以影响结构体大小——通过对齐
#[repr(C)]
struct WithAlignedZst {
    zst: CacheLineAligned,  // align=64 的 ZST
    data: u64,              // 8 字节
}
// #[repr(C)] 下，整个结构体的对齐 = max(zst.align, data.align) = 64
// size = 64（data 占 8 字节，但在 64 对齐下需要填充到 64）
// 实际上这会变成至少 64 字节，因为有对齐
```

## 2. ZST 的运行时行为——一切都被优化掉了

### 2.1 ZST 的操作编译为零指令

```rust
// 创建一个 () 值：
let unit: () = ();
// 编译后：没有 CPU 指令。() 不需要任何寄存器或内存来表示。

// 传递一个 ZST 参数：
fn take_zst(x: PhantomData<i32>) {}
take_zst(PhantomData);
// 编译后：函数签名被单态化——调用时没有 push/lea，函数体内没有使用 x

// 返回一个 ZST：
fn return_zst() -> [u8; 0] {
    []
}
// 编译后：函数"返回"零字节——返回值优化为 nothing

// 在栈上"存储"一个 ZST：
let zst_array: [PhantomData<i32>; 1000] = [PhantomData; 1000];
// 编译后：栈上不分配任何空间。
// 注意：尽管数组长度是 1000，但每个元素大小是 0——总大小 = 0
assert_eq!(mem::size_of_val(&zst_array), 0);
```

### 2.2 Vec<ZST>——只有长度，没有数据

```rust
use std::mem;

// Vec<u8> 在堆上分配内存存储元素
let mut bytes: Vec<u8> = Vec::with_capacity(10);
// 分配了 10 字节的堆内存

// Vec<()> 不分配任何堆内存！
let mut units: Vec<()> = Vec::new();
units.push(());
units.push(());
units.push(());

assert_eq!(units.len(), 3);       // 长度跟踪仍然有效
assert_eq!(units.capacity(), 4);  // capacity 可能 > 0

// 但是堆上没有分配任何内存！Vec 的指针是 dangling（悬垂）的
// Vec 内部表示：
// ptr: NonNull<()> —— dangling 指针（对齐为 1 的悬垂指针）
// len: 3
// cap: 4

// 用 push 不需要分配：
units.push(());  // 只增加 len，不做堆操作

// 用 pop：
assert_eq!(units.pop(), Some(()));  // 只减少 len

// 遍历——循环执行但 body 可能什么都不做：
for _ in &units {
    // 循环迭代次数 = len
    // 但每次迭代不需要读取任何数据
}
// LLVM 可能将整个 for 循环优化为 nothing

// 底层实现（Vec 源码中的概念）：
// impl<T> Vec<T> {
//     fn push(&mut self, value: T) {
//         if self.len == self.cap { self.grow(); }
//         unsafe {
//             // 对于 ZST：ptr::write 是 no-op
//             ptr::write(self.ptr.as_ptr().add(self.len), value);
//         }
//         self.len += 1;
//     }
// }
// grow() 对于 ZST 也被特殊处理——实际上不需要任何分配
```

### 2.3 ZST 与迭代器的魔法

```rust
// Range<usize> 不是 ZST（大小通常是 16 字节）
use std::mem;
assert_eq!(mem::size_of::<std::ops::Range<usize>>(), 16);

// 但是 Range<()> 不是合法类型——但相关类型的迭代器可能是 ZST
// 例如：std::iter::Empty<T> 是 ZST！

let empty: std::iter::Empty<i32> = std::iter::empty();
assert_eq!(mem::size_of_val(&empty), 0);

// 标准库中的 ZST 迭代器：
// std::iter::Empty<T>     —— 永远不产出元素的迭代器
// std::iter::Repeat<T>    —— 不是 ZST（可能包含要重复的值，或者通过引用）
// std::ops::RangeFull (..) —— ZST！代表整个范围

assert_eq!(mem::size_of::<std::ops::RangeFull>(), 0);
assert_eq!(mem::size_of::<std::ops::RangeTo<usize>>(), 0);
// RangeToInclusive 也不是 ZST
assert_eq!(mem::size_of::<std::ops::RangeFrom<usize>>(), 0);

// 当迭代器是 ZST 时，for 循环可以被大量优化
let v = vec![1, 2, 3];
for i in 0..v.len() {
    // 0..v.len() 是 Range<usize>——不是 ZST，占 16 字节
    // 但 LLVM 通常能完全展开这个循环——不依赖 Range 的大小
}

// 用 RangeFrom 做无限迭代：
let from_zero = 0..;  // RangeFrom<i32>——ZST
for i in from_zero.take(5) {
    // RangeFrom 本身是 ZST——但 take(5) 产生的 Take<RangeFrom<i32>> 不是 ZST
    // 因为需要存储计数器
}
```

## 3. PhantomData——类型级标记，零运行时开销

### 3.1 PhantomData 的基本用途

```rust
use std::marker::PhantomData;

// 场景 1：标记"拥有"某类型——即使不直接持有它
struct MyBuffer<T> {
    ptr: *const u8,         // 裸指针，不表达所有权
    len: usize,
    _owner: PhantomData<T>, // 告诉编译器：MyBuffer<T> 拥有 T 类型的数据
    // PhantomData<T> 的大小是 0——不占空间
    // 但 Drop checker 知道 MyBuffer<T> 可能负责释放 T
}

// 场景 2：标记生命周期关系
struct BorrowPair<'a, 'b, T: 'a> {
    first: &'a T,
    _second_marker: PhantomData<&'b T>,  // 表明逻辑上也与 'b 有关
}

// 场景 3：实现未使用的类型参数
struct MyHashMap<K, V> {
    buckets: Vec<Bucket<V>>,  // 键 K 没有出现在任何字段中
    _key_type: PhantomData<K>,  // 让类型参数 K 有用
}
// 没有 PhantomData<K>：
// struct MyHashMap<K, V> { buckets: Vec<Bucket<V>> }
// 编译器警告：type parameter K is never used

// 场景 4：模拟所有权/借用语义
struct Owned<T> {
    _marker: PhantomData<T>,  // 协变
}

// PhantomData 与变性的关系在 2.6 章节已深度讨论
```

### 3.2 PhantomData 的更多模式

```rust
use std::marker::PhantomData;
use std::ptr::NonNull;

// 通过 PhantomData 模拟各种所有权关系：

// PhantomData<fn() -> T> —— 逆变（函数参数位置）
struct ContraVariant<T> {
    _marker: PhantomData<fn() -> T>,
}

// PhantomData<fn(T)> —— 协变（函数返回位置，等同于 PhantomData<T>）
struct CoVariant<T> {
    _marker: PhantomData<fn(T)>,
}

// PhantomData<*const T> —— 不变（裸指针位置）
struct InVariant<T> {
    _marker: PhantomData<*const T>,
}

// 实用例子：RawVec 的核心结构
struct RawVec<T> {
    ptr: NonNull<T>,             // 指向堆内存
    cap: usize,                  // 容量
    _marker: PhantomData<T>,     // 表明拥有 T，且与 T 协变
}
// 没有 _marker 的话，T 只出现在 NonNull<T> 中
// NonNull<T> 对 T 是协变的——和不加 PhantomData 一样
// 但 PhantomData<T> 确保了意图明确，且 Drop checker 正确工作
```

### 3.3 使用多个 PhantomData 字段

```rust
use std::marker::PhantomData;

// 一个结构体可以有多个 PhantomData，编码多种关系：

struct ComplexRelation<'a, K, V>
where
    K: 'a,
    V: 'a,
{
    // 数据指针
    ptr: *const u8,
    
    // K 被借用（有生命周期 'a）
    _borrow_k: PhantomData<&'a K>,
    
    // V 被拥有
    _own_v: PhantomData<V>,
    
    // 大小 = 8 字节（只有 ptr）
}

// 编译器检查：
// - ComplexRelation<String, Vec<u8>> 假设它拥有 Vec<u8>
// - 如果包含引用 'a，则受限于生命周期 'a
// 所有这些约束都在类型系统中，运行时零开销

// 你甚至可以用 PhantomData 强制非 Send / 非 Sync：
struct NotSend {
    _marker: PhantomData<*const ()>,  // *const T 不是 Send
    // 或者更明确：
    // _marker: PhantomData<std::rc::Rc<()>>,  // Rc 不是 Send 或 Sync
}

// 用 PhantomData<Cell<()>> 同时标记非 Send 和非 Sync：
struct NotSendNotSync {
    _marker: PhantomData<std::cell::Cell<()>>,
}
```

## 4. ZST 作为类型级状态——Typestate 模式

### 4.1 什么是 Typestate

```rust
use std::marker::PhantomData;
use std::mem;

// Typestate 模式：用类型参数编码对象的状态
// 使得非法状态转换在编译时被拒绝

// 状态标签——都是 ZST
struct Locked;
struct Unlocked;

// 一个简单的 Mutex（概念演示）
struct Mutex<T, State = Unlocked> {
    data: T,
    _state: PhantomData<State>,
}

// 构造函数——总是从 Unlocked 开始
impl<T> Mutex<T, Unlocked> {
    fn new(data: T) -> Self {
        Mutex {
            data,
            _state: PhantomData,
        }
    }
    
    // 只有 Unlocked 状态才能 lock
    fn lock(self) -> Mutex<T, Locked> {
        Mutex {
            data: self.data,
            _state: PhantomData,
        }
    }
}

// Locked 状态的方法
impl<T> Mutex<T, Locked> {
    // 只有 Locked 状态才能访问数据
    fn get(&self) -> &T {
        &self.data
    }
    
    fn get_mut(&mut self) -> &mut T {
        &mut self.data
    }
    
    fn unlock(self) -> Mutex<T, Unlocked> {
        Mutex {
            data: self.data,
            _state: PhantomData,
        }
    }
}

// 使用——编译时保证安全性：
fn main() {
    let m = Mutex::new(42);  // Mutex<i32, Unlocked>
    
    // m.get();  // ❌ 编译错误——Unlocked 没有 get 方法
    
    let locked = m.lock();   // Mutex<i32, Locked>
    println!("{}", locked.get());  // ✅ 合法
    
    let unlocked = locked.unlock();  // Mutex<i32, Unlocked>
    // unlocked 可以再次 lock...
    
    // 完全没有运行时开销——PhantomData<Locked> 和 PhantomData<Unlocked>
    // 在编译后都完全消失
    assert_eq!(mem::size_of::<Mutex<i32, Locked>>(), 4);   // 只有 i32
    assert_eq!(mem::size_of::<Mutex<i32, Unlocked>>(), 4); // 只有 i32
}
```

### 4.2 Typestate 在 Builder 模式中的应用

```rust
use std::marker::PhantomData;

// Builder 模式——在编译时检查必填字段

// 状态标签
struct NoHost;
struct NoPort;
struct NoPath;
struct Ready;

struct Host(String);
struct Port(u16);
struct Path(String);

// Builder——泛型参数编码"已设置了什么"
struct RequestBuilder<H = NoHost, P = NoPort, Ph = NoPath> {
    host: Option<Host>,
    port: Option<Port>,
    path: Option<Path>,
    _h: PhantomData<H>,
    _p: PhantomData<P>,
    _ph: PhantomData<Ph>,
}

impl RequestBuilder {
    fn new() -> RequestBuilder<NoHost, NoPort, NoPath> {
        RequestBuilder {
            host: None,
            port: None,
            path: None,
            _h: PhantomData,
            _p: PhantomData,
            _ph: PhantomData,
        }
    }
}

// 在任何状态下都可以设置字段：
impl<H, P, Ph> RequestBuilder<H, P, Ph> {
    fn host(mut self, h: &str) -> RequestBuilder<Host, P, Ph> {
        self.host = Some(Host(h.to_string()));
        RequestBuilder {
            host: self.host,
            port: self.port,
            path: self.path,
            _h: PhantomData,
            _p: PhantomData,
            _ph: PhantomData,
        }
    }
    
    fn port(mut self, p: u16) -> RequestBuilder<H, Port, Ph> {
        self.port = Some(Port(p));
        RequestBuilder {
            host: self.host,
            port: self.port,
            path: self.path,
            _h: PhantomData,
            _p: PhantomData,
            _ph: PhantomData,
        }
    }
    
    fn path(mut self, ph: &str) -> RequestBuilder<H, P, Path> {
        self.path = Some(Path(ph.to_string()));
        RequestBuilder {
            host: self.host,
            port: self.port,
            path: self.path,
            _h: PhantomData,
            _p: PhantomData,
            _ph: PhantomData,
        }
    }
}

// build 只能在这一组具体类型上调用：
impl RequestBuilder<Host, Port, Path> {
    fn build(self) -> Request {
        Request {
            host: self.host.unwrap(),
            port: self.port.unwrap(),
            path: self.path.unwrap(),
        }
    }
}

struct Request {
    host: Host,
    port: Port,
    path: Path,
}

fn main() {
    // ✅ 正确顺序：
    let req = RequestBuilder::new()
        .host("example.com")
        .port(443)
        .path("/api/v1")
        .build();
    
    // ❌ 忘了 port——编译错误：
    // let req = RequestBuilder::new()
    //     .host("example.com")
    //     .path("/api/v1")
    //     .build();  // error: method `build` not found
    
    // ❌ 忘了 host——编译错误：
    // let req = RequestBuilder::new()
    //     .port(443)
    //     .path("/api/v1")
    //     .build();  // error: method `build` not found
    
    // 这是"使非法状态不可表示"的完美例子
    // 所有 PhantomData 字段——编译后完全消失
}
```

### 4.3 State Machine 与 ZST

```rust
use std::marker::PhantomData;

// 完整的 TCP 连接状态机（简化）

// 状态
struct Closed;
struct Listen;
struct SynSent;
struct Established;
struct FinWait1;
struct FinWait2;
struct TimeWait;

// TCP 连接——状态在类型参数中
struct TcpConnection<State = Closed> {
    _state: PhantomData<State>,
    socket_fd: i32,  // 实际的 socket 文件描述符
}

impl TcpConnection<Closed> {
    fn new() -> Self {
        // socket() 系统调用
        let fd = 0; // 模拟
        TcpConnection {
            _state: PhantomData,
            socket_fd: fd,
        }
    }
    
    fn bind(self, _port: u16) -> Self {
        self  // 状态不变，还是 Closed
    }
    
    fn listen(self) -> TcpConnection<Listen> {
        println!("Listening...");
        TcpConnection {
            _state: PhantomData,
            socket_fd: self.socket_fd,
        }
    }
}

impl TcpConnection<Listen> {
    fn accept(self) -> (TcpConnection<Established>, TcpConnection<Listen>) {
        println!("Accepted connection");
        let new_fd = 1; // accept() 返回的新 fd
        (
            TcpConnection::<Established> {
                _state: PhantomData,
                socket_fd: new_fd,
            },
            self,  // 监听 socket 继续监听
        )
    }
}

impl TcpConnection<Established> {
    fn read(&self, buf: &mut [u8]) -> usize {
        // read() 系统调用
        println!("Reading from established connection");
        0
    }
    
    fn write(&self, data: &[u8]) -> usize {
        // write() 系统调用
        println!("Writing to established connection");
        data.len()
    }
    
    fn close(self) -> TcpConnection<FinWait1> {
        println!("Closing...");
        TcpConnection {
            _state: PhantomData,
            socket_fd: self.socket_fd,
        }
    }
}

impl TcpConnection<FinWait1> {
    fn fin_ack(self) -> TcpConnection<FinWait2> {
        TcpConnection {
            _state: PhantomData,
            socket_fd: self.socket_fd,
        }
    }
}

impl TcpConnection<FinWait2> {
    fn fin(self) -> TcpConnection<TimeWait> {
        TcpConnection {
            _state: PhantomData,
            socket_fd: self.socket_fd,
        }
    }
}

impl TcpConnection<TimeWait> {
    fn wait_timeout(self) -> TcpConnection<Closed> {
        TcpConnection {
            _state: PhantomData,
            socket_fd: self.socket_fd,
        }
    }
}

fn main() {
    let conn = TcpConnection::new();
    let listening = conn.bind(8080).listen();
    let (established, _listener) = listening.accept();
    
    let mut buf = [0u8; 1024];
    established.read(&mut buf);
    established.write(b"HTTP/1.1 200 OK\r\n");
    
    let closing = established.close();
    let fin2 = closing.fin_ack();
    let time_wait = fin2.fin();
    let _closed = time_wait.wait_timeout();
    
    // 在编译时保证：
    // - Closed 状态不能 read/write
    // - Established 状态不能 listen
    // - 所有非法转换被阻止
    // - 完全零运行时开销
}
```

## 5. 空枚举——Uninhabited Type（不可构造类型）

### 5.1 空枚举 vs 单元类型

```rust
// () 是"恰好一个值的类型"——可以构造、传递、匹配
let unit: () = ();
// 它是 inhabited type（可居住类型）

// 空枚举是"零个值的类型"——永远无法构造
enum Void {}
// 它是 uninhabited type（不可居住类型）

// 关键区别：
// - () 可以构造——() 是一个值
// - Void 不能构造——没有任何变体

// 为什么需要 Void？
// 1. 表达"永远不会发生"的错误类型
// 2. 在泛型代码中"删除"某个分支
// 3. 与 Result<T, Void> = 不可能出错的 Result（但实际上推荐用 Infallible）
```

### 5.2 Void 的实际应用

```rust
use std::convert::Infallible;

// Infallible 本质上就是空枚举：
// pub enum Infallible {}

// 场景 1：永远不失败的转换
fn parse_always_succeeds(s: &str) -> Result<i32, Infallible> {
    // 这个函数保证永远不会返回 Err
    Ok(s.parse().unwrap_or(0))
}

// 调用者知道不需要处理 Err：
let result: Result<i32, Infallible> = parse_always_succeeds("42");
let value = match result {
    Ok(v) => v,
    Err(e) => match e {},  // 这行永远不会执行
    // match e {} 是"空匹配"——因为 e: Infallible 没有任何可能的值
    // 编译器知道这段代码不可达——可以删除它
};

// 甚至更简单——unwrap 对 Result<T, Infallible> 总是成功：
// 实际上 unwrap 不需要 panic——编译器保证不会走 Err 分支

// 场景 2：类型级别的"永不"
enum Never {}  // 等价于 Infallible 和 !

fn never_returns() -> Never {
    loop {
        // 无限循环——never 返回
        std::thread::sleep(std::time::Duration::from_secs(1));
    }
}
// Never 类型保证这个函数"永远不会正常返回"
// （目前稳定版用 Infallible 或 ! ——never type 还未完全稳定）

// 场景 3：消除泛型的某个特化
trait Handler<T> {
    type Error;
    fn handle(&self, input: T) -> Result<(), Self::Error>;
}

struct NoErrorHandler;

impl Handler<String> for NoErrorHandler {
    type Error = Infallible;  // 这个 handler 永远不会出错
    
    fn handle(&self, _: String) -> Result<(), Infallible> {
        Ok(())
    }
}

// 场景 4：在类型系统中"证明"某个分支不存在
enum Either<L, R> {
    Left(L),
    Right(R),
}

// Either<T, Void> 总是 Left——Right 永远构造不出来
type AlwaysLeft<T> = Either<T, Infallible>;
// 编译器可以利用这一点删除 Right 分支的所有代码
```

### 5.3 空枚举的"Niche"优化

```rust
use std::mem;

enum Void {}

// Option<Void> 只有一个可能的值：None
// Some(v) 永远无法构造——因为 v: Void 无法构造
assert_eq!(mem::size_of::<Option<Void>>(), 0);

// Result<Void, E> 总是 Err——Ok 分支不可能
assert_eq!(mem::size_of::<Result<Void, i32>>(), mem::size_of::<i32>());

// Result<T, Void> 总是 Ok——Err 分支不可能
assert_eq!(mem::size_of::<Result<i32, Void>>(), mem::size_of::<i32>());

// 这展示了 Rust 的"Niche Optimization"（niche = 壁龛/利基）：
// 编译器利用"这个值不可能存在"的事实来优化内存布局
// 
// 具体来说：
// - Option<&T>：因为 &T 不能是 null（NonNull），null 可以作为 None 的 niche
//   sizeof(Option<&T>) = sizeof(&T) = 8 字节（64 位）
//
// - Option<bool>：bool 只有 0 和 1，2-255 都是 niche
//   sizeof(Option<bool>) = 1 字节（比预期的 2 字节小）
//
// - Option<Infallible>：Infallible 没有任何值，None 是唯一可能
//   sizeof(Option<Infallible>) = 0 字节！

// 完整的 niche 优化列表：
assert_eq!(mem::size_of::<Option<&i32>>(), 8);       // null = None
assert_eq!(mem::size_of::<Option<bool>>(), 1);        // 2-255 = None
assert_eq!(mem::size_of::<Option<std::cmp::Ordering>>(), 1);  // same as bool
assert_eq!(mem::size_of::<Option<std::num::NonZeroU8>>(), 1);  // 0 = None
assert_eq!(mem::size_of::<Option<Infallible>>(), 0);  // only None possible
```

## 6. ZST 在 FFI 中的行为

### 6.1 `#[repr(C)]` ZST 在 C 中也是零大小

```rust
// Rust 的 ZST 在 #[repr(C)] 下，C 编译器也看到零大小

#[repr(C)]
struct RustZst {
    // 空结构体
}

// 等价 C 代码：
// struct RustZst {
//     // 空的
// };
// sizeof(RustZst) 在 C 中是 0（GCC/Clang 扩展）或 1（标准 C 中空 struct 至少 1 字节）
//
// ⚠️ 重要警告：C 标准规定空 struct 大小至少为 1（以便有地址）
// 但 GCC 和 Clang 在 GNU 扩展模式下允许大小为 0
// Rust 的 FFI 与 GCC/Clang 的 GNU 行为一致——大小为 0

use std::mem;

#[repr(C)]
struct Empty1;
assert_eq!(mem::size_of::<Empty1>(), 0);

#[repr(C)]
struct Empty2 {}
assert_eq!(mem::size_of::<Empty2>(), 0);

// 如果 C 编译器给空 struct 分配 1 字节（严格标准模式），
// Rust 的 0 字节布局可能与 C 不兼容——这是已知的 FFI 边界问题
// 实践中，大多数 C 编译器在 Rust 使用的平台上与 Rust 一致（0 字节）
```

### 6.2 包含 ZST 的联合体和结构体在 FFI

```rust
// 当结构体包含 ZST 时，FFI 布局可能出乎意料

#[repr(C)]
struct HasZst {
    data: u32,
    marker: (),  // ZST
}
// sizeof = 4（marker 不占空间）
// 等价 C struct：struct { uint32_t data; }

#[repr(C)]
struct OnlyZsts {
    a: (),
    b: (),
}
// sizeof = 0
// 在 C 中取决于编译器和标准模式

// FFI 中的经验法则：
// 1. 不要将包含 ZST 的结构体传给 C（除非 ZST 是 PhantomData）
// 2. 直接传字段，不要依赖 ZST 不影响布局的假设
// 3. 用 #[repr(C)] 但没有 ZST 的结构体总是安全的

// ✅ 安全的 FFI 结构：
#[repr(C)]
struct SafeFfiStruct {
    data: u32,
    // 没有 PhantomData 或其他 ZST——或至少知道它们不影响 FFI
    _marker: std::marker::PhantomData<()>,  // PhantomData 总是 0 大小
}
// 但还是建议在 FFI 中避免 PhantomData——保持简单
```

### 6.3 C 中的 Zero-Sized 模拟——EBO 和 no_unique_address

```cpp
// C++ 的"接近 ZST"方案

// 1. Empty Base Optimization (EBO)
struct Empty {};

struct Foo : Empty {
    int x;
};
// sizeof(Foo) == sizeof(int) = 4
// 基类 Empty 不占空间——这就是 EBO

// 但在成员位置不行：
struct Bar {
    Empty e;   // 即使是空的，也占 1 字节（标准要求）
    int x;
};
// sizeof(Bar) == 8（1 + padding(3) + 4）

// 解决：[[no_unique_address]] (C++20)
struct Baz {
    [[no_unique_address]] Empty e;  // 不占空间！
    int x;
};
// sizeof(Baz) == 4（与 Foo 相同）

// Rust 的 ZST 总是等价于 [[no_unique_address]]
// 不需要特殊注解——这是 Rust 布局规则的默认行为
// 这使得 Rust 的零成本抽象比 C++ 更自然和普遍
```

## 7. Compiler Optimizations——编译器如何消除 ZST

### 7.1 MIR 层面的 ZST 优化

```rust
// 当你写这样的代码：
fn process_zst() {
    let zst: () = ();
    let another: PhantomData<i32> = PhantomData;
    let arr: [(); 10] = [(); 10];
    
    for _ in arr.iter() {
        do_something_with_zst(PhantomData::<i32>);
    }
}

fn do_something_with_zst<T>(_x: PhantomData<T>) {}

// 在 MIR（Mid-level IR）层面：
// - let zst: () = ();  → MIR 中的 place 的 type 是 ()
//   → 在 codegen 时，没有对应的 LLVM alloca 指令
// - PhantomData::<i32> → 这是 ZST 构造——在 MIR 中是一个 constant
//   → 不需要运行时计算
// - arr: [(); 10] → 类型系统知道大小 = 0
//   → 在 LLVM IR 中，arr 不分配栈空间
// - for _ in arr.iter() → 迭代器中只有 len 被使用
//   → LLVM 展开或删除循环（取决于上下文）

// 关键的 MIR 优化 pass：
// 1. ConstProp：传播 ZST 常量
// 2. DeadStoreElimination：删除对 ZST 的写入
// 3. 类型级别的 size_of::<ZST>() = 0 → 影响所有布局计算
```

### 7.2 LLVM IR 中的 ZST

```rust
// 以下 Rust 代码对应的 LLVM IR：

fn return_zst() -> PhantomData<i32> {
    PhantomData
}

// 编译后的 LLVM IR（简化，release 模式）：
// define void @return_zst() {
//     ret void
// }
// 注意：返回类型是 void——因为 PhantomData<i32> 大小是 0
// 函数"返回"零字节——LLVM 将其映射为 void

// --------------------

fn take_zst(x: PhantomData<i32>) -> i32 {
    42
}

// LLVM IR：
// define i32 @take_zst() {
//     ret i32 42
// }
// 注意：参数 x 被完全删除了！调用方也不需要传任何东西

// --------------------

struct WithZst {
    data: i64,
    marker: (),
}

fn read_zst(s: &WithZst) -> () {
    s.marker
}

// LLVM IR：
// define void @read_zst(%WithZst* %s) {
//     ret void
// }
// 对 s.marker 的读取被优化为 nothing

// --------------------

// 对比——非 ZST 版本：
struct NoZst {
    data: i64,
    marker: u8,  // 1 字节
}

fn read_no_zst(s: &NoZst) -> u8 {
    s.marker
}

// LLVM IR：
// define i8 @read_no_zst(%NoZst* %s) {
//     %marker_ptr = getelementptr %NoZst, %NoZst* %s, i32 0, i32 1
//     %val = load i8, i8* %marker_ptr
//     ret i8 %val
// }
// 有实际的指针计算和 load 指令！
```

### 7.3 ZST 与函数单态化

```rust
use std::marker::PhantomData;
use std::mem;

// 泛型函数——包含 ZST 参数：

fn generic_with_zst<T>(data: i32, _tag: PhantomData<T>) -> i32 {
    // _tag 没人用——编译器完全删除
    data * 2
}

// 单态化为：
// generic_with_zst::<String> → 函数被生成，但 PhantomData 参数没影响
// generic_with_zst::<Vec<u8>> → 另一个副本
// 两个副本的机器码完全相同！
// LLVM 会合并它们（identical code folding）

// 验证——这两个调用生成相同的机器码：
fn call_with_different_zst() {
    let a = generic_with_zst(21, PhantomData::<String>);
    let b = generic_with_zst(21, PhantomData::<Vec<u8>>);
    // LLVM 可能将两者合并为同一个函数
    assert_eq!(a, 42);
    assert_eq!(b, 42);
}

// 但如果泛型参数影响实际行为（如 size_of），则不能合并：
fn generic_layout_dependent<T>() -> usize {
    mem::size_of::<T>()  // 非 ZST 的类型参数实际上影响返回值
}

fn call_different() {
    let a = generic_layout_dependent::<u32>();  // 返回 4
    let b = generic_layout_dependent::<u64>();  // 返回 8
    // 这两个调用的机器码不同——不能合并
}

// ZST 的优雅之处：泛型参数的类型信息用于编译时决策，
// 但不产生运行时开销
```

### 7.4 ZST 与汇编——亲眼看看

```rust
// 用 cargo-asm 或 godbolt.org 查看以下代码的汇编：

use std::marker::PhantomData;

pub struct MyVec<T> {
    inner: Vec<T>,
    _marker: PhantomData<T>,
}

pub fn push_zst(v: &mut MyVec<()>) {
    v.inner.push(());
}

pub fn push_u32(v: &mut MyVec<u32>, val: u32) {
    v.inner.push(val);
}

// x86_64 汇编对比（简化）：

// push_zst:
//     mov rax, qword ptr [rdi + 8]    ; 加载 len
//     cmp rax, qword ptr [rdi + 16]   ; 比较 len vs cap
//     je  .grow                        ; 如果需要扩容
//     inc qword ptr [rdi + 8]         ; len += 1
//     ret
// .grow:
//     ; ... 扩容对于 ZST 是特殊的——不需要分配

// push_u32:
//     mov rax, qword ptr [rdi + 8]    ; 加载 len
//     cmp rax, qword ptr [rdi + 16]   ; 比较 len vs cap
//     je  .grow_u32
//     mov rcx, qword ptr [rdi]        ; 加载 ptr
//     mov dword ptr [rcx + 4*rax], esi ; 存储值——ZST 版本没有这条！
//     inc qword ptr [rdi + 8]         ; len += 1
//     ret
// .grow_u32:
//     ; ... 实际分配内存并复制

// 关键区别：push_zst 没有"写入内存"这一步——没有 memcpy/mov
// 因为 () 不需要存储任何数据
```

## 8. 常见陷阱

### 陷阱 1：认为 `()` 是"没有值"——其实是"只有一个值"

```rust
// () 有一个值：()
// 这不是"null"、"undefined"、"nil"——这是"恰好一个可能的值"

let unit: () = ();
// 你总能构造一个 ()，就像你能构造 true 一样
// 只是 () 的"信息量"更少

// 这意味着：
// 1. 对 () 的模式匹配总是成功
match () {
    () => println!("always matches"),
}
// 2. let () = expr; 可以用于检查类型
let _: () = {
    println!("side effect");
    // 返回 () —— 隐式的
};
// 3. 函数如果没有指定返回值 → 返回 ()

// 常见误解："ZST 是没有值" → 错误
// 正确：ZST 是"大小为零的值" → ()、空结构体、PhantomData 都有值
// 只有空枚举（Infallible/Void）才是"没有值"的
```

### 陷阱 2：PhantomData 不是零成本——它影响 Drop Check

```rust
use std::marker::PhantomData;

// PhantomData<T> 是零大小——但它在编译器内部不是"免费的"
// Drop checker 会检查 PhantomData<T> —— 即使 T 没有被实际持有

struct OwningWrapper<T> {
    ptr: *const u8,
    _marker: PhantomData<T>,  // ← Drop checker 认为你拥有 T
}

// 这可能导致令人惊讶的编译错误：
struct Container<'a> {
    data: &'a str,
}

fn phantom_drop_check_problem() {
    let container = Container { data: "hello" };
    let wrapper = OwningWrapper {
        ptr: std::ptr::null(),
        _marker: PhantomData::<Container>,  // PhantomData 认为拥有 Container
    };
    
    // drop(wrapper);
    // 编译器可能要求 Container 在 wrapper 之后 drop
    // 但 container 还在存活——没问题
    
    // 但如果 PhantomData 被放在错误的位置，可能导致"借用冲突"的假阳性
    
    mem::drop(wrapper);  // 实际上 drop 了 PhantomData（no-op）
    // container 现在可以安全使用
    println!("{}", container.data);  // ✅ OK
}

fn mem<T>(_val: T) {}  // 只是一个消耗任何东西的函数
```

### 陷阱 3：ZST 的 Clone/Copy 有隐藏的代价吗

```rust
// ZST 几乎总是 Copy + Clone（没有任何数据要复制）
// 但手动实现的 Clone 可能有意外的副作用：

#[derive(Debug)]
struct LoggingZst;

impl Clone for LoggingZst {
    fn clone(&self) -> Self {
        println!("Cloning a ZST!");  // 副作用！
        LoggingZst
    }
}

// 这种情况下，x.clone() 不是"零开销"的——因为有 println
// 但大多数标准库的 ZST 的 Clone/Copy 都是真正的 no-op

// 标准库 ZST 的 Clone 实现（概念）：
// impl Clone for PhantomData<T> {
//     fn clone(&self) -> Self { PhantomData }
// }
// → 编译器内联 → 只有 PhantomData 的构造 → LLVM 删除 → 零指令

// 相似的，() 的 Clone：
// impl Clone for () {
//     fn clone(&self) -> () { () }
// }
// → 被完全优化掉
```

### 陷阱 4：`[ZST; N]` 实际上是 ZST——即使 N 很大

```rust
use std::mem;

// [PhantomData<i32>; 1_000_000] 的大小是 0！
let huge_zst_array: [PhantomData<i32>; 1_000_000] = [PhantomData; 1_000_000];
assert_eq!(mem::size_of_val(&huge_zst_array), 0);

// 但这不能滥用——虽然大小是 0，但类型系统中"数组长度"仍然起作用：
// 
// 索引检查仍然发生：
// let x = huge_zst_array[999_999];  // ✅ 不需要——0 大小
// let x = huge_zst_array[1_000_000]; // ❌ 编译错误——索引越界（const 求值）
//
// 但运行时索引不受影响——对于 ZST 元素，ptr::add(offset) 始终相同

// 迭代：
for (i, _) in huge_zst_array.iter().enumerate() {
    // 循环执行 1,000,000 次——但 body 中没有任何数据访问
    // LLVM 很难完全优化掉这个循环——除非 body 也是纯副作用
    if i > 10 {
        break;  // 早点退出以避免性能问题
    }
}

// ⚠️ 陷阱：当泛型代码在 T 上时，如果 T 是 ZST：
fn create_and_fill<T: Default>(len: usize) -> Vec<T> {
    let mut v = Vec::with_capacity(len);
    for _ in 0..len {
        v.push(T::default());  // 如果是 ZST，T::default() 是 trivial
    }
    v
}
// 如果 T = u64，这是 N 次堆写入
// 如果 T = PhantomData<i32>，这是 N 次循环迭代 + N 次 trivial 函数调用
// → 通常被 LLVM 优化掉，但不要在性能关键路径中依赖这个
```

## 10. 练习答案

### 练习 1 答案

```
(a) struct A;                          → ZST (size = 0)
(b) struct B { x: () }                 → ZST (size = 0, 字段是 ZST)
(c) struct C { x: PhantomData<i32> }   → ZST (size = 0)
(d) struct D { x: *const u8 }          → 不是 ZST (size = 8, 裸指针)
(e) enum E { Variant }                 → ZST (size = 0, 只有一个不带数据的变体)
(f) struct F { data: u64, tag: () }    → 不是 ZST (size = 8, data 是 u64)
(g) struct G([u8; 0]);                → ZST (size = 0, 零长度数组)
(h) enum H {}                          → ZST (size = 0, 空枚举/uninhabited)
(i) struct I(PhantomData<String>);     → ZST (size = 0)
(j) struct J { x: (), y: PhantomData<String>, z: [u8; 0] }
                                       → ZST (size = 0, 所有字段都是 ZST)
```

### 练习 2 答案

```rust
use std::marker::PhantomData;

// 状态标签
struct Closed;
struct Open;

struct File<State = Closed> {
    _state: PhantomData<State>,
    fd: i32,  // 模拟文件描述符
}

impl File<Closed> {
    fn new() -> Self {
        File {
            _state: PhantomData,
            fd: -1,  // 无效 fd
        }
    }
    
    fn open(self, path: &str) -> std::io::Result<File<Open>> {
        // 模拟 open() 系统调用
        let fd = path.len() as i32; // 模拟
        if fd == 0 {
            return Err(std::io::Error::new(
                std::io::ErrorKind::NotFound,
                "file not found",
            ));
        }
        println!("Opened file: {}", path);
        Ok(File::<Open> {
            _state: PhantomData,
            fd,
        })
    }
}

impl File<Open> {
    fn read(&self, buf: &mut [u8]) -> std::io::Result<usize> {
        // 模拟 read
        println!("Reading {} bytes from fd {}", buf.len(), self.fd);
        Ok(0)
    }
    
    fn write(&self, data: &[u8]) -> std::io::Result<usize> {
        // 模拟 write
        println!("Writing {} bytes to fd {}", data.len(), self.fd);
        Ok(data.len())
    }
    
    fn close(self) -> File<Closed> {
        println!("Closing fd {}", self.fd);
        File::<Closed> {
            _state: PhantomData,
            fd: -1,
        }

    // file.read(buf);  // ❌ Closed 状态没有 read 方法
```

```rust
use std::convert::Infallible;

// (a) HashMap<K, ()> ↔ HashSet<K>
// 双向转换——零开销（ZST 的值被优化掉）

fn hashmap_to_hashset<K: Eq + Hash + Clone>(map: HashMap<K, ()>) -> HashSet<K> {
    map.into_keys().collect()
    // into_keys() 丢弃 () 值——drop(())) 是 no-op
    // HashSet<K> 内部就是 HashMap<K, ()>
    // 所以这实际上是零开销的类型重标
}

fn hashset_to_hashmap<K: Eq + Hash>(set: HashSet<K>) -> HashMap<K, ()> {
    set.into_iter().map(|k| (k, ())).collect()
}

// 实际上，std 内部就是这样实现的：
// HashSet<K> 就是 HashMap<K, ()>
// 转换是 O(1) 时间 —— 只是类型参数变了

// (b) Option<Infallible> ↔ ()
// Infallible 没有任何值 → Option<Infallible> 只有 None 一种可能
// () 只有一个值

fn option_infallible_to_unit(opt: Option<Infallible>) -> () {
    match opt {
        None => (),
        Some(infallible) => match infallible {},  // 不可达
    }
}

fn unit_to_option_infallible((): ()) -> Option<Infallible> {
    None  // 唯一可能的返回值
}

// 两个转换都编译为零指令

// Result<Infallible, E> 只能是 Err(e)

fn result_to_e<E>(r: Result<Infallible, E>) -> E {
    match r {
        Err(e) => e,
        Ok(infallible) => match infallible {},
    }
}

fn e_to_result<E>(e: E) -> Result<Infallible, E> {
    Err(e)
}

    arr1.map(|_| PhantomData)
}
```

```
    → None 用 0 → 1 字节

    → None = null → 和 &T 一样大

(e) Option<Box<T>>
    size = 8 (64-bit)

    → None = null → 和 Box<T> 一样大
    → 8 字节

(f) Option<Infallible>
    size = 0

    size = 8 (64-bit)

    size = 1
    Ordering 只有 3 个值（Less=-1, Equal=0, Greater=1）
    → 其他 i8 值都是 niche → 1 字节
```

## 11. 深度扩展

### 11.1 C++ 的 ZST 方案——EBO 和 no_unique_address

```cpp
// C++ 的零开销抽象面临的问题：

// 问题 1：空类在 C++ 中至少占 1 字节
struct Empty {};
static_assert(sizeof(Empty) >= 1);  // 标准保证

// 为什么？因为不同的对象必须有不同的地址：
Empty e1, e2;
assert(&e1 != &e2);  // 必须为真
// 如果 sizeof(Empty) == 0，e1 和 e2 的地址会相同

// 问题 2：EBO 只对基类有效
struct HasEBO : Empty {
    int x;
};
static_assert(sizeof(HasEBO) == sizeof(int));  // 4 字节 → EBO 生效！

struct NoEBO {
    [[no_unique_address]] Empty e;  // C++20 才支持
    int x;
};
// C++17 及之前：sizeof(NoEBO) == 8（1 + 3 padding + 4）
// C++20 之后：sizeof(NoEBO) == 4（Empty e 不占空间）

// 问题 3：no_unique_address 不强制零大小
// 它只是说"不要求唯一地址"——编译器 MAY 优化，而非 MUST
// 实际中 GCC/Clang/MSVC 都会优化，但这不是语言保证

// Rust 的对比：
// - 所有 ZST 的大小始终是 0（语言保证）
// - 没有 C++ 的"至少 1 字节"规则
// - 不需要特殊的 [[no_unique_address]] 注解
// - PhantomData<T> 不需要 EBO 那样的"继承"技巧

// Rust ZST 的关键设计选择：
// 放弃"每个值有唯一地址"的承诺（ZST 没有地址）
// 换取真正的零开销抽象
// 这是 Rust 和 C++ 在"零开销"哲学上的一个核心分歧
```

### 11.2 如果没有 ZST——PhantomData 的代价

```rust
// 假设 Rust 没有 ZST，所有类型至少占 1 字节：

// PhantomData<i32> 至少要 1 字节
// 每一个使用 PhantomData 的结构体多付出 1 字节

// 示例：标准库中的 HashMap
// 实际结构（简化）：
struct HashMap<K, V> {
    // ... buckets, len, etc.
    _marker: PhantomData<(K, V)>,  // 0 字节
}

// 如果没有 ZST，_marker 至少 1 字节
// 这对于 Rust 标准库中上千个使用 PhantomData 的地方来说
// 累积的内存浪费是显著的

// 但更糟的是：typestate 模式将完全不可行
// 原因：如果 typestate 标签有大小，那它就不再是"零成本抽象"了
struct File<State> {
    _state: State,  // 如果 State 至少 1 字节——那就不是零开销了
    fd: i32,
}
// 而且 State: Locked / Unlocked 必须是具体的值，不能用 PhantomData 了

// Typestate 模式依赖 ZST：
// - PhantomData<State> 是信息量的"证明"（类型层面）
// - 在运行时完全消失
// - 如果 PhantomData 有大小，每种状态组合都需要额外的内存
```

### 11.3 设计哲学——ZST 与"零成本抽象"

```rust
// ZST 是 Rust "零成本抽象"哲学的最纯粹体现：

// C++ 的口号："Don't pay for what you don't use"
// Rust 的口号：  "Don't pay for what you don't use" +
//               "What you do use, you couldn't hand-code any better"

// ZST 完美体现了这两点：
// 1. 你为用到的东西付费：
//    - 如果你不用 PhantomData，完全不付费
//    - 如果你用了 PhantomData，你支付的是 0 字节 + 0 周期
//
// 2. "亲手写也不会更好"：
//    - 没有 PhantomData，你只能用注释或运行时标记来编码类型关系
//    - 注释会过时，运行时标记有开销
//    - ZST 的 PhantomData 方案是"人类能写出的最优解"

// ZST 在 Rust 设计中不是事后想到的——
// 它是整个类型系统和所有权系统的基础部分：
//
// - 单元类型 () 是默认返回值——如果不 ZST，每个函数都有开销
// - PhantomData 是 trait 约束、Drop check、变性的基础
// - typestate 模式允许编译时验证——在 C++ 中远不如 Rust 自然
// - 零长度数组 [T; 0] 允许统一的数组处理逻辑

// 这不是"优化"——这是 Rust 类型系统的根基
```

### 11.4 ZST 与 Rust 编译器内部

```rust
// 在 rustc 源码中，ZST 的判断是布局计算的重要部分：

// 编译器中的关键函数（概念）：
// fn is_zst<T: Layout>() -> bool {
//     size_of::<T>() == 0
// }

// ZST 在编译器的多个阶段被特殊处理：
//
// 1. 类型检查阶段：
//    - PhantomData<T> 的 variance 推导
//    - ZST 的 Copy/Clone 自动实现检查
//
// 2. MIR 构建阶段：
//    - let zst: () = () 不需要对应的 MIR statement
//    - 对 ZST 的赋值被省略
//
// 3. MIR 优化阶段：
//    - 死代码消除（ZST 操作通常被视为"无副作用"）
//    - 常量传播（ZST 值的传播没有代价）
//
// 4. LLVM codegen 阶段：
//    - Layout 计算中 size=0 影响对齐和偏移
//    - 对 ZST 的存储指令被删除
//    - 函数参数中 ZST 被移除

// 这就是为什么 ZST 能"完全消失"——
// 不是 LLVM 的神奇优化，而是 rustc 在每个阶段都有特殊处理
```

### 11.5 ZST 与 Any / TypeId——运行时类型识别

```rust
use std::any::{Any, TypeId};
use std::mem;

// TypeId 是 16 字节（u128）——不是 ZST
assert_eq!(mem::size_of::<TypeId>(), 16);

    fn type_id() -> TypeId {

// 可以在运行时通过 TypeId::of::<T>() 获得类型标识

// - bevy 游戏引擎的 Component 类型注册
// - actix-web 的 extractor 类型识别

```

### 11.6 `[T; 0]`——零长度数组的实用价值

```rust
// int arr[0];  // C 中"灵活数组成员"——末尾的不完整数组
```

### 11.7 ZST 与 Zero-Sized 集合的性能陷阱

```rust
use std::collections::HashMap;

// 虽然 ZST 元素是零成本的，但循环遍历它们不是：

fn zip_zst_and_data<const N: usize>(data: &[u8]) {
    let zst_arr: [(); N] = [(); N];
    
    // ⚠️ 这个循环执行 N 次！
    for (_, byte) in zst_arr.iter().zip(data.iter()) {
        // 如果 N 很大而 data 很短，会 panic（zip 按最短的迭代器终止）
        // 但如果 N 很大且 data 也很长——这个循环确实执行 N 次
        // 即使 () 不需要数据访问，循环控制流仍然存在
    }
}

// HashMap<K, ()> — set-like 结构：
fn hashset_equivalent() {
    let mut map: HashMap<String, ()> = HashMap::new();
    
    // insert 仍然做完整的工作：
    map.insert("hello".to_string(), ());
    // - 哈希 "hello"
    // - 寻找/创建 bucket
    // - 存储 key（字符串）
    // 只是不存储 value——但 key 的开销一分不少
    
    // get 也是一样：
    if map.contains_key("hello") {
        // - 哈希 "hello"
        // - 查找 bucket
        // - 比较 key
        // 只是返回 () 不需要额外空间
    }
    
    // 总结：ZST 优化的是"有 value 但不需要存储"的场景
    // 不是"整个操作都不需要做"的魔法
}
```

### 11.8 ZST 在嵌入式和无 std 环境中的价值

```rust
#![no_std]

use core::marker::PhantomData;
use core::mem;

// 嵌入式系统中，每个字节都宝贵
// ZST 允许你在不消耗内存的情况下使用 Rust 的类型系统

// 例子：硬件寄存器访问——状态用 ZST 编码
struct GpioPin<const PIN: u8, Mode> {
    _mode: PhantomData<Mode>,
}

struct Input;
struct Output;
struct AlternateFunction;

impl<const PIN: u8> GpioPin<PIN, Input> {
    fn read(&self) -> bool {
        // 读取 GPIO 输入寄存器
        // 实际硬件访问通过 MMIO
        unsafe {
            let gpio_input_reg = 0x40020000 as *const u32;
            ((*gpio_input_reg) >> PIN) & 1 != 0
        }
    }
}

impl<const PIN: u8> GpioPin<PIN, Output> {
    fn set_high(&self) {
        unsafe {
            let gpio_set_reg = 0x40020004 as *mut u32;
            core::ptr::write_volatile(gpio_set_reg, 1 << PIN);
        }
    }
    
    fn set_low(&self) {
        unsafe {
            let gpio_clear_reg = 0x40020008 as *mut u32;
            core::ptr::write_volatile(gpio_clear_reg, 1 << PIN);
        }
    }
}

impl<const PIN: u8, Mode> GpioPin<PIN, Mode> {
    fn into_input(self) -> GpioPin<PIN, Input> {
        // 重新配置硬件为输入模式
        GpioPin { _mode: PhantomData }
    }
    
    fn into_output(self) -> GpioPin<PIN, Output> {
        // 重新配置硬件为输出模式
        GpioPin { _mode: PhantomData }
    }
}

// 使用——编译时保证安全性，运行时零开销：
fn embedded_example() {
    let led: GpioPin<13, Output> = GpioPin { _mode: PhantomData };
    led.set_high();
    
    // let value = led.read();  // ❌ 编译错误——Output 没有 read 方法
    
    let button: GpioPin<5, Input> = GpioPin { _mode: PhantomData };
    let pressed = button.read();  // ✅ Input 有 read 方法
    
    // 所有 PhantomData 字段占用 0 字节内存
    // 在嵌入式 64KB RAM 的芯片上，这是有意义的
    assert_eq!(mem::size_of::<GpioPin<13, Output>>(), 0);
    assert_eq!(mem::size_of::<GpioPin<5, Input>>(), 0);
}
```

## 12. 回顾

```
ZST 学习清单：

□ 理解 ZST 的核心定义：大小 = 0 的类型（()、空结构体、PhantomData、空枚举、[T; 0]）

□ 知道 niche 优化：Option<ZST> 可能比 Option<非ZST> 更小
□ 理解 ZST 在 FFI 中的行为：#[repr(C)] ZST 在 C 中也是 0 大小

□ 知道 ZST 在集合（Vec<()>、HashMap<K, ()>）中的特殊处理
```

> **一句话总结**：ZST（Zero-Sized Types）是大小为 0 字节的类型——`()`、空结构体、`PhantomData<T>`、空枚举（uninhabited）、`[T; 0]`。在运行时，ZST 的所有操作被编译为零指令——它们不占内存、不消耗 CPU、不产生栈帧。但它们在类型层面承载了丰富的信息：PhantomData 编码了所有权和生命周期，空结构体作为 typestate 标签实现了编译时状态验证，空枚举证明了某个分支"永远不会发生"。ZST 是 Rust "零成本抽象"哲学的最纯粹体现——你在运行时什么都不付，但在编译时获得了完整的类型安全保证。
