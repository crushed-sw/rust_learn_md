# 2.14 惰性初始化——OnceCell / OnceLock / LazyLock

> 全局可变状态是系统编程的"房间里的大象"。C 语言的静态变量在编译期就必须初始化，C++ 的 `static` 局部变量虽然支持惰性初始化，却付出了线程安全的隐藏代价（曾经需要手动加锁，后来自动加锁但可能引入异常安全问题）。Rust 的答案是一组类型安全的惰性初始化原语：**OnceCell**（单线程）、**OnceLock**（多线程）、**LazyLock**（自动惰性初始化）。它们确保数据"恰好初始化一次"，并且在未完全初始化时无法被访问——所有不变量都在类型系统中得到保证。没有 `unsafe`，没有 data race，没有"检查是否已初始化"的人为错误。

---

## 1. 问题的起源——为什么需要惰性初始化

### 1.1 全局状态的困境

```rust
// ❌ 问题 1：编译期常量无法表达复杂初始化
// 你没法在编译期打开一个文件、连接到数据库、读取环境变量…

// ❌ 问题 2：运行期初始化 vs 静态生命周期不兼容
// static 要求 'static + Sync + 编译期常量
// 但很多数据结构初始化需要运行期计算

// ❌ 问题 3：初始化顺序依赖
// 两个全局变量 A 和 B，B 的初始化依赖 A
// C++ 中的 "static initialization order fiasco"
```

### 1.2 各种语言的处理方式

```cpp
// C++ 的方式：局部 static + 魔术
std::string& get_config() {
    static std::string config = read_config_file();  // C++11 保证线程安全
    return config;
}
// 问题：
// - 隐式加锁——不是零成本
// - 异常不安全（如果 read_config_file 抛异常，后续调用会重试——但这是 C++17 才明确的）
// - "Magic Statics"——你甚至不知道编译器为你加了什么代码

// Go 的方式：sync.Once
// var once sync.Once
// once.Do(func() { ... })
// 问题：需要闭包——有堆分配开销

// Java 的方式：静态初始化块 + volatile + double-checked locking
// 最容易出错的并发编程模式之一（直到 Java 5 才修复了 bug）
```

## 2. Rust 的惰性初始化三件套

### 2.1 `OnceCell<T>` —— 单线程惰性初始化（关键类型）

```rust
use std::cell::OnceCell;
use std::collections::HashMap;

fn main() {
    let cell = OnceCell::new();
    
    // 检查是否已初始化
    assert_eq!(cell.get(), None);
    
    // 设置值（get_or_init 保证只初始化一次）
    let map = cell.get_or_init(|| {
        let mut m = HashMap::new();
        m.insert("key", 42);
        m
    });
    
    // 再次调用 get_or_init——不会重新执行闭包
    let map2 = cell.get_or_init(|| {
        // 这个闭包永远不会被调用！
        unreachable!();
    });
    
    assert_eq!(map.get("key"), Some(&42));
    assert_eq!(std::ptr::eq(map, map2), true);  // 同一块内存
    
    // 直接设置（如果已初始化则返回 Err）
    let cell = OnceCell::new();
    assert_eq!(cell.set("hello"), Ok(()));
    assert_eq!(cell.set("world"), Err("world"));  // 已经设置过了
    assert_eq!(cell.get(), Some(&"hello"));
}
```

### 2.2 `OnceLock<T>` —— 多线程惰性初始化

```rust
use std::sync::OnceLock;
use std::thread;
use std::sync::Arc;

fn main() {
    let lock = Arc::new(OnceLock::new());
    
    // 多个线程尝试初始化同一个 OnceLock
    let mut handles = vec![];
    for i in 0..10 {
        let lock = lock.clone();
        handles.push(thread::spawn(move || {
            let value = lock.get_or_init(|| {
                // 只有一个线程会执行这个闭包
                // 其他线程会阻塞等待
                format!("initialized by thread {}", i)
            });
            value.clone()
        }));
    }
    
    let results: Vec<_> = handles
        .into_iter()
        .map(|h| h.join().unwrap())
        .collect();
    
    // 所有线程拿到相同的值（由"第一次"初始化决定）
    assert!(results.iter().all(|s| s == &results[0]));
}
```

### 2.3 `LazyLock<T>` —— 首次访问时自动初始化

```rust
use std::sync::LazyLock;

// 全局 LazyLock——在首次访问时自动初始化
static CONFIG: LazyLock<String> = LazyLock::new(|| {
    // 模拟读取配置文件
    std::fs::read_to_string("/etc/app.conf")
        .unwrap_or_default()
});

static CACHE: LazyLock<std::collections::HashMap<String, i32>> = LazyLock::new(|| {
    let mut m = std::collections::HashMap::new();
    m.insert("default".to_string(), 0);
    m
});

fn main() {
    // 直到这里才真正初始化
    println!("Config: {}", *CONFIG);
    println!("Cache: {:?}", *CACHE);
}
```

## 3. 内部实现原理

### 3.1 OnceCell 的简化实现

```rust
// OnceCell 的核心结构（简化版）：
use std::cell::UnsafeCell;

struct OnceCell<T> {
    // 内部状态：UNINIT | INITIALIZING | INITIALIZED
    // 单线程版本用 Cell<State> —— 不需要原子操作
    state: std::cell::Cell<State>,
    value: std::cell::UnsafeCell<Option<T>>,
}

#[derive(Clone, Copy, PartialEq)]
enum State {
    Uninit,
    Initializing,
    Initialized,
}

// 关键实现：get_or_init
impl<T> OnceCell<T> {
    fn get_or_init<F: FnOnce() -> T>(&self, f: F) -> &T {
        match self.get() {
            Some(value) => return value,
            None => {}
        }
        
        // 设置状态为"初始化中"
        self.state.set(State::Initializing);
        
        // 执行初始化
        // 注意：如果 f 中又调用了 get_or_init（递归），会 panic
        let value = f();
        
        // 写入值
        unsafe {
            (*self.value.get()) = Some(value);
        }
        
        // 标记为已初始化
        self.state.set(State::Initialized);
        
        // 返回引用
        unsafe { (*self.value.get()).as_ref().unwrap() }
    }
}
```

### 3.2 OnceLock 的简化实现

```rust
use std::sync::atomic::{AtomicU8, Ordering};

struct OnceLock<T> {
    state: AtomicU8,
    // OnceLock 和 OnceCell 的关键区别：
    // - OnceCell 用 Cell（非原子） → !Send + !Sync
    // - OnceLock 用 Atomic（原子） → Send + Sync
    value: UnsafeCell<Option<T>>,
}

// 和 OnceCell 的区别：
// get_or_init 在设置 state 时用 compare_exchange 而非 set
// → 多线程安全：只有一个线程进入初始化
    
// 简化版 get_or_init：
impl<T> OnceLock<T> {
    fn get_or_init<F: FnOnce() -> T>(&self, f: F) -> &T {
        if let Some(value) = self.get() {
            return value;
        }
        
        // CAS：只有状态为 UNINIT 的线程才进入初始化
        if self.state.compare_exchange(
            0, // UNINIT
            1, // INITIALIZING
            Ordering::Acquire,
            Ordering::Acquire,
        ).is_ok() {
            // 我是初始化者
            let value = f();
            unsafe { (*self.value.get()) = Some(value); }
            self.state.store(2, Ordering::Release); // INITIALIZED
        } else {
            // 其他线程——自旋等待
            while self.state.load(Ordering::Acquire) != 2 {
                std::hint::spin_loop();
            }
        }
        
        // 所有线程都从这里返回
        unsafe { (*self.value.get()).as_ref().unwrap() }
    }
}
```

### 3.3 LazyLock 的简化实现

```rust
struct LazyLock<T, F = fn() -> T> {
    cell: OnceLock<T>,
    init: std::cell::Cell<Option<F>>,
}

impl<T, F: FnOnce() -> T> LazyLock<T, F> {
    fn new(f: F) -> LazyLock<T, F> {
        LazyLock {
            cell: OnceLock::new(),
            init: Cell::new(Some(f)),
        }
    }
}

impl<T, F: FnOnce() -> T> std::ops::Deref for LazyLock<T, F> {
    type Target = T;
    
    fn deref(&self) -> &T {
        self.cell.get_or_init(|| {
            let f = self.init.take().unwrap();
            f()
        })
    }
}
// LazyLock 本质：OnceLock + 闭包存储 + Deref 自动触发初始化
```

## 4. 对比：lazy_static! / once_cell crate / std

### 4.1 历史演进

```
2016 ── lazy_static! crate 诞生
         ├── 用宏模拟惰性初始化
         ├── 基于 std::sync::Once
         └── 问题：宏语法不自然，类型必须满足 Sync

2019 ── once_cell crate (matklad 创建)
         ├── OnceCell<T> 和 OnceCell<T>（sync 模块）
         ├── 类型系统方案——不需要宏
         └── 被社区广泛采用

2023 ── OnceCell / OnceLock 进入 std (Rust 1.70)
         └── 基于 once_cell 的设计，做了清理

2024 ── LazyCell / LazyLock 进入 std (Rust 1.80)
         └── 进一步简化全局变量的惰性初始化
```

### 4.2 语法对比

```rust
// === 旧：lazy_static! 宏 ===
use lazy_static::lazy_static;

lazy_static! {
    static ref CONFIG: String = read_config();  // ← 函数调用
    static ref CACHE: Mutex<HashMap<i32, String>> = Mutex::new(HashMap::new());
}

// === 中：once_cell crate ===
use once_cell::sync::Lazy;

static CONFIG: Lazy<String> = Lazy::new(|| read_config());
// 或
static CONFIG: Lazy<String> = Lazy::new(read_config);

// === 新：标准库 LazyLock ===
use std::sync::LazyLock;

static CONFIG: LazyLock<String> = LazyLock::new(|| read_config());
// 或（Rust 1.80+）
static CONFIG: LazyLock<String> = LazyLock::new(read_config);

// 演进方向：
// 宏 → 类型系统 → 更简洁的类型系统
// 每一项都减少了一个依赖
```

### 4.3 选择指南

```
场景                              最佳选择
──────────────────────────────────────────────────
全局 static（多线程）              std::sync::LazyLock<T>
全局 static（单线程，只有 main）     std::cell::LazyCell<T> 或 OnceCell<T>
结构体中的惰性字段                 OnceCell<T> / OnceLock<T>
需要重置/重新初始化                once_cell::OnceCell（支持 take）
需要多线程 call_once_init         std::sync::Once + unsafe
避免任何依赖                      std（1.80+ 就全有了）
从旧代码迁移                      先看 lazy_static! → LazyLock 怎么转
```

## 5. 常见陷阱

### 陷阱 1：OnceCell 的 Send/Sync 陷阱

```rust
use std::cell::OnceCell;
use std::sync::OnceLock;

// OnceCell<T> 是 !Sync —— 不能在多线程间共享
// 即使用 Arc 也不行（可 Send 但不可 Sync）

// ❌ 错误：OnceCell 跨线程共享
use std::sync::Arc;
use std::thread;
use std::cell::OnceCell;

let cell = Arc::new(OnceCell::new());
// let cell2 = cell.clone();
// thread::spawn(move || {
//     cell2.get_or_init(|| 42);  // 编译错误：OnceCell 不是 Sync
// });

// ✅ 正确：用 OnceLock
use std::sync::OnceLock;

let lock = Arc::new(OnceLock::new());
let lock2 = lock.clone();
thread::spawn(move || {
    lock2.get_or_init(|| 42);
}).join().unwrap();
```

### 陷阱 2：get_or_init 中的恐慌

```rust
use std::sync::OnceLock;

let lock = OnceLock::<String>::new();

// ⚠️ 如果闭包 panic——OnceLock 被"毒化"
// 当前实现中：如果 get_or_init 的闭包 panic，
// OnceLock 会停留在 "INITIALIZING" 状态
// → 后续调用 get_or_init 会看到这个状态并 panic（"poisoned"）

// 这是和 std::sync::Once 一致的行为——因为无法知道
// 闭包在 panic 前做了多少工作（状态是否一致）

// 解决方案：确保闭包不 panic
lock.get_or_init(|| {
    // ✅ 用 Result 处理可能的错误
    "safe_value".to_string()
});
```

### 陷阱 3：LazyLock 的循环依赖

```rust
use std::sync::LazyLock;

// ❌ 循环初始化——会导致"already borrowed" panic 或死锁
static A: LazyLock<String> = LazyLock::new(|| {
    B.clone()  // B 又回调 A
});
static B: LazyLock<String> = LazyLock::new(|| {
    A.clone()  // A 正在初始化中
});

// 运行时：访问 A → 初始化 A → 需要 B → 初始化 B → 需要 A
// → B 的闭包中访问 A → A 正在初始化 → panic!

// ✅ 正确：设计无环的初始化图
```

### 陷阱 4：LazyLock 不是零成本

```rust
use std::sync::LazyLock;

// LazyLock 每次 Deref 都需要检查"是否已初始化"
// 虽然检查是原子操作——但在热路径中不应该用 LazyLock

// ❌ 热循环中频繁 Deref LazyLock：
static EXPENSIVE: LazyLock<String> = LazyLock::new(|| "hello".to_string());

fn hot_loop() {
    for _ in 0..1_000_000 {
        let _ = EXPENSIVE.len();  // 每次都要原子加载检查状态
    }
}

// ✅ 正确：在热循环外部先缓存：
fn optimized() {
    let cached: &String = &EXPENSIVE;  // 只检查一次
    for _ in 0..1_000_000 {
        let _ = cached.len();
    }
}
```

### 陷阱 5：LazyLock 闭包的 FnOnce 限制

```rust
use std::sync::LazyLock;

// LazyLock 的初始化函数是 FnOnce——因为它会被消耗
// 不能捕获在运行时改变的外部变量

// ❌ 错误：闭包捕获运行时值
fn create_lazy(prefix: String) -> LazyLock<String> {
    LazyLock::new(|| format!("{}-suffix", prefix))
    // ✅ 这个其实可以——prefix 被 move 进闭包
}

// 但更常见的问题是：
// ❌ 想用一个函数签名为 Fn() -> T 的函数：
// if LazyLock::new(my_fn)  // 如果 my_fn 需要多次调用——但 LazyLock 只调用一次
// → 这不是问题，LazyLock::new 接受 FnOnce
```

## 7. 练习答案

### 练习 1 答案

```rust
use std::sync::{OnceLock, Arc, Mutex};
use std::collections::VecDeque;

struct ConnectionPool {
    connections: VecDeque<Connection>,
}

struct Connection {
    id: usize,
}

static POOL: OnceLock<Mutex<ConnectionPool>> = OnceLock::new();

fn get_pool() -> &'static Mutex<ConnectionPool> {
    POOL.get_or_init(|| {
        let connections = (0..10).map(|id| Connection { id }).collect();
        Mutex::new(ConnectionPool { connections })
    })
}

fn get_connection() -> Option<Connection> {
    get_pool().lock().unwrap().connections.pop_front()
}

fn return_connection(conn: Connection) {
    get_pool().lock().unwrap().connections.push_back(conn);
}

// 关键设计：
// 1. OnceLock<Mutex<Pool>> —— OnceLock 保证初始化一次，Mutex 保证访问安全
// 2. get_pool() 总是返回 &'static Mutex<Pool>
// 3. 第一次调用 get_connection 时才真正连接数据库
```

### 练习 2 答案

```rust
use std::sync::Once;
use std::collections::HashMap;

static mut MAP: Option<HashMap<String, i32>> = None;
static INIT: Once = Once::new();

fn get_map() -> &'static HashMap<String, i32> {
    unsafe {
        INIT.call_once(|| {
            MAP = Some(HashMap::from([
                ("key".to_string(), 42),
            ]));
        });
        MAP.as_ref().unwrap()
        // ⚠️ 问题：返回了 &'static HashMap——对全局可变状态的引用
        // 如果别处修改了 MAP，这个引用就成了 dangling reference
    }
}

// 相比 OnceLock 缺失的安全保证：
// 1. 没有所有权模型——MAP 是全局可变静态变量
// 2. 可能产生多个 &mut 引用（没有借用检查）
// 3. 如果 call_once 的闭包 panic——MAP 仍然是 None
//    → 下次调用 INIT.call_once 不会重新执行（Once 被"污染"）
//    → get_map() 会遇到 unwrap() panic
// 4. 需要 unsafe 块——任何修改都可能破坏不变量
```

### 练习 3 答案

```rust
use std::sync::{LazyLock, Mutex, RwLock};

// lazy_static! 版本：
// lazy_static! {
//     static ref COUNTER: Mutex<u64> = Mutex::new(0);
//     static ref DATA: RwLock<Vec<String>> = RwLock::new(Vec::new());
// }

// LazyLock 版本：
static COUNTER: LazyLock<Mutex<u64>> = LazyLock::new(|| Mutex::new(0));
static DATA: LazyLock<RwLock<Vec<String>>> = LazyLock::new(|| RwLock::new(Vec::new()));

fn main() {
    // 使用方式完全相同：
    let mut count = COUNTER.lock().unwrap();
    *count += 1;
    
    let mut data = DATA.write().unwrap();
    data.push("hello".to_string());
}

// 语义差异：
// 1. lazy_static! 使用前会自动 Deref → 语法甜一点
//    lazy_static!: 直接访问 COUNTER.lock()
//    LazyLock:    需要 *COUNTER 或 COUNTER.lock()
//                 但 AutoDeref 使两者几乎无差别
//
// 2. lazy_static! 生成的类型实现 Deref<Target=T>
//    LazyLock<T> 也实现 Deref<Target=T> → 行为一致
//
// 3. lazy_static! 内部用 std::sync::Once + unsafe
//    LazyLock 内部用 OnceLock —— 更安全的抽象
//
// 4. 迁移时注意：lazy_static! 变量名后不需要写 : Type
//    而 LazyLock 需要显式类型标注（或让编译器推断）
```

### 练习 4 答案

```rust
use std::sync::Mutex;

struct ResettableLock<T> {
    inner: Mutex<Option<T>>,
}

impl<T> ResettableLock<T> {
    fn new() -> Self {
        ResettableLock {
            inner: Mutex::new(None),
        }
    }
    
    fn set(&self, value: T) -> Result<(), T> {
        let mut guard = self.inner.lock().unwrap();
        if guard.is_some() {
            return Err(value);
        }
        *guard = Some(value);
        Ok(())
    }
    
    fn get(&self) -> Option<std::sync::MutexGuard<'_, Option<T>>> {
        // 返回 MutexGuard 而非 &T —— 因为 T 可能被 take
        // 实际上这个设计不太实用，因为 guard 持有锁
        // 更好的方案见下方
        Some(self.inner.lock().unwrap())
    }
    
    fn take(&self) -> Option<T> {
        self.inner.lock().unwrap().take()
    }
    
    fn get_or_init<F: FnOnce() -> T>(&self, f: F) -> &T {
        // 这返回引用就有问题了——如何在使用引用时保持锁？
        // 标准 OnceLock 不用 Mutex——所以没有这个问题
        todo!("需要更复杂的设计")
    }
}

// 问题分析：
// ResettableLock 用 Mutex 保护 → 每次访问都要加锁
// 标准 OnceLock 用 atomic flag → 初始化后就是零成本读取
// 这就是"可重置"的代价——你失去了"初始化后零成本读取"的保证

// once_cell::sync::OnceCell 的 take()：
// 需要 &mut self —— 独占引用保证线程安全
// 这比 Mutex 方案更高效，但限制了 API 的使用方式

// 标准库 OnceLock 不支持 take() 是因为：
// 1. 语义复杂——"已初始化"是一劳永逸的承诺
// 2. 如果支持 take，需要处理"take 后有人持有旧引用"
// 3. 保持简单、不可变的 API 更容易推理
```

## 8. 深度扩展

### 8.1 C++ 的 `static` 局部变量 vs LazyLock——两种惰性初始化哲学

```cpp
// C++ 的 "Magic Statics"（C++11 起）：
std::string& get_global_config() {
    static std::string config = read_config_file();
    return config;
}
// 编译器生成的代码（简化）：
//   static bool initialized = false;
//   static std::aligned_storage_t<sizeof(std::string)> storage;
//   if (!initialized) {
//       // 加锁（编译器自动插入的 mutex guard）
//       if (!initialized) {
//           new (&storage) std::string(read_config_file());
//           initialized = true;
//       }
//   }
//   return *reinterpret_cast<std::string*>(&storage);

// Rust LazyLock——同样的模式，但类型系统可见：
static CONFIG: LazyLock<String> = LazyLock::new(|| read_config());
// 初始化在类型中——不是编译器的隐式魔术

// 关键差异：
// ┌─────────────────────┬──────────────────────────────┐
// │ C++ Magic Statics   │ Rust LazyLock                 │
// ├─────────────────────┼──────────────────────────────┤
// │ 编译器隐式生成       │ 类型显式表达                 │
// │ 破坏异常不安全       │ panic = poison（显式处理）   │
// │ 一次初始化            │ 一次初始化                   │
// │ 语言特性，不可组合   │ 库类型，可组合                │
// │ 零成本读取（GCC/Clang│ 原子操作检查（但 CPU 预测    │
// │  生成无分支代码）    │  使此开销可忽略）             │
// │ 线程安全（C++11+）   │ 线程安全（总是）             │
// │ 全局函数访问         │ 全局 static 变量             │
// └─────────────────────┴──────────────────────────────┘
```

### 8.2 如果没有 OnceCell/OnceLock——Rust 中怎么做惰性初始化

```rust
// 方案 A：std::sync::Once + unsafe（标准库 1.0 时代的唯一选择）
use std::sync::Once;

static mut VALUE: Option<String> = None;
static INIT: Once = Once::new();

fn get_value() -> &'static String {
    unsafe {
        INIT.call_once(|| {
            VALUE = Some("initialized".to_string());
        });
        VALUE.as_ref().unwrap()
    }
}
// 问题：
// 1. unsafe 块——每次访问都需要
// 2. 返回的 &'static String 违反了借用规则（实际上是对全局可变状态的引用）
// 3. 如果闭包 panic——INIT 被"毒化"——VALUE 仍是 None
//    → 下次调用 call_once 不会重新执行 → get_value() 会 unwrap panic
// 4. 如果有人在别处修改 VALUE，所有已有引用都变成 dangling

// 方案 B：Mutex<Option<T>>（安全但低效）
use std::sync::Mutex;

static VALUE: Mutex<Option<String>> = Mutex::new(None);

fn get_value() -> std::sync::MutexGuard<'static, Option<String>> {
    let mut guard = VALUE.lock().unwrap();
    if guard.is_none() {
        *guard = Some("initialized".to_string());
    }
    guard
}
// 问题：
// 1. 每次都加锁——即使初始化完成后也要加锁
// 2. 返回 MutexGuard——限制了使用模式（不能返回 &T）
// 3. 锁竞争——多线程性能灾难

// 方案 C：Option<&T> 的全局变量（根本行不通）
// static VALUE: Option<&String> = None;
// → 编译错误！&'static String 只有在编译期就存在的字符串才行

// 结论：OnceCell/OnceLock 是这三个方案的"统一最优解"
```

### 8.3 OnceLock 的原子操作 vs Mutex——性能对比

```rust
// OnceLock 的 get_or_init：
// - 初始化前：一个 CAS（compare_exchange）+ 可能的自旋等待
// - 初始化后：一个 Load（原子读）—— 比 Mutex::lock 快 10-100x

// Mutex<Option<T>> 的方案：
// - 初始化前：Mutex::lock（系统调用可能） + 检查 + 写入 + unlock
// - 初始化后：Mutex::lock + 检查 + unlock —— 每次都加锁

// 基准测试（概念性的）：
// ┌──────────────┬──────────────┬──────────────┐
// │ 操作          │ OnceLock     │ Mutex        │
// ├──────────────┼──────────────┼──────────────┤
// │ 初始化（首次）│ ~CAS latency │ ~syscall     │
// │ 已初始化后读取│ ~Load (1ns)  │ ~lock (20ns) │
// │ 高并发读取    │ ~Load (1ns)  │ ~contention  │
// └──────────────┴──────────────┴──────────────┘

// 这就是 OnceLock 存在的原因——它是初始化场景的专用工具
// Mutex 是通用锁——功能更强但效率更低
```

### 8.4 为什么 OnceCell 不是 Sync——设计哲学

```rust
use std::cell::OnceCell;

// OnceCell 用 `Cell` 追踪状态（非原子操作）
// → 单线程保证安全
// → 编译器标记为 !Sync

// 为什么不用 Atomic 而用 Cell？
// 1. 性能：Cell 就是裸内存读写——比 Atomic 快
// 2. 语义：OnceCell 承诺是单线程的——加了 Atomic 也不会变多线程安全
//    （因为 OnceCell 不保证可见性——其他线程可能看到过时的状态）
// 3. 编译期错误 > 运行时错误：!Sync 让编译器帮你拒绝多线程使用
// 4. 最小化 API：如果你需要多线程——用 OnceLock
//    Rust 不试图用"一个类型解决所有问题"——而是提供精确的工具

// C++ 的教训：std::shared_ptr 的控制块总是原子的
// → 即使只用单线程也要付出原子操作的开销
// → Rust 的 Rc（非原子）vs Arc（原子）就是吸取了这个教训
```

### 8.5 LazyLock 和 const 的交互

```rust
use std::sync::LazyLock;

// LazyLock::new 是 const fn —— 可以在编译期创建
static CONFIG: LazyLock<String> = LazyLock::new(|| {
    "default".to_string()
});

// 为什么重要？
// - 传统惰性初始化需要 Once::new() 在运行期调用（非 const）
// - LazyLock::new 是 const → 编译期就可以分配并初始化结构体
// - → 在 static 初始化中无需任何运行期代码

// 内幕：LazyLock 的初始化函数存储在一个 Cell<Option<F>> 中
// const LazyLock::new 创建的结构体的所有字段都是 const 可构造的
// → OnceLock::new() 是 const（原子变量可以 const 初始化）
// → Cell::new(Some(f)) 中的 f 被"冻结"在二进制中
```

### 8.6 与 `std::sync::Once` 的关系

```rust
use std::sync::Once;

// Once 是更低层的原语——OnceLock 用 Once 实现吗？
// 不。OnceLock 有自己的实现（基于 AtomicU8）。
// 但 Once 仍然有其用途：

// Once 的独特用途：初始化不返回值的资源
static INIT: Once = Once::new();
static mut LOG_FILE: Option<std::fs::File> = None;

fn init_logging() {
    INIT.call_once(|| {
        unsafe {
            LOG_FILE = Some(std::fs::File::create("app.log").unwrap());
        }
    });
}

// 这种"初始化副作用"模式是 Once 的特长
// OnceCell/OnceLock 更关注"初始化一个值并返回引用"
// 两种工具互补——不是互相替代
```

### 8.7 LazyLock 与依赖注入——惰性初始化作为"穷人的 DI"

```rust
use std::sync::LazyLock;

// 在没有 DI 框架的情况下，LazyLock 提供了一种轻量级替代：

trait Database {
    fn query(&self, sql: &str) -> Vec<String>;
}

struct RealDatabase;
impl Database for RealDatabase {
    fn query(&self, sql: &str) -> Vec<String> {
        vec!["result".to_string()]
    }
}

// 全局的"默认"数据库——惰性初始化
static DB: LazyLock<Box<dyn Database + Sync>> = LazyLock::new(|| {
    Box::new(RealDatabase)
});

// 测试中可以替换：
#[cfg(test)]
fn setup_test_db() {
    // 虽然 LazyLock 不能直接替换——但可以设计更灵活的模式
}

// 更灵活的模式：让调用者传入依赖
fn do_query(db: &dyn Database) -> Vec<String> {
    db.query("SELECT * FROM users")
}

// 全局 LazyLock 作为"便利入口"——不是唯一的访问方式
fn default_query() -> Vec<String> {
    do_query(&*DB)
}

// 设计原则：
// - LazyLock 提供便利——避免每个函数都传 db 参数
// - 但核心逻辑接受 trait object——保持可测试性
// - 这是"实用主义 Rust"的典型模式
```

### 8.8 未来展望——`LazyLock` 还能怎么进化

```
未来可能的改进：

1. try_get_or_init —— 返回 Result 的初始化
   - 当前：闭包 panic → OnceLock 被毒化
   - 未来：可能支持 Result 返回，失败时允许重试

2. LazyLock 的 const 初始化函数改进
   - 当前：LazyLock::new(|...闭包...|) —— 闭包在 static 中
   - 可能的改进：让 LazyLock 的初始化函数更灵活

3. OnceLock 的 try_take（需要 &mut self）
   - 在特定条件下支持"安全重置"
   - 类似于 once_cell 的 take() 但更安全

4. 更细粒度的锁——sharded lazy initialization
   - 大规模惰性初始化的并行模式
   - 类似 Java 的 ConcurrentHashMap 的惰性初始化模式

5. const LazyLock for non-'static
   - 当前 LazyLock 主要用于 static
   - 未来可能在栈上使用 const LazyLock（无需 static）

6. 与线程局部存储的结合
   - thread-local + lazy init 的组合模式
   - 每个线程惰性初始化自己的值
```

## 9. 回顾

```
惰性初始化学习清单：

□ 理解惰性初始化的三大场景：全局变量、结构体字段、资源延迟创建
□ OnceCell<T> ── 单线程惰性初始化，!Sync，用 Cell 追踪状态
□ OnceLock<T> ── 多线程惰性初始化，Sync，用 Atomic 追踪状态
□ LazyLock<T> ── 首次 Deref 时自动初始化，OnceLock + 闭包 + Deref
□ 理解初始化后"零成本读取"（只有一次原子 Load，无锁）
□ 知道 get_or_init 闭包 panic 会"毒化"OnceLock
□ 能从 lazy_static! 迁移到 LazyLock（宏 → 类型系统）
□ 对比 C++ Magic Statics 的隐式线程安全初始化
□ 理解为什么 OnceCell 不用 Atomic（单线程语义 ≠ 多线程安全）
□ "如果没有"：Once + unsafe → Mutex<Option<T>> → OnceCell/OnceLock 的进化线
```

> **一句话总结**：OnceCell/OnceLock/LazyLock 将"惰性初始化"从 `unsafe` 和宏的领域带入类型系统。OnceLock 初始化后只需一次原子 Load（无锁），比 `Mutex<Option<T>>` 方案快 10 倍以上。C++ 的 Magic Statics 提供了类似功能，但实现是编译器隐式魔术——不可见、不可组合。Rust 用三个明确的类型覆盖了所有惰性初始化场景：单线程用 OnceCell、多线程用 OnceLock、自动触发用 LazyLock。
