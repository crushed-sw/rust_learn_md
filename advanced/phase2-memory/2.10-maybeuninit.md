# 2.10 MaybeUninit & 未初始化内存——Rust 安全网下的"手工区"

> Rust 的铁律之一：每个值在使用前必须被初始化。但有些场景你无法一步到位——你需要分配一块空间，然后分步填充它；你需要从 C 函数接收输出参数；你需要实现一个比你更聪明的高性能数据结构。`MaybeUninit<T>` 就是为这些场景开的"后门"：它在你和 unsafe 之间划了一条清晰的边界——"这块内存可能未初始化，你负责在读取前初始化它"。这是 unsafe Rust 最基础也最重要的积木。

---

## 1. MaybeUninit 的核心概念

### 1.1 Rust 的初始化保证——和它的局限性

```rust
// Rust 正常情况下不允许未初始化变量：
// let x: i32;       // ❌ 编译错误——必须初始化
// println!("{}", x);

// 这在大多数场景下是好的——防止读取垃圾数据导致 UB
// 但在以下场景就成了障碍：
// 1. 数组分步初始化——你想 [a, b, c] 但 a、b、c 不是同一时间能得到的
// 2. FFI 输出参数——C 函数通过指针写入结果
// 3. 复用已分配的内存——避免反复分配/释放
```

### 1.2 MaybeUninit 的基本用法

```rust
use std::mem::MaybeUninit;

// MaybeUninit<T>：一块大小和对齐与 T 相同的内存，但可能在未初始化状态

// 创建未初始化的内存：
let mut uninit: MaybeUninit<i32> = MaybeUninit::uninit();
// 此时 uninit 内部是未初始化的——读取它是 UB

// 初始化：
uninit.write(42);
// write() 在不 drop 旧值的情况下写入新值（因为旧值不存在）

// 安全地取出已初始化的值：
let value: i32 = unsafe { uninit.assume_init() };
assert_eq!(value, 42);
// assume_init() 是 unsafe——你承诺这块内存已经被正确初始化了
```

## 2. 数组分步初始化

### 2.1 问题：无法逐步填充数组

```rust
// ❌ 你不能逐步填充数组：
// let mut arr: [String; 3];
// arr[0] = String::from("a");  // ❌ arr 未初始化
// arr[1] = String::from("b");
// arr[2] = String::from("c");

// ✅ 用 MaybeUninit：
use std::mem::MaybeUninit;

let mut arr: [MaybeUninit<String>; 3] = unsafe {
    MaybeUninit::uninit().assume_init()
};
// 现在 arr 中每个元素都是未初始化的

// 逐步初始化：
arr[0].write(String::from("a"));
arr[1].write(String::from("b"));
arr[2].write(String::from("c"));

// 安全地转换为 [String; 3]：
let arr: [String; 3] = unsafe {
    std::mem::transmute::<[MaybeUninit<String>; 3], [String; 3]>(arr)
};
// transmute 在这里是安全的——因为 MaybeUninit 和 T 有相同的布局
// 且我们保证所有元素都已初始化

assert_eq!(arr, ["a", "b", "c"]);
```

### 2.2 用 MaybeUninit 实现安全的数组初始化器

```rust
use std::mem::MaybeUninit;

// 从迭代器创建固定大小数组（不需要 Default）：
fn array_from_iter<T, const N: usize>(mut iter: impl Iterator<Item = T>) -> [T; N] {
    let mut arr: [MaybeUninit<T>; N] = unsafe { MaybeUninit::uninit().assume_init() };

    for (i, item) in (0..N).zip(&mut iter) {
        arr[i].write(item);
    }

    // 如果迭代器提前结束——这里有泄漏风险！
    // 但为了简化，假设迭代器有足够的元素

    unsafe { std::mem::transmute_copy(&arr) }
}

let arr: [String; 3] = array_from_iter(
    ["a", "b", "c"].map(String::from).into_iter()
);
```

### 2.3 安全的部分初始化处理

```rust
use std::mem::MaybeUninit;

struct PartialArray<T, const N: usize> {
    len: usize,
    data: [MaybeUninit<T>; N],
}

impl<T, const N: usize> Drop for PartialArray<T, N> {
    fn drop(&mut self) {
        // 只 drop 已初始化的元素
        for i in 0..self.len {
            unsafe { self.data[i].assume_init_drop(); }
        }
        // 未初始化的位置不需要 drop——它们没有值
    }
}
```

## 3. FFI 输出参数

### 3.1 C 函数通过指针返回结果

```rust
use std::mem::MaybeUninit;

mod ffi {
    // C 函数签名：
    // int compute_value(int* out_result);
    extern "C" {
        pub fn compute_value(out: *mut i32) -> i32;
    }
}

fn call_ffi() -> Result<i32, &'static str> {
    // 创建未初始化的内存供 C 写入
    let mut out = MaybeUninit::<i32>::uninit();

    let status = unsafe {
        ffi::compute_value(out.as_mut_ptr())
    };

    if status == 0 {
        // C 函数成功写入了值
        Ok(unsafe { out.assume_init() })
    } else {
        Err("ffi call failed")
    }
    // 如果失败——out 没有被初始化——不要 assume_init()
    // MaybeUninit 在 drop 时不会释放任何东西（因为没有值需要析构）
}
```

### 3.2 结构体 FFI 输出

```rust
use std::mem::MaybeUninit;

#[repr(C)]
struct FfiResult {
    value: f64,
    error_code: i32,
}

mod ffi {
    extern "C" {
        pub fn get_result(out: *mut FfiResult) -> i32;
    }
}

fn get_ffi_result() -> Option<FfiResult> {
    let mut result = MaybeUninit::<FfiResult>::uninit();

    let status = unsafe { ffi::get_result(result.as_mut_ptr()) };

    if status == 0 {
        Some(unsafe { result.assume_init() })
    } else {
        None
    }
}
```

## 4. 与 ManuallyDrop 的配合

### 4.1 从已初始化值"取出"而不析构

```rust
use std::mem::{MaybeUninit, ManuallyDrop};

// 场景：从一个 Vec 中逐个取出元素而不触发析构
fn take_elements<T>(vec: Vec<T>) -> Vec<T> {
    let mut vec = ManuallyDrop::new(vec);  // 阻止 Vec 自己和元素的析构

    let ptr = vec.as_mut_ptr();
    let len = vec.len();

    let mut result = Vec::with_capacity(len);
    for i in 0..len {
        unsafe {
            // 读取指针处的值但并不析构原位置的元素
            result.push(ptr.add(i).read());
        }
    }

    // ManuallyDrop 的 drop 不会发生——Vec 的内存泄漏了
    result
}
// 这是 unsafe 但正确的——我们手动"偷"了元素
```

### 4.2 复用分配

```rust
use std::mem::MaybeUninit;

// 场景：重用 Vec 的底层分配来存储不同类型的数据
fn reuse_allocation() {
    let mut buf = Vec::<u32>::with_capacity(100);
    buf.extend(0..50u32);

    // 取走底层分配的所有权（但不清除内容）：
    let ptr = buf.as_mut_ptr() as *mut MaybeUninit<f64>;
    let cap = buf.capacity();
    let len = buf.len();
    std::mem::forget(buf);  // 忘记 Vec——不释放底层内存

    // 现在用 ptr 作为 f64 的数组：
    let mut f64_buf = unsafe {
        Vec::from_raw_parts(ptr as *mut f64, 0, cap /* 可能不同 */)
    };
    // ⚠️ 这极度 unsafe——实际代码中需要仔细处理对齐和大小
}
```

## 5. 常见陷阱

### 陷阱 1：忘记 assume_init——或"assume_init"未初始化的值

```rust
// ❌ 读取未初始化的数据——UB
let uninit = MaybeUninit::<i32>::uninit();
// let val: i32 = unsafe { uninit.assume_init() };  // UB！没初始化

// ❌ 部分初始化的数组——UB
let mut arr: [MaybeUninit<i32>; 3] = unsafe { MaybeUninit::uninit().assume_init() };
arr[0].write(1);
arr[1].write(2);
// arr[2] 未初始化
// let arr: [i32; 3] = unsafe { transmute(arr) };  // UB！arr[2] 未初始化
```

### 陷阱 2：MaybeUninit 的 Drop

```rust
// MaybeUninit 的 Drop 不调用 T 的 Drop
// 这是设计使然——如果值是未初始化的，不能调用其析构

// ❌ 如果你初始化了值但没有取走——泄漏！
let mut uninit = MaybeUninit::<String>::uninit();
uninit.write(String::from("hello"));
// uninit 离开作用域——String 的 Drop 未被调用——内存泄漏！

// ✅ 在 drop 前取走值或手动析构：
let mut uninit = MaybeUninit::<String>::uninit();
uninit.write(String::from("hello"));
let s = unsafe { uninit.assume_init() };  // 取走所有权
// s 离开作用域时 String 正常析构
```

### 陷阱 3：write() 覆盖已初始化的值

```rust
let mut uninit = MaybeUninit::<String>::uninit();
uninit.write(String::from("first"));  // 初始化
uninit.write(String::from("second")); // 覆盖——但 "first" 的 String 没有被析构！
// → 内存泄漏："first" 的堆分配永远丢失了

// ✅ 如果需要替换：先取走旧值
let old = unsafe { uninit.assume_init_read() };  // 读取但不析构
uninit.write(String::from("second"));
drop(old);  // 手动析构旧值
```

## 7. 练习答案

### 练习 1 答案

```rust
use std::mem::MaybeUninit;

fn collect_initialized<T>(arr: &mut [MaybeUninit<T>], count: usize) -> Vec<T> {
    let mut result = Vec::with_capacity(count);
    for i in 0..count {
        let value = unsafe { arr[i].assume_init_read() };
        result.push(value);
    }
    result
}
// 注意：assume_init_read 读取值但不析构——原位置变为"未初始化"
```

### 练习 2 答案

```rust
use std::mem::MaybeUninit;

fn get_points(max: usize) -> Vec<(f64, f64)> {
    let mut xs: Vec<MaybeUninit<f64>> = Vec::with_capacity(max);
    let mut ys: Vec<MaybeUninit<f64>> = Vec::with_capacity(max);

    // 为 C 函数预留空间
    unsafe { xs.set_len(max); }
    unsafe { ys.set_len(max); }

    let count = unsafe {
        get_points(
            xs.as_mut_ptr() as *mut f64,
            ys.as_mut_ptr() as *mut f64,
            max as i32,
        )
    };

    let mut points = Vec::with_capacity(count as usize);
    for i in 0..count as usize {
        let x = unsafe { xs[i].assume_init_read() };
        let y = unsafe { ys[i].assume_init_read() };
        points.push((x, y));
    }

    points
}
```

### 练习 3 答案

```rust
// MaybeUninit<()> 大小 = 0（因为 () 大小是 0）
// MaybeUninit<()>::uninit() 也是安全的——ZST 不需要初始化
// 实际上 ZST 没有实际的"未初始化"状态——它们永远是"有效的"

// 但 MaybeUninit::<()>::uninit().assume_init() 仍然是 unsafe
// 因为类型层面的 unsafe 合约——即使 ZST 没有实际的数据
```

### 练习 4 答案

```rust
use std::mem::MaybeUninit;

fn try_array_from_fn<T, E, const N: usize>(
    mut f: impl FnMut(usize) -> Result<T, E>
) -> Result<[T; N], E> {
    let mut arr: [MaybeUninit<T>; N] =
        unsafe { MaybeUninit::uninit().assume_init() };

    let mut initialized = 0;

    for i in 0..N {
        match f(i) {
            Ok(value) => {
                arr[i].write(value);
                initialized += 1;
            }
            Err(e) => {
                // 回滚——析构已初始化的元素
                for j in 0..initialized {
                    unsafe { arr[j].assume_init_drop(); }
                }
                return Err(e);
            }
        }
    }

    // 所有元素都已初始化
    Ok(unsafe { std::mem::transmute_copy(&arr) })
}
```

## 8. 深度扩展

### 8.1 C/C++ 的未初始化内存——更自由，也更危险

```cpp
// C++ 中未初始化变量是常见的问题来源：

void dangerous() {
    int x;  // 未初始化——包含任意垃圾值
    printf("%d\n", x);  // UB！读取未初始化的值

    // C++ 可以部分初始化数组：
    int arr[3];
    arr[0] = 1;
    // arr[1], arr[2] 未初始化——编译器不阻止
    // 读取它们 = UB
}

// C++ 的未初始化问题：
// 1. 编译器通常会警告但不阻止（取决于编译选项）
// 2. 静态分析工具可以检测（但不完美）
// 3. 运行时 sanitizer（MSan）可以检测——但需要专门的构建

// Rust 的 MaybeUninit 设计：
// 1. 明确区分"已初始化"和"未初始化"——类型层面
// 2. assume_init() 标记为 unsafe——强制程序员确认
// 3. 正常的 T 类型保证已初始化——不需要 MaybeUninit

// 这比 C++ 的"所有地方都可能未初始化"的模型安全得多
```

### 8.2 如果没有 MaybeUninit——与未初始化内存交互的三种痛苦方案

```rust
// 方案 A：用 Option<T> —— 浪费空间和性能
let mut arr: [Option<String>; 3] = [None, None, None];
arr[0] = Some(String::from("a"));
// 每个元素多了一个 discriminant（判别位）+ 运行时检查

// 方案 B：用零值初始化——不是所有类型都有零值
let mut arr: [i32; 3] = [0; 3];  // i32 有零值——但 String 没有
// 零值可能被误用为"有效值"——语义不清晰

// 方案 C：用 ManuallyDrop + 手动追踪初始化状态
struct ManualArray<T, const N: usize> {
    data: [ManuallyDrop<T>; N],  // 需要 T: Default + Clone
    initialized: [bool; N],
}
// 非常复杂——容易出错
// MaybeUninit 是最小化、最安全的方案
```

### 8.3 `assume_init` 和 `assume_init_read` 的微妙差异

```rust
use std::mem::MaybeUninit;

let mut uninit = MaybeUninit::new(42i32);

// assume_init(): &mut self → T（移动值）
// 消耗 MaybeUninit，返回 T
// → MaybeUninit 不再存在
let val: i32 = unsafe { uninit.assume_init() };

// assume_init_read(): &self → T（复制值但不析构原位置）
let mut uninit2 = MaybeUninit::new(String::from("hello"));
let s: String = unsafe { uninit2.assume_init_read() };
// s 是 String 的 bitwise 副本（通过 ptr::read）
// uninit2 仍然"好像"有值——但实际上它的值被"偷"了
// → 需要手动防止双重析构

// 选择：
// - assume_init: 拥有所有权——正常的"取走"
// - assume_init_read: 不拥有——类似 ptr::read ——需要手动管理
```

### 8.4 MaybeUninit 与 niche 优化的交互

```rust
use std::mem::{MaybeUninit, size_of};

// MaybeUninit 阻止 niche 优化：
assert_eq!(size_of::<MaybeUninit<bool>>(), 1);   // 1 字节
assert_eq!(size_of::<MaybeUninit<&i32>>(), 8);   // 8 字节（指针大小）

// 对比 Option<T>：
assert_eq!(size_of::<Option<bool>>(), 1);        // 1 字节（niche）
assert_eq!(size_of::<Option<&i32>>(), 8);       // 8 字节（niche）

// MaybeUninit 不享受 niche 优化——因为它就是一块"原始内存"
// 没有"某个位模式表示 None"的概念
```

### 8.5 `MaybeUninit::zeroed()` —— 零初始化

```rust
use std::mem::MaybeUninit;

// zeroed() 将内存填充为 0
let x = MaybeUninit::<i32>::zeroed();
// 对 i32 来说，0 是有效的——安全
unsafe { assert_eq!(x.assume_init(), 0); }

// 但对引用类型：
let r = MaybeUninit::<&i32>::zeroed();
// r 是全零——a null reference——无效！
// unsafe { let _ = r.assume_init(); }  // UB！null 引用

// ⚠️ zeroed() 对引用、NonNull、bool（除了 0 和 1）等类型是无效的
// 只在类型的所有位模式都有效时才安全（如 u8、i32、[u8; N]）
```

### 8.6 `MaybeUninit::uninit_array()` 和 `MaybeUninit::uninit_slice()`

```rust
use std::mem::MaybeUninit;

// 创建未初始化的数组（const 上下文可用）：
let arr: [MaybeUninit<i32>; 16] = MaybeUninit::uninit_array();

// 创建未初始化的装箱切片：
let slice: Box<[MaybeUninit<i32>]> = MaybeUninit::uninit_slice(100);

// 这些比手动 MaybeUninit::uninit() 更安全——因为编译器保证类型正确
```

### 8.7 `MaybeUninit::fill` —— 批量初始化

```rust
use std::mem::MaybeUninit;

let mut buf: [MaybeUninit<u8>; 10] = MaybeUninit::uninit_array();

// 全部填充为同一个值：
MaybeUninit::fill(&mut buf, 42u8);
// 等价于 buf.iter_mut().for_each(|m| m.write(42));

// 但 fill 可能生成 SIMD 优化——比手动循环快
```

### 8.8 MaybeUninit 在现代 Rust 中的演进

```
1.36: MaybeUninit<T> 稳定
     - 替代了 mem::uninitialized()（已被废弃——极度危险）

1.51: MaybeUninit::assume_init() 变为 const fn
     - 可以在 const 上下文中使用

1.55: MaybeUninit::write_slice() 稳定
     - 批量写入到 MaybeUninit 切片

1.71: MaybeUninit::transpose() 稳定
     - MaybeUninit<Option<T>> ↔ Option<MaybeUninit<T>>

未来可能：
     - const 上下文中更广泛的使用
     - 更安全的 API（减少 unsafe 需求）
```

```

□ 理解 MaybeUninit<T> 的核心用途：代表可能未初始化的 T 内存
□ 掌握 write() 初始化、assume_init() 取走值、assume_init_read() 读取

```

> **一句话总结**：`MaybeUninit<T>` 是 Rust 安全网下的"手工区"——它用类型层面区分"已初始化"和"未初始化"，把 unsafe 的范围精确限制在 `assume_init()` 调用点上。C++ 的未初始化变量散落在代码各处（读取就是 UB），Rust 把它隔离到了类型系统中——你不会"意外"读取未初始化的内存。
