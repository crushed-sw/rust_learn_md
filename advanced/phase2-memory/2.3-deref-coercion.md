# 2.3 Deref & DerefMut 强制解引用——编译器的隐式魔法

> 你写过 `&*x` 吗？你不需要——因为 Rust 编译器在背后自动做了这件事。`Deref` 和 `DerefMut` 是 Rust 最常被使用但最不被注意的 trait。当你写 `s.len()` 而 `s` 是 `String` 时，编译器自动插入了解引用：`(*s).len()` → `str::len`。当你把 `&String` 传给接受 `&str` 的函数时——自动解引用又在工作了。这一课揭开编译器"隐式魔法"的面纱，也告诉你什么时候应该**停止**实现 Deref。

---

## 1. Deref 的基本机制

### 1.1 什么是 Deref

```rust
use std::ops::Deref;

// Deref trait 的定义：
pub trait Deref {
    type Target: ?Sized;
    fn deref(&self) -> &Self::Target;
}

// 当你写 *x 时，Rust 实际上调用 x.deref()
// Box<T> 的实现：
impl<T: ?Sized> Deref for Box<T> {
    type Target = T;
    fn deref(&self) -> &T {
        // Box 内部存储的是堆指针
        unsafe { &**self }  // 解引用两次：&*Box → &*ptr → &T
    }
}

let b = Box::new(42);
let r: &i32 = &*b;   // 显式解引用
let r: &i32 = &b;     // 隐式强制解引用——编译器自动做了
```

### 1.2 自动解引用的三个场景

```rust
// 场景 1：方法调用——编译器沿 Deref 链查找方法
let s = String::from("hello");
s.len();  // String 没有 len() → 编译器自动解引用到 str → 找到 str::len()

// 场景 2：函数参数——&T 自动转为 &Target
fn takes_str(s: &str) {
    println!("{}", s);
}
let s = String::from("hello");
takes_str(&s);  // &String → &str（Deref 强制）

// 场景 3：运算符——* 操作符
let b = Box::new(42);
let val: i32 = *b;  // 解引用：Box<i32> → i32
```

### 1.3 Deref 链的工作方式

```rust
// 编译器会沿着 Deref 链连续查找方法：

struct Wrapper<T>(T);

impl<T> Deref for Wrapper<T> {
    type Target = T;
    fn deref(&self) -> &T { &self.0 }
}

let w = Wrapper(Wrapper(String::from("hello")));
// w: Wrapper<Wrapper<String>>

// 调用 .len() 时——编译器的查找过程：
// 1. Wrapper<Wrapper<String>> 有 len() 吗？→ 没有
// 2. deref → Wrapper<String> 有 len() 吗？→ 没有
// 3. deref → String 有 len() 吗？→ 有！调用 String::len()

w.len();  // ✅ 自动解引用两次找到 String::len()

// ⚠️ 但 Deref 链只在方法的 self 参数上生效——不用于其他参数
// 而且 Deref 链只在编译期展开——没有运行时循环
```

## 2. DerefMut —— 可变解引用

### 2.1 DerefMut trait

```rust
use std::ops::{Deref, DerefMut};

pub trait DerefMut: Deref {
    fn deref_mut(&mut self) -> &mut Self::Target;
}

// Box<T> 的 DerefMut 实现：
impl<T: ?Sized> DerefMut for Box<T> {
    fn deref_mut(&mut self) -> &mut T {
        unsafe { &mut **self }
    }
}

// 自动解引用也适用于可变上下文：
let mut b = Box::new(42);
*b = 100;  // *b 调用 deref_mut() → 返回 &mut i32

// 方法调用中也会触发 DerefMut：
let mut v = vec![1, 2, 3];
v.push(4);  // push 需要 &mut self → 不需要解引用
// 但如果 v 是 Box<Vec<i32>>：
let mut bv = Box::new(vec![1, 2, 3]);
bv.push(4);  // Box<Vec<i32>> 没有 push → deref_mut → Vec::push
```

### 2.2 Deref 与 DerefMut 的协同

```rust
// 编译器使用 DerefMut 的条件：

// 对于 *x = val —— 必须用 DerefMut
// 对于 x.method() 如果 method(&mut self) —— 尝试 DerefMut

// 规则：
// - 如果上下文需要 &T → 使用 Deref
// - 如果上下文需要 &mut T → 使用 DerefMut
// - DerefMut 要求 Deref 已经实现（trait bound）

// 不是所有能 Deref 的类型都能 DerefMut
// 例如：Rc<T> 只实现了 Deref，没有 DerefMut
//       （因为 Rc 是共享所有权——不能返回 &mut T）
let rc = std::rc::Rc::new(42);
// *rc = 100;  // ❌ Rc 没有实现 DerefMut
```

## 3. 实际案例：Deref 在日常中的表现

### 3.1 `&String` → `&str`

```rust
// String 实现了 Deref<Target = str>
// 这是 Rust 中最常见的自动解引用：

fn greet(name: &str) {
    println!("Hello, {}!", name);
}

let s = String::from("Alice");
greet(&s);          // ✅ &String → &str（自动）
greet(s.as_str());  // ✅ 显式——也是 &str
greet(&*s);         // ✅ 手动解引用——效果一样

// 为什么 String 的 Deref 到 str？
// String 拥有堆上的 UTF-8 数据 + len + cap（24 字节）
// str 只是 UTF-8 字节的引用（16 字节胖指针）
// Deref<String> → &str —— 让你能在 String 上调用所有 &str 的方法
```

### 3.2 `&Vec<T>` → `&[T]`

```rust
// Vec<T> 实现了 Deref<Target = [T]>

fn sum_slice(s: &[i32]) -> i32 {
    s.iter().sum()
}

let v = vec![1, 2, 3, 4];
sum_slice(&v);  // ✅ &Vec<i32> → &[i32]（自动）

// 所有 [T] 的方法都可以在 Vec<T> 上直接调用：
v.len();       // [T]::len
v.is_empty();  // [T]::is_empty
v.first();     // [T]::first
```

### 3.3 `&Box<T>` → `&T`

```rust
// Box<T> 实现了 Deref<Target = T>

fn takes_i32_ref(x: &i32) {
    println!("{}", x);
}

let b = Box::new(42);
takes_i32_ref(&b);  // ✅ &Box<i32> → &i32（自动）

// 层级穿透：
trait Animal { fn speak(&self) -> &str; }
struct Dog;
impl Animal for Dog { fn speak(&self) -> &str { "woof" } }

let b: Box<dyn Animal> = Box::new(Dog);
b.speak();  // Box<dyn Animal> → dyn Animal → Dog::speak
```

## 4. 什么时候**不要**实现 Deref

### 4.1 Deref 是"is-a"关系，不是"has-a"

```rust
// ❌ 错误：用 Deref 模拟继承
struct User {
    name: String,
    email: String,
}

struct Admin {
    user: User,       // Admin "有一个" User（has-a）
    permissions: Vec<String>,
}

// ❌ 不要这样：
// impl Deref for Admin {
//     type Target = User;
//     fn deref(&self) -> &User { &self.user }
// }
// 问题：
// 1. Admin 不是"一种" User——它是"有一个" User
// 2. Deref 传递了"is-a"的语义暗示——调用方可能误用
// 3. admin.email 可以通过 Deref 工作——封装被破坏了

// ✅ 正确做法：显式访问或提供方法
impl Admin {
    fn user(&self) -> &User { &self.user }
    fn email(&self) -> &str { &self.user.email }
}
```

### 4.2 Deref 不应该有副作用

```rust
// ❌ 错误：Deref 不应该做"工作"
struct LazyString {
    cache: Option<String>,
    source: String,
}

// ❌ 不要这样：
// impl Deref for LazyString {
//     type Target = str;
//     fn deref(&self) -> &str {
//         // deref 是 &self —— 无法修改 cache
//         // 但你可能想"惰性填充 cache" —— 做不到
//     }
// }

// ✅ Deref 应该只是"返回一个引用"
// 如果需要在访问时做计算——用普通方法
```

### 4.3 Deref 的决策规则

实现 `Deref` 之前，需要判断你是否真的应该实现它。

**应该实现 `Deref` 的场景：** 你是一个智能指针（如 `Box<T>`、`Rc<T>`、`Arc<T>`、`Cow<T>`）；你是在 newtype 模式中对同类型的一层透明包装且确有必要；`Target` 和 `Self` 有语义上的"is-a"关系。

**不应该实现 `Deref` 的场景：** 你是"has-a"关系（封装了另一个类型）；`Deref` 会破坏封装，暴露不该暴露的方法；`Deref` 需要做实际的计算、IO 或分配工作；`Target` 的接口中有些方法在 `Self` 上没有合理的语义。

## 5. 常见陷阱

### 陷阱 1：方法解析歧义

```rust
// 如果两个 Deref 路径都能到达同一个方法名：

struct A;
impl A { fn foo(&self) { println!("A::foo"); } }

struct B(A);
impl Deref for B { type Target = A; fn deref(&self) -> &A { &self.0 } }
impl B { fn foo(&self) { println!("B::foo"); } }

let b = B(A);
b.foo();  // 打印 "B::foo" ——自身方法优先于 Deref 方法

// 但如果 B 没有 foo：
// b.foo() 会调用 A::foo（通过 Deref）
```

### 陷阱 2：DerefMut 的可变性要求

```rust
// DerefMut 要求 self 是 &mut —— 不能在共享引用上调用

let b = Box::new(42);
// b 是不可变的 → *b 返回 &i32（不是 &mut i32）

let mut b = Box::new(42);
*b = 100;  // b 是可变的 → *b 使用 DerefMut → 可以赋值
```

### 陷阱 3：无限 Deref 循环

```rust
// ❌ 危险的递归 Deref
struct Infinite;
impl Deref for Infinite {
    type Target = Infinite;
    fn deref(&self) -> &Infinite {
        self  // 返回自己——编译器阻止了无限循环
        // 但实际上编译器会报错：
        // "overflow evaluating the requirement"
    }
}

// Rust 编译器对 Deref 递归有深度限制
// 但你仍然不应该设计递归的 Deref
```

## 7. 练习答案

### 练习 1 答案

```
x.len() 的方法查找路径：

1. A 有 len() 吗？→ 没有
2. deref → B 有 len() 吗？→ 没有
3. deref → C 有 len() 吗？→ 没有
4. deref → String 有 len() 吗？→ 有！调用 String::len()
   （实际上 String 的 len 也是通过 Deref 到 str ——
    但编译器只会沿一层 Deref 链查找方法名，
    然后直接解析到最终定义）

注意：编译器不是"每次解引用一层"——它会展开整个 Deref 链，
然后在沿途所有类型上查找方法名。如果找到，就用那层的类型。
```

### 练习 2 答案

```
(a) FilePath → Deref ✅
    "文件路径就是一种字符串" — is-a 关系
    调用方应该能在 FilePath 上使用所有 &str 的方法
    风险：FilePath 的内部格式暴露了——但这是设计意图

(b) Database → AsRef/方法 ✅
    "数据库有一个连接" — has-a 关系
    不应该让 Database 透明地暴露 Connection 的所有方法
    Deref 会破坏封装——调用方可以绕过 Database 直接操作 Connection

(c) MyBox → Deref ✅
    "自定义指针包装" — is-a 关系
    智能指针是 Deref 最标准的用例
    调用方应该透明地访问 T 的方法
```

### 练习 3 答案

```rust
use std::ops::{Deref, DerefMut};

struct MyPtr<T> {
    ptr: *mut T,
}

impl<T> Deref for MyPtr<T> {
    type Target = T;
    fn deref(&self) -> &T {
        unsafe { &*self.ptr }
        // SAFETY：调用方保证 ptr 非空且有效
    }
}

impl<T> DerefMut for MyPtr<T> {
    fn deref_mut(&mut self) -> &mut T {
        unsafe { &mut *self.ptr }
        // SAFETY：调用方保证 ptr 非空、有效、且没有其他引用
    }
}

// ⚠️ 这是 unsafe 的典型例子——MyPtr 的构造和析构也需要相应保证
```

### 练习 4 答案

```
Cow<'a, str> 实现 Deref<Target = str> → ✅ 合理
原因：
- Cow 是"借用或拥有"的字符串——它的语义就是"str 的智能指针"
- Borrowed 变体时，*cow 直接返回借用
- Owned 变体时，*cow 解引用到 String 然后到 str
- 调用方不关心数据是借用的还是拥有的——Deref 隐藏了这个实现细节

MutexGuard<T> 实现 Deref<Target = T> → ✅ 也合理
原因：
- MutexGuard 是"加了锁的 T"——语义上是 T + 临时锁
- 访问被锁保护的数据时，Guard 应该像 T 一样工作
- 但 DerefMut 是危险的——如果在持有 DerefMut 引用时释放了锁（drop guard）
  那 DerefMut 返回的引用就悬垂了——这由 Rust 的借用规则防止
```

## 8. 深度扩展

### 8.1 C++ 的 operator-> 和 operator* vs Rust 的 Deref

```cpp
// C++ 的智能指针解引用：两个独立的操作符
template <typename T>
class MyPtr {
    T* ptr;
public:
    T& operator*() { return *ptr; }   // 解引用 → T&
    T* operator->() { return ptr; }   // 成员访问 → T*
};

// Rust 只有一个 Deref trait：
// *x     → x.deref() → &T
// x.foo  → (&*x).foo（编译器自动做）

// 关键差异：
// 1. Rust 的自动解引用是递归的（沿着链），C++ 只做一层
//    Rust: wrapper.method()  → deref→inner.method() → deref→inner2.method()
//    C++:  wrapper->method() → inner.method()      (一次)

// 2. C++ 的 operator-> 返回指针——有 null 风险
//    Rust 的 Deref 返回引用——保证非空

// 3. C++ 的 operator* 可以返回任何类型（不需要 Target 关联类型）
//    Rust 的 Deref 必须声明 Target——类型更严格

// 4. C++ 的 -> 可以递归（如果返回类型也实现了 operator->）
//    但这是通过返回值的类型推导——Rust 通过 trait 系统更明确
```

### 8.2 如果没有 Deref 强制——你需要写多少 `&*` 和 `.as_ref()`

```rust
// 假设 Rust 没有自动 Deref 强制：

fn greet(name: &str) { println!("Hello, {}", name); }

let s = String::from("Alice");
// greet(&s);         // ❌ 不能传 &String
greet(s.as_str());    // ✅ 需要手动 .as_str()
greet(&*s);           // ✅ 需要手动解引用

let v = vec![1, 2, 3];
// v.len();           // ❌ Vec<i32> 没有 len（是 [i32] 的 len）
// v.first();         // ❌ Vec<i32> 没有 first
v.as_slice().len();   // ✅ 手动 as_slice
(&*v).first();         // ✅ 手动解引用

let b = Box::new(42);
// takes_i32_ref(&b);  // ❌ 不能传 &Box<i32>
takes_i32_ref(&*b);    // ✅ 手动解引用

// Deref 强制让这些"手感好"——但代价是"魔法"
// 理解魔法背后的机制 = 成为 Rust 专家的必要条件
```

### 8.3 Deref 和方法解析的优先级

Rust 方法解析的完整优先级（从高到低）是：

1. **固有方法**（inherent methods）—— 在 `impl Type` 中直接定义的
2. **直接 trait 方法** —— trait 在当前作用域且 `Type` 直接实现了该 trait
3. **Deref 链上的固有方法** —— 通过 Deref 转换后找到的固有方法
4. **Deref 链上的 trait 方法** —— 通过 Deref 转换后找到的 trait 方法

当存在歧义时，可以用完全限定语法（UFCS）来显式指定调用路径：

```rust
let s = String::from("hello");
// String::len(&s);                              // 固有方法
// str::len(&*s);                                // Deref 后的方法
// <String as Deref>::deref(&s).len();           // 显式链
```

### 8.4 为什么 `Deref` 是 unsafe trait 的一个关键依赖

`Deref` 直接影响 `Pin` 的安全性。`Pin<P>` 依赖 `P` 的 `Deref` 实现——例如 `Pin<Box<T>>` 利用 `Box<T>` 的 `Deref` 到 `T` 的转换，使得 `Pin` 可以安全地从 `Pin<Box<T>>` "投影"到 `Pin<&mut T>`（或 `Pin<&T>`）。

如果有人错误实现了 `Deref`（返回了不同对象的引用），`Pin` 的保证就会被破坏。但 Rust 的安全性检查阻止了这种情况，因为 `Deref` trait 强制要求返回对同一对象的引用。

只有 `Target: Unpin` 时才能安全地从 `Pin<P>` 获取 `&mut Target`：

```rust
impl<P: DerefMut<Target: Unpin>> DerefMut for Pin<P> {
    fn deref_mut(&mut self) -> &mut P::Target {
        Pin::get_mut(Pin::as_mut(self))
    }
}
```

### 8.5 多重 Deref 和 trait 的特化问题

```rust
// 当一个类型通过 Deref 链实现了同一个 trait 多次：

trait Describe {
    fn describe(&self) -> String;
}

impl Describe for String {
    fn describe(&self) -> String { format!("String: {}", self) }
}

impl Describe for str {
    fn describe(&self) -> String { format!("str: {}", self) }
}

let s = String::from("hello");
// s.describe();  // ⚠️ 歧义！String 有 Describe，str 也有 Describe
// 编译器选择 String 的（固有方法优先）
// 但如果你想要 str 的：
// <str as Describe>::describe(&s);  // UFCS 消除歧义
```

### 8.6 Deref 与自动引用和解引用的配合

```rust
// Rust 的"自动引用/解引用"不仅影响方法调用，也影响比较和运算符：

let s1 = String::from("hello");
let s2 = String::from("hello");

// 比较 &String 和 &String：
assert_eq!(s1, s2);  // PartialEq<String>::eq —— 直接实现

// 比较 &String 和 &str：
assert_eq!(&s1, "hello");  // Deref 强制：&String → &str
                           // PartialEq<str>::eq

// 加法——String + &str（通过 Deref）：
// let s3 = s1 + " world";  // Deref 让 &str 可以作为 Add 的右操作数
```

### 8.7 `DerefPure` —— 未来可能的改进

目前 `Deref` 存在一个限制：编译器假设 `deref()` 可能有副作用，即使实际上大部分 `Deref` 实现都是"纯"的（只是指针解引用）。这可能会影响某些优化——编译器不能"自由地"重复调用 `deref()`。未来 Rust 可能会引入 `DerefPure` 标记 trait（类似于 `const fn` 的概念），让编译器可以在更多场景下放心地优化。

在 const 上下文中，`Deref` 目前无法使用，例如 `const X: &str = &*String::from("hello");` 会编译失败，因为 `Deref` 不是 `const fn`。随着 const trait 的发展，`Deref` 在 const 上下文中的使用会逐步得到拓展。

### 8.8 Deref 的运行时成本——零成本抽象

```rust
// Deref 是真正的零成本抽象：

// 这两段代码生成相同的机器码：
fn use_box(b: &Box<i32>) -> i32 {
    **b  // 显式解引用
}

fn use_box_deref(b: &Box<i32>) -> i32 {
    **b  // Deref::deref(b) 被内联为指针解引用
}

// 汇编层面：
// mov rax, [rdi]    ; 从 Box 读取指针
// mov eax, [rax]    ; 从堆上读取 i32
// 完全相同！
//
// Deref 的方法调用在编译期被完全消除（内联）
// → 零运行时开销
```

```
□ 掌握自动解引用的三个场景：方法调用、函数参数、运算符
□ 知道编译器沿 Deref 链连续查找方法

□ 掌握方法解析歧义时使用 UFCS 消除
```

> **一句话总结**：`Deref` 和 `DerefMut` 让 Rust 编译器能自动插入解引用操作——`&String` 自动变 `&str`、`Box<T>` 自动像 `T` 一样调用方法。C++ 的 `operator*`/`operator->` 做类似的事但不会递归，且返回指针可能为 null。实现 Deref 的核心原则：**只有"is-a"关系（智能指针、newtype）才实现 Deref；"has-a"关系（封装）不要实现 Deref**——否则你会破坏封装，让不该暴露的方法"泄露"到外层。
