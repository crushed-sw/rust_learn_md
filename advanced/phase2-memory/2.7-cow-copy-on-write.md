# 2.7 Cow (Copy on Write)——只在需要时才克隆

> 你从 API 收到了一个字符串，90% 的情况你只是读取它——但偶尔需要修改。如果每次都 `.clone()`，你浪费了 90% 的分配和复制。`Cow<'a, B>`（Clone on Write）解决了这个问题：先把数据**借用**（零成本），只在真正需要修改时才**克隆**。`Cow<str>` 和 `Cow<[T]>` 是 Rust 标准库中最被低估的优化工具——它们在你不知不觉中把"懒"变成了性能优势。

---

## 1. Cow 的基本概念

### 1.1 Cow 的定义

```rust
use std::borrow::Cow;

// Cow 的定义（简化）：
pub enum Cow<'a, B: ?Sized + 'a>
where
    B: ToOwned,
{
    Borrowed(&'a B),        // 借用——不拥有数据
    Owned(<B as ToOwned>::Owned),  // 拥有——克隆了一份
}

// 常用形式：
// Cow<'a, str>    → Borrowed(&'a str) | Owned(String)
// Cow<'a, [T]>    → Borrowed(&'a [T]) | Owned(Vec<T>)
// Cow<'a, Path>   → Borrowed(&'a Path) | Owned(PathBuf)
```

### 1.2 Cow 的核心思想

```rust
use std::borrow::Cow;

// 场景：处理输入字符串，大部分时间只读，偶尔需要修改

// ❌ 不用 Cow：总是复制
fn process_always_clone(input: &str) -> String {
    let mut data = input.to_string();  // 总是分配——即使不修改！
    if data.contains("error") {
        data.push_str(" (fixed)");
    }
    data
}

// ✅ 用 Cow：只在需要时复制
fn process_cow(input: &str) -> Cow<'_, str> {
    let mut data = Cow::Borrowed(input);  // 不分配——只是借用
    if data.contains("error") {
        data.to_mut().push_str(" (fixed)");  // 第一次修改时才克隆！
    }
    data
}

// to_mut() 的神奇之处：
// - 如果当前是 Borrowed → 克隆成 Owned → 返回 &mut String
// - 如果当前是 Owned   → 直接返回 &mut String
// - 零开销的"惰性克隆"
```

## 2. Cow 的工作机制

### 2.1 to_mut() —— 惰性克隆的入口

```rust
use std::borrow::Cow;

let mut cow = Cow::Borrowed("hello");

// to_mut() 内部做了：
// if cow is Borrowed {
//     let owned = cow.to_owned();  // "hello" -> String::from("hello")
//     cow = Cow::Owned(owned);
// }
// &mut Cow::Owned(owned)

let s: &mut String = cow.to_mut();
s.push_str(" world");

match cow {
    Cow::Owned(s) => assert_eq!(s, "hello world"),  // ✅ 现在是 Owned
    _ => unreachable!(),
}
```

### 2.2 into_owned() —— 转换为拥有值

```rust
use std::borrow::Cow;

let cow = Cow::Borrowed("hello");

// into_owned() 总是返回拥有的类型：
let s: String = cow.into_owned();
// 如果是 Borrowed → 克隆
// 如果是 Owned   → 直接返回（零成本）

// 与 to_mut() 的区别：
// - to_mut() 返回 &mut，Cow 仍然存活
// - into_owned() 消耗 Cow，返回 T::Owned
```

### 2.3 Cow 的 Deref 行为

```rust
use std::borrow::Cow;

let cow = Cow::Borrowed("hello");
// Deref<Target = str>:
println!("{}", cow);         // ✅ Display
println!("{}", cow.len());   // ✅ len
println!("{}", cow.contains("ell")); // ✅ contains

// Cow 无论 Borrowed 还是 Owned 都像 &str / &[T] 一样使用
```

## 3. Cow 的实际应用

### 3.1 处理可选修改的数据

```rust
use std::borrow::Cow;

// 场景：去除字符串首尾空白——如果没变化，不分配
fn trim_whitespace<'a>(input: &'a str) -> Cow<'a, str> {
    let trimmed = input.trim();
    if trimmed == input {
        Cow::Borrowed(input)  // 没变化——不分配
    } else {
        Cow::Owned(trimmed.to_string())  // 有变化——返回新 String
    }
}

let s1 = trim_whitespace("hello");    // Borrowed——0 分配
let s2 = trim_whitespace("  hello "); // Owned——1 分配
```

### 3.2 JSON/配置处理

```rust
use std::borrow::Cow;
use std::collections::HashMap;

// 场景：解析 JSON 中的字符串——大部分是原样使用，少数需要脱敏
fn sanitize_value<'a>(value: &'a str, sensitive_keys: &HashMap<&str, bool>) -> Cow<'a, str> {
    if sensitive_keys.contains_key(value) {
        Cow::Owned("***".to_string())
    } else {
        Cow::Borrowed(value)
    }
}

// 大 JSON 文件：10000 个值，只有 10 个需要脱敏
// → 9990 个 Borrowed（零分配） + 10 个 Owned
// → 如果用 String：10000 个分配——浪费 99.9% 的分配！
```

### 3.3 从 Vec 到切片的零拷贝转换

```rust
use std::borrow::Cow;

fn process_numbers<'a>(input: &'a [i32]) -> Cow<'a, [i32]> {
    if input.iter().any(|&x| x < 0) {
        // 有负数——替换为绝对值
        let owned: Vec<i32> = input.iter().map(|&x| x.abs()).collect();
        Cow::Owned(owned)
    } else {
        Cow::Borrowed(input)  // 全正数——零拷贝
    }
}
```

## 4. ToOwned trait —— Cow 的底层机制

### 4.1 ToOwned 的定义

```rust
pub trait ToOwned {
    type Owned: Borrow<Self>;
    fn to_owned(&self) -> Self::Owned;
    fn clone_into(&self, target: &mut Self::Owned) { ... }
}

// str 的实现：
impl ToOwned for str {
    type Owned = String;
    fn to_owned(&self) -> String {
        // 分配新的 String + 复制内容
        self.to_string()
    }
}

// [T] 的实现：
impl<T: Clone> ToOwned for [T] {
    type Owned = Vec<T>;
    fn to_owned(&self) -> Vec<T> {
        self.to_vec()
    }
}
```

### 4.2 Cow 的泛型约束

```rust
// Cow<'a, B> 要求 B: ToOwned
// 这就是为什么 Cow<str> 可以工作（str 实现了 ToOwned）
// 但 Cow<Path> 也可以——Path 实现了 ToOwned<Owned = PathBuf>

use std::borrow::Cow;
use std::path::{Path, PathBuf};

fn get_path<'a>(is_temp: bool) -> Cow<'a, Path> {
    if is_temp {
        Cow::Owned(PathBuf::from("/tmp/data"))
    } else {
        Cow::Borrowed(Path::new("/var/data"))
    }
}
```

## 5. 常见陷阱

### 陷阱 1：to_mut() 后原 Borrowed 不受影响——但你已经有了克隆

```rust
use std::borrow::Cow;

let original = String::from("hello world");
let mut cow = Cow::Borrowed(original.as_str());

// to_mut() 会克隆：
cow.to_mut().push_str("!");  // original 的内容被克隆到新的 String

// original 不受影响：
assert_eq!(original, "hello world");  // ✅ 还在

// 但 cow 现在是 Owned：
match cow {
    Cow::Owned(s) => assert_eq!(s, "hello world!"),
    _ => unreachable!(),
}
```

### 陷阱 2：Cow 的生命周期总是和借用数据绑定

```rust
use std::borrow::Cow;

fn return_cow() -> Cow<'static, str> {
    let local = String::from("hello");
    // Cow::Borrowed(&local)  // ❌ local 的生命周期不够
    Cow::Owned(local)         // ✅ Owned——数据所有权在 Cow 内
}

// 如果需要返回 Cow<'static, ...>，必须用 Owned 变体
```

### 陷阱 3：to_mut() 后不能再访问原始借用——move 了

```rust
use std::borrow::Cow;

let original = String::from("hello");
let mut cow = Cow::Borrowed(original.as_str());

// original 仍然可用——Cow 只是借用了它
println!("{}", original);  // ✅

cow.to_mut().push_str(" world");
// to_mut() 克隆了 original 的数据——original 不受影响
println!("{}", original);  // ✅ 仍然可用（未被 move）
```

## 8. 深度扩展

### 8.1 C++ 的 Copy-on-Write——std::string 的 COW 历史

```cpp
// C++ 曾经尝试在 std::string 中实现 COW（GCC 的 libstdc++）
// 但 C++11 之后被废弃了

// 历史（GCC < 5）：std::string 使用引用计数 + COW
std::string s1 = "hello";
std::string s2 = s1;  // 不分配新内存——共享同一块 buffer
// s2 修改时才拷贝（COW）

// 为什么被废弃了？
// 1. 多线程性能灾难——引用计数需要原子操作
// 2. operator[] 返回非 const 引用时无法判断"是否会修改"
//    s2[0] 可能修改也可能只是读取——无法延迟 COW 决策
// 3. 与 C++11 的移动语义和 noexcept 的交互问题
// 4. 小字符串优化（SSO）已经足够好

// Rust 的 Cow 的优势：
// 1. 类型系统强制"你只能通过 to_mut() 修改"——不存在 operator[] 的歧义
// 2. 单线程场景 Rc 没有原子开销
// 3. 显式——程序员选择什么时候用 Cow——不隐藏开销

// C++ 的 COW 是隐式的——Rust 的 Cow 是显式的
// 这是两种语言哲学差异的缩影
```

### 8.2 如果没有 Cow——要么全分配，要么全借用

```rust
// 假设没有 Cow，处理"可选修改"的三种方式：

// 方式 A：总是返回 String——浪费
fn upper_if_shout(s: &str) -> String {
    if s.contains('!') { s.to_uppercase() }
    else { s.to_string() }  // ← 即使没变化也分配!
}

// 方式 B：返回两个不同的类型——API 不统一
fn upper_if_shout(s: &str) -> Result<&str, String> {
    if s.contains('!') { Err(s.to_uppercase()) }
    else { Ok(s) }  // ← API 很奇怪
}

// 方式 C：让调用方处理
fn needs_upper(s: &str) -> bool { s.contains('!') }
// 调用方：if needs_upper(s) { s.to_uppercase() } else { s.to_string() }
// ← 把"分配 vs 不分配"的选择推给了调用方——封装被破坏

// Cow 统一了三种方式的优点：
// ✅ 统一返回类型（Cow<str>）
// ✅ 零分配的借用路径
// ✅ 需要时的分配路径
```

### 8.3 Cow 与序列化的典型配合

```rust
use std::borrow::Cow;
use serde::{Deserialize, Serialize};

// Cow 在 serde 的零拷贝反序列化中广泛应用：

#[derive(Deserialize)]
struct User<'a> {
    #[serde(borrow)]
    name: Cow<'a, str>,       // 如果可能就用借用，否则拥有
    email: Cow<'a, str>,
}

// serde 的零拷贝模式：
// 1. 解析 JSON 字节时，如果字段内容不需要转义
//    → 直接借用原始 buffer 中的 &str
// 2. 如果内容包含转义序列（如 \n）
//    → 需要修改 → 克隆为 String

// 这就是 Cow 在生态中最重要的应用场景之一
// serde_json::from_str::<User>() 会自动决定 Borrowed 还是 Owned
```

### 8.4 Cow 在 CString/CStr 中的使用

```rust
use std::borrow::Cow;
use std::ffi::{CStr, CString};

// CStr 也支持 Cow：
fn get_c_str<'a>(owned: Option<CString>, borrowed: Option<&'a CStr>)
    -> Cow<'a, CStr>
{
    match owned {
        Some(s) => Cow::Owned(s),
        None => Cow::Borrowed(borrowed.unwrap()),
    }
}

// 这对于 FFI 中"有时拥有、有时借用"的场景很有用
```

### 8.5 `clone_into` —— 重用已有分配

```rust
// ToOwned::clone_into 允许你重用已有的分配：

let mut buffer = String::with_capacity(100);
let source: &str = "hello world";

// 传统方式——每次 to_owned() 都重新分配
// let owned = source.to_owned();  // 新分配

// 用 clone_into 重用 buffer：
source.clone_into(&mut buffer);
// buffer 现在是 "hello world"——如果 capacity 足够就不重新分配

// 这在循环中很有价值：
for line in lines {
    line.clone_into(&mut buffer);
    // 处理 buffer...
    // buffer.clear(); 的情况下可以多次重用同一块内存
}
```

### 8.6 Cow 和类型转换——From 实现

```rust
use std::borrow::Cow;

// Cow 提供了丰富的 From 实现：

// &str → Cow<str>（Borrowed）
let c1: Cow<str> = "hello".into();  // Cow::Borrowed("hello")

// String → Cow<str>（Owned）
let c2: Cow<str> = String::from("hello").into();  // Cow::Owned

// &String → Cow<str>（Borrowed）
let s = String::from("hello");
let c3: Cow<str> = (&s).into();  // Cow::Borrowed(&s)

// 这些 From 实现让 Cow 在 API 中很好用——
// 调用方可以传 &str、String、&String
fn accept_cow<'a>(input: impl Into<Cow<'a, str>>) {
    let cow: Cow<str> = input.into();
    // ...
}
```

### 8.7 Cow 的 Deref 实现与零成本抽象

`Cow` 在两种变体上的 `Deref` 实现都是零成本抽象：

- **Borrowed 变体：** `Deref` 直接返回 `&'a B`，完全零成本——编译后就是 `mov rax, rdi`（直接传递指针）。
- **Owned 变体：** `Deref` 通过 `Borrow<B>` trait 返回 `&B`，即 `O as Borrow<B>::borrow(&self.0)`。这个调用总是被编译器内联，生成和直接写 `&s[..]` 相同的机器码。

两者在编译后都不产生任何额外开销。

### 8.8 为什么 `Cow` 需要 `ToOwned` trait —— 而不直接用 Clone

`Cow` 之所以需要 `ToOwned` trait 而不是直接用 `Clone`，源于两者语义的根本差异：

- `Clone` 是 `T -> T`（同类型复制）：例如 `&String -> String`，这需要两个 `String` 在堆上各自拥有内存。
- `ToOwned` 是 `&T -> Owned`（从引用创建拥有值）：例如 `&str -> String`，`str` 本身不拥有内存，需要分配新内存才能"拥有"数据。

如果用 `Clone` 来代替，会遇到签名上的问题：`Clone` 的签名是 `fn clone(&self) -> Self`，如果对 `str` 实现 `Clone`，返回值必须是 `str`——但 `str` 是 DST，不能作为返回值。而 `ToOwned` 的签名是 `fn to_owned(&self) -> Self::Owned`，返回关联类型 `Owned`，可以是 `Sized` 的类型。因此 `ToOwned` 是比 `Clone` 更通用的"从引用创建拥有值"的抽象。
