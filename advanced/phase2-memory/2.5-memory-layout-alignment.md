# 2.5 内存布局与对齐——字段在你的结构体里到底怎么放的

> 你定义了 `struct Foo { a: u8, b: u32, c: u16 }`。在内存里，`a`、`b`、`c` 是紧挨着放的吗？不是。编译器在它们之间插入了不可见的"填充"（padding），并且可能重新排列字段的顺序——默认的 `repr(Rust)` 布局下，字段的顺序**没有任何保证**。如果你需要 C 兼容的布局，必须显式指定 `#[repr(C)]`。理解内存布局是 FFI、序列化、性能优化的基础——也是"不能依赖默认布局"的根本原因。

---

## 1. 对齐的基本概念

### 1.1 什么是内存对齐

```rust
// 对齐（alignment）：一个值在内存中的起始地址必须是其对齐值的倍数
// 自然对齐：类型的对齐值通常等于其大小（对基本类型）

// 示例——在 64 位系统上：
// u8   : size=1, align=1  — 可以放在任何地址
// u16  : size=2, align=2  — 必须放在偶数地址
// u32  : size=4, align=4  — 必须放在 4 的倍数地址
// u64  : size=8, align=8  — 必须放在 8 的倍数地址
// f64  : size=8, align=8
// &T   : size=8, align=8
// Box<T> : size=8, align=8

use std::mem;

assert_eq!(mem::size_of::<u8>(), 1);
assert_eq!(mem::align_of::<u8>(), 1);

assert_eq!(mem::size_of::<u32>(), 4);
assert_eq!(mem::align_of::<u32>(), 4);

// 为什么需要对齐？
// CPU 访问未对齐的内存可能：
// - x86: 更慢（需要两次内存访问）
// - ARM/某些架构: 直接崩溃（SIGBUS）
// - Atomics（原子操作）: 必须对齐——未对齐会导致 UB
```

### 1.2 填充（padding）如何产生

```rust
use std::mem;

// 由于对齐要求，字段之间可能有不可见的"填充字节"：
struct Foo {
    a: u8,   // offset 0, size 1, align 1
    //      [padding: 3 bytes]   ← a 后面有 3 字节填充！
    b: u32,  // offset 4, size 4, align 4  （4 是 4 的倍数）
    c: u16,  // offset 8, size 2, align 2
    //      [padding: 6 bytes]   ← 结构体尾部有 6 字节填充！
    //                             结构体的整体对齐 = max(align(a), align(b), align(c)) = 4
    //                             结构体大小必须是 4 的倍数 → 2+6 = 8 ← 不对
}
//              实际大小 = 12 bytes
//              对齐要求 = 4

// 计算：8 (b的offset) + 2 (c的size) = 10
//      10 不是 4 的倍数 → pad to 12

assert_eq!(mem::size_of::<Foo>(), 12);
assert_eq!(mem::align_of::<Foo>(), 4);
```

## 2. repr(Rust) —— 默认布局（无保证）

### 2.1 编译器可以重排字段

```rust
// repr(Rust) 是默认的——编译器拥有最大自由度：
// 1. 可以重新排列字段顺序（为了减小 padding）
// 2. 可以添加填充
// 3. 字段顺序没有任何保证
// 4. 同类型的两次编译可能产生不同的布局！

struct Foo {
    a: u8,
    b: u32,
    c: u16,
}

// 编译器可能优化为：
// struct Foo_optimized {
//     b: u32,  // offset 0 (对齐 4)
//     c: u16,  // offset 4 (对齐 2)
//     a: u8,   // offset 6 (对齐 1)
//              // padding 1 byte → size 8!
// }
// 优化后：从 12 字节降到 8 字节——节省 33%！

// 编译器具体怎么优化：
// 1. 最大对齐的字段放最前
// 2. 减少字段之间的空洞
// 3. Niche 优化——利用非法位模式存储额外信息
```

### 2.2 不能依赖 repr(Rust) 的顺序

```rust
// ⚠️ 绝对不要写这样的代码：
struct Foo { x: u8, y: u32 }
let foo = Foo { x: 1, y: 2 };
let ptr = &foo as *const Foo as *const u8;

// ❌ 天真的假设：
// unsafe { assert_eq!(*ptr, 1); }           // 第一个字节是 x？不一定！
// unsafe { assert_eq!(*ptr.add(4), 2); }    // 偏移 4 是 y？不一定！

// repr(Rust) 下字段顺序无保证——
// 用于 FFI、序列化、裸指针操作时必须指定 repr(C)！
```

## 3. repr(C) —— C 兼容布局

### 3.1 基本规则

```rust
#[repr(C)]
struct Foo {
    a: u8,   // offset 0, size 1
    //      [padding: 3 bytes]
    b: u32,  // offset 4, size 4
    c: u16,  // offset 8, size 2
    //      [padding: 2 bytes] → total 12
}

// repr(C) 的规则——严格保证：
// 1. 字段按声明顺序排列（不重排）
// 2. 每个字段按自身对齐要求对齐
// 3. 结构体整体对齐 = max(所有字段的对齐)
// 4. 结构体大小 = 整体对齐的倍数

use std::mem;

// repr(C) 的 Foo 布局：
// offset 0: a (u8,    1 byte, align 1)
//           [3 bytes padding]
// offset 4: b (u32,   4 bytes, align 4)
// offset 8: c (u16,   2 bytes, align 2)
//           [2 bytes padding]  → max_align = 4, total 12

assert_eq!(mem::size_of::<Foo>(), 12);
assert_eq!(mem::align_of::<Foo>(), 4);
// 这是确定性的——在所有平台、所有编译器版本上都一样

// offset_of!（nightly/1.77+）可以验证：
// assert_eq!(mem::offset_of!(Foo, a), 0);
// assert_eq!(mem::offset_of!(Foo, b), 4);
// assert_eq!(mem::offset_of!(Foo, c), 8);
```

### 3.2 repr(C) 用于 FFI

```rust
// 与 C 库交互时必须用 #[repr(C)]：

#[repr(C)]
struct RustStruct {
    x: i32,
    y: f64,
    flag: bool,
}

// 等价的 C 定义：
// struct CStruct {
//     int32_t x;
//     double  y;
//     bool    flag;
// };

// 由于 repr(C)，RustStruct 和 CStruct 的布局完全相同
// → 可以安全地通过 FFI 传递

extern "C" {
    fn process_struct(s: *const RustStruct);
}
```

## 4. repr(align) 和 repr(packed)

### 4.1 `#[repr(align(N))]` — 提高对齐

```rust
// 强行设置结构体的对齐值：
#[repr(align(64))]
struct CacheLine {
    data: [u8; 64],
}

use std::mem;
assert_eq!(mem::align_of::<CacheLine>(), 64);  // 强制 64 字节对齐
assert_eq!(mem::size_of::<CacheLine>(), 64);   // 大小不变（刚好 64）
// 用途：避免 false sharing、SIMD 对齐要求

// 对齐可以比默认大，但不能比字段的对齐小：
// #[repr(align(1))]
// struct Bad { x: u32 }   // ❌ 编译错误——u32 需要 4 字节对齐
// 你不能降低字段的对齐要求
```

### 4.2 `#[repr(packed)]` — 紧凑布局（危险）

```rust
// packed：移除所有填充，字段紧密排列
#[repr(packed)]
struct Packed {
    a: u8,   // offset 0, size 1
    b: u32,  // offset 1, size 4  ← 未对齐！b 在奇数地址
    c: u16,  // offset 5, size 2  ← 也未对齐！
    // total: 7 bytes（没有尾部填充）
}

use std::mem;
assert_eq!(mem::size_of::<Packed>(), 7);
assert_eq!(mem::align_of::<Packed>(), 1);  // 整体对齐降为 1

// ⚠️ 危险：
let p = Packed { a: 1, b: 2, c: 3 };
// let r = &p.b;  // ⚠️ 这是对未对齐内存的引用！
//                   在 x86 上"可行"但慢，在 ARM 上可能崩溃
// Rust 编译器会对此发出警告（或错误）

// 如果你需要 packed 的引用——用 unsafe 和 ptr::read_unaligned：
let b_val = unsafe { std::ptr::addr_of!(p.b).read_unaligned() };
```

### 4.3 `repr(C, packed)` — C 兼容 + 紧凑

```rust
// 同时指定 C 和 packed：
#[repr(C, packed)]
struct Cpacked {
    a: u8,
    b: u32,
    c: u16,
}

// 用途：解析网络协议包、二进制文件格式
// 风险：未对齐字段 → unsafe 访问
```

## 5. 编译器自动重排字段

### 5.1 repr(Rust) 的优化能力

```rust
// repr(Rust) 下编译器会重排字段以最小化 size：

struct Unoptimized {
    a: u8,    // align 1
    b: u64,   // align 8
    c: u8,    // align 1
    d: u32,   // align 4
    e: u8,    // align 1
}
// 如果按声明顺序：1 + 7(pad) + 8 + 1 + 3(pad) + 4 + 1 + 7(pad) = 32 bytes

// repr(Rust) 自动优化：
// 编译器可能重排为：b(u64) + d(u32) + a(u8) + c(u8) + e(u8) + pad
//                   8     + 4        + 1     + 1     + 1     + 1(pad) = 16 bytes
// 节省 50%！

// 验证（不保证——可能不同编译器版本有不同结果）：
use std::mem;
// 至少在优化后 size ≤ 在未优化版本
```

### 5.2 自动重排的局限性

`repr(Rust)` 的自动重排优化有一定的边界限制。编译器不会改变：枚举的判别（discriminant）位置、trait object 的 vtable 指针、DST 的元数据位置。优化也不能穿透 `unsafe` 代码的假设（因为 `unsafe` 代码可能依赖特定布局），并且 `repr(C)` 或 `repr(packed)` 会明确禁用这些优化。

## 6. 常见陷阱

### 陷阱 1：假设 repr(Rust) 的布局是稳定的

```rust
// ❌ 序列化 repr(Rust) 结构体是危险的
struct Config {
    enabled: bool,
    port: u16,
    host: [u8; 16],
}

// let bytes: &[u8] = unsafe { std::slice::from_raw_parts(
//     &config as *const Config as *const u8,
//     std::mem::size_of::<Config>(),
// ) };
// ❌ bytes 的内容没有稳定保证！字段顺序可能不同
// 下次编译或不同版本可能得到不同的 bytes

// ✅ 序列化时用 serde 或手动指定字段顺序
```

### 陷阱 2：repr(packed) + 引用 = UB

```rust
#[repr(packed)]
struct Packed {
    x: u32,
    y: u8,
}

let p = Packed { x: 42, y: 1 };
// let r: &u32 = &p.x;  // ⚠️ 未对齐引用 → 可能 UB！

// ✅ 安全方式：用 ptr::addr_of! + read_unaligned
let x_val = unsafe { std::ptr::addr_of!(p.x).read_unaligned() };
```

### 陷阱 3：ZST（零大小类型）在对齐中的作用

```rust
// ZST 占用 0 字节但不影响对齐——除了作为字段影响整体对齐？
struct WithZst {
    x: u32,
    y: (),      // ZST —— 大小 0，但对齐是 1
}

// () 的对齐是 1——不影响结构体对齐
assert_eq!(mem::align_of::<WithZst>(), 4);  // 和 u32 一样

// 但 #[repr(align(8))] 的 ZST 会影响整体对齐：
#[repr(align(8))]
struct Align8Zst;  // size=0, align=8

struct WithAlignZst {
    x: u32,
    y: Align8Zst,  // size=0, align=8
}
assert_eq!(mem::align_of::<WithAlignZst>(), 8);  // ZST 的对齐影响结构体！
```

## 8. 练习答案

### 练习 1 答案

```
(a) A { u8, u16, u8 }
    u8  @0 (1B) + pad(1B)
    u16 @2 (2B)
    u8  @4 (1B) + pad(1B) → total 6, align 2
    size=6, align=2

(b) B { u8, u32, u8, u64 }
    u8  @0 (1B) + pad(3B)
    u32 @4 (4B)
    u8  @8 (1B) + pad(7B)
    u64 @16 (8B) → total 24, align 8
    size=24, align=8

(c) C { u8, [u8; 3], u32 }
    u8   @0 (1B)
    [u8;3] @1 (3B)  — align 1, 不产生填充
    u32  @4 (4B)    — 4 是 4 的倍数!
    → total 8, align 4
    size=8, align=4

(d) D { u8, &i32, u8 }
    u8  @0 (1B) + pad(7B)
    &i32 @8 (8B)  — 引用在 64 位系统是 8 字节，align 8
    u8  @16 (1B) + pad(7B) → total 24, align 8
    size=24, align=8
```

### 练习 2 答案

```rust
// 原始：a(u8)+b(u64)+c(u8)+d(u16)+e(u32)+f(u8)
// 优化排列：最大对齐的字段放前面
#[repr(C)]
struct Optimized {
    b: u64,  // offset 0, 8B
    e: u32,  // offset 8, 4B
    d: u16,  // offset 12, 2B
    a: u8,   // offset 14, 1B
    c: u8,   // offset 15, 1B
    f: u8,   // offset 16, 1B
    // pad: 7B → total 24, align 8
}

// 24 字节 vs 未优化的约 40 字节——省了约 40%
```

### 练习 3 答案

```
(1) ❌ 不安全。repr(Rust) 下字段顺序无保证——偏移可能不对
(2) ✅ 安全。repr(C) 布局与 C 兼容——就是为这个场景设计的
(3) ❌ 不可靠。repr(Rust) 可能重排字段、插入填充——不保证连续性
(4) ❌ 危险。除非是 repr(C) 且所有字节都有效——否则容易 UB
```

### 练习 4 答案

```rust
impl NetworkPacket {
    fn version(&self) -> u8 {
        unsafe { std::ptr::addr_of!(self.version).read_unaligned() }
    }

    fn set_version(&mut self, v: u8) {
        unsafe { std::ptr::addr_of_mut!(self.version).write_unaligned(v); }
    }

    fn length(&self) -> u16 {
        unsafe { std::ptr::addr_of!(self.length).read_unaligned() }
    }

    fn set_length(&mut self, l: u16) {
        unsafe { std::ptr::addr_of_mut!(self.length).write_unaligned(l); }
    }
}
// read_unaligned / write_unaligned 专门处理未对齐访问
```

## 9. 深度扩展

### 9.1 C/C++ 的内存布局——相同的规则，不同的默认值

```cpp
// C/C++ 的默认布局 = Rust 的 repr(C)（不是 repr(Rust)）
// C++ 也需要显式 packed/alignas

struct Foo {
    uint8_t  a;   // offset 0
    // 3 bytes padding
    uint32_t b;   // offset 4
    uint16_t c;   // offset 8
    // 2 bytes padding → 12 bytes total
};

// C/C++ 编译器也能重排字段——如果启用了优化：
// gcc -fpack-struct 或 __attribute__((packed))
// 但默认不会——这与 Rust 相反

// C++ 的对齐控制：
// alignas(64) struct CacheAligned { char data[64]; };
// struct __attribute__((packed)) Packed { ... };

// 核心差异：
// - C++ 默认 = Rust 的 #[repr(C)]（字段顺序保证）
// - Rust 默认 = repr(Rust)（字段顺序无保证，编译器可优化）
// - 这反映了两种语言的默认哲学：
//   C++ 假设你需要 C 兼容（保守默认）
//   Rust 假设你不需要 C 兼容（优化默认）
```

### 9.2 如果没有 `#[repr(C)]`——FFI 的全部痛苦

假设 Rust 只有 `repr(Rust)` 而没有 `repr(C)`，你要和 C 交互。C 侧有 `struct CStruct { int x; double y; };`，布局是 `x@0, padding@4, y@8`。但 Rust 侧如果用 `repr(Rust)` 定义：

```rust
struct RustStruct {
    y: f64,   // repr(Rust) 可能把 y 放前面！
    x: i32,   // repr(Rust) 可能把 x 放后面！
}
```

把这个 `RustStruct` 传给 C 会导致字段不对应，造成逻辑错误乃至未定义行为。唯一的解法是手工计算偏移、写成两个 `u64`、自己序列化……这就是为什么 `repr(C)` 不是"可选"的——它是 FFI 的必需品。

### 9.3 `std::alloc::Layout` —— 动态布局

```rust
use std::alloc::Layout;

// Layout 是编译期不知道大小/对齐时使用的
let layout = Layout::new::<u32>();  // 静态布局
assert_eq!(layout.size(), 4);
assert_eq!(layout.align(), 4);

// 动态计算布局（大小在运行时才知道）：
let layout = Layout::array::<u32>(100).unwrap();
assert_eq!(layout.size(), 400);  // 100 * 4

// Layout 还支持：
let layout = Layout::from_size_align(1024, 64).unwrap();
// 动态指定大小和对齐
// 用于自定义 allocator、arena、内存池

// C++ 对比：
// C 的 malloc/free 不记录对齐——需要 aligned_alloc 或 posix_memalign
// C++ 的 operator new 支持对齐（C++17）
// Rust 的 Layout 统一了大小和对齐——更清晰的 API
```

### 9.4 Niche 优化与内存布局的关系

Niche 优化利用类型中"不可能的值"来节省空间。以 `OptionBool` 为例：`bool` 只有 0 和 1 两个有效值，但底层的 `u8` 有 256 个值，因此 2-255 都是"不可能的"。编译器可以利用这一点，用 2 表示 `None`，用 0/1 表示 `Some(false)`/`Some(true)`。结果 `OptionBool` 的大小仅为 1 字节，而不是 2 字节（判别 + bool）。

Niche 检测仅限于 `repr(Rust)`——`repr(C)` 禁用 niche 优化以保持与 C 的兼容性，`repr(packed)` 也会禁用。作为对比，C++ 没有 Niche 优化，`std::optional<bool>` 至少占用 2 字节。

### 9.5 `repr(transparent)` —— 单字段类型的零成本包装

```rust
// #[repr(transparent)] 保证单字段类型与内部类型布局完全相同：

#[repr(transparent)]
struct Wrapper<T>(T);

// 保证：
// - size_of::<Wrapper<T>>() == size_of::<T>()
// - align_of::<Wrapper<T>>() == align_of::<T>()
// - Wrapper<T> 可以通过 transmute 与 T 互转

// 用途：newtype 模式、FFI 包装、安全抽象

// 对比：
// - repr(C) 不保证单字段类型与内部类型布局相同（可能添加填充）
// - repr(Rust) 不保证（编译器可能优化）
// - repr(transparent) 是唯一保证"A 就是 B"的 repr
```

### 9.6 枚举的布局——discriminant 和变体

```rust
// 枚举的内存布局：

// 简单枚举（无数据）：
enum Color { Red, Green, Blue }
// size = 1（u8 足够 3 个变体）
// 默认从 0 开始：Red=0, Green=1, Blue=2

// 带数据的枚举：
enum Value {
    Int(i32),       // variant 0
    Float(f64),     // variant 1
    Str(String),    // variant 2
}
// 布局（repr(Rust)）：
// discriminant: usize（或更小，如果只有几个变体）
// data: union of { i32, f64, String }
// size = max(discriminant + pad, max_variant_size)
// 编译器可能把 discriminant 塞进 padding 或使用 niche

// 指定 discriminant：
#[repr(u8)]
enum Status {
    Ok = 200,
    NotFound = 404,
    Error = 500,
}
// size = 1（u8 discriminant）

// C 风格的枚举也用 #[repr(C)]：
#[repr(C)]
enum CEnum {
    A(i32),
    B(f64, i32),
}
// 布局：tag(usize) + union —— 与 C 兼容
```

### 9.7 虚表（vtable）的内存布局

trait object（如 `&dyn Display`）在内存中由两个指针组成：数据指针 `data_ptr: *const ()` 和虚表指针 `vtable: *const VTable`。VTable 的结构（编译器内部）包含了析构函数、值的大小与对齐信息，以及各个 trait 方法的函数指针（按方法声明顺序排列）。

vtable 的布局是稳定的 ABI（application binary interface），这意味着来自不同 crate 的 trait object 可以互相兼容。

与 C++ 对比：C++ 的 vtable 也有固定的布局（取决于 ABI 标准，如 Itanium C++ ABI），但 C++ 的 vtable 包含了 `type_info`（RTTI）、虚基类偏移等更多信息。Rust 的 vtable 更加轻量——只包含必要的信息。

```rust
struct VTable {
    drop_in_place: fn(*mut ()),     // 析构函数
    size: usize,                    // 值的大小
    align: usize,                   // 值的对齐
    // ...然后是各个 trait 方法的函数指针（按方法声明顺序）
    fmt: fn(*const (), &mut Formatter<'_>) -> fmt::Result,
}
```

### 9.8 `std::mem::transmute` 与布局假设

```rust
// transmute 的合法性取决于布局：

// ✅ 安全：repr(transparent) 保证布局相同
#[repr(transparent)]
struct Wrapper(u32);
let w = Wrapper(42);
let u: u32 = unsafe { std::mem::transmute(w) };

// ❌ 危险：repr(Rust) 不保证布局
struct Foo { x: u32, y: u16 }
let bytes: [u8; 8] = unsafe { std::mem::transmute(Foo { x: 1, y: 2 }) };
// bytes 的内容没有稳定保证

// transmute 的黄金规则：
// 只有两者 size 相同 + 目标类型的每个位模式都是有效的 + 布局已知
// 才能安全 transmute
// repr(C) 和 repr(transparent) 是确保布局已知的关键
```

## 10. 回顾

```
内存布局与对齐学习清单：

□ 理解对齐的基本概念：类型的起始地址必须是对齐值的倍数
□ 知道 repr(C) 的规则：字段声明顺序、自然对齐、尾部填充
□ 知道 repr(Rust) 的规则：编译器可重排字段——无顺序保证
□ 理解 repr(align(N)) 提高对齐、repr(packed) 移除填充
□ 知道 repr(packed) 的危险：未对齐引用 = UB
□ 了解编译器的自动重排优化（节省内存）
□ 知道何时需要 repr(C)：FFI、序列化、裸指针操作
□ 理解 C/C++ 默认 = repr(C)、Rust 默认 = repr(Rust) 的哲学差异
```

> **一句话总结**：默认的 `repr(Rust)` 给编译器最大自由——重排字段、利用 niche、优化大小，但字段顺序没有保证（不能用于 FFI）。`#[repr(C)]` 保证 C 兼容的确定性布局（声明顺序 + 自然对齐填充），是 FFI 的生命线。C++ 默认就是 C 布局（字段顺序保证），Rust 默认是优化布局（性能优先）——这是两种语言"兼容优先"vs"优化优先"哲学的体现。
