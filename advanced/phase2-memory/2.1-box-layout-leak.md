# 2.1 Box 内存布局 & Box::leak

> `Box<T>` 看起来简单——"就是把值放在堆上"。但它的内部有大量精妙设计：它是唯一"拥有堆分配但不可复制"的指针；`Box<dyn Trait>` 自动变成胖指针带 vtable；`Box::leak` 能故意泄漏出 `&'static mut T`；而 `Box<[T]>` 知道自己的长度却不存额外信息。理解 Box 就是理解 Rust 堆分配的全部核心机制。

---

## 1. Box 的基本语义——"唯一的拥有型堆指针"

### 1.1 Box 的三个独特性

```rust
// Rust 的指针类型光谱：
// &T       — 共享引用，不拥有，栈上
// &mut T   — 独占引用，不拥有，栈上
// Rc<T>    — 共享所有权，堆分配，可复制
// Arc<T>   — 共享所有权（线程安全），堆分配，可 Clone
// Box<T>   — 独占所有权，堆分配，不可 Clone/Copy

// Box<T> 的独特性：
// 1. 拥有 T —— Box 被 drop 时，T 也被 drop
// 2. 不可复制 —— 没有 Clone/Copy（除非 T: Clone 且显式 .clone()）
// 3. 独占 —— 只有一个所有者
// 4. 大小固定 —— 总是指针大小（usize），即使 T 是 DST

let b: Box<i32> = Box::new(42);
// 栈上：b 占 8 字节（64 位）= 指向堆的指针
// 堆上：42 占 4 字节

// 移动 Box 只移动指针，不移动堆数据：
let b2 = b;  // 栈上的指针被复制，堆上的 42 还在原地
// b 不再可用——所有权已转移
```

### 1.2 Box 的内存布局

```rust
// Box<T> 的布局（当 T: Sized）：
//
//   栈                         堆
//  ┌───────────┐             ┌────────┐
//  │   ptr     │────────────→│   T    │
//  │ (8 bytes) │             │(N bytes)│
//  └───────────┘             └────────┘
//
// Box<T> 本身只有一个裸指针的大小——总是 usize

use std::mem::size_of;

assert_eq!(size_of::<Box<i32>>(), 8);          // 指针
assert_eq!(size_of::<Box<[i32; 1000]>>(), 8);  // 还是指针
assert_eq!(size_of::<Box<String>>(), 8);       // 仍是指针
```

### 1.3 Box 不是智能指针——越少越好的设计

```rust
// C++ 有 std::unique_ptr、std::shared_ptr、std::weak_ptr
// 每种都有很多方法（get、reset、release、swap、owner_before...）

// Rust 只有 Box<T>——它实现了 Deref 和 Drop，除此之外没有多余的方法
// 原因：Box 只负责两件事
// 1. 在堆上分配 T
// 2. 当 Box 离开作用域时释放 T

// Box<T> 不提供：
// - 无"空状态"(没有 null——用 Option<Box<T>>)
// - 无自定义 deleter（用 Drop trait）
// - 无 release/swap/set 等方法（用标准库函数或 DerefMut）
```

## 2. Box 的胖指针形态

### 2.1 `Box<dyn Trait>` —— 胖指针：ptr + vtable

```rust
use std::fmt::Display;

let s: Box<dyn Display> = Box::new(String::from("hello"));

// Box<dyn Display> 的内存布局（胖指针——16 字节）：
//
//   栈（16 bytes）             堆
//  ┌─────────────────┐       ┌──────────────┐
//  │  data_ptr (8B)  │──────→│ "hello" +    │
//  │                 │       │ 容量 + 长度  │
//  ├─────────────────┤       └──────────────┘
//  │  vtable_ptr (8B)│───┐
//  └─────────────────┘   │
//                        │    ┌──────────────────┐
//                        └───→│ drop: fn(*mut ()) │
//                             │ size: usize       │
//                             │ align: usize      │
//                             │ fmt: fn(...)      │
//                             └──────────────────┘

use std::mem::size_of;
assert_eq!(size_of::<Box<dyn Display>>(), 16);  // 两个指针
assert_eq!(size_of::<Box<String>>(), 8);         // 一个指针
```

**vtable 包含的内容**：

```rust
// vtable 实际上是一个结构体（编译器生成），包含：
// - drop: fn(*mut ())          — 析构函数
// - size: usize                — 值的大小
// - align: usize               — 值的对齐
// - method_ptrs: [...]         — trait 方法的函数指针
//
// Box<dyn Display> 的 vtable 包含：
// - drop for String
// - size = std::mem::size_of::<String>()
// - align = std::mem::align_of::<String>()
// - fmt: <String as Display>::fmt
```

### 2.2 `Box<[T]>` —— 胖指针：ptr + len

```rust
// Box<[T]> 是"拥有堆分配但大小未知"的切片

// 从 Vec 转换：
let v = vec![1, 2, 3, 4, 5];
let b: Box<[i32]> = v.into_boxed_slice();

// Box<[i32]> 的内存布局（胖指针——16 字节）：
//
//   栈（16 bytes）           堆
//  ┌─────────────────┐     ┌───┬───┬───┬───┬───┐
//  │  data_ptr (8B)  │────→│ 1 │ 2 │ 3 │ 4 │ 5 │
//  ├─────────────────┤     └───┴───┴───┴───┴───┘
//  │  len: 5   (8B)  │
//  └─────────────────┘

use std::mem::size_of;
assert_eq!(size_of::<Box<[i32]>>(), 16);  // 胖指针
assert_eq!(size_of::<Box<Vec<i32>>>(), 8); // 普通指针

// Box<[T]> vs Vec<T>：
// - Vec<T> 是 {ptr, len, cap} — 24 字节（可以增长）
// - Box<[T]> 是 {ptr, len} — 16 字节（固定大小）
// - Box<[T]> 不能增长（没有 cap）——但省了 8 字节
```

### 2.3 其他胖指针对比

```rust
// Rust 中的胖指针光谱：

// &[T] / Box<[T]>          → {ptr, len}      = 16 字节   (切片)
// &dyn Trait / Box<dyn Trait> → {ptr, vtable} = 16 字节  (trait object)
// &str                      → {ptr, len}      = 16 字节   (字符串切片)
// &Path                     → {ptr, len}      = 16 字节   (路径切片)
// &T (Sized)                → {ptr}           = 8 字节    (普通引用)

// 胖指针的共同特征：
// 1. 指针本身大于 usize（通常是 2*usize）
// 2. 包含了在编译期大小未知的类型所需的元数据
// 3. 都是 DST（dynamically sized type）的引用/指针
```

## 3. Box::leak —— 故意泄漏

### 3.1 基本用法

```rust
// Box::leak 消耗 Box 并返回 &'static mut T
let b = Box::new(String::from("hello"));
let leaked: &'static mut String = Box::leak(b);

// leaked 现在是 'static 生命周期的可变引用
// 这个 String 永远不会被释放——程序结束时由 OS 回收
// 用途：全局配置、单例、FFI 传给 C 的长生命周期数据
```

### 3.2 什么时候用 Box::leak

```rust
// 场景 1：初始化一次、使用全程的全局状态
use std::collections::HashMap;

fn get_global_config() -> &'static HashMap<String, String> {
    let mut config = HashMap::new();
    config.insert("host".into(), "localhost".into());
    config.insert("port".into(), "8080".into());
    Box::leak(Box::new(config))
}

// 场景 2：C FFI —— C 期望一个永远有效的指针
// extern "C" {
//     fn register_callback(cb: *const ());
//     fn set_global_buffer(buf: *mut u8, len: usize);
// }

fn setup_ffi_buffer(data: Vec<u8>) {
    let buf: Box<[u8]> = data.into_boxed_slice();
    let leaked: &'static mut [u8] = Box::leak(buf);
    // 现在可以安全地把指针传给 C，知道它不会被 Rust 释放
    // unsafe { set_global_buffer(leaked.as_mut_ptr(), leaked.len()); }
}

// 场景 3：在嵌入式/无 OS 环境中手动管理内存
// 在这些环境中，你不需要"归还"内存——泄漏是故意的设计
```

### 3.3 Box::leak 与 Drop 的交互

```rust
// Box::leak 阻止了 Drop 被调用！
struct HasDrop {
    data: String,
}

impl Drop for HasDrop {
    fn drop(&mut self) {
        println!("Dropping: {}", self.data);
    }
}

{
    let b = Box::new(HasDrop { data: "important".into() });
    let _leaked = Box::leak(b);
    // HasDrop 的 drop 不会被调用！
    // 没有 "Dropping: important" 输出
}  // _leaked 是 &'static mut T——离开作用域不触发任何析构

// ⚠️ 所以 Box::leak 是"泄漏资源"——包括内存和析构逻辑
// 只在你知道"这个资源需要活到程序结束"时使用
```

## 4. Box 的内部实现原理

### 4.1 简化版 Box 实现

```rust
use std::alloc::{Layout, alloc, dealloc};
use std::ops::{Deref, DerefMut};

struct MyBox<T> {
    ptr: *mut T,
}

impl<T> MyBox<T> {
    fn new(value: T) -> Self {
        // 在堆上分配内存
        let layout = Layout::new::<T>();
        let ptr = unsafe { alloc(layout) as *mut T };

        if ptr.is_null() {
            // 分配失败——Rust 默认 abort（不是返回 null）
            std::alloc::handle_alloc_error(layout);
        }

        // 将值写入堆
        unsafe { ptr.write(value); }

        MyBox { ptr }
    }
}

impl<T> Deref for MyBox<T> {
    type Target = T;
    fn deref(&self) -> &T {
        unsafe { &*self.ptr }
    }
}

impl<T> DerefMut for MyBox<T> {
    fn deref_mut(&mut self) -> &mut T {
        unsafe { &mut *self.ptr }
    }
}

impl<T> Drop for MyBox<T> {
    fn drop(&mut self) {
        let layout = Layout::new::<T>();
        unsafe {
            // 先析构值
            self.ptr.drop_in_place();
            // 再释放内存
            dealloc(self.ptr as *mut u8, layout);
        }
    }
}
```

### 4.2 Box 与编译器优化

```rust
// Box<T> 的引用和 &mut T 的引用可能指向同一内存吗？
// 不能——Box<T> 是独占所有者。这是"noalias"保证的来源

fn takes_box(b: &mut Box<i32>) {
    // 编译器知道：只有 Box 拥有堆上的 i32
    // 没有任何其他引用指向那个 i32（除非你用 unsafe 创建了别名）
    // → 可以应用更激进的优化
}

// 这被 LLVM 编码为 noalias（restrict）注解
// 对比原始指针 *mut T —— 编译器不做 noalias 保证
```

## 5. 常见陷阱

### 陷阱 1：误以为 `Box<T>` 可以 Copy

```rust
let b1 = Box::new(42);
let b2 = b1;           // ❌ b1 被 move
// println!("{}", b1); // 编译错误：use of moved value

// Box<T> 是 Move 类型——即使 T 是 Copy
// 如果 T: Clone，你可以：
let b3 = Box::new(42);
let b4 = b3.clone();  // ✅ 在堆上新分配一份
```

### 陷阱 2：`Box<[T]>` 的内存不能增长

```rust
let v = vec![1, 2, 3];
let b: Box<[i32]> = v.into_boxed_slice();
// b.push(4);  // ❌ Box<[i32]> 没有 push 方法——没有 cap
// 如果需要对切片做变长操作，保持 Vec<T>
```

### 陷阱 3：Box<dyn Trait> 无法向下转型到具体类型

```rust
// Box<dyn Any> 可以向下转型，但普通 Box<dyn Trait> 不行

let b: Box<dyn std::fmt::Display> = Box::new(42i32);
// let i: Box<i32> = b.downcast::<i32>().unwrap();  // ❌ 没有 downcast 方法
// downcast 只在 Box<dyn Any> 上可用

// 如果你需要向下转型，用 Box<dyn Any>：
let b: Box<dyn std::any::Any> = Box::new(42i32);
let i: Box<i32> = b.downcast::<i32>().unwrap();
```

### 陷阱 4：Box::leak 导致"不可见"的内存泄漏

```rust
// 在循环中使用 Box::leak 是危险的：
fn bad_pattern() {
    for i in 0..1_000_000 {
        let data = vec![0u8; 1024];  // 1KB
        let _leaked = Box::leak(Box::new(data));
        // 每次迭代泄漏 1KB——很快耗尽内存
    }
}
// Box::leak 只应该在"整个程序生命周期"的场景中使用
```

## 7. 练习答案

### 练习 1 答案

```
(a) Box<i32>           → 8 字节（普通指针）
(b) Box<[i32]>         → 16 字节（胖指针：ptr + len）
(c) Box<dyn Display>   → 16 字节（胖指针：ptr + vtable）
(d) Box<Vec<i32>>      → 8 字节（普通指针——Vec 内部已经包含指针）
(e) Box<Box<i32>>      → 8 字节（指针指向另一个指针）
(f) Option<Box<i32>>   → 8 字节（Niche 优化：全 0 表示 None，非零指针表示 Some）
```

### 练习 2 答案

```rust
use std::alloc::{Layout, alloc, dealloc};

struct MyBox<T> {
    ptr: *mut T,
    released: bool,
}

impl<T> MyBox<T> {
    fn new(value: T) -> Self {
        let layout = Layout::new::<T>();
        let ptr = unsafe { alloc(layout) as *mut T };
        assert!(!ptr.is_null());
        unsafe { ptr.write(value); }
        MyBox { ptr, released: false }
    }
}

impl<T> Drop for MyBox<T> {
    fn drop(&mut self) {
        if self.released {
            return;  // 已经手动释放——跳过
        }
        let layout = Layout::new::<T>();
        unsafe {
            self.ptr.drop_in_place();
            dealloc(self.ptr as *mut u8, layout);
        }
    }
}
// 关键：released 标志防止 double-free
```

### 练习 3 答案

```
创建 Box<dyn Animal>：

1. 用户写：let animal: Box<dyn Animal> = Box::new(Dog { name: "Rex".into() });
2. 编译器生成 Dog 的 vtable（包含 speak 函数指针、drop 函数指针、size、align）
3. Box::new 在堆上分配 sizeof(Dog) 字节，写入 Dog 实例
4. Box<dyn Animal> 在栈上存储 { data_ptr → 堆上的 Dog, vtable_ptr → Dog_vtable }

调用 animal.speak()：
1. 从 animal 的 vtable_ptr 找到 speak 的函数指针
2. 以 data_ptr 为 self 调用该函数指针
3. 调用发生在编译时确定的地址——没有额外的间接调用（除了 vtable 本身的一层）
```

### 练习 4 答案

```
(a) 临时 buffer → ❌ 不适合
   用普通 Vec<u8> 或 Box<[u8]>，让 RAII 自动回收

(b) 全局日志配置 → ✅ 合理
   初始化一次，整个程序生命周期使用，永不释放
   等价于 &'static 的语义

(c) C 回调 → ✅ 合理（最常见场景）
   C 库不知道 Rust 的借用规则——它需要一个永久有效的指针
   Box::leak 保证了这一点

(d) 每秒替换的大块数据 → ❌ 不适合
   旧数据会一直泄漏——用循环创建/替换数据时绝不用 Box::leak
   用普通 Box<T>，让旧数据被正确 drop
```

## 8. 深度扩展

### 8.1 C++ 的 unique_ptr vs Rust 的 Box——同源异构

```cpp
// C++ unique_ptr——Box<T> 的 C++ 对应物
#include <memory>

// 基本用法类似：
auto p1 = std::make_unique<int>(42);
auto p2 = std::move(p1);  // p1 变为 nullptr——转移所有权

// 但 C++ 有 Rust 没有的东西：
// 1. null 状态 —— unique_ptr 可以是 nullptr
auto p3 = std::unique_ptr<int>();  // nullptr
// Rust：没有 null——用 Option<Box<T>>

// 2. 自定义 deleter —— unique_ptr<int, MyDeleter>
// Rust：没有——Box 只能用全局 allocator（ nightly 有自定义 allocator）

// 3. release() —— 放弃所有权，返回裸指针
int* raw = p2.release();  // p2 不再管理内存——调用方负责 delete
// Rust：Box::into_raw(b) 返回 *mut T

// 4. 数组特化 —— unique_ptr<int[]>
auto arr = std::make_unique<int[]>(100);  // int[100]
// Rust：Box<[i32]> 等价，且通过胖指针自动存储长度

// 关键差异：
// - C++ 的 unique_ptr 可以"空"（nullptr）→ 每次使用前需要检查
// - Rust 的 Box 永远非空（除非用 unsafe）→ 编译器可以优化 noalias
// - C++ 的 move 后原变量仍存在（处于"合法但未指定"状态）
// - Rust 的 move 后原变量不可用（编译错误）
```

### 8.2 如果没有 `Box<T>`——如何手动管理堆分配

```rust
// 如果 Rust 没有 Box<T>，你需要手动做三件事：
// 1. 分配内存
// 2. 初始化值
// 3. 确保正确释放

use std::alloc::{Layout, alloc, dealloc};

// Rust 的解决方案（只关注正确性，不关注性能）：
fn on_heap_manual() {
    unsafe {
        // 1. 分配
        let layout = Layout::new::<String>();
        let ptr = alloc(layout) as *mut String;

        // 2. 初始化（写入值）
        ptr.write(String::from("hello"));

        // 3. 使用
        println!("{}", *ptr);

        // 4. 析构 + 释放
        ptr.drop_in_place();         // 先析构 String
        dealloc(ptr as *mut u8, layout); // 再释放内存
    }
}

// 问题：
// 1. 每一步都是 unsafe——容易出错
// 2. 在 2 和 4 之间 panic → 内存泄漏
// 3. 忘记 drop_in_place → String 内部的堆内存泄漏
// 4. 忘记 dealloc → 外层内存泄漏

// Box<T> 自动化了全部——零安全性开销
// 它用 Drop trait 保证"要么成功，要么不分配"
```

### 8.3 Box 与 allocator API——为什么 Rust 的分配是可替换的

Rust 的 `Box<T, A>` 在 nightly 上支持自定义 allocator。标准库的 `Global` allocator 实现了 `Allocator` trait，`Box<T, A>` 可以用第二个泛型参数指定分配器。

实际应用场景包括：内存池（arena allocator）实现批量分配/释放、共享内存（通过 mmap）、GPU 显存分配，以及嵌入式场景下的固定大小 bump allocator。

与 C++ 的 `std::allocator<T>` 和 `std::pmr`（polymorphic memory resource）对比，Rust 的 `#[global_allocator]` 全局替换机制更简单、更安全，但在细节丰富度上不如 C++17 的 pmr。

```rust
#![feature(allocator_api)]

use std::alloc::{Allocator, Global, Layout};
```

### 8.4 `Box<[T]>` 与切片 DST 的分配魔法

```rust
// Box<[T]> 怎么知道 T 的数量？
// 它使用了一个未稳定的机制：CoerceUnsized

// 简化理解：
let vec = vec![1, 2, 3];
let boxed: Box<[i32]> = vec.into_boxed_slice();

// Vec<T> 到 Box<[T]> 的转换不是简单的强制类型转换
// Vec 的 into_boxed_slice() 做了什么：
// 1. 拿到 {ptr, len, cap}
// 2. 如果 len < cap，用 realloc 缩小分配（shrink_to_fit）
// 3. 把 {ptr, len} 封装进 Box<[i32]>
// 4. 忘记 cap——Box<[T]> 不需要 cap

// Box<[T]> 分配大小 = len * size_of::<T>() 字节（精确的）
// Vec<T> 分配大小 >= cap * size_of::<T>() 字节（可能更大）
```

### 8.5 Box 与 Pin 的关系

```rust
use std::pin::Pin;

// Box<T> 是指向堆的指针——值在堆上
// 这意味着把值"移动"出 Box 需要显式操作（*box）
// → Box<T> 天然适合 Pin

// 创建 Pin<Box<T>>：
let pinned: Pin<Box<i32>> = Box::pin(42);
// Pin<Box<T>> 保证 T 不会被移动（因为堆上的位置固定）

// 这比 Pin<&mut T> 有优势：
// - Pin<Box<T>> 拥有 T——生命周期独立
// - Pin<&mut T> 借用 T——受限于栈帧生命周期

// async 块生成的 Future 通常被 Pin<Box<dyn Future>> 包裹
// 因为：
// 1. Future 可能是 !Unpin（自引用）
// 2. Pin<Box<...>> 可以放在 Vec 中或在线程间传递
// 3. Box 的大小固定（指针大小）——方便存储
```

### 8.6 Box<dyn Trait> 的 vtable 生成时机

```rust
// vtable 是什么时候生成的？
// 答：在编译期，为每个 (Trait, ConcreteType) 对生成一个 vtable

trait Animal {
    fn speak(&self) -> String;
    fn eat(&self);
}

struct Dog { name: String }

impl Animal for Dog {
    fn speak(&self) -> String { format!("Woof! I'm {}", self.name) }
    fn eat(&self) { println!("{} eats kibble", self.name); }
}

// 编译器生成的 vtable（伪代码）：
// const DOG_ANIMAL_VTABLE: AnimalVTable = AnimalVTable {
//     drop: <Dog as Drop>::drop,  // 或编译器生成的析构
//     size: std::mem::size_of::<Dog>(),
//     align: std::mem::align_of::<Dog>(),
//     speak: <Dog as Animal>::speak,
//     eat: <Dog as Animal>::eat,
// };

// 当你写：
let animal: Box<dyn Animal> = Box::new(Dog { name: "Rex".into() });
// 编译器使用 DOG_ANIMAL_VTABLE 初始化 Box 的 vtable 指针

// 每个 (trait, impl) 对生成一个 vtable
// 如果 Animal 有 5 个实现，就有 5 个 vtable
// 但同名 trait 的 vtable 布局相同（相同的方法顺序）——这是 ABI 兼容的基础
```

### 8.7 Box 与 zero-sized types (ZST)

```rust
// Box<()> —— ZST 的堆分配会发生什么？

let b = Box::new(());  // () 大小是 0

// Box<()> 的指针指向哪里？
// 答：指向一个"悬垂的"对齐地址（dangling alignment address）
// 具体来说，指向 NonNull::dangling()——一个满足对齐但不指向实际内存的指针

// 为什么这样设计？
// - 分配 0 字节是未定义行为（在某些 allocator 中）
// - 但仍然需要一个"非空"的指针给 Box
// - NonNull::dangling() 返回一个按 T 对齐的非零指针

// 所以 Box<()> 实际上不分配任何堆内存！
// Box<T>（其中 T 是 ZST）是一个特例——不调用 alloc

// C++ 对比：
// unique_ptr<void> 或 unique_ptr<int[0]> 的行为是实现定义的
```

### 8.8 Box::leak 的底层实现

```rust
// Box::leak 的近似实现：
impl<T: ?Sized> Box<T> {
    fn leak<'a>(b: Self) -> &'a mut T
    where
        T: 'a,  // 实际上 'a 会被推断为 'static（通过类型推断）
    {
        // 1. 获取裸指针
        let ptr = Box::into_raw(b);
        // 2. 从裸指针构造引用
        //    SAFETY: Box 保证 ptr 非空且有效
        unsafe { &mut *ptr }
        // 3. Box 被消耗了——没有 Drop 被调用
        //    因为 into_raw 不会调用 drop
    }
}

// 关键：Box::into_raw 消耗 Box 但不调用 T 的 Drop
// 返回的裸指针需要调用方负责管理（从 Box::from_raw 重建 Box 并触发 Drop）
// Box::leak 跳过了 from_raw——故意永久放弃所有权
```

## 9. 回顾

```
Box 内存布局 & Box::leak 学习清单：

□ 理解 Box<T> 的独占所有权语义：不可 Copy，只有一个所有者
□ 知道 Box<dyn Trait> 是胖指针（ptr + vtable），16 字节
□ 知道 Box<[T]> 是胖指针（ptr + len），16 字节
□ 理解 Box::leak 返回 &'static mut T 并跳过 Drop
□ 知道 Box::leak 的合理使用场景（全局状态、FFI、嵌入式）
□ 了解 Box 与 allocator API 的关系
□ 知道 ZST 的 Box 不分配实际内存
□ 理解 Box 与 C++ unique_ptr 的关键差异（null 状态、自定义 deleter）
```

> **一句话总结**：`Box<T>` 是 Rust 唯一的"拥有堆分配但不可复制"的智能指针——它在栈上只是一个指针，但通过胖指针机制（vtable/len）能处理 DST（`Box<dyn Trait>`、`Box<[T]>`），通过 `Box::leak` 能故意泄漏出 `&'static mut T`。和 C++ 的 `unique_ptr` 不同，Box 没有 null 状态（编译器因此能做 noalias 优化）、没有自定义 deleter——它的设计哲学是"只做必要的事，剩下的交给 Rust 的类型系统"。
