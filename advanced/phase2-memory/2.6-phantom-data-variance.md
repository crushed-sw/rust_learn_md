# 2.6 PhantomData & 型变——生命周期子类型的"方向"

> Rust 的类型系统在幕后悄悄做着一件事：**型变**（variance）。当你写 `fn foo<'a>(x: &'a str)` 时，编译器知道 `'long` 可以放进需要 `'short` 的位置——这是**协变**。当你写 `fn bar(f: fn(&'a str))` 时，参数生命周期只能"拉长"不能"缩短"——这是**逆变**。而 `&'a mut T` 既不能缩短也不能拉长——这是**不变**。理解型变是编写正确的 unsafe 代码、理解 `PhantomData` 的全部用途、以及理解为什么 `Cell<T>` 是不变的——的钥匙。

---

## 1. 型变——生命周期的子类型关系

### 1.1 什么是子类型

```rust
// Rust 中的子类型关系只存在于生命周期之间：
// 'long: 'short 表示 'long 是 'short 的子类型
// → 'long 的值可以用在需要 'short 的地方

// 例如：
let long_str: &'static str = "hello";  // 'static 是最大的生命周期
{
    let short_str: &str = long_str;     // ✅ 'static 可以当短生命周期用
    // 'static 是 'short 的子类型
}
```

### 1.2 型变——子类型关系的传播

型变描述的是：当类型参数有子类型关系时，复合类型如何传播这个关系。Rust 中有三种型变：

- **协变（Covariant）：** 保持子类型关系的方向。例如 `&'a T` 在 `'a` 上是协变的——如果 `'long: 'short`，则 `&'long T: &'short T`。
- **逆变（Contravariant）：** 反转子类型关系的方向。例如 `fn(T)` 在参数上是逆变的——如果 `'long: 'short`，则 `fn(&'short T): fn(&'long T)`（注意方向反转了）。
- **不变（Invariant）：** 不传播子类型关系。例如 `&'a mut T` 在 `'a` 上是不变的——即使 `'long: 'short`，也不能用 `&'long mut T` 替代 `&'short mut T`。

---

## 2. 协变、逆变、不变——详细规则

### 2.1 协变（Covariant）——最普遍

```rust
// 以下类型在 T 上是协变的：
// &'a T          — 共享引用
// Box<T>         — 拥有型指针
// Vec<T>         — 容器
// Option<T>      — Option
// [T; N]         — 固定数组

// 协变意味着：T 的子类型关系正向传播
fn accept_ref<'a>(r: &'a i32) {
    println!("{}", r);
}

let x: &'static i32 = &42;  // 'static 是 'a 的子类型
accept_ref(x);               // ✅ 可以传——&'static i32 是 &'a i32 的子类型

// 实际上 Rust 编译器不显式"转换"——它只是允许子类型出现在需要父类型的位置
```

### 2.2 逆变（Contravariant）——函数参数的例外

```rust
// 函数指针在参数上是逆变的：

fn takes_fn(f: fn(&'static str)) {
    f("hello");
}

// 你可以传一个接受 "任何生命周期" 的函数的指针：
fn my_func(s: &str) {  // 隐式 for<'a> fn(&'a str)——接受任意生命周期
    println!("{}", s);
}

takes_fn(my_func);  // ✅ 可以传！
// fn(&str) 是 fn(&'static str) 的子类型（逆变！）
// 因为 fn(&str) 可以接受 &'static str，反之不行

// 逆变的理解：
// "我需要的只是能接受 'static 的函数"
// "你给了我一个能接受**任何生命周期**的函数" → 更好！
// "子类型"关系方向：更强的函数 → 更弱的函数（在参数上是逆变的）
```

### 2.3 不变（Invariant）——可变引用的特殊性

`&'a mut T` 在 `'a` 上必须是不变的。来看一个例子：

```rust
fn accept_mut<'a>(r: &'a mut i32) -> &'a mut i32 {
    r
}

let mut x: i32 = 42;
{
    let r: &'static mut i32 = &mut x;
    // accept_mut(r);  // 如果允许协变，这会缩短生命周期
}
```

为什么 `&mut T` 必须是不变的？如果 `&'long mut T` 能协变为 `&'short mut T`，那么原始的长引用还在，缩短后的引用作为一个新的可变引用也存在，此时就有两个可变引用指向同一数据——直接违反了 Rust 的借用规则，会导致未定义行为。

## 3. PhantomData 控制型变

### 3.1 PhantomData 的型变行为

```rust
use std::marker::PhantomData;

// PhantomData 的型变取决于它的类型参数：

// PhantomData<T>         → 在 T 上是协变的
// PhantomData<fn(T)>     → 在 T 上是逆变的（函数参数）
// PhantomData<fn() -> T> → 在 T 上是协变的（函数返回值）
// PhantomData<fn(T) -> T> → 在 T 上是不变的（同时出现在参数和返回）

// 这让你可以精确控制自定义类型的型变！
```

### 3.2 用 PhantomData 设置不变

```rust
use std::marker::PhantomData;

// 你有一个包含裸指针的类型：
struct MySlice<'a, T> {
    ptr: *const T,
    len: usize,
    _marker: PhantomData<&'a T>,
    // &'a T 在 T 上是协变的，在 'a 上也是协变的
    // → MySlice<T> 在 T 上是协变的
    // 这和 &'a [T] 的型变一致（因为裸指针不是 T 的"拥有者"）
}

// 如果你需要不变的型变（类似 &'a mut T）：
struct MyMutSlice<'a, T> {
    ptr: *mut T,
    len: usize,
    _marker: PhantomData<&'a mut T>,
    // &'a mut T 在 'a 上是不变的，在 T 上也是不变的
    // → MyMutSlice 在 T 上是不变的
}
```

### 3.3 用 `fn(T) -> T` 控制不变性

```rust
use std::marker::PhantomData;

// 最常用的"强制不变"技巧：
struct Invariant<T> {
    _marker: PhantomData<fn(T) -> T>,
    // fn(T) -> T 在参数和返回值中都用了 T
    // → 在 T 上是不变的
}

// 用于需要"不变"的场景：
struct RawCell<T> {
    value: std::cell::UnsafeCell<T>,  // UnsafeCell 默认不变
    // 但如果你的类型没有 UnsafeCell 但需要不变行为：
    _invariant: PhantomData<fn(T) -> T>,
}
```

## 4. 为什么 Cell/RefCell 是不变的——实际案例

### 4.1 如果 Cell 是协变的——灾难

假设 `Cell<T>` 在 `T` 上是协变的，会发生什么？考虑以下步骤：

1. 先创建一个生命周期为 `'long` 的 `Cell<&'long str>`。
2. 由于协变，可以"缩短"为 `Cell<&'short str>`——编译器允许这种隐式转换。
3. 现在你有一个表面上是 `Cell<&'short str>` 的值，但内部实际存储的引用类型仍然是 `&'long str`。通过 `Cell::set` 放入一个 `&'short str` 的引用，原始的长引用就被短的覆盖了。当 `'short` 生命周期结束后，`Cell` 内部就存在悬垂引用。

这就是为什么 `Cell<T>` 在 `T` 上必须是不变的——它阻止了生命周期被"缩短"的可能性，保护了内存安全。

### 4.2 UnsafeCell 奠定了不变性的基础

```rust
// Cell<T> 内部是 UnsafeCell<T>
// UnsafeCell<T> 在 T 上是不变的——这是 Rust 语言层面的保证
// Cell/RefCell 继承了这一不变性

// 标准库的设计链条：
// UnsafeCell<T>（不变）
//   → Cell<T>（不变——继承自 UnsafeCell）
//   → RefCell<T>（不变——继承自 UnsafeCell）

// 这一不变性防止了通过协变绕过借用检查
```

## 5. 常见陷阱

### 陷阱 1：忘记标注 PhantomData 导致型变错误

```rust
// ❌ 忘记 PhantomData——裸指针不携带型变信息
struct MyRef<'a, T> {
    ptr: *const T,
    // 忘记 PhantomData！
}

// 问题：*const T 上没有生命周期信息
// MyRef<'a, T> 不会"绑定" 'a 和 T
// → MyRef 在 'a 和 T 上的型变取决于编译器推断——可能是错误的
// → 编译器会报 unused parameter 错误——但如果你 suppress 了这个错误：
//    MyRef { ptr, _unused: PhantomData } // 忘记给正确的型变标记

// ✅ 正确：
struct MyRef<'a, T> {
    ptr: *const T,
    _marker: PhantomData<&'a T>,  // 协变——类似 &'a [T]
}
```

### 陷阱 2：对型变理解不足导致的 unsafe 错误

```rust
// 你写的 unsafe 代码可能依赖特定的型变

// ❌ 如果类型是协变的，但你不期望它协变：
struct Cache<K, V> {
    data: HashMap<K, V>,
    // 你认为 Cache 在 K 上应该不变（类似 &mut HashMap<K, V>）
    // 但没有加 PhantomData 控制型变
}

// 你可能在 unsafe 代码中依赖"K 的类型不会变"
// 但编译器推断 Cache 在 K 上是协变的 → 潜在的 UB
```

### 陷阱 3：函数指针型变与闭包的差异

函数指针 `fn(T)` 在 `T` 上是逆变的，但闭包类型（如 `Fn(T)`）的型变由编译器推导决定。在编写涉及函数类型的 `unsafe` 代码时，需要特别注意：`fn(&'a str)` 中 `'a` 是逆变参数，而 `for<'a> fn(&'a str)` 是高级生命周期（HRTB），不依赖特定 `'a`。两者的型变行为不同——理解这个差异是正确编写 unsafe 代码的重要前提。

## 7. 练习答案

### 练习 1 答案

```
(a) &'a T         → 协变（共享引用——'long 可以缩短为 'short）
(b) &'a mut T     → 不变（可变引用——不能缩短或拉长）
(c) Box<&'a T>    → 协变（Box 在内部类型上是协变的）
(d) fn(&'a T) → &'a T → 不变（'a 在参数中是逆变的，在返回值中是协变的 → 合起来是不变的）
(e) *const &'a T  → 协变（裸指针本身在生命周期上不携带型变，
                   但内部 &'a T 是协变的——\n                    最终取决于指针被如何使用，通常由 PhantomData 决定）
```

### 练习 2 答案

```rust
use std::marker::PhantomData;

struct InvariantBox<T> {
    inner: Box<T>,
    _invariant: PhantomData<fn(T) -> T>,  // 强制 T 不变
}

impl<T> InvariantBox<T> {
    fn new(value: T) -> Self {
        InvariantBox {
            inner: Box::new(value),
            _invariant: PhantomData,
        }
    }
}
// fn(T) -> T 在 T 上既在参数位置（逆变）又在返回值位置（协变）
// → T 只能是不变的
// 效果：InvariantBox<&'long str> 不能当作 InvariantBox<&'short str> 使用
```

### 练习 3 答案

```rust
// assign 的签名：fn assign<T>(target: &mut T, source: T)
// 其中 target: &mut T 意味着 T 是不变的（因为 &mut T 在 T 上不变）

// 调用 assign(&mut x, y) 时：
// x: &str（其实 x 的类型是 &'x str，'x 是某个生命周期）
// y: &'static str
// T 被推断为 &'x str
// 但 y 是 &'static str —— T 应该也接受 'static
// 如果 &mut T 是协变的——可以
// 但 &mut T 是不变的 → 'x 和 'static 不兼容 → 编译错误

// 修复：显式标注生命周期使 T 在这两个位置一致
```

### 练习 4 答案

```rust
// 假设 Cell<T> 是协变的（实际上它是不变的）：
fn create<'a>(s: &'a str) -> &'a str {
    let cell: Cell<&'a str> = Cell::new(s);

    // 如果 Cell 是协变的：
    // 可以创建一个生命周期更短的 Cell：
    fn evil<'a, 'b>(cell: Cell<&'a str>) -> Cell<&'b str> {
        cell  // 协变允许 Cell<&'long str> → Cell<&'short str>
    }

    let short_cell: Cell<&'b str> = evil(cell);
    // short_cell 内部还是 &'a str，但类型系统认为它是 &'b str

    let dangling: &'b str = String::from("temp").as_str();
    short_cell.set(dangling);  // 把短引用写进 cell（覆盖了长引用）
    // 'b 结束后，short_cell 内部是悬垂引用

    // 这就是为什么 Cell 必须是不变的
    // 否则内部可变性 + 协变 = 悬垂引用
}
```

## 8. 深度扩展

### 8.1 C++ 的型变——模板参数、继承和 const 的交互

```cpp
// C++ 没有生命周期，所以型变体现在别的维度：

// 1. 指针的协变（const 方面）
//    int* 可以转为 const int*（协变）
//    int** 不能转为 const int**（为什么？）
int x = 42;
int* p1 = &x;
const int* p2 = p1;  // ✅ OK

// int** → const int** 的问题：
// const int** pp = &p1;  // ❌ 这是错误的——如果可以：
// *pp = &some_const_int;  // 通过 pp 修改了 p1
// 现在 p1 指向 const int——但 p1 是 int* ——可以修改！→ UB

// 2. 模板参数的型变
//    std::vector<T> 在 T 上是协变的吗？
//    class Base {};
//    class Derived : public Base {};
//    std::vector<Derived*> → std::vector<Base*>  // 不能直接转换
//    // C++ 的模板参数默认是不变的
//    // 除非使用 std::shared_ptr（支持协变转换）

// 3. shared_ptr 的协变
//    std::shared_ptr<Derived> → std::shared_ptr<Base>  // ✅ OK（通过 template ctor）
//    这是 C++ 唯一内置的智能指针协变

// Rust 对比：
// Rust 没有继承——型变只在生命周期层面存在
// 这简化了型变模型，但增加了 PhantomData 的需要
```

### 8.2 如果没有型变控制——全部默认不变或协变

假设 Rust 没有型变系统，所有类型参数都是协变的：`Cell<T>` 因为协变而可以通过生命周期缩短绕过借用规则，导致悬垂引用，内存安全被破坏。反过来，如果所有类型都是不变的，那么 `&'static str` 就不能传给需要 `&str` 的函数——每次调用都需要显式"缩短"生命周期，非常繁琐。

型变系统的价值在于它提供了四种精细的控制层次：协变作为默认（大多数类型是协变的，方便使用）；不变作为特例（保护内部可变性——`Cell`、`RefCell`、`UnsafeCell`、`&mut T`）；逆变作为另一种特例（让函数参数更灵活）；`PhantomData` 控制（允许自定义型变行为）。

### 8.3 PhantomData 的型变表

```
PhantomData<T>                  → 在 T 上是协变的
PhantomData<fn(T)>              → 在 T 上是逆变的
PhantomData<fn() -> T>          → 在 T 上是协变的
PhantomData<fn(T) -> T>         → 在 T 上是不变的（最常用）
PhantomData<Cell<T>>            → 在 T 上是不变的（继承 Cell 的型变）
PhantomData<*const T>           → 在 T 上是协变的
PhantomData<*mut T>             → 在 T 上是不变的

选择口诀：
- 想要协变 → PhantomData<T>
- 想要逆变 → PhantomData<fn(T)>
- 想要不变 → PhantomData<fn(T) -> T> 或 PhantomData<Cell<T>>
- 想要 &mut T 的不变 → PhantomData<&'a mut T>
```

### 8.4 型变检查——编译器如何推导型变

编译器通过分析类型的使用方式来自动推导型变。一个类型的型变等于其所有字段在对应参数上型变的"合取"：

- 如果所有字段都是协变的，则类型是协变的。
- 如果任意一个字段是不变的，则类型是不变的。
- 如果某个字段是逆变的而另一个是协变的，则类型是不变的（逆变 + 协变 = 不变）。

`PhantomData` 像普通字段一样参与型变推导，因此你可以通过 `PhantomData` 精确控制类型的最终型变行为。

```rust
struct MyType<'a, T> {
    x: &'a T,       // &T 在 T 上协变 → MyType 在 T 上协变
}

struct Mixed<'a, T> {
    a: fn(T),          // 逆变
    b: fn() -> T,      // 协变
    // 逆变 + 协变 → 不变！
}
```

### 8.5 型变与 trait object 的交互

```rust
// trait object 的型变：

// &'a dyn Trait —— 在 'a 上协变，在 Trait 参数上协变
// Box<dyn Trait + 'a> —— 在 'a 上协变

// 但 trait object 的生命周期 bound 影响 Send/Sync：
// dyn Trait + 'static: Send（如果 Trait: Send）
// dyn Trait + 'a:      Send（如果 Trait: Send + 'a 足够长）

// 型变让 trait object 的生命周期可以"缩短"：
fn use_dyn(x: &dyn Fn() -> i32) {
    x();
}

let closure: &'static dyn Fn() -> i32 = &|| 42;
use_dyn(closure);  // ✅ 'static 协变为更短的生命周期
```

### 8.6 `for<'a>` 高阶生命周期与型变

```rust
// for<'a> 创建了"对所有生命周期都成立"的类型：

// for<'a> fn(&'a str) → &'a str
// 这个函数在 'a 上是"全称量化"的——不涉及特定生命周期

// 与型变的交互：
fn takes_hr(f: for<'a> fn(&'a str) -> &'a str) {
    let result = f("hello");
    println!("{}", result);
}

fn specific_fn<'a>(s: &'a str) -> &'a str { s }

// takes_hr(specific_fn);  // ✅ specific_fn 可以视为 for<'a> 的实例
// 因为 specific_fn 对任何 'a 都有效

// 型变在这个场景中的角色：
// fn(&'static str) → &'static str 不能当作 for<'a> fn(&'a str) → &'a str
// 因为前者只对 'static 有效——不够"通用"
```

### 8.7 型变与 unsafe 代码

```rust
// unsafe 代码中，型变错误可能导致 undefine behavior：

struct UnsafeContainer<T> {
    ptr: *mut T,
}

// 裸指针不推导型变——编译器会回退到不变或协变（取决于具体类型）
// 如果你在 unsafe 代码中假设了错误的型变：

impl<T> UnsafeContainer<T> {
    fn get(&self) -> &T {
        unsafe { &*self.ptr }
    }

    fn set(&self, value: T) {
        // ⚠️ 如果 UnsafeContainer<T> 在 T 上是协变的
        // 且你用 set 替换了内部值——可能破坏生命周期假设
        unsafe { self.ptr.write(value); }
    }
}

// 在这种情况下，UnsafeContainer<T> 应该是不变的！
// 你需要 PhantomData<fn(T) -> T> 来强制执行
```

### 8.8 为什么 Rust 的型变只在生命周期上——类型没有子类型

Rust 不像 C++ 那样有基于继承的类型子类型关系。这带来了几个好处：

1. **不用继承，类型子类型关系不存在。** 型变只需要处理生命周期，模型更加简单。
2. **trait 系统提供了替代的多态机制。** `fn foo(x: &dyn Trait)` 通过 trait object 提供了类似"子类型"的效果，不需要继承链。
3. **生命周期型变更"窄"但更"安全"。** C++ 的型变可能导致 `std::vector<Derived*>` 不是 `std::vector<Base*>` 的子类型的混淆，程序员可能误用导致 UB。Rust 没有这个问题——生命周期型变由编译器自动处理，不会让程序员手动操作。
4. **未来前景。** 如果 Rust 引入了 delegation 或 specialization，型变系统可能需要扩展，但目前保持简单是明智的选择。

## 9. 回顾

```
PhantomData & 型变学习清单：

□ 理解型变的三种形式：协变（保持方向）、逆变（反转方向）、不变（禁止传播）
□ 知道 &T 协变、&mut T 不变、fn(T) 逆变
□ 理解为什么 Cell/RefCell 在 T 上必须是不变的
□ 掌握用 PhantomData 控制型变：
    - PhantomData<T> → 协变
    - PhantomData<fn(T)> → 逆变
    - PhantomData<fn(T) -> T> → 不变
□ 能判断自定义类型的型变并纠正错误
□ 理解 C++ 的型变维度（继承、const）和 Rust 的差异
□ 知道 unsafe 代码中错误的型变假设可能导致 UB
```

> **一句话总结**：型变（variance）描述了生命周期子类型关系如何在复合类型中传播——`&T` 协变（正向）、`&mut T` 不变（禁止传播）、`fn(T)` 逆变（反向）。`Cell<T>` 的不变性是防止"通过生命周期缩短绕过借用检查"的安全屏障。`PhantomData` 不仅是"占有类型标记"——它更重要的角色是通过 `PhantomData<fn(T)>` 和 `PhantomData<fn(T)->T>` 精确控制自定义类型的型变行为。
