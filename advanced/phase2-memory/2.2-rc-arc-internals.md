# 2.2 Rc/Arc 内部实现——引用计数在堆上的秘密

> `Rc::clone()` 看起来像是在"克隆"数据，但它实际上只是在增加一个整数——引用计数。这个计数不在栈上、不在 Rc 结构体内部，而是在**堆上紧挨着数据**存放。理解 `Rc` 和 `Arc` 的内部布局后，你会知道为什么 `Rc` 不能跨线程（非原子计数）、`Weak` 如何检测"悬垂"、`make_mut` 的写时复制何时真正触发克隆。这是理解 Rust 共享所有权的基石。

---

## 1. Rc 的内存布局——堆上的三位一体

### 1.1 基本布局

```rust
use std::rc::Rc;

let rc = Rc::new(String::from("hello"));

// Rc<String> 的内存布局：
//
//   栈（8 bytes）             堆
//  ┌─────────────┐          ┌──────────────────────────┐
//  │   ptr       │─────────→│  strong_count: 1 (usize)  │
//  │ (指向堆块)  │          │  weak_count:   0 (usize)  │
//  └─────────────┘          │  value: "hello"           │
//                            │  (String 的数据)          │
//                            └──────────────────────────┘
//
// Rc<T> 栈上只有一个指针（8 字节）
// 堆上包含：strong_count + weak_count + T 的数据

use std::mem::size_of;
assert_eq!(size_of::<Rc<String>>(), 8);  // 只一个指针
```

### 1.2 引用计数的位置

`Rc` 的堆分配结构在内部是一个 `RcBox<T>`，包含三个字段：`strong: Cell<usize>`（强引用计数）、`weak: Cell<usize>`（弱引用计数）和 `value: T`（实际数据）。`Rc<T>` 的指针指向 `RcBox<T>` 的起始位置，也就是 `strong` 字段的地址。在 64 位架构上，`value` 的偏移量等于两个 `usize` 的大小，即 16 字节。

这意味着当你调用 `Rc::clone()` 时，完全不需要访问 `T` 的数据本身——只需要在堆上增加 `strong_count`（对 `Rc` 而言是非原子操作），然后复制栈上的指针即可。

### 1.3 Rc::clone 的零拷贝性质

```rust
use std::rc::Rc;

let rc1 = Rc::new(vec![1u8; 1_000_000]);  // 1MB 数据在堆上
let rc2 = Rc::clone(&rc1);
//   ^^^^^^^^ 没有分配新内存——没有复制 1MB 的数据
//   ^^^^^^^^ 只增加了引用计数（堆上的一个整数）
//   ^^^^^^^^ 非常轻量！

// Rc::clone 做了什么（近似）：
// 1. RcBox.strong += 1  （通过 Cell::set，非原子地）
// 2. 复制 ptr 到新的 Rc

// 对比 Box<T>：
// Box<T> 没有 Clone——你不能"共享"所有权
```

## 2. Strong Count 和 Weak Count 的协作

### 2.1 引用计数的规则

```rust
use std::rc::{Rc, Weak};

let strong: Rc<i32> = Rc::new(42);
// strong_count = 1, weak_count = 0

let weak: Weak<i32> = Rc::downgrade(&strong);
// strong_count = 1, weak_count = 1

let strong2 = weak.upgrade().unwrap();
// strong_count = 2, weak_count = 1

drop(strong);
// strong_count = 1, weak_count = 1  ← 数据还在！

drop(strong2);
// strong_count = 0, weak_count = 1
// → 数据被析构，但 RcBox 的外壳还在！
// → 因为 weak_count > 0

drop(weak);
// weak_count = 0
// → RcBox 的外壳也被释放
```

### 2.2 当 strong_count == 0 时发生了什么

```rust
use std::rc::{Rc, Weak};

let weak: Weak<String> = {
    let strong = Rc::new(String::from("hello"));
    Rc::downgrade(&strong)
    // strong 离开作用域——strong_count → 0
    // String 被析构（内存被释放）
    // 但 RcBox 的"外壳"还在——因为 weak_count = 1
};

// weak.upgrade() 返回 None——数据已经没了
assert!(weak.upgrade().is_none());

// Weak 指针本身仍然有效（可以安全查询）
// 但不能再 upgrade 到 Rc
```

### 2.3 Weak 如何检测"悬垂"

`Weak::upgrade()` 的核心逻辑是先读取 `strong_count`，如果大于零就增加计数并返回 `Some(Rc { ptr })`，否则返回 `None`。

关键点在于：增加 `strong_count` 必须是"检查 + 增加"两步操作。在 `Rc`（单线程）中不存在并发问题——检查之后不会有其他线程修改 `strong_count`。但在 `Arc`（多线程）中，必须用原子操作完成：先做 `fetch_add(1)`，如果之前的值大于 0 则成功，否则需要减回去并返回 `None`。

```rust
impl<T> Weak<T> {
    fn upgrade(&self) -> Option<Rc<T>> {
        if self.inner().strong() > 0 {
            self.inner().inc_strong();
            Some(Rc { ptr: self.ptr })
        } else {
            None
        }
    }
}
```

## 3. Arc —— 线程安全的 Rc

### 3.1 唯一区别：原子 vs 非原子

`Rc` 和 `Arc` 唯一的语义区别在于引用计数的操作方式。`Rc<T>` 在单线程环境下使用 `Cell<usize>` 进行非原子读写，`Rc::clone()` 只做一次 `.set()` 操作，零同步开销。`Arc<T>` 在多线程环境下使用 `AtomicUsize` 进行原子读写（涉及内存屏障），`Arc::clone()` 执行一次 `fetch_add(1, Relaxed)`，有内存屏障开销。

性能差异显著：`Rc::clone` 大约是一条 `mov` 指令（非原子），而 `Arc::clone` 大约是一条 `lock inc` 指令（原子操作，约 20-50 个时钟周期）。在单线程中，`Rc` 的 clone 操作比 `Arc` 快约 5-10 倍。

### 3.2 Arc 的 Send/Sync 推导

```rust
use std::sync::Arc;
use std::rc::Rc;

// Arc<T> 是 Send ⇔ T: Send + Sync
//            （注意需要 Sync——为什么？）
//
// 因为多个线程可以持有同一个 Arc<T>，它们都可以通过 &T（共享引用）访问 T
// → T 必须是 Sync（共享引用可以跨线程）

// Arc<T> 是 Sync ⇔ T: Send + Sync
// 同理，Arc 的引用可以跨线程共享 → 内部 T 必须是 Sync

// Rc<T> 既不是 Send 也不是 Sync——因为非原子的计数在多线程中不安全

fn require_send_sync<T: Send + Sync>(_: T) {}

let arc: Arc<i32> = Arc::new(42);
require_send_sync(arc.clone());  // ✅ Arc<i32> 是 Send + Sync

// let rc: Rc<i32> = Rc::new(42);
// require_send_sync(rc);  // ❌ Rc<i32> 不是 Send 也不是 Sync
```

## 4. Rc/Arc 的写时复制：make_mut

### 4.1 make_mut 的工作机制

```rust
use std::rc::Rc;

let mut rc = Rc::new(String::from("hello"));

// 情况 1：只有一个强引用——直接修改
{
    let s: &mut String = Rc::make_mut(&mut rc);
    // Rc::make_mut 检查 strong_count：
    // strong_count == 1 → 不需要克隆 → 返回 &mut T
    s.push_str(" world");
}
// rc 现在指向 "hello world"

// 情况 2：有多个强引用——克隆后修改
let rc_clone = rc.clone();  // strong_count = 2
{
    let s: &mut String = Rc::make_mut(&mut rc);
    // strong_count == 2 → 需要克隆！
    // 1. 分配新的 RcBox，深拷贝 String 数据
    // 2. 对拷贝进行修改
    // 3. 返回 &mut 指向新拷贝
    // 4. 原来的 rc_clone 不受影响——还指向 "hello world"
    s.push_str("!");
}
// rc 现在指向 "hello world!"（新的独立副本）
// rc_clone 仍然指向 "hello world"（原始数据）
```

### 4.2 Arc::make_mut 的线程安全版本

```rust
use std::sync::Arc;

let arc = Arc::new(vec![1, 2, 3]);

// 并行场景：
let mut arc1 = arc.clone();
std::thread::spawn(move || {
    // make_mut 对 Arc 同样适用
    // 但需要 &mut self——确保没有其他线程在同时做 make_mut
    //（Rust 的类型系统保证了这一点）
    let v: &mut Vec<i32> = Arc::make_mut(&mut arc1);
    v.push(4);
}).join().unwrap();
```

### 4.3 get_mut —— 无克隆的版本

```rust
use std::rc::Rc;

let mut rc = Rc::new(42);

// get_mut 只在 strong_count == 1 时返回 Some(&mut T)
// 不会克隆——要么成功，要么失败
match Rc::get_mut(&mut rc) {
    Some(val) => *val = 100,  // ✅ strong_count == 1
    None => println!("shared—can't get mut ref"),
}

let rc2 = rc.clone();
match Rc::get_mut(&mut rc) {
    Some(_) => unreachable!(),  // strong_count == 2
    None => println!("正确—get_mut 返回 None"),
}
```

## 5. 常见陷阱

### 陷阱 1：循环引用导致内存泄漏

```rust
use std::rc::{Rc, Weak};
use std::cell::RefCell;

struct Node {
    parent: RefCell<Weak<Node>>,     // ✅ 用 Weak 打破循环
    children: RefCell<Vec<Rc<Node>>>,
}

// 如果 parent 也用 Rc：
// struct Node {
//     parent: RefCell<Rc<Node>>,  // ❌ 循环引用！
//     children: RefCell<Vec<Rc<Node>>>,
// }
// → parent → child → parent → child → ... 内存泄漏

// 正确：parent 用 Weak — 不影响 strong_count
let leaf = Rc::new(Node {
    parent: RefCell::new(Weak::new()),
    children: RefCell::new(vec![]),
});

let branch = Rc::new(Node {
    parent: RefCell::new(Weak::new()),
    children: RefCell::new(vec![leaf.clone()]),
});

*leaf.parent.borrow_mut() = Rc::downgrade(&branch);
// ✅ leaf 不阻止 branch 被释放
```

### 陷阱 2：内部可变性与 Rc 的配合

```rust
use std::rc::Rc;
use std::cell::RefCell;

// Rc<T> 只提供 &T（不可变引用）
// 如果你需要修改 Rc 内部的数据，需要 RefCell：
let shared: Rc<RefCell<String>> = Rc::new(RefCell::new(String::from("hello")));

let shared2 = shared.clone();
shared2.borrow_mut().push_str(" world");

// 但注意：RefCell 的借用规则在运行时检查——
// 如果同时持有两个 borrow_mut → panic!

// 对于多线程，用 Arc<Mutex<T>> 或 Arc<RwLock<T>>
```

### 陷阱 3：Rc 不是 Send/Sync——不要"我以为它是"

```rust
// 如果你尝试把 Rc 传到另一个线程：
let rc = std::rc::Rc::new(42);
// std::thread::spawn(move || {
//     println!("{}", rc);  // ❌ Rc<i32> 不是 Send
// }).join().unwrap();
// 编译错误！

// 需要用 Arc：
let arc = std::sync::Arc::new(42);
std::thread::spawn(move || {
    println!("{}", arc);  // ✅ Arc<i32> 是 Send
}).join().unwrap();
```

### 练习 2 答案

```rust
use std::cell::Cell;
use std::ops::Deref;
use std::ptr::NonNull;

struct MyRcInner<T> {
    strong: Cell<usize>,
    value: T,
}

struct MyRc<T> {
    ptr: NonNull<MyRcInner<T>>,
}

impl<T> MyRc<T> {
    fn new(value: T) -> Self {
        let inner = Box::new(MyRcInner {
            strong: Cell::new(1),
            value,
        });
        MyRc {
            ptr: unsafe { NonNull::new_unchecked(Box::into_raw(inner)) },
        }
    }
}

impl<T> Clone for MyRc<T> {
    fn clone(&self) -> Self {
        let inner = unsafe { self.ptr.as_ref() };
        inner.strong.set(inner.strong.get() + 1);
        MyRc { ptr: self.ptr }
    }
}

impl<T> Deref for MyRc<T> {
    type Target = T;
    fn deref(&self) -> &T {
        unsafe { &self.ptr.as_ref().value }
    }
}

impl<T> Drop for MyRc<T> {
    fn drop(&mut self) {
        let inner = unsafe { self.ptr.as_ref() };
        let count = inner.strong.get() - 1;
        inner.strong.set(count);
        if count == 0 {
            unsafe {
                // 重建 Box 来触发 value 的析构
                drop(Box::from_raw(self.ptr.as_ptr()));
            }
        }
    }
}
```

### 练习 3 答案

```rust
// Owner → Gadgets (Rc) → Owner (Rc) → 循环！
// Owner 持有 Gadget 的 Rc
// Gadget 持有 Owner 的 Rc
// → strong_count 永远不会到 0 → 内存泄漏

// 修复：Gadget 用 Weak<RefCell<Owner>> 替代 Rc
struct OwnerFixed {
    gadgets: Vec<Rc<RefCell<GadgetFixed>>>,
```

### 练习 4 答案

```
Rc::clone 的操作：
1. 读取 ptr → inner.strong_count
2. Cell::set(strong_count + 1)
   → 编译为：mov [rcx + 0], rax  （普通的 store）
   ≈ 1-2 CPU cycles

Arc::clone 的操作：
1. 读取 ptr → inner.strong_count
2. AtomicUsize::fetch_add(1, Relaxed)
   → 编译为：lock xadd [rcx + 0], 1  （原子的 fetch-and-add）
   ≈ 20-50 CPU cycles（取决于 CPU 架构）

差异来源：
- lock 前缀触发总线锁定或缓存行锁定
- 内存屏障阻止 CPU 的乱序执行优化
- 多核间需要同步缓存行（MESI 协议）

结论：Arc::clone 比 Rc::clone 慢约 10-50x
      但这只是 clone 操作——实际应用中的瓶颈通常在数据访问而非引用计数
```

## 8. 深度扩展

### 8.1 C++ 的 shared_ptr——比 Arc 更"重"的设计

```cpp
// C++ std::shared_ptr —— Arc<T> 的对应物
#include <memory>

auto sp1 = std::make_shared<int>(42);
auto sp2 = sp1;  // 引用计数 +1

// C++ shared_ptr 和 Rust Arc 的关键差异：

// 1. 控制块分离——C++ 的引用计数不一定和 T 连续
//    make_shared 会把 T 和计数放在一起（类似 RcBox）
//    但 shared_ptr(new T(...)) 会分开分配——两次堆分配！
//    Rust 的 Arc 总是一次分配（类似 make_shared）

// 2. C++ 有 use_count()——可以查询引用计数
int count = sp1.use_count();  // 2
// Rust 也有 Arc::strong_count(&arc)——但不鼓励依赖它做逻辑判断

// 3. C++ 的 weak_ptr::lock() 返回 shared_ptr
//    Rust 的 Weak::upgrade() 返回 Option<Arc<T>>

// 4. C++ 的 shared_ptr 有 aliasing constructor
//    shared_ptr<T> sp(shared_ptr<U>, T* ptr);
//    允许"共享所有权但指向不同对象"
//    Rust 没有等价物——用 Arc::ptr_eq 或重新设计

// 5. C++ 的 shared_ptr 有自定义 deleter
//    Rust 的 Arc 总是用 Drop（或全局 allocator）

// 6. C++ 的 enable_shared_from_this
//    struct MyClass : public enable_shared_from_this<MyClass> { ... };
//    从 this 获取 shared_ptr —— Rust 需要手动用 Weak 或外部方案
```

### 8.2 如果没有 `Rc<RefCell<T>>` —— 如何共享可变状态

`Rc<RefCell<T>>` 是 Rust 中最常见的"共享可变状态"模式。如果没有这个组合，你有哪些替代方案？

- 方案 A：只用 `Rc<T>` —— 无法修改，因为只有 `&T` 而没有 `&mut T`。
- 方案 B：用 `unsafe` 和裸指针绕过借用检查 —— 失去了 Rust 的安全保证。
- 方案 C：用其他内部可变性模式 —— 比如把修改推迟到"最后一个引用持有者"，或者用消息传递代替共享状态。
- 方案 D：`Arc<Mutex<T>>` —— 在多线程场景下适用，但在单线程中 `Mutex` 太重（需要系统调用或原子操作）。

`Rc<RefCell<T>>` 的价值就在于：在单线程中实现共享可变状态（`RefCell` 使用非原子操作，速度很快），同时保留编译期不变性检查和运行时借用检查，且 `RefCell` 的 borrow 检查只是简单的计数器比较，属于零抽象成本。

### 8.3 Arc 的原子操作与内存序选择

```rust
// Arc 内部的原子操作选择：
// Arc::clone() 使用 Ordering::Relaxed
// 为什么不是 Acquire/Release 或 SeqCst？

// 原因：
// - clone 只增加计数——不需要与其他内存操作同步
// - 只要计数增加是"原子的"就够了
// - Relaxed 是最快的——没有内存屏障

// Arc::drop() 使用 Ordering::Release
// - 在释放 T 的数据之前，需要保证其他线程能看到修改
// - Release 确保所有之前的写操作在计数减少前可见

// Arc 的内部实现（简化）：
// fn clone(arc: &Arc<T>) -> Arc<T> {
//     arc.inner().strong.fetch_add(1, Ordering::Relaxed);
//     // fetch_add 保证原子性，Relaxed 不需要同步其他内存
// }

// fn drop(arc: &mut Arc<T>) {
//     if arc.inner().strong.fetch_sub(1, Ordering::Release) == 1 {
//         // 我们是最后一个强引用
//         // 需要 Acquire 屏障来看到其他线程在 T 上的修改
//         std::sync::atomic::fence(Ordering::Acquire);
//         unsafe { drop_in_place(arc.inner().value) };
//         // 之后处理 weak_count...
//     }
// }
```

### 8.4 弱引用计数的"延迟释放"机制

当所有强引用消失时，`T` 的数据被析构（`value` 不再存在），但 `RcBox` 的外壳依然保留（只要 `weak_count > 0`）。此时弱引用仍然可以查询"数据还在吗"，但无法访问数据本身。当所有弱引用也消失后，`RcBox` 的外壳才被释放。

这种两阶段释放设计的优点在于：`Weak` 指针始终有效（不会悬垂），`Weak::upgrade()` 可以安全地检查强引用是否还存在，并且无需"通知所有 `Weak` 数据已失效"——`Weak` 直接查询 `strong_count` 即可。唯一的代价是：当 `weak_count > 0` 时，即使数据已经析构，`RcBox` 仍然占用内存。

C++ 的 `shared_ptr`/`weak_ptr` 采用了同样的设计方案：控制块（control block）独立于数据，数据可以提前释放，`weak_ptr::lock()` 通过查询 `use_count()` 来判断数据是否仍然有效。

### 8.5 `Rc::ptr_eq` —— 检查是否指向同一分配

```rust
use std::rc::Rc;

let a = Rc::new(42);
let b = a.clone();
let c = Rc::new(42);

// 指针相等——同一个 RcBox
assert!(Rc::ptr_eq(&a, &b));  // a 和 b 指向同一个 RcBox

// 数据相等但不指向同一个 RcBox
assert_eq!(*a, *c);           // 值相等
assert!(!Rc::ptr_eq(&a, &c)); // 但不是同一个 RcBox

// ptr_eq 在需要"身份"比较（而不仅仅是"值"比较）时很有用
// 例如，判断两个引用是否来自同一个分配——在图的算法中很常见
```

### 8.7 Rc 与异步——为什么 async 块中的 Rc 有陷阱

```rust
// ❌ 危险——Rc 不是 Send，async 块可能被调度到不同线程！
async fn bad_async() {
    let rc = std::rc::Rc::new(42);
    tokio::spawn(async move {
        println!("{}", rc);  // ❌ Rc 不是 Send
    });
}

// async 块生成的 Future 必须满足 Send（如果需要在多线程运行时中执行）
// Rc 不是 Send → 包含 Rc 的 Future 也不是 Send

// 解决方案：
// 1. 用 Arc 代替 Rc
// 2. 用 tokio::task::LocalSet 在当前线程中运行 !Send 的 future
// 3. 确保 Rc 不会跨越 .await 点（只在同步代码块中使用）

async fn good_async() {
    let rc = std::rc::Rc::new(42);
    // 在 .await 之前使用 rc
    println!("{}", rc);
    // rc 在这里被 drop——不跨越 .await

    some_async_fn().await;  // .await 时 rc 已经不存在了
}
```

### 8.8 引用计数的性能权衡——何时用 Rc/Arc，何时不要

引用计数的优点在于：确定性的释放（离开作用域就减计数，到零就释放），不需要借用检查器的帮助，编译器帮你管理生命周期，适合复杂的所有权图（如有向无环图 DAG）。

但引用计数也有明显的缺点：每次 clone/drop 都要修改计数带来性能开销，循环引用永远不会被释放（除非用 `Weak`），`RcBox` 在堆上导致缓存不友好（访问数据需要两次解引用），每个 `RcBox` 多了 2 个 `usize` 的头部内存开销。

以下场景不应使用 `Rc`/`Arc`：所有权关系简单时用 `Box<T>` 或 `&T`；所有权是树状结构时用普通 Drop 级联；只需要临时共享时用 `&T`（借用）；高性能场景下考虑 arenas 或索引方案。

与 C++ 的 `shared_ptr` 对比，两者有相似的问题，但 C++ 还有额外的负担：可能的两次分配（`shared_ptr` + `new`），即使在不需要的场景下引用计数也总是线程安全的（C++ 的 `shared_ptr` 总是使用原子操作），以及如果有自定义 deleter 则会有更重的 vtable 查找开销。

## 9. 回顾

```
Rc/Arc 内部实现学习清单：

□ 理解 Rc/Arc 的内存布局：栈上一个指针 → 堆上 {strong, weak, value}
□ 知道 Rc::clone 只是增加引用计数——不复制数据
□ 掌握 Strong vs Weak 的计数规则：数据在 strong==0 时析构，RcBox 在 weak==0 时释放
□ 理解 Weak::upgrade 的工作机制——检查 strong_count
□ 知道 Arc 的唯一区别：AtomicUsize vs Cell<usize>——多线程安全
□ 掌握 make_mut 的写时复制：strong==1 直接改，strong>1 克隆后改
□ 能诊断循环引用并用 Weak 打破
□ 理解 C++ shared_ptr/weak_ptr 的关键差异（控制块分离、自定义 deleter）
```

> **一句话总结**：`Rc<T>` 和 `Arc<T>` 在栈上只是一个指针，但指向了堆上的"引用计数头部 + 数据体"。`Rc::clone` 只增加一个整数而不复制数据；`Weak` 通过 `strong_count == 0` 检测悬垂；`Arc` 把 `Cell<usize>` 换成 `AtomicUsize`，用约 20-50 个 CPU cycle 的代价换取线程安全。C++ 的 `shared_ptr` 功能更丰富（自定义 deleter、aliasing constructor），但可能产生两次堆分配——Rust 的 Arc 始终只分配一次，设计哲学是"越少选择，越少出错"。
