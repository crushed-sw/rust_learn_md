# 2.12 自引用结构 & Pin（下）——Pin 投影、async 状态机和 `pin_project`

> 你已经知道 Pin 可以阻止值被移动——但当你有一个被 Pin 住的结构体时，如何安全地访问它的**字段**？如何从 `Pin<&mut Struct>` 拿到 `Pin<&mut Field>`？这就是 **Pin 投影**。它看似简单，实则是 Pin API 中最微妙的部分——错误的投影会悄无声息地破坏 Pin 的保证。这一课深入到 Pin 投影、`pin_project` crate、以及 async 状态机内部如何利用这些机制。

## 1. Pin 投影——从 Pin 的结构体到 Pin 的字段

### 1.1 问题

```rust
use std::pin::Pin;
use std::marker::PhantomPinned;

struct TwoFields {
    a: String,
    b: String,
    _pin: PhantomPinned,
}

let mut boxed = Box::pin(TwoFields {
    a: "hello".into(),
    b: "world".into(),
    _pin: PhantomPinned,
});

// 你有一个 Pin<&mut TwoFields>
// 你想获取 Pin<&mut String> 指向 boxed.a
// 如何做？

// ❌ 不能直接：
// let a: &mut String = &mut boxed.a;  // Pin 不自动 DerefMut（TwoFields: !Unpin）

// ❌ 不能用 get_unchecked_mut 然后取字段：
// let entire: &mut TwoFields = unsafe { Pin::get_unchecked_mut(boxed.as_mut()) };
// let a: &mut String = &mut entire.a;
// // 现在你有 &mut String —— 可以通过 std::mem::replace 移动 a
// // → 破坏了 Pin 的保证（虽然 String 是 Unpin，但原则问题）
```

### 1.2 安全的 Pin 投影

```rust
use std::pin::Pin;

// 对于实现了 Unpin 的字段——安全投影是允许的
impl TwoFields {
    // 手动实现"安全的"投影
    fn a(self: Pin<&mut Self>) -> Pin<&mut String> {
        // SAFETY: String 是 Unpin——即使拿到 &mut String 也不会破坏 Pin 保证
        unsafe {
            let this = Pin::get_unchecked_mut(self);
            Pin::new_unchecked(&mut this.a)
        }
    }

    fn b(self: Pin<&mut Self>) -> Pin<&mut String> {
        unsafe {
            let this = Pin::get_unchecked_mut(self);
            Pin::new_unchecked(&mut this.b)
        }
    }
}

// 使用：
let mut boxed: Pin<Box<TwoFields>> = /* ... */;
boxed.as_mut().a().push_str(" modified");
// ✅ 安全——因为 String: Unpin
```

### 1.3 Pin 投影必须遵守的规则

```
规则 1：如果字段类型是 !Unpin，投影返回的 Pin<&mut Field> 必须保证
        你不会通过其他路径获取到同一个字段的 &mut Field

规则 2：如果结构体实现了 Drop，在 Drop 被调用时 Pin 保证已经不存在了
        ——Drop 接收 &mut Self，不通过 Pin

规则 3：投影方法应该消耗 self（Pin<&mut Self>），防止多次投影到同一个字段

规则 4：不能同时投影到两个字段都返回 Pin<&mut>（除非它们彼此独立）
```

## 2. `pin_project` crate —— 自动生成安全投影

### 2.1 基本用法

```rust
use pin_project::pin_project;
use std::pin::Pin;

// #[pin_project] 自动生成安全的投影方法
#[pin_project]
struct MyStruct {
    #[pin]  // 标记需要 Pin 访问的字段
    future: std::future::Ready<i32>,
    name: String,  // 未标记 #[pin] —— 认为是 Unpin
}

impl MyStruct {
    fn poll_me(self: Pin<&mut Self>) {
        let this = self.project();  // 自动生成的 project() 方法
        // this.future: Pin<&mut Ready<i32>>
        // this.name:  &mut String（不是 Pin——因为没有 #[pin]）

        // this.future.poll(...);  // 直接 poll
    }
}
```

### 2.2 pin_project 如何工作

```rust
// #[pin_project] 宏展开后大致生成：

struct MyStruct {
    future: std::future::Ready<i32>,
    name: String,
}

// 生成的投影结构体（零成本——只是指针变换）：
struct __MyStructProjection<'__pin> {
    future: Pin<&'__pin mut std::future::Ready<i32>>,  // #[pin] → Pin<&mut>
    name: &'__pin mut String,  // 没有 #[pin] → &mut
}

impl MyStruct {
    fn project<'__pin>(self: Pin<&'__pin mut Self>) -> __MyStructProjection<'__pin> {
        unsafe {
            let this = Pin::get_unchecked_mut(self);
            __MyStructProjection {
                future: Pin::new_unchecked(&mut this.future),
                name: &mut this.name,
            }
        }
    }
}

// 关键：project 方法使用了 Pin::get_unchecked_mut —— unsafe
// 但 pin_project 宏被社区广泛审查——认为是安全的
```

### 2.3 PinnedDrop —— 被 Pin 的结构体的自定义 Drop

```rust
use pin_project::{pin_project, pinned_drop};
use std::pin::Pin;

#[pin_project(PinnedDrop)]
struct MyFuture {
    #[pin]
    inner: SomeInnerFuture,
    buf: Vec<u8>,
}

#[pinned_drop]
impl PinnedDrop for MyFuture {
    fn drop(self: Pin<&mut Self>) {
        // self: Pin<&mut Self> —— Pin 的保证还在！
        // 可以在 Pin 的保证下执行清理逻辑
        let this = self.project();
        // this.inner: Pin<&mut SomeInnerFuture>
        // this.buf: &mut Vec<u8>

        // 在 Pin 内安全地清理...
    }
}
// 注意：PinnedDrop 和标准 Drop 不能共存
// PinnedDrop 让你在 Pin 被释放前执行清理（而不是 Drop 之后）
```

## 3. async/await 内部——Pin 的杀手级应用

### 3.1 async 状态机的自引用本质

```rust
// 这个 async 块的内部结构：
async fn fetch_and_process(url: &str) -> String {
    let data = fetch(url).await;        // data 存储在状态机中
    let processed = process(&data);     // processed 可能引用 data
    format!("Result: {}", processed)    // 返回拥有 String
}

// 编译器生成的状态机（简化）：
enum FetchAndProcessFuture<'a> {
    Start { url: &'a str },
    AwaitFetch { /* fetch 的子 Future */ },
    AfterFetch {
        data: String,                   // ← 状态机字段
        processed: String,              // ← 可能自引用 data
    },
    Done,
}

// 问题：从 AwaitFetch 切换到 AfterFetch 时
//       状态机的内存被"重新解释"为不同的枚举变体
//       如果 AfterFetch 的 processed 引用了 data
//       而这个"引用"隐含了"data 的地址"——那就不能移动！
//
// 编译器解决：Future 自动 !Unpin
```

### 3.2 Future::poll 的签名——为什么接收 Pin

```rust
use std::pin::Pin;
use std::task::{Context, Poll};

pub trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
    //       ^^^^^^^^^^^^^^^^
    //       为什么是 Pin<&mut Self> 而不是 &mut Self？
    //
    // 因为：
    // 1. Future 状态机可能自引用 → !Unpin
    // 2. 每次 poll 之间，Future 必须留在同一地址
    // 3. Pin 保证 poll 不会移动 Future
    //
    // 如果 poll 是 &mut Self：
    // - 调用方可以在两次 poll 之间 swap/move Future
    // - 自引用指针就悬垂了
    // - Pin 阻止了这一点
}

// 这也意味着：你只能在 Pin 内 poll Future
// → tokio::spawn 要求 Pin<Box<dyn Future>>
// → 手动 poll 需要 pin! 宏或 Box::pin
```

### 3.3 为什么 async 块生成的状态机需要 Pin

```rust
// 考虑这个 async 块：
async fn reference_across_await() {
    let local = String::from("hello");
    let ref_to_local: &String = &local;
    // ref_to_local 指向 local
    // local 在状态机的字段 A
    // ref_to_local 存储在状态机的字段 B

    some_async_fn().await;
    // ↑ .await 是挂起点——状态机在这里暂停
    // ctx 保存了整个状态机

    println!("{}", ref_to_local);
    // 当恢复时——local 和 ref_to_local 必须还在同一个相对位置！
    // 如果状态机被移动了——ref_to_local 成为悬垂引用 → UB
}

// 所以编译器：
// 1. 检测到自引用 → 给 Future 加 PhantomPinned
// 2. Future 变成 !Unpin
// 3. 只有通过 Pin 才能 poll 它
// 4. 运行时（如 tokio）使用 Pin<Box<dyn Future>> 来存储
```

## 4. 常见陷阱

### 陷阱 1：在 Pin 投影中获取 &mut T 然后用 swap

```rust
// ❌ 即使是通过 Pin 投影获取的 Pin<&mut T>，你也不能 swap

let mut boxed: Pin<Box<MyStruct>> = /* ... */;
// let this = boxed.as_mut().project();

// 如果你通过某种方式拿到了没有 Pin 的 &mut T：
// unsafe {
//     let entire: &mut MyStruct = Pin::get_unchecked_mut(boxed.as_mut());
//     std::mem::swap(entire, &mut another_struct);  // 移动了！→ 自引用指针悬垂
// }
```

### 陷阱 2：PinnedDrop 和 Drop 不能共存

```rust
// ❌ 不能同时实现 Drop 和 PinnedDrop
#[pin_project(PinnedDrop)]
struct MyType {
    data: String,
}

// impl Drop for MyType { ... }  // ❌ 和 PinnedDrop 冲突

// 选择 PinnedDrop — 当结构体使用 #[pin] 投影时
// 选择 Drop — 当不需要 Pin 的保证时
```

### 陷阱 3：忘记 #[pin] 标注

```rust
#[pin_project]
struct MyFuture {
    #[pin]
    inner: SomeFuture,  // ✅ 标注了 #[pin]
    buf: Vec<u8>,       // ✅ 正常——Vec 是 Unpin
}

// 如果 inner 是 !Unpin 且你没标注 #[pin]：
// pin_project 会给 inner 生成 &mut SomeFuture（而不是 Pin<&mut SomeFuture>）
// → 可能导致 inner 被误移动 → UB
```

## 6. 练习答案

### 练习 1 答案

```rust
impl Parser {
    fn source(self: Pin<&mut Self>) -> Pin<&mut String> {
        // String: Unpin —— 安全投影
        unsafe {
            let this = Pin::get_unchecked_mut(self);
            Pin::new_unchecked(&mut this.source)
        }
    }

    fn advance(self: Pin<&mut Self>, n: usize) {
        unsafe {
            let this = Pin::get_unchecked_mut(self);
            // SAFETY: cursor 指向 source 内——source 在 Pin 内不会移动
            this.cursor = this.cursor.add(n);
        }
    }
}
```

### 练习 2 答案

```
inner: tokio::io::ReadHalf → 需要用 #[pin]
  原因：ReadHalf 是一个 Future（!Unpin）——不能安全移动
  需要用 Pin<&mut ReadHalf> 来保护它

buf: Vec<u8> → 不需要 #[pin]
  原因：Vec<u8> 是 Unpin——安全移动

len: usize → 不需要 #[pin]
  原因：usize 是 Unpin
```

### 练习 3 答案

```
分析 async 块：
1. let s = String::from("hello");     → s 在状态机中（字段）
2. let r = &s;                        → r 指向 s —— 自引用！
3. .await                             → 挂起点
4. r.to_owned()                       → 恢复后访问 r

编译器：
1. 检测到 r 是对 s 的引用
2. 检测到引用跨越了 .await（挂起点）
3. 状态机需要 PhantomPinned → Future 是 !Unpin
4. 只有 Pin<&mut Self> 才能 poll

如果编译器没检测到——自引用指针在状态机移动后悬垂——UB
```

### 练习 4 答案

```
Pin<Box<dyn Future>> 可以存入 Vec：
- Box<dyn Future> 是 Sized（胖指针 16 字节）
- Pin<Box<...>> 也是 Sized
- Box 拥有 Future——Vec 存储 Box 指针
- 每个 Box 指向独立的堆分配——互不影响

Pin<&mut dyn Future> 不能存入 Vec：
- &mut dyn Future 是胖指针（16 字节）
- 但它是一个借用——生命周期受限于被引用者
- Vec 要求 'static（或至少长寿命）——借用不满足
- 并且多个 Pin<&mut> 可能指向同一数据——违反借用规则

所以 Pin<Box<dyn Future>> 是异步运行时的基础积木
```

## 7. 深度扩展

### 7.1 Pin 投影的安全推理——为什么它必须是 unsafe

```rust
// Pin 投影的本质：从 Pin<&mut Struct> 到 Pin<&mut Field>
// 为什么这个转换必须有 unsafe？

// 原因 1：防止多次获取 &mut 到同一字段
// 如果有两个 Pin<&mut Field> 指向同一字段 → 违反 Rust 的 &mut 独占性

// 原因 2：防止通过一个字段访问另一个字段时破坏 Pin 保证
// 如果 Pin<&mut FieldA> 给你 &mut FieldB → 你可以 swap FieldB
// → FieldB 被移动 → 自引用指针悬垂

// 原因 3：结构体内部字段可能有"跨字段引用"
// 从 FieldA 到 FieldB 的指针——它们必须一起保持在 Pin 内

// 所以安全的 Pin 投影需要满足：
// 1. 每次投影消耗 self（不能重复投影）
// 2. 投影的字段彼此独立（没有跨字段引用）
// 3. 投影的字段类型是 Unpin（或者投影返回 Pin<&mut>）
```

### 7.2 Generator 和 async 块的 Pin 关系

```rust
// Generator（生成器）是 async/await 的底层机制：

// async 块本质上是 Generator 的语法糖：
// async { ... } ≈ Gen::new(|| { ... })

// Generator 的 resume 方法：
// fn resume(self: Pin<&mut Self>, arg: ResumeArg) -> GeneratorState<Yield, Return>

// Generator 是 !Unpin —— 原因和 Future 一样：状态机可能自引用

// async 块 ~= Generator<Yield = (), Return = T>
// .await ~= generator.resume(...)  → 挂起点
```

### 7.3 C++ 的协程 vs Rust 的 async——两种移动策略

```cpp
// C++20 协程：
// - 协程帧（coroutine frame）在堆上分配
// - 协程帧的地址在协程生命周期内不变（堆分配）
// - 自引用指针放在协程帧内——天然安全（堆地址不变）
// - 不需要 Pin 等价物——因为协程帧本身就是堆分配的

// Rust 的 async：
// - Future 可以在栈上（用 pin! 宏）——零分配
// - 也可以 Pin<Box<dyn Future>> 在堆上——手动选择
// - 编译器不强制堆分配——给程序员选择权
// - 但代价是 Pin 的复杂性

// 两种哲学的取舍：
// C++：隐式堆分配 → 简单但总是有分配开销
// Rust：显式 Pin → 复杂但可以选择零分配
```

### 7.4 如果没有 Pin 投影和 `pin_project`——手动 unsafe 投影的代价

```rust
// 假设没有 pin_project 宏——你需要手动为每个结构体写 unsafe 投影代码

use std::pin::Pin;
use std::marker::PhantomPinned;

struct FutureState {
    buffer: Vec<u8>,
    offset: usize,
    _pin: PhantomPinned,
}

// ❌ 没有 pin_project：手动投影的灾难
impl FutureState {
    // 方案 A：不投影——每次 poll 都在 unsafe 中访问字段
    fn poll_manual(self: Pin<&mut Self>) {
        // 你对结构体有 Pin<&mut FutureState>
        // 但你无法安全地获取 Pin<&mut buffer>
        
        unsafe {
            let this = self.get_unchecked_mut();
            // 现在 this 是 &mut FutureState
            // 任何人都可以用 this 做 swap/move——Pin 保证被绕过了！
            // 每次访问字段都需要在 unsafe 块中——没有编译器的帮助
            this.buffer.push(42); // unsafe 上下文
        }
    }
    
    // 方案 B：自己实现安全的投影方法（每个结构体都要写一遍）
    fn project(self: Pin<&mut Self>) -> Projection<'_> {
        unsafe {
            let this = self.get_unchecked_mut();
            Projection {
                buffer: Pin::new_unchecked(&mut this.buffer),
                offset: &mut this.offset,  // usize: Unpin, 不需要 Pin
            }
        }
    }
    // 但每次添加/删除字段都要手动更新 Projection 结构体！
}

struct Projection<'a> {
    buffer: Pin<&'a mut Vec<u8>>,
    offset: &'a mut usize,
}

// pin_project 自动生成了上面所有代码——省去数百行重复的 unsafe 投影

// 更糟的情况：嵌套结构体
struct OuterState {
    inner: FutureState,  // !Unpin（包含 PhantomPinned）
}

// 没有 pin_project：
// 你需要为 OuterState 也写一个 project() 方法
// 然后调用 inner.project() —— 两层 unsafe
impl OuterState {
    fn project(self: Pin<&mut Self>) -> /* ... */ {
        unsafe {
            let this = self.get_unchecked_mut();
            // 现在需要 InnerProjection —— 但 inner 是 !Unpin
            // 需要 Pin::new_unchecked(&mut this.inner)
            // 三层嵌套的 unsafe —— 极易出错
            todo!("每次都要写这些——pin_project 一行宏就解决了")
        }
    }
}

// 如果没有 pin_project 的现实影响：
// 1. 每个 !Unpin 结构体的字段访问都在 unsafe 中——丧失了类型安全
// 2. 字段重命名/增删 → 所有投影代码都要手动更新 → 维护噩梦
// 3. 新手无法安全地使用 Pin——unsafe 门槛过杆
// 4. 生态系统中各种手工投影方案碎片化——互不兼容
// 5. 写异步运行时的开发者苦不堪言——每增加一个 future 都要写一堆 unsafe

// pin_project 的核心价值：
// - 它不引入新语义——只是自动生成本来就需要手写的 unsafe 代码
// - 但它让 unsafe 代码正确性由"宏的一次审查"保证——而不是 N 个使用者的 N 次审查
// - 这就是 Rust 的哲学：用宏消除样板 unsafe，隔离真正的 unsafe 逻辑
```

### 7.5 `Pin::static_ref` 和 `Pin::static_mut`

```rust
use std::pin::Pin;

// 从 'static 引用创建 Pin（1.68+）
static mut DATA: String = String::new();

fn get_pinned() -> Pin<&'static mut String> {
    unsafe { Pin::static_mut(&mut DATA) }
}
// Pin 保证 DATA 不会被移动——static 的数据天然不会被移动
// Pin::static_mut 和 Pin::static_ref 是安全的——'static 保证了这一点
```

### 7.6 `Pin::into_inner` 和 `Pin::get_mut`

```rust
use std::pin::Pin;

// Pin::into_inner —— 拆开 Pin，返回内部指针（P）
let boxed = Box::pin(42);
let boxed: Box<i32> = Pin::into_inner(boxed);  // 不再被 Pin

// Pin::get_mut —— 获取 &mut T（只在 T: Unpin 时可用）
let mut x = 42;
let mut pinned = Pin::new(&mut x);
let r: &mut i32 = Pin::get_mut(&mut pinned);  // i32: Unpin
*r = 100;
// 安全——不会破坏 Pin 的保证（i32 是 Unpin）
```

### 7.7 Pin 和 `std::mem::replace` / `std::mem::swap`

```rust
// 为什么 std::mem::replace 对 Pin 是危险的：

use std::pin::Pin;
use std::marker::PhantomPinned;

struct SelfRef {
    data: String,
    ptr: *const String,

let mut pinned: Pin<Box<SelfRef>> = Box::pin(/* ... */);

// Pin 的类型系统防止了这个——只有 unsafe 代码能绕过

```

### 7.8 `Unpin` 的 auto trait 推导与 `PhantomPinned`

```rust
use std::marker::{PhantomPinned, Unpin};

// PhantomPinned 是 !Unpin —— 所以任何包含它的类型也是 !Unpin
struct MyType {
    data: String,
    _pin: PhantomPinned,
}

// 即使不是 auto trait，你也可以手动实现 !Unpin：
// impl !Unpin for MyType {}  // nightly

// 但 PhantomPinned 在 stable 中是最简单的方式

// 如果类型不包含 PhantomPinned 也不包含任何 !Unpin 的字段：
// → 自动实现 Unpin（auto trait）
struct NormalType {
    data: String,       // Unpin
    nums: Vec<i32>,     // Unpin
}
// NormalType 自动是 Unpin
```

### 7.9 Pin 在异步生态系统中的角色总结

```
┌─────────────────────────────────────────────────┐
│                Rust 异步生态的 Pin 全景         │
├─────────────────────────────────────────────────┤
│                                                 │
│  async fn / async {}                            │
│       │                                         │
│       ▼                                         │
│  impl Future (自动 !Unpin)                      │
│       │                                         │
│       ├──→ pin! 宏 (栈)                         │
│       │        → Pin<&mut impl Future>           │
│       │        → 在当前栈帧内 poll               │
│       │                                         │
│       ├──→ Box::pin (堆)                        │
│       │        → Pin<Box<impl Future>>           │
│       │        → 可以 move/存储                  │
│       │                                         │
│       └──→ tokio::spawn / spawn_local           │
│                → 要求 Send / !Send               │
│                → 内部用 Pin<Box<dyn Future>>     │
│                                                 │
│  pin_project 宏                                  │
│       └──→ 安全的 Pin 投影                      │
│             → Pin<&mut Field> 从 Pin<&mut Struct>│
│                                                 │
│  PinnedDrop                                     │
│       └──→ 在 Pin 内的清理逻辑                  │
│                                                 │
└─────────────────────────────────────────────────┘
```

## 8. 回顾

```
自引用结构 & Pin（下）学习清单：

□ 理解 Pin 投影的概念：从 Pin<&mut Struct> → Pin<&mut Field>
□ 掌握 pin_project crate 的基本用法：#[pin] 标注和 project() 方法
□ 知道 PinnedDrop 和普通 Drop 的区别——以及为什么不能共存
□ 理解 async 状态机的自引用本质——为什么需要 !Unpin
□ 知道 Future::poll 为什么接收 Pin<&mut Self> 而不是 &mut Self
□ 了解 C++ 协程的堆分配策略 vs Rust 的可选零分配 Pin
□ 知道 Pin::static_ref / Pin::into_inner / Pin::get_mut 的用法
□ 知道如果没有 pin_project——需要手动写 unsafe 投影代码的代价
```

> **一句话总结**：Pin 投影让你从被 Pin 住的结构体安全地访问内部字段——`pin_project` 宏自动化了 `#[pin]` 标注和安全的 `project()` 方法。async 块被编译为自引用的 Future 状态机（`!Unpin`），这就是为什么 `Future::poll` 必须接收 `Pin<&mut Self>`。C++ 协程通过强制堆分配避免了 Pin 的复杂性，Rust 则选择了"让程序员选择零分配（Pin）还是堆分配（Box::pin）"——复杂但零成本的哲学。
