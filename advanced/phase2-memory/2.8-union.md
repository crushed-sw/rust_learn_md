# 2.8 Union（联合体）——当所有变体共用同一块内存

> 你有一个 `u32`，有时需要把它当成 `f32` 来读；你在写一个网络协议解析器，同一个字节位置可能是 4 种不同的消息类型；你在做 FFI，C 的 `union` 需要直接映射到 Rust。你可能会想到 `enum`——但 `enum` 带标签（discriminant），占用额外空间，而且运行时才知道是哪个变体。`union` 是更低层的工具：所有字段**共享同一块内存**，没有标签，没有运行时检查，大小等于最大字段的大小。它不安全，但给了你完全的控制权——就像 C 的 union，但规则更清晰。

---

## 1. Union 的基本概念

### 1.1 Union 的定义

```rust
// 一个最简单的 union：4 字节内存，可以被解释为 u32 或 f32
union IntOrFloat {
    i: u32,
    f: f32,
}

// 大小 = max(sizeof(u32), sizeof(f32)) = 4 字节
// 对齐 = max(align_of(u32), align_of(f32)) = 4
// 没有 discriminant——编译器不知道当前存的是哪个变体

fn main() {
    let u = IntOrFloat { i: 42 };
    
    // 读取字段需要 unsafe：
    unsafe {
        println!("as u32: {}", u.i);  // 42
        println!("as f32: {}", u.f);  // 重新解释位模式——不是 42.0
    }
}
```

### 1.2 Union vs Enum——核心区别

```rust
// Enum：带标签的联合体（tagged union）
enum IpAddr {
    V4(u8, u8, u8, u8),   // discriminant = 0，数据 = 4 字节
    V6(String),            // discriminant = 1，数据 = 堆指针 + cap + len (24 字节)
}
// sizeof(IpAddr) = max(变体大小) + discriminant 大小
// 通常 = 24 + 8 = 32 字节（在 64 位平台上）

// Union：无标签的联合体（untagged union）
union IpAddrUnion {
    v4: [u8; 4],
    v6: *const u8,  // 实际上 String 不能放在 union 里（见后文）
}
// sizeof(IpAddrUnion) = max(sizeof([u8; 4]), sizeof(*const u8)) = 8 字节
// 没有 discriminant——你不知道当前是 v4 还是 v6
```

**关键差异对照表：**

| 特性 | `enum` | `union` |
|------|--------|---------|
| 标签（discriminant） | 有，占用空间 | 无，零开销 |
| 运行时知道类型 | 是（`match` 自带检查） | 不是（你自己跟踪） |
| 安全访问 | `match` / `if let` 安全 | 需要 `unsafe` |
| FFI 兼容性 | 需要 `#[repr(C)]` 处理 | 天然对应 C union |
| 大小 | max + discriminant | max 字段大小 |
| 字段类型限制 | 任意 | 不能是 Drop 类型（默认情况） |

### 1.3 Union 的内存布局

```rust
use std::mem;

union Mixed {
    a: u8,      // 1 字节
    b: u32,     // 4 字节
    c: u64,     // 8 字节
}

fn main() {
    // 大小 = max(sizeof(a), sizeof(b), sizeof(c)) = 8 字节
    assert_eq!(mem::size_of::<Mixed>(), 8);
    
    // 对齐 = max(align_of(a), align_of(b), align_of(c)) = 8
    assert_eq!(mem::align_of::<Mixed>(), 8);
    
    // 内存布局（假设小端）：
    //
    // 字节:  | 0  | 1  | 2  | 3  | 4  | 5  | 6  | 7  |
    // a:     | a0 | -- | -- | -- | -- | -- | -- | -- |  只用第 1 字节
    // b:     | b0 | b1 | b2 | b3 | -- | -- | -- | -- |  用前 4 字节
    // c:     | c0 | c1 | c2 | c3 | c4 | c5 | c6 | c7 |  用全部 8 字节
    //
    // 所有字段的起始地址相同（第一个字节）
    
    let mut u = Mixed { c: 0x0102030405060708_u64 };
    
    unsafe {
        // 写入 c 会影响所有字段的字节：
        assert_eq!(u.c, 0x0102030405060708);
        // 读 b 得到 c 的低 4 字节（小端）：
        assert_eq!(u.b, 0x05060708);
        // 读 a 得到 c 的最低 1 字节（小端）：
        assert_eq!(u.a, 0x08);
    }
}
```

## 2. Union 的字段类型限制

### 2.1 默认限制：不能包含 Drop 类型

```rust
// ❌ 编译错误：String 实现了 Drop
// union BadUnion {
//     s: String,   // error: union fields must not have drop glue
//     n: i32,
// }

// 原因：union 字段共享内存——如果 drop 了 union，
// 编译器不知道哪个字段真正"活跃"，不知道该调用谁的 Drop。
//
// String 有 Drop（释放堆内存），所以不能直接放在 union 中。

// ✅ 可以包含的类型：所有 Copy 类型、不实现 Drop 的类型
union GoodUnion {
    i: i32,
    f: f64,
    p: *const u8,  // 裸指针没有 Drop
    a: [u8; 16],
    b: bool,
}
```

### 2.2 用 ManuallyDrop 绕过限制

```rust
use std::mem::ManuallyDrop;

// ManuallyDrop<T> 是一个 wrapper——它"抑制"了 T 的 Drop 实现
// 你自己负责在合适的时机手动释放

union StringOrInt {
    s: ManuallyDrop<String>,  // ✅ 现在可以了——ManuallyDrop<String> 不 Drop
    i: i64,
}

fn main() {
    let mut u = StringOrInt { i: 42 };
    
    unsafe {
        assert_eq!(u.i, 42);
    }
    
    // 写入 String：
    u.s = ManuallyDrop::new(String::from("hello"));
    
    unsafe {
        // 读取 String：
        assert_eq!(&*u.s, "hello");  // *ManuallyDrop<T> → T，需要 Deref
    }
    
    // ⚠️ 在 u 被 drop 之前，你必须手动释放 ManuallyDrop<String>：
    unsafe {
        ManuallyDrop::drop(&mut u.s);  // 手动调用 String 的 Drop
    }
    // 如果不手动 drop，String 的堆内存会泄漏！
}
```

### 2.3 手动管理 Drop 的模式

当 union 包含非 Copy 类型（如 `String`），必须面对一个问题：**谁负责释放？** 编译器不知道 union 中哪个字段是"活跃的"，所以它不会为 union 生成 `Drop` 实现。你需要自己跟踪活跃变体并手动调用析构。

下面的 `SafeFlexibleValue` 展示了完整的手动管理方案：

- 用一个 `ActiveVariant` 枚举跟踪"当前哪个变体是活的"
- `String` 变体用 `ManuallyDrop<String>` 包装，阻止编译器自动 drop
- `set_integer` 在切换变体时**先释放旧的 String**（如果当前活跃的是 String）
- 自定义的 `Drop` impl 在结构体销毁时释放最后一个活跃的 String

**为什么不用标准 enum？** 标准 enum 自带 discriminant（标签），会占用额外空间。union + 手动 discriminant 让你可以在需要极致紧凑布局时（比如 8 字节的数据 + 1 字节的 discriminant = 仅 9 字节），自行控制内存布局。但代价是所有正确性保证都由你负责。

```rust
use std::mem::ManuallyDrop;

union FlexibleValue {
    integer: i64,
    float: f64,
    string: ManuallyDrop<String>,
}

// 安全包装——自己跟踪活跃变体
enum ActiveVariant {
    Integer,
    Float,
    String,
}

struct SafeFlexibleValue {
    value: FlexibleValue,
    active: ActiveVariant,
}

impl SafeFlexibleValue {
    fn new_integer(n: i64) -> Self {
        SafeFlexibleValue {
            value: FlexibleValue { integer: n },
            active: ActiveVariant::Integer,
        }
    }
    
    fn new_string(s: String) -> Self {
        SafeFlexibleValue {
            value: FlexibleValue { string: ManuallyDrop::new(s) },
            active: ActiveVariant::String,
        }
    }
    
    fn get_integer(&self) -> Option<i64> {
        if matches!(self.active, ActiveVariant::Integer) {
            unsafe { Some(self.value.integer) }
        } else {
            None
        }
    }
    
    fn get_string(&self) -> Option<&str> {
        if matches!(self.active, ActiveVariant::String) {
            unsafe { Some(&self.value.string) }
        } else {
            None
        }
    }
    
    fn set_integer(&mut self, n: i64) {
        // 如果当前是 String，需要先释放它
        if matches!(self.active, ActiveVariant::String) {
            unsafe { ManuallyDrop::drop(&mut self.value.string); }
        }
        self.value.integer = n;
        self.active = ActiveVariant::Integer;
    }
}

impl Drop for SafeFlexibleValue {
    fn drop(&mut self) {
        // 只在活跃变体是 String 时才释放
        if matches!(self.active, ActiveVariant::String) {
            unsafe { ManuallyDrop::drop(&mut self.value.string); }
        }
        // 整数和浮点数没有 Drop——不需要特殊处理
    }
}
```

## 3. Safe Wrapping Patterns——安全地使用 Union

### 3.1 Enum + Union——手动实现 Tagged Union

```rust
// 场景：实现一个值可以是 i32、f64、或 String
// 标准 enum 即可做到——但如果你需要精确控制内存布局呢？

use std::mem::ManuallyDrop;

union ValueData {
    int: i32,
    float: f64,
    string: ManuallyDrop<String>,
}

#[derive(Debug, PartialEq)]
enum ValueKind {
    Int,
    Float,
    String,
}

struct Value {
    kind: ValueKind,
    data: ValueData,
}

impl Value {
    fn new_int(n: i32) -> Self {
        Value {
            kind: ValueKind::Int,
            data: ValueData { int: n },
        }
    }
    
    fn new_float(f: f64) -> Self {
        Value {
            kind: ValueKind::Float,
            data: ValueData { float: f },
        }
    }
    
    fn new_string(s: String) -> Self {
        Value {
            kind: ValueKind::String,
            data: ValueData { string: ManuallyDrop::new(s) },
        }
    }
    
    fn as_int(&self) -> Option<i32> {
        if self.kind == ValueKind::Int {
            unsafe { Some(self.data.int) }
        } else {
            None
        }
    }
    
    fn as_float(&self) -> Option<f64> {
        if self.kind == ValueKind::Float {
            unsafe { Some(self.data.float) }
        } else {
            None
        }
    }
    
    fn as_str(&self) -> Option<&str> {
        if self.kind == ValueKind::String {
            unsafe { Some(&self.data.string) }
        } else {
            None
        }
    }
}

impl Drop for Value {
    fn drop(&mut self) {
        if self.kind == ValueKind::String {
            unsafe { ManuallyDrop::drop(&mut self.data.string); }
        }
    }
}

// 这本质上是手动实现了 enum——但你可以完全控制布局
// 标准 enum (enum Value { Int(i32), Float(f64), String(String) })
// 让编译器处理，通常更安全。只在需要特殊布局控制时手动实现
```

### 3.2 MaybeUninit + Union——延迟初始化模式

**延迟初始化**是指：先分配内存空间，但稍后才写入有效数据。这在以下场景中非常有用：
- 固定大小的环形缓冲区——槽位可能为空，也可能有值
- 无锁数据结构——需要原子地"发布"一个值到槽位中
- 避免在构造时立即初始化大数组的成本

`MaybeUninit<T>` 是 Rust 提供的"未初始化内存"抽象——它告诉编译器"这块内存可能不包含有效的 T"。与 union 配合时，可以把 `MaybeUninit<T>` 和 `T` 放在同一个 union 中，切换状态时不需要移动数据——只需写入后"声明"它为已初始化。

`MaybeUninitOr<T>` 展示了这个组合：同一块内存，既可以当作未初始化的（`raw`），也可以当作已初始化的（`value`），由外部逻辑跟踪状态。

下面的 `Slot<T>` 实现了一个有"active"标记的固定大小槽位：
- `insert`：检查是否空闲 → 写入值 → 标记活跃
- `remove`：检查是否活跃 → 读取值 → 标记空闲
- `get`：安全地返回活跃值的引用

```rust
use std::mem::MaybeUninit;

// MaybeUninit<T> 允许"未初始化"状态
// 与 union 配合可以实现高性能的延迟初始化

union MaybeUninitOr<T: Copy> {
    raw: MaybeUninit<T>,
    value: T,
}

// 这种模式在需要"可能未初始化"的固定大小缓冲区中常见
// 比如无锁数据结构中的"槽位"（slot）

use std::mem;

struct Slot<T: Copy> {
    // 每个槽位要么是活跃的（有值），要么是空的
    // 用 union 避免占用双倍空间
    data: MaybeUninit<T>,
    active: bool,
}

impl<T: Copy> Slot<T> {
    fn new() -> Self {
        Slot {
            data: MaybeUninit::uninit(),
            active: false,
        }
    }
    
    fn insert(&mut self, value: T) -> bool {
        if self.active {
            return false;
        }
        self.data.write(value);
        self.active = true;
        true
    }
    
    fn remove(&mut self) -> Option<T> {
        if !self.active {
            return None;
        }
        self.active = false;
        unsafe { Some(self.data.assume_init_read()) }
    }
    
    fn get(&self) -> Option<&T> {
        if self.active {
            unsafe { Some(self.data.assume_init_ref()) }
        } else {
            None
        }
    }
}
```

### 3.3 类型重解释模式——bitcast

**类型重解释**（bitcast）是指：不改变任何 bit，只改变编译器对同一块内存的"解读方式"。例如：把一个 `f32` 的 32 个 bit 原封不动地读作 `u32`。

**为什么需要？**
- **IEEE 754 浮点数的位级操作**：提取符号位、指数、尾数；实现快速近似倒数平方根（Quake III 的那个著名 hack）
- **自定义序列化**：需要把浮点数的原始字节写入网络包
- **性能**：`f32::to_bits()` 在标准库中已有——但了解 union 实现帮你理解底层原理

**和 `transmute` 的区别**：`std::mem::transmute<f32, u32>(val)` 做同样的事，但 `transmute` 要求源类型和目标类型大小相同（编译期检查），而 union 的 bitcast 不限制类型——你可以从 `u32` 读到 `[u8; 4]`。union 的 bitcast 是更底层的工具，`transmute` 常用于简单的一对一转换。

**安全注意**：从 union 的另一个字段读取是 unsafe 的——你保证这个位模式对目标类型是有效的。`f32` → `u32` 总是安全的（所有 u32 都是有效的），但反过来 `u32` → `f32` 可能产生 NaN。

```rust
// 最常见的 union 用途之一：在数值类型间重新解释位模式

union FloatBits {
    f: f32,
    bits: u32,
}

fn float_to_bits(f: f32) -> u32 {
    let u = FloatBits { f };
    unsafe { u.bits }
}

fn bits_to_float(bits: u32) -> f32 {
    unsafe { FloatBits { bits }.f }
}

fn main() {
    let pi = 3.14159_f32;
    let bits = float_to_bits(pi);
    println!("{} as bits: {:#034b}", pi, bits);
    // 3.14159 as bits: 0b01000000010010010000111111010010
    
    let back = bits_to_float(bits);
    assert_eq!(pi, back);
    
    // 提取符号位、指数、尾数：
    let u = FloatBits { f: pi };
    unsafe {
        let sign = (u.bits >> 31) & 1;
        let exponent = (u.bits >> 23) & 0xFF;
        let mantissa = u.bits & 0x7FFFFF;
        println!("sign={}, exp={}, mantissa={:#x}", sign, exponent, mantissa);
    }
}
```

## 4. FFI——与 C Union 互操作

### 4.1 映射 C Union

```rust
// C 代码中的 union：
//
// typedef union {
//     int   i;
//     float f;
// } Number;
//
// void print_number(Number* n, int type);

// Rust 映射——需要 #[repr(C)] 确保 ABI 兼容：

#[repr(C)]
union Number {
    i: i32,
    f: f32,
}

// extern 函数声明
extern "C" {
    fn print_number(n: *const Number, type_: i32);
}

fn main() {
    let n = Number { i: 42 };
    unsafe {
        print_number(&n, 0);  // type=0 表示 int
    }
    
    let n = Number { f: 3.14 };
    unsafe {
        print_number(&n, 1);  // type=1 表示 float
    }
}
```

### 4.2 复杂 C Union 示例——网络协议解析

```rust
// C 中常见的网络协议头（如 IP 头 + TCP 头组合）：
//
// typedef union {
//     struct { uint32_t src; uint32_t dst; } ip;
//     struct { uint16_t sport; uint16_t dport; uint32_t seq; } tcp;
// } PacketHeader;

// Rust 版本：

#[repr(C)]
union PacketHeader {
    ip: IpHeader,
    tcp: TcpHeader,
}

#[repr(C)]
#[derive(Debug, Clone, Copy)]
struct IpHeader {
    src: u32,
    dst: u32,
}

#[repr(C)]
#[derive(Debug, Clone, Copy)]
struct TcpHeader {
    sport: u16,
    dport: u16,
    seq: u32,
}

fn parse_packet(bytes: &[u8]) {
    // 确保不对空切片操作
    if bytes.len() < 4 {
        return;
    }
    
    let proto = bytes[0];  // 协议类型——第 0 字节
    
    unsafe {
        // 将 bytes 的指针解释为 PacketHeader
        // 注意：这要求字节序和 ABI 匹配机器的架构
        let header_ptr = bytes.as_ptr() as *const PacketHeader;
        let header = &*header_ptr;
        
        match proto {
            1 => {
                // IP 协议
                println!("IP: src={}, dst={}", header.ip.src, header.ip.dst);
            }
            2 => {
                // TCP 协议
                println!("TCP: sport={}, dport={}, seq={}",
                    header.tcp.sport, header.tcp.dport, header.tcp.seq);
            }
            _ => println!("Unknown protocol"),
        }
    }
}
```

### 4.3 C Union 中的 enum 判别器模式

```cpp
// C 代码：
// struct TaggedNumber {
//     enum { INT, FLOAT } tag;
//     union {
//         int i;
//         float f;
//     } data;
// };

// Rust 映射：

#[repr(C)]
struct TaggedNumber {
    tag: NumberTag,
    data: NumberData,
}

#[repr(C)]
#[derive(Debug, PartialEq)]
enum NumberTag {
    Int = 0,
    Float = 1,
}

#[repr(C)]
union NumberData {
    i: i32,
    f: f32,
}

impl TaggedNumber {
    fn get_value(&self) -> f64 {
        unsafe {
            match self.tag {
                NumberTag::Int => self.data.i as f64,
                NumberTag::Float => self.data.f as f64,
            }
        }
    }
    
    fn set_int(&mut self, v: i32) {
        self.tag = NumberTag::Int;
        self.data.i = v;
    }
    
    fn set_float(&mut self, v: f32) {
        self.tag = NumberTag::Float;
        self.data.f = v;
    }
}

fn main() {
    let mut tn = TaggedNumber {
        tag: NumberTag::Int,
        data: NumberData { i: 42 },
    };
    
    assert_eq!(tn.get_value(), 42.0);
    
    tn.set_float(3.14);
    assert!((tn.get_value() - 3.14).abs() < 0.001);
}
```

## 5. Union 与 Transmute——两种位重解释方式

### 5.1 Transmute 的方式

```rust
// transmute：直接把一种类型的位模式解释为另一种类型
// 极度不安全——没有任何检查

let i: u32 = 0x40490FDB;  // 3.14159 的 IEEE 754 表示
let f: f32 = unsafe { std::mem::transmute::<u32, f32>(i) };
println!("{}", f);  // 3.14159

// transmute 的危险：
// 1. 如果大小不同，编译失败——但如果你编译通过，可能是 UB
// 2. 没有意图表达——代码读者不知道"这里在做重解释"
// 3. 无法部分读取——只能整值 transmute
```

### 5.2 Union 的方式——更安全

```rust
union U32F32 {
    i: u32,
    f: f32,
}

let u = U32F32 { i: 0x40490FDB };
let f: f32 = unsafe { u.f };
println!("{}", f);  // 3.14159

// union 的优势：
// 1. 意图清晰——union 的定义就是"这些类型共享内存"
// 2. 编译器检查字段类型匹配——写错字段名会报错
// 3. 可以读取部分字段（如从 u64 union 中读 u32）
// 4. 可以单独修改一个字段而不影响语义
```

### 5.3 什么时候用 Transmute，什么时候用 Union

```rust
// 使用 transmute 的场景：
// - 单次的、临时的类型重解释（如 f64::from_bits 内部）
// - 类型的 safety wrapper 是可见的（如 bytemuck::cast）
// - 两个类型大小不同也需要重解释（如 &[u8] → &[u32]）

// 使用 union 的场景：
// - 持续存在的多态数据（"这个位置可以是 A 或 B"）
// - FFI 映射 C union
// - 构造复杂内存布局（如 tagged union）
// - 意图需要被文档化和类型系统编码

// 经验法则：如果一段内存持续被当成两种类型使用 → union
//           如果是临时的类型转换 → transmute（但优先用 safe wrapper）
```

## 6. 常见陷阱

### 陷阱 1：读取非活跃字段——未定义行为

```rust
union IntOrFloat {
    i: u32,
    f: f32,
}

fn main() {
    let u = IntOrFloat { i: 42 };
    
    // ⚠️ 读取 f 在技术上是安全的（所有位模式都是有效的 f32）
    // 但 Rust 编译器不保证这一点——它可能假设你只读"活跃"的字段
    // 并基于这个假设做优化
    
    unsafe {
        let f = u.f;  // 从 u.i=42 的角度看，这个 f 是什么？
        println!("{}", f);  // 极其小的浮点数：5.885e-44
    }
    
    // 对于整数 → 浮点数重解释，这恰好是安全的
    // 但对于包含指针或引用的 union，读取非活跃字段是 UB
}
```

### 陷阱 2：Drop 字段导致内存泄漏

```rust
use std::mem::ManuallyDrop;

union LeakyUnion {
    s: ManuallyDrop<String>,
    n: i64,
}

fn leak_example() {
    let mut u = LeakyUnion { s: ManuallyDrop::new(String::from("hello")) };
    
    // 切换活跃字段：
    u.n = 42;  // ⚠️ 覆盖了 String 的指针——堆内存泄漏！
    
    // String 的指针和 capacity 被 i64 覆盖了——
    // 堆上 "hello" 的内存永远不会被释放
}
// u 被 drop——但只 drop 了 i64（什么都不做）
// String 的堆分配泄漏了

// 修复：先手动 drop String
fn fixed_example() {
    let mut u = LeakyUnion { s: ManuallyDrop::new(String::from("hello")) };
    
    unsafe {
        ManuallyDrop::drop(&mut u.s);  // ✅ 先释放
    }
    
    u.n = 42;  // 现在安全了
}
```

### 陷阱 3：Union 字段的引用和指针问题

```rust
union RefOrPtr {
    r: &'static str,
    p: *const u8,
}

fn dangerous() {
    let s = String::from("hello");
    // ⚠️ 把局部变量的引用放进 union：
    // 不能在 union 中存活超过 s 的生命周期
    let r: &str = &s;
    
    // 如果用 transmute 把 &str 放进 union... 已进入危险区域
    // 正确做法是只在 union 中放 'static 引用或裸指针
}

// 安全版本——只用 'static 生命周期
union SafeRefOrPtr {
    r: &'static str,
    n: usize,
}

fn safe() {
    let u = SafeRefOrPtr { r: "static string literal" };
    unsafe {
        println!("{}", u.r);  // ✅ 'static 数据永远活着
    }
}
```

### 陷阱 4：对齐问题——不同架构上的差异

```rust
// 在 x86-64 上，未对齐访问是"慢但正确"的
// 在 ARM 上，未对齐访问可能是 UB
// 在 WASM 上，行为不同

union PotentiallyUnaligned {
    bytes: [u8; 8],
    value: u64,  // 要求 8 字节对齐
}

fn read_from_buffer(buf: &[u8], offset: usize) -> u64 {
    // ⚠️ offset 可能不对齐
    let ptr = &buf[offset] as *const u8 as *const PotentiallyUnaligned;
    unsafe {
        // 如果 ptr 不是 8 字节对齐的，在 ARM 上 UB
        (*ptr).value
    }
}

// 安全方式——用 ptr::read_unaligned 或逐字节构造
fn read_safe(buf: &[u8], offset: usize) -> u64 {
    let bytes: [u8; 8] = buf[offset..offset + 8].try_into().unwrap();
    u64::from_le_bytes(bytes)
}
```

### 练习 2 答案

```rust
union FloatBits {
    f: f32,
    bits: u32,
}

struct FloatInspector {
    inner: FloatBits,
}

impl FloatInspector {
    fn new(f: f32) -> Self {
        FloatInspector {
            inner: FloatBits { f },
        }
    }
    
    fn bits(&self) -> u32 {
        unsafe { self.inner.bits }
    }
    
    fn sign_bit(&self) -> u32 {
        (self.bits() >> 31) & 1
    }
    
    fn exponent_bits(&self) -> u32 {
        (self.bits() >> 23) & 0xFF
    }
    
    fn mantissa_bits(&self) -> u32 {
        self.bits() & 0x7FFFFF
    }
    
    fn print_parts(&self) {
        println!("Float: {}", unsafe { self.inner.f });
        println!("  Bits:     {:032b}", self.bits());
        println!("  Sign:     {}", self.sign_bit());
        println!("  Exponent: {} (0x{:02x})", self.exponent_bits(), self.exponent_bits());
        println!("  Mantissa: {} (0x{:06x})", self.mantissa_bits(), self.mantissa_bits());
    }
    
    fn is_positive_zero(&self) -> bool {
        self.bits() == 0
    }
    
    fn is_negative_zero(&self) -> bool {
        self.bits() == 0x80000000
    }
    
    fn is_any_zero(&self) -> bool {
        self.is_positive_zero() || self.is_negative_zero()
    }
    
    fn is_nan(&self) -> bool {
        self.exponent_bits() == 0xFF && self.mantissa_bits() != 0
    }
    
    fn is_signaling_nan(&self) -> bool {
        // 信号 NaN：尾数最高位为 0
        self.is_nan() && (self.mantissa_bits() & 0x400000) == 0
    }
    
    fn is_quiet_nan(&self) -> bool {
        // 安静 NaN：尾数最高位为 1
        self.is_nan() && (self.mantissa_bits() & 0x400000) != 0
    }
    
    fn nan_kind(&self) -> Option<&'static str> {
        if !self.is_nan() {
            return None;
        }
        if self.is_signaling_nan() {
            Some("signaling NaN")
        } else {
            Some("quiet NaN")
        }
    }
}

fn main() {
    let inspector = FloatInspector::new(-3.14_f32);
    inspector.print_parts();
    // Float: -3.14
    //   Bits:     11000000010010001111010111000011
    //   Sign:     1
    //   Exponent: 128 (0x80)
    //   Mantissa: 4205555 (0x4028f5)
```

### 练习 3 答案

```rust
use std::mem::ManuallyDrop;

union StringOrFloat {
    s: ManuallyDrop<String>,
    f: f64,
}

// 问题分析：
// 1. process() 中，item.f > 0.0 不是可靠的方式来判断"数据是 float"
//    ——字符串被误读为 f64 会产生任意位模式
// 2. 当 item.s 是活跃的 String 时，item.f 读取到的是 String 的指针/容量/长度
//    的位——不是有效的 f64 值，也不是与 0.0 比较的正确方法
// 3. 直接用 item.f = len 覆盖 ManuallyDrop<String> 会导致 String 泄漏
// 4. 没有 tracking 机制来区分活跃变体

// 修复方案 1：添加 tag 字段（推荐）
#[derive(Debug)]
enum SFKind {
    String,
    Float,
}

struct TaggedSF {
    kind: SFKind,
    value: StringOrFloat,
}

impl TaggedSF {
    fn new_string(s: String) -> Self {
        TaggedSF {
            kind: SFKind::String,
            value: StringOrFloat { s: ManuallyDrop::new(s) },
        }
    }
    
    fn new_float(f: f64) -> Self {
        TaggedSF {
            kind: SFKind::Float,
            value: StringOrFloat { f },
        }
    }
}

impl Drop for TaggedSF {
    fn drop(&mut self) {
        if matches!(self.kind, SFKind::String) {
            unsafe { ManuallyDrop::drop(&mut self.value.s); }
        }
    }
}

fn process_fixed(items: &mut Vec<TaggedSF>) {
    for item in items.iter_mut() {
        match item.kind {
            SFKind::Float => continue,
            SFKind::String => unsafe {
                let len = item.value.s.len() as f64;
                // ✅ 先释放 String
                ManuallyDrop::drop(&mut item.value.s);
                // 修改 kind 和 value
                item.kind = SFKind::Float;
                item.value.f = len;
            },
        }
    }
}

// 修复方案 2：如果不需要 Drop（只用 i64/f64 等 Copy 类型），
// 直接用简单的 tag field 即可，不需要 ManuallyDrop
union SimpleUnion {
    i: i64,
    f: f64,
}

struct SimpleTagged {
    kind: SimpleKind,
    value: SimpleUnion,
}

#[derive(Debug, PartialEq)]
enum SimpleKind {
    Int,
    Float,
}

impl SimpleTagged {
    fn process_and_convert(&mut self) {
        match self.kind {
            SimpleKind::Int => {
                let int_val = unsafe { self.value.i };
                self.kind = SimpleKind::Float;
                self.value.f = int_val as f64;
            }
            SimpleKind::Float => {} // 已经是 float
        }
    }
}
```

### 练习 4 答案

```rust
use std::mem::ManuallyDrop;

#[derive(Debug, PartialEq)]
enum Which {
    First,
    Second,
}

union DataUnion<T1, T2> {
    first: ManuallyDrop<T1>,
    second: ManuallyDrop<T2>,
}

struct TaggedUnion<T1, T2> {
    which: Which,
    data: DataUnion<T1, T2>,
}

impl<T1, T2> TaggedUnion<T1, T2> {
    fn new_first(v: T1) -> Self {
        TaggedUnion {
            which: Which::First,
            data: DataUnion {
                first: ManuallyDrop::new(v),
            },
        }
    }
    
    fn new_second(v: T2) -> Self {
        TaggedUnion {
            which: Which::Second,
            data: DataUnion {
                second: ManuallyDrop::new(v),
            },
        }
    }
    
    fn is_first(&self) -> bool {
        self.which == Which::First
    }
    
    fn is_second(&self) -> bool {
        self.which == Which::Second
    }
    
    fn as_first(&self) -> Option<&T1> {
        if self.is_first() {
            unsafe { Some(&self.data.first) }
        } else {
            None
        }
    }
    
    fn as_second(&self) -> Option<&T2> {
        if self.is_second() {
            unsafe { Some(&self.data.second) }
        } else {
            None
        }
    }
    
    fn as_first_mut(&mut self) -> Option<&mut T1> {
        if self.is_first() {
            unsafe { Some(&mut self.data.first) }
        } else {
            None
        }
    }
    
    fn as_second_mut(&mut self) -> Option<&mut T2> {
        if self.is_second() {
            unsafe { Some(&mut self.data.second) }
        } else {
            None
        }
    }
    
    fn set_first(&mut self, v: T1) {
        // 如果当前是 Second，先 drop
        if self.is_second() {
            unsafe { ManuallyDrop::drop(&mut self.data.second); }
        }
        self.which = Which::First;
        self.data.first = ManuallyDrop::new(v);
    }
    
    fn set_second(&mut self, v: T2) {
        // 如果当前是 First，先 drop
        if self.is_first() {
            unsafe { ManuallyDrop::drop(&mut self.data.first); }
        }
        self.which = Which::Second;
        self.data.second = ManuallyDrop::new(v);
    }
}

impl<T1, T2> Drop for TaggedUnion<T1, T2> {
    fn drop(&mut self) {
        match self.which {
            Which::First => unsafe { ManuallyDrop::drop(&mut self.data.first); },
            Which::Second => unsafe { ManuallyDrop::drop(&mut self.data.second); },
        }
    }
}

// 使用示例：
fn main() {
    let mut tu = TaggedUnion::<i32, String>::new_first(42);
    assert!(tu.is_first());
    assert_eq!(tu.as_first(), Some(&42));
    
    tu.set_second(String::from("hello"));
    assert!(tu.is_second());
    assert_eq!(tu.as_second(), Some(&"hello".to_string()));
    
    // tu 在离开作用域时自动 Drop String
}
```

### 9.1 C++ 的 Union——历史演变和现状

```cpp
// C++98/03：union 成员必须是 POD 类型（Plain Old Data）
// 不能有构造函数、析构函数、虚函数
union OldStyle {
    int i;
    float f;
    // std::string s;  // ❌ 编译错误——有构造/析构函数
};

// C++11：引入了"无限制联合体"（unrestricted union）
// 可以有非 POD 成员——但需要手动管理生命周期
union Unrestricted {
    int i;
    std::string s;  // ✅ C++11 允许——但不自动调用构造/析构
    
    // 你必须用 placement new 和显式析构来管理
    Unrestricted() : i(0) {}   // 默认初始化为 int
    ~Unrestricted() {}          // 不自动析构——你自己负责
};

// 使用 C++11 unrestricted union：
void example() {
    Unrestricted u;
    u.i = 42;  // int 活跃
    
    // 切换到 string：
    new (&u.s) std::string("hello");  // placement new——调用构造
    u.s.append(" world");             // 使用 string
    
    // ⚠️ 必须手动析构
    u.s.~basic_string();  // 显式调用析构函数
    // 不析构的话，"hello world" 的内存泄漏
}

// C++17：std::variant——安全的 tagged union
#include <variant>
std::variant<int, float, std::string> v;
v = 42;                               // 自动管理
v = std::string("hello");             // 自动析构旧的，构造新的
// std::variant 自动处理：存储 discriminant + 内存 + 生命周期
// std::visit() 做安全访问

// 对比 Rust：
// C++ union (C++03)  ≈ Rust union（都无标签，需要 unsafe，手动管理）
// std::variant        ≈ Rust enum（带标签，安全访问）
// 但 Rust 的 enum 比 std::variant 更高效——不需要额外堆分配
```

### 9.2 如果没有 Union——transmute 和指针转换的泛滥

```rust
// 假设 Rust 没有 union，你要实现类型的重解释：

// 方案 A：到处使用 transmute
fn get_float_bits(f: f32) -> u32 {
    unsafe { std::mem::transmute::<f32, u32>(f) }
}
// 问题：大小不匹配时直接编译失败（但一旦编译通过，仍可能是 UB）

// 方案 B：用裸指针重解释
fn get_float_bits_ptr(f: f32) -> u32 {

    unsafe { *(ptr as *const u32) }
}
// 问题：严格别名（strict aliasing）违规——LLVM 可能做意外的优化
//   不确定这是否符合 Rust 的 aliasing 模型

// 方案 C：用 MaybeUninit + ptr::read
fn get_float_bits_maybe(f: f32) -> u32 {
    let mut bytes = [0u8; 4];
    bytes.copy_from_slice(&f.to_le_bytes());
    u32::from_le_bytes(bytes)
}
// 问题：可能有额外拷贝——性能不如 union

// union 解决了所有这些问题：
// - 意图明确（类型系统编码了"共享内存"的语义）
// - 编译器友好（union 的 aliasing 规则清晰）
// - 零开销（和 transmute 生成相同的机器码）

// LLVM 对 union 和 transmute 的处理：
// 两者在优化后通常生成完全相同的 IR——
// 但 union 在 Rust 类型层面提供了更强的语义保证
```

### 9.3 Union 与编译器优化——Aliasing 分析

```rust
// Rust 的 &mut 是独占引用——编译器利用这一点做优化
// union 引入了"共享可变性"——同一个内存位置可以同时被多个
// 字段的引用访问——这会影响别名分析

union AliasingTest {
    a: u32,
    b: u16,
}

fn union_aliasing() {
    let mut u = AliasingTest { a: 0x12345678 };
    
    // 获取不同字段的引用：
    let a_ref = unsafe { &u.a };    // *const u32
    let b_ref = unsafe { &u.b };    // *const u16
    
    // a_ref 和 b_ref 指向同一块内存的（部分）重叠区域
    // 编译器不能假设它们不重叠——可能抑制优化
    
    unsafe {
        // 编译器的视角：
        // a_ref 和 b_ref 是"may-alias"关系
        // 不能假设 *a_ref 在写入 *b_ref 后不变
        u.b = 0;
        // 如果编译器假设 a_ref 不变... 但实际上变了！
        assert_eq!(u.a, 0x12340000);  // 高 2 字节保留，低 2 字节清零
    }
}

// 这就是为什么 union 字段访问需要 unsafe：
// 编译器放弃了关于 union 内部 aliasing 的一些假设
// 你需要保证你的访问模式是安全的
```

### 9.4 Rust 编译器对 Union 的特殊处理

```rust
// Rust 编译器对 union 做了以下特殊处理：

// 1. 没有自动 Drop
//    union 的析构是 no-op——编译器不生成任何 drop glue
//    即使所有字段都是 Drop 类型，也不调用它们的析构

// 2. 移动语义
//    默认情况下，union 不能包含非 Copy 类型
//    （需要 ManuallyDrop wrapper）

// 3. 初始化
//    只能初始化一个字段
//    不能同时初始化多个字段——语法上就不允许
//    let u = MyUnion { a: 1, b: 2.0 };  // ❌ 编译错误

// 4. 模式匹配
//    union 不支持模式匹配（没有 discriminant）
//    let MyUnion { a } = u;  // ❌ 编译错误

// 5. PartialEq / Debug 等 trait
//    编译器不为 union 自动派生任何 trait
//    需要手动 impl

// 6. 布局
//    默认布局（Rust repr）不保证与 C 兼容
//    用 #[repr(C)] 获取与 C 兼容的布局

// 示例——手动实现 Debug：
use std::fmt;

union DebuggableUnion {
    int: i32,
    float: f32,
}

impl fmt::Debug for DebuggableUnion {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        // 默认只打印一个字段——但哪个？
        // 通常的做法是由外部 tag 选择
        unsafe {
            write!(f, "Union({})", self.int)
        }
    }
}
```

### 9.5 实际应用——网络协议头解析

```rust
// 以太网帧头（简化的）：
// offset 0: 目标 MAC (6 bytes)
// offset 6: 源 MAC (6 bytes)
// offset 12: EtherType (2 bytes)
// offset 14: payload (变长，取决于 EtherType)
//
// EtherType = 0x0800 → IPv4
// EtherType = 0x0806 → ARP
// EtherType = 0x86DD → IPv6

use std::mem;

#[repr(C, packed)]  // packed——禁止填充，保证字段紧密排列
struct MacAddr([u8; 6]);

#[repr(C, packed)]
struct EthHeader {
    dst_mac: MacAddr,
    src_mac: MacAddr,
    ethertype: u16,  // 网络字节序（大端）
}

#[repr(C, packed)]
struct ArpPayload {
    htype: u16,
    ptype: u16,
    hlen: u8,
    plen: u8,
    oper: u16,
    // ... 更多字段
}

#[repr(C, packed)]
struct Ipv4Payload {
    version_ihl: u8,
    dscp_ecn: u8,
    total_length: u16,
    // ... 更多字段
}

// 用 union 把不同协议 payload 放在同一位置：
#[repr(C)]
union Payload {
    arp: ArpPayload,
    ipv4: Ipv4Payload,
    raw: [u8; 0],  // 零大小——只是一个类型标记
}

#[repr(C)]
struct EthernetFrame {
    header: EthHeader,
    payload: Payload,
}

fn parse_frame(bytes: &[u8]) {
    if bytes.len() < 14 {
        return;
    }
    
    // 读出 EtherType
    let ethertype = u16::from_be_bytes([bytes[12], bytes[13]]);
    
    let payload_start = 14;
    let payload_bytes = &bytes[payload_start..];
    
    unsafe {
        match ethertype {
            0x0800 => {
                if payload_bytes.len() < mem::size_of::<Ipv4Payload>() {
                    return;
                }
                let ptr = payload_bytes.as_ptr() as *const Ipv4Payload;
                let ipv4 = &*ptr;
                let version = ipv4.version_ihl >> 4;
                let ihl = ipv4.version_ihl & 0x0F;
                println!("IPv4: version={}, IHL={}", version, ihl);
            }
            0x0806 => {
                if payload_bytes.len() < mem::size_of::<ArpPayload>() {
                    return;
                }
                let ptr = payload_bytes.as_ptr() as *const ArpPayload;
                let arp = &*ptr;
                println!("ARP: oper={}", u16::from_be(arp.oper));
            }
            _ => println!("Unknown EtherType: {:#06x}", ethertype),
        }
    }
}
```

### 9.6 设计哲学——Rust 的 Union 为什么"不给你更多魔法"

```rust
// Rust 的 union 刻意保持最小化：
// - 没有 discriminant（你需要自己跟踪）
// - 没有自动 Drop（你需要 ManuallyDrop + 手动管理）
// - 没有安全构造（你需要 unsafe）
// - 没有 derive（你需要手动 impl）

// 为什么不像 C++17 std::variant 一样给个自动类型安全的 union？
// 因为那样做就是 enum——Rust 已经有了。

// Rust 的设计哲学：
// enum = 安全正确的情况——日常使用
// union = 你需要完全控制底层布局的情况——FFI、性能、位操作
//
// 这个二分法和 Rust 其他设计一致：
// Rc/Arc = 安全的共享所有权
// *const T/*mut T = 你需要原始指针的完全控制
// String = 安全的字符串
// *const u8/CStr = FFI 的字符串

// 对比 C++：
// union（C++03）= 有限制，但不需要 unsafe block
// union（C++11）= 无限制，几乎可以放任何东西，但无安全机制
// std::variant（C++17）= 单线程安全，但有堆分配的风险
//
// Rust：清晰的两级——你明确知道什么时候在"不安全"区域
```

### 9.7 `#[repr(C)]` Union 的布局保证

```rust
// #[repr(C)] 保证 union 的布局与 C 兼容：
// 1. 对齐 = 所有字段的最大对齐
// 2. 大小 = 最大字段的大小（对齐后）
// 3. 字段从偏移 0 开始（所有字段的地址相同）

#[repr(C)]
union CCompatible {
    a: u8,
    b: u32,
    c: f64,
}

// C 等价代码：
// typedef union {
//     uint8_t  a;
//     uint32_t b;
//     double   c;
// } CCompatible;

// 从 C 传过来的 union 可以直接 reinterpret cast：
extern "C" {
    fn get_some_union() -> CCompatible;
}

// 但如果不用 #[repr(C)]，Rust 编译器可以自由重排字段
union RustLayout {
    a: u8,
    b: u32,
}
// Rust 的默认布局：不保证与任何 C 编译器兼容
// 不要把它传给 C 代码！

// 还有 #[repr(transparent)] —— 用于单字段 union
// （实际中很少见）
// #[repr(packed)] —— 减少对齐但不跨越 C ABI 边界
```

### 9.8 什么时候该用 Union——决策指南

```rust
// 决策流程图：

// Q1: 你的变体是不是"互斥的数据类型"（一个值可以是 i32 或 String）？
//     是 → Q2
//     不是（只是在做类型重解释，如 f32 ↔ u32）→ Q5

// Q2: 你是不是在封装一个 C union（FFI）？
//     是 → 使用 #[repr(C)] union + 安全 wrapper
//     不是 → Q3

// Q3: 普通 enum 能满足你的需求吗？
//     - 你不需要精确控制布局
//     - 你能接受 discriminant 的开销
//     - 你需要安全模式匹配
//     是 → 使用 enum！这是 Rust 的推荐方式
//     不是 → Q4

// Q4: 你是不是在优化一个被 profiling 证明是瓶颈的 enum？
//     是 → 考虑手动 tagged union（enum tag + union data）
//     不是 → 重新考虑——union 增加了维护负担

// Q5: 你是不是在实现像 f64::from_bits 这样的类型转换？
//     是 → 使用 union（或直接用 to_bits/from_bits 方法）
//     不是 → Q6

// Q6: 你是不是在实现特殊的内存布局（如无锁数据结构的槽位）？
//     是 → union 是合理的选择
//     不是 → 可能不需要 union——用 transmute 更简单

// 黄金法则：先用 enum。只有在以下情况才使用 union：
// 1. FFI 需求（必须映射 C union）
// 2. 被 profiling 证实的性能瓶颈（enum discriminant 开销不可接受）
// 3. 需要精确的内存布局控制（如与硬件寄存器映射）
// 4. 类型重解释（f32/u32 互转——但 transmute 通常也可以）
```

## 10. 回顾

```
Union 学习清单：

□ 理解 union 和 enum 的核心区别：untagged vs tagged
□ 知道 union 的内存布局：size = max(field_sizes)，align = max(field_aligns)
□ 知道 union 的所有字段共享同一块内存——读写同一个地址
□ 会用 unsafe { union.field } 安全地读取数据
□ 理解为什么 union 默认不能包含 Drop 类型（及 ManuallyDrop 的作用）
□ 能手写安全的 tagged union 包装（enum tag + union data）
□ 知道 union 与 C FFI 的对应关系——#[repr(C)] 保证 ABI 兼容
□ 知道 union vs transmute 的选择原则——union 意图更清晰
□ 知道网络协议解析、硬件寄存器映射等 union 的实际应用场景
□ 理解为什么 Rust 的 union 刻意"功能最小化"——enum 做日常工作，union 做底层控制
```

> **一句话总结**：`union` 是所有字段共享同一块内存的无标签联合体——大小等于最大字段，无 discriminant 开销，无运行时类型检查。它是 Rust 安全模型中的一个"受控逃生口"：日常代码用 `enum`（安全、模式匹配、自动 Drop），当你需要精确控制内存布局（FFI、手工 tagged union、位重解释）时用 `union`。Rust 通过 `unsafe` 块、`ManuallyDrop` 和清晰的规则，让 union 的使用比 C 更容易正确。
