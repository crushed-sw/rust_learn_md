# 2.11 自引用结构 & Pin（上）——为什么移动会杀死自引用

> 你在 Rust 中写过这样的结构体吗？一个字段是数据，另一个字段是指向**这个数据的指针**。看起来没问题——直到你把这个结构体**移动**了位置。数据换了地址，指针却还指向旧地址——悬垂！`Pin` 是 Rust 为这个场景设计的解决方案：它保证被 Pin 住的值不会再被移动。理解 Pin 不仅是为了 async/await——它是理解 Rust 如何在"零成本抽象"的前提下保证自引用结构安全的关键。

---

## 1. 自引用结构——移动的致命问题

### 1.1 什么是自引用结构

```rust
// 自引用结构：一个字段包含指向同结构体另一个字段的指针

// 这个结构体还不能直接写成安全的 Rust（之后会说明为什么）：
struct SelfRef {
    data: String,
    ptr: *const String,  // 指向 self.data
}

impl SelfRef {
    fn new() -> Self {
        let mut s = SelfRef {
            data: String::from("hello"),
            ptr: std::ptr::null(),
        };
        s.ptr = &s.data as *const String;  // ptr 指向 data
        s
    }
}
```

### 1.2 移动破坏了指针

```rust
// 问题：当你移动 SelfRef 时

let s1 = SelfRef::new();
// s1 在栈地址 A：
//   data 在 A+0
//   ptr  = A+0（指向 data）

let s2 = s1;  // s1 被 move 到 s2
// s2 在栈地址 B：
//   data 在 B+0  ← 数据被复制到新地址
//   ptr  = A+0   ← 指针还指向旧地址！

// → ptr 现在是悬垂指针！
// unsafe { println!("{}", *s2.ptr); }  // UB！
```

**视觉化**：

```
移动前：                    移动后：
栈地址 A:                  栈地址 B:
┌──────────────┐           ┌──────────────┐
│ data: "hello" │←──────┐  │ data: "hello" │
│ ptr:  ────────┘       │  │ ptr:  ────────┼──→ 指向 A！（悬垂！）
└──────────────┘         │  └──────────────┘
                         │
                         栈地址 A:（可能被其他变量覆盖）
```

## 2. Pin 的核心语义

### 2.1 Pin 的定义

```rust
use std::pin::Pin;

// Pin 的定义（简化）：
pub struct Pin<P> {
    pointer: P,  // P 是某种指针类型（如 &mut T, Box<T>, Rc<T>）
}

// Pin 的保证：
// 如果 P 指向的类型 T 实现了 !Unpin，
// 那么 Pin<P> 保证 T 不会被移动（它的内存地址不会改变）
```

### 2.2 Pin 防止移动——但不能防止"换内容"

```rust
use std::pin::Pin;

let mut value = String::from("hello");
let mut pinned = Pin::new(&mut value);
// Pin::new 要求 T: Unpin —— 所以 String（Unpin）可以创建 Pin

// 但 Pin<&mut String> 保护的是什么？
// - 保护 value 不被移动（不能 std::mem::swap 或 ptr::read）
// - 但 value 的内容可以被修改（String 是 Unpin 的）

// 对于 !Unpin 类型（自引用结构），Pin 才真正"锁住"地址
```

### 2.3 Unpin trait —— "我可以在被 Pin 后还能安全移动"

```rust
use std::marker::Unpin;

// Unpin 是一个 auto trait——绝大多数类型自动实现
// Unpin 的语义："即使我在 Pin 内部，也可以安全地移动我"

// 标准库中自动实现 Unpin 的类型：
// - 所有基本类型：i32, bool, f64, char...
// - 大多数标准类型：String, Vec<T>, Box<T>...
// - &T, &mut T, *const T, *mut T
// - fn(...), 等

// 不自动实现 Unpin 的类型（!Unpin）：
// - async 块生成的 Future
// - Generator（生成器）
// - PhantomPinned —— 标记类型，强制 !Unpin

// 大部分类型是 Unpin 的——所以 Pin 对它们"没效果"
// 只有 !Unpin 的类型才真正受益于 Pin 的保护
```

## 3. PhantomPinned —— 让自己的类型变为 !Unpin

### 3.1 基本用法

```rust
use std::marker::PhantomPinned;
use std::pin::Pin;

// 包含 PhantomPinned 的类型自动是 !Unpin
struct SelfReferential {
    data: String,
    ptr: *const String,
    _pin: PhantomPinned,  // ← 这行让 SelfReferential 变成 !Unpin
}

// 现在 SelfReferential 不能通过"正常方式"移动了
// 如果它在 Pin 内部——移动被阻止
```

### 3.2 安全的构造——用 `Pin<Box<T>>`

```rust
use std::marker::PhantomPinned;
use std::pin::Pin;

struct SelfRef {
    data: String,
    ptr: *const String,
    _pin: PhantomPinned,
}

impl SelfRef {
    fn new(data: String) -> Pin<Box<Self>> {
        // 1. 在堆上分配
        let mut boxed = Box::pin(SelfRef {
            data,
            ptr: std::ptr::null(),
            _pin: PhantomPinned,
        });

        // 2. 设置自引用指针
        // SAFETY: boxed 被 Pin 住了——它的地址不会改变
        //         所以 boxed.data 的地址是稳定的
        let self_ptr: *const String = &boxed.data;
        unsafe {
            let mut_ref: Pin<&mut SelfRef> = Pin::as_mut(&mut boxed);
            Pin::get_unchecked_mut(mut_ref).ptr = self_ptr;
        }

        boxed
    }

    fn get_data(self: Pin<&Self>) -> &String {
        &self.data
    }
}
// 现在 SelfRef 的 ptr 始终有效——因为 Box 的堆地址不会变
```

### 3.3 为什么 `Pin<Box<T>>` 特别适合

```rust
// Pin<Box<T>> 的优势：
// 1. Box<T> 将值放在堆上——堆地址在值生命周期内不变
//    除非显式 *box 或 Box::into_inner
// 2. Box::pin(value) 直接在堆上构造 Pin 值
// 3. Pin<Box<T>> 拥有 T —— 生命周期独立

// Pin<&mut T> 的限制：
// - 借用栈上的值——生命周期受限于栈帧
// - 值在栈上——Pin 虽然阻止"通过 Pin"移动，但不能阻止栈帧弹出

// 所以 Pin<Box<T>> 是自引用结构最常见的形式
```

## 4. Pin 的安全边界

### 4.1 安全的 Pin 操作

```rust
use std::pin::Pin;

// ✅ 安全的 Pin 操作：
// - Pin::new(&mut T) —— 如果 T: Unpin
// - Pin::as_ref() / Pin::as_mut() —— 重新 Pin
// - Pin::get_ref() —— 获取 &T（不破坏 Pin 的保证）
// - Deref / DerefMut —— 如果 Target: Unpin

let mut x = 42;
let pinned = Pin::new(&mut x);  // ✅ i32: Unpin
*pinned = 100;                   // ✅ i32: Unpin → DerefMut 安全
```

### 4.2 不安全的 Pin 操作

```rust
use std::pin::Pin;

// ⚠️ unsafe 的 Pin 操作：
// - Pin::new_unchecked(&mut T) —— 承诺 T: Unpin 或值不会被移动
// - Pin::get_unchecked_mut(Pin<&mut T>) —— 返回 &mut T（不通过 Pin）
// - Pin::into_inner_unchecked(Pin<P>) —— 拆开 Pin

// 这些之所以 unsafe：
// 如果 T 是 !Unpin，通过这些方法获取 &mut T 后，
// 可以用 std::mem::swap 或 std::mem::replace 移动 T
// → 破坏了 Pin 的保证 → 悬垂指针 → UB
```

### 4.3 为什么 Pin::new 要求 T: Unpin

```rust
use std::pin::Pin;

// Pin::new 的签名：
// impl<P: Deref<Target: Unpin>> Pin<P> {
//     pub fn new(pointer: P) -> Pin<P> { ... }
// }

// 为什么需要 Unpin？
// 因为 Pin::new 通过一个"可能不是 Pin 的"指针创建 Pin
// 如果在创建时 T 不是 Unpin——可能有其他引用在移动 T
// → Pin 的保证在创建时就被破坏了

// 对于 !Unpin 类型：
// - Box::pin(value) —— 直接在堆上构造，没有中间引用
// - pin! 宏 —— 在栈上构造，用 shadowing 防止移动
// - unsafe Pin::new_unchecked —— 你承诺不会移动
```

## 5. 常见陷阱

### 陷阱 1：试图移动 `Pin<Box<T>>` 内的值

```rust
use std::pin::Pin;

let pinned: Pin<Box<i32>> = Box::pin(42);

// ❌ 不能：Pin<Box<T>> 没有实现 DerefMut（如果 T: !Unpin）
// 即使 T: Unpin 也不能"移出" Pin 中的值

// let val = *pinned;  // ❌ 不能移出（Deref 返回引用）
// std::mem::swap(&mut *pinned, &mut 0);  // ❌ Pin 阻止了这个
```

### 陷阱 2：混淆 Pin 保护和 Drop 的顺序

```rust
// Pin 保证"地址不变"——但在 Drop 前后 Pin 就不存在了
// Pin 不能阻止值的析构——只阻止在 Pin 存在期间的移动
```

### 陷阱 3：`Pin<&mut T>` 和借用规则

```rust
use std::pin::Pin;

let mut x = 42;
let pinned = Pin::new(&mut x);

// Pin 持有一个可变引用 → 在 Pin 存在期间不能有其他可变引用
// let y = &mut x;  // ❌ 已有一个可变引用（通过 Pin）
// ✅ Pin::as_mut() 可以创建新的 Pin 投影
```

## 7. 练习答案

### 练习 1 答案

```
(a) i32 → Unpin。基本类型都是 Unpin
(b) String → Unpin。大多数标准类型都是 Unpin（自移动是安全的）
(c) Box<i32> → Unpin。Box 拥有堆数据——移动 Box 只移动指针
(d) 包含 PhantomPinned → !Unpin。PhantomPinned 是标记类型——强制 !Unpin
(e) async { 42 } → !Unpin。编译器生成的 Future 状态机可能是自引用的
(f) &mut T → Unpin。引用本身可以安全移动（复制）
```

### 练习 2 答案

```rust
// String 是 Unpin——Pin<&mut String> 的 Pin 保护是"多余的"
// 因为 String 在被移动时不会产生悬垂指针（内部的堆指针是独立的）
// 但 API 仍然正常工作——Pin 只是不提供额外保护

// ✅ 安全。s 不会被移动（因为有 Pin 的可变引用在），
//    但即使没有 Pin，push_str 也是安全的——它修改 *s 的内容，不移动 s
```

### 练习 3 答案

```rust
use std::marker::PhantomPinned;
use std::pin::Pin;

struct Tokenizer {
    source: String,
    current: *const u8,  // 指向 source 内
    _pin: PhantomPinned,
}

impl Tokenizer {
    fn new(source: String) -> Pin<Box<Self>> {
        let mut boxed = Box::pin(Tokenizer {
            source,
            current: std::ptr::null(),
            _pin: PhantomPinned,
        });

        // 获取 source 的起始地址
        let src_ptr = boxed.source.as_ptr();
        unsafe {
            let mut_ref = Pin::as_mut(&mut boxed);
            Pin::get_unchecked_mut(mut_ref).current = src_ptr;
        }

        boxed
    }

    fn next_token(self: Pin<&mut Self>) -> Option<&str> {
        // SAFETY: current 始终指向 self.source 内部——Pin 保证了 source 没被移动
        // 这里省略了具体解析逻辑
        None
    }
}
```

### 练习 4 答案

```
对于 Unpin 类型：
- Pin<&mut T> 的 DerefMut 返回 &mut T
- 这是安全的——因为 T: Unpin 意味着"即使被移动也不会产生悬垂指针"
- 移动 Unpin 类型不会破坏任何内部指针

对于 !Unpin 类型：
- Pin<&mut T> 不能安全实现 DerefMut
- 如果 DerefMut 返回 &mut T，调用方可以用它调用 std::mem::swap
- swap 会移动值 → 破坏 Pin 的保证 → 悬垂指针

所以 Pin 对 !Unpin 类型强制使用 unsafe 的 get_unchecked_mut
→ 每次获取 &mut T 时，程序员都需要手动保证不会用它来移动值
```

## 8. 深度扩展

### 8.1 C++ 的自引用——move 构造函数的责任

```cpp
// C++ 的解决方案：移动构造函数 + 移动赋值
struct SelfRef {
    std::string data;
    const char* ptr;  // 指向 data

    SelfRef(std::string s) : data(std::move(s)), ptr(nullptr) {
        ptr = data.c_str();  // 指向自己的 data
    }

    // 移动构造函数——必须更新 self-pointer！
    SelfRef(SelfRef&& other) noexcept
        : data(std::move(other.data)), ptr(nullptr)
    {
        ptr = data.c_str();  // 重新指向新的 data
        // other.ptr 现在是悬垂的——但 other 即将被析构
    }

    // 移动赋值——也要更新
    SelfRef& operator=(SelfRef&& other) noexcept {
        data = std::move(other.data);
        ptr = data.c_str();
        return *this;
    }
};
// C++ 的问题：
// 1. 程序员必须**记得**写正确的移动构造/赋值——容易遗漏
// 2. 如果移动构造抛异常——自引用指针可能处于无效状态
// 3. 标准库容器在重新分配时可能"移动"元素
//    → std::vector<SelfRef> 的 reallocation 可能破坏 self-pointer
//    → 需要 noexcept 移动构造 + 正确的实现

// Rust 的 Pin 方案：
// 1. 不需要手动实现移动构造（移动在 Rust 中是 memcpy——不能自定义）
// 2. Pin 在类型层面阻止移动——编译期保证"不会移动" = "不会破坏自引用"
// 3. async/await 的 Future 自动生成 !Unpin 状态机——不需要程序员操心
```

### 8.2 如果没有 Pin——自引用只能用 unsafe 和约定

```rust
// 假设没有 Pin，你要实现自引用结构：

struct SelfRef {
    data: String,
    ptr: *const String,
}

impl SelfRef {
    fn new(data: String) -> Self {
        let mut s = SelfRef { data, ptr: std::ptr::null() };
        s.ptr = &s.data;
        s
    }

    // ⚠️ 调用方必须承诺"不会移动"这个结构体
    // 但这是"文档约定"——编译器不强制执行
}

// 如果调用方忘了：
let s1 = SelfRef::new("hello".into());
let s2 = s1;  // move —— ptr 悬垂！
// → UB，但编译器不报错

// 这就是为什么 Pin 是必要的——
// 它把"不要移动"从文档约定变成了类型系统约束
```

### 8.3 Pin 和 Drop 的交互——drop 时 Pin 已经不存在

```rust
use std::pin::Pin;
use std::marker::PhantomPinned;

struct SelfRef {
    data: String,
    ptr: *const String,
    _pin: PhantomPinned,
}

impl Drop for SelfRef {
    fn drop(&mut self) {
        // Drop 接收 &mut self —— 不通过 Pin
        // 在 drop 被调用时，Pin 的保证已经不存在了
        // 但这也是 OK 的——因为数据即将被释放
        // 只要你不"移动" self（即不把所有权传给其他地方）

        // ✅ 安全：只是读取
        unsafe { println!("dropping: {}", *self.ptr); }
    }

    // ❌ 危险：在 Drop 中把 self 的所有权传给其他地方（比如放到 Vec）
    // 但 Rust 的类型系统通常阻止这种情况——&mut self 不能"移走"
}
```

### 8.4 Pin 和 trait object——Box<dyn Future>

```rust
use std::pin::Pin;
use std::future::Future;

// async fn 返回的 Future 是 !Unpin
// 为了存储在 Vec 中或在线程间传递——需要 Pin<Box<dyn Future>>

fn store_futures() -> Vec<Pin<Box<dyn Future<Output = i32>>>> {
    let f1 = async { 1 };
    let f2 = async { 2 };

    vec![
        Box::pin(f1),  // Pin<Box<impl Future>> → Pin<Box<dyn Future>>
        Box::pin(f2),
    ]
}

// 这是 tokio::spawn 的基础——
// 它接受 Pin<Box<dyn Future<Output = T> + Send>>
// 因为：
// 1. Box —— 大小固定（可以在堆上管理）
// 2. Pin —— 保证 Future 不会被移动
// 3. dyn Future —— 类型擦除——不同 async 块可以放在一起
// 4. Send —— 可以在线程间传递
```

### 8.5 Pin 的 "structural pinning" vs "non-structural pinning"

```rust
// Structural Pinning（结构性 Pin）：
// Pin 的保证延伸到结构体的**所有字段**
// 通过 Pin<&mut Struct> 获取 Pin<&mut Field>
// → 字段也被 Pin 住了

// Non-structural Pinning（非结构性 Pin）：
// 你承诺"这个字段的地址可以改变，不影响 Pin 的保证"
// 这种情况很少见——大多数时候你用的是结构性 Pin

// 实际中：
// - Pin<Box<Self>>：Box 的所有权意味着结构性 Pin——整个 Self 在堆上固定
// - Pin<&mut Self>：如果 Self 有 PhantomPinned，所有字段都不应该被移动
```

### 8.6 `pin!` 宏——栈上 Pin

```rust
use std::pin::pin;

// pin! 宏在栈上创建 Pin 值（nightly → stable 1.68）
async fn example() {
    let future = async { 42 };
    let pinned = pin!(future);  // Pin<&mut impl Future> 在栈上

    // pinned 现在是 Pin<&mut impl Future>
    // 它把 future "锁"在了栈上当前位置
    // 直到 pinned 离开作用域

    // 限制：
    // - 生命周期绑定到当前栈帧
    // - 不能从函数返回（借用）
}
```

### 8.7 `Pin::set` —— 替换 Pin 内的值

```rust
use std::pin::Pin;

let mut x = 42;
let mut pinned = Pin::new(&mut x);

// Pin::set 可以替换 Pin 内的值：
pinned.set(100);
// 这安全吗？对 Unpin 类型——是的（i32: Unpin）
// 对 !Unpin 类型——Pin::set 不可用

// 原理：
// set 相当于 *pinned = 100——直接覆盖内存
// 不移动地址——只改变内容
// 所以不会破坏自引用指针（如果它们指向的是这块内存的"偏移"而不是"旧数据"）
```

### 8.8 Pin 和 async 块——编译器自动生成 !Unpin

```rust
// async 块被编译为状态机：
// async fn foo() {
//     let x = 42;
//     some_async_fn().await;  // .await 点 = 状态机挂起点
//     println!("{}", x);      // x 必须在 .await 前后都在同一个地址
// }

// 编译器生成的 Future 状态机：
// enum FooFuture {
//     Start,
//     AfterAwait { x: i32, sub_future: SomeAsyncFuture },
//     Done,
// }

// 问题：枚举变体切换时——相当于"移动"
// 但 async 块内部可能有自引用——所以编译器给 Future 加了 PhantomPinned
// → Future 是 !Unpin

// 这就是为什么你需要 Pin<&mut Future> 或 Pin<Box<dyn Future>> 来 poll
// Pin 保证 Future 状态机的地址在 poll 之间不变
```

```
□ 知道 async 块的 Future 是 !Unpin——这就是为什么需要 Pin
```

> **一句话总结**：自引用结构的致命弱点是"移动会导致自指针悬垂"。C++ 通过自定义移动构造函数来"修复"移动后的指针，但依赖程序员记得写（且写对）。Rust 的 `Pin` 走了完全不同的路——它在类型层面阻止移动：`Pin<Box<T>>` 把值钉在堆上，`Pin<&mut T>` 把值钉在栈上，`!Unpin` 的类型在 Pin 内部彻底不可移动。async/await 的 Future 状态机天然是 `!Unpin`——这就是为什么你每次 `.await` 的背后都有 Pin 在默默工作。
