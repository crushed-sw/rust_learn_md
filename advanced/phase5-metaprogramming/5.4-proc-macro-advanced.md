# 5.4 过程宏进阶 —— 从"能跑"到"专业"

> 能生成代码只是入门——能让使用者愉快地调试、能提供精准的编译错误信息、能在 CI 中被自动化测试验证，这才是专业的宏开发者。Span 诊断是过程宏的"错误处理系统"，trybuild 是宏的"单元测试框架"，性能意识是"对同事编译时间的尊重"。这三者加在一起，构成了从玩具宏到生产级宏的质变。

## 1. Span 诊断系统精讲

### 1.1 什么是 Span ——"代码的 GPS 坐标"

Span 是编译器用来追踪"这段代码来自源文件的哪个位置"的数据结构。当你写一个过程宏时，输入的 `TokenStream` 中的每一个 token 都携带 Span 信息——它记录了该 token 在源文件中的行号、列号、文件路径。Span 是编译器诊断系统的基础：只有知道"哪里出了问题"，你才能给出精准的错误提示。

```rust
// 过程宏的本质：操纵带位置信息的 token
// TokenStream = 一串 (TokenTree × Span) 的序列
//
// 当你写：
//   #[my_macro]
//   struct Foo { x: i32 }
//
// proc_macro 收到的 TokenStream 中：
// - struct → Span(行2, 列1)
// - Foo    → Span(行2, 列8)
// - {      → Span(行2, 列12)
// - x      → Span(行2, 列14)
// - :      → Span(行2, 列15)
// - i32    → Span(行2, 列17)
// - }      → Span(行2, 列20)
//
// 有了这些位置信息，你就能告诉用户：
// "第2行第17列的 i32 有问题——这里应该用 String"
```

### 1.2 proc_macro::Span vs proc_macro2::Span —— 为什么要两个 Span 类型

新手常有一个困惑：为什么有 `proc_macro::Span` 和 `proc_macro2::Span` 两个东西？为什么几乎所有第三方宏都用 `proc_macro2` 而不是直接用标准库的 `proc_macro`？

```rust
//──────────────────────────────────────────────────────────────
// 核心原因：proc_macro 只能在 proc-macro = true 的 crate 中使用
//──────────────────────────────────────────────────────────────

// ❌ 这样做不了——proc_macro 的 TokenStream/Span 不实现 Send/Sync
// 不能在普通 crate 中引用、不能在线程间传递、不能用常规测试框架测试
//
// 但你的宏逻辑通常很复杂——你想把它拆分成：
//   my_macro/           # proc-macro = true   → 薄包装层
//   my_macro_core/      # 普通 crate          → 所有宏逻辑 + 可测试
//
// proc_macro2 就是 bridge——它提供的类型和 proc_macro 中的对应类型
// 可以互相转换（通过 .into()），但 proc_macro2 的类型可以在任何 crate 中使用。

use proc_macro2::TokenStream as TokenStream2;
use proc_macro2::Span as Span2;

// 在 proc-macro crate 的入口函数中转换：
// #[proc_macro_derive(MyDerive)]
// pub fn my_derive(input: proc_macro::TokenStream) -> proc_macro::TokenStream {
//     let input: TokenStream2 = input.into();  // proc_macro → proc_macro2
//     let output = my_core_logic(input);       // 在普通 crate 中实现的逻辑
//     output.into()                            // proc_macro2 → proc_macro
// }
```

### 1.3 Span 的关键方法

```rust
use proc_macro2::Span;

//──────────────────────────────────────────────────────────────
// 方法 1: Span::call_site()
//──────────────────────────────────────────────────────────────
// 含义：宏的"调用位置"。生成的代码会被视为写在调用宏的那个位置。
//
//   // 在 crate A 中定义：
//   macro_rules! my_macro {
//       () => { /* Span::call_site() → crate A 中的 my_macro!() 调用点 */ }
//   }
//
//   // 在 crate B 中使用：
//   my_macro!();  // call_site = crate B 中的 my_macro!() 这一行
//
// 效果：以 call_site 为 Span 生成的标识符只能在调用方的上下文中解析。
//       如果生成的代码引用了调用方的局部变量——能工作。
//       如果生成的代码引用了宏定义方的私有函数——不能工作（不在作用域内）。

//──────────────────────────────────────────────────────────────
// 方法 2: Span::mixed_site()
//──────────────────────────────────────────────────────────────
// 含义：半 hygienic。标识符的"定义上下文"是宏的定义位置，
//       但可见性检查在调用位置进行。
//
// 这解决了声明宏（macro_rules!）的一个经典问题：
//   宏内部生成的辅助函数/类型对调用方不可见（因为是在 def_site 定义的）
//   但宏又想引用调用方上下文中的名字
//
// 对于过程宏，mixed_site 是常用选择——
// 你既要引用自己 crate 中的辅助类型，又要让生成的代码适应用户的上下文。

//──────────────────────────────────────────────────────────────
// 方法 3: Span::def_site()（需要 nightly）
//──────────────────────────────────────────────────────────────
// 含义：宏的"定义位置"。生成的代码被视为写在宏定义的那个位置。
// 用于解决跨 crate 的 hygiene 问题——让宏生成的标识符能访问宏定义 crate 的私有项。

//──────────────────────────────────────────────────────────────
// 方法 4: span.source_text()
//──────────────────────────────────────────────────────────────
// 含义：获取该 Span 对应的源代码文本。
// 极其强大——你可以在宏中读到使用者写的内容，做更精准的错误提示。

fn example_source_text() {
    // 假设用户在 struct Foo { x: i32 } 上使用了你的宏
    // 你可以遍历字段，对每个字段取出它的 source_text()：
    //   span.source_text() → Some("x")
    //   span.source_text() → Some("i32")
    //
    // 然后可以用它构建更友好的错误信息：
    //   "字段 'x' 的类型 i32 不被支持——请使用 String 或 i64"
}
```

### 1.4 从 token 创建 Span —— 保留源位置

```rust
use proc_macro2::{TokenStream, Span};
use quote::quote;

// 当 quote! 生成代码时，你可以用 quote_spanned! 来指定 Span：
fn demonstrate_quote_spanned() {
    let some_type: TokenStream = syn::parse_str("MyType").unwrap();
    // some_type 中的 token 带有调用方的 Span 信息

    // 方式 A：quote! —— 使用 Span::call_site()
    let code_a = quote! {
        impl #some_type {
            fn generated_method(&self) {}
        }
    };
    // impl MyType 的 Span = call_site
    // generated_method 的 Span = call_site

    // 方式 B：quote_spanned! —— 使用用户提供的 Span
    // 假设你从用户输入的 #[attr] 中拿到了 attr 的 Span
    let attr_span = Span::call_site(); // 实际中从 syn::Attribute 拿
    let code_b = quote_spanned! { attr_span =>
        impl #some_type {
            fn generated_method(&self) {}
        }
    };
    // 现在整个 impl 块的 Span = attr_span
    //
    // 为什么重要？—— 错误信息会指向 attr 的位置，而不是宏内部生成的代码
    // 用户看到："第5行的 #[my_attr] 有问题" ← 有意义的错误
    // 而不是："<proc_macro generated code>:1:1: error" ← 无意义的错误

    // 方式 C：在 quote! 中混用 —— 给局部内容指定 Span
    let field_span = Span::call_site(); // 实际中从字段名取
    let code_c = quote! {
        impl SomeTrait for #some_type {
            fn field_name() -> &'static str {
                #field_span "my_field"  // 这个字符串字面量的 Span = field_span
            }
        }
    };
    // 字符串字面量 "my_field" 会绑定到 field_span——
    // 如果这里出了问题，编译器指向的就是用户的字段位置。
}
```

### 1.5 call_site vs mixed_site 的完整对比示例

```rust
//──────────────────────────────────────────────────────────────
// 场景：你在写一个 derive 宏，生成的代码需要访问自己 crate 中的辅助函数
//──────────────────────────────────────────────────────────────

// 在你的宏 crate 中：
mod helpers {
    pub fn secret_helper() -> String {
        "hello from helper".to_string()
    }
}

// 你的 derive 宏生成了如下代码（简化）：
// 假设 Span 用的是 call_site：
//
//   impl MyTrait for UserType {
//       fn my_method(&self) -> String {
//           // 编译错误！helpers::secret_helper 不在调用方的上下文中
//           // 因为 call_site 指的是用户的 crate
//           helpers::secret_helper()
//       }
//   }
//
// 这会导致编译失败——用户在 crate 中（调用方）没有 helpers 这个模块。
//
// 解决方案 1（推荐）：用你的 crate 的绝对路径
fn solution_1() {
    // 把生成的代码改成：
    //   ::my_macro_crate::helpers::secret_helper()
    // 而不是：
    //   helpers::secret_helper()
    //
    // 这样无论 Span 是什么，路径都是确定的。
}

// 解决方案 2：用 mixed_site
fn solution_2() {
    // Span::mixed_site() 允许生成的标识符在宏定义方的上下文中解析
    // 但使用上有限制——不是所有场景都适用
    // 更好的实践永远是显式使用绝对路径。
}
```

## 2. 编译错误诊断 —— Span::error / warning / note / help

### 2.1 syn::Error —— 过程宏的"自定义编译错误"

过程宏不能在编译期执行 `println!`，也不能 `panic!`（panic 在宏中会导致 ICE——Internal Compiler Error，这对用户来说完全是黑盒）。正确的做法是用 `syn::Error`：

```rust
use proc_macro2::Span;
use syn::{Error, LitStr};

//──────────────────────────────────────────────────────────────
// 基本模式：Error::new(span, message) → into_compile_error()
//──────────────────────────────────────────────────────────────
fn basic_error_demo(input_span: Span) -> proc_macro::TokenStream {
    // 创建一个错误
    let err = Error::new(input_span, "这里有问题：字段名必须以 'my_' 开头");

    // 将其转化为 TokenStream —— 编译器会将其作为编译错误输出
    err.into_compile_error().into()
    // 注意：你仍然需要返回一个 TokenStream！
    // into_compile_error() 返回的是 proc_macro2::TokenStream
    // 然后 .into() 转为 proc_macro::TokenStream
}

//──────────────────────────────────────────────────────────────
// 多部分诊断：error + note + help
//──────────────────────────────────────────────────────────────
fn multi_part_diagnostic(struct_name: &syn::Ident, field_span: Span) {
    // 基础错误
    let mut err = Error::new(
        struct_name.span(),
        format!("结构体 `{}` 缺少必需的属性", struct_name),
    );

    // 添加 note —— 就像你在编译错误中看到的 "= note: ..."
    err.combine(Error::new(
        field_span,
        "这个字段没有 #[required] 属性",
    ));

    // 方式 1：用 .note() 添加补充说明（配合 nightly 的 proc_macro_diagnostic）
    // （当前稳定版 syn 使用 combine 来组合多个 Error）
    let note_err = Error::new(
        field_span,
        "注意：所有公开字段都需要标记 #[required] 或 #[optional]",
    );

    // 方式 2：添加 help 信息
    let help_err = Error::new(
        field_span,
        "help: 请添加 #[required] 属性，例如：\n    #[required]\n    pub field: String",
    );

    // 将所有错误合并
    let combined = vec![err, note_err, help_err]
        .into_iter()
        .fold(Error::new(Span::call_site(), ""), |mut acc, e| {
            acc.combine(e);
            acc
        });
    // 注意：这个 fold 是以一个"空" Error 为起点的，
    // 最终返回的 Error 会包含所有子诊断。
    // 实际编译时：
    //   error: 结构体 `Foo` 缺少必需的属性
    //     --> src/main.rs:5:1
    //   note: 这个字段没有 #[required] 属性
    //     --> src/main.rs:7:5
    //   help: 请添加 #[required] 属性...
}

//──────────────────────────────────────────────────────────────
// 收集多个错误再统一发射 —— "不要一遇到错误就 return"
//──────────────────────────────────────────────────────────────
fn collect_all_errors(fields: &[syn::Field]) -> Result<TokenStream, Error> {
    let mut errors = Vec::new();

    for field in fields {
        // 做各种检查，把错误收集起来
        if field.ident.as_ref().map(|i| i.to_string().starts_with("_")).unwrap_or(false) {
            errors.push(Error::new(
                field.span(),
                "字段名不能以下划线开头——这会让字段被忽略",
            ));
        }

        // 继续检查其他条件...不提前 return
    }

    if errors.is_empty() {
        Ok(/* 成功的 TokenStream */)
    } else {
        // 合并所有错误
        let mut combined = errors.remove(0);
        for e in errors {
            combined.combine(e);
        }
        Err(combined)
    }
}
// 为什么收集所有错误？
//   ❌ 坏做法：遇到第一个错误就 return Error → 用户修完一个又看到一个
//   ✅ 好做法：收集所有错误一次性报告 → 用户可以一次性全部修完
//   就像 rustc 本身做的那样——尽量多报告错误，而不是在第一个错误处停下来。
```

### 2.2 实战案例：`#[sorted]` derive 的诊断设计

下面是一个具体例子——一个检查 enum variant 是否按字母排序的 derive 宏。我们重点看它的诊断输出。

```rust
use proc_macro2::{Span, TokenStream};
use quote::quote;

// 用户写了：
//   #[derive(Sorted)]
//   enum MyEnum {
//       Zebra,   // ← "Z"
//       Apple,   // ← "A" —— 不在字母序！应该排在 Zebra 前面
//       Banana,  // ← "B"
//   }
//
// 我们应该发出的诊断信息（理想的）：
//   error: variants are not in alphabetical order
//     --> src/main.rs:3:5
//      |
//    3 |     Zebra,
//      |     ^^^^^ first out-of-order variant
//    4 |     Apple,
//      |     ----- "Apple" should come before "Zebra"
//      |
//   help: move the "Apple" variant above "Zebra"

fn sorted_diagnostic(variants: &[syn::Variant]) -> Result<TokenStream, syn::Error> {
    for window in variants.windows(2) {
        let a = &window[0].ident.to_string();
        let b = &window[1].ident.to_string();

        if a > b {
            // 发现无序：a 应该排在 b 之后（因为 a > b 在字母序上）
            // 但 a 在 b 的前面 → 错误！b 应该排在 a 前面
            let mut err = syn::Error::new(
                variants[0].span(), // 指向 enum 的定义位置
                "variants are not in alphabetical order",
            );

            // 标出 b 的 Span，说明它不在正确位置
            err.combine(syn::Error::new(
                variants[window_start + 1].ident.span(),
                format!(r#""{}" should come before "{}""#, b, a),
            ));

            // 帮助信息
            err.combine(syn::Error::new(
                variants[window_start + 1].ident.span(),
                format!(r#"help: move the "{}" variant above "{}""#, b, a),
            ));

            return Err(err);
        }
    }

    // 全部有序，生成空的 impl（只做编译期检查）
    Ok(quote! {
        // proc_macro 宏至少需要输出一些代码
        // 我们输出一个空的 impl 块——真正的"工作"只有编译期检查
        const _: () = ();
    })
}
// 注意：这个例子展示的是原理。实际的 #[sorted] 需要处理更多情况，
// 比如带字段的变体、带属性的变体等。章节末尾的练习 3 会要求你完整实现。
```

### 2.3 诊断信息质量的对比 —— Before / After

```rust
//──────────────────────────────────────────────────────────────
// BEFORE：没有 Span，没有 note，没有 help —— 糟糕的诊断
//──────────────────────────────────────────────────────────────
fn bad_diagnostic_wrong_type(field: &syn::Field) -> TokenStream {
    // ❌ 最糟糕的做法：没有 Span，没有有帮助的信息
    syn::Error::new(
        Span::call_site(),  // 指向宏调用位置——用户不知道到底是哪里
        "type error",        // 什么样的 type error？哪个字段？
    ).into_compile_error()
}

// 编译输出：
//   error: type error
//     --> src/main.rs:1:1   ← 指向宏调用位置，毫无帮助
//      |
//    1 | #[derive(MyDerive)]
//      | ^^^^^^^^^^^^^^^^^^^
//      |
//   (没有 note，没有 help，用户完全不知道该怎么办)


//──────────────────────────────────────────────────────────────
// AFTER：精确定位 + 上下文 + 修复建议 —— 专业的诊断
//──────────────────────────────────────────────────────────────
fn good_diagnostic_wrong_type(field: &syn::Field) -> TokenStream {
    let field_name = field.ident.as_ref().unwrap();
    let ty = &field.ty;

    let mut err = syn::Error::new(
        ty.span(),  // ← 精确指向类型的 Span
        format!(
            "字段 `{}` 的类型不被支持",
            field_name,
        ),
    );

    // 提取源代码文本（如果能的话）
    if let Some(source) = ty.span().source_text() {
        err.combine(syn::Error::new(
            ty.span(),
            format!("不支持的类型: `{}`", source),
        ));
    }

    err.combine(syn::Error::new(
        ty.span(),
        "note: 仅支持以下类型: String, i32, bool, Vec<String>",
    ));

    err.combine(syn::Error::new(
        field.span(),
        format!(
            "help: 将字段 `{}` 的类型改为支持的类型，例如:\n    {}: String",
            field_name, field_name,
        ),
    ));

    err.into_compile_error()
}

// 编译输出：
//   error: 字段 `name` 的类型不被支持
//     --> src/main.rs:10:17    ← 精确指向 name 字段的类型位置
//      |
//   10 |     pub name: u64,
//      |               ^^^
//      |
//   note: 不支持的类型: `u64`
//     --> src/main.rs:10:17
//   note: 仅支持以下类型: String, i32, bool, Vec<String>
//     --> src/main.rs:10:17
//   help: 将字段 `name` 的类型改为支持的类型，例如:
//         name: String
//     --> src/main.rs:10:5
//
// 这种诊断质量就是 "专业" 和 "玩具" 的分界线。
```

## 3. trybuild —— 测试编译期行为

### 3.1 哲学：测试"应该编译失败"的代码

传统单元测试验证的是"代码运行时行为正确"。但过程宏的很多行为是**编译期**的：
- "当用户遗漏了必需属性，宏应该**拒绝编译**并给出清晰的错误信息"
- "当 enum variant 无序时，宏应该**拒绝编译**并指出哪个 variant 有问题"

这些行为没法用 `#[test]` 测试——因为被测代码根本通不过编译。

trybuild 的答案是：

> 把"编译失败的代码"作为测试用例，把"编译器错误输出"作为预期结果。如果代码按预期失败且错误信息匹配——测试通过。

```toml
# Cargo.toml
[dev-dependencies]
trybuild = "1.0"

# 同时确保你的宏 crate 本身在 dev-dependencies 也可用
# [dev-dependencies]
# my_macro = { path = "." }  # 如果宏和测试在同一 crate
```

### 3.2 项目结构

```
my_macro/
├── Cargo.toml           # proc-macro crate
├── src/
│   └── lib.rs           # #[proc_macro_derive(Sorted)]
└── tests/
    ├── ui/                  # trybuild 测试用例
    │   ├── pass_sorted.rs       # 应该编译通过的用例
    │   ├── fail_unsorted.rs     # 应该编译失败的用例
    │   ├── fail_unsorted.stderr # ← 预期的错误输出（关键文件）
    │   └── fail_not_enum.rs     # 用在 struct 上应该失败
    │   └── fail_not_enum.stderr
    ├── progress.rs         # 可选：如果用例需要逐步测试
    └── trybuild_test.rs    # 驱动 trybuild 的 #[test]
```

### 3.3 trybuild 完整工作流

```rust
//──────────────────────────────────────────────────────────────
// Step 1: 创建测试驱动文件 tests/trybuild_test.rs
//──────────────────────────────────────────────────────────────

#[test]
fn ui_tests() {
    let t = trybuild::TestCases::new();

    // 应该编译通过的文件
    t.pass("tests/ui/pass_sorted.rs");

    // 应该编译失败的文件（需要有对应的 .stderr）
    t.compile_fail("tests/ui/fail_unsorted.rs");
    t.compile_fail("tests/ui/fail_not_enum.rs");
}

//──────────────────────────────────────────────────────────────
// Step 2: 编写测试用例 tests/ui/fail_unsorted.rs
//──────────────────────────────────────────────────────────────
use my_macro::Sorted;

#[derive(Sorted)]
enum Unsorted {
```

## 4. proc_macro2 深度

### 4.1 为什么 proc_macro2 是必需品

```rust
//──────────────────────────────────────────────────────────────
// proc_macro 的致命限制：
//──────────────────────────────────────────────────────────────
// 1. proc_macro::TokenStream 只能在 proc-macro crate 的
//    #[proc_macro]、#[proc_macro_derive]、#[proc_macro_attribute]
//    函数中使用。你不能在普通函数中使用它。
//
// 2. proc_macro::TokenStream 不实现 Send / Sync ——
//    不能用在线程池、异步任务中。
//
// 3. proc_macro::TokenStream 不实现 Clone ——
//    每次操作后原始的 TokenStream 被消费。
//
// 4. proc_macro::Span 没有 source_text() ——
//    你需要 proc_macro2::Span 来获取源代码文本。
//
// 5. proc_macro 的类型不能在普通测试中构造 ——
//    不能写 #[test] fn test_my_macro() { ... }
//    因为 proc_macro::TokenStream 只能在编译器进程中使用。

//──────────────────────────────────────────────────────────────
// proc_macro2 的解决方案：
//──────────────────────────────────────────────────────────────
// 1. proc_macro2::TokenStream 可以在任何 crate 中使用。
// 2. 实现了 Send + Sync + Clone。
// 3. 支持 source_text()、join()、located_at() 等高级 Span 操作。
// 4. 可以在 #[test] 中构造和操作。
// 5. 通过 .into() 与 proc_macro::TokenStream 互转。

// 标准的三层架构：
//
// ┌────────────────────────────────────────────────────┐
// │ my_macro/  (proc-macro = true)                     │
// │   src/lib.rs                                       │
// │     #[proc_macro_derive(MyDerive)]                 │
// │     pub fn my_derive(input: TS) -> TS {            │
// │       let input = TS2::from(input);  ← 入口转换    │
// │       let output = my_core::derive(input);         │
// │       TS::from(output)               ← 出口转换    │
// │     }                                              │
// └────────────────────────────────────────────────────┘
//           ↓ 依赖
// ┌────────────────────────────────────────────────────┐
// │ my_core/  (普通 crate)                             │
// │   包含所有宏逻辑 + 辅助函数 + 类型定义              │
// │   全部使用 proc_macro2::TokenStream                │
// │   可以被 #[test] 测试                               │
// └────────────────────────────────────────────────────┘
```

### 4.2 proc_macro2 的 feature flags

```toml

# 默认只有基础功能

# ── 关键 feature ──

# 只在你的 proc-macro crate 中需要！
# my_core/ 不需要此 feature
proc-macro2 = { version = "1.0", features = ["proc-macro"] }

# "span-locations": 启用 Span 的行列号查询

# 仅在需要构建自定义诊断格式化时使用
proc-macro2 = { version = "1.0", features = ["span-locations"] }

# "extra-traits": 为 Span 实现 Debug / PartialEq / Eq / Hash
# 在测试代码中非常有用——你可以 assert_eq!(span1, span2)
[dev-dependencies]

```

```rust
// feature "proc-macro" 的核心作用：
// 启用后，proc_macro2::TokenStream 实现了：
//   From<proc_macro::TokenStream> for proc_macro2::TokenStream
//   From<proc_macro2::TokenStream> for proc_macro::TokenStream
```

### 4.3 TokenStream 的手动操作 —— 不用 quote!

```rust
use proc_macro2::{TokenStream, TokenTree, Ident, Punct, Literal, Group, Delimiter, Span};

//──────────────────────────────────────────────────────────────
// TokenStream 的手动构造 —— 完全绕过 quote! 宏
//──────────────────────────────────────────────────────────────
// 
// 在某些极端性能场景下（需要生成海量代码），
// quote! 的运行时开销可能成为瓶颈。此时可以直接构造 token。

fn manual_token_construction() -> TokenStream {
    // 目标：生成 impl Foo { fn bar(&self) -> i32 { 42 } }

    // Step 1: 构造 impl 关键字
    let impl_kw = TokenTree::Ident(Ident::new("impl", Span::call_site()));

    // Step 2: 构造类型名
    let type_name = TokenTree::Ident(Ident::new("Foo", Span::call_site()));

    // Step 3: 构造花括号内的内容
    let mut inner = TokenStream::new();

    // fn bar(&self) -> i32 { 42 }
    inner.extend(vec![
        TokenTree::Ident(Ident::new("fn", Span::call_site())),
        TokenTree::Ident(Ident::new("bar", Span::call_site())),
        // (&self)
        TokenTree::Group(Group::new(
            Delimiter::Parenthesis,
            {
                let mut args = TokenStream::new();
                args.extend(vec![
                    TokenTree::Punct(Punct::new('&', proc_macro2::Spacing::Alone)),
                    TokenTree::Ident(Ident::new("self", Span::call_site())),
                ]);
                args
            },
        )),
        // ->
        TokenTree::Punct(Punct::new('-', proc_macro2::Spacing::Joint)),
        TokenTree::Punct(Punct::new('>', proc_macro2::Spacing::Alone)),
        // i32
        TokenTree::Ident(Ident::new("i32", Span::call_site())),
        // { 42 }
        TokenTree::Group(Group::new(
            Delimiter::Brace,
            {
                let mut body = TokenStream::new();
                body.extend(vec![
                    TokenTree::Literal(Literal::i32_suffixed(42)),
                ]);
                body
            },
        )),
    ]);

    let impl_body = TokenTree::Group(Group::new(Delimiter::Brace, inner));

    // Step 4: 组装完整的 impl 块
    let mut result = TokenStream::new();
    result.extend(vec![impl_kw, type_name, impl_body]);

    result
}

// 输出（概念上）：
// impl Foo {
//     fn bar(&self) -> i32 { 42i32 }
// }

// 对比用 quote! 的版本：
fn with_quote() -> TokenStream {
    quote::quote! {
        impl Foo {
            fn bar(&self) -> i32 { 42i32 }
        }
    }
}
// quote! 版本在可读性和可维护性上完胜。
// 手动构造仅在极端性能场景下考虑——绝大多数情况下不需要。
```

### 4.4 TokenTree 枚举的四种变体

```rust
// TokenTree 是 Rust 语法树的最小单元：
//
// pub enum TokenTree {
//     Group(Group),    // { ... }、[ ... ]、( ... )
//     Ident(Ident),    // 标识符：foo、self、i32、crate
//     Punct(Punct),    // 标点符号：+、-、>、::、;
//     Literal(Literal),// 字面量："hello"、42、3.14、true
// }
//
// 注意：KeyWord 不是独立的 TokenTree！
// fn、struct、impl 等在 proc_macro2 中都是 Ident。

use proc_macro2::{Group, Delimiter, Spacing};

// Ident 的构造
let ident = Ident::new("my_function", Span::call_site());
// Ident::new 会检查是否是 Rust 关键字——如果是，不会 panic，
// 但生成的代码可能在后续编译中由于关键字冲突而报错。

// Punct 的 Spacing 语义
let plus_alone = Punct::new('+', Spacing::Alone);  // 独立的 +
let arrow_joint = Punct::new('-', Spacing::Joint); // - 后面必须紧跟 >
let arrow_alone = Punct::new('>', Spacing::Alone); // > 可以独立存在
// Spacing::Joint 表示这个标点与其后的标点构成一个复合标点（如 ->、::、+=）

// Group 的 Delimiter
let brace_group = Group::new(Delimiter::Brace, inner_tokens);      // { ... }
let bracket_group = Group::new(Delimiter::Bracket, inner_tokens);   // [ ... ]
let paren_group = Group::new(Delimiter::Parenthesis, inner_tokens); // ( ... )
let none_group = Group::new(Delimiter::None, inner_tokens);         // ...（无定界符）

// Literal 的构造
let int_lit = Literal::i32_suffixed(42);       // 42i32
let str_lit = Literal::string("hello");        // "hello"
let byte_str = Literal::byte_string(b"hello"); // b"hello"
let char_lit = Literal::character('x');        // 'x'
let bool_lit_true = Literal::usize_suffixed(1); // 只能用数字模拟...
// 注意：proc_macro2::Literal 没有 bool 构造函数，
// 通常用 Ident::new("true", span) 来表示布尔字面量。
```

## 5. 性能优化实战

### 5.1 过程宏的编译期成本来源

```rust
// 过程宏的性能瓶颈不是"生成的代码跑得慢"——
// 而是"生成代码的过程太慢"。每次宏调用都是：
//
//   1. 解析 TokenStream → AST（如果有 syn，这一步最重）
//   2. 分析/转换 AST
//   3. 生成 TokenStream → 输出
//
// 对于 derive 宏，每个被标注的类型都是一次宏调用。
// 如果一个 crate 中有 100 个结构体 #[derive(MyDerive)]，
// 你的宏就会被调用 100 次。

// 典型的热点：
// 1. syn::parse2() / syn::parse_macro_input!() —— 全量解析
// 2. 大量的 String 操作（format!、to_string()）
// 3. 复杂的类型分析（syn 的类型遍历）
// 4. 组合爆炸（为每个字段 × 每个类型组合生成代码）
```

### 5.2 优化策略详解

```rust
//──────────────────────────────────────────────────────────────
// 策略 1: 最小化 syn 的使用 —— 用轻量级解析代替全量解析
//──────────────────────────────────────────────────────────────

// ❌ 重方案：全量解析整个输入
fn heavy_parse(input: proc_macro2::TokenStream) {
    let parsed: syn::DeriveInput = syn::parse2(input).unwrap();
    // syn::DeriveInput 包含了整个类型的所有信息：
    // 泛型参数、where clause、所有属性、所有字段、文档注释... 
    // 即使你只需要字段名列表，也要全部解析。
}

// ✅ 轻方案：只解析需要的信息
fn light_parse(input: proc_macro2::TokenStream) {
    // 如果你只需要知道结构体的字段名，
    // 可以用 proc_macro2 的 TokenTree API 自己遍历，
    // 不必调用 syn 的全量解析。

    let mut field_names = Vec::new();
    let mut depth = 0;
    let mut in_struct = false;

    for tt in input {
        match tt {
            proc_macro2::TokenTree::Ident(id) if id == "struct" && depth == 0 => {
                in_struct = true;
            }
            proc_macro2::TokenTree::Group(g) => {
                if in_struct && g.delimiter() == proc_macro2::Delimiter::Brace {
                    // 进入结构体体
                    depth += 1;
                    // 这里可以进一步解析字段名...
                }
            }
            _ => {}
        }
    }
    // 注意：这个方法很脆弱——它假设了特定的语法模式
    // 权衡：性能提升 vs 代码健壮性
    // 对于简单的 derive，轻量级解析通常"够用"
}

//──────────────────────────────────────────────────────────────
// 策略 2: 提取逻辑到非 proc-macro 的辅助 crate
//──────────────────────────────────────────────────────────────
//
// 这个策略的本质是"减少 proc-macro crate 的编译负担"：
//
// my_macro/  (proc-macro = true)
//   只包含 #[proc_macro_derive(...)] 的入口函数
//   ── 薄如蝉翼
//
// my_macro_impl/ (普通 crate)
//   包含所有的解析逻辑、代码生成逻辑、辅助类型
//   ── 这里可以正常测试、正常使用任何依赖
//
// 好处：
// 1. my_macro_impl/ 作为普通 crate，可以被 cargo 增量编译缓存
// 2. 可以在 my_macro_impl/ 中写 #[test]
// 3. 依赖更新时，只需要重编译 my_macro_impl/
//    而 my_macro/ 的重编译成本极小

//──────────────────────────────────────────────────────────────
// 策略 3: 避免不必要的分配
//──────────────────────────────────────────────────────────────

// ❌ 频繁的 String 转换
fn wasteful_allocation(fields: &[syn::Field]) -> TokenStream {
    let mut code = String::new();
    for field in fields {
        let name = field.ident.as_ref().unwrap().to_string(); // 分配
        let ty = quote::quote!(#field.ty).to_string();        // 分配
        code.push_str(&format!("// field: {} type: {}\n", name, ty)); // 分配
    }
    code.parse().unwrap() // 又分配一次
}

// ✅ 直接用 TokenStream
fn efficient_tokens(fields: &[syn::Field]) -> TokenStream {
    let mut tokens = TokenStream::new();
    for field in fields {
        let name = &field.ident;          // &Ident —— 无分配
        let ty = &field.ty;               // &Type —— 无分配
        tokens.extend(quote::quote! {
            // field: #name type: #ty
        });
        // quote! 生成的代码直接合并到 TokenStream
        // 全程零 String 转换
    }
    tokens
}

//──────────────────────────────────────────────────────────────
// 策略 4: 缓存（慎用！）
//──────────────────────────────────────────────────────────────

// 场景：你的宏需要读取 Cargo.toml 或某个配置文件
// 如果每次调用都读一次文件——非常慢

use std::sync::OnceLock;

static CONFIG: OnceLock<MyConfig> = OnceLock::new();

fn get_config() -> &'static MyConfig {
    CONFIG.get_or_init(|| {
        // 只读一次文件
        MyConfig::from_file("config.toml")
    })
}

// ⚠️ 警告：缓存 Span 是极其危险的！
// static BAD_IDENT: OnceLock<Ident> = OnceLock::new();
// BAD_IDENT.get_or_init(|| Ident::new("foo", Span::call_site()));
// → Span::call_site() 在每次调用时不同！缓存的 Ident 带错误的 Span！
//
// 规则：永远不要缓存 Span 或带 Span 的 token。
//       只能缓存 Span 无关的数据（配置、常量表等）。

//──────────────────────────────────────────────────────────────
// 策略 5: 延迟操作 —— 先检查再解析
//──────────────────────────────────────────────────────────────

// ❌ 先解析整个输入再做检查
fn parse_then_check(input: TokenStream) -> TokenStream {
    let parsed = syn::parse2::<syn::DeriveInput>(input).unwrap();
    // 昂贵的解析已经发生了...
    if !has_required_attr(&parsed.attrs) {
        return syn::Error::new(parsed.ident.span(), "缺少必需属性")
            .into_compile_error();
    }
    // ...
}

// ✅ 先快速扫描检查，只有通过解析才继续
fn check_then_parse(input: TokenStream) -> TokenStream {
    // 快速扫描：只检查前几个 token 中是否有 #[my_attr]
    let has_attr = input.to_string().contains("#[my_attr]");

    if !has_attr {
        return syn::Error::new(Span::call_site(), "缺少必需属性")
            .into_compile_error();
    }

    // 只有通过检查才做昂贵的解析
    let parsed = syn::parse2::<syn::DeriveInput>(input).unwrap();
    // ...
}
```

### 5.3 性能测试与基准

### 5.4 优化案例：从一个慢速宏说起

```rust
//──────────────────────────────────────────────────────────────
// 问题场景：一个 derive 宏，每次调用 ~500ms
//──────────────────────────────────────────────────────────────
//
// 用户报告：一个中等规模的项目（~50 个结构体使用此 derive）

// 分析过程：
// 1. cargo build --timings → 发现 my_derive 占总时间的 80%
// 2. 审查代码 → 发现几个问题：

// 问题 1: 在宏内部读取文件
fn slow_read_file() {
    // ❌ 每次宏调用都读取 Cargo.toml
    let _cargo_toml = std::fs::read_to_string("Cargo.toml").unwrap();
    // 50 个结构体 × 读文件 = 50 次磁盘 I/O（在编译期！）
}

// 修复：用 OnceLock 缓存
fn fast_cache() {
    static CARGO_TOML: OnceLock<String> = OnceLock::new();
    let _cargo_toml = CARGO_TOML.get_or_init(|| {
        std::fs::read_to_string("Cargo.toml").unwrap()
    });
}

// 问题 2: 过多的 String 转换
fn slow_string_ops(fields: &[syn::Field]) {
    for field in fields {
        let _s = format!("{}", quote!(#field)); // String 往返转换
        let _t: TokenStream = _s.parse().unwrap(); // 再次解析
    }
}

// 修复：直接用 TokenStream 操作
fn fast_token_ops(fields: &[syn::Field]) -> TokenStream {
    let mut tokens = TokenStream::new();
    for field in fields {
        tokens.extend(quote!(#field));
    }
    tokens
}

// 问题 3: 不必要的全量解析
fn slow_full_parse(input: TokenStream) {
    // ❌ 即使只需要字段名，也解析了整个结构体（包括泛型、where clause）
    let _: syn::DeriveInput = syn::parse2(input).unwrap();
}

// 修复：如果可能，用轻量解析
// 但如果确实需要完整的类型信息，全量解析是无法避免的。
// 此时应该考虑：你真的需要 derive 宏吗？用 attribute 宏是否可以？
// derive 宏每个类型调用一次，attribute 宏每个模块调用一次。


//──────────────────────────────────────────────────────────────
// 优化结果：
//──────────────────────────────────────────────────────────────
// 修复后，每次调用从 ~500ms 降到 ~5ms
// 50 个结构体的总编译时间从 25s 降到 0.25s
//
// 核心教训：
// 1. 不要在宏内做 I/O（除非有缓存）
// 2. 避免 TokenStream → String → TokenStream 的往返转换
// 3. 审慎使用 syn 的全量解析
// 4. 将复杂逻辑移到辅助 crate（利用增量编译）
```

## 6. Crate 隔离约束

### 6.1 proc-macro crate 的沙箱规则

proc-macro crate 有着严格的编译器层面的约束，这些约束是 Rust 的安全性决定的。

```toml
# Cargo.toml
[lib]
proc-macro = true
# 一旦设置了这个，编译器就会施加以下限制：

# 限制 1: 只能导出过程宏（不能导出 pub fn）
# proc-macro crate 不能有常规的公开函数——pub fn 对外部不可见。

# 限制 2: 只能依赖其他 proc-macro crate 或普通 crate
# 不能有一个 proc-macro crate 依赖另一个 proc-macro crate（极少例外）

# 限制 3: proc-macro crate 不能作为普通依赖
# 其他 crate 不能 [dependencies] my_proc_macro = "..."
# 只有 proc-macro 的特殊属性可以被调用
```

```rust
//──────────────────────────────────────────────────────────────
// 限制 1 的具体表现：
//──────────────────────────────────────────────────────────────

// ❌ 不能被外部调用
pub fn helper_function() {
    // 即使在 proc-macro crate 中定义，外部也无法 use
}

// ✅ 只能暴露以下四种函数：
#[proc_macro]
pub fn my_macro(input: proc_macro::TokenStream) -> proc_macro::TokenStream {
    input
}

#[proc_macro_derive(MyDerive)]
pub fn my_derive(input: proc_macro::TokenStream) -> proc_macro::TokenStream {
    input
}

#[proc_macro_derive(MyDeriveWithHelper, attributes(my_attr))]
pub fn my_derive_attr(input: proc_macro::TokenStream) -> proc_macro::TokenStream {
    input
}

#[proc_macro_attribute]
pub fn my_attribute(
    attr: proc_macro::TokenStream,
    item: proc_macro::TokenStream,
) -> proc_macro::TokenStream {
    item
}

// 没有任何其他方式对外暴露功能。
// 这就是你需要"辅助 crate"的根本原因——
// 把你真正的实现逻辑放在一个普通的 crate 中。
```

```rust
//──────────────────────────────────────────────────────────────

//──────────────────────────────────────────────────────────────
pub struct Diagnostic {
    pub errors: Vec<syn::Error>,
}

impl Diagnostic {
    pub fn new() -> Self {
        Diagnostic { errors: Vec::new() }
    }

    pub fn error(&mut self, span: proc_macro2::Span, msg: &str) {
        self.errors.push(syn::Error::new(span, msg));
    }

    pub fn check(self) -> Result<(), syn::Error> {
        if self.errors.is_empty() {
            return Ok(());
        }
        let mut combined = self.errors.into_iter().reduce(|mut a, b| {
            a.combine(b);
            a
        }).unwrap();
        Err(combined)
    }
}

#[cfg(test)]

    #[test]
    fn test_empty_diagnostic_ok() {

        assert!(diag.check().is_ok());
    }

    #[test]
    fn test_diagnostic_with_errors() {
        let mut diag = Diagnostic::new();
        diag.error(proc_macro2::Span::call_site(), "test error");
        assert!(diag.check().is_err());
    }
}
```

```rust

// 转换全景图：
//──────────────────────────────────────────────────────────────

// 进入 proc-macro crate：
//   proc_macro::TokenStream  →  proc_macro2::TokenStream
//   (来自编译器的输入)            (用于你的逻辑)

let input: proc_macro::TokenStream = /* 编译器传入 */;
let input2: proc_macro2::TokenStream = input.into();

//   proc_macro2::TokenStream  →  proc_macro::TokenStream

let output2: proc_macro2::TokenStream = /* 你的逻辑产出 */;
let output: proc_macro::TokenStream = output2.into();
// 同上，需要 "proc-macro" feature

// Span 的转换同理：
//   proc_macro::Span  →  proc_macro2::Span
//   proc_macro2::Span  →  proc_macro::Span
```

## 9. 练习答案

### 练习 1 答案：改善错误信息质量

```rust
use proc_macro::TokenStream;
use proc_macro2::Span;
use quote::quote;
use syn::{Data, DataStruct, DeriveInput, Error, Fields};

#[proc_macro_derive(CheckedStruct, attributes(required))]
pub fn checked_struct(input: TokenStream) -> TokenStream {
    let input: proc_macro2::TokenStream = input.into();
    let parsed: DeriveInput = match syn::parse2(input) {
        Ok(p) => p,
        Err(e) => return e.into_compile_error().into(),
    };

    // 收集所有错误，而不是遇到第一个就返回
    let mut errors: Vec<Error> = Vec::new();

    // 检查结构体级别
    let has_required = parsed.attrs.iter().any(|attr| {
        attr.path().is_ident("required")
    });

    if !has_required {
        // ✅ 改进 1: 用结构体名的 Span，精确指位置
        let mut err = Error::new(
            parsed.ident.span(),
            format!(
                "结构体 `{}` 缺少必需的属性",
                parsed.ident,
            ),
        );

        // ✅ 改进 2: 添加 note 说明缺少的属性
        err.combine(Error::new(
            parsed.ident.span(),  // note 的 Span 可以和 error 一样
            format!(
                "note: `{}` 需要标注 #[required] 属性",
                parsed.ident,
            ),
        ));

        // ✅ 改进 3: 添加 help 展示正确语法
        err.combine(Error::new(
            parsed.ident.span(),
            format!(
                "help: 请添加 #[required] 属性，例如:\n\n\
                 #[required]\n\
                 struct {} {{ ... }}",
                parsed.ident,
            ),
        ));

        errors.push(err);
    }

    // ✅ 改进 4: 检查字段级别的诊断
    // 如果结构体有 named fields，检查每个字段是否有 required 属性
    if let Data::Struct(DataStruct {
        fields: Fields::Named(ref fields),
        ..
    }) = parsed.data
    {
        for field in &fields.named {
            let field_has_required = field.attrs.iter().any(|attr| {
                attr.path().is_ident("required")
            });

            if !field_has_required {
                let field_name = field.ident.as_ref().unwrap();
                let mut err = Error::new(
                    field_name.span(),
                    format!(
                        "字段 `{}` 缺少 #[required] 属性",
                        field_name,
                    ),
                );

                // 提供帮助信息
                err.combine(Error::new(
                    field_name.span(),
                    format!(
                        "help: 添加 #[required] 属性到字段上:\n\
                         #[required]\n\
                         {}: /* type */",
                        field_name,
                    ),
                ));

                errors.push(err);
            }
        }
    }

    // 合并所有错误，一次性报告
    if !errors.is_empty() {
        let combined = errors
            .into_iter()
            .reduce(|mut acc, e| {
                acc.combine(e);
                acc
            })
            .unwrap();
        return combined.into_compile_error().into();
    }

    // 检查通过，生成代码
    let name = &parsed.ident;
    let output = quote! {
        impl #name {
            fn _checked_struct_verified() {}
        }
    };
    output.into()
}

// 诊断输出效果对比：
//
// BEFORE（改进前）:
//   error: missing attribute
//     --> src/main.rs:1:1
//
// AFTER（改进后）:
//   error: 结构体 `MyStruct` 缺少必需的属性
//     --> src/main.rs:5:8
//      |
//    5 | struct MyStruct {
//      |        ^^^^^^^^
//   note: `MyStruct` 需要标注 #[required] 属性
//     --> src/main.rs:5:8
//   help: 请添加 #[required] 属性，例如:
//
//         #[required]
//         struct MyStruct { ... }
//
//   error: 字段 `name` 缺少 #[required] 属性
//     --> src/main.rs:6:9
//      |
//    6 |     pub name: String,
//      |         ^^^^
//   help: 添加 #[required] 属性到字段上:
//         #[required]
//         name: /* type */
```

### 练习 3 答案：实现 `#[sorted]` derive 宏

```rust
//──────────────────────────────────────────────────────────────
// 文件: my_macro/src/lib.rs（proc-macro crate）
//──────────────────────────────────────────────────────────────
use proc_macro::TokenStream;
use syn::{DeriveInput, Error};

#[proc_macro_derive(Sorted)]
pub fn sorted_derive(input: TokenStream) -> TokenStream {
    let input: proc_macro2::TokenStream = input.into();
    let parsed: DeriveInput = match syn::parse2(input) {
        Ok(p) => p,
        Err(e) => return e.into_compile_error().into(),
    };

    // 检查是否是 enum
    let variants = match &parsed.data {
        syn::Data::Enum(data_enum) => &data_enum.variants,
        syn::Data::Struct(data_struct) => {
            return Error::new(
                parsed.ident.span(),
                format!(
                    "`#[derive(Sorted)]` 只能用于枚举 (enum)，不能用于结构体 (struct)。\n\
                     note: 结构体 `{}` 不是枚举类型",
                    parsed.ident,
                ),
            )
            .into_compile_error()
            .into();
        }
        syn::Data::Union(data_union) => {
            return Error::new(
                parsed.ident.span(),
                format!(
                    "`#[derive(Sorted)]` 只能用于枚举 (enum)，不能用于联合体 (union)。\n\
                     note: 联合体 `{}` 不是枚举类型",
                    parsed.ident,
                ),
            )
            .into_compile_error()
            .into();
        }
    };

    // 空 enum 和单 variant enum 总是"有序"的
    if variants.len() <= 1 {
        return generate_pass_through(&parsed).into();
    }

    // 检查字母顺序
    for window in variants.windows(2) {
        let prev = window[0].ident.to_string();
        let curr = window[1].ident.to_string();

        if prev > curr {
            // 发现无序：curr 应该排在 prev 前面
            let curr_ident = &window[1].ident;

            // 构建完整诊断
            let mut err = Error::new(
                variants.first().unwrap().ident.span(),
                // 指向第一个 variant 的 Span（让用户能看到整个 enum）
                "variants are not in alphabetical order",
            );

            err.combine(Error::new(
                curr_ident.span(),
                format!(r#""{}" should come before "{}""#, curr, prev),
            ));

            err.combine(Error::new(
                curr_ident.span(),
                format!(r#"help: move the "{}" variant above "{}""#, curr, prev),
            ));

            return err.into_compile_error().into();
        }
    }

    // 检查通过——生成一个空实现（避免"未生成任何代码"的警告）
    generate_pass_through(&parsed).into()
}

fn generate_pass_through(input: &DeriveInput) -> proc_macro2::TokenStream {
    let name = &input.ident;
    let (impl_generics, ty_generics, where_clause) = input.generics.split_for_impl();

    quote::quote! {
        // 自动生成：验证 enum variant 已按字母排序
        // 这个 impl 块仅用于编译期检查，没有运行时开销
        impl #impl_generics #name #ty_generics #where_clause {
            #[doc(hidden)]
            fn __sorted_derive_verified() {
                // 编译期验证通过
                // 如果枚举 variant 没有按字母顺序排列，
                // 上面的检查会发出编译错误而不是到这里。
                #[allow(clippy::no_effect)]
                const _: () = ();
            }
        }
    }
}

//──────────────────────────────────────────────────────────────
// Cargo.toml 配置
//──────────────────────────────────────────────────────────────
/*
[package]
name = "sorted-derive"
version = "0.1.0"
edition = "2021"

[lib]
proc-macro = true

[dependencies]
syn = { version = "2.0", features = ["full"] }
quote = "1.0"

trybuild = "1.0"
*/
```

## 10. 深度扩展

### 9.1 Span hygiene 的底层原理

Hygiene 是 Rust 宏系统中防止"名字意外冲突"的机制。它的核心思想是：宏生成的标识符不应该意外地与调用方代码中的标识符冲突（反之亦然）。

```rust
// 老式 C 预处理器宏的经典 bug（Rust 中不会出现）：
// #define SWAP(a, b) { int tmp = a; a = b; b = tmp; }
//
// int tmp = 42;
// SWAP(x, tmp);  // 展开后：int tmp = x; x = tmp; tmp = tmp;
//                // ↑ 新声明的 tmp 遮蔽了外面的 tmp——行为错误！
//
// Rust 的 hygiene 确保宏内部的 tmp 和外面的 tmp 是不同的变量。

// 在 proc_macro 中，hygiene 通过 Span 实现：
// - 每个 token 的 Span 不仅包含位置信息，
//   还包含一个 SyntaxContext（语法上下文）
// - 宏内部的 token 的 SyntaxContext 不同于调用方的 SyntaxContext
// - 相同名字但不同 SyntaxContext → 编译器视为不同的标识符

// 你可以强行"破坏"hygiene（通过 Span::mixed_site/def_site）
// 但通常保持 hygiene 是正确的默认行为。
```

### 9.2 与声明宏（macro_rules!）的对比

| 特性 | macro_rules! | proc_macro |
|:---|:---|:---|
| **Hygiene** | 自动 + 半 hygienic | 需手动控制 Span |
| **错误诊断** | 编译器自动 | 需手动实现 `syn::Error` |
| **表达能力** | 模式匹配 + 递归 | 完整的 Rust 图灵完备 |
| **编译速度** | 极快（声明式） | 较慢（运行 Rust 代码） |
| **可测试性** | 有限（通过尝试编译） | 完整（trybuild + 单元测试） |
| **IDE 支持** | 好（rust-analyzer 可展开） | 有限（需执行编译期代码） |
| **适用场景** | 简单模板、语法糖 | 复杂代码生成、trait 实现、DSL |

### 9.3 proc_macro_error —— 社区诊断增强方案

除了手写 `syn::Error`，社区提供了一个更强大的诊断框架：

```rust
// - emit_error!() —— 发出错误但继续执行
// - emit_warning!() —— 发出警告

#[proc_macro_error::proc_macro_error]
#[proc_macro_derive(MyDerive)]

    let input: proc_macro2::TokenStream = input.into();

// proc_macro_error 的主要优势：
// 1. 简化的 API —— 不需要手动 combine Error
// 2. 自动处理 Span 的层级关系
```

### 9.4 实战建议 —— 过程宏开发的清单

**开发一个生产级过程宏的完整检查清单：**

- [ ] 1. 三层架构：proc-macro crate + 逻辑 crate + 测试 crate
- [ ] 2. 所有错误诊断使用 syn::Error（永不 panic!）
- [ ] 3. 收集所有错误再一次性返回（不要遇到第一个错误就停止）
- [ ] 4. 每个错误都包含：精确 Span + 清晰的错误描述 + help 信息
- [ ] 5. trybuild 测试覆盖每个错误分支 + 边界情况
- [ ] 6. 使用 cargo build --timings 检查编译性能
- [ ] 7. 避免在宏中做 I/O（或使用 OnceLock 缓存）
- [ ] 8. 避免 TokenStream → String → TokenStream 往返
- [ ] 9. 为所有生成的代码使用合适的 Span（quote_spanned!）
- [ ] 10. 文档中展示完整的错误信息示例（用户会参考）
- [ ] 11. 在 README 中列出所有可能的错误信息及其解决方案
- [ ] 12. CI 中运行 cargo test（包括 trybuild）+ cargo clippy
