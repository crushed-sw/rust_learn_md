# 5.3 Attribute 与 Function-like 宏 —— 修改与创造的两种姿态

> "Derive 宏回答了'为一个类型自动生成 trait 实现'的问题。但元编程的疆域远不止于此——Attribute 宏让你拦截并重写任意语法项，Function-like 宏让你创造全新的语法。两者底层是同一套管线——`TokenStream` 进，`TokenStream` 出。区别只在于：Attribute 宏接受被标注的 item 作为输入，而 Function-like 宏像声明宏一样接受自由格式的 token 流。"

---

## 一、设计哲学：三种宏的职责边界

在深入技术细节之前，先建立心智模型。Rust 过程宏有三种形态，它们各自承担不同的元编程职责：

| 宏种类 | 标注语法 | 输入 | 输出 | 典型场景 |
|--------|----------|------|------|----------|
| **Derive 宏** | `#[derive(Trait)]` | `TokenStream`（类型的定义） | `TokenStream`（通常是 `impl Trait for Type`） | 自动生成 trait 实现 |
| **Attribute 宏** | `#[my_attr] fn/struct/... { }` | `(TokenStream, TokenStream)`（属性参数 + 被标注项） | `TokenStream`（替换整项） | 包装函数、修改结构体、编译期注册 |
| **Function-like 宏** | `my_macro!(...)` | `TokenStream`（自由格式的 token 流） | `TokenStream`（任意代码） | DSL、自定义语法 |

核心哲学：

- **Derive** 是"增强"——它给已有类型添加新的能力（trait 实现），不改变类型本身。
- **Attribute** 是"拦截"——它截获一个语法项，可以修改、替换、甚至删除它。
- **Function-like** 是"创造"——它接受完全自定义的语法，生成任意 Rust 代码。

三者的关系不是相互替代，而是互为补充。一个成熟的 proc macro crate 常常同时使用三种形式：Function-like 宏提供 DSL 入口，Attribute 宏进行编译期注册，Derive 宏为类型生成样板代码。

## 二、深度原理

### 2.1 `#[proc_macro_attribute]` 精讲

#### 函数签名

```rust
#[proc_macro_attribute]
pub fn my_attribute(attr: TokenStream, item: TokenStream) -> TokenStream {
    // attr: 属性自身的参数，如 #[my_attribute(args...)]
    // item: 被标注的语法项，如一个 fn、struct、enum 等
    // 返回值：替换掉原始的 item
}
```

两个参数的含义必须理解透彻：

- **`attr: TokenStream`** — 属性括号中的内容。例如 `#[route(GET, "/users")]` 中，`attr` 就是 `GET, "/users"` 这个 token 流。如果属性不带参数，如 `#[log_call]`，则 `attr` 是空的 `TokenStream`。
- **`item: TokenStream`** — 被标注的整个语法项。包括它的所有属性、可见性修饰符、关键字、签名、以及函数体/字段定义等全部内容。
- **返回值** — 返回的 `TokenStream` 将**完全替换**原始的 `item`。这意味着你可以：
  - 修改原始 item 后返回它
  - 返回原始 item 加上额外的代码
  - 完全替换为新的 item
  - 返回多个 item（原始 item 的修改版本 + 新的辅助 item）

#### 最简单的示例：`#[log_call]`

我们从一个可以立刻运行的例子开始。这个 attribute 宏会在函数调用前后打印日志：

```rust
// 在 proc-macro crate: my_macros/src/lib.rs
use proc_macro::TokenStream;
use quote::quote;

#[proc_macro_attribute]
pub fn log_call(_attr: TokenStream, item: TokenStream) -> TokenStream {
    // 1. 将 item 解析为一个函数
    let input_fn: syn::ItemFn = syn::parse_macro_input!(item);

    let fn_name = &input_fn.sig.ident;
    let fn_block = &input_fn.block;
    let fn_vis = &input_fn.vis;
    let fn_sig = &input_fn.sig;
    let fn_attrs = &input_fn.attrs;
    let fn_generics = &input_fn.sig.generics;

    // 2. 获取参数名列表用于打印
    let param_names: Vec<_> = fn_sig
        .inputs
        .iter()
        .filter_map(|arg| {
            if let syn::FnArg::Typed(pat_type) = arg {
                if let syn::Pat::Ident(pat_ident) = &*pat_type.pat {
                    return Some(pat_ident.ident.to_string());
                }
            }
            None
        })
        .collect();
    let param_debug_str = param_names.join(", ");

    let is_async = fn_sig.asyncness.is_some();

    // 3. 生成包装后的函数
    let expanded = if is_async {
        quote! {
            #(#fn_attrs)*
            #fn_vis #fn_sig {
                println!("Entering {} with args: ({})", stringify!(#fn_name), #param_debug_str);
                let result = { #fn_block };
                println!("Exiting {} with result: {:?}", stringify!(#fn_name), result);
                result
            }
        }
    } else {
        // 检查返回类型是否为单元类型
        let output = &fn_sig.output;
        quote! {
            #(#fn_attrs)*
            #fn_vis #fn_sig {
                println!("Entering {} with args: ({})", stringify!(#fn_name), #param_debug_str);
                let __result = (|| #output { #fn_block })();
                println!("Exiting {} with result: {:?}", stringify!(#fn_name), __result);
                __result
            }
        }
    };

    expanded.into()
}
```

```rust
// 在使用方 crate:
use my_macros::log_call;

/// 这是一个测试函数
#[log_call]
fn add(a: i32, b: i32) -> i32 {
    a + b
}

// 调用 add(3, 4) 时会打印：
// Entering add with args: (a, b)
// Exiting add with result: 7
```

注意几个关键点：
1. `#(#fn_attrs)*` — 这保留了原始函数的属性（如 `#[inline]`、doc comments 等）。**忘记保留 attrs 是 attribute 宏最常见的 bug**。
2. 使用 `stringify!(#fn_name)` 在展开时获取函数名的字符串字面量。
3. 整个 item 被原地替换——原始函数不再存在，只有包装后的函数。

#### 可以添加额外的 item

Attribute 宏的一个强大特性是可以在修改后的 item 之后追加新的 item：

```rust
#[proc_macro_attribute]
pub fn register_handler(attr: TokenStream, item: TokenStream) -> TokenStream {
    let input_fn = syn::parse_macro_input!(item as syn::ItemFn);
    let fn_name = &input_fn.sig.ident;
    let fn_name_str = fn_name.to_string();

    // 生成注册入口的静态项
    let registration_name = syn::Ident::new(
        &format!("__REGISTRATION_{}", fn_name_str.to_uppercase()),
        fn_name.span(),
    );

    let expanded = quote! {
        #input_fn

        #[doc(hidden)]
        #[allow(non_upper_case_globals)]
        static #registration_name: RegistrationEntry = RegistrationEntry {
            name: #fn_name_str,
            func: #fn_name,
        };
    };

    expanded.into()
}
```

这样，标注为 `#[register_handler]` 的函数会自动生成对应的全局注册项。

### 2.2 Attribute 宏的常见模式

#### 模式一：Wrapper（包装）

最经典的模式——在函数调用前后插入代码。

**使用场景**：日志、性能测量、权限检查、事务管理。

```rust
// 简化版 #[instrument] — 类似 tracing crate
#[proc_macro_attribute]
pub fn instrument(attr: TokenStream, item: TokenStream) -> TokenStream {
    let fn_item = syn::parse_macro_input!(item as syn::ItemFn);
    let fn_name = fn_item.sig.ident.to_string();
    let fn_block = &fn_item.block;

    // 解析属性参数，如 #[instrument(level = "debug")]
    let level = if !attr.is_empty() {
        let args: syn::LitStr = syn::parse_macro_input!(attr);
        args.value()
    } else {
        "info".to_string()
    };

    let level_ident = syn::Ident::new(&level, proc_macro2::Span::call_site());

    let expanded = quote! {
        fn #fn_name() {
            log::#level_ident!("[{}] Entering", #fn_name);
            let __instant = std::time::Instant::now();
            let __result = (|| { #fn_block })();
            log::#level_ident!(
                "[{}] Exiting (elapsed: {:?})",
                #fn_name,
                __instant.elapsed()
            );
            __result
        }
    };

    expanded.into()
}
```

#### 模式二：Registration（编译期注册）

通过 `inventory` crate 或 linkme 类似的机制，attribute 宏可以在编译期收集信息。

**使用场景**：Web 框架的路由注册、插件的自动发现、测试用例的自动收集。

```rust
// 一个简化版的路由注册宏
#[proc_macro_attribute]
pub fn route(attr: TokenStream, item: TokenStream) -> TokenStream {
    let fn_item = syn::parse_macro_input!(item as syn::ItemFn);
    let fn_name = &fn_item.sig.ident;

    // 解析 attr: 期望格式为 METHOD, "/path"
    let attr_str = attr.to_string();
    let parts: Vec<&str> = attr_str.split(',').map(|s| s.trim()).collect();
    let method = parts[0];
    let path = parts[1];

    let registration_name = syn::Ident::new(
        &format!("__ROUTE_{}", fn_name.to_string().to_uppercase()),
        fn_name.span(),
    );

    // 将原始函数标记为 async（如果需要）
    let expanded = quote! {
        #fn_item

        #[doc(hidden)]
        #[link_section = ".routes"]
        #[no_mangle]
        #[used]
        pub static #registration_name: RouteEntry = RouteEntry {
            method: #method,
            path: #path,
            handler: #fn_name,
        };
    };

    expanded.into()
}
```

#### 模式三：Modification（修改）

直接修改被标注项的内部结构。

**使用场景**：为结构体自动添加字段、修改函数签名、注入依赖。

```rust
// 自动为 struct 添加 created_at 字段
#[proc_macro_attribute]
pub fn with_timestamp(_attr: TokenStream, item: TokenStream) -> TokenStream {
    let mut input_struct = syn::parse_macro_input!(item as syn::ItemStruct);

    // 添加一个 created_at 字段
    let timestamp_field: syn::Field = syn::parse_quote! {
        pub created_at: std::time::Instant
    };

    match &mut input_struct.fields {
        syn::Fields::Named(named) => {
            named.named.push(timestamp_field);
        }
        syn::Fields::Unnamed(_) => {
            // 对于元组结构体，通常报错或做特殊处理
            return syn::Error::new(
                proc_macro2::Span::call_site(),
                "with_timestamp requires a named-field struct"
            )
            .to_compile_error()
            .into();
        }
        syn::Fields::Unit => {
            // 将单元结构体转换为命名字段结构体
            input_struct.fields = syn::Fields::Named(
                syn::parse_quote! { { pub created_at: std::time::Instant } }
            );
        }
    }

    quote! { #input_struct }.into()
}
```

#### 模式四：Validation（编译期验证）

在编译期检查不变量，如果不符合要求则产生编译错误。

**使用场景**：状态机完整性检查、SQL 查询的语法验证、API 端点的契约检查。

```rust
#[proc_macro_attribute]
pub fn validate_params(attr: TokenStream, item: TokenStream) -> TokenStream {
    let fn_item = syn::parse_macro_input!(item as syn::ItemFn);

    // 从 attr 中获取期望的参数类型列表
    let expected_types: Vec<String> = attr.to_string()
        .split(',')
        .map(|s| s.trim().to_string())
        .collect();

    let actual_params: Vec<_> = fn_item.sig.inputs.iter().collect();

    if expected_types.len() != actual_params.len() {
        return syn::Error::new(
            proc_macro2::Span::call_site(),
            format!(
                "Function expects {} parameters, but {} types were specified in #[validate_params]",
                actual_params.len(),
                expected_types.len()
            )
        )
        .to_compile_error()
        .into();
    }

    // 可以进一步检查类型匹配...

    quote! { #fn_item }.into()
}
```

### 2.3 `#[proc_macro]` 精讲（Function-like 宏）

#### 函数签名与本质

```rust
#[proc_macro]
pub fn my_macro(input: TokenStream) -> TokenStream {
    // input: 调用时 !(...) 中的全部内容
    // 返回值：替换调用的代码
}
```

Function-like 宏是最自由的过程宏。它接受括号内的任意 token 流——没有语法限制，没有结构要求。它的本质与 `macro_rules!` 相同（都接受 token 流，都产生代码），但拥有过程式编程的全部能力：

- 可以做复杂的类型检查
- 可以访问文件系统（虽然不推荐）
- 可以进行任意计算
- 可以产生带有 Span 信息的诊断

#### 核心 API：`syn::parse_macro_input!` 与自定义 Parser

```rust
#[proc_macro]
pub fn my_macro(input: TokenStream) -> TokenStream {
    // 与 Derive 宏相同的解析模式
    let parsed_input = syn::parse_macro_input!(input as MySyntax);
    // ...
}
```

不同之处在于，你需要自己定义 `MySyntax` 的 `Parse` trait 实现——因为这个语法是由你创造的，`syn` 无法为它提供内置的解析器。

#### 示例 1：`sql!` 宏（简化版）

```rust
use proc_macro::TokenStream;
use syn::{parse::{Parse, ParseStream}, *};

// 第一步：定义我们自己的语法结构
mod sql_syntax {
    use super::*;

    pub struct SqlSelect {
        pub columns: Vec<String>,
        pub table: String,
        pub where_clause: Option<WhereClause>,
    }

    pub struct WhereClause {
        pub column: String,
        pub operator: String,
        pub value: String,
    }

    impl Parse for SqlSelect {
        fn parse(input: ParseStream) -> syn::Result<Self> {
            // 期望: SELECT col1, col2 FROM table WHERE col = value
            // 简化版，不支持 FULL SQL

            // 跳过 SELECT
            let select_kw: Ident = input.parse()?;
            if select_kw.to_string().to_uppercase() != "SELECT" {
                return Err(input.error("expected SELECT"));
            }

            // 解析列名
            let mut columns = Vec::new();
            loop {
                let col: Ident = input.parse()?;
                columns.push(col.to_string());

                if input.peek(Token![,]) {
                    input.parse::<Token![,]>()?;
                } else {
                    break;
                }
            }

            // 跳过 FROM
            let from_kw: Ident = input.parse()?;
            if from_kw.to_string().to_uppercase() != "FROM" {
                return Err(input.error("expected FROM"));
            }

            // 解析表名
            let table: Ident = input.parse()?;
            let table = table.to_string();

            // 可选的 WHERE 子句
            let where_clause = if input.peek(Token![WHERE]) || {
                // peek by string
                let fork = input.fork();
                let kw: Result<Ident, _> = fork.parse();
                kw.map(|i| i.to_string().to_uppercase() == "WHERE")
                    .unwrap_or(false)
            } {
                let _where_kw: Ident = input.parse()?; // WHERE
                let column: Ident = input.parse()?;
                let op: Token![=] = input.parse()?;
                let value: LitStr = input.parse()?;

                Some(WhereClause {
                    column: column.to_string(),
                    operator: "=".to_string(),
                    value: value.value(),
                })
            } else {
                None
            };

            Ok(SqlSelect { columns, table, where_clause })
        }
    }
}

#[proc_macro]
pub fn sql(input: TokenStream) -> TokenStream {
    let query = syn::parse_macro_input!(input as sql_syntax::SqlSelect);

    // 第三步：生成 Rust 代码
    let table = syn::Ident::new(
        &format!("{}", query.table),
        proc_macro2::Span::call_site()
    );

    let columns = query.columns;
    let column_idents: Vec<syn::Ident> = columns
        .iter()
        .map(|c| syn::Ident::new(c, proc_macro2::Span::call_site()))
        .collect();

    // 生成一个查询构建器
    let expanded = quote::quote! {
        {
            use crate::db::QueryBuilder;
            QueryBuilder::new(stringify!(#table))
                #(.select(stringify!(#column_idents)))*
                .build()
        }
    };

    expanded.into()
}
```

#### 示例 2：`html!` 宏（JSX 风格）

```rust
mod html_syntax {
    use super::*;

    pub struct HtmlElement {
        pub tag: String,
        pub attributes: Vec<(String, String)>,
        pub children: Vec<HtmlNode>,
    }

    pub enum HtmlNode {
        Element(HtmlElement),
        Text(String),
        Expression(proc_macro2::TokenStream),
    }

    impl Parse for HtmlElement {
        fn parse(input: ParseStream) -> syn::Result<Self> {
            // < 开始
            input.parse::<Token![<]>()?;

            // 标签名
            let tag: Ident = input.parse()?;
            let tag = tag.to_string();

            // 属性（简化：不支持属性值中的表达式）
            let mut attributes = Vec::new();
            while !input.peek(Token![>]) && !input.peek(Token![/]) {
                let attr_name: Ident = input.parse()?;
                input.parse::<Token![=]>()?;
                let attr_value: LitStr = input.parse()?;
                attributes.push((attr_name.to_string(), attr_value.value()));
            }

            // 自闭合标签 <br />
            let self_closing = input.peek(Token![/]);
            if self_closing {
                input.parse::<Token![/]>()?;
            }
            input.parse::<Token![>]>()?;

            let mut children = Vec::new();
            if !self_closing {
                // 解析子节点直到闭合标签
                loop {
                    if input.peek(Token![<]) && input.peek2(Token![/]) {
                        // </tag> — 闭合标签
                        input.parse::<Token![<]>()?;
                        input.parse::<Token![/]>()?;
                        let closing_tag: Ident = input.parse()?;
                        input.parse::<Token![>]>()?;
                        if closing_tag.to_string() != tag {
                            return Err(input.error(format!(
                                "mismatched closing tag: expected </{}>, found </{}>",
                                tag, closing_tag
                            )));
                        }
                        break;
                    }

                    if input.peek(Token![<]) {
                        // 嵌套元素
                        children.push(HtmlNode::Element(input.parse()?));
                    } else if input.peek(Token![{]) {
                        // 表达式插值 { expr }
                        input.parse::<Token![{]>()?;
                        let expr: syn::Expr = input.parse()?;
                        input.parse::<Token![}]>()?;
                        children.push(HtmlNode::Expression(quote::quote! { #expr }));
                    } else {
                        // 文本节点
                        let text: LitStr = input.parse()?;
                        children.push(HtmlNode::Text(text.value()));
                    }
                }
            }

            Ok(HtmlElement { tag, attributes, children })
        }
    }

    impl Parse for HtmlNode {
        fn parse(input: ParseStream) -> syn::Result<Self> {
            if input.peek(Token![<]) {
                Ok(HtmlNode::Element(input.parse()?))
            } else if input.peek(Token![{]) {
                input.parse::<Token![{]>()?;
                let expr: syn::Expr = input.parse()?;
                input.parse::<Token![}]>()?;
                Ok(HtmlNode::Expression(quote::quote! { #expr }))
            } else {
                let text: LitStr = input.parse()?;
                Ok(HtmlNode::Text(text.value()))
            }
        }
    }
}

#[proc_macro]
pub fn html(input: TokenStream) -> TokenStream {
    let element = syn::parse_macro_input!(input as html_syntax::HtmlElement);
    generate_html(element).into()
}

fn generate_html(element: html_syntax::HtmlElement) -> proc_macro2::TokenStream {
    let tag = &element.tag;
    let attrs = element.attributes.iter().map(|(name, value)| {
        quote::quote! {
            builder.attr(#name, #value);
        }
    });
    let children = element.children.iter().map(|child| match child {
        html_syntax::HtmlNode::Text(text) => {
            quote::quote! { builder.text(#text); }
        }
        html_syntax::HtmlNode::Element(el) => {
            let nested = generate_html(el.clone());
            quote::quote! { builder.child(|| { #nested }); }
        }
        html_syntax::HtmlNode::Expression(expr) => {
            quote::quote! { builder.child(#expr); }
        }
    });

    quote::quote! {
        {
            let mut builder = ::dom::Builder::new(#tag);
            #(#attrs)*
            #(#children)*
            builder.finish()
        }
    }
}
```

#### Function-like 宏不像声明宏那样能展开

一个关键区别：Function-like 宏的内部内容**不能先被声明宏展开再传入**。例如：

```rust
macro_rules! my_vec {
    ($($x:expr),*) => { vec![$($x),*] };
}
// my_vec! 展开后，vec! 又可以被展开——声明宏是逐层展开的。

// 但 Function-like 宏：
// sql!(SELECT * FROM users WHERE name = my_stringify!(hello));
// my_stringify! 不会被先展开再传给 sql!，因为 Function-like 宏
// 接受的是原始 token 流。
```

### 2.4 Attribute vs Function-like vs Derive 决策指南

#### 决策流程图

当你在设计一个宏时，按以下顺序问自己：

1. **是否需要为一个类型生成 trait 实现？**
   - 是 → 使用 **Derive 宏**
   - 部分需要修改类型本身 → 继续往下

2. **是否需要修改/包装/验证一个已存在的 Rust 语法项（fn、struct、enum 等）？**
   - 是 → 使用 **Attribute 宏**
   - 不是 → 继续往下

3. **是否需要接受自由格式的输入，创造新的语法？**
   - 是 → 使用 **Function-like 宏**
   - 不是 → 用普通函数或 trait 即可

#### 详细对比表

| 维度 | Derive 宏 | Attribute 宏 | Function-like 宏 |
|------|-----------|--------------|-------------------|
| **语法位置** | `#[derive(Macro)]` 在类型定义上 | `#[macro]` 在任何 item 上 | `macro!(...)` 在任何表达式/语句位置 |
| **输入限制** | 只能是类型定义 | 有效的 Rust item | 任意 token 流 |
| **能否修改输入** | 否（只能添加 impl） | 是（完全替换 item） | N/A（输入只是 tokens） |
| **能否添加新 item** | 否 | 是 | 是 |
| **实现复杂度** | 低-中（syn 提供自动解析） | 中（需手动处理 item） | 中-高（需自定义 Parser） |
| **常见应用** | Serialize, Deserialize, Debug | 路由注册, 日志注入, AOP | DSL, 模板引擎, 自定义语法 |
| **学习曲线** | 最低 | 中等 | 最高 |
| **错误处理** | 标注在类型上 | 标注在 item 上 | 发生在宏调用处 |

#### 组合使用的力量

三种宏可以协同工作：

```rust
// 1. Function-like 宏定义 DSL
routes! {
    GET "/users" => list_users,
    POST "/users" => create_user,
};

// 2. Attribute 宏标注单个路由处理函数
#[auth(role = "admin")]
fn create_user(form: CreateUserForm) -> Result<UserId> {
    // ...
}

// 3. Derive 宏为表单类型生成验证
#[derive(Validate)]
struct CreateUserForm {
    #[validate(email)]
    email: String,
    #[validate(length(min = 8))]
    password: String,
}
```

三个宏分别承担了：DSL 定义、编译期切面注入、样板代码生成的职责。

### 2.5 实战：`#[route]` 宏完整实现

现在实现一个完整的、接近生产级别的 `#[route]` 宏。

#### 目标

```rust
use my_macros::route;

#[route(GET, "/users")]
fn list_users() -> Vec<User> {
    // ...
}

#[route(GET, "/users/{id}")]
fn get_user(id: u64) -> User {
    // ...
}

#[route(POST, "/users")]
fn create_user(body: Json<CreateUserForm>) -> StatusCode {
    // ...
}
```

#### 需求

1. 解析 HTTP 方法和路径
2. 提取函数名、参数、返回类型
3. 生成路由表的注册代码
4. 保留原始函数，在调用时通过路由表分发

#### 完整实现

```rust
// 在 proc-macro crate 中

use proc_macro::TokenStream;
use quote::{quote, format_ident};
use syn::{parse_macro_input, ItemFn, LitStr, Ident, parse::Parse, parse::ParseStream, Token};

// 解析属性参数: GET, "/path"
struct RouteArgs {
    method: Ident,
    path: LitStr,
}

impl Parse for RouteArgs {
    fn parse(input: ParseStream) -> syn::Result<Self> {
        let method: Ident = input.parse()?;
        input.parse::<Token![,]>()?;
        let path: LitStr = input.parse()?;
        Ok(RouteArgs { method, path })
    }
}

#[proc_macro_attribute]
pub fn route(attr: TokenStream, item: TokenStream) -> TokenStream {
    // 解析属性参数和方法
    let args = parse_macro_input!(attr as RouteArgs);
    let fn_item = parse_macro_input!(item as ItemFn);

    let method = &args.method;       // GET, POST, PUT, DELETE...
    let path_str = &args.path;       // "/users" 或 "/users/{id}"

    let fn_name = &fn_item.sig.ident;
    let fn_name_str = fn_name.to_string();
    let fn_vis = &fn_item.vis;
    let fn_attrs = &fn_item.attrs;
    let fn_sig = &fn_item.sig;
    let fn_block = &fn_item.block;

    // 提取路径参数名: "/users/{id}" -> ["id"]
    let path_params = extract_path_params(&path_str.value());

    // 提取函数参数（跳过 self）
    let handler_params: Vec<_> = fn_sig.inputs.iter()
        .filter_map(|arg| {
            if let syn::FnArg::Typed(pat) = arg {
                if let syn::Pat::Ident(ident) = &*pat.pat {
                    return Some((ident.ident.clone(), pat.ty.clone()));
                }
            }
            None
        })
        .collect();

    // 校验：路径参数必须在函数参数中存在
    for param in &path_params {
        if !handler_params.iter().any(|(name, _)| name.to_string() == *param) {
            return syn::Error::new(
                param.span(),
                format!(
                    "Path parameter `{}` in route '{}' has no matching function parameter",
                    param,
                    path_str.value()
                )
            )
            .to_compile_error()
            .into();
        }
    }

    // 生成路由注册表条目名称
    let register_ident = format_ident!("__ROUTE_REGISTRY_{}", fn_name_str.to_uppercase());

    // 构建路径参数提取代码
    let param_extractions = path_params.iter().map(|param_name| {
        let ident = Ident::new(param_name, proc_macro2::Span::call_site());
        quote! {
            let #ident: #ident = request.path_param(stringify!(#ident))
                .ok_or(Response::bad_request(format!(
                    "Missing path parameter: {}",
                    stringify!(#ident)
                )))?;
        }
    });

    // 构建函数调用
    let arg_idents: Vec<_> = handler_params.iter()
        .map(|(name, _)| name.clone())
        .collect();

    let method_upper = method.to_string().to_uppercase();

    let expanded = quote! {
        // 保留原始函数（可以设为 private）
        #(#fn_attrs)*
        #fn_vis #fn_sig #fn_block

        // 生成路由注册项
        #[doc(hidden)]
        #[linkme::distributed_slice(ROUTES)]
        static #register_ident: RouteEntry = RouteEntry {
            method: #method_upper,
            path: #path_str,
            handler: |request: &Request| -> Response {
                #(#param_extractions)*
                let response = #fn_name(#(#arg_idents),*);
                Response::ok(response)
            },
        };
    };

    TokenStream::from(expanded)
}

fn extract_path_params(path: &str) -> Vec<String> {
    let mut params = Vec::new();
    let mut chars = path.chars().peekable();

    while let Some(ch) = chars.next() {
        if ch == '{' {
            let mut name = String::new();
            while let Some(&next) = chars.peek() {
                if next == '}' {
                    chars.next(); // consume }
                    break;
                }
                name.push(chars.next().unwrap());
            }
            if !name.is_empty() {
                params.push(name);
            }
        }
    }

    params
}
```

使用 `linkme` (或者 `inventory`)，路由在程序启动时自动被收集到全局路由表中。

### 2.6 实战：`sql!` 宏实现（简化版）

#### 目标

```rust
use my_macros::sql;

let query = sql!(SELECT id, name, email FROM users WHERE status = "active");
// 展开为：
let query = Query {
    columns: vec!["id", "name", "email"],
    table: "users",
    condition: Some(Condition {
        column: "status",
        operator: "=",
        value: "active",
    }),
};
```

#### 需求

1. 解析 `SELECT col1, col2, ... FROM table`
2. 解析可选的 `WHERE col = value` 子句
3. 根据列名映射到 Rust 类型（编译时类型检查）
4. 生成查询结构体

#### 语法定义

```
sql_query := "SELECT" columns "FROM" table ("WHERE" condition)?
columns   := identifier ("," identifier)*
table     := identifier
condition := identifier operator literal
operator  := "=" | "!=" | ">" | "<" | ">=" | "<="
```

#### 完整实现

```rust
use proc_macro::TokenStream;
use quote::quote;
use syn::{
    parse::{Parse, ParseStream},
    parse_macro_input, Ident, LitStr, Token,
};

// ---- 语法定义 ----

struct SqlQuery {
    columns: Vec<Ident>,
    table: Ident,
    condition: Option<SqlCondition>,
}

struct SqlCondition {
    column: Ident,
    operator: SqlOperator,
    value: LitStr,
}

enum SqlOperator {
    Eq,
    NotEq,
    Gt,
    Lt,
    Gte,
    Lte,
}

impl Parse for SqlQuery {
    fn parse(input: ParseStream) -> syn::Result<Self> {
        // SELECT
        let kw: Ident = input.parse()?;
        if kw.to_string().to_uppercase() != "SELECT" {
            return Err(syn::Error::new(kw.span(), "expected SELECT"));
        }

        // 列名列表
        let mut columns = Vec::new();
        loop {
            let col: Ident = input.parse()?;
            columns.push(col);

            if input.peek(Token![,]) {
                input.parse::<Token![,]>()?;
            } else {
                break;
            }
        }

        // FROM
        let from: Ident = input.parse()?;
        if from.to_string().to_uppercase() != "FROM" {
            return Err(syn::Error::new(from.span(), "expected FROM"));
        }

        let table: Ident = input.parse()?;

        // 可选的 WHERE
        let condition = if !input.is_empty() {
            let where_kw: Ident = input.parse()?;
            if where_kw.to_string().to_uppercase() != "WHERE" {
                return Err(syn::Error::new(where_kw.span(), "expected WHERE"));
            }

            let column: Ident = input.parse()?;
            let operator = input.parse::<SqlOperator>()?;
            let value: LitStr = input.parse()?;

            Some(SqlCondition { column, operator, value })
        } else {
            None
        };

        Ok(SqlQuery { columns, table, condition })
    }
}

impl Parse for SqlOperator {
    fn parse(input: ParseStream) -> syn::Result<Self> {
        if input.peek(Token![=]) {
            input.parse::<Token![=]>()?;
            Ok(SqlOperator::Eq)
        } else if input.peek(Token![!]) {
            input.parse::<Token![!]>()?;
            input.parse::<Token![=]>()?;
            Ok(SqlOperator::NotEq)
        } else if input.peek(Token![>]) {
            input.parse::<Token![>]>()?;
            if input.peek(Token![=]) {
                input.parse::<Token![=]>()?;
                Ok(SqlOperator::Gte)
            } else {
                Ok(SqlOperator::Gt)
            }
        } else if input.peek(Token![<]) {
            input.parse::<Token![<]>()?;
            if input.peek(Token![=]) {
                input.parse::<Token![=]>()?;
                Ok(SqlOperator::Lte)
            } else {
                Ok(SqlOperator::Lt)
            }
        } else {
            Err(input.error("expected one of: =, !=, >, <, >=, <="))
        }
    }
}

// ---- 类型映射 ----

/// 根据表名和列名返回 Rust 类型（编译期映射）
fn map_sql_type(table: &str, column: &str) -> &'static str {
    // 在生产代码中，这会查找数据库 schema
    // 这里简化为硬编码映射
    match (table, column) {
        (_, "id") => "i64",
        (_, "name") | (_, "email") => "String",
        (_, "age") => "i32",
        (_, "active") | (_, "status") => "String",
        _ => "String", // 默认
    }
}

// ---- 宏主体 ----

#[proc_macro]
pub fn sql(input: TokenStream) -> TokenStream {
    let query = parse_macro_input!(input as SqlQuery);

    let table_name = query.table.to_string();
    let table_name_lit = &query.table;
    let columns: Vec<_> = query.columns.iter().map(|c| c.to_string()).collect();

    // 为每列生成类型安全的字段
    let column_fields = query.columns.iter().map(|col| {
        let col_name = col.to_string();
        let col_ident = Ident::new(&col_name, col.span());
        let rust_type_str = map_sql_type(&table_name, &col_name);
        let rust_type: syn::Type = syn::parse_str(rust_type_str)
            .unwrap_or_else(|_| syn::parse_quote! { String });
        quote! { pub #col_ident: #rust_type }
    });

    let column_strings: Vec<String> = columns.clone();

    // 生成条件结构体字段（如果有 WHERE 子句）
    let condition_struct = if let Some(ref cond) = query.condition {
        let col_name = cond.column.to_string();
        let rust_type_str = map_sql_type(&table_name, &col_name);
        let rust_type: syn::Type = syn::parse_str(rust_type_str)
            .unwrap_or_else(|_| syn::parse_quote! { String });
        let col_ident = &cond.column;
        quote! {
            pub condition: Option<Condition<#rust_type>>,
        }
    } else {
        quote! {}
    };

    // 生成查询构建器
    let build_condition = if let Some(cond) = &query.condition {
        let col_str = cond.column.to_string();
        let value = &cond.value;
        let operator_str = match cond.operator {
            SqlOperator::Eq => "=",
            SqlOperator::NotEq => "!=",
            SqlOperator::Gt => ">",
            SqlOperator::Lt => "<",
            SqlOperator::Gte => ">=",
            SqlOperator::Lte => "<=",
        };
        quote! {
            sql.push_str(&format!(
                " WHERE {} {} {}",
                #col_str, #operator_str, #value
            ));
        }
    } else {
        quote! {}
    };

    // 生成最终的代码
    let col_strs: Vec<&str> = column_strings.iter().map(|s| s.as_str()).collect();

    let expanded = quote! {
        {
            // 为查询结果生成一个新的 struct（在同一作用域内）
            let mut sql = String::new();
            sql.push_str(&format!(
                "SELECT {} FROM {}",
                [#(#col_strs),*].join(", "),
                stringify!(#table_name_lit)
            ));
            #build_condition

            // 返回一个包含 SQL 字符串和类型信息的查询对象
            QueryBuilder::new(sql)
        }
    };

    expanded.into()
}

// 使用方 crate 需要定义:
// struct QueryBuilder { sql: String }
// impl QueryBuilder {
//     fn new(sql: String) -> Self { Self { sql } }
//     fn execute<T>(&self) -> Result<Vec<T>, DbError> { /* ... */ }
// }
```

#### 关于完整 SQL 解析的说明

这个简化版 `sql!` 宏仅支持最简单的 `SELECT ... FROM ... WHERE` 形式。真正的 SQL 解析是一个极其复杂的工程：

- 需要完整的解析器（支持 JOIN、GROUP BY、ORDER BY、子查询、CTE、窗口函数等）
- 需要数据库连接以获取 schema 信息进行类型检查
- 需要 SQL 方言感知（MySQL vs PostgreSQL vs SQLite 语法差异）

这就是为什么在实际项目中，我们使用 `sqlx`（编译时查询检查）、`diesel`（编译时 schema 映射）或 `rusqlite`（运行时 SQL）——它们已经解决了这些问题。这里的 `sql!` 宏的价值在于展示 Function-like 宏的工作原理，而不是作为一个生产级的 SQL 解析库。

### 2.7 🔧 调试本节：Attribute 与 Function-like 宏的特殊调试挑战

过程宏的调试是 Rust 开发中最困难的部分之一。以下是针对 Attribute 和 Function-like 宏的专项调试技巧。

#### 2.7.1 Attribute 宏的双重输入调试

Attribute 宏同时接收 `attr` 和 `item` 两个参数，任何一方的误解都会导致 bug。

**基本调试策略**——打印两个输入：

```rust
#[proc_macro_attribute]
pub fn my_attr(attr: TokenStream, item: TokenStream) -> TokenStream {
    eprintln!("===== ATTR =====");
    eprintln!("{}", attr);
    eprintln!("===== ITEM =====");
    eprintln!("{}", item);

    // ... 宏逻辑 ...

    // 也打印输出
    let output: TokenStream = /* ... */;
    eprintln!("===== OUTPUT =====");
    eprintln!("{}", output);
    output
}
```

`eprintln!` 在 proc macro 中的输出会显示在 `cargo build` 的输出流中（以 `note:` 或 `warning:` 的形式）。

#### 2.7.2 Item 破坏问题——最常见的 Attribute 宏 Bug

Attribute 宏最常见的错误是**在返回的代码中遗漏了原始 item 的部分内容**。

**典型症状**：

```rust
// 原始代码
/// 这是一个被良好注释的函数
#[inline(always)]
#[my_attr]
fn do_something(x: i32) -> i32 {
    x + 1
}

// my_attr 宏展开后，doc comment 和 #[inline(always)] 消失了！
fn do_something(x: i32) -> i32 {
    // ... 包装代码 ...
}
```

**根因**：宏作者在 `quote!` 中只保留了部分 item，忘记保留 `attrs`。

```rust
// 错误写法——丢失了 attrs
let fn_name = &input_fn.sig.ident;
let fn_block = &input_fn.block;
let expanded = quote! {
    fn #fn_name() { /* wrapper */ #fn_block }
};

// 正确写法——保留 attrs
let fn_attrs = &input_fn.attrs;  // ← 关键！
let expanded = quote! {
    #(#fn_attrs)*
    fn #fn_name() { /* wrapper */ #fn_block }
};
```

**验证方法**：使用 `cargo expand` 对比展开前后的代码。

```bash
# 安装 cargo-expand
cargo install cargo-expand

# 查看展开结果
cargo expand

# 只看特定函数
cargo expand -- --test my_test_fn
```

#### 2.7.3 Function-like 宏的 Parser 调试

自定义 Parser 是 Function-like 宏的核心，也是最容易出错的地方。

**策略一：打印解析后的 AST 结构**

```rust
impl Parse for MySyntax {
    fn parse(input: ParseStream) -> syn::Result<Self> {
        eprintln!("Parsing: {}", input);

        // 逐个 token 调试
        let lookahead = input.lookahead1();
        eprintln!("Lookahead peeks: {:?}", lookahead);

        // ...
        let result = /* 解析逻辑 */;
        eprintln!("Parsed: {:#?}", result);
        Ok(result)
    }
}
```

**策略二：使用 `syn::parse2` 进行"尝试性解析"**

```rust
use proc_macro2::TokenStream as TokenStream2;

fn try_parse_variants(input: TokenStream2) -> syn::Result<TokenStream2> {
    // 克隆 token 流以便多次尝试
    let ts = input.clone();
    
    // 尝试解析为变体 A
    if let Ok(parsed_a) = syn::parse2::<VariantA>(ts.clone()) {
        return Ok(generate_a(parsed_a));
    }
    
    // 尝试解析为变体 B
    if let Ok(parsed_b) = syn::parse2::<VariantB>(ts) {
        return Ok(generate_b(parsed_b));
    }
    
    Err(syn::Error::new(
        proc_macro2::Span::call_site(),
        "failed to parse as any known variant"
    ))
}
```

**策略三：识别最常见的解析错误**

| 错误类型 | 原因 | 解决办法 |
|----------|------|----------|
| `unexpected token` | 用户输入与预期不符 | 检查 `input.peek()` 是否正确 |
| `expected identifier` | 期望标识符但收到关键字 | 检查是否漏了解析某个关键字前的步骤 |
| `cannot parse` | 忘了消费某个 token | 加 `eprintln!("after X: remaining = {}", input)` |
| 泛型解析失败 | `<` 被解析为比较运算符 | 使用 `syn::parse2` 配合 `syn::AngleBracketedGenericArguments` |

#### 2.7.4 `compile_error!` 断言式调试

在 proc macro 的输出中嵌入 `compile_error!`——这相当于在宏展开阶段插入断言。

```rust
#[proc_macro_attribute]
pub fn debug_attr(_attr: TokenStream, item: TokenStream) -> TokenStream {
    let fn_item = syn::parse_macro_input!(item as syn::ItemFn);
    let fn_name_str = fn_item.sig.ident.to_string();
    let fn_attrs_count = fn_item.attrs.len();

    // 在展开结果中插入编译错误，携带调试信息
    let debug_info = quote! {
        compile_error!(concat!(
            "DEBUG: function name = ",
            stringify!(#fn_name_str),
            ", attrs count = ",
            stringify!(#fn_attrs_count)
        ));
    };

    let expanded = quote! {
        #debug_info
        // ... 正常代码 ...
    };

    expanded.into()
}
```

当编译器遇到生成的 `compile_error!` 时，会在错误消息中打印你的调试信息。这个方法比 `eprintln!` 更可靠，因为 `compile_error!` 会出现在 IDE 的诊断输出中。

#### 2.7.5 Token Stream 实时查看

```rust
fn inspect_tokens(ts: &proc_macro2::TokenStream) {
    for token in ts.clone().into_iter() {
        eprintln!("Token: {:?} (span: {:?})", token, token.span());
    }
}
```

这个函数逐个打印 token 流中的 token，对于理解复杂的解析失败非常有用。

#### 2.7.6 常见 Span 问题

```rust
// 错误：使用 call_site 导致编译错误指向宏调用处而非具体代码
let bad_span = proc_macro2::Span::call_site();

// 正确：使用原始 token 的 span
let good_span = input_fn.sig.ident.span(); // 指向函数名

// 在错误中使用正确的 span
return syn::Error::new(good_span, "this function must be async").to_compile_error().into();
```

### 练习 2（实现题）

实现一个完整的 `#[log_call]` attribute 宏，满足以下要求：

1. 在函数调用前打印：`"Entering <函数名> with args: (<参数名列表>)"`
2. 在函数返回后打印：`"Exiting <函数名> with result: {:?}"`
3. 正确处理有返回值和无返回值的函数（返回值用 `{:?}` 需要 `Debug` trait）
4. 保留原始函数的 doc comments (`/// ...`)、attributes (`#[inline]` 等)、visibility (`pub` 等)
5. 支持泛型函数（保留泛型参数）
6. 函数体中对 `?` 运算符的使用不破坏控制流（提示：将函数体包装在一个闭包中可能有问题；考虑更健壮的做法）

**提示**：对于要求 6，`(|| { #fn_block })()` 可以工作，但当 `fn_block` 中有 `return` 或 `?` 时需要特别处理。可以接受一个简化方案：只处理没有 `?` 和 `return` 的情况，并在注释中说明限制。

### 练习 3（设计题）

设计并实现一个 `calc!` function-like 宏，它是一个编译期算术表达式求值器。

**需求**：

```rust
// 基本使用
let x: i64 = calc!(1 + 2 * 3);       // → 7  (展开为 7i64)
let y: i64 = calc!((1 + 2) * 3);     // → 9
let z: i64 = calc!(100 / (2 + 3));   // → 20

// 支持运算符：+ - * /，正确处理优先级（乘除优先于加减）
// 支持括号嵌套
// 对非法输入给出有意义的编译错误
```

**解析思路**：

虽然可以手写递归下降解析器，但更优雅的方式是利用 Rust 自身的表达式解析能力。

```rust
// 思路：将 calc!(...) 中的算术表达式转换为匿名 const 的初始化表达式
// 例如 calc!(1 + 2 * 3) → { const RESULT: i64 = 1 + 2 * 3; RESULT }

// 这样 Rust 编译器会替你做运算符优先级解析和常量求值！
```

**额外挑战**：不依赖 Rust 编译器的 `const` 求值，自己实现一个简单的递归下降解析器和求值器。这样你就能在错误消息中给出更精确的诊断（比如分割子表达式的值）。

## 四、答案

### 练习 1 答案

**Bug 诊断**：

`#[log_call]` 宏在生成输出代码时，**没有保留原始函数的 attributes**（包括 doc comments）。

当用户写：
```rust
/// 计算两个数的和
#[log_call]
fn add(a: i32, b: i32) -> i32 { a + b }
```

宏接收到的 `item` 中包含 `/// 计算两个数的和` 这个 attribute。但宏在 `quote!` 中没有引用 `fn_item.attrs`，所以展开后的代码丢失了 doc comment。

**修复方案**：

```rust
#[proc_macro_attribute]
pub fn log_call(_attr: TokenStream, item: TokenStream) -> TokenStream {
    let fn_item = syn::parse_macro_input!(item as syn::ItemFn);
    let fn_name = fn_item.sig.ident.clone();
    let fn_block = fn_item.block.clone();
    let fn_sig = fn_item.sig.clone();
    let fn_vis = fn_item.vis.clone();
    let fn_attrs = fn_item.attrs.clone(); // ← 补充这行

    let output = quote! {
        #(#fn_attrs)*  // ← 补充这行：展开所有原始 attribute（包括 doc comments）
        #fn_vis #fn_sig {
            println!("Calling {}", stringify!(#fn_name));
            let __result = (|| #fn_block)();
            println!("{} returned", stringify!(#fn_name));
            __result
        }
    };

    output.into()
}
```

`#(#fn_attrs)*` 是 `quote!` 的重复插值语法——它遍历 `fn_attrs` 中的每个元素并展开。对于 doc comments，`syn` 会将 `/// ...` 解析为一个 `Attribute`，包含在 `attrs` 向量中。

**验证**：使用 `cargo expand` 查看展开结果，应该能看到 doc comments 被保留了。

### 练习 2 答案

```rust
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, ItemFn, ReturnType, FnArg, Pat};

#[proc_macro_attribute]
pub fn log_call(_attr: TokenStream, item: TokenStream) -> TokenStream {
    // 1. 解析被标注的函数
    let fn_item = parse_macro_input!(item as ItemFn);

    // 2. 提取所有需要保留的元数据
    let fn_attrs = &fn_item.attrs;       // doc comments + other attrs
    let fn_vis = &fn_item.vis;           // pub, pub(crate), etc.
    let fn_sig = &fn_item.sig;           // 签名（含泛型）
    let fn_block = &fn_item.block;       // 函数体
    let fn_name = &fn_item.sig.ident;    // 函数名
    let fn_name_str = fn_name.to_string();

    // 3. 提取参数名（用于打印）
    let param_names: Vec<String> = fn_sig
        .inputs
        .iter()
        .filter_map(|arg| {
            if let FnArg::Typed(pat_type) = arg {
                if let Pat::Ident(pat_ident) = &*pat_type.pat {
                    return Some(pat_ident.ident.to_string());
                }
            }
            None
        })
        .collect();

    let param_str = param_names.join(", ");

    // 4. 检查返回类型
    let return_type = &fn_sig.output;
    let has_return_value = !matches!(return_type, ReturnType::Default);

    // 5. 根据是否有返回值，生成不同的包装代码
    let expanded = if has_return_value {
        // 有返回值：闭包包装，捕获结果并打印
        quote! {
            #(#fn_attrs)*
            #fn_vis #fn_sig {
                println!(
                    "Entering {} with args: ({})",
                    #fn_name_str,
                    #param_str
                );
                // 使用闭包包装函数体，确保 ?、return 等在正确的作用域
                let __result = (|| #fn_block)();
                println!(
                    "Exiting {} with result: {:?}",
                    #fn_name_str,
                    __result
                );
                __result
            }
        }
    } else {
        // 无返回值（单元类型）：直接包装
        quote! {
            #(#fn_attrs)*
            #fn_vis #fn_sig {
                println!(
                    "Entering {} with args: ({})",
                    #fn_name_str,
                    #param_str
                );
                #fn_block;
                println!("Exiting {}", #fn_name_str);
            }
        }
    };

    expanded.into()
}
```

**关键设计决策说明**：

1. **`#(#fn_attrs)*`** — 保留了原始的所有 attributes，包括 doc comments、`#[inline]`、`#[allow(...)]` 等。
2. **`(#fn_sig)`** — 使用 `#fn_sig` 展开完整签名（含泛型参数、where 子句），保留了泛型支持。
3. **`(|| #fn_block)()` 的限制** — 这个闭包包装模式对 `return` 语句和 `?` 运算符的处理有限制：
   - `?` 运算符在闭包中要求闭包返回 `Result` 或 `Option`，如果函数返回的不是这些类型则会出错
   - `return` 只会从闭包中返回，不会从外层函数返回
   - **生产级解决方案**：不包装函数体，而是直接在函数体前后注入代码，但需要更复杂的宏来分割 `block.stmts`
4. **分支处理返回值** — 当返回 `()`（单元类型）时，`__result` 不可用 `{:?}` 打印（需要 `Debug` trait），所以分支为不打结果。

**改进版（处理 `?` 运算符）**：

```rust
// 对于有 ? 运算符的函数，更健壮的方案是展开 block 内部的语句：
let stmts = &fn_block.stmts;
let expanded = quote! {
    #(#fn_attrs)*
    #fn_vis #fn_sig {
        println!("Entering {} with args: ({})", #fn_name_str, #param_str);
        // 直接内联 stmts，不包装闭包
        let __result = {
            #(#stmts)*
        };
        println!("Exiting {} with result: {:?}", #fn_name_str, __result);
        __result
    }
};
```

### 练习 3 答案

#### 方案 A：利用 Rust 编译器的 const 求值（推荐）

```rust
use proc_macro::TokenStream;
use quote::quote;
use syn::Expr;

#[proc_macro]
pub fn calc(input: TokenStream) -> TokenStream {
    // 尝试将输入解析为 Rust 表达式
    let expr = match syn::parse::<Expr>(input) {
        Ok(expr) => expr,
        Err(e) => {
            // 如果解析失败，返回编译错误
            return e.to_compile_error().into();
        }
    };

    // 包装为 const 上下文，让编译器在编译期求值
    let expanded = quote! {
        {
            const __RESULT: i64 = #expr;
            __RESULT
        }
    };

    expanded.into()
}
```

**原理**：`const` 上下文中的表达式在编译期求值。`1 + 2 * 3` 作为 const 初始化器，Rust 编译器会自动应用运算符优先级规则并计算出 `7`。这使得我们的宏极其简洁且正确。

**优缺点**：
- 优点：简单、正确、自动支持所有算术运算
- 缺点：不允许使用变量（因为 const 上下文中不能引用运行时变量），但这恰好符合 "编译期求值" 的语义

#### 方案 B：手写递归下降解析器（完整控制）

```rust
use proc_macro::TokenStream;
use quote::quote;
use syn::{
    parse::{Parse, ParseStream},
    parse_macro_input, LitInt, Token,
};

// ---------- AST 定义 ----------

#[derive(Debug, Clone)]
enum ExprNode {
    Number(i64),
    BinaryOp {
        left: Box<ExprNode>,
        op: BinOp,
        right: Box<ExprNode>,
    },
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord)]
enum BinOp {
    Add,
    Sub,
    Mul,
    Div,
}

impl BinOp {
    fn precedence(&self) -> u8 {
        match self {
            BinOp::Add | BinOp::Sub => 1,
            BinOp::Mul | BinOp::Div => 2,
        }
    }
}

// ---------- 解析器 ----------

impl Parse for ExprNode {
    fn parse(input: ParseStream) -> syn::Result<Self> {
        // 使用 Pratt 解析器处理运算符优先级
        parse_expr(input, 0)
    }
}

fn parse_expr(input: ParseStream, min_prec: u8) -> syn::Result<ExprNode> {
    // 解析左操作数（原子表达式）
    let mut lhs = parse_atom(input)?;

    // 持续解析二元运算符，直到遇到优先级更低的运算符或输入结束
    loop {
        // 检查下一个 token 是否是二元运算符
        let op = if input.peek(Token![+]) {
            input.parse::<Token![+]>()?;
            BinOp::Add
        } else if input.peek(Token![-]) {
            input.parse::<Token![-]>()?;
            BinOp::Sub
        } else if input.peek(Token![*]) {
            input.parse::<Token![*]>()?;
            BinOp::Mul
        } else if input.peek(Token![/]) {
            input.parse::<Token![/]>()?;
            BinOp::Div
        } else {
            // 不是运算符，结束
            break;
        };

        let prec = op.precedence();
        if prec < min_prec {
            // 当前运算符优先级低于最小优先级，停止
            break;
        }

        // 递归解析右操作数，传递当前运算符优先级（左结合）
        let rhs = parse_expr(input, prec + 1)?;

        lhs = ExprNode::BinaryOp {
            left: Box::new(lhs),
            op,
            right: Box::new(rhs),
        };
    }

    Ok(lhs)
}

fn parse_atom(input: ParseStream) -> syn::Result<ExprNode> {
    if input.peek(syn::token::Paren) {
        // 括号表达式
        let content;
        syn::parenthesized!(content in input);
        let inner = parse_expr(&content, 0)?;
        Ok(inner)
    } else if input.peek(LitInt) {
        // 整数字面量
        let lit: LitInt = input.parse()?;
        let value: i64 = lit.base10_parse()
            .map_err(|e| syn::Error::new(lit.span(),
                format!("invalid integer literal: {}", e)))?;
        Ok(ExprNode::Number(value))
    } else if input.is_empty() {
        Err(syn::Error::new(
            input.span(),
            "unexpected end of input: expected an expression"
        ))
    } else {
        let lookahead = input.lookahead1();
        Err(lookahead.error())
    }
}

// ---------- 求值器 ----------

fn evaluate(node: &ExprNode) -> Result<i64, String> {
    match node {
        ExprNode::Number(n) => Ok(*n),
        ExprNode::BinaryOp { left, op, right } => {
            let l = evaluate(left)?;
            let r = evaluate(right)?;
            match op {
                BinOp::Add => Ok(l.checked_add(r).ok_or("overflow in addition")?),
                BinOp::Sub => Ok(l.checked_sub(r).ok_or("overflow in subtraction")?),
                BinOp::Mul => Ok(l.checked_mul(r).ok_or("overflow in multiplication")?),
                BinOp::Div => {
                    if r == 0 {
                        Err("division by zero".to_string())
                    } else {
                        Ok(l / r)
                    }
                }
            }
        }
    }
}

// ---------- 宏入口 ----------

#[proc_macro]
pub fn calc(input: TokenStream) -> TokenStream {
    let expr = parse_macro_input!(input as ExprNode);

    match evaluate(&expr) {
        Ok(result) => {
            let expanded = quote! { #result };
            expanded.into()
        }
        Err(msg) => {
            // 返回编译错误
            syn::Error::new(
                proc_macro2::Span::call_site(),
                format!("calc! error: {}", msg),
            )
            .to_compile_error()
            .into()
        }
    }
}
```

**Pratt 解析器说明**：

`parse_expr(input, min_prec)` 实现了 Pratt 解析算法：

1. 先解析一个原子表达式（数字或括号）
2. 循环检查下一个 token：
   - 如果是运算符，比较其优先级与 `min_prec`：
     - 优先级 >= `min_prec`：解析运算符，递归解析右操作数（用 `prec + 1` 确保左结合）
     - 优先级 < `min_prec`：停止，返回左侧结果

示例执行过程 `1 + 2 * 3`：

```
parse_expr(min_prec=0):
  atom: Number(1)
  peek '+', prec=1 >= 0 → parse '+'
    parse_expr(min_prec=2, 右操作数):
      atom: Number(2)
      peek '*', prec=2 >= 2 → parse '*'
        parse_expr(min_prec=3, 右操作数):
          atom: Number(3)
          no more ops → return Number(3)
      → BinaryOp(2, *, 3)
    → BinaryOp(1, +, BinaryOp(2, *, 3))
```

最终 AST: `(1 + (2 * 3))`，求值得 `7`。

**优先级验证**：

```
calc!((1 + 2) * 3)  →  (1 + 2) 先求值 → 3, 然后 3 * 3 = 9  ✓
calc!(1 + 2 * 3)    →  2 * 3 = 6, 然后 1 + 6 = 7           ✓
calc!(100 / (2 + 3)) → 2 + 3 = 5, 然后 100 / 5 = 20       ✓
```
