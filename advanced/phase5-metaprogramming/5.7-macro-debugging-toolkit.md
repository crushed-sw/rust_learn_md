## 设计哲学：宏调试为什么这么难

在 Rust 的所有子系统中，宏的调试难度是公认最高的。这不是偶然——它是由元编程的本质决定的：

**问题 1：间接层**

宏是"写代码的代码"。当你的 derive 宏出错时，编译器报告的错误信息指向的是你的宏**生成**的代码，而不是你**写的**宏代码。你和错误之间隔着整整一个间接层。这就像你写了一封信，邮差把信弄丢了，警察来问你为什么把信弄丢了——你面对的根本不是你做过的事情。

**问题 2：黑箱执行**

Proc macro 在编译时以动态库的形式被编译器加载和执行。它们运行在一个受限的环境中：没有 std::io::stdout()（被编译器占用了），无法 attach 调试器（如果你强行 attach，你调试的是 rustc 进程而不是你的宏），panic 会跨越 FFI 边界引发 Internal Compiler Error (ICE)。Rust 编译器团队已经尽力改善这个体验了，但"编译时运行"这个根本事实不变，调试就不会变得像普通代码一样简单。

**问题 3：Hygiene 的幽灵**

Rust 的声明宏有一套复杂的卫生（hygiene）系统。它大部分时候默默地做正确的事情，但当它出错时，出错的症状极其诡异：一个变量明明定义了却报"cannot find value"，一个类型明明导入了却报"unresolved import"。问题的根源在于：宏展开后的标识符在"语法上下文"(Syntax Context) 层面携带了看不见的信息，而这些信息决定了名称解析的结果。"宏出错了不可怕，可怕的是，展开后的代码看起来完全正确。"

**问题 4：错误信息的中断链**

在普通 Rust 代码中，如果函数 A 调用函数 B 调用函数 C 出错，你得到的是一个完整的调用栈。在宏中，这个链条是断裂的：

```text
普通代码:
  main() → func_a() → func_b() → func_c()  // 完整的回溯

宏代码:
  main.rs → #[derive(MyMacro)] → ??? → 展开后的代码中有错误
                                    ↑
                              没有任何回溯信息
```

编译器知道展开后的代码中有错误，但它不知道这个错误是宏生成的。它只是忠实地报告：第 X 行第 Y 列，类型不匹配。而第 X 行第 Y 列对应的是展开后的 token 流，不是你写的 proc macro 源码。

**解决方案：系统化方法论**

好消息是：所有这些问题都有系统化的解决方案。本课将四大调试工具、十类反模式、一套完整诊断流程组合成一个闭环：

```text
    ┌─────────────────────────────────────┐
    │         宏调试系统化方法论           │
    │                                     │
    │  1. cargo expand  —— 看见展开结果   │
    │  2. trace_macros!  —— 追踪展开过程  │
    │  3. 三板斧        —— 介入 proc macro│
    │  4. trybuild      —— 测试错误路径   │
    │                                     │
    │  + 十类反模式     —— 知识库匹配     │
    │  + 五步诊断法     —— 流程化解决     │
    └─────────────────────────────────────┘
```

这四个工具不是孤立的——它们在五步诊断流程中按顺序使用，形成一个从"发现问题"到"修复验证"的完整闭环。

## 工具一：cargo expand —— 看见不可见之物

### 为什么 cargo expand 是第一工具

如果你只能记住本节课的一个要点，请记住这个：**在尝试理解任何宏的行为之前，先用 `cargo expand` 看它生成了什么。**

编译器报错说第 42 行有类型错误。但第 42 行在你的源代码中是什么？是一个 `#[derive(MyMacro)]` 注解。真正的错误代码在第 42 行展开后的结果中，你看不到它。"cargo expand 把宏展开后的代码打印出来——让不可见变成可见。这是调试的第一步，也是最关键的一步。"

### 安装

```bash
cargo install cargo-expand
```

`cargo-expand` 本质上是对 `rustc -Zunpretty=expanded` 的一个友好封装。它处理了 nightly 工具链切换、模块路径解析、以及格式化输出。你不需要手动用 nightly——cargo-expand 会自动为展开操作使用 nightly 工具链（而你的项目仍然可以用 stable 编译）。

### 基础用法

```bash
# 展开整个 crate
cargo expand

# 展开特定的模块或路径
cargo expand path::to::item

# 展开测试模块
cargo expand --tests

# 仅展开库
cargo expand --lib

# 展开特定的 binary target
cargo expand --bin my_binary

# 展开特定模块（常见的 Debug 场景）
cargo expand my_module::my_function
```

**最佳实践：永远从最小范围开始。** 不要一上来就 `cargo expand` 整个 crate——数千行展开代码会淹没你的问题。先用 `cargo expand path::to::specific_item` 只看你关心的那条。如果问题不在那里，再扩大范围。

### 解读 cargo expand 的输出

cargo expand 的输出不是 Rust 源代码——它是编译器在宏展开后、类型检查前的中间表示。"学会'读' expand 输出是宏调试的核心技能。"

以下是解读时需要关注的关键点：

#### 1. 区分原始代码和展开代码

```rust
// cargo expand 的输出中：
// 原始代码（你写的）和展开后的代码（宏生成的）混合在一起

// 你写的不变代码：
struct Point {
    x: i32,
    y: i32,
}

// #[derive(Debug)] 展开后的代码（插入到同一个文件中）：
#[automatically_derived]
impl ::core::fmt::Debug for Point {
    fn fmt(&self, f: &mut ::core::fmt::Formatter) -> ::core::fmt::Result {
        // ... 宏生成的代码
    }
}
```

寻找标记：`#[automatically_derived]` 是 derive 宏生成代码的标志。对于声明宏，你通常可以看到从宏调用处展开出来的大段代码。

#### 2. 识别 Hygiene 标记

展开后的代码中，你可能会看到 `r#` 前缀的标识符：

```rust
// 声明宏中使用了 let x = 1;
// 展开后可能变成：
let r#x_123abc = 1;  // 编译器为了卫生性重命名了 x
```

`r#` 是 raw identifier 语法，通常不是 hygiene 的直接产物，但如果你的宏中使用了 raw identifier，展开后要特别注意。更常见的 hygiene 表现是：某些标识符被附加了不可见的 Syntax Context 信息，使得它们虽然名字相同，但在编译器的眼中是"不同的变量"——这就是著名的"hygiene bug"。

#### 3. 发现意外的代码生成

对比"你期望生成的代码"和"实际展开的代码"：

```rust
// 你期望宏生成：
impl MyTrait for MyType {
    fn my_method(&self) -> u32 { 42 }
}

// 实际展开结果：
impl MyTrait for MyType {
    fn my_method(&self) -> u32 {
        42  // ← 有多余的分号？少了 return？类型不对？
    }
}
```

任何差异都是问题所在——cargo expand 让你看到这个差异。

#### 4. 识别丢失的属性或文档注释

一个常见的 bug：derive 宏或 attribute 宏"吞掉"了用户的属性：

```rust
// 用户写了：
#[my_attribute]
/// Documentation for the struct
#[derive(Debug, Clone)]
struct MyStruct { field: i32 }

// 如果 #[my_attribute] 不正确处理属性传递，
// expand 的结果中会丢失 #[derive(Debug, Clone)] 和文档注释
```

### 实战走读

#### 示例 1：展开 #[derive(Debug)]

创建一个简单的项目：

```bash
cargo new --lib expand_demo
cd expand_demo
```

在 `src/lib.rs` 中写入：

```rust
#[derive(Debug)]
struct Point {
    x: i32,
    y: i32,
}
```

运行 `cargo expand`：

```rust
// cargo expand 的输出（简化版，展示关键结构）：
#![feature(prelude_import)]
#[prelude_import]
use std::prelude::rust_2021::*;
#[macro_use]
extern crate std;

struct Point {
    x: i32,
    y: i32,
}

#[automatically_derived]
impl ::core::fmt::Debug for Point {
    #[inline]
    fn fmt(&self, f: &mut ::core::fmt::Formatter) -> ::core::fmt::Result {
        // debug_struct 方法：创建一个 struct 格式的调试输出
        ::core::fmt::Formatter::debug_struct_field2_finish(
            f,
            "Point",          // struct 的名称
            "x",              // 字段 1 的名称
            &self.x,          // 字段 1 的引用
            "y",              // 字段 2 的名称
            &self.y,          // 字段 2 的引用
        )
    }
}
```

仔细观察：
1. `#[automatically_derived]` 标记表明这是 derive 宏生成的
2. 调用的是 `debug_struct_field2_finish`——注意 "field2"，编译器为 2 个字段的 struct 优化了专门的格式化函数
3. 字段名是字符串字面量——这是在编译时确定的，不是运行时反射
4. 使用的是完全限定路径 `::core::fmt::Debug`——绝不依赖 use 语句

#### 示例 2：展开 `vec![1, 2, 3]`

```rust
// 源码：
let v = vec![1, 2, 3];

// cargo expand 输出（关键部分）：
let v = {
    // 创建一个 Vec
    let mut v = ::std::vec::Vec::new();
    // 逐个 push 元素
    v.push(1);
    v.push(2);
    v.push(3);
    // 返回 v
    v
};
// 如果元素数量大于等于 capacity，会调用 with_capacity 预分配
// 这里 3 个元素在 Vec 的默认容量内，所以用 new()
```

注意几个细节：
- 整个 `vec![]` 展开为一个块表达式 `{ ... }`
- 内部使用 `$crate::vec::Vec` 来确保路径正确
- 对于更大的序列，编译器可能会使用 `::std::vec::Vec::with_capacity(n)` 预分配

#### 示例 3：发现 hygiene bug

假设你写了一个声明宏来简化某个重复操作：

```rust
macro_rules! with_temp {
    ($expr:expr) => {
        let temp = 42;
        $expr
    };
}

fn main() {
    let temp = 10;
    with_temp!(println!("{}", temp)); // 期望打印 42，实际打印 10
}
```

运行 `cargo expand`：

```rust
// 展开结果：
fn main() {
    let temp = 10;           // 用户的 temp
    let r#temp_abc = 42;     // 宏的 temp（被 hygiene 重命名）
    {
        println!("{}", temp); // 解析到用户的 temp（10），不是宏的 temp（42）
    };
}
```

"这个展开结果直接揭示了问题：宏中的 `temp` 和用户代码中的 `temp` 在 hygiene 系统中是不同的标识符。宏生成的 `temp` 不会遮蔽用户的 `temp`——这是 hygiene 的保护机制。如果你确实需要宏引入一个变量供用户代码使用，你需要让调用者传入标识符名。"

**修复方案**：

```rust
macro_rules! with_temp {
    ($name:ident, $expr:expr) => {
        let $name = 42;
        $expr
    };
}

with_temp!(temp, println!("{}", temp)); // 现在打印 42
```

## 工具二：trace_macros! / log_syntax! —— 声明宏的慢镜头

### trace_macros!

`trace_macros!` 是编译器内置的调试工具，它会打印每个宏调用的展开过程。

```rust
#![feature(trace_macros)]

fn main() {
    trace_macros!(true);
    vec![1, 2, 3];
    trace_macros!(false);
}
```

编译时的输出：

```text
note: trace_macro
 --> src/main.rs:5:5
  |
5 |     vec![1, 2, 3];
  |     ^^^^^^^^^^^^^
  |
  = note: expanding `vec! { 1, 2, 3 }`
  = note: to `$crate :: __rust_force_expr ! (:: std :: vec :: Vec :: new ())`
  = note: expanding `__rust_force_expr! { :: std :: vec :: Vec :: new () }`
  = note: to `:: std :: vec :: Vec :: new ()`
  // ... 递归展开过程
```

**用法限制**：
- 需要 `#![feature(trace_macros)]`——只能在 nightly Rust 上使用
- 只对声明宏（macro_rules!）有效——proc macro 不受影响
- 输出可能非常冗长——配合 grep 或重定向到文件

### log_syntax!

`log_syntax!` 是更细粒度的工具——它打印单个 token：

```rust
#![feature(log_syntax)]

macro_rules! debug_tokens {
    ($($t:tt)*) => {
        log_syntax!($($t)*);
        // 对每个 token 单独 log
        $( log_syntax!(token: $t); )*
    };
}

// 使用：
debug_tokens!(hello 42 + world);
// 编译时输出：token: hello, token: 42, token: +, token: world
```

### 使用场景

#### 场景 1：理解递归宏的展开顺序

```rust
#![feature(trace_macros)]

macro_rules! recurse {
    // 终止条件
    ($e:expr) => { $e };
    // 递归情况
    ($e:expr, $($rest:expr),*) => {
        recurse!($($rest),*) + $e
    };
}

fn main() {
    trace_macros!(true);
    let _x = recurse!(1, 2, 3);
    trace_macros!(false);
}
```

**trace_macros! 的输出会依次展示**：
1. `recurse!(1, 2, 3)` → `recurse!(2, 3) + 1`
2. `recurse!(2, 3)` → `recurse!(3) + 2`
3. `recurse!(3)` → `3`
4. 最终结果：`((3) + 2) + 1`

"trace_macros! 让你看到的是宏展开的**顺序**——谁先展开，谁后展开，递归如何深入和退出。cargo expand 让你看到的是展开的**最终结果**。两者互补。"

#### 场景 2：调试 TT muncher

TT muncher 是一种常见的声明宏模式——逐 token 地"啃食"输入，通常在每次迭代中处理最前面的 token(s)，然后递归调用自身处理剩余的：

```rust
macro_rules! tt_muncher {
    // 基础情况：没有 token 了
    () => {};
    // 处理 `let $name = $value;`
    (let $name:ident = $value:expr; $($rest:tt)*) => {
        // 在这里做某些操作
        println!("defined: {}", stringify!($name));
        tt_muncher!($($rest)*);
    };
    // 跳过未知的 token
    ($first:tt $($rest:tt)*) => {
        // 吞掉这个 token
        tt_muncher!($($rest)*);
    };
}
```

当 TT muncher "吃到"了你不期望的 token 时，使用 `log_syntax!` 来追踪：

```rust
#![feature(log_syntax)]

macro_rules! debug_muncher {
    () => {};
    ($first:tt $($rest:tt)*) => {
        log_syntax!(FIRST_TOKEN: $first);
        debug_muncher!($($rest)*);
    };
}
```

#### 场景 3：trace_macros! vs cargo expand —— 何时用哪个

| 维度 | trace_macros! | cargo expand |
|------|---------------|--------------|
| 展示内容 | 展开过程（步骤） | 展开结果（最终） |
| 适用宏类型 | 仅声明宏 | 所有宏 |
| 需要 nightly | 是 | cargo-expand 自动处理 |
| 输出量 | 中等（每步一行） | 大（完整展开代码） |
| 最适合 | 递归逻辑、匹配顺序 | 最终代码正确性 |
| 典型问题 | "为什么选择了这个分支" | "为什么展开后类型不对" |

**经验法则**：如果问题是"宏匹配了错误的分支"（编译通过了但逻辑不对），用 trace_macros!。如果问题是"宏展开后编译器报错"（编译不通过），用 cargo expand。

## 工具三：Proc Macro 调试三板斧

这是本课最核心的部分。Proc macro 的调试是 Rust 元编程中最困难的场景。常规的调试手段（println!、gdb/lldb、panic 回溯）在 proc macro 中要么不可用，要么行为异常。以下是**经过实战检验**的三板斧。

### 板斧 1：eprintln! 大法

**最简单、最直接、最有效。** 如果 proc macro 调试有"Hello World"，那就是 eprintln!。

```rust
use proc_macro::TokenStream;
use syn::{parse_macro_input, DeriveInput};

#[proc_macro_derive(MyDerive)]
pub fn my_derive(input: TokenStream) -> TokenStream {
    // ===== 调试入口：打印原始输入 =====
    eprintln!("[MyDerive] ===== RAW INPUT =====");
    eprintln!("[MyDerive] {}", input);

    // 解析输入
    let input = parse_macro_input!(input as DeriveInput);

    // ===== 调试中点：打印解析后的 AST =====
    eprintln!("[MyDerive] ===== PARSED AST =====");
    eprintln!("[MyDerive] ident: {}", input.ident);
    eprintln!("[MyDerive] generics: {}", input.generics.params.iter().count());
    for attr in &input.attrs {
        eprintln!("[MyDerive] attr: {}", attr.path.get_ident().map_or("<path>".into(), |i| i.to_string()));
    }

    // 生成代码
    let output = quote! {
        impl MyTrait for #my_type {
            fn my_method(&self) -> String {
                "hello".to_string()
            }
        }
    };

    // ===== 调试出口：打印生成的代码 =====
    eprintln!("[MyDerive] ===== GENERATED OUTPUT =====");
    eprintln!("[MyDerive] {}", output);

    output.into()
}
```

**核心技术要点**：

1. **eprintln! 输出到 stderr**。Cargo 在编译过程中会显示 stderr 的输出（以黄色/红色文本），你可以直接看到它。

2. **使用统一的前缀标签**。`[MyDerive]` 这样的标签让你可以用 grep 过滤输出：
   ```bash
   cargo build 2>&1 | grep '\[MyDerive\]'
   ```

3. **打印三个关键点**：input（接受到的 token）→ parsed（解析后的 AST）→ output（生成的 token）。哪个环节出问题一目了然。

4. **在关键分支路径上打印**：
   ```rust
   match &input.data {
       syn::Data::Struct(data) => {
           eprintln!("[MyDerive] Processing STRUCT: {}", input.ident);
           // ...
       }
       syn::Data::Enum(data) => {
           eprintln!("[MyDerive] ERROR: expected struct, found enum: {}", input.ident);
           // ... generate compile_error!
       }
       syn::Data::Union(data) => {
           eprintln!("[MyDerive] ERROR: expected struct, found union: {}", input.ident);
       }
   }
   ```

**限制**：

- **stderr 是共享的**。如果你的项目中有 50 个 crate 在同时编译，你的 eprintln! 输出会混在数千行编译输出中。使用唯一的前缀标签来 grep。
- **无法在 proc macro 中使用 println!**。proc macro 运行在编译器进程中，stdout 被编译器占用。eprintln! 才是正确的选择。
- **格式化可能被截断**。如果 token stream 非常大，eprintln! 的输出可能只显示前几百字符。用 Debug 格式打印时注意这一点。

### 板斧 2：将 proc macro 提取为独立 binary 或单元测试

"eprintln! 告诉你发生了什么，但无法让你单步调试。如果你需要设置断点、逐行执行、检查变量状态——你需要把 proc macro 的逻辑提取到一个可以独立运行的环境中。"

**方案 A：提取核心逻辑到辅助 crate**

这是推荐的项目结构：

```text
my_proc_macro/
├── Cargo.toml               # workspace
├── my-proc-macro/           # proc-macro crate（thin wrapper）
│   ├── Cargo.toml           # proc-macro = true
│   └── src/lib.rs            # 仅仅调用 my-proc-macro-core
├── my-proc-macro-core/       # 核心逻辑（普通 crate）
│   ├── Cargo.toml
│   └── src/lib.rs            # 包含所有解析和代码生成逻辑
└── tests/
    └── integration_test.rs   # 可以正常调试的测试
```

`my-proc-macro-core/src/lib.rs` 包含纯逻辑：

```rust
use proc_macro2::TokenStream;
use syn::{parse2, DeriveInput};

/// 纯函数：接收 proc_macro2 TokenStream，返回 proc_macro2 TokenStream。
/// 这个函数可以在单元测试中调用，可以在独立 binary 中调试。
pub fn my_derive_impl(input: TokenStream) -> syn::Result<TokenStream> {
    let input: DeriveInput = parse2(input)?;

    let name = &input.ident;
    // ... 核心逻辑 ...

    let output = quote! {
        impl MyTrait for #name {
            fn my_method(&self) -> String {
                // ...
            }
        }
    };

    Ok(output)
}
```

`my-proc-macro/src/lib.rs` 是 thin wrapper：

```rust
use proc_macro::TokenStream;
use my_proc_macro_core::my_derive_impl;

#[proc_macro_derive(MyDerive)]
pub fn my_derive(input: TokenStream) -> TokenStream {
    let input: proc_macro2::TokenStream = input.into();
    match my_derive_impl(input) {
        Ok(output) => output.into(),
        Err(e) => e.to_compile_error().into(),
    }
}
```

**方案 B：利用 proc_macro2 的测试能力**

如果你的逻辑不太复杂，可以直接在 proc-macro crate 中写测试（需要利用 `proc_macro2` 的非 proc-macro 模式）：

```rust
// 在 proc-macro crate 的 lib.rs 中：
#[cfg(not(feature = "proc-macro"))]  // 当不以 proc-macro 模式编译时
mod tests {
    use proc_macro2::TokenStream;

    #[test]
    fn test_my_derive_logic() {
        // 构造测试输入
        let input: TokenStream = quote! {
            struct TestStruct {
                field1: i32,
                field2: String,
            }
        };

        // 调用核心逻辑
        let output = my_derive_impl(input).unwrap();

        // 检查输出
        let output_str = output.to_string();
        assert!(output_str.contains("impl MyTrait for TestStruct"));
        assert!(output_str.contains("fn my_method"));
    }

    #[test]
    fn test_error_on_enum() {
        let input: TokenStream = quote! {
            enum TestEnum { A, B }
        };
        let result = my_derive_impl(input);
        assert!(result.is_err());
    }
}
```

这种方法的关键是：在 `Cargo.toml` 中配置 feature flag，使得测试时不以 proc-macro 模式编译：

```toml
[lib]
proc-macro = true

[features]
# 当你不以 proc-macro 模式编译时（如测试），不设置此 feature
proc-macro = []

[dev-dependencies]
# dev 依赖不自动启用 proc-macro feature，所以测试会走非 proc-macro 路径
```

实际上，更常见的做法是直接用 `cargo test -p my-proc-macro-core` 来测试提取出来的核心库。

### 板斧 3：compile_error! 断言式调试

当 eprintln! 不够直观，或者你想让"调试输出"以编译错误的形式展示时，使用 compile_error!。

```rust
use proc_macro::TokenStream;
use quote::quote;

#[proc_macro_derive(MyDerive)]
pub fn my_derive(input: TokenStream) -> TokenStream {
    let input_clone = input.clone();  // 需要在原始 input 被消费前克隆

    // 1. 先检查输入
    let debug_input = quote! {
        compile_error!(concat!("[MyDerive] Input tokens: ", stringify!(#input_clone)));
    };
    eprintln!("[MyDerive] Input: {}", input_clone);  // 也打印到 stderr

    // 2. 解析输入
    let input = syn::parse_macro_input!(input as syn::DeriveInput);

    // 3. 调试：验证字段名提取
    if let syn::Data::Struct(ref data) = input.data {
        for field in &data.fields {
            let field_name = field.ident.as_ref().unwrap();
            let debug_field = quote! {
                compile_error!(concat!(
                    "[MyDerive] Processing field: ",
                    stringify!(#field_name),
                    ", type: ",
                    stringify!(#field.ty)
                ));
            };
            // 在正常代码中不要包含 compile_error!，这里仅用于调试：
            // eprintln!("[MyDerive] Field: {:?} -> {:?}", field_name, field.ty);
        }
    }

    let name = &input.ident;
    let expanded = quote! {
        impl MyTrait for #name {
            fn my_method(&self) -> String {
                String::from("hello")
            }
        }
    };

    expanded.into()
}
```

**compile_error! 调试技巧**：

1. **作为"编译期断点"**：在怀疑有问题的代码路径上插入 `compile_error!`，如果它触发了，说明你的怀疑正确——代码确实走到了这条路径。

2. **验证中间状态**：
   ```rust
   // 在宏生成的代码中插入：
   let debug_state = quote! {
       compile_error!(concat!(
           "State at this point: generics_count=",
           stringify!(#generics_count),
           ", fields_count=",
           stringify!(#fields_count),
       ));
   };
   ```

3. **对比 cargo expand 和 compile_error! 的输出顺序**：如果你在宏输出中同时包含 `compile_error!` 和实际代码，编译器会先报告 compile_error! 然后停止，而不是继续报告后续的类型错误。这对于隔离问题非常有用。

### Span 调试专项

Span 是 proc macro 调试中最容易被忽视但最重要的部分。"正确的 Span 信息是好的编译器错误信息的灵魂。一个没有正确 Span 的错误信息等同于'你代码里的某个地方有问题'——这比没有错误信息更令人沮丧。"

#### 检查 Span 信息

```rust
use proc_macro2::{TokenStream, Span};
use syn::{parse_macro_input, DeriveInput};

// 需要 proc_macro2 的 "extra-traits" feature
// 在 Cargo.toml 中：
// [dependencies]
// proc_macro2 = { version = "1", features = ["extra-traits"] }

#[proc_macro_derive(MyDerive)]
pub fn my_derive(input: TokenStream) -> TokenStream {
    let input: proc_macro2::TokenStream = input.into();

    // 遍历每个 token，检查其 span
    for token in input.clone() {
        eprintln!(
            "[SpanDebug] token={:?}, span={:?}",
            token,
            token.span()
        );
    }

    // 检查 span 的源码位置
    // 注意：source_file() 和 source_text() 只在 proc_macro2 中有
    // 它们依赖 nightly 或者特定条件
    let parsed = parse_macro_input!(input as DeriveInput);
    let ident_span = parsed.ident.span();

    // 在 proc_macro crate (不是 proc_macro2) 中可以用：
    // span.unwrap().source_file().path()   // 文件路径
    // span.unwrap().start()               // 起始行列号
    // span.unwrap().end()                 // 结束行列号

    // ...
}
```

#### Span 最佳实践

```rust
use quote::quote_spanned;

// 坏做法：使用 call_site() —— 错误信息都指向相同的位置
let bad = quote! {
    impl #trait_ident for #type_ident {
        // 所有错误都会指向宏调用处，而不是具体的字段或类型
    }
};

// 好做法：使用 quote_spanned! 保留原始 token 的 span
let good = quote_spanned! { type_ident.span() =>
    impl #trait_ident for #type_ident {
        fn method(&self) -> #output_type {
            // 错误会指向 type_ident 的实际位置
        }
    }
};

// 更好：在错误信息中使用具体的 span
use syn::spanned::Spanned;

fn my_macro_impl(input: DeriveInput) -> syn::Result<TokenStream> {
    let data = match &input.data {
        syn::Data::Struct(data) => data,
        syn::Data::Enum(data) => {
            // 错误指向 enum 关键字的位置
            return Err(syn::Error::new(
                data.enum_token.span,
                "MyDerive can only be derived for structs, not enums"
            ));
        }
        _ => {
            return Err(syn::Error::new(
                input.span(),
                "MyDerive can only be derived for structs"
            ));
        }
    };
    // ...
}
```

**Span 调试的核心原则**：

> "永远不要创建新的 Span。保留输入 token 的 span，使用 `quote_spanned!` 传播它们，在错误信息中使用 syn::Error::new(span, msg)。这样用户看到的错误信息会指向他们代码中的正确位置——而不是一个随机的宏调用处。"

## 工具四：trybuild —— 编译失败的测试

### 设计哲学

> "测试'代码能编译'是不够的——你还需要测试'错误的代码产生好的错误信息'。宏的用户体验不仅取决于正确情况下的行为，更取决于错误情况下的诊断质量。trybuild 让你能够像测试运行时行为一样测试编译时错误。"

### 项目结构

```text
my_proc_macro/
├── Cargo.toml
├── src/
│   └── lib.rs              # 你的 proc macro 代码
└── tests/
    ├── ui.rs               # trybuild 测试驱动程序
    └── ui/                  # 测试用例目录
        ├── pass/            # 应该编译成功的代码
        │   ├── 01-basic.rs
        │   └── 02-generics.rs
        └── fail/            # 应该编译失败的代码
            ├── 01-enum-not-struct.rs
            ├── 01-enum-not-struct.stderr  # 期望的错误输出
            ├── 02-missing-attr.rs
            ├── 02-missing-attr.stderr
            └── 03-invalid-value.rs
            └── 03-invalid-value.stderr
```

### 设置 trybuild

Cargo.toml 添加依赖：

```toml
[dev-dependencies]

```

```bash

# 后续运行：检查测试通过
cargo test

# 如果修改了错误信息，重新生成 .stderr
TRYBUILD=overwrite cargo test
```

### 关键原则

1. **.stderr 文件是你的 API 的一部分**。用户依赖你的错误信息来诊断问题。每次修改错误信息，都是在改变 API。你的 CI 应该运行 trybuild 测试以确保不会意外地改变错误信息。

2. **好的错误信息 = Span 精确 + 消息有帮助**。不要满足于 "compile_error!"，而要使用 `syn::Error::new(span, helpful_message)`：

   ```rust
   // 差：
   Err(syn::Error::new(Span::call_site(), "invalid"))

   // 好：
   Err(syn::Error::new(
       attr.span(),
       "expected #[my_attr(name = \"...\")], found an unsupported argument. \
        Valid arguments are: \"name\".",
   ))
   ```

3. **Pass 测试不应太少**。除了基础用法，还应覆盖：
   - 泛型类型
   - 生命周期参数
   - 带 where clause 的泛型
   - 空 struct / 元组 struct
   - 带其他 derive 的其他用法

4. **Fail 测试覆盖主要错误路径**。至少应该覆盖：
   - 应用到错误的语法类别（struct/enum/union/function）
   - 缺少必需属性
   - 属性值语法错误
   - 类型约束冲突
   - 命名冲突

## 反模式大全 —— 元编程十大陷阱

"以下十个反模式是从真实世界的 Rust 宏代码中收集的。每一个反模式都遵循相同的结构：症状 → 根因 → 诊断方法 → 修复方案。当你遇到宏问题时，可以对号入座。"

---

### 反模式 1：声明宏忘记 $crate

**症状**：
- 宏在定义它的 crate 中工作正常
- 当其他 crate 通过 `use` 导入这个宏后，编译失败
- 错误信息：`cannot find type/function/module MyType`

**根因**：
声明宏展开后，其中的路径（如 `MyType`、`my_function`）相对于**调用宏的 crate** 解析，而不是**定义宏的 crate**。如果调用方没有导入这些名称，就会报"找不到"。

**诊断方法**：
`cargo expand` 展开宏调用处，检查展开后的路径。如果路径缺少 crate 前缀（如 `::my_crate::MyType`），就是这个问题。

**修复方案**：
在宏内部使用 `$crate::` 前缀：

```rust
// 错误：
macro_rules! my_macro {
    () => {
        MyType::new()              // 相对于调用方解析
        crate::helper_function()   // crate 指向调用方的 crate
    };
}

// 正确：
macro_rules! my_macro {
    () => {
        $crate::MyType::new()      // 始终解析到定义宏的 crate
        $crate::helper_function()
    };
}

// 对于标准库路径，使用绝对路径：
macro_rules! with_vec {
    ($($x:expr),*) => {
        ::std::vec![$($x),*]       // 绝对路径，不依赖调用方
    };
}
```

**经验法则**："在宏内部，永远不要使用相对于调用方的路径。要么用 `$crate::`（引用你自己的 crate），要么用绝对路径（`::std::...`、`::core::...`）。"

---

### 反模式 2：proc macro 中 panic 导致 ICE

**症状**：
- 编译器崩溃，输出 "internal compiler error" (ICE)
- 不显示你期望的用户友好的错误信息
- 编译器的 panic message 中可能提到你的 proc macro

**根因**：
Proc macro 运行在编译器进程内部。当你的 proc macro 调用 `panic!()` 或 `unwrap()` 遇到 `None`/`Err` 时，panic 会跨越 FFI 边界（proc macro 通过 C ABI 与编译器通信），导致未定义行为。Rust 编译器会尝试捕获这个 panic 并转换为编译错误，但这不是总能成功——结果就是 ICE。

**诊断方法**：
- 如果编译器报告 ICE，首先检查你的 proc macro 中是否有 `unwrap()`、`expect()`、`panic!()`
- 在关键路径上插入 `eprintln!` 来找到 panic 的位置

**修复方案**：
在 proc macro 中，**永远不要 panic**。使用 `syn::Error` 或返回 `TokenStream::from(compile_error!(...))`：

```rust
use proc_macro::TokenStream;

// 错误：会导致 ICE
#[proc_macro_derive(MyDerive)]
pub fn my_derive(input: TokenStream) -> TokenStream {
    let input = syn::parse_macro_input!(input as syn::DeriveInput);

    // 如果解析到 enum 就直接 unwrap —— 可能导致 ICE
    let data = match &input.data {
        syn::Data::Struct(data) => data,
        _ => panic!("MyDerive only supports structs"), // ← BAD
    };
    // ...
}

// 正确：优雅地处理错误
#[proc_macro_derive(MyDerive)]
pub fn my_derive(input: TokenStream) -> TokenStream {
    let input = syn::parse_macro_input!(input as syn::DeriveInput);

    let data = match &input.data {
        syn::Data::Struct(data) => data,
        _ => {
            // 方式 1：使用 syn::Error（推荐）
            let err = syn::Error::new(
                proc_macro2::Span::call_site(),
                "MyDerive only supports structs"
            );
            return err.to_compile_error().into();
        }
    };
    // ...
}

// 更好的模式：用 Result 包裹整个逻辑
fn my_derive_impl(input: syn::DeriveInput) -> syn::Result<proc_macro2::TokenStream> {
    let name = &input.ident;
    let data = match &input.data {
        syn::Data::Struct(data) => data,
        _ => return Err(syn::Error::new(input.span(), "MyDerive only supports structs")),
    };
    // ... 核心逻辑 ...
    Ok(quote! { /* ... */ })
}

#[proc_macro_derive(MyDerive)]
pub fn my_derive(input: TokenStream) -> TokenStream {
    let input = syn::parse_macro_input!(input as syn::DeriveInput);
    match my_derive_impl(input) {
        Ok(tokens) => tokens.into(),
        Err(e) => e.to_compile_error().into(),
    }
}
```

**关键规则**：proc macro 的入口函数中，任何可能失败的操作都应该返回 `syn::Error` 而不是 panic。用 `?` 传播错误，用 `to_compile_error()` 转换为编译错误。

---

### 反模式 3：derive 宏不处理泛型参数

**症状**：
- derive 宏在具体类型上工作正常：`struct Foo { x: i32 }` 正常
- 在泛型类型上编译失败：`struct Foo<T> { x: T }` 报错
- 错误信息：`the trait bound T: MyTrait is not satisfied` 或 `cannot find type T in this scope`

**根因**：
derive 宏生成的 impl 块没有包含泛型参数和 where clause。例如，你的 derive 生成了：

```rust
impl MyTrait for Foo { ... }     // 没有 <T>，没有 where clause
```

而正确的应该是：

```rust
impl<T> MyTrait for Foo<T> where T: MyTrait { ... }
```

**诊断方法**：
`cargo expand` 展开泛型类型的 derive，检查生成的 impl 块：
- 是否有 `<T>`（等等泛型参数）？
- impl 块是否包含了正确的 where clause？
- 字段类型 `T` 是否需要额外的 trait bound？

**修复方案**：
使用 `syn::Generics` API 正确提取和处理泛型信息：

```rust
use syn::{DeriveInput, Generics};

fn process_generics(input: &DeriveInput) -> proc_macro2::TokenStream {
    let name = &input.ident;

    // 1. 提取泛型信息
    let generics = &input.generics;

    // 2. 分离泛型参数（impl 后面需要）
    let (impl_generics, ty_generics, where_clause) = generics.split_for_impl();

    // 3. 可能需要对泛型添加额外的 trait bound
    //    比如：如果字段类型都是 T，且你需要 Clone，就需要 T: Clone
    let mut generics_with_bounds = generics.clone();
    // 为每个泛型参数添加额外的 bound
    // （简化示例；实际中应基于字段分析决定哪些 bound 是必需的）
    //
    // for param in &mut generics_with_bounds.params {
    //     if let syn::GenericParam::Type(type_param) = param {
    //         type_param.bounds.push(syn::parse_quote!(MyTrait));
    //     }
    // }

    let (impl_generics_with_bounds, _, where_clause_with_bounds) =
        generics_with_bounds.split_for_impl();

    quote! {
        // 完整的 impl，包含泛型参数、类型参数、where clause
        impl #impl_generics_with_bounds MyTrait for #name #ty_generics
            #where_clause_with_bounds
        {
            fn my_method(&self) -> String {
                format!("{}", stringify!(#name))
            }
        }
    }
}
```

**`split_for_impl()` 的三元组详解**：

```rust
// 对于 struct Foo<'a, T: Display> where T: Clone { ... }
// split_for_impl() 返回：

// impl_generics:    <'a, T: Display>
// ty_generics:      <'a, T>
// where_clause:     where T: Clone
//
// 组合方式：
// impl <'a, T: Display> MyTrait for Foo <'a, T>
//     where T: Clone
// { ... }
```

**要点**：
- 泛型 trait 的 derive（如 `PartialEq`）会自动推断额外的 bound，但你的自定义 derive 需要自己处理
- 如果字段类型是 `T`，且你的代码中调用了 `T` 的某个方法，你需要在 where clause 中添加对应的 trait bound
- 如果字段中有 `PhantomData<T>`，而 `T` 不在其他任何字段中使用，你需要特殊处理（通常是添加 `T: Sized` 或对应的 bound）

### 反模式 4：Attribute 宏覆盖用户属性

**症状**：
- 应用你的 attribute 宏后，struct 失去了文档注释（`/// ...`）或其他 derive 宏
- 应用 `#[my_attr]` 后，`#[derive(Debug, Clone)]` 消失
- 用户需要重新添加你宏"吞掉"的属性

**根因**：
Attribute 宏替换整个 item。你的 proc macro 处理完属性后，返回了一个**不包含原始属性的**新 item。例如：

```rust
// 用户写了：
#[my_attr]
/// My documented struct
#[derive(Debug)]
struct MyStruct {
    field: i32,
}

// 你的宏返回：
struct MyStruct {
    field: i32,
}
// ← 文档注释和 #[derive(Debug)] 丢失
```

**诊断方法**：
`cargo expand` 看 attribute 宏展开后的 item，检查所有非 `#[my_attr]` 的属性是否还在。

**修复方案**：
在生成输出 token 时，**保留并重新发射非你的宏的属性**：

```rust
use proc_macro::TokenStream;
use syn::{parse_macro_input, Attribute, ItemStruct};

#[proc_macro_attribute]
pub fn my_attr(attr: TokenStream, item: TokenStream) -> TokenStream {
    let attr_args = parse_macro_input!(attr as syn::AttributeArgs);
    let mut input = parse_macro_input!(item as ItemStruct);

    // 1. 保存原始属性（除了我们自己的）
    let my_attr_names: &[&str] = &["my_attr"];
    let preserved_attrs: Vec<&Attribute> = input.attrs.iter()
        .filter(|attr| {
            !my_attr_names.iter().any(|name| attr.path.is_ident(name))
        })
        .collect();

    // 2. 处理我们的属性
    // ... 读取 attr_args ...

    // 3. 生成带"原始属性"和"生成的代码"的 item
    let generated_items = quote! { /* ... 生成的代码 ... */ };

    let name = &input.ident;
    let fields = &input.fields;

    let output = quote! {
        // 保留原始属性
        #(#preserved_attrs)*
        // 保留原始 item
        struct #name #fields

        // 生成额外的代码
        #generated_items
    };

    output.into()
}
```

**更健壮的方式——使用 outer_attrs API**：

```rust
// 对于 syn 2.x，可以直接通过 attrs 字段访问
// 并在生成时重复原始属性：

let attrs_to_preserve: Vec<_> = item_struct.attrs.iter()
    .filter(|attr| !attr.path().is_ident("my_attr"))
    .collect();

let result = quote! {
    #(#attrs_to_preserve)*
    #item_struct
    // ... 生成的代码 ...
};
```

### 反模式 5：TT muncher 无限递归

**症状**：
- 编译永远不结束（挂起）或者报 "recursion limit reached"
- 或者：宏展开到某个深度后编译器放弃

**根因**：
TT muncher 的递归规则没有正确消费 token。最常见的原因是：所有递归分支都匹配了当前输入，但没有任何一个真正消费了 token，导致每次递归调用都传入相同的输入。

**诊断方法**：
1. `cargo expand` 看展开是否正常终止
2. `trace_macros!(true)` 看是否在某个点无限循环
3. 检查每个递归分支，确保它至少消费了一个 token

**修复方案 1：添加显式的递归计数保护**

```rust
macro_rules! tt_muncher_safe {
    // 终止条件 1：没有 token 了
    (@done $($output:tt)*) => {
        $($output)*
    };
    // 终止条件 2：递归深度达到限制
    (@limit $count:expr, $($rest:tt)*) => {
        compile_error!(concat!(
            "tt_muncher recursion depth ",
            stringify!($count),
            " exceeded"
        ))
    };
    // 入口：从 0 开始计数
    ($($input:tt)*) => {
        tt_muncher_safe!(@recurse 0, $($input)*)
    };
    // 递归处理
    (@recurse $count:expr, let $name:ident = $value:expr; $($rest:tt)*) => {
        // 处理 let 语句
        // 递归调用，count + 1
        tt_muncher_safe!(@recurse ($count + 1), $($rest)*)
    };
    // 检查深度限制
    (@recurse $count:expr, $($rest:tt)*) => {
        {
            // 如果 $count > 1000，触发 @limit 分支
            // （这需要编译时计算能力，通常在 nightly 上更好用）
            tt_muncher_safe!(@check_limit ($count < 1000), $count, $($rest)*)
        }
    };
}
```

**修复方案 2：确保每个分支都消费了 token**

```rust
macro_rules! tt_muncher {
    // 基础情况：空输入
    () => {};

    // 这个分支消费了 "let ... = ...; " 然后是剩余的
    (let $name:ident = $value:expr; $($rest:tt)*) => {
        // 处理...
        tt_muncher!($($rest)*)          // ← 剩余部分变小了
    };

    // 保底分支：吞掉一个 token（关键——必须消费至少一个 token）
    ($head:tt $($rest:tt)*) => {
        // 忽略这个 token
        tt_muncher!($($rest)*)          // ← 剩余部分变小了
    };

    // 注意：如果这里写的是 tt_muncher!($head $($rest)*)，会导致无限递归
    // 因为 $head 没有被消费
}
```

**调试技巧**：在 TT muncher 的每个递归分支开头插入 `compile_error!` 来追踪匹配了哪个分支：

```rust
macro_rules! tt_muncher_debug {
    () => {};
    (let $name:ident = $value:expr; $($rest:tt)*) => {
        // compile_error!(concat!("DEBUG: matched 'let' branch, name=", stringify!($name)));
        tt_muncher_debug!($($rest)*)
    };
    ($head:tt $($rest:tt)*) => {
        // compile_error!(concat!("DEBUG: matched 'skip' branch, token=", stringify!($head)));
        tt_muncher_debug!($($rest)*)
    };
}
```

### 反模式 6：Hygiene 变量名冲突

**症状**：
- 宏看起来访问了正确的变量，但运行时值不对
- 或者在作用域内明明有变量，但编译报 "cannot find value"
- 症状在不同调用上下文中不一致

**根因**：
Rust 的声明宏 hygiene 系统是**部分**的：
- `let` 绑定的变量名：hygienic（不会污染调用方的作用域）
- 函数调用、类型引用、模块路径：non-hygienic（在调用方的上下文解析）
- 标识符到底以哪种方式解释，取决于它在宏定义中的**位置**和**角色**

这导致了微妙的 bug——某些看起来应该属于宏的变量泄漏到了调用方，或反之。

**诊断方法**：
`cargo expand` 查看展开后代码中的变量名，特别注意是否有被重命名的标识符。

**修复方案 1：在宏内部使用唯一的内部变量名**

```rust
// 错误：宏内部的 temp 可能与用户变量冲突
// （虽然 Rust 会做 hygiene 处理，但某些边界情况下可能出错）
macro_rules! swap {
    ($a:ident, $b:ident) => {
        let temp = $a;
        $a = $b;
        $b = temp;
    };
}

// 修复：使用极不可能冲突的名称
macro_rules! swap {
    ($a:ident, $b:ident) => {
        let __macro_swap_temp = $a;
        $a = $b;
        $b = __macro_swap_temp;
    };
}

// 更现代的修复：使用 paste crate 生成真正唯一的标识符
// （如果你需要宏引入多个内部变量）
use paste::paste;

macro_rules! with_internals {
    ($($body:tt)*) => {
        paste! {
            let [<__internal _ a>] = 1;
            let [<__internal _ b>] = 2;
            $($body)*
        }
    };
}
```

**修复方案 2：使用 $crate 确保路径解析正确**

参考反模式 1。

**修复方案 3：了解什么时候 hygiene 会"失效"**

```rust
// 场景：macro_rules! 中的函数调用
macro_rules! call_my_fn {
    () => {
        my_fn()  // 在调用方上下文解析 my_fn
    };
}

// 如果调用方没有 import my_fn，会报错。
// 修复：使用完全限定路径
macro_rules! call_my_fn {
    () => {
        $crate::my_fn()  // 始终在定义方的 crate 中解析
    };
}
```

### 反模式 7：proc_macro crate 依赖非 proc_macro 库

**症状**：
- 在 proc-macro crate 中 `cargo build` 失败
- 错误信息：某个依赖无法解析或不可用
- 或者：编译成功，但在其他 crate 使用你的宏时报错

**根因**：
Proc-macro crate 有严格的依赖限制。它们只能链接到 `proc_macro` crate（编译器内置）和 `proc_macro2`、`syn`、`quote` 等少量辅助 crate。如果你的 proc-macro crate 直接依赖了 `serde`、`tokio`、`reqwest` 等库，就会出现问题。

**诊断方法**：
检查你的 proc-macro crate 的 `Cargo.toml` 中的 `[dependencies]`，确认每个依赖是否是"proc-macro 安全的"。

**修复方案**：使用辅助 crate 模式

```text
my_proc_macro/
├── Cargo.toml               # workspace
├── my-proc-macro/           # proc-macro crate（最小依赖）
│   ├── Cargo.toml           # proc-macro = true
│   │   [dependencies]
│   │   proc-macro2 = "1"
│   │   syn = { version = "2", features = ["full"] }
│   │   quote = "1"
│   │   my-proc-macro-core = { path = "../my-proc-macro-core" }
│   └── src/lib.rs           # thin wrapper：30-50 行
│       use proc_macro::TokenStream;
│       use my_proc_macro_core::my_derive_impl;
│       #[proc_macro_derive(MyDerive)]
│       pub fn my_derive(input: TokenStream) -> TokenStream {
│           my_derive_impl(input.into())
│               .unwrap_or_else(|e| e.to_compile_error().into())
│               .into()
│       }
└── my-proc-macro-core/       # 核心逻辑（普通 crate）
    ├── Cargo.toml
    │   [dependencies]
    │   proc-macro2 = "1"
    │   syn = "2"
    │   quote = "1"
    │   serde = "1"          # ← 安全地放在这里
    │   any_other_lib = "x"
    └── src/lib.rs           # 所有逻辑在这里，可以正常测试
```

**关键设计原则**：

> "proc-macro crate 应该是一个'薄壳'——只负责从编译器接收 TokenStream 并返回 TokenStream，真正的逻辑全部在另一个普通的 Rust crate 中。这不仅是绕过依赖限制的方式，也是让代码可测试、可调试的正确架构。"

### 反模式 8：Span 信息丢失

**症状**：
- 你的宏产生的错误信息都指向同一个位置（如宏调用处）
- 错误信息说的是："there's a problem somewhere in this vast struct / trait impl / function"
- 用户必须猜到底是哪个字段或哪个参数有问题

**根因**：
你创建了新的 Span（通常是 `Span::call_site()`），丢弃了输入 token 中的原始 span 信息。

**诊断方法**：
检查 proc macro 代码中是否有 `Span::call_site()` 或 `quote!` 中使用了新的标识符而没有使用 `quote_spanned!`。

**修复方案**：传播 Span

```rust
use quote::{quote, quote_spanned};
use proc_macro2::Span;

// 坏做法 —— 所有内容都指向宏调用处
fn bad_derive(input: &syn::DeriveInput) -> syn::Result<proc_macro2::TokenStream> {
    let name = &input.ident;

    // 所有 Token 都隐式使用了 call_site span
    Ok(quote! {
        impl MyTrait for #name {
            fn my_method(&self) -> u32 {
                // 如果这里有错误，指向宏调用处 —— 没用
                42
            }
        }
    })
}

// 好做法 —— 传播 span
fn good_derive(input: &syn::DeriveInput) -> syn::Result<proc_macro2::TokenStream> {
    let name = &input.ident;
    let name_span = name.span();  // 保留 ident 的 span

    Ok(quote_spanned! { name_span =>
        // 在这个 quote 块中，编译错误会指向 name_span
        impl MyTrait for #name {
            fn my_method(&self) -> u32 {
                // 错误会指向 name.span()
                42
            }
        }
    })
}

// 最佳实践 —— 精确的 Span
fn best_derive(input: &syn::DeriveInput) -> syn::Result<proc_macro2::TokenStream> {
    let name = &input.ident;

    let data = match &input.data {
        syn::Data::Struct(s) => s,
        _ => {
            // 错误精确指向输入类型的开始位置
            return Err(syn::Error::new(
                input.span(),
                "MyDerive can only be derived for structs",
            ));
        }
    };

    // 处理字段，每个字段保留其 span
    let methods: Vec<_> = data.fields.iter().map(|field| {
        let field_name = field.ident.as_ref().unwrap();
        let field_span = field_name.span();

        quote_spanned! { field_span =>
            // 这里生成的代码如果有错误，会指向具体字段
            let _ = &self.#field_name;
        }
    }).collect();

    Ok(quote! {
        impl MyTrait for #name {
            fn my_method(&self) {
                #(#methods)*
            }
        }
    })
}
```

**Span 在错误信息中的视觉差异**：

```text
// 使用 call_site() span（坏）：
error[E0277]: the trait bound `T: Display` is not satisfied
 --> src/main.rs:3:10   ← 指向 #[derive(MyTrait)]，没有帮助
  |
3 | #[derive(MyTrait)]
  |          ^^^^^^^ the trait `Display` is not implemented for `T`

// 使用正确的 span（好）：
error[E0277]: the trait bound `T: Display` is not satisfied
 --> src/main.rs:4:5    ← 指向具体的字段
  |
4 |     value: T,        ← 非常清楚：这个字段需要 Display
  |     ^^^^^
```

### 反模式 9：quote! 插值类型错误

**症状**：
- 编译 proc macro 时报错：`the trait bound `Foo: quote::ToTokens` is not satisfied`
- 你想在 `quote!` 中插入某个值，但它没有实现 `ToTokens`

**根因**：
`quote!` 宏使用 `#var` 语法进行插值。变量 `var` 必须实现 `quote::ToTokens` trait。大多数常见类型（`Ident`、`Type`、`Expr`、`TokenStream`）都实现了 `ToTokens`，但某些中间类型（如 `syn::Field`、自定义 struct、`String`）没有。

**诊断方法**：
看编译器错误——它会准确告诉你哪个类型没有实现 `ToTokens`。

**修复方案**：

```rust
// 问题 1：String 不能直接插值
let name_string: String = "my_struct".to_string();
// let tokens = quote! { struct #name_string {} };  // 错误！String 没有 ToTokens

// 修复：转换为 Ident 后再插值
let name_ident = syn::Ident::new(&name_string, Span::call_site());
let tokens = quote! { struct #name_ident {} };

// 通用修复：将值转换为 TokenStream
let my_value: MyType = ...;
let value_tokens: proc_macro2::TokenStream = my_value.into();  // 如果有 impl
let tokens = quote! { fn foo() -> #value_tokens {} };

// 问题 2：无法直接插值 Vec<Something>
let items: Vec<syn::ItemFn> = ...;
// let tokens = quote! { #items };  // Vec<ItemFn> 没有 ToTokens

// 修复：使用 #(#var)* 重复插值语法
let tokens = quote! {
    #(#items)*  // 展开为每个 item 的 tokens
};

// 如果要带分隔符：
let tokens = quote! {
    #(#items),*  // 用逗号分隔
};

// 问题 3：插值 Option<Something>
let maybe_name: Option<syn::Ident> = ...;

// 修复：手动处理
let tokens = quote! {
    fn #maybe_name() {}  // 错误：Option<Ident> 没有 ToTokens
};

// 正确做法：
let tokens = if let Some(ref name) = maybe_name {
    quote! { fn #name() {} }
} else {
    quote! { fn anonymous() {} }
};

// 或使用 quote 的条件：
let tokens = quote! {
    #(fn #maybe_name() {})*  // 如果 maybe_name 是 None，生成空
};
```

**常见的 ToTokens 实现**：

| 类型 | 可直接插值？ | 如果不可以，怎么办 |
|------|------------|------------------|
| `Ident`, `Type`, `Expr`, `Pat`, `Lit` | 是 | — |
| `proc_macro2::TokenStream` | 是 | — |
| `proc_macro2::TokenTree` | 是 | — |
| `String` | 否 | `syn::Ident::new(&s, span)` 或 `syn::LitStr::new(&s, span)` |
| `&str` | 否 | 同上 |
| `Vec<T>` where T: ToTokens | 否 | 使用 `#(#vec)*` 重复语法 |
| `Option<T>` where T: ToTokens | 否 | 手动处理，或使用条件插值 |
| `bool` | 否 | 转换为 `syn::LitBool` |
| 数值类型 (`i32`, `u64`, ...) | 否 | 转换为 `syn::LitInt` |
| 自定义类型 | 否 | 实现 `quote::ToTokens` trait |

### 反模式 10：build.rs 路径错误

**症状**：
- cargo build 成功，但找不到生成的文件
- 或者：生成的代码总是旧的（上次构建的结果）
- 错误信息：`cannot find module generated` 或 `file not found`

**根因**：
- 写入到了错误的目录（当前目录而非 `OUT_DIR`）
- 或者：没有使用 `cargo:rerun-if-changed` 导致增量编译缓存了旧结果

**诊断方法**：
- 在 build.rs 中打印 `OUT_DIR` 的值：`eprintln!("OUT_DIR={}", env::var("OUT_DIR").unwrap());`
- 检查生成的文件的绝对路径是否正确
- 对比 include! 路径和实际写入路径是否一致

**修复方案**：

```rust
// ===== 错误的 build.rs =====
use std::env;
use std::fs;
use std::path::Path;

fn main() {
    // 错误 1：写入当前目录（不是 OUT_DIR）
    let out_path = Path::new("src/generated.rs");
    fs::write(out_path, "pub fn hello() { println!(\"hello\"); }").unwrap();
    // 这会写入到项目根目录下的 src/generated.rs
    // 可能与源代码管理冲突，且 cargo clean 不会清理它
}

// ===== 正确的 build.rs =====
use std::env;
use std::fs;
use std::path::Path;

fn main() {
    // 1. 使用 OUT_DIR
    let out_dir = env::var("OUT_DIR")
        .expect("OUT_DIR not set — are you running in a build script?");
    let dest_path = Path::new(&out_dir).join("generated.rs");

    // 2. 写入文件
    let generated_code = r#"
        pub fn hello() {
            println!("Hello from generated code!");
        }
    "#;
    fs::write(&dest_path, generated_code)
        .expect("Failed to write generated file");

    // 3. 告诉 cargo 何时重新运行 build.rs
    // 如果 build.rs 自己没变，且这些文件没变，就不要重新运行
    println!("cargo:rerun-if-changed=build.rs");

    // 如果你的 build.rs 读取了某些配置文件：
    println!("cargo:rerun-if-changed=config/proto_files.toml");
    println!("cargo:rerun-if-changed=../proto/");

    // 4. 在 lib.rs 或 main.rs 中引用：
    // include!(concat!(env!("OUT_DIR"), "/generated.rs"));
}

// ===== lib.rs 中使用 =====
// 用 include! 宏在编译时把生成的文件内容嵌入到当前文件
// 注意：include! 展开的是文件内容作为 Rust 代码，不是模块导入
include!(concat!(env!("OUT_DIR"), "/generated.rs"));

fn main() {
    hello(); // 来自生成的代码
}
```

**注意区分 `include!` 和 `mod` + `include!` 的区别**：

```rust
// 方式 1：include! 直接展开代码（好像代 码就写在这里）
include!(concat!(env!("OUT_DIR"), "/generated.rs"));
// 等价于在 build.rs 生成内容后，它变成了当前文件的一部分

// 方式 2：通过 mod 引用（如果生成的文件定义了模块）
// 使用 #[path] 属性指向生成的文件
```

## 完整诊断流程 —— 从问题到修复的五步法

"以下流程不是理论——它是从数百个真实世界的宏调试经验中提炼出来的。按顺序执行这五个步骤，你能解决 90% 以上的宏问题。"

### 步骤 1：宏观诊断 —— 先看到展开后的全貌

**目标**：回答"展开后的代码对吗？"

**操作**：
```bash
# 展开有问题的 item（不要展开整个 crate）
cargo expand path::to::problematic_item
```

**判断逻辑**：

```text
展开后的代码看起来是否正确？
│
├── 否 → 宏逻辑本身有 bug
│   ├── 需要修：宏的代码生成逻辑
│   └── 工具：eprintln! / 提取逻辑为测试
│
├──看起来对，但编译报错
│   ├── 是 "cannot find type/function/name" → 转步骤 2A（hygiene/路径）
│   ├── 是 "trait bound not satisfied" → 转步骤 2B（泛型处理）
│   └── 是 "expected type, found type" → 转步骤 2C（token 插值）
│
└── 展开对了、编译过了、运行不对
    ├── 逻辑在正确的展开代码中是否如预期？
    └── 检查宏的运行时行为（这种情况较少见，通常是类型不匹配）
```

### 步骤 2：编译错误定位 —— 错误信息指纹识别

根据错误信息的类型，快速定位可能的原因：

**指纹 A：`cannot find type/function/module/value`**

```text
可能原因：
├── 声明宏缺少 $crate 前缀（反模式 1）
├── Hygiene 导致名称解析到错误的作用域（反模式 6）
├── 宏引用了调用方没有的 import
└── 宏展开后的模块路径与实际不符

诊断步骤：
├── cargo expand → 找到报错的标识符 → 它当前的完全限定路径是什么？
├── 追溯回原始宏代码 → 这个标识符在宏中是如何生成的？
└── 添加 $crate:: 或 ::full::path:: 前缀
```

**指纹 B：`trait bound ... is not satisfied`**

```text
可能原因：
├── 缺少泛型参数 → impl 块没有 <T> 或 where T: Trait（反模式 3）
├── 泛型的 trait bound 太弱 → 生成的代码调用了需要某个 trait 的方法
└── 字段类型 T 需要额外的 bound 但没有生成

诊断步骤：
├── cargo expand → 查看 impl 块的泛型部分
├── 有没有 <T> / <'a> 等泛型参数？
├── where clause 是否包含了所有需要的 bound？
└── 字段中的类型变量 T 是否需要额外的 trait？
```

**指纹 C：`expected type, found type` / `mismatched types`**

```text
可能原因：
├── 宏生成的代码中有类型不匹配
├── quote! 插值时传入了错误类型的表达式
├── 宏生成的函数返回类型与 trait 定义不匹配
└── Span 问题导致错误的类型推断（罕见）

诊断步骤：
├── cargo expand → 找到报错的行 → 对比预期类型和实际类型
├── 回溯到宏代码 → 生成这个 token 的代码在哪里？
└── 检查 quote! 中的插值表达式
```

**指纹 D：`internal compiler error (ICE)`**

```text
可能原因：
└── proc macro 中 panic 了（反模式 2）

诊断步骤：
├── 检查所有 unwrap() / expect() / panic!() 调用
├── 在关键位置添加 eprintln! 定位 panic 点
├── 将 panic 替换为 syn::Error / to_compile_error()
└── 如果无法定位，使用板斧 2（提取为 binary）进行调试
```

### 步骤 3：追踪溯源 —— 介入宏的执行过程

**对于声明宏**：

**对于 Derive 宏**：

## 答案

### 练习 1 答案：修复有问题的 derive 宏

#### 步骤 1：识别三种反模式

**反模式 1：Proc macro 中 panic** —— 第 16-22 行

```rust
// 原始代码：
syn::Data::Enum(data) => {
    panic!("BuggyIntrospect only supports structs, not enums"); // ← 反模式 2
}
```

**症状**：当用户在一个 enum 上使用 `#[derive(BuggyIntrospect)]` 时，编译器会崩溃并报告 "internal compiler error" 而不是清晰的错误信息。

**反模式 2：不处理泛型** —— 第 35 行

```rust
// 原始代码：
impl BuggyIntrospect for #name {  // ← 缺少 <T> 或 impl_generics
```

**症状**：当用户使用 `#[derive(BuggyIntrospect)]` 在 `struct Wrapper<T> { value: T }` 上时，编译器报错 "cannot find type T in this scope" 或 impl 块的泛型约束不匹配。

**反模式 3：Span 信息丢失** —— 第 34 行

```rust
// 原始代码：
let expanded = quote! {  // ← 用的是 quote! 不是 quote_spanned!
```

**症状**：所有编译错误都指向 `BuggyIntrospect` 的调用处，而不是 struct 的字段或其他具体位置。用户不知道具体哪里出了问题。

#### 步骤 2 和 3：修复后的代码

```rust
// ===== 修复后的核心逻辑 crate =====
// 文件: buggy-proc-macro-core/src/lib.rs

use proc_macro2::{Span, TokenStream};
use quote::{quote, quote_spanned};
use syn::{parse2, spanned::Spanned, DeriveInput, Fields};

/// 核心实现函数 —— 返回 Result，可以被单元测试
pub fn buggy_introspect_impl(input: TokenStream) -> syn::Result<TokenStream> {
    let input: DeriveInput = parse2(input)?;
    let name = &input.ident;

    // ===== 修复 1：用 syn::Error 替代 panic =====
    // 错误信息指向具体的 enum 或 union 位置
    let fields = match &input.data {
        syn::Data::Struct(data) => &data.fields,
        syn::Data::Enum(data) => {
            return Err(syn::Error::new(
                data.enum_token.span,  // Span 指向 enum 关键字
                format!(
                    "BuggyIntrospect can only be derived for structs, \
                     but `{}` is an enum. Consider providing a manual \
                     implementation instead.",
                    name
                ),
            ));
        }
        syn::Data::Union(data) => {
            return Err(syn::Error::new(
                data.union_token.span,  // Span 指向 union 关键字
                format!(
                    "BuggyIntrospect can only be derived for structs, \
                     but `{}` is a union. Unions are not supported.",
                    name
                ),
            ));
        }
    };

    // ===== 修复 2：正确处理泛型 =====
    let generics = &input.generics;
    let (impl_generics, ty_generics, where_clause) = generics.split_for_impl();

    // 收集字段名，保留每个字段的 Span
    let field_names_with_spans: Vec<_> = fields.iter()
        .filter_map(|f| {
            f.ident.as_ref().map(|ident| {
                let name_str = ident.to_string();
                let span = ident.span();
                (name_str, span)
            })
        })
        .collect();

    let field_count = field_names_with_spans.len();

    // 为每个字段生成带正确 Span 的 field_info
    let field_info_items: Vec<_> = field_names_with_spans.iter()
        .map(|(name_str, span)| {
            let name_lit = syn::LitStr::new(name_str, *span);
            quote! { #name_lit }
        })
        .collect();

    // ===== 修复 3：使用 quote_spanned! 传播 Span =====
    // name 的 Span 用于 impl 块的整体位置
    let expanded = quote_spanned! { name.span() =>
        // impl 块包含完整的泛型参数
        impl #impl_generics BuggyIntrospect for #name #ty_generics
            #where_clause
        {
            fn introspect() -> String {
                format!(
                    "Struct `{}` has {} fields: [{}]",
                    stringify!(#name),
                    #field_count,
                    vec![#(#field_info_items),*].join(", "),
                )
            }
        }
    };

    Ok(expanded)
}
```

#### 步骤 4：cargo expand 展开结果分析

以下是 `cargo expand` 的核心输出（带注释）：

```rust
// ================================================================
// Serde Serialize 展开分析
// ================================================================

// 1. 生成的 impl 块 —— Serialize trait 的核心实现
#[automatically_derived]
impl serde::Serialize for Person {
    fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
    where
        S: serde::Serializer,
    {
        // 2. 调用 serializer 的 serialize_struct 方法
        //    参数 1: struct 名称 —— "Person"（编译时确定）
        //    参数 2: 字段数量 —— 3（编译时计数）
        let mut serde_state = serde::Serializer::serialize_struct(
            &serializer,
            "Person",
            // 字段数量（根据 skip_serializing_if 动态调整）
            false as usize + true as usize + {
                let mut __field_count = 1;
                // 如果 email 被跳过，字段数减 1
                if Option::is_none(&self.email) {
                    __field_count -= 1;
                }
                __field_count
            },
        )?;

        // 3. 序列化 name 字段（无条件的）
        serde::ser::SerializeStruct::serialize_field(
            &mut serde_state,
            "name",          // 字段名 —— 编译时字符串
            &self.name,      // 字段的引用
        )?;

        // 4. 序列化 age 字段（无条件的）
        serde::ser::SerializeStruct::serialize_field(
            &mut serde_state,
            "age",
            &self.age,
        )?;

        // 5. 序列化 email 字段（有条件的 —— skip_serializing_if）
        //    关键设计：只有在条件不满足时才序列化
        if !(Option::is_none(&self.email)) {
            serde::ser::SerializeStruct::serialize_field(
                &mut serde_state,
                "email",
                &self.email,
            )?;
        }

        // 6. 完成序列化
        serde::ser::SerializeStruct::end(serde_state)
    }
}
```

**问题回答**：

**Q: Serde 的 `Serialize` 展开后为 `Person` 实现了什么 trait？核心方法签名是什么？**

实现了 `serde::Serialize` trait。核心方法是 `fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error> where S: serde::Serializer`。这是一个泛型方法，其泛型参数 `S` 受 `serde::Serializer` trait 约束。这种设计允许一个 Serialize 实现适配多种输出格式（JSON、YAML、MessagePack 等）。

**Q: `serialize_struct` 的三个参数分别是什么？**

1. `&serializer`：序列化器的引用
2. `"Person"`：struct 的名称，作为字符串字面量（在序列化为 JSON 等格式时用作 key 或标识符）
3. `字段数量`：一个在编译时计算（对非条件字段是常量 `true as usize`/`false as usize`）和运行时计算（对 `skip_serializing_if` 字段动态判断）混合的表达式

**Q: `skip_serializing_if` 是如何体现在展开后的代码中的？**

通过一个 if 守卫包裹 `serialize_field` 调用：

```rust
if !(Option::is_none(&self.email)) {
    serde::ser::SerializeStruct::serialize_field(
        &mut serde_state, "email", &self.email
    )?;
}
```

并且字段计数也动态调整：

```rust
{
    let mut __field_count = 1;
    if Option::is_none(&self.email) {
        __field_count -= 1;
    }
    __field_count
}
```

注意：`serialize_struct` 的字段数参数必须与实际调用 `serialize_field` 的次数一致。如果字段数不对，序列化器可能会错误地生成输出。

**Q: 如果将 `Person` 改为 `Person<T>`，展开后的 impl 块有什么变化？**

展开后的 impl 块会变成：

```rust
impl<T> serde::Serialize for Person<T>
where
    // Serde 自动为 T 添加了 Serialize bound
    // 因为 Person 中的字段是 T 类型
    T: serde::Serialize,
{
    fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
    where
        S: serde::Serializer,
    {
        // ... struct body 类似，但字段引用 &self.age 的类型变为 &T
        // 而 T: Serialize 保证了它可以序列化
    }
}
```

Serde 会自动分析字段类型并为泛型参数添加对应的 trait bound。这是 Serde 相比于手工实现的最核心价值之一。

**Q: 展开代码中一处精巧的设计及原因**

最精巧的设计在于**字段计数的编译时/运行时混合计算**。对于无条件字段，Serde 使用 `true as usize` (=1) 和 `false as usize` (=0) 让编译器在优化阶段就计算好这些常量。对于有条件字段，使用运行时的 `if` 判断来动态调整。结果是：如果 struct 中没有任何 `skip_serializing_if` 字段，字段计数完全是编译时常量——零运行时开销。

#### 步骤 5：添加 `#[serde(rename)]` 后的差异

添加 `#[serde(rename = "full_name")]` 到 `name` 字段后运行 `cargo expand`：

```rust
// name 字段的 serialize_field 调用变为：
serde::ser::SerializeStruct::serialize_field(
    &mut serde_state,
    "full_name",    // ← 不再是 "name"，改为了 "full_name"
    &self.name,
)?;
```

**分析**：`rename` 属性的实现非常直接——它只改变传给 `serialize_field` 的字符串字面量。对 Serde 的数据模型来说，struct 内部的字段名和序列化输出中的 key 名是两个独立的概念。`rename` 就是把"映射表"的这个映射关系改变了。展开代码中没有任何运行时查找——字符串字面量是在编译时确定并直接嵌入的。这也是 Serde 零开销抽象的一个体现。

#### 测试设计哲学

```text
/*
 * ===== 测试设计哲学 =====
 *
 * 一个好的宏测试套件应该覆盖三个维度：
 *
 * 1. 维度一：正确路径（Pass 测试）
 *    - 基本用法：最简单的正确用例
 *    - 边界用法：泛型、复杂组合、极限值
 *    - 回归用例：以前修过的 bug 场景
 *
 * 2. 维度二：错误路径（Fail 测试）
 *    - 语法错误：应用到错误的语法类别（enum/union/fn）
 *    - 约束错误：缺少必需属性/参数
 *    - 语义错误：属性值无效/冲突
 *
 * 3. 维度三：用户体验（.stderr 文件）
 *    - Span 精确：错误指向问题的实际位置
 *    - 消息有帮助：告诉用户"出了什么问题"和"如何修复"
 *    - 一致性：相似的错误使用相似的措辞
 *
 * 设计原则：
 * - "错误信息是API的一部分" —— 每次修改错误信息都是一种 breaking change
 * - "测试即文档" —— 通过这些测试，新用户可以快速了解什么是允许的、什么是不允许的
 * - "Fail 测试驱动错误UX设计" —— 在写宏之前先写测试，强迫你思考错误场景
 */
```
