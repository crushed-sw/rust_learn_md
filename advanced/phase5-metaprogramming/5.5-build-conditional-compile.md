# 5.5 build.rs 与条件编译 —— 编译器启动前的代码生成

> **build.rs 是 Rust 的"编译器预处理器"——在 cargo 开始编译你的 crate 之前，先执行一段 Rust 代码来生成源码、配置编译选项、检测外部依赖。它与宏的关键区别在于：宏在编译过程中运行，操作的是 Rust AST；build.rs 在编译前运行，可以直接执行任意系统命令、读取文件系统、调用外部工具。而 cfg 和 feature flags 则让你在代码中做"条件编译"——某些代码只在特定平台、特定 feature 组合下才被编译。**

---

## 1. build.rs 生命周期与机制

### 1.1 build.rs 何时运行——理解编译时序

build.rs 是放在 crate 根目录下的一个特殊 Rust 源文件。当 cargo 构建你的 crate 时，它遵循一个精确的编译时序：

1. **解析依赖关系**：cargo 首先解析 `Cargo.toml`，下载并检查所有依赖
2. **编译 build-dependencies**：cargo 编译 `[build-dependencies]` 中声明的 crate（这些 crate 只在构建时使用）
3. **编译并运行 build.rs**：cargo 将 build.rs 编译为一个可执行文件并运行它
4. **读取 cargo 指令**：cargo 解析 build.rs 输出中所有以 `cargo:` 开头的行，作为构建指令
5. **编译主 crate**：cargo 使用 build.rs 输出的指令来编译你的 crate

关键理解：build.rs 是在 **你的 crate 编译之前** 运行的。这意味着 build.rs 不能依赖你的 crate 中的任何代码（除了它自身的 [build-dependencies]）。build.rs 的输出（生成的代码、设置的 cfg 标志、链接指令）会成为后续编译阶段的输入。

```rust
// 一个最小的 build.rs —— 什么都不做，仅仅存在
fn main() {
    // build.rs 必须有一个 main 函数
    // 如果没有 cargo: 指令输出，cargo 不做任何额外处理
}
```

### 1.2 cargo 协议——stdout 即 API

build.rs 和 cargo 之间的通信协议极其简单：build.rs 往 stdout 写入特定的行，cargo 读取并解析它们。任何以 `cargo:` 开头（注意冒号后面的部分）的行都被 cargo 解释为指令。

以下是所有 `cargo:` 指令的完整列表及其用途：

```rust
// build.rs —— cargo 指令速查手册
fn main() {
    // ===== 链接指令 =====

    // 告诉 rustc 链接到某个原生库（等价于 gcc 的 -l 标志）
    // cargo:rustc-link-lib=NAME
    // 静态链接：cargo:rustc-link-lib=static=foo
    // 动态链接：cargo:rustc-link-lib=dylib=foo
    // 框架链接（macOS）：cargo:rustc-link-lib=framework=CoreFoundation
    println!("cargo:rustc-link-lib=ssl");
    println!("cargo:rustc-link-lib=crypto");

    // 添加库搜索路径（等价于 gcc 的 -L 标志）
    // 可以是绝对路径或相对于 build.rs 所在目录的相对路径
    println!("cargo:rustc-link-search=native=/usr/local/lib");
    println!("cargo:rustc-link-search=native=./vendor/lib");

    // 设置搜索路径的类型：
    //   native —— 原生库搜索路径
    //   dependency —— 依赖 crate 搜索路径（很少用）
    //   framework —— macOS 框架搜索路径（仅 macOS）
    println!("cargo:rustc-link-search=framework=/Library/Frameworks");

    // ===== 编译配置指令 =====

    // 设置自定义 cfg 标志 —— 在代码中可以用 #[cfg(my_flag)] 检测
    println!("cargo:rustc-cfg=has_openssl");
    println!("cargo:rustc-cfg=target_vendor=\"myvendor\"");

    // 设置环境变量 —— 在代码中可以用 env!("MY_VAR") 或 option_env!("MY_VAR") 读取
    println!("cargo:rustc-env=GIT_HASH=abc123def");
    println!("cargo:rustc-env=BUILD_TIMESTAMP=2026-07-05T12:00:00Z");

    // ===== 增量编译控制 =====

    // 指定何时需要重新运行 build.rs（如果这些文件变化了）
    println!("cargo:rerun-if-changed=src/config.toml");
    println!("cargo:rerun-if-changed=proto/user.proto");

    // 指定何时需要重新运行 build.rs（如果这些环境变量变化了）
    println!("cargo:rerun-if-env-changed=OPENSSL_DIR");
    println!("cargo:rerun-if-env-changed=CC");

    // ===== 用户通知 =====

    // 向用户显示一条警告信息（不会阻止编译）
    println!("cargo:warning=OpenSSL 未找到，使用 Rustls 作为后备");
}
```

**关键陷阱**：cargo 只解析以 `cargo:` 开头的行。如果你不小心在前面加了空格或者用了 `eprintln!`（输出到 stderr），cargo 不会识别。同样，不要在 `println!` 的字符串开头使用除 `cargo:` 以外的其他前缀——那些行会被 cargo 忽略。

### 1.3 OUT_DIR —— 生成文件的归宿

build.rs 不应该向源代码目录写入任何文件。所有生成的文件都应该写入 `OUT_DIR` 环境变量指向的目录。这是一个 cargo 管理的临时目录，位于 `target/debug/build/<crate-name>-<hash>/out/`。

```rust
// build.rs —— 正确的文件生成方式
use std::env;
use std::fs;
use std::path::PathBuf;

fn main() {
    // 获取 OUT_DIR 环境变量 —— cargo 在运行 build.rs 时设置
    let out_dir = env::var("OUT_DIR").unwrap();
    let dest_path = PathBuf::from(&out_dir).join("generated.rs");

    // 将生成的代码写入 OUT_DIR，绝不写入 src/ 目录
    fs::write(
        &dest_path,
        "pub fn generated_function() -> &'static str { \"I was generated!\" }\n"
    ).unwrap();

    // 告诉 cargo 这个文件的变化需要触发 crate 重新编译
    // 注意：这里通常不需要 rerun-if-changed，因为文件总是会被重新生成
}
```

在库代码中，使用 `include!` 宏将生成的文件嵌入：

```rust
// src/lib.rs —— 引用 build.rs 生成的文件
// concat! + env! + include! 是引用生成文件的标准模式
include!(concat!(env!("OUT_DIR"), "/generated.rs"));
```

为什么不能直接写 `include!("/path/to/out_dir/generated.rs")` ？因为 OUT_DIR 的路径包含哈希值（由 cargo 计算），它在不同的机器、不同的 cargo 版本上都不同。`env!("OUT_DIR")` 是唯一可移植的方式。

### 1.4 rerun-if-changed —— 增量编译的关键

build.rs 最容易被忽视但影响最大的特性是 `rerun-if-changed` 指令。默认情况下，如果 Cargo.toml 中没有任何 `rerun-if-changed` 指令，cargo 会在以下情况重新运行 build.rs：

- crate 根目录下的任何文件发生变化
- Cargo.toml 中列出的任何依赖发生变化

这意味着：如果你修改了 `src/lib.rs` 中的一个函数注释，build.rs 会被重新运行——即使它的输出完全没有变化。对于慢速的 build.rs（比如 protobuf 代码生成、C 库编译），这严重拖慢了增量编译。

```rust
// build.rs —— 精确控制 rebuild 触发条件的最佳实践
fn main() {
    // 列举 build.rs 真正依赖的所有文件
    println!("cargo:rerun-if-changed=build.rs");
    // ↑ 这个很常用 —— build.rs 本身变了当然要重新运行

    println!("cargo:rerun-if-changed=proto/user.proto");
    println!("cargo:rerun-if-changed=proto/order.proto");
    // ↑ 只在这些 proto 文件变化时才重新生成代码

    println!("cargo:rerun-if-changed=src/config/default.toml");
    // ↑ build.rs 读取了一个配置文件，配置文件变了才重新运行

    println!("cargo:rerun-if-env-changed=OPENSSL_DIR");
    // ↑ 如果 OpenSSL 的安装路径变了，需要重新检测

    // ⚠️ 常见错误：忘记写 rerun-if-changed
    // 后果：每次都重新运行 build.rs，拖慢开发迭代速度
    // ⚠️ 常见错误：写了 rerun-if-changed 却没写 build.rs 本身
    // 后果：修改了 build.rs 逻辑后不会自动重新运行
    // ⚠️ 常见错误：通配符不被支持
    // ❌ println!("cargo:rerun-if-changed=proto/*.proto");  // 不工作！
    // ✅ 必须显式列出每个文件，或在 build.rs 中使用 glob 库动态生成列表
}
```

**一个重要细节**：一旦你在 build.rs 中使用了任何 `rerun-if-changed` 指令，cargo 的默认"扫描一切"行为就被关闭了。这意味着你必须精确列举所有需要触发 rebuild 的路径。如果你加了一个 `rerun-if-changed` 却忘了列举另一个重要的依赖文件，修改那个文件将不会触发 rebuild。

### 1.5 完整示例——从 git 生成版本字符串

下面是一个生产级别的 build.rs 示例，它从 git 仓库中提取提交哈希，并生成一个包含版本信息的 Rust 源文件。

```rust
// build.rs —— 从 git 生成版本信息
use std::env;
use std::fs;
use std::path::PathBuf;
use std::process::Command;

fn main() {
    // 获取 OUT_DIR
    let out_dir = env::var("OUT_DIR").unwrap();
    let dest_path = PathBuf::from(&out_dir).join("version_info.rs");

    // 尝试获取 git commit hash
    let git_hash = Command::new("git")
        .args(["rev-parse", "--short", "HEAD"])
        .output()
        .ok()
        .and_then(|output| {
            if output.status.success() {
                Some(String::from_utf8_lossy(&output.stdout).trim().to_string())
            } else {
                None
            }
        })
        .unwrap_or_else(|| "unknown".to_string());

    // 获取 git 分支名
    let git_branch = Command::new("git")
        .args(["rev-parse", "--abbrev-ref", "HEAD"])
        .output()
        .ok()
        .and_then(|output| {
            if output.status.success() {
                Some(String::from_utf8_lossy(&output.stdout).trim().to_string())
            } else {
                None
            }
        })
        .unwrap_or_else(|| "unknown".to_string());

    // 检查工作区是否干净（是否有未提交的修改）
    let is_dirty = Command::new("git")
        .args(["diff-index", "--quiet", "HEAD", "--"])
        .status()
        .map(|status| !status.success())
        .unwrap_or(true);

    // 获取构建时的 target triple
    let target = env::var("TARGET").unwrap_or_else(|_| "unknown".to_string());

    // 获取 profile（debug 或 release）
    let profile = env::var("PROFILE").unwrap_or_else(|_| "unknown".to_string());

    // 生成 Rust 源代码
    let dirty_suffix = if is_dirty { "-dirty" } else { "" };
    let output = format!(
        r#"
// 此文件由 build.rs 自动生成，请勿手动编辑
// Generated at build time by build.rs

/// Git commit hash（短格式），如果 git 不可用则为 "unknown"
pub const GIT_HASH: &str = "{hash}{dirty}";

/// Git 分支名，如果 git 不可用则为 "unknown"
pub const GIT_BRANCH: &str = "{branch}";

/// 构建目标三元组（target triple）
pub const TARGET_TRIPLE: &str = "{target}";

/// 构建 profile（"debug" 或 "release"）
pub const BUILD_PROFILE: &str = "{profile}";

/// 工作区是否包含未提交的修改
pub const IS_DIRTY: bool = {dirty_flag};
"#,
        hash = git_hash,
        dirty = dirty_suffix,
        branch = git_branch,
        target = target,
        profile = profile,
        dirty_flag = is_dirty,
    );

    // 写入 OUT_DIR（绝不写入 src/）
    fs::write(&dest_path, output).unwrap();

    // 增量编译控制：只在 build.rs 自身或 .git/HEAD 变化时重新运行
    println!("cargo:rerun-if-changed=build.rs");
    println!("cargo:rerun-if-changed=.git/HEAD");

    // 也通知 cargo 我们的 OUT_DIR 中写了一个文件
    // （虽然不是必须的，但这是好习惯）
}
```

在 lib.rs 中引用：

```rust
// src/lib.rs
include!(concat!(env!("OUT_DIR"), "/version_info.rs"));

pub fn version_string() -> String {
    let dirty = if IS_DIRTY { " (dirty)" } else { "" };
    format!(
        "{} [{}] ({}){}",
        env!("CARGO_PKG_VERSION"),
        GIT_HASH,
        BUILD_PROFILE,
        dirty,
    )
}
```

## 2. 代码生成模式

### 2.1 模式一——写入 Rust 源文件，使用 include! 嵌入

这是最常见的 build.rs 使用模式。build.rs 执行一些逻辑（读取文件、调用外部工具、计算值），然后将结果写入 OUT_DIR 中的 `.rs` 文件，最后主代码通过 `include!` 宏将其嵌入。

```rust
// 示例：生成 protobuf 绑定
// build.rs
fn main() {
    // 使用 prost-build crate 编译 .proto 文件
    prost_build::compile_protos(&["src/protos/user.proto"], &["src/protos/"])
        .expect("Failed to compile protobuf definitions");

    // prost_build 内部已经将生成的代码写入 OUT_DIR
    // 我们不需要手动做任何事情
    println!("cargo:rerun-if-changed=src/protos/user.proto");
}

// src/lib.rs
// prost 生成的模块通过标准的 include! 模式引入
// prost 实际生成的代码在 OUT_DIR/<package_name>.rs
include!(concat!(env!("OUT_DIR"), "/_.user.rs"));
```

### 2.2 模式二——生成配置常量

build.rs 可以在编译时计算常量值，然后生成包含这些常量的 Rust 代码。这对于嵌入编译时信息（时间戳、git 哈希、target 信息）非常有用。

```rust
// build.rs —— 生成编译时常量
use std::env;
use std::fs;
use std::path::PathBuf;
use std::time::{SystemTime, UNIX_EPOCH};

fn main() {
    let out_dir = env::var("OUT_DIR").unwrap();
    let dest_path = PathBuf::from(&out_dir).join("build_constants.rs");

    // 获取构建时的 UNIX 时间戳
    let build_timestamp = SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .unwrap_or_default()
        .as_secs();

    // 获取 Rust 编译器版本
    let rustc_version = env::var("RUSTC")
        .ok()
        .unwrap_or_else(|| "unknown".to_string());

    // 生成代码
    let code = format!(
        r#"
/// 构建时的 UNIX 时间戳（秒）
pub const BUILD_TIMESTAMP_SECS: u64 = {ts};

/// 此 crate 的 Cargo.toml 中声明的版本
pub const PACKAGE_VERSION: &str = "{ver}";

/// 此 crate 的 Cargo.toml 中声明的名称
pub const PACKAGE_NAME: &str = "{name}";
"#,
        ts = build_timestamp,
        ver = env::var("CARGO_PKG_VERSION").unwrap_or_default(),
        name = env::var("CARGO_PKG_NAME").unwrap_or_default(),
    );

    fs::write(&dest_path, code).unwrap();
    println!("cargo:rerun-if-changed=build.rs");
}
```

### 2.3 模式三——检测系统能力并设置 cfg 标志

build.rs 可以检测外部系统是否安装了特定的库或工具，然后通过 `rustc-cfg` 设置自定义 cfg 标志。主代码可以使用 `#[cfg(flag)]` 根据检测结果进行条件编译。

```rust
// build.rs —— 检测系统能力
use std::process::Command;

fn main() {
    // 检测 OpenSSL 是否可用
    let has_openssl = Command::new("pkg-config")
        .args(["--exists", "openssl"])
        .status()
        .map(|s| s.success())
        .unwrap_or(false);

    if has_openssl {
        println!("cargo:rustc-cfg=has_openssl");
        println!("cargo:warning=检测到 OpenSSL，启用原生 TLS 支持");
    } else {
        println!("cargo:warning=未检测到 OpenSSL，将使用 rustls 作为后备");
    }

    // 检测是否为 CI 环境
    if env::var("CI").is_ok() {
        println!("cargo:rustc-cfg=ci_build");
    }

    // 检测 CPU 是否支持特定指令集
    // 注意：这里仅是示例，实际应使用 is_x86_feature_detected! 而不是 build.rs
    // cfg(target_feature = "avx2") 由编译器自动设置
}

// src/lib.rs —— 根据检测结果选择实现
#[cfg(has_openssl)]
mod tls {
    // 使用 OpenSSL 的实现
    pub fn make_connection() { /* ... */ }
}

#[cfg(not(has_openssl))]
mod tls {
    // 使用 rustls 的实现
    pub fn make_connection() { /* ... */ }
}
```

**重要限制**：通过 `cargo:rustc-cfg` 设置的 cfg 标志**只对包含 build.rs 的那个 crate 有效**。如果你的 crate 是一个库，下游使用者不会看到这些 cfg 标志——这正是设计意图：cfg 标志是 crate 内部的实现细节，不应泄露给下游。

### 2.4 模式四——编译并链接 C/C++ 代码

build.rs 最常见的用途之一就是使用 `cc` crate 编译 C/C++ 源文件，并将生成的目标文件链接到 Rust 二进制中。

```rust
// build.rs —— 编译 C 代码并链接
fn main() {
    // 使用 cc crate 编译 C 源文件
    // 在 Cargo.toml 的 [build-dependencies] 中添加：cc = "1"
    cc::Build::new()
        .file("src/c/checksum.c")         // C 源文件
        .file("src/c/crc32.c")            // 可以有多个源文件
        .include("src/c/include")          // 头文件搜索路径
        .define("VERSION", Some("\"1.0\"")) // 预定义宏
        .flag("-O3")                       // 编译标志
        .warnings_into_errors(true)        // 把警告当错误
        .compile("myclib");                // 输出库名（libmyclib.a）

    // 增量编译：只在 C 源文件变化时重新编译
    println!("cargo:rerun-if-changed=src/c/checksum.c");
    println!("cargo:rerun-if-changed=src/c/crc32.c");
    println!("cargo:rerun-if-changed=src/c/include/checksum.h");
}

// src/lib.rs —— 声明外部 C 函数并使用
extern "C" {
    fn compute_checksum(data: *const u8, len: usize) -> u32;
    fn compute_crc32(data: *const u8, len: usize) -> u32;
}

pub fn checksum(data: &[u8]) -> u32 {
    unsafe { compute_checksum(data.as_ptr(), data.len()) }
}

pub fn crc32(data: &[u8]) -> u32 {
    unsafe { compute_crc32(data.as_ptr(), data.len()) }
}
```

### 2.5 完整示例——从数据文件生成 Rust 代码

下面的示例演示了一个完整的工作流：build.rs 读取一个 JSON 配置文件（config schema），生成对应的 Rust 结构体定义和反序列化代码，然后主代码通过 `include!` 引用。

```rust
// build.rs —— 从 JSON schema 生成 Rust 代码
use std::env;
use std::fs;
use std::path::PathBuf;

fn main() {
    // 读取配置 schema 文件
    let schema_path = "src/schema/app_config_schema.json";
    let schema_content = fs::read_to_string(schema_path)
        .expect("无法读取配置 schema 文件");

    // 解析 JSON schema（简化示例——实际应使用 serde_json）
    // 假设 schema 格式：{ "fields": [ { "name": "...", "type": "..." }, ... ] }
    let generated_code = generate_rust_struct(&schema_content);

    // 写入 OUT_DIR
    let out_dir = env::var("OUT_DIR").unwrap();
    let dest_path = PathBuf::from(&out_dir).join("generated_config.rs");
    fs::write(&dest_path, generated_code).unwrap();

    // 增量编译控制
    println!("cargo:rerun-if-changed={}", schema_path);
    println!("cargo:rerun-if-changed=build.rs");
}

fn generate_rust_struct(schema_json: &str) -> String {
    // 这是一个简化的代码生成器
    // 在实际项目中，你会使用 serde_json 解析 JSON
    let mut code = String::new();
    code.push_str("// 此文件由 build.rs 从 app_config_schema.json 自动生成\n");
    code.push_str("// 请勿手动编辑\n\n");
    code.push_str("use serde::{Deserialize, Serialize};\n\n");
    code.push_str("#[derive(Debug, Clone, Serialize, Deserialize)]\n");
    code.push_str("pub struct AppConfig {\n");

    // 假设我们从 schema 中解析出了以下字段
    code.push_str("    /// 服务器监听地址\n");
    code.push_str("    pub bind_address: String,\n");
    code.push_str("    /// 服务器监听端口\n");
    code.push_str("    pub port: u16,\n");
    code.push_str("    /// 数据库连接字符串\n");
    code.push_str("    pub database_url: String,\n");
    code.push_str("    /// 最大连接数\n");
    code.push_str("    pub max_connections: u32,\n");
    code.push_str("    /// 是否启用日志\n");
    code.push_str("    pub enable_logging: bool,\n");
    code.push_str("}\n\n");
    code.push_str("impl Default for AppConfig {\n");
    code.push_str("    fn default() -> Self {\n");
    code.push_str("        Self {\n");
    code.push_str("            bind_address: \"127.0.0.1\".to_string(),\n");
    code.push_str("            port: 8080,\n");
    code.push_str("            database_url: \"postgres://localhost/mydb\".to_string(),\n");
    code.push_str("            max_connections: 10,\n");
    code.push_str("            enable_logging: true,\n");
    code.push_str("        }\n");
    code.push_str("    }\n");
    code.push_str("}\n");

    code
}
```

## 3. include! / include_str! / include_bytes! —— 编译期文件嵌入

### 3.1 三个宏的完整对比

Rust 提供了三个编译期文件嵌入宏，它们都在**编译时**将文件内容嵌入到二进制中——用户运行时不需要文件系统中存在这些文件。

```rust
// 三个编译期文件嵌入宏的对比

// include! —— 将其他 Rust 源文件的内容作为代码插入当前位置
// 像 C 的 #include —— 文件内容被解析为 Rust 代码
include!("path/to/module.rs");
include!(concat!(env!("OUT_DIR"), "/generated.rs"));  // 最常见的用法

// include_str! —— 将文本文件的内容作为 &'static str 嵌入
// 文件内容不经过 Rust 解析，只是一个字符串
const README: &str = include_str!("../README.md");
const SQL: &str = include_str!("queries/users.sql");

// include_bytes! —— 将二进制文件的内容作为 &'static [u8; N] 嵌入
// 数组长度 N 在编译时确定
const LOGO: &[u8] = include_bytes!("assets/logo.png");
const FONT: &[u8] = include_bytes!("fonts/Roboto-Regular.ttf");
```

### 3.2 关键特性与限制

```rust
// ===== 特性 =====

// 1. 路径是相对于当前源文件所在的目录
// 如果你的文件是 src/lib.rs，那么 include_str!("data.txt") 查找的是 src/data.txt
const DATA: &str = include_str!("data.txt");

// 2. 结果是 'static 生命周期的 —— 在程序的整个生命周期内有效
// 不需要堆分配，不需要运行时 I/O
fn serve_logo() -> &'static [u8] {
    include_bytes!("assets/logo.png")
}

// 3. 编译时求值 —— 如果文件不存在，你会得到一个编译错误而不是运行时错误
// const MISSING: &str = include_str!("this_file_does_not_exist.txt");
// ↑ 编译错误：error: couldn't read ... : 系统找不到指定的文件。

// ===== 限制 =====

// 4. 不支持动态路径 —— 路径必须是字符串字面量或 concat!/env! 宏
// const path = "data.txt";
// const DATA: &str = include_str!(path);  // ❌ 编译错误
// 这是因为 include_str! 是编译时宏，路径必须在编译时已知

// 5. 文件大小会影响编译时间
// 嵌入一个 50MB 的文件会让编译器处理 50MB 的数据
// 对于大文件，考虑运行时加载或使用 include_bytes! + 压缩
const LARGE_DATA: &[u8] = include_bytes!("large_dataset.bin");
// 50MB 的嵌入可能导致编译内存溢出或极长的编译时间
```

### 3.3 何时使用编译期嵌入 vs 运行时读取

| 场景 | 编译期嵌入 | 运行时读取 |
|------|-----------|-----------|
| 小配置文件（< 1KB） | include_str! | fs::read_to_string |
| SQL 迁移文件 | include_str! | 运行时读取目录 |
| 图片/字体资源 | include_bytes! | 运行时加载 |
| 大型数据集（> 10MB） | 不要嵌入 | 流式读取 |
| 用户可编辑的配置 | 不要嵌入 | 运行时读取 |
| TLS 证书 | include_bytes! | 运行时加载 |
| HTML 模板 | include_str! | 模板引擎加载 |

**核心原则**：如果数据在你的控制之下、不会频繁修改、且大小合理，编译期嵌入是最佳选择——零运行时开销，零文件 I/O 错误。如果用户可能需要修改数据，或者数据很大，运行时读取更合适。

### 3.4 完整示例——将 SQL 迁移文件嵌入二进制

这是一个生产模式：把数据库迁移 SQL 文件编译到二进制中，运行时从内存执行迁移。

```rust
// src/migrations.rs —— 编译期内嵌 SQL 迁移脚本
//
// 目录结构：
// src/
//   lib.rs
//   migrations.rs
//   sql/
//     001_create_users.sql
//     002_create_posts.sql
//     003_add_indexes.sql

/// 一个迁移项：版本号 + SQL 语句
pub struct Migration {
    pub version: u32,
    pub name: &'static str,
    pub sql: &'static str,

/// 所有已知迁移的列表（编译时确定）
pub const MIGRATIONS: &[Migration] = &[
    Migration {

    },
    Migration {
        version: 2,
        name: "002_create_posts",
        sql: include_str!("sql/002_create_posts.sql"),
    },
    Migration {
        version: 3,
        name: "003_add_indexes",
        sql: include_str!("sql/003_add_indexes.sql"),
    },
];

/// 获取最新的迁移版本号
pub const LATEST_VERSION: u32 = 3;

// 使用示例：
// let conn = Database::connect("postgres://localhost/mydb")?;
// for migration in MIGRATIONS {
//     conn.execute(migration.sql)?;
// }

// 在一个更完整的实现中，你会：
// 1. 在数据库中维护一个迁移版本表
// 2. 只执行版本号大于当前数据库版本的迁移
// 3. 支持回滚（down migrations）
```

## 4. #[cfg] 与 cfg! —— 条件编译主线

### 4.1 两个 cfg 面孔——编译期移除 vs 编译期布尔

`cfg` 在 Rust 中以两种形式出现，它们的作用时机截然不同：

```rust
// #[cfg(condition)]  —— 条件编译属性，编译期决定代码是否存在
// cfg!(condition)     —— 编译期宏，展开为 true/false 布尔值

// ===== #[cfg] 示例 =====

// 这段代码只在 Linux 上被编译
#[cfg(target_os = "linux")]
fn get_process_id() -> u32 {
    unsafe { libc::getpid() }
}

// 这段代码只在 Windows 上被编译
#[cfg(target_os = "windows")]
fn get_process_id() -> u32 {
    unsafe { windows_sys::Win32::System::Threading::GetCurrentProcessId() }
}

// 关键差异：在 macOS 上编译时，Linux 版本和 Windows 版本的
// get_process_id 函数都【不存在】——编译器根本看不见它们。
// 你可以使用任何平台的独有 API，不存在跨平台兼容问题。

// ===== cfg! 示例 =====

// cfg! 宏返回 true/false，但它仍然是编译时求值的
const IS_LINUX: bool = cfg!(target_os = "linux");

// 编译器会优化掉永远不会执行的分支
fn print_platform_info() {
    if cfg!(target_os = "linux") {
        println!("Running on Linux");
    } else if cfg!(target_os = "windows") {
        println!("Running on Windows");
    } else {
        println!("Running on an unknown platform");
    }
}

// 在 Linux 上编译后，上面的函数等价于：
// fn print_platform_info() {
//     println!("Running on Linux");
// }
// 其他分支被 LLVM 死代码消除优化掉了

// 关键限制：cfg! 内的代码【仍然会被解析和部分类型检查】
// 以下代码无法通过编译：
// fn broken() {
//     if cfg!(target_os = "windows") {
//         WindowsOnlyType::new();  // ❌ 在 Linux 上编译时找不到这个类型
//     }
// }
// 必须用 #[cfg] 代替：
// #[cfg(target_os = "windows")]
// fn use_windows_type() { WindowsOnlyType::new(); }
```

**核心教训**：`#[cfg]` 移除代码，`cfg!()` 只移除分支但仍需代码可解析。当你需要引用平台特定的类型或函数时，必须用 `#[cfg]`。

### 4.2 常用 cfg 谓词速查表

```rust

#[cfg(target_os = "linux")]       // Linux（包括 Android）
#[cfg(target_os = "macos")]       // macOS
#[cfg(target_os = "windows")]     // Windows
#[cfg(target_os = "ios")]         // iOS
#[cfg(target_os = "android")]     // Android
#[cfg(target_os = "freebsd")]     // FreeBSD
#[cfg(target_os = "dragonfly")]   // DragonFly BSD
#[cfg(target_os = "netbsd")]      // NetBSD
#[cfg(target_os = "openbsd")]     // OpenBSD

// ===== 目标架构 =====
#[cfg(target_arch = "x86")]       // x86 (32-bit)
#[cfg(target_arch = "x86_64")]    // x86-64 (64-bit)
#[cfg(target_arch = "arm")]       // ARM (32-bit)
#[cfg(target_arch = "aarch64")]   // ARM64 / AArch64
#[cfg(target_arch = "riscv64")]   // RISC-V 64-bit
#[cfg(target_arch = "mips")]      // MIPS (32-bit)
#[cfg(target_arch = "powerpc")]   // PowerPC

// ===== 指针宽度 =====
#[cfg(target_pointer_width = "64")]  // 64-bit 平台
#[cfg(target_pointer_width = "32")]  // 32-bit 平台
#[cfg(target_pointer_width = "16")]  // 16-bit 平台（嵌入式）

// ===== 目标环境 =====

#[cfg(target_env = "musl")]       // musl libc 环境

#[cfg(target_vendor = "pc")]      // 通用 PC

#[cfg(windows)]                   // 所有 Windows 系列

// ===== CPU 特性（SIMD 等） =====
#[cfg(target_feature = "avx2")]   // AVX2 指令集

#[cfg(test)]                      // 运行测试时（cargo test）
#[cfg(not(test))]                 // 非测试模式

// ===== 自定义 cfg（通过 build.rs 设置） =====
#[cfg(has_openssl)]
#[cfg(target_vendor = "myvendor")]
```

### 4.3 cfg 组合逻辑——any / all / not

```rust
// cfg 的条件组合使用标准的逻辑运算

// cfg(any(...))  —— 逻辑或：满足任意一个条件即匹配
#[cfg(any(target_os = "linux", target_os = "macos"))]
fn unix_specific_function() {
    // 在 Linux 和 macOS 上都编译此函数
}

// cfg(all(...))  —— 逻辑与：所有条件都必须满足
#[cfg(all(unix, target_pointer_width = "64"))]
fn unix_64bit_specific() {
    // 只在 64-bit Unix 系统上编译
}

// cfg(not(...))  —— 逻辑非
#[cfg(not(windows))]
fn non_windows_function() {
    // 在所有非 Windows 系统上编译
}

// 嵌套组合
#[cfg(all(
    unix,
    not(target_os = "macos"),
    target_pointer_width = "64"
))]
fn linux_or_bsd_64bit() {
    // 64-bit Linux 或 64-bit BSD，但不包括 macOS
}

}

    use std::fs::File;
    use std::os::windows::fs::OpenOptionsExt;

    pub fn acquire_lock(file: &File) -> std::io::Result<()> {
        // 使用 LockFileEx
        // ...
        Ok(())
    }
```

### 4.4 cfg_attr —— 条件属性

`cfg_attr` 让你根据条件给一个项附加或不附加属性。它避免了重复声明。

```rust
// cfg_attr 的基本语法
// #[cfg_attr(condition, attribute_to_apply)]
// 如果 condition 为真，则应用 attribute_to_apply

// 示例 1：只在测试时派生 PartialEq
#[cfg_attr(test, derive(PartialEq))]
#[derive(Debug)]
struct MyStruct {
    value: i32,
}

// 示例 2：条件性地添加文档注释
#[cfg_attr(
    feature = "serde",
    doc = "当启用 `serde` feature 时，此类型支持序列化"
)]
struct MyType {
    // ...
}

// 示例 3：给 cfg_attr 传递多个属性
#[cfg_attr(
    all(unix, feature = "io-uring"),
    allow(dead_code),
    doc = "仅在 Linux+io-uring feature 下使用"
)]
mod io_uring_backend {
    // ...
}

// 示例 4：最常见的用法 —— 条件派生
// 不用 cfg_attr——需要手动重复结构体定义
// #[cfg(feature = "serde")]
// #[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
// #[cfg(not(feature = "serde"))]
// #[derive(Debug, Clone, PartialEq)]
// pub struct Config { ... }  // ← 重复了！

// 用 cfg_attr——定义一次
#[derive(Debug, Clone, PartialEq)]
#[cfg_attr(feature = "serde", derive(Serialize, Deserialize))]
pub struct Config {
    pub database_url: String,
    pub max_connections: u32,
}
```

### 4.5 完整示例——跨平台 getpid 实现

这是一个完整的跨平台函数实现，展示了 `#[cfg]` 的实际用法。

```rust
// src/process.rs —— 跨平台获取进程 ID

/// 获取当前进程的 PID
/// 使用条件编译为不同平台提供不同的实现
pub fn getpid() -> u32 {
    // cfg! 宏用于单一的运行时检查——编译器会优化掉死分支
    #[cfg(target_os = "linux")]
    {
        // SAFETY: getpid() 总是安全的系统调用
        unsafe { libc::getpid() }
    }

    #[cfg(target_os = "macos")]
    {
        unsafe { libc::getpid() }
    }

    #[cfg(target_os = "windows")]
    {
        unsafe {
            // 使用 Windows API
            windows_sys::Win32::System::Threading::GetCurrentProcessId()
        }
    }

    // 对于不支持的平台，提供一个编译时错误
    // 而不是运行时 panic —— 更早发现问题
    #[cfg(not(any(
        target_os = "linux",
        target_os = "macos",
        target_os = "windows"
    )))]
    {
        compile_error!("getpid 不支持当前平台");
    }
}

// 更好的模式 —— 每个平台独立模块
#[cfg(target_os = "linux")]
mod platform {
    pub fn getpid() -> u32 {
        unsafe { libc::getpid() }
    }
}

#[cfg(target_os = "macos")]
mod platform {
    pub fn getpid() -> u32 {
        unsafe { libc::getpid() }
    }
}

#[cfg(target_os = "windows")]
mod platform {
    pub fn getpid() -> u32 {
        unsafe {
            windows_sys::Win32::System::Threading::GetCurrentProcessId()
        }
    }
}

// 统一的公有 API
pub fn getpid_v2() -> u32 {
    platform::getpid()
}
```

## 5. Feature Flags 设计

### 5.1 Cargo.toml 中的 Feature 声明

Feature 是 Rust 条件编译的第二大支柱（与 `#[cfg]` 并列）。它们在 Cargo.toml 中声明，在代码中通过 `#[cfg(feature = "name")]` 使用。

```toml
# Cargo.toml —— Feature 定义的完整示例

[package]
name = "my-library"
version = "0.1.0"
edition = "2021"

# ===== Feature 定义 =====
[features]
# default feature —— cargo build 不指定 feature 时启用
default = ["std", "logging"]

# 一个简单的 feature —— 只是一个名称，没有额外含义
std = []

# 一个 feature 可以启用其他 feature
serde = ["dep:serde", "std"]

# 一个 feature 可以启用依赖的特定 feature
tls-rustls = ["dep:rustls"]
tls-native = ["dep:openssl"]

# 聚合 feature —— 一键启用一组相关功能
full = ["serde", "tls-rustls", "logging", "async"]

# ===== 可选依赖 =====
[dependencies]
# 仅当 "serde" feature 启用时才编译这个依赖
serde = { version = "1", features = ["derive"], optional = true }
rustls = { version = "0.21", optional = true }
openssl = { version = "0.10", optional = true }
tokio = { version = "1", optional = true }

# 非可选依赖不受 feature 影响
log = "0.4"
```

**`dep:serde` 语法**：`dep:` 前缀明确表示 "我引用的是一个可选依赖"。这在 feature 名称和依赖名称不同时特别重要。例如，如果你的 feature 名叫 `json-support` 但依赖项叫 `serde_json`，你可以写 `json-support = ["dep:serde_json"]`。

### 5.2 在代码中使用 Feature

```rust
// src/lib.rs —— 使用 feature flags 进行条件编译

// 条件编译模块
#[cfg(feature = "serde")]
pub mod serialization {
    use serde::{Serialize, Deserialize};

    pub fn to_json<T: Serialize>(value: &T) -> String {
        serde_json::to_string(value).unwrap()
    }

    pub fn from_json<T: for<'de> Deserialize<'de>>(json: &str) -> T {
        serde_json::from_str(json).unwrap()
    }
}

// 条件编译函数
#[cfg(feature = "logging")]
pub fn init_logger() {
    env_logger::init();
}

#[cfg(not(feature = "logging"))]
pub fn init_logger() {
    // logging 未启用时，这是一个空操作
}

// 更好的模式 —— 编译期禁用整个功能
#[cfg(feature = "async")]
pub mod async_api {
    pub async fn fetch_data(url: &str) -> Result<String, Error> {
        // 异步实现
    }
}

// 在结构体上使用 cfg_attr
#[derive(Debug, Clone)]
#[cfg_attr(feature = "serde", derive(Serialize, Deserialize))]
pub struct User {
    pub id: u64,
    pub name: String,
    #[cfg_attr(feature = "serde", serde(skip_serializing_if = "Option::is_none"))]
    pub email: Option<String>,
}
```

### 5.3 Feature 设计原则

Feature flags 的设计直接影响 crate 的可维护性和用户的体验。以下是经过社区验证的最佳实践：

```rust
// ===== 原则 1：Feature 必须是可加的（additive） =====
//
// 启用更多 feature 不应该改变已有 feature 的行为或破坏已有功能。
// 这是 Rust feature 系统最核心的设计约束。

// ✅ 好 —— feature 添加功能
#[cfg(feature = "json")]
pub fn to_json<T: Serialize>(value: &T) -> String { /* ... */ }

// ❌ 坏 —— feature 改变行为的语义
#[cfg(feature = "fast_mode")]
fn calculate() -> u64 { /* 快速但近似 */ }
#[cfg(not(feature = "fast_mode"))]
fn calculate() -> u64 { /* 慢速但精确 */ }
// 问题：下游 crate 无法同时依赖"快速版"和"精确版"

// ===== 原则 2：避免互斥 feature =====
//
// 两个不应同时启用的 feature 应该用 cfg flag 或编译期错误来表达

// ❌ 坏 —— 互斥 feature（用户可能同时启用）
// [features]
// backend-postgres = ["dep:postgres"]
// backend-mysql = ["dep:mysql"]
// backend-sqlite = ["dep:rusqlite"]

// ✅ 好 —— 编译期检测冲突
#[cfg(all(feature = "backend-postgres", feature = "backend-mysql"))]
compile_error!("backend-postgres 和 backend-mysql 不能同时启用。请选择一个。");

// ===== 原则 3：合理的 Feature 粒度 =====
//
// 一个 feature 应该代表一个有意义的用户可见功能单元。

// ❌ 太细 —— 用户会被 feature 数量淹没
// [features]
// use-ahash = []
// use-fxhash = []
// use-siphash = []

// ✅ 合适 —— 提供合理的默认值
// [features]
// default = ["std-hash"]
// std-hash = []
// fast-hash = ["dep:ahash"]  // 启用 ahash 作为替代

// ===== 原则 4：Feature 命名约定 =====
//
// - 使用 kebab-case 命名 feature
// - 与依赖同名的 feature 自动映射（不需要 dep: 前缀）
// - 使用描述性名称

// ✅ 好的命名
// [features]
// enable-tracing = ["dep:tracing"]
// vendored-openssl = ["openssl/vendored"]

// ❌ 不好的命名
// [features]
// f1 = []
// use_tokio = ["dep:tokio"]  // 应使用 kebab-case
```

### 5.4 cargo-hack —— 验证所有 Feature 组合

手动测试所有 feature 组合是不可行的（N 个 feature 有 2^N 种组合）。cargo-hack 是专门解决这个问题的工具。

```bash
# 安装 cargo-hack
cargo install cargo-hack

# 检查每个 feature 单独启用时是否编译通过
cargo hack check --each-feature

# 检查所有 feature 组合的 powerset（2^N 种组合）
cargo hack check --feature-powerset

# 排除已知冲突的组合
cargo hack check --feature-powerset \
  --group-features backend-postgres,backend-mysql,backend-sqlite

# 在 CI 中运行（跳过默认 feature，测试所有组合）
cargo hack test --feature-powerset --no-default-features
```

## 6. 互斥 Feature 与组合爆炸

### 6.1 问题的本质

如果你有 N 个独立的布尔 feature，理论上的组合数是 2^N。当其中某些 feature 互斥时，测试这么多组合既浪费计算资源，又无法覆盖所有实际使用场景。

```text
实际场景分析：
- 3 个后端 feature（postgres, mysql, sqlite）：2^3 = 8 种组合
- 但只有 3 种有意义（每次只能选一个后端）
- 如果加上 serde 和 tracing（可加的）：总共 3 × 2^2 = 12 种有效组合
- 如果不做任何处理需要测试 2^5 = 32 种组合
```

### 6.2 解决方案

```rust
// 方案 1：使用 cfg flag + compile_error! 防止无效组合

// Cargo.toml
// [features]
// backend-postgres = ["dep:postgres"]
// backend-mysql = ["dep:mysql"]
// backend-sqlite = ["dep:rusqlite"]

// src/lib.rs
// 在编译时强制互斥
#[cfg(all(feature = "backend-postgres", feature = "backend-mysql"))]
compile_error!("不能同时启用 postgres 和 mysql 后端");

#[cfg(all(feature = "backend-postgres", feature = "backend-sqlite"))]
compile_error!("不能同时启用 postgres 和 sqlite 后端");

#[cfg(all(feature = "backend-mysql", feature = "backend-sqlite"))]
compile_error!("不能同时启用 mysql 和 sqlite 后端");

#[cfg(not(any(
    feature = "backend-postgres",
    feature = "backend-mysql",
    feature = "backend-sqlite"
)))]
compile_error!("必须至少启用一个后端 feature");

// 方案 2：使用单个 cfg flag（推荐，但需要额外配置）
//
// 不在 Cargo.toml 中为后端创建 feature，而是让用户通过 cfg 选择：
// 在用户的 .cargo/config.toml 中：
// [build]
// rustflags = ["--cfg", "db_backend=\"postgres\""]

// 代码中：
#[cfg(db_backend = "postgres")]
mod backend_impl { /* ... */ }

#[cfg(db_backend = "mysql")]
mod backend_impl { /* ... */ }

#[cfg(db_backend = "sqlite")]
mod backend_impl { /* ... */ }
// 编译器天然保证只有一个 cfg 值生效

// 方案 3：使用 --group-features 和 cargo-hack（测试层面的解决方案）
//
// 不改变代码，只在 CI 层面减少测试组合
// .github/workflows/ci.yml
// - name: Check feature combinations
//   run: |
//     cargo hack check --feature-powerset \
//       --group-features backend-postgres,backend-mysql,backend-sqlite \
//       --no-default-features
//
// --group-features 将互斥的 feature 分到同一组
// cargo-hack 会为每组中的每个成员分别测试，而不是测试所有组合

// 方案 4：提供 "full" feature（用户体验层面的解决方案）
//
// Cargo.toml
// [features]
// full = ["serde", "tracing", "backend-postgres"]
// 用户只需要 cargo add mylib --features full
// 而不是 cargo add mylib --features serde,tracing,backend-postgres
```

## 答案

### 练习 1 答案

**问题诊断**：

1. **文件写入位置错误**：`fs::write("generated_timestamp.rs", code)` 将文件写入**当前工作目录**（crate 根目录），而不是 `OUT_DIR`。这污染了源代码目录，并且 `include!` 解析路径时会出现问题。

2. **`include!` 路径不匹配**：`include!("generated_timestamp.rs")` 查找的是相对于 `src/lib.rs` 所在目录的文件（即 `src/generated_timestamp.rs`）。但 build.rs 写入的是 crate 根目录的 `generated_timestamp.rs`。即使文件被正确放置，也不应该让生成的文件污染 `src/` 目录。

3. **缺少 rerun-if-changed**：build.rs 没有任何 `rerun-if-changed` 指令。这意味着 cargo 默认在**任何文件变化时**都重新运行 build.rs，严重拖慢增量编译。

**修复后代码**：

```rust
// build.rs —— 修复后的版本
use std::env;
use std::fs;
use std::path::PathBuf;
use std::time::{SystemTime, UNIX_EPOCH};

fn main() {
    // 修复 1：写入 OUT_DIR 而不是当前目录
    let out_dir = env::var("OUT_DIR").expect("OUT_DIR 未设置——build.rs 必须在 cargo 环境中运行");
    let dest_path = PathBuf::from(&out_dir).join("generated_timestamp.rs");

    let timestamp = SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .expect("系统时间异常：当前时间早于 UNIX 纪元")
        .as_secs();

    let code = format!(
        "// 此文件由 build.rs 自动生成，请勿手动编辑\n\
         pub const BUILD_TIMESTAMP: u64 = {};\n",
        timestamp
    );

    fs::write(&dest_path, code).expect("无法写入生成文件到 OUT_DIR");

    // 修复 3：正确的增量编译控制
    // build.rs 本身变化时需要重新运行
    println!("cargo:rerun-if-changed=build.rs");
```

```rust
/// 将 UNIX 时间戳转换为人类可读的 UTC 时间字符串
fn format_utc(unix_secs: u64) -> String {
    // 简化版：从常用的基准日期计算
    // 实际项目中请使用 chrono 或 time crate
    let secs_per_day: u64 = 86400;
    let secs_per_hour: u64 = 3600;

    // 计算从 1970-01-01 开始的天数
    let total_days = unix_secs / secs_per_day;
    let remaining = unix_secs % secs_per_day;

    let hours = remaining / secs_per_hour;
    let minutes = (remaining % secs_per_hour) / 60;
    let seconds = remaining % 60;

    // 计算年、月、日（简化算法，没有处理闰年细节）
    // 这是一个准确的实现——使用整数运算
    let mut days_remaining = total_days;
    let mut year: u64 = 1970;

    loop {
        let days_in_year = if is_leap_year(year) { 366 } else { 365 };
        if days_remaining < days_in_year {
            break;
        }
        days_remaining -= days_in_year;
        year += 1;
    }

    let days_in_months: [u64; 12] = [
        31, // Jan
        if is_leap_year(year) { 29 } else { 28 }, // Feb
        31, 30, 31, 30, 31, 31, 30, 31, 30, 31,
    ];

    let mut month: u64 = 1;
    for &dim in &days_in_months {
        if days_remaining < dim {
            break;
        }
        days_remaining -= dim;
        month += 1;
    }

    let day = days_remaining + 1;

    format!(
        "{:04}-{:02}-{:02} {:02}:{:02}:{:02}",
        year, month, day, hours, minutes, seconds
    )
}

fn is_leap_year(y: u64) -> bool {
    (y % 4 == 0 && y % 100 != 0) || (y % 400 == 0)
}
```

```rust
                Ok(PostgresDb { client })
            })
        }

            // 启用 WAL 模式以提升并发性能
            conn.execute_batch("PRAGMA journal_mode=WAL;")

            Ok(SqliteDb { conn })
        }

    impl Database for SqliteDb {
        type Connection = SqliteDb;

        fn connect(url: &str) -> Result<Self::Connection, DbError> {
            // SQLite URL 格式：sqlite:///path/to/db.sqlite
            // 或者简单使用文件路径
            let path = url
                .strip_prefix("sqlite://")
                .unwrap_or(url);
            SqliteDb::new(path)
        }

        fn execute(&self, query: &str) -> Result<u64, DbError> {
            self.conn
                .execute(query, [])
                .map(|n| n as u64)
                .map_err(|e| DbError::Query(e.to_string()))
        }

        fn query(&self, query: &str) -> Result<Vec<Row>, DbError> {
            let mut stmt = self
                .conn
                .prepare(query)
                .map_err(|e| DbError::Query(e.to_string()))?;

            let column_names: Vec<String> = stmt
                .column_names()
                .iter()
                .map(|s| s.to_string())
                .collect();

            let rows = stmt
                .query_map([], |row| {
                    let mut values = Vec::with_capacity(column_names.len());
                    for i in 0..column_names.len() {
                        let value = row.get::<_, rusqlite::types::Value>(i).unwrap_or(rusqlite::types::Value::Null);
                        values.push(match value {
                            rusqlite::types::Value::Null => Value::Null,
                            rusqlite::types::Value::Integer(i) => Value::Int(i),
                            rusqlite::types::Value::Real(f) => Value::Float(f),
                            rusqlite::types::Value::Text(s) => Value::Text(s),
                            rusqlite::types::Value::Blob(_) => Value::Text("[binary data]".to_string()),
                        });
                    }
                    Ok(Row {
                        columns: column_names.clone(),
                        values,
                    })
                })
                .map_err(|e| DbError::Query(e.to_string()))?;

            let mut result = Vec::new();
            for row in rows {
                result.push(row.map_err(|e| DbError::Conversion(e.to_string()))?);
            }
            Ok(result)
        }
```

**d) 为什么使用 compile_error! 而不是运行时 panic**

使用 `compile_error!` 检测无效的 feature 组合有四个关键优势：

1. **更早发现错误**：编译错误在开发者敲下 `cargo build` 的那一刻就暴露出来，而运行时 panic 可能要到生产环境执行到那条代码路径时才会触发。编译期错误 = 零运行时开销，零未覆盖路径。

2. **不可能被绕过**：运行时 panic 只有在代码被实际执行时才触发。如果无效的 feature 组合对应的代码路径只在特定条件下触发，测试可能漏掉它，最终在用户环境中崩溃。compile_error! 根本不让代码编译通过。

3. **panic 本身就是互斥 feature 设计中的一个逻辑矛盾**：如果两个 feature 互斥，那么同时启用它们时应该使用哪个？panic 意味着"程序不知道该怎么处理这种情况"——而编译期错误意味着"这种情况不应该发生，编译器阻止了它"。

4. **符合 Rust 的编译期安全哲学**：Rust 的整体设计理念是尽可能在编译期捕获错误。三个互斥的 feature 被同时启用是一个可以在编译期检测到的"类型错误"（feature 级别的类型错误），compile_error! 是自然的表达方式。
