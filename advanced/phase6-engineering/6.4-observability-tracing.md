# 6.4 可观测性 —— 让系统"透明化"

> 一个没有日志的服务就像一个没有仪表盘的飞机——你在飞，但不知道高度、速度、油量，也不知道引擎是不是着火了。`println!` 调试在本地开发时够用，但在生产环境中它是无用的：输出混在一堆容器日志里、没有结构化字段、无法按级别过滤、无法关联到具体的请求。Rust 的 `tracing` 生态解决了所有这些问题：结构化 key=value 日志、span 追踪跨函数边界、OpenTelemetry 分布式追踪、Prometheus metrics 导出——让你在任何时候都能回答："系统现在在做什么？某条请求走了哪些函数？每个函数花了多少时间？"

---

## 1. `tracing` 基础 —— 超越 `println!`

### 1.1 从 `println!` 到 `tracing`

```toml
[dependencies]
tracing = "0.1"
tracing-subscriber = "0.3"
```

```rust
use tracing::{info, warn, error, debug, trace, Level};

fn main() {
    // 必须先安装 subscriber，否则所有 tracing 宏都不会输出
    tracing_subscriber::fmt::init();

    // ━━━ tracing 宏替代 println! ━━━
    info!("server started on port 8080");

    let user_id = 42;
    let username = "alice";

    // ━━━ 结构化字段：不是字符串拼接 ━━━
    // ❌ println! —— 丢失了字段信息
    println!("user {} ({}) logged in", user_id, username);

    // ✅ tracing —— 保留了字段为结构化数据
    info!(user_id = user_id, username = %username, "user logged in");
    // 输出: 2024-01-01T10:30:00.000Z  INFO user logged in user_id=42 username=alice

    // ━━━ 字段简写：当变量名和字段名相同时 ━━━
    info!(user_id, %username, "user logged in"); // ← user_id 自动取变量值
}
```

### 1.2 五种日志级别

```rust
use tracing::{error, warn, info, debug, trace};

// 级别从高到低
error!("database connection lost: {}", e);  // 需要立即关注的故障
warn!("retry attempt {} of {}", n, max);    // 潜在问题，不紧急
info!("server started");                     // 正常业务流程的里程碑
debug!("parsed {} records", count);          // 调试时才有用的信息
trace!("buffer position moved to {}", pos);  // 极其详细的追踪

// 运行时配置级别
// RUST_LOG=info cargo run    → 只输出 info、warn、error
// RUST_LOG=debug cargo run   → 输出 debug 及以上
// RUST_LOG=my_crate=trace cargo run → 只看 my_crate 的 trace 日志
```

### 1.3 字段记录的三种模式

```rust
use tracing::info;

#[derive(Debug)]
struct User {
    id: u64,
    name: String,
}

fn field_patterns() {
    let user = User { id: 1, name: "alice".into() };
    let count = 42;

    // ━━━ % 显示值（Display trait）━━━
    info!(%count, "increment");          // count=42

    // ━━━ ? 调试值（Debug trait）━━━
    info!(?user, "user object");         // user=User { id: 1, name: "alice" }
    info!(user.id, %user.name);          // user.id=1 user.name=alice

    // ━━━ = 字面值（不出现在结构化字段中）━━━
    // 注意：= 后面的内容直接用 Display 输出
}
```

## 2. Span —— 跨函数边界的执行追踪

### 2.1 Span 是什么？

```
没有 Span 的日志：
  INFO connecting to database
  INFO querying user   ← 这些日志片段毫无关系
  INFO sending response

有 Span 的日志：
  INFO connecting to database            [request_id=abc123]
  INFO querying user                     [request_id=abc123]
  INFO sending response                  [request_id=abc123]
  ↑ 所有日志共享同一个 request_id，可以关联到一起
```

### 2.2 手动创建 Span

```rust
use tracing::{info, span, Level};

fn handle_request(request_id: u64) {
    // 创建 span
    let request_span = span!(Level::INFO, "request", request_id);
    // _enter 让当前上下文进入这个 span
    let _enter = request_span.enter();

    info!("handling request");

    // 在 span 内部的日志自动关联这个 span
    authenticate_user();
    process_data();

    // 离开作用域时 span 自动结束
}

fn authenticate_user() {
    // 这里的 info! 自动继承父 span（request）
    info!("authenticating user");  // ← 输出中包含 request_id
}

fn process_data() {
    info!("processing data");      // ← 同样包含 request_id
}
```

### 2.3 `#[instrument]` —— 自动 Span

```rust
use tracing::{info, instrument};

// #[instrument] 自动为函数创建同名 span，捕获所有参数
#[instrument]
fn process_order(order_id: u64, user_id: u64) {
    info!("checking inventory");
    check_inventory(order_id);
    info!("order processed successfully");
}

#[instrument]
fn check_inventory(order_id: u64) {
    // 嵌套 span 形成调用树
    info!("inventory checked");

    // ━━━ 自定义输出：skip 跳过敏感参数 ━━━
}

// ━━━ #[instrument] 的高级用法 ━━━
#[instrument(
    skip(connection_pool),          // ← 不记录 connection_pool（太重/不实现 Debug）
    fields(
        db_name = %connection_pool.name(), // ← 但记录 pool 的名字
        request_id = %uuid::Uuid::new_v4(),
    )
)]
async fn query_database(
    connection_pool: &Pool,
    query: &str,
) -> Result<Vec<Row>, DbError> {
    info!("executing query");
    // ...
}
```

### 2.4 Span 的父子关系形成调用树

```rust
#[instrument]
fn handle_request() {
    info!("start");
    query_database();
    render_response();
    info!("done");
}

#[instrument]
fn query_database() {
    info!("querying");
    // ...
}

#[instrument]
fn render_response() {
    info!("rendering");
    // ...
}

// 日志输出（用 tracing-subscriber 的 JSON formatter）：
// { "name": "handle_request", "event": "start" }
// { "name": "query_database",  "parent": "handle_request", "event": "querying" }
// { "name": "render_response", "parent": "handle_request", "event": "rendering" }
// { "name": "handle_request", "event": "done" }
```

## 3. `tracing-subscriber` —— 控制日志输出

### 3.1 基础配置

```rust
use tracing_subscriber::{fmt, EnvFilter};

fn main() {
    // ━━━ 方案 1：纯文本格式（适合本地开发）━━━
    tracing_subscriber::fmt()
        .with_env_filter(EnvFilter::from_default_env()) // RUST_LOG=info
        .with_target(false)        // 不显示模块路径
        .with_thread_ids(true)     // 显示线程 ID
        .with_line_number(true)    // 显示行号
        .init();

    // ━━━ 方案 2：JSON 格式（适合生产环境 / ELK / Loki）━━━
    tracing_subscriber::fmt()
        .json()
        .with_current_span(true)   // 在事件中显示当前 span
        .init();
}
```

### 3.2 Layer 组合 —— 同时输出到终端和文件

```rust
use tracing_subscriber::{fmt, layer::SubscriberExt, util::SubscriberInitExt};

fn main() {
    // ━━━ 同时输出到 stdout 和文件 ━━━
    let stdout_layer = fmt::layer()
        .with_target(false)
        .pretty();  // 人类可读的格式

    let file = std::fs::File::create("app.log").unwrap();
    let file_layer = fmt::layer()
        .json()
        .with_writer(file);

    tracing_subscriber::registry()
        .with(stdout_layer)
        .with(file_layer)
        .init();
}
```

### 3.3 环境变量过滤

```rust
// RUST_LOG 的语法：
// RUST_LOG=info                  → 所有 crate 的 info 及以上
// RUST_LOG=my_app=debug          → 只看 my_app 的 debug+
// RUST_LOG=my_app=debug,other_crate=warn → 分别配置
// RUST_LOG=my_app::db=trace      → 只追踪 db 子模块
// RUST_LOG=error                  → 生产环境的默认配置

fn main() {
    tracing_subscriber::fmt()
        .with_env_filter(
            tracing_subscriber::EnvFilter::try_from_default_env()
                .unwrap_or_else(|_| "info".into())
        )
        .init();
}
```

## 4. OpenTelemetry 分布式追踪

### 4.1 为什么需要分布式追踪？

```
在微服务架构中，一个请求可能跨越 5 个服务：
  Nginx → API Gateway → Auth Service → User Service → Database

没有分布式追踪：
  每个服务的日志是孤立的——你无法把一个慢查询关联到它来自哪个 API 请求

有分布式追踪：
  同一个 trace_id 贯穿所有服务，你可以看到：
  - 请求在 Gateway 花了 2ms
  - 在 Auth Service 花了 50ms
  - 在 User Service 花了 200ms  ← 瓶颈！
  - 在 Database 花了 180ms
```

### 4.2 `tracing-opentelemetry`

```toml
[dependencies]
tracing-opentelemetry = "0.23"
opentelemetry = "0.22"
opentelemetry_sdk = "0.22"
opentelemetry-otlp = "0.15"
```

```rust
use opentelemetry::trace::TracerProvider as _;
use opentelemetry_sdk::trace::TracerProvider;
use opentelemetry_otlp::WithExportConfig;
use tracing_subscriber::layer::SubscriberExt;
use tracing_subscriber::util::SubscriberInitExt;
use tracing_opentelemetry::OpenTelemetryLayer;

fn init_tracing() {
    // 创建 OTLP exporter（发送到 Jaeger / Grafana / Datadog）
    let exporter = opentelemetry_otlp::new_exporter()
        .tonic()
        .with_endpoint("http://jaeger:4317");

    let provider = TracerProvider::builder()
        .with_batch_exporter(exporter, opentelemetry_sdk::runtime::Tokio)
        .build();

    let tracer = provider.tracer("my-service");

    // 创建 OpenTelemetry layer
    let telemetry_layer = OpenTelemetryLayer::new(tracer);

    tracing_subscriber::registry()
        .with(tracing_subscriber::fmt::layer())
        .with(telemetry_layer)
        .init();
}
```

## 5. Metrics —— 量化系统行为

### 5.1 `metrics` crate 生态

```toml
[dependencies]
metrics = "0.22"
metrics-exporter-prometheus = "0.15"
```

```rust
use metrics::{counter, gauge, histogram, describe_counter};

fn record_metrics() {
    // ━━━ Counter（计数器）—— 只增不减 ━━━
    counter!("http.requests.total", 1, "method" => "GET", "status" => "200");
    counter!("http.requests.total", 1, "method" => "POST", "status" => "500");

    // ━━━ Gauge（仪表）—— 可增可减 ━━━
    gauge!("active.connections", 42.0);
    gauge!("memory.usage.mb", 128.5);

    // ━━━ Histogram（直方图）—— 分布统计 ━━━
    histogram!("http.request.duration.ms", 12.5, "endpoint" => "/api/users");
    histogram!("http.request.duration.ms", 250.0, "endpoint" => "/api/users");
}
```

### 5.2 导出 Prometheus 指标

```rust
use axum::{Router, routing::get};
use metrics_exporter_prometheus::PrometheusBuilder;

#[tokio::main]
async fn main() {
    // 安装 Prometheus recorder
    let recorder = PrometheusBuilder::new()
        .install_recorder()
        .unwrap();

    // 暴露 /metrics 端点
    let app = Router::new()
        .route("/metrics", get(move || async move { recorder.render() }));

    let listener = tokio::net::TcpListener::bind("0.0.0.0:3001").await.unwrap();
    axum::serve(listener, app).await.unwrap();
}
```

### 5.3 关键指标设计

```
┌───────────────────┬─────────┬──────────────────────────────────────┐
│ 指标类别           │ 指标名  │ 为什么重要                           │
├───────────────────┼─────────┼──────────────────────────────────────┤
│ 吞吐量             │ QPS     │ 知道系统负载，容量规划               │
│ 延迟               │ P50/P99 │ P99 比平均值更重要——少数慢请求       │
│ 错误率             │ 5xx/4xx │ 发布后错误率变化是回滚的第一信号     │
│ 资源使用           │ CPU/Mem │ 内存泄漏或 CPU 异常是最早的告警信号  │
│ 业务指标           │ 注册/订单│ 运维和产品的共同语言                 │
└───────────────────┴─────────┴──────────────────────────────────────┘
```

## 7. 练习答案

```rust
use tracing::{info, warn, error, instrument};
use tracing_subscriber::EnvFilter;

fn main() {
    tracing_subscriber::fmt()
        .with_env_filter(EnvFilter::from_default_env())
        .init();

    info!("Starting data import...");
    match read_csv("data.csv") {
        Ok(records) => {
            info!(count = records.len(), "Read records");
            validate_records(&records);
            info!("Import completed successfully");
        }
        Err(e) => {
            error!(%e, "Import failed");
            std::process::exit(1);
        }
    }
}

#[instrument]
fn read_csv(path: &str) -> Result<Vec<String>, std::io::Error> {
    info!(%path, "reading CSV file");
    let content = std::fs::read_to_string(path)?;
    Ok(content.lines().map(String::from).collect())
}

#[instrument(skip(records))]
fn validate_records(records: &[String]) {
    let invalid = records.iter().filter(|r| r.is_empty()).count();
    if invalid > 0 {
        warn!(invalid, "found invalid records");
    }
}
```
