# 6.13 API 设计原则与模式 —— 让你的 crate "想都不用想就能用对"

> 你的 crate 的公共 API 是你的产品。它不是"内部代码的窗口"——它是你和用户的契约。一个设计良好的 API 让用户"想都不用想就能用对"：函数签名自解释、错误类型可操作、泛型约束在合理的位置、Builder 模式让复杂构造变得清晰。一个设计糟糕的 API 则相反：用户需要翻源码才知道怎么用、错误信息是 `Box<dyn Error>` 根本没法匹配、泛型的传染性让简单的调用也要写三行 turbofish。本节总结 Rust API 设计的核心原则和常用模式——不是让你记住所有 Rust API Guidelines 的 200 条规则，而是建立一种直觉：**什么样的 API 是"地道"的？什么样的设计是可演化的？三个月后的新同事能不能看懂？**

---

## 1. Rust API Guidelines 精要

### 1.1 命名约定

```rust
// ━━━ 方法命名：用 Rust 社区的"通用动词" ━━━
impl MyType {
    pub fn new() -> Self { ... }              // 构造函数
    pub fn with_capacity(n: usize) -> Self { ... }  // 带容量的构造

    // 转换方法：前缀约定
    pub fn as_bytes(&self) -> &[u8] { ... }   // 低成本借用转换
    pub fn to_owned(&self) -> Owned { ... }   // 昂贵的有所有权的转换
    pub fn into_bytes(self) -> Vec<u8> { ... } // 消耗 self 的转换

    // 布尔判断
    pub fn is_empty(&self) -> bool { ... }    // 不带参数
    pub fn has_children(&self) -> bool { ... }

    // 迭代器：单复数形式
    pub fn iter(&self) -> Iter { ... }        // 不可变借用迭代
    pub fn iter_mut(&mut self) -> IterMut { ... } // 可变借用迭代
    // IntoIterator 由类型实现（不是方法）
}

// ━━━ 不要做的事 ━━━
impl MyType {
    // ❌ 别自创谓词——用社区约定的动词
    // pub fn check_if_empty(&self) -> bool { ... }  // 没有人这么写

    // ❌ 别把类型名放进方法名（已经是 impl MyType 了）
    // pub fn my_type_new() -> Self { ... }
}
```

### 1.2 泛型参数的位置

```rust
// ━━━ 泛型放在"有意义"的位置 ━━━
// 原则：泛型参数应该出现在"用户能推断出来"的地方

// ✅ 泛型在输入参数中（用户调用时可以推断）
fn parse<T: FromStr>(s: &str) -> Result<T, T::Err> {
    s.parse()
}
let x: i32 = parse("42")?;   // T 从返回类型推断
let y = parse::<f64>("3.14")?; // 或者用 turbofish

// ✅ 泛型在返回类型中
fn read_config<T: Deserialize>(path: &str) -> Result<T> { ... }
let cfg: MyConfig = read_config("config.toml")?;

// ❌ 泛型只在内部使用，不出现在签名中
// fn process<T: Display>(data: &Data) -> String {
//     // T 只在函数体内使用 → 不应该作为泛型参数
//     // 应该用 trait object 或具体类型
// }
```

### 1.3 错误类型的 Self 描述

```rust
// ✅ 好的错误类型让调用者能做出反应
pub enum ClientError {
    #[error("connection refused: {addr}")]
    ConnectionRefused { addr: String },  // → 调用者可以重试另一个地址

    #[error("request timeout")]
    Timeout,                            // → 调用者可以重试

    #[error("rate limited, retry after {retry_after:?}")]
    RateLimited { retry_after: Duration }, // → 调用者知道该等多久

    #[error("{0}")]
    Other(#[from] Box<dyn std::error::Error + Send + Sync>), // 兜底
}

// ❌ 不可操作的错误
// pub enum BadError {
//     #[error("error: {0}")]
//     Error(String),  // 调用者只能打印，做不了任何决策
// }
```

## 2. Builder 模式 —— 让复杂构造变得清晰

### 2.1 基础 Builder

```rust
#[derive(Debug)]
pub struct Server {
    host: String,
    port: u16,
    workers: usize,
    timeout: Duration,
    tls: Option<TlsConfig>,
}

pub struct ServerBuilder {
    host: String,
    port: u16,
    workers: usize,
    timeout: Duration,
    tls: Option<TlsConfig>,
}

impl ServerBuilder {
    pub fn new(host: impl Into<String>, port: u16) -> Self {
        Self {
            host: host.into(),
            port,
            workers: 4,                    // ← 合理的默认值
            timeout: Duration::from_secs(30),
            tls: None,
        }
    }

    pub fn workers(mut self, n: usize) -> Self {
        self.workers = n;
        self
    }

    pub fn timeout(mut self, d: Duration) -> Self {
        self.timeout = d;
        self
    }

    pub fn tls(mut self, config: TlsConfig) -> Self {
        self.tls = Some(config);
        self
    }

    pub fn build(self) -> Server {
        Server {
            host: self.host,
            port: self.port,
            workers: self.workers,
            timeout: self.timeout,
            tls: self.tls,
        }
    }
}

// 使用
let server = ServerBuilder::new("0.0.0.0", 8080)
    .workers(8)
    .timeout(Duration::from_secs(10))
    .tls(tls_config)
    .build();

// ✅ 清晰：每个可选参数都有语义化的方法名
// ✅ 默认值：用户只需要关心他们想改的参数
// ✅ 链式调用：IDE 自动补全直观
```

### 2.2 Typestate Builder —— 编译期保证正确性

```rust
// ━━━ Typestate Builder：用类型系统确保"必须先设 A 才能设 B" ━━━
// 场景：数据库连接 —— 必须先指定地址，再指定凭据，再连接

pub struct ConnectionBuilder<State> {
    url: Option<String>,
    credentials: Option<Credentials>,
    _state: std::marker::PhantomData<State>,
}

// 状态标记类型
pub struct NoUrl;
pub struct HasUrl;
pub struct Ready;

impl ConnectionBuilder<NoUrl> {
    pub fn new() -> Self {
        Self { url: None, credentials: None, _state: std::marker::PhantomData }
    }

    pub fn url(mut self, url: impl Into<String>) -> ConnectionBuilder<HasUrl> {
        self.url = Some(url.into());
        ConnectionBuilder { url: self.url, credentials: self.credentials, _state: std::marker::PhantomData }
    }
}

impl ConnectionBuilder<HasUrl> {
    pub fn credentials(mut self, creds: Credentials) -> ConnectionBuilder<Ready> {
        self.credentials = Some(creds);
        ConnectionBuilder { url: self.url, credentials: self.credentials, _state: std::marker::PhantomData }
    }
}

impl ConnectionBuilder<Ready> {
    pub fn connect(self) -> Result<Connection> {
        let url = self.url.unwrap();
        let creds = self.credentials.unwrap();
        Connection::new(&url, &creds)
    }
}

// 使用
let conn = ConnectionBuilder::new()
    .url("postgres://localhost/db")
    .credentials(my_creds)
    .connect()?;

// conn = ConnectionBuilder::new()
//     .connect()?;  // ← 编译错误！必须先 .url() 再 .connect()
```

## 3. Extension Trait —— 给外部类型加方法

```rust
// ━━━ Extension Trait 模式 ━━━
// 你无法为外部类型（如 Vec<T>）impl 你自己的 trait（孤儿规则）
// 但你可以定义新 trait 并为外部类型实现它

pub trait VecExt<T> {
    fn to_sorted(self) -> Vec<T>;
    fn median(&self) -> Option<&T>;
}

impl<T: Ord> VecExt<T> for Vec<T> {
    fn to_sorted(mut self) -> Vec<T> {
        self.sort();
        self
    }

    fn median(&self) -> Option<&T> {
        if self.is_empty() {
            return None;
        }
        let mut sorted: Vec<&T> = self.iter().collect();
        sorted.sort();
        Some(sorted[sorted.len() / 2])
    }
}

// 使用
use my_crate::VecExt;
let v = vec![3, 1, 4, 1, 5];
let sorted = v.clone().to_sorted();
let med = v.median();
```

## 4. RAII 守卫模式

```rust
// ━━━ RAII Guard：获取资源返回守卫，守卫 drop 时自动释放资源 ━━━

pub struct FileLock {
    path: PathBuf,
}

pub struct FileGuard {
    path: PathBuf,
}

impl FileLock {
    pub fn acquire(path: impl Into<PathBuf>) -> Result<FileGuard> {
        let path = path.into();
        // 创建 .lock 文件
        std::fs::write(&path, "")?;
        Ok(FileGuard { path })
    }
}

impl Drop for FileGuard {
    fn drop(&mut self) {
        // 自动删除 .lock 文件
        let _ = std::fs::remove_file(&self.path);
    }
}

// 使用
fn process_file() -> Result<()> {
    let _lock = FileLock::acquire("data.lock")?;  // 获取锁
    // ... 处理数据 ...
    Ok(())
    // _lock 离开作用域 → Drop → 自动释放锁
}  // ← 即使 panic 也会释放！
```

## 5. API 演化策略

### 5.1 `#[deprecated]` —— 温和地淘汰旧 API

```rust
impl MyType {
    /// 创建一个新的实例（已废弃，请使用 `new`）
    #[deprecated(since = "1.5.0", note = "use `new()` instead")]
    pub fn create() -> Self {
        Self::new()
    }

    pub fn new() -> Self {
        Self { }
    }
}
// 用户调用 create() 时会看到编译警告，但不是错误——给他们迁移时间
```

### 5.2 兼容性的铁律

```
✅ 这些在 MINOR 版本中是安全的：
  - 增加新的 pub 函数/方法
  - 增加新的 pub 类型/trait
  - 在 trait 中增加带默认实现的方法
  - 放宽泛型约束（impl<T> → impl<T: Debug>）

❌ 这些需要 MAJOR 版本：
  - 删除任何 pub API
  - 修改 pub 函数签名（参数类型、返回类型）
  - 在 trait 中增加不带默认实现的方法
  - 收紧泛型约束（impl<T: Debug> → impl<T>）
  - 修改 enum 变体（增加或删除变体）
```

## 6. 常见反模式

```rust
// ━━━ 反模式 1：过度泛型化 ━━━
// ❌ 为了"灵活性"把每个参数都泛型化
pub fn save<T: AsRef<Path>, U: Serialize>(path: T, data: &U) -> Result<()> { ... }
// 调用者被迫写 turbofish 或依赖类型推断——增加认知负担

// ✅ 只有在"调用者确实会用不同类型"时才泛型化
pub fn save(path: &Path, data: &impl Serialize) -> Result<()> { ... }

// ━━━ 反模式 2：暴露内部类型 ━━━
// ❌ 在公共 API 中返回第三方 crate 的类型
pub fn get_client() -> reqwest::Client { ... }
// 调用者被迫依赖 reqwest（即使你的库可能将来换 HTTP 客户端）

// ✅ 返回自己的类型，内部包装
pub struct HttpClient { inner: reqwest::Client }
pub fn get_client() -> HttpClient { ... }

// ━━━ 反模式 3：字符串类型的 API ━━━
// ❌ 魔法字符串
pub fn set_mode(&mut self, mode: &str) { ... }
// set_mode("fast")? set_mode("Fast")? set_mode("FAST")? 哪个是对的？

// ✅ enum
pub enum Mode { Fast, Safe, Balanced }
pub fn set_mode(&mut self, mode: Mode) { ... }
```
