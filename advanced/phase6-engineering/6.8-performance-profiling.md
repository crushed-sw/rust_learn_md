# 6.8 性能基准测试与 Profiling —— 先测量，再优化

> "过早优化是万恶之源"——Knuth 这句名言被引用了太多次，但它后面的半句话常常被忽略："但 97% 的情况下，我们也不应该放过那 3% 的关键机会。" 关键机会在哪里？你不测量就不知道。太多程序员凭直觉优化："这个 HashMap 肯定很慢，我换成 BTreeMap"，或者"我加了个 `#[inline]`，肯定更快了"——结果呢？可能 HashMap 更快，可能 BTreeMap 更快，可能没区别，可能更慢了。**没有测量数据的优化是信仰，不是工程。** 本节建立性能优化的第一原则：先建立可信的基准测试，然后用 profiling 工具定位真正的瓶颈，再用基准测试验证优化效果。这不是优化教学——这是教你如何不被自己骗到。

---

## 1. Criterion —— 统计意义上的基准测试

### 1.1 为什么不能用 `std::time::Instant`？

```rust
// ❌ 简易的"基准测试"——结果不可信
fn bad_bench() {
    let start = std::time::Instant::now();
    for _ in 0..1000 {
        my_function();
    }
    let duration = start.elapsed();
    println!("{} ns per iter", duration.as_nanos() / 1000);
}
```

**问题**：
- CPU 频率缩放：你的 CPU 可能正在降频省电，导致测量值偏高
- 缓存预热：第一次运行可能更慢（冷缓存），后续运行更快（热缓存）
- 编译器优化：如果 `my_function()` 的返回值没有被使用，编译器可能**完全消除**你的基准代码
- 统计噪声：单次测量受 OS 调度、中断等因素影响极大
- 趋势分析：你需要知道这次 commit 是不是把性能搞砸了

### 1.2 `criterion` 入门

```toml
[dev-dependencies]
criterion = { version = "0.5", features = ["html_reports"] }

[[bench]]
name = "my_benchmark"
harness = false  # ← 关键：禁用内置的 benchmark harness，用 criterion 的
```

```rust
// benches/my_benchmark.rs
use criterion::{black_box, criterion_group, criterion_main, Criterion, BenchmarkId};

// 被测试的函数
fn fibonacci(n: u64) -> u64 {
    match n {
        0 => 0,
        1 => 1,
        _ => fibonacci(n - 1) + fibonacci(n - 2),
    }
}

fn fibonacci_iterative(n: u64) -> u64 {
    let mut a = 0u64;
    let mut b = 1u64;
    for _ in 0..n {
        let tmp = a;
        a = b;
        b = tmp + b;
    }
    a
}

fn bench_fibonacci(c: &mut Criterion) {
    // ━━━ 基础基准 ━━━
    c.bench_function("fib 20", |b| b.iter(|| fibonacci(black_box(20))));
    //                                        ^^^^^^^^^^  black_box 阻止编译器优化

    // ━━━ 参数化基准（比较不同输入的差异）━━━
    let mut group = c.benchmark_group("Fibonacci Comparison");
    for n in [10, 20, 30].iter() {
        group.bench_with_input(
            BenchmarkId::new("recursive", n),
            n,
            |b, n| b.iter(|| fibonacci(black_box(*n)))
        );
        group.bench_with_input(
            BenchmarkId::new("iterative", n),
            n,
            |b, n| b.iter(|| fibonacci_iterative(black_box(*n)))
        );
    }
    group.finish();
}

criterion_group!(benches, bench_fibonacci);
criterion_main!(benches);
```

```bash
# 运行
cargo bench

# 输出（简化）：
# fib 20                 time:   [18.231 µs 18.341 µs 18.452 µs]
#                         change: [-0.5% +0.8% +1.2%] (p = 0.45 > 0.05)
#                         No change in performance detected.
```

**`black_box` 的作用**：
```rust
// ❌ 没有 black_box：编译器知道你从不使用返回值，直接消除整个函数！
let result = fibonacci(20);  // 编译器："这个值没人用，我删掉"

// ✅ 有 black_box：编译器不知道返回值是否被使用，必须计算
let result = fibonacci(black_box(20));  // 编译器："black_box 可能用这个值，我得留着"
```

### 1.3 Criterion 的核心参数

```rust
use criterion::Criterion;
use std::time::Duration;

fn configure_criterion() -> Criterion {
    Criterion::default()
        .warm_up_time(Duration::from_secs(3))    // 预热时间
        .measurement_time(Duration::from_secs(5)) // 测量时间
        .sample_size(100)                          // 采样数量
        .noise_threshold(0.02)                     // 噪声阈值（2%）
        .significance_level(0.05)                  // 显著性水平
}
```

### 1.4 基准测试的最佳实践

```
✅ 应该基准测试的：
  - 核心算法（排序、哈希、查找、序列化）
  - 热路径上的函数（被频繁调用的）
  - 不同实现的对比（Vec vs SmallVec，BTreeMap vs HashMap）

❌ 不应该基准测试的：
  - I/O 操作（网络请求、文件读写——延迟波动太大，无意义）
  - 一次性的初始化代码
  - 过小的函数（纳秒级测量噪声太大）

⚠️ 关键陷阱：
  1. 不要在笔记本电脑电池供电时跑基准测试（CPU 会降频）
  2. 关闭其他程序（浏览器、IDE、音乐播放器）
  3. 基准测试应该在一个"安静"的机器上跑
  4. 如果要在 CI 中跑，使用 GitHub Actions 的专用 runner（而非共享 runner）
```

### 1.5 Criterion 的 HTML 报告

```bash
cargo bench
# 生成 target/criterion/report/index.html

# 打开它：
# 可以看到：
#   - 每次基准的统计摘要（均值、标准差、中位数）
#   - 历次运行的变化趋势（性能是变好了还是变差了）
#   - 两个实现的对比图（A vs B）
```

## 2. `perf` —— Linux 性能分析瑞士军刀

### 2.1 `perf stat` —— 宏观性能计数器

```bash
# 安装（Linux）
# Ubuntu: sudo apt install linux-tools-generic

# 统计程序运行期间的 CPU 事件
perf stat target/release/my_binary

# 输出：
#      15,234.12 msec task-clock       # CPU 利用率
#         1,234      context-switches  # 上下文切换次数
#           123      cpu-migrations    # CPU 迁移
#        12,345      page-faults       # 缺页异常
#    45,678,901      cycles            # CPU 周期
#    12,345,678      instructions      # 执行的指令数
#     3,456,789      branches          # 分支指令
#       123,456      branch-misses     # 分支预测失败
#      0.45          IPC               # 每条指令的周期数（越高越好）
```

**关键指标解读**：
| 指标 | 含义 | 好/坏 |
|------|------|-------|
| IPC（instructions per cycle） | > 1.0 是好，< 0.5 是瓶颈信号 | 越高越好 |
| branch-misses / branches | 分支预测失败率 | < 1% 是好 |
| cache-misses / cache-references | 缓存命中率 | > 95% 是好 |
| context-switches | 过多意味着 I/O 等待或锁竞争 | 越少越好 |

### 2.2 `perf record` + `perf report` —— 找到热点

```bash
# 1. 记录性能数据（采样，对程序影响很小）
perf record --call-graph dwarf target/release/my_binary

# 2. 查看报告
perf report

# 输出（交互式 TUI）：
#  Overhead  Command    Shared Object    Symbol
#   45.23%   my_binary  my_binary        [.] json_parse_value
#   18.45%   my_binary  my_binary        [.] allocate_buffer
#   12.33%   my_binary  my_binary        [.] hash_lookup
#    5.21%   my_binary  my_binary        [.] string_compare
#   ...
# ← 45% 的时间花在 json_parse_value —— 这是你的优化目标！
```

### 2.3 火焰图 —— 可视化热点

```bash
# 需要安装 FlameGraph 工具
# git clone https://github.com/brendangregg/FlameGraph

# 生成火焰图
perf record -F 99 --call-graph dwarf target/release/my_binary
perf script | stackcollapse-perf.pl | flamegraph.pl > flamegraph.svg
```

**读火焰图**：
```
横轴：函数调用栈（按字母顺序排列）
纵轴：调用深度（从下往上是调用链）
宽度：CPU 时间占比（越宽 → 越热 → 越值得优化）

  ┌──────────────────────────────┐
  │        json_parse_value       │  ← 最宽 = 最大热点
  │  ┌──────────────────────┐    │
  │  │    allocate_buffer     │    │
  │  │  ┌──────┐ ┌────────┐ │    │
  │  │  │malloc│ │memcpy  │ │    │
  └──┴──┴──────┴─┴────────┴─┴────┘
```

## 3. 辅助 Profiling 工具

### 3.1 `cachegrind` —— 缓存行为分析

```bash
# valgrind --tool=cachegrind target/release/my_binary
# 输出：L1/L2/LLC 缓存的命中和未命中次数

# 关键指标：D1mr（L1 数据缓存未命中率）
# 如果某个函数的 D1mr 很高 → 可能数据布局不好
```

### 3.2 `heaptrack` —— 内存分配分析

```bash
# 安装：sudo apt install heaptrack
heaptrack target/release/my_binary

# 输出 GUI 报告：
# - 总共分配了多少内存
# - 分配次数最多的函数
# - 临时分配（分配后很快释放）的热点
# - 内存泄漏（如果有的话）
```

### 3.3 `cargo-flamegraph` —— Rust 友好的火焰图

```bash
cargo install flamegraph

# 一行命令生成火焰图
cargo flamegraph --bin my_binary -- --my-arg

# 输出 flamegraph.svg —— 直接用浏览器打开
# perf 权限：sudo sysctl kernel.perf_event_paranoid=-1（降低 perf 权限要求）
```

## 4. 性能分析工作流 —— 4 步法

### 4.1 完整的优化工作流

```
Step 1: 基准测试（Criterion）
  → 建立性能基线："当前版本处理 1000 条记录需要 2.3ms"
  → 回答："它现在多快？"

Step 2: Profiling（perf + flamegraph）
  → 找到热点："45% 时间花在 json_parse_value"
  → 回答："是什么让它慢？"

Step 3: 分析瓶颈
  → 是算法问题？（O(n²) vs O(n log n)）
  → 是内存问题？（分配太多？缓存未命中？）
  → 是 I/O 问题？（fsync 太频繁？）
  → 回答："为什么慢？"

Step 4: 优化 + 验证（Criterion 再次运行）
  → 实施优化
  → 跑相同的基准测试
  → 对比："2.3ms → 1.1ms，提升 52%"
  → 回答："优化有没有效果？"
```

### 4.2 实战案例：优化一个 JSON 解析器

```rust
// ━━━ Step 1: 基准测试 ━━━
use criterion::{black_box, Criterion};

fn parse_benchmark(c: &mut Criterion) {
    let json_data = r#"{"users": [{"name": "Alice", "age": 30}, ...]}"#;
    c.bench_function("parse large json", |b| {
        b.iter(|| my_json_parser::parse(black_box(json_data)))
    });
}
// 基线：parse large json: [2.34 ms 2.41 ms 2.48 ms]

// ━━━ Step 2: Profiling ━━━
// $ cargo flamegraph --example parse_bench
// → 火焰图显示：65% 时间在 string_unescape()

// ━━━ Step 3: 分析 ━━━
// string_unescape() 每次遇到 \uXXXX 都要分配新的 String
// → 分配次数过多，触发大量 malloc/free

// ━━━ Step 4: 优化 ━━━
// 优化方案：对于没有转义字符的常用情况，避免分配
pub fn string_unescape_optimized(input: &str) -> Cow<str> {
    if !input.contains('\\') {
        return Cow::Borrowed(input);  // ← 零分配路径！
    }
    Cow::Owned(unescape_inner(input))
}

// ━━━ Step 5: 验证 ━━━
// parse large json: [0.89 ms 0.91 ms 0.93 ms]
// 提升：2.41ms → 0.91ms = 62% 提升！
```

## 6. 练习答案

```rust
fn bench_contains(c: &mut Criterion) {
    let mut group = c.benchmark_group("Contains Comparison");
    for size in [10, 100, 1000, 10000].iter() {
        let vec: Vec<i32> = (0..*size).collect();
        let set: std::collections::HashSet<i32> = vec.iter().cloned().collect();
        let target = size / 2;

        group.bench_with_input(BenchmarkId::new("Vec::contains", size), size, |b, _| {
            b.iter(|| vec.contains(black_box(&target)))
        });
        group.bench_with_input(BenchmarkId::new("HashSet::contains", size), size, |b, _| {
            b.iter(|| set.contains(black_box(&target)))
        });
    }
    group.finish();
}

// 预期结果：
//   10 个元素：   Vec O(n) ≈ HashSet O(1)  (开销差异不明显)
//   10000 个元素：Vec 明显更慢（线性 vs 常数）
//   结论：小数据集用 Vec（缓存友好），大数据集用 HashSet（O(1) 查找）
```
