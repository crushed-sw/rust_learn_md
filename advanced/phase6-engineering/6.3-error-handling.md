# 6.3 错误处理架构 —— 从 `unwrap()` 到结构化错误

> `unwrap()` 是 Rust 代码里的定时炸弹。每写一个 `unwrap()`，你都在说："这里绝对不可能出错。" 但生产环境的数据永远比你想的更脏：文件不存在、网络超时、JSON 格式错误、数据库连接断开。当 `unwrap()` 在凌晨三点的生产环境 panic 时，你那行"绝对不可能出错"的注释只会让 on-call 的同事更愤怒。Rust 的类型系统给了你 `Result<T, E>`——一个极其强大的错误处理基础——但用好它需要设计：库的该用什么错误类型？应用的该怎么添加上下文？错误链怎么追溯？这节不是教你"怎么用 `?` 操作符"，而是教你"怎么设计一个让三个月后的同事能看懂（且能 debug）的错误体系"。

---

## 1. 库 vs 应用 —— 两种错误处理哲学

### 1.1 根本分歧

```
┌──────────────────┬──────────────────────────┬───────────────────────────┐
│                  │ 库（Library）             │ 应用（Application）        │
├──────────────────┼──────────────────────────┼───────────────────────────┤
│ 谁处理错误？     │ 调用者                    │ 应用自身                   │
│ 错误类型         │ 结构化、可匹配（enum）     │ 灵活、有上下文（Box<dyn>） │
│ 是否应该 panic？ │ 几乎从不                  │ 特定情况下可以             │
│ 错误信息         │ 给程序员看                │ 给程序员 + 运维看          │
│ 首选 crate       │ thiserror                 │ anyhow / eyre             │
└──────────────────┴──────────────────────────┴───────────────────────────┘
```

**一句话总结**：库返回结构化的错误让调用者能**分情况处理**；应用把错误层层传递并**添加上下文**，最后统一报告或恢复。

### 1.2 库的错误类型：`thiserror`

```toml
[dependencies]
thiserror = "1"
```

```rust
use thiserror::Error;

// ━━━ 好的库错误类型设计 ━━━
#[derive(Error, Debug)]
pub enum MyLibError {
    #[error("I/O error: {0}")]
    Io(#[from] std::io::Error),  // ← #[from] 自动生成 From impl

    #[error("JSON parse error at line {line}: {msg}")]
    JsonParse { line: usize, msg: String },

    #[error("invalid configuration: {0}")]
    Config(String),

    #[error("timeout after {duration:?}")]
    Timeout { duration: std::time::Duration },

    #[error("resource not found: {resource}")]
    NotFound { resource: String },
}

// pub fn do_something() -> Result<(), MyLibError> { ... }
```

**`thiserror` 做了什么**：
- `#[derive(Error)]` 自动实现 `std::error::Error` trait
- `#[error("...")]` 自动生成 `Display` impl
- `#[from]` 自动生成 `From<T> for MyLibError`（支持 `?` 自动转换）
- `#[source]` 标记错误链中的"源错误"

### 1.3 应用的错误处理：`anyhow`

```toml
[dependencies]
anyhow = "1"
```

```rust
use anyhow::{Context, Result, anyhow, bail};

// ━━━ anyhow 的核心 API ━━━

// 1. context() —— 为错误添加上下文
fn read_config(path: &str) -> anyhow::Result<String> {
    let content = std::fs::read_to_string(path)
        .with_context(|| format!("failed to read config from {}", path))?;
    //          ^^^^^^^^^^^^ 如果 read_to_string 出错，附加这条信息
    Ok(content)
}

// 2. anyhow!() —— 从零创建错误
fn validate_config(content: &str) -> anyhow::Result<()> {
    if content.is_empty() {
        return Err(anyhow!("config file is empty"));
    }
    Ok(())
}

// 3. bail!() —— 等价于 return Err(anyhow!(...))
fn check_port(port: u16) -> anyhow::Result<()> {
    if port < 1024 {
        bail!("port {} is privileged (must be >= 1024)", port);
    }
    Ok(())
}

// 4. 串联使用
fn load_and_validate(path: &str) -> anyhow::Result<String> {
    let content = read_config(path)?;          // ← 已经在 read_config 内部加了 context
    validate_config(&content)
        .context("config validation failed")?;  // ← 额外添加上下文
    Ok(content)
}

// ━━━ anyhow 的显示效果 ━━━
// 如果 config.txt 不存在：
//
// Error: failed to read config from config.txt
// Caused by:
//     No such file or directory (os error 2)
```

### 1.4 `thiserror` + `anyhow` 的分工

```rust
// ━━━ 典型的分层架构 ━━━

// lib.rs —— 库层：用 thiserror 定义结构化错误
#[derive(thiserror::Error, Debug)]
pub enum DbError {
    #[error("connection failed: {0}")]
    Connection(String),
    #[error("query failed: {0}")]
    Query(String),
}

pub fn query_user(id: u64) -> Result<User, DbError> {
    // ...
}

// main.rs —— 应用层：用 anyhow 添加调用上下文
use anyhow::Context;

fn handle_request(user_id: u64) -> anyhow::Result<User> {
    let user = query_user(user_id)
        .context("failed to query user in handle_request")?;
    Ok(user)
}

fn main() -> anyhow::Result<()> {
    let user = handle_request(42)?;
    println!("User: {}", user.name);
    Ok(())
}
```

## 2. 错误链与 `source()`

### 2.1 错误不是孤立的——它是一串因果链

```rust
// 场景：一个 HTTP 请求失败了，原因是什么？

// ❌ 只报告最上层，丢失根本原因
"request failed"

// ✅ 保留完整因果链
"request failed"
  → Caused by: failed to connect to database
    → Caused by: DNS resolution timeout for db.example.com
```

### 2.2 用 `thiserror` 构建错误链

```rust
use thiserror::Error;

#[derive(Error, Debug)]
enum AppError {
    #[error("request to {url} failed")]
    HttpRequest {
        url: String,
        #[source]  // ← 标记为"源错误"
        source: reqwest::Error,
    },

    #[error("database operation failed")]
    Database {
        #[source]
        source: DbError,
    },

    #[error("failed to parse response")]
    Parse {
        #[source]
        source: serde_json::Error,
        raw: String,
    },
}

// ━━━ 打印完整错误链 ━━━
fn print_error_chain(error: &dyn std::error::Error) {
    eprintln!("Error: {}", error);
    let mut source = error.source();
    while let Some(e) = source {
        eprintln!("  Caused by: {}", e);
        source = e.source();
    }
}
```

### 2.3 `anyhow` 的自动 `source()` 链

```rust
fn read_config() -> anyhow::Result<String> {
    std::fs::read_to_string("config.toml")
        .context("failed to read config")?;    // ← anyhow 自动建立 source 链
    // ...
}

fn main() {
    if let Err(e) = do_work() {
        // anyhow 的 Display 自动打印完整链：
        // Error: failed to read config
        // Caused by:
        //     0: failed to read config
        //     1: No such file or directory (os error 2)
        eprintln!("{:?}", e); // Debug 格式包含 Backtrace
    }
}
```

## 3. `eyre` —— `anyhow` 的增强版

### 3.1 `eyre` 比 `anyhow` 多了什么？

```toml
[dependencies]
eyre = "0.6"
```

```rust
use eyre::{Context, Result, eyre, WrapErr};

// ━━━ 1. 自动捕获 Backtrace（在 nightly 或设置了 RUST_BACKTRACE=1 时）━━━
fn might_fail() -> eyre::Result<()> {
    Err(eyre!("something went wrong"))
    // 错误自动附带 backtrace
}

// ━━━ 2. WrapErr trait —— 比 context() 更灵活的链式调用 ━━━
fn parse_port(s: &str) -> eyre::Result<u16> {
    s.parse::<u16>()
        .wrap_err("failed to parse port number")?; // ← 类似 context 但可链式
    // ...
}

// ━━━ 3. 自定义错误报告格式 ━━━
// 安装 color-eyre 以获得彩色输出
// fn main() {
//     color_eyre::install()?;
//     // 现在所有错误输出都有语法高亮
// }

// ━━━ 4. Section —— 为错误附加结构化信息 ━━━
fn connect_to_db(url: &str) -> eyre::Result<()> {
    Err(eyre!("connection failed"))
        .suggestion("try running `docker-compose up db`")
        .suggestion("check if the database port is correct")
        // .with_section(|| format!("url: {}", url).header("Details:"))
}
```

### 3.2 `anyhow` vs `eyre` 的选择

| 特性 | anyhow | eyre |
|------|--------|------|
| 基本错误上下文 | ✅ | ✅ |
| Backtrace | 手动 | 自动 |
| 建议/帮助信息 | ❌ | ✅ `suggestion()` |
| 结构化附加信息 | ❌ | ✅ `Section` |
| 生态成熟度 | 更高（tokio 也在用） | 中等 |
| 选择建议 | **默认选择** | 需要更好诊断时 |

## 4. 错误处理设计原则

### 4.1 原则一：error 变体应该是调用者可操作的

```rust
// ❌ 坏设计：调用者拿到错误后什么都做不了
#[derive(Error, Debug)]
pub enum BadError {
    #[error("something went wrong: {0}")]
    SomethingWrong(String), // ← 只能打印，无法匹配
}

// ✅ 好设计：调用者可以根据错误类型做出决策
#[derive(Error, Debug)]
pub enum GoodError {
    #[error("resource not found")]
    NotFound,                      // ← 可以重试或返回 404

    #[error("permission denied")]
    PermissionDenied,              // ← 可以提示用户登录

    #[error("rate limit exceeded, retry after {retry_after:?}")]
    RateLimited { retry_after: std::time::Duration }, // ← 可以等待后重试

    #[error("internal error: {0}")]
    Internal(#[from] Box<dyn std::error::Error + Send + Sync>), // ← 兜底
}
```

### 4.2 原则二：不要泄露实现细节

```rust
// ❌ 泄露了"我们用的是 PostgreSQL" 这个实现细节
#[derive(Error, Debug)]
pub enum BadApiError {
    #[error("PostgreSQL error: {0}")]
    PgError(sqlx::Error),  // ← 调用者不该关心数据库类型
}

// ✅ 用语义化的错误变体包装
#[derive(Error, Debug)]
pub enum GoodApiError {
    #[error("data access error: {0}")]
    DataAccess(#[source] Box<dyn std::error::Error + Send + Sync>),
    // 内部可能来自 PostgreSQL, MySQL, Redis, 文件系统——调用者不关心
}
```

### 4.3 原则三：错误信息写给"凌晨三点的自己"

```
好的错误信息：
  "failed to parse config file at /etc/myapp/config.toml: invalid TOML syntax at line 42, column 7"
  → 告诉了你：什么操作失败了、哪个文件、什么位置、什么原因

坏的错误信息：
  "parse error"
  → 你什么都不知道
```

## 5. `tracing-error` —— 让错误和 Span 关联

```rust
// 在 tracing span 中，错误会被自动关联到当前 span

use tracing::instrument;
use tracing_error::ErrorLayer;
use tracing_subscriber::prelude::*;

#[instrument]
fn process_order(order_id: u64) -> anyhow::Result<()> {
    let order = load_order(order_id)
        .context("failed to load order")?;  // ← 这个错误会关联到 process_order span

    validate_order(&order)
        .context("order validation failed")?;

    Ok(())
}

// 在日志中，错误和 span 的关系被保留
// {"span": {"name": "process_order", "order_id": 42},
//  "error": "failed to load order",
//  "error.source": "database connection timeout"}
```
