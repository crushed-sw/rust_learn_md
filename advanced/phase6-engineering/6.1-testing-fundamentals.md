# 6.1 测试体系基础 —— 从 `#[test]` 到测试金字塔

> 测试不是"写完代码后补一下"的附属品。测试是你对代码行为的**可执行规范**。它回答一个比你想象中更重要的问题：**"我凭什么相信这段代码是对的？"** 写测试的那一刻，你从"我觉得这个函数应该能工作"切换到"我有证据证明它在这些条件下正确工作"。Rust 的测试基础设施与语言特性深度集成——`#[test]`、`#[should_panic]`、文档测试、`#[cfg(test)]` 条件编译——这些不是附加功能，它们是 Rust 开发生态的一等公民。本节从最基础的测试写法开始，一路走到 Mock 策略和 Snapshot 测试，建立你的测试直觉。

## 1. 单元测试 —— 测试金字塔的基座

### 1.1 最基础的 `#[test]`

```rust
// src/lib.rs 或 src/main.rs（实践中推荐放在 lib.rs 中）

/// 计算两个数的最大公约数
pub fn gcd(mut a: u64, mut b: u64) -> u64 {
    while b != 0 {
        (a, b) = (b, a % b);
    }
    a
}

// ━━━ 单元测试模块 ━━━
#[cfg(test)]           // 只在 cargo test 时编译
mod tests {
    use super::*;      // 导入父模块的所有内容

    #[test]            // 标记为测试函数
    fn test_gcd() {
        assert_eq!(gcd(12, 8), 4);
        assert_eq!(gcd(7, 13), 1);
        assert_eq!(gcd(0, 5), 5);
    }

    #[test]
    fn test_gcd_same_number() {
        // 每个测试只验证一个行为（单一职责）
        assert_eq!(gcd(10, 10), 10);
    }
}
```

**关键设计决策**：为什么把测试写在同一个文件里？

Rust 社区普遍使用 `#[cfg(test)] mod tests { ... }` 模式，把测试写在被测代码的**同一个文件**中。这样做的理由：
- 测试能访问 `pub(crate)` 和私有函数（只要在同一个 crate 内且模块路径正确）
- 代码和测试距离近，修改代码后测试就在眼前
- 不需要维护单独的测试文件目录结构

### 1.2 断言语义族

Rust 提供了精确的断言工具，用对工具能让测试失败时的输出更易理解：

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn assertion_family() {
        // assert! —— 验证 bool 条件
        let v = vec![1, 2, 3];
        assert!(v.len() == 3);

        // assert_eq! —— 验证相等（比 assert! 更好：失败时打印两个值）
        assert_eq!(v.len(), 3);

        // assert_ne! —— 验证不等
        assert_ne!(v.len(), 0);

        // ━━━ 高级：带自定义错误信息的断言 ━━━
        let result = "hello world";
        assert!(
            result.len() > 5,
            "expected length > 5, got {} with value '{}'",
            result.len(),
            result
        );

        // ━━━ 条件编译的调试断言 ━━━
        // debug_assert! 只在 debug 模式下运行，release 中会被移除
        // 适合放在热路径中的"理论上不应失败"的检查
        debug_assert!(!v.is_empty(), "vector should never be empty here");
    }
}
```

### 1.3 `#[should_panic]` —— 验证错误行为

不是所有测试都验证"成功路径"。当你设计一个函数在某些条件下应该 panic 时，`#[should_panic]` 就是你的工具：

```rust
/// 安全的除法，分母不能为零
pub fn safe_divide(a: i32, b: i32) -> i32 {
    if b == 0 {
        panic!("division by zero");
    }
    a / b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_normal_division() {
        assert_eq!(safe_divide(10, 2), 5);
    }

    #[test]
    #[should_panic(expected = "division by zero")]
    fn test_divide_by_zero_should_panic() {
        safe_divide(10, 0); // 期望 panic
    }

    // ━━━ 不带 expected 参数：任何 panic 都算通过 ━━━
    #[test]
    #[should_panic]
    fn test_any_panic_is_ok() {
        panic!("any message works");
    }

    // ━━━ 使用 expected 精确匹配 panic 消息 ━━━
    // 如果 panic 消息不包含 "division by zero"，测试会失败
    // 这比不带 expected 更好：能区分"是我预期的那种 panic"还是"意外的 panic"
}
```

### 1.4 `Result<(), E>` 作为测试返回类型

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use std::io;

    fn setup_temp_file() -> io::Result<String> {
        // 可能失败的准备操作
        Ok("temp_path".to_string())
    }

    // ✅ 测试返回 Result —— 可以在测试中用 ? 操作符
    #[test]
    fn test_with_result() -> io::Result<()> {
        let path = setup_temp_file()?; // ← 不用 unwrap，失败就自动 fail
        assert!(!path.is_empty());
        Ok(())
    }

    // ✅ 也可以配合 should_panic
    #[test]
    #[should_panic]
    fn test_result_with_panic() {
        let _ = std::fs::File::open("/nonexistent/path").unwrap();
    }
}
```

**什么时候用 `Result` 而不是 `panic`？**
- 测试中有多个可能失败的准备步骤 → 用 `Result` + `?` 更干净
- 测试本身验证的是"成功或失败"逻辑 → 用 `assert!` 更直白
- 简单测试没有 fallible 准备步骤 → 用 `assert!` 就好

## 2. 集成测试 —— 验证多个模块协作

### 2.1 `tests/` 目录结构

集成测试放在项目根目录的 `tests/` 目录下，每个 `.rs` 文件是一个独立的 crate：

```
my_project/
  Cargo.toml
  src/
    lib.rs           ← 库代码 + 单元测试
    main.rs          ← 二进制入口（如果有）
  tests/
    integration_test.rs    ← 集成测试文件 1
    api_test.rs            ← 集成测试文件 2
    common/                ← 共享测试工具
      mod.rs
```

**关键规则**：
- `tests/` 下的每个 `.rs` 文件是**独立的 crate**
- 它们只能访问你库的**公共 API**（`pub` 函数、`pub` 类型）
- 私有函数无法在集成测试中访问——这迫使你把"需要测试的行为"放在公共 API 中

### 2.2 集成测试实战

```rust
// tests/api_test.rs —— 集成测试：验证公开 API 的行为
use my_project::UserService;   // 导入你的库

#[test]
fn test_create_and_get_user() {
    let mut service = UserService::new();
    service.create_user("alice", "alice@example.com").unwrap();
    
    let user = service.get_user("alice").unwrap();
    assert_eq!(user.email, "alice@example.com");
}

#[test]
fn test_get_nonexistent_user() {
    let service = UserService::new();
    let result = service.get_user("nobody");
    assert!(result.is_err());
}
```

### 2.3 共享测试辅助函数

```rust
// tests/common/mod.rs —— 共享的测试工具模块
// ⚠️ 注意：必须是 common/mod.rs 结构，不能是 common.rs
// 因为 common.rs 会被 cargo 当作独立的测试 crate

use my_project::UserService;

/// 创建一个预配置的测试服务
pub fn setup_test_service() -> UserService {
    let mut service = UserService::new();
    service.create_user("test_user", "test@example.com").unwrap();
    service
}

/// 清理测试数据
pub fn teardown(service: &mut UserService) {
    service.clear();
}
```

```rust
// tests/integration_test.rs
mod common;  // ← 引入共享模块

use common::setup_test_service;

#[test]
fn test_with_setup() {
    let service = setup_test_service();  // ← 复用 setup
    let user = service.get_user("test_user").unwrap();
    assert!(!user.email.is_empty());
}
```

### 2.4 单元测试 vs 集成测试 —— 什么时候用什么？

```
┌───────────────────┬──────────────────┬──────────────────────┐
│ 维度              │ 单元测试          │ 集成测试              │
├───────────────────┼──────────────────┼──────────────────────┤
│ 访问权限          │ private + pub    │ 仅 pub API           │
│ 速度              │ 极快（微秒级）    │ 较快（毫秒级）        │
│ 隔离性            │ 高（独立模块）    │ 中（多模块协作）      │
│ 定位 bug          │ 精准（函数级别）  │ 模糊（组件级别）      │
│ 维护成本          │ 低               │ 中                   │
│ 信心级别          │ "这个函数对了"    │ "这些组件配合对了"     │
└───────────────────┴──────────────────┴──────────────────────┘
```

**决策**：
- 每个函数的核心逻辑 → 单元测试
- 多个模块交互的流程（如"注册→登录→发帖→查看"）→ 集成测试
- 外部依赖的接入（数据库、文件系统、网络）→ 集成测试（可能需要 mock 或 testcontainers）
- 不要用集成测试覆盖可以用单元测试轻易覆盖的内容

## 3. 文档测试 —— 确保示例代码真是对的

### 3.1 基础文档测试

```rust
/// 计算斐波那契数列的第 n 项
///
/// # Examples
///
/// ```
/// use my_project::fibonacci;
///
/// assert_eq!(fibonacci(0), 0);
/// assert_eq!(fibonacci(1), 1);
/// assert_eq!(fibonacci(10), 55);
/// ```
///
/// # Panics
///
/// 当 n 太大时不会 panic，但会溢出（u64 的 wrap-around）。
/// 对于大 n，请使用 `fibonacci_bigint` 版本。
pub fn fibonacci(n: u64) -> u64 {
    match n {
        0 => 0,
        1 => 1,
        _ => fibonacci(n - 1) + fibonacci(n - 2),
    }
}
```

当运行 `cargo test` 时，上面的 `/// ``` ` 块会被提取、编译并运行。如果 `assert_eq!(fibonacci(10), 55)` 失败，测试就失败。

### 3.2 文档测试的高级用法

```rust
/// 创建用户，返回新用户的 ID
///
/// # Examples
///
/// ```rust
/// # use my_project::UserService;    // ← # 开头的行在文档中不显示
/// # let mut service = UserService::new();
/// let id = service.create_user("bob", "bob@example.com").unwrap();
/// assert!(id > 0);
/// # service.clear();                // ← 隐藏的清理代码
/// ```
///
/// ```rust,should_panic
/// # use my_project::UserService;
/// # let mut service = UserService::new();
/// // 重复创建应该 panic
/// service.create_user("bob", "bob@example.com").unwrap();
/// service.create_user("bob", "bob@example.com").unwrap();
/// ```
///
/// ```rust,no_run
/// // 这段代码不会被运行（但会被编译检查类型）
/// // 适合演示需要外部资源的代码（如网络请求）
/// let response = reqwest::blocking::get("https://api.example.com/users")?;
/// ```
///
/// ```rust,ignore
/// // 这段代码既不会被编译也不会被运行
/// // 适合放伪代码或部分不完整的示例
/// ```
pub fn create_user(&mut self, name: &str, email: &str) -> Result<u64> {
    // ...
}
```

**常见注解**：
| 注解 | 含义 |
|------|------|
| 无注解 | 编译并运行，panic 则失败 |
| `should_panic` | 编译并运行，panic 才算通过 |
| `no_run` | 编译但不运行（做类型检查） |
| `ignore` | 不编译也不运行 |
| `compile_fail` | 期望编译失败（用于展示类型错误） |

## 4. Mock 策略 —— 隔离外部依赖

### 4.1 问题：测试依赖外部服务太脆弱

```rust
// ❌ 直接依赖外部 API 的代码难以测试
pub struct WeatherService;

impl WeatherService {
    pub async fn get_temperature(&self, city: &str) -> Result<f64> {
        let url = format!("https://api.weather.com/v1/{}", city);
        let resp = reqwest::get(&url).await?;  // ← 网络请求！
        let body = resp.json::<WeatherResponse>().await?;
        Ok(body.temp)
    }
}

// 测试这台 WeatherService：
// - 需要网络连接
// - 结果不稳定（温度会变）
// - API 可能限流或宕机
// - 测试很慢（网络延迟）
```

### 4.2 方案一：Trait 抽象 + 手动 Mock

```rust
// ━━━ 第一步：定义 trait 抽象 ━━━
#[async_trait::async_trait]  // 需要 async-trait crate
pub trait TemperatureProvider: Send + Sync {
    async fn get_temperature(&self, city: &str) -> Result<f64>;
}

// ━━━ 第二步：真实实现 ━━━
pub struct WeatherApiProvider {
    api_key: String,
}

#[async_trait::async_trait]
impl TemperatureProvider for WeatherApiProvider {
    async fn get_temperature(&self, city: &str) -> Result<f64> {
        // 真实的 HTTP 请求
        // ...
        Ok(25.0)
    }
}

// ━━━ 第三步：Mock 实现（用于测试）━━━
struct MockWeatherProvider {
    temperatures: std::collections::HashMap<String, f64>,
}

#[async_trait::async_trait]
impl TemperatureProvider for MockWeatherProvider {
    async fn get_temperature(&self, city: &str) -> Result<f64> {
        self.temperatures
            .get(city)
            .copied()
            .ok_or_else(|| anyhow::anyhow!("unknown city: {}", city))
    }
}

// ━━━ 第四步：业务逻辑依赖 trait，不依赖具体实现 ━━━
pub struct WeatherReporter<P: TemperatureProvider> {
    provider: P,
}

impl<P: TemperatureProvider> WeatherReporter<P> {
    pub async fn report(&self, city: &str) -> String {
        match self.provider.get_temperature(city).await {
            Ok(temp) => format!("{}: {}°C", city, temp),
            Err(e) => format!("Error: {}", e),
        }
    }
}

// ━━━ 测试 ━━━
#[cfg(test)]
mod tests {
    use super::*;

    #[tokio::test]
    async fn test_weather_report() {
        let mut mock = MockWeatherProvider {
            temperatures: std::collections::HashMap::new(),
        };
        mock.temperatures.insert("Beijing".to_string(), 20.0);
        mock.temperatures.insert("Shanghai".to_string(), 25.0);

        let reporter = WeatherReporter { provider: mock };
        assert_eq!(reporter.report("Beijing").await, "Beijing: 20°C");
        assert_eq!(reporter.report("Shanghai").await, "Shanghai: 25°C");
    }
}
```

**Trait 抽象 + Mock 的优缺点**：
- ✅ 完全 Rust 原生，不需要额外的 crate
- ✅ 类型安全，编译期保证
- ❌ 需要为每个依赖手动写 trait 抽象和 mock 实现
- ❌ 泛型参数会"感染"使用方（`WeatherReporter<P>` 而不是 `WeatherReporter`）
- ❌ 当依赖很多时，trait 数量爆炸

### 4.3 方案二：mockall —— 自动生成 Mock

`mockall` 提供了 `#[automock]` 属性，自动为 trait 生成 mock 实现：

```rust
use mockall::automock;
use mockall::predicate::*;

// ━━━ 用 #[automock] 标注 trait ━━━
#[automock]
#[async_trait::async_trait]
pub trait TemperatureProvider: Send + Sync {
    async fn get_temperature(&self, city: &str) -> Result<f64>;
}

#[cfg(test)]
mod tests {
    use super::*;

    #[tokio::test]
    async fn test_with_mockall() {
        let mut mock = MockTemperatureProvider::new();  // ← 自动生成的结构体！

        // 设置期望：当 get_temperature("Beijing") 被调用时返回 Ok(20.0)
        mock.expect_get_temperature()
            .with(eq("Beijing"))        // 参数匹配器
            .times(1)                    // 期望被调用 1 次
            .returning(|_city| Ok(20.0)); // 返回闭包

        mock.expect_get_temperature()
            .with(eq("Shanghai"))
            .returning(|_city| Ok(25.0));

        let reporter = WeatherReporter { provider: mock };
        assert_eq!(reporter.report("Beijing").await, "Beijing: 20°C");
        assert_eq!(reporter.report("Shanghai").await, "Shanghai: 25°C");
        // mock 在 drop 时自动验证：expect 的调用次数是否匹配？
    }
}
```

**mockall 的核心 API**：
| 方法 | 用途 |
|------|------|
| `expect_xxx()` | 期望被调用，如果没被调用测试会失败 |
| `returning(closure)` | 指定返回值 |
| `with(matcher)` | 参数匹配（`eq`、`gt`、`lt`、`always`） |
| `times(n)` | 指定期望调用次数 |
| `never()` | 期望从不被调用 |
| `return_const(val)` | 总是返回固定值 |

### 4.4 两种方案的选型决策

| 场景 | 推荐方案 |
|------|---------|
| 只有 1-2 个外部依赖 | Trait 抽象 + 手动 Mock（更轻量） |
| 3 个以上依赖需要 mock | mockall（避免样板代码爆炸） |
| 库的公共 API（需要稳定） | Trait 抽象（mockall 不应出现在你的 pub API 中） |
| 应用内部测试 | mockall（开发效率优先） |
| 异步代码 | 两者都支持（需要 `async-trait` + `mockall` 的 async feature） |

## 5. Snapshot 测试 —— 捕获复杂输出

### 5.1 什么场景需要 snapshot？

当你的测试输出不是"一个数字"或"一个 bool"，而是**一段结构化的文本**时，手写 `assert_eq!` 变得不可能：

```rust
// ❌ 这种输出你怎么 assert？
// {
//   "name": "Alice",
//   "age": 30,
//   "children": [
//     {"name": "Bob", "age": 5},
//     {"name": "Carol", "age": 3}
//   ]
// }

// ❌ 手写 assert_eq 极其繁琐且容易出错
assert_eq!(json, "{\n  \"name\": \"Alice\",\n  ...太长了...");
```

Snapshot 测试的策略：**第一次运行时把输出保存为一个 `.snap` 文件，后续运行时自动对比。**

### 5.2 `insta` —— Rust 的 snapshot 测试库

```toml
# Cargo.toml
[dev-dependencies]
insta = { version = "1", features = ["json"] }
```

```rust
use serde::Serialize;

#[derive(Serialize)]
struct User {
    name: String,
    age: u32,
    children: Vec<User>,
}

fn serialize_user(user: &User) -> String {
    serde_json::to_string_pretty(user).unwrap()
}

#[cfg(test)]
mod tests {
    use super::*;
    use insta::assert_json_snapshot;

    #[test]
    fn test_user_serialization() {
        let user = User {
            name: "Alice".into(),
            age: 30,
            children: vec![
                User { name: "Bob".into(), age: 5, children: vec![] },
                User { name: "Carol".into(), age: 3, children: vec![] },
            ],
        };

        // 第一次运行：创建 snapshots/xxx.snap 文件
        // 后续运行：自动对比
        insta::assert_json_snapshot!(serialize_user(&user));
    }
}
```

**运行流程**：
```bash
# 第一次运行：创建快照文件
cargo test
# → 生成 tests/snapshots/xxx.snap

# 如果输出变了（比如你改了字段名）
cargo test
# → FAIL：快照不匹配！
# → 终端显示 diff

# 审查 diff 后，如果变更是有意的：
cargo insta review
# → 交互式审查：a(accept 接受) / r(reject 拒绝) / s(skip 跳过)

# 或者直接接受所有变更：
cargo insta accept
```

### 5.3 insta 常用宏

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use insta::*;

    #[test]
    fn snapshot_variants() {
        // ━━━ 字符串快照 ━━━
        assert_snapshot!("hello world");  // → 保存为 .snap 文件

        // ━━━ 带名称的快照（区分同一个函数中的多个 snapshot）━━━
        assert_snapshot!("greeting", "hello world");

        // ━━━ Debug 输出快照 ━━━
        let vec = vec![1, 2, 3, 4, 5];
        assert_debug_snapshot!(vec);
        // 保存内容：[1, 2, 3, 4, 5]

        // ━━━ JSON 快照（需要 features = ["json"]）━━━
        let json_str = r#"{"key": "value"}"#;
        assert_json_snapshot!(json_str);

        // ━━━ 序列化快照（需要 features = ["ron"] 等）━━━
        // assert_ron_snapshot!(my_struct);
    }
}
```

### 5.4 Snapshot 测试的最佳实践

```
✅ 适合 snapshot 测试的场景：
  - 序列化/反序列化输出（JSON, YAML, TOML）
  - CLI 命令的输出
  - 模板渲染结果（HTML, Markdown, 邮件正文）
  - 复杂数据结构的 Debug 输出
  - HTTP API 响应体的快照

❌ 不适合 snapshot 测试的场景：
  - 单个数字或布尔值（手写 assert_eq! 更清晰）
  - 包含时间戳或随机数的输出（需要先 normalize）
  - 包含不稳定的排序结果（需要先排序）

⚠️ 黄金法则：
  - Snapshot 文件必须提交到 git（它们是测试的一部分）
  - Review PR 时，检查 .snap 文件的变更和代码变更要一起看
  - 太大的 snapshot（>100 行）是设计问题的信号——考虑重构
```

### 6.3 测试覆盖率不是目标

```
常见的误区："我们要达到 90% 的代码覆盖率！"

真相：
  - 覆盖率告诉你"哪些代码被测试执行到了"，不告诉你"测试有没有验证正确性"
  - 100% 覆盖率 + 没有 assert 的测试 = 0% 信心
  - 追求覆盖率往往导致"为了覆盖而覆盖"的无意义测试

更好的目标：
  - 核心业务逻辑 80%+ 覆盖率（有 assert 的）
  - 简单的 getter/setter/constructor 不需要单独测试
  - 每个 pub 函数至少有一个"好路径"测试 + 1-2 个边界条件测试
  - 错误路径比成功路径更需要测试（因为平时不会走到）
```

```rust
use anyhow::Result;

pub trait EmailSender {
    fn send(&self, to: &str, subject: &str, body: &str) -> Result<()>;
}

pub struct RegistrationService<E: EmailSender> {
    email_sender: E,
}

impl<E: EmailSender> RegistrationService<E> {
    pub fn new(email_sender: E) -> Self {
        Self { email_sender }
    }

    pub fn register(&self, email: &str) -> Result<()> {
        // 注册逻辑...
        // 注册成功后发送欢迎邮件
        self.email_sender.send(email, "Welcome!", "Thanks for registering!")?;
        Ok(())
    }
}

// ━━━ 手动 Mock ━━━
#[cfg(test)]
mod tests {
    use super::*;
    use std::cell::RefCell;

    struct MockEmailSender {
        sent_emails: RefCell<Vec<(String, String, String)>>,
    }

    impl EmailSender for MockEmailSender {
        fn send(&self, to: &str, subject: &str, body: &str) -> Result<()> {
            self.sent_emails.borrow_mut().push((
                to.to_string(),
                subject.to_string(),
                body.to_string(),
            ));
            Ok(())
        }
    }

    #[test]
    fn test_registration_sends_welcome_email() {
        let mock = MockEmailSender { sent_emails: RefCell::new(vec![]) };
        let service = RegistrationService::new(mock);

        service.register("alice@example.com").unwrap();

        let emails = service.email_sender.sent_emails.borrow();
        assert_eq!(emails.len(), 1);
        assert_eq!(emails[0].0, "alice@example.com");
        assert_eq!(emails[0].1, "Welcome!");
    }
}
