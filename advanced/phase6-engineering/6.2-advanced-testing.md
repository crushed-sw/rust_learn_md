# 6.2 高级测试技术 —— 找到你不知道的未知

> 单元测试验证的是"你想到的边界"——你写了 `test_divide_by_zero`，所以你验证了分母为零的情况。但**你没想到的边界呢**？那个只在输入是质数且长度超过 4096 字节时才触发的 bug？那个只有在线程 A 先执行到第 47 行、线程 B 恰好卡在锁上时才出现的数据竞争？单元测试不会替你想到这些。高级测试技术的价值就在这里——**它们不依赖你的想象力**。属性测试自动生成海量用例来检验数学不变量；模糊测试用半随机数据疯狂轰炸你的解析器；Miri 在 MIR 层面追踪每一次内存访问；Loom 穷举所有可能的线程调度。当你把这四种技术加入测试套件时，你不再只是"觉得自己的代码没问题"——你有了证据。

## 1. 属性测试（Property-Based Testing）—— 验证不变量

### 1.1 从"用例"到"属性"的思维转换

```
单元测试的思维：
  "对于输入 [1, 2, 3]，排序后应该是 [1, 2, 3]"
  "对于输入 [3, 1, 2]，排序后应该是 [1, 2, 3]"
  "对于输入 []，排序后应该是 []"
  → 你只能列出你能想到的有限几个例子

属性测试的思维：
  "对于任意 Vec<i32>，sort 之后，len 不变"
  "对于任意 Vec<i32>，sort 之后，每个元素都在原数组中"
  "对于任意 Vec<i32>，sort 之后，对于所有 i<j，arr[i] <= arr[j]"
  → 这些 **属性** 对所有输入都应该成立
```

### 1.2 `proptest` 入门

```toml
# Cargo.toml
[dev-dependencies]
proptest = "1"
```

```rust
use proptest::prelude::*;

// 被测试的函数
fn reverse_vector<T>(mut v: Vec<T>) -> Vec<T> {
    v.reverse();
    v
}

#[cfg(test)]
mod tests {
    use super::*;
    use proptest::prelude::*;

    proptest! {
        // ━━━ 属性 1：反转两次等于原数组 ━━━
        #[test]
        fn test_reverse_twice_equals_original(
            vec in prop::collection::vec(any::<i32>(), 0..100)
            //                    ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
            //                    生成 0 到 100 个 i32 的随机 Vec
        ) {
            let original = vec.clone();
            let reversed_once = reverse_vector(vec);
            let reversed_twice = reverse_vector(reversed_once);
            assert_eq!(original, reversed_twice);
        }

        // ━━━ 属性 2：长度不变 ━━━
        #[test]
        fn test_reverse_preserves_length(
            vec in prop::collection::vec(any::<String>(), 0..50)
        ) {
            let original_len = vec.len();
            let reversed = reverse_vector(vec);
            assert_eq!(original_len, reversed.len());
        }
    }
}
```

**运行过程**：
```bash
cargo test
# proptest 会生成 256 个随机测试用例（默认配置）
# 如果全部通过 → OK
# 如果某个失败 → proptest 自动"缩小"（shrink）到最小反例
```

### 1.3 Shrink —— proptest 的超能力

当 proptest 发现一个失败用例时，它不是直接报告"`[83, -42, 7, 15, ...]` 失败了"，而是**自动缩小输入**以找到最小的反例：

```rust
proptest! {
    #[test]
    fn test_divide(
        (a, b) in (any::<i32>(), any::<i32>())
    ) {
        let result = a / b; // ← 如果 b = 0 会 panic
        // ... 验证一些属性
    }
}
// 运行结果：
// proptest: found minimal failing case: (0, 0)
//           原始发现的是 (8347, 0)，自动 shrink 到 (0, 0)
```

**Shrink 的价值**：你不会拿到一个 500 个元素的 Vec 说"这个测试失败了"，而是拿到最小可能的反例——比如 `[0]` 或 `[-1]`。

### 1.4 常用策略（Strategy）

```rust
proptest! {
    #[test]
    fn common_strategies(
        // ━━━ 基础类型 ━━━
        age in 0u32..150,                    // 范围
        name in "[a-zA-Z]{1,20}",            // 正则
        flag in any::<bool>(),               // 任意值
        opt in prop::option::of(any::<i32>()), // Option<T>

        // ━━━ 集合 ━━━
        vec in prop::collection::vec(any::<i32>(), 0..100),     // Vec
        set in prop::collection::hash_set(any::<i64>(), 0..50), // HashSet
        map in prop::collection::hash_map(                       // HashMap
            any::<String>(), any::<i32>(), 0..20
        ),

        // ━━━ 自定义策略组合 ━━━
        tuple in (any::<i32>(), any::<String>(), any::<bool>()),
    ) {
        // 使用这些生成的值
        assert!(age < 150);
        assert!(!name.is_empty() || name.len() <= 20);
    }
}
```

### 1.5 实战：为 JSON 解析器写属性测试

```rust
// 场景：你写了一个 JSON 解析器，想验证 "序列化→反序列化" 的不变性

#[derive(Debug, Clone, PartialEq, proptest_derive::Arbitrary)]
struct Person {
    name: String,
    age: u32,
    active: bool,
}

fn serialize_person(p: &Person) -> String {
    format!(r#"{{"name":"{}","age":{},"active":{}}}"#, p.name, p.age, p.active)
}

fn deserialize_person(s: &str) -> Option<Person> {
    // 一个简易的手写 JSON 解析器
    // ...解析逻辑...
    None // 占位
}

proptest! {
    #[test]
    fn roundtrip_property(person in any::<Person>()) {
        let json = serialize_person(&person);
        let decoded = deserialize_person(&json).unwrap();
        assert_eq!(person, decoded);
        // 属性："序列化后再反序列化，结果等于原值"
    }
}
```

### 1.6 属性测试的适用场景

```
✅ 强烈推荐：
  - 序列化/反序列化（roundtrip 不变性）
  - 编码/解码（encode → decode = identity）
  - 排序算法（输出有序 + 元素不变 + 长度不变）
  - 集合操作（并集、交集满足数学定律）
  - 压缩/解压（compress → decompress = identity）
  - 任何有"数学不变量"的场景

⚠️ 不太适合：
  - 复杂的业务逻辑（属性难以形式化）
  - 高度依赖外部状态（数据库、网络、文件系统）
  - UI 渲染（"看起来对不对"不是属性）
```

## 2. 模糊测试（Fuzzing）—— 用随机数据寻找崩溃

### 2.1 Fuzzing vs Property-Based Testing

```
两者的区别：

属性测试（proptest）：
  "我定义了不变量，你帮我找反例"
  → 需要你知道"什么是对的"
  → 适合：已定义清晰属性的纯逻辑

模糊测试（fuzzing）：
  "我喂你随机数据，只要你没崩溃就行"
  → 不需要定义"什么是对的"
  → 适合：解析器、协议处理、图像/音频解码
```

### 2.2 `cargo-fuzz` 实战

```bash
# 安装
cargo install cargo-fuzz

# 在你的项目里创建 fuzz target
cargo fuzz init
# → 生成 fuzz/fuzz_targets/fuzz_target_1.rs
```

```rust
// fuzz/fuzz_targets/parse_url.rs
#![no_main]

use libfuzzer_sys::fuzz_target;
use my_project::parse_url;  // 你的 URL 解析函数

fuzz_target!(|data: &[u8]| {
    // data 是 libFuzzer 生成的半随机字节流
    // 你的目标：不能 panic、不能无限循环、不能 UB

    if let Ok(s) = std::str::from_utf8(data) {
        // 尝试解析任意字符串为 URL
        let _ = parse_url(s);
        // parse_url 可能返回 Err（解析失败），但绝对不应该 panic
    }
});
```

```bash
# 运行模糊测试（建议在 CI 中跑 5-10 分钟）
cargo fuzz run parse_url -- -max_total_time=300

# 如果发现崩溃，libFuzzer 会保存触发崩溃的输入：
# fuzz/artifacts/parse_url/crash-*
```

### 2.3 编写好的 Fuzz Target

```rust
// ━━━ 原则 1：结构化输入 → 尽量解析，而非直接操作字节 ━━━
fuzz_target!(|data: &[u8]| {
    // ✅ 好：尝试将字节解析为有意义的结构
    if let Ok(input) = parse_my_protocol(data) {
        process(input);
    }
});

// ━━━ 原则 2：差分 Fuzzing —— 对比两个实现 ━━━
fuzz_target!(|data: &[u8]| {
    // 假设你有两个解析器：一个快速的（手写的），一个正确的（参考实现）
    let result_fast = fast_parser(data);
    let result_correct = reference_parser(data);
    // 如果都能解析，结果应该一致
    if let (Ok(a), Ok(b)) = (result_fast, result_correct) {
        assert_eq!(a, b);
    }
});

// ━━━ 原则 3：结构感知 Fuzzing —— 用 Arbitrary trait ━━━
use arbitrary::Arbitrary;

#[derive(Arbitrary, Debug)]
struct MyInput {
    command: u8,
    payload: Vec<u8>,
    flags: u32,
}

fuzz_target!(|input: MyInput| {
    // 直接拿到结构化的输入，不需要手动解析字节
    process_command(input.command, &input.payload, input.flags);
});
```

### 2.4 Fuzzing 适合的场景

```
✅ 极其有效：
  - 解析器（JSON, YAML, URL, HTML, 二进制协议）
  - 编解码器（图像、音频、视频、压缩）
  - 正则表达式引擎
  - 网络协议实现

❌ 不太有效：
  - 纯算法逻辑（proptest 更好）
  - 需要复杂状态机的场景（考虑 stateful fuzzing）
  - 高度依赖外部 I/O 的代码
```

## 3. Miri —— 在 MIR 层面检测未定义行为

### 3.1 Miri 是什么？

```
Miri 是一个 MIR（Mid-level Intermediate Representation）解释器。
它不编译你的代码为机器码——它在 MIR 上逐条解释执行。

好处：每执行一条 MIR 指令，Miri 都会检查：
  - 是否有未初始化的内存被读取？
  - 是否有指针越界？
  - 是否有对齐违规？
  - 是否有 data race？
  - 是否有整数溢出（即使是 release 模式）？
  - dangling pointer 解引用？

这使得 Miri 的检测比 sanitizer 更精确——因为它在抽象层面上执行。
```

### 3.2 运行 Miri

```bash
# 安装
rustup +nightly component add miri

# 运行测试
cargo +nightly miri test

# 运行特定测试
cargo +nightly miri test -- test_unsafe_code

# 列出所有受支持的检查
cargo +nightly miri test -- -Zmiri-env-set=MIRI_LOG=info
```

### 3.3 Miri 能检测到的问题

```rust
// ━━━ 问题 1：读取未初始化的内存 ━━━
#[test]
fn test_uninit_read() {
    let x: u8 = unsafe { std::mem::MaybeUninit::uninit().assume_init() };
    // Miri: error: reading from uninitialized memory
    println!("{}", x);
}

// ━━━ 问题 2：指针越界 ━━━
#[test]
fn test_oob_pointer() {
    let arr = [1u8, 2, 3];
    let ptr = arr.as_ptr();
    unsafe {
        let _ = *ptr.add(5); // ← Miri: error: pointer arithmetic with out-of-bounds offset
    }
}

// ━━━ 问题 3：use-after-free ━━━
#[test]
fn test_use_after_free() {
    let ptr = {
        let v = vec![1, 2, 3];
        v.as_ptr()
    }; // v 被丢弃
    unsafe {
        let _ = *ptr; // ← Miri: error: dangling pointer dereference
    }
}

// ━━━ 问题 4：错误的指针转换 ━━━
#[test]
fn test_bad_cast() {
    let x = 42u32;
    let ptr = &x as *const u32 as *const u64;
    unsafe {
        let _ = *ptr; // ← Miri: error: accessing memory with incorrect alignment
    }
}

// ━━━ 问题 5：并发 data race ━━━
#[test]
fn test_data_race() {
    use std::sync::atomic::{AtomicU64, Ordering};
    use std::thread;

    let x = AtomicU64::new(0);
    let x_ref = &x;

    let t1 = thread::spawn(move || {
        x_ref.store(42, Ordering::Relaxed);
    });
    let t2 = thread::spawn(move || {
        x_ref.store(99, Ordering::Relaxed);
    });
    t1.join().unwrap();
    t2.join().unwrap();

    // Atomic 操作本身没有 data race（但上面有逻辑上的竞态）
    // 真正的问题是：非 Atomic 类型在多线程中的 data race
}
```

### 3.4 Miri 的局限性

```
Miri 不能检测的：
  ✗ 调用了 C FFI 的代码（Miri 不理解 C 代码）
  ✗ 调用了内联汇编的代码
  ✗ 调用了某些平台特定的系统调用
  ✗ 没有被执行到的代码路径（和所有动态分析工具一样）
  ✗ 直接操作文件系统或网络的代码

Miri 比 sanitizer 更好的地方：
  ✓ 不需要插桩编译（不影响代码布局）
  ✓ 能精确报告哪一行触发了 UB
  ✓ 能检测更多类型的 UB（sanitizer 主要检测内存错误）
  ✓ 能在任何平台上运行（因为是解释器，不依赖特定 CPU 特性）
```

## 4. Loom —— 穷举并发排列

### 4.1 并发测试的困境

```rust
// ❌ 这个测试可能"今晚通过了，明早 CI 失败了"
#[test]
fn test_concurrent_counter() {
    let counter = std::sync::Arc::new(std::sync::atomic::AtomicU64::new(0));
    let c1 = counter.clone();
    let c2 = counter.clone();

    let t1 = std::thread::spawn(move || {
        for _ in 0..1000 {
            c1.fetch_add(1, std::sync::atomic::Ordering::SeqCst);
        }
    });
    let t2 = std::thread::spawn(move || {
        for _ in 0..1000 {
            c2.fetch_add(1, std::sync::atomic::Ordering::SeqCst);
        }
    });

    t1.join().unwrap();
    t2.join().unwrap();

    assert_eq!(counter.load(std::sync::atomic::Ordering::SeqCst), 2000);
    // 这个测试用 AtomicU64 是对的——但如果有人不小心改成了普通的 u64 + Mutex，
    // 并发 bug 可能跑 10000 次才出现一次
}
```

### 4.2 Loom：模拟所有可能的线程交错

```toml
[dev-dependencies]
loom = "0.7"
```

```rust
// tests/loom_counter.rs
use loom::sync::atomic::AtomicU64;
use loom::sync::atomic::Ordering;
use loom::thread;

#[test]
fn test_counter_under_loom() {
    // loom::model 会穷举当前测试中所有可能的执行排列
    loom::model(|| {
        let counter = loom::sync::Arc::new(AtomicU64::new(0));
        let c1 = counter.clone();
        let c2 = counter.clone();

        let t1 = thread::spawn(move || {
            for _ in 0..2 { // ← Loom 下循环次数要小（状态爆炸）
                c1.fetch_add(1, Ordering::SeqCst);
            }
        });
        let t2 = thread::spawn(move || {
            for _ in 0..2 {
                c2.fetch_add(1, Ordering::SeqCst);
            }
        });

        t1.join().unwrap();
        t2.join().unwrap();

        assert_eq!(counter.load(Ordering::SeqCst), 4);
    });
}
```

**Loom 的工作原理**：
```
你的代码中有两个线程，每个线程有 2 个 fetch_add 操作。
Loom 穷举所有可能的交错序列：
  t1.add → t1.add → t2.add → t2.add
  t1.add → t2.add → t1.add → t2.add
  t1.add → t2.add → t2.add → t1.add
  t2.add → t1.add → t1.add → t2.add
  ...（所有排列）

如果任何一种排列下 assert 失败 → 报告具体哪种排列触发了失败
```

### 4.3 Loom 的局限

```
⚠️ 状态爆炸：
  - 循环次数不能太大（通常 ≤ 3-5 次迭代）
  - 线程数不能太多（通常 ≤ 2-3 个）
  - 每个原子的操作会指数级增加排列数

⚠️ loom 替换了 std 的类型：
  - 需要改用 loom::sync::* 而不是 std::sync::*
  - 需要 #[cfg(loom)] 做条件编译

✅ Loom 最擅长检测：
  - 错误的 Ordering（Relaxed 导致的逻辑错误）
  - 细粒度锁协议中的死锁
  - 自定义同步原语中的 bug
```

### 4.4 条件编译：测试时用 loom，生产用 std

```rust
// 生产代码中可以用 cfg 切换

#[cfg(not(loom))]
use std::sync::atomic::{AtomicU64, Ordering};

#[cfg(loom)]
use loom::sync::atomic::{AtomicU64, Ordering};

pub struct MyCounter {
    count: AtomicU64,
}

impl MyCounter {
    pub fn new() -> Self {
        Self { count: AtomicU64::new(0) }
    }

    pub fn increment(&self) {
        self.count.fetch_add(1, Ordering::SeqCst);
    }

    pub fn get(&self) -> u64 {
        self.count.load(Ordering::SeqCst)
    }
}

#[cfg(test)]
mod loom_tests {
    use super::*;

    #[test]
    fn test_counter_concurrent() {
        loom::model(|| {
            let counter = loom::sync::Arc::new(MyCounter::new());
            let c1 = counter.clone();
            let c2 = counter.clone();

            let t1 = loom::thread::spawn(move || {
                for _ in 0..3 { c1.increment(); }
            });
            let t2 = loom::thread::spawn(move || {
                for _ in 0..3 { c2.increment(); }
            });
            t1.join().unwrap();
            t2.join().unwrap();
            assert_eq!(counter.get(), 6);
        });
    }
}
```

## 5. CI 集成 —— 让高级测试常驻 CI

### 5.1 GitHub Actions 配置示例

```yaml
# .github/workflows/ci.yml
name: CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable

      # ━━━ 常规测试 ━━━

      # ━━━ 属性测试：运行更多用例 ━━━
      - run: cargo test -- --ignored

  miri:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@nightly
        with:
          components: miri
      - run: cargo miri test
        # 只对 unsafe-heavy 的 crate 运行

  fuzz:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@nightly
      - run: cargo install cargo-fuzz
      # 每个 fuzz target 跑 5 分钟
      - run: cargo fuzz run fuzz_target_1 -- -max_total_time=300
      - run: cargo fuzz run fuzz_target_2 -- -max_total_time=300
```

## 6. 练习

### 6.1 练习：为 `remove_duplicates` 写属性测试

```rust
/// 从已排序的 Vec 中移除重复元素（原地修改）
pub fn remove_duplicates<T: PartialEq>(v: &mut Vec<T>) {
    v.dedup();
}
```

## 7. 练习答案

```rust
proptest! {
    #[test]
    fn test_remove_duplicates_properties(
        mut vec in prop::collection::vec(any::<i32>(), 0..50)
    ) {
        let original_len = vec.len();
        let original = vec.clone();

        vec.sort();
        remove_duplicates(&mut vec);

        // 属性 1：长度不会增加
        assert!(vec.len() <= original_len);

        // 属性 2：没有相邻重复
        for i in 1..vec.len() {
            assert_ne!(vec[i], vec[i - 1]);
        }

        // 属性 3：所有输出元素都在原数组中
        for item in &vec {
            assert!(original.contains(item));
        }
    }
}
```
