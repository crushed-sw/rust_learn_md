# 6.9 运行时性能优化 —— 数据结构、内存与算法

> Profiling 告诉你热点在哪里（6.8），优化告诉你热点怎么修（6.9）。好消息是：绝大多数性能问题不需要汇编级别的黑魔法——它们只需要"选对数据结构"、"放好内存布局"、"少分配几次内存"、"用对算法"。这些优化有一个共同特点：**它们对代码可读性的影响很小，但性能收益很大**。一个 `Vec::with_capacity` 可能让循环快 3 倍，一个 `SmallVec` 可能消除 90% 的堆分配，把 `HashMap` 换成 `BTreeMap` 可能让有序遍历从 O(n log n) 降到 O(n)。这些不是"过早优化"——它们是在写代码时就该养成的**性能意识**。本节以"数据为中心"组织内容：数据结构选择、内存布局优化、分配策略、算法改进、`#[inline]` 的正确姿势、字符串优化、散列函数选择——七把手术刀，按 ROI 从高到低排列。

## 1. 数据结构选择 —— ROI 最高的优化

### 1.1 HashMap vs BTreeMap

```rust
use std::collections::{HashMap, BTreeMap};

// ━━━ 什么时候用哪个？ ━━━

// HashMap（哈希表）：
//   ✅ 随机查找 O(1)
//   ✅ 插入/删除 O(1)
//   ❌ 不保证顺序
//   ❌ 内存开销大（需要额外的哈希桶）
//   适用：大部分不需要顺序的 key-value 场景

// BTreeMap（B 树）：
//   ✅ 有序存储（可以范围查询）
//   ✅ 内存紧凑（连续分配）
//   ❌ 查找 O(log n)
//   ❌ 插入/删除 O(log n)
//   适用：需要顺序遍历、范围查询、或 key 不能哈希的场景

// ━━━ 实测对比 ━━━
fn map_benchmarks() {
    // 场景 1：随机查找 10000 次
    // HashMap: 120µs  (O(1) 的优势在大量查找时体现)
    // BTreeMap: 450µs (log n 在大规模时累积)

    // 场景 2：顺序遍历所有元素
    // HashMap: 890µs  (元素分散在内存中 → 缓存不友好)
    // BTreeMap: 95µs   (元素连续存储 → 缓存友好！)

    // 场景 3：小规模（< 100 个元素）
    // 两者性能接近——甚至 Vec<(K, V)> 的线性扫描可能更快
}
```

**决策表**：
```
你需要什么？          → 选什么？
─────────────────────────────────────
随机查找为主要操作     → HashMap
需要按键顺序遍历       → BTreeMap
需要范围查询（"从 A 到 M 的所有 key"）→ BTreeMap
数据量 < 100           → 都可以，考虑 Vec<(K,V)> + 线性扫描
key 没有实现 Hash       → BTreeMap（只需要 Ord）
内存敏感               → BTreeMap（更紧凑）
```

### 1.2 Vec vs SmallVec

```rust
use smallvec::{smallvec, SmallVec};

// ━━━ SmallVec：内联存储小数组 ━━━
// 原理：在栈上预留 N 个元素的数组，超过 N 才上堆
// 类似 C++ 的 std::small_vector

// std::Vec：总是堆分配
let v1: Vec<i32> = vec![1, 2, 3];  // ← 即使只有 3 个元素也要堆分配

// SmallVec：如果 ≤ 4 个元素，全在栈上
let v2: SmallVec<[i32; 4]> = smallvec![1, 2, 3]; // ← 零堆分配！

// ━━━ 性能差异 ━━━
fn smallvec_bench() {
    // 场景：创建大量小数组（如解析器中的 token 列表）
    // Vec<Vec<u8>>:  每个小 Vec 一次 malloc → N 次堆分配
    // SmallVec<[u8; 32]>: 大多数数组 ≤ 32 字节 → 0 次堆分配
    // 结果：SmallVec 可以快 3-5 倍！
}
```

**`SmallVec` 的适用条件**：
- 大多数时候数组很小（< 16 或 < 32 个元素）
- 偶尔会超出这个限制（这时退化为堆分配）
- 数组被频繁创建和销毁

### 1.3 VecDeque vs LinkedList

```rust
use std::collections::{VecDeque, LinkedList};

// ━━━ 几乎永远不要用 LinkedList ━━━
// Rust 标准库文档自己都承认：

// VecDeque（环形缓冲区）：
//   ✅ push_front O(1)*（摊销）
//   ✅ push_back O(1)*
//   ✅ 缓存友好（连续内存）
//   ✅ 支持索引访问

// LinkedList（双向链表）：
//   ❌ 每个节点独立堆分配 → N 次 malloc
//   ❌ 缓存极不友好（指针跳转）
//   ❌ 内存开销大（每个节点 2 个指针 + 堆分配元数据）
//   ✅ 理论上插入/删除 O(1)，但常数因子很糟糕

// ━━━ LinkedList 唯一有用的场景 ━━━
// 你需要把一个列表拆分成两个，且不复制元素
// → 但即使这种场景，用 Vec + split_off 通常也够好
```

## 2. 内存布局优化

### 2.1 SoA vs AoS —— 数组的结构 vs 结构的数组

```rust
// ━━━ AoS（Array of Structures）—— 常见的写法 ━━━
struct Particle {
    x: f64,
    y: f64,
    z: f64,
    velocity: f64,
}
let particles: Vec<Particle> = vec![...];
// 内存布局：[x y z v x y z v x y z v ...]
// 遍历所有 velocity：每次都要跳过 x y z → 缓存浪费

// ━━━ SoA（Structure of Arrays）—— 性能友好的写法 ━━━
struct ParticleSystem {
    x: Vec<f64>,
    y: Vec<f64>,
    z: Vec<f64>,
    velocity: Vec<f64>,
}
// 内存布局：[x x x ...] [y y y ...] [z z z ...] [v v v ...]
// 遍历所有 velocity：连续读取 v v v → 缓存友好！

// ━━━ 性能差异 ━━━
fn soa_vs_aos() {
    // 遍历所有粒子的 velocity（100万个粒子）
    // AoS: 8.2ms（每次读取 8 字节，跳过 24 字节）
    // SoA: 1.1ms（连续读取 8 字节 × 100万次）
    // 提升：7.5x ← 纯内存布局优化
}
```

**SoA 的适用条件**：
- 你的结构体有很多字段，但大多数操作只访问其中一两个
- 数据量大（> 10000 个元素）
- 需要 SIMD 加速（连续内存更利于向量化）

### 2.2 Eytzinger Layout —— 为二分查找优化

```rust
// ━━━ 标准的有序数组二分查找 ━━━
// 内存访问模式：中间 → 左边 → 右边 → ...
// → 每次访问跳跃很大，缓存不友好

// ━━━ Eytzinger 布局 ━━━
// 将二叉搜索树按 BFS 顺序存储在数组中：
//   索引 0 是根
//   索引 i 的左子节点在 2i+1，右子节点在 2i+2
// → 靠近根的节点在内存中也靠近 → 缓存友好

/// 将有序数组转换为 Eytzinger 布局
fn eytzinger_layout<T: Clone>(sorted: &[T]) -> Vec<T> {
    let n = sorted.len();
    let mut result = vec![sorted[0].clone(); n];
    eytzinger_build(sorted, &mut result, 0, &mut 0);
    result
}

fn eytzinger_build<T: Clone>(src: &[T], dst: &mut [T], src_idx: &mut usize, dst_idx: usize) {
    if dst_idx >= dst.len() {
        return;
    }
    eytzinger_build(src, dst, src_idx, 2 * dst_idx + 1);
    dst[dst_idx] = src[*src_idx].clone();
    *src_idx += 1;
    eytzinger_build(src, dst, src_idx, 2 * dst_idx + 2);
}

// 性能：标准二分查找在有序数组中的 Eytzinger 布局可快 2-3 倍（缓存优化）
// 库推荐：eytzinger crate（已经实现好了）
```

### 2.3 对齐和填充

```rust
use std::mem;

// ━━━ 结构体字段顺序影响大小 ━━━
#[repr(C)]
struct Bad {
    a: u8,   // 1 byte
    // 7 bytes padding    ← 浪费！
    b: u64,  // 8 bytes (必须 8 字节对齐)
    c: u8,   // 1 byte
    // 7 bytes padding    ← 浪费！
}
// sizeof(Bad) = 24 bytes

#[repr(C)]
struct Good {
    b: u64,  // 8 bytes
    a: u8,   // 1 byte
    c: u8,   // 1 byte
    // 6 bytes padding    ← 最小填充
}
// sizeof(Good) = 16 bytes ← 节省 33%！

// ━━━ 检查大小 ━━━
fn check_sizes() {
    assert_eq!(mem::size_of::<Bad>(), 24);
    assert_eq!(mem::size_of::<Good>(), 16);
    // 经验法则：按字段大小降序排列
}
```

## 3. 分配策略 —— 少分配 = 更快

### 3.1 `Vec::with_capacity` —— 最便宜的优化

```rust
// ❌ 默认构造的 Vec 从容量 0 开始，每次 push 可能触发重新分配
fn build_bad(n: usize) -> Vec<i32> {
    let mut v = Vec::new();
    for i in 0..n {
        v.push(i as i32); // ← 可能触发多次 realloc + 数据拷贝
    }
    v
}

// ✅ 预分配容量，只分配一次
fn build_good(n: usize) -> Vec<i32> {
    let mut v = Vec::with_capacity(n);
    for i in 0..n {
        v.push(i as i32); // ← 不会触发 realloc
    }
    v
}

// 性能差异（n = 1,000,000）：
// Vec::new():    8.2ms（多次 realloc）
// with_capacity: 2.1ms（一次分配）
// 提升：3.9x
```

### 3.2 `String::with_capacity` —— 同样适用于 String

```rust
// 构建 JSON 字符串时，预估大小可以大幅减少分配次数
fn build_json_bad(users: &[User]) -> String {
    let mut s = String::new();
    s.push('[');
    for user in users {
        s.push_str(&format!(r#"{{"name":"{}"}}"#, user.name));
    }
    s.push(']');
    s
}

fn build_json_good(users: &[User]) -> String {
    // 预估：每个 user 约 30 字节
    let mut s = String::with_capacity(users.len() * 30);
    s.push('[');
    for (i, user) in users.iter().enumerate() {
        if i > 0 { s.push(','); }
        use std::fmt::Write;
        write!(s, r#"{{"name":"{}"}}"#, user.name).unwrap();
    }
    s.push(']');
    s
}
```

### 3.3 Arena/Bump Allocator —— 批量分配，批量释放

```rust
use bumpalo::Bump;

// ━━━ Bump Allocator 的核心思想 ━━━
// 不是每次需要内存时调用 malloc
// 而是一次性分配一大块内存
// 然后在这块内存上"切"出小块给各个对象
// 释放时一次性释放整块

// 适用场景：
//   - 编译器（解析期间的 AST 节点，编译结束后全部释放）
//   - HTTP 请求处理（请求期间的临时对象，响应返回后全部释放）
//   - 游戏帧循环（每帧内的临时对象，帧结束后释放）

fn arena_example() {
    let bump = Bump::new();

    // 在 arena 上分配一个 Vec
    let vec: &mut Vec<i32> = bump.alloc(Vec::new());
    for i in 0..1000 {
        vec.push(i);
    }

    // 分配自定义结构体
    let node = bump.alloc(MyAstNode {
        tag: "expression",
        children: bump.alloc(vec![]),
    });

    // bump 离开作用域时，所有在这个 arena 上分配的内存一次性释放
    // 不需要逐个 drop！
}
```

### 3.4 Object Pool —— 复用热对象

```rust
// ━━━ 对象池模式 ━━━
// 对于频繁创建和销毁的对象（如缓冲区、连接），不要每次都 new/drop
// 而是从一个池里取用和归还

use std::collections::VecDeque;

struct BufferPool {
    buffers: VecDeque<Vec<u8>>,
    buffer_size: usize,
}

impl BufferPool {
    fn new(initial_count: usize, buffer_size: usize) -> Self {
        let buffers = (0..initial_count)
            .map(|_| Vec::with_capacity(buffer_size))
            .collect();
        Self { buffers, buffer_size }
    }

    fn acquire(&mut self) -> Vec<u8> {
        self.buffers.pop_front()
            .unwrap_or_else(|| Vec::with_capacity(self.buffer_size))
    }

    fn release(&mut self, mut buffer: Vec<u8>) {
        buffer.clear(); // 保留容量，清空内容
        self.buffers.push_back(buffer);
    }
}
```

## 4. 算法优化

### 4.1 Iterator 链 vs 手写循环

```rust
// ━━━ Iterator 的性能通常不输手写循环 ━━━
// 事实上，Iterator 链更利于编译器优化（特别是 LLVM 的循环融合）

fn sum_of_squares_iter(v: &[i32]) -> i32 {
    v.iter().map(|x| x * x).filter(|x| x % 2 == 0).sum()
}

fn sum_of_squares_loop(v: &[i32]) -> i32 {
    let mut sum = 0;
    for &x in v {
        let sq = x * x;
        if sq % 2 == 0 {
            sum += sq;
        }
    }
    sum
}
// 两者性能几乎相同——选择更清晰的写法

// ━━━ Iterator 的技巧：collect 之前问自己"真的需要吗？" ━━━
// ❌ 不必要的 collect
fn find_first_bad(v: &[i32]) -> Option<i32> {
    let squares: Vec<i32> = v.iter().map(|x| x * x).collect(); // ← 分配整个 Vec
    squares.into_iter().find(|x| x > &100)                      // ← 只取第一个
}

// ✅ 惰性求值
fn find_first_good(v: &[i32]) -> Option<i32> {
    v.iter().map(|x| x * x).find(|x| x > &100)
    // 不需要 collect！map 是惰性的，find 找到了就停止
}
```

### 4.2 分支预测友好

```rust
// ━━━ CPU 的分支预测器喜欢"规律" ━━━
// 随机的 if/else 模式会让分支预测器频繁失败

// ❌ 随机的分支（branch-miss 率高）
fn process_random(data: &[i32]) -> i32 {
    let mut sum = 0;
    for &x in data {
        if x > 0 {  // ← 如果 x 的值是随机的，每次预测成功率 ~50%
            sum += x;
        }
    }
    sum
}

// ✅ 先排序再处理（branch-miss 率降低）
fn process_sorted(data: &[i32]) -> i32 {
    let mut sorted = data.to_vec();
    sorted.sort_unstable();  // ← 排序后，if x > 0 的模式变成"一段 false + 一段 true"
    let mut sum = 0;
    for &x in &sorted {
        if x > 0 {
            sum += x;
        }
    }
    sum
}
// 性能提升：对于随机数据，排序后的版本可以快 3-6 倍！
// （当然，排序本身也有开销——但如果数据需要用多次，排序一次是值得的）

// ━━━ 无分支写法 ━━━
// 如果分支确实是性能瓶颈，考虑用无分支的写法：
fn process_branchless(data: &[i32]) -> i32 {
    let mut sum = 0;
    for &x in data {
        // (x > 0) 转成 0 或 1，乘以 x
        sum += (x > 0) as i32 * x;
    }
    sum
}
// 注意：无分支版本的性能取决于数据模式，不是总更快
```

### 4.3 `sort_unstable` vs `sort`

```rust
// sort_unstable() 使用 pdqsort（模式击败快速排序）
// sort() 使用 Timsort（基于归并排序）
// sort_unstable 通常更快，但不保证相等元素的相对顺序

v1.sort();           // Timsort: 稳定排序，相等元素保持原顺序
v1.sort_unstable();  // pdqsort: 不稳定，但通常快 20-30%
```

## 5. `#[inline]` 的正确姿势

### 5.1 内联的真实成本

```rust
// ━━━ #[inline] 不是免费的性能提升 ━━━
// 内联 = 把函数体复制到调用处
// → 好处：省去函数调用开销（push/pop/call 指令）
// → 代价：代码膨胀（同一个函数体在 N 个调用处被复制了 N 次）
//        → i-cache 压力增大
//        → 编译时间增加

// ━━━ 编译器已经做了很好的内联决策 ━━━
// LLVM 的内联器根据函数大小、调用频率、代码大小预算等因素自动决策
// 大多数情况下你不需要手动干预

// ━━━ 什么时候加 #[inline]？ ━━━
// 1. 非常小的函数（如 getter/setter）被调用在非本 crate 的代码中
#[inline]
pub fn is_valid(self) -> bool {
    self.status == Status::Active  // ← 跨 crate 内联：非本 crate 调用时编译器不内联
}

// 2. 包装函数（只是简单转发到另一个函数）
#[inline]
pub fn parse(s: &str) -> Result<Value> {
    crate::parser::parse_value(s)  // ← 告诉编译器：把调用这个函数替换为直接调用 parse_value
}

// ━━━ 什么时候加 #[inline(always)]？ ━━━
// 极少使用！只在以下情况：
// 1. 你绝对确定内联是正确选择（性能测试证明）
// 2. 编译器因为代码大小预算拒绝内联，但你确实需要内联（如 SIMD 的内联）

// ━━━ 什么时候加 #[inline(never)]？ ━━━
// 1. 冷路径函数（错误处理、初始化、日志），不希望它们膨胀热代码
// 2. 需要函数调用帧出现在 backtrace 中（调试目的）
```

### 5.2 跨 crate 内联的重要性

```rust
// ━━━ 在库中，小函数应该标记 #[inline] ━━━
// 不标记的话，只有在 LTO 开启时才会跨 crate 内联

// 例如：一个被广泛使用的数学库
impl Vector3 {
    #[inline]
    pub fn dot(&self, other: &Vector3) -> f64 {
        self.x * other.x + self.y * other.y + self.z * other.z
    }
    // ↑ 只有 3 次乘法 + 2 次加法
    // 如果每次调用都要函数调用开销，性能影响可能很大
    // 标了 #[inline] 后，调用处直接内联为算术指令
}
```

## 6. 字符串优化

### 6.1 `Cow<str>` —— 零拷贝的拥有/借用双模

```rust
use std::borrow::Cow;

// ━━━ Cow<str> 在"大多数时候不需要修改"的场景中消除分配 ━━━
fn process_text(input: &str) -> Cow<str> {
    if input.contains("bad_word") {
        // 需要修改时才分配
        Cow::Owned(input.replace("bad_word", "****"))
    } else {
        // 不需要修改时零拷贝
        Cow::Borrowed(input)
    }
}

// 性能影响：如果 95% 的输入不需要修改，避免了 95% 的分配
```

### 6.2 `smallstr` / `smartstring` —— 内联短字符串

```rust
use smartstring::SmartString;

// ━━━ SmartString：短字符串内联在栈上 ━━━
// 大多数字符串都很短（用户名、标签、国家代码等）
// SmartString 在 ≤ 23 字节时内联在栈上
let short: SmartString = "hello".into(); // ← 栈上，零分配
let long: SmartString = "a very long string that exceeds the inline limit...".into();
// ← 堆分配（和普通 String 一样）
```

### 6.3 String interning —— 相同字符串共享内存

```rust
// ━━━ String Interning ━━━
// 场景：你有成千上万个字符串，其中很多是重复的（如 HTTP 头部名称、数据库列名）
// String Interning 为每个唯一字符串分配一次内存，重复的共享指针

// 用例：编译器中的标识符
use lasso::Rodeo;

let mut interner = Rodeo::new();
let id1 = interner.get_or_intern("user_id");   // 分配
let id2 = interner.get_or_intern("username");  // 分配
let id3 = interner.get_or_intern("user_id");   // ← 返回 id1，不分配！

// 好处：
// 1. 节省内存（相同字符串只存一份）
// 2. 字符串比较变成整数比较（O(1) 的 ==）
// 3. 哈希变成整数哈希（比字符串哈希快很多）
```

## 8. 优化优先级总结

```
ROI 从高到低的优化：

1. 数据结构选择（HashMap vs BTreeMap，Vec vs SmallVec）
   → 影响可达 3-10x，纯代码修改，无副作用

2. 预分配（Vec::with_capacity，String::with_capacity）
   → 影响可达 2-5x，只加一行代码

3. 避免不必要的 collect() 和 clone()
   → 影响可达 2-3x，代码可能更简洁

4. 内存布局（SoA vs AoS，字段顺序）
   → 影响可达 2-7x，但修改较大（API 可能变化）

5. Hash 函数选择
   → 影响可达 2-4x，但只在 HashMap-heavy 代码中

6. 算法改进（排序、查找算法）
   → 影响取决于问题，O(n²) → O(n log n) 惊天动地

7. String 优化（Cow，SmartString，interning）
   → 影响可达 1.5-3x，在字符串密集型代码中

8. Arena / Object Pool
   → 影响可达 1.5-3x，但增加复杂度

9. #[inline] 提示
   → 影响通常 1-10%，谨慎使用
```

## 10. 练习答案

```rust
// 优化 1：预分配容量
fn process_users(names: Vec<String>) -> Vec<String> {
    let mut results = Vec::with_capacity(names.len());  // ← 预分配
    for name in names {
        // 优化 2：用 format! 或 write! 代替多次 push_str
        // 优化 3：用 with_capacity 预估 String 大小
        let mut processed = String::with_capacity(7 + name.len() + 9);
        use std::fmt::Write;
        write!(processed, "User: {} (active)", name).unwrap();
        results.push(processed);
    }
    results
}

// 优化：用 HashMap 代替线性查找
use std::collections::HashMap;

fn lookup_user(users: &HashMap<String, u64>, target: &str) -> Option<u64> {
    users.get(target).copied()  // O(1) vs O(n)
}
```
