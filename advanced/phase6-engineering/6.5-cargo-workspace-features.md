# 6.5 Cargo 工作空间与 Feature 工程 —— 管理多 crate 项目

> 当你只有一个 `main.rs` 时，项目结构不是问题。当你有了三个 crate（核心库、CLI 工具、Web 服务），五个 feature flag（`serde`、`cli`、`server`、`tls`、`metrics`），和二十个依赖时——如果没有任何组织策略，`cargo build` 会变成一场噩梦：每个 crate 独立编译依赖、feature 组合爆炸导致 CI 跑一小时、依赖版本不一致引入诡异的编译错误。Cargo workspace 和 feature flags 是 Rust 为多 crate 项目提供的两把手术刀——用得好的话，你的项目结构像积木一样清晰；用不好的话，你会花一半的时间在和 Cargo 作斗争。本节不是教你"怎么创建 workspace"，而是教你"怎么设计一个让三个月后的自己不会恨你的项目结构"。

---

## 1. Cargo Workspace —— 多 crate 的统一管理

### 1.1 什么时候需要 Workspace？

```
单 crate 项目（不需要 workspace）：
  my_project/
    Cargo.toml
    src/
      main.rs
      lib.rs

多 crate 项目（需要 workspace）：
  my_project/
    Cargo.toml          ← workspace 根配置
    crates/
      my-core/          ← 核心库（纯逻辑，无 I/O）
        Cargo.toml
        src/lib.rs
      my-cli/           ← CLI 工具（依赖 my-core）
        Cargo.toml
        src/main.rs
      my-server/        ← HTTP 服务（依赖 my-core）
        Cargo.toml
        src/main.rs
```

**触发 workspace 的信号**：
- 你有 3 个以上 crate 共享相同的依赖
- 你想同时发布多个 crate（如 `serde` + `serde_json` + `serde_derive`）
- 你希望所有 crate 共享一个 `Cargo.lock`（保证版本一致）
- 你的 CI 希望一次构建所有 crate（而非逐个 `cd` 进去 `cargo build`）

### 1.2 Workspace 根配置

```toml
# Cargo.toml（项目根目录 —— workspace 配置）
[workspace]
members = [
    "crates/my-core",
    "crates/my-cli",
    "crates/my-server",
]

# 还可以用 glob 模式
# members = ["crates/*"]

resolver = "2"  # 使用 Rust 2021 edition 的依赖解析器

# ━━━ workspace.dependencies —— 统一依赖版本 ━━━
[workspace.dependencies]
serde = { version = "1", features = ["derive"] }
tokio = { version = "1", features = ["full"] }
anyhow = "1"
thiserror = "1"
tracing = "0.1"

# ━━━ workspace.package —— 统一 crate 元数据 ━━━
[workspace.package]
version = "0.1.0"
edition = "2021"
license = "MIT"
repository = "https://github.com/me/my_project"
```

```toml
# crates/my-core/Cargo.toml —— 子 crate 使用 workspace 依赖
[package]
name = "my-core"
version.workspace = true   # ← 引用 workspace.package.version
edition.workspace = true   # ← 引用 workspace.package.edition

[dependencies]
serde.workspace = true     # ← 引用 workspace.dependencies.serde
tokio.workspace = true
thiserror.workspace = true
```

**`workspace.dependencies` 的价值**：
- 所有 crate 共享同一版本的 `serde` → 不会出现 crate A 依赖 `serde 1.0.100`，crate B 依赖 `serde 1.0.200` 的"依赖地狱"
- 升级依赖时只改一处（根 `Cargo.toml`），所有子 crate 自动跟随
- 新 crate 加入时直接 `serde.workspace = true`，不需要查版本号

### 1.3 Workspace 内的 crate 间依赖

```toml
# crates/my-cli/Cargo.toml
[package]
name = "my-cli"
version.workspace = true
edition.workspace = true

[dependencies]
my-core = { path = "../my-core" }  # ← 引用 workspace 内的另一个 crate
# 依赖关系：my-cli → my-core → serde, tokio, ...
clap = { version = "4", features = ["derive"] }  # CLI 专用依赖
```

### 1.4 Workspace 中的共享 target/ 目录

```
没有 workspace：
  my-core/target/   ← 独立编译，600MB
  my-cli/target/    ← 独立编译，800MB（重复编译了 my-core 的依赖）
  my-server/target/ ← 独立编译，800MB

有 workspace：
  target/           ← 共享编译，1.2GB（依赖只编译一次）
  ↑ 所有 crate 共享同一个 target/ 目录
  ↑ cargo build/check/test 在 workspace 根目录运行
```

## 2. Feature Flags 设计 —— 让用户只编译需要的

### 2.1 Feature 是什么？

```toml
# Cargo.toml
[features]
default = ["std"]       # 默认开启的 features
std = []                # 使用 std 库
serde = ["dep:serde"]   # 开启 serde 支持
tls = ["tokio-native-tls"]  # 开启 TLS

[dependencies]
serde = { version = "1", optional = true }  # optional = 不是必须依赖
tokio-native-tls = { version = "0.3", optional = true }
```

```rust
// src/lib.rs

// ━━━ 条件编译 ━━━
#[cfg(feature = "serde")]
use serde::{Serialize, Deserialize};

#[cfg(feature = "serde")]
#[derive(Serialize, Deserialize)]
pub struct MyStruct {
    pub name: String,
}

// 如果没有开启 serde feature，这段代码不会编译
// 用户看不到 MyStruct，二进制里也没有 serde 的代码

#[cfg(feature = "tls")]
pub fn connect_secure(url: &str) {
    // 使用 TLS 连接
}

#[cfg(not(feature = "tls"))]
pub fn connect_secure(url: &str) {
    compile_error!("TLS feature is required for secure connections");
}
```

### 2.2 Feature 的设计原则

```toml
# ━━━ 好的 feature 设计 ━━━
[features]
# 名词式：一个 feature 对应一个功能
default = ["std"]
std = []
json = ["dep:serde_json"]
yaml = ["dep:serde_yaml"]
tls = ["dep:rustls"]
async = ["dep:tokio"]

# ━━━ 坏的 feature 设计 ━━━
[features]
# 动词式/模糊式：feat 的名字说不清它做什么
extended = []     # "extended" 是什么？
v2 = []           # "v2" 是 API 版本吗？
experimental = [] # 哪些功能是实验性的？
```

**设计原则**：
1. **用名词命名**：feature 代表一个可选的**功能**，不是一种**配置模式**
2. **一个 feature = 一个明确的用户可见的功能**：`json`、`tls`、`async`
3. **避免互斥 features**：不要设计"开了 `fast` 就不能开 `safe`"的 feature——用 `cfg` 宏做编译期检查
4. **default 要保守**：默认 features 应该只包含最基本的功能

### 2.3 互斥 Features 的处理

```rust
// ━━━ 编译期检查互斥 features ━━━

// 方案 1：用 cfg 宏在编译期报错
#[cfg(all(feature = "rustls", feature = "native-tls"))]
compile_error!("rustls and native-tls are mutually exclusive");

// 方案 2：选择一个默认
#[cfg(all(feature = "rustls", feature = "native-tls"))]
use rustls as tls_backend;

#[cfg(all(feature = "rustls", not(feature = "native-tls")))]
use rustls as tls_backend;

#[cfg(all(not(feature = "rustls"), feature = "native-tls"))]
use native_tls as tls_backend;
```

### 2.4 `cargo-hack` —— 测试所有 feature 组合

```bash
# 安装
cargo install cargo-hack

# 测试所有 feature 组合
cargo hack test --feature-powerset
# --feature-powerset 测试所有 feature 的幂集（所有可能的组合！）

# 排除某些已知不可行的组合
cargo hack test --feature-powerset \
    --skip "rustls native-tls"  # ← rustls 和 native-tls 不能同时开启

# 检查所有 feature 组合下是否都能编译
cargo hack check --feature-powerset
```

**为什么需要 `cargo-hack`？**
```
你的 Cargo.toml 有 4 个可选 features:
  json, yaml, tls, async

可能的组合数：2^4 = 16 种

你手动测试了：json、yaml、json+yaml、tls、...
你忘了测试：json+tls、yaml+async+tls、无 feature

cargo-hack 不会忘——它穷举所有组合
```

### 2.5 Feature 文档化

```rust
// src/lib.rs 顶部或模块文档

//! # Features
//!
//! - **`json`** (默认): 启用 JSON 序列化/反序列化支持
//! - **`yaml`**: 启用 YAML 支持
//! - **`tls`**: 启用 TLS 加密连接（基于 rustls）
//! - **`async`**: 启用异步 API（需要 tokio runtime）
//!
//! ## 示例
//!
//! ```toml
//! [dependencies]
//! my_crate = { version = "1", features = ["json", "tls"] }
//! ```
```

## 3. 项目结构最佳实践

### 3.1 推荐的 Workspace 布局

```
my_project/
  Cargo.toml              ← workspace 根
  Cargo.lock              ← 共享的 lock 文件
  README.md
  .github/
    workflows/
      ci.yml              ← CI 配置

  crates/
    my-core/              ← 核心库（最底层，尽量少依赖）
      Cargo.toml
      src/
        lib.rs
        models.rs
        error.rs

    my-cli/               ← CLI 工具
      Cargo.toml
      src/
        main.rs
        commands/
          serve.rs
          migrate.rs

    my-server/            ← HTTP 服务
      Cargo.toml
      src/
        main.rs
        routes/
        middleware/

  tests/                  ← 集成测试（跨 crate）
    integration_test.rs

  examples/               ← 使用示例
    basic_usage.rs
```

### 3.2 依赖层级原则

```
依赖应该形成单向的有向无环图（DAG）：

❌ 循环依赖：
  my-core → my-utils → my-core  // ← 编译会报错

✅ 分层依赖：
  my-server ──┐
              ├──→ my-core ──→ serde, thiserror
  my-cli ─────┘
              └──→ clap, indicatif (各自的专用依赖)

✅ 依赖方向：上层 → 下层（单向）
```

## 4. 练习

### 4.1 练习：设计一个 Workspace

```
场景：你要做一个"代码统计工具"（类似 tokei），包含：
  - 核心库：文件扫描、语言检测、代码行统计
  - CLI 工具：命令行界面
  - 可选功能：JSON 输出、YAML 输出、多线程分析

任务：
1. 设计 workspace 结构（画出 crate 依赖图）
2. 编写根 Cargo.toml（workspace 配置 + workspace.dependencies）
3. 设计 feature flags（哪些 feature？默认开哪些？）
4. 写一个子 crate 的 Cargo.toml（引用 workspace 依赖）
```

## 5. 练习答案

```toml
# 根 Cargo.toml
[workspace]
members = ["crates/code-stats-core", "crates/code-stats-cli"]
resolver = "2"

[workspace.dependencies]
serde = { version = "1", features = ["derive"] }
serde_json = "1"
serde_yaml = "0.9"
rayon = "1"
clap = { version = "4", features = ["derive"] }
thiserror = "1"

[workspace.package]
version = "0.1.0"
edition = "2021"
license = "MIT"
```

```toml
# crates/code-stats-core/Cargo.toml
[package]
name = "code-stats-core"
version.workspace = true
edition.workspace = true

[features]
default = ["json"]
json = ["dep:serde_json"]
yaml = ["dep:serde_yaml"]
parallel = ["dep:rayon"]

[dependencies]
serde.workspace = true
serde_json = { workspace = true, optional = true }
serde_yaml = { workspace = true, optional = true }
rayon = { workspace = true, optional = true }
thiserror.workspace = true
```

```
依赖图：
  code-stats-cli
    ├── code-stats-core
    │   ├── serde
    │   ├── serde_json (optional)
    │   ├── serde_yaml (optional)
    │   └── rayon       (optional)
    └── clap
```
