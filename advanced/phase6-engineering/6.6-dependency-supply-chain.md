# 6.6 依赖管理与供应链安全 —— 你的 `Cargo.toml` 不是你的代码

> 你写的代码只占最终二进制文件的 10%——另外 90% 来自你的依赖。这 90% 你基本上看都没看过：它有安全漏洞吗？它用了什么许可证？它会不会在下个 minor 版本中破坏你的 API？它是不是已经被作者遗弃了？Rust 的包管理体系（crates.io + Cargo）在可用性上极其出色——一个 `cargo add` 就能引入任何 crate。但这种便利性也带来了风险：你的依赖树可能有 200 个 crate，其中任何一个出了安全问题都可能是你的问题。本节教你建立一套"依赖卫生"流程——不是让你审查每一行依赖代码（那不现实），而是用自动化工具持续监控依赖的健康状况，在有问题的第一时间收到警报。

## 1. `cargo-audit` —— 检测已知安全漏洞

### 1.1 快速开始

```bash
# 安装
cargo install cargo-audit

# 扫描当前项目的依赖
cargo audit

# 输出示例（你的项目安全）：
# Success - No vulnerable packages found

# 输出示例（发现漏洞）：
# Crate:     time
# Version:   0.1.44
# Title:     Potential segfault in the time crate
# Date:      2020-11-18
# ID:        RUSTSEC-2020-0071
# URL:       https://rustsec.org/advisories/RUSTSEC-2020-0071
# Solution:  Upgrade to >=0.2.23
```

### 1.2 工作原理

`cargo-audit` 查询 [RustSec Advisory Database](https://github.com/RustSec/advisory-db/)——一个社区维护的 Rust crate 已知漏洞数据库。当你的依赖版本与数据库中记录的受影响版本匹配时，它会发出警告。

### 1.3 在 CI 中运行

```yaml
# .github/workflows/security.yml
name: Security Audit
on:
  schedule:
    - cron: '0 0 * * *'  # 每天运行
  push:
    paths:
      - '**/Cargo.toml'
      - '**/Cargo.lock'

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - run: cargo install cargo-audit
      - run: cargo audit
```

## 2. `cargo-deny` —— 许可证合规与依赖策略

### 2.1 `cargo-audit` 只能做安全，`cargo-deny` 能做更多

```bash
cargo install cargo-deny

# 初始化配置
cargo deny init
# → 生成 deny.toml（配置文件）
```

### 2.2 三大功能

```toml
# deny.toml

# ━━━ 1. 安全漏洞（类似 cargo-audit）━━━
[advisories]
vulnerability = "deny"   # 有已知漏洞 → 拒绝
unmaintained = "warn"    # 未维护的 crate → 警告
yanked = "deny"          # 已被 yank 的版本 → 拒绝

# ━━━ 2. 许可证合规 ━━━
[licenses]
unlicensed = "deny"      # 没有许可证 → 拒绝
allow = [
    "MIT",
    "Apache-2.0",
    "BSD-3-Clause",
    "ISC",
    "Unicode-DFS-2016",
]
copyleft = "deny"        # GPL/AGPL 等 copyleft 许可证 → 拒绝
# 注意：GPL 可能不适合你想保持闭源的商业项目

# 对特定 crate 例外
[[licenses.exceptions]]
name = "ring"
allow = ["ISC", "MIT", "OpenSSL"]

# ━━━ 3. 依赖来源与重复检查 ━━━
[sources]
unknown-registry = "deny"  # 未知来源的依赖 → 拒绝
unknown-git = "deny"       # 未指定的 git 依赖 → 拒绝

[bans]
multiple-versions = "deny" # 依赖树中有同一个 crate 的多个版本 → 拒绝
# 例如：crate A 依赖 serde 1.0.100，crate B 依赖 serde 1.0.200
# 这会导致 serde 被编译两次，增加二进制大小

# 禁止特定 crate
[[bans.deny]]
name = "openssl"           # ← 强制使用 rustls 而非 openssl

# 允许特定 crate 的重复
[[bans.skip]]
name = "syn"               # syn 随 proc-macro 版本变化是正常的
```

### 2.3 运行检查

```bash
# 检查所有三类问题
cargo deny check advisories
cargo deny check licenses
cargo deny check bans
```

## 3. `cargo-outdated` —— 保持依赖更新

### 3.1 检查过时依赖

```bash
cargo install cargo-outdated

cargo outdated
# 输出示例：
# Name         Project  Compat  Latest   Kind    Platform
# serde        1.0.100  1.0.200 1.0.200  Normal  ---
# tokio        1.20.0   1.35.0  1.36.0   Normal  ---
# clap         3.2.0    3.2.25  4.4.0    Normal  ---
# ...
# "Compat" = 最新的兼容版本（按照你的 Cargo.toml 里的 semver 约束）
# "Latest" = 绝对最新版本（可能包含 breaking changes）
```

### 3.2 更新策略

```
依赖更新的节奏：

🚨 安全补丁：立即更新（当天）
  → cargo-audit 报警 → cargo update -p vulnerable_crate

🔄 兼容更新（minor/patch）：定期批量更新（每两周或每月）
  → cargo update → 跑测试 → 合并

⏳ Major 版本升级：计划性升级（每季度或半年）
  → 阅读 CHANGELOG → 估算工作量 → 专门的分支 + 完整测试

⚠️ 不要做的事情：
  - "永远不升级" → 安全漏洞积累
  - "每次发布都升到最新" → breaking changes 让你疲于奔命
  - "闭着眼睛 cargo update" → 不看 changelog 的升级是赌博
```

## 4. Semver 在 Rust 中的实际应用

### 4.1 Rust 的 Semver 约定

```
版本号：MAJOR.MINOR.PATCH（例如 1.2.3）

MAJOR：不兼容的 API 修改
  → 1.2.3 → 2.0.0：你的代码可能需要修改才能编译

MINOR：向后兼容的新功能
  → 1.2.3 → 1.3.0：新增了 API，但不影响现有代码

PATCH：向后兼容的 bug 修复
  → 1.2.3 → 1.2.4：只修复了 bug，没有任何新功能

Cargo.toml 中的版本约束：
  serde = "1.0"      → 等价于 >=1.0.0, <2.0.0（接受所有 1.x）
  serde = ">=1.0.100" → 明确的范围
  serde = "=1.0.100"  → 精确锁定（通常不推荐）
```

### 4.2 作为库作者：什么算 Breaking Change？

```rust
// 以下都是 MAJOR 版本变更（应该从 1.x → 2.0）：

// 1. 删除 pub 函数/类型/trait
// pub fn old_function() {}  → 删除

// 2. 修改 pub 函数签名
// pub fn process(data: &str) {} → pub fn process(data: String) {}

// 3. 在 trait 中增加方法（不带默认实现）
// pub trait Processor {
//     fn process(&self);         // ← 1.0 版本
//     fn validate(&self);        // ← 2.0 版本：所有实现者必须更新
// }

// 4. 修改 enum 变体
// pub enum Status { Active, Inactive }
// → pub enum Status { Active, Inactive, Suspended }  ← MAJOR！

// ━━━ 以下 NOT 是 Breaking Change ━━━
// 1. 增加新的 pub 函数/类型/trait
// 2. 在 trait 中增加带默认实现的方法
// 3. 修改私有代码（不影响公开 API）
// 4. 放宽 trait 约束（impl<T> → impl<T: Debug> 是 BREAKING！但反过来不是）
```

### 4.3 MSRV (Minimum Supported Rust Version)

```toml
# Cargo.toml
[package]
edition = "2021"
rust-version = "1.70"  # ← MSRV：你的 crate 需要 Rust 1.70+
```

**MSRV 策略建议**：
- 如果只是为了用一个新特性，不要急着提高 MSRV
- 可以参考"当前 stable - 2 个小版本"作为 MSRV（给用户缓冲）
- 如果你的 crate 很底层（如 `libc`），MSRV 应该尽量低

## 5. 自动依赖更新 —— Dependabot / Renovate

### 5.1 Dependabot（GitHub 内置）

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "cargo"
    directory: "/"
    schedule:
      interval: "weekly"      # 每周检查一次
    open-pull-requests-limit: 5
    labels:
      - "dependencies"
    # 只允许 patch 和 minor 自动更新
    versioning-strategy: auto
```

### 5.2 Renovate（更灵活的替代方案）

```json
// renovate.json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["config:base"],
  "packageRules": [
    {
      "matchManagers": ["cargo"],
      "matchUpdateTypes": ["patch"],
      "automerge": true       // patch 更新自动合并
    },
    {
      "matchManagers": ["cargo"],
      "matchUpdateTypes": ["minor"],
      "automerge": false      // minor 更新需要人工审查
    }
  ]
}
```

## 6. 最小化依赖原则

### 6.1 依赖是有成本的

```
每增加一个依赖，你都要付出：
  - 编译时间：更多 crate 要编译
  - 二进制大小：更多代码链接到最终二进制
  - 维护成本：需要跟踪更新、安全漏洞、breaking changes
  - 信任成本：你必须信任依赖作者的代码质量和安全意识
  - 供应链风险：依赖的依赖的依赖可能有漏洞
```

### 6.2 决策：引入依赖前问自己

```
1. 我能用 std 库实现吗？
   → 例如：用 std::sync::Mutex 而不是 parking_lot（除非你真的需要更快）

2. 我能用更轻量的替代吗？
   → 例如：用 ureq（HTTP 客户端）而不是 reqwest（如果你不需要 async）

3. 这个依赖被积极维护吗？
   → 检查：最后 commit 是什么时候？有多少 open issue？有响应吗？
   → 工具：crates.io 上的 "Last Updated"

4. 这个依赖的依赖树有多深？
   → 用 cargo tree 查看：cargo tree -e features
   → 如果一个小功能引入了 50 个传递依赖，三思

5. 我能自己写吗？
   → 如果功能只有 30 行代码，就不要引入一个 5000 行的 crate
   → 但如果你写的代码可能引入 bug（如密码学），用现成的审查过的 crate 更安全
```
