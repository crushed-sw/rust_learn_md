# 6.14 no_std 与嵌入式 Rust —— 当 std 不可用时

> 绝大多数 Rust 代码运行在有操作系统的环境里——Linux、macOS、Windows。但有的时候，你的代码跑在裸机上：一个 STM32 微控制器、一个 WebAssembly 模块、一个操作系统内核、一个 UEFI 引导程序。这些环境没有文件系统、没有 `HashMap`（需要堆分配器）、没有 `println!`（没有 stdout）——连 `main` 函数都不一定有。Rust 为这些场景提供了 `#![no_std]` 模式：stdlib 被移除，你只剩下 `core`（语言最核心的类型和 trait）和可选的 `alloc`（需要你提供堆分配器）。这节不是教你写一个完整的嵌入式项目——那是另一本书的内容——而是给你一个 no_std 世界的地图：什么可用、什么不可用、怎么填补缺口，以及在嵌入式 Rust 生态中如何找到自己的方向。

---

## 1. `core` vs `alloc` vs `std` —— 三层能力边界

```
┌────────────────────────────────────────────────────┐
│                      std                           │
│  ┌──────────────────────────────────────────────┐ │
│  │                    alloc                      │ │
│  │  ┌────────────────────────────────────────┐  │ │
│  │  │                core                     │  │ │
│  │  │                                        │  │ │
│  │  │  • 基本类型：i32, bool, str, ...       │  │ │
│  │  │  • 运算符重载：Add, Eq, PartialOrd     │  │ │
│  │  │  • Option / Result                      │  │ │
│  │  │  • 迭代器：Iterator trait               │  │ │
│  │  │  • Clone, Copy, Send, Sync              │  │ │
│  │  │  • 标记类型：PhantomData, NonNull       │  │ │
│  │  │  • 内存操作：mem::size_of, ptr::read    │  │ │
│  │  │  • 切片：&[T], &str 的方法             │  │ │
│  │  │  • 数字运算：pow, abs, to_be_bytes     │  │ │
│  │  │  • Future / async 基础                  │  │ │
│  │  └────────────────────────────────────────┘  │ │
│  │                                              │ │
│  │  • Vec<T>, String, Box<T>                    │ │
│  │  • Rc<T>, Arc<T>                             │ │
│  │  • BTreeMap, BTreeSet, BinaryHeap            │ │
│  │  • format!() 宏                               │ │
│  │  • LinkedList, VecDeque                      │ │
│  └──────────────────────────────────────────────┘ │
│                                                   │
│  • HashMap, HashSet (需要随机数种子)               │
│  • std::io (文件, 网络, 管道)                     │
│  • std::thread, std::sync::mpsc                  │
│  • std::time::Instant                            │
│  • std::env, std::fs                             │
│  • println!, eprintln!                            │
└────────────────────────────────────────────────────┘
```

**关键记忆**：
- `core` 有 `Iterator`、`Option`、`Result`、`Clone`、`Send`、`Sync`、`Future`
- `alloc` 有 `Vec`、`String`、`Box`、`Rc`、`Arc`、`format!`
- `std` 有所有剩下的：I/O、线程、网络、`HashMap`、`println!`

## 2. `#![no_std]` 入门

### 2.1 最小的 no_std 库

```rust
// src/lib.rs
#![no_std]  // ← 这一行告诉编译器：不链接 std

// ✅ core 中的东西可以直接用
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

pub fn first<T>(slice: &[T]) -> Option<&T> {
    slice.iter().next()
}

// ❌ std 中的东西不能用了
// use std::collections::HashMap;  // 编译错误！
// println!("hello");               // 编译错误！
```

### 2.2 no_std + alloc —— 需要堆空间

```rust
// src/lib.rs
#![no_std]

extern crate alloc;  // ← 声明使用 alloc crate

use alloc::vec::Vec;
use alloc::string::String;
use alloc::format;

pub fn build_greeting(name: &str) -> String {
    format!("Hello, {}!", name)  // ✅ format! 来自 alloc
}

pub fn collect_even(n: u32) -> Vec<u32> {
    (0..n).filter(|x| x % 2 == 0).collect()  // ✅ Vec 来自 alloc
}
```

### 2.3 自定义 panic handler

```rust
// src/lib.rs
#![no_std]

use core::panic::PanicInfo;

/// 在 no_std 环境中，你必须自己定义 panic 的行为
#[panic_handler]
fn panic(_info: &PanicInfo) -> ! {
    // 在裸机环境中，通常：
    // 1. 闪烁 LED
    // 2. 写入调试日志
    // 3. 无限循环（或重启 CPU）
    loop {
        // 可能触发一个 breakpoint
        unsafe { core::arch::asm!("bkpt") };
    }
}
```


## 3. `heapless` —— 无堆数据结构

### 3.1 当 alloc 也不可用时

```rust
// heapless 的所有数据结构都在栈上分配，不需要堆
// 代价：需要提前指定最大容量

use heapless::Vec;       // ← heapless::Vec，不是 std::Vec
use heapless::String;    // ← heapless::String
use heapless::LinearMap; // ← heapless 的 HashMap

// 固定容量 32 的 Vec
let mut v: Vec<i32, 32> = Vec::new();  // 最多 32 个元素，全在栈上
v.push(1).unwrap();
v.push(2).unwrap();
// v.push(3).unwrap(); // ... 到第 33 个时返回 Err

// 固定容量 64 字节的 String
let mut s: String<64> = String::new();
use core::fmt::Write;
write!(s, "Hello, {}!", "world").unwrap();
```

**heapless 的优势**：
- 零堆分配（所有数据在栈上）
- 适合确定性实时系统（没有 malloc 的不确定性）
- 容量溢出是可预测的（返回 `Err` 而不是 panic）

## 4. 嵌入式 Rust 生态概览

### 4.1 embedded-hal —— 硬件抽象层

```rust
// embedded-hal 定义了 GPIO、I2C、SPI、UART 等硬件接口的 trait
// 具体芯片的 crate 实现这些 trait

use embedded_hal::digital::v2::OutputPin;

// 你的代码只依赖 trait，不依赖具体芯片
fn blink<P: OutputPin>(led: &mut P, delay: &mut impl DelayMs<u32>) {
    loop {
        led.set_high().unwrap();
        delay.delay_ms(500u32);
        led.set_low().unwrap();
        delay.delay_ms(500u32);
    }
}
// 这个 blink 函数可以跑在 STM32、nRF52、ESP32 等任何实现了 OutputPin 的芯片上！
```

### 4.2 RTIC —— 实时并发框架

```rust
// RTIC (Real-Time Interrupt-driven Concurrency)
// 基于硬件中断的并发模型

// #[rtic::app] 宏生成 main、中断处理、资源管理等所有样板代码
// 特点：
//   - 编译期保证无数据竞争（资源通过优先级天花板协议共享）
//   - 极低的开销（无 RTOS 的上下文切换）
//   - 确定性延迟（最高优先级中断总是立即响应）
```

### 4.3 `defmt` —— 嵌入式日志

```rust
// defmt 是嵌入式优化的日志框架
// 常规的 format! 在嵌入式上太重了（格式字符串很大）
// defmt 将格式字符串压缩为索引，在主机端解码

use defmt::{info, error};

fn process_sensor(reading: u16) {
    info!("sensor reading: {}", reading);
    if reading > 1000 {
        error!("reading out of range!");
    }
}
// 字符串 "sensor reading: {}" 被压缩为一个字节的索引
// 实际的字符串存储在主机端的 ELF 文件中
```

### 4.4 `probe-rs` —— 调试与烧录

```bash
# 通过调试探针烧录和调试嵌入式程序
probe-rs run --chip STM32F407VG target/thumbv7em-none-eabihf/debug/my_app
# 不需要 OpenOCD、不需要 GDB server！
# probe-rs 直接通过 CMSIS-DAP / J-Link / ST-Link 与芯片通信
```

## 5. no_std 中的 std 替代方案

```
你需要 std::X         → no_std 替代方案
───────────────────────────────────────────────────────
println!               → defmt (嵌入式), ufmt (轻量)
format!                → alloc::format (需要 alloc)
HashMap                → heapless::LinearMap, hashbrown
Vec                    → heapless::Vec, tinyvec
Mutex                  → cortex_m::interrupt::Mutex (嵌入式)
thread                 → RTIC tasks (嵌入式), embassy (异步)
Instant::now()         → 硬件定时器 + embedded-hal
fs::read_to_string     → 不适用（没有文件系统）
TcpStream              → embedded-nal (嵌入式网络抽象)
```
