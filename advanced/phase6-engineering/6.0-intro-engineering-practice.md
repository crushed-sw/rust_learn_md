# 6.0 工程实践概述 —— 从"能写对的代码"到"能领导项目的技术架构"

> 前五个 Phase 你学的是 Rust 语言本身——类型系统、内存控制、并发异步、底层突破、元编程。你学会了如何用 Rust 表达想法，如何在编译期捕获错误，如何写出零成本抽象的代码。但语言能力只是工程师能力的一半。另一半是：**如何让一段代码变成可维护、可观测、高性能、能持续演进的软件产品**。这就是 Phase 6 要解决的问题。它不再问你"这段代码对吗？"——它问你"这段代码三个月后还好改吗？出问题时几分钟内能定位？CI 跑一次要多久？依赖有安全漏洞怎么办？发布流程能不能一键完成？" 这是从"匠人"到"架构师"的思维转换。

## 1. 为什么工程实践被放在 Phase 6？

### 1.1 知识栈的依赖关系

```
你的 Rust 知识栈（从底层到上层）
┌────────────────────────────────────────────────────┐
│  Phase 6: 工程实践 ← 你现在在这里                    │
│  测试 · 错误 · 观测 · CI/CD · 性能 · API · 分发     │
├────────────────────────────────────────────────────┤
│  Phase 5: 元编程                                    │
│  Phase 4: 底层突破                                  │
│  Phase 3: 并发异步          语言能力层               │
│  Phase 2: 内存控制                                  │
│  Phase 1: 类型系统                                  │
└────────────────────────────────────────────────────┘
```

工程实践在语言能力之上。**你不能在不懂类型系统的时候设计好的 API——因为你不理解泛型、trait 和生命周期的表达能力；你不能在不懂并发模型的时候做性能优化——因为你不理解 async runtime 的开销和锁竞争的本质；你不能在不懂 unsafe 的时候深入 profiling——因为你不知道编译器优化在底层做了什么。**

这就是 Phase 6 被放在最后的原因：**它需要前面所有 Phase 的知识作为基础**，但它教授的不是语言特性本身，而是如何将这些特性组合成一个可维护的系统。

### 1.2 "能写出正确代码"和"能交付可维护项目"之间差了多远？

下面是一个典型的差距对照表：

| 维度 | 能写对的代码 | 能交付的项目 |
|------|------------|------------|
| **测试** | 写了几个 `#[test]` | 单元+集成+属性+模糊测试全覆盖，CI 中自动运行 |
| **错误** | `unwrap()` / `?` 往上抛 | 结构化错误类型，有上下文，可追溯，可恢复 |
| **日志** | `println!` 调试 | 结构化 tracing，按级别过滤，生产环境可查询 |
| **项目** | 一个 `main.rs` | Cargo workspace，feature flags 精细控制，模块边界清晰 |
| **性能** | "跑得挺快" | Criterion 基准，perf 火焰图，PGO 优化，有量化指标 |
| **CI/CD** | 手动 `cargo test` | GitHub Actions 自动测试+格式+安全检查+发布 |
| **API** | "能用就行" | 遵循 API Guidelines，考虑演化，有 deprecation 路径 |
| **分发** | `cargo build --release` | 交叉编译 × 多平台，Docker 镜像优化，docs.rs 文档 |

**Phase 6 的目标就是填平这个差距。**

## 3. 五大支柱详解

### 3.1 质量保证（6.1-6.2）

代码的正确性不靠"我觉得对"，靠测试来证明。Rust 的测试体系远超 `#[test]`：

```
测试金字塔
         ┌──────┐
         │ Fuzz │  ← 少量：模糊测试找未知边界
        ┌┴──────┴┐
        │  Miri  │  ← 少量：MIR 层面检测 UB
       ┌┴────────┴┐
       │ Property │  ← 中量：属性测试自动生成用例
      ┌┴──────────┴┐
      │ Integration│  ← 中量：集成测试验证组件协作
     ┌┴────────────┴┐
     │    Unit       │  ← 大量：单元测试覆盖每个函数
     └───────────────┘
```

- **单元测试**：最便宜，最多，覆盖每个函数的每种情况
- **集成测试**：验证多个模块协作正确
- **文档测试**：确保 README 和文档里的示例真的能跑
- **属性测试**（proptest）：不是"我这个测试用例"，而是"对所有满足条件的输入，这个属性都成立"
- **模糊测试**（cargo-fuzz）：用随机数据疯狂调用你的解析器/协议处理，直到崩溃
- **Miri**：在 MIR 层面解释执行你的 unsafe 代码，检测 sanitzer 都可能漏掉的 UB

### 3.2 可靠性（6.3）

错误处理不是"能用 `?` 就行"。一个好的错误处理架构需要回答：
- 这个错误是**可恢复的**还是**不可恢复的**？
- 调用者拿到这个错误后**能做什么**？
- 这个错误在日志里**长什么样**？（有上下文吗？有 backtrace 吗？）
- 库的调用者和应用的最终用户看到的是**同一个错误吗**？

Rust 提供了三种主流的错误处理范式：
- `thiserror`：库开发者用 derive 定义结构化错误类型
- `anyhow` / `eyre`：应用开发者用 context 链添加上下文
- `tracing-error`：把错误和 tracing span 关联起来

### 3.3 可观测性（6.4）

当你的服务跑在生产环境，`println!` 帮不了你。你需要：

- **结构化日志**（tracing）：不是字符串，是 key=value 的结构化字段
- **分布式追踪**（Span + OpenTelemetry）：一个请求跨了 5 个微服务的调用链
- **指标**（metrics + Prometheus）：QPS、P99 延迟、错误率、内存使用

```rust
// ❌ println! 调试
println!("user {} logged in at {}", user_id, now);

// ✅ tracing 结构化日志
info!(user_id = %user.id, "user logged in");
// 输出 JSON: {"level":"INFO","user_id":"42","message":"user logged in","timestamp":"..."}
```

### 3.4 项目工程（6.5-6.7）

当项目从"一个 main.rs"长到"10 个 crate + 50 个依赖"时：
- **Cargo workspace**：共享 `Cargo.lock`、共享 `target/`、统一版本管理
- **Feature flags**：让用户只编译他们需要的功能（`cargo add serde -F derive`）
- **依赖管理**：定期 `cargo audit`、`cargo deny` 检查许可证合规
- **CI/CD**：每次 PR 自动跑测试+lint+安全检查，merge 后自动发布

### 3.5 性能工程（6.8-6.12）🔥

这是 Phase 6 的重头戏——5 节完整的性能优化课程。

```
性能优化闭环
     ┌──────────────────────────────────────┐
     │                                      │
     ▼                                      │
┌─────────┐    ┌─────────┐    ┌─────────┐   │
│ 6.8     │───→│ 6.9     │───→│ 6.12    │   │
│ Profiling│   │ 运行时  │    │ 闭环    │   │
│ 测量    │    │ 优化    │    │ 验证    │───┘
└─────────┘    └─────────┘    └─────────┘
     │              │              │
     │         ┌─────────┐         │
     └────────→│ 6.10    │←────────┘
               │ 编译时  │
               └─────────┘
                    │
               ┌─────────┐
               │ 6.11    │
               │ PGO/    │
               │ BOLT    │
               └─────────┘
```

**关键原则**：
- **先测后优**：没有 profiling 数据的优化都是猜测（6.8）
- **运行时第一**：大部分性能问题在数据结构和算法层面（6.9）
- **工具链次之**：更快的链接器、更好的编译配置也是收益（6.10）
- **PGO/BOLT 是终局武器**：编译器和 CPU 比你更懂你的代码热点（6.11）
- **闭环验证**：优化不是一次性的——建立可重复的优化工作流（6.12）

### 3.6 交付分发（6.13-6.14）

代码写完了，怎么让别人用？
- **API 设计**：你的 crate 的公共接口就是你的产品——命名、签名、错误类型、trait 设计都要遵循 Rust API Guidelines
- **跨平台分发**：交叉编译、静态链接 musl、Docker 镜像优化
- **嵌入式**：当 std 不能用时（no_std），怎么在裸机上跑 Rust

## 4. 学习建议

### 4.1 推荐学习顺序

Phase 6 的课程分为两组——"必学基础"和"性能专题"：

```
必学基础（顺序递进）
6.1 测试基础 → 6.2 高级测试 → 6.3 错误处理 → 6.4 可观测性
   → 6.5 工作空间 → 6.6 依赖管理 → 6.7 CI/CD

性能专题（🔥 核心系列）
6.8 Profiling → 6.9 运行时优化 → 6.10 编译时优化
   → 6.11 PGO/BOLT → 6.12 闭环实战

交付与收尾
6.13 API 设计 → 6.14 no_std/嵌入式 → 6.15 综合实战
```

**建议策略**：
1. **先走完"必学基础"**（6.1-6.7）。这些是日常开发每天都会用到的东西，不需要等到全部精通，但至少要知道每个维度的"最佳实践大概长什么样"。
2. **然后投入"性能专题"**（6.8-6.12）。这是 Phase 6 最有价值的部分——5 节课程形成一个完整的性能优化体系。**6.9（运行时优化）是最重要的一节**：数据结构选择、内存布局、分配策略——这些是你能做的 ROI 最高的优化。6.10（编译时优化）和 6.11（PGO/BOLT）是"锦上添花"——它们可以带来额外 10-25% 的提升，但前提是你的运行时已经优化过了。
3. **最后完成 API 设计、嵌入式和综合实战**（6.13-6.15）。

### 4.2 工程实践不是"一次性学完"的

Phase 6 的内容量很大——14 节课 + 综合实战。但不同于 Phase 1-5 的"概念掌握型"学习，Phase 6 是"经验积累型"。你不必一次性全部精通：

- **6.1-6.2（测试）**：第一个月就用上 `insta` 和 `proptest`
- **6.3（错误）**：下个项目就用 `thiserror` 代替手写 `Display`/`Error`
- **6.4（观测）**：任何非玩具项目都应该接入 `tracing`
- **6.5-6.7（项目工程）**：当你的项目超过 3 个 crate 时回来重读
- **6.8-6.12（性能）**：当你有性能需求时再深入。**但 6.8（Profiling 方法论）应该在性能需求出现之前就学会——因为"如何测量性能"本身就是一项基本技能。**

### 4.3 最重要的学习原则

**1. 不要试图一次性记住所有工具和配置**

Phase 6 涉及大量外部工具和 crate：`criterion`、`proptest`、`tracing`、`cargo-deny`、`cargo-hack`、`sccache`、`mold`、`cross`... 你不需要记住它们的 CLI 参数和配置项。你需要记住的是：**什么时候该用哪个工具，它能解决什么问题**。具体用法查文档即可。

**2. 在实践中学习**

Phase 6 的每一节课都包含"练习"和"练习答案"。如果你在做一个真实的 Rust 项目，在学完某一节课后马上去项目里实践——例如学完 6.5 就把你的项目改成 workspace 结构，学完 6.8 就给你的核心函数写 criterion benchmark。

**3. 优化不是目的，可维护性是目的**

性能优化（6.8-6.12）是 Phase 6 的亮点，但不要沉迷。一个跑得飞快但没人能维护的系统，不如一个跑得够快且团队能迭代的系统。**在工程实践中，可读性 > 可测试性 > 可观测性 > 性能。只有当性能被证明是瓶颈时，才值得牺牲前三者。**

**4. 工程决策没有标准答案**

Phase 1-5 的很多问题有明确的"对"和"错"——生命周期标注错了编译器会报错，Send/Sync 不满足会编译失败。但 Phase 6 的问题往往没有唯一正确答案：
- 用 `thiserror` 还是 `anyhow`？看场景。
- 用 HashMap 还是 BTreeMap？看数据量和 key 类型。
- 加还是不加 LTO？看你的发布策略。

本 Phase 的每一节课都会给出"决策表"或"选择建议"，但最终决策需要你在具体上下文中做出。

---

> **Phase 6 的意义**：前五个 Phase 你学会的是"怎么写出 Rust 编译器能验证通过的代码"。Phase 6 教你的是"怎么写出团队能持续维护的代码"。编译器验证的是正确性；工程实践验证的是可持续性。编译器报错只会在你写代码的时候发生；但糟糕的工程实践会在每一个深夜 on-call 的时刻报复你——当你面对一个没有日志、没有测试、没有 CI、错误信息全是 `unwrap()` panic 的生产环境时，你会感谢自己认真学完了 Phase 6 的每一节课。
>
> 现在，从 [[6.1-testing-fundamentals|测试体系基础]] 开始，建立你的工程直觉。
