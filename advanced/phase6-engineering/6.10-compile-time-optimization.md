# 6.10 编译时优化 —— 让编译器更快地生成更快的代码

> 性能优化有两个维度：代码跑得有多快（运行时），以及代码需要多久才能开始跑（编译时）。这两者通常是一个权衡——更激进的优化（如 LTO、codegen-units=1）让代码跑得更快，但编译更慢。但有些优化能**同时**改善两者：更快的链接器（mold）、编译缓存（sccache）、debug 构建加速（cranelift）。本节覆盖了"编译时"的两层含义：**如何配置编译器让生成的代码更优**（运行时性能），以及**如何加速编译过程本身**（开发体验）。这两个目标有时一致（如 mold），有时冲突（如 LTO），理解这个权衡是做编译配置决策的前提。

## 1. `cargo build --timings` —— 知道时间花在哪

### 1.1 分析编译耗时

```bash
# 生成编译耗时报告
cargo build --timings

# 输出 target/cargo-timings/cargo-timings.html
# 用浏览器打开，可以看到：
#   - 每个 crate 的编译时间（条形图）
#   - 依赖编译 vs 你的代码编译
#   - 哪些依赖是编译时间的大头
```

**典型发现**：
- `syn` + `quote` + `serde_derive`：proc macro 依赖是编译时间大户
- 大型 crate（如 `aws-sdk-*`）：每个都是分钟级的编译
- 你的代码：如果用了大量泛型，单态化时间可能很长

### 1.2 读 timings 报告的技巧

```
HTML 报告中关注：
  1. "Crate compilation" 图表：哪个 crate 最耗时？
     → 考虑替换为更轻的替代品

  2. 你的 crate 的"monomorphization"时间
     → 如果很高，说明泛型膨胀严重

  3. "Linker" 时间
     → 如果很高，考虑换 mold/lld（见第 2 节）

  4. "Proc macro" 时间
     → 考虑减少 derive 的数量
```

## 2. 更快链接器 —— mold / lld

### 2.1 链接器是编译时间的关键瓶颈

```
编译流程：
  源代码 → 编译(.rs → .o) → 链接(.o → 可执行文件)
  ├──── 70% ────┤  ├─── 30% ───┐
                                  ↑
                            链接器在这里！
                            默认的 GNU ld 很慢
```

### 2.2 mold —— 目前最快的链接器

```bash
# 安装 mold
# Ubuntu: sudo apt install mold
# macOS: brew install mold  (需要手动编译)
# Arch: sudo pacman -S mold

# 使用 mold 链接
# 方法 1：命令行指定
cargo build --release --config "target.x86_64-unknown-linux-gnu.linker=\"clang\""
# + .cargo/config.toml:
# [target.x86_64-unknown-linux-gnu]
# rustflags = ["-C", "link-arg=-fuse-ld=mold"]

# 方法 2：在 .cargo/config.toml 中全局配置
```

```toml
# .cargo/config.toml
[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = ["-C", "link-arg=-fuse-ld=mold"]
```

**性能对比（链接阶段）**：
| 链接器 | 时间 | 相对速度 |
|--------|------|---------|
| GNU ld | 12.3s | 1.0x |
| gold | 8.1s | 1.5x |
| lld | 3.2s | 3.8x |
| **mold** | **0.8s** | **15x** |

### 2.3 macOS 上的选择

```
macOS 默认使用 Apple 的 ld，性能中等。

替代方案：
  - lld：通过 brew install llvm 获得
  - mold：需要从源码编译，不如 Linux 上简单

对于 macOS，lld 通常是可用的最快选择
```

## 3. sccache —— 编译缓存

### 3.1 原理

```
第一次编译 crate A（无变化）：
  rustc → .o → sccache 存储

第二次编译 crate A（无变化）：
  rustc 查询 sccache → 命中 → 直接返回缓存的 .o
  ↑ 完全跳过编译！
```

### 3.2 安装和配置

```bash
# 安装
cargo install sccache

# 配置环境变量
export RUSTC_WRAPPER=sccache

# 或在 .cargo/config.toml 中
```

```toml
# .cargo/config.toml
[build]
rustc-wrapper = "sccache"
```

```bash
# 查看缓存命中率
sccache --show-stats
# Compile requests: 1234
# Cache hits: 987      (80%)
# Cache misses: 247    (20%)
```

### 3.3 sccache 在 CI 中的配置

```yaml
# GitHub Actions
- name: Setup sccache
  uses: mozilla-actions/sccache-action@v0.0.3

- name: Build
  run: cargo build --release
  env:
    SCCACHE_GHA_ENABLED: "true"
    RUSTC_WRAPPER: "sccache"
```

## 4. Cranelift 后端 —— debug 构建加速

### 4.1 问题

Rust 有两套代码生成后端：
- **LLVM 后端**（默认）：优化质量极高，但编译慢。debug 构建中 90% 的编译时间花在 LLVM 上
- **Cranelift 后端**（实验性）：优化质量低，但编译极快

**关键洞察**：debug 构建你不需要极致优化！你只需要"能跑就行"。

### 4.2 使用 Cranelift

```bash
# 安装 cranelift 组件
rustup component add rustc-codegen-cranelift-preview

# 用 cranelift 编译
cargo build -Zcodegen-backend=cranelift-preview

# 或配置 .cargo/config.toml
```

```toml
# .cargo/config.toml（仅用于 debug 构建）
[profile.dev]
codegen-backend = "cranelift"
```

**性能对比**：
| 后端 | debug 构建时间 | debug 运行时性能 |
|------|-------------|---------------|
| LLVM | 45s | 100% (基准) |
| Cranelift | 12s (3.7x 快) | ~50-70% (慢一些) |
| Cranelift + opt=1 | 18s | ~80% |

**结论**：Cranelift 不适合 release 构建（性能损失太大），但极适合 debug 迭代。

## 5. 编译配置调优

### 5.1 `Cargo.toml` Profile 配置

```toml
# Cargo.toml

# ━━━ Release 优化配置 ━━━
[profile.release]
opt-level = 3         # 优化级别：0=不优化, 1=基本, 2=默认, 3=激进, s=优化大小, z=极端优化大小
lto = "fat"           # Link-Time Optimization: false | "thin" | "fat"
codegen-units = 1     # 代码生成单元：1=最大优化但最慢编译，16=最快编译但优化少
panic = "abort"       # panic=abort 移除 unwinding 支持，减小二进制 10-20%
strip = true          # 移除符号表
debug = 0             # 不生成调试信息

# ━━━ Debug 优化配置（开发体验优先）━━━
[profile.dev]
opt-level = 0         # 不优化（编译最快）
debug = 2             # 完整调试信息

# ━━━ 自定义 profile（需要快速但仍可调试的性能）━━━
[profile.dev-opt]
inherits = "dev"      # 继承 dev profile
opt-level = 1         # 轻度优化（debug 信息仍基本可用）
# 用途：当 debug 模式太慢但 release 模式无法调试时
```

### 5.2 关键参数详解

| 参数 | 作用 | 取舍 |
|------|------|------|
| `opt-level` | 控制 LLVM 优化强度 | 越高 → 运行时越快，编译越慢 |
| `lto = "fat"` | 全程序跨 crate 优化 | 运行时 ~5-15% 提升，链接时间 × 2-5 |
| `lto = "thin"` | 轻量跨 crate 优化 | 运行时 ~3-8% 提升，链接时间 × 1.2-2 |
| `codegen-units = 1` | 单个代码生成单元 | 最优优化，编译最慢 |
| `codegen-units = 16` | 16 个并行生成单元 | 编译最快，优化最差 |
| `panic = "abort"` | 移除 unwinding | 减小二进制 10-20%，但不能 catch_unwind |
| `strip = true` | 移除符号表 | 大幅减小二进制，但 backtrace 无函数名 |

### 5.3 典型配置方案

```toml
# ━━━ 方案 1：追求极致运行时性能 ━━━
[profile.release]
opt-level = 3
lto = "fat"
codegen-units = 1
panic = "abort"

# ━━━ 方案 2：平衡编译时间和性能 ━━━
[profile.release]
opt-level = 3
lto = "thin"        # ← 比 fat 快很多，性能损失很小
codegen-units = 16  # ← 编译快，优化略差
panic = "unwind"    # ← 保留 catch_unwind 能力

# ━━━ 方案 3：追求小体积（嵌入式/IoT）━━━
[profile.release]
opt-level = "z"     # ← 优化体积优先
lto = "fat"
codegen-units = 1
panic = "abort"
strip = true
```

## 6. 泛型单态化成本

### 6.1 泛型不是免费的

```rust
// 这行代码会导致什么？
let v: Vec<String> = vec!["hello".into()];
let v2: Vec<i32> = vec![1, 2, 3];
let v3: Vec<f64> = vec![1.0, 2.0];
// 编译器为 Vec::<String>, Vec::<i32>, Vec::<f64> 各生成一份代码
// → 3 倍的代码量，3 倍的编译时间
```

### 6.2 减少单态化膨胀

```rust
// ━━━ 技术 1：提取非泛型的内层函数 ━━━
pub fn process<T: Display>(items: &[T]) {
    // 泛型的外层——为每个 T 生成
    let mut buf = String::new();
    for item in items {
        process_inner(item, &mut buf);  // ← 调用了内层非泛型函数
    }
}

// 非泛型的内层——只生成一份代码
fn process_inner(item: &dyn Display, buf: &mut String) {
    use std::fmt::Write;
    write!(buf, "{}", item).unwrap();
}

// ━━━ 技术 2：用 trait 对象代替泛型（当泛型参数很多时）━━━
// ❌ 每个 (T, U) 组合生成一份代码
fn process_pair<T: Display, U: Display>(a: &T, b: &U) { ... }

// ✅ 只生成一份代码（但多了虚函数调用开销）
fn process_pair(a: &dyn Display, b: &dyn Display) { ... }
```

## 7. 增量编译

### 7.1 增量编译的工作原理

```
第一次编译：
  crate A → .o（全部编译）
  crate B → .o（全部编译）

修改了 crate B 的一行代码：
  非增量：crate A → .o  crate B → .o  （全部重新编译）
  增量：  只重新编译 crate B 中变化的部分
          ↑ Rust 默认开启增量编译
```

```toml
# Cargo.toml
[profile.dev]
incremental = true   # 默认开启，debug 模式推荐

[profile.release]
incremental = false  # 默认关闭，因为会影响优化质量
# 但如果你在频繁修改 + 测试 release 二进制，可以打开
# incremental = true
```

## 8. 实用速查

```bash
# ━━━ 编译时间分析 ━━━
cargo build --timings                    # 生成 HTML 报告
cargo clean && time cargo build          # 测量全量编译时间
cargo bloat --release --crates           # 分析二进制大小（按 crate）
cargo tree --edges features              # 查看 feature 传递链

# ━━━ 快速迭代配置 ━━━
# .cargo/config.toml
[build]
rustc-wrapper = "sccache"               # 编译缓存

[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = ["-C", "link-arg=-fuse-ld=mold"]  # 快 15x 的链接器

[profile.dev]
opt-level = 0                           # 极快编译
codegen-backend = "cranelift"           # 实验性，但快

# ━━━ 极限 release 配置 ━━━
[profile.release]
opt-level = 3
lto = "fat"
codegen-units = 1
panic = "abort"
strip = true
```
