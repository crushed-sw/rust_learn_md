# 6.11 PGO/BOLT/LTO 与二进制分发 —— 性能优化的终局武器

> 代码层面的优化（6.9）和编译配置的优化（6.10）你已经掌握了。但编译器有一个根本的局限：它在编译时不知道你的代码在运行时**哪些路径最热**。LTO 可以看到整个 crate 的代码，但它看不到 runtime 的执行频率。PGO（Profile-Guided Optimization）填补了这个空白：你先跑一遍程序，收集真实的执行 profile（哪个函数被调用了多少次、哪个分支通常走 true），然后把这些 profile 喂给编译器——编译器据此重新编排代码布局，把热路径放在一起，热函数内联更激进，冷路径移开。BOLT 更进一步：它不碰源代码，直接在二进制层面重新排列函数布局。PGO + BOLT 的组合可以带来 10-25% 的性能提升——**不需要改一行代码**。本节带你走通 PGO、BOLT、LTO 的完整流程，然后讨论如何把优化后的二进制分发到用户手中。

---

## 1. LTO —— 全程序优化

### 1.1 Fat LTO vs Thin LTO

```
LTO（Link-Time Optimization）= 链接期优化

没有 LTO：
  crate A 编译 → crate A.o
  crate B 编译 → crate B.o
  链接：把 .o 拼在一起 → 可执行文件
  ↑ 编译器看不到跨 crate 的优化机会！

有 LTO：
  crate A → LLVM IR（不是机器码！）
  crate B → LLVM IR
  链接时：对所有 LLVM IR 做全程序优化 → 机器码
  ↑ 编译器可以跨 crate 内联、消除死代码、常量传播！
```

### 1.2 配置

```toml
# Cargo.toml
[profile.release]
lto = "fat"   # 最大优化，链接时间最长（可能 ×3-5）
# 或
lto = "thin"  # 较轻的优化，链接时间较短（可能 ×1.5-2）
# 或
lto = false   # 无 LTO（默认）
```

**决策**：
| LTO 模式 | 性能提升 | 链接时间增加 | 适用场景 |
|---------|---------|------------|---------|
| `false` | 0% | 0% | 开发阶段 |
| `thin` | 3-8% | 50-100% | CI / 发布候选 |
| `fat` | 5-15% | 200-500% | 最终发布 |

## 2. PGO —— 用真实数据指导编译器

### 2.1 PGO 三步流程

```
Step 1: 编译 instrumented 版本
  → 编译器在代码中插入"计数桩"（这个函数被调了多少次？这个分支通常走哪个方向？）

Step 2: 运行 instrumented 版本（用真实工作负载）
  → 收集 profile 数据

Step 3: 重新编译（使用收集的 profile 数据）
  → 编译器根据 profile 做更好的优化决策
```

### 2.2 完整操作流程

```bash
# ━━━ Step 1：编译 instrumented 版本 ━━━
RUSTFLAGS="-C profile-generate=/tmp/pgo-data" cargo build --release

# ━━━ Step 2：运行（用典型的工作负载）━━━
# 关键：工作负载要代表真实使用场景！
./target/release/my_binary --workload typical
./target/release/my_binary --workload edge-case
# 重复多次以提高数据质量

# 合并多次运行的 profile 数据（可选）
llvm-profdata merge -o /tmp/pgo-data/merged.profdata /tmp/pgo-data

# ━━━ Step 3：用 profile 数据重新编译 ━━━
RUSTFLAGS="-C profile-use=/tmp/pgo-data/merged.profdata" cargo build --release

# 生成的二进制就是 PGO 优化过的！
```

### 2.3 PGO 的实际效果

```
典型收益：
  - 编译器（rustc 自身也用 PGO）：~7-15% 提升
  - JSON 解析器：~10-20% 提升
  - 数据库：~5-15% 提升
  - 网络服务：~5-10% 提升
  - 游戏引擎：~10-20% 提升

PGO 对哪些代码最有效？
  ✅ 分支密集的代码（PGO 能优化分支预测）
  ✅ 有很多函数调用层次的热路径（PGO 能优化内联决策）
  ✅ 间接调用多的代码（如虚函数调用、trait 对象调用）

PGO 对哪些代码效果有限？
  ❌ 已经高度优化的线性代码（无分支 = 无额外信息）
  ❌ I/O bound 的代码（瓶颈不在 CPU）
```

### 2.4 在 CI 中自动化 PGO

```yaml
# .github/workflows/release-pgo.yml
- name: Build instrumented
  run: RUSTFLAGS="-C profile-generate=/tmp/pgo-data" cargo build --release

- name: Run benchmark workload
  run: ./scripts/run_benchmarks.sh  # 你的基准测试脚本

- name: Merge profiles
  run: |
    llvm-profdata merge -o /tmp/pgo-data/merged.profdata /tmp/pgo-data

- name: Build PGO-optimized
  run: RUSTFLAGS="-C profile-use=/tmp/pgo-data/merged.profdata" cargo build --release
```

## 3. BOLT —— 二进制层面的布局优化

### 3.1 BOLT 是什么？

```
PGO 的局限：PGO 在编译期工作——它优化了 LLVM IR 层面的决策
           但函数在二进制文件中的物理布局是链接器决定的
           → 热函数可能分散在二进制文件的不同位置 → i-cache miss

BOLT 做了什么：
  1. 拿到 perf 采样的真实执行数据
  2. 分析：哪些函数是热的？热函数内部的哪个 basic block 最热？
  3. 重新排列二进制中的函数布局：
     - 热函数 → 放在一起（i-cache 友好）
     - 函数内部的热 basic block → 放在一起（减少跳转）
     - 冷代码 → 移到文件末尾（不污染 i-cache）
```

### 3.2 BOLT 操作流程

```bash
# ━━━ 前提：编译时保留重定位信息 ━━━
RUSTFLAGS="-C link-arg=-Wl,--emit-relocs" cargo build --release

# ━━━ Step 1：perf 采样 ━━━
perf record -e cycles:u -j any,u -o perf.data -- ./target/release/my_binary --workload typical

# ━━━ Step 2：转换为 BOLT 格式 ━━━
perf2bolt -p perf.data -o perf.fdata ./target/release/my_binary

# ━━━ Step 3：BOLT 优化二进制 ━━━
llvm-bolt ./target/release/my_binary \
    -o ./target/release/my_binary.bolt \
    -data perf.fdata \
    -reorder-blocks=ext-tsp \
    -reorder-functions=hfsort \
    -split-functions \
    -split-all-cold

# my_binary.bolt 就是优化后的二进制
```

### 3.3 PGO + BOLT 组合

```
PGO + BOLT 不是替代关系——是互补关系：

PGO：
  - 优化编译期决策（内联、分支概率、寄存器分配）
  - 需要修改编译流程

BOLT：
  - 优化二进制布局（函数排列、basic block 排列）
  - 不碰编译流程，只处理最终二进制

组合效果：
  - 单独的 PGO：~10% 提升
  - 单独的 BOLT：~5-8% 提升
  - PGO + BOLT：~15-25% 提升（1+1 > 2！）

最佳实践：
  1. 先用 PGO 编译（profile-generate → run → profile-use）
  2. 然后在 PGO 优化后的二进制上运行 BOLT
```

## 4. 交叉编译

### 4.1 `cross` —— 零配置交叉编译

```bash
# 安装
cargo install cross

# 一行命令：在 Linux 上编译 Windows 二进制
cross build --target x86_64-pc-windows-msvc --release

# 在 x86_64 上编译 ARM64
cross build --target aarch64-unknown-linux-gnu --release
```

**`cross` 做了什么**：
- 自动拉取预配置的 Docker 镜像（包含对应平台的 toolchain）
- 在容器内运行编译
- 你不需要手动安装任何交叉编译工具链

### 4.2 musl —— 静态链接的 Linux 二进制

```bash
# 添加 musl target
rustup target add x86_64-unknown-linux-musl

# 编译——生成完全静态链接的二进制
cargo build --target x86_64-unknown-linux-musl --release

# 验证：无动态链接依赖
ldd target/x86_64-unknown-linux-musl/release/my_binary
# 输出：statically linked（或 not a dynamic executable）
```

**musl 的优势**：
- 完全静态链接 → 可以在任何 Linux 发行版上运行（不需要 glibc 版本匹配）
- 生成的 Docker 镜像可以基于 `scratch`（最小镜像，< 2MB）
- 但 musl 的 allocator 性能略低于 glibc 的 allocator

### 4.3 常用 Target 一览

| Target | 平台 | 用途 |
|--------|------|------|
| `x86_64-unknown-linux-gnu` | Linux x86_64 (glibc) | 通用 Linux |
| `x86_64-unknown-linux-musl` | Linux x86_64 (musl) | 静态链接/Docker |
| `aarch64-unknown-linux-gnu` | Linux ARM64 | ARM 服务器 |
| `x86_64-apple-darwin` | macOS Intel | macOS |
| `aarch64-apple-darwin` | macOS Apple Silicon | M1/M2/M3 Mac |
| `x86_64-pc-windows-msvc` | Windows x86_64 | Windows |

## 5. Docker 镜像优化

### 5.1 多阶段构建 + musl

```dockerfile
# ━━━ Dockerfile：极小的 Rust 镜像 ━━━

# Stage 1: 编译
FROM rust:1.75-alpine AS builder
RUN apk add --no-cache musl-dev
WORKDIR /app
COPY . .
RUN cargo build --release --target x86_64-unknown-linux-musl

# Stage 2: 最小运行时（只有 ~8MB！）
FROM scratch
COPY --from=builder /app/target/x86_64-unknown-linux-musl/release/my_app /my_app
ENTRYPOINT ["/my_app"]
```

```bash
# 构建
docker build -t my_app .

# 检查镜像大小
docker images my_app
# REPOSITORY  TAG     SIZE
# my_app     latest   8.2MB   ← 极小的镜像！

# 对比：如果用 ubuntu 作为基础镜像，至少 80MB+
```

### 5.2 distroless —— 带有 CA 证书的极简镜像

```dockerfile
# 如果你的应用需要 HTTPS（需要 CA 证书）
FROM gcr.io/distroless/cc-debian12 AS runtime
# distroless 包含最小的运行时（libc、CA 证书、时区数据），但没有 shell
COPY --from=builder /app/target/release/my_app /my_app
ENTRYPOINT ["/my_app"]
```

## 6. `cargo publish` 与 docs.rs

### 6.1 发布前检查清单

```bash
# 1. 确保能编译
cargo build --release

# 2. 确保测试通过
cargo test --all-features

# 3. 检查文档
cargo doc --no-deps --open

# 4. 检查是否有未提交的文件
git status

# 5. 更新版本号
# Cargo.toml: version = "0.2.0"

# 6. 打包检查（模拟发布）
cargo package --allow-dirty  # 如果是 git 仓库
# 或 cargo publish --dry-run

# 7. 发布！
cargo publish
```

### 6.2 docs.rs 配置

```toml
# Cargo.toml
[package]
name = "my_crate"
# docs.rs 自动读取以下配置

[package.metadata.docs.rs]
all-features = true       # 启用所有 features 构建文档
rustdoc-args = ["--cfg", "docsrs"]  # 传递参数给 rustdoc
```

```rust
// src/lib.rs —— 利用 docsrs cfg 在文档中显示 feature 标记
#![cfg_attr(docsrs, feature(doc_cfg))]

#[cfg(feature = "json")]
#[cfg_attr(docsrs, doc(cfg(feature = "json")))]
pub mod json_support {
    // 文档中会显示：Available on crate feature json only.
}
```

1. 编译：release + fat LTO + codegen-units=1

3. 交叉编译：Linux musl + Windows MSVC
4. Docker：多阶段构建 + scratch
5. 发布：cargo publish + GitHub Releases
