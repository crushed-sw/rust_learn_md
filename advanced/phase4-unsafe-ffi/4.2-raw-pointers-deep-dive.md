# 4.2 裸指针深入 —— 从 ptr::read 到指针出处（Provenance）

> 裸指针是 unsafe Rust 的 "汇编语言"。当你写下 `*mut T` 时，你放弃了 Rust 引用提供的全部自动保护——生命期追踪、借用规则、自动解引用——换来的是完全的内存操控能力。但这种能力不是没有规则的：`std::ptr` 模块定义了一套精密的"指针操作原语"，每一条原语都附带着特定的安全契约。`ptr::read` 在读取内存时绕过借用规则；`ptr::copy` 在复制字节时不关心类型系统；`ptr::write` 在写入时不调用目标的 drop——它们共同构成了 Rust 的"手动内存管理工具箱"。理解这些原语不是可选的——如果你要写任何超越平凡安全抽象边界的 Rust 代码，这套原语就是你的基本工具。而更进一步，"指针出处"（provenance）定义了指针本身具备的"身份"——一个指针不只是地址，它还携带着它"从哪来"的历史。丢失这个历史，编译器可以做任意优化——包括让你的代码行为变得不可预测。

## 1. ptr::read —— 不借用就读取

### 1.1 绕过借用检查器的"就地读取"

`std::ptr::read` 是我们遇到的第一把"尖刀"。它的语义很简单：从指定地址读取一个 `T` 类型的值，返回这个值的所有权。但它的实现方式极为激进——它**根本不经过借用检查器**。

```rust
// ptr::read 的签名（概念版）
// pub unsafe fn read<T>(src: *const T) -> T;
//
// 它的作用：把 src 指向的内存中的 bit pattern 复制出来，
// 作为新的 T 返回，同时 src 指向的内存保持原样——不调用 drop

// 使用场景演示：
let mut values = vec![1, 2, 3, 4, 5];

// 场景：在不持有引用的前提下从 Vec 中提取一个元素
// 如果使用 safe 方式（swap_remove 等），可能需要移动其他元素
// 使用 ptr::read 可以直接"偷走"元素而不触发移动语义

let ptr = &values[2] as *const i32; // 指向值 3 的裸指针

let stolen: i32 = unsafe {
    // 安全条件：
    // 1. ptr 不为 null（它是从有效引用转换来的）
    // 2. ptr 正确对齐（i32 的引用保证对齐）
    // 3. ptr 指向完全初始化的 i32 值
    // 4. ptr 指向的内存不会被同时写入
    // 5. 归还所有权后，我们手动管理 values 的 len
    std::ptr::read(ptr)
};

// 现在 stolen 拥有值 3，但 values[2] 仍然是 3（bit-wise）
// values[2] 处于"逻辑上已移走"的状态——double-free 的定时炸弹！
// 这是 ptr::read 的危险：它不改变源内存，也不调用 drop
println!("stolen: {}", stolen); // 3

// 必须手动处理：将 Vec 的 len 设为只有前 2 个元素有效
// 或者用 ptr::write 在同一个位置写入新值
unsafe {
    // 重写 values[2] 位置，防止 Vec drop 时二次释放
    std::ptr::write(&mut values[2] as *mut i32, 0);
}
// 现在 values[2] = 0，Vec 可以安全 drop
```

`ptr::read` 的"原罪"在于它给了你移走一个值而不通知原主人的能力。这很有用——比如在实现 `Vec::remove` 时需要不破坏性地移走元素——但也极易出错。记住：**每次 `ptr::read` 都意味着你手动接管了对那片内存的责任**。

### 1.2 ptr::read 的典型应用场景

以下是 `ptr::read` 真正有价值的几个场景——它们都涉及在不触发借用规则检查的前提下操作内存：

```rust
// 场景 1：实现 swap——交换两个值而不创建中间副本
// 注意：标准库的 std::mem::swap 使用 ptr::swap_nonoverlapping 或等价机制
// 这里展示原理
fn my_swap<T>(a: &mut T, b: &mut T) {
    unsafe {
        // 把 a 的值读到栈上临时变量中
        let tmp = std::ptr::read(a);
        
        // 把 b 的值写到 a（覆盖 a 的旧值——不调用 a 的 drop）
        // 因为 tmp 已经接管了 a 的旧值，不会泄漏
        std::ptr::write(a, std::ptr::read(b));
        
        // 把 tmp（a 的旧值）写到 b
        std::ptr::write(b, tmp);
    }
    // tmp 在离开作用域时被 drop——如果 T 实现了 Drop 的话
    // 但注意：tmp 在 write(b, tmp) 时已经被移动了，不会被 drop 两次
    // 因为 ptr::write 需要第二个参数为值类型，发生了所有权转移
}

// 场景 2：从连续的裸内存中读取数组元素（常见于解析二进制格式时）
fn read_u32_from_buffer(buffer: *const u8, offset: usize) -> Option<u32> {
    // 检查对齐——u32 需要 4 字节对齐
    let ptr = unsafe { buffer.add(offset) } as *const u32;
    if !ptr.is_aligned() {
        return None; // 未对齐的读取在某些平台上会导致段错误
    }
    unsafe {
        // 安全：已验证对齐，调用者保证 buffer 足够大
        Some(std::ptr::read(ptr))
    }
}
```

`ptr::read` 的价值在于它让你能精确控制"何时读取"和"如何处理读取后的源内存"。在 safe Rust 中，读取一个值通常意味着借用它（通过引用），而借用检查器会限制你同时还能做什么。`ptr::read` 移除了这些限制——代价是你要自己保证正确性。

### 1.3 ptr::read 与 drop 的相互作用

`ptr::read` 一个最容易犯错的方面是它与 `Drop` 的关系。当你 `ptr::read` 一个实现了 `Drop` 的类型时，源内存中的值仍然存在（bit-wise），但"逻辑上"它已经不属于任何所有者了。如果那份内存稍后被 drop（比如包含它的 `Vec` 被销毁），就会发生 double-free。

```rust
// ptr::read + 有 Drop 的类型的危险
let mut s = String::from("hello");
let ptr = &mut s as *mut String;

// 读走了 String 的内容——现在 s 处于"被移动"的状态
let moved_out = unsafe { std::ptr::read(ptr) };

// 此时 s 的内部仍然有原来的指针（指向堆上的 "hello" 缓冲区）
// 但"逻辑上"它已经被 moved_out 接管

// 如果不做任何事，s 会在作用域结束时被 drop——
// drop 会释放"hello"缓冲区，而 moved_out 也会释放它 → double-free!

// 解决方案：在 ptr::read 后用 std::mem::forget 或 ManuallyDrop
std::mem::forget(s); // 阻止 s 被 drop——moved_out 现在唯一拥有"hello"缓冲区

println!("{}", moved_out); // "hello"——正常打印
// moved_out 在这里被 drop——正确释放内存
```

这个例子揭示了 `ptr::read` 的哲学：它是"不带行李的移动"——你拿走了值，但源位置没有收到通知。你必须手动处理这笔"债务"。

## 2. ptr::write —— 覆盖内存而不调用 drop

### 2.1 直接写入原始内存

`ptr::write` 是 std::ptr 模块的第二把尖刀。它在指定地址写入一个值，覆盖该地址已有的内容而不先调用 drop。这与 safe Rust 中赋值操作的行为根本不同——赋值操作会先 drop 旧值，再放置新值。

```rust
// ptr::write 的签名（概念版）
// pub unsafe fn write<T>(dst: *mut T, src: T);
//
// 语义：把 src 移动到 dst 指向的位置，
// 不调用 dst 原来指向的值的 drop（如果有的话）

// 基本使用：
let mut value = 42_i32;
let ptr = &mut value as *mut i32;

unsafe {
    // 安全：
    // 1. ptr 非 null，指向有效的已初始化的 i32
    // 2. i32 没有 Drop，所以旧值不需要释放
    // 3. ptr 正确对齐
    std::ptr::write(ptr, 100);
}

println!("{}", value); // 100

// 更关键的用例：写入未初始化的内存
// 在实现 Vec 时，需要在未初始化的内存中构造新元素
let mut buffer: Vec<u8> = Vec::with_capacity(10);
let ptr = buffer.as_mut_ptr();

unsafe {
    // 在未初始化的位置写入值
    // 这是 ptr::write 的最核心用法——
    // 你可以在"空白"的内存上构造值而不先 drop（因为没有东西需要 drop）
    std::ptr::write(ptr.add(0), b'H');
    std::ptr::write(ptr.add(1), b'i');
    std::ptr::write(ptr.add(2), b'!');
}
// 必须手动设置 Vec 的 len，让 Vec 知道这些元素已经存在
unsafe { buffer.set_len(3); }
assert_eq!(&buffer[..], b"Hi!");
```

### 2.2 ptr::write 与所有权转移

`ptr::write(dst, src)` 在语义上是一个"移动"操作——`src` 的所有权被转移到 `dst` 指向的位置。这意味着在 `write` 之后，你不应该再使用 `src`（编译器会阻止你，因为 `src` 是按值传递的）。

```rust
// ptr::write 的所有权语义
let mut destination: String = String::from("old");
let source = String::from("new");

// 记录 source 的堆指针以便后续对比
let source_ptr_before = source.as_ptr();

let dst_ptr = &mut destination as *mut String;

unsafe {
    // 把 source 移动到 destination 的位置
    // destination 的旧值 ("old") 不会被 drop！
    // 这会导致 "old" 的内存泄漏
    std::ptr::write(dst_ptr, source);
    // source 在这里已经被移动，不能再使用
}

// source 不能再被访问——编译器会报错如果尝试
// println!("{}", source); // 编译错误：source 已被移动

// destination 现在包含 "new"——source 的堆缓冲区被转移了
println!("{}", destination); // "new"

// 内存泄漏！"old" 的堆缓冲区永远不会被释放
// destination 的旧值被覆盖了，没有调用它的 drop
//
// 解决方案：在 ptr::write 之前手动 drop 旧值
// unsafe {
//     std::ptr::drop_in_place(dst_ptr); // 调用旧值的析构器
//     std::ptr::write(dst_ptr, source);  // 然后写入新值
// }
```

这个例子说明了 `ptr::write` 的核心注意事项：**它不调用目标位置的旧值的 drop**。如果旧值的类型实现了 `Drop`（如 `String`、`Vec`、`Box`），你必须在 `write` 之前手动调用 `ptr::drop_in_place` 来释放旧值，否则导致内存泄漏。

## 3. ptr::copy 与 ptr::copy_nonoverlapping —— Rust 版的 memcpy 和 memmove

### 3.1 copy_nonoverlapping：当两块内存互不重叠

`std::ptr::copy_nonoverlapping` 相当于 C 的 `memcpy`——它在两块不重叠的内存之间按字节复制数据。如果两块内存有任何重叠，行为就是未定义的。编译器和 LLVM 利用"不重叠"的承诺进行激进优化——包括使用 SIMD 指令进行大块复制。

```rust
// copy_nonoverlapping 的签名（概念版）
// pub unsafe fn copy_nonoverlapping<T>(src: *const T, dst: *mut T, count: usize);
//
// 语义：将 count 个 T 从 src 复制到 dst
// 等价于：for i in 0..count { *dst.add(i) = *src.add(i); }
// 但不通过引用，且可能使用硬件加速的批量复制

// 使用示例：在两个 Vec 之间批量移动数据
fn copy_vec_to_buffer(src: &Vec<u32>, dst: &mut [u32]) {
    let count = src.len().min(dst.len());
    if count == 0 {
        return;
    }
    
    unsafe {
        // 安全条件：
        // 1. src.as_ptr() 指向至少 count 个有效的 u32 值
        // 2. dst.as_mut_ptr() 指向至少 count 个可写的 u32 位置
        // 3. 两块内存不重叠（不同 Vec 的缓冲区不会重叠）
        //    ——如果可能重叠（比如 src 和 dst 是同一个切片的两个子切片），
        //       必须使用 copy（下方讨论）
        std::ptr::copy_nonoverlapping(
            src.as_ptr(),
            dst.as_mut_ptr(),
            count,
        );
    }
}

let src = vec![1u32, 2, 3, 4, 5];
let mut dst = vec![0u32; 5];
copy_vec_to_buffer(&src, &mut dst);
assert_eq!(dst, vec![1, 2, 3, 4, 5]);
```

`copy_nonoverlapping` 是 Rust 世界中速度最快的内存批量复制方式。当你知道源和目标不重叠时，这是正确的选择。但如果你不确定——或者两块内存可能重叠——你需要使用 `copy`。

### 3.2 copy：允许重叠的复制

`std::ptr::copy` 相当于 C 的 `memmove`。它处理两块可能重叠的内存，保证结果为"如同先把源复制到临时缓冲区再从临时缓冲区复制到目标"。在 Rust 中，一个常见的重叠场景是：在同一个切片（或 Vec 的缓冲区）内移动元素。

```rust
// copy 的签名（概念版）
// pub unsafe fn copy<T>(src: *const T, dst: *mut T, count: usize);
// 与 copy_nonoverlapping 的唯一区别：允许 src 和 dst 重叠

// 典型场景：在 Vec 中删除一个元素，需要把后面的元素向前移动
fn remove_at(vec: &mut Vec<i32>, index: usize) -> i32 {
    assert!(index < vec.len(), "index 越界");
    
    let ptr = vec.as_mut_ptr();
    let len = vec.len();
    
    // 先读出要删除的元素
    let removed = unsafe {
        // 安全：index < len 已验证
        std::ptr::read(ptr.add(index))
    };
    
    // 把后面的元素向前移动一位
    // 这里源和目标在同一个 Vec 的缓冲区中——必然重叠
    // 所以必须使用 copy 而不是 copy_nonoverlapping
    let src = unsafe { ptr.add(index + 1) };
    let dst = unsafe { ptr.add(index) };
    let count = len - index - 1;
    
    if count > 0 {
        unsafe {
            // 安全：src 和 dst 都在 Vec 的有效范围内
            // copy 处理重叠——方向由实现保证正确
            std::ptr::copy(src, dst, count);
        }
    }
    
    vec.pop(); // 移除最后一个元素（现在它是重复的）
    removed
}
```

注意在这个实现中，`copy` 会自动选择正确的复制方向（从前向后或从后向前），以避免破坏尚未复制的数据。而 `copy_nonoverlapping` 没有这种保证——它可能以任意顺序复制。

### 3.3 copy vs copy_nonoverlapping 的选择指南

选择哪个函数取决于你对重叠性的把握。一个简单的决策框架：

```rust
// 决策逻辑（伪代码）：
// 
// if 我确信源和目标完全不重叠 {
//     使用 copy_nonoverlapping  // 更快——编译器可以做更多优化
// } else {
//     使用 copy  // 正确性优先——处理重叠
// }
//
// 注意：即使使用 copy_nonoverlapping 在重叠的内存上可能恰好工作，
// 但这是 UB（未定义行为）——编译器将来可能优化掉"恰好工作"的情况

// 验证两块内存是否重叠（概念上的检查）
fn ranges_overlap<T>(a_start: *const T, a_count: usize, 
                      b_start: *const T, b_count: usize) -> bool {
    let a_start = a_start as usize;
    let a_end = a_start + a_count * std::mem::size_of::<T>();
    let b_start = b_start as usize;
    let b_end = b_start + b_count * std::mem::size_of::<T>();
    
    // 两个区间重叠：一个区间的起始在另一个区间之内
    a_start < b_end && b_start < a_end
}
// 这个函数可以在 debug 模式下用作 debug_assert 检查
// 但在 release 中建议移除——地址比较本身有 minor overhead
```

## 4. 指针算术——在裸内存中移动游标

### 4.1 add / offset / sub / wrapping_add

裸指针支持算术运算，让你能在内存中向前或向后移动。但指针算术不是简单的整数加法——它涉及到对齐、边界、以及指针出处（provenance）的语义。

```rust
// 指针算术的核心方法

// ptr.add(count)     —— 偏移 count 个 T 的大小（不是字节！）
// ptr.offset(count)  —— 与 add 相同（语义上），count 可以是负数（*const 也接受负的 offset）
// ptr.sub(count)     —— 向后偏移 count 个 T
// ptr.wrapping_add(count) —— 环绕加法，不触发 UB（但结果可能无效）

let values = [10i32, 20, 30, 40, 50];
let ptr: *const i32 = values.as_ptr();

unsafe {
    // add 以元素大小为单位移动，不是以字节为单位
    // values[0] = 10
    println!("values[0] = {}", *ptr);           // 10
    
    // ptr.add(1) 的值 = ptr + 1 * size_of::<i32>() 字节
    // values[1] = 20
    println!("values[1] = {}", *ptr.add(1));    // 20
    
    // ptr.add(3) 的值 = values[3] = 40
    println!("values[3] = {}", *ptr.add(3));    // 40
    
    // sub 向后移动
    let ptr_to_30 = ptr.add(2); // 指向 30
    println!("values[1] = {}", *ptr_to_30.sub(1)); // 20
    
    // 注意：add 超出已分配内存边界 = UB
    // ptr.add(5)  // UB! 指向 values 之后的一个位置
    //              // 即使你不解引用它，仅计算地址也可能是 UB
    //              // (严格来说，可以计算到"past-the-end"，但不能再向前)
}
```

### 4.2 offset vs add —— 微妙但重要的区别

`offset` 和 `add` 对正数偏移有相同的语义，但 `offset` 还接受负数偏移（即使指针是 `*const T`）：

```rust
// offset 和 add/sub 的区别
let values = [10i32, 20, 30, 40, 50];

// 从中间元素开始
let mid_ptr: *const i32 = &values[2] as *const i32; // 指向 30

unsafe {
    // add 只能正向偏移，不接受负数
    // let _ = mid_ptr.add(-1);  // 编译错误！expected usize, got integer
    
    // sub 只能反向偏移
    println!("{}", *mid_ptr.sub(1));  // 20
    
    // offset 可以正也可以负（参数是 isize）
    println!("{}", *mid_ptr.offset(-2));  // 10——前移两个元素
    println!("{}", *mid_ptr.offset(1));   // 40——后移一个元素
    
    // 实际上，add(n) 内部就是 offset(n as isize)
    // sub(n) 内部就是 offset(-(n as isize))
}
```

在日常使用中，`add`/`sub` 通常更可读——意图明确。只有当你需要根据一个 `isize` 变量来动态决定前进还是后退时，`offset` 才是正确的选择。

### 4.3 wrapping_add —— 故意包裹的指针运算

`wrapping_add` 对基本类型的整数会做环绕。对指针来说，它在地址层面做环绕加法，**不**假定结果指针指向有效的已分配内存。这在实现一些底层数据结构（如环形缓冲区用裸指针实现）时有用：

```rust
// wrapping_add 的用例：环形缓冲区用裸指针实现
struct RingBuffer {
    buf: *mut u8,
    capacity: usize,
    write_pos: usize,
}

impl RingBuffer {
    /// 获取写入位置的裸指针，处理环绕
    fn write_ptr(&self) -> *mut u8 {
        // wrapping_add 不会因为超出已分配范围而产生 UB
        // 它只是在位级执行加法然后返回结果
        unsafe { self.buf.wrapping_add(self.write_pos) }
        // 注意：我们仍然需要确保 write_pos 不会超出 capacity
        // （由外部逻辑保证）
    }
    
    /// 推进写入位置（带环绕）
    fn advance_write(&mut self, bytes: usize) {
        self.write_pos = (self.write_pos + bytes) % self.capacity;
        // 后续调用 write_ptr() 会自动通过 wrapping_add 返回正确地址
    }
}
```

`wrapping_add` 的关键区别在于：**它不会触发 UB 即使结果指针超出了已分配内存的边界**。但解引用这个结果仍然可能 UB——只是计算地址本身是安全的。这使得它适合实现那些"地址本身有意义但不一定可解引用"的算法（如标记指针、环形缓冲区的边界检测等）。

## 5. NonNull\<T\> —— 承诺非空的指针

### 5.1 为什么需要 NonNull

`NonNull<T>` 是对 `*mut T` 的一个极简包装，它在类型层面保证了指针不为 null。这个看似微小的区别带来了两个重要的实际好处：

1. **Niche 优化**：`Option<NonNull<T>>` 可以用 null 值表示 `None`，因此 `Option<NonNull<T>>` 的大小等于 `NonNull<T>` 的大小（等于一个指针的大小），无额外开销
2. **API 自文档化**：函数签名中的 `NonNull<T>` 明确告诉调用者"这个指针不能是 null"

```rust
use std::ptr::NonNull;

// NonNull 的基本使用
fn process_data(ptr: NonNull<u8>, len: usize) {
    // ptr.as_ptr() 返回 *mut T
    // ptr.as_ref() 返回 &T（unsafe——因为需要保证引用的有效性）
    
    unsafe {
        // as_ptr 返回内部的裸指针——safe 操作
        let raw: *mut u8 = ptr.as_ptr();
        
        // as_ref 尝试返回一个引用——unsafe 操作
        // 因为编译器不能保证 ptr 指向有效的已初始化内存
        let reference: &u8 = ptr.as_ref();
        println!("第一个字节: {}", *reference);
    }
}

// 从安全的引用构造 NonNull
let x = 42i32;
let nn = NonNull::from(&x); // NonNull::from(&T) -> NonNull<T>
// 现在 nn 保证非空——因为它来自一个有效的引用

// Option<NonNull<T>> 的 niche 优化
let opt: Option<NonNull<i32>> = Some(nn);
// opt 的大小就是 NonNull<i32> 的大小（指针大小，8 字节在 64 位平台上）
// 而不需要额外的 tag 字节——因为 null 被用作 None 的表示
assert_eq!(std::mem::size_of::<Option<NonNull<i32>>>(), 
           std::mem::size_of::<NonNull<i32>>());
```

### 5.2 NonNull 在标准库中的运用

`NonNull<T>` 是标准库中大多数"内部使用裸指针"的数据结构的基础。`Vec<T>`、`Box<T>`、`LinkedList<T>`、`HashMap<K, V>`——它们内部都使用 `NonNull<T>` 而不是裸的 `*mut T` 来存储指向堆内存的指针。

```rust
// 使用 NonNull 重写安全缓冲区（对比上一课的 SafeBuffer）
use std::ptr::NonNull;

struct BetterBuffer {
    // 使用 NonNull 而不是 *mut u8
    // 好处：语义上 guarantee 这个指针不为 null
    ptr: NonNull<u8>,
    len: usize,
    cap: usize,
}

impl BetterBuffer {
    pub fn new(cap: usize) -> Self {
        assert!(cap > 0, "capacity 必须大于 0");
        
        let layout = std::alloc::Layout::array::<u8>(cap).unwrap();
        let raw_ptr = unsafe { std::alloc::alloc(layout) };
        
        if raw_ptr.is_null() {
            std::alloc::handle_alloc_error(layout);
        }
        
        // NonNull::new 返回 Option<NonNull<T>>
        // 如果 raw_ptr 为 null，返回 None
        let ptr = NonNull::new(raw_ptr)
            .expect("alloc 返回的指针不应为 null");
        
        BetterBuffer { ptr, len: 0, cap }
    }
    
    pub fn as_mut_ptr(&self) -> *mut u8 {
        self.ptr.as_ptr() // NonNull 到 *mut T 的转换是 safe 的
    }
    
    // 从 NonNull + len 构造 &[u8]
    pub fn as_slice(&self) -> &[u8] {
        unsafe {
            std::slice::from_raw_parts(self.ptr.as_ptr(), self.len)
        }
    }
}
```

可以看到，`NonNull` 带来的增量价值在于：你不再需要在每次使用指针时心里默念"这个指针理论上不会是 null"——类型系统帮你记住了这个承诺。

### 5.3 NonNull 不能替代所有验证

`NonNull<T>` 只保证非空——它不保证对齐、不保证指向已初始化内存、不保证生命期。这些仍然是程序员的责任：

```rust
// NonNull 不保证的项——仍需程序员验证
unsafe fn dangerous_deref(ptr: NonNull<i32>) -> i32 {
    // NonNull 保证 non-null——但下面这些条件仍需程序员保证：
    // 1. ptr 指向已初始化的 i32
    // 2. ptr 满足 i32 的对齐要求
    // 3. ptr 指向的内存尚未释放
    // 4. 没有其他线程在同时写入
    *ptr.as_ptr() // 仍然需要 unsafe 来解引用
}
```

## 6. 从安全世界到裸指针世界——构造和转换

### 6.1 从引用到裸指针

从引用（`&T` 和 `&mut T`）转换为裸指针是最安全的裸指针获取路径。因为引用本身已经保证了非空、已对齐、指向已初始化内存——转换后的裸指针继承了这些保证（虽然类型系统不再追踪它们）：

```rust
// 从引用到裸指针的所有路径

// &T → *const T
let x = 42;
let ptr1: *const i32 = &x as *const i32;    // 方式 1: as 转换
let ptr2: *const i32 = &x;                   // 方式 2: 自动强制转换（coercion）

// &mut T → *mut T
let mut y = 99;
let ptr3: *mut i32 = &mut y as *mut i32;    // as 转换
let ptr4: *mut i32 = &mut y;                 // 自动强制转换

// &mut T → *const T（失去可变性——安全的降级）
let ptr5: *const i32 = &mut y as *const i32; // 从可变引用创建不可变裸指针
let ptr6: *const i32 = &mut y;               // 自动强制转换

// &T → *mut T 是非法的（不能从不可变引用创建可变裸指针）
// let ptr7: *mut i32 = &x as *mut i32;  // 编译错误！
// 这防止了通过裸指针绕过不可变引用限制
```

关键规则：**你可以从 `&mut T` 创建 `*const T`（失去写权限），但不能从 `&T` 创建 `*mut T`（获得写权限）**。这种"只降级不升级"的规则防止了最明显的一类别名违规。

### 6.2 std::ptr::addr_of! —— 解引用操作的替代

`std::ptr::addr_of!` 和 `std::ptr::addr_of_mut!` 宏提供了在不实际解引用的情况下获取某个位置的地址的能力。这在处理可能未对齐或要求精确控制时非常有用：

```rust
// addr_of! 的必要性：获取结构体字段的指针而不触发借用

struct MyStruct {
    a: u32,
    b: u64, // 可能有 padding
}

let mut s = MyStruct { a: 1, b: 2 };

// 方式 1: &s.b as *const u64 —— 创建了对 s.b 的引用
// 这会临时借用 s.b，可能在某些情况下不可行

// 方式 2: addr_of! —— 创建裸指针而不创建引用
let ptr_to_b: *const u64 = std::ptr::addr_of!(s.b);
// 这不会创建任何引用——纯粹是计算偏移量

let ptr_to_a_mut: *mut u32 = std::ptr::addr_of_mut!(s.a);
unsafe {
    *ptr_to_a_mut = 42;
}
// 注意：这里 ptr_to_a_mut 和 ptr_to_b 同时存在——
// 一个是 *mut u32，一个是 *const u64
// 裸指针没有借用规则——编译器不会阻止这种别名
// 但不正确的使用仍然是 UB
```

`addr_of!` 的核心价值在于：**它允许你创建指向某个位置的裸指针而不需要通过引用**。这在以下场景中至关重要：
- 字段可能未初始化（在实现 unsafe 数据结构时）
- 字段可能未对齐（在解析二进制协议时）
- 你需要在创建指针的同时避免触发借用检查器的限制

### 6.3 空指针、悬垂指针和地址构造

`std::ptr` 模块提供了几种从"无"构造指针的方式：

```rust
// 空指针——最常见的"不指向任何东西"的指针
let null_const: *const i32 = std::ptr::null();
let null_mut: *mut i32 = std::ptr::null_mut();

// dangling() —— 创建一个非空但不指向有效内存的指针
// 用于"占位"——比如在数据结构中标记一个指针尚未初始化
// 注意：绝对不能解引用 dangling 指针
let dangler: *mut i32 = std::ptr::dangling();
// dangling 返回一个"对齐良好但无意义"的地址
// 通常用于 NonNull 的占位值

// 从 NonNull 创建 dangling
let nn_dangler: std::ptr::NonNull<i32> = std::ptr::NonNull::dangling();

// 内存地址到指针的转换（极度危险，只在嵌入式和 FFI 中使用）
// let mmio_addr = 0x4000_0000 as *mut u32;  // 内存映射 I/O
// 仅在目标地址确实有有效内存映射时才安全
```

对于 `std::ptr::dangling()` 的使用，一个重要的实际场景是在实现那些需要"伪初始化"的数据结构时（如在 `Vec` 的 `with_capacity` 中，`ptr` 字段可能先用 `dangling()` 占位，直到真正分配后才更新）。

## 7. 指针对齐——CPU 的隐式契约

### 7.1 什么是对齐，为什么它重要

对齐（alignment）要求一个类型的值的内存地址必须是某个数的整数倍。例如，`u32` 通常要求 4 字节对齐，意味着 `u32` 的地址必须能被 4 整除。如果访问未对齐的内存，在某些 CPU 架构（如 ARM、MIPS）上会触发硬件异常；在 x86 上通常"能工作但更慢"，而编译器会假定对齐成立并据此生成优化代码。

```rust
// 对齐的基本概念
// size_of::<T>()  = 类型的字节大小
// align_of::<T>() = 类型的对齐要求（字节数）

// 查看常见类型的对齐
println!("u8:    size={}, align={}", 
    std::mem::size_of::<u8>(), std::mem::align_of::<u8>());
// u8: size=1, align=1 —— 任意地址都可以放 u8

println!("u32:   size={}, align={}", 
    std::mem::size_of::<u32>(), std::mem::align_of::<u32>());
// u32: size=4, align=4 —— 地址必须能被 4 整除

println!("u64:   size={}, align={}", 
    std::mem::size_of::<u64>(), std::mem::align_of::<u64>());
// u64: size=8, align=8 —— 地址必须能被 8 整除

// 结构体的对齐等于其最严格对齐字段的对齐
#[repr(C)]
struct Mixed {
    a: u8,   // align 1
    b: u32,  // align 4
}
// Mixed 的 align = max(1, 4) = 4
// Mixed 的 size = 8（包含 padding 以满足 b 的对齐）
```

### 7.2 检查和管理对齐

在 unsafe 代码中，检查指针对齐是一项基础操作。`ptr.is_aligned()` 方法是 Rust 1.79+ 中新增的便捷检查：

```rust
// 检查和修复对齐
fn read_u32_unaligned(data: &[u8], offset: usize) -> Option<u32> {
    // 获取指向目标位置的裸指针
    let ptr = unsafe { data.as_ptr().add(offset) } as *const u32;
    
    // 检查对齐
    if ptr.is_aligned() {
        // 对齐正确——直接读取
        unsafe { Some(*ptr) }
    } else {
        // 未对齐——使用 std::ptr::read_unaligned
        // read_unaligned 对流式对齐没有要求（可能更慢但更安全）
        unsafe { Some(std::ptr::read_unaligned(ptr)) }
    }
}

// 手动对齐指针的经典方法
fn align_ptr_up(ptr: *const u8, alignment: usize) -> *const u8 {
    let addr = ptr as usize;
    let aligned = (addr + alignment - 1) & !(alignment - 1);
    aligned as *const u8
}
// 示例：ptr = 0x1003, alignment = 4
// addr + 4 - 1 = 0x1006
// !(4 - 1) = !3 = !0b11 = 0xFFFF...FC
// 0x1006 & 0xFFFF...FC = 0x1004 — 向上对齐到 4 的倍数
```

在实现自定义分配器时，对齐处理尤其重要——你分配的内存必须满足调用者指定的对齐要求。

### 7.3 未对齐访问的 UB 风险

编译器生成的代码**假设**所有通过引用或常规指针解引用的访问都满足对齐要求。如果违反这一假设，即使是 x86 平台也可能出现令人困惑的行为——编译器的优化可能基于对齐假设而重排或向量化内存访问：

```rust
// 危险的未对齐访问——即使可能在 x86 上"恰好工作"也是 UB
fn dangerous_unaligned_read(ptr: *const u32) -> u32 {
    unsafe {
        *ptr  // 如果 ptr 未对齐，这是 UB
              // 编译器可能生成 movdqa（需要 16 字节对齐）或其他对齐敏感的指令
              // 或者基于"对齐"假设删除某些代码路径
    }
}

// 安全的替代方案 1：read_unaligned
fn safe_unaligned_read(ptr: *const u32) -> u32 {
    unsafe {
        std::ptr::read_unaligned(ptr)
        // read_unaligned 明确告诉编译器"这个指针可能未对齐"
        // 编译器会生成适当的指令来处理未对齐情况
    }
}

// 安全的替代方案 2：手动按字节组装
fn safe_unaligned_read_manual(ptr: *const u8) -> u32 {
    unsafe {
        let bytes = [
            *ptr,
            *ptr.add(1),
            *ptr.add(2),
            *ptr.add(3),
        ];
        u32::from_le_bytes(bytes) // 或 from_be_bytes，取决于字节序
    }
}
```

## 8. 指针出处（Provenance）——地址不是一切

### 8.1 指针不只是一个整数

很多 C 和 Rust 程序员有一个直觉上的误解：指针只是内存地址的另一种写法。但事实上，指针携带着**出处（provenance）信息**——关于它"从哪个分配（allocation）来"的信息。这个信息对编译器的别名分析（alias analysis）至关重要。

```rust
// 指针出处：两个相同地址的裸指针可能是"不同"的

// 一个简化但能说明问题的例子（概念演示，实际行为依赖编译器版本和优化级别）
let mut x = 42u32;
let ptr_a: *mut u32 = &mut x;          // 出自 x
let ptr_b: *mut u32 = &mut x as *mut u32; // 也出自 x，通过 x

// ptr_a 和 ptr_b 的地址相同，出处相同——它们代表"同一个指针"

// 但如果这样做：
let addr = &x as *const u32 as usize;
let ptr_c: *mut u32 = addr as *mut u32;  // 从整数还原的指针
// ptr_c 的地址和 ptr_a/ptr_b 相同，
// 但 ptr_c 的出处是"无出处"（或者说"来自整数"）
// 
// 在某些优化级别下，通过 ptr_c 的访问可能被编译器视为
// 与通过 ptr_a 的访问不冲突——导致错误的优化

// 这是 Rust 安全模型的一个开放领域——
// 当前的最佳实践是：避免从整数构造指针，除非在 FFI 中必须如此
```

指针出处（provenance）是目前 Rust 不安全代码指南（Unsafe Code Guidelines）中最复杂的主题之一。虽然完整的模型还在社区的讨论和迭代中，但一个基本准则很清晰：**尽量让指针的生命周期"连续"**——一个指针通过引用创建，通过 `add`/`sub` 偏移，通过 `cast` 转换类型——但不要从中"跳出去"（转为整数再转回来）又期望完全相同的行为。

### 8.2 Strict Provenance —— 未来的方向

Rust 社区正在推动"严格出处"（strict provenance）模型，它明确定义了指针和整数之间的关系。核心思想是：**指针 ≠ 地址**；指针带有出处；在指针和整数之间转换时明确"暴露出处"（expose provenance）的语义。

```rust
// Strict Provenance API（实验性，在 nightly 中可用）
// 这些 API 的设计目标是让指针出处显式化

// 传统方式（模糊出处）：
// let addr = ptr as usize;       // 指针 → 整数，出处信息"丢失"
// let ptr2 = addr as *mut u8;    // 整数 → 指针，出处信息从何而来？

// Strict Provenance 方式：
// let addr = ptr.addr();         // 获取地址（丢弃出处）
// let ptr2 = ptr.with_addr(new_addr);  // 创建同出处但不同地址的指针
// let ptr3 = ptr.map_addr(|a| a + offset); // 偏移地址保持出处
//
// 严格出处的关键原则：
// 1. 指针转整数用 .addr() —— 得到纯地址，丢失出处
// 2. 整数转指针用 .with_addr() —— 保留出处，只改变地址
// 3. 不应该有"从纯整数凭空创建出处"的操作（除非 FFI）
```

Strict Provenance 目前仍然在 nightly 中迭代（通过 `#![feature(strict_provenance)]` 启用），但它的方向已经相当清晰。在写今天的产品代码时，最好的实践是：**尽量避免指针和整数的来回转换**；如果你必须转换，确保在同一块分配内操作，且转换是"无损往返"的。

### 8.3 出处违规的实际后果

出处违规可能在优化级别较高时产生微妙且难以调试的错误：

```rust
// 一个被编译优化"揭穿"的出处违规示例（概念演示）
// 
// 假设你有两个不同的分配（alloc_a 和 alloc_b），
// 它们的地址恰好相邻或是某个关系
// 你从 alloc_a 的指针计算出一个恰好指向 alloc_b 边界的地址，
// 然后通过这个地址访问 alloc_b 的内存。
// 这在美国法律上类比你拿到 A 公寓的钥匙，
// 发现它恰好也能开 B 公寓的门——
// 但这不代表你有权进入 B 公寓。
//
// 编译器的别名分析会基于"不同的分配永远不会重叠"来优化，
// 如果发现你的代码通过不同出处访问了重叠的内存，
// 优化可能产生与你的直觉不同的结果。

// 最佳实践：
// 1. 不要"跨分配"使用指针
// 2. 不要依赖两个无关分配的地址关系
// 3. 在单个分配内使用 add/sub/offset 偏移指针
// 4. 如果需要在不同分配之间操作，使用索引而不是指针
```

## 9. 指针类型转换——*const T ↔ *const U

### 9.1 as 转换 vs .cast() 方法

在裸指针之间转换类型时，`as` 关键字和 `.cast()` 方法完成相同的事情，但 `.cast()` 更显式和可读：

```rust
let values: [u32; 4] = [0x01020304, 0x05060708, 0x090A0B0C, 0x0D0E0F10];
let ptr_u32: *const u32 = values.as_ptr();

// 方式 1: as 转换
let ptr_u8_a: *const u8 = ptr_u32 as *const u8;

// 方式 2: .cast() 方法（推荐——意图更明确）
let ptr_u8_b: *const u8 = ptr_u32.cast::<u8>();

// 现在 ptr_u8_b 指向同一个内存地址
// 但以 u8 为单位来解释——你可以逐字节读取 u32 的内容
unsafe {
    // 读取第一个 u32 的四个字节
    let byte0 = *ptr_u8_b;          // 0x04（小端序——最低字节在前）
    let byte1 = *ptr_u8_b.add(1);   // 0x03
    let byte2 = *ptr_u8_b.add(2);   // 0x02
    let byte3 = *ptr_u8_b.add(3);   // 0x01
    
    println!("bytes: {:02x} {:02x} {:02x} {:02x}", 
        byte0, byte1, byte2, byte3);
    // 在 x86（小端序）上输出: bytes: 04 03 02 01
}
```

使用 `.cast()` 而不是 `as` 的好处在于，`.cast()` 只改变类型参数而不改变 mutability——如果你有一个 `*const T`，`.cast::<U>()` 返回 `*const U`；如果你有一个 `*mut T`，`.cast::<U>()` 返回 `*mut U`。这防止了无意中的可变性改变。

### 9.2 指针转换的安全边界

并非所有的指针类型转换都是安全的。核心规则是：目标类型的对齐要求不能比源类型更严格（除非你已验证对齐满足要求），而且你需要确保大小和布局的一致性：

```rust
// 安全的指针类型转换与不安全的转换

// ✓ 安全：u32 → u8（u8 的 align 为 1，比 u32 的 4 更宽松）
let u32_ptr: *const u32 = &42u32;
let u8_ptr: *const u8 = u32_ptr.cast::<u8>();

// ✓ 安全但需要注意：u8 → u32（需要保证 u8 指针恰好满足 u32 对齐）
let bytes: [u8; 8] = [1, 0, 0, 0, 2, 0, 0, 0];
let bytes_ptr: *const u8 = bytes.as_ptr();
let u32_ptr_from_bytes: *const u32 = bytes_ptr.cast::<u32>();
// cast 本身总是安全的——它不检查对齐
// 但解引用 u32_ptr_from_bytes 前必须验证对齐
if u32_ptr_from_bytes.is_aligned() {
    unsafe {
        let val = *u32_ptr_from_bytes; // 需要 unsafe 的解引用
    }
}

// ✗ 绝不安全：非法的类型大小关系
// 从 *const u64 转换到 *const u128 可能导致读取超出范围的内存
```

### 9.3 在实现中使用指针转换——Vec 内部的类型转换

在实现自定义容器时，指针类型转换是一个核心工具。例如，在一个类型泛型的 `Vec<T>` 实现中，你需要在底层操作 `*const u8`（字节级别的内存管理），但在读写元素时使用 `*const T`（类型级别的操作）：

```rust
// 概念演示：Vec<T> 内部的指针类型转换
struct MyVec<T> {
    // 底层存储为 *mut u8——字节级别管理内存
    raw_buffer: *mut u8,
    len: usize,
    cap: usize,
    _marker: std::marker::PhantomData<T>,
}

impl<T> MyVec<T> {
    // 获取指向第 index 个元素的 T 类型指针
    fn element_ptr(&self, index: usize) -> *const T {
        assert!(index < self.len);
        // 从字节指针转换为元素类型指针
        let byte_ptr = unsafe { self.raw_buffer.add(index * std::mem::size_of::<T>()) };
        byte_ptr.cast::<T>()
    }
    
    // 获取指向第 index 个元素的可变 T 类型指针
    fn element_ptr_mut(&mut self, index: usize) -> *mut T {
        assert!(index < self.len);
        let byte_ptr = unsafe { self.raw_buffer.add(index * std::mem::size_of::<T>()) };
        byte_ptr.cast::<T>()
    }
    
    // 读取第 index 个元素——使用 element_ptr + ptr::read
    fn get_copy(&self, index: usize) -> T 
    where T: Copy 
    {
        unsafe {
            std::ptr::read(self.element_ptr(index))
        }
    }
}
```

注意上面代码中 `raw_buffer` 是 `*mut u8`（字节指针），而 `element_ptr` 方法通过 `.cast::<T>()` 将其转换为元素类型的指针。这种"底层字节、高层类型"的双重视图是几乎所有 Rust 容器数据结构的核心模式。

## 10. 用裸指针构建自定义 Vec——综合实战

### 10.1 整体结构设计

让我们把所有学到的裸指针知识综合起来，实现一个简化但功能完整的 `RawVec<T>`——一个管理堆内存的底层类型。这不是生产级别的代码，但它包含了你在真实 unsafe 编程中遇到的几乎所有模式：

```rust
use std::alloc::{self, Layout};
use std::ptr::NonNull;

/// 一个管理堆上连续内存的底层类型
/// 不关心元素是否初始化——那是上层的责任
struct RawVec<T> {
    // NonNull 承诺这个指针不为 null（当 cap > 0 时）
    ptr: NonNull<T>,
    cap: usize,
    _marker: std::marker::PhantomData<T>,
}

// 当 T: Send 时，RawVec<T> 是 Send
// 因为我们对 ptr 的使用是单所有者的
unsafe impl<T: Send> Send for RawVec<T> {}
unsafe impl<T: Sync> Sync for RawVec<T> {}
```

这里有几个值得注意的设计决策：使用 `NonNull<T>` 而不是 `*mut T` 表明我们保证了这个指针在 `cap > 0` 时非空；使用 `PhantomData<T>` 让编译器知道这个结构体"逻辑上拥有" T 类型的值，从而正确处理 drop 检查和 variance。

### 10.2 内存分配和释放

```rust
impl<T> RawVec<T> {
    /// 创建一个容量为 cap 的 RawVec
    fn new(cap: usize) -> Self {
        if cap == 0 {
            // 零容量——不需要分配，使用 dangling 占位
            return RawVec {
                ptr: NonNull::dangling(),
                cap: 0,
                _marker: std::marker::PhantomData,
            };
        }
        
        // 计算 T 类型的内存布局
        // Layout::array 在 len * size_of::<T>() 溢出时返回 Err
        let layout = Layout::array::<T>(cap)
            .expect("capacity 过大导致内存布局计算溢出");
        
        // 分配内存
        let raw_ptr = unsafe { alloc::alloc(layout) };
        
        // 处理分配失败
        let ptr = match NonNull::new(raw_ptr as *mut T) {
            Some(p) => p,
            None => alloc::handle_alloc_error(layout),
        };
        
        RawVec {
            ptr,
            cap,
            _marker: std::marker::PhantomData,
        }
    }
    
    /// 获取原始指针
    fn ptr(&self) -> *mut T {
        self.ptr.as_ptr()
    }
    
    /// 获取容量
    fn cap(&self) -> usize {
        self.cap
    }
}
```

### 10.3 扩容——裸指针迁移数据

```rust
impl<T> RawVec<T> {
    /// 扩容到 new_cap（new_cap 必须 > self.cap）
    fn grow(&mut self, new_cap: usize) {
        assert!(new_cap > self.cap, "grow 的 new_cap 必须大于当前容量");
        
        let new_layout = Layout::array::<T>(new_cap)
            .expect("new_cap 过大");
        
        let old_layout = Layout::array::<T>(self.cap)
            .expect("old_cap 不应该溢出");
        
        let new_ptr: *mut u8;
        
        if self.cap == 0 {
            // 从零容量扩容——直接分配
            new_ptr = unsafe { alloc::alloc(new_layout) };
        } else {
            // 已有分配——使用 realloc 扩大
            unsafe {
                new_ptr = alloc::realloc(
                    self.ptr.as_ptr() as *mut u8,
                    old_layout,
                    new_layout.size(),
                );
            }
        }
        
        // 处理分配失败
        self.ptr = match NonNull::new(new_ptr as *mut T) {
            Some(p) => p,
            None => alloc::handle_alloc_error(new_layout),
        };
        self.cap = new_cap;
    }
    
    /// 收缩容量
    fn shrink(&mut self, new_cap: usize) {
        assert!(new_cap <= self.cap, "shrink 的 new_cap 不能大于当前容量");
        
        if new_cap == 0 {
            // 收缩到零——释放旧内存
            if self.cap > 0 {
                let old_layout = Layout::array::<T>(self.cap)
                    .expect("old_cap 不应该溢出");
                unsafe {
                    alloc::dealloc(self.ptr.as_ptr() as *mut u8, old_layout);
                }
                self.ptr = NonNull::dangling();
                self.cap = 0;
            }
            return;
        }
        
        let new_layout = Layout::array::<T>(new_cap)
            .expect("new_cap 溢出");
        
        let old_layout = Layout::array::<T>(self.cap)
            .expect("old_cap 不应溢出");
        
        // realloc 可以缩小分配
        let new_ptr = unsafe {
            alloc::realloc(
                self.ptr.as_ptr() as *mut u8,
                old_layout,
                new_layout.size(),
            )
        };
        
        self.ptr = match NonNull::new(new_ptr as *mut T) {
            Some(p) => p,
            None => alloc::handle_alloc_error(new_layout),
        };
        self.cap = new_cap;
    }
}
```

### 10.4 Drop 实现——释放内存资源

```rust
impl<T> Drop for RawVec<T> {
    fn drop(&mut self) {
        if self.cap == 0 {
            return;
        }
        
        let layout = Layout::array::<T>(self.cap)
            .expect("cap 在构造时已验证，这里不应溢出");
        
        unsafe {
            // 注意：这里只释放原始内存，不调用元素的 drop
            // 如果元素有 Drop 实现，调用者必须在上层先 drop 每个元素
            // RawVec 只负责原始内存的管理
            alloc::dealloc(self.ptr.as_ptr() as *mut u8, layout);
        }
    }
}
```

这个 `RawVec<T>` 展示了裸指针编程的完整图景：`NonNull` 用于保证非空、`alloc` 进行堆分配、`Layout` 处理对齐和大小、`Drop` 确保资源释放。注意一个重要的设计决策：`RawVec` 不负责元素的初始化和析构——这是上层类型（如完整的 `Vec<T>`）的责任。这种关注点分离是 Rust 标准库中实际采用的设计。

## 11. Drop Check 与裸指针——为什么裸指针不实现 Drop

### 11.1 裸指针的 Drop 豁免

`*const T` 和 `*mut T` 都不实现 `Drop`。这不是偶然的——这是因为裸指针不代表所有权。一个裸指针不知道它指向的内存是否需要释放、由谁释放、何时释放。自动 Drop 一个裸指针几乎肯定是错误的行为：

```rust
// 为什么裸指针不实现 Drop：它们不知道如何处理指向的内存

let mut s = String::from("owned data");
let ptr: *mut String = &mut s;

// 当 ptr 离开作用域时——什么都不发生
// 这很好！因为 ptr 不拥有 String，s 仍然持有所有权
// 如果 *mut T 实现了 Drop 并尝试释放 *ptr，那将是灾难

// 对比：当一个包含裸指针的类型离开作用域时——
// 必须由该类型的 Drop 实现来释放裸指针指向的内存
// （如前面 RawVec 的 Drop 实现）
```

### 11.2 Drop Check 与 PhantomData

如果你在类型中包含裸指针但不加 `PhantomData<T>`，编译器可能会做出错误的 Drop 推断：

```rust
use std::marker::PhantomData;

// ❌ 没有 PhantomData —— 编译器认为这个类型不拥有 T
struct BadVec<T> {
    ptr: *mut T,
    len: usize,
    cap: usize,
}
// 问题：编译器可能认为 BadVec<T> 在 T 被 drop 之前可以被安全地 drop
// 但实际上 BadVec<T> 逻辑上拥有 T 类型的值——先 drop BadVec 会导致 use-after-free

// ✓ 有 PhantomData —— 编译器知道这个类型逻辑上拥有 T
struct GoodVec<T> {
    ptr: *mut T,
    len: usize,
    cap: usize,
    _marker: PhantomData<T>,
}
// PhantomData<T> 告诉编译器："虽然我没有直接包含 T，但我逻辑上拥有 T"
// 这确保 drop checker 正确推理生命周期
// 也可以使用 std::mem::ManuallyDrop<T> 来更精确地表达所有权
```

`PhantomData<T>` 是一个零大小的类型标记，它的唯一作用是参与类型检查和 Drop Check。在编写包含裸指针的容器类型时，你应该总是评估是否需要它——这是一个容易忽略但影响深远的设计细节。

```rust
use std::ptr::NonNull;

struct ArrayIter {
    current: NonNull<i32>,
    end: *const i32,
}

impl ArrayIter {
    fn new(arr: &[i32; 5]) -> Self {
        let start = arr.as_ptr() as *mut i32;
        let end = unsafe { start.add(5) } as *const i32;
        ArrayIter {
            current: NonNull::new(start).expect("数组指针不为 null"),
            end,
        }
    }
}

impl Iterator for ArrayIter {
    type Item = i32;

    fn next(&mut self) -> Option<Self::Item> {
        // 检查是否已到达末端
        if self.current.as_ptr() as *const i32 == self.end {
            return None;
        }
        
        unsafe {
            // 读取当前位置的元素
            // 安全条件：
            // 1. current 指向有效的已初始化 i32（在数组范围内）
            // 2. current 正确对齐（数组保证对齐）
            // 3. 不会 double-read 同一位置的元素——每次 next 后 current 前进
            let value = std::ptr::read(self.current.as_ptr());
            
            // 前进到下一个元素
            self.current = NonNull::new(self.current.as_ptr().add(1))
                .expect("不会越界——已检查 end");
            
            Some(value)
        }
    }
}

// 测试：
// let arr = [10, 20, 30, 40, 50];
```

### 练习 2 答案

```rust
// 问题分析：
// copy_nonoverlapping 要求源和目标不重叠。
// 在这个例子中，源是 ptr（指向位置 0），目标是 ptr.add(1)（指向位置 1），
// 复制长度是 len - 1。
// 
// 以 vec![1, 2, 3] 为例：
//   源区域：位置 [0..2] → [1, 2]
//   目标区域：位置 [1..3] → [2, 3]
//   它们重叠了！（位置 1 和 2 在源和目标中都存在）
//
//   这就是 UB——copy_nonoverlapping 假设不重叠，行为未定义。
//   实际效果可能是 [1, 1, 1] 或 [1, 1, 2] 或任意值——

fn duplicate_first_element_fixed(v: &mut Vec<u32>) {
    if v.len() < 2 {
        return; // 至少需要两个元素
    }
    
    // 先扩容一个位置
    v.push(0); // 临时值，会被覆盖
    
    unsafe {
        let ptr = v.as_mut_ptr();
        let len = v.len();
        
        // 修复：使用 copy 而不是 copy_nonoverlapping
        // copy 能正确处理重叠内存——它会自动选择安全的复制方向
        // 从后向前复制（先把最后一个元素移到新位置）
        std::ptr::copy(
            ptr,       // 源: [0..len-2]
            ptr.add(1), // 目标: [1..len-1]
            len - 2,   // 复制除最后一个元素外的所有有效元素
        );
        
        // 现在位置 1 和位置 0 已经有了相同的值（都是原来的位置 0 的值）
        // ptr[0] 仍然是原来的值——因为从后向前复制，它没被覆盖
    }
    
    // 或者更简单的方式（更符合 Rust 习惯）：
    // let first = v[0];
    // v.insert(1, first);
}

// 实际测试：
// let mut v = vec![1, 2, 3];
// duplicate_first_element_fixed(&mut v);
// assert_eq!(v, vec![1, 1, 2, 3]);
// 
// 注意：上面的修复实现比较复杂，
// 实际项目中直接使用 v.insert(1, v[0]) 是最佳选择
```

### 练习 3 答案

```rust
use std::alloc::{self, Layout};
use std::ptr::NonNull;

struct RingBuffer {
    buf: NonNull<i32>,
    cap: usize,
    // 读取位置（出队位置）
    read_pos: usize,
    // 写入位置（入队位置）
    write_pos: usize,
    // 当前元素数量
    count: usize,
}

impl RingBuffer {
    fn new(capacity: usize) -> Self {
        assert!(capacity > 0, "容量必须大于 0");
        
        let layout = Layout::array::<i32>(capacity)
            .expect("capacity 过大");
        let raw_ptr = unsafe { alloc::alloc(layout) };
        let buf = NonNull::new(raw_ptr as *mut i32)
            .unwrap_or_else(|| alloc::handle_alloc_error(layout));
        
        RingBuffer {
            buf,
            cap: capacity,
            read_pos: 0,
            write_pos: 0,
            count: 0,
        }
    }
    
    fn push(&mut self, value: i32) -> Result<(), &'static str> {
        if self.count == self.cap {
            return Err("缓冲区已满");
        }
        
        unsafe {
            // 安全：write_pos < cap，buf 指向有效的分配
            let target = self.buf.as_ptr().add(self.write_pos);
            std::ptr::write(target, value);
        }
        
        self.write_pos = (self.write_pos + 1) % self.cap;
        self.count += 1;
        Ok(())
    }
    
    fn pop(&mut self) -> Option<i32> {
        if self.count == 0 {
            return None;
        }
        
        let value = unsafe {
            // 安全：read_pos < cap，该位置已通过 push 初始化
            let source = self.buf.as_ptr().add(self.read_pos);
            std::ptr::read(source)
        };
        
        self.read_pos = (self.read_pos + 1) % self.cap;
        self.count -= 1;
        Some(value)
    }
    
    fn len(&self) -> usize {
        self.count
    }
    
    fn is_empty(&self) -> bool {
        self.count == 0
    }
    
    fn is_full(&self) -> bool {
        self.count == self.cap
    }
}

impl Drop for RingBuffer {
    fn drop(&mut self) {
        // 先 drop 所有剩余的有效元素
        while let Some(_) = self.pop() {
            // pop 已经通过 ptr::read 取出了值，
            // 返回的 Option<i32> 会在离开作用域时被 drop
        }
        
        // 然后释放底层内存
        let layout = Layout::array::<i32>(self.cap)
            .expect("cap 在构造时已验证");
        unsafe {
            alloc::dealloc(self.buf.as_ptr() as *mut u8, layout);
        }
    }
}

// 测试：
// let mut rb = RingBuffer::new(3);
// assert!(rb.push(1).is_ok());
// assert!(rb.push(2).is_ok());
// assert!(rb.push(3).is_ok());
// assert!(rb.push(4).is_err());  // 已满
// assert_eq!(rb.pop(), Some(1));
// assert_eq!(rb.pop(), Some(2));
// assert!(rb.push(4).is_ok());   // 现在有空间了
// assert_eq!(rb.pop(), Some(3));
// assert_eq!(rb.pop(), Some(4));
// assert_eq!(rb.pop(), None);    // 空
```

> **转型之术**：裸指针是 unsafe Rust 的"汇编层"——它们给了你 C 语言的灵活性，但也要求你像 C 程序员一样小心内存管理。掌握了 ptr::read/write/copy 这套原语，你就拥有了手动内存管理的全套工具。下一个课题 [[4.3-transmute-type-punning]] 将进入更危险的领域——byte-level 的类型重解释（type punning）。`transmute` 让你无视类型系统，将一个类型的 bit pattern 重新解释为另一个类型——它极度强大，也极度危险。理解 transmute 的边界和替代方案（union-based punning、bytemuck 等）是你 Rust unsafe 工具箱中的最后一件利器。
