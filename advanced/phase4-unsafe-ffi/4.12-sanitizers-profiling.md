# 4.12 Sanitizers & Profiling —— 找到隐藏的 bug，找到真正的瓶颈

> Rust 的 borrow checker 在编译时消灭了 90% 的内存错误——这是 Rust 引以为傲的成就。但剩下的 10% 发生在 unsafe 代码、FFI 边界和不安全的底层操作中。这些错误不会触发编译错误，也不会总是触发 panic。它们静静地潜伏在你的代码中，等待最不幸的时刻出现——生产环境的凌晨三点，一个特定的输入触发了 buffer overflow。Sanitizers（消毒器）正是为"剩下的 10%"设计的——它们注入运行时检查，让你在测试阶段就能捕获这些错误。ASan 检测内存错误，TSan 检测数据竞争，LSan 检测内存泄漏，MSan 检测未初始化内存的读取。配合 Miri（MIR 解释器）检测纯 Rust 中的 UB，以及 profiling 工具链定位性能瓶颈，你得到的不仅是"Rust 的安全保证"，而是一个完整的"发现-诊断-修复"工作流。

## 1. Sanitizers 概述——运行时检测的四种武器

### 1.1 什么是 Sanitizer——编译时注入的运行时检测

Sanitizer 不是独立的工具。它们是编译器（LLVM）在编译时向你的代码中注入的额外检测代码。当你编写 `a[i] = x` 时，ASan 编译的版本会额外检查 `i < len`、`指针有效`、`内存未被释放`。这些检查在正常编译中不存在，只在启用 sanitizer 时才注入。

Sanitizer 注入的检测代码是高效的——在编译器层面融合，而非普通的函数调用。尽管如此，它们仍然会引入显著的开销。ASan 的典型开销是 2 倍慢，TSan 是 5-15 倍慢，MSan 是约 3 倍慢。你只在开发和测试阶段启用它们，从不用于生产环境（受影响的不仅是性能，还包括内存开销——ASan 额外消耗 2-3 倍内存）。

### 1.2 Sanitizer 与 Rust 的编译器检查——互补关系

Rust 的编译器已经帮你消灭了至少以下错误类别：use-after-free（所有权系统）、double-free（所有权系统）、null pointer dereference（`Option<&T>` 系统）、data races（`Send + Sync` trait 系统）、iterator invalidation（borrow checker）。但以下错误仍然可能发生：unsafe 代码中的 raw pointer 错误、FFI 调用 C 库导致的内存错误、`unsafe` trait 实现中的不变量违反、多线程中的 lock-free 数据结构错误、未初始化内存的读取（通过 `MaybeUninit` 或 FFI）、栈缓冲区溢出。

Sanitizer 填补的正是这个空白——它们不依赖类型系统，而是通过运行时检测捕获所有实际发生的错误（不仅有 unsafe Rust 的，还有 C 库的）。

## 2. Address Sanitizer (ASan) —— 内存错误的终极检测

### 2.1 ASan 检测什么——四类内存错误

ASan 是目前最成熟的 sanitizer，由 Google 的 Konstantin Serebryany 开发。它检测四类最致命的内存错误。

第一类是 heap use-after-free（堆内存释放后使用）——通过将已释放的内存标记为"红区"，任何对红区的访问都会触发 ASan 报告。第二类是 heap buffer overflow（堆缓冲区溢出）——在每个堆分配周围放置守卫区（red zone），溢出即报错。第三类是 stack buffer overflow（栈缓冲区溢出）——通过栈帧中的红区保护局部数组。第四类是 use-after-return（全局空间中的使用后返回错误）——通过将栈变量保存到 shadow stack 检测。

ASan 不直接检测未初始化内存的读取（那是 MSan 的职责）和内存泄漏（那是 LSan 的职责，但 LSan 通常与 ASan 一起启用）。

### 2.2 启用 ASan —— RUSTFLAGS 配置

在 Rust 中启用 ASan 非常简单——通过 `RUSTFLAGS` 环境变量传递 `-Z sanitizer=address` 给编译器。

启用 ASan 的完整步骤：
1. 确保你使用的是 nightly Rust（`rustup default nightly` 或 `cargo +nightly`）
2. 设置环境变量：`RUSTFLAGS="-Z sanitizer=address" cargo +nightly test`
3. ASan 将在运行测试时检测所有内存错误

```bash
# 在命令行中（Linux/macOS）
# 对整个项目运行测试并启用 ASan
RUSTFLAGS="-Z sanitizer=address" cargo +nightly test

# 你也可以将其写在 .cargo/config.toml 中（只用于特定 target）
# [target.'cfg(all())']
# rustflags = ["-Z", "sanitizer=address"]

# 编译并运行特定的二进制文件（而非测试）
RUSTFLAGS="-Z sanitizer=address" cargo +nightly run

# 重要：ASan 在 macOS 上需要额外的链接器标志
# macOS: RUSTFLAGS="-Z sanitizer=address -C link-arg=-fsanitize=address" cargo +nightly test
```

ASan 将接管程序的 `malloc`/`free`——它用自己的实现替代系统分配器，在每个分配周围添加红区（red zone）。当你的程序访问红区时，ASan 报告错误。当你的程序释放内存后访问它时，ASan 将那块内存放入"隔离区"（quarantine zone），请求即触发报告。

### 2.3 ASan 捕获的实际错误示例

以下是一些 ASan 可以捕获的典型 Rust 错误（都涉及 unsafe 代码或 FFI）。

```rust
// 示例 1：use-after-free（堆释放后使用）
// ASan 会捕获这个错误
fn use_after_free() {
    let ptr: *mut u32;
    {
        let val = Box::new(42u32);
        ptr = Box::into_raw(val);
        // Box 被手动释放（相当于 free）
        unsafe { drop(Box::from_raw(ptr)); }
    }
    // ptr 现在是悬垂指针——ASan 会在这里报错
    unsafe {
        // ASan 报告：heap-use-after-free
        // println!("{}", *ptr); // ← 取消注释将触发 ASan 错误
    }
}

// 示例 2：heap buffer overflow（堆缓冲区溢出）
fn buffer_overflow() {
    let v: Vec<u8> = vec![0u8; 10];
    let ptr = v.as_ptr();
    unsafe {
        // 写入超出分配范围的内存——ASan 报告：heap-buffer-overflow
        // *(ptr.add(20) as *mut u8) = 42; // ← 取消注释将触发 ASan 错误
    }
}

// 示例 3：stack buffer overflow（栈缓冲区溢出）
fn stack_overflow() {
    let buf: [u8; 10] = [0u8; 10];
    let ptr = buf.as_ptr();
    unsafe {
        // 写入超出栈数组的内存——ASan 报告：stack-buffer-overflow
        // *(ptr.add(15) as *mut u8) = 99; // ← 取消注释将触发 ASan 错误
    }
}
```

每个 ASan 的报告都会包含完整的调用栈回溯——告诉你错误发生在哪个文件的哪一行、是谁分配的这块内存、是谁释放的。这在调试复杂的 unsafe 代码时是无价的。

### 2.4 ASan 的局限性——什么不会被检测到

ASan 不会检测几种错误。如果程序没有执行到有 bug 的代码路径，ASan 无法检测。如果错误是数据竞争（多线程同时读写同一变量——需要 TSan），ASan 可能检测到内存错误，但不会报告它的根因。如果读取了未初始化的内存（需要 MSan），ASan 对此视而不见。如果错误发生在 ASan 未覆盖的库中（比如预编译的第三方 C 库，除非你也用 ASan 编译它们），ASan 也无法检测。

关键启示：ASan 只能检测"被执行到的代码路径"中的错误。这就是为什么 sanitizer 应该与良好的测试覆盖率配合使用——测试越全面，sanitizer 发现的问题越多。

## 3. Thread Sanitizer (TSan) —— 发现数据竞争

### 3.1 TSan 如何工作——Happens-Before 关系图

TSan 通过构建"Happens-Before"关系图来检测数据竞争。它记录每一次内存访问（读或写）和每一次同步操作（mutex lock/unlock、atomic store/load with ordering）。如果两个线程在没有同步的情况下同时访问同一块内存，且至少有一个是写操作，TSan 报告数据竞争。

TSan 的检测不是静态的——它在程序运行时维护一个状态机，追踪每个内存位置被哪些线程在哪些同步点访问过。如果它发现一个写的 happens-before 关系无法与其他访问建立，那就是数据竞争。

### 3.2 启用 TSan 和典型错误模式

启用方式与 ASan 类似，但将 sanitizer 名称改为 `thread`：

```bash
# 启用 Thread Sanitizer
RUSTFLAGS="-Z sanitizer=thread" cargo +nightly test

# 注意：TSan 不能与 ASan 同时使用——它们是互斥的
# 而且 TSan 必须在所有被链接的代码中都启用，否则可能漏报
```

以下是一些 TSan 可以从 Rust 代码中检测到的数据竞争（这些代码在 safe Rust 中无法编译，但可以通过 unsafe 绕过）：

```rust
use std::sync::Arc;
use std::thread;

/// TSan 可检测的竞争条件示例
fn data_race_example() {
    // 使用 Arc 共享可变数据——这是"故意"的竞争
    let shared = Arc::new(std::cell::UnsafeCell::new(0u64));
    let shared2 = shared.clone();

    let t1 = thread::spawn(move || {
        for _ in 0..1_000_000 {
            unsafe {
                let ptr = shared.get();
                *ptr += 1; // ← 写操作——无同步保护
                // TSan 会在这里报告：数据竞争
            }
        }
    });

    let t2 = thread::spawn(move || {
        for _ in 0..1_000_000 {
            unsafe {
                let ptr = shared2.get();
                *ptr += 1; // ← 同时写——与 t1 的写操作竞争
            }
        }
    });

    t1.join().unwrap();
    t2.join().unwrap();
}
```

TSan 不仅会报告有竞争，还会告诉你竞争的两个线程的完整调用栈——每一方在什么时间点访问了什么内存地址。

### 3.3 TSan 能捕获而 Rust 编译器会遗漏的场景

Rust 的 `Send + Sync` 检查在编译时防止了大多数数据竞争，但存在盲区。第一是 raw pointer 操作——`*mut T` 是 `Send` 的（你可以将 raw pointer 发送到另一个线程），但通过它进行无保护的读写是 unsafe 的。第二是 `UnsafeCell`——它是安全 Rust 中实现内部可变性的基础，但直接使用需要 unsafe。第三是 FFI 中 C 库内部的数据竞争。第四是 lock-free 数据结构的手动实现——如果你用 `Atomic*` 自己实现并发容器，错误的 `Ordering` 可能导致事实上的数据竞争（而不仅是逻辑错误——TSan 会检测到）。

```rust
use std::sync::atomic::{AtomicU64, Ordering};

/// 看似正确但实际有 bug 的无锁计数器
/// TSan 可以检测这种错误
struct BuggyCounter {
    value: AtomicU64,
}

impl BuggyCounter {
    fn increment(&self) -> u64 {
        // 这是一个经典的"check-then-act"竞争
        // 读取当前值（load）然后写入新值（store）不是原子的
        // TSan 在这里报告数据竞争——因为 load + store 之间没有 happens-before
        let current = self.value.load(Ordering::Relaxed);
        self.value.store(current + 1, Ordering::Relaxed);
        current
        // 正确做法是使用 fetch_add
        // self.value.fetch_add(1, Ordering::Relaxed)
    }
}
```

## 4. Leak Sanitizer (LSan) —— 检测内存泄漏

### 4.1 LSan 的工作原理

LSan 在程序退出时扫描所有仍然存活的内存分配。对于每个分配，它检查是否有任何可达的指针指向它（从栈、寄存器、全局变量开始的可达性分析）。如果一块内存在程序退出时无法被任何变量访问，LSan 报告泄漏。

LSan 作为 ASan 的一部分运行——通常在启用 ASan 时自动包含。你也可以独立运行它（`-Z sanitizer=leak`），但这种情况很少见——单独用 LSan 的开销通常不如直接用全套 ASan 更划算。

```bash
# LSan 通常和 ASan 一起运行（ASan 会自动包含 LSan）
RUSTFLAGS="-Z sanitizer=address" cargo +nightly test

# 或者显式单独启用 LSan
# RUSTFLAGS="-Z sanitizer=leak" cargo +nightly test
```

在 Rust 中，borrow checker 消除了绝大多数"忘记释放"类型的错误。但以下情况仍可能导致泄漏：使用 `Box::leak` 或 `Vec::leak`（故意的，但可能意外）、`Rc`/`Arc` 引用循环（两个 Rc 互相持对方，形成环形引用）、FFI 中 C 库分配的内存在 Rust 侧没有对应的释放代码、全局集合无限增长（不是泄漏但行为类似）。

```rust
use std::rc::Rc;
use std::cell::RefCell;

/// Rc 引用循环——LSan 可以检测到的泄漏
fn rc_cycle_leak() {
    // 构造一个简单的循环引用：a → b → a
    struct Node {
        next: RefCell<Option<Rc<Node>>>,
    }

    let a = Rc::new(Node { next: RefCell::new(None) });
    let b = Rc::new(Node { next: RefCell::new(None) });

    // 建立循环：a.next = b, b.next = a
    *a.next.borrow_mut() = Some(b.clone());
    *b.next.borrow_mut() = Some(a.clone());

    // a 和 b 离开作用域时——refcount 都为 1（互相持有）
    // 两者都不会被 drop——这就是泄漏！
    // LSan 会在程序退出时报告这两个 Node 从未被释放
}
```

处理 Rc 循环的标准做法是使用 `Weak`——将其中一个方向的引用改为 `Weak` 引用（不增加引用计数）。`std::rc::Weak` 正是为了防止这种泄漏而设计的。

## 5. Memory Sanitizer (MSan) —— 读取未初始化内存的守卫

### 5.1 MSan 的独特职责

MSan 是 sanitizer 家族中最特殊的一个——它检测的是"读取未初始化内存"的行为。这种错误在 C 和 C++ 中极其常见，在 Rust 中主要发生在涉及 `MaybeUninit`、FFI 边界、或手动管理的未初始化缓冲区时。

Rust 的 safe 代码理论上不会有这种问题——每个变量必须初始化才能使用。但 unsafe 代码中的 `MaybeUninit<T>`、FFI 中 C 函数填充的缓冲区、手动 `alloc` 后未初始化的内存——这些都是 MSan 的舞台。

MSan 会追踪每一位（bit）的内存初始化状态。当程序尝试读取尚未被写入的字节时，MSan 报告错误。这需要编译器和运行时库的配合——MSan 维护一个 shadow memory 区域，标记哪些字节已初始化。

```bash
# 启用 Memory Sanitizer
# 注意：MSan 的支持晚于 ASan 和 TSan，在某些平台上可能需要更多配置
RUSTFLAGS="-Z sanitizer=memory" cargo +nightly test

# MSan 不能与 ASan、TSan、LSan 同时使用
# 它是独立的——因为它的 shadow memory 表示与其他 sanitizer 不兼容
```

### 5.2 Rust 中需要 MSan 的典型场景

```rust
use std::mem::MaybeUninit;

/// 错误使用 MaybeUninit——MSan 可以捕获
fn uninit_read_example() {
    // 创建一个未初始化的 u32
    let uninit: MaybeUninit<u32> = MaybeUninit::uninit();

    // 危险：假设已初始化的读取
    unsafe {
        // MSan 在这里报告：读取未初始化的内存
        // let value = uninit.assume_init(); // ← 取消注释将触发 MSan 错误
        // println!("{}", value);
    }
}

/// FFI 场景：C 函数应该填充缓冲区但实际没有
fn ffi_uninit_example() {
    let mut buf: [u8; 64] = [0u8; 64]; // Rust 此处已初始化——但 FFI 应该重新填充

    // 模拟：C 函数只填充了前 32 字节，忘了填充后 32 字节
    // 在真实的 FFI 中，这可能是 C 库的 bug
    // unsafe { some_c_function(buf.as_mut_ptr(), 32); }

    // Rust 代码假设整个 buf 被填充了——MSan 报告后 32 字节未初始化
    // process_buffer(&buf); // ← buf[32..64] 是"已初始化但实际应来自 C 函数"
}
```

`MaybeUninit` 是 Rust 中处理未初始化内存的标准类型。正确使用它的模式是：先用 `MaybeUninit::uninit()` 分配空间，然后用 `ptr::write()` 或类似的写入操作初始化，最后用 `assume_init()` 转换为已初始化类型。MSan 帮你验证这个流程中没有遗漏初始化步骤。

## 6. Sanitizers 的局限性与 Miri 的补充

### 6.1 Sanitizers 的通用局限性

所有 sanitizer 共享一个根本局限：它们只能检测被执行到的代码路径中的错误。如果 bug 隐藏在 `if config.enable_experimental_feature` 的分支中，而你的测试从未覆盖这个分支，sanitizer 就不会报告。

此外，sanitizer 有平台限制。ASan 在 Linux（x86_64、aarch64）和 macOS 上可用。TSan 主要在 Linux x86_64 上最成熟。MSan 主要在 Linux x86_64 上可用。Windows 上的 sanitizer 支持一直是滞后的（虽然 Rust 团队在推进）。

内存开销也是一个限制。ASan 会使程序内存使用量增加 2-3 倍。TSan 增加 5-10 倍。MSan 增加约 2-3 倍。对于已经接近内存上限的程序，启用 sanitizer 可能导致无法运行。

### 6.2 Miri——在 MIR 层面解释执行，发现纯 Rust 中的 UB

Miri 是一个与 sanitizer 互补的工具——它在编译器 MIR（中级中间表示）层面解释执行你的 Rust 代码。与 sanitizer 不同，Miri 不编译为机器码然后运行并注入检测。它直接在 `rustc` 的 MIR 上运行一个解释器，对每个操作进行细粒度的 UB 检查。

Miri 的优势是：它检测的是 Rust 抽象机的 UB，而不是"在特定平台上的行为"。这使它更严格、更全面。Miri 能发现：整数溢出（即使启用了 wrapping 行为）、未对齐的指针解引用（直接是 UB）、使用 dangling pointer、无效的 bool 值（不是 0 或 1）、创建无效引用（比如 `&*null()`）等。这些都是 sanitizer 难以检测的（因为它们在你平台的 CPU 上可能"碰巧工作"）。

启用 Miri 的方式：

```bash
# 安装 Miri（需要 nightly）
rustup +nightly component add miri

# 运行测试——Miri 会解释执行每个测试
cargo +nightly miri test

# 运行特定的测试
cargo +nightly miri test -- test_name

# 运行特定的示例
cargo +nightly miri run --example my_example

# 重要：Miri 不能执行 FFI 调用——如果代码中有 extern "C"，
# Miri 会返回错误，除非你事先注册了模拟函数
```

Miri 的局限也很明确。它不支持 FFI——所有外部 C 函数调用都会失败。它比 native 执行慢很多（大约 100-1000 倍，取决于代码特征）。它不能检测多线程相关的 bug（Miri 目前不支持并发）。它依赖于你的测试覆盖——只检测被执行到的路径。

### 6.3 Sanitizer vs Miri——选择指南

在实际工作中，你应该将两者结合使用，而非二选一。以下是一个快速决策指南：

- 代码涉及 FFI 调用 C 库：必须用 ASan（Miri 不支持 FFI）
- 代码是纯 Rust（也许有些 unsafe）：先用 Miri（更严格），再用 ASan（覆盖 Miri 不支持的部分）
- 代码涉及多线程：必须用 TSan（Miri 目前不支持多线程）
- 代码中使用 `MaybeUninit` 或手动内存管理：MSan + Miri
- 开发阶段的快速反馈：Miri（`cargo miri test` 更快集成到 CI）
- 发布前最后一轮的全面检查：ASan + TSan + LSan（覆盖面最广）

```bash
# 推荐的 CI 脚本流程
# 1. Miri——最严格，最快在 CI 中运行（不涉及 FFI 的测试）
cargo +nightly miri test

# 2. ASan——检测 FFI 和其他 Miri 覆盖不到的错误
RUSTFLAGS="-Z sanitizer=address" cargo +nightly test

# 3. TSan——如果代码中有多线程（单独运行，不能与 ASan 同时）
RUSTFLAGS="-Z sanitizer=thread" cargo +nightly test
```

## 7. Profiling 工具——从"正确"到"快速"

### 7.1 perf (Linux) —— CPU 性能分析的终极工具

Linux 内核自带的 `perf` 是最强大的 CPU profiling 工具。它可以直接读取 CPU 的硬件性能计数器（PMU），告诉你程序在哪些函数上花了多少 CPU 周期、发生了多少次缓存未命中、多少次分支预测失败。

基本用法如下（这些是 bash 命令，不是 Rust 代码）：

```bash
# 记录程序的性能数据
perf record --call-graph dwarf ./target/release/my_program

# 查看报告（按 CPU 周期排序）
perf report

# 生成可视化——火焰图
perf script | stackcollapse-perf.pl | flamegraph.pl > flamegraph.svg

# 实时查看
perf top -p $(pgrep my_program)
```

使用 `perf` 的关键准备步骤：确保你的二进制文件包含调试符号（在 `Cargo.toml` 的 `[profile.release]` 中设置 `debug = true`——不影响优化，只添加符号名称）。使用 `--call-graph dwarf` 或 `--call-graph fp`（frame-pointer）来获取调用栈回溯。DWARF 提供最准确的调用图，但开销较大。

### 7.2 samply —— 跨平台采样 profiler

`samply` 是一个基于 Firefox profiler 的跨平台 CPU 采样工具。它支持 Linux、macOS 和 Windows，使用每个平台的最佳 profiling 机制（Linux 上使用 perf_event_open，macOS 上使用 PC 采样线程，Windows 上使用 ETW）。

```bash
# 安装 samply
cargo install samply

# 对 Rust 程序进行 CPU 采样
samply record ./target/release/my_program

# 自动打开 Firefox profiler 显示结果
# samply 会生成一个包含调用树的火焰图 profile

# 你也可以附加到已运行的进程
samply record -p $(pgrep my_program)
```

`samply` 的优势在于它的可视化——它集成 Firefox profiler，你可以在时间线上查看 CPU 使用情况、调用栈的火焰图、每个函数的自用时间/总时间。它尤其擅长多线程程序的 profiling——你可以按线程查看性能分布。

### 7.3 cargo-flamegraph —— Rust 专用的火焰图生成器

`cargo-flamegraph` 将 `perf` 和火焰图生成封装为一个简单的 cargo 命令。

```bash
# 安装
cargo install flamegraph

# 生成火焰图（需要 Linux，使用 perf 作为后端）
cargo flamegraph --bin my_program

# 指定输出文件
cargo flamegraph --bin my_program -o flame.svg

# 传递参数给程序
cargo flamegraph --bin my_program -- --input data.txt

# 只对特定函数生成火焰图（以减少开销）
cargo flamegraph --bin my_program -- --bench

# 在 macOS 上使用 DTrace（需要 sudo）
cargo flamegraph --root --bin my_program
```

火焰图的可视化规则：横轴是代码行/函数的栈占比（越宽表示 CPU 时间越多），纵轴是调用栈深度（从底部的 main 到顶部的叶子函数），颜色无特殊含义（默认用随机暖色调区分函数）。你寻找的是"宽而平"的顶部（hot spots）——这些是性能优化的目标。

### 7.4 cargo-instruments (macOS) —— Xcode Instruments 集成

在 macOS 上，`cargo-instruments` 提供了与 Xcode Instruments 的集成，可以使用 Apple 的 Time Profiler、Allocations、Leaks 等模板。

```bash
# 安装
cargo install cargo-instruments

# CPU 时间分析（Time Profiler 模板）
cargo instruments -t time --bin my_program

# 内存分配分析（Allocations 模板）
cargo instruments -t alloc --bin my_program

# 系统调用追踪（System Trace 模板）
cargo instruments -t sys --bin my_program

# 输出到指定文件
cargo instruments -t time --bin my_program -o trace_output
```

Xcode Instruments 的优势是它的图形界面——你可以拖动时间线、查看每个分配的生命周期、追踪每个线程的状态转换。对于内存问题（泄漏、过度分配、碎片），Allocations 模板的"Generation"分析是无价的。

### 7.5 Criterion —— 统计严格的基准测试

`criterion` 是 Rust 生态中最广泛使用的基准测试框架。它不仅仅测量运行时间——它进行统计显著性分析，检测异常值，并生成可视化的性能报告。

```rust
// 使用 criterion 进行基准测试
use criterion::{black_box, criterion_group, criterion_main, Criterion};

fn fibonacci(n: u64) -> u64 {
    match n {
        0 => 0,
        1 => 1,
        _ => fibonacci(n - 1) + fibonacci(n - 2),
    }
}

fn bench_fib(c: &mut Criterion) {
    c.bench_function("fib 20", |b| {
        b.iter(|| fibonacci(black_box(20)))
    });
}

// 多组基准测试的比较
fn bench_vec_push(c: &mut Criterion) {
    let mut group = c.benchmark_group("vec_push");

    group.bench_function("preallocated", |b| {
        b.iter(|| {
            let mut v = Vec::with_capacity(1000);
            for i in 0..1000 {
                v.push(black_box(i));
            }
            black_box(v)
        })
    });

    group.bench_function("no_preallocation", |b| {
        b.iter(|| {
            let mut v = Vec::new();
            for i in 0..1000 {
                v.push(black_box(i));
            }
            black_box(v)
        })
    });

    group.finish();
}

criterion_group!(benches, bench_fib, bench_vec_push);
criterion_main!(benches);
```

`black_box` 是 criterion 基准测试中的关键元素。它阻止编译器优化掉你的代码——编译器不允许假设 `black_box` 返回的值是已知的，因此被 `black_box` 包装的值和操作无法被优化移除。没有 `black_box`，编译器可能完全删除你的基准测试代码。

`criterion` 生成的 HTML 报告包含：每个基准测试的均值、标准差、分布曲线、变化趋势（如果比较多个提交的性能）、p 值和置信区间。这些统计数据帮助你判断一个性能变化是真实的还是随机的噪声。

## 8. 实际工作流——将工具集成到开发周期中

### 8.1 开发阶段：快速反馈循环

在开发新功能或重构 unsafe 代码时，你应该配置一个快速的"预提交检查"：

```bash
# 开发阶段的快速检查（在提交前运行）
# 1. 编译检查
cargo check

# 2. 单元测试（常规测试——快速）
cargo test

# 3. Miri（针对涉及 unsafe 的模块——比 ASan 快，更严格）
# 只对特定测试文件运行以节省时间
cargo +nightly miri test -- my_unsafe_module

# 4. Clippy——catching common errors and style issues
cargo clippy -- -D warnings
```

这个循环应该在 1-2 分钟内完成，给你快速的反馈。如果你在开发 unsafe 代码（比如自定义分配器或 FFI 绑定），将 Miri 加入这个循环是特别有价值的。

### 8.2 CI 阶段：全面消毒

在 CI 中（当你的 PR 被推送或合并前），运行完整的 sanitizer 检查。这比开发循环慢，但覆盖面更广。

```bash
# CI 阶段的完整检查（示例脚本）
# 这些步骤需要在 nightly Rust 上运行

# 步骤 1：Miri——捕获纯 Rust 中的 UB
cargo +nightly miri test

# 步骤 2：ASan——捕获内存错误（包括 FFI 错误）
RUSTFLAGS="-Z sanitizer=address" cargo +nightly test

# 步骤 3：TSan——捕获多线程数据竞争
# 只对有并发代码的测试运行
RUSTFLAGS="-Z sanitizer=thread" cargo +nightly test -- --test-threads=1

# 步骤 4：LSan——作为 ASan 的一部分自动运行
# 但在步骤 2 中已是

# 步骤 5：MSan——如果涉及 MaybeUninit 或 FFI 缓冲区
# RUSTFLAGS="-Z sanitizer=memory" cargo +nightly test
```

重要提示：`cargo test -- --test-threads=1` 在 TSan 启用时很重要——多个测试同时运行可能增加 TSan 检测的复杂性和性能开销。顺序执行测试（`--test-threads=1`）是推荐的做法。

### 8.3 发布前：性能分析

当功能实现完成、所有 sanitizer 检查通过后，是 profiling 的时候。性能分析在前——不要浪费时间优化一段原本就足够快的代码。

```bash

# 1. 确保 release profile 有足够的调试信息
# 在 Cargo.toml 中添加：

# 或
samply record ./target/release/my_program
# 或
cargo flamegraph --bin my_program

# 4. 内存 profiling——找到分配热点
# 使用之前课程中介绍的 StatsAllocator 包装器
# 或使用 heaptrack（Linux）
# heaptrack ./target/release/my_program

# 5. 基准测试——测量优化的效果
cargo bench

# 6. 对比优化前后的基准测试报告
# criterion 会自动对比（如果保存了历史基线）
```

## 9. 练习

### 练习 1：诊断一个 ASan bug

以下代码包含一个 bug。运行测试，使用 ASan 找出错误，然后修复它。

```rust
/// 一个简化版的"向量"——使用 unsafe 手动管理内存
struct SimpleVec {
    ptr: *mut u32,
    len: usize,
    cap: usize,
}

impl SimpleVec {
    fn new() -> Self {
        SimpleVec {
            ptr: std::ptr::null_mut(),
            len: 0,
            cap: 0,
        }
    }

    fn push(&mut self, val: u32) {
        if self.len == self.cap {
            let new_cap = if self.cap == 0 { 4 } else { self.cap * 2 };
            let layout = std::alloc::Layout::array::<u32>(new_cap).unwrap();
            let new_ptr = unsafe { std::alloc::alloc(layout) as *mut u32 };
            unsafe {
                std::ptr::copy_nonoverlapping(self.ptr, new_ptr, self.len);
            }
            self.ptr = new_ptr;
            self.cap = new_cap;
        }
        unsafe {
            self.ptr.add(self.len).write(val);
            // BUG：如果 len == cap 且 len == 0，上面的分支不会触发
            // ptr 仍然是 null_mut
        }
        self.len += 1;
    }
}

impl Drop for SimpleVec {
    fn drop(&mut self) {
        // BUG：没有释放之前分配的旧内存
        if !self.ptr.is_null() {
            unsafe {
                let layout = std::alloc::Layout::array::<u32>(self.cap).unwrap();
                std::alloc::dealloc(self.ptr as *mut u8, layout);
            }
        }
    }
}
```

找出 SimpleVec 中的所有 bug，使用 ASan 验证（描述 ASan 会如何报告每个 bug）。

## 10. 练习答案

### 练习 1 答案

SimpleVec 中有以下 bug（ASan 会如何检测）：

**Bug 1：双重释放**——`push` 中 `old_ptr` 没有被释放。每次扩容时（`new_cap = old_cap * 2`），`old_ptr` 指向的内存被泄漏，但新的 `ptr` 覆盖了 `old_ptr`。在 `drop` 中只释放最后一个 `ptr`。ASan 会在程序退出时（通过 LSan）报告泄漏。

**Bug 2：空指针解引用**——当 `len == cap` 且初始容量为 0 时（第一次 push），`std::ptr::copy_nonoverlapping(self.ptr, new_ptr, self.len)` 中的 `self.len == 0`，所以 copy 0 个元素。但之后如果不小心使用了 `self.ptr`，它就是 null。ASan 会报告 null pointer dereference。

**Bug 3：分配大小不匹配**——`drop` 中使用 `self.cap` 而不是初始的布局参数——如果 `cap` 在某处被错误修改，可能导致释放的大小不匹配。ASan 会报告分配-释放不匹配。

**Bug 4：缺少 dealloc 旧内存**——扩容时旧的 `self.ptr` 没有被 dealloc。ASan 的 LSan 组件会报告泄漏。

修复方案如下所示。

```rust
struct SimpleVecFixed {
    ptr: *mut u32,
    len: usize,
    cap: usize,
}

impl SimpleVecFixed {
    fn new() -> Self {
        SimpleVecFixed {
            ptr: std::ptr::null_mut(),
            len: 0,
            cap: 0,
        }
    }

    fn push(&mut self, val: u32) {
        if self.len == self.cap {
            let new_cap = if self.cap == 0 { 4 } else { self.cap * 2 };
            let layout = std::alloc::Layout::array::<u32>(new_cap).unwrap();
            let new_ptr = unsafe { std::alloc::alloc(layout) as *mut u32 };
            // 修复：只复制非零的数据，且不依赖旧指针（当 old cap 为 0 时无需复制）
            if self.len > 0 {
                unsafe {
                    std::ptr::copy_nonoverlapping(self.ptr, new_ptr, self.len);
                }
            }
            // 修复：释放旧内存
            if !self.ptr.is_null() {
                unsafe {
                    let old_layout = std::alloc::Layout::array::<u32>(self.cap).unwrap();
                    std::alloc::dealloc(self.ptr as *mut u8, old_layout);
                }
            }
            self.ptr = new_ptr;
            self.cap = new_cap;
        }
        unsafe {
            self.ptr.add(self.len).write(val);
        }
        self.len += 1;
    }
}

impl Drop for SimpleVecFixed {
    fn drop(&mut self) {
        if !self.ptr.is_null() {
            unsafe {
                let layout = std::alloc::Layout::array::<u32>(self.cap).unwrap();
                std::alloc::dealloc(self.ptr as *mut u8, layout);
            }
        }
    }
}
```

### 练习 2 答案

```rust
use std::sync::atomic::{AtomicU64, Ordering};
use std::sync::Arc;
use std::thread;

/// 使用错误的 Ordering 导致的竞争
/// 问题不在于最终值（最后的值总是 100_000 + 100_000 = 200_000），
/// 而在于两个线程之间存在数据竞争——TSan 检测的是"未同步的并发访问"，
/// 而非"最终值是否正确"
fn relaxed_race() {
    let counter = Arc::new(AtomicU64::new(0));
    let counter2 = counter.clone();

    let t1 = thread::spawn(move || {
        for _ in 0..100_000 {
            // fetch_add 本身是原子的，但使用 Relaxed 意味着
            // 这个操作与其他线程的 fetch_add 之间没有 happens-before 关系
            // 这在这个特定例子中不会导致错误的值，
            // 但在更复杂的算法中可能产生问题
            counter.fetch_add(1, Ordering::Relaxed);
        }
    });

    let t2 = thread::spawn(move || {
        for _ in 0..100_000 {
            counter2.fetch_add(1, Ordering::Relaxed);
        }
    });

    t1.join().unwrap();
    t2.join().unwrap();

    println!("counter = {}", counter.load(Ordering::Relaxed));
}

// 修复：使用更强的 Ordering
fn relaxed_race_fixed() {
    let counter = Arc::new(AtomicU64::new(0));
    let counter2 = counter.clone();

    let t1 = thread::spawn(move || {
        for _ in 0..100_000 {
            // SeqCst 确保所有线程看到一致的顺序
            // 对于简单的递增操作，这虽然可能过强，
            // 但确保了正确性
            counter.fetch_add(1, Ordering::SeqCst);
        }
    });

    let t2 = thread::spawn(move || {
        for _ in 0..100_000 {
            counter2.fetch_add(1, Ordering::SeqCst);
        }
    });

    t1.join().unwrap();
    t2.join().unwrap();

    println!("counter = {}", counter.load(Ordering::SeqCst));
}

// 为什么 Rust 编译器无法捕获？
// Rust 编译器检查的是 Send + Sync 安全——Relaxed 原子操作是合法的。
// 它检查的是"类型是否在线程间传递时可能导致 undefined behavior"，
// 而不是"业务逻辑是否正确"。
// AtomicU64 是 Send + Sync 的，Relaxed 是 Ordering 的合法变体。
// 只有 TSan 能在运行时检测到数据竞争——因为它看的是实际的内存访问模式。
```

关于为何 Rust 编译器无法捕获：Rust 的 `Send + Sync` 检查确保的是内存安全（防止 UB），而不是逻辑正确性。`Ordering::Relaxed` 本身不会导致 UB——它可能导致"弱一致性"的结果，但 Rust 类型系统关心的是前者。TSan 在这里扮演了"逻辑正确性守卫"的角色——它检测的是"在运行时发生的未同步并发访问"，即使这些访问没有导致 UB。

### 练习 3 答案

```rust
/// 瓶颈分析：
/// result = result + &i.to_string() + ", ";
/// 这行代码每次都会创建一个新的 String。
/// `result + &s` 会分配新的内存来容纳 result + s。
/// 在 1000 次迭代中，这导致 1000 次分配，
/// 每次分配的大小递增，总分配量为 O(n^2)。
///
/// 这被称为 Schlemiel the Painter 算法。
/// 对于 n=1000，总分配大小约为 500,000 字节（而非最优的 ~20,000 字节）。
/// 这就是 10x 慢的原因。
///
/// ASan/StatsAllocator 会显示大量的分配事件。

/// 修复方案 1：使用 push_str（原地追加）
fn build_string_fixed() -> String {
    let mut result = String::new();
    for i in 0..1000 {
        // push_str 在 result 的容量足够时是 O(1) 的——不需要分配
        // String 有指数增长的容量策略，因此 push_str 摊销 O(1)
        use std::fmt::Write;
        write!(&mut result, "{}, ", i).unwrap();
    }
    result
}

/// 修复方案 2：使用预分配 + push_str
fn build_string_fixed_prealloc() -> String {
    // 预估最终大小——减少重新分配次数
    let estimated_size = 1000 * 6; // 每个数字约 4 位 + ", "
    let mut result = String::with_capacity(estimated_size);
    for i in 0..1000 {
        use std::fmt::Write;
        write!(&mut result, "{}, ", i).unwrap();
    }
    result
}

/// 修复方案 3：使用 join with sep（最简洁）
fn build_string_join() -> String {
    let parts: Vec<String> = (0..1000).map(|i| i.to_string()).collect();
    parts.join(", ")
}
// 方案 3 分配了一个 Vec 和最终的 String，共 2 次主要分配
// 而原方案是 1000 次分配。差异是 500x 分配次数。
```

性能差异（使用 criterion 基准测试）：原方案约 300-500 微秒，push_str 方案约 10-20 微秒，join 方案约 15-25 微秒。优化程度：15-50 倍（取决于输入大小）。
