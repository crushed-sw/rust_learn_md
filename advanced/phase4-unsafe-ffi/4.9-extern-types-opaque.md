# 4.9 extern 类型（Opaque FFI Types）—— 包装未知大小的 C 句柄

> 不是所有类型都需要暴露它们内部的一切。C 库中最常见的设计模式之一是"不透明句柄"（opaque handle）——库向外提供一个指针类型（如 `FILE*`），但从不公开其内部结构。客户端代码只能通过库的函数来操作这些句柄。这种设计在 C 中被称为 forward declaration（前向声明）或不完整类型——"我知道这个类型的存在，但我不知道它的大小。"当 Rust 需要和这类 C 类型交互时，我们面临一个根本性的挑战：Rust 的类型系统强烈倾向于"必须知道所有类型的大小"。`extern type` 是这个张力的解决方案——它在 Rust 的类型系统中为"大小在编译时未知"的类型留了一个合法的位置。

---

## 1. `extern type` —— 大小和布局未知的类型

### 1.1 问题：C 的不完整类型与 Rust 的 Sized 约束

在 C 中，不完整类型（incomplete type）是一种常见的封装手段：

```c
// C 头文件 —— 库对外暴露不完整类型
// 前向声明：编译器知道 MyLibHandle 是一个结构体，但不知道它的大小
struct MyLibHandle;

// 客户端只能通过以下函数操作句柄：
struct MyLibHandle* mylib_open(const char* config);
int mylib_process(struct MyLibHandle* handle, const char* input);
void mylib_close(struct MyLibHandle* handle);
```

客户端代码永远不知道 `struct MyLibHandle` 内部有什么字段、占用多少字节。这是 C 中实现封装的经典手段——类似于面向对象中的"私有成员"。

当 Rust 通过 FFI 和这种库交互时，碰到的第一个问题是：Rust 默认要求所有类型实现 `Sized` trait（即在编译时知道大小）。你不能声明一个 `struct MyLibHandle;` 然后在 `let h: MyLibHandle;` 中使用它——编译器会报"recursive type with infinite size"或类似的错误。

```rust
// 这在 Rust 中是非法的——类型大小未知，无法实例化
// struct MyLibHandle;  // 零大小结构体——但 C 侧不是零大小！
// let h: MyLibHandle;  // 编译错误：类型大小未知
```

Rust 的 `extern type` 特性（目前在 nightly 中可用，通过 `#![feature(extern_types)]`）专门解决这个问题——它声明一个外部类型，其大小和对齐在编译时无法确定。

```rust
// nightly 功能——extern type 声明
#![feature(extern_types)]

extern "C" {
    // 声明一个 C 侧的类型——大小和布局对 Rust 编译器透明
    // Rust 只知道它是一个类型，对于其他信息一无所知
    type MyLibHandle;
}
```

### 1.2 extern type 的使用限制

由于大小未知，`extern type` 有一组严格的使用限制。这些限制不是任意的——每一条都源自"不知道大小意味着什么"这个核心事实。

```rust
#![feature(extern_types)]

extern "C" {
    type OpaqueType;
}

// ✅ 允许：通过指针使用（指针的大小是已知的——就是 usize）
fn use_by_pointer(h: *mut OpaqueType) {
    // 可以通过指针读写
    // unsafe { (*h).do_something(); }  // 但这依赖 OpaqueType 的实际布局
}

// ✅ 允许：通过引用使用（&T / &mut T 的指针大小已知）
fn use_by_reference(h: &OpaqueType) {
    // 引用背后是指针——大小已知
}

// ✅ 允许：放在指针包装中
fn use_box_like() -> *mut OpaqueType {
    // 可以用 Box::into_raw 的模式，但需要 unsafe 的构造方式
    std::ptr::null_mut()
}

// ❌ 不允许：直接使用值——编译器无法计算目标大小
// fn use_by_value(h: OpaqueType) {}           // 编译错误
// let h: OpaqueType;                          // 编译错误
// fn array_of_opaque() -> [OpaqueType; 10] {} // 编译错误
// fn return_opaque() -> OpaqueType {}         // 编译错误

// ❌ 不允许：Box<T> 要求 T: Sized —— 非泛型参数有隐式 Sized bound
// let b: Box<OpaqueType> = Box::new(???);      // 编译错误

// ❌ 不允许：放在没有指针间接的结构体中
// struct Wrapper { inner: OpaqueType }         // 编译错误
// 但这是允许的：
// struct Wrapper { inner: *mut OpaqueType }    // ✅
```

核心原则是：`extern type` 只能通过一层指针间接（`*const T`、`*mut T`、`&T`、`&mut T`）使用，永远不能作为值来传递、返回或存储。这是因为编译器需要为栈上的每个值分配空间，而不知道大小的类型无法分配。

### 1.3 稳定 Rust 中的替代方案

在稳定 Rust 中（没有 `extern_types` feature），有三种主要的替代方案来包装不透明类型：

**替代方案 1：零大小 DST 模拟（bindgen 的方式）。** 声明一个有私有零大小字段的结构体。这个结构体本身是 Sized 的（大小 = 0），但它无法被外部构造。

```rust
// 稳定 Rust 中最常见的替代——零大小占位结构体
#[repr(C)]
pub struct OpaqueHandle {
    _private: [u8; 0],  // 零大小字段——防止外部构造
    // 关键：_private 是私有字段，外部 crate 无法构造此结构体
}

// 使用方式——和 extern type 一样，通过指针使用
fn create_handle() -> *mut OpaqueHandle {
    // 只有定义 OpaqueHandle 的模块可以"知道"如何构造
    // 外部只能通过 FFI 函数获取指针
    std::ptr::null_mut()
}
```

这种方式的局限是 `OpaqueHandle` 在 Rust 侧的大小是 0，而 C 侧的实际大小不是 0。不过由于我们只通过指针使用它，这个大小不一致不会造成实际问题。bindgen 默认使用这种方式生成不透明类型的绑定。

**替代方案 2：`*mut c_void` 完全擦除类型信息。** 将句柄完全擦除为 `*mut c_void`，不给它任何 Rust 类型标识。

```rust
use std::os::raw::c_void;

// 最简单的替代——完全类型擦除
extern "C" {
    fn open_resource(config: *const std::os::raw::c_char) -> *mut c_void;
    fn process_resource(handle: *mut c_void) -> i32;
    fn close_resource(handle: *mut c_void);
}
```

这种方式的代价是损失了类型安全——编译器不会帮你区分 `*mut c_void` 是一个文件句柄还是数据库连接。你可能会意外地将一个句柄传给错误的函数，编译器无法检测到。

**替代方案 3：不透明新类型包装 `*mut c_void`。** 用一个单元结构体包裹 `*mut c_void` 恢复类型安全。

```rust
use std::os::raw::c_void;

/// 一个类型安全的不透明句柄包装
/// 在类型系统中将不同类型的不透明句柄区分开
pub struct FileHandle(*mut c_void);
pub struct DbHandle(*mut c_void);
pub struct NetworkHandle(*mut c_void);

// 现在编译器不会让你把 FileHandle 传给需要 DbHandle 的函数
fn use_file(h: &FileHandle) { /* ... */ }
fn use_db(h: &DbHandle) { /* ... */ }

// use_file(&db_handle);  // 编译错误——类型不匹配！
```

这种方案在稳定 Rust 中广泛使用，在保持类型安全的同时不依赖任何 nightly feature。

## 2. `Box<OpaqueHandle>` 的问题与 `NonNull` 方案

### 2.1 为什么 Box\<extern type\> 不可行

`Box<T>` 在其标准定义中要求 `T: Sized`。这是因为 `Box<T>` 在构造时需要知道要分配多少字节的内存——如果 `T` 的大小未知，`Box::new()` 无法决定向分配器请求多少字节。

但 `Box` 对于不透明类型的实际使用场景几乎是完美的——它表示唯一所有权、提供自动释放、可以安全地转移所有权。问题是 `T: Sized` 的约束阻止了直接使用。

```rust
#![feature(extern_types)]

extern "C" {
    type MyLibHandle;
}

// ❌ 编译错误：MyLibHandle 没有实现 Sized
// let box_handle: Box<MyLibHandle> = ...;
```

对于 `extern type`，我们不能使用 `Box`。但对于稳定 Rust 中的零大小占位结构体方案，`Box` 可以工作——只是它分配的大小可能和 C 侧的实际大小不一致（因为结构体编译为 0 大小），这会导致 UB。

### 2.2 `NonNull<T>` —— 非空指针包装

`NonNull<T>` 是替代 `Box<T>` 处理不透明类型的最佳工具。它表示一个永远不为 null 的裸指针，在编译时优化了 `Option<NonNull<T>>` 的大小（利用 null 值表示 None）。

```rust
use std::ptr::NonNull;

// NonNull 是对不透明指针的最佳包装——不像 Box，它不要求 T: Sized
pub struct SafeHandle {
    ptr: NonNull<OpaqueHandle>,
    // NonNull 保证内部指针不为 null——去除了 null 检查的需要
}

impl SafeHandle {
    /// 从 C 函数创建 SafeHandle——检查 null 返回
    pub fn from_raw(raw: *mut OpaqueHandle) -> Option<Self> {
        NonNull::new(raw).map(|ptr| SafeHandle { ptr })
    }

    /// 获取内部的原始指针（用于传给 C 函数）
    pub fn as_ptr(&self) -> *mut OpaqueHandle {
        self.ptr.as_ptr()
    }
}
```

`NonNull<T>` 不需要 `T: Sized`（这是它相对于 `Box<T>` 的核心优势），但它需要 `T` 是一个合法的类型。结合零大小占位结构体或 nightly 的 `extern type`，`NonNull` 提供了不透明类型的最佳指针包装。

### 2.3 对比各种包装方案

以下表格总结了在不同场景下应该使用的指针类型：

| 场景 | Rust 类型 | T: Sized 需求 | 说明 |
|---|---|---|---|
| 拥有所有权的句柄（稳定） | `NonNull<Wrapper>` | 否 | 最佳选择——类型安全 + 非空 + 无自动 Drop |
| 拥有所有权 + 自动释放 | 自定义 Drop 包装 `NonNull<T>` | 否 | 手动实现 Drop 调用 C 的 free 函数 |
| null-able 句柄 | `*mut T` 或 `Option<NonNull<T>>` | 否 | Option<NonNull> 的大小优化（与裸指针相同） |
| 借用句柄 | `&T` 或 `*const T` | 否 | 短生命周期——调用者保证有效性 |
| 共享所有权句柄 | `Arc<HandleWrapper>` | 是（HandleWrapper 是 Sized） | 多个所有者——Rust 侧管理引用计数 |
| 完全类型擦除 | `*mut c_void` | N/A | 最简单但损失类型安全 |

## 3. Drop 包装器 —— 为不透明句柄创建 RAII 安全封装

### 3.1 基础模式：newtype + Drop

将不透明指针包装为 Rust 新类型，并在 `Drop` 中调用 C 的释放函数，这是处理 FFI 句柄的最基本模式。以下是一个通用模板：

```rust
use std::ptr::NonNull;

// 第一步：声明不透明类型（零大小占位结构体）
#[repr(C)]
pub struct OpaqueHandle {
    _private: [u8; 0],
}

// 第二步：声明 C 函数
extern "C" {
    fn resource_create() -> *mut OpaqueHandle;
    fn resource_destroy(handle: *mut OpaqueHandle);
    fn resource_use(handle: *const OpaqueHandle) -> i32;
}

// 第三步：创建安全的 RAII 包装
pub struct Resource {
    handle: NonNull<OpaqueHandle>,
}

impl Resource {
    /// 创建资源——可能失败（C 函数返回 null 表示失败）
    pub fn new() -> Option<Self> {
        let raw = unsafe { resource_create() };
        NonNull::new(raw).map(|handle| Resource { handle })
    }

    /// 使用资源——安全地委托给 C 函数
    pub fn use_resource(&self) -> i32 {
        unsafe { resource_use(self.handle.as_ptr() as *const OpaqueHandle) }
    }

    /// 获取原始指针——用于高级场景
    pub fn as_ptr(&self) -> *mut OpaqueHandle {
        self.handle.as_ptr()
    }
}

// 第四步：实现 Drop —— 在 Resource 离开作用域时自动调用 C 的销毁函数
impl Drop for Resource {
    fn drop(&mut self) {
        unsafe {
            resource_destroy(self.handle.as_ptr());
        }
    }
}

// 使用示例：
// {
//     let res = Resource::new().expect("failed to create resource");
//     let status = res.use_resource();
//     println!("resource status: {}", status);
// } // ← Drop 自动调用 resource_destroy——不会忘记清理
```

这个模式解决了 C 资源管理中最令人头疼的问题——忘记释放。由于 `Drop` 在作用域退出时自动运行，即使是提前返回、panic 展开（前提是 `extern "C"` 函数使用了 `catch_unwind`），资源也会被正确清理。

### 3.2 增强模式：防止 use-after-drop 和 double-drop

基础的 Drop 包装有一个缺陷：如果在 `drop` 后代码仍然持有句柄的副本（裸指针），就可能导致 use-after-free。我们可以通过以下增强来缓解：

```rust
use std::ptr::NonNull;

pub struct Resource {
    handle: NonNull<OpaqueHandle>,
    // 添加一个"已释放"标记——防止 double-free
    // 实际上 Drop 只调用一次；这个标记主要用于调试
    dropped: bool,
}

impl Resource {
    pub fn new() -> Option<Self> {
        let raw = unsafe { resource_create() };
        NonNull::new(raw).map(|handle| Resource {
            handle,
            dropped: false,
        })
    }

    /// 消费 Resource 并返回原始指针（转移所有权给调用者）
    /// 使用 std::mem::forget 防止 Drop 被调用
    pub fn into_raw(mut self) -> *mut OpaqueHandle {
        let ptr = self.handle.as_ptr();
        // 标记为已释放——如果由于某种原因 Drop 被调用（比如 panic），
        // 它能检测到并跳过 C 的销毁调用
        self.dropped = true;
        // forget(self) 阻止 Drop——所有权已转移
        std::mem::forget(self);
        ptr
    }

    /// 从原始指针重建 Resource（接管所有权）
    /// 安全性：ptr 必须来自 Resource::into_raw() 或 C 的 create 函数
    pub unsafe fn from_raw(ptr: *mut OpaqueHandle) -> Self {
        Resource {
            handle: NonNull::new_unchecked(ptr),
            dropped: false,
        }
    }
}

impl Drop for Resource {
    fn drop(&mut self) {
        if !self.dropped {
            unsafe { resource_destroy(self.handle.as_ptr()); }
            self.dropped = true;
        }
    }
}
```

`into_raw` / `from_raw` 这一对方法是非 `Copy` 所有权的经典模式——它们允许在特殊场景下将所有权转移出安全的 Rust 包装，常见于需要把句柄传给 C 侧长期持有的情况。

### 3.3 实现 `Send` 和 `Sync`

不同的 C 库对其句柄的线程安全性有不同的保证。你应该根据 C 库的实际线程模型来决定是否为包装类型实现 `Send` 和 `Sync`。

```rust
// 场景 1：库文档声明其句柄可以安全地在线程间移动
// 例如：libgit2 的 git_repository 可以跨线程 move
unsafe impl Send for RepositoryHandle {}

// 场景 2：库文档声明其句柄可以安全地被多个线程共享
// 例如：引用计数的数据库连接句柄
unsafe impl Sync for ConnectionHandle {}

// 场景 3：库文档声明其句柄不是线程安全的
// 不实现 Send —— 编译器会在编译时阻止跨线程移动
// （默认情况下：包含 *mut T 的类型是 !Send 的，所以不需要手动标记）
```

`unsafe impl Send` 和 `unsafe impl Sync` 是编译器的信任声明——你声明你已验证了 C 库在多线程下的行为。错误的标记可能导致数据竞争和 UB，需要非常谨慎。

## 4. Phantom Type 模式 —— 给不透明句柄加上类型状态

### 4.1 动机：用类型系统编码句柄状态

许多 C 库的不透明句柄有隐含的状态机——比如一个数据库连接可能是"已连接"或"已断开"，一个文件句柄可能是"已打开"或"已关闭"。在 C 中，这些状态通常通过运行时检查和错误码来表示——如果对已关闭的连接调用查询，返回一个错误码。

Rust 的类型系统可以做得更好——通过 phantom type（幽灵类型），我们可以将状态编码到类型中，在编译时阻止非法操作。

```rust
use std::marker::PhantomData;
use std::ptr::NonNull;

// 定义状态标记——零大小的单元结构体
// 这些永远不持有数据，只作为编译时的类型标签
pub struct Connected;
pub struct Disconnected;

// 泛型包装——H 是不透明句柄类型，S 是状态标记
pub struct Database<H, S> {
    handle: NonNull<H>,
    // PhantomData 表示"这个类型在逻辑上拥有 H 和 S，但不需要在运行时存储它们"
    // 它告诉 drop checker 和类型系统正确的 ownership 关系
    _phantom: PhantomData<(H, S)>,
}

// Disconnected 状态的方法
impl<H> Database<H, Disconnected> {
    /// 创建一个新的、断开状态的数据库包装
    /// 安全性：ptr 必须来自 C 的 open 函数
    pub unsafe fn new(ptr: *mut H) -> Self {
        Database {
            handle: NonNull::new_unchecked(ptr),
            _phantom: PhantomData,
        }
    }

    /// 连接到数据库——状态从 Disconnected 变为 Connected
    pub fn connect(self) -> Result<Database<H, Connected>, String>
    where
        H: Connectable,
    {
        unsafe { H::connect(self.handle.as_ptr()) }?;
        Ok(Database {
            handle: self.handle,
            _phantom: PhantomData,
        })
        // self 被消费——旧的 Database<H, Disconnected> 不再存在
        // 返回新的 Database<H, Connected>
    }
}

// Connected 状态的方法
impl<H> Database<H, Connected> {
    /// 执行查询——只能在 Connected 状态下调用
    pub fn execute_query(&self, sql: &str) -> Result<String, String>
    where
        H: Queryable,
    {
        unsafe { H::execute(self.handle.as_ptr(), sql) }
    }

    /// 断开连接——状态从 Connected 变为 Disconnected
    pub fn disconnect(self) -> Database<H, Disconnected> {
        // 断开连接（类型状态变化——不需要调用 C，这里是类型层面的操作）
        Database {
            handle: self.handle,
            _phantom: PhantomData,
        }
    }
}

// 为包装实现 Drop（不管状态，最终都需要释放句柄）
impl<H, S> Drop for Database<H, S> {
    fn drop(&mut self) {
        // unsafe { H::destroy(self.handle.as_ptr()); }
    }
}

// 为 H 类型所需的 trait——实际的 FFI 调用在这里
// （在真实项目中，这些 trait 方法由 bindgen 生成的函数实现）
trait Connectable {
    unsafe fn connect(ptr: *mut Self) -> Result<(), String>;
}

trait Queryable {
    unsafe fn execute(ptr: *const Self, sql: &str) -> Result<String, String>;
}
```

使用 phantom type 的效果是：

```rust
// 使用示例（概念性——展示类型状态的编译时保护）

// 假设 SqliteHandle 实现了 Connectable 和 Queryable
// let db = unsafe { Database::<SqliteHandle, Disconnected>::new(raw_ptr) };

// let db = db.connect().expect("connection failed");
// // db 的类型现在是 Database<SqliteHandle, Connected>

// let result = db.execute_query("SELECT 1")
//     .expect("query failed");

// let db = db.disconnect();
// // db 的类型变回 Database<SqliteHandle, Disconnected>
// // db.execute_query("SELECT 1");  // 编译错误——Disconnected 没有 execute_query 方法
```

这种模式将运行时的状态检查变为编译时的类型检查——不可能在断开的数据库上执行查询，因为编译器不允许。这是零成本的：phantom type 在编译后完全消失，运行时没有任何额外开销。

### 4.2 PhantomData 的多种用途

`PhantomData<T>` 在 phantom type 模式中扮演了多重角色：

```rust
use std::marker::PhantomData;

// 用途 1：标记所有权——告诉 drop checker "我逻辑上拥有 T"
struct Owns<T> {
    _marker: PhantomData<T>,  // T 的 drop 检查会影响 Owns<T>
}

// 用途 2：标记生命周期——将生命周期与结构体关联
struct Borrows<'a, T: 'a> {
    _marker: PhantomData<&'a T>,  // Borrows<'a, T> 不活得比 'a 长
}

// 用途 3：实现 Send/Sync 的精细控制——包含 PhantomData<*const T> 阻止自动实现 Send
struct NotSend {
    _marker: PhantomData<*const ()>,  // *const T 是 !Send 的 → NotSend 也是
}

// 用途 4：类型状态标记——如上节的 Connected / Disconnected
// PhantomData<S> 让 S 的类型信息参与类型检查，但不占用运行时内存
```

在 phantom type 模式中，`PhantomData<(H, S)>` 同时完成了两件事：它告诉 drop checker `Database<H, S>` 在逻辑上拥有 `H`（因为 `H` 可能包含需要 drop 的资源）；它将 `S` 绑定到类型参数中，使得 `Database<H, Connected>` 和 `Database<H, Disconnected>` 成为不同的类型。

## 5. 实战 —— 封装 libgit2 的 `git_repository`

### 5.1 libgit2 的 API 模式

libgit2 是一个用 C 编写的 Git 库，广泛使用不透明句柄。以 `git_repository` 为例：

```c
// C 头文件 git2/repository.h —— libgit2 的 API 声明
typedef struct git_repository git_repository;

int git_repository_open(git_repository **out, const char *path);
int git_repository_is_bare(const git_repository *repo);
int git_repository_state(const git_repository *repo);
const char *git_repository_path(const git_repository *repo);
void git_repository_free(git_repository *repo);
```

每个函数返回 `int`——0 表示成功，负数表示错误码。`git_repository_open` 通过输出参数返回句柄。`git_repository_free` 释放句柄。

### 5.2 Rust 侧的类型声明

```rust
use std::ffi::{CStr, CString};
use std::os::raw::{c_char, c_int};
use std::ptr::NonNull;

// 不透明类型占位——对应 C 的 git_repository
#[repr(C)]
pub struct GitRepository {
    _private: [u8; 0],
}

// FFI 声明（实际项目中由 bindgen 生成）
extern "C" {
    fn git_repository_open(
        out: *mut *mut GitRepository,
        path: *const c_char,
    ) -> c_int;

    fn git_repository_is_bare(repo: *const GitRepository) -> c_int;

    fn git_repository_state(repo: *const GitRepository) -> c_int;

    fn git_repository_path(repo: *const GitRepository) -> *const c_char;

    fn git_repository_free(repo: *mut GitRepository);
}
```

### 5.3 安全的 Rust 包装

```rust
/// 拥有所有权的 git_repository 包装——RAII
pub struct Repository {
    handle: NonNull<GitRepository>,
}

impl Repository {
    /// 打开一个 git 仓库
    pub fn open(path: &str) -> Result<Self, String> {
        let c_path = CString::new(path).map_err(|e| format!("invalid path: {}", e))?;
        let mut raw: *mut GitRepository = std::ptr::null_mut();

        // 调用 C 函数——通过输出参数返回句柄
        let rc = unsafe {
            git_repository_open(
                &mut raw as *mut *mut GitRepository,
                c_path.as_ptr(),
            )
        };

        if rc != 0 {
            return Err(format!("git_repository_open failed: {}", rc));
        }

        let handle = NonNull::new(raw).ok_or("repository pointer is null after open")?;
        Ok(Repository { handle })
    }

    /// 检查当前仓库是否为 bare 仓库
    pub fn is_bare(&self) -> bool {
        let ret = unsafe { git_repository_is_bare(self.handle.as_ptr()) };
        ret != 0  // 0 = false, 非零 = true (libgit2 约定)
    }

    /// 获取仓库的工作路径
    /// 返回值由 libgit2 内部管理——不需要调用者释放
    pub fn path(&self) -> Option<String> {
        let ptr = unsafe { git_repository_path(self.handle.as_ptr()) };
        if ptr.is_null() {
            None
        } else {
            unsafe { CStr::from_ptr(ptr) }
                .to_str()
                .ok()
                .map(|s| s.to_owned())
        }
    }

    /// 获取仓库状态
    pub fn state(&self) -> i32 {
        unsafe { git_repository_state(self.handle.as_ptr()) }
    }

    /// 消费 Repository 并返回原始指针（所有权转移给调用者）
    /// 调用者负责之后调用 git_repository_free
    pub fn into_raw(self) -> *mut GitRepository {
        let raw = self.handle.as_ptr();
        // 防止 Drop 释放——所有权已转移
        std::mem::forget(self);
        raw
    }

    /// 从原始指针重建——接管所有权
    /// 安全性：ptr 必须是一个有效的 git_repository 指针
    pub unsafe fn from_raw(ptr: *mut GitRepository) -> Self {
        Repository {
            handle: NonNull::new_unchecked(ptr),
        }
    }
}

// Drop 实现——在 Repository 离开作用域时自动释放
impl Drop for Repository {
    fn drop(&mut self) {
        unsafe {
            git_repository_free(self.handle.as_ptr());
        }
    }
}

// 安全标记——libgit2 文档声明 git_repository 可以在线程间安全移动
unsafe impl Send for Repository {}
// 但默认不实现 Sync——多个线程共享同一个 git_repository 是不安全的
```

### 5.4 使用示例

```rust
fn example_git_operations() -> Result<(), String> {
    // 打开一个 git 仓库
    let repo = Repository::open("/path/to/some/git/repo")?;

    // 查询仓库属性
    if repo.is_bare() {
        println!("This is a bare repository");
    } else {
        println!("Repository path: {:?}", repo.path());
    }

    println!("Repository state: {}", repo.state());

    // repo 离开作用域——自动调用 git_repository_free
    Ok(())
}
```

这个例子展示了包裹 libgit2 的完整模式：用 `NonNull` 和 Drop 实现拥有所有权的安全包装、通过 `CStr` 处理 C 返回的字符串、在文档的基础上安全地实现 `Send`。

## 6. extern type 的进展与未来

### 6.1 RFC 1861 状态

`extern type` 作为语言特性源于 RFC 1861，提出了在 Rust 中声明"大小未知的外部类型"的语法。这个 RFC 已经部分实现（当前在 nightly 中可用，feature gate 是 `extern_types`），但距离完全稳定还有一段距离。

目前的主要未解决问题包括：
- **Extern type 的元数据**：编译器在编译单元中需要多少关于 extern type 的信息才能生成正确的代码？
- **与 trait 系统的交互**：extern type 是否可以实现 trait？默认情况下哪些 auto trait（Send、Sync、Unpin、Sized）被实现？
- **Drop 检查的边界**：如果一个结构体包含指向 extern type 的指针（而不是直接包含 extern type），其 Drop 行为应该如何分析？

### 6.2 当前最佳实践

对于稳定 Rust 项目，建议使用以下优先级：

1. **零大小占位结构体 + `NonNull` + Drop** —— 这是最成熟的模式，95% 的情况下足够使用。
2. **`*mut c_void` + 新类型包装** —— 如果你需要绝对最小化的绑定（如在 embedded 或内核模块中）。
3. **bindgen 生成的 opaque type** —— bindgen 自动应用方案 1，适合大型 C 库。
4. **nightly `extern type`** —— 仅在需要编译器的额外保证（如确保 value 永远不会被使用）时考虑。

在实际项目中，方案 1 几乎总是正确的选择。`extern type` 在当前阶段主要是为未来做准备的特性——它的设计方向是正确的，但稳定化的细节仍需社区的进一步讨论。

### 练习 1 答案

```rust
use std::ffi::CString;
use std::os::raw::{c_char, c_int, c_long, c_void};
use std::ptr::NonNull;

// 不透明类型占位
#[repr(C)]
pub struct FileHandle {
    _private: [u8; 0],
}

// FFI 声明
extern "C" {
    fn file_open(path: *const c_char, mode: *const c_char) -> *mut FileHandle;
    fn file_read(f: *mut FileHandle, buf: *mut c_void, count: usize) -> usize;
    fn file_write(f: *mut FileHandle, buf: *const c_void, count: usize) -> usize;
    fn file_seek(f: *mut FileHandle, offset: c_long, whence: c_int) -> c_int;
    fn file_close(f: *mut FileHandle) -> c_int;
}

// whence 常量
const SEEK_SET: c_int = 0;
const SEEK_CUR: c_int = 1;
const SEEK_END: c_int = 2;

pub enum SeekFrom {
    Start(i64),
    Current(i64),
    End(i64),
}

pub struct SafeFile {
    handle: NonNull<FileHandle>,
}

impl SafeFile {
    pub fn open(path: &str, mode: &str) -> Result<Self, String> {
        let c_path = CString::new(path).map_err(|e| format!("invalid path: {}", e))?;
        let c_mode = CString::new(mode).map_err(|e| format!("invalid mode: {}", e))?;

        let raw = unsafe { file_open(c_path.as_ptr(), c_mode.as_ptr()) };
        let handle = NonNull::new(raw).ok_or_else(|| {
            format!("failed to open file: {}", path)
        })?;

        Ok(SafeFile { handle })
    }

    pub fn read(&self, buf: &mut [u8]) -> Result<usize, String> {
        if buf.is_empty() {
            return Ok(0);
        }
        let n = unsafe {
            file_read(
                self.handle.as_ptr(),
                buf.as_mut_ptr() as *mut c_void,
                buf.len(),
            )
        };
        Ok(n)
    }

    pub fn write(&self, data: &[u8]) -> Result<usize, String> {
        if data.is_empty() {
            return Ok(0);
        }
        let n = unsafe {
            file_write(
                self.handle.as_ptr(),
                data.as_ptr() as *const c_void,
                data.len(),
            )
        };
        Ok(n)
    }

    pub fn seek(&self, offset: i64, whence: SeekFrom) -> Result<(), String> {
        let (offset, whence_code) = match whence {
            SeekFrom::Start(o) => (o, SEEK_SET),
            SeekFrom::Current(o) => (o, SEEK_CUR),
            SeekFrom::End(o) => (o, SEEK_END),
        };
        let ret = unsafe {
            file_seek(
                self.handle.as_ptr(),
                offset as c_long,
                whence_code,
            )
        };
        if ret == 0 {
            Ok(())
        } else {
            Err(format!("seek failed with code: {}", ret))
        }
    }
}

impl Drop for SafeFile {
    fn drop(&mut self) {
        unsafe {
            file_close(self.handle.as_ptr());
        }
    }
}
```

> **类型即契约**：`extern type` 和 opaque pattern 体现了 Rust 类型系统的一个核心哲学——类型不仅仅描述数据的内存布局，更描述一段代码的"使用协议"。通过将"你只能用指针传递它，不能构造或检查它"这一约束编码为类型层面的规则，Rust 在编译时就防止了一整类 FFI bug。在 [[4.10-custom-allocators]] 中，我们将进一步探索 Rust 如何与底层内存分配交互——当 `Box<T>` 使用全局分配器不够用时，如何自定义每一块内存的来源和去向。
