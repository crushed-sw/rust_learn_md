# 4.1 Unsafe Rust 基础 —— 掌控编译器的五项超级力量

> Safe Rust 的借用检查器像一个警觉的警卫，阻止你做出任何它无法验证安全性的操作。`unsafe` 关键字不是"关闭借用检查器"的开关——它是你和编译器之间的一份契约。在这块标记为 `unsafe` 的区域内，编译器退后一步，把验证的责任交给你：你保证自己遵守 Rust 的所有安全规则，即使编译器无法自动证明这一点。理解 unsafe Rust 就是理解 Rust 安全模型的"边界"——每一行 safe Rust 代码的基石，最终都建立在经过精心封装的 unsafe 实现之上。`Vec`、`Box`、`Arc`、`Mutex`——所有这些类型内部都包含着 unsafe 代码，但它们向外暴露的是百分百 safe 的 API。这就是 unsafe Rust 的最高实践准则：**把 unsafe 关在最小的笼子里，用 safe 的围墙包住它。**

## 1. unsafe 的两面性——关键字不等于"随便写"

### 1.1 什么是 unsafe Rust 真正"解锁"的

很多初学者听到 "unsafe" 就以为 Rust 开了一个后门，允许你无视所有权和借用规则。真相恰恰相反：**unsafe 代码中，所有权和借用规则依然有效**。`unsafe` 关键字只解锁了五类额外的能力，其余所有 Rust 的静态检查——类型检查、模式匹配、trait 解析——全部原封不动地运行。

我把这五种能力称为"unsafe 的五项超级力量"。每项力量都对应着一种编译器无法自动验证的操作，需要程序员通过阅读文档、理解内存模型和遵守安全契约来手动保证正确性。

### 1.2 五项超级力量一览

在深入每项能力之前，先建立全局视图。以下五项操作只能在 `unsafe` 块内进行：

1. **解引用裸指针**（`*const T` / `*mut T`）：裸指针不跟踪生命周期、不保证非空、可以被任意复制——编译器无法验证解引用是否安全
2. **调用 unsafe 函数**：标记为 `unsafe fn` 的函数在其文档中声明了调用者必须遵守的前置条件
3. **访问或修改可变静态变量**：全局可变状态天然存在数据竞争风险，编译器无法静态保证线程安全
4. **实现 unsafe trait**：如 `Send` / `Sync`，编译器信任实现者保证了 trait 所宣称的安全性语义
5. **访问 union 的字段**：union 的多个字段共享同一块内存，编译器不知道当前哪个变体是"活跃的"

下面的代码展示了 unsafe 块的基本语法——所有五种操作都必须在 `unsafe { }` 块中进行：

```rust
// unsafe 块的基本形式：大括号内的代码获得了使用"超级力量"的权限
// 但在 unsafe 块内部，借用检查器依然工作——你不能在同一块内创建两个可变引用

// 一个裸指针——仅创建不需要 unsafe
let raw_ptr: *const i32 = &42 as *const i32;

unsafe {
    // 解引用裸指针——这是五项超级力量之一，需要 unsafe
    let value = *raw_ptr;
    println!("裸指针指向的值: {}", value);
}

// 注意：raw_ptr 的创建在 unsafe 块之外——因为"创建裸指针"不是五项超级力量之一
// 只有"解引用"裸指针才需要 unsafe
```

## 2. 裸指针——*const T 和 *mut T 的本质

### 2.1 裸指针 vs 引用：一张对比表胜过千言万语

裸指针（`*const T` 和 `*mut T`）和引用（`&T` 和 `&mut T`）看起来相似——它们都指向某个内存地址——但它们的"社会契约"完全不同。引用受到借用检查器的严格约束，而裸指针几乎没有任何编译期的限制。这种"自由"是把双刃剑：它让你能够实现那些借用检查器无法表达的模式（自引用结构、侵入式链表、自定义内存分配器），但也把全部验证负担转移到了你身上。

```rust
// 引用：编译器保证的所有安全属性
let x = 42;
let r: &i32 = &x;
// ✓ r 保证非空（引用永远不能是 null）
// ✓ r 保证对齐正确
// ✓ r 保证指向已初始化的内存
// ✓ r 的生命周期由借用检查器追踪
// ✓ 在 r 存在期间，x 不能被移动或释放

// 裸指针：以上保证全部由程序员负责
let p: *const i32 = &x as *const i32;
// ? p 可能为 null——编译器不关心
// ? p 可能未对齐——编译器不知道
// ? p 可能指向未初始化的内存——编译器不追踪
// ? p 的生命周期——编译器不追踪，可能成为悬垂指针
// ? p 可以被任意复制——没有所有权语义
// ? x 可能被移动——编译器不阻止
```

裸指针和引用之间的这些差异不是 Rust 设计上的缺陷，而是一种深思熟虑的权衡。Safe Rust 的引用让你可以放心地编写百分之九十九的代码而无需担心内存安全；而裸指针则在剩下的百分之一场景中给了你 C 语言的灵活性和控制力——但你需要自己承担责任。

### 2.2 创建裸指针——不需要 unsafe

裸指针的**创建**本身是 safe 操作。你可以随时将引用转换为裸指针，也可以凭空创建空指针。这是 Rust 的一个重要设计选择：如果你不小心创建了一个裸指针，这本身不会造成未定义行为——只有当你试图**解引用**它时，危险才开始。

```rust
// 三种创建裸指针的方式——都不需要 unsafe 块

// 方式 1：从引用转换（最安全的方式——你知道引用是有效的）
let x = 100;
let const_ptr: *const i32 = &x as *const i32;     // 不可变引用 → *const
let mut y = 200;
let mut_ptr: *mut i32 = &mut y as *mut i32;        // 可变引用 → *mut

// 方式 2：使用 std::ptr 模块的工厂函数
let null_const: *const i32 = std::ptr::null();      // 空指针——*const
let null_mut: *mut i32 = std::ptr::null_mut();        // 空指针——*mut
// 注意：std::ptr::null() 返回的是 *const T，不是 *mut T
// 这是类型系统在帮你区分"可能被写的指针"和"只读的指针"

// 方式 3：从整数地址构造（极端危险——只在 FFI 或嵌入式场景中使用）
let arbitrary_addr = 0x1000 as *const u8;  // 把任意整数强转为指针
// 别这么做除非你 100% 确定这个地址是有效的映射内存
```

创建裸指针是 safe 的，这个设计决策的深层原因在于：你持有裸指针不会破坏任何已有对象的内存安全——直到你**解引用**它。这个分界让 unsafe 的范围最小化成为可能——你可以在 safe 代码中准备数据、创建指针，只在真正需要访问内存的那一瞬间进入 unsafe 块。

### 2.3 解引用裸指针——进入 unsafe 世界

解引用操作是 unsafe Rust 中最常见的 unsafe 操作。当你写下 `*ptr` 时，编译器信任你做了以下验证：

```rust
// 解引用裸指针的完整安全检查清单
// 下面每一项都是【程序员的责任】，编译器不做任何保证

let x = 42;
let ptr: *const i32 = &x as *const i32;

unsafe {
    // 在这个 unsafe 块中，编译器信任你已验证以下条件：
    // 1. ptr 不是 null（解引用 null 指针 = UB）
    // 2. ptr 指向已初始化的 i32 值（读取未初始化内存 = UB）
    // 3. ptr 满足 i32 的对齐要求（通常是 4 字节对齐，在大多数平台上）
    // 4. ptr 指向的内存尚未被释放（悬垂指针解引用 = UB）
    // 5. 如果 ptr 是 *mut，且存在其他读写者，需要保证数据竞争自由
    let value: i32 = *ptr;
    println!("成功读取: {}", value);
}
// unsafe 块结束——回到 safe Rust 的怀抱
```

你注意到这段代码中，unsafe 块只包围了真正需要 unsafe 的那一行——解引用操作。其他代码（创建指针、打印）都在 safe Rust 中进行。这正是"最小化 unsafe 范围"的体现。

## 3. 调用 unsafe 函数——理解安全契约

### 3.1 unsafe fn 的语义

一个函数被标记为 `unsafe fn`，意味着它有一个**安全契约**——一组调用者必须满足的前置条件。编译器的借用检查器无法验证这些条件是否成立，所以调用者必须阅读文档，理解契约，然后通过 `unsafe { }` 块来承诺自己遵守了契约。

标准库中充满了 unsafe 函数——它们是构建 safe 抽象的砖石。以下是一些常见例子及其安全契约：

```rust
// 标准库中常见的 unsafe 函数及其安全契约

// 示例 1: std::slice::from_raw_parts
// 安全契约：
//   - data 必须指向至少 len 个已初始化的 T 类型值
//   - data 必须满足 T 的对齐要求
//   - 返回的切片的生命周期不能超出 data 指向的内存的生命周期
//   - data 不能是 null（除非 len == 0）

// 示例 2: Box::from_raw
// 安全契约：
//   - raw 必须是之前通过 Box::into_raw 产生的指针
//   - raw 不能已经被释放或通过其他方式回收
//   - 调用后不能再使用该指针

// 示例 3: std::mem::transmute
// 安全契约（极度严格）：
//   - 源类型和目标类型的 size 必须完全相等
//   - 源值的每个 bit pattern 必须是目标类型的合法值
//   - 必须考虑对齐差异（即使 size 相同，对齐可能不同）
```

### 3.2 调用 unsafe fn 的正确方式

调用 unsafe 函数时，最好的实践是把调用本身放在 `unsafe { }` 块中，并在块的外部（或紧邻的注释中）说明为什么调用是安全的：

```rust
// 正确模式：unsafe 块内调用 unsafe fn，块外的注释说明理由

/// 从一个 Vec<T> 中手动夺取底层缓冲区——实际代码中请使用 Vec::into_raw_parts
fn vec_to_raw_parts<T>(v: Vec<T>) -> (*mut T, usize, usize) {
    // 阻止 Vec 的 Drop 实现释放缓冲区
    let mut v = std::mem::ManuallyDrop::new(v);

    let ptr = v.as_mut_ptr();
    let len = v.len();
    let cap = v.capacity();

    // 安全条件验证：
    // 1. ptr 是由 Vec 的分配器分配的，满足对齐要求
    // 2. len 不会超过 cap
    // 3. 返回的指针在 ManuallyDrop 的保护下不会被 double-free
    (ptr, len, cap)
}

// 把这个指针重新组装成 Vec
// # Safety
// ptr 必须:
//   - 由相同的全局分配器分配
//   - 指向至少 cap 个 T 类型的连续内存
//   - len <= cap
//   - 前 len 个元素是有效的 T 值
unsafe fn raw_parts_to_vec<T>(ptr: *mut T, len: usize, cap: usize) -> Vec<T> {
    // 这里调用的是假设的 from_raw_parts，不是标准库的 API
    // 实际标准库中 Vec::from_raw_parts 存在类似功能
    //
    // 安全：调用者保证 ptr/len/cap 满足 Vec 的内部不变量
    Vec::from_raw_parts(ptr, len, cap)
}

// 使用示例：
let v = vec![1, 2, 3, 4, 5];
let (ptr, len, cap) = vec_to_raw_parts(v);

// unsafe 块尽可能小——只包围真正需要的调用
unsafe {
    // 安全：ptr/len/cap 来自同一个 Vec，刚刚被取出，没有其他使用者
    let restored_v = raw_parts_to_vec(ptr, len, cap);
    assert_eq!(restored_v, vec![1, 2, 3, 4, 5]);
    // restored_v 在这里被 drop——缓冲区被正确释放
}
```

### 3.3 安全契约文档的写法

当你定义自己的 `unsafe fn` 时，必须在文档中写出 `# Safety` 部分。这是 Rust 社区的强约定，也是编译器 lint 会提示的内容。一个好的 `# Safety` 部分应该列出调用者需要保证的每一条前置条件：

```rust
/// 从裸指针和长度构造一个切片引用。
///
/// # Safety
///
/// 调用者必须保证以下所有条件都成立：
///
/// * `data` 必须为 [非空] 且 [正确对齐]，指向至少 `len` 个连续的、已初始化的 `T` 类型值。
/// * `data` 指向的内存必须在返回的引用的整个生命周期内保持有效且不被 mutably aliased。
/// * 总内存大小 `len * size_of::<T>()` 不能超过 `isize::MAX`。
///
/// # 为什么这些条件是必要的
///
/// 这个函数生成一个 `&[T]`（引用），而 Rust 的引用需要上述保证以维护内存安全。
/// 任何条件的违反都可能导致未定义行为——包括但不限于段错误、数据损坏和逻辑错误。
pub unsafe fn from_raw_parts<'a, T>(data: *const T, len: usize) -> &'a [T] {
    // 实际实现会做 debug_assert! 检查（仅在 debug build 中）
    debug_assert!(!data.is_null() || len == 0, "data 不能为 null 除非 len 为 0");
    debug_assert!(data.is_aligned(), "data 必须满足 T 的对齐要求");
    std::slice::from_raw_parts(data, len)
}
```

`# Safety` 部分的编写是 unsafe Rust 编程中最严肃的环节。你的调用者将基于这些文档来决定是否可以安全地调用你的函数。因此，每条条件都应该是可检查的、明确的、完整的。

## 4. 实现 unsafe trait——"我担保这个类型满足契约"

### 4.1 unsafe trait 的哲学

在 Rust 中，大多数 trait 是"自动验证"的——编译器会检查你的 impl 是否满足 trait 的所有要求。但有一类 trait 携带了**编译器无法验证的语义承诺**。最典型的例子就是 `Send` 和 `Sync`。

当你为一个类型实现 `Send` 时，你告诉编译器："这个类型可以安全地在线程之间转移所有权。"如果你的类型包含裸指针，编译器无法判断你是否手动管理了线程安全——它只能信任你。

```rust
// unsafe trait 的典型——Send 和 Sync
// 
// Send: 类型的值可以安全地转移到另一个线程
// Sync: 类型的共享引用（&T）可以安全地被多个线程访问
//
// 大多数类型自动推导 Send/Sync，但当你的类型包含裸指针时，
// 推导被抑制——编译器说"我无法判断，你来决定"

// 一个包含裸指针的类型——自动不是 Send 也不是 Sync
struct RawBuffer {
    ptr: *mut u8,
    len: usize,
}

// 如果你清楚 ptr 指向的内存是线程安全管理的（比如通过原子操作或单独的锁），
// 你可以手动声明 Send——这需要 unsafe impl
//
// 安全：RawBuffer 的 ptr 指向由外部锁保护的内存，
//       且 len 只在持有锁时被修改
unsafe impl Send for RawBuffer {}

// 类似地，如果 &RawBuffer 可以被共享访问
// 安全：读取操作不修改 ptr/len，写入操作受锁保护
unsafe impl Sync for RawBuffer {}
```

手动实现 `Send` 和 `Sync` 是最常见的 unsafe trait 场景。你不应该随意添加这些实现——如果你的类型实际上不满足线程安全要求，你将引入隐蔽的数据竞争 bug。务必像上面这样在注释中说明你满足条件的理由。

### 4.2 其他 unsafe trait 的例子

除了 `Send` 和 `Sync`，Rust 标准库中还有其他 unsafe trait。它们的共同点是：实现者需要保证的语义超出了编译器可以静态验证的范围。

```rust
use std::marker::PhantomData;

// unsafe trait 示例 1: GlobalAlloc —— 自定义全局分配器
// 安全契约：alloc/dealloc/realloc 的实现必须满足标准分配器语义
// （返回正确对齐的内存、不返回重叠的内存块等）

// unsafe trait 示例 2: std::ops::DerefMut（间接 unsafe）
// DerefMut 本身是 safe trait，但如果你手动实现它并且返回了
// 错误的引用，你可能破坏借用规则

// unsafe trait 示例 3: 自定义 unsafe trait
/// 标记一个类型为"不可移动"——一旦固定在内存中，不能改变地址
///
/// # Safety
///
/// 实现者必须保证：
/// * 此类型的值一旦创建，其地址在生命周期结束前不会改变
/// * 此类型不应实现任何可能改变地址的操作（如通过 DerefMut 移动内容）
unsafe trait Immovable {
    // 可以包含 safe 方法——unsafe 仅针对的是"实现"这个 trait
    fn is_pinned(&self) -> bool;
}

// 实现 unsafe trait 需要在 unsafe impl 块中
struct FixedBuffer {
    data: [u8; 1024],
}

// 安全：FixedBuffer 包含纯内联数据（没有堆指针），
//       移动它是安全的，因此 Immovable 的语义不适用。
//       这里仅作语法演示——实际上 FixedBuffer 不应该实现这个 trait。
unsafe impl Immovable for FixedBuffer {
    fn is_pinned(&self) -> bool {
        false // FixedBuffer 可以被正常移动
    }
}
```

理解 unsafe trait 的关键是：**unsafe 的是"实现"这个 trait，而不是"使用"它**。一旦一个类型正确地实现了 `Send`，其他代码就可以在泛型约束 `T: Send` 下使用它，完全不需要 unsafe。这种设计——把 unsafe 限制在定义的层面——是 Rust "安全封装"哲学的核心体现。

## 5. 可变静态变量——全局状态的原罪与救赎

### 5.1 为什么可变静态变量危险

静态变量（`static`）存储在程序的固定内存地址中，整个程序生命周期内都存在。如果它是可变的（`static mut`），任何代码片段都可以在任何时刻读写它。在没有同步机制的情况下，这天然是数据竞争的温床。

```rust
// 不可变静态变量——safe，因为只读
static MAX_CONNECTIONS: u32 = 100;
// 所有线程都可以读取 MAX_CONNECTIONS，没有任何安全问题
// 事实上，编译器可能会把它内联到使用处——零运行时开销

// 可变静态变量——读写都需要 unsafe
static mut COUNTER: u32 = 0;
// COUNTER 可以被程序的任何部分修改，没有任何编译期保护

fn increment() {
    unsafe {
        // 读写可变静态变量需要 unsafe 块
        // 编译器在说："我无法保证没有其他线程在同一时刻修改 COUNTER"
        COUNTER += 1;
    }
}

fn read_counter() -> u32 {
    unsafe {
        COUNTER  // 读取也需要 unsafe
    }
}
```

Rust 要求读写 `static mut` 必须在 unsafe 块中进行，这本身就是一个巨大的警告信号。如果你的代码中有很多 `static mut`，你应该停下来思考是否可以用其他方式实现——在多线程环境中，`static mut` 几乎总是错误的答案。

### 5.2 安全地管理全局可变状态

Rust 提供了多种安全地管理全局可变状态的机制。这些方案本质上都是把 unsafe 封装在安全的 API 之后：

```rust
use std::sync::Mutex;
use std::sync::atomic::{AtomicU32, Ordering};
use std::sync::OnceLock;

// 方案 1: 原子类型——轻量级、lock-free
// AtomicU32 的 store/load 操作是安全的，不需要外部同步
static SAFE_COUNTER: AtomicU32 = AtomicU32::new(0);

fn safe_increment() {
    // fetch_add 是 safe 操作——原子性由硬件保证
    SAFE_COUNTER.fetch_add(1, Ordering::SeqCst);
}

// 方案 2: Mutex<static>——需要内部可变性
// 注意：static Mutex 需要 Sync 实现（Mutex<T> 当 T: Send 时是 Sync）
static SAFE_SHARED_DATA: Mutex<Vec<String>> = Mutex::new(Vec::new());

fn add_data(item: String) {
    // lock() 返回 MutexGuard，提供了排他访问
    SAFE_SHARED_DATA.lock().unwrap().push(item);
}

// 方案 3: OnceLock / LazyLock——延迟初始化
// 适合需要在运行时初始化的全局配置
static CONFIG: OnceLock<Config> = OnceLock::new();

fn get_config() -> &'static Config {
    CONFIG.get_or_init(|| {
        // 首次访问时初始化——线程安全，只执行一次
        Config::load_from_file("config.toml")
    })
}

// 方案 4: thread_local! —— 线程局部存储
// 每个线程有自己的副本，天然无数据竞争
use std::cell::RefCell;
thread_local! {
    static THREAD_LOCAL_COUNTER: RefCell<u32> = RefCell::new(0);
}
```

这些替代方案覆盖了绝大多数全局可变状态的需求。如果你发现自己真的需要 `static mut`，那通常意味着你正在与 C 代码交互（FFI 中的全局变量），或者你在实现一个自定义的同步原语（此时 unsafe 是不可避免的）。

### 5.3 何时 static mut 不可避免

在某些场景下，`static mut` 确实是正确的选择——但仅限于你完全理解并发语义的情况：

```rust
// 场景：FFI 中 C 库提供全局变量
// C 头文件：extern int errno;
// Rust 等价声明：
extern "C" {
    // 这是一个"外部"的静态变量——由 C 代码维护
    static mut errno: i32;
}

// 读取 errno 需要 unsafe——因为它是外部的可变状态
fn get_errno() -> i32 {
    unsafe { errno }
}

// 场景：实现全局分配器时可能需要 static mut 来跟踪分配状态
// 但即使是这种场景，也应该优先考虑 Atomic 类型和 Mutex
// 只在性能分析证明这些同步机制成为瓶颈时才使用 static mut
```

## 6. Union——当多个类型共享同一块内存

### 6.1 Union 的基本概念

Union（联合体）是一种数据结构，它的所有字段共享同一块内存。在任意时刻，只有一个字段是"活跃的"（即包含有效的值）。编译器不追踪哪个字段是活跃的——这是程序员的责任。

```rust
// union 的语法类似 struct，但用 union 关键字
union IntOrFloat {
    i: i32,
    f: f32,
}
// 注意：这个 union 的大小等于 max(size_of::<i32>(), size_of::<f32>()) = 4 字节
// 而不是 struct 的 4 + 4 = 8 字节

fn use_union() {
    let mut u = IntOrFloat { i: 42 };
    
    // 读取 union 字段需要 unsafe
    unsafe {
        println!("作为 i32 读取: {}", u.i);  // 输出: 42
        
        // 如果此时读取 u.f，你会得到一个"被重新解释"的 bit pattern
        // 42 作为 i32 的 bit pattern 是 0x0000002A
        // 被解释为 f32 后可能是一个极小的小数
    }
    
    // 写入另一个字段
    u.f = 3.14;
    unsafe {
        println!("作为 f32 读取: {}", u.f);  // 输出: 3.14
        // 注意：此时 u.i 不再包含之前的值——内存被 f 覆盖了
    }
}
```

Union 在 Rust 中不像在 C 中那样常用，这主要是出于两个原因：第一，Rust 的 enum 已经提供了类型安全的"或"关系；第二，访问 union 字段需要 unsafe，提醒你这是危险操作。Union 的主要用途在 FFI 边界——当 C 库使用 union 时，Rust 侧需要用同样的布局来交互。

### 6.2 Union 与 FFI 的经典模式

在 C 语言中，union 常用于实现"带标签的联合体"（tagged union）——这正是 Rust 的 `enum` 所抽象的。但当你需要精确匹配 C 的内存布局时，必须使用 `union`：

```rust
// C 端定义：
// typedef struct {
//     int type;  // 0 = int, 1 = float
//     union {
//         int int_value;
//         float float_value;
//     } data;
// } TaggedValue;

// Rust 端对应的定义
#[repr(C)]
struct TaggedValue {
    value_type: i32,     // 标签：0 = 整数，1 = 浮点数
    data: ValueUnion,    // 联合体数据
}

#[repr(C)]
union ValueUnion {
    int_value: i32,
    float_value: f32,
}

impl TaggedValue {
    /// 根据 value_type 安全地读取数据
    fn get_value(&self) -> Result<Value, &'static str> {
        match self.value_type {
            0 => unsafe {
                // 安全：value_type == 0 表示 int_value 是活跃字段
                Ok(Value::Int(self.data.int_value))
            },
            1 => unsafe {
                // 安全：value_type == 1 表示 float_value 是活跃字段
                Ok(Value::Float(self.data.float_value))
            },
            _ => Err("未知的 value_type"),
        }
    }
}

enum Value {
    Int(i32),
    Float(f32),
}
```

这种模式——"tag 字段 + union + 安全的访问方法"——是 FFI 中处理 C 联合体的标准做法。关键设计是：`get_value` 方法是 safe 的，因为它在内部验证了 tag 字段，然后才访问 union 的正确字段。调用者不需要写 unsafe。

### 6.3 Union 用于类型转换（type punning）

Union 也可以用来在不同类型之间重新解释 bit pattern——这被称为 type punning。虽然 Rust 提供了 `transmute`（见下一课），但 union-based punning 更加显式：

```rust
// 使用 union 进行 bit-level 的类型重新解释
// 这种模式明确指出"这里发生了类型转换"
union FloatBits {
    float: f32,
    bits: u32,
}

fn float_to_bits(value: f32) -> u32 {
    let converter = FloatBits { float: value };
    unsafe {
        // 安全：f32 和 u32 具有相同的大小（都是 4 字节）
        // 且任意 u32 的 bit pattern 都是一个有效的 f32（反之亦然）
        converter.bits
    }
}

fn bits_to_float(bits: u32) -> f32 {
    let converter = FloatBits { bits };
    unsafe {
        converter.float
    }
}

// 使用示例：
let f = 3.14_f32;
let bits = float_to_bits(f);
println!("{:.32}", bits);  // 打印 f32 的内部表示

let restored = bits_to_float(bits);
assert_eq!(f, restored);
```

注意这个例子与直接使用 `transmute` 的区别：`transmute` 可以吞下任意类型而没有任何结构上的提示；union 的方式明确列出了"哪些类型可能参与转换"。虽然两者都是 unsafe 的，但 union 方式的可读性和可审计性更好——读者一眼就能看到哪些类型参与了这个 "reinterpretation game"。

## 7. unsafe 块的范围——大括号不是装饰

### 7.1 unsafe 块到底"标记"了什么

`unsafe { }` 块标记了一个区域，在这个区域中程序员承担了额外的安全验证责任。但 `unsafe` 不是一个"全特性解锁"——它只解锁五项超级力量。理解这个限制是安全封装的关键。

```rust
// 演示：unsafe 块的范围和限制

let mut x = 5;
let r1 = &x;
let r2 = &x; // 两个不可变引用——在 safe Rust 中是允许的

unsafe {
    // 在 unsafe 块内，借用检查器仍然工作！
    //
    // 下面这行如果取消注释会编译失败：
    // let r3 = &mut x;  // 错误！已经存在不可变引用 r1, r2
    //
    // unsafe 不会关闭借用检查。它只解锁：
    // 1. 解引用裸指针
    let ptr = &x as *const i32;
    let val = *ptr;  // 这行在 safe 中无法编译
    
    // 2. 调用 unsafe fn
    // 3. 访问 static mut（如果有的话）
    // 4. 实现 unsafe trait（在 impl 块中）
    // 5. 访问 union 字段
    
    // 除此之外，一切照旧
}

// 关键洞察：unsafe 块不是"做任何事都合法"的区域
// 它是"我承诺额外的安全条件"的声明
```

一个常见的误解是"在 unsafe 块里什么都能做"。实际上，你仍然不能：
- 同时持有对一个值的可变引用和任何其他引用（借用检查器不看你的 unsafe 块）
- 无视 Send/Sync 约束在线程间移动数据（除非使用 unsafe trait impl 解除）
- 跳过类型转换的合法性检查（非法的 `as` 转换仍然会编译报错）
- 无视生命周期约束

### 7.2 过大的 unsafe 块是坏味道

`unsafe` 块应该尽可能小——精确包围需要超级力量的操作，仅此而已。过大的 unsafe 块会使代码审查变得困难：审查者需要审计块中的每一行代码，判断它们是否可能引入未定义行为。

```rust
// ❌ 糟糕：过大的 unsafe 块——难以审查
fn bad_pattern(buffer: *mut u8, len: usize) -> Vec<u8> {
    unsafe {
        // 这里包含了大量 safe 逻辑和 unsafe 操作混在一起
        let slice = std::slice::from_raw_parts(buffer, len);
        let mut v = Vec::with_capacity(len);
        v.extend_from_slice(slice);
        // ... 100 行混合了 safe 和 unsafe 的代码 ...
        v
    }
}

// ✓ 良好：最小 unsafe 块，每个 unsafe 块有明确的理由
fn good_pattern(buffer: *mut u8, len: usize) -> Vec<u8> {
    // 安全：extend_from_slice 是 safe 的，
    // 但我们需要一个切片来调用它
    let slice = unsafe {
        // 安全条件已验证：
        // buffer 不为 null（除非 len == 0）
        // buffer 指向至少 len 个初始化的字节
        // buffer 在调用期间保持有效
        std::slice::from_raw_parts(buffer, len)
    };
    
    // 这里全部是 safe 代码
    let mut v = Vec::with_capacity(len);
    v.extend_from_slice(slice);
    v
}

// 另一个例子：更精细的 unsafe 划分
fn fine_grained_unsafe(ptr: *const i32) -> Option<i32> {
    // safe: 检查 null
    if ptr.is_null() {
        return None;
    }
    
    // safe: 检查对齐
    if !ptr.is_aligned() {
        return None;
    }
    
    // 只有这里需要 unsafe——解引用操作
    unsafe {
        // 安全：ptr 非 null、已对齐，且指向有效的已初始化 i32
        // 这由调用者提供的安全契约保证
        Some(*ptr)
    }
}
```

最小化 unsafe 块的意义不仅是审查便利性。当你需要追查一个未定义行为的源头时，小块的 unsafe 就像是精确的断点——你知道问题一定出在某个 unsafe 块中，而不是整个函数都值得怀疑。

### 7.3 unsafe 和函数边界

一个函数的整体可以是 `unsafe fn`，但这通常意味着该函数有一个**调用者必须满足的安全契约**。这与在 safe 函数内部使用 `unsafe { }` 块是两种不同的模式：

```rust
// 模式 A：safe 函数，内部使用 unsafe 块（封装模式）
// 这是标准库的主要模式——把 unsafety 封装在 safe API 后面
pub fn split_at_mut(values: &mut [i32], mid: usize) -> (&mut [i32], &mut [i32]) {
    let len = values.len();
    let ptr = values.as_mut_ptr();
    
    assert!(mid <= len);  // 安全验证在 safe 代码中进行
    
    unsafe {
        // 安全：mid <= len 已通过 assert 验证
        // 我们使用裸指针构造两个不重叠的 &mut [i32]
        (
            std::slice::from_raw_parts_mut(ptr, mid),
            std::slice::from_raw_parts_mut(ptr.add(mid), len - mid),
        )
    }
}

// 模式 B：unsafe fn，安全契约由调用者负责（委托模式）
// 这是"无法在函数内部验证安全条件"时的选择
pub unsafe fn raw_buffer_to_vec(buffer: *const u8, len: usize) -> Vec<u8> {
    // 函数内部假设调用者已验证安全条件
    let slice = std::slice::from_raw_parts(buffer, len);
    slice.to_vec()
}
// 调用者需要写：
// unsafe {
//     raw_buffer_to_vec(buf_ptr, buf_len);
// }
```

选择模式 A 还是模式 B 的核心判断标准是：**你能不能在函数内部充分验证安全条件**。如果能——用模式 A（safe 函数 + 内部 unsafe 块）。如果不能（比如需要的信息只在调用者的上下文中可用）——用模式 B（unsafe fn），把验证责任上移。

## 8. 构建安全抽象——标准库的模式

### 8.1 Vec 的核心：unsafe 基础上的 safe 城堡

`Vec<T>` 是 Rust 中最常用的数据结构之一，你每天使用它而无需写一行 unsafe。但它的内部实现大量使用了 unsafe。让我们看看标准库是如何做到这一点的。

`Vec` 的核心是一个“堆分配的连续缓冲区”。这个缓冲区是通过裸指针管理的——因为借用检查器无法表达“我拥有这块堆内存，但它的生命周期可以动态增长”：

```rust
// Vec 的简化内部结构（概念版本，非标准库实际代码）
pub struct Vec<T> {
    // 指向堆上分配的缓冲区的裸指针
    ptr: *mut T,
    // 当前已初始化的元素数量
    len: usize,
    // 缓冲区总容量（以 T 的数量计）
    cap: usize,
}
// 注意：ptr 是 *mut T，裸指针
// 这意味着 Vec<T> 默认不是 Send 也不是 Sync
// 但 Vec<T> 实际上在安全文档中保证了：
// 当 T: Send 时，Vec<T>: Send
// 当 T: Sync 时，Vec<T>: Sync
// 这通过 unsafe impl Send/Sync 实现
```

`Vec` 的所有 safe API——`push`, `pop`, `index`, `len`, `iter`——都是通过安全地使用内部的原始指针来构建的。关键模式是：在 safe 函数中**先验证所有安全条件**，然后在最小的 unsafe 块中执行操作。

### 8.2 push 操作的实现剖析

让我们解剖 `Vec::push` 的实现，看看标准库如何用大面积的 safe 验证包裹微小而精确的 unsafe 操作：

```rust
// Vec::push 的简化实现——展示安全封装模式
impl<T> Vec<T> {
    pub fn push(&mut self, value: T) {
        // 阶段 1: 纯 safe 检查——是否需要扩容
        if self.len == self.cap {
            // 需要扩容——reserve 是 safe 方法（内部包含 unsafe）
            self.reserve(1);
            // 此处的 unsafe 不变量: reserve 成功后 cap > len
        }
        
        // 阶段 2: unsafe 操作——向缓冲区的空位写入
        // 安全条件已验证:
        //   1. self.len < self.cap（因为 reserve 确保如此）
        //   2. self.ptr 指向有效的已分配内存
        //   3. self.ptr 在 self.len 位置之后有空位
        //   4. self.ptr.add(self.len) 是有效且对齐的写入位置
        unsafe {
            // std::ptr::write 覆盖目标位置的内存，不调用 drop
            // 因为那个位置尚未初始化——没有旧值需要释放
            std::ptr::write(self.ptr.add(self.len), value);
        }
        
        // 阶段 3: 更新长度——仍然在 safe 代码中
        self.len += 1;
        // 现在 value 已经"属于" Vec，它的 drop 责任被 Vec 接管
    }
}
```

这个模式可以总结为：
1. **Safe 验证**：检查前置条件（`len < cap`），如果不满足就通过安全操作创造条件（`reserve`）
2. **最小的 unsafe 块**：执行裸指针写操作——仅此一行
3. **Safe 收尾**：更新逻辑状态（`len += 1`）

这种“三阶段”模式——验证、操作、记账——是 Rust 标准库中 unsafe 封装的标准写法。

### 8.3 Drop 和裸指针——为什么不能忘写 Drop

当你的类型包含裸指针时，你必须手动实现 `Drop` 来释放指针指向的内存。Rust 不会为裸指针自动生成 Drop——因为编译器不知道这些指针是否指向堆内存、是由谁分配的、该如何释放：

```rust
// 一个使用裸指针的自定义缓冲区类型
struct OwnedBuffer {
    ptr: *mut u8,
    len: usize,
    cap: usize,
}

impl OwnedBuffer {
    fn new(cap: usize) -> Self {
        let layout = std::alloc::Layout::array::<u8>(cap).unwrap();
        let ptr: *mut u8;
        unsafe {
            // 安全：layout.size() > 0，Layout::array 已验证参数不溢出
            ptr = std::alloc::alloc(layout);
        }
        if ptr.is_null() {
            std::alloc::handle_alloc_error(layout);
        }
        OwnedBuffer { ptr, len: 0, cap }
    }
}

// 必须手动实现 Drop——否则内存泄漏
impl Drop for OwnedBuffer {
    fn drop(&mut self) {
        // 先 drop 所有已初始化的元素（如果有的话）
        // 对于 u8 来说不需要，但如果 T 是有 Drop 的类型则需要
        // 这里 ptr 指向 u8，可以直接释放
        
        if self.cap > 0 {
            let layout = std::alloc::Layout::array::<u8>(self.cap).unwrap();
            unsafe {
                // 安全：ptr 由 GlobalAlloc 分配，未被释放过
                // cap 匹配分配时使用的大小
                std::alloc::dealloc(self.ptr, layout.cast());
            }
        }
    }
}
// 注意：不能为 OwnedBuffer derive(Copy)，因为 Drop 和 Copy 互斥
```

这段代码展示了一个关键教训：当你接手了内存管理（通过裸指针），你就必须对那个内存的整个生命周期负责——分配、使用、释放。遗忘任何一步要么导致内存泄漏（没写 Drop），要么导致 double-free（在 Dropped 后又释放），要么导致 use-after-free（释放后继续使用指针）。

## 9. 常见陷阱与最佳实践

### 9.1 陷阱 1：过大的 unsafe 块

我们已在前文讨论过这个问题，但它值得再次强调——因为这是在实际代码审查中最常见的 unsafe 问题。

过大的 unsafe 块就像没有格式化过的代码：它"能工作"，但会让维护者（包括三个月后的你）置身于一片危险的模糊地带——不知道哪一行是安全敏感的。

作为经验法则：**一个 unsafe 块通常不应超过 3-5 行**。如果超过了，你应该问自己：这个块包含了多项操作，能不能拆分成多个？那些看起来 safe 的代码（如整数算术、布尔判断）能不能移出块外？

### 9.2 陷阱 2：忘记安全不变量

`unsafe` 代码中的安全性依赖于**不变量**——那些在 unsafe 块内外必须始终为真的属性。最常见的错误是在修改数据结构时忘记维护这些不变量。

```rust
// 一个有 bug 的 Vec 简化实现——展示"忘记不变量"的后果
struct BuggyVec<T> {
    ptr: *mut T,
    len: usize,
}

impl<T> BuggyVec<T> {
    // ❌ BUG：pop 减少了 len，但没有调用被移除元素的 drop
    fn buggy_pop(&mut self) -> Option<T> {
        if self.len == 0 {
            return None;
        }
        // 减少 len——但旧元素的内存仍然在那里
        // 如果 T 有 Drop（如 String, Vec），这导致内存泄漏
        self.len -= 1;
        unsafe {
            // 读取"已移出"的元素——在某些情况下可能正常
            // 但语义上已经是错误的
            Some(std::ptr::read(self.ptr.add(self.len)))
        }
    }
    
    // ✓ 修复：正确管理 len 不变量
    fn correct_pop(&mut self) -> Option<T> {
        if self.len == 0 {
            return None;
        }
        self.len -= 1;
        unsafe {
            // 安全：len 在 read 之前减少，所以 self.ptr.add(self.len)
            // 指向的是一个"逻辑上不再属于 Vec"的元素
            // ptr::read 后调用者获得所有权，不会再被 Vec 的 drop 释放
            Some(std::ptr::read(self.ptr.add(self.len)))
        }
    }
}

// 然而 correct_pop 在 BuggyVec 被 drop 时仍然有问题——
// 因为 BuggyVec 没有实现 Drop，ptr 指向的内存会被泄漏！
```

### 9.3 陷阱 3：unsafe 代码中的 panic 安全性

在 unsafe 块中发生 panic 时，Rust 的 unwinding 机制仍然会正常运行——栈会展开，Drop 实现会被调用。但如果你的 unsafe 块正在操作原始内存，panic 可能导致数据结构处于不一致状态：

```rust
// panic 在 unsafe 块中的危险
impl<T> Vec<T> {
    // push 操作中如果 value 的 clone/drop 会 panic 怎么办？
    // 标准库的解决方案：使用 ptr::write，它只是覆盖内存不运行 drop
    // 
    // 而如果有中间操作可能 panic（比如 reserve 中的内存分配），
    // Vec 会确保在 panic 发生时不变量仍然完好（len 尚未更新）
    //
    // 这是一个普遍的 unsafe 编程原则：
    // "如果 panic 发生在中间，数据结构的状态必须仍然是自洽的"
}
```

关键原则：**在 unsafe 块中修改数据结构后，如果后续操作可能 panic，数据结构的不变量必须仍然成立**。这通常意味着：先准备好所有资源，再一次性修改状态。

### 9.4 最佳实践清单

以下 checklist 来自 Rust 社区的集体经验，可作为每次编写或审查 unsafe 代码时的参考：

1. **unsafe 块最小化**——每个 unsafe 块只包含必要的操作，最好只有 1-3 行
2. **安全条件文档化**——用 `// 安全：...` 注释解释为什么这个 unsafe 块是安全的
3. **对 unsafe fn 写 `# Safety` 文档**——列出调用者必须满足的所有前置条件
4. **优先使用安全的替代方案**——在考虑 unsafe 之前，检查是否可以用 safe API 实现同样的功能
5. **使用调试断言**——`debug_assert!` 在开发阶段验证关键不变量，release 构建中零开销
6. **运行 Miri**——在测试中使用 `cargo miri test` 检测未定义行为
7. **代码审查**——所有包含 unsafe 的 PR 应该有一个额外的审阅者

### 练习 2 答案

```rust
struct SafeBuffer {
    ptr: *mut u8,
    len: usize,
    cap: usize,
}

impl SafeBuffer {
    pub fn new(capacity: usize) -> Self {
        let layout = std::alloc::Layout::array::<u8>(capacity)
            .expect("capacity 不能为零且不能溢出");
        let ptr = unsafe {
            // 安全：layout 有效
            std::alloc::alloc(layout)
        };
        if ptr.is_null() {
            std::alloc::handle_alloc_error(layout);
        }
        SafeBuffer {
            ptr: ptr as *mut u8,
            len: 0,
            cap: capacity,
        }
    }
    
    /// 写入数据到缓冲区，返回实际写入的字节数
    pub fn write(&mut self, data: &[u8]) -> usize {
        let available = self.cap - self.len;
        let to_write = data.len().min(available);
        
        if to_write == 0 {
            return 0;
        }
        
        // 安全条件：
        // 1. self.ptr 非 null，指向至少 self.cap 字节的有效内存
        // 2. self.ptr.add(self.len) 不会越界（因为 self.len + to_write <= self.cap）
        // 3. 目标区域当前未初始化，使用 ptr::copy_nonoverlapping 直接写入
        unsafe {
            std::ptr::copy_nonoverlapping(
                data.as_ptr(),
                self.ptr.add(self.len),
                to_write,
            );
        }
        
        self.len += to_write;
        to_write
    }
    
    /// 读取已写入数据的不可变引用
    pub fn read(&self) -> &[u8] {
        if self.len == 0 {
            return &[];
        }
        unsafe {
            // 安全：
            // 1. self.ptr 非 null，指向至少 self.len 个已初始化的字节
            // 2. self.len <= self.cap（不变量在 write 中维护）
            // 3. 返回的引用与 &self 共享生命周期
            std::slice::from_raw_parts(self.ptr, self.len)
        }
    }
    
    /// 清空缓冲区（逻辑上清空，不回收内存）
    pub fn clear(&mut self) {
        self.len = 0;
        // 注意：对于 u8 来说不需要逐个 drop
        // 因为 u8 没有 Drop 实现
        // 如果 SafeBuffer 是泛型的 SafeBuffer<T>，
        // 则需要先 drop 每个元素再设置 len = 0
    }
    
    pub fn len(&self) -> usize {
        self.len
    }
    
    pub fn capacity(&self) -> usize {
        self.cap
    }
}

impl Drop for SafeBuffer {
    fn drop(&mut self) {
        if self.cap > 0 {
            let layout = std::alloc::Layout::array::<u8>(self.cap)
                .expect("cap 在构造时已验证有效");
            unsafe {
                // 安全：ptr 由 GlobalAlloc 分配，使用相同的 layout
                // cap 匹配分配时的大小
                std::alloc::dealloc(self.ptr, layout);
            }
        }
    }
}
```

### 练习 3 答案

```rust
// 问题分析：
//
// ThreadSafeCounter 的问题在于 Cell<u64> 不是 Sync。
// Cell 的 get() 和 set() 方法使用非原子操作，在多线程环境中存在数据竞争：
//
// 线程 A: self.value.get()   → 读到 5
// 线程 B: self.value.get()   → 也读到 5
// 线程 A: get() + 1 = 6, set(6)
// 线程 B: get() + 1 = 6, set(6)
// 结果：两次 increment 但只增加了 1 —— 经典的读-改-写竞态
//
// 此外，unsafe impl Sync for ThreadSafeCounter {} 是错误的：
// Sync 要求 &Self 可以被多个线程安全地共享访问，
// 但 Cell<u64> 在多线程环境中不是安全的共享访问类型。
// Cell 是 !Sync 的，为包含 Cell 的类型实现 Sync 破坏了 Safety 契约。
//
// 正确做法：使用 AtomicU64
use std::sync::atomic::{AtomicU64, Ordering};

struct CorrectThreadSafeCounter {
    value: AtomicU64,
}

// AtomicU64 已经是 Send + Sync 的——编译器自动推导
// 不需要手动 unsafe impl

impl CorrectThreadSafeCounter {
    fn increment(&self) {
        self.value.fetch_add(1, Ordering::SeqCst);
    }
    
    fn get(&self) -> u64 {
        self.value.load(Ordering::SeqCst)
    }
}

// 关键教训：
// 1. 不要为包含 !Send/!Sync 字段的类型实现 Send/Sync
//    （除非你通过外部同步机制保证了安全性）
// 2. 使用 unsafe impl 时，必须在注释中说明为什么满足安全条件
// 3. 优先使用原子类型而非 Cell/RefCell 来处理跨线程状态
```
