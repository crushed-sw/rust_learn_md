# 4.10 自定义分配器 —— 当默认分配器成为瓶颈

> Rust 的默认分配器是一般化的——它必须处理任意大小的分配、任意时机的释放、多线程竞争。这种通用性是有代价的。但如果你确切知道你的程序如何分配内存——比如所有分配的寿命相同（arena）、所有对象大小相同（slab）、或者你的多线程工作负载需要更智能的锁策略（jemalloc）——你就可以用专用分配器替换默认行为。自定义分配器不是 Rust 特有的技巧，它是所有系统编程语言的必修课。Rust 提供了两个层次的接口：`GlobalAlloc` 让你替换整个程序的全局分配器，`Allocator` trait（nightly）让你为单个集合指定分配器。两者都需要 unsafe 代码，因为分配器的第一法则和第二法则是——绝不返回重叠的内存，绝不释放不属于你的内存。一个字节的错误就能制造灾难。

---

## 1. 为什么需要自定义分配器——通用方案的代价

### 1.1 默认分配器的设计权衡

Rust 默认使用系统分配器——在 Linux 上是 glibc 的 `malloc`（ptmalloc3），在 Windows 上是 HeapAlloc，在 macOS 上是 malloc zone。这些通用分配器必须同时满足以下要求：
- 处理从 1 字节到数 GB 的任意大小分配
- 支持任意时机的释放——分配和释放的顺序完全不可预测
- 最小化碎片——长期运行的程序不能因为碎片而耗尽内存
- 多线程安全——多个线程同时分配和释放互不干扰

满足所有要求的结果是：每次 `malloc` 调用可能涉及复杂的空闲链表遍历、锁获取、甚至系统调用（`mmap`/`brk`）。对于大多数程序这不是问题——内存分配相对于 I/O 和业务逻辑的开销微不足道。但当你的程序以分配为主要工作（比如构建 AST 树的编译器，或每秒处理百万请求的服务器），默认分配器会实实在在地拖慢你。

### 1.2 自定义分配器的三类场景

自定义分配器在三种场景下能带来显著收益。第一是 arena（竞技场/区域）分配：当你有一批寿命完全相同的对象——比如一个 HTTP 请求处理中的所有临时数据，或一个编译器 pass 中的所有 AST 节点——你可以用一个简单的 bump 指针分配它们，然后在请求结束时一次性释放整块内存。第二是固定大小对象池（slab/pool）：当你的程序分配了大量相同大小的对象——网络数据包、粒子系统中的粒子、数据库的页面缓存——你可以预分配一个固定大小的区块并为每个对象使用一个槽位。第三是更换全局分配器策略：如果你知道你的程序有特殊的多线程分配模式，jemalloc 或 mimalloc 可能比默认的 glibc malloc 快 2-3 倍。

## 2. GlobalAlloc Trait —— 替换全局分配器

### 2.1 GlobalAlloc 的核心签名

`GlobalAlloc` 是 Rust 标准库中用于替换全局分配器的 trait。它只有两个必须实现的方法——`alloc` 和 `dealloc`——外加两个有默认实现的变体（`alloc_zeroed` 和 `realloc`）。

```rust
use std::alloc::{GlobalAlloc, Layout};

// GlobalAlloc 在 std::alloc 中的定义（简化版）
// pub unsafe trait GlobalAlloc {
//     unsafe fn alloc(&self, layout: Layout) -> *mut u8;
//     unsafe fn dealloc(&self, ptr: *mut u8, layout: Layout);
//     
//     unsafe fn alloc_zeroed(&self, layout: Layout) -> *mut u8 { /* 默认：alloc + 清零 */ }
//     unsafe fn realloc(&self, ptr: *mut u8, layout: Layout, new_size: usize) -> *mut u8 { /* 默认 */ }
// }
```

`Layout` 参数承载了分配请求的元信息：大小（size）和对齐（align）。`alloc` 返回一个指向至少 `layout.size()` 字节、对齐到 `layout.align()` 的内存块指针。如果分配失败，返回 `std::ptr::null_mut()`——注意不是返回 `Result`，而是返回 null 指针。Rust 的 `Box`、`Vec` 等标准类型在收到 null 后会触发 `alloc::handle_alloc_error`（通常是 abort）。

### 2.2 编写第一个自定义全局分配器

理解 `GlobalAlloc` 最好的方式是先写一个什么都不做的 pass-through 分配器——它把实际工作委托给系统分配器。这个练习展示了 trait 的骨架。

```rust
use std::alloc::{GlobalAlloc, Layout, System};

/// 一个记录每次分配和释放的跟踪分配器
/// 实际工作委托给 System 分配器
struct TrackingAllocator;

// 关键设计点：GlobalAlloc 是 unsafe trait
// 因为 alloc 和 dealloc 的实现者必须遵守内存安全合约
unsafe impl GlobalAlloc for TrackingAllocator {
    unsafe fn alloc(&self, layout: Layout) -> *mut u8 {
        // 记录分配事件（在生产代码中这里可能写入日志或更新计数器）
        eprintln!("alloc: {} bytes, align {}", layout.size(), layout.align());

        // 委托给系统分配器
        System.alloc(layout)
    }

    unsafe fn dealloc(&self, ptr: *mut u8, layout: Layout) {
        eprintln!("dealloc: {} bytes", layout.size());

        // 必须使用与 alloc 完全相同的 layout
        System.dealloc(ptr, layout)
    }
}
```

这个跟踪分配器本身没有做任何优化，但它展示了两个关键点：第一，`alloc` 和 `dealloc` 都接收 `Layout`——释放时传入的 `Layout` 必须与分配时使用的完全一致；第二，`GlobalAlloc` 的实现是全局的——注册后，程序中所有的 `Box`、`Vec`、`String`、`Arc` 都会使用这个分配器。

### 2.3 注册全局分配器：`#[global_allocator]`

`#[global_allocator]` 属性将一个静态变量标记为程序的全局分配器。该静态变量必须是实现了 `GlobalAlloc` 的类型的实例。

```rust
// 在生产代码中，你需要在 crate 根模块注册全局分配器
// 例如在 main.rs 或 lib.rs 中：

use std::alloc::System;

// 方式 1：使用标准库提供的系统分配器（这是默认行为）
// #[global_allocator]
// static GLOBAL: System = System;

// 方式 2：使用自定义分配器
// #[global_allocator]
// static GLOBAL: TrackingAllocator = TrackingAllocator;

// 关键约束：
// 1. #[global_allocator] 只能在 crate 根模块使用
// 2. 只能有一个全局分配器
// 3. 静态变量名必须是 GLOBAL（约定，非强制但强烈建议）
// 4. alloc 和 dealloc 必须是 unsafe fn——由调用者保证 layout 正确
```

注意 `GlobalAlloc` 的 `alloc` 和 `dealloc` 接收的是 `&self`（共享引用）而不是 `&mut self`。这意味着全局分配器必须是线程安全的——它需要在多个线程同时分配时正确处理并发。通常这意味着内部使用原子操作或互斥锁。

### 2.4 Layout 参数详解——大小与对齐

`Layout` 不只是简单的 (size, align)。它的构造方法包含了一些有用的函数。

```rust
use std::alloc::Layout;

// Layout 的常用构造方式
fn layout_examples() {
    // 方式 1：从大小和对齐直接构造
    // 对齐必须是 2 的幂
    let layout1 = Layout::from_size_align(64, 8).unwrap();

    // 方式 2：对于简单类型，使用泛型函数
    let layout2 = Layout::new::<u64>();          // size=8, align=8
    let layout3 = Layout::new::<(u32, u64)>();   // size=16, align=8

    // 方式 3：对于数组
    let layout4 = Layout::array::<u32>(100).unwrap(); // size=400, align=4

    // 方式 4：重复——从已有 layout 扩展为 N 个元素的数组
    let elem = Layout::new::<u64>();
    let array = elem.repeat(100).unwrap(); // (size=800, align=8)
    // 返回 (Layout, usize) —— layout 是整个数组的，usize 是填充后的元素步长

    // 方式 5：对齐到更大的边界
    let padded = layout1.align_to(64).unwrap();

    // 方式 6：扩展到容纳另一个 layout
    let combined = layout1.extend(Layout::new::<u32>()).unwrap();
    // 返回 (Layout, usize) —— (合并后的 layout, 第二个字段的偏移)
}
```

理解 `Layout` 的意义在于：分配器实现者必须尊重对齐要求。如果你返回一个对齐 1 但要求对齐 8 的指针，程序会在 SIMD 指令或某些平台上的原子操作中崩溃。未对齐的访问是未定义行为。

## 3. Arena / Bump 分配器 —— 最简单也最高效的策略

### 3.1 Arena 分配的基本原理

Arena 分配器（也叫 bump allocator）是所有分配策略中最简单的：你预分配一大块内存，然后维护一个"当前位置"指针（cursor/bump pointer）。每次分配，指针向前移动请求的大小并返回旧位置。释放时什么都不做——所有内存在 arena 被销毁时一次性回收。

Arena 分配器之所以高效，是因为它的分配操作只需要一次指针加法加一次边界检查（通常只需要几个 CPU 指令）。没有空闲链表遍历，没有碎片整理，没有任何锁争用（在单线程场景）。代价是你无法在 arena 生命周期内释放单个对象——要么全留要么全丢。

```rust
/// 一个最小可行的 bump 分配器
/// 使用预先分配的 u8 数组作为后备存储
struct BumpAllocator<const N: usize> {
    memory: [u8; N],     // 后备存储——栈上分配的固定大小数组
    position: usize,      // 当前分配位置——cursor
}

impl<const N: usize> BumpAllocator<N> {
    fn new() -> Self {
        BumpAllocator {
            memory: [0u8; N],
            position: 0,
        }
    }

    /// 从 arena 中分配一块内存
    /// 返回 None 表示空间不足
    fn allocate(&mut self, layout: Layout) -> Option<*mut u8> {
        // 步骤 1：计算对齐位置
        let align = layout.align();
        let curr = self.position;
        // 向上对齐到 align 的倍数
        let aligned = (curr + align - 1) & !(align - 1);

        // 步骤 2：检查是否有足够空间
        let new_position = aligned + layout.size();
        if new_position > N {
            return None; // 空间不足
        }

        // 步骤 3：移动 cursor 并返回旧位置
        self.position = new_position;
        Some(self.memory[aligned..].as_mut_ptr())
    }

    /// 重置 arena——释放所有已分配的内存
    fn reset(&mut self) {
        self.position = 0;
        // 注意：这不会调用任何对象的 drop！
        // arena 的使用者必须确保在 reset 之前手动清理
    }
}
```

这个实现仅有三步操作：对齐计算、边界检查、cursor 移动。三步都是 O(1) 的。这比 `malloc` 快了数个数量级——一个 bump 分配大约 2-3 个 CPU 周期，而 `malloc` 可能需要几十甚至几百个周期。

### 3.2 将 BumpAllocator 适配为 GlobalAlloc

要让 arena 分配器作为全局分配器使用，需要处理一个关键问题：`GlobalAlloc` 的 `alloc` 接收的是 `&self`，但我们的 `BumpAllocator` 需要 `&mut self` 来修改 cursor。解决方法是在内部使用 `UnsafeCell` 实现内部可变性。

```rust
use std::alloc::{GlobalAlloc, Layout};
use std::cell::UnsafeCell;

/// 一个使用 bump 策略的全局分配器
/// 注意：这只是一个教学示例——真正的全局 bump 分配器无法处理释放
struct BumpGlobalAllocator<const N: usize> {
    memory: UnsafeCell<[u8; N]>,  // UnsafeCell 提供内部可变性
    position: UnsafeCell<usize>,
}

// 必须标记为 Sync——如果用于多线程程序
// 这里我们假设单线程使用
unsafe impl<const N: usize> Sync for BumpGlobalAllocator<N> {}

impl<const N: usize> BumpGlobalAllocator<N> {
    const fn new() -> Self {
        BumpGlobalAllocator {
            memory: UnsafeCell::new([0u8; N]),
            position: UnsafeCell::new(0),
        }
    }
}

unsafe impl<const N: usize> GlobalAlloc for BumpGlobalAllocator<N> {
    unsafe fn alloc(&self, layout: Layout) -> *mut u8 {
        let pos = &mut *self.position.get();
        let align = layout.align();
        let aligned = (*pos + align - 1) & !(align - 1);
        let new_pos = aligned + layout.size();

        if new_pos > N {
            std::ptr::null_mut() // 分配失败
        } else {
            *pos = new_pos;
            let mem = &mut *self.memory.get();
            mem[aligned..].as_mut_ptr()
        }
    }

    unsafe fn dealloc(&self, _ptr: *mut u8, _layout: Layout) {
        // Arena 分配器不处理单次释放
        // 在真正的 arena 使用中，这是有意为之
        // 但对于 GlobalAlloc，这会导致内存永远不会被回收
        // 这也是为什么 arena 通常不作为全局分配器使用
    }
}
```

这个实现揭示了 arena 策略的本质限制：`dealloc` 不做任何事情，这意味着在全局分配器层面使用 arena 策略会导致内存只增不减。在真实的生产代码中，arena 分配器通常不是全局的——它被用于一个特定作用域（一个请求、一个编译 pass），在该作用域结束时整体释放。

### 3.3 Arena 分配的典型应用场景

Arena 分配最适合以下三种场景。第一是编译器——每个编译 pass 产生大量 AST/IR 节点，在 pass 结束或编译完成时一次性释放。第二是游戏引擎——每帧的临时数据（碰撞检测结果、粒子位置、渲染命令）在帧末清空。第三是 HTTP 服务器——每个请求的处理过程中的临时分配在请求结束时批量回收。

这三种场景的共同特点是：对象的寿命是"批量的"——所有对象同时诞生，同时死亡。使用 arena 分配器，你不仅避免了 `malloc/free` 的开销，还避免了逐个调用 `drop` 的开销（如果你处理的是不需要 `Drop` 的纯数据类型）。

```rust
// 概念性示例：在 HTTP 请求处理中使用 arena
// 这不是可运行的代码，而是展示 arena 的使用模式

// struct RequestContext {
//     arena: BumpAllocator<65536>,  // 64KB arena per request
//     // ... 其他请求级数据
// }
//
// fn handle_request(ctx: &RequestContext) -> Response {
//     // 所有临时分配都走 arena
//     let parsed_json = parse_json_into_arena(body, &ctx.arena);
//     let validation_result = validate_in_arena(&parsed_json, &ctx.arena);
//     let response_json = build_response_in_arena(validation_result, &ctx.arena);
//     // arena 在请求结束时被 reset——所有临时数据被抛弃
//     // 不需要任何 free 调用
// }
```

## 4. Slab 分配器 —— 固定大小对象池

### 4.1 Slab 的设计动机

当你需要频繁分配和释放同一大小的对象时，通用分配器会浪费大量时间在空闲链表上搜索合适大小的块。Slab 分配器（也叫 pool allocator）解决这个问题：它预分配一个包含 N 个固定大小槽位的数组，每个槽位要么是"已占用"要么是"空闲"。分配时找一个空闲槽位并标记为占用，释放时标记为空闲。时间复杂度 O(1)——不需要搜索，不需要合并碎片。

典型使用场景包括：网络服务器的连接对象（每个 TCP 连接一个固定大小的结构体）、编译器的 AST 节点（大部分节点大小相近）、游戏引擎的粒子系统（每个粒子相同大小）。

### 4.2 Slab 分配器的核心实现

下面是一个使用空闲链表追踪空闲槽位的 slab 分配器实现。每个空闲槽位用于存储下一个空闲槽位的索引——这本质上是一个侵入式链表，无需额外内存。

```rust
use std::alloc::{alloc, dealloc, Layout};
use std::ptr;

/// 一个固定大小对象的 slab 分配器
struct SlabAllocator<T> {
    // 后备存储——一片连续内存，划分为 slot_size 大小的槽位
    memory: *mut u8,
    capacity: usize,        // 最大槽位数
    slot_size: usize,       // 每个槽位的实际大小（>= size_of::<T>()）
    // 空闲链表的头——指向第一个空闲槽位
    free_head: Option<usize>,
    // 已分配槽位的计数
    allocated: usize,
}

impl<T> SlabAllocator<T> {
    /// 创建一个可以容纳 capacity 个 T 类型对象的 slab
    fn new(capacity: usize) -> Self {
        let slot_size = std::mem::size_of::<T>().max(std::mem::size_of::<usize>());
        let align = std::mem::align_of::<T>();
        let layout = Layout::from_size_align(slot_size * capacity, align).unwrap();

        let memory = unsafe { alloc(layout) };
        if memory.is_null() {
            panic!("slab allocation failed");
        }

        // 初始化空闲链表：将所有槽位链接起来
        // slot 0 → slot 1 → slot 2 → ... → None
        unsafe {
            for i in 0..capacity - 1 {
                let slot_ptr = memory.add(i * slot_size) as *mut usize;
                *slot_ptr = i + 1; // 指向下一个空闲槽位
            }
            // 最后一个槽位指向 None
            let last_ptr = memory.add((capacity - 1) * slot_size) as *mut usize;
            *last_ptr = usize::MAX; // 哨兵值——表示链表结束
        }

        SlabAllocator {
            memory,
            capacity,
            slot_size,
            free_head: Some(0),
            allocated: 0,
        }
    }

    /// 分配一个槽位——返回指向该槽位的指针
    fn allocate(&mut self) -> Option<*mut T> {
        let index = self.free_head?;
        let slot_ptr = unsafe { self.memory.add(index * self.slot_size) };

        // 更新空闲链表头
        let next_free = unsafe { *(slot_ptr as *const usize) };
        self.free_head = if next_free == usize::MAX {
            None
        } else {
            Some(next_free)
        };

        self.allocated += 1;
        Some(slot_ptr as *mut T)
    }

    /// 释放一个之前分配的槽位
    unsafe fn deallocate(&mut self, ptr: *mut T) {
        let offset = (ptr as usize) - (self.memory as usize);
        let index = offset / self.slot_size;
        debug_assert!(index < self.capacity, "pointer does not belong to this slab");

        // 将该槽位的引用写回——存放下一个空闲槽位的索引
        let slot_ptr = self.memory.add(index * self.slot_size) as *mut usize;
        *slot_ptr = self.free_head.unwrap_or(usize::MAX);

        // 更新空闲链表头为当前释放的槽位
        self.free_head = Some(index);
        self.allocated -= 1;
    }
}

// 当 SlabAllocator 销毁时，释放所有后备存储
impl<T> Drop for SlabAllocator<T> {
    fn drop(&mut self) {
        unsafe {
            let layout = Layout::from_size_align(
                self.slot_size * self.capacity,
                std::mem::align_of::<T>(),
            ).unwrap();
            dealloc(self.memory, layout);
        }
    }
}
```

需要注意一个关键细节：空闲槽位用于存储 `usize`——下一个空闲槽位的索引。这意味着 `slot_size` 必须足够大以容纳一个 `usize`。如果 `T` 小于 `usize`（比如 `u8`），槽位大小必须提升到 `max(size_of::<T>(), size_of::<usize>())`。这是侵入式数据结构的代价——槽位必须足够大才能兼作链表节点。

### 4.3 Slab 分配器 vs 通用分配器——性能差异的来源

通用分配器在分配固定大小对象时也要经历以下步骤：检查线程本地缓存 → 检查对应大小类别的空闲链表 → 如果空闲链表为空，向操作系统申请更多内存并切分 → 返回指针。Slab 分配器则只需：检查空闲链表头 → 弹出第一个节点 → 返回指针。前者可能需要数十个 CPU 周期，后者通常只需要 2-3 个。

更重要的是碎片问题：通用分配器在长期运行的程序中会产生外部碎片——虽然有空闲内存，但没有一块足够大的连续区域来满足分配请求。Slab 分配器因为所有槽位大小相同且预分配，完全不会有碎片问题。

## 5. Allocator Trait（nightly）—— 为单个集合指定分配器

### 5.1 `std::alloc::Allocator` —— 非全局的分配器 trait

截至本文编写时，`Allocator` trait 仍在 nightly 通道中（需要 `#![feature(allocator_api)]`）。与 `GlobalAlloc` 不同，`Allocator` trait 允许你为单个 `Vec`、`Box`、`BTreeMap` 等集合指定不同的分配器，而无需替换全局分配器。

`Allocator` trait 的核心方法签名与 `GlobalAlloc` 类似，但使用 `NonNull` 和 `Result` 而不是裸指针和 null：

```rust
// 需要 nightly Rust： #![feature(allocator_api)]
// 以下是概念性的 trait 定义（简化版）

use std::alloc::{AllocError, Allocator, Layout};
use std::ptr::NonNull;

// pub unsafe trait Allocator {
//     fn allocate(&self, layout: Layout) -> Result<NonNull<[u8]>, AllocError>;
//
//     unsafe fn deallocate(&self, ptr: NonNull<u8>, layout: Layout);
//
//     // 还有 allocate_zeroed, grow, shrink, grow_zeroed, shrink_zeroed
// }
```

与 `GlobalAlloc` 的关键区别在于：`Allocator` 用 `NonNull<[u8]>` 返回指针+大小的胖指针，用 `Result` 而非 null 处理错误，以及它可以是局部的——你可以有多个不同的 allocator 实例在同一个程序中共存。

### 5.2 `new_in` 系列 API——为集合指定分配器

使用 `allocator_api` feature 后，标准库的集合类型会获得一个"带分配器"的变体构造方法：

```rust
// 需要 nightly Rust
// #![feature(allocator_api)]

use std::vec::Vec;
use std::boxed::Box;
// 假设 MyAllocator 实现了 Allocator trait

// let my_alloc = MyAllocator::new();

// Box::new_in(42, &my_alloc)       —— 在指定分配器中分配 Box<i32>
// Vec::new_in(&my_alloc)           —— 使用指定分配器的 Vec<T, &MyAllocator>
// String::new_in(&my_alloc)        —— 使用指定分配器的 String
// BTreeMap::new_in(&my_alloc)      —— 使用指定分配器的 BTreeMap

// 类型的泛型参数会包含分配器类型
// Vec<i32, &MyAllocator> —— 不再是 Vec<i32>（后者默认使用 Global）
```

这意味着你可以为一个 `Vec` 使用 arena 分配器，而程序的其他部分仍然使用全局分配器。这是对内存控制粒度的巨大改进——你不再需要因为一个数据结构的需求而改变整个程序的行为。

### 5.3 使用 Arena 作为 Allocator（nightly 示例）

将之前定义的 arena 分配器适配为 `Allocator` trait 是连接两个概念的桥梁。

```rust
// 需要 nightly Rust
// #![feature(allocator_api)]

use std::alloc::{AllocError, Allocator, Layout};
use std::ptr::NonNull;
use std::cell::RefCell;

/// 一个实现了 Allocator trait 的 Bump 分配器
/// 使用 RefCell 实现 &self 方法中的内部可变性
struct BumpAlloc {
    memory: RefCell<Vec<u8>>,
    position: RefCell<usize>,
}

impl BumpAlloc {
    fn new(capacity: usize) -> Self {
        BumpAlloc {
            memory: RefCell::new(vec![0u8; capacity]),
            position: RefCell::new(0),
        }
    }

    fn reset(&self) {
        *self.position.borrow_mut() = 0;
    }
}

unsafe impl Allocator for BumpAlloc {
    fn allocate(&self, layout: Layout) -> Result<NonNull<[u8]>, AllocError> {
        let mut pos = self.position.borrow_mut();
        let mut mem = self.memory.borrow_mut();
        let align = layout.align();
        let aligned = (*pos + align - 1) & !(align - 1);
        let new_pos = aligned + layout.size();

        if new_pos > mem.len() {
            return Err(AllocError);
        }

        *pos = new_pos;
        let slice = &mut mem[aligned..new_pos];
        let ptr = NonNull::new(slice.as_mut_ptr()).unwrap();
        Ok(NonNull::slice_from_raw_parts(ptr, layout.size()))
    }

    unsafe fn deallocate(&self, _ptr: NonNull<u8>, _layout: Layout) {
        // arena 分配器不单独释放
        // 所有内存在 reset() 或 drop 时回收
    }
}
```

```rust
//     // 所有 AST 节点都分配在 arena 中
//     let ast: Vec<AstNode, &BumpAlloc> = Vec::new_in(arena);
//     let name: String = String::from("example");
//     // ... 构建 AST ...
//     // arena.reset() —— 整个 AST 树被丢弃
//     Ok(())
```

## 6. 更换全局分配器 —— jemalloc 和 mimalloc

### 6.1 为什么换分配器——默认分配器的问题

不同分配器在不同工作负载下表现差异巨大。glibc 的默认 `malloc`（ptmalloc3）在以下场景可能表现不佳：高并发多线程分配（锁争用严重）、大量小对象分配（元数据开销大）、频繁分配-释放循环（碎片累积快）。jemalloc 和 mimalloc 针对这些场景做了大量优化。

jemalloc（由 FreeBSD 和 Facebook 开发）擅长多线程环境——它的线程缓存设计减少了锁争用，并且对碎片控制极为出色。mimalloc（由微软开发）使用极致的延迟释放和分片设计，在通用场景和多线程场景都表现出色，尤其适合频繁的小对象分配。

### 6.2 在 Rust 中集成 jemalloc

在 Rust 中使用 jemalloc 非常直接——使用 `jemallocator` crate，然后用 `#[global_allocator]` 注册。

要在 `Cargo.toml` 中添加依赖：

```toml
[dependencies]
jemallocator = "0.5"

# 如果要控制 jemalloc 的特性
# [dependencies.jemallocator]
# version = "0.5"
# features = ["profiling", "stats", "unprefixed_malloc_on_supported_platforms"]
```

然后在主模块中注册：

```rust
// 在 main.rs 或 lib.rs 中

use jemallocator::Jemalloc;

#[global_allocator]
static GLOBAL: Jemalloc = Jemalloc;

// 程序中的所有 Box、Vec、String 等现在使用 jemalloc
// 而不是默认的系统分配器

fn main() {
    // jemalloc 的多线程分配性能在此处发挥作用
    let v: Vec<u32> = (0..1_000_000).collect();
    println!("vec capacity: {}", v.capacity());
}
```

jemalloc 有一个额外的优势：它内置了强大的性能统计和分析能力。通过 `jemalloc_ctl` crate，你可以在运行时查询分配量、碎片率、线程缓存命中率等指标——这对于定位内存相关的性能问题极为有用。

### 6.3 在 Rust 中集成 mimalloc

mimalloc 的集成同样简单——使用 `mimalloc` crate：

在 `Cargo.toml` 中：

```toml
[dependencies]
mimalloc = "0.1"

# mimalloc 的特性：
# - "secure": 启用安全检查（稍微增加分配开销）
# - "local_dynamic_tls": 在某些平台上解决 TLS 问题
```

注册如下：

```rust
// 在 main.rs 或 lib.rs 中

use mimalloc::MiMalloc;

#[global_allocator]
static GLOBAL: MiMalloc = MiMalloc;

fn main() {
    // 所有分配现在通过 mimalloc
    let mut v = Vec::new();
    for i in 0..100_000 {
        v.push(i);
    }
    println!("len: {}", v.len());
}
```

mimalloc 的一个独特优势是它在 Windows 上对默认分配器的性能提升最为显著（因为 Windows 的默认分配器性能较差），同时它在 Linux 和 macOS 上也表现优异。它在 FireFox 中作为默认分配器经过了大量生产环境验证。

### 6.4 如何选择 jemalloc vs mimalloc vs 默认

没有"一刀切"的答案，但有一些启发式规则。如果你的程序是典型的服务器（多线程、长运行、中等大小对象的分配为主），jemalloc 通常是一个安全的默认选择——它的碎片控制经历了近 20 年的优化。如果你的程序分配大量小对象（比如 AST 节点、小型的网络消息），mimalloc 可能表现更好——它的小对象路径经过了极端优化。如果你在做嵌入式开发或对二进制大小敏感，默认分配器是最好的选择——它不需要额外的依赖，且已经链接在你的程序中。

最可靠的方法是实际测试。用 `cargo bench` 配合不同的分配器，测量分配密集型操作的性能差异。不要仅凭直觉——分配器的表现在不同硬件、不同工作负载和不同对象大小分布下差异巨大。

## 7. 分配器 API 的安全性要求——你必须遵守的合约

### 7.1 alloc 的安全性合约

实现 `GlobalAlloc` 或 `Allocator` 是 unsafe 的——你承担了以下合约责任。违反任何一条都可能导致未定义行为，而 UB 意味着从"似乎正常工作"到"随机崩溃"到"安全漏洞"的任何情况。

`alloc` 方法必须保证：
- 返回的指针对于 `layout.size()` 字节必须是有效的
- 返回的指针必须对齐到 `layout.align()`
- 返回的内存块不能与任何其他已分配的内存块重叠
- 如果 `layout.size() == 0`，行为是未指定的——可以返回任何对齐的非空悬垂指针或 null
- 分配失败时必须返回 null（GlobalAlloc）或 `Err(AllocError)`（Allocator）

### 7.2 dealloc 的安全性合约

`dealloc` 的合约更加苛刻——它保证你是这块内存的"合法所有者"：
- `ptr` 必须是通过同一个分配器的 `alloc` 获得的
- `layout` 必须与分配时使用的完全一致——大小和对齐都必须匹配
- 同一块内存不能被释放两次（double-free）
- `ptr` 在释放后不能被使用（use-after-free）

这些规则不是你"应该遵守"的建议——违反它们直接导致 UB。Rust 的 borrow checker 在 safe Rust 中帮你防止了这些错误，但 unsafe 分配器代码中，你完全是自己的安全守卫。

### 7.3 实现分配器时的常见陷阱

编写自定义分配器时最容易犯几个错误。下面用注释标注了危险点：

```rust
use std::alloc::{GlobalAlloc, Layout};

// 一个"有 bug"的分配器——逐行标注了陷阱
struct BuggyAllocator {
    // ...
}

unsafe impl GlobalAlloc for BuggyAllocator {
    unsafe fn alloc(&self, layout: Layout) -> *mut u8 {
        // 陷阱 1：忽略对齐——返回了可能不对齐的指针
        // 如果调用者期望 8 字节对齐用于 u64/f64/SIMD，未对齐 = UB
        // let ptr = get_some_memory();
        // ptr  // ← 错误：可能只有 1 字节对齐

        // 陷阱 2：零大小分配没有正确处理
        // 如果 layout.size() == 0，null_mut 是合法的
        // 但在某些调用者那里，非空的悬垂指针更好——两者都可以

        // 陷阱 3：线程不安全——多个线程同时调用 alloc
        // 如果内部修改了共享状态而没有同步，就是数据竞争 = UB
        // self.internal_counter += 1; // ← 没有原子操作或锁 = 数据竞争

        // 陷阱 4：内存泄漏——分配的内存永远不会被释放
        // 这不是 UB，但会导致程序的常驻内存不断增长

        std::ptr::null_mut()
    }

    unsafe fn dealloc(&self, ptr: *mut u8, layout: Layout) {
        // 陷阱 5：释放后立即重用内存
        // free(ptr);
        // let new_ptr = alloc(next_layout); // ← 可能重用同一块内存
        // 这不是 UB，但可能在多线程下导致竞争

        // 陷阱 6：忘记检查 ptr 是否为 null
        // 如果 ptr 是 null 且 layout.size() > 0，这可能是个 bug
        // 如果 ptr 是 null 且 layout.size() == 0，什么也不做是安全的
    }
}
```

## 8. 用自定义包装器分析分配行为

### 8.1 构建分配统计包装器

最简单的性能分析工具是一个记录分配统计的包装分配器。它可以帮你回答以下问题：程序分配了多少次？总共分配了多少字节？峰值内存是多少？分配大小分布如何？

```rust
use std::alloc::{GlobalAlloc, Layout, System};
use std::sync::atomic::{AtomicU64, Ordering};

/// 一个记录分配统计信息的全局分配器包装器
/// 实际分配委托给 System，仅记录统计
struct StatsAllocator;

// 使用原子变量在无锁的情况下安全地跨线程计数
static ALLOC_COUNT: AtomicU64 = AtomicU64::new(0);
static ALLOC_BYTES: AtomicU64 = AtomicU64::new(0);
static DEALLOC_COUNT: AtomicU64 = AtomicU64::new(0);
static DEALLOC_BYTES: AtomicU64 = AtomicU64::new(0);
static PEAK_BYTES: AtomicU64 = AtomicU64::new(0);

unsafe impl GlobalAlloc for StatsAllocator {
    unsafe fn alloc(&self, layout: Layout) -> *mut u8 {
        ALLOC_COUNT.fetch_add(1, Ordering::Relaxed);
        let current = ALLOC_BYTES.fetch_add(layout.size() as u64, Ordering::Relaxed);
        let new_total = current + layout.size() as u64;
        // 更新峰值——使用 fetch_max 或 loop-CAS 模式
        let mut peak = PEAK_BYTES.load(Ordering::Relaxed);
        while new_total > peak {
            match PEAK_BYTES.compare_exchange_weak(
                peak, new_total, Ordering::Relaxed, Ordering::Relaxed
            ) {
                Ok(_) => break,
                Err(actual) => peak = actual,
            }
        }
        System.alloc(layout)
    }

    unsafe fn dealloc(&self, ptr: *mut u8, layout: Layout) {
        DEALLOC_COUNT.fetch_add(1, Ordering::Relaxed);
        DEALLOC_BYTES.fetch_add(layout.size() as u64, Ordering::Relaxed);
        System.dealloc(ptr, layout)
    }
}

// 查询统计信息的 API
impl StatsAllocator {
    fn report() {
        let allocs = ALLOC_COUNT.load(Ordering::Relaxed);
        let alloc_bytes = ALLOC_BYTES.load(Ordering::Relaxed);
        let deallocs = DEALLOC_COUNT.load(Ordering::Relaxed);
        let dealloc_bytes = DEALLOC_BYTES.load(Ordering::Relaxed);
        let peak = PEAK_BYTES.load(Ordering::Relaxed);

        eprintln!("=== Allocator Stats ===");
        eprintln!("  Allocations:   {} ({} bytes)", allocs, alloc_bytes);
        eprintln!("  Deallocations: {} ({} bytes)", deallocs, dealloc_bytes);
        eprintln!("  Current usage: {} bytes", 
                  alloc_bytes.saturating_sub(dealloc_bytes));
        eprintln!("  Peak usage:    {} bytes", peak);
        if allocs > 0 {
            eprintln!("  Avg alloc:     {} bytes", alloc_bytes / allocs);
        }
    }
}
```

这个包装器揭示了程序的内存行为——你可以通过它发现意外的分配热点（比如在热循环中分配临时 String）、泄露（分配数远超释放数）、或过大的分配请求。

### 8.2 在程序退出时打印统计

在程序退出时自动打印分配统计需要一个能感知程序生命周期的钩子。`Drop` 不能用于全局静态变量（它们不会被 drop），但你可以用 `libc::atexit`（在 Unix 上）或直接在主函数结束前调用。

```rust

// 或者直接在 main 函数最后打印

// 使用一个显式的 drop guard——在 main 中使用它
```

## 10. 练习答案

### 练习 1 答案

```rust
use std::alloc::Layout;
use std::mem;

struct Bump {
    // 所有 chunk 的列表——新 chunk 总是追加
    chunks: Vec<Vec<u8>>,
    position: usize,      // 当前 chunk 中的分配位置
    chunk_size: usize,    // 每个新 chunk 的初始大小
    current_chunk: usize, // chunks 中当前正在使用的 chunk 的索引
}

impl Bump {
    fn new(chunk_size: usize) -> Self {
        let mut bump = Bump {
            chunks: Vec::new(),
            position: 0,
            chunk_size,
            current_chunk: 0,
        };
        // 预分配第一个 chunk
        bump.chunks.push(vec![0u8; chunk_size]);
        bump
    }

    fn alloc<T>(&mut self, val: T) -> &mut T {
        let layout = Layout::new::<T>();
        let ptr = self.alloc_raw(layout) as *mut T;
        unsafe {
            ptr.write(val);
            &mut *ptr
        }
    }

    fn alloc_slice<T: Default + Copy>(&mut self, len: usize) -> &mut [T] {
        let layout = Layout::array::<T>(len).expect("layout overflow");
        let ptr = self.alloc_raw(layout) as *mut T;
        unsafe {
            for i in 0..len {
                ptr.add(i).write(T::default());
            }
            std::slice::from_raw_parts_mut(ptr, len)
        }
    }

    fn alloc_raw(&mut self, layout: Layout) -> *mut u8 {
        let align = layout.align();
        let pos = self.position;
        let aligned = (pos + align - 1) & !(align - 1);

        if aligned + layout.size() <= self.chunks[self.current_chunk].len() {
            // 当前 chunk 有足够空间
            self.position = aligned + layout.size();
            unsafe { self.chunks[self.current_chunk].as_mut_ptr().add(aligned) }
        } else {
            // 需要新的 chunk——至少能容纳当前分配
            let new_chunk_size = self.chunk_size.max(layout.size() + align);
            self.chunks.push(vec![0u8; new_chunk_size]);
            self.current_chunk += 1;
            self.position = layout.size(); // 新 chunk 从 0 开始，不需要对齐
            unsafe { self.chunks[self.current_chunk].as_mut_ptr() }
        }
    }

    fn reset(&mut self) {
        // 保留第一个 chunk，只重置位置
        self.position = 0;
        self.current_chunk = 0;
        // 丢弃多余的 chunks，释放内存
        self.chunks.truncate(1);
    }
}
```

### 练习 3 答案

解决此问题有几种思路，取决于你对 nightly 的接受度。

**方案 A（nightly——使用 allocator_api）**：实现 `Allocator` trait，然后用 `Box::new_in(node, &slab_alloc)` 创建每个 `Expr` 节点的 `Box`。类型签名变为 `Box<Expr, &SlabAllocator>`。优点是完全利用标准库的分配器抽象，所有 `Box` 的行为（drop、移动等）完全正常。缺点是只能在 nightly 上运行，且每个 `Box` 的类型参数中嵌入了分配器引用——导致不同分配器的 `Box` 之间类型不兼容。

**方案 B（stable——原始指针 + 手动管理）**：不依赖 `Box` 的分配器参数化，而是用原始指针手动实现。将 `Expr` 中的 `Box<Expr>` 替换为 `*const Expr`，在 slab 上手动分配节点并通过原始指针引用。drop 时遍历 slab 并手动调用 drop。优点是在 stable Rust 上可运行。缺点是需要更繁琐的 unsafe 代码，失去了 `Box` 的自动内存管理。

**方案 C（stable——重新定义节点类型）**：不使用递归的 `Box<Expr>`，而将所有节点存储在一个扁平的数组中，用索引而非指针引用子节点。例如 `Expr` 变为 `enum Expr { Number(i64), Add(usize, usize), Mul(usize, usize) }` 其中 `usize` 是索引，实际节点存储在 `Vec<Expr>` 中。这实际上是 arena 的另一种形式——`Vec` 充当了分配器。优点是纯 safe Rust，性能极好（连续内存，cache 友好）。缺点是失去了递归数据结构的直观性。

```
自定义分配器 学习清单：
```
