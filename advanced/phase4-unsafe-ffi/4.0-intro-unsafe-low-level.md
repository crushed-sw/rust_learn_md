# 4.0 底层突破 —— 从"安全区内写代码"到"理解机器的真实运作"

> Rust 的安全保证——所有权、借用、生命周期——在 99% 的场景下让你无需担心内存错误。但剩下的 1% 呢？当你需要直接操作硬件寄存器、调用 C 库、实现一个比标准库更高效的数据结构、或者仅仅是想理解"编译器到底把我的代码编译成了什么"的时候——你就需要进入 unsafe Rust 的世界。这不是一个"危险区域"，而是一个**责任转移区**：编译器把一部分证明义务从自己身上转移给了你。本 Phase 的目标不是让你变成一个"unsafe 狂魔"，而是让你理解 unsafe 的边界在哪里、什么时候该跨过去、跨过去后要遵守什么契约——以及最重要的：**如何用 unsafe 构建安全抽象，让调用者永远不需要碰 unsafe。**

---

## 1. 为什么 Rust 需要 unsafe？

### 1.1 编译器不是全能的

Rust 的借用检查器能证明很多事情：没有悬垂指针、没有数据竞争、没有 use-after-free。但它证明不了的也很多：

```
编译器能证明的（safe Rust）：
  ✅ &T 不会在 T 被释放后使用
  ✅ &mut T 是独占的
  ✅ 多线程不会数据竞争
  ✅ 没有空指针解引用（Option<T> 强制检查）

编译器证明不了的（需要 unsafe）：
  ❌ 这段裸指针指向的内存是否有效
  ❌ 这个 union 的哪个字段是活跃的
  ❌ 这个 C 函数会不会破坏 Rust 的不变量
  ❌ 这个 transmute 是否合法
  ❌ 这个汇编指令是否正确
```

unsafe 不是"禁用借用检查器"——它只是在特定位置让你做编译器无法验证的操作。在 unsafe 块之外，所有安全保证依然生效。

### 1.2 unsafe 的真实含义

一个常见的误解：**unsafe = 危险的代码**。

真相：**unsafe 的代码不一定是危险的；安全的代码也不一定是正确的。** 

```
unsafe 的真实含义：
  编译器说："我无法证明这段代码满足所有安全不变量。
            我选择信任你——你在 unsafe 块里写的东西，
            你来保证它不违反 Rust 的安全规则。"

Safe Rust 的契约：
  无论你写什么 safe Rust 代码，都不会导致未定义行为（UB）。
  （逻辑错误不算 UB——比如死循环、业务逻辑错误）

Unsafe Rust 的契约：
  你可以在 unsafe 块中做 safe Rust 做不到的事。
  但你必须保证：从外部看，这段代码的行为不违反 safe Rust 的安全不变量。
```

这就是为什么标准库里到处是 unsafe——`Vec`、`String`、`Box`、`Arc`、`Mutex` 内部都用了 unsafe，但它们暴露给用户的 API 是安全的。**用 unsafe 实现安全抽象，是 Rust 最核心的设计模式之一。**

## 2. 五条 unsafe 超能力——你的"逃生舱"清单

在 safe Rust 中不能做但在 unsafe 中能做的，只有这五件事：

```
┌─────────────────────────────────────────────────────────────┐
│              五条 unsafe 超能力（Rustonomicon）               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 解引用裸指针（*const T / *mut T）                         │
│     → safe Rust 中只能通过引用访问数据                          │
│                                                             │
│  2. 调用 unsafe 函数或方法                                    │
│     → 包括 FFI 函数、编译器内建函数等                           │
│                                                             │
│  3. 访问或修改可变静态变量（static mut）                        │
│     → 数据竞争风险，safe Rust 禁止                              │
│                                                             │
│  4. 实现 unsafe trait                                       │
│     → Send / Sync 等，你承诺满足它们的契约                      │
│                                                             │
│  5. 访问 union 的字段                                        │
│     → 编译器不知道哪个字段是活跃的                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

注意什么**不在**这个清单里：
- 绕过借用检查器？❌ 借用检查器在 unsafe 中仍然工作
- 关闭生命周期检查？❌ 生命周期的基本规则仍然适用
- 随意 transmute？❌ transmute 本身是 unsafe 函数（属于第 2 条），但不能乱用

unsafe 没有"关掉安全机制"，它只是给你打开了五扇特定的门。

## 3. unsafe 的责任模型——抽象安全边界

### 3.1 核心原则：unsafe 是实现细节

```rust
// ✅ 好的设计：unsafe 被封装在安全 API 内部
pub struct MyVec<T> {
    ptr: *mut T,      // unsafe 数据
    len: usize,
    cap: usize,
}

impl<T> MyVec<T> {
    /// 这个函数是 safe 的——调用者不需要写 unsafe
    pub fn push(&mut self, value: T) {
        // 内部实现用了 unsafe，但调用者不关心
        // ... unsafe { /* 裸指针操作 */ }
    }
}

// ❌ 坏的设计：unsafe 泄漏到了公共 API
impl<T> MyVec<T> {
    /// 调用者被迫写 unsafe ——说明抽象有泄漏
    pub unsafe fn get_ptr(&self) -> *const T {
        self.ptr
    }
}
```

**安全抽象的铁律**：如果你的 unsafe 代码可能被 safe 代码以某种方式触发 UB，那么你的 unsafe 代码就是错的——不是调用者的错。

### 3.2 安全不变量文档

写 unsafe 代码时，你必须清楚地文档化你的"安全不变量"——safe 调用者依赖的前提条件：

```rust
/// 从裸指针构造一个 `MyBox<T>`。
///
/// # Safety
///
/// 调用者必须保证：
/// - `ptr` 是通过 `Box::into_raw` 获得的
/// - `ptr` 指向一个有效的、已初始化的 `T`
/// - `ptr` 没有被其他代码释放
/// - 这个函数是唯一使用该指针的地方
pub unsafe fn from_raw(ptr: *mut T) -> MyBox<T> {
    // ...
}
```

这是 Rust 社区的约定——`# Safety` 段说明调用者需要满足什么条件。如果你看标准库的 unsafe 函数文档，每个都有这样的说明。

## 5. 学习建议

1. **先理解"为什么"再学"怎么做"。** 每个 unsafe 特性都有一个明确的存在理由——通常是因为 safe Rust 在某个特定场景下做不到某件事。理解这个"做不到"比记住 API 更重要。

2. **unsafe 代码量越少越好。** 好的 unsafe 代码是小块集中、边界清晰的。如果你发现自己在写大段大段的 unsafe，很可能应该重新审视设计。

3. **使用 Miri 测试 unsafe 代码。** `cargo miri test` 可以在 MIR 层面检测 UB——这是测试 unsafe 代码最重要的工具，比 sanitizer 更精确（但更慢）。

4. **读标准库源码。** `Vec::push`、`Box::new`、`Arc::clone` 的内部实现是学习 unsafe Rust 最好的教材——它们经过数千名开发者的审查和数十亿次运行。

5. **学习目标不是"会用 unsafe"，而是"让调用者不需要 unsafe"。** 你的最终产物应该是一个安全 API，unsafe 只是实现细节。

---

> 💡 **最重要的思维转换**：在 safe Rust 中，编译器的报错是"你不应该这样做"。在 unsafe Rust 中，编译器的沉默是"我信任你——但如果你违反了契约，后果自负"。safe Rust 的哲学是"编译器替你想"，unsafe Rust 的哲学是"你替调用者想"。当你写 `unsafe { ... }` 时，你是在对每一个调用你代码的人做出承诺——"我已经验证过了，这段代码对外部是安全的。" 这是一个沉重的责任，也是 Rust 赋予专家的信任。
