# 4.11 SIMD & 可移植向量化 —— 一条指令，多个数据

> 我们花了半个世纪把 CPU 越做越快——更深的分支预测、更大的缓存、更多的核心。但有一个方向被大多数程序员忽略了：一条指令同时处理多个数据。SIMD（Single Instruction, Multiple Data）正是这个方向的答案。想象你有 256 个 u8 需要加 1：标量循环需要 256 次加法指令；AVX2 只需要 8 次（每次处理 32 个 u8）；AVX-512 只需要 4 次。这就是 32 倍到 64 倍的吞吐量差异——不靠更快的时钟频率，不靠更多的核心，只靠更聪明地使用已经存在的晶体管。Rust 在 SIMD 领域的布局分为两层：`std::simd` 提供可跨平台移植的抽象，`std::arch` 提供对 x86_64 和 ARM 每条指令的精确控制。两个层级的工具配合使用，你可以在保证可移植性的同时榨干特定 CPU 的每一滴性能。

---

## 1. SIMD 是什么——从标量到向量的心智转变

### 1.1 标量处理 vs 向量处理

在标量世界，一条指令处理一对数据：`a + b = c`。如果你想处理 8 对数据，你需要 8 条指令。在 SIMD 世界，一条指令处理一个向量中的所有数据：`[a1, a2, ..., a8] + [b1, b2, ..., b8] = [c1, c2, ..., c8]`。8 对数据仅需 1 条指令。

这不是算法层面的优化——这是硬件级别的并行。现代 x86_64 CPU 都有 AVX2 支持（256 位寄存器，可一次处理 8 个 f32 或 32 个 u8），较新的 CPU 还有 AVX-512（512 位寄存器，一次 16 个 f32 或 64 个 u8）。ARM 平台有 NEON（128 位寄存器，一次 4 个 f32 或 16 个 u8）。WebAssembly 也有自己的 SIMD 提案（128 位）。

```rust
// 标量方式——8 条加法指令，8 个结果
fn scalar_add(a: [f32; 8], b: [f32; 8]) -> [f32; 8] {
    let mut result = [0.0f32; 8];
    for i in 0..8 {
        result[i] = a[i] + b[i];
    }
    result
}

// SIMD 方式（概念）——1 条加法指令，8 个结果同时计算
// fn simd_add(a: Simd<f32, 8>, b: Simd<f32, 8>) -> Simd<f32, 8> {
//     a + b  // ← 这编译成一条 vaddps 指令（AVX）
// }
```

理解 SIMD 的关键是转变你的思维方式：不要想"我如何为一个元素写代码"，而要想"我如何让一个操作同时作用于 4/8/16/32/64 个元素"。这需要重新组织数据布局（数组结构体 vs 结构体数组），重新设计算法（消除分支以使向量化可行），但在正确的场景下，收益是巨大的。

### 1.2 SIMD 的硬件历史——从 MMX 到 AVX-512

了解硬件演进有助于理解为什么有这么多不同的 SIMD 指令集。Intel 的旅途从 MMX（1997，64 位，整数）开始，然后是 SSE（1999，128 位，浮点），SSE2（2000，128 位，双精度浮点和更多整数指令），SSE3/SSSE3/SSE4（2004-2008，专用指令如点积、字节混洗），AVX（2011，256 位，三操作数格式），AVX2（2013，256 位，完整的整数指令），AVX-512（2017，512 位，掩码操作）。

ARM 的旅途：NEON（2009，128 位），SVE（2018，可变长度向量——非常先进的设计），SVE2（2021）。

Rust 的 `std::simd` 正是为了让你不用关心这些差异——它自动映射到目标平台的最佳指令集。

## 2. std::simd —— 可移植的向量化抽象（nightly）

### 2.1 Simd<T, N> 类型——便携 SIMD 的核心

`std::simd` 模块（需要 `#![feature(portable_simd)]`）提供了 `Simd<T, N>` 类型，其中 `T` 是元素类型，`N` 是元素个数。`N` 必须是 2 的幂，且 `T` 必须是"可 SIMD 的"基本类型（整数、浮点、布尔等）。

```rust
// 需要 nightly Rust： #![feature(portable_simd)]

use std::simd::{Simd, simd_swizzle};

fn simd_basics() {
    // 创建一个 Simd 向量——4 个 f32 同时处理
    let a: Simd<f32, 4> = Simd::from_array([1.0, 2.0, 3.0, 4.0]);
    let b: Simd<f32, 4> = Simd::from_array([5.0, 6.0, 7.0, 8.0]);

    // 基本的算术运算——每条在线路上都编译成一条 SIMD 指令
    let sum = a + b;    // [6.0, 8.0, 10.0, 12.0]
    let product = a * b; // [5.0, 12.0, 21.0, 32.0]
    let min_vals = a.min(b); // [1.0, 2.0, 3.0, 4.0]
    let max_vals = a.max(b); // [5.0, 6.0, 7.0, 8.0]

    // lane-wise（逐道）访问
    let first = a[0];  // 提取第一道的值（标量）
    let array: [f32; 4] = sum.to_array(); // 转换为普通数组

    // f32 特有的操作
    let abs_vals = a.abs();     // 绝对值
    let sqrt_vals = sum.sqrt(); // 平方根（每条道独立计算）
}
```

关键点：`Simd<f32, 4>` 的大小是 16 字节（4 * 4），刚好放入一个 SSE 寄存器（128 位）。在支持更宽寄存器的 CPU 上，如果你使用 `Simd<f32, 8>`（32 字节），它会映射到 AVX 的 YMM 寄存器（256 位）。如果你写 `Simd<f32, 16>`（64 字节），在某些平台上它会编译为两条 AVX 指令。

### 2.2 通用 SIMD 操作——对任何元素类型都有效的方法

`std::simd` 提供了一组对所有 SIMD 类型通用的操作。

```rust
use std::simd::{Simd, Mask};

fn generic_simd_ops() {
    let v: Simd<i32, 4> = Simd::from_array([1, 2, 3, 4]);
    let w: Simd<i32, 4> = Simd::from_array([3, 2, 1, 0]);

    // splat：将单个值广播到所有 lane
    let all_twos: Simd<i32, 4> = Simd::splat(2); // [2, 2, 2, 2]

    // 比较——返回 Mask 类型（每条 lane 的布尔值）
    let mask: Mask<i32, 4> = v.lanes_gt(w); // [false, false, true, true]
    // Mask 可用于条件选择
    let selected = mask.select(v, w); // [3, 2, 3, 4]——true 选 v，false 选 w

    // cast：类型转换（保持 lane 数不变）
    let as_f32: Simd<f32, 4> = v.cast(); // [1.0, 2.0, 3.0, 4.0]

    // reduce：将向量归约为单个值
    let sum_all: i32 = v.reduce_sum(); // 1 + 2 + 3 + 4 = 10
    let product_all: i32 = v.reduce_product(); // 1 * 2 * 3 * 4 = 24
    let min_all: i32 = v.reduce_min(); // 1
    let max_all: i32 = v.reduce_max(); // 4

    // gather/scatter：从内存中按索引收集/分散
    let indices = Simd::from_array([0, 2, 1, 3]);
    let source = [10i32, 20, 30, 40, 50];
    // 注意：gather/scatter API 仍在演进中，以下为概念形式
    // let gathered = Simd::<i32, 4>::gather(&source, indices); // [10, 30, 20, 40]
}

// Mask 类型也可以通过位运算组合
fn mask_operations() {
    let a: Simd<i32, 4> = Simd::from_array([1, 2, 3, 4]);
    let b: Simd<i32, 4> = Simd::from_array([4, 3, 2, 1]);

    let gt_mask = a.lanes_gt(b);   // [false, false, true, true]
    let lt_mask = a.lanes_lt(b);   // [true, false, false, false]
    let combined = gt_mask | lt_mask; // [true, false, true, true]

    // 直接创建 mask——只有第 0 和第 3 lane 被选中
    let custom_mask = Mask::from_array([true, false, false, true]);
    let selected = custom_mask.select(a, b); // [1, 3, 2, 1]
}
```

`Mask` 类型是 SIMD 分支消除的关键——传统代码中的 `if` 可以用 `select` 替代，从而避免条件分支（条件分支会阻止向量化）。

### 2.3 Swizzle（混洗）——重新排列向量中的元素

swizzle 操作（也叫 shuffle）让你按照一个索引模式重新排列向量中的元素。这是许多 SIMD 算法的核心——矩阵乘法、快速傅里叶变换、颜色空间转换等都依赖它。

```rust
use std::simd::{Simd, simd_swizzle, Which};

fn swizzle_examples() {
    let v: Simd<f32, 4> = Simd::from_array([1.0, 2.0, 3.0, 4.0]);
    let w: Simd<f32, 4> = Simd::from_array([5.0, 6.0, 7.0, 8.0]);

    // 基本 swizzle——从 v 和 w 中选取元素组成新向量
    // swizzle!(v, w, [i0, i1, i2, i3])
    // i0..i3 的取值为 0-7：
    //   0-3 表示 v 的 lane 0-3
    //   4-7 表示 w 的 lane 0-3

    // 反转 v 的顺序
    let reversed = simd_swizzle!(v, [3, 2, 1, 0]); // [4.0, 3.0, 2.0, 1.0]

    // 交错 v 和 w：取 v[0], w[0], v[1], w[1]
    let interleaved = simd_swizzle!(v, w, [0, 4, 1, 5]); // [1.0, 5.0, 2.0, 6.0]

    // 广播 v 的第一个元素到所有 lane
    let broadcast = simd_swizzle!(v, [0, 0, 0, 0]); // [1.0, 1.0, 1.0, 1.0]

    // 对两个向量进行交错操作（常用于图像处理的 RGBA 通道重排）
    let rgba1 = Simd::from_array([255.0, 0.0, 0.0, 255.0]);     // RGBA
    // 将 RGBA 转为 BGRA：交换 R 和 B 通道（lane 0 和 lane 2）
    let bgra = simd_swizzle!(rgba1, [2, 1, 0, 3]); // [0.0, 0.0, 255.0, 255.0]
}
```

swizzle 操作直接映射到 x86_64 的 `pshufd`/`vpermilps` 或 ARM NEON 的 `vtbl` 指令——它们是 0 个额外周期的寄存器操作（swizzle 在内核内部完成，不消耗额外的执行单元）。

## 3. 平台特定 SIMD —— x86_64 架构

### 3.1 `std::arch::x86_64` —— Intel intrinsic 的直接映射

当 `std::simd` 不够用（或者你需要使用 nightly 之前的 stable Rust），`std::arch::x86_64` 模块提供了 x86 SIMD 指令的一对一映射。每个函数对应一条汇编指令，函数的命名直接沿用了 Intel 的 intrinsic 命名惯例。

命名规则解读很重要：
- 前缀 `_mm` 表示 SSE（128 位）；`_mm256` 表示 AVX（256 位）；`_mm512` 表示 AVX-512（512 位）
- `_add` 加法，`_mul` 乘法，`_min`/`_max` 极值，`_load` 从内存加载，`_store` 存入内存
- `_ps`（packed single precision）= 打包的单精度浮点（4 个 f32）
- `_pd`（packed double precision）= 打包的双精度浮点（2 个 f64）
- `_epi32`（extended packed integer 32）= 打包的 32 位整数

```rust
// 需要目标平台支持
// #![feature(stdsimd)]  // 对于某些 intrinsic 需要

#[cfg(target_arch = "x86_64")]
use std::arch::x86_64::*;

// SSE 示例——4 个 f32 一次处理
#[cfg(target_arch = "x86_64")]
fn sse_example() {
    unsafe {
        // 创建两个 SSE 向量
        let a = _mm_set_ps(4.0, 3.0, 2.0, 1.0); // __m128
        let b = _mm_set_ps(8.0, 7.0, 6.0, 5.0); // __m128

        // _mm_add_ps —— 一次加法，4 个结果
        let sum = _mm_add_ps(a, b); // [6.0, 8.0, 10.0, 12.0]

        // _mm_mul_ps —— 一次乘法，4 个结果
        let product = _mm_mul_ps(a, b);

        // 从浮点数转换为整数（截断）
        let as_ints: __m128i = _mm_cvttps_epi32(sum);

        // 从数组加载和存储
        let data: [f32; 4] = [1.0, 2.0, 3.0, 4.0];
        let loaded = _mm_loadu_ps(data.as_ptr()); // 未对齐的 load
        // _mm_load_ps 要求 16 字节对齐——更快但更严格

        // 存储回数组
        let mut result: [f32; 4] = [0.0; 4];
        _mm_storeu_ps(result.as_mut_ptr(), loaded);
    }
}
```

关键点：所有平台特定的 intrinsic 都是 `unsafe` 的——编译器无法验证你是否正确地使用了它们。调用 `_mm_load_ps` 时如果指针未对齐 16 字节，就是未定义行为。而且这些函数仅在目标 CPU 支持对应指令集时可用——在不支持的 CPU 上调用会导致 `SIGILL`（非法指令信号）。

### 3.2 运行时特性检测——安全地使用 AVX2 和 AVX-512

你不能假设所有 x86_64 CPU 都支持 AVX2 或 AVX-512。需要使用 `is_x86_feature_detected!` 宏在运行时检测 CPU 特性，然后根据结果选择不同的代码路径。

```rust
#[cfg(target_arch = "x86_64")]
use std::arch::x86_64::*;

/// 计算两个 f32 数组的点积
/// 根据 CPU 特性自动选择 SSE4.1 或 AVX2 实现
#[cfg(target_arch = "x86_64")]
fn dot_product(a: &[f32], b: &[f32]) -> f32 {
    assert_eq!(a.len(), b.len());

    if is_x86_feature_detected!("avx2") {
        // AVX2 路径——每次处理 8 个 f32
        unsafe { dot_product_avx2(a, b) }
    } else if is_x86_feature_detected!("sse4.1") {
        // SSE4.1 路径——每次处理 4 个 f32，
        // 同时可以使用 _mm_dp_ps 点积指令
        unsafe { dot_product_sse(a, b) }
    } else {
        // 标量回退——在任何 x86_64 CPU 上都能工作
        a.iter().zip(b.iter()).map(|(x, y)| x * y).sum()
    }
}

#[cfg(target_arch = "x86_64")]
unsafe fn dot_product_sse(a: &[f32], b: &[f32]) -> f32 {
    // 使用 SSE _mm_dp_ps 指令——直接计算 4 元素向量点积
    let mut sum = _mm_setzero_ps();
    let chunks = a.len() / 4;

    for i in 0..chunks {
        let idx = i * 4;
        let va = _mm_loadu_ps(a.as_ptr().add(idx));
        let vb = _mm_loadu_ps(b.as_ptr().add(idx));
        // 0xFF = 所有通道参与，结果广播到所有通道
        let dp = _mm_dp_ps(va, vb, 0xFF);
        sum = _mm_add_ps(sum, dp);
    }

    // 从 SIMD 寄存器提取标量和
    let mut result: [f32; 4] = [0.0; 4];
    _mm_storeu_ps(result.as_mut_ptr(), sum);
    result[0] + result[1] + result[2] + result[3] // 额外项用于 overflow 为 safety
}

// AVX2 的 dot product 实现在下一节展示
```

`is_x86_feature_detected!` 是一个编译时展开为运行时检测的宏。它在程序启动时一次性检测 CPU 特性，然后将结果缓存为布尔值。开销极小——本质上是读取 CPUID 结果。

### 3.3 AVX2 实例——一次处理 8 个 f32

AVX2 使用 256 位寄存器（`__m256`），可以容纳 8 个 f32 或 4 个 f64。相比 SSE（4 个 f32），吞吐量翻倍。

```rust
#[cfg(target_arch = "x86_64")]
use std::arch::x86_64::*;

/// AVX2 实现的点积——每次处理 8 个 f32
#[cfg(target_arch = "x86_64")]
unsafe fn dot_product_avx2(a: &[f32], b: &[f32]) -> f32 {
    // 累加器——初始化为 0（8 个 lane 各自为 0）
    let mut acc = _mm256_setzero_ps();
    let chunks = a.len() / 8;

    for i in 0..chunks {
        let idx = i * 8;
        // _mm256_loadu_ps：从未对齐的内存加载 8 个 f32 到 YMM 寄存器
        let va = _mm256_loadu_ps(a.as_ptr().add(idx));
        let vb = _mm256_loadu_ps(b.as_ptr().add(idx));
        // FMA（Fused Multiply-Add）：acc = acc + va * vb —— 一条指令完成
        acc = _mm256_fmadd_ps(va, vb, acc);
    }

    // 将 8 个累加器归约为一个标量
    // 水平加法：将 256 位向量的上下 128 位相加
    // 然后将 128 位的 4 个 f32 水平相加
    let hadd = _mm256_hadd_ps(acc, acc);  // [a+b, c+d, a+b, c+d, e+f, g+h, e+f, g+h]
    let hadd = _mm256_hadd_ps(hadd, hadd); // [a+b+c+d, ..., e+f+g+h, ...]
    // 提取前两个 lane 并相加得到总和
    let mut result: [f32; 8] = [0.0; 8];
    _mm256_storeu_ps(result.as_mut_ptr(), hadd);
    result[0] + result[4] // 两个 128 位的和
}
```

AVX2 引入了 FMA（Fused Multiply-Add）——一条指令完成 `a * b + c`。这不仅减少了指令数，还提高了精度（中间乘法结果保持在扩展精度中）。在矩阵乘法和卷积等计算中，FMA 可以带来巨大的性能提升。

## 4. 平台特定 SIMD —— ARM NEON

### 4.1 `std::arch::aarch64` —— ARM 平台的原生 SIMD

ARM 平台的 NEON 指令集通过 `std::arch::aarch64` 模块暴露。NEON 使用 128 位寄存器，可以容纳 4 个 f32、2 个 f64、4 个 i32、8 个 i16、16 个 i8。NEON 的命名规则与 x86 完全不同：`vaddq_s32` 表示"对 128 位（q=quad）寄存器进行有符号 32 位整数加法"，`vld1q_s32` 表示"从内存加载 4 个有符号 32 位整数到 128 位寄存器"。

```rust
#[cfg(target_arch = "aarch64")]
use std::arch::aarch64::*;

/// ARM NEON 实现的两个 f32 数组加法
#[cfg(target_arch = "aarch64")]
unsafe fn vector_add_neon(a: &[f32], b: &[f32], result: &mut [f32]) {
    let chunks = a.len() / 4;

    for i in 0..chunks {
        let idx = i * 4;
        // vld1q_f32：从内存加载 4 个 f32 到 NEON 寄存器（128 位）
        let va = vld1q_f32(a.as_ptr().add(idx));
        let vb = vld1q_f32(b.as_ptr().add(idx));

        // vaddq_f32：128 位 f32 加法——4 个同时完成
        let vresult = vaddq_f32(va, vb);

        // vst1q_f32：将结果存储回内存
        vst1q_f32(result.as_mut_ptr().add(idx), vresult);
    }

    // 处理剩余的元素（tail——不足 4 个的部分）
    for i in (chunks * 4)..a.len() {
        result[i] = a[i] + b[i];
    }
}

/// ARM NEON 的点积实现
#[cfg(target_arch = "aarch64")]
unsafe fn dot_product_neon(a: &[f32], b: &[f32]) -> f32 {
    // 累加器——两个 128 位向量的 4 个 f32，初始为 0
    let mut acc0 = vdupq_n_f32(0.0); // 累加器 1
    let mut acc1 = vdupq_n_f32(0.0); // 累加器 2——双累加器提升吞吐量
    let chunks = a.len() / 8; // 每次处理 8 个（两个 NEON 寄存器）

    for i in 0..chunks {
        let idx = i * 8;
        // 加载 8 个 f32 到两个 NEON 寄存器
        let va0 = vld1q_f32(a.as_ptr().add(idx));
        let vb0 = vld1q_f32(b.as_ptr().add(idx));
        let va1 = vld1q_f32(a.as_ptr().add(idx + 4));
        let vb1 = vld1q_f32(b.as_ptr().add(idx + 4));

        // vmlaq_f32：FMA 指令——acc = acc + a * b
        acc0 = vmlaq_f32(acc0, va0, vb0);
        acc1 = vmlaq_f32(acc1, va1, vb1);
    }

    // 归约：将两个 128 位向量相加，然后水平求和
    let acc = vaddq_f32(acc0, acc1);
    let mut arr: [f32; 4] = [0.0; 4];
    vst1q_f32(arr.as_mut_ptr(), acc);
    arr[0] + arr[1] + arr[2] + arr[3]
}
```

注意 ARM NEON 代码中的一个性能技巧：使用双累加器（`acc0` 和 `acc1`）。ARM 的某些微架构可以每个周期执行两条 NEON 指令——如果它们使用不同寄存器（打破数据依赖），则可以并行执行。使用双累加器让处理器更容易发现这种并行性。

### 4.2 可移植代码——在 x86 和 ARM 之间共享

使用条件编译，你可以编写在 x86_64 和 aarch64 上都高效运行的代码。核心模式是定义一个抽象层——一个函数根据编译目标选择实现。

```rust
/// 可移植的 f32 向量点积——x86 用 AVX2/SSE，ARM 用 NEON
fn portable_dot_product(a: &[f32], b: &[f32]) -> f32 {
    assert_eq!(a.len(), b.len());

    #[cfg(target_arch = "x86_64")]
    {
        if is_x86_feature_detected!("avx2") {
            unsafe { return dot_product_avx2(a, b); }
        }
        if is_x86_feature_detected!("sse4.1") {
            unsafe { return dot_product_sse(a, b); }
        }
        // 回退到标量
        return a.iter().zip(b.iter()).map(|(x, y)| x * y).sum();
    }

    #[cfg(target_arch = "aarch64")]
    {
        // ARM NEON 在 AArch64 上几乎是通用的——不需要运行时检测
        unsafe { return dot_product_neon(a, b); }
    }

    #[cfg(not(any(target_arch = "x86_64", target_arch = "aarch64")))]
    {
        // 其他平台——使用标量回退或 std::simd（如果可用）
        a.iter().zip(b.iter()).map(|(x, y)| x * y).sum()
    }
}
```

更好的方案是使用 `std::simd`——它为你处理了所有的平台差异，包括回退逻辑。但当你需要访问特定平台的专用指令（比如 AVX-512 的 vpcompress 或 NEON 的 vtbl）时，`std::arch` 是唯一的选择。

## 5. 自动向量化——当编译器帮你写 SIMD

### 5.1 LLVM 的自动向量化器——编译器什么时候能帮你

LLVM（Rust 的编译器后端）有强大的自动向量化能力。当它遇到简单的循环——没有复杂的分支、没有跨迭代的数据依赖、内存访问模式可预测——它会自动生成 SIMD 指令。

自动向量化的核心条件如下。数据必须对齐或编译器能处理未对齐的加载（`movups` 而非 `movaps`）。循环的迭代次数必须足够大（至少几倍于 SIMD 宽度）。循环体内不能有会导致提前退出的分支（`break`、`return`、`continue` 在某些复杂条件下会阻止向量化）。跨迭代不能有数据依赖（第 N 次迭代不能依赖第 N-1 次迭代的结果，除非是归约模式如累加）。

```rust
/// 自动向量化的理想候选——编译器几乎肯定会自动向量化
fn auto_vectorizable(a: &[f32], b: &[f32]) -> Vec<f32> {
    // 简单的逐元素操作
    // 无分支（除了循环条件）
    // 无跨迭代依赖
    // 内存访问模式——连续、可预测
    a.iter()
        .zip(b.iter())
        .map(|(&x, &y)| x * y + x.sin())
        .collect()
}

/// 难以自动向量化的例子——编译器通常无法向量化
fn hard_to_vectorize(input: &[f32]) -> Vec<f32> {
    let mut result = Vec::new();
    for &x in input {
        if x > 0.0 {
            // 分支——但不是致命的。编译器可以用掩码来向量化
            // 但如果分支体太复杂，向量化收益会被掩盖
            result.push(x.sqrt());
        } else if x < -10.0 {
            result.push(x.abs());
        } else {
            result.push(x.powi(3));
        }
    }
    result
}
```

要查看编译器是否自动向量化了你的代码，可以使用 `cargo rustc --release -- --emit asm` 查看生成的汇编代码（搜索 `%xmm` 或 `%ymm`），或者使用 `-C remark=loop-vectorize`（在 Linux 上通过 LLVM remark）。更简单的做法是测量性能——如果 `-C target-cpu=native` 显著提升了循环性能，自动向量化很可能发生了。

### 5.2 编译器标志——让 LLVM 使用所有可用的 ISA 扩展

默认情况下，`rustc` 为"最低公分母" CPU 编译——通常是 x86-64-v1（仅 SSE2）或 ARMv8-a。这意味着即使你的 CPU 支持 AVX2 或 NEON 的高级指令，编译器也不会使用它们。

你可以通过以下方式启用所有可用的指令集扩展：

```toml
# 在 .cargo/config.toml 中
[build]
rustflags = ["-C", "target-cpu=native"]

# 或者为发布版本指定
[target.'cfg(not(debug_assertions))']
rustflags = ["-C", "target-cpu=native"]

# 你也可以指定特定的目标 CPU
# rustflags = ["-C", "target-cpu=skylake"]
# rustflags = ["-C", "target-cpu=apple-m1"]
```

`-C target-cpu=native` 让 LLVM 为当前 CPU 的所有特性编译——包括 AVX2、AVX-512、FMA、BMI 等。这通常带来 10-30% 的性能提升，仅通过重新编译（而不改变任何代码）。代价是生成的二进制文件可能无法在较旧的 CPU 上运行。另一个有用的标志是 `-C target-feature=+avx2` 用于启用特定特性（在 `native` 不够精确或你需要特定指令集时使用）。

## 6. 实际应用——可运行的真实案例

### 6.1 向量化点积（Dot Product）——完整实现

点积是许多科学计算和机器学习算法的核心操作。这里给出一个使用 `std::simd` 的完整可运行实现（需要 nightly）：

```rust
// 需要 nightly Rust
// #![feature(portable_simd)]

use std::simd::{Simd, simd_swizzle};

/// 使用可移植 SIMD 计算两个 f32 数组的点积
fn dot_product_simd(a: &[f32], b: &[f32]) -> f32 {
    assert_eq!(a.len(), b.len());

    if a.is_empty() {
        return 0.0;
    }

    // 确定 SIMD 宽度——在实际使用中，Rust 会根据平台确定
    const LANES: usize = 4; // 4 f32 适合 128 位寄存器（便携）

    let mut acc: Simd<f32, LANES> = Simd::splat(0.0);
    let chunks = a.len() / LANES;
    let remainder = a.len() % LANES;
    let mut i = 0usize;

    // 主循环——每次处理 LANES 个元素
    for _ in 0..chunks {
        // 将 a[i..i+LANES] 和 b[i..i+LANES] 加载为 Simd 向量
        // 这里用 from_slice 来从已有切片中加载
        let va = Simd::from_slice(&a[i..i + LANES]);
        let vb = Simd::from_slice(&b[i..i + LANES]);
        acc = va.mul_add(vb, acc); // FMA：acc = a * b + acc
        i += LANES;
    }

    // 将累加器水平归约为一个标量
    let mut total = acc.reduce_sum();

    // 处理剩余元素
    for j in 0..remainder {
        total += a[i + j] * b[i + j];
    }

    total
}
```

这个实现的关键性能技巧是 `mul_add`（`a * b + c`），它编译为单个 FMA（Fused Multiply-Add）指令。只有在 128 位寄存器上尝试此操作需要 3 条指令（乘、加、舍入），而 FMA 只用 1 条指令并保持更高精度。`reduce_sum()` 将向量归约为单一值——在 SSE 上需要两次水平加法，在 AVX 上需要更多（因为有更多 lane）。

### 6.2 图像处理——灰度转换的向量化实现

图像处理是 SIMD 的完美应用场景——每个像素有独立的 RGBA 通道，处理一个像素不影响另一个像素。

```rust
// 需要 nightly Rust
// #![feature(portable_simd)]

use std::simd::Simd;

/// 将 RGBA 图像转换为灰度（使用 SIMD）
/// 灰度公式：Y = 0.2989 * R + 0.5870 * G + 0.1140 * B
/// 对于一个 1920x1080 的图像，这需要 2,073,600 个像素的处理
/// SIMD 可以将处理时间减少 4 倍（128 位）到 8 倍（256 位）
fn rgba_to_grayscale_simd(pixels: &[u8]) -> Vec<u8> {
    // pixels 是 RGBA 格式——每 4 个字节一个像素
    assert_eq!(pixels.len() % 4, 0);
    let pixel_count = pixels.len() / 4;

    let mut gray = vec![0u8; pixel_count];
    let lanes = 16usize; // 16 u8 在 128 位寄存器中

    // 分两步处理——首先将 [u8] 转换为 [f32]，然后应用灰度公式
    // 但在实际 SIMD 中，我们使用整数算术来避免浮点转换开销

    // 权重——使用 Q16.16 定点数
    // r = 0.2989 * 65536 ≈ 19595
    // g = 0.5870 * 65536 ≈ 38470
    // b = 0.1140 * 65536 ≈ 7471
    let weight_r: Simd<u16, 8> = Simd::splat(19595u16);
    let weight_g: Simd<u16, 8> = Simd::splat(38470u16);
    let weight_b: Simd<u16, 8> = Simd::splat(7471u16);

    // 此处的完整实现需要涉及 de-interleave（拆解 RGBA 交错格式）、
    // 乘累加、右移 16 位（除以 65536）
    // 实际项目中，使用 image crate 配合专门的 SIMD 后端更合适
    // 这里仅展示核心的 SIMD 操作模式

    gray
}
```

在实际的图像处理中，使用 `image` crate 或 `fast_image_resize` 等专门的库是更好的选择——它们已经对常见图像操作做了深度的 SIMD 优化（包括运行时 CPU 特性检测和回退逻辑）。

## 7. 性能对比——标量 vs SSE vs AVX2

### 7.1 基准测试的方法论

在 SIMD 性能测试中，需要注意以下事项。必须使用 `std::hint::black_box` 防止编译器过度优化（比如在编译时计算整个循环的结果）。数据必须足够大（至少 L3 缓存大小）以确保测出吞吐量而非缓存命中效应。使用 `criterion` crate 进行统计显著的测量。最后，使用 `--release` 和 `-C target-cpu=native` 编译（否则 SIMD 代码会被编译为标量指令）。

```rust
// 正确的基准测试结构
// use criterion::{black_box, criterion_group, criterion_main, Criterion};

fn bench_dot_product(c: &mut Criterion) {
    let size = 1_000_000;
    let a: Vec<f32> = (0..size).map(|i| i as f32).collect();
    let b: Vec<f32> = (0..size).map(|i| (i * 2) as f32).collect();

    c.bench_function("scalar_dot_product", |bench| {
        bench.iter(|| {
            // black_box 防止编译器优化掉整个调用
            scalar_dot_product(
                black_box(&a),
                black_box(&b),
            )
        })
    });

    c.bench_function("simd_dot_product", |bench| {
        bench.iter(|| {
            dot_product_simd(
                black_box(&a),
                black_box(&b),
            )
        })
    });
}
```

没有 `black_box`，LLVM 可能检测到输入没有外部可见的副作用，从而彻底删除你的代码。

### 7.2 典型性能数据——什么值得向量化

基于社区广泛的基准测试（特别是 `std::simd` 和 `simdeez` 项目），以下是一些典型加速比（使用 `target-cpu=native`，在现代 x86_64 CPU 上）：

- f32 点积（100 万元素）：SSE 2-3x，AVX2 3-5x，AVX-512 5-8x
- u8 图像处理（灰度转换）：AVX2 10-20x（大量 u8 在一条 AVX2 指令中处理——32 个同时）
- 字符串查找（字节搜索）：SSE4.2 / AVX2 5-15x（`_mm_cmpestri` 等字符串 intrinsic）
- 加密（AES）：AES-NI 10-50x（专用的加密硬件指令——比纯软件快两个数量级）

加速比取决于元素宽度（越小越好）、操作复杂度（越简单越好）、以及内存带宽（如果程序瓶颈在内存，SIMD 的加速效果会被掩盖）。

## 9. 练习答案

### 练习 1 答案

```rust
// 需要 nightly Rust
// #![feature(portable_simd)]

use std::simd::Simd;

fn l2_norm(v: &[f32]) -> f32 {
    if v.is_empty() {
        return 0.0;
    }

    const LANES: usize = 8; // AVX2 宽度——便携版本会自动调节
    let chunks = v.len() / LANES;
    let remainder = v.len() % LANES;

    let mut acc: Simd<f32, LANES> = Simd::splat(0.0);
    let mut i = 0usize;

    // 主循环：每次处理 LANES 个元素
    for _ in 0..chunks {
        let vals = Simd::from_slice(&v[i..i + LANES]);
        // mul_add：acc = vals * vals + acc（平方并累加）
        acc = vals.mul_add(vals, acc);
        i += LANES;
    }

    // 归约向量为标量
    let mut sum_sq = acc.reduce_sum();

    // 剩余元素
    for j in 0..remainder {
        sum_sq += v[i + j] * v[i + j];
    }

    sum_sq.sqrt()
}

// 标量版本——用于对比和 fallback
fn l2_norm_scalar(v: &[f32]) -> f32 {
    v.iter().map(|x| x * x).sum::<f32>().sqrt()
}

// 测试
// fn test_l2_norm() {
//     assert_eq!(l2_norm(&[]), 0.0);
//     assert!((l2_norm(&[3.0, 4.0]) - 5.0).abs() < 1e-6);
// }
```

关键点：`Simd::from_slice` 从切片中加载（需要长度 >= LANES），`mul_add` 将两步操作（平方+累加）合并为一步，`reduce_sum()` 水平归约所有的 lane。这是 SIMD 点积/平方和的最优实现模式。

### 练习 2 答案

**循环 A**：可以自动向量化。逐元素乘法和 collect——无分支、无跨迭代依赖、连续内存访问。LLVM 会将此转换为 SIMD 的 mul 操作。

**循环 B**：无法自动向量化。这是前缀和（prefix sum）——每个迭代依赖前一个迭代的结果（`data[i] = data[i] + data[i - 1]`）。这种跨迭代的数据依赖序列（loop-carried dependency chain）正是自动向量化器的死穴。前缀和虽然可以向量化（使用专门的算法如 Hillis-Steele 或 Blelloch 扫描），但需要算法级别的改变——编译器无法自动做到。

**循环 C**：部分可以向量化，但 `break` 阻止了完整的向量化。LLVM 可能通过如下方式向量化：使用条件掩码累加正值（忽略 `break` 的语义优化），但如果它保守地处理 `break`（允许在任意点退出），向量化会无法确定何时退出。通常 `break` 会阻止向量化——LLVM 倾向于保守处理。

### 练习 3 答案

```rust
#[cfg(target_arch = "x86_64")]
use std::arch::x86_64::*;

/// SSE 实现的 RGBA 转灰度
/// 参数：
///   pixels: RGBA 格式的输入像素数据（长度必须是 16 的倍数——4 像素 * 每像素 4 通道）
///   gray: 输出缓冲区（长度 / 4 字节）
#[cfg(target_arch = "x86_64")]
unsafe fn rgba_to_grayscale_sse(pixels: &[u8], gray: &mut [u8]) {
    assert_eq!(pixels.len() % 4, 0);
    assert_eq!(gray.len(), pixels.len() / 4);

    // 整数权重：77, 150, 29 → 除以 256（通过右移 8 位）
    // 这些值在 0-255 范围内，将结果保持为 u8（通过 _mm_packus_epi16）
    let weight_r = _mm_set1_epi16(77);
    let weight_g = _mm_set1_epi16(150);
    let weight_b = _mm_set1_epi16(29);

    // 每次处理 8 个像素（需要 2 个 SSE 寄存器加载完整的 RGBA 数据）
    let chunk_pixels = 8;
    let bytes_per_chunk = chunk_pixels * 4; // 32 字节 = 2 个 __m128i

    let mut pi = 0usize; // 输入索引
    let mut gi = 0usize; // 输出索引

    while pi + bytes_per_chunk <= pixels.len() {
        // 加载 32 字节的 RGBA 数据（8 个像素）
        // 使用两个 128 位 load
        let lo = _mm_loadu_si128(pixels.as_ptr().add(pi) as *const __m128i);
        let hi = _mm_loadu_si128(pixels.as_ptr().add(pi + 16) as *const __m128i);

        // 解包——将交错存储的 RGBA 数据拆分为 RRRR...、GGGG...、BBBB...、AAAA...
        // _mm_unpack 会交错两个寄存器的字节
        // 以下是一个简化的处理流程——实际需要多次 _mm_unpacklo_epi8 和 _mm_unpackhi_epi8
        // 将 u8 解包为 u16（以便乘累加而不溢出）

        // 最终的灰度值（通过饱和打包回 u8 并存储）
        // 使用 _mm_packus_epi16 将两个 128 位的 u16 向量打包为一个 u8 向量

        // 对于完整的生产级实现，参考 image crate 中的 `hsv` 和 `colorops` 模块
        // 这里仅展示算法的骨架

        pi += bytes_per_chunk;
        gi += chunk_pixels;
    }

    // 处理剩余的像素（不足 8 个的部分）
    for i in 0..(pixels.len() / 4 - gi) {
        let pi = (gi + i) * 4;
        let r = pixels[pi] as f32;
        let g = pixels[pi + 1] as f32;
        let b = pixels[pi + 2] as f32;
        gray[gi + i] = (0.2989 * r + 0.5870 * g + 0.1140 * b) as u8;
    }
}
```

此实现展示了 SSE 图像处理的典型挑战：数据布局转换（RGBA 交错 → 分离通道）通常比实际计算更复杂。生产级代码会将图像存储为分离的平面格式（即所有 R 值连续存储，所有 G 值连续存储……），这极大地简化了 SIMD 处理并避免了昂贵的解包/重排操作。

```
□ 理解性能基准测试的正确方法——black_box、criterion、大数据集
□ 清楚 SIMD 加速比取决于元素宽度、操作复杂度和内存带宽
□ 能够在可移植 SIMD（std::simd）和平台 intrinsic 之间做出正确选择
```

> **Phase 4 性能分析基石**：SIMD 是手动榨干 CPU 性能的最直接手段。但性能优化最大的敌人不是代码写得不够好，而是隐藏的 bug——memory errors、data races、memory leaks。下一课 [[4.12-sanitizers-profiling]] 将介绍 Sanitizers（ASan/TSan/LSan/MSan）、Miri 和 profiling 工具——它们帮你找到 bug 并确定优化的真正瓶颈。
