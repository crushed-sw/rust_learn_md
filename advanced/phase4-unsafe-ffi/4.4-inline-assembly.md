# 4.4 内联汇编 (asm!) —— 当 Safe Rust 不够快时，你需要的最后武器

> 内联汇编是 Rust 与硬件之间的终极桥梁。`asm!` 宏不是一根救命稻草——它是一把手术刀。你告诉编译器"我要在这一点上完全控制 CPU"，编译器退后一步，交出寄存器分配和指令选择的主权。但这份权力带着沉重的责任：你必须告诉编译器你修改了什么（clobbers），你的输入输出去向何处（constraints），以及这段汇编代码是否会影响内存和栈（options）。漏掉任何一个信息，都可能让编译器的优化产生错误的假设——在汇编层面，错误不是 panic，是静默的数据损坏。

## 1. asm! 宏 —— 稳定版的内联汇编入口

### 1.1 历史与稳定性

Rust 的内联汇编经历了漫长的演进。在 1.59 版本之前，内联汇编只能通过 `llvm_asm!` 宏使用，且被标记为不稳定的（unstable）。`llvm_asm!` 是 LLVM 内联汇编语法的直接映射，语法晦涩，且缺乏对寄存器操作的类型安全约束。

从 Rust 1.59 开始，全新的 `asm!` 宏作为稳定功能进入标准库。新宏提供了更清晰的语法、更强的类型检查、更好的跨平台支持，以及与 Rust 的所有权和借用系统更一致的语义。`llvm_asm!` 在 2023 年被正式移除——所有的内联汇编代码都应当迁移到 `asm!`。

### 1.2 基本语法结构

`asm!` 宏的核心语法分为四个部分：汇编指令字符串、输出操作数、输入操作数、以及选项配置。下面是一个最简单的示例——执行一次空操作（nop）：

```rust
use std::arch::asm;

// 最基本的 asm! 调用：执行一条 nop 指令
fn do_nothing() {
    unsafe {
        asm!("nop");
    }
}
```

`asm!` 的第一个参数是汇编模板字符串——它是一个格式字符串，可以包含 `{}` 占位符来引用操作数。占位符中的数字对应操作数列表中从 0 开始的索引，这类似于 `format!` 宏的工作方式。但请注意，`asm!` 的参数顺序是：先写所有输出操作数，再写所有输入操作数——这意味着索引 0 属于第一个输出操作数（如果有的话）。

### 1.3 一个真正的例子：读取 CPU 时间戳计数器

x86_64 架构上的 `rdtsc` 指令将 CPU 时间戳计数器的 64 位值分别存储在 EDX（高 32 位）和 EAX（低 32 位）中。下面展示如何使用 `asm!` 读取它：

```rust
use std::arch::asm;

/// 使用 rdtsc 指令读取时间戳计数器
/// 返回一个 64 位值，低 32 位来自 EAX，高 32 位来自 EDX
fn read_timestamp_counter() -> u64 {
    let low: u32;
    let high: u32;
    unsafe {
        asm!(
            "rdtsc",              // x86_64: 读取 TSC 到 edx:eax
            out("eax") low,       // eax → low
            out("edx") high,      // edx → high
        );
    }
    // 拼接高 32 位和低 32 位为一个 u64
    ((high as u64) << 32) | (low as u64)
}
```

这段代码展示了 `asm!` 的几个关键元素：`out("eax") low` 声明 `eax` 寄存器值将被写入 Rust 变量 `low`；`out("edx") high` 同理。`asm!` 宏会自动处理值的传递——你不需要手动操作寄存器，只需要声明"哪个寄存器对应哪个 Rust 变量"即可。

## 2. x86_64 汇编基础 —— 在 Rust 中使用 x86_64 汇编

### 2.1 寄存器速览

x86_64 有 16 个通用 64 位寄存器。在 `asm!` 宏中，你可以按三种大小引用每个寄存器：

下表列出了最常用的通用寄存器及其不同位宽的命名方式：

| 64 位 | 32 位 | 16 位 | 8 位 | 典型用途 |
|-------|-------|-------|------|---------|
| rax | eax | ax | al | 返回值、累加器 |
| rbx | ebx | bx | bl | 基址寄存器（被调用保存） |
| rcx | ecx | cx | cl | 计数器、第四个参数 |
| rdx | edx | dx | dl | 第三个参数、I/O 端口 |
| rsi | esi | si | sil | 第二个参数（源索引） |
| rdi | edi | di | dil | 第一个参数（目标索引） |
| rsp | esp | sp | spl | 栈指针 |
| rbp | ebp | bp | bpl | 帧指针（被调用保存） |
| r8-r15 | r8d-r15d | r8w-r15w | r8b-r15b | 额外通用寄存器 |

在 `asm!` 中使用寄存器时，你需要确保大小匹配：操作 64 位整数用 `rax`，32 位用 `eax`，16 位用 `ax`，8 位用 `al`。

### 2.2 Intel 语法与 AT&T 语法

Rust 的 `asm!` 宏默认使用 Intel 语法——这是 Windows 世界的标准，也是大多数现代 x86 文档偏好的语法。Intel 语法的特点是 `指令 目的操作数, 源操作数`。

```rust
use std::arch::asm;

/// Intel 语法：mov 目的, 源
fn intel_syntax_example() -> u64 {
    let result: u64;
    unsafe {
        asm!(
            "mov {0}, 42",  // 把立即数 42 放入 result 对应的寄存器
            "add {0}, 8",   // 加上 8 → {0} 现在是 50
            out(reg) result,
        );
    }
    result
}

assert_eq!(intel_syntax_example(), 50);
```

如果你需要 AT&T 语法（某些 Linux 工具链的偏好），可以使用 `options(att_syntax)`。AT&T 语法的特点是 `指令 源操作数, 目的操作数`，且寄存器前有 `%` 前缀，立即数前有 `$`：

```rust
use std::arch::asm;

/// AT&T 语法：mov 源, 目的
fn att_syntax_example() -> u64 {
    let result: u64;
    unsafe {
        asm!(
            "movq $$42, %[r]",
            "addq $$8, %[r]",
            r = out(reg) result,
            options(att_syntax),
        );
    }
    result
}
```

注意 AT&T 语法中操作数用 `%[name]` 而不是 `{n}` 的形式引用命名操作数。两种引用方式在两种语法中都可以使用——`{0}` 或 `%[name]`——选择哪个取决于可读性和个人偏好。

### 2.3 常见 x86_64 指令速查

在 Rust 内联汇编中频繁出现的指令包括：

数据传送指令是内联汇编中最基础的指令族。以下是编写内联汇编时最常用的几个：

```
mov  dst, src   —— 将 src 的值复制到 dst
lea  dst, [addr] —— 加载有效地址（不访问内存，仅做地址计算）
xchg dst, src   —— 交换两个操作数的值
push src        —— 将值压入栈顶
pop  dst        —— 从栈顶弹出值
```

算术和逻辑指令则是实际计算的执行者：

```
add  dst, src   —— dst = dst + src
sub  dst, src   —— dst = dst - src
inc  dst        —— dst = dst + 1
dec  dst        —— dst = dst - 1
xor  dst, src   —— 按位异或（常见用法：xor reg, reg 清空寄存器）
and  dst, src   —— 按位与
or   dst, src   —— 按位或
shl  dst, count —— 逻辑左移
shr  dst, count —— 逻辑右移
```

条件测试和跳转指令用于实现分支逻辑：

```
cmp  a, b       —— 比较 a 和 b（设置状态标志位）
test a, b       —— 测试 a 和 b 的按位与（设置状态标志位）
je   label      —— 如果相等则跳转
jne  label      —— 如果不相等则跳转
jg   label      —— 如果大于（有符号）则跳转
jl   label      —— 如果小于（有符号）则跳转
```

## 3. ARM64 基础 —— 跨平台视角

### 3.1 ARM64 寄存器模型

当你需要编写同时支持 x86_64 和 ARM64（aarch64）的内联汇编时，理解两种架构的差异至关重要。ARM64 有 31 个 64 位通用寄存器：`x0-x30`。其中 `x0-x7` 用于传递函数参数，`x0` 也用于返回值。

```rust
#[cfg(target_arch = "aarch64")]
use std::arch::asm;

/// ARM64 版本的读取 CPU 周期计数器
/// ARM64 使用不同指令集，但 asm! 宏的约束语法相同
#[cfg(target_arch = "aarch64")]
fn read_cycle_counter() -> u64 {
    let result: u64;
    unsafe {
        // ARM64 使用 MRS 指令读取系统寄存器
        // PMCCNTR_EL0 是性能监视器周期计数寄存器
        asm!(
            "mrs {}, pmccntr_el0",
            out(reg) result,
        );
    }
    result
}
```

注意 `asm!` 宏的条件编译——这是编写跨平台内联汇编的标准做法。你应当将不同架构的汇编代码放在 `#[cfg(target_arch = "...")]` 属性下。

### 3.2 x86_64 与 ARM64 的关键差异

了解下面这些架构差异可以帮助你编写更安全的跨平台汇编代码：

对于内存访问模式，x86_64 允许大多数指令直接操作内存操作数（例如 `add rax, [memory_location]`），而 ARM64 严格执行加载-存储架构——你必须先用 `ldr` 从内存加载到寄存器，运算，再用 `str` 写回。这意味着 ARM64 的内联汇编通常需要更多的指令来完成同样的逻辑。

```rust
// ARM64 加载-存储风格的代码在 asm! 中的表现
// x86_64 上的一条指令"add rax, [ptr]"在 ARM64 上需要两条：
// "ldr x1, [x0]"  加载内存值到 x1
// "add x0, x0, x1"  执行加法
```

在立即数范围方面，ARM64 的立即数编码更加受限。不是所有的 32 位或 64 位值都可以直接作为指令的立即数——这被称为"立即数编码约束"。ARM64 使用一种位模式编码方案，只有特定的位模式可以编码为立即数。如果立即数不符合编码规则，编译器会自动生成常量加载序列。

在原子操作和内存屏障方面，ARM64 的内存模型比 x86_64 更弱（更接近 ARM 传统的弱一致性模型），因此在内联汇编中执行原子操作时，你可能需要显式添加 `dmb`（数据内存屏障）或 `dsb`（数据同步屏障）指令。

## 4. 约束 (Constraints) —— 告诉编译器你的数据流向

### 4.1 约束类型全景

`asm!` 宏的约束系统定义了汇编代码和 Rust 变量之间的数据流动关系。五种约束类型涵盖了所有可能的交互模式：

- `in(reg)` —— 输入约束：值从 Rust 变量流向寄存器，汇编代码可以读取它
- `out(reg)` —— 输出约束：值从寄存器流向 Rust 变量，汇编代码写入它
- `inout(reg)` —— 输入输出约束：值先从 Rust 变量流向寄存器，汇编代码修改后流回 Rust 变量
- `lateout(reg)` —— 延迟输出约束：类似于 out，但保证在所有输入被读取之后才写入
- `inlateout(reg)` —— 延迟输入输出约束：inout 的延迟版本

### 4.2 out 约束详解

`out(reg)` 告诉编译器"这个寄存器将被汇编代码写入，你不需要为其提供初始值"。

```rust
use std::arch::asm;

/// 使用 BSR 指令计算最高置位的位置
/// BSR (Bit Scan Reverse): 从最高位向最低位扫描，返回第一个 1 的位置
fn highest_set_bit(value: u64) -> Option<u32> {
    let result: u32;
    // ZF (零标志) 在 BSR 指令后被设置——如果输入为 0，ZF = 1
    let mut zero: u8;
    
    unsafe {
        asm!(
            "bsr {result:r}, {value:r}",  // 扫描 value 的最高置位，结果放入 result 寄存器
            "sete {zero:r}",               // 如果 ZF=1 (value 为 0)，设置 zero = 1
            value = in(reg) value,
            result = out(reg) result,
            zero = out(reg_byte) zero,
        );
    }
    
    if zero != 0 {
        None // 输入为 0，没有置位
    } else {
        Some(result)
    }
}
```

这个例子中的 `result` 使用 `out(reg)` 约束——汇编代码写入它，但不需要初始值。注意 `reg_byte` 是一个寄存器类，表示我们只需要寄存器的低 8 位（匹配 `sete` 指令的 8 位目标）。

### 4.3 inout 约束——原地修改

`inout(reg)` 是"输入并输出"约束——汇编代码读取并修改寄存器的值。它特别适用于需要累积计算结果或修改输入参数的场景。

```rust
use std::arch::asm;

/// 原子地对一个 u64 值加 1
/// 使用 x86_64 的 "lock inc" 指令——保证多线程安全
fn atomic_increment(value: &mut u64) {
    unsafe {
        asm!(
            "lock inc qword ptr [{ptr}]",  // 锁定内存总线，原子地递增
            ptr = in(reg) value,
            options(nostack, preserves_flags),
        );
    }
}
```

这里使用 `in(reg)` 而不是 `inout(reg)`，因为我们传递的是指针——汇编指令直接修改内存，不需要寄存器输出。指针本身只被读取（`in`），不会被汇编代码修改。

### 4.4 lateout —— 何时需要它

`lateout` 和 `out` 的关键区别在于写入时机。`lateout` 保证在**所有输入寄存器被读取之后**才写入，这对于那些输入和输出需要使用**同一个寄存器**的场景至关重要。

```rust
use std::arch::asm;

/// 使用 XADD 指令进行原子交换并加法
/// XADD 将 src 加到 dst 上，并将 dst 的原始值放入 src 中
/// 问题：如果我们把同一个寄存器用于输入和 lateout，编译器可以复用寄存器
fn xadd_example(operand: &mut u64, x: u64) -> u64 {
    let old: u64;
    unsafe {
        asm!(
            "lock xadd [{operand}], {x}",
            "mov {old}, {x}",             // 将 x 中的旧值复制到 old
            operand = in(reg) operand,
            x = inlateout(reg) x => old,  // x 输入到汇编，结果输出到 old
            // 使用 inlateout 确保 x 寄存器在所有读取完成后才被写回
        );
    }
    old
}
```

`inlateout` 确保了即使编译器将 `x` 和 `old` 分配到同一个物理寄存器，也不会发生覆盖问题——汇编代码先读取 `x`（输入），然后写入 `old`（输出），不会出现提前覆盖。

### 4.5 寄存器类

除了通用的 `reg` 类，`asm!` 支持多种特定的寄存器类，允许你对寄存器选择进行精确控制：

`reg` 是通用寄存器类，编译器在 r0-r15（ARM64）或 rax/rbx/rcx/rdx/rsi/rdi/r8-r15（x86_64）中选择。这是最常用的选择，让编译器自由分配能提高寄存器分配效率。

`reg_byte` 限制为 8 位寄存器的低字节部分（x86 上的 al/bl/cl/dl, sil, dil, r8b-r15b）。当你用 `sete` 等 8 位指令时，必须使用这个类。

`reg_abcd` 是 x86 上传统命名的 32 位寄存器（eax/ebx/ecx/edx）。某些指令（如 `mul`）隐式使用 eax 和 edx，在这种情况下你还需要使用 `in("eax")` 这样的显式寄存器名。

## 5. Clobber 声明 —— 告诉编译器你破坏了什么

### 5.1 为什么需要 Clobber

编译器在寄存器分配和指令调度方面做出了很多假设。当你编写内联汇编时，你的汇编代码可能在编译器不知情的情况下修改了某些寄存器的内容。如果不声明这些被修改的寄存器（clobbers），编译器会在错误的假设上继续优化——结果可能是不可预测的数据损坏。

```rust
use std::arch::asm;

/// 一个 64 位整数的乘法——获取 128 位结果
/// x86_64 的 mul 指令隐式修改 rdx:rax
fn multiply_with_full_result(a: u64, b: u64) -> u128 {
    let hi: u64;
    let lo: u64;
    unsafe {
        asm!(
            "mul {b}",                    // rax * b → rdx:rax
            // mul 指令：
            //   - 读取 rax（由 a 的 in 约束填充）
            //   - 读取 b（由 b 的 in 约束填充）
            //   - 写入 rdx:rax（128 位结果）
            //   - 会破坏（clobber）rdx
            in("rax") a,                   // 显式指定 rax 作为输入
            b = in(reg) b,                 // b 放入任意通用寄存器
            lateout("rax") lo,            // rax 的最终值就是乘积的低 64 位
            out("rdx") hi,                // rdx 接收乘积的高 64 位
        );
    }
    // 拼接高低部分为 u128
    (lo as u128) | ((hi as u128) << 64)
}
```

在这个例子中，`out("rdx") hi` 隐式地将 `rdx` 声明为被 clobber 的——编译器知道这个寄存器的值在 asm! 之后发生了变化。任何显式列在输出中的寄存器都被视为 clobber。但**隐式**被修改的寄存器——如 `rdx` 被 `mul` 隐式修改——如果没有被声明为输出或 clobber，就会导致错误。

### 5.2 显式 Clobber 声明

`clobber_abi` 是一种声明一组寄存器的便捷方式。当你调用遵循特定调用约定的函数时，你可以告诉编译器"所有该调用约定中非保存寄存器都可能被修改"：

```rust
use std::arch::asm;

/// 使用 cpuid 指令获取 CPU 信息
/// cpuid 指令使用 eax（输入/输出 leaf 选择）和 ebx/ecx/edx（输出）
fn get_cpu_vendor() -> [u8; 12] {
    let mut vendor = [0u8; 12];
    let ebx: u32;
    let edx: u32;
    let ecx: u32;
    
    unsafe {
        asm!(
            "xor eax, eax",       // EAX = 0: 查询厂商 ID 字符串
            "cpuid",               // 执行 CPUID
            // 结果：
            // EBX 包含 "Genu" ("GenuineIntel" 的前 4 字节)
            // EDX 包含 "ineI"
            // ECX 包含 "ntel"
            inout("eax") 0u32 => _,  // 丢弃 eax 的返回值
            out("ebx") ebx,
            out("edx") edx,
            out("ecx") ecx,
            options(nostack, preserves_flags),  // 不修改栈和标志位
        );
    }
    // 将三个 u32 按小端序写入 u8 数组
    vendor[0..4].copy_from_slice(&ebx.to_le_bytes());
    vendor[4..8].copy_from_slice(&edx.to_le_bytes());
    vendor[8..12].copy_from_slice(&ecx.to_le_bytes());
    vendor
}
```

`options(nostack, preserves_flags)` 是 clobber 管理的另一种形式——选项告诉编译器"这段汇编代码不会修改栈指针（nostack），也不会修改 CPU 标志寄存器（preserves_flags）"。这使得编译器不需要在 asm! 前后保存和恢复栈状态和标志位。

### 5.3 内存 Clobber

如果你的内联汇编代码写入了任意内存位置（而不仅仅是那些通过指针操作数指定的位置），你需要通知编译器。虽然当前稳定版的 `asm!` 不直接支持 `"memory"` clobber，但你可以通过 `nomem` 选项的反面（即不指定 `nomem`）来表明代码**可能**访问内存——默认行为就是如此。

如果你指定了 `options(nomem)`，编译器就会假设汇编代码不读取或写入任何内存（除了通过显式操作数指定的位置）。这是更强的优化假设，但如果你违反了它，后果严重。

## 6. Options —— 给编译器的优化提示

### 6.1 可用 Options 全景

`asm!` 的 `options()` 接受以下标志来控制编译器对汇编代码的假设：

- `pure` —— 汇编代码没有副作用（除了通过输出操作数）。编译器可以消除纯汇编代码的死调用。
- `nomem` —— 汇编代码不读取或写入任何内存（除了通过显式操作数指定的位置）。这允许编译器更激进的重排。
- `readonly` —— 汇编代码只读取内存但不写入。比 nomem 稍弱，但仍有助于优化。
- `preserves_flags` —— 汇编代码不修改 CPU 状态标志（EFLAGS）。省略对标志位的保存和恢复。
- `nostack` —— 汇编代码不修改栈指针，也不依赖栈的特定对齐。允许编译器跳过栈对齐。
- `att_syntax` —— 使用 AT&T 语法而非默认的 Intel 语法。

### 6.2 nomem 的威力与风险

`nomem` 是最强大的优化提示之一，但也是风险最高的。下面用对比示例展示它的影响：

```rust
use std::arch::asm;

/// 一个纯计算的内联汇编——非常适合 nomem
/// 统计 u64 中值为 1 的位的数量（popcount）
/// x86_64 有专用的 popcnt 指令
fn count_set_bits(value: u64) -> u32 {
    let result: u32;
    unsafe {
        asm!(
            "popcnt {result:r}, {value:r}",
            value = in(reg) value,
            result = out(reg) result,
            options(nomem, pure, preserves_flags),
            // nomem:  不访问内存
            // pure:   无副作用（输出仅通过 result 操作数）
            // preserves_flags:  popcnt 确实修改标志位...
            //   ...但这里我们不需要标志位，且声明 preserves_flags
            //   会减少编译器的保存/恢复开销
        );
    }
    result
}
```

注意 `preserves_flags` 在这里实际上是**不准确**的（`popcnt` 确实修改 ZF、CF 等标志位），但因为我们在 asm! 之后不读取标志位，这个选项是安全的——它只是告诉编译器"你之后不需要依赖 asm! 之前的标志位状态"。

### 6.3 正确组合 Options

Options 的组合应当遵循你的汇编代码的精确语义。下表列出了常见场景和推荐选项：

对于纯算术计算（如 `popcnt`、`bsr`、位运算），推荐使用 `options(nomem, pure)`——这些指令不访问内存，没有副作用。

对于读取但不写入内存的汇编（如从内存指针读取数据），推荐只使用 `options(readonly)`——告诉编译器内存被读取但未被修改，允许编译器把读取操作移到 asm! 之前（如果逻辑允许）。

对于通过指针写入内存的汇编（如 `lock inc qword ptr [ptr]`），不要使用 nomem——因为确实访问了内存。默认行为已经足够，编译器会保守地处理内存访问。

```rust
use std::arch::asm;

/// 只读内存的汇编——适合 readonly 选项
/// 校验一个字节与内存中的值是否一致
fn check_byte_equal(ptr: *const u8, expected: u8) -> bool {
    let result: u8;
    unsafe {
        asm!(
            "cmp byte ptr [{ptr}], {expected}",
            "sete {result}",
            ptr = in(reg) ptr,
            expected = in(reg_byte) expected,
            result = out(reg_byte) result,
            options(readonly, preserves_flags),
            // readonly: 这段代码从内存读取但不写入
            // preserves_flags: sete 的结果通过 result 操作数传出，
            //   我们不依赖 cmp 设置的标志位
        );
    }
    result != 0
}
```

## 7. 实战场景

### 7.1 Linux 系统调用——用汇编执行 write

在 x86_64 Linux 上，系统调用通过 `syscall` 指令执行，参数通过寄存器传递。以下代码展示了如何使用内联汇编实现 `write` 系统调用——这是理解系统调用机制的最佳起点：

```rust
use std::arch::asm;

/// 使用内联汇编直接执行 Linux 的 write 系统调用
/// 这绕过了 libc 和标准库，直接操作内核接口
/// 
/// Linux x86_64 系统调用约定：
///   - 系统调用号 → rax
///   - 参数 1 → rdi
///   - 参数 2 → rsi
///   - 参数 3 → rdx
///   - 返回值 → rax
///   - 使用 syscall 指令触发
fn raw_write_to_stdout(message: &[u8]) -> isize {
    let fd: u64 = 1; // stdout 文件描述符
    let buf_ptr = message.as_ptr() as u64;
    let count = message.len() as u64;
    let ret: u64;

    unsafe {
        // write 系统调用在 x86_64 Linux 上的编号是 1
        asm!(
            "syscall",
            inout("rax") 1u64 => ret,      // 系统调用号 = 1 (write), 返回值在 rax
            in("rdi") fd,                   // 第 1 个参数: 文件描述符
            in("rsi") buf_ptr,              // 第 2 个参数: 缓冲区指针
            in("rdx") count,                // 第 3 个参数: 字节数
            // syscall 指令会 clobber rcx 和 r11
            // 但因为我们使用 out("rcx") _ 和 out("r11") _ 已经隐式告诉编译器了
            out("rcx") _,                   // syscall 使用 rcx 保存返回地址
            out("r11") _,                   // syscall 使用 r11 保存 RFLAGS
            options(nostack, preserves_flags),
            // nostack: 不修改栈 —— syscall 在特权级切换时使用内核栈，不碰用户栈
            // preserves_flags: syscall 前后标志位由内核控制，从用户态看来是一致的
        );
    }

    ret as isize
}
```

这个例子是理解系统调用机制的优秀教材。注意我们将系统调用号 `1`（对应 `__NR_write`）放入 `rax`，将文件描述符、缓冲区指针和长度分别放入 `rdi`、`rsi` 和 `rdx`。`syscall` 指令触发了从用户态到内核态的切换，内核处理请求后将返回值放回 `rax`。

在实际项目中，你几乎不会直接写这样的代码——标准库的 `std::io::Write` 已经完美封装了这些操作。但这个例子说明了为什么 Rust 可以用于裸机（no_std）和操作系统内核开发：`asm!` 提供了直接与硬件和内核对话的能力。

### 7.2 SIMD 操作——手动使用 SSE/AVX 指令

现代编译器对 SIMD 自动向量化做得相当好，但有时你需要手动控制 SIMD 指令以获得最佳性能。以下示例展示如何使用 `asm!` 调用 SSE 指令进行 128 位向量加法：

```rust
use std::arch::asm;

/// 使用 SSE 指令同时处理 4 个 f32 的加法
/// 这是一个教学示例——实际项目中应该使用 std::arch::x86_64 模块中的包装函数
fn sse_add_four_floats(a: [f32; 4], b: [f32; 4]) -> [f32; 4] {
    // 将 f32 数组转换为 __m128（128 位 SSE 向量）
    let a_bytes = unsafe { std::mem::transmute::<[f32; 4], u128>(a) };
    let b_bytes = unsafe { std::mem::transmute::<[f32; 4], u128>(b) };
    let result_bytes: u128;
    
    // 注意：实际项目中应使用 std::arch::x86_64::_mm_add_ps
    // 这里用 asm! 是为了展示内联汇编的用法
    unsafe {
        asm!(
            // movups: 将 128 位数据从内存移到 xmm 寄存器（不对齐版本）
            "movups xmm0, xmmword ptr [{a_ptr}]",   // 加载 a 到 xmm0
            "movups xmm1, xmmword ptr [{b_ptr}]",   // 加载 b 到 xmm1
            "addps  xmm0, xmm1",                     // xmm0 = xmm0 + xmm1（4 个 f32 并行加法）
            "movups xmmword ptr [{result_ptr}], xmm0", // 结果存回内存
            a_ptr = in(reg) &a_bytes,
            b_ptr = in(reg) &b_bytes,
            result_ptr = in(reg) &result_bytes,
            // xmm0 和 xmm1 被修改——通过读写内存的方式，
            // 我们没有在约束中声明，但它们的内容确实变化了
            // 这在实际代码中是一个问题：如果编译器在 asm! 前后有数据在
            // xmm0/xmm1 中，这些数据会被静默覆盖
            // 解决方案：使用 out 约束或直接使用 std::arch 包装函数
            out("xmm0") _,  // 废弃 xmm0 的值
            out("xmm1") _,  // 废弃 xmm1 的值
            options(nostack, preserves_flags),
        );
    }
    
    unsafe { std::mem::transmute::<u128, [f32; 4]>(result_bytes) }
}
```

注意代码中指出的问题：我们显式使用了 xmm0 和 xmm1，但没有通过约束声明这些寄存器的使用。我们通过 `out("xmm0") _` 告诉编译器"这些寄存器的值发生了变化"，这样编译器就知道不要依赖 asm! 之前 xmm0/xmm1 中的值。在实际项目中，更安全和更推荐的方式是使用 `std::arch` 中的包装函数（如 `_mm_add_ps`），它们已经正确封装了这些约束。

### 7.3 裸机 / Bootloader 代码

在操作系统内核或 bootloader 开发中，`asm!` 是不可或缺的工具。以下代码展示了如何切换到保护模式或执行端口 I/O——这些操作是无法用 safe Rust 表达的：

```rust
use std::arch::asm;

/// 向 x86 I/O 端口写入一个字节
/// 在用户态中这通常被禁止（需要 IO 特权级），但在内核态中是标准操作
/// 
/// out 指令: 将 AL/AX/EAX 中的值输出到 DX 指定的 I/O 端口
#[cfg(target_arch = "x86_64")]
pub unsafe fn outb(port: u16, value: u8) {
    asm!(
        "out dx, al",
        in("dx") port,       // 端口号放入 dx
        in("al") value,      // 要输出的值放入 al
        options(nomem, preserves_flags),
    );
}

/// 从 x86 I/O 端口读取一个字节
#[cfg(target_arch = "x86_64")]
pub unsafe fn inb(port: u16) -> u8 {
    let value: u8;
    asm!(
        "in al, dx",
        in("dx") port,
        out("al") value,
        options(nomem, preserves_flags),
    );
    value
}

/// 禁用中断——清除 EFLAGS 中的 IF 位
/// cli 指令清除中断标志位
#[cfg(target_arch = "x86_64")]
pub unsafe fn disable_interrupts() {
    asm!("cli", options(nomem));
}

/// 启用中断——设置 EFLAGS 中的 IF 位
#[cfg(target_arch = "x86_64")]
pub unsafe fn enable_interrupts() {
    asm!("sti", options(nomem));
}
```

这些函数是操作系统内核开发的基本原语。注意它们被标记为 `#[cfg(target_arch = "x86_64")]`——不支持 x86_64 的平台上这些函数不会被编译。

## 8. asm! 与 LLVM 内联汇编的关系

### 8.1 编译流水线中的位置

Rust 的 `asm!` 宏最终被翻译为 LLVM IR 中的内联汇编表达式。理解这个翻译过程有助于调试和优化：

```
你的 Rust 代码: asm!("add {a}, {b}", a = in(reg) a, ...)
    ↓ 编译器前端（rustc）
LLVM IR: call void asm sideeffect "add $0, $1", "r,r"(i64 %a, i64 %b)
    ↓ LLVM 后端
目标机器码: add rax, rbx  （实际指令取决于寄存器分配结果）
```

LLVM 的内联汇编系统提供了比 `asm!` 更底层的控制——但 `asm!` 已经是稳定 Rust 的最小封装，你通常不需要直接关心 LLVM 层面。

### 8.2 阅读 LLVM IR 验证汇编语义

当你怀疑 `asm!` 的行为不符合预期时，可以通过阅读 LLVM IR 来确认编译器的理解是否正确。假设你写了一段调用 `cpuid` 的 asm! 代码，但结果总是错误的——你可以用 `cargo rustc -- --emit=llvm-ir` 生成 LLVM IR，检查寄存器约束是否和你的预期一致。

以下是一个概念性的 LLVM IR 对应的 asm 表达式示例：

```llvm
; LLVM IR 中的内联汇编表示
; 下面的 IR 对应 Rust 代码 asm!("add {a}, {b}", a = inout(reg) a, b = in(reg) b)

%1 = call i64 asm sideeffect alignstack inteldialect 
    "add $0, $1",                    ; 汇编模板字符串
    "=r,{imregister},r,~{dirflag},~{fpsr},~{flags}"  ; 约束字符串
    (i64 %a, i64 %b)                 ; 操作数
    ; 约束解释：
    ; =r      → 输出操作数在通用寄存器中
    ; r       → 输入操作数在通用寄存器中  
    ; ~{...}  → clobber 声明（被修改的寄存器/状态）
```

理解这个映射有助于你在遇到 LLVM 相关的 asm 错误时进行调试。大多数情况下你不需要关心这个层面，但了解它的存在是有价值的。

## 9. 常见陷阱

### 9.1 忘记声明 Clobbers

这是内联汇编中最常见的 bug——也是后果最严重的一类。编译器不知道你的汇编代码修改了哪些寄存器，除非你通过约束或 clobber 声明告知它。

考虑以下危险代码：

```rust
use std::arch::asm;

/// ❌ 严重错误：忘记声明 rcx 被修改
/// 这段代码使用了 "cpuid" 指令，它会修改 eax、ebx、ecx、edx，
/// 但我们只声明了 ebx 和 edx 为输出——eax 和 ecx 的变化被隐藏了
/// 编译器可能已经将重要的数据放在 eax 或 ecx 中，它们会被静默覆盖
fn dangerous_cpuid() -> u32 {
    let ebx: u32;
    unsafe {
        asm!(
            "mov eax, 1",
            "cpuid",
            out("ebx") ebx,
            // ❌ 缺少 out("eax") _ 和 out("ecx") _ 以及 out("edx") _
            // 编译器不知道 eax、ecx、edx 被 cpuid 修改了
            // 如果这些寄存器中有活跃的变量，会被静默覆盖
        );
    }
    ebx
}

/// ✅ 正确做法：将所有被隐式修改的寄存器声明为输出（即使你不关心它们的值）
fn safe_cpuid(leaf: u32) -> (u32, u32, u32, u32) {
    let eax: u32;
    let ebx: u32;
    let ecx: u32;
    let edx: u32;
    unsafe {
        asm!(
            "cpuid",
            inout("eax") leaf => eax,
            out("ebx") ebx,
            out("ecx") ecx,
            out("edx") edx,
            options(nostack, preserves_flags),
        );
    }
    (eax, ebx, ecx, edx)
}
```

### 9.2 操作数大小不匹配

寄存器类必须与你使用的汇编指令的操作数大小匹配。这是一个常见的错误源：

```rust
use std::arch::asm;

fn operand_size_mismatch() {
    let value: u32 = 42;
    let result: u64;
    unsafe {
        // ❌ 错误：popcnt 接收 64 位操作数，但我们提供了 u32
        // 而且 result 是 u64，但使用 reg（64 位）没问题
        // 问题在于 value 被当作 64 位使用
        // 下面这个实际是正确的（因为 reg 会自动扩展），但语义上需要留意
        asm!(
            "popcnt {result:r}, {value:r}",
            value = in(reg) value,
            result = out(reg) result,
            options(nomem, pure, preserves_flags),
        );
        // 注意：u32 放入 64 位寄存器时高 32 位是未定义的
        // popcnt 会对整个 64 位计数——这会导致错误结果
        // 正确的做法是使用 "popcnt {result:e}, {value:e}" (32 位版本)
        // 或者先用 "mov {value:e}, ..." 确保高位清零
    }
}
```

正确的做法是匹配指令的操作数大小。如果需要 32 位操作数，使用 `r` 寄存器类的 32 位版本（即 `eax/ebx/ecx/edx/r8d-r15d`）——你可以通过使用 `in(eax)` 显式指定，或者确保 Rust 变量的类型与指令期望完全一致。

### 9.3 在 asm! 中引用局部变量的地址

`asm!` 宏中的指针操作数是数值——编译器将变量的地址作为立即数传递给寄存器。但你需要确保变量在 asm! 执行期间存活且不被移动：

```rust
use std::arch::asm;

/// ✅ 正确：通过输入约束传递指针
/// 编译器负责加载地址到寄存器，并保证变量的有效性
fn correct_pointer_usage(data: &[u8]) -> u8 {
    let first_byte: u8;
    unsafe {
        asm!(
            "mov {0}, byte ptr [{1}]",  // 从 data[0] 读取一个字节
            out(reg_byte) first_byte,
            in(reg) data.as_ptr(),
        );
    }
    first_byte
}
```

指针通过 `in(reg)` 传递后，编译器将变量的地址放入一个寄存器中。只要变量的生命周期覆盖 asm! 的执行区间，指针就是有效的。

### 9.4 过度使用 nomem

`nomem` 是一个强大的优化选项，但它排除了内存访问。如果在标记为 `nomem` 的 asm! 块中访问了内存（即使是通过指针操作数），行为是未定义的。只有在你 100% 确定汇编代码只操作寄存器和操作数时才使用 `nomem`。

```rust
// ❌ 危险：标记了 nomem 但通过指针读取了内存
// asm!("mov {0}, [{1}]", ..., options(nomem));
// 这是未定义行为——nomem 告诉编译器"不访问内存"但实际访问了

// ✅ 正确：移除 nomem 选项
// asm!("mov {0}, [{1}]", ...);
// 默认行为允许内存访问
```

## 10. 安全性——为什么 asm! 总是 unsafe

### 10.1 unsafe 的根本原因

`asm!` 宏永远在 `unsafe` 块中使用，这不是一个可选的安全带——它是 Rust 安全模型的核心设计决策。`asm!` 必须 unsafe 的原因有三层：

第一层是内存安全：编译器无法验证你的汇编指令是否正确使用了内存。一条错误的 `mov` 指令可以覆盖任意内存地址——编译器无法阻止，也无法检测。

第二层是类型安全：汇编指令不关心 Rust 的类型系统。你可以将一个 `&str` 的指针当作 `u64` 传给加法指令，编译器不会报错——但在运行时，结果将是毫无意义的。

第三层是控制流安全：汇编指令可以跳转到任意地址，绕过 Rust 的所有控制流检查。栈帧、返回地址、异常处理表——一切都可能被错误的 `jmp` 或 `ret` 破坏。

### 10.2 你必须维护的不变量

当你编写 `unsafe { asm!(...) }` 时，你向编译器和 Rust 社区做出了以下承诺：

你必须保证操作数的有效性——输入操作数的值是寄存器大小的有意义的值，输出操作数最终会被赋予有效的值。

你必须诚实地声明 clobbers——所有被修改的寄存器都必须出现在输出约束或 clobber 声明中。不诚实的 clobber 声明会导致未定义行为。

你必须正确使用 options——`nomem` 意味着真的不访问内存，`pure` 意味着真的没有副作用，`preserves_flags` 意味着真的不修改标志位（或者你不依赖修改前的标志位）。

你必须确保 asm! 块内部的跳转不会逃逸——`jmp` 到 asm! 块外部的地址是未定义行为（除非你使用的是 `global_asm!` 宏）。

### 练习 1：实现一个安全的 rdtsc 包装

编写一个函数，使用 `asm!` 读取 CPU 时间戳计数器（`rdtsc` 指令），并安全地将结果作为 `u64` 返回。同时添加一个辅助函数，在同一段 asm! 中读取 TSC 和 CPU 标识（`rdtscp` 指令，读取 TSC 和处理器 ID）。

`rdtscp` 指令将 TSC 值写入 edx:eax，并将 IA32_TSC_AUX MSR 的值写入 ecx。

### 练习 2：分析 asm! 块的 LLVM IR

对于以下 `asm!` 调用，推测它在 LLVM IR 中的表示，并指出潜在的问题：

```rust
let a: u64 = 100;
let b: u64 = 200;
let sum: u64;
let carry: u64;
unsafe {
    asm!(
        "add {sum}, {a}, {b}",
        "adc {carry}, 0",
        a = in(reg) a,
        b = in(reg) b,
        sum = lateout(reg) sum,
        carry = lateout(reg) carry,
        options(pure, nomem),
    );
}
```

问题：这段代码存在什么缺陷？`adc {carry}, 0` 的语义是什么？为什么 `lateout` 在这里是必要的？

### 练习 3：跨平台的内联汇编

编写一个函数 `cpu_relax`，在 x86_64 上执行 `pause` 指令（提示 CPU 处于自旋等待），在 ARM64 上执行 `yield` 指令（提示 CPU 可以放弃执行资源）。两种实现都标记为 `options(nomem, preserves_flags)`。思考为什么这样的指令在自旋锁中至关重要。

## 12. 练习答案

### 练习 1 答案

```rust
use std::arch::asm;

/// 读取 CPU 时间戳计数器
/// 返回 64 位的 TSC 值
pub fn read_tsc() -> u64 {
    let low: u32;
    let high: u32;
    unsafe {
        asm!(
            "rdtsc",            // 读取 TSC → edx:eax
            out("eax") low,     // 低 32 位
            out("edx") high,    // 高 32 位
            options(nomem),
        );
    }
    ((high as u64) << 32) | (low as u64)
}

/// 读取 TSC 和处理器 ID（通过 rdtscp 指令）
/// rdtscp 将 TSC 写入 edx:eax，将 IA32_TSC_AUX MSR 写入 ecx
pub fn read_tsc_with_processor_id() -> (u64, u32) {
    let low: u32;
    let high: u32;
    let processor_id: u32;
    unsafe {
        asm!(
            "rdtscp",                       // 读取 TSC 和 processor ID
            out("eax") low,
            out("edx") high,
            out("ecx") processor_id,        // ecx 接收 IA32_TSC_AUX 的值
            options(nomem),
        );
    }
    let tsc = ((high as u64) << 32) | (low as u64);
    (tsc, processor_id)
}
```

关键点：`rdtscp` 比 `rdtsc` 多了一个输出（`ecx`），并且它还是一个序列化指令——它保证所有之前的指令在 TSC 被读取之前完成。如果你漏掉了 `out("ecx") processor_id`，编译器不知道 `ecx` 被修改了——经典的 clobber 遗漏错误。

### 练习 2 答案

分析这段代码的问题：

```rust
// 问题 1: 指令错误
// "add {sum}, {a}, {b}" 不是有效的 x86_64 指令
// x86_64 的 add 指令格式是 "add dst, src"（两个操作数），不是三个操作数
// 正确的写法是：
// "add {sum}, {a}"   → sum = a
// "add {sum}, {b}"   → sum = sum + b
// 或者：
// "mov {sum}, {a}"
// "add {sum}, {b}"

// 问题 2: adc 不匹配
// "adc {carry}, 0" 也是无效的——adc 需要寄存器操作数
// 正确的写法是将 carry 清零，然后执行 adc
// 例如："setc {carry:b}" 将进位标志保存到一个字节寄存器

// 问题 3: lateout 的原因
// lateout 是必要的，因为 {a} 和 {b} 是输入，{sum} 是输出
// 如果编译器将 sum 和 a 分配到同一个寄存器（对于 out 约束是可以的），
// 那么 sum 在 a 被读取之前就被覆盖了
// lateout 保证 sum 在所有输入被读取之后才被写入
// 这也解释了为什么这两个操作数不能被分配在同一个寄存器

// 问题的根本：这段代码试图做加法并检测加法溢出
// 正确的实现：
fn add_with_carry(a: u64, b: u64) -> (u64, bool) {
    let sum: u64;
    let carry: u8;
    unsafe {
        asm!(
            "add {sum}, {b}",        // sum (初始 = a) + b
            "setc {carry}",          // 如果加法导致进位（无符号溢出），设置 carry = 1
            sum = inout(reg) a => sum,
            b = in(reg) b,
            carry = out(reg_byte) carry,
            options(nomem, preserves_flags),
        );
    }
    (sum, carry != 0)
}
```

注意修复后的版本使用了 `inout(reg) a => sum`——`sum` 初始化为 `a`，然后 `add {sum}, {b}` 将 `b` 加到 `sum` 上。`setc` 指令检查进位标志（CF），如果加法产生进位则将 `carry` 设为 1。

### 练习 3 答案

```rust
use std::arch::asm;

/// 向 CPU 提示当前处于自旋等待状态
/// x86_64: 执行 PAUSE 指令
/// ARM64:  执行 YIELD 指令
///
/// 这些指令告知 CPU 当前线程正处于忙等待（busy-wait）循环中，
/// CPU 可以：
///   - 降低功耗（避免在空循环中全速运行）
///   - 在超线程中为另一个逻辑线程腾出执行资源
///   - 避免内存序违规导致的流水线清空
#[cfg(target_arch = "x86_64")]
pub fn cpu_relax() {
    unsafe {
        asm!("pause", options(nomem, preserves_flags));
    }
}

#[cfg(target_arch = "aarch64")]
pub fn cpu_relax() {
    unsafe {
        // ARM64 的等价指令——提示 CPU 可以放弃执行资源
        // ISB 是可选的，它保证 yield 之前的指令在 yield 之后才重排
        asm!("yield", options(nomem, preserves_flags));
    }
}

/// 自旋锁中 cpu_relax 的使用示例
use std::sync::atomic::{AtomicBool, Ordering};

pub struct SpinLock {
    locked: AtomicBool,
}

impl SpinLock {
    pub const fn new() -> Self {
        SpinLock {
            locked: AtomicBool::new(false),
        }
    }

    pub fn lock(&self) {
        // 自旋等待直到获取锁
        while self.locked.swap(true, Ordering::Acquire) {
            // 在自旋等待期间调用 cpu_relax 是性能关键
            // 没有 PAUSE/YIELD，CPU 会在循环中全速空转，
            // 浪费功耗并可能在超线程系统中拖慢另一个逻辑线程
            cpu_relax();
        }
    }

    pub fn unlock(&self) {
        self.locked.store(false, Ordering::Release);
    }
}
```

这个练习的核心洞察是：`pause`/`yield` 指令本身不改变程序的正确性——它们只是性能提示。但忽略它们会导致显著的性能退化，特别是在超线程（Hyper-Threading）系统中。自旋锁的忙等待循环中加上 `cpu_relax` 是标准做法，tokio、std 的内部锁实现无一例外都使用了这个技巧。

```
内联汇编 (asm!) 学习清单：

□ 理解 asm! 宏的语法：模板字符串、输出、输入、options 四个区域
□ 知道 x86_64 的基本寄存器名和位宽：rax/eax/ax/al
□ 能区分 Intel 语法和 AT&T 语法，知道 Rust 默认使用 Intel 语法
□ 理解 ARM64 的基本寄存器模型和与 x86_64 的主要差异
□ 掌握五种约束类型：in / out / inout / lateout / inlateout
□ 能够解释 lateout 的使用场景：输入和输出共享同一个寄存器时
□ 理解 clobber 的必要性：被修改的寄存器必须声明
□ 掌握六种 options 的含义：pure / nomem / readonly / preserves_flags / nostack / att_syntax
□ 能够实现简单的 Linux 系统调用：write 的汇编版本
□ 知道常见的 asm! 陷阱：忘记 clobbers、操作数大小不匹配、误用 nomem
□ 理解 asm! 永远 unsafe 的原因：编译器无法验证内存安全、类型安全、控制流安全
□ 能够编写跨平台的内联汇编：使用 #[cfg(target_arch)] 条件编译
```

> **Phase 4 核心武器**：`asm!` 是 Rust 工具箱中最锋利但也最危险的刀。当你需要在特定平台上挤出最后一丝性能，或者需要执行 safe Rust 无法表达的硬件操作时，它是无可替代的。但请记住——大多数情况下 `std::arch` 模块中的包装函数已经足以满足你的需求。[[4.5-compiler-intrinsics]] 将展示另一种底层机制：编译器内建函数——它们比 asm! 更便携，但同样需要 unsafe 和深刻的理解。
