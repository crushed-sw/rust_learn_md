# 4.8 FFI 进阶：绑定生成 —— 让机器替你写 FFI 代码

> 手动书写 FFI 绑定如同手工翻译一本外语小说——每行都需要仔细核对，任何一个微小的偏差都可能导致灾难性的未定义行为。当你面对一个拥有数百个函数、数十个结构体、层层嵌套的宏定义的 C 头文件时，手动翻译不仅枯燥，而且极易出错。`bindgen` 的核心理念是："C 头文件已经精确描述了这个库的接口——为什么不用它作为真理来源？"它将 C 类型系统映射为 Rust 类型系统，处理好对齐、填充、平台差异，让你只需关注如何用 Rust 的风格封装那些原始的 unsafe 函数。`cxx` 则走得更远——它不仅是绑定生成器，更是一个安全的 C++/Rust 双向互操作桥梁，让两种语言的类型系统在编译期就能互相理解。

---

## 1. `bindgen` —— 从 C 头文件生成 Rust 绑定

### 1.1 bindgen 是什么

`bindgen` 是一个工具，它解析 C（和部分 C++）头文件，生成对应的 Rust FFI 绑定代码。它基于 Clang/LLVM 的 libclang 库，这意味着它能理解完整的 C 类型系统——不仅仅是"看到"类型名，而是深入到 typedef 展开、宏求值、结构体布局的层面。

bindgen 的工作流程是：读取 C 头文件 → libclang 解析为 AST → 遍历 AST 节点 → 映射为 Rust 的 `extern "C"` 块、结构体定义、常量声明等 → 输出一个 Rust 源文件（通常叫 `bindings.rs`）。

最简单的使用方式是通过命令行：

```
bindgen input.h -o bindings.rs
```

这行命令背后，bindgen 在做什么？它打开 `input.h`，用 clang 的预处理器和解析器处理它，得到完整的抽象语法树。然后它遍历这个 AST——遇到函数声明时生成 `extern "C" { fn ... }` 块；遇到结构体时生成 `#[repr(C)] struct`；遇到 `#define` 常量时生成 `const`；遇到 `typedef` 时根据目标类型生成 Rust 的类型别名。

在真实项目中，bindgen 一般通过 `build.rs` 构建脚本集成——这意味着每次 `cargo build` 都会自动重新生成绑定，确保它们始终与原 C 库的当前版本同步。

```rust
// 一个典型的 build.rs——在编译时生成 FFI 绑定
fn main() {
    // 步骤 1：告诉 bindgen 解析哪个头文件
    let bindings = bindgen::Builder::default()
        // 输入头文件
        .header("wrapper.h")     // wrapper.h 包含所有需要的 #include
        // 解析选项
        .clang_arg("-I./include") // 添加 include 搜索路径
        .clang_arg("-DUSE_OPENSSL") // 预定义宏
        // 生成选项
        .generate_comments(true)  // 保留 C 注释作为 Rust doc comment
        .layout_tests(true)      // 生成 #[test] 验证结构体大小和对齐
        .derive_debug(true)      // 自动为结构体派生 Debug
        .derive_default(true)    // 自动为结构体派生 Default
        // 调用 generate() 触发解析
        .generate()
        .expect("Unable to generate bindings");

    // 步骤 2：把生成的代码写入文件
    let out_path = std::path::PathBuf::from(
        std::env::var("OUT_DIR").unwrap()
    );
    bindings
        .write_to_file(out_path.join("bindings.rs"))
        .expect("Couldn't write bindings!");
}
```

关键点在于 `OUT_DIR` 环境变量——这是 cargo 在 `target/` 目录下为当前 crate 分配的临时目录。生成的 `bindings.rs` 被写入这个目录，然后在 Rust 源代码中通过 `include!` 宏引入：

```rust
// 在 src/lib.rs 或 src/main.rs 中引入生成的绑定
// OUT_DIR 中的文件不能直接 use，需要用 include! 宏
include!(concat!(env!("OUT_DIR"), "/bindings.rs"));
```

`include!` 宏在编译时将指定文件的内容直接嵌入到当前位置——就像你把那段代码直接复制粘贴过来一样。结合 `env!("OUT_DIR")` 宏在编译期读取 `OUT_DIR` 的值，我们就实现了在编译时生成代码、编译时引入代码的完整管线。

### 1.2 处理复杂的 C 结构

真实的 C 头文件通常比简单的函数声明复杂得多。以下是一个典型的 scenario 以及 bindgen 的处理方式：

假设我们有以下 C 头文件 `wrapper.h`：

```c
// wrapper.h —— 包含一些典型的 C 声明模式
#include <stdint.h>
#include <stddef.h>

// 宏定义常量
#define MAX_CONNECTIONS 128
#define VERSION_MAJOR 2
#define VERSION_STRING "2.0.0"

// 不透明类型（只需指针使用）
typedef struct DatabaseHandle DatabaseHandle;

// 有具体定义的结构体
typedef struct {
    uint32_t id;
    const char *name;
    uint8_t flags;
} Record;

// 函数声明
DatabaseHandle* db_open(const char *path);
int db_close(DatabaseHandle *handle);
int db_get_record(DatabaseHandle *handle, uint32_t id, Record *out);

// 回调类型
typedef void (*log_callback)(int level, const char *message);
void set_log_callback(log_callback cb);
```

bindgen 会将这些 C 声明转换为以下 Rust 绑定（简化版，实际生成的代码更多）：

```rust
// bindgen 生成的 Rust 绑定（概念性简化版）
/* automatically generated by bindgen */

// 宏常量 → Rust 常量
pub const MAX_CONNECTIONS: u32 = 128;
pub const VERSION_MAJOR: u32 = 2;
// 字符串宏 → Rust &str（静态生命周期）
pub const VERSION_STRING: &[u8; 6usize] = b"2.0.0\0";

// 不透明类型 → 用零大小数组模拟（不能在 safe Rust 中直接构造）
#[repr(C)]
#[derive(Debug)]
pub struct DatabaseHandle {
    _unused: [u8; 0],
}

// 具体结构体 → #[repr(C)] struct
#[repr(C)]
#[derive(Debug)]
pub struct Record {
    pub id: u32,
    pub name: *const std::os::raw::c_char,
    pub flags: u8,
}

// 函数声明 → extern "C" 块
extern "C" {
    pub fn db_open(path: *const std::os::raw::c_char) -> *mut DatabaseHandle;
    pub fn db_close(handle: *mut DatabaseHandle) -> std::os::raw::c_int;
    pub fn db_get_record(
        handle: *mut DatabaseHandle,
        id: u32,
        out: *mut Record,
    ) -> std::os::raw::c_int;
}

// 函数指针类型 → 类型别名
pub type log_callback = ::std::option::Option<
    unsafe extern "C" fn(
        level: std::os::raw::c_int,
        message: *const std::os::raw::c_char,
    ),
>;

extern "C" {
    pub fn set_log_callback(cb: log_callback);
}
```

有几个值得注意的细节：bindgen 将 C 的 `const char *` 映射为 `*const c_char`（不是 `&str`），因为 C 字符串不一定符合 Rust 的 UTF-8 和生命周期保证；`log_callback` 被包裹在 `Option` 中——这是 bindgen 的对齐优化，因为函数指针和可空指针在 ABI 层面是兼容的。

### 1.3 allowlist 和 blocklist

当封装一个大型 C 库时，你可能只关心其中一小部分 API。bindgen 的 allowlist（白名单）和 blocklist（黑名单）让你精确控制生成哪些绑定。

```rust
// build.rs —— 精细控制生成内容
let bindings = bindgen::Builder::default()
    .header("wrapper.h")
    // —— 函数控制 ——
    .allowlist_function("db_.*")          // 只生成以 "db_" 开头的函数
    .blocklist_function("db_internal_.*") // 排除内部函数
    .allowlist_function("log_callback")   // 精确添加单个函数
    // —— 类型控制 ——
    .allowlist_type("Record")             // 只生成 Record 类型
    .blocklist_type("InternalState")      // 隐藏内部实现细节
    // —— 变量控制 ——
    .allowlist_var("VERSION_.*")          // 只生成版本相关常量
    .blocklist_var("DEBUG_FLAG")          // 隐藏调试标志
    // —— 整体控制 ——
    .blocklist_item("InternalModule_.*")  // 黑名单中可以是任意条目
    // opaque_type 用于告诉 bindgen 某些类型不要生成字段（转为不透明句柄）
    .opaque_type("DatabaseHandle")        // DatabaseHandle 已经是 opaque 的
    .generate()
    .expect("Unable to generate bindings");
```

allowlist/blocklist 通过正则表达式匹配名称，使用 `db_.*` 这样的模式来匹配以 `db_` 开头的所有标识符。黑名单的优先级高于白名单——如果你同时 allow 和 block 一个项，它不会被生成。

### 1.4 处理 C 的 inline 函数和宏

bindgen 默认会跳过 C 的 `static inline` 函数和宏——因为它只处理头文件中"声明"的内容，而不是函数体。对于 inline 函数，你通常有两个选择：

**选择 1：在 C 侧创建一个非 inline 的包装函数。** 这是最可靠的方法——创建一个额外的 `.c` 文件，调用 inline 函数然后用正常的外部链接导出。

```c
// wrapper_helpers.c —— 为非 inline 的函数体提供外部链接
#include "original_library.h"

// 原库中的 static inline 函数 inline int add(int a, int b);
// 我们提供一个非 inline 的包装
int add_helper(int a, int b) {
    return add(a, b);  // 调用 inline 函数，但我们的包装有外部链接
}
```

**选择 2：在 Rust 侧用 `extern "C"` 重新实现。** 对于简单的函数（如数学运算或位操作），在 Rust 侧重写通常更好——既减少了对 C 编译器的依赖，又能利用 Rust 的更好的泛型和类型安全。

对于宏常量，bindgen 默认生成 `pub const` 声明。但并非所有宏都是常量——类似 `#define MAX(a, b) ((a) > (b) ? (a) : (b))` 这样的函数式宏无法直接翻译为 Rust，你需要手动处理（通常是重写为 Rust 函数或 `macro_rules!`）。

## 2. `cbindgen` —— 从 Rust 生成 C 头文件

### 2.1 使用场景

`bindgen` 的方向是 C → Rust。但有时方向是反过来的——你需要一个 C 库调用你的 Rust 代码。例如：你有一个 Rust 实现的加密库，需要被 C 项目使用；或者你正在将一个项目从 C 逐步迁移到 Rust，需要 Rust 提供兼容的 C 头文件。

`cbindgen` 就是解决这个问题的工具。它解析 Rust 源代码，提取带有 `#[no_mangle]` 和 `pub extern "C"` 标记的函数和类型，生成对应的 C 头文件。

### 2.2 基本使用

首先写 Rust 代码。cbindgen 要生成的 C 头文件来源于以下标记了特定属性的 Rust 函数和类型：

```rust
// src/lib.rs —— 带有 C FFI 导出标记的 Rust 代码
// cbindgen 会扫描这个文件，提取 pub extern "C" 项

/// 表示文件打开模式的枚举
/// cbindgen: 在生成的 C 头文件中记录这个注释
#[repr(C)]
pub enum OpenMode {
    ReadOnly = 0,
    ReadWrite = 1,
    Append = 2,
}

/// Opaque handle to a database connection —— 不透明类型
/// C 侧只能通过指针使用，不能直接访问
pub struct Database {
    // 私有字段——cbindgen 发现没有 pub 字段时，会生成为不透明类型
    internal_path: String,
    internal_handle: i32,
}

/// 打开数据库连接
#[no_mangle]
pub extern "C" fn database_open(path: *const std::os::raw::c_char) -> *mut Database {
    // 实现省略
    std::ptr::null_mut()
}

/// 关闭数据库连接
#[no_mangle]
pub extern "C" fn database_close(db: *mut Database) {
    if !db.is_null() {
        unsafe { drop(Box::from_raw(db)); }
    }
}

/// 执行查询
#[no_mangle]
pub extern "C" fn database_execute(
    db: *const Database,
    query: *const std::os::raw::c_char,
) -> std::os::raw::c_int {
    // 实现省略
    0
}
```

你可以通过命令行运行 cbindgen：

```
cbindgen --config cbindgen.toml --crate my_crate --output my_header.h
```

或者在 Cargo.toml 中配置：

```toml
# Cargo.toml
[package]
name = "my_rust_lib"
version = "0.1.0"

[lib]
crate-type = ["cdylib", "staticlib"]  # 生成 .so / .a

# cbindgen 配置
[package.metadata.cbindgen]
# 如果配置简单，可以直接放在 Cargo.toml 中
```

cbindgen 生成的 C 头文件大致如下：

```c
// cbindgen 自动生成的头文件 my_header.h
#include <stdarg.h>
#include <stdbool.h>
#include <stdint.h>
#include <stdlib.h>

/// 表示文件打开模式的枚举
/// cbindgen: 在生成的 C 头文件中记录这个注释
enum OpenMode {
  ReadOnly = 0,
  ReadWrite = 1,
  Append = 2,
};
typedef uint8_t OpenMode;  // cbindgen 使用判别类型（默认为 u8）

/// Opaque handle to a database connection —— 不透明类型
/// C 侧只能通过指针使用，不能直接访问
typedef struct Database Database;

/// 打开数据库连接
struct Database *database_open(const char *path);

/// 关闭数据库连接
void database_close(struct Database *db);

/// 执行查询
int database_execute(const struct Database *db, const char *query);
```

### 2.3 cbindgen.toml 配置

cbindgen 可以通过 `cbindgen.toml` 文件进行精细控制：

```toml
# cbindgen.toml —— 精细控制 C 头文件生成
language = "C"           # 目标语言：C 或 C++

# 头文件包含
[export]
include = ["Database", "database_open", "database_close"]  # 只导出这些项
exclude = ["internal_*"]  # 排除内部项

# 样式设置
[style]
tag = "enum"             # enum 的 tag 命名风格
type = "PascalCase"      # 类型的命名风格：PascalCase, camelCase, snake_case

# 枚举设置
[enum]
rename_variants = "None"     # 不变更变体名称
prefix_with_name = true      # 变体自动加类型名前缀
derive_helper_methods = true # 生成辅助函数

# 宏定义
[defines]
"feature = serde" = "HAS_SERDE"  # 条件编译
"target_os = windows" = "WINDOWS"

# 文档
[documentation]
style = "doxy"           # Doxygen 风格的注释
```

## 3. C++ 互操作 —— `cxx` crate

### 3.1 cxx 与 bindgen 的本质区别

`bindgen` 解决的是"我有 C 库，需要 Rust 绑定"的问题。`cxx` 解决的是"我同时有 C++ 和 Rust 代码，它们需要互相调用"的问题——它生成 Rust 和 C++ 两端的胶水代码，使得跨语言调用在编译期就是类型安全的。

cxx 的核心设计是**桥接模块**（bridge module）。你在 Rust 侧写一个 `#[cxx::bridge]` 模块，声明哪些 Rust 类型和函数要暴露给 C++，以及哪些 C++ 类型和函数要暴露给 Rust。cxx 的代码生成器同时生成 Rust 粘合代码和 C++ 粘合代码——两端都经过编译检查，消除 ABI 不匹配的可能性。

```rust
// Rust 侧——定义 cxx bridge 模块
// 这个模块不是普通的 Rust 模块——它被 cxx 宏处理后产生真实的 FFI 绑定

#[cxx::bridge]
mod ffi {
    // ========== 从 C++ 侧引入的类型和函数 ==========

    // 声明 C++ 中存在一个类型 Client
    // 在 Rust 侧它是"不透明的"——只能通过引用使用
    extern "C++" {
        type Client;  // C++ 侧的 Client 类

        // C++ 的构造函数 → Rust 调用它创建 Client
        fn new_client(address: &str) -> UniquePtr<Client>;

        // C++ 的方法 → Rust 可以通过 &self 调用
        fn connect(self: &Client) -> Result<()>;
        fn send_message(self: &Client, msg: &str) -> Result<()>;
        fn disconnect(self: &Client);
    }

    // ========== 暴露给 C++ 的 Rust 类型和函数 ==========

    // Rust 函数暴露给 C++ —— C++ 可以调用
    extern "Rust" {
        fn process_response(data: &[u8]) -> String;
        fn log_message(level: u32, message: &str);
    }
}

// 在 bridge 外实现 Rust 侧的函数
fn process_response(data: &[u8]) -> String {
    format!("processed {} bytes", data.len())
}

fn log_message(level: u32, message: &str) {
    println!("[{}] {}", level, message);
}
```

C++ 侧需要包含 cxx 生成的头文件并提供实现：

```cpp
// C++ 侧——实现 bridge 中声明的 C++ 类型和方法
#include "my_project/src/main.rs.h"  // cxx 自动生成的头文件
#include <string>
#include <iostream>

class Client {
public:
    Client(const std::string& address) : address_(address) {}

    void connect() {
        // Rust 的 Result<()> 映射为 C++ 异常或返回码
        // cxx 处理了这个转换
        std::cout << "Connecting to " << address_ << std::endl;
        // ... 实际连接逻辑 ...
    }

    void send_message(const std::string& msg) {
        // 调用 Rust 的 log_message 函数
        log_message(2, msg);  // ← 这是 Rust 函数！cxx 处理了 FFI 边界
    }

    void disconnect() {
        std::cout << "Disconnecting" << std::endl;
    }

private:
    std::string address_;
};

// 实现 cxx 需要的工厂函数
std::unique_ptr<Client> new_client(const std::string& address) {
    return std::make_unique<Client>(address);
}
```

cxx 的类型映射是它最大的价值所在。它不仅仅是在语言边界上传递裸指针——cxx 理解并安全地转换多种类型：

- **基础类型**：`i32`, `u64`, `f64`, `bool` —— 直接映射
- **字符串类型**：`&str` / `String` ↔ `rust::Str` / `rust::String` —— 自动处理 UTF-8 和生命周期
- **切片类型**：`&[T]` ↔ `rust::Slice<T>` —— 自动传递长度信息
- **Box 和 UniquePtr**：`Box<T>` ↔ `std::unique_ptr<T>` —— 所有权转移
- **Vec**：`Vec<T>` ↔ `rust::Vec<T>` —— 支持 push/pop/迭代
- **CxxString**：`CxxString` ↔ `std::string` —— 通过引用借用
- **SharedPtr**：`SharedPtr<T>` ↔ `std::shared_ptr<T>` —— 共享所有权

这是 cxx 相较于 bindgen 的根本优势——bindgen 给你 `*const c_char`，你需要手动转换为 `&str`；cxx 直接给你 `&str`，因为它在生成的胶水代码中处理了所有转换。

### 3.2 cxx vs bindgen —— 如何选择

总结选择决策树：

- **你要从 Rust 调用一个现有的、成熟的 C 库**（如 sqlite3, zlib, openssl）→ 用 bindgen。这些库的 C 头文件是真理来源，bindgen 直接从它们生成绑定。
- **你要从 C 调用 Rust 库** → 用 cbindgen 生成 C 头文件，或者直接用 cxx。
- **你在写一个同时包含 Rust 和 C++ 的新项目**，并且需要双向调用 → 用 cxx。它比 bindgen/cbindgen 更安全，提供自动类型转换，并且两端的代码都经过编译检查。
- **你需要调用 C++ 特有的功能**（类、模板、虚函数、重载、异常）→ 只能用 cxx。bindgen 对 C++ 的支持仅限于 C-like 的子集（POD 类型和自由函数）。
- **你只需要调用一个简单的 C 函数**（比如 `getenv`、`cos`）→ 手动写 `extern "C"` 声明。不需要引入额外的构建依赖。

## 4. 跨 FFI 的字符串类型 —— CString / CStr

### 4.1 CString —— 拥有所有权的 C 字符串

Rust 的 `String` 和 `&str` 不保证内部有结尾 `\0`（空终止符），而 C 的字符串（`char*`）要求以 `\0` 结尾。`CString` 和 `CStr` 分别对应 `String` 和 `&str`，专门用于 FFI。

`CString` 是一个拥有所有权的类型，保证内部字节序列以 `\0` 结尾，且中间不包含 `\0`。它可以从 `String`、`&str` 或 `Vec<u8>` 创建。

```rust
use std::ffi::{CString, CStr};
use std::os::raw::c_char;

// CString::new() —— 安全构造，检查中间没有 \0
let rust_string = String::from("hello world");
let c_string = CString::new(rust_string)
    .expect("CString::new failed: string contains null byte");

// CString::new() 的安全保证：
// 1. 检查内部没有 \0 字节（如果有，C 函数会截断字符串）
// 2. 在末尾添加 \0 终止符
// 3. 返回的 CString 可以安全地传递给 C 函数

// 如果确定字节序列没有 null 且在末尾有 null：
// CString::from_vec_unchecked() —— unsafe，不做检查
let mut bytes = b"hello world\0".to_vec();
let c_string = unsafe {
    CString::from_vec_unchecked(bytes)
};

// 获取裸指针传给 C
extern "C" {
    fn puts(s: *const c_char) -> i32;
}
unsafe { puts(c_string.as_ptr()); }
```

获取指向内部缓冲区的指针有多种方式。`as_ptr()` 返回 `*const c_char`（只读指针）；`into_raw()` 消耗 `CString`，返回 `*mut c_char`（所有权转移给调用者）；相应的，`from_raw` 从裸指针重建 `CString`（重新获得所有权）。

```rust
// into_raw() / from_raw() —— 所有权转移模式
let c_string = CString::new("owned by C").unwrap();

// 将所有权转移给 C 代码——C 代码负责释放
let raw_ptr: *mut c_char = c_string.into_raw();

// ... C 侧使用 raw_ptr ...

// C 代码完成后，Rust 重新接管——CString 的 Drop 会正确释放内存
// 注意：这要求 C 侧没有释放这个指针，且 ptr 来自 into_raw
unsafe {
    let _reclaimed = CString::from_raw(raw_ptr);
}
```

### 4.2 CStr —— 借用的 C 字符串

`CStr` 是 C 字符串的借用形式，对应 `&str`。它不拥有数据，只是一个指向以 `\0` 结尾的字节序列的引用。

```rust
use std::ffi::CStr;
use std::os::raw::c_char;

extern "C" {
    // C 函数返回一个指向静态字符串的指针
    fn strerror(errnum: i32) -> *const c_char;
}

fn get_error_message(errnum: i32) -> String {
    unsafe {
        let ptr = strerror(errnum);
        // CStr::from_ptr —— 从裸指针创建 &CStr
        // 安全性要求：
        // 1. ptr 不为 null
        // 2. ptr 指向一个以 \0 结尾的有效字节序列
        // 3. ptr 指向的数据在 &CStr 的生命周期内有效
        let c_str = CStr::from_ptr(ptr);

        // 转换为 Rust String
        // to_string_lossy() —— 丢失性转换：非 UTF-8 字符被 � (U+FFFD) 替换
        c_str.to_string_lossy().into_owned()
    }
}
```

`CStr::from_ptr()` 是最常见的从 C 函数返回值创建 `&CStr` 的方式——但它需要你保证裸指针的生命周期和有效性。如果你不确定 C 字符串是否是合法的 UTF-8，使用 `to_string_lossy()`；如果你确定（或者想处理错误），使用 `to_str().map_err(...)`。

### 4.3 OsString —— 平台原生路径

在 Unix 和 Windows 上，系统路径和文件名不一定是有效的 UTF-8。`OsString`/`OsStr` 是 Rust 对平台原生字符串的抽象——在 Unix 上它和 `Vec<u8>` 等价，在 Windows 上它使用 WTF-8（一种 UTF-8 的宽松超集）来兼容非法的 Unicode 序列。

```rust
use std::ffi::{OsString, OsStr};
use std::path::Path;

// 从平台原生路径创建
let path = OsString::from("/home/user/file.txt");
// 在 Windows 上，也可以是：
// let path = OsString::from("C:\\Users\\user\\file.txt");

// 转换为 CString 以传给 C 函数
// 在 Unix 上这基本上是零成本的（OsString 内部就是 Vec<u8>）
// 在 Windows 上可能需要编码转换
// let c_path = CString::new(path.as_encoded_bytes())?;
```

在实际项目中，路径操作应该优先使用 `std::path::Path`/`PathBuf`，它们在内部使用 `OsStr`/`OsString`，提供了跨平台的路径操作能力。

## 5. 实战 —— 用 bindgen 为 zlib 生成绑定

### 5.1 项目设置

以下是一个完整的小项目，展示从零开始用 bindgen 为 zlib 生成 Rust 绑定。zlib 是一个广泛使用的压缩库，头文件简洁，是学习 bindgen 的理想目标。

项目结构：

```
zlib-sys/
  Cargo.toml
  build.rs
  wrapper.h
  src/
    lib.rs
```

Cargo.toml 的配置：

```toml
# Cargo.toml
[package]
name = "zlib-sys"
version = "0.1.0"
edition = "2021"

[build-dependencies]
bindgen = "0.70"   # 构建时依赖——bindgen 只在 build.rs 中运行

[dependencies]
libc = "0.2"       # 用于 C 兼容类型的补充

[build]
# （可选）告诉 cargo 在构建前检查 zlib 是否可用
# rustc-link-lib 指令在 build.rs 中处理
```

wrapper.h 文件包含我们想要生成绑定的所有声明：

```c
// wrapper.h —— 汇聚所有需要的 zlib 声明
// 这是一个"包裹头文件"，包含了 zlib 中我们需要的所有头文件
#include <zlib.h>
// zlib.h 包含以下我们关心的声明：
// - compress() / uncompress()
// - compressBound()
// - z_stream 结构体
// - Z_OK / Z_STREAM_END / Z_ERRNO 等常量
```

build.rs 中的完整配置：

```rust
// build.rs —— 项目构建脚本
use std::env;
use std::path::PathBuf;

fn main() {
    // 步骤 1：检查系统是否有 zlib
    // 如果系统有 pkg-config（Linux/macOS），用它来定位 zlib
    // 否则手动指定路径
    if let Ok(lib) = pkg_config::Config::new()
        .atleast_version("1.2")
        .probe("zlib")
    {
        // pkg-config 自动添加了 link 指令
        // 我们只需要打印它的 include 路径供 bindgen 使用
        for path in &lib.include_paths {
            println!("cargo:include={}", path.display());
        }
    } else {
        // 没有 pkg-config——手动链接 zlib
        println!("cargo:rustc-link-lib=z");
        // Windows 上可能是 zdll.lib 或 zlibwapi.lib
    }

    // 步骤 2：配置 bindgen
    let mut builder = bindgen::Builder::default()
        .header("wrapper.h")
        // 只生成 zlib 公共 API
        .allowlist_function("compress.*")
        .allowlist_function("uncompress.*")
        .allowlist_function("deflate.*")
        .allowlist_function("inflate.*")
        .allowlist_function("gz.*")
        .allowlist_type("z_stream")
        .allowlist_var("Z_OK")
        .allowlist_var("Z_ERRNO")
        .allowlist_var("Z_STREAM_END")
        // 生成测试以验证布局
        .layout_tests(true)
        // 派生常用 trait
        .derive_debug(true)
        .derive_default(true)
        // 生成 Rust 风格的枚举来表示常量组
        .rustified_enum(".*")
        // 对于不透明类型使用 opaque 标记
        .opaque_type("gzFile_s")
        // C 标准库类型应该用 libc crate 的类型
        .ctypes_prefix("libc");

    // 如果有额外的 include 路径
    if let Ok(include_path) = env::var("ZLIB_INCLUDE_DIR") {
        builder = builder.clang_arg(format!("-I{}", include_path));
    }

    // 步骤 3：生成绑定
    let bindings = builder
        .generate()
        .expect("Unable to generate zlib bindings");

    // 步骤 4：写入输出文件
    let out_path = PathBuf::from(env::var("OUT_DIR").unwrap());
    bindings
        .write_to_file(out_path.join("bindings.rs"))
        .expect("Couldn't write zlib bindings!");
}
```

src/lib.rs 是库的入口点——引入生成的绑定并提供安全的 Rust 包装：

```rust
// src/lib.rs —— 引入 bindgen 生成的绑定并提供安全 API

// 引入生成的所有绑定——这些是原始的不安全 FFI 声明
include!(concat!(env!("OUT_DIR"), "/bindings.rs"));

// 安全封装 1：compress —— 自动分配输出缓冲区
pub fn compress(data: &[u8]) -> Result<Vec<u8>, String> {
    // 计算最大可能的压缩后大小
    let bound = unsafe { compressBound(data.len() as u64) } as usize;

    let mut dest = vec![0u8; bound];
    let mut dest_len = bound as u64; // zlib 使用 u64 作为长度类型

    let ret = unsafe {
        compress(
            dest.as_mut_ptr(),
            &mut dest_len as *mut u64,
            data.as_ptr(),
            data.len() as u64,
        )
    };

    if ret != Z_OK {
        return Err(format!("compression failed with code: {}", ret));
    }

    // 截断到实际压缩后的大小
    unsafe { dest.set_len(dest_len as usize) };
    Ok(dest)
}

// 安全封装 2：uncompress —— 需要预先知道解压后的大小
pub fn uncompress(data: &[u8], expected_size: usize) -> Result<Vec<u8>, String> {
    let mut dest = vec![0u8; expected_size];
    let mut dest_len = expected_size as u64;

    let ret = unsafe {
        uncompress(
            dest.as_mut_ptr(),
            &mut dest_len as *mut u64,
            data.as_ptr(),
            data.len() as u64,
        )
    };

    if ret != Z_OK {
        return Err(format!("decompression failed with code: {}", ret));
    }

    unsafe { dest.set_len(dest_len as usize) };
    Ok(dest)
}
```

### 5.2 完整的使用示例

```rust
// 使用我们刚刚创建的 zlib-sys crate
use zlib_sys::{compress, uncompress};

fn main() {
    let original = b"The quick brown fox jumps over the lazy dog. "
        .repeat(100);  // 重复 100 遍以产生足够的数据

    let compressed = compress(original.as_slice())
        .expect("compression failed");
    println!(
        "Original: {} bytes, Compressed: {} bytes",
        original.len(),
        compressed.len()
    );

    let decompressed = uncompress(&compressed, original.len())
        .expect("decompression failed");
    assert_eq!(original.as_slice(), decompressed.as_slice());
    println!("Decompression successful: data matches original");
}
```

这个示例展示了从零开始封装一个真实 C 库的完整流程——build.rs 配置、wrapper.h、bindgen 设置、安全的 Rust API。你可以在任何需要封装 C 库的项目中复用这个模式。

### 练习 2 答案

原代码的问题：
1. `CString::from_raw` 在结果可能是 NULL 时直接调用——如果 `result` 是 null，`from_raw` 是 UB
2. `s`（CString）在获取 `as_ptr()` 之后、FFI 调用完成之前就可能被 drop——引用的生命周期错误（实际上这里没有生命周期问题，因为 `let ptr = s.as_ptr();` 的生命周期到 unsafe 块之后，但更安全的是确保 `s` 的生命周期覆盖整个调用）
3. 由 C 函数分配的内存应该用 C 的 `free()` 释放，而不是 `CString::from_raw`——后者会使用 Rust 的分配器释放，造成 allocator mismatch

```rust
use std::ffi::{CStr, CString};
use std::os::raw::{c_char, c_void};

extern "C" {
    fn get_home_dir(username: *const c_char) -> *mut c_char;
    // 假设 C 侧的释放函数：
    // void free_home_dir(char *ptr);
    // 如果不提供 free 函数，调用者可以用 libc::free
    fn free_home_dir(ptr: *mut c_void);
}

fn safe_get_home_dir(username: &str) -> Option<String> {
    // 步骤 1：构造 CString——开头的 null 检查
    let c_username = CString::new(username).ok()?;

    // 步骤 2：调用 C 函数
    let result: *mut c_char = unsafe { get_home_dir(c_username.as_ptr()) };

    // 步骤 3：检查 NULL
    if result.is_null() {
        return None;
    }

    // 步骤 4：从 C 指针创建 &CStr——这时候只有借用
    let c_str = unsafe { CStr::from_ptr(result) };

    // 步骤 5：转换为 Rust String（复制数据）——在释放 C 内存之前完成
    let rust_string = c_str.to_string_lossy().into_owned();

    // 步骤 6：释放 C 分配的内存
    unsafe { free_home_dir(result as *mut c_void); }

    Some(rust_string)
}

// 如果没有专门的 free_home_dir 函数：
// 使用 #[link(name = "c")] 引入 libc::free
// unsafe { libc::free(result as *mut c_void); }
```

> **以不变应万变**：无论是自动生成还是手动编写，FFI 绑定的本质始终不变——它们都是 unsafe 的承诺，由你（而非编译器）保证其正确性。bindgen 和 cxx 将机械的翻译工作自动化，让你能把精力投入到"如何设计安全的 Rust 封装"上。但工具不能完全替代理解——当你查看生成的 `bindings.rs` 时，你需要能读懂每一个 `*const c_char` 的含义。下一课 [[4.9-extern-types-opaque]] 将深入探索 C 库中的不透明类型（opaque type）——那些只知道名字、不知道大小的类型——以及 Rust 如何安全地包装它们。
