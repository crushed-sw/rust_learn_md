# 4.14 操作系统接口 —— 从系统调用到内核模块

> 每一个 Rust 程序最终都要穿过一层薄薄的抽象，进入操作系统的内核。哪怕是最简单的 `println!("hello")`，在底层都翻译成了 `write(1, buf, len)` 系统调用——用户态到内核态的切换、寄存器的保存与恢复、权限检查——这一切都在每一次"平凡"的 I/O 操作中默默发生。理解系统调用不仅仅是理解"怎么用 libc"，更是理解"计算机程序如何与硬件世界对话"。本章从最原始的 `syscall` 指令开始，穿越 Linux 的 epoll 和 io_uring、macOS 的 kqueue，最终到达"用 Rust 写 Linux 内核模块"的边界——让你看见 Rust 作为系统语言的终极表达。

---

## 1. 系统调用——程序与内核之间的"电话热线"

### 1.1 libc 包装 vs 原始系统调用

当你在 Rust 中调用 `std::fs::write("file.txt", b"hello")` 时，背后经历了多层封装：

```
你的 Rust 代码
  → std::fs::write()                      (Rust 标准库)
    → libc::write()                        (libc 包装函数)
      → syscall(SYS_write, ...)             (libc syscall 包装)
        → syscall 指令 (x86-64)             (CPU 指令)
          → 内核态 write 系统调用实现        (Linux 内核)
            → 硬件操作                       (磁盘控制器)
```

大多数情况下你不需要跳过 libc——libc 提供了一层经过数十年优化的、安全的包装。但在某些场景下，直接发出系统调用可以让你绕过 libc 的开销，或者使用 libc 尚未封装的 Linux 特有功能。

### 1.2 x86-64 系统调用约定

在 x86-64 Linux 上，系统调用遵循一套精确的寄存器使用约定：

```
x86-64 Linux 系统调用约定：

寄存器          用途
──────────────────────────────────────
rax             系统调用号（例如 SYS_write = 1）
rdi             第 1 个参数
rsi             第 2 个参数
rdx             第 3 个参数
r10             第 4 个参数（代替 rcx，因为 syscall 指令会破坏 rcx）
r8              第 5 个参数
r9              第 6 个参数
──────────────────────────────────────
rax             返回值（成功时）或负的 errno 值（失败时）
rcx             被 syscall 指令破坏（保存的是返回地址）
r11             被 syscall 指令破坏（保存的是 RFLAGS）
```

最多 6 个参数可以通过寄存器传递——这是系统调用的一个基本限制。更多参数需要通过内存传递（极少见）。

### 1.3 用 Rust 内联汇编发出原始系统调用

以下是在 Rust 中使用内联汇编直接执行 `write` 系统调用的完整示例——绕过 libc，直接与内核对话：

```rust
use std::arch::asm;

/// 不使用 libc，直接通过 syscall 指令调用 write(2)
/// 这在 x86-64 Linux 上工作，在其他平台上不可移植
fn raw_write(fd: i32, buf: &[u8]) -> isize {
    let mut result: isize;

    unsafe {
        asm!(
            "syscall",              // 单条指令：陷入内核
            inlateout("rax") 1isize => result,  // rax: SYS_write 号 (1)，返回时存放结果
            in("rdi") fd,            // rdi: 第一个参数——文件描述符
            in("rsi") buf.as_ptr(),  // rsi: 第二个参数——缓冲区地址
            in("rdx") buf.len(),     // rdx: 第三个参数——字节数
            // syscall 会破坏 rcx 和 r11——告知编译器
            lateout("rcx") _,
            lateout("r11") _,
            options(nostack),        // 不破坏栈指针
        );
    }

    result
}

// 使用示例：
fn main() {
    let msg = b"直接从 Rust 发出系统调用！\n";
    let ret = raw_write(1, msg); // fd=1 是标准输出
    if ret < 0 {
        eprintln!("write 系统调用失败: {}", -ret);
    }
}
```

这段代码的每一行都有存在的理由。`inlateout("rax")` 告诉编译器：`rax` 既作为输入（系统调用号 1，对应 write），又作为输出（系统调用的返回值）。`lateout("rcx") _` 和 `lateout("r11") _` 告知编译器这两个寄存器被 `syscall` 指令破坏了——如果不声明，编译器可能认为这些寄存器在 asm 块中保持不变，导致难以调试的 heisenbug。

### 1.4 更安全的替代：libc crate 的 syscall 函数

如果你不想写内联汇编（大多数情况下不应该），`libc` crate 提供了一个 `syscall` 函数作为折中——它仍然绕过 libc 包装函数，但跨平台移植由 crate 负责：

```rust
use libc::{syscall, SYS_write, STDOUT_FILENO};

/// 使用 libc::syscall —— 比内联汇编更安全，但仍然绕过 libc 包装
fn safe_raw_write(buf: &[u8]) -> isize {
    unsafe {
        // syscall() 是一个变长参数的 C 函数
        // 参数：系统调用号，之后是系统调用的参数
        syscall(
            SYS_write,               // rax: 系统调用号
            STDOUT_FILENO,           // rdi: fd
            buf.as_ptr(),            // rsi: buffer
            buf.len(),               // rdx: count
        ) as isize
    }
}
```

`libc::syscall` 的优点是它自动处理跨平台差异（x86-64、aarch64、riscv64 各有不同的系统调用约定），并且不需要内联汇编。但代价是——它仍然是 `unsafe` 的（因为编译器不能验证系统调用的类型安全），而且需要 libc crate 依赖。

### 1.5 什么时候该（不该）绕过 libc

绕过 libc 直接发系统调用不是常规操作——它是一种"在你确切知道自己在做什么"时的优化手段：

```rust
// 使用 libc 包装（标准做法）：
// ✅ 安全、可移植、经过充分测试
// ✅ 处理了 EINTR 重试、errno 线程安全等细节
// ✅ 你写的 "just works"
use std::io::Write;
let mut stdout = std::io::stdout();
stdout.write_all(b"normal way\n").unwrap();

// 使用 raw syscall（特殊场景）：
// ✅ 在 no_std 环境中不依赖 libc
// ✅ 调用 libc 未封装的 Linux 特有系统调用
// ✅ 极致的性能优化（避免 libc 包装的开销）
// ❌ 不可移植——针对特定 CPU 架构和操作系统
// ❌ 需要手动处理 EINTR 重试
// ❌ 不会被 valgrind/ASan 等工具正确追踪（它们 hook libc 而非 syscall 指令）
```

如果你的程序运行在正常的有 libc 的用户空间环境中，绝大多数情况下不需要也不应该绕过 libc——libc 包装的开销相比系统调用本身的内核态切换开销来说微乎其微。

## 2. epoll —— Linux 异步 I/O 的引擎

### 2.1 为什么 epoll 是 tokio 的心脏

epoll 是 Linux 提供的一个 I/O 事件通知机制——它允许你同时监控成百上千个文件描述符，并只在其中某个变得"可读"或"可写"时才收到通知。tokio、mio、async-std——所有 Rust 异步运行时在 Linux 上的 I/O 驱动层最终都建立在 epoll 之上。

```
epoll 的核心思想：与其主动询问每个 socket "你有数据吗？"（poll/select 的方式），
不如告诉内核"这几个 fd 我关心"，然后睡眠——内核会在数据到达时唤醒你并告诉你
具体是哪个 fd 就绪了。

传统方式 (select/poll)：
  应用: "fd1 有数据吗？fd2 有数据吗？fd3 有数据吗？..." (每次循环都要遍历所有 fd)
  → O(n) 时间复杂度

epoll 方式：
  应用: epoll_wait(epfd, events, max, timeout) → "哪些 fd 就绪了？"
  内核: "fd3, fd7, fd15 就绪了"
  → O(1) 只返回就绪的 fd
```

epoll 的三个核心 API 构成了完整的事件循环：`epoll_create` 创建一个 epoll 实例，`epoll_ctl` 向该实例注册/修改/删除要监控的文件描述符，`epoll_wait` 阻塞等待事件发生。

### 2.2 epoll 完整的生命周期——从创建到等待

以下代码展示了 epoll 的完整 API 调用流程——这是 tokio 运行时 I/O 驱动层工作原理的缩影：

```rust
use std::io::{self, Read, Write};
use std::net::TcpListener;
use std::os::unix::io::{AsRawFd, RawFd};
use std::os::raw::c_int;

/// 一个极简的 epoll 事件循环——展示 epoll 的核心机制
/// 这不是生产级代码——tokio 的实现比这复杂上千倍——
/// 但核心流程是一样的
fn demo_epoll() -> io::Result<()> {
    // 步骤 1：创建 epoll 实例
    // epoll_create1(0) 等价于 epoll_create 但没有过时的 size 参数
    let epoll_fd = unsafe { libc::epoll_create1(0) };
    if epoll_fd < 0 {
        return Err(io::Error::last_os_error());
    }

    // 步骤 2：创建一个 TCP 监听 socket 并加入 epoll
    let listener = TcpListener::bind("127.0.0.1:0")?;
    listener.set_nonblocking(true)?;
    let listener_fd = listener.as_raw_fd();

    // 步骤 3：通过 epoll_ctl 注册要监控的事件
    let mut event = libc::epoll_event {
        events: (libc::EPOLLIN | libc::EPOLLET) as u32,  // 可读 + 边缘触发
        u64: listener_fd as u64,                          // 用户数据：保存 fd
    };
    let ret = unsafe {
        libc::epoll_ctl(
            epoll_fd,
            libc::EPOLL_CTL_ADD,  // 操作：添加监控
            listener_fd,
            &mut event,
        )
    };
    if ret < 0 {
        return Err(io::Error::last_os_error());
    }

    // 步骤 4：阻塞等待事件——这就是事件循环的核心
    let mut events: [libc::epoll_event; 64] = unsafe { std::mem::zeroed() };
    loop {
        let nfds = unsafe {
            libc::epoll_wait(
                epoll_fd,
                events.as_mut_ptr(),
                events.len() as c_int,
                -1,  // 超时时间（毫秒）：-1 表示无限等待
            )
        };

        if nfds < 0 {
            return Err(io::Error::last_os_error());
        }

        // 步骤 5：处理就绪的事件
        for i in 0..nfds as usize {
            let fd = events[i].u64 as RawFd;
            let ev = events[i].events as c_int;

            if (ev & libc::EPOLLIN as c_int) != 0 {
                // 这个 fd 可读了
                if fd == listener_fd {
                    // 监听 socket 可读 → 有新连接来了
                    match listener.accept() {
                        Ok((stream, addr)) => {
                            println!("新连接来自: {}", addr);
                            stream.set_nonblocking(true)?;
                            // 将新连接也加入 epoll 监控
                            let stream_fd = stream.as_raw_fd();
                            let mut conn_event = libc::epoll_event {
                                events: (libc::EPOLLIN | libc::EPOLLET) as u32,
                                u64: stream_fd as u64,
                            };
                            unsafe {
                                libc::epoll_ctl(
                                    epoll_fd,
                                    libc::EPOLL_CTL_ADD,
                                    stream_fd,
                                    &mut conn_event,
                                );
                            }
                            // stream 离开作用域...但是 fd 还在 epoll 中
                            // 生产代码中需要妥善管理 fd 的生命周期
                        }
                        Err(ref e) if e.kind() == io::ErrorKind::WouldBlock => {
                            // 非阻塞模式下没有连接可用，正常
                        }
                        Err(e) => eprintln!("accept 错误: {}", e),
                    }
                } else {
                    // 普通连接可读 → 读取数据
                    // 实际代码中需要通过 fd 找到对应的 TcpStream 对象
                    println!("fd {} 可读", fd);
                }
            }
        }
    }
}
```

这个示例揭示了几个重要的设计细节。`EPOLLET`（边缘触发）告诉内核只在状态从"未就绪"变为"就绪"时才通知一次——之后如果不读取数据，内核不会再发送通知。这是高性能服务器的标准选择。`u64` 字段存储用户自定义数据——一般是文件描述符号或指向连接状态结构体的指针。tokio 使用这个字段存储指向任务 Waker 的指针，从而将"epoll 事件"和"唤醒协程"连接起来。

### 2.3 边缘触发 vs 水平触发

epoll 提供两种触发模式，选择哪种取决于你的数据处理策略：

```rust
// 水平触发（Level-Triggered，默认）：
// → 只要 fd 还处于可读状态，每次 epoll_wait 都返回此事件
// → 类似于问"你现在准备好了吗？"——内核每次都说"是的"，直到你读完所有数据
// → 更宽容——即使你没有一次读完所有数据，下次 epoll_wait 还会通知你
// → 适合简单的、逐次处理的应用

// 边缘触发（Edge-Triggered，EPOLLET）：
// → 只有 fd 从"未就绪"变为"就绪"时才通知一次
// → 类似于问"有变化吗？"——内核只在状态变化时说"有变化了"
// → 更高效——减少了无意义的事件通知
// → 要求你在收到通知后一次性读完所有数据（循环 read 直到 EAGAIN）
// → 适合高性能服务器（tokio 使用边缘触发）

// 代码对比：
let events_default = libc::EPOLLIN as u32;           // 水平触发
let events_et = (libc::EPOLLIN | libc::EPOLLET) as u32; // 边缘触发
```

如果你使用边缘触发，但只调用一次 `read` 而没有循环读到底（读到 `EWOULDBLOCK`），剩余数据可能永远不会触发新事件——这是一个经典且隐蔽的 bug。tokio 通过内部的事件循环正确实现了循环读，你作为 tokio 用户不需要关心这些细节。

## 3. io_uring —— 下一代异步 I/O 接口

### 3.1 为什么需要 io_uring

epoll 解决了"批量监控多个 fd"的问题，但它有一个根本性的限制：**它只是通知机制，不是执行机制**。数据实际的读写仍然需要你主动调用 `read()`/`write()`——每次都是一次额外的系统调用。对于极高性能场景（每秒百万级 I/O 操作），这些系统调用的开销累积起来是显著的。

io_uring 是 Linux 5.1 引入的下一代异步 I/O 接口——它的核心创新是使用**共享内存环形缓冲区**替代系统调用进行 I/O 操作的提交和完成通知。

```
epoll 的工作方式：
  1. epoll_wait() → 系统调用（内核态切换）
  2. fd N 就绪了
  3. read(fd, buf) → 又一轮系统调用（再次内核态切换）
  4. 数据到手

io_uring 的工作方式：
  1. 在共享内存的提交队列（SQ）中写入 I/O 请求
  2. 调用一次 io_uring_enter() 提交批量请求（或使用 SQPOLL 模式完全零系统调用）
  3. 内核处理请求，将结果写入共享内存的完成队列（CQ）
  4. 轮询 CQ 获取完成结果——纯用户态操作，无系统调用！
```

### 3.2 两个队列——提交与完成

io_uring 的核心是两个环形缓冲区（ring buffer），它们在内核和用户空间之间共享内存：

```rust
// 概念性展示：io_uring 的双队列设计
// 实际使用需要 io-uring crate——此处展示的是设计原理

// 提交队列条目（Submission Queue Entry，SQE）：
// 你向 SQ 写入"请帮我读取 fd 3 的数据到地址 0x..."
// struct io_uring_sqe {
//     opcode: u8,      // 操作码：IORING_OP_READ, IORING_OP_WRITE, IORING_OP_ACCEPT...
//     flags: u8,
//     ioprio: u16,
//     fd: i32,         // 目标文件描述符
//     off: u64,        // 文件偏移（对于普通文件）
//     addr: u64,       // 缓冲区地址
//     len: u32,        // 长度
//     user_data: u64,  // 用户数据——完成时原样返回，用于关联请求和响应
//     // ... 更多字段
// }

// 完成队列条目（Completion Queue Entry，CQE）：
// 内核向 CQ 写入"fd 3 的读操作完成了，读了 1024 字节"
// struct io_uring_cqe {
//     user_data: u64,  // 从 SQE 原样返回——用于匹配请求
//     res: i32,        // 结果：>= 0 表示成功（字节数），< 0 表示错误（-errno）
//     flags: u32,
// }
```

`user_data` 字段是关键——它让你能够将完成事件与原始请求关联起来。典型的模式是：提交时在 SQE 中填入指向请求上下文结构体的指针作为 `user_data`；完成时从 CQE 中取出 `user_data`，直接访问该请求的完整上下文（缓冲区地址、回调函数等）。

### 3.3 使用 io-uring crate

`io-uring` crate 为 Rust 提供了 io_uring 的安全绑定。以下是一个使用 io_uring 读取文件的示例：

```rust
// Cargo.toml:
// [dependencies]
// io-uring = "0.6"

use io_uring::{opcode, types, IoUring};
use std::fs::File;
use std::os::unix::io::AsRawFd;

fn io_uring_file_read() -> anyhow::Result<()> {
    // 步骤 1：创建 io_uring 实例——参数是 SQ 和 CQ 的条目数
    // entries 必须是 2 的幂
    let mut ring = IoUring::new(64)?;

    // 步骤 2：打开文件（io_uring 对常规文件效果最好）
    let file = File::open("/etc/hostname")?;
    let fd = types::Fd(file.as_raw_fd());

    // 步骤 3：准备缓冲区——必须是 'static 或受 ring 管理的
    let mut buf = vec![0u8; 4096];

    // 步骤 4：构建读取请求的 SQE
    let read_e = opcode::Read::new(fd, buf.as_mut_ptr(), buf.len() as u32)
        .offset(0)                   // 从文件第 0 字节开始读
        .build()
        .user_data(0x42);            // 用户数据——完成时用于识别此请求

    // 步骤 5：将 SQE 放入提交队列
    unsafe {
        ring.submission()
            .push(&read_e)
            .map_err(|_| anyhow::anyhow!("SQ 已满"))?;
    }

    // 步骤 6：提交所有待处理的 SQE 并等待至少一个完成
    ring.submit_and_wait(1)?;

    // 步骤 7：处理完成队列
    let cqe = ring.completion().next().expect("至少应该有一个完成事件");
    assert_eq!(cqe.user_data(), 0x42);

    let bytes_read = cqe.result();
    if bytes_read < 0 {
        anyhow::bail!("读取失败: errno={}", -bytes_read);
    }

    // 步骤 8：读取数据
    let content = std::str::from_utf8(&buf[..bytes_read as usize])?;
    println!("文件内容: {}", content.trim());

    Ok(())
}
```

io_uring 的性能优势在批量 I/O 时最为明显——你可以一次向 SQ 提交数十个 I/O 请求，然后通过一次 `submit_and_wait` 等待它们全部完成。这意味着每批数十个 I/O 操作只需要一次系统调用（而不是数十次），极大地减少了内核态切换开销。

### 3.4 tokio-uring —— 将 io_uring 融入 async 生态

`tokio-uring` crate 将 io_uring 的能力包装为 Rust 的 Future，使其可以自然地集成到 tokio 异步代码中：

```rust
// Cargo.toml:
// [dependencies]
// tokio-uring = "0.4"

// use tokio_uring::fs::File;

// #[tokio::main]
// async fn main() -> anyhow::Result<()> {
//     // tokio-uring 的 File 提供了与 tokio::fs::File 类似的 API
//     // 但底层使用 io_uring 而非线程池
//     let file = File::open("/etc/hostname").await?;
//
//     let buf = vec![0u8; 4096];
//     let (res, buf) = file.read_at(buf, 0).await;
//     let n = res?;
//
//     println!("文件内容: {}", std::str::from_utf8(&buf[..n])?);
//     Ok(())
// }

// 注意：tokio-uring 使用单线程运行时——io_uring 本身是线程不安全的
// 这与 tokio 的多线程工作窃取模型不同
// 因此 tokio-uring 适用于 I/O 密集型而非 CPU 密集型的工作负载
```

io_uring 虽然强大但并非万能——它对常规文件 I/O 最有效（epoll 不支持常规文件的非阻塞 I/O），对 socket I/O 也有优势（减少系统调用次数）。但目前 Rust 生态的主流仍然是 epoll（通过 tokio/mio），io_uring 正在逐步成熟。

## 4. 其他 Linux 特有接口——signalfd 与 memfd

### 4.1 signalfd —— 把信号变成文件描述符

传统的 Unix 信号处理是异步的、重新进入的，对程序状态管理造成了极大的困难。`signalfd` 将信号传递从"异步回调"变为"文件描述符上的可读事件"——这意味着你可以用 epoll 同时监控 I/O 事件和信号：

```rust
use std::io;
use std::os::unix::io::AsRawFd;

/// 使用 signalfd 将 SIGINT 和 SIGTERM 变为可 epoll 的文件描述符
fn setup_signal_handler() -> io::Result<()> {
    // 步骤 1：阻塞默认的信号处理——防止信号被默认行为处理
    let mut sigset = unsafe { std::mem::zeroed::<libc::sigset_t>() };
    unsafe {
        libc::sigaddset(&mut sigset, libc::SIGINT);
        libc::sigaddset(&mut sigset, libc::SIGTERM);
        // 在当前线程中阻塞这些信号
        libc::pthread_sigmask(libc::SIG_BLOCK, &sigset, std::ptr::null_mut());
    }

    // 步骤 2：创建 signalfd
    let sfd = unsafe {
        libc::signalfd(-1, &sigset, libc::SFD_NONBLOCK | libc::SFD_CLOEXEC)
    };
    if sfd < 0 {
        return Err(io::Error::last_os_error());
    }

    // 步骤 3：现在 sfd 可以像普通文件描述符一样使用
    // 你可以把它注册到 epoll 中，当信号到达时，sfd 变为可读
    // 然后 read(sfd, &info, sizeof(signalfd_siginfo)) 读取信号详情
    println!("signalfd 已创建: fd={}", sfd);

    // 在真实应用中，这里会把 sfd 注册到 epoll 或 tokio 的 I/O 驱动中
    // 从而实现在异步上下文中优雅地处理信号
    Ok(())
}
```

`signalfd` 使得"在异步事件循环中优雅关闭"成为可能——你可以将所有 I/O 事件和信号统一处理，而不需要担心在信号处理器中能做什么（异步信号安全函数的限制）。

### 4.2 memfd_create —— 存在于内存中的"文件"

`memfd_create` 创建一个匿名的、存在于内存中的文件——它没有对应的磁盘路径，行为类似于普通文件，但页存储在 RAM 中：

```rust
use std::io::{self, Write, Read, Seek, SeekFrom};
use std::os::unix::io::{FromRawFd, AsRawFd};
use std::fs::File;

/// 使用 memfd_create 创建一个内存文件
fn demo_memfd() -> io::Result<()> {
    // 创建匿名内存文件
    let fd = unsafe {
        libc::memfd_create(
            b"my_shared_buffer\0".as_ptr() as *const libc::c_char,
            libc::MFD_CLOEXEC | libc::MFD_ALLOW_SEALING,
        )
    };

    if fd < 0 {
        return Err(io::Error::last_os_error());
    }

    // 从原始 fd 构建 Rust File——安全抽象！
    let mut file = unsafe { File::from_raw_fd(fd) };

    // 像普通文件一样使用
    file.write_all(b"This is stored purely in memory.\n")?;
    file.set_len(4096)?; // 设置文件大小（分配页）
    file.seek(SeekFrom::Start(0))?;

    let mut content = String::new();
    file.read_to_string(&mut content)?;
    println!("从内存文件读取: {}", content.trim());

    // 用途：
    // 1. IPC——通过 fd 传递给子进程（fork 后 fd 被继承）
    // 2. 沙箱——将数据放入 sealed memfd，防止被修改
    // 3. 匿名共享内存——比 shm_open 更简单的接口

    // 注意：memfd 在文件系统中不可见（没有路径）
    // 对它的所有操作都通过文件描述符进行

    Ok(())
}
```

`memfd_create` 在某些场景下是可以替代临时文件的——如果数据不需要持久化也不需要跨系统共享，使用 memfd 可以避免磁盘 I/O，同时保持标准文件 API 的便利性。它特别适合进程间共享只读数据（通过 `MFD_ALLOW_SEALING` 创建后 seal，防止子进程修改）。

## 5. nix crate —— 类型安全的 Unix API

### 5.1 为什么用 nix 而不是 raw libc

`libc` crate 提供了 C 函数和常量的薄绑定——它几乎是 C 头文件的直译，类型是 C 风格的基础类型。`nix` crate 在此基础上提供了 Rust 习惯用法：类型安全的包装、RAII 资源管理、Result 错误处理。

```rust
// 使用 libc（原始绑定）：
// 不安全、需要手动类型转换、容易遗漏错误检查
unsafe {
    let fd = libc::open(
        b"/tmp/test.txt\0".as_ptr() as *const libc::c_char,
        libc::O_RDONLY
    );
    if fd < 0 {
        let err = std::io::Error::last_os_error();
        // ...
    }
    // 需要对 fd 做 close，容易忘记
}

// 使用 nix（安全包装）：
// 类型安全、自动资源管理、Result 错误处理
// use nix::fcntl::{open, OFlag};
// use nix::sys::stat::Mode;
//
// let fd = open("/tmp/test.txt", OFlag::O_RDONLY, Mode::empty())?;
// fd 的类型实现了 Drop——自动 close
```

nix 通过类型系统的力量消除了整类错误——标志位不能用错误的整数组合（因为它们是枚举而非裸 int），文件描述符自动关闭（通过 RAII），错误通过 `nix::Result` 而非 `errno` 传递。

### 5.2 nix 的生态系统——涵盖哪些领域

nix crate 覆盖了 Unix 系统编程的主要领域。以下是一个展示 nix 多样性的示例集合：

```rust
// nix 提供的安全抽象覆盖了 Unix 系统编程的主要领域
// 这些是 nix 中最常用的几个模块：

// use nix::unistd;           // fork, pipe, close, read, write, sleep
// use nix::fcntl;            // open, fcntl, flock
// use nix::sys::socket;      // socket, bind, listen, accept, send, recv
// use nix::sys::signal;      // sigaction, kill, signal blocking
// use nix::sys::wait;        // waitpid (等待子进程)
// use nix::sys::epoll;       // epoll_create, epoll_ctl, epoll_wait
// use nix::sys::stat;        // stat, fstat, mkdir
// use nix::sys::mman;        // mmap, munmap (内存映射)
// use nix::sys::ioctl;       // ioctl (设备控制)
// use nix::sys::termios;     // 终端控制 (termios)
// use nix::sys::eventfd;     // eventfd (线程/进程间事件通知)
// use nix::sys::timerfd;     // timerfd (基于 fd 的定时器)
// use nix::poll;             // poll (多路复用，类似 select)
// use nix::sched;            // sched_setaffinity (CPU 亲和性)
// use nix::sys::resource;    // getrlimit/setrlimit (资源限制)
```

以下是一个使用 nix 创建 TCP 服务器的简单示例——展示了 nix 的 API 风格：

```rust
// Cargo.toml:
// [dependencies]
// nix = { version = "0.29", features = ["socket", "ioctl"] }

use nix::sys::socket::{
    socket, bind, listen, accept,
    AddressFamily, SockType, SockFlag, SockaddrIn,
};
use nix::unistd::{read, write, close};
use std::os::unix::io::RawFd;

fn nix_tcp_echo_server() -> nix::Result<()> {
    // 创建 socket——参数不再是 raw int，而是类型安全的枚举
    let listen_fd = socket(
        AddressFamily::Inet,    // AF_INET
        SockType::Stream,       // SOCK_STREAM
        SockFlag::empty(),       // flags=0
        None,                    // protocol=0（自动选择）
    )?;

    // 绑定地址
    let addr = SockaddrIn::new(127, 0, 0, 1, 0); // 端口 0 = 让内核分配
    bind(listen_fd, &addr)?;

    // 开始监听
    listen(&listen_fd, 128)?;

    // 获取实际分配的端口（用于日志）
    // 实际应用中通过 getsockname 获取

    // 接受连接
    let conn_fd = accept(listen_fd)?;

    // 读取和回复——nix 的 read/write 返回 Result<usize>
    let mut buf = [0u8; 1024];
    let n = read(conn_fd, &mut buf)?;
    write(conn_fd, &buf[..n])?;

    // fd 可以手动关闭——但如果是生产代码，应该包装进 RAII 结构体
    close(conn_fd)?;
    close(listen_fd)?;

    Ok(())
}
```

注意 `accept(listen_fd)` 不需要接收 `sockaddr` 参数——nix 简化了 API，去掉了 C 接口中那些通常传 NULL 的参数。`bind` 接受 `&SockaddrIn` 而非 `*const sockaddr`——类型安全确保了你不会不小心传入不匹配的地址族。

## 7. 用 Rust 写 Linux 内核模块

### 7.1 Rust for Linux 项目概述

"Rust for Linux"（RFL）是一个社区驱动的项目，目标是将 Rust 作为 Linux 内核的第二种官方语言。自 Linux 6.1 起，Rust 的基础设施已合并入主线内核。这意味着你可以用 Rust 编写内核模块——在内核空间运行、直接操作硬件、没有标准库、只有 core 和 alloc 以及内核提供的 crate。

```
Rust 内核模块的基本架构：

你的 Rust 代码
  → kernel crate（内核提供的 API：文件系统、设备驱动、网络等）
    → bindings crate（自动生成的 C 内核 API 的 unsafe 绑定）
      → C 内核函数

编译时：
  - 使用内核的 Makefile 系统 + Rust 编译器
  - target: x86_64-unknown-none（没有操作系统、没有 libc）
  - 使用内核提供的 allocator 替代 std 的全局分配器
```

### 7.2 一个最小的 Rust 内核模块

以下是一个"hello world"级别的 Rust 内核模块——编译后可以通过 `insmod` 加载到 Linux 内核中：

```rust
// 这不是可以直接在你机器上编译运行的代码——
// 需要在内核源码树中通过内核的构建系统编译
// 此处展示结构和概念

//! SPDX-License-Identifier: GPL-2.0
//! 一个最简的 Rust 内核模块

use kernel::prelude::*;

// 声明这是一个内核模块
module! {
    type: RustHelloModule,
    name: "rust_hello",
    author: "Rust Learner",
    description: "A minimal Rust kernel module",
    license: "GPL",
}

/// 模块的主结构体——代表一个已加载的模块实例
struct RustHelloModule;

impl kernel::Module for RustHelloModule {
    /// 模块初始化函数——在 insmod 时被调用
    fn init(_module: &'static ThisModule) -> kernel::error::Result<Self> {
        // pr_info! 是内核的日志宏，输出到 dmesg
        pr_info!("🦀 Rust 内核模块已加载！Hello from kernel space!\n");

        Ok(RustHelloModule)
    }
}

/// Drop trait 充当模块的退出函数——在 rmmod 时被调用
impl Drop for RustHelloModule {
    fn drop(&mut self) {
        pr_info!("Rust 内核模块正在卸载。Goodbye from kernel space!\n");
    }
}
```

这个模块的代码量极小，但揭示了许多核心设计：`kernel::Module` trait 定义了模块的初始化契约（对应 C 的 `module_init`）；`Drop` 实现作为模块退出清理函数（对应 C 的 `module_exit`）；`pr_info!` 是内核日志系统（输出到 `dmesg` 或 `journalctl -k`）。

### 7.3 内核模块中的限制——没有 std，只有 core 和 alloc

在内核中运行 Rust 代码与用户空间有本质区别：

```rust
// 内核模块中可用的（通过 kernel crate + core + alloc）：
// use core::mem;          // ✅ core crate 完整可用
// use alloc::vec::Vec;    // ✅ alloc crate 可用（使用内核分配器）
// use alloc::boxed::Box;  // ✅ Box 可用
// use alloc::string::String; // ✅ String 可用

// 内核模块中不可用的（std 依赖操作系统抽象）：
// use std::fs::File;      // ❌ 没有文件系统抽象（可以用 kernel::fs 替代）
// use std::net::TcpStream; // ❌ 没有用户空间网络栈
// use std::thread;         // ❌ 没有用户空间线程（可以用内核线程替代）
// use std::time::Instant;  // ❌ 没有用户空间时钟（可以用 kernel::jiffies 替代）
// println!("");           // ❌ 没有 stdout（用 pr_info! 替代）
```

内核 Rust 代码使用 `core` crate（语言的核心功能，不依赖分配器或操作系统）和 `alloc` crate（需要分配器，内核提供了自己的分配器实现），但完全没有 `std`。这是理解 Rust 模块化设计的绝佳视角：core（零依赖）> alloc（需分配器）> std（需操作系统抽象）。

### 7.4 实际现状——从实验到生产

Rust for Linux 项目目前处于活跃开发阶段——Android 团队已经在使用 Rust 内核模块保护 Binder 驱动程序免受内存安全漏洞的影响。Asahi Linux（Apple Silicon Mac 上运行的 Linux）的 GPU 驱动程序完全用 Rust 编写。然而，对于大多数开发者来说，学习 Rust 内核模块的意义在于**理解操作系统内核是如何工作的**——即使你永远不会写一行内核代码，理解你的程序在内核层面上发生了什么也是成为一名系统级程序员的关键。

### 8.2 为什么这很重要

理解从顶层到底层的完整路径，让你在调试时能够"穿越抽象层级"进行推理：

当你的 `tokio::spawn` 任务"卡住了"：
- 第一层（Rust 层面）：Future 是否返回了 Pending 但没有注册 Waker？
- 第二层（运行时层面）：tokio 的 I/O 驱动是否正确地调用了 epoll_ctl？
- 第三层（系统调用层面）：epoll_wait 返回了什么事件？fd 是否被正确设置？
- 第四层（内核层面）：内核网络栈是否收到了数据包？中断是否触发？

每一层都有对应的工具和方法——但前提是你知道这些层级存在。

### 练习 2：分析 tokio 与 epoll 的交互

阅读以下描述并回答问题：

tokio 在 Linux 上使用 epoll 作为其 I/O 驱动层。当一个 `TcpStream::read()` 在 tokio 中被 `.await` 时，大致流程如下：

1. Future 的 `poll()` 方法被调用
2. 尝试非阻塞读取——如果返回 `WOULDBLOCK`，说明数据尚未到达
3. 将当前任务的 Waker 注册到 mio（mio 是 tokio 依赖的事件循环库）的内部数据结构中
4. 向 mio 注册对该 socket 的 "可读" 事件感兴趣
5. mio 内部调用 `epoll_ctl(EPOLL_CTL_ADD, socket_fd, EPOLLIN | EPOLLET)`
6. Future 返回 `Poll::Pending`
7. tokio 运行时调用 `epoll_wait()` 等待事件
8. 数据到达 → `epoll_wait` 返回 → mio 找到对应的 Waker → 调用 `waker.wake()`
9. tokio 将任务重新放入调度队列 → 再次 `poll()` → 这次 `read()` 成功

问题：
- 为什么步骤 2 中尝试非阻塞读取不会真正消耗数据？（提示：`WOULDBLOCK` 的性质）
- 为什么使用边缘触发（EPOLLET）而非水平触发？
- 如果 socket 在步骤 4 和步骤 5 之间（注册事件前）收到了数据——会出现什么问题？如何解决？

```rust
use std::arch::asm;

/// getpid 系统调用（x86-64 Linux）
/// 系统调用号 = 39，无参数，返回进程 ID
fn raw_getpid() -> i32 {
    let pid: i32;
    unsafe {
        asm!(
            "syscall",
            inlateout("rax") 39isize => pid,  // 39 = SYS_getpid
            // 没有参数寄存器需要设置
            lateout("rcx") _,
            lateout("r11") _,
            options(nostack),
        );
    }
    pid
}

/// exit 系统调用（x86-64 Linux）
/// 系统调用号 = 60，参数：退出码
/// 注意：这是个发散函数（!）——不会返回
fn raw_exit(code: i32) -> ! {
    unsafe {
        asm!(
            "syscall",
            in("rax") 60isize,   // 60 = SYS_exit
            in("rdi") code,      // 第一个参数：退出码
            // 不需要 lateout——这个函数不返回
            options(nostack, noreturn),
        );
    }
}

/// write 系统调用（x86-64 Linux）
/// 系统调用号 = 1，参数：fd, buf, count
fn raw_write(fd: i32, buf: &[u8]) -> isize {
    let result: isize;
    unsafe {
        asm!(
            "syscall",
            inlateout("rax") 1isize => result,   // 正 result 表示成功，负 result 表示 -errno
            in("rdi") fd,
            in("rsi") buf.as_ptr(),
            in("rdx") buf.len(),
            lateout("rcx") _,
            lateout("r11") _,
            options(nostack),
        );
    }
    result
}

// 使用示例：
// fn main() {
//     let pid = raw_getpid();
//     let msg = format!("Process {}: hello via raw syscall!\n", pid);
//     raw_write(1, msg.as_bytes());
//     raw_exit(0);
// }
```

关键要点：`inlateout` 用于既是输入又是输出的寄存器——这里 `rax` 输入是系统调用号，输出是返回值。传递切片引用 `buf: &[u8]` 时，其指针和长度被分隔装入 `rsi` 和 `rdx`。

### 练习 2 答案

```
问题 1：步骤 2 中非阻塞读取为什么不会消耗数据？

非阻塞读操作 read(fd, buf, len) 在数据未就绪时返回错误 EWOULDBLOCK
（或 EAGAIN），此时 buf 的内容未被修改——数据仍然在 socket 的内核缓冲区中。
如果数据已经就绪，那么非阻塞读会正常返回数据——
但这意味着 Future 可以直接返回 Ready，不需要走 epoll 路径。
所以这个"先试读一下"是一种优化：如果数据已经在，就省去 epoll 的往返。

问题 2：为什么使用边缘触发（EPOLLET）而非水平触发？

边缘触发下，内核只在"未就绪 → 就绪"的转换时刻通知一次。
水平触发下，只要就绪就一直通知——每次 epoll_wait 都返回这个 fd。

边缘触发的优势：
- 减少事件通知次数（每个"数据到达"事件只通知一次）
- tokio 在收到通知后循环读取直到 EAGAIN，确保一次事件就消费完所有数据
- 减少不必要的 epoll_wait 返回，提高 CPU 效率

水平触发的问题：
- 如果收到通知但没读完数据，每次 epoll_wait 都再次返回此事件
- 浪费 CPU 周期——你已经在 epoll_wait 的返回路径上，反复返回一个你不打算读取的 fd

问题 3：注册事件前收到数据的竞争问题（race condition）。

时间线：
  T1: 尝试非阻塞读 → EWOULDBLOCK（数据未到达）
  T2: 数据到达！socket 变为可读
  T3: epoll_ctl(EPOLL_CTL_ADD, fd, EPOLLIN | EPOLLET)
      → 边缘触发：此时 fd 已经是可读状态，但尚未"转变"——不会触发事件
  T4: epoll_wait() → 可能永远等不到此 fd 的事件！

解决方案：在 epoll_ctl 注册之后，再尝试一次非阻塞读（double check）——
这就是课程中提到的"注册后检查"（register-then-check）模式。

修正流程：
  1. 尝试非阻塞读 → EWOULDBLOCK
  2. epoll_ctl(ADD, fd, EPOLLIN | EPOLLET) // 先注册
  3. 再次尝试非阻塞读 // 再检查
  4a. 如果读到数据 → 直接 Ready，从 epoll 中移除
  4b. 如果 EWOULDBLOCK → 返回 Pending，epoll_wait 会正确通知下一次事件
```

以上 Cargo.toml 声明了对 nix crate 的依赖，启用了 socket、epoll 和 ioctl 相关的 feature。配合以下 Rust 源码即可编译运行。

```rust
// src/main.rs
use nix::errno::Errno;
use nix::fcntl::{fcntl, FcntlArg, OFlag};
use nix::sys::epoll::{
    epoll_create1, epoll_ctl, epoll_wait,
    EpollCreateFlags, EpollEvent, EpollFlags, EpollOp,
};
use nix::sys::socket::{
    accept, bind, listen, setsockopt, socket,
    AddressFamily, SockType, SockFlag, SockaddrIn, SockaddrStorage,
    SetSockOpt,
};
use nix::unistd::{read, write, close};
use std::collections::HashMap;
use std::io::{self, Write};
use std::os::unix::io::RawFd;

const MAX_EVENTS: usize = 64;
const BUFFER_SIZE: usize = 4096;

fn set_nonblocking(fd: RawFd) -> nix::Result<()> {
    let flags = fcntl(fd, FcntlArg::F_GETFL)?;
    let mut oflags = OFlag::from_bits_truncate(flags);
    oflags.insert(OFlag::O_NONBLOCK);
    fcntl(fd, FcntlArg::F_SETFL(oflags))?;
    Ok(())
}

fn main() -> nix::Result<()> {
    // 步骤 1：创建 epoll 实例
    let epoll_fd = epoll_create1(EpollCreateFlags::EPOLL_CLOEXEC)?;

    // 步骤 2：创建监听 socket
    let listen_fd = socket(
        AddressFamily::Inet,
        SockType::Stream,
        SockFlag::empty(),
        None,
    )?;
    setsockopt(&listen_fd, SetSockOpt::ReuseAddr, &true)?;
    set_nonblocking(listen_fd)?;

    let addr = SockaddrIn::new(127, 0, 0, 1, 8080);
    bind(listen_fd, &addr)?;
    listen(&listen_fd, 128)?;
    println!("服务器启动在 127.0.0.1:8080");

    // 步骤 3：将监听 socket 注册到 epoll
    let mut listen_event = EpollEvent::new(
        EpollFlags::EPOLLIN | EpollFlags::EPOLLET,
        listen_fd as u64,
    );
    epoll_ctl(epoll_fd, EpollOp::EpollCtlAdd, listen_fd, Some(&mut listen_event))?;

    // 步骤 4：事件循环
    let mut events = vec![EpollEvent::empty(); MAX_EVENTS];
    // 维护活跃连接——用于回显数据后找到对应的 fd
    let mut connections: HashMap<RawFd, ()> = HashMap::new();

    loop {
        // 阻塞等待事件（timeout = -1 = 无限等待）
        let nfds = epoll_wait(epoll_fd, &mut events, -1)?;

        for i in 0..nfds {
            let event_fd = events[i].data() as RawFd;
            let event_flags = events[i].events();

            // 错误或挂断
            if event_flags.intersects(
                EpollFlags::EPOLLERR | EpollFlags::EPOLLHUP
            ) {
                println!("连接 {} 错误或断开", event_fd);
                epoll_ctl(epoll_fd, EpollOp::EpollCtlDel, event_fd, None)
                    .or_else(|e| {
                        if e == Errno::EBADF {
                            Ok(()) // fd 已经被关闭了，忽略
                        } else {
                            Err(e)
                        }
                    })?;
                connections.remove(&event_fd);
                let _ = close(event_fd);
                continue;
            }

            if event_fd == listen_fd {
                // 监听 socket 就绪 → 接受新连接
                // 边缘触发模式：循环 accept 直到 EAGAIN
                loop {
                    match accept(listen_fd) {
                        Ok((conn_fd, _addr)) => {
                            set_nonblocking(conn_fd)?;
                            let mut conn_event = EpollEvent::new(
                                EpollFlags::EPOLLIN | EpollFlags::EPOLLET,
                                conn_fd as u64,
                            );
                            epoll_ctl(
                                epoll_fd,
                                EpollOp::EpollCtlAdd,
                                conn_fd,
                                Some(&mut conn_event),
                            )?;
                            connections.insert(conn_fd, ());
                            println!("新连接: fd={}", conn_fd);
                        }
                        Err(Errno::EAGAIN) => {
                            // 没有更多连接了——跳出循环
                            break;
                        }
                        Err(e) => {
                            eprintln!("accept 错误: {}", e);
                            break;
                        }
                    }
                }
            } else if event_flags.contains(EpollFlags::EPOLLIN) {
                // 客户端数据可读 → 边缘触发模式：循环读直到 EAGAIN
                let mut closed = false;
                loop {
                    let mut buf = [0u8; BUFFER_SIZE];
                    match read(event_fd.as_raw_fd(), &mut buf) {
                        Ok(0) => {
                            // 读到 0 字节 → 客户端关闭了连接
                            println!("客户端 {} 关闭连接", event_fd);
                            closed = true;
                            break;
                        }
                        Ok(n) => {
                            // 回显：把读到的数据原样写回去
                            // 生产代码中 write 也应该处理 EAGAIN
                            match write(event_fd.as_raw_fd(), &buf[..n]) {
                                Ok(written) if written < n => {
                                    eprintln!("部分写入: {}/{} 字节", written, n);
                                }
                                Err(e) => {
                                    eprintln!("write 错误: {}", e);
                                    closed = true;
                                    break;
                                }
                                _ => {}
                            }
                        }
                        Err(Errno::EAGAIN) => {
                            // 所有可用数据已读完——正常退出循环
                            break;
                        }
                        Err(e) => {
                            eprintln!("read 错误: {}", e);
                            closed = true;
                            break;
                        }
                    }
                }

                if closed {
                    // 清理断开的连接
                    epoll_ctl(epoll_fd, EpollOp::EpollCtlDel, event_fd, None)
                        .or_else(|e| if e == Errno::EBADF { Ok(()) } else { Err(e) })?;
                    connections.remove(&event_fd);
                    let _ = close(event_fd);
                }
            }
        }
    }
}
```

这个回显服务器的核心设计在于：边缘触发模式下，accept 和 read 都在循环中执行直到遇到 `EAGAIN`——确保一次事件通知消费完所有可用资源。不这样做的后果是：剩余可用数据可能永远不会触发新事件（因为边缘触发需要"未就绪到就绪"的状态转换，但 fd 已经处于"就绪"状态），数据就会"滞留"在内核缓冲区中。
