# 4.5 编译器内建函数 —— 从 LLVM IR 直通 Rust 的秘密通道

> 编译器内建函数（Compiler Intrinsics）是 Rust 标准库地板之下的地下室。当 `std::ptr::copy` 还不够快时，当分支预测需要手动介入时，当数学运算需要直接映射为 CPU 指令时——intrinsics 是你最后的加速手段。它们不是 Rust 的函数——它们是 LLVM 的函数。每一个 intrinsic 调用都被编译器翻译为 LLVM IR 中的对应指令，绕过类型检查、借用规则、甚至函数调用的开销。但这份能力是有代价的：绝大多数 intrinsics 被锁定在 nightly 频道中，因为它们暴露了编译器内部的不稳定接口。理解 intrinsics，就是理解 Rust 编译器"向下"的边界。

---

## 1. 什么是编译器内建函数

### 1.1 定义与本质

编译器内建函数（intrinsics）是**由编译器而非标准库提供的函数**。当你调用 `std::intrinsics::sqrtf64(2.0)` 时，rustc 不会生成一个函数调用指令——它会将这次调用替换为 LLVM 的 `@llvm.sqrt.f64` intrinsic，LLVM 后端再将其映射为目标平台的 `sqrtsd`（x86_64）或 `fsqrt`（ARM64）指令。

这不是通过"优化"实现的——intrinsic 的整个生命周期在编译器中就结束了。前端识别 intrinsic 名称，生成 LLVM IR 中的对应节点；中端（优化管道）可能将其内联、常量折叠或重排；后端生成目标机器码。整个过程没有函数调用开销，甚至没有跳转指令。

### 1.2 Intrinsics 与普通函数的核心区别

普通 Rust 函数和编译器 intrinsics 之间存在根本性的语义鸿沟。以下对比表列出了关键差异：

对于调用开销，普通函数在未内联时有 `call` 指令开销（保存/恢复寄存器、栈帧操作），而 intrinsic 被编译器识别后直接替换为对应的 LLVM 操作，零开销。

对于优化可见性，普通函数体在跨 crate 调用时对优化器不透明（除非 LTO），而 intrinsic 对优化器完全透明——LLVM 知道它的精确语义，可以自由地常量折叠、重排或消除。

对于类型检查，普通函数受到 Rust 类型系统的完整保护，而 intrinsic 可能绕过类型检查——例如 `std::intrinsics::transmute` 允许任意的类型转换。

对于稳定性，普通函数在稳定 Rust 中可用（除非标记为 `#[unstable]`），而大多数 intrinsics 被锁定在 `#[rustc_intrinsic]` 下，需要 nightly 编译器。

### 1.3 如何探索 intrinsics 列表

`std::intrinsics` 模块包含了 Rust 编译器支持的所有内置 intrinsic 函数。在 nightly Rust 中，你可以直接在代码中引用它们（需要 `#![feature(core_intrinsics)]`）。以下代码展示了如何列出一些常见的 intrinsics：

```rust
// 注意：此代码需要 nightly Rust
// #![feature(core_intrinsics)]

// 常见 intrinsic 的分类概览：
// 
// === 数学运算 ===
// std::intrinsics::sqrtf64(x: f64) -> f64      —— 平方根
// std::intrinsics::powf64(a: f64, x: f64) -> f64 —— 幂运算
// std::intrinsics::fabsf64(x: f64) -> f64       —— 绝对值
// std::intrinsics::copysignf64(x: f64, y: f64) -> f64 —— 复制符号
//
// === 类型信息 ===
// std::intrinsics::size_of::<T>() -> usize      —— 类型大小（编译时常量）
// std::intrinsics::min_align_of::<T>() -> usize —— 最小对齐
// std::intrinsics::type_name::<T>() -> &'static str —— 类型名称（用于反射）
//
// === 内存操作 ===
// std::intrinsics::copy::<T>(src: *const T, dst: *mut T, count: usize)
// std::intrinsics::copy_nonoverlapping::<T>(src: *const T, dst: *mut T, count: usize)
// std::intrinsics::write_bytes::<T>(dst: *mut T, val: u8, count: usize)
//
// === 类型转换 ===
// std::intrinsics::transmute::<T, U>(value: T) -> U  —— 位级别类型转换
//
// === 控制流 ===
// std::intrinsics::unreachable() -> !               —— 标记不可达代码
// std::intrinsics::assume(condition: bool)          —— 声明条件恒为真
```

每个 intrinsic 的名称和签名在 `std::intrinsics` 的 Rustdoc 中都有记录。但请注意——这个模块的文档明确标注为"编译器内部接口，不保证稳定"。

## 2. 分支预测 Hint —— likely 与 unlikely

### 2.1 什么是分支预测

现代 CPU 使用分支预测器来猜测条件跳转的方向。当预测正确时，指令流水线保持满载；当预测错误时，流水线必须清空并重新填充——在深度流水化的 CPU 上，这可能导致 10-20 个时钟周期的性能损失。`likely` 和 `unlikely` intrinsics 允许你向编译器和 CPU 提示哪个分支更有可能被执行。

```rust
// 需要 nightly Rust: #![feature(core_intrinsics)]
#![feature(core_intrinsics)]

use std::intrinsics::{likely, unlikely};

/// 错误处理路径——几乎从不执行
/// unlikely 告诉编译器和 CPU："这个分支几乎不可能进入"
fn process_value(value: i32) -> Result<i32, &'static str> {
    if unlikely(value < 0) {
        // 编译器和 CPU 预期不会进入这个分支
        // 编译器会将这个分支的代码移到函数的"冷"区域
        // CPU 不会为这个分支预热流水线
        return Err("negative value");
    }
    // 主路径——编译器将这里设为"热"代码
    // 指令流水线会持续预取这里的指令
    Ok(value * 2)
}

/// 常见情况提示
fn common_case() {
    let x = 42;
    if likely(x > 0) {
        // 编译器预期这个分支几乎总是被执行
        // main path 的代码被紧密排列在一起，提高 I-cache 命中率
        println!("positive");
    } else {
        println!("non-positive");
    }
}
```

`likely` 和 `unlikely` 在 LLVM IR 中被翻译为 `@llvm.expect.i1` intrinsic，后者在大多数目标平台上会生成相应的分支预测提示。在 x86_64 上，这些提示通过指令编码中的前缀字节（prefix）实现——它们不改变程序的正确性，只影响 CPU 的分支预测器策略和编译器的代码布局决策。

### 2.2 CPU 的分支预测器如何工作

分支预测器是现代 CPU 中最精密的组件之一。它维护着一个分支目标缓冲区（Branch Target Buffer, BTB），记录每条分支指令的历史跳转方向。对于有规律的分支（如循环计数器），预测准确率可以超过 99%；对于无规律的分支（如错误处理），预测器只能猜测。

`likely` 和 `unlikely` 的作用是**初始化预测器的状态**——它们告诉预测器"这个分支的默认预测方向是什么"。对于冷路径（几乎不执行的分支），使用 `unlikely` 避免预测器浪费资源去跟踪它的历史。

### 2.3 稳定替代方案

在稳定 Rust 中，虽然没有 `likely`/`unlikely` intrinsics，但你可以通过以下方式暗示分支倾向：

```rust
/// 在稳定 Rust 中表达分支预测倾向
/// 方法：将"热"分支放在 if 条件的前面
/// 
/// 编译器通常会假设 if 的第一个分支更容易执行
/// ——这不是硬性保证，但在大多数优化场景中是有效的启发式
fn stable_likely_example(value: i32) -> i32 {
    // 将常见情况放在 if 的第一个分支
    if value >= 0 {
        value * 2
    } else {
        -1
    }
}

/// 使用 match 表达多路分支——编译器对第一个臂有偏向
fn stable_match_example(code: u8) -> &'static str {
    match code {
        200 => "OK",           // 最常见的状态码放在最前面
        404 => "Not Found",
        500 => "Server Error",
        _ => "Unknown",
    }
}
```

在稳定 Rust 中，你也可以使用 `#[cold]` 属性标记几乎不执行的函数——编译器会将标记为 `#[cold]` 的函数体移到代码段的冷区域，并避免为其优化。

## 3. 编译时类型信息 —— size_of / min_align_of / type_name

### 3.1 size_of 与 min_align_of

`size_of::<T>()` 和 `min_align_of::<T>()` 返回 `T` 的大小和对齐要求。它们是**编译时常量**——编译器在编译阶段就知道答案，因此调用它们不会产生任何运行时开销：

```rust
// 需要 nightly: #![feature(core_intrinsics)]
#![feature(core_intrinsics)]

use std::intrinsics;

fn demonstrate_type_info() {
    // size_of 返回类型的内存大小（以字节为单位）
    let u32_size = intrinsics::size_of::<u32>();       // 编译时常量：4
    let u64_size = intrinsics::size_of::<u64>();       // 8
    let usize_size = intrinsics::size_of::<usize>();   // 8 (64 位平台) 或 4 (32 位平台)
    
    // 结构体的大小受 padding 和 alignment 影响
    struct MyStruct {
        a: u8,
        b: u32,
    }
    let struct_size = intrinsics::size_of::<MyStruct>(); // 8 (1 字节 + 3 字节 padding + 4 字节)
    
    // min_align_of 返回最小对齐要求
    let u32_align = intrinsics::min_align_of::<u32>(); // 4
    let u64_align = intrinsics::min_align_of::<u64>(); // 8
    
    println!("u32: size={}, align={}", u32_size, u32_align);
    println!("u64: size={}, align={}", u64_size, u64_align);
}
```

在稳定 Rust 中，这些 intrinsic 被封装在 `std::mem::size_of::<T>()` 和 `std::mem::align_of::<T>()` 中——它们的实现本质上就是调用对应的 intrinsic。因此，在稳定代码中使用标准库版本即可获得完全相同的效果。

### 3.2 type_name —— 编译期反射

`type_name::<T>()` 返回一个 `&'static str`，包含 `T` 的类型名称。这是 Rust 中为数不多的"反射"能力之一，完全在编译时完成：

```rust
// 需要 nightly: #![feature(core_intrinsics)]
#![feature(core_intrinsics)]

use std::intrinsics::type_name;

fn explore_type_names() {
    // type_name 返回的字符串是编译时常量
    // 它包含完整的类型路径信息
    
    let int_name = type_name::<i32>();
    println!("type of i32: {}", int_name);  // "i32"
    
    let vec_name = type_name::<Vec<String>>();
    println!("type of Vec<String>: {}", vec_name);  // "alloc::vec::Vec<alloc::string::String>"
    
    let slice_name = type_name::<&[u8]>();
    println!("type of &[u8]: {}", slice_name);  // "&[u8]"
    
    // type_name 在泛型代码中特别有用——用于调试输出
    fn debug_type<T>() -> String {
        format!("T = {}", type_name::<T>())
    }
    
    println!("{}", debug_type::<f64>());  // "T = f64"
}
```

在稳定 Rust 中，`std::any::type_name::<T>()` 提供了等价的功能——它是对 `intrinsics::type_name` 的稳定封装。因此，你不需要 nightly Rust 就能使用编译期类型名反射。

## 4. needs_drop —— 类型是否需要释放资源

### 4.1 语义

`needs_drop::<T>()` 返回一个 `bool`，指示类型 `T` 是否实现了 `Drop` 或包含实现了 `Drop` 的字段。它是一个**编译时常量**——编译器在编译时就知道答案。

```rust
// 需要 nightly: #![feature(core_intrinsics)]
#![feature(core_intrinsics)]

use std::intrinsics::needs_drop;

fn demonstrate_needs_drop() {
    // 基本类型不需要 drop
    assert!(!needs_drop::<i32>());
    assert!(!needs_drop::<u64>());
    assert!(!needs_drop::<f64>());
    
    // 包含堆分配的类型需要 drop
    assert!(needs_drop::<String>());
    assert!(needs_drop::<Vec<i32>>());
    
    // 引用不需要 drop（它们不拥有资源）
    assert!(!needs_drop::<&String>());
    assert!(!needs_drop::<&mut Vec<u8>>());
    
    // 手动实现 Drop 的类型
    struct HasDrop;
    impl Drop for HasDrop {
        fn drop(&mut self) {
            println!("dropping!");
        }
    }
    assert!(needs_drop::<HasDrop>());
}
```

### 4.2 needs_drop 的实际用途

`needs_drop` 主要用于编写高性能的容器和内存管理代码。当一个数据结构需要批量构造和销毁元素时，知道元素是否需要 drop 可以让你跳过不必要的释放逻辑：

```rust
// 概念性示例：一个自定义的 Vec，在清空时使用 needs_drop 优化
// 这个例子展示 needs_drop 的使用场景——实际代码会更复杂

#![feature(core_intrinsics)]

use std::intrinsics::needs_drop;
use std::alloc::{alloc, dealloc, Layout};
use std::ptr;

struct OptimizedVec<T> {
    ptr: *mut T,
    len: usize,
    cap: usize,
}

impl<T> OptimizedVec<T> {
    /// 清空 Vec 中的所有元素
    /// 根据 needs_drop 选择不同的实现路径——这是编译期计算
    fn clear(&mut self) {
        if needs_drop::<T>() {
            // 类型需要 drop：逐个调用 drop
            // 编译器知道这个分支的存在，但不会编译这个分支的代码
            // （如果 T 不需要 drop 的话）
            // 这取决于单态化——每个 T 的实例化会产生独立的代码
            for i in 0..self.len {
                unsafe {
                    ptr::drop_in_place(self.ptr.add(i));
                }
            }
        }
        // 如果 T 不需要 drop，上面的 if 分支被编译器消除
        // 只留下设置长度为 0 的操作
        self.len = 0;
    }
}
```

由于 `needs_drop` 的返回值是编译时常量，编译器可以将 `if needs_drop::<T>()` 优化为死代码消除——如果 `T` 不需要 drop，整个 `if` 块都会被移除。这种"零成本抽象"是 Rust 容器类高性能的关键因素之一。

在稳定 Rust 中，`std::mem::needs_drop::<T>()` 提供了等价的功能，底层同样调用 `intrinsics::needs_drop`。

## 5. copy 与 copy_nonoverlapping —— 内存操作的本质

### 5.1 底层语义

`copy::<T>(src: *const T, dst: *mut T, count: usize)` 是 `std::ptr::copy` 的底层实现。它将 `count` 个 `T` 类型元素从 `src` 复制到 `dst`，**允许**源和目标重叠。在 LLVM IR 中，它被翻译为 `@llvm.memmove`——一个智能的内存移动指令，能自动处理重叠情况。

`copy_nonoverlapping::<T>(src: *const T, dst: *mut T, count: usize)` 是 `std::ptr::copy_nonoverlapping` 的底层实现。它同样复制 `count` 个元素，但**要求**源和目标不重叠。在 LLVM IR 中，它被翻译为 `@llvm.memcpy`——一个更高效的内存复制指令，但假设没有重叠。

```rust
// 需要 nightly: #![feature(core_intrinsics)]
#![feature(core_intrinsics)]

use std::intrinsics;

/// 演示 copy 和 copy_nonoverlapping 的区别
fn demonstrate_copy_intrinsics() {
    let src = [1u32, 2, 3, 4, 5];
    let mut dst = [0u32; 5];
    
    // unsafe: 需要确保 src 和 dst 都不为 null，
    // count 不超过分配的内存区域
    unsafe {
        // copy: 允许重叠——编译器使用 memmove
        // 适用于在数组内部滑动元素
        intrinsics::copy(src.as_ptr(), dst.as_mut_ptr(), 5);
    }
    assert_eq!(dst, [1, 2, 3, 4, 5]);
    
    let mut buf = [1u32, 2, 3, 4, 5];
    unsafe {
        // copy_nonoverlapping: 禁止重叠——编译器使用更快的 memcpy
        // 在 buf 内部移动元素不能使用这个方法！
        intrinsics::copy_nonoverlapping(
            buf.as_ptr().add(2),  // src: [3, 4, 5, ...]
            buf.as_mut_ptr(),      // dst: 从 buf[0] 开始
            2,                      // count: 2 个元素
        );
        // 这没问题——src[3,4] 和 dst[buf[0], buf[1]] 不重叠
    }
    assert_eq!(buf[0..2], [3, 4]);
}
```

### 5.2 copy 与 copy_nonoverlapping 的性能差异

`copy_nonoverlapping`（memcpy）比 `copy`（memmove）更快的原因是：

`memmove` 必须在运行时检查源和目标是否重叠。如果重叠且目标在源之后，`memmove` 必须**从后向前**复制以避免覆盖。这个运行时检查额外消耗几个时钟周期。

`memcpy` 不需要此检查——它直接使用最快的复制策略（通常是 SIMD 向量化复制，利用 SSE/AVX 寄存器一次复制 16/32/64 字节）。对于大块内存复制，这个差异可以非常显著。

在稳定 Rust 中，你应该总是通过 `std::ptr::copy` 和 `std::ptr::copy_nonoverlapping` 来使用这些功能——它们是稳定的封装，不需要 nightly。

## 6. unreachable —— 标记不可能到达的代码

### 6.1 基本用法

`unreachable` intrinsic 告诉编译器"这段代码永远不会被执行"。这是一个**致命的**承诺——如果程序运行到了 `unreachable()` 的位置，行为是**立即可触发未定义行为**（UB）的。

```rust
// 需要 nightly: #![feature(core_intrinsics)]
#![feature(core_intrinsics)]

use std::intrinsics::unreachable;

/// 使用 unreachable 消除不可能的匹配分支
fn classify_number(n: u8) -> &'static str {
    match n {
        0..=9   => "single digit",
        10..=99 => "double digit",
        100..=255 => "triple digit (max)",
        // 理论上 u8 只有 0..=255，上面的分支已经完整覆盖
        // 但编译器不知道这一点——所以要求一个通配分支
        // unreachable 告诉编译器："相信我，这个分支不可能到达"
        // 编译器会把 _ => 分支完全消除，不生成任何代码
        _ => unsafe { unreachable() },
    }
}

/// unreachable 对性能的影响：消除不可能的检查
fn get_elem_unchecked<T>(slice: &[T], index: usize) -> &T {
    // 假设调用者已经验证了 index 的有效性
    // 使用 unreachable 告诉编译器跳过边界检查
    
    // 这只是概念性的展示——实际应使用 get_unchecked
    // 或者依赖已有验证，让编译器优化掉重复检查
    
    // 下面的模式在实际项目中不推荐，因为如果 index 越界，
    // unreachable 会导致 UB
    if index < slice.len() {
        &slice[index]
    } else {
        unsafe { unreachable() }
    }
}
```

### 6.2 unreachable 与优化

`unreachable` 的价值在于它允许编译器进行更激进的优化。当编译器知道某段代码不可达时，它可以：

消除对不可达代码的条件检查。例如在 `match` 中，如果编译器知道某个变体永远不会出现，它不需要为那个变体生成任何分支指令。

将不可达分支后的代码视为可更自由地重排和优化。编译器可以假设控制流永远不会进入不可达代码。

在 LLVM IR 中，`unreachable` 被翻译为 `unreachable` 指令——LLVM 内部的一个标记，表示执行到此处是未定义行为。LLVM 的优化器会利用这个信息进行死代码消除和条件简化。

### 6.3 稳定 Rust 中的替代方案

在稳定 Rust 中，有几种方式表达"不可达"语义，每种有不同的安全级别：

`unreachable!()` 宏是**安全**的替代方案——如果被触发，它会 panic（而不是 UB）。在 release 模式中，编译器仍然可以利用 `unreachable!()` 进行死代码优化，但加上了一层安全网：如果假定出错，程序会崩溃并给出清晰的错误信息，而不是进入未定义行为状态。

`std::hint::unreachable_unchecked()` 是 `intrinsics::unreachable` 的稳定包装——但它仍然需要 `unsafe`，触发时会直接导致 UB。我们在下一课（4.6）中会详细讨论。

```rust
/// 安全的 unreachable: unreachable!() 宏——触发时 panic
fn safe_unreachable(x: Option<i32>) -> i32 {
    match x {
        Some(v) => v,
        None => unreachable!("should never be called with None"),
    }
}

/// unsafe 但更快的 unreachable: unreachable_unchecked
fn unsafe_unreachable(x: Option<i32>) -> i32 {
    match x {
        Some(v) => v,
        // 如果传入了 None，这是 UB——程序可能做任何事情
        None => unsafe { std::hint::unreachable_unchecked() },
    }
}
```

## 7. assume —— 告诉编译器"相信我"

### 7.1 assume 的语义

`assume(condition: bool)` 告诉编译器"这个条件在运行时**永远**为真"。与 `unreachable` 类似，这是一个强力的承诺——如果条件为假，行为是未定义的。

```rust
// 需要 nightly: #![feature(core_intrinsics)]
#![feature(core_intrinsics)]

use std::intrinsics::assume;

/// 使用 assume 帮助编译器优化——假设指针非空
fn process_buffer(buf: *const u8, len: usize) -> u8 {
    unsafe {
        // 告诉编译器：buf 不为空，len 大于 0
        // 编译器可以据此消除空指针检查和零长度分支
        assume(!buf.is_null());
        assume(len > 0);
        
        // 由于上面的 assume，编译器知道：
        // 1. buf 不是 null——不需要空指针检查
        // 2. len > 0——循环至少执行一次
        let mut sum: u8 = 0;
        for i in 0..len {
            sum = sum.wrapping_add(*buf.add(i));
        }
        sum
    }
}

/// assume 用于除法优化——假设除数非零
fn safe_divide(dividend: i32, divisor: i32) -> i32 {
    unsafe {
        // 告诉编译器：divisor 永远不是 0
        // 编译器可以生成更高效的除法指令
        // （不做除零检查，不做除零跳转）
        assume(divisor != 0);
        dividend / divisor
    }
}
```

### 7.2 assume 如何影响 LLVM 优化

在 LLVM IR 中，`assume(condition)` 被翻译为 `@llvm.assume(i1 condition)`。LLVM 的优化器利用 assume 信息进行多种优化：

常量传播：如果 assume 可以推导出某个变量的值范围，优化器会在后续代码中使用这个约束。

条件消除：如果 assume 断言 `x > 0`，那么任何检查 `x <= 0` 的代码都可以被消除。

循环优化：assume 可以告知循环的边界和步长信息，帮助 LLVM 进行循环展开和向量化。

在稳定 Rust 中，没有 `assume` 的直接替代品。最接近的工具是通过 `unreachable!()` 或 `unreachable_unchecked()` 将假定条件转化为不可达代码——但这不如 `assume` 精确。

## 8. 数学 Intrinsics —— 直接映射为 CPU 指令

### 8.1 sqrtf64 和其他数学 intrinsic

数学 intrinsics 直接映射为 CPU 的浮点运算指令。以 `sqrtf64` 为例：

```rust
// 需要 nightly: #![feature(core_intrinsics)]
#![feature(core_intrinsics)]

use std::intrinsics;

fn demonstrate_math_intrinsics() {
    let x: f64 = 16.0;
    
    // sqrtf64 直接在 LLVM IR 中翻译为 @llvm.sqrt.f64
    // 在 x86_64 上，这进一步映射为 sqrtsd 指令
    let sqrt_x = intrinsics::sqrtf64(x);
    assert!((sqrt_x - 4.0).abs() < 1e-10);
    
    // 其他数学 intrinsics
    let _abs_x = intrinsics::fabsf64(-5.0);      // fabs —— 绝对值
    let _pow = intrinsics::powf64(2.0, 10.0);    // powf —— 幂运算（2^10 = 1024）
    let _copysign = intrinsics::copysignf64(-1.0, 2.0); // copysign —— 复制符号
    
    // 三角函数也有对应的 intrinsics
    let _sin = intrinsics::sinf64(std::f64::consts::PI / 2.0); // sin
    let _cos = intrinsics::cosf64(0.0);                         // cos
    
    // 指数和对数
    let _exp = intrinsics::expf64(1.0);  // e^1
    let _ln = intrinsics::logf64(std::f64::consts::E); // ln(e)
}
```

### 8.2 为什么使用数学 intrinsic 而不是标准库方法

`x.sqrt()`（标准库方法）和 `intrinsics::sqrtf64(x)` 之间有两个关键区别：

标准库的 `f64::sqrt()` 是一个普通的 trait 方法调用，它在编译时**没有**对编译器的语义保证——编译器可能将其内联，也可能不内联。如果函数跨 crate 边界调用，LTO 之前优化器看不到函数体。

`intrinsics::sqrtf64` 被编译器识别为 LLVM 的 `@llvm.sqrt.f64`，优化器**总是**知道它的精确语义。这意味着优化器可以常量折叠（`sqrtf64(4.0)` 在编译时就被计算为 `2.0`）、与其他操作结合（`sqrtf64(x) * sqrtf64(x)` 可以被优化为 `fabsf64(x)`）、并且在 SIMD 向量化中与其他运算一起被向量化。

在稳定 Rust 中，`f64::sqrt()` 等标准库方法通常已经足够好——LLVM 优化器在 LTO 或内联后可以识别出这些方法是"已知的数学函数"，并把它们当作 intrinsic 一样对待。但对于需要确保最优代码生成的场景（如高性能计算库），直接使用 intrinsic 可以消除任何不确定性。

## 9. 何时使用 Intrinsics vs 安全抽象

### 9.1 决策框架

使用 intrinsics 的决策应该遵循以下优先级：

第一步：尝试使用 safe Rust 的标准库接口（`f64::sqrt()`、`std::ptr::copy`、`std::mem::size_of`）。在 95% 的场景中，标准库已经是最优实现。

第二步：如果性能分析显示某个标准库调用是热点，检查是否可以通过算法改进或其他 safe 方式解决。

第三步：如果确实需要 intrinsics 级别的控制，优先使用 `std::intrinsics` 中的**稳定包装**（`std::mem::size_of`、`std::mem::needs_drop`、`std::ptr::copy_nonoverlapping`）。

第四步：只有在前三步都不满足的情况下，才退回到需要使用 `#![feature(core_intrinsics)]` 的 nightly-only intrinsics——并准备好为每个 Rust 版本更新维护它们。

### 9.2 稳定性矩阵

以下是常见 intrinsic 及其稳定替代品的对照表：

| Intrinsic | 稳定替代 | 是否需要 nightly |
|-----------|---------|-----------------|
| `size_of::<T>()` | `std::mem::size_of::<T>()` | 否 |
| `min_align_of::<T>()` | `std::mem::align_of::<T>()` | 否 |
| `needs_drop::<T>()` | `std::mem::needs_drop::<T>()` | 否 |
| `type_name::<T>()` | `std::any::type_name::<T>()` | 否 |
| `copy::<T>()` | `std::ptr::copy::<T>()` | 否 |
| `copy_nonoverlapping::<T>()` | `std::ptr::copy_nonoverlapping::<T>()` | 否 |
| `unreachable()` | `unreachable!()` / `std::hint::unreachable_unchecked()` | 否 |
| `likely()` / `unlikely()` | 无稳定替代（使用代码结构暗示） | 是 |
| `assume()` | 无稳定替代 | 是 |
| `sqrtf64()` | `f64::sqrt()` | 否 |
| `transmute()` | `std::mem::transmute()` | 否 |
| `volatile_load()` / `volatile_store()` | `std::ptr::read_volatile()` / `write_volatile()` | 否 |
| `pref_align_of::<T>()` | `std::mem::align_of::<T>()` | 否 |
| `variant_count::<T>()` | [RFC #3573 待稳定](https://github.com/rust-lang/rust/issues/73662) | 是 |

## 10. 阅读 LLVM IR —— 理解 Intrinsic 做了什么

### 10.1 生成和阅读 LLVM IR

你可以通过 `cargo rustc -- --emit=llvm-ir` 命令生成 LLVM IR 文件。生成的文件以 `.ll` 扩展名结尾，是文本格式的 LLVM 中间表示。

以下是一个 Rust 函数及其对应的 LLVM IR 的概念性对比。请想象一个使用 `sqrtf64` intrinsic 的函数：

```rust
// Rust 源码：
// pub fn sqrt(x: f64) -> f64 {
//     intrinsics::sqrtf64(x)
// }

// 对应的 LLVM IR（简化版）：
// 
// define double @sqrt(double %x) unnamed_addr {
// start:
//   ; sqrtf64 被翻译为 @llvm.sqrt.f64 intrinsic
//   %0 = call fast double @llvm.sqrt.f64(double %x)
//   ret double %0
// }
//
// 关键观察：
// 1. 没有函数调用开销——@llvm.sqrt.f64 是 LLVM intrinsic
// 2. fast 标记表示使用宽松的浮点语义（允许更多优化）
// 3. 后端将把 @llvm.sqrt.f64 直接映射为 sqrtsd（x86_64）或 fsqrt（ARM64）
```

### 10.2 用 LLVM IR 验证优化

阅读 LLVM IR 的一个最大的实际用途是**验证编译器是否正确理解了你的意图**。特别是当你使用 `likely`、`unlikely`、`unreachable` 和 `assume` 时，LLVM IR 中可以看到这些 intrinsic 如何影响控制流和优化决策。

```llvm
; 使用 unreachable intrinsic 的 Rust 代码：
; match value {
;     0..=9 => "single",
;     10..=99 => "double",
;     _ => unreachable(),
; }
;
; 对应的 LLVM IR（简化版）：
;
;   %cmp_single = icmp ult i8 %value, 10
;   br i1 %cmp_single, label %single, label %check_double
; check_double:
;   %cmp_double = icmp ult i8 %value, 100
;   br i1 %cmp_double, label %double, label %unreachable
; unreachable:
;   unreachable   ; LLVM 的 unreachable 指令
;                 ; 优化器知道这个位置永远不会被到达
;                 ; 因此可以：
;                 ; 1. 消除 %cmp_double 检查（因为到这里的 value 一定 >= 10）
;                 ; 2. 推断 value 一定 < 100（否则会到 unreachable）
;
; 编译器可以将这个逻辑优化为：
;   ; value 在 [0, 99] 范围内
;   %is_single = icmp ult i8 %value, 10
;   ; 单个条件跳转即可，不需要 %cmp_double 检查
```

当你怀疑优化器没有如预期般工作，或者 intrinsic 没有产生期望的效果时，LLVM IR 是调试的最佳起点。

## 12. 练习答案

### 练习 1 答案

```rust
// needs_drop 是编译时常量，因此它在编译时被评估。
// 
// 对于 T = i32：
// needs_drop::<i32>() 在编译时被计算为 false。
// 编译器在单态化（monomorphization）阶段将整个 if 块消除。
// 
// 预期的 LLVM IR（简化版）：
// define void @optimized_clear_i32(%Vec* %data) {
//   ; needs_drop::<i32>() = false → if 块被消除
//   ; 函数体为空——什么都不需要做
//   ret void
// }
//
// 对于 T = String：
// needs_drop::<String>() 在编译时被计算为 true。
// 编译器保留 if 块中的 clear() 调用。
//
// 预期的 LLVM IR（简化版）：
// define void @optimized_clear_string(%Vec* %data) {
//   call void @Vec_String_clear(%Vec* %data)
//   ret void
// }
//
// 关键洞察：needs_drop 的"代价"完全在编译时支付。
// 运行时没有分支，没有检查，没有条件跳转——
// 因为 needs_drop 的结果在编译时就已经确定。
// 单态化（monomorphization）确保了每个类型实例化获得最优化的代码。
```

### 练习 2 答案

```rust
// 需要 nightly: #![feature(core_intrinsics)]

use std::intrinsics::assume;

/// 对齐优化的数组求和
/// 前提条件：
///   - data 是 16 字节对齐的
///   - len 是 4 的倍数（4 × f32 = 16 字节，匹配 SSE 寄存器宽度）
pub fn sum_aligned(data: *const f32, len: usize) -> f32 {
    unsafe {
        // 向编译器承诺对齐和长度属性
        assume(data as usize % 16 == 0);   // 16 字节对齐
        assume(len % 4 == 0);              // 长度是 4 的倍数
        
        let mut total: f32 = 0.0;
        for i in 0..len {
            total += *data.add(i);
        }
        total
    }
}

// 为什么这些 assume 有助于 SIMD 向量化：
//
// 1. assume(data as usize % 16 == 0):
//    告诉编译器数据指针是 16 字节对齐的。
//    SSE 的 movaps（对齐加载）比 movups（不对齐加载）更快。
//    编译器现在可以使用 movaps 而不是 movups。
//
// 2. assume(len % 4 == 0):
//    告诉编译器循环次数是 4 的倍数。
//    编译器知道它可以将循环展开为每次处理 4 个 f32 的迭代，
//    使用单个 addps（SSE 打包单精度加法）指令一次加 4 个数。
//    不需要为"剩余元素"生成尾循环（tail loop）——因为 len % 4 == 0。
//
// LLVM 的自动向量化器（SLP Vectorizer 和 Loop Vectorizer）
// 会利用这些信息生成如下模式：
//
// .loop:
//   movaps xmm0, [data + offset]    ; 加载 4 个 f32（16 字节，对齐加载）
//   addps  xmm1, xmm0               ; 一次累加 4 个值
//   add offset, 16                   ; 前进 16 字节
//   cmp offset, len_times_4
//   jl .loop
```

### 练习 3 答案

```rust
// 问题分析：
// 
// parse_bool 函数接收任意 &str 输入，但使用了 unreachable()
// 当输入不是预定义的真值/假值字符串时（如 "hello"），
// 程序会触发不可恢复的未定义行为——而不是返回一个合理的错误。
//
// 错误类型：误用 unreachable() 处理可预期的输入错误
// unreachable 应该用于逻辑上不可能发生的情况，
// 而不是用于"我懒得处理"的输入验证。

/// ✅ 修正方案 1（推荐）：使用 Result 类型表达可能失败
fn parse_bool_safe(s: &str) -> Result<bool, String> {
    match s.to_lowercase().as_str() {
        "true" | "1" | "yes" => Ok(true),
        "false" | "0" | "no" => Ok(false),
        other => Err(format!("invalid bool string: '{}'", other)),
    }
}

/// ✅ 修正方案 2：使用 unreachable!() 宏而非 intrinsic unreachable
/// 如果调用方已经验证了输入，但你需要处理"I know this can't happen"
/// 的情况—用 unreachable!()——它 panic 而不是 UB
fn parse_bool_checked(s: &str) -> bool {
    match s.to_lowercase().as_str() {
        "true" | "1" | "yes" => true,
        "false" | "0" | "no" => false,
        other => {
            // unreachable!() 会 panic——至少是可控的崩溃
            // 而 unreachable() 会导致 UB——可能出现任意行为
            unreachable!("parse_bool called with invalid input: '{}'", other)
        }
    }
}

/// ✅ 修正方案 3：提供默认值——最安全的选择
fn parse_bool_with_default(s: &str, default: bool) -> bool {
    match s.to_lowercase().as_str() {
        "true" | "1" | "yes" => true,
        "false" | "0" | "no" => false,
        _ => default,
    }
}

// 核心教训：
// 1. intrinsic::unreachable() 是给编译器看的——优化提示，触发即 UB
// 2. unreachable!() 宏是给人看的——错误处理，触发即 panic
// 3. 不要将 intrinsic::unreachable() 用于处理可预期的输入错误
// 4. 如果你不确定某段代码是否真的不可达，使用 unreachable!() 更安全
```
