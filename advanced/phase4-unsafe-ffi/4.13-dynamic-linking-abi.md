# 4.13 动态链接与 ABI —— 从符号可见性到运行时插件系统

> 静态链接是你把库的代码"搬运"到你自己的二进制里；动态链接是你在二进制里留下一个"电话号簿"——运行时的动态链接器会按图索骥，找到真正的函数地址填进去。Rust 没有稳定的 ABI——`extern "Rust"` 随每个编译器版本变化——这意味着你不能在不同 rustc 版本间共享 Rust 编写的动态库。但通过 `extern "C"`、`#[no_mangle]` 和平台的动态链接机制，Rust 可以完美地融入 C 的生态。更进一步，通过 `libloading` 实现运行时动态加载，你可以构建"编译时不依赖具体库、运行时按需加载"的插件系统——这是理解 Rust 如何与操作系统链接模型交互的最后一课。

---

## 1. 动态库的两种面孔——cdylib 与 dylib

### 1.1 Cargo 的库类型分类

Rust 的 `Cargo.toml` 中通过 `[lib]` 的 `crate-type` 字段指定编译产物类型。与动态链接相关的有两种，它们的语义有本质区别。

```toml
# 编译为 C 兼容的动态库（.so / .dylib / .dll）
# 供任何语言通过 C ABI 调用——库中只暴露 extern "C" 的符号
[lib]
crate-type = ["cdylib"]

# 编译为 Rust 特定的动态库（.so / .dylib / .dll）
# 仅供 Rust 代码使用——内部使用 extern "Rust" 的符号
[lib]
crate-type = ["dylib"]
```

这两种类型的区别不仅仅是名字上的——它们决定了库的 ABI 接口、依赖管理方式和可使用范围。

### 1.2 cdylib —— 嵌入 C 生态的桥梁

`cdylib` 中的 "c" 代表 **C 兼容**。一个 cdylib 编译出来的动态库，遵循目标平台的标准 C ABI（Linux 上的 System V AMD64 ABI、Windows 上的 MSVC ABI），可以被任何支持 C FFI 的语言调用。

编写一个 cdylib 时，你只通过 `extern "C"` 暴露函数——库内部可以用 Rust 的所有特性，但公共接口必须是 C 兼容的。

```rust
// src/lib.rs —— 一个 cdylib 的完整示例
// Cargo.toml 需要 [lib] crate-type = ["cdylib"]

use std::ffi::CStr;
use std::os::raw::c_char;

/// 一个可以被 C 调用的加法函数
/// #[no_mangle] 阻止 Rust 的名称修饰（name mangling）
/// extern "C" 确保使用 C 的函数调用约定
#[no_mangle]
pub extern "C" fn add_numbers(a: i32, b: i32) -> i32 {
    a + b
}

/// 一个处理 C 字符串的函数
/// 调用者负责分配和释放字符串——这是 C ABI 的标准模式
#[no_mangle]
pub extern "C" fn greet(name: *const c_char) -> *mut c_char {
    let c_str = unsafe {
        if name.is_null() {
            return std::ptr::null_mut();
        }
        CStr::from_ptr(name)
    };

    let name_str = match c_str.to_str() {
        Ok(s) => s,
        Err(_) => return std::ptr::null_mut(),
    };

    let greeting = format!("你好，{}！", name_str);
    // 使用 CString 确保末尾有 \0
    std::ffi::CString::new(greeting)
        .unwrap_or_default()
        .into_raw() // 转移所有权给调用者
}

/// 释放由 greet 分配的字符串
/// C 调用者必须手动调用此函数来释放内存
#[no_mangle]
pub extern "C" fn free_greeting(ptr: *mut c_char) {
    if !ptr.is_null() {
        unsafe {
            // 重新获得所有权并让 Rust 释放它
            let _ = std::ffi::CString::from_raw(ptr);
        }
    }
}
```

设计 cdylib 的公共 API 时，有一条铁律：**永远不要跨越 FFI 边界传递 Rust 独有的类型**——`String`、`Vec`、`&str`、`Result` 的布局是 Rust 编译器私有的，在 FFI 边界上使用它们是未定义行为。始终使用 `*const c_char`、`i32`、`f64` 等 C 兼容类型，或强制转换为 `*mut c_void` 的不透明指针。

### 1.3 dylib —— Rust 生态内部的动态链接

`dylib` 编译出的动态库使用 `extern "Rust"` ABI——这是 Rust 编译器私有的调用约定，不承诺跨版本兼容。一个 `dylib` 可以被其他 Rust crate 通过 `extern crate` 或 `Cargo.toml` 依赖引用。

编译一个 dylib 时，你不需要 `#[no_mangle]` 或 `extern "C"`——正常的 Rust 函数就足够了。

```rust
// src/lib.rs —— 一个 dylib 的简单示例
// Cargo.toml 需要 [lib] crate-type = ["dylib"]

// 正常的 Rust 函数——自动使用 extern "Rust" ABI
pub fn compute(value: u64) -> u64 {
    value * 2 + 1
}

pub struct DataProcessor {
    multiplier: u64,
}

impl DataProcessor {
    pub fn new(factor: u64) -> Self {
        DataProcessor { multiplier: factor }
    }

    pub fn process(&self, input: u64) -> u64 {
        input * self.multiplier
    }
}
```

但这里有一个关键限制：**依赖此 dylib 的 crate 必须使用完全相同版本的 rustc 编译**。因为 `extern "Rust"` 的名称修饰（name mangling）、类型布局、泛型单态化策略都可能随编译器版本变化。如果你的项目和它的 dylib 依赖编译时使用了不同的编译器版本，可能会在运行时出现难以调试的崩溃。

### 1.4 选择指南：何时用哪种

没有唯一正确的选择——取决于你的使用场景。以下是决策流程：

```rust
// 决策逻辑（伪代码，表达的是选择过程而非可执行代码）
//
// if 库需要被 C / Python / Java / Go 等非 Rust 语言调用 {
//     → 使用 cdylib + extern "C" + #[no_mangle]
// }
// else if 库只在 Rust 生态内使用 && 可以保证统一的编译器版本 {
//     → 使用 dylib（或者更好的做法：使用 rlib 静态链接）
// }
// else if 需要在运行时动态发现和加载组件（插件系统） {
//     → 使用 cdylib + libloading（见第 4 节）
// }
//
// 在大多数 Rust 项目的实际实践中：
// rlib（静态链接） > cdylib（FFI 场景） > dylib（极其少见）
```

绝大多数 Rust 项目默认使用的是 `rlib`（Rust 库的静态链接格式）——它没有动态链接的任何复杂性，编译时直接内联优化。`dylib` 主要用于需要多个 Rust crate 共享同一个 Rust 标准库的场景（比如编译时间极长的项目想通过动态链接减少重复编译），在实践中使用较少。

## 2. 符号可见性——控制链接器能看到什么

### 2.1 #[no_mangle] —— 禁止名称修饰

Rust 编译器会对其生成的所有符号进行名称修饰（name mangling）——将函数名、泛型参数、所在的模块路径编码成一个唯一的字符串。如果你用 `nm` 或 `objdump` 查看一个 Rust 编译出的 `.o` 文件，你会看到类似 `_ZN7mycrate3foo17h3f8a9b2c1d4e5f6E` 的符号名。`#[no_mangle]` 阻止这种修饰，让符号名以你在代码中写的名字直接出现在输出文件中。

```rust
// 没有 #[no_mangle] —— 链接器看到的是类似
// _ZN12my_library11add_numbers17h0123456789abcdefE
// 这样的修饰后的符号名
pub fn add_numbers(a: i32, b: i32) -> i32 {
    a + b
}

// 有 #[no_mangle] —— 链接器看到的就是 "add_numbers"
#[no_mangle]
pub extern "C" fn add_numbers_raw(a: i32, b: i32) -> i32 {
    a + b
}
```

`#[no_mangle]` 的两个隐含要求：第一，因为 Rust 不允许在同一个 crate 中有两个同名的"正常"函数，如果两个 `#[no_mangle]` 函数同名，链接阶段会报符号冲突错误；第二，符号名必须符合目标平台的命名规则（Windows 上函数名不分大小写，Linux 上没有这个限制）。

### 2.2 #[export_name] —— 自定义导出符号名

`#[no_mangle]` 简单粗暴——导出的符号名就是函数名本身。`#[export_name]` 允许你指定任意字符串作为符号名，这在需要与特定二进制接口匹配时非常有用。

```rust
// 导出一个自定义名称的符号
// 这在需要与预定义的符号名匹配时非常有用
#[export_name = "my_custom_symbol_v1"]
pub extern "C" fn actual_function(arg: i32) -> i32 {
    arg * 3
}

// 一个实际场景：导出多个版本的 API
// 调用者可以根据需要选择不同版本
#[export_name = "process_data_v1"]
pub extern "C" fn process_v1(data: *const u8, len: usize) -> i32 {
    // 版本 1 的实现
    1
}

#[export_name = "process_data_v2"]
pub extern "C" fn process_v2(data: *const u8, len: usize, options: u32) -> i32 {
    // 版本 2 的增强实现
    2
}
```

使用 `#[export_name]` 时有一个陷阱需要注意：如果你为两个函数指定了相同的导出名，编译器不会报错——而是在链接阶段产生难以理解的"重复符号"错误。另外，`#[export_name]` 隐含了 `#[no_mangle]` 的效果，你不需要同时使用两者。

### 2.3 pub 与符号可见性的关系

在 Rust 的模块系统中，`pub` 控制的是 Rust 层面的可见性——它决定其他 Rust crate 能否访问某个项。但在编译为动态库时，`pub` 不影响符号是否被导出到动态符号表。

```rust
// 编译为 cdylib 时：
// pub 的 extern "C" fn 会被导出
#[no_mangle]
pub extern "C" fn visible_to_all() {}  // ✅ 在动态符号表中

// 非 pub 的 extern "C" fn 也会被导出（如果有 #[no_mangle]）
#[no_mangle]
extern "C" fn also_visible() {}  // ⚠️ 也在动态符号表中！

// 普通的 Rust 函数（pub 或非 pub）：
pub fn rust_function() {}  // cdylib 中：❌ 不在动态符号表（extern "Rust"）
fn private_rust_fn() {}    // cdylib 中：❌ 不在动态符号表
```

关键规则：在 `cdylib` 中，只有 `extern "C"`（或 `extern "system"` 等明确 ABI 的）函数会被放入动态符号表。普通的 Rust 函数（`extern "Rust"` ABI）不会被导出——它们仍然存在于库的代码段中，但不在动态符号表里，外部链接器看不到它们。

## 3. PLT 与 GOT —— 动态链接器的"电话交换系统"

### 3.1 为什么需要 PLT 和 GOT

当你的程序调用一个动态库中的函数时，编译器在编译时不知道这个函数的最终地址——因为动态库可能被加载到内存的任何位置（地址空间布局随机化 ASLR）。PLT（Procedure Linkage Table，过程链接表）和 GOT（Global Offset Table，全局偏移表）共同解决了"编译时不知道地址、运行时要能调得到"的问题。

```
程序的调用链（首次调用一个动态库函数）：

1. call printf@PLT       ← 你的代码调用 PLT 中的一个桩函数
2. PLT 桩：jmp *GOT[n]   ← 跳转到 GOT 表项中存储的地址
3. GOT[n] 初始值 → PLT 桩的下一条指令（动态链接器的入口）
4. 动态链接器查找 "printf" 在 libc.so 中的真实地址
5. 动态链接器把真实地址写入 GOT[n]
6. 控制流跳转到真正的 printf 函数
7. 第二次调用 printf 时：
   call printf@PLT → jmp *GOT[n] → 直接跳转（GOT 已有真实地址）
```

这就是"延迟绑定"（lazy binding）——函数地址只在第一次调用时才被解析，避免在程序启动时解析所有符号。

### 3.2 从 Rust 代码窥视 PLT 的工作

我们可以通过一个简单的 Rust 程序来理解 PLT 的存在。虽然你不需要修改 PLT/GOT，但理解它的存在对于性能调试至关重要。

```rust
// 这个程序调用 libc 的 write 函数
// write 在 libc.so 中，而非你的二进制里
// 每次对 write 的调用都会经过 PLT

use std::os::unix::io::RawFd;

// 通过 libc crate 调用 C 函数——经过 PLT
extern "C" {
    fn write(fd: RawFd, buf: *const u8, count: usize) -> isize;
}

fn main() {
    let msg = b"Hello via PLT!\n";
    unsafe {
        // 这次调用会经过 PLT 跳板（trampoline）
        // 第一次调用：PLT → 动态链接器 → 解析 → GOT → 真实函数
        // 后续调用：PLT → GOT → 真实函数（直接跳转，无开销）
        write(1, msg.as_ptr(), msg.len());
    }
}
```

每次通过 `extern "C"` 调用的外部函数都会经过 PLT——即使它已经被解析过了（后续调用只是多了一次内存间接跳转，开销很小但非零）。

### 3.3 PLT 的性能代价与 `-C relocation-model=static`

PLT 的间接跳转意味着：每次调用外部函数，CPU 都需要先读取 GOT 表项中的地址，然后跳转到那个地址。对于热点路径上的频繁调用，这个间接跳转的分支预测可能会失败，导致流水线停顿。

在某些场景下，你可以通过编译选项绕过 PLT：

```bash
# 在 .cargo/config.toml 中设置：
# [build]
# rustflags = ["-C", "relocation-model=static"]

# 或者在命令行：
# RUSTFLAGS="-C relocation-model=static" cargo build --release
```

`relocation-model=static` 告诉链接器：在编译时尽可能直接解析外部符号的地址（而不是通过 PLT）。这要求外部符号在链接时就已经确定位置——对于静态链接的库没有问题，但对于动态库，只有当你设置 `-fno-plt` 风格的编译选项或使用 `LD_BIND_NOW` 环境变量时才真正生效。

```rust
// 没有 -C relocation-model=static 时：
// call write@PLT  → 间接跳转

// 有 -C relocation-model=static 且链接器支持时：
// call write       → 直接跳转（免去 PLT 间接寻址）

// 适用场景：
// - 静态编译的二进制（所有依赖已嵌入）
// - 对性能极度敏感的调用循环
// - 不适用场景：
// - 需要 LD_PRELOAD 拦截函数调用（绕过 PLT 会使拦截失效）
```

多数项目不需要修改此设置——PLT 的开销在绝大多数应用中是微不足道的。只有在性能剖析（profiling）明确指出 PLT 是瓶颈时，才值得启用此选项。

## 4. dlopen / dlsym —— 在运行时决定链接什么

### 4.1 动态加载的核心理念

静态链接和编译时动态链接都有一个共同前提：你要链接的库在编译时已知。但有些场景下，你想在运行时决定加载哪个库——插件系统就是最典型的例子。操作系统提供了 `dlopen`（加载库）、`dlsym`（查找符号）、`dlclose`（卸载库）的 API。

### 4.2 libloading crate —— 安全包装操作系统 API

直接使用 `libc::dlopen` / `libc::dlsym` 需要大量 unsafe 代码和平台相关的条件编译。`libloading` crate 提供了一个跨平台的安全包装——在 Linux 上使用 `dlopen`，在 macOS 上使用 `dlopen`，在 Windows 上使用 `LoadLibrary`，但提供统一的 Rust 接口。

首先在 `Cargo.toml` 中添加依赖：

```toml
[dependencies]
libloading = "0.8"
```

以下是最基本的使用模式——加载一个动态库并调用其中的函数：

```rust
use libloading::{Library, Symbol};
use std::path::Path;

/// 动态加载一个库并从其中查找和调用函数
fn call_plugin(plugin_path: &Path) -> Result<(), Box<dyn std::error::Error>> {
    // 步骤 1：加载动态库
    // unsafe：加载的库可能包含任意代码（包括静态析构函数等）
    let library = unsafe { Library::new(plugin_path)? };

    // 步骤 2：查找符号——按名称获取函数指针
    // unsafe：我们必须保证符号的类型签名与实际一致
    let func: Symbol<unsafe extern "C" fn(i32, i32) -> i32> = unsafe {
        library.get(b"multiply\0")?  // b"\0" 确保 C 风格的以 null 结尾
    };

    // 步骤 3：调用函数
    // unsafe：调用者负责确保参数和返回值类型正确
    let result = unsafe { func(6, 7) };
    println!("plugin multiply(6, 7) = {}", result);

    // library 离开作用域时自动关闭（相当于 dlclose）
    Ok(())
}

// 实际使用：
fn main() {
    let path = Path::new("./plugins/libmyplugin.so");
    if path.exists() {
        call_plugin(path).expect("Failed to load plugin");
    } else {
        println!("Plugin not found at {:?}", path);
    }
}
```

`Library` 在 drop 时自动调用 `dlclose`（或等效 API）——这是 RAII 在 FFI 领域的一个经典应用。`Symbol<T>` 是一个泛型包装，`T` 通常是函数指针类型。

### 4.3 常见的动态加载陷阱

动态加载在灵活性提升的同时也引入了新的风险。以下是三个最常见的陷阱及其解决方案。

当库中的符号名与你期望的不符时，`library.get()` 会返回错误。这在跨平台开发中尤为常见——Windows 的 DLL 可能使用 `_my_function` 或 `my_function@8`（名称修饰因调用约定而生）而 Linux 的 `.so` 使用 `my_function`。解决方案是使用一致的 `extern "C"` 和 `#[no_mangle]`，或者在平台上做适配。

加载的库在 `Library` 被 drop 之前一直驻留在进程地址空间中。如果在此期间你持有来自该库的函数指针或数据指针，在库被卸载后使用这些指针会导致段错误（segfault）。确保 `Library` 的生命周期覆盖所有从该库获取的 `Symbol<T>`。

不同的动态库可能使用不同版本的 Rust 标准库，导致存在两份标准库代码在同一个进程中。这通常不是问题——标准库被设计为可重入的——但如果两个版本之间标准库的内部类型布局发生了变化，通过 FFI 传递标准库类型就会产生问题。因此永远只通过 C ABI 类型跨越动态加载的边界。

## 5. Rust ABI 的不稳定性——为什么你不能用 dylib 做插件

### 5.1 extern "Rust" 没有稳定保证

Rust 的标准库和编译器对 `extern "Rust"` ABI 不做任何稳定性承诺。每次编译器发布新版本，内部可能改变以下任何一项：

```
每次 rustc 版本迭代可能变化的内容：
- 名称修饰（name mangling）算法（v0 仍在演进）
- 类型的内部布局（repr(Rust) 是编译器私有算法）
- 泛型单态化策略（何时产生、何时合并）
- 函数调用约定（参数传递的寄存器分配）
- panic 展开的运行时机制
```

因此，使用 `dylib`（依赖 `extern "Rust"` ABI）时，编译 dylib 和使用 dylib 的 crate 必须使用**完全相同**的 rustc 版本。这是一个硬性约束，任何版本差异都可能导致运行时未定义行为。

### 5.2 运行时检测编译器版本

如果你出于某种原因必须使用 `dylib`，至少在运行时验证编译器版本是否匹配：

```rust
/// 在插件加载代码中检查编译器版本
/// 使用 rustc_version_runtime crate 获取当前运行时的编译器版本
fn check_rustc_compatibility(plugin_version: &str) -> Result<(), String> {
    let host_version = rustc_version_runtime::version();

    if host_version.to_string() != plugin_version {
        return Err(format!(
            "rustc 版本不匹配：plugin 用 {} 编译，host 用 {} 编译",
            plugin_version, host_version
        ));
    }
    Ok(())
}

// 一个可行的方案是让插件导出一个返回版本信息的函数
// 然后在加载插件时调用它进行验证：
//
// 插件侧：
// #[no_mangle]
// pub extern "C" fn plugin_rustc_version() -> *mut c_char {
//     let version = format!("{}", rustc_version_runtime::version());
//     CString::new(version).unwrap().into_raw()
// }
//
// 宿主侧：加载后调用 plugin_rustc_version()，比较并决策
```

但即使版本匹配，`dylib` 仍有许多微妙的问题（如全局 allocator 冲突、panic hook 覆盖等），所以大多数项目选择完全避免它。

### 5.3 abi_stable crate —— 在 Rust 中构建稳定 ABI

`abi_stable` crate 提供了一个解决方案：它定义了一套"稳定的 Rust ABI"——库和调用者不依赖 `extern "Rust"`，而是使用 `abi_stable` 定义的类型和宏，这些类型承诺在 `abi_stable` 的 semver 兼容范围内保持布局一致。

使用 `abi_stable` 构建一个插件接口：

```rust
// 使用 abi_stable crate
// Cargo.toml:
// [dependencies]
// abi_stable = "0.11"

use abi_stable::{
    export_root_module,
    prefix_type::PrefixTypeTrait,
    sabi_extern_fn,
    std_types::{RString, RStr},
    StableAbi,
};

/// 使用 abi_stable 定义插件接口——不需要 extern "C"
/// #[sabi_trait] 宏自动生成 FFI-safe 的虚表（vtable）
#[abi_stable::sabi_trait]
pub trait Plugin {
    fn name(&self) -> RString;
    fn process(&self, input: RStr<'_>) -> RString;
}

/// 实现插件的具体类型
#[derive(StableAbi)]
#[repr(C)]
pub struct MyPlugin;

impl Plugin for MyPlugin {
    fn name(&self) -> RString {
        RString::from("MyPlugin v1.0")
    }

    fn process(&self, input: RStr<'_>) -> RString {
        RString::from(format!("处理结果: {}", input))
    }
}

// abi_stable 提供了安全的跨版本动态加载机制
// 这比手工使用 extern "C" + libloading 更安全且更易维护
```

`abi_stable` 的实现原理是在幕后生成 `extern "C"` 函数和 `#[repr(C)]` 类型——它并没有使用任何编译器魔法，而是自动化了手工做起来繁琐且容易出错的工作。但要注意：`abi_stable` 本身的版本也会通过 semver 管理兼容性——如果你从 `0.11` 升级到 `0.12`，可能需要重新编译所有插件。

## 6. 静态链接与动态链接的博弈

### 6.1 #[link] 属性——在代码中声明链接方式

Rust 允许通过 `#[link]` 属性直接在代码中声明对外部库的链接方式：

```rust
// 动态链接 libfoo（在 Linux 上搜索 libfoo.so）
// libfoo 需要在编译时存在于链接搜索路径中
#[link(name = "foo", kind = "dylib")]
extern "C" {
    fn bar() -> i32;
}

// 静态链接 libfoo（在 Linux 上搜索 libfoo.a）
// 库的代码会被完整复制到最终的二进制中
#[link(name = "foo", kind = "static")]
extern "C" {
    fn bar() -> i32;
}

// 链接框架（macOS 特有）——同时包含动态库和头文件的预编译包
#[cfg(target_os = "macos")]
#[link(name = "CoreFoundation", kind = "framework")]
extern "C" {
    fn CFRunLoopRun();
}
```

`kind = "dylib"` 和 `kind = "static"` 告诉 rustc 在链接时搜索什么类型的库文件。在 Linux 上：`.so`（dylib）或 `.a`（static）；在 macOS 上：`.dylib` 或 `.a`；在 Windows 上：`.dll` + `.lib`（dylib）或 `.lib`（static）。

### 6.2 两种方式的利弊分析

选择静态还是动态链接是一个权衡多种因素的工程决策：

```
静态链接（static）：
优点：
  ✅ 单一二进制文件——部署简单，不需要安装依赖
  ✅ 编译器可以做全程序优化（LTO）——内联跨库边界
  ✅ 没有版本兼容问题——所有代码编译时确定
  ✅ 没有"找不到 .so"的运行时错误
缺点：
  ❌ 二进制体积更大（每个库的代码都包含在内）
  ❌ 更新库需要重新编译整个程序
  ❌ 多程序共享同一库时浪费内存（每个进程各自加载）

动态链接（dylib）：
优点：
  ✅ 二进制更小（库代码在外部文件中）
  ✅ 更新库不需要重新编译程序（只要 ABI 兼容）
  ✅ 多程序可共享同一份库的内存页（节省 RAM）
  ✅ 支持运行时插件（dlopen）
缺点：
  ❌ "dependency hell"——版本不匹配导致运行时错误
  ❌ 部署复杂——需要确保目标环境有正确的 .so 版本
  ❌ PLT 间接调用的微小性能开销
  ❌ 安全更新需要同时更新库和程序
```

大多数 Rust 应用默认全静态链接（包括 Rust 标准库），这也是为什么 Rust 编译的二进制文件通常比 C 程序更大但更"即开即用"——你不需要安装任何运行时依赖。

## 7. 跨平台动态链接——一个世界，三种方言

### 7.1 文件命名约定

不同操作系统对动态库有不同的命名约定和搜索规则：

```
平台        动态库后缀    静态库后缀    环境变量          搜索路径
────────────────────────────────────────────────────────────────
Linux       .so          .a          LD_LIBRARY_PATH    /lib, /usr/lib, /usr/local/lib
macOS       .dylib       .a          DYLD_LIBRARY_PATH  /usr/lib, /usr/local/lib
Windows     .dll + .lib  .lib        PATH               当前目录, System32, PATH 中的目录
```

在 Windows 上有一个独特之处：`.dll` 是运行时的动态库，但编译时链接器还需要一个 `.lib`（导入库，import library）——这个 `.lib` 包含 PLT 桩代码，告诉程序如何跳转到 DLL 中的真实函数。Rust 通过 MSVC/MinGW 工具链处理这个复杂性。

### 7.2 RPATH 与 LD_LIBRARY_PATH

当程序启动时，动态链接器会在特定位置搜索 `.so` 文件。搜索顺序由 RPATH（嵌入在 ELF 二进制中的搜索路径）和 `LD_LIBRARY_PATH` 环境变量共同决定。

在 Rust 中设置 RPATH 的方法：

```bash
# 在 .cargo/config.toml 中设置 RPATH
# 让编译后的二进制能够在相对路径 ../lib 中查找 .so
[target.x86_64-unknown-linux-gnu]
rustflags = ["-C", "link-args=-Wl,-rpath,'$ORIGIN/../lib'"]

# $ORIGIN 是一个特殊的占位符，展开为二进制文件所在的目录
# 这使得你的程序可以"随身携带"它的动态依赖
```

或者在编译时通过环境变量指定搜索目录：

```bash
# 编译时指定链接搜索路径
RUSTFLAGS="-L /path/to/libs" cargo build

# 运行时指定动态库搜索路径
LD_LIBRARY_PATH=/path/to/libs ./my_program
```

RPATH 的优点是它嵌入在二进制中，不需要用户手动设置环境变量——适合分发给最终用户的应用程序。缺点是它使二进制不那么"可移植"——如果目标机器的路径结构不同就会出现找不到库的错误。

### 7.3 Rust 中的跨平台条件编译

处理跨平台差异的标准方式是使用 `#[cfg]` 属性：

```rust
/// 跨平台加载动态库的示例
/// 使用 cfg 属性适配不同平台

#[cfg(target_os = "linux")]
const LIB_EXTENSION: &str = "so";

#[cfg(target_os = "macos")]
const LIB_EXTENSION: &str = "dylib";

#[cfg(target_os = "windows")]
const LIB_EXTENSION: &str = "dll";

/// 根据当前平台构建插件文件路径
fn plugin_path(name: &str) -> std::path::PathBuf {
    let filename = format!(
        "{}{}{}",
        // Linux/macOS 惯例：lib 前缀
        if cfg!(target_os = "windows") { "" } else { "lib" },
        name,
        // 后缀
        format!(".{}", LIB_EXTENSION)
    );
    std::path::PathBuf::from("plugins").join(filename)
}

// 在 Linux 上：plugin_path("myplugin") → "plugins/libmyplugin.so"
// 在 macOS 上：plugin_path("myplugin") → "plugins/libmyplugin.dylib"
// 在 Windows 上：plugin_path("myplugin") → "plugins/myplugin.dll"
```

注意 Windows 不使用 `lib` 前缀——这与 Linux/macOS 的惯例不同。当你在编写跨平台代码时，这些细节决定了你的程序能否在不同系统上正确找到动态库。

## 8. 实战：构建一个完整的 Rust 插件系统

### 8.1 架构概览

我们将构建一个简单的插件系统：宿主程序定义插件接口，通过 `libloading` 在运行时加载插件目录中的所有 `.so`/`.dylib`/`.dll` 文件，每个插件导出一个统一的 C ABI 函数供宿主调用。

```
项目结构：
plugin-host/        ← 宿主程序
  src/main.rs
  Cargo.toml
plugin-interface/   ← 插件接口定义（cdylib 都依赖它）
  src/lib.rs
  Cargo.toml
plugins/
  hello-plugin/     ← 一个示例插件
    src/lib.rs
    Cargo.toml
```

### 8.2 插件接口定义

接口 crate 定义了宿主和插件之间的"合同"——双方都依赖这个 crate，确保类型一致：

```rust
// plugin-interface/src/lib.rs
// 这个 crate 定义了插件必须实现的接口
// 因为通过 FFI 传递，所有类型必须用 #[repr(C)]

use std::ffi::c_char;

/// 插件元数据——描述插件的基本信息
#[repr(C)]
pub struct PluginInfo {
    /// 插件名称（UTF-8 编码，末尾有 \0）
    pub name: *const c_char,
    /// 插件版本
    pub version_major: u32,
    pub version_minor: u32,
}

/// 插件 VTable（虚函数表）——插件提供的操作集合
/// 通过函数指针表来模拟面向对象的接口
#[repr(C)]
pub struct PluginVTable {
    /// 获取插件元信息
    pub get_info: unsafe extern "C" fn() -> PluginInfo,
    /// 执行插件的主要功能
    /// 参数：input 为输入的 C 字符串
    /// 返回：输出 C 字符串（调用者负责释放）
    pub execute: unsafe extern "C" fn(input: *const c_char) -> *mut c_char,
    /// 释放由 execute 返回的字符串
    pub free_string: unsafe extern "C" fn(ptr: *mut c_char),
}
```

注意所有类型使用 `#[repr(C)]`——这是跨越 FFI 边界的前提。`PluginVTable` 模拟了面向对象接口在 C ABI 层面的实现方式：一组函数指针（相当于 C++ 虚表）。使用这种方法可以定义任意复杂的插件接口。

### 8.3 一个示例插件

插件是一个编译为 `cdylib` 的 crate，导出 `PluginVTable` 中的函数：

```rust
// plugins/hello-plugin/src/lib.rs
// Cargo.toml 中：crate-type = ["cdylib"]

use std::ffi::{CStr, CString};
use std::os::raw::c_char;

// 引用接口 crate 中的类型
use plugin_interface::{PluginInfo, PluginVTable};

#[no_mangle]
pub unsafe extern "C" fn hello_get_info() -> PluginInfo {
    PluginInfo {
        name: CString::new("hello-plugin")
            .unwrap()
            .into_raw() as *const c_char,
        version_major: 1,
        version_minor: 0,
    }
}

#[no_mangle]
pub unsafe extern "C" fn hello_execute(input: *const c_char) -> *mut c_char {
    if input.is_null() {
        return std::ptr::null_mut();
    }

    let input_str = match CStr::from_ptr(input).to_str() {
        Ok(s) => s,
        Err(_) => return std::ptr::null_mut(),
    };

    let output = format!("[hello-plugin] 收到: \"{}\" —— 处理后返回此消息", input_str);
    CString::new(output).unwrap_or_default().into_raw()
}

#[no_mangle]
pub unsafe extern "C" fn hello_free_string(ptr: *mut c_char) {
    if !ptr.is_null() {
        drop(CString::from_raw(ptr));
    }
}

/// 使用 static 来导出全局的 VTable——这是宿主导入的入口点
/// 宿主通过 dlsym("PLUGIN_VTABLE") 获取指向此结构的指针
#[no_mangle]
pub static PLUGIN_VTABLE: PluginVTable = PluginVTable {
    get_info: hello_get_info,
    execute: hello_execute,
    free_string: hello_free_string,
};
```

在这个设计中，每个插件导出一个名为 `PLUGIN_VTABLE` 的全局静态符号——宿主程序只需要通过 `dlsym` 查找这一个符号，就能获取插件的所有功能。这比逐个导出和查找每个函数指针要优雅得多。

### 8.4 宿主程序——加载和调用插件

宿主程序扫描插件目录，加载所有符合条件的库并调用它们：

```rust
// plugin-host/src/main.rs
// Cargo.toml:
// [dependencies]
// libloading = "0.8"
// plugin-interface = { path = "../plugin-interface" }
// walkdir = "2"

use libloading::{Library, Symbol};
use plugin_interface::PluginVTable;
use std::path::Path;
use std::ffi::CStr;

/// 已加载的插件——持有库句柄和 VTable 引用
struct LoadedPlugin {
    _library: Library,          // 必须保持存活——否则库被卸载
    vtable: *const PluginVTable, // 指向库中的静态结构的指针
    name: String,
}

impl LoadedPlugin {
    /// 从动态库加载插件
    unsafe fn load(path: &Path) -> Result<Self, Box<dyn std::error::Error>> {
        let library = Library::new(path)?;

        // 查找 PLUGIN_VTABLE 符号——这是插件导出的唯一入口
        let vtable_sym: Symbol<*const PluginVTable> = library.get(b"PLUGIN_VTABLE")?;
        let vtable = *vtable_sym;

        // 调用 get_info 获取插件名称
        let info = ((*vtable).get_info)();
        let name = CStr::from_ptr(info.name).to_string_lossy().into_owned();

        Ok(LoadedPlugin {
            _library: library,
            vtable,
            name,
        })
    }

    /// 调用插件的 execute 函数
    unsafe fn execute(&self, input: &str) -> String {
        let c_input = std::ffi::CString::new(input).unwrap();
        let result_ptr = ((*self.vtable).execute)(c_input.as_ptr());

        if result_ptr.is_null() {
            return String::from("(plugin returned null)");
        }

        let result = CStr::from_ptr(result_ptr).to_string_lossy().into_owned();
        ((*self.vtable).free_string)(result_ptr);
        result
    }
}

fn main() {
    let plugin_dir = Path::new("./plugins/compiled");
    if !plugin_dir.exists() {
        println!("插件目录不存在: {:?}", plugin_dir);
        return;
    }

    let mut plugins: Vec<LoadedPlugin> = Vec::new();

    // 遍历插件目录，加载所有 .so/.dylib/.dll 文件
    for entry in std::fs::read_dir(plugin_dir).unwrap() {
        let path = entry.unwrap().path();
        let ext = path.extension().and_then(|e| e.to_str()).unwrap_or("");

        if ext == "so" || ext == "dylib" || ext == "dll" {
            println!("加载插件: {:?}", path.file_name().unwrap());
            match unsafe { LoadedPlugin::load(&path) } {
                Ok(plugin) => {
                    println!("  插件名称: {}", plugin.name);
                    plugins.push(plugin);
                }
                Err(e) => {
                    eprintln!("  加载失败: {}", e);
                }
            }
        }
    }

    // 调用所有已加载的插件
    let input = "你好，这是一个测试消息";
    for plugin in &plugins {
        let output = unsafe { plugin.execute(input) };
        println!("[{}] 输出: {}", plugin.name, output);
    }
}
```

这个设计的关键在于 `LoadedPlugin` 中的 `_library` 字段——它确保 `Library` 的生命周期覆盖 `vtable` 指针的使用。如果 `_library` 被丢弃，动态库被卸载，`vtable` 就变成悬垂指针——下次通过它调用函数会导致段错误。

## 10. 练习答案

### 练习 1 答案

```rust
// === Rust 侧：cdylib crate ===
// Cargo.toml:
// [lib]
// crate-type = ["cdylib"]

use std::ffi::CString;
use std::os::raw::c_char;

/// 加法——最简单的 FFI 函数
#[no_mangle]
pub extern "C" fn add(a: i32, b: i32) -> i32 {
    a + b
}

/// 乘法
#[no_mangle]
pub extern "C" fn multiply(a: i32, b: i32) -> i32 {
    a * b
}

/// 获取版本号——返回一个 C 字符串
/// 调用者必须使用 free_string 释放返回的字符串
#[no_mangle]
pub extern "C" fn get_version() -> *mut c_char {
    CString::new("cdylib-demo v1.0.0")
        .unwrap()
        .into_raw()
}

/// 释放由 get_version 分配的字符串
/// 为什么调用者不能直接用 free()？
/// → Rust 的 CString::into_raw() 使用 Rust 的全局分配器分配内存
/// → C 的 free() 使用 libc 的分配器
/// → 两个分配器可能不兼容（不同的内存池、不同的簿记结构）
/// → 因此必须由分配方的分配器来释放
#[no_mangle]
pub extern "C" fn free_string(ptr: *mut c_char) {
    if !ptr.is_null() {
        unsafe { drop(CString::from_raw(ptr)); }
    }
}

// === C 侧 ===
// #include <stdio.h>
// #include <dlfcn.h>
//
// int main() {
//     void *lib = dlopen("./libmathtools.so", RTLD_NOW);
//     if (!lib) { fprintf(stderr, "%s\n", dlerror()); return 1; }
//
//     typedef int (*add_fn)(int, int);
//     typedef int (*mul_fn)(int, int);
//     typedef char *(*ver_fn)(void);
//     typedef void (*free_fn)(char *);
//
//     add_fn add = dlsym(lib, "add");
//     mul_fn multiply = dlsym(lib, "multiply");
//     ver_fn get_version = dlsym(lib, "get_version");
//     free_fn free_string = dlsym(lib, "free_string");
//
//     printf("add(3, 4) = %d\n", add(3, 4));
//     printf("multiply(3, 4) = %d\n", multiply(3, 4));
//
//     char *version = get_version();
//     printf("version: %s\n", version);
//     free_string(version);  // 必须通过 Rust 提供的释放函数
//
//     dlclose(lib);
// }
```

关键学习点：跨 FFI 边界的内存管理是单向的——谁分配谁释放。这是因为不同的运行时使用不同的分配器，你不能混用。

### 练习 2 答案

```rust
// 分析 main 中的三次调用：

// 调用 1：local_function(42)
// → 不经过 PLT
// → local_function 在同一个编译单元内，地址在编译时确定
// → 编译器可能内联它

// 调用 2：unsafe { printf(b"hello\n\0".as_ptr()) }
// → 经过 PLT
// → printf 在 libc.so 中，编译时地址未知
// → 链接器生成 PLT 桩：call printf@PLT → jmp *GOT[printf]

// 调用 3：stdout.write_all(b"world\n").unwrap()
// → 表层不经过 PLT（Rust 代码中的函数调用）
// → 但 write_all 内部最终会调用 libc::write，那一步经过 PLT
// → 这个是"间接"经过 PLT——你不需要关心，但 profiling 能看到
//
// 总结规则：只有 extern 块中声明的外部符号调用才直接经过 PLT。
// 纯 Rust 代码调用不经过 PLT——但它们的底层实现可能会。
```

### 练习 3 答案

```rust
use libloading::{Library, Symbol};
use std::sync::{Arc, Mutex};
use std::time::{Duration, Instant};
use std::panic;

/// 沙箱插件加载器
struct SandboxedLoader {
    expected_api_version: u32,
}

impl SandboxedLoader {
    fn new(expected_api_version: u32) -> Self {
        SandboxedLoader { expected_api_version }
    }

    /// 安全加载并执行一个插件
    fn call_plugin_safely(
        &self,
        path: &std::path::Path,
        input: &str,
        timeout: Duration,
    ) -> Result<String, String> {
        // 步骤 1：加载库
        let library = unsafe {
            Library::new(path).map_err(|e| format!("加载失败: {}", e))?
        };

        // 步骤 2：验证 API 版本
        type VersionFn = unsafe extern "C" fn() -> u32;
        let version_func: Symbol<VersionFn> = unsafe {
            library.get(b"plugin_api_version\0")
                .map_err(|_| "插件未导出 plugin_api_version".to_string())?
        };
        let version = unsafe { version_func() };
        if version != self.expected_api_version {
            return Err(format!(
                "API 版本不匹配: 期望 {}, 实际 {}",
                self.expected_api_version, version
            ));
        }

        // 步骤 3：安装 panic hook——防止插件 panic 导致宿主崩溃
        // std::panic::catch_unwind 只能捕获 unwinding panic，
        // 不能捕获 abort，所以这只是"尽力而为"的沙箱
        let result = Arc::new(Mutex::new(None));
        let result_clone = result.clone();

        let plugin_thread = std::thread::spawn(move || {
            // 在子线程中执行插件代码
            let prev_hook = panic::take_hook();
            panic::set_hook(Box::new(|_info| {
                // 抑制插件的 panic 输出，避免污染宿主的日志
                // 真实实现中可以记录到单独的日志文件
            }));

            let res = panic::catch_unwind(panic::AssertUnwindSafe(|| {
                type ExecuteFn = unsafe extern "C" fn(*const i8) -> *mut i8;
                let func: Symbol<ExecuteFn> = unsafe {
                    library.get(b"plugin_execute\0").unwrap()
                };
                let c_input = std::ffi::CString::new(input).unwrap();
                unsafe { func(c_input.as_ptr()) }
            }));

            panic::set_hook(prev_hook);

            *result_clone.lock().unwrap() = Some(res);
        });

        // 步骤 4：等待执行结果，带超时
        let start = Instant::now();
        loop {
            if start.elapsed() > timeout {
                return Err(format!(
                    "插件执行超时 ({}ms)",
                    timeout.as_millis()
                ));
            }

            if let Some(ref res) = *result.lock().unwrap() {
                match res {
                    Ok(ptr) => {
                        // 步骤 5：处理结果
                        if ptr.is_null() {
                            return Ok("(插件返回空指针)".into());
                        }
                        let s = unsafe {
                            std::ffi::CStr::from_ptr(*ptr)
                                .to_string_lossy()
                                .into_owned()
                        };
                        return Ok(s);
                    }
                    Err(_panic_info) => {
                        return Err("插件执行时 panicked".into());
                    }
                }
            }

            std::thread::sleep(Duration::from_millis(10));
        }
    }
}
```

关键设计要点：版本检查放在调用任何其他函数之前——这是防御性编程的第一道防线。`catch_unwind` 提供了基本的 panic 隔离，但不能捕获 `panic = "abort"` 或未定义行为导致的崩溃；真正的进程级隔离需要将插件放进子进程。超时机制通过轮询实现，但更好的方案是使用 `recv_timeout` 或条件变量——这只是教学示例。所有这些机制共同构成了"不安全代码的安全抽象"——调用者不需要写任何 `unsafe`，但内部实现大量使用了它。
