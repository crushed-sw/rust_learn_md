# 4.7 FFI C 互操作 —— 从 Rust 调用 C，从 C 调用 Rust

> FFI（Foreign Function Interface）是 Rust 通向外部世界的桥梁。Rust 的 `extern "C"` 不仅仅是一个关键字——它是对 C 调用约定的承诺：参数按什么顺序入栈、谁负责清理栈、返回值通过哪个寄存器传递。当你写下 `unsafe { some_c_function() }` 时，你是在对编译器说："我知道自己在做什么——我保证这个函数的签名正确、生命周期合理、内存管理清晰。"而编译器则退后一步，把所有这些责任交还给你。FFI 是 Rust 中最需要谨慎的领域之一——但也是 Rust 能够嵌入现有生态系统、与数十年积累的 C 库无缝协作的关键能力。

## 1. `extern "C"` —— 声明外部函数

### 1.1 基本语法和调用约定

Rust 通过 `extern` 块声明来自外部语言的函数和变量。`extern` 后面跟一个 ABI 字符串——最常见的是 `"C"`，表示使用 C 语言的调用约定（calling convention）。

调用约定定义了一个协议：参数以什么顺序传递（寄存器还是栈）、使用哪些寄存器、返回值如何传递、调用者还是被调用者负责清理栈。当你声明 `extern "C"` 时，Rust 编译器按照目标平台的 C ABI 生成调用代码，使得 Rust 代码可以调用 C 编译的目标文件，反之亦然。

```rust
// 声明一个外部 C 函数——Rust 编译器会按 C ABI 生成调用代码
extern "C" {
    // fn 后面的函数名是 C 侧的名字
    // Rust 中调用它时用同一个名字（除非你用 #[link_name] 指定不同的名字）
    fn abs(input: i32) -> i32;
    fn sqrt(input: f64) -> f64;
}
```

Rust 自身支持多种 ABI 字符串。最常见的包括 `"C"`（默认 C 调用约定）、`"system"`（目标平台的系统调用约定——Windows 上是 `stdcall`，其他地方是 `"C"`）、以及各种明确的调用约定如 `"stdcall"`、`"fastcall"`、`"cdecl"` 等。

当你声明了外部函数后，调用它必须在 `unsafe` 块中进行，因为编译器无法验证 C 侧的实现是否真的匹配你的声明——签名不匹配是未定义行为。

```rust
// 在 unsafe 块中调用外部函数
extern "C" {
    fn abs(input: i32) -> i32;
}

fn main() {
    let x: i32 = -42;
    // 调用 FFI 函数必须使用 unsafe——编译器无法验证 C 侧实现的正确性
    let result: i32 = unsafe { abs(x) };
    assert_eq!(result, 42);
}
```

### 1.2 Rust 函数暴露给 C

`extern "C"` 也可以用在 Rust 函数定义上，使其可以被 C 代码调用。注意此时函数体内不能使用 Rust 特有的不可移植特性（比如非 C 兼容的 enum 作为参数），但函数体内部可以正常使用任何 Rust 特性。

```rust
// 定义一个 Rust 函数，使用 C ABI 导出——C 代码可以调用它
#[no_mangle]  // 防止 Rust 编译器修改函数名（name mangling）
pub extern "C" fn add_numbers(a: i32, b: i32) -> i32 {
    // 函数体内部可以使用任何 Rust 特性——对外接口是 C 兼容的
    let result = a.checked_add(b).unwrap_or(i32::MAX);
    println!("Rust side: computing {} + {} = {}", a, b, result);
    result
}
```

`#[no_mangle]` 属性告诉 Rust 编译器不要对函数名进行名称修饰（name mangling）。名称修饰是 Rust 将函数名转换为链接器友好的唯一标识符的过程——它包含模块路径、泛型参数等信息。对于 C FFI，我们必须关闭名称修饰，这样 C 侧才能通过 `add_numbers` 这个名字找到我们的函数。

### 1.3 `extern "C" { fn ... }` 与 `unsafe` 的关系

一个值得深思的问题是：为什么每个 FFI 调用都必须是 `unsafe` 的？根本原因在于 Rust 编译器无法检查以下任何一项：

**签名不匹配。** 如果你声明 C 函数 `fn foo(x: i32) -> i32` 而实际的 C 实现是 `void foo(double)`，编译器不会报错，但运行时会读取错误的内存、产生未定义行为。

**内存安全性。** C 函数可能返回一个指针，指向它内部栈上分配的缓冲区——Rust 无法跟踪这个指针的生命周期。C 函数可能在任何时候 free 掉 Rust 还持有的内存。C 函数可能写入数组越界的位置。

**线程安全性。** C 函数可能不是线程安全的，但 Rust 编译器没有关于它的任何信息——它可能在持有全局锁的情况下被调用，导致死锁。

**纯度。** Rust 假设函数调用不会修改"不相关"的内存——但 C 函数可能通过全局变量、`longjmp` 或其他机制破坏这个假设。

```rust
// 外部函数声明本身不需要 unsafe——声明是一种信任的承诺
extern "C" {
    fn strlen(s: *const std::os::raw::c_char) -> usize;
}

// 调用时才需要 unsafe——实际执行那个承诺
fn my_strlen(s: &str) -> usize {
    // 使用 CString 确保空终止
    let c_str = std::ffi::CString::new(s).expect("no null bytes in input");
    unsafe {
        // unsafe 块表示"我已检查过此调用的前提条件"
        strlen(c_str.as_ptr())
    }
}
```

## 2. `#[link]` —— 链接外部库

### 2.1 链接 C 标准库中的函数

Rust 默认链接了 C 的运行时库（如 libc），所以声明 `extern "C"` 块中的标准 C 库函数通常可以直接使用。但对于非标准库，你需要用 `#[link]` 属性告诉链接器去哪里找。

```rust
// 链接一个叫 "m" 的库——这是 Unix 上的 C 数学库（libm.so / libm.a）
// 注意：name 参数不带 "lib" 前缀和扩展名
#[link(name = "m")]
extern "C" {
    fn cos(x: f64) -> f64;
    fn sin(x: f64) -> f64;
    fn tan(x: f64) -> f64;
}

fn compute_trig(angle_degrees: f64) -> (f64, f64, f64) {
    let rad = angle_degrees * std::f64::consts::PI / 180.0;
    unsafe {
        (cos(rad), sin(rad), tan(rad))
    }
}
```

### 2.2 静态链接 vs 动态链接

`#[link]` 支持通过修饰符控制链接方式。静态链接（`kind = "static"`）将库的代码直接嵌入到可执行文件中；动态链接（`kind = "dylib"`）在运行时加载库。默认行为取决于目标平台上能找到什么类型的库文件。

```rust
// 静态链接——库代码嵌入最终二进制，不依赖运行时的 .so/.dll
#[link(name = "z", kind = "static")]
extern "C" {
    // zlib 的压缩函数
    fn compress(
        dest: *mut u8,
        dest_len: *mut u64,
        source: *const u8,
        source_len: u64,
    ) -> i32;
}

// 动态链接——需要 .so（Linux）/ .dylib（macOS）/ .dll（Windows）在运行时可用
#[link(name = "ssl", kind = "dylib")]
extern "C" {
    fn SSL_library_init() -> i32;
}
```

静态链接和动态链接各有利弊。静态链接的产物不依赖外部库文件，部署简单，但二进制体积更大，且库的 bug 修复需要重新编译你的程序。动态链接共享系统的库文件，体积小、可以享受系统的安全更新，但存在"依赖地狱"——用户系统上必须有正确版本的动态库。

除了 `static` 和 `dylib`，Rust 还支持其他链接方式：`static-nobundle` 强制静态链接但不把库打包进 rlib；`framework` 用于 macOS 的 Framework bundle；`raw-dylib` 用于 Windows 的延迟加载 DLL。

### 2.3 通过 build.rs 指定链接路径

当库不在系统默认搜索路径中时，我们需要告诉 Rust 编译系统去哪里找。这一般通过 `build.rs` 构建脚本完成，它会在编译之前运行。

以下是一个 build.rs 的典型模式：

```rust
// build.rs —— 在编译前运行，负责配置链接路径
fn main() {
    // 告诉 cargo 在哪里找库文件
    println!("cargo:rustc-link-search=native=/path/to/custom/libs");
    // 告诉 cargo 链接什么库
    println!("cargo:rustc-link-lib=static=my_custom_lib");

    // 如果库路径需要在编译的 crate 的目录结构中找到：
    let manifest_dir = std::env::var("CARGO_MANIFEST_DIR").unwrap();
    println!("cargo:rustc-link-search=native={}/native-libs", manifest_dir);

    // 对于有不同平台版本的库，可以条件编译：
    #[cfg(target_os = "linux")]
    println!("cargo:rustc-link-search=native={}/linux-libs", manifest_dir);
    #[cfg(target_os = "windows")]
    println!("cargo:rustc-link-search=native={}/win-libs", manifest_dir);
}
```

## 3. C 类型映射 —— 从 c_int 到 size_t

### 3.1 基础类型映射表

FFI 的核心挑战之一是类型系统的对齐：C 的类型是平台相关的（`int` 在 16 位平台上是 2 字节，在 32/64 位平台上是 4 字节），而 Rust 的固定宽度类型（`i32` 始终是 4 字节）提供了更强的保证。

Rust 标准库的 `std::os::raw` 模块提供了 C 兼容的类型别名，通常通过 `libc` crate 使用：

| C 类型 | Rust 类型 (libc / std::os::raw) | 备注 |
|---|---|---|
| `char` | `c_char` | 在大多数平台是 `i8`，但 ARM/AArch64 上可能是 `u8` |
| `signed char` | `c_schar` | 明确的有符号 char，`i8` |
| `unsigned char` | `c_uchar` | 明确的无符号 char，`u8` |
| `short` | `c_short` | `i16` |
| `unsigned short` | `c_ushort` | `u16` |
| `int` | `c_int` | `i32` |
| `unsigned int` | `c_uint` | `u32` |
| `long` | `c_long` | 平台相关：32位=4字节 / 64位 Unix=8字节 / 64位 Windows=4字节 |
| `unsigned long` | `c_ulong` | 对应 c_long 的无符号版本 |
| `long long` | `c_longlong` | `i64` |
| `unsigned long long` | `c_ulonglong` | `u64` |
| `float` | `c_float` | `f32` |
| `double` | `c_double` | `f64` |
| `void*` | `*mut c_void` | 指向任意类型的指针 |
| `size_t` | `size_t` (libc) / `usize` | 无符号整数，足以表示对象大小 |
| `ssize_t` | `ssize_t` (libc) / `isize` | 有符号的 size_t |
| `int8_t` (C99) | `i8` | 精确宽度的有符号整数 |
| `uint32_t` (C99) | `u32` | 精确宽度的无符号整数 |

### 3.2 直接使用这些类型的示例

```rust
// 使用 libc crate 提供的 C 兼容类型声明 FFI 函数
// 在 Cargo.toml 中添加：libc = "0.2"
use libc::{c_int, c_char, c_void, size_t, ssize_t};

extern "C" {
    // 对应 C 的 ssize_t write(int fd, const void *buf, size_t count);
    fn write(fd: c_int, buf: *const c_void, count: size_t) -> ssize_t;

    // 对应 C 的 char *strerror(int errnum);
    fn strerror(errnum: c_int) -> *mut c_char;
}

fn write_message(fd: i32, msg: &str) -> Result<usize, String> {
    let bytes = msg.as_bytes();
    let ret: isize = unsafe {
        write(
            fd as c_int,
            bytes.as_ptr() as *const c_void,
            bytes.len() as size_t,
        )
    };
    if ret >= 0 {
        Ok(ret as usize)
    } else {
        // 获取 errno 对应的错误描述
        // 实际项目中应使用 libc::__errno_location() 等
        Err(format!("write failed with return code {}", ret))
    }
}
```

### 3.3 指针类型映射

C 中的指针通常映射为 Rust 的裸指针类型。一般规则是：如果 C 函数只读取指针指向的内容，用 `*const T`；如果它会修改，用 `*mut T`。

```rust
use std::os::raw::{c_char, c_int};

extern "C" {
    // C: int printf(const char *format, ...);
    // format 是只读的，所以用 *const c_char
    fn printf(format: *const c_char, ...) -> c_int;

    // C: void *memset(void *s, int c, size_t n);
    // s 会被修改，所以用 *mut c_void
    fn memset(s: *mut std::os::raw::c_void, c: c_int, n: usize) -> *mut std::os::raw::c_void;
}
```

特别注意 `*const c_char` 和 `*const u8` 是不同的类型——虽然它们在内存层面的表示相同，但 Rust 的类型系统将它们视为不同的类型。如果你有一个 Rust `&str`，需要通过 `CString` 来创建 C 兼容的字符串，再用 `.as_ptr()` 获取裸指针。

## 4. `#[repr(C)]` —— 结构体内存布局兼容

### 4.1 为什么需要 repr(C)

Rust 默认的结构体布局（`repr(Rust)`）不做任何关于字段顺序、对齐或填充的保证。编译器有权重新排列字段以减小结构体大小（reorder optimization），或者在不同版本间改变布局。这对于 C FFI 是不可接受的——C 代码期望结构体的字段按声明顺序排列，并且遵循 C 的对齐规则。

`#[repr(C)]` 锁定结构体的内存布局，使其与 C 编译器生成的布局一致——字段按声明顺序排列，每个字段按其自然对齐放置，必要时插入填充（padding）。

```rust
// 没有 #[repr(C)]——编译器可能重新排列字段
// 此结构体在 FFI 中使用是危险的
struct RustDefault {
    a: u8,
    b: u32,
    c: u16,
}
// 编译器可能把 b 放在最前面，a 和 c 放在后面——C 侧无法预测

// 有 #[repr(C)]——布局与 C 完全一致
#[repr(C)]
struct CCompatible {
    a: u8,   // 偏移 0，占 1 字节
    // 填充 3 字节——b 的对齐要求是 4 字节（偏移必须是 4 的倍数）
    b: u32,  // 偏移 4，占 4 字节
    c: u16,  // 偏移 8，占 2 字节
    // 填充 2 字节——结构体大小必须是最大对齐（4）的倍数
    // 总大小：12 字节
}
```

### 4.2 analizing a #[repr(C)] 结构体

你可以用标准库的 `mem::offset_of!` 宏（Rust 1.77+）或通过 `unsafe` 检查字段偏移来验证 repr(C) 的布局：

```rust
use std::mem::{offset_of, size_of};

#[repr(C)]
struct Example {
    x: u8,    // 1 字节
    y: u32,   // 4 字节，需要 4 字节对齐 → 前面填充 3 字节
    z: u16,   // 2 字节，需要 2 字节对齐 → 刚好在偏移处
    // 尾部填充 2 字节，使总大小是 4 的倍数
}

fn verify_layout() {
    // 验证字段偏移
    assert_eq!(offset_of!(Example, x), 0);
    assert_eq!(offset_of!(Example, y), 4);  // 1 (x) + 3 (padding)
    assert_eq!(offset_of!(Example, z), 8);  // 4 + 4 (y)
    // 验证总大小
    assert_eq!(size_of::<Example>(), 12);   // 8 + 2 (z) + 2 (tail padding)
}
```

### 4.3 repr(C) 的枚举

Rust 的枚举在 C 中没有直接的对应物。`#[repr(C)]` 用于 enum 时的行为取决于变体是否有数据：

```rust
// 简单枚举（没有数据字段）——可以 repr(C)，行为类似 C enum
#[repr(C)]
enum Color {
    Red = 0,
    Green = 1,
    Blue = 2,
}

// 带数据的枚举——不能用 repr(C) 导出到 C！
// 因为 Rust 的枚举是有标签联合（tagged union），C 没有原生对应
// #[repr(C)]
// enum Value {
//     Int(i32),     // C 不能原生表示这个
//     Float(f64),   //
// }
// 对于带数据的枚举，需要使用 #[repr(C, u8)] 指定标签类型 + 手动布局
// 或者使用 struct + union 的组合
```

对于 C 侧的 `union`，Rust 也有对应的 `#[repr(C)] union` 语法。不过更常见的方式是使用 `libc` 或 `bindgen` 自动生成绑定来解决这些复杂问题。

## 5. 回调 —— 从 C 调用 Rust 函数指针

### 5.1 传递 extern "C" fn 作为回调

C 代码经常通过函数指针接受回调。Rust 可以把 `extern "C" fn` 类型转换为 C 可调用的函数指针，传给 C 侧的 qsort、信号处理、事件处理等函数。

```rust
use libc::{c_int, c_void};

// 模拟 C 的 qsort 函数签名
extern "C" {
    fn qsort(
        base: *mut c_void,
        num: usize,
        size: usize,
        compar: Option<extern "C" fn(*const c_void, *const c_void) -> c_int>,
    );
}

// 定义 Rust 侧的比较函数——必须是 extern "C" fn
extern "C" fn compare_ints(a: *const c_void, b: *const c_void) -> c_int {
    let a = unsafe { *(a as *const i32) };
    let b = unsafe { *(b as *const i32) };
    a.cmp(&b) as c_int
}

fn sort_with_qsort(data: &mut [i32]) {
    unsafe {
        qsort(
            data.as_mut_ptr() as *mut c_void,
            data.len(),
            std::mem::size_of::<i32>(),
            Some(compare_ints),  // 函数指针传给 C
        );
    }
}
```

### 5.2 闭包模拟——Trampoline 模式

C 的回调没有"上下文"概念——函数指针不带数据。但在实际场景中，我们经常需要在回调中访问一些状态。trampoline（蹦床）模式解决了这个问题：通过一个薄的 `extern "C"` 包装函数把上下文转换出来。

```rust
use std::ffi::c_void;

// 假设 C 库提供了一个遍历 API，回调签名是 void (*)(void *element, void *user_data)
// 我们把 user_data 指针转换为 Rust 的闭包引用
extern "C" {
    fn traverse_list(
        list: *mut c_void,
        callback: Option<extern "C" fn(element: *mut c_void, user_data: *mut c_void)>,
        user_data: *mut c_void,
    );
}

// Trampoline——extern "C" 包装函数
// 它的工作很简单：把 user_data 还原为 Rust 闭包，然后调用它
extern "C" fn trampoline(element: *mut c_void, user_data: *mut c_void) {
    // user_data 是我们传入的 Box<&mut dyn FnMut(*mut c_void)> 的裸指针
    let callback: &mut &mut dyn FnMut(*mut c_void) = unsafe {
        &mut *(user_data as *mut &mut dyn FnMut(*mut c_void))
    };
    callback(element);
}

// 安全的 Rust 包装——对外暴露熟悉的 Rust API
fn traverse<F: FnMut(*mut c_void)>(list: *mut c_void, mut callback: F) {
    // 把闭包包装为一个 trait object，然后 Box 它并转换为裸指针
    let mut cb: &mut dyn FnMut(*mut c_void) = &mut callback;
    let user_data: *mut c_void = &mut cb as *mut _ as *mut c_void;

    unsafe {
        traverse_list(list, Some(trampoline), user_data);
    }
}
```

Trampoline 模式的关键在于：`user_data` 指针是一个"胖指针"（trait object），包含数据指针和 vtable 指针。`extern "C" fn` 不能是闭包，所以我们需要这一层间接——用一个独立的函数提取上下文，再分派到真正的闭包。

## 6. Panic 穿越 FFI 边界 —— `catch_unwind`

### 6.1 为什么 panic 穿越 FFI 是 UB

Rust 的 panic 通过栈展开（stack unwinding）实现——调用链上的每个函数都运行其析构函数（drop），直到被 `catch_unwind` 捕获或终止线程。问题在于：C 栈帧不知道 Rust 的展开机制。

如果 Rust 的 panic 在 `extern "C" fn` 中没有被捕获，unwinding 会尝试展开 C 的栈帧。C 的栈帧没有 drop 信息、没有 landing pad、没有适当的异常处理表。结果是未定义行为——最常见的是程序直接崩溃（segfault 或 abort）。

```rust
// BAD: panic 可能穿越 FFI 边界
pub extern "C" fn risky_callback(value: i32) -> i32 {
    if value < 0 {
        // 这个 panic 会穿越 C ABI 边界——未定义行为！
        panic!("value cannot be negative");
    }
    value * 2
}
```

### 6.2 用 catch_unwind 防止 panic 穿越

每个暴露给 C 的 `extern "C" fn` 都应该在顶层包装 `catch_unwind`。

```rust
use std::panic::{catch_unwind, AssertUnwindSafe};

// GOOD: 用 catch_unwind 守卫 FFI 边界
pub extern "C" fn safe_callback(value: i32) -> i32 {
    // catch_unwind 捕获任何 panic，防止它穿越 FFI
    let result = catch_unwind(AssertUnwindSafe(|| {
        if value < 0 {
            panic!("value cannot be negative");
        }
        value * 2
    }));

    match result {
        Ok(ret) => ret,
        Err(_) => {
            // panic 被捕获了——我们可以：
            // 1. 返回错误码
            // 2. 记录日志
            // 3. 通过某个全局变量传递错误
            eprintln!("Rust callback panicked with value {}", value);
            -1  // 返回一个错误哨兵值
        }
    }
}
```

`AssertUnwindSafe` 是一个零大小的标记类型，表示"我断言这段代码即使 unwind 也是安全的"。这不是真正的安全保证——它只是绕过 `catch_unwind` 的类型检查，允许你捕获可能不满足 `UnwindSafe` 的闭包中的 panic。在 FFI 场景中这通常是正确的做法，因为防止 UB 比维护 UnwindSafe 的不变量更重要。

### 6.3 跨 FFI 边界的最佳实践模式

综合以上知识，以下是一个安全的 FFI 导出函数的完整模板：

```rust
use std::panic::{catch_unwind, AssertUnwindSafe};

/// 一个展示完整 FFI 注册模式的 Rust 导出函数
/// 包含：catch_unwind 守卫 + 错误处理 + 日志
#[no_mangle]
pub extern "C" fn rust_process_data(
    input: *const u8,
    input_len: usize,
    output: *mut u8,
    output_capacity: usize,
    output_len: *mut usize,
) -> i32 {
    // 步骤 1：catch_unwind 作为最外层——防止 panic 穿越 FFI
    let result = catch_unwind(AssertUnwindSafe(|| {
        // 步骤 2：验证并转换入参
        if input.is_null() || output.is_null() || output_len.is_null() {
            return Err("null pointer argument".to_string());
        }

        let input_slice = unsafe {
            std::slice::from_raw_parts(input, input_len)
        };

        // 步骤 3：核心业务逻辑
        let processed = process_inner(input_slice);

        // 步骤 4：检查输出缓冲区大小
        if processed.len() > output_capacity {
            return Err(format!(
                "output buffer too small: need {}, got {}",
                processed.len(),
                output_capacity
            ));
        }

        // 步骤 5：写入输出
        unsafe {
            std::ptr::copy_nonoverlapping(
                processed.as_ptr(),
                output,
                processed.len(),
            );
            *output_len = processed.len();
        }

        Ok(())
    }));

    // 步骤 6：返回 C 友好的错误码
    match result {
        Ok(Ok(())) => 0,          // 成功
        Ok(Err(msg)) => {
            eprintln!("[rust_process_data] error: {}", msg);
            -1  // 逻辑错误
        }
        Err(_) => {
            eprintln!("[rust_process_data] panic occurred");
            -2  // panic——严重错误
        }
    }
}

// 内部实现——使用纯 Rust，可以 panic（会被外层捕获）
fn process_inner(data: &[u8]) -> Vec<u8> {
    data.iter().map(|b| b.wrapping_mul(2)).collect()
}
```

## 7. 跨 FFI 的内存管理 —— 谁分配，谁释放

### 7.1 基本原则

跨 FFI 的内存管理是 FFI 中最容易出问题的领域。核心原则是：**分配内存的代码负责释放内存**，或者双方约定明确的规则。常见的做法有几种：

**模式 A：调用者分配，调用者释放** —— C 侧分配缓冲区，Rust 写入数据。这是最安全的模式之一。

**模式 B：被调用者分配，调用者释放** —— Rust 通过 `Box::into_raw` 将所有权转移给 C，C 用完后通过特定的 free 函数归还给 Rust。

**模式 C：C 分配，Rust 包装为安全类型** —— C 创建对象，Rust 通过 `Drop` trait 在 Rust 类型销毁时自动调用 C 的 free 函数。

**模式 D：共享所有权** —— 双方各持有一个引用计数（如 Arc），释放协调一致。

### 7.2 模式 A：调用者分配

```rust
use std::os::raw::c_char;

// C 侧分配缓冲区，Rust 侧填充数据
// 对应 C 签名：int fill_buffer(char *buf, size_t buf_size, size_t *written);
extern "C" {
    fn fill_buffer(
        buf: *mut c_char,
        buf_size: usize,
        written: *mut usize,
    ) -> i32;
}

fn safe_fill() -> Result<String, String> {
    let mut buf: Vec<u8> = vec![0u8; 1024];  // Rust 侧分配
    let mut written: usize = 0;

    let ret = unsafe {
        fill_buffer(
            buf.as_mut_ptr() as *mut c_char,
            buf.len(),
            &mut written as *mut usize,
        )
    };

    if ret != 0 {
        return Err(format!("C function fill_buffer failed: {}", ret));
    }

    // 截断到实际写入的长度并转换为 String
    buf.truncate(written);
    String::from_utf8(buf).map_err(|e| format!("invalid UTF-8: {}", e))
}
```

### 7.3 模式 B：Rust 分配，C 释放

当 Rust 需要在堆上创建一个 C 结构体并交给 C 管理时，使用 `Box::into_raw` 将所有权转移为裸指针：

```rust
#[repr(C)]
struct RustObject {
    id: u32,
    name: *mut std::os::raw::c_char,
    data: *mut std::os::raw::c_void,
}

/// C 调用这个函数来创建一个 RustObject
#[no_mangle]
pub extern "C" fn rust_object_create(id: u32) -> *mut RustObject {
    let obj = Box::new(RustObject {
        id,
        name: std::ptr::null_mut(),
        data: std::ptr::null_mut(),
    });
    // Box::into_raw 将所有权转移给调用者——调用者负责释放
    Box::into_raw(obj)
}

/// C 调用这个函数来释放 RustObject
/// 关键：由 Rust 侧执行释放，确保所有 Rust 字段的 Drop 被正确调用
#[no_mangle]
pub extern "C" fn rust_object_destroy(ptr: *mut RustObject) {
    if ptr.is_null() {
        return;
    }
    // unsafe: 从裸指针重建 Box——Rust 的 Drop 被正确调用
    unsafe {
        // 先释放内部的 C 字符串（如果有的话）
        let obj = &mut *ptr;
        if !obj.name.is_null() {
            // 使用 CString::from_raw 来正确释放 C 字符串
            let _ = std::ffi::CString::from_raw(obj.name);
        }

        // Box::from_raw 重建所有权并立即 drop
        drop(Box::from_raw(ptr));
    }
}
```

`Box::from_raw` 是 `Box::into_raw` 的逆操作——它从裸指针重建 `Box<T>`，使 Rust 重新获得所有权并能在离开作用域时调用 `T` 的 `Drop`。这里有一个重要的注意事项：`Box::from_raw` 的指针必须是从 `Box::into_raw` 获得的，且只能调用一次——多次 `from_raw` 同一个指针是 double-free。

### 7.4 模式 C：RAII Drop 包装器

这是最优雅的模式——创建一个 Rust 新类型包装 C 的不透明句柄，在 `Drop` 中调用 C 的销毁函数：

```rust
// 假设有一个 C 库提供文件句柄 API：
//   FileHandle* file_open(const char *path);
//   int file_read(FileHandle* f, char *buf, size_t len);
//   void file_close(FileHandle* f);

/// 安全的 RAII 包装器——包装 C 的文件句柄
pub struct SafeFile {
    handle: *mut libc::c_void,  // 不透明句柄
}

impl SafeFile {
    /// 创建——调用 C 的 open 函数
    pub fn open(path: &str) -> Result<Self, String> {
        let c_path = std::ffi::CString::new(path)
            .map_err(|e| format!("invalid path: {}", e))?;
        // 假设 file_open 是外部 C 函数
        // let handle = unsafe { file_open(c_path.as_ptr()) };
        // 这里用占位符
        let handle: *mut libc::c_void = std::ptr::null_mut();

        if handle.is_null() {
            Err(format!("failed to open file: {}", path))
        } else {
            Ok(SafeFile { handle })
        }
    }

    /// 读取——调用 C 的 read 函数
    pub fn read(&self, buf: &mut [u8]) -> Result<usize, String> {
        // unsafe {
        //     let n = file_read(self.handle, buf.as_mut_ptr() as *mut _, buf.len());
        //     if n < 0 { Err("read failed".into()) } else { Ok(n as usize) }
        // }
        Ok(0)
    }
}

impl Drop for SafeFile {
    fn drop(&mut self) {
        // 当 SafeFile 离开作用域时，自动调用 C 的 close 函数
        if !self.handle.is_null() {
            // unsafe { file_close(self.handle); }
        }
        // 注意：将 handle 置为 null 防止 double-free
        // （虽然 Drop 一般只调用一次）
    }
}

// 使用示例：File 在离开作用域时自动关闭——RAII 保护
// {
//     let file = SafeFile::open("/tmp/test.txt")?;
//     let mut buf = [0u8; 1024];
//     file.read(&mut buf)?;
// } // ← Drop 自动运行，file_close 被调用
```

## 8. 实战示例 —— 包装 SQLite 的 C API 核心步骤

### 8.1 SQLite 核心 C API 回顾

SQLite 的 C API 是理解 FFI 封装的绝佳案例——它涉及不透明指针、字符串处理、错误检查和资源管理。以下是三个核心函数：

```c
// C 侧的核心 API（简化）
sqlite3 *db;
int rc = sqlite3_open("test.db", &db);  // 打开数据库，失败返回非零
int sqlite3_exec(sqlite3*, const char *sql, callback, void*, char **errmsg);
int sqlite3_close(sqlite3*);             // 关闭数据库
```

### 8.2 Rust 侧的类型声明

```rust
use std::ffi::{CStr, CString};
use std::os::raw::{c_char, c_int, c_void};

// 用空枚举模拟 SQLite 的不透明类型——无法在 Rust 中直接创建值
// 只能通过指针使用（类似于 extern type）
#[repr(C)]
pub struct sqlite3 {
    _private: [u8; 0],  // 零大小字段——不能被外部构造
}

// FFI 函数声明（通常由 bindgen 自动生成，这里手动声明作为示例）
extern "C" {
    fn sqlite3_open(filename: *const c_char, pp_db: *mut *mut sqlite3) -> c_int;
    fn sqlite3_close(db: *mut sqlite3) -> c_int;
    fn sqlite3_exec(
        db: *mut sqlite3,
        sql: *const c_char,
        callback: Option<
            extern "C" fn(
                user_data: *mut c_void,
                argc: c_int,
                argv: *mut *mut c_char,
                az_col_name: *mut *mut c_char,
            ) -> c_int,
        >,
        user_data: *mut c_void,
        errmsg: *mut *mut c_char,
    ) -> c_int;
    fn sqlite3_free(ptr: *mut c_void);
}
```

### 8.3 安全的 Rust 包装

```rust
/// 安全的数据库连接包装器
pub struct Database {
    db: *mut sqlite3,
}

impl Database {
    pub fn open(path: &str) -> Result<Self, String> {
        let c_path = CString::new(path).map_err(|e| format!("invalid path: {}", e))?;
        let mut db: *mut sqlite3 = std::ptr::null_mut();

        let rc = unsafe {
            sqlite3_open(c_path.as_ptr(), &mut db as *mut *mut sqlite3)
        };

        if rc != 0 {
            // 即使打开失败，SQLite 仍可能分配了 db 结构体——需要关闭
            if !db.is_null() {
                unsafe { sqlite3_close(db); }
            }
            return Err(format!("sqlite3_open failed with code {}", rc));
        }

        Ok(Database { db })
    }

    pub fn exec(&self, sql: &str) -> Result<(), String> {
        let c_sql = CString::new(sql).map_err(|e| format!("invalid SQL: {}", e))?;
        let mut errmsg: *mut c_char = std::ptr::null_mut();

        let rc = unsafe {
            sqlite3_exec(
                self.db,
                c_sql.as_ptr(),
                None,                    // 不需要回调
                std::ptr::null_mut(),     // 不需要 user_data
                &mut errmsg as *mut *mut c_char,
            )
        };

        if rc != 0 {
            let error_msg = if !errmsg.is_null() {
                let msg = unsafe {
                    CStr::from_ptr(errmsg).to_string_lossy().into_owned()
                };
                // 必须用 sqlite3_free 释放 SQLite 分配的错误消息字符串
                unsafe { sqlite3_free(errmsg as *mut c_void); }
                msg
            } else {
                format!("unknown error code {}", rc)
            };
            return Err(error_msg);
        }

        Ok(())
    }
}

impl Drop for Database {
    fn drop(&mut self) {
        if !self.db.is_null() {
            unsafe { sqlite3_close(self.db); }
        }
    }
}
```

### 8.4 使用示例

```rust
fn example_sqlite_usage() -> Result<(), String> {
    let db = Database::open(":memory:")?;  // 内存数据库

    db.exec(
        "CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT NOT NULL);"
    )?;

    db.exec(
        "INSERT INTO users (name) VALUES ('Alice'), ('Bob');"
    )?;

    // db 离开作用域时自动调用 sqlite3_close
    Ok(())
}
```

这个示例展示了 FFI 封装的完整模式：用 `CString` 转换字符串、用裸指针操作不透明类型、在 `Drop` 中自动释放资源、正确释放 C 分配的错误消息。这是你在所有 C FFI 项目中都会反复使用的模式。

## 9. 常见陷阱与调试方法

### 9.1 最常见的错误清单

FFI 编程中的许多错误都可以归为以下几类。每一类我都给出一个具体的错误示例和正确做法。

**陷阱 1：忘记 `#[repr(C)]`。** 这是最常见的 FFI bug。没有 `#[repr(C)]`，Rust 编译器可能重新排列结构体字段或使用与 C 不同的对齐方式。

```rust
// WRONG —— 缺少 #[repr(C)]，编译器可能重新布局
struct BadFFI {
    count: u32,
    data: *const u8,
}

// RIGHT —— 显式指定 C 布局
#[repr(C)]
struct GoodFFI {
    count: u32,
    data: *const u8,
}
```

**陷阱 2：调用约定不匹配。** 特别是在 Windows 上，C 函数的默认调用约定和 Win32 API 的 `stdcall` 不同。混用会导致栈损坏。

```rust
// 在 Windows 上这是错误的 —— Win32 API 通常使用 stdcall
// extern "C" {
//     fn MessageBoxW(
//         hwnd: *mut c_void, text: *const u16, caption: *const u16, utype: u32,
//     ) -> i32;
// }

// 在 Windows 上是正确的 —— 使用 "system" 或显式的 "stdcall"
#[cfg(target_os = "windows")]
extern "system" {
    fn MessageBoxW(
        hwnd: *mut std::os::raw::c_void,
        text: *const u16,
        caption: *const u16,
        utype: u32,
    ) -> i32;
}
```

**陷阱 3：字符串缺少空终止符。** C 字符串必须以 `\0` 结尾。Rust 的 `&str` 没有这个保证。必须使用 `CString` 而不是直接传 `str.as_ptr()`。

```rust
// WRONG —— Rust 字符串不一定有空终止符
// unsafe { some_c_function(my_str.as_ptr() as *const c_char); }

// RIGHT —— 用 CString 创建空终止的字符串
let c_string = CString::new("hello").unwrap();
unsafe { some_c_function(c_string.as_ptr()); }
```

**陷阱 4：悬垂指针。** C 函数可能返回只在调用期间有效的指针。你需要立即复制数据而不是保存原始指针。

```rust
// WRONG —— c_str 在语句结束后释放，ptr 变为悬垂指针
// let ptr = CString::new("hello").unwrap().as_ptr();

// RIGHT —— 保持 CString 所有权
let c_str = CString::new("hello").unwrap();
let ptr = c_str.as_ptr(); // ptr 只在 c_str 的生命周期内有效
```

**陷阱 5：类型宽度不匹配。** C 的 `long` 在 64 位 Linux 上是 8 字节，在 64 位 Windows 上是 4 字节。如果你用 `i64` 映射它，在 Windows 上可能溢出或读取越界。

```rust
// 用 c_long 而不是直接猜测宽度
use std::os::raw::c_long;

extern "C" {
    fn takes_long(val: c_long);
}
```

### 9.2 调试 FFI 的工具和技术

**方法 1：Valgrind / AddressSanitizer。** 这些工具可以检测到 FFI 中最危险的 bug——内存越界、use-after-free、double-free。Rust 可以通过 nightly 编译器的 sanitizer 支持来使用它们。

```rust
// 用 sanitizer 编译（nightly）：
// RUSTFLAGS="-Z sanitizer=address" cargo build
// 或者添加：
// [build]
// rustflags = ["-Z", "sanitizer=address"]
```

**方法 2：`#[link_name]` 验证。** 如果链接器报 "undefined reference"，可以先检查你声明的函数名是否与 C 侧完全一致。可以在 C 侧先用 `nm` 或 `dumpbin` 检查导出符号列表。

**方法 3：分步测试。** 不要一次性封装一个完整的 C 库。先测试一个最简单的函数调用，验证链接、类型映射和调用约定都正确，然后逐步扩展。

```rust
use std::time::{Duration, SystemTime, UNIX_EPOCH};

// 与 C struct stat 兼容的布局
// 这里只声明我们需要用到的字段，但必须保持 repr(C)
#[repr(C)]
struct Stat {
    st_dev: u64,
    st_ino: u64,
    st_mode: u32,
    st_nlink: u32,
    st_uid: u32,
    st_gid: u32,
    st_rdev: u64,
    st_size: i64,     // off_t 通常是 i64
    st_blksize: i64,
    st_blocks: i64,
    st_atime: i64,    // time_t
    st_mtime: i64,    // time_t——最后修改时间
    st_ctime: i64,    // time_t
}

extern "C" {
    // stat 传递路径和输出缓冲区，成功返回 0，失败返回 -1
    fn stat(path: *const c_char, buf: *mut Stat) -> c_int;
}

struct FileInfo {
    size: u64,
    modified: SystemTime,
    is_dir: bool,
}

fn file_metadata(path: &str) -> Result<FileInfo, String> {
    let c_path = CString::new(path).map_err(|e| format!("invalid path: {}", e))?;
    let mut stat_buf: Stat = unsafe { std::mem::zeroed() };

    let rc = unsafe { stat(c_path.as_ptr(), &mut stat_buf as *mut Stat) };
    if rc != 0 {
        return Err(format!("stat failed for path: {}", path));
    }

    let size = stat_buf.st_size as u64;

    // 将 Unix 时间戳转换为 SystemTime
    let modified = UNIX_EPOCH + Duration::from_secs(stat_buf.st_mtime as u64);

    // S_ISDIR 检查：模式位掩码 0170000 表示文件类型，0040000 表示目录
    // POSIX 定义：S_IFMT = 0o170000, S_IFDIR = 0o040000
    const S_IFMT: u32 = 0o170000;
    const S_IFDIR: u32 = 0o040000;
    let is_dir = (stat_buf.st_mode & S_IFMT) == S_IFDIR;

    Ok(FileInfo { size, modified, is_dir })
}
```

### 练习 2 答案

原代码的问题：
1. `getenv` 的签名错误——C 的 `getenv` 接受 `*const c_char`（空终止 C 字符串），返回 `*mut c_char`（可能为 NULL），不是 Rust 的 `&str` 和 `String`
2. 没有正确处理空指针——`getenv` 在变量不存在时返回 null
3. 没有用 `CString` 来构造输入参数——直接传 `&str` 没有空终止符

```rust
use std::ffi::{CStr, CString};
use std::os::raw::c_char;

// 修复后的外部函数声明
extern "C" {
    // C 签名：char *getenv(const char *name);
    // 返回 NULL 表示变量不存在
    fn getenv(name: *const c_char) -> *mut c_char;
}

fn safe_getenv(key: &str) -> Option<String> {
    // 步骤 1：构造空终止的 C 字符串
    let c_key = CString::new(key).ok()?;

    // 步骤 2：调用外部函数
    let ptr = unsafe { getenv(c_key.as_ptr()) };

    // 步骤 3：检查空指针（变量不存在）
    if ptr.is_null() {
        return None;
    }

    // 步骤 4：将 C 字符串转换为 Rust String
    // 注意：不要释放 ptr——getenv 返回的指针由 C 运行时管理
    unsafe {
        let c_str = CStr::from_ptr(ptr);
        Some(c_str.to_string_lossy().into_owned())
    }
}
```

### 练习 3 答案

```rust
use std::ffi::{c_void, CStr};
use std::os::raw::c_int;

// FFI 声明
type TraverseCallback = extern "C" fn(id: c_int, name: *const std::os::raw::c_char, user_data: *mut c_void);

extern "C" {
    fn traverse_records(cb: TraverseCallback, user_data: *mut c_void) -> c_int;
}

// Trampoline 函数——将 C 回调转换为 Rust 闭包调用
extern "C" fn trampoline(
    id: c_int,
    name: *const std::os::raw::c_char,
    user_data: *mut c_void,
) {
    // 从 user_data 还原 Rust 闭包
    let callback: &mut &mut dyn FnMut(i32, &str) -> bool = unsafe {
        &mut *(user_data as *mut &mut dyn FnMut(i32, &str) -> bool)
    };

    let name_str = if name.is_null() {
        ""
    } else {
        unsafe { CStr::from_ptr(name) }.to_str().unwrap_or("<invalid utf-8>")
    };

    callback(id as i32, name_str);
}

/// 安全的 Rust API——用户传递一个闭包
fn traverse_records_safe<F>(mut callback: F) -> i32
where
    F: FnMut(i32, &str) -> bool,
{
    // 把闭包转换为 trait object 的裸指针
    let mut cb: &mut dyn FnMut(i32, &str) -> bool = &mut callback;
    let user_data: *mut c_void = &mut cb as *mut _ as *mut c_void;

    unsafe {
        traverse_records(trampoline, user_data)
    }
}
```
