# 9.2 贡献 rustc 与 Clippy：从 Phase 7 到实战

> Phase 7 让你理解了编译器内部——现在是把理论转化为贡献的时候了。为 rustc 和 Clippy 做贡献是 Rust 生态中"技术含金量最高"的贡献形式之一——它直接改进每个 Rust 开发者的日常体验。但不要被"编译器贡献"这个词吓到——很多有价值的贡献不需要你理解整个编译管线。改进一条错误信息、修复一个 ICE、添加一个 Clippy lint——这些都在你的能力范围内。本节给你从"找到第一个 issue"到"PR 被 merge"的完整操作手册。

---

## 1. 从 Phase 7 到实战：你的起点

### 1.1 Phase 7 知识如何转化为贡献

```
Phase 7 的知识              →  可以直接做的贡献
─────────────────────────────────────────────────────────
7.1 编译管线（HIR/MIR）     →  写 Clippy lint（遍历 HIR）
                             →  添加 MIR 优化 pass
                             
7.2 借用检查器              →  改进 NLL 错误信息
                             →  修 borrow checker 的 false positive bug

7.3 Trait 解析              →  修 trait solver 的 bug
                             →  改进 overlap check 的错误信息

7.5 MIR 优化                →  实现新的 MIR 优化 pass
                             →  改进已有 pass 的覆盖范围

7.6 类型推导                →  修类型推导的 edge case
                             →  改进"类型不够"时的错误建议

7.7 增量编译                →  修增量编译的缓存失效 bug
                             →  优化 query 的依赖图

7.8 dev workflow            →  你已经知道 ./x.py 怎么用了！
```

### 1.2 最容易的首次贡献：改进诊断信息

```rust
// 这不是改编译器逻辑——是改编译器说的话
// 每个 Rust 开发者都遇到过不知所云的编译错误
// 改进一条错误信息 = 帮助数百万 Rust 开发者

// ━━━ 例子：改进一条 confusing 的错误信息 ━━━

// 现状：错误信息让人困惑
// error[E0382]: borrow of moved value: `x`
//   → 这告诉了你"发生了什么"，但没说"怎么修"

// 贡献：添加帮助信息
// error[E0382]: borrow of moved value: `x`
//   help: consider cloning `x` before the move
//   help: or, if you don't need `x` afterwards, use `let _ = x;`
//
// 这种改动通常在 20 行代码以内就完成了！
```

## 2. Bug 修复工作流

### 2.1 从 ICE 到 PR 的完整流程

```
┌─────────────────────────────────────────────────────────┐
│  Bug 修复五步法                                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Step 1: 最小化复现（MCVE）                              │
│    用户报告了一个 ICE（Internal Compiler Error）          │
│    → 把 1000 行的项目代码缩减为 10 行的重现用例           │
│    → 移除不相关的依赖、类型、逻辑                        │
│                                                         │
│  Step 2: 定位问题代码                                    │
│    → rustc 报错时会打印 backtrace                        │
│    → 从 backtrace 找到 panic/crash 的文件和行号          │
│    → 用 rust-analyzer "go to definition" 追踪逻辑        │
│                                                         │
│  Step 3: 理解为什么 crash                                │
│    → 读相关代码——为什么这里假设了某个条件成立？           │
│    → 在你的 MCVE 上，这个条件为什么被违反了？             │
│    → 画一张小图理解状态流转                              │
│                                                         │
│  Step 4: 写修复                                         │
│    → 通常是：添加一个检查、修改一个条件、处理 edge case   │
│    → 添加你的 MCVE 作为测试用例                          │
│                                                         │
│  Step 5: 提交 PR                                        │
│    → PR 描述：before（ICE）→ after（正常或更好的错误）    │
│    → 标注 @rustbot ready                                │
│    → 等待 review                                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.2 ICE 的最小化技巧

```bash
# 工具：cargo-minimize（自动缩减导致 ICE 的代码）
cargo install cargo-minimize

# 使用：
cargo minimize  # 在触发 ICE 的项目目录下运行
# → 自动删除不相关的代码，保留最小的触发用例

# 手动最小化的思路：
# 1. 删除项目依赖，用 std 类型替代
# 2. 删除不相关的函数和数据结构
# 3. 简化类型参数（把泛型换成具体类型试试）
# 4. 把复杂的表达式拆开（逐步删除子表达式）
# 目标：10-20 行的独立文件，可以直接放在 issue 里
```

## 3. Clippy 贡献深化

### 3.1 从 7.10 到真实贡献

> Phase 7.10 的 Clippy 综合实战让你写了一个 lint——现在我们要讨论的是"如何让你的 lint 被接受"。

```
Phase 7.10 教了你              Phase 9.2 补充
─────────────────────         ─────────────────────
怎么写一个 lint              怎么写一个"好"的 lint
怎么注册和测试                 Clippy 团队的 review 标准
怎么运行 ./x.py test           怎么让你的 PR 减少 review 轮数
                              误报率（false positive）的自我评估
                              Applicability 选择的深层思考
```

### 3.2 Clippy Lint 审核标准

```
Clint 团队在审核 lint 时会问四个问题：

1. 这个 lint 真的有用吗？
   - 被 lint 的代码模式真的"有问题"吗？
   - 还是只是"审美偏好"？
   → 主观的 lint 通常会被拒绝（如 "use match instead of if"）

2. 误报率（False Positive Rate）是多少？
   - 对真实代码（如 crates.io 上的 crate）跑过测试吗？
   - 可以用 clippy 的 lintcheck 工具批量测试：
     $ cargo lintcheck
   → 如果误报率 > 10% ——基本会被拒绝

3. 建议是安全的吗？
   - 如果 lint 提供自动修复（MachineApplicable）
   - 这个修复在所有情况下都是语义等价的吗？
   → 不确定就用 Unspecified，不要冒险用 MachineApplicable

4. 和已有 lint 有重叠吗？
   - 检查 clippy_lints/src/ 目录
   - 用 cargo clippy -- -W clippy::all 检查你关心的模式
   → 如果已有 lint 覆盖了大部分场景——在已有的上面改
```

### 3.3 Lintcheck 工具

```bash
# lintcheck 是 Clippy 用来在大规模代码上测试 lint 的工具
# 它下载 crates.io 上的热门 crate，对它们运行 Clippy

# 运行 lintcheck：
cd src/tools/clippy
cargo lintcheck

# 只测试你的新 lint：
cargo lintcheck --filter your_lint_name

# 输出：
# clippy::your_lint_name  1  (0 genuine / 1 false positive)
# → 如果你的 lint 在大规模代码上只有 false positive——需要改进

# 查看触发 lint 的代码：
cargo lintcheck --recursive  # 保留下载的 crate 源码
# → 在 lintcheck-logs/ 目录查看触发位置
```

## 4. 与编译器团队协作

### 4.1 机器人助手

```
rust-lang/rust 仓库有几个机器人帮你管理 PR：

@rustbot:
  → @rustbot claim           # 认领一个 issue（避免冲突）
  → @rustbot ready             # 标记 PR 准备好 review
  → @rustbot author            # 标记 PR 需要作者修改
  → @rustbot label +S-waiting-on-review
  → @rustbot review             # 请求 review

@bors:
  → @bors r+                    # 批准合并（需要 reviewer 权限）
  → @bors try                   # 触发 CI 测试
  → @bors delegate+             # 委托合并权限
  → @bors retry                  # 重试失败的 CI
```

### 4.2 Review 流程

```
┌─────────────────────────────────────────────────────────┐
│  rustc PR 的 Typical Review 流程                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 提交 PR                                              │
│     → @rustbot ready                                     │
│                                                         │
│  2. CI 自动运行                                          │
│     → 编译检查（全平台）                                  │
│     → 测试（全平台 + 各种配置）                           │
│     → perf 检查（性能退化检测）                           │
│                                                         │
│  3. Reviewer 被分配                                      │
│     → 编译器团队有 review 轮转制度                       │
│     → 小 PR 可能在 1-3 天被 review                      │
│     → 大 PR 可能需要 1-2 周                              │
│                                                         │
│  4. Review 反馈循环                                      │
│     → Reviewer 提出问题/建议                             │
│     → 你修改 → push → CI 重跑                           │
│     → Reviewer 再次审查                                  │
│     → 重复直到 reviewer 满意或提出更多问题                 │
│                                                         │
│  5. 批准                                                 │
│     → @bors r+                                           │
│     → Bors 做最终检查（CI 全绿 + 无冲突）                 │
│     → Auto-merge！                                       │
│                                                         │
│  整个流程对首次贡献者可能需要 2-4 周                      │
│  （CI 慢是主要瓶颈——rustc CI 可能需要 2-4 小时）          │
└─────────────────────────────────────────────────────────┘
```

## 5. 实战建议：你的前 5 个 rustc PR

### 5.1 推荐的贡献顺序

```
PR 1: 改进诊断信息（1-2 天）
  → 找一条你曾经困惑过的错误信息
  → 在 rustc 源码中搜索这条信息
  → 添加 `span_help!()` 或改进措辞
  → 这是最快的"入门贡献"——几乎不需要理解编译器逻辑

PR 2: 修 clippy lint 的 false positive（3-7 天）
  → 在 clippy issue tracker 找 "false-positive" 标签
  → 复现误报 → 理解为什么 lint 在这里触发了
  → 在 lint 代码中添加一个排除条件
  → 这让你熟悉 clippy 的 HIR 遍历模式

PR 3: 修复 ICE（1-2 周）
  → 在 issue tracker 找 "I-ICE" + "E-mentor" 标签
  → 按上面的五步法修复
  → 这让你深入理解编译器的一个子系统

PR 4: 写一个新的 clippy lint（2-4 周）
  → 参考 Phase 7.10 的实战经验
  → 在 clippy issue tracker 找 "good-first-issue" + "new lint"
  → 这是你第一个完整的"从 idea 到 merge"的贡献

PR 5: 改进 MIR 优化或 trait 解析（4-8 周）
  → 现在你有足够的编译器经验了
  → 跟着你的兴趣——哪个子系统你最熟悉？
```
