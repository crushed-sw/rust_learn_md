# 9.1 Rust 语言演进：RFC 流程深度解析

> Rust 可能是世界上"民主化"程度最高的主流编程语言——任何一个有 GitHub 账号的人都可以提交 RFC，任何一个有合理论证的人都可以影响语言的设计决策。但 RFC 流程不是"投票"——它是"说服"。你需要用动机（为什么现在不够好）、用例（谁受益）、代价（谁受损）、替代方案（为什么别的方案不行）来说服团队和社区。本节拆解 RFC 流程的每一个阶段，分析三个改变了 Rust 的关键 RFC 案例，并教你如何写出有说服力的 RFC 和 RFC 评论。

---

## 1. RFC 全流程

### 1.1 从 Idea 到 Stable 的完整旅程

```
┌─────────────────────────────────────────────────────────┐
│  RFC 生命周期（通常 6 个月 ~ 3 年）                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. IDEA：有人觉得"Rust 这里应该更好"                     │
│     │  ↓ 通常先在 URLO / Zulip / Reddit 讨论              │
│  2. Pre-RFC 讨论：在社区收集反馈，磨利提案                 │
│     │  ↓ "这个 idea 有坑吗？有更好的做法吗？"              │
│  3. RFC PR：在 rust-lang/rfcs 仓库提交正式的 RFC 文档     │
│     │  ↓ 格式: 0000-template.md → 被分配一个编号          │
│  4. 社区讨论 + 团队 review：                              │
│     │  - 社区成员在 PR 下评论（任何人都可以！）             │
│     │  - 相关团队成员（Lang/Libs/Compiler）深入审查        │
│     │  ↓ 可能需要多次修改（RFC PR 经常有 50+ 评论）        │
│  5. FCP（Final Comment Period）：                        │
│     │  - 团队认为 RFC 已经充分讨论                         │
│     │  - 开启 10 天静默期——最后的机会提反对意见             │
│     │  ↓ 如果 FCP 期间没有新的实质性反对……                │
│  6. MERGE：RFC 被合并！进入 rfcs 仓库的 accepted 目录     │
│     │  ↓ 但合并 ≠ 实现——现在需要有人来写代码               │
│  7. Implementation：                                     │
│     │  - 通常由 RFC 作者或志愿者实现                       │
│     │  - 需要配合 Tracking Issue（在 rust-lang/rust 仓库） │
│     │  - 可能分多个 PR 逐步实现                            │
│     │  ↓ 实现在 nightly 中可用                             │
│  8. Stabilization：                                      │
│     │  - 在 nightly 中测试足够长时间后                      │
│     │  - 提交 Stabilization Report（总结反馈和测试结果）    │
│     │  - 团队 FCP → 在 stable 中可用！                     │
│     └────────────────────────────────────────────────────│
│                                                         │
│  注意：被否决的 RFC 不意味着"永远不行"                     │
│  → 改进提案后可以重新提交（有些 RFC 经历了 3-4 次迭代）     │
└─────────────────────────────────────────────────────────┘
```

### 1.2 追踪工具

```bash
# RFC 仓库
https://github.com/rust-lang/rfcs
# → 所有 RFC（proposed / accepted / rejected）都在这里

# Tracking Issues（在 rust-lang/rust 仓库）
# 一旦 RFC 被接受，在 rust-lang/rust 会创建一个 tracking issue
# 例如: "Tracking issue for RFC 3498: Lifetime Capture Rules 2024"
https://github.com/rust-lang/rust/issues?q=label:C-tracking-issue

# rfcbot（RFC 机器人）
# 团队成员用 @rfcbot 命令管理 RFC 流程
# @rfcbot fcp merge    → 开启 FCP（提议合并）
# @rfcbot fcp close    → 开启 FCP（提议关闭）
# @rfcbot concern xxx  → 提出关切（block FCP 直到解决）
# @rfcbot reviewed     → 标记已审查

# 查看当前活跃的 RFC 讨论
https://github.com/rust-lang/rfcs/pulls
```

## 2. 如何写一个好的 RFC

### 2.1 RFC 模板结构

```markdown
# RFC: [Feature Name]

## Summary（摘要）
用一段话概括：这个 RFC 要做什么，为什么。

## Motivation（动机）
- 当前 Rust 在这个领域有什么不足？
- 这些不足导致了什么实际问题？（具体！给代码示例！）
- 为什么现有的 workaround 不够好？

## Guide-level explanation（用户视角）
- 这个功能稳定后，Rust 用户会看到什么变化？
- 用一个完整的代码示例展示"之前 vs 之后"
- 如果有新的错误信息——展示它们长什么样
- 这部分应该让**不懂编译器内部的人**也能看懂

## Reference-level explanation（实现视角）
- 具体的语法/API 定义
- 新的类型、trait、属性
- 编译器内部需要做哪些改动
- 对现有代码的影响（破坏性变更？向后兼容？）

## Drawbacks（代价）
> 这是最重要的部分之一——诚实面对代价
- 为什么我们**不应该**做这个？
- 会增加语言复杂度吗？
- 会影响编译速度吗？
- 有合理的反对意见吗？

## Rationale and alternatives（设计理由与替代方案）
- 为什么选这个方案而不是别的？
- 考虑过哪些替代方案？为什么它们不够好？
- 这个设计和过去类似的 RFC/feature 有什么关系？
- 未来有什么扩展空间？

## Prior art（先例）
- 其他语言是怎么解决这个问题的？
- 从它们的经验中我们能学到什么？
- 相关的前置 RFC/feature 有哪些？

## Unresolved questions（未解决的问题）
- 还有哪些设计细节需要讨论？
- 有哪些边界情况还没想清楚？

## Future possibilities（未来可能）
- 这个 RFC 为未来的哪些扩展打开了大门？
- 需要注意不要堵死哪些未来路径？
```

### 2.2 好 RFC vs 差 RFC

```
好 RFC 的特征：
  ✓ Motivation 部分有具体的代码示例，展示"现在的痛苦"
  ✓ 诚实地列举 Drawbacks（反而增强了可信度）
  ✓ "我们考虑过方案 B，但因为它有 X 问题所以没选"
  ✓ Guide-level 部分让非编译器开发者也能评价设计
  ✓ Unresolved questions 列出已知的盲点（坦率比完美更有说服力）

差 RFC 的特征：
  ✗ Motivation 太抽象："这样会更好"而没有具体场景
  ✗ 回避 Drawbacks（reviewer 会帮你找出来）
  ✗ 不讨论替代方案（显得你没想到过其他可能性）
  ✗ Guide-level 太技术化（非专家无法参与讨论）
  ✗ 篇幅太短（<500 字通常意味着想得不够深）
```

## 3. 如何有效参与 RFC 讨论

### 3.1 评论的类型

```markdown
┌─────────────────────┬─────────────────────────────────────┐
│ 评论类型             │ 示例                                │
├─────────────────────┼─────────────────────────────────────┤
│ 用例补充             │ "我在 XX 项目中也需要这个，补充一个  │
│                     │  用例：当我们处理 Y 时……"             │
├─────────────────────┼─────────────────────────────────────┤
│ 边界情况提醒         │ "如果 T 是 !Send 会怎样？RFC 没有    │
│                     │  讨论这个"                           │
├─────────────────────┼─────────────────────────────────────┤
│ 替代方案提出         │ "考虑过用 const generics 实现吗？   │
│                     │  似乎比新语法更简单"                 │
├─────────────────────┼─────────────────────────────────────┤
│ 先例引用             │ "Haskell 的 X 扩展有类似的设计，     │
│                     │  他们的经验是……"                     │
├─────────────────────┼─────────────────────────────────────┤
│ 实现反馈（如果你实   │ "我在 nightly 试了原型实现，发现    │
│  现了原型）          │  性能比预期差 30%……"                 │
└─────────────────────┴─────────────────────────────────────┘
```

### 3.2 评论礼仪

```
DO:
  ✓ 用代码示例说话——一个 10 行的例子胜过 100 行的论述
  ✓ 提问而非断言："有没有考虑过 X 情况？" 优于 "你没有考虑 X"
  ✓ 假设作者已经深思熟虑过——他们可能已经排除了你想到的方案
  ✓ 把反对意见包装为"潜在的担忧"而非"这个设计错了"

DON'T:
  ✗ "这个设计很糟糕"——没有帮助，攻击性
  ✗ "+1" / "同感"——没有增加信息量，用 emoji reaction 即可
  ✗ 在 FCP 期间提出之前已经充分讨论过的问题
  ✗ 把个人偏好当作技术必然
```

## 4. 关键 RFC 案例分析

### 4.1 RFC 1236: `?` 运算符（2015）

```
背景：
  Rust 早期用 try!() 宏做错误传播：
    let x = try!(fallible_func());
  
  问题是：嵌套时很难读
    let x = try!(try!(try!(a.b()).c()).d());

提案：
  引入 ? 后缀运算符：
    let x = fallible_func()?;

争议：
  - "? 会让 Rust 看起来像 Ruby/JS？" — 最终风格胜出
  - "? 的语义是什么？" — 最终定义为 try!() 的语法糖
  - "会不会被滥用？" — 实践证明不会

教训：
  → 小的语法糖可以显著改善人体工程学
  → 好名字很重要——? 比 try 更短，视觉负担更小
  → RFC 1236 从提案到稳定用了约 1 年
```

### 4.2 RFC 1598: GAT（Generic Associated Types，2016→2021）

```
背景：
  关联类型不能有泛型参数：
    trait Foo {
        type Bar<T>;  // ← 在 2016 年这是语法错误
    }

提案：
  允许关联类型有泛型参数（包括生命周期参数）

为什么花了 5 年？
  - 编译器实现极其复杂（涉及 trait solver 的重大改动）
  - 和 borrow checker 的交互需要仔细设计
  - 多次被其他更高优先级的特性挤占资源
  - 最终由 Jack Huey 完成了实现

教训：
  → 大的语言特性需要数年时间——耐心是必需的
  → RFC 被接受不等于会立即实现
  → 一个人（champion）的持续推动至关重要
```

### 4.3 RFC 3498: Lifetime Capture Rules 2024（2023）

```
背景：
  Rust 2021 Edition 中，闭包捕获的生命周期规则有问题：
    let x = String::from("hello");
    let f = || drop(x);  // f 捕获了 x
    // 但 f 的生命周期标注让某些模式无法表达

提案：
  在 Rust 2024 Edition 中修改闭包的生命周期捕获规则

争议：
  - Edition 变更必须通过 cargo fix 自动迁移
  - 对已有代码的影响需要仔细评估（通过 crater 跑所有公共 crate）

教训：
  → Edition 机制让"修复过去的错误设计"成为可能
  → 但修复必须有自动迁移路径（cargo fix）
  → crater（大规模代码分析）是 Edition 变更的安全网
```

## 5. 从 RFC 读者到 RFC 作者

### 5.1 你的进阶路径

```
阶段 1: 读者
  → 每 2-3 周浏览一次 rust-lang/rfcs/pulls
  → 读 2-3 个正在活跃讨论的 RFC
  → 试着形成自己的判断："如果让我投票，我会支持吗？"

阶段 2: 评论者
  → 在你熟悉的领域（Phase 7 编译器？Phase 8 的 crate？）
  → 给正在讨论的 RFC 留下有信息量的评论
  → 目标：你的评论被团队成员的 emoji reaction 认可

阶段 3: Pre-RFC 参与者
  → 在 URLO / Zulip 上参与新 idea 的早期讨论
  → 帮助提案者打磨 idea——这比写正式 RFC 评论更有影响力

阶段 4: RFC 作者
  → 当你遇到一个 Rust 的痛点，并且有了清晰的解决方案
  → 先写 Pre-RFC → 收集反馈 → 提正式 RFC PR
  → 准备好花 3-6 个月持续推动它
```

### 5.2 你应该写什么样的 RFC？

```
不适合首次 RFC 的主题：
  ✗ 添加全新的语言特性（如 "Rust 应该支持 inheritance"）
  ✗ 改变借用检查器的核心规则
  ✗ 移除已有的 stable 特性

适合首次 RFC 的主题：
  ✓ 标准库 API 补充（添加一个方法到已有类型）
  ✓ 小的语法糖（减小常见模式的样板代码）
  ✓ lint 提议（将常见错误模式提升为编译器警告）
  ✓ 文档/诊断信息的标准化
```
