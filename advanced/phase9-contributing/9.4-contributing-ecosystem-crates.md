# 9.4 贡献生态 Crate：从 User 到 Maintainer

> 标准库贡献需要 1-2 年的耐心。rustc 贡献需要理解编译器内部。但生态 crate 贡献——你每天 `cargo add` 的那个库——是你最快能"落地"的贡献形式。而且最有意思的是：你不需要成为"那个 crate 的专家"。你只需要是一个"被 bug 折磨过的用户"——因为你已经理解了问题，你只需要修它。本节提供从选目标、读指南、写 PR、到处理 review 反馈的完整方法论——以及如何从"一次贡献"开始，走上 maintainer 之路。

---

## 1. 找对第一个贡献目标

### 1.1 目标选择的四个维度

```
找 crate 的 Checklist：

□ 你在用吗？
  → 最理想：被它的 bug 折磨过的 crate
  → 最不理想：从来没听说过的 crate
  → 你已经是 user → 你已经理解了问题域

□ 有 "good first issue" 标签吗？
  → GitHub issues → Labels → "good first issue" / "E-easy" / "help wanted"
  → 这些是维护者特意标注的"新人友好"任务

□ 维护者活跃吗？
  → 看最近 1 个月是否有 PR 被 merge
  → 看 issue 是否有人回复
  → 如果最近一次 commit 是 2 年前——换一个

□ 贡献指南清晰吗？
  → 有 CONTRIBUTING.md 吗？
  → 有 DEVELOPMENT.md 或 ARCHITECTURE.md 吗？
  → CI 配置是否包含 clippy/fmt（说明有自动化检查）
```

### 1.2 推荐的首批贡献目标

```
难度 ⭐：拼写错误 / 文档修复
  → 任何 crate
  → 在 README / doc comment 中找 typos
  → 改 1-2 行，但让你熟悉整个 PR 流程

难度 ⭐⭐：测试补充
  → 找 codecov 覆盖率低的模块
  → 添加缺失的单元测试
  → 这让你深入理解代码逻辑

难度 ⭐⭐⭐：小 bug 修复
  → "good first issue" 标签的 bug
  → 通常涉及 10-50 行代码改动
  → 这是最有成就感的首次贡献

难度 ⭐⭐⭐⭐：小 feature 添加
  → 维护者标记为 "help wanted" 的 feature
  → 需要理解更多上下文
  → 通常涉及 50-200 行代码
```

## 2. 读贡献指南

### 2.1 CONTRIBUTING.md 的典型结构

```markdown
# 典型的 CONTRIBUTING.md 内容

## Development Setup
# 如何设置本地开发环境
git clone ...
cargo build

## Code Style
# 代码风格要求
- cargo fmt (rustfmt 配置)
- cargo clippy (clippy 配置)
- 我们使用 4 空格缩进

## Testing
# 如何运行测试
cargo test
cargo test --all-features
# 特殊测试（如集成测试、benchmark）

## Commit Messages
# commit message 格式
# 惯例：遵循 Conventional Commits
feat: add X feature
fix: fix Y bug
docs: improve Z documentation

## PR Process
# PR 的流程
- 先开 issue 讨论（对于大改动）
- 一个 PR 只做一件事
- 必须通过 CI
- 需要至少 1 个 reviewer 批准
```

### 2.2 如果没有 CONTRIBUTING.md？

```
# 隐性规则（大部分 Rust crate 都遵循）

1. 用 cargo fmt 格式化代码
2. 用 cargo clippy 检查代码
3. 跑 cargo test 确保测试通过
4. PR 标题用英文，描述清晰
5. 一个 PR 一个目的

# 在开 PR 之前，可以先开一个 issue：
"Hi! I'd like to contribute. I see that [problem].
 I'm thinking of [approach]. Would that be welcome?"
→ 维护者可以提前告诉你"这个方向对吗"
→ 避免花时间实现了不被接受的方向
```

## 3. PR 的黄金法则

### 3.1 四要素

```
┌─────────────────────────────────────────────────────────┐
│  PR 黄金法则                                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 小（Small）                                          │
│     → 目标: <200 行改动的 PR                             │
│     → 大功能？拆成 3-5 个 PR，每个是独立的逻辑步骤       │
│     → 越小的 PR 越快被 review，越快被 merge              │
│                                                         │
│  2. 聚焦（Focused）                                      │
│     → 一个 PR = 一个意图                                 │
│     → "修 bug X" 或 "加 feature Y"——不是两者都要         │
│     → 不要在 PR 中"顺便重构"无关代码                     │
│                                                         │
│  3. 有测试（Tested）                                     │
│     → 修 bug 的 PR？测试证明 bug 被修了                   │
│     → 加 feature 的 PR？测试证明 feature 正常工作         │
│     → 测试比代码更能说服 reviewer："这是对的"             │
│                                                         │
│  4. 有上下文（Contextualized）                            │
│     → PR description 解释 WHY 而不只是 WHAT              │
│     → WHAT（做了什么改动）→ 看 diff 就知道了              │
│     → WHY（为什么这样做）→ 需要在 description 中说明      │
│     → 如果有相关 issue → 用 "Fixes #123" 链接             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 PR Description 模板

```markdown
## Summary
一句话说明：这个 PR 做了什么，为什么。

## Problem
[如果有相关 issue，链接它]
描述问题：用户遇到了什么？什么场景下出现？

## Solution
描述解决方案：
- 为什么选这个方案而不是别的？
- 有什么 tradeoff？
- 如果需要，提供设计决策的理由

## Testing
- [ ] 添加了新的测试
- [ ] 所有已有测试继续通过
- [ ] 手动测试：[描述如何手动验证]

## Screenshots / Logs（如果适用）
[对于 UI/CLI crate，展示 before/after]
```

## 4. 处理 Code Review 反馈

### 4.1 心态

```
Review 不是对你的攻击——是对代码的检查。

Reviewer 的目标：确保合并后的代码不会引入问题。
你的目标：让 reviewer 理解你的意图并批准合并。

常见情绪陷阱：
  "我花了两天写的代码，被 reviewer 说"重写吧""
  → review 不会否定你的努力——reviewer 只是看到了你没看到的路径

  "这个 reviewer 太挑刺了，这些都是小问题"
  → reviewer 通常是在帮你防止未来的 bug
  → 他们曾经被同样的小问题坑过

  "我不回复了，让这个 PR 烂掉吧"
  → 沉默比争论更不利于你的声誉
  → 如果你不同意某个 review 意见——解释为什么。这是正常的对话。
```

### 4.2 如何处理不同类型的 Review 反馈

```markdown
┌───────────────────┬─────────────────────────────────────┐
│ 反馈类型           │ 如何应对                             │
├───────────────────┼─────────────────────────────────────┤
│ "应该用 X 而不是 Y"│ 如果同意：感谢指出，修改后 push       │
│  （实现选择）      │ 如果不同意：解释为什么 Y 更好，      │
│                   │ 附上代码示例或性能数据               │
├───────────────────┼─────────────────────────────────────┤
│ "这个 edge case    │ 感谢！补充处理这个 edge case         │
│  没处理"           │ 加测试用例覆盖这个 edge case         │
├───────────────────┼─────────────────────────────────────┤
│ "代码风格不符合"   │ 直接改——风格问题不值得争论          │
│                   │ 下次先跑 cargo fmt + cargo clippy    │
├───────────────────┼─────────────────────────────────────┤
│ "这个 PR 太大了，  │ 接受。拆成多个小 PR。                │
│  拆开吧"           │ 这很常见——大部分人第一次 PR 都太    │
│                   │ 大。这不是批评，是经验。              │
├───────────────────┼─────────────────────────────────────┤
│ "为什么要这样做？" │ 解释。这不是质疑你的能力——         │
│                   │ reviewer 只是在建立上下文来理解代码   │
└───────────────────┴─────────────────────────────────────┘
```

### 4.3 Review 轮回的节奏

```
一轮 review 的典型时间线：

PR Day 1: 提交 PR → CI 跑（30 分钟-2 小时）
PR Day 2-5: Reviewer 开始看（维护者有自己的工作）
             → 留下 3-10 条评论
PR Day 6: 你回应评论并 push 修改
PR Day 7: Reviewer 再看 → 留下 0-3 条评论
PR Day 8: 你修复最后的问题
PR Day 9: Approved! → 合并

总共: 1.5-2 周（对于小 PR）
大 PR / 复杂的 PR 可能 4-8 周

保持耐心——维护者通常是志愿工作。
如果超过 2 周没有回应：
  → 发一条礼貌的 @ 提醒
  → "Hi @maintainer, just checking if you had a chance to look at this?"
```

## 5. 从一次贡献到 Maintainer

### 5.1 进阶路径

```
阶段 1: One-time Contributor（第 1-3 个 PR）
  → 修 bug、加测试、改文档
  → 维护者记住你的名字了吗？——这是关键

阶段 2: Repeat Contributor（第 4-8 个 PR）
  → 你对代码库越来越熟悉
  → 开始 review 别人的 PR（在 comment 中留下建设性意见）
  → 开始在 issue 讨论中提供帮助

阶段 3: Triage（被邀请或自然形成）
  → 你有"管理 issue"的经验（标记、关闭无效 issue）
  → 维护者可能会给你 triage 权限
  → 你开始承担"维护者的左手"角色

阶段 4: Co-maintainer（被邀请加入）
  → 你对代码库有深刻理解
  → 你可以 review 和批准 PR
  → 你参与设计决策
  → 你有发布新版本的权限

阶段 5: Maintainer（核心维护者离开或退休时）
  → 你成为项目的"face"
  → 你决定项目的方向和优先级
  → 你培养新的贡献者——完成循环
```

### 5.2 如何被注意到

```
不只是写代码——还要：

1. 写好的 commit message
   → 清晰地解释 WHY
   → 维护者看到高质量的 commit → "这个人靠谱"

2. Review 别人的 PR
   → 在还没有 merge 权限时就可以 review
   → 留下有建设性的 comment
   → "这里可能需要处理 X 的 edge case" → 展示你的细致

3. 在 Issue 讨论中提供帮助
   → 回答用户的问题（比维护者更快）
   → 复现报告者遇到的 bug（节省维护者时间）

4. 一致地出现
   → 每月 2-3 个小贡献 → 几个月后你是最活跃的贡献者之一
   → 一次性大贡献 → 只是统计中的一个数据点
```

## 6. 回顾

- [ ] 掌握选择第一个贡献目标的四个维度（在用、好 issue、活跃、有指南）
- [ ] 知道如何读 CONTRIBUTING.md 和隐性规则
- [ ] 理解 PR 黄金法则：小（<200 行）、聚焦、有测试、有上下文
- [ ] 会用 PR Description 模板（Summary → Problem → Solution → Testing）
- [ ] 掌握处理五种 Code Review 反馈的策略
- [ ] 理解从 one-time contributor → maintainer 的五个阶段
- [ ] 知道"被注意到"的三个非代码行动：commit message、review 别人、保持一致性

---

> 生态 crate 贡献是最快的"正向反馈"——你昨天用的库，今天有你写的一行代码。但贡献不只在代码库里——[[9.5-community-engagement-brand|9.5 社区参与与个人品牌]] 将探索代码之外的贡献形式：写博客、做演讲、在论坛回答问题——这些"软贡献"同样能让你在 Rust 社区中建立声誉。
