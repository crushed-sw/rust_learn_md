# 9.3 贡献标准库：API 设计的实战

> 为 rustc 贡献是改"编译器怎么说"，为标准库贡献是改"程序员怎么写"。标准库的贡献门槛比编译器低——不需要理解 borrow checker 的实现细节，不需要修改 MIR——但它有另一种难度：**API 设计**。为标准库添加一个方法不是"能不能实现"的问题——标准库团队内部的工程师完全有能力实现任何合理的功能——而是"应不应该有"的问题。这个 API 风格一致吗？对得起"forever stable"的承诺吗？能被 100% 的 Rust 用户直观理解吗？本节教你如何在标准库的严格约束下做出优雅的 API 设计。

---

## 1. 标准库的 API 设计原则

### 1.1 Rust std 的设计哲学

```
Rust 标准库不同于 Python/Go 的"batteries included"：
  → std 很小——HTTP、JSON、数据库、加密都在生态里
  → std 加东西极其谨慎——加了就永远不能删

std API 设计的四条铁律：

1. 一致性（Consistency）
   → 新 API 必须和已有 API 的命名、签名、行为风格一致
   → 例: 如果已有 .is_some()，新方法应该叫 .is_sorted() 而非 .sorted()

2. 最小性（Minimality）
   → 只加"明显的缺口"，不加"可能有用的"
   → 如果 90% 的用户不需要，就留给生态 crate
   → 如果可以用已有 API 组合实现，就不值得加

3. 稳定性（Stability）
   → 一旦 stable，永远不能改签名、不能改语义
   → 可以有更快的实现，但不能有不同的行为
   → → 这意味着 API 设计必须在第一版就"对"

4. 可理解性（Understandability）
   → 方法名要"不用查文档就知道做什么"
   → 泛型约束要最小化（不要因为一个罕见用例加大约束）
   → 文档示例应该展示最常见用法
```

### 1.2 "这个要加吗？"——一个自检清单

```
在提议标准库新 API 之前，问自己：

□ 这个问题足够普遍吗？
  → "我在 XX 项目中需要它" 不够
  → "我在 3 个项目中都手动实现了它" ——开始值得考虑

□ 现有 API 能做吗？即使啰嗦？
  → 如果 let x = a.iter().all(|x| x & 1 == 0); 能实现
  → 那 iter::all_even() 就不值得加

□ API 签名是"唯一显然"的吗？
  → 如果 fn foo(x: u32) 和 fn foo(x: i32) 都有合理理由
  → 说明设计还不成熟——先讨论清楚

□ 和其他已有方法风格一致吗？
  → 查 Option、Result、Iterator 上的已有方法
  → 你的方法名是否遵循相同的命名模式？

□ 文档写出来能"一看就懂"吗？
  → 写一个 5 行的示例——它是否展示了最常见用法？
  → 如果没有明确的"最常见用法"——可能不需要这个 API
```

## 2. 常见贡献类型

### 2.1 为已有类型添加新方法

```rust
// 这是最常见的标准库贡献类型
// 例：为 Iterator 添加 is_sorted 方法

// ━━━ 提案 ━━━
// ACP（API Change Proposal）: Iterator::is_sorted

// 动机（Motivation）:
// 检查"是否已排序"是常见操作，目前需要：
let is_sorted = data.windows(2).all(|w| w[0] <= w[1]);
// 或
let is_sorted = data.iter().is_sorted();  // ← 不存在！需要从 itertools 引入

// 提案的 API:
impl<T: Iterator> Iterator for T
where
    T::Item: PartialOrd,
{
    fn is_sorted(self) -> bool
    where
        T::Item: PartialOrd,
    {
        // 实现：遍历相邻对
    }
}

// ━━━ API 设计的思考过程 ━━━
// Q: 用 PartialOrd 还是 Ord？
// A: PartialOrd——浮点数 NaN 不是全序，但对大多数类型足够
// Q: 要不要返回 Result<bool, NotSortedError>?
// A: 不需要——false 就够了，过度设计
// Q: 名字用 is_sorted 还是 sorted?
// A: is_sorted——以 is_ 前缀遵循 Rust 命名惯例（is_some, is_none, is_empty）
```

### 2.2 为已有类型添加 trait 实现

```rust
// 例：为标准库类型添加新的 trait 实现

// 提案：为 Range 添加 Copy trait
// （如果 Range 的边界类型实现了 Copy）

impl<T: Copy> Copy for std::ops::Range<T> {}

// 这看起来简单——但审查过程会问：
// Q: 为什么现在不加？为什么以前没加？
// A: 以前没加是因为——早期设计时保守处理
// Q: 加了会 break 吗？
// A: 不会——添加 trait impl 是向后兼容的
// Q: 有没有什么场景下你不希望 Range 是 Copy？
// A: 没有——Range 只是两个数的组合，Copy 是合理的

// 这种贡献通常很小（10 行），但能显著改善人体工程学！
```

### 2.3 改进文档

```rust
// 文档贡献极容易被忽视，但价值极高

// ━━━ 好文档 vs 已有的“刚好够用”的文档 ━━━

// 现状（std 中的典型文档）：
/// Returns true if the option is a Some value.
///
/// # Examples
///
/// ```
/// let x: Option<u32> = Some(2);
/// assert_eq!(x.is_some(), true);
/// ```
pub fn is_some(&self) -> bool { ... }

// 改进后：
/// Returns `true` if the option is a [`Some`] value.
///
/// # Examples
///
/// Basic usage:
/// ```
/// let x: Option<u32> = Some(2);
/// assert_eq!(x.is_some(), true);
///
/// let x: Option<u32> = None;
/// assert_eq!(x.is_some(), false);
/// ```
///
/// # When to use
/// Prefer `if let` or `match` when you need to use the inner value:
/// ```
/// if let Some(val) = x {
///     println!("got {}", val);
/// }
/// ```
///
/// `is_some()` is most useful in iterator chains:
/// ```
/// let count = items.iter().filter(|x| x.is_some()).count();
/// ```
///
/// # See also
/// - [`is_some_and`] for checking a condition on the contained value
/// - [`is_none`] for the opposite check
pub fn is_some(&self) -> bool { ... }

// 文档贡献的价值：
// → 不需要深入理解实现
// → 直接提升每个 Rust 开发者的体验
// → 是学习标准库的好方式（读一个方法 → 理解它 → 改进文档）
```

## 3. 标准库贡献的特殊约束

### 3.1 不能 break 任何 stable 特性

```
标准库的铁律:

  "任何在 stable Rust 上编译通过的代码，
   在新版本中必须继续编译通过，且行为一致。"

这意味着:
  ✗ 不能删除任何 pub 方法
  ✗ 不能修改 pub 方法的签名
  ✗ 不能改变既有方法的语义（即使旧语义"不合理"）
  ✗ 不能为已有类型移除 trait 实现

这意味着你可以:
  ✓ 添加新的 pub 方法
  ✓ 添加新的 trait 实现
  ✓ 优化内部实现（只要外部行为不变）
  ✓ 改进文档
  ✓ 使用 #[deprecated] 标记（但永不删除）
```

### 3.2 性能不能退化

```
标准库是 Rust 生态的性能基线。
如果你改动了标准库代码，perf.rust-lang.org 会自动检测。

关键性能指标：
  → 编译时间：新 API 会不会拖慢编译？
  → 运行时性能：新实现比旧实现更快吗？
  → 二进制大小：新 API 会不会增加每个程序的体积？

提交 std PR 时的性能 checklist：
  □ 用 cargo bench 在改动前后对比
  □ 检查是否引入了不必要的分配
  □ 如果可能，提供编译期计算（const fn）实现
  □ 检查是否意外增加了泛型单态化的副本数
```

### 3.3 平台兼容性

```
标准库必须在所有 Tier 1 平台上行为一致：
  → x86_64-unknown-linux-gnu
  → x86_64-apple-darwin
  → x86_64-pc-windows-msvc
  → ...等

平台问题的典型场景：
  → 文件路径处理（Windows 用 \，Unix 用 /）
  → 数字格式化（不同 locale 的小数点和千位分隔符）
  → 线程和进程行为（Windows 没有 fork()）
  → 网络 I/O 行为差异

测试策略：
  → rustc CI 在所有 Tier 1 平台上运行完整测试套件
  → 你的 PR 必须通过所有平台的 CI
  → 如果你在 Linux 上开发——至少在逻辑中考虑其他平台
```

## 4. libs-api 团队的 Review 流程

### 4.1 ACP（API Change Proposal）流程

```
标准库 API 变更的完整流程：

1. ACP Issue（在 rust-lang/libs-team 仓库）
   → 用标准模板描述你提议的 API
   → 社区和团队成员讨论
   
2. libs-api 会议（通常是 Zulip 非同步讨论）
   → 团队成员逐个审查
   → 核心问题："这个 API 符合 std 的设计哲学吗？"
   
3. 决定：
   ✓ 接受 → 在 rust-lang/rust 开 tracking issue
   ✗ 拒绝 → 通常会说明原因（"可以用已有 API 组合"最常见）
   ? 修改 → 建议修改 API 签名后重新提案
   
4. 实现 PR
   → 实际代码 + 测试 + 文档
   
5. FCP to stabilize
   → 标记 #[stable] 后合并

整个流程通常需要 2-6 个月
（比编译器贡献慢——因为 API 设计比 bug 修复更需要共识）
```

### 4.2 最常见的拒绝原因

```
1. "可以用已有 API 组合实现"
    → 你的 API 没有提供新的能力，只是提供了新的语法
    → 如果 vec.iter().all(|x| x.is_some()) 已经够用了
    → iter::all_some() 就不会被接受

2. "不够通用"
    → 你用了一个太具体的名字或签名
    → 例: fn to_uppercase_first_letter() —— 为什么只有第一个字母？

3. "太通用了" 
    → 名字太泛，行为不明显
    → 例: fn process() ——干什么的？

4. "没有充分的动机"
    → "有时候可能会用到" 不是充分理由
    → 需要展示"在真实项目中，这是一个痛点"

5. "和已有 API 命名风格不一致"
    → 例: 用 contains_element 而非 contains（和 Iterator::contains 风格冲突）
```

## 5. 典型案例：`Option::insert` 的完整旅程

```
Option::insert 为 Option<T> 添加了一个方法：
  fn insert(&mut self, value: T) -> &mut T
  
  如果是 Some → 替换值，返回新值的可变引用
  如果是 None → 插入值，返回新值的可变引用

这个提案从 idea 到 stable 用了约 1.5 年：

2017.04: 有人提出 "为什么 Option 没有 insert 方法？"
2017.05: ACP 在 rust-lang/rfcs 仓库讨论
2017.06: 争论焦点：
  - insert 的返回值应该是什么？&mut T？还是旧的 Option<T>？
  - 最终决定：&mut T（最常见的用法是 insertion 后立即使用）
2017.08: RFC 被接受（FCP 通过）
2017.09: 实现 PR 提交
2017.10: 在 nightly 中可用
2018.04: 经历 6 个月的 nightly 测试
2018.08: 稳定化 FCP 通过，在 Rust 1.28 中 stable！

教训：
  → 即使是"显而易见"的 API（Option 当然应该有 insert！）
  → 也需要经过完整的讨论流程
  → 返回值的设计需要仔细权衡（&mut T vs Option<T> 的讨论持续了 2 个月）
  → 从 idea 到 stable 通常需要 1-2 年
```
