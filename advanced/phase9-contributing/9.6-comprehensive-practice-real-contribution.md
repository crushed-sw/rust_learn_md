# 9.6 Phase 9 综合实战：完成一次真正的开源贡献

> Phase 9 的 5 节课告诉你：为什么贡献（9.0）、语言怎么决策（9.1）、怎么改编译器（9.2）、怎么设计 API（9.3）、怎么成为 maintainer（9.4）、怎么建立声誉（9.5）。现在，它们需要被组合成一次完整的行动。这个综合实战不是虚构的练习——你需要在 GitHub 上找一个真实的 issue，走完完整的贡献周期：fork → branch → implement → test → PR → review → revise → merge。这是你从"Rust 学习者"变成"Rust 贡献者"的毕业礼。

---

## 1. 你的任务

### 1.1 核心要求

```
完成一次真实的开源贡献（非虚构的 toy project）：
  ✓ 必须是一个真实存在的 GitHub issue
  ✓ 必须通过 PR 提交（不是直接 push 到 main）
  ✓ 必须经历至少一轮 review（reviewer 至少有一条反馈）
  ✓ 必须被 merge（如果最终被拒绝——记录原因并重新选题）

贡献可以是任何形式：
  → 修 bug（最常见）
  → 加测试（最容易开始）
  → 改文档（最快被 merge）
  → 加小 feature（最有成就感）
  → review 别人的 PR（如果你还没准备好写代码）
```

### 1.2 推荐选题

```
按难度排序的选题建议：

Level 1: 文档修复（1-2 小时）
  → 在 README / doc comment 中找拼写错误
  → 改进已有的文档示例（让它更完整或更清楚）
  → 例如: "这个函数的文档缺少一个 error case 的示例"

Level 2: 测试补充（2-4 小时）
  → 找一个 codecov 覆盖率 < 80% 的模块
  → 添加缺失的单元测试
  → 例如: "这个方法对空输入的测试不存在"

Level 3: bug 修复（1-3 天）
  → GitHub issues → "good first issue" 标签
  → 在 issue 下面说 "I'd like to work on this!"
  → 例如: "clap 在 Windows 上处理路径时 panic"

Level 4: Clippy lint（3-7 天）
  → 在 clippy issue tracker 找 "good first issue"
  → 结合 Phase 7.10 的 Clippy 经验
  → 例如: "new lint: detect unnecessary .clone() on Copy types"

Level 5: 标准库方法（1-4 周）
  → 先写 ACP，通过后再实现
  → 需要一个足够明显的"为什么这个 API 应该存在于 std"的理由
  → 例如: "Iterator::is_sorted_by"
```

│                                                         │

## 3. 案例学习：成功的首次贡献

### 3.1 案例 1: 为 `clap` 改进文档示例

```
贡献者背景: Rust 学习 6 个月，第一次开源贡献
选择的 crate: clap（命令行参数解析）
Issue: "derive API 的文档缺少一个完整的 app/subcommand 示例"

过程:
  Day 1: 读 clap 源码理解 derive API
  Day 2: 写了 40 行的完整示例代码
  Day 3: 提交 PR（只改了文档）
  Day 5: Reviewer 建议"添加 error handling 到示例中"
  Day 6: 修改 + push
  Day 7: Merged！

贡献者收获:
  → "原来文档贡献也这么有价值"
  → "通过写示例，我对 clap 的理解深了很多"
  → "维护者非常友好——让我对后续贡献有了信心"

关键点:
  → 文档贡献 ≈ 0 行代码改动，但改进了每个用户的体验
  → 这可能是 PR 被 merge 最快的方式
  → 写示例的过程 = 学习 API 的最佳方式
```

### 3.2 案例 2: 修 `serde_json` 的 edge case

```
贡献者背景: 工作中重度使用 serde_json
选择的 crate: serde_json
Issue: "serde_json::from_str 对某些 Unicode 字符处理不正确"

过程:
  Day 1-2: 复现问题（构造了 5 行的最小复现用例）
  Day 3: 定位到 serde_json 的 read.rs 中的字符解析逻辑
  Day 4: 理解为什么当前逻辑不对（读了 Unicode 规范）
  Day 5: 实现了修复（+30 行）和测试（+15 行）
  Day 6: 提交 PR
  Day 8: Reviewer 指出"你的修复对另一种 Unicode 不工作"
  Day 9-10: 修改 + 补充更多测试
  Day 12: Merged！

贡献者收获:
  → "对 Unicode 的理解从 surface level 到真的懂了"
  → "reviewer 的反馈让我意识到我的修复不够全面"
  → "这个 bug 修好后，下一个 release 就包含了我的代码"

关键点:
  → 你不需要是 crate 的专家——你是 bug 的"受害者"
  → 最小复现用例 = 你最好的贡献工具
  → reviewer 的反馈让你学到比修 bug 更多的东西
```

### 3.3 案例 3: 为 `tracing` 添加新的 Subscriber

```
贡献者背景: 在自己的项目中使用 tracing，需要特定功能
选择的 crate: tracing
Issue: "添加一个将 span 转换为 OpenTelemetry 格式的 Layer"

过程:
  第 1 周: 在 issue 中讨论设计（"应该做成 Layer 还是 Subscriber？"）
  第 2 周: 实现了基础版本（200 行）
  第 3 周: Review → "应该用 trait abstraction 让用户自定义转换"
  第 4 周: 重构——用 trait 模式替代硬编码转换
  第 5 周: 第二轮 review → 通过
  第 6 周: Merged！

贡献者收获:
  → "学会了如何在贡献前先讨论设计——节省了大量时间"
  → "trait abstraction 的设计让我的代码更有价值"
  → "这个 feature 后来成为了 tracing-opentelemetry crate 的基础"

关键点:
  → 大 feature 在写代码前先讨论设计（9.1 RFC 思维）
  → review 不一定是"你错了"——有时是"你的抽象层级不够"
  → 一个被 merge 的 PR 可以催生一个全新的 crate
```

## 4. 贡献之后：下一步

### 4.1 Celebrate + Reflect

```
完成了！庆祝一下。

这不是小事——大部分开发者永远不会提交第一个 PR。
你跨过了那条最重要的线："消费者 → 贡献者"。

然后反思（用实战日志模板）：
  - 贡献过程中最享受的是什么？
  - 哪部分最痛苦？是技术问题还是沟通问题？
  - 这种类型的贡献适合你吗？

→ 答案决定了你的下一步方向。
```

### 4.2 三条路

```
路 1: 深耕同一个 crate
  → 你已经熟悉了它的代码库
  → 再贡献 2-3 个 PR → 你可能被邀请成为 maintainer
  → 适合: 喜欢深度、想在某个领域成为专家

路 2: 扩展到编译器 / 标准库
  → 在生态 crate 之后，挑战更高难度的目标
  → rustc bug 修复或 std API 提案
  → 适合: 对 Rust 本身感兴趣，想影响语言演进

路 3: 开始写自己的 RFC
  → 你已经是一个"有经验"的贡献者了
  → 从使用者角度发现的语言痛点形成了清晰的想法
  → 在 URLO 上写 pre-RFC → 收集反馈 → 正式提案
  → 适合: 有设计冲动，想塑造 Rust 的未来

Note: 这三条路不是互斥的——它们可以并行。
```
