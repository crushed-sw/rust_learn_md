# 9.5 社区参与与个人品牌

> 在 GitHub 上提交代码是最"可见"的贡献——但它们不是唯一的贡献。Rust 社区中，回答问题的人、写博客的人、做演讲的人、review 代码的人——他们的工作同样受到尊重，同样建立声誉，同样推动 Rust 向前。而且——这些"软贡献"往往是代码贡献的催化剂：别人看了你的博客所以信任你的 PR，听过你的演讲所以来找你合作，被你的回答帮助过所以愿意 review 你的代码。本节教你如何通过非代码方式参与 Rust 社区，建立持久的技术声誉。

---

## 1. 写技术博客

### 1.1 选题策略

```
Rust 博客的"热度金字塔"：

L3: 观点 / 趋势 / 预测（如 "Rust in 2027"）
    → 最有争议，最有传播力，也最容易过时

L2: 教程 / 入门指南（如 "Rust async from scratch"）
    → 受众最广，但竞争最多（已经有很多教程）

L1: 踩坑记录 / 深度分析（如 "How I debugged a LLVM miscompilation in rustc"）
    → 受众窄但粘性极高——读这类文章的人最可能是未来的合作者
    → 最持久——5 年后仍然有人搜索到你的文章

建议：从 L1 开始。
  - 你不需要是专家——你只需要"比昨天的自己知道得多一点"
  - "我是怎么修好这个 bug 的" > "Rust 的终极指南"
  - 一篇好的踩坑记录 = 问题背景 + 排查过程 + 解决方案 + 学到的教训
```

### 1.2 写作原则

```markdown
# 好 Rust 博客的特征

1. 代码要能跑
   → 每个代码块要么是 "cargo run 就能跑" 的完整示例
   → 要么明确标注 "// 伪代码，不能直接编译"
   → 你的读者会复制粘贴你的代码——别让他们踩坑

2. 深度优于广度
   → "Rust 所有权的 10 个要点" ← 太泛，没人记住
   → "为什么这个 Pin<Box<dyn Future>> 不能 Unpin" ← 窄但深
   → 狭窄的深度文章让你成为"这个领域的那个人"

3. 诚实比权威更重要
   → "我不确定这是最佳方案，但它是有效的" > "这是正确的做法"
   → 承认不确定性反而增强可信度
   → 读者会帮你验证和纠正——这是博客的"peer review"

4. 标题要具体
   → "Rust async 笔记" ← 没人会点
   → "Debugging a tokio task that never wakes up: a 6-hour investigation"
   → 具体标题 = 读者预知文章是否相关

5. 以一个问题开始
   → "上周我的服务在凌晨 3 点挂了……"
   → 故事 > 手册。人天生对叙事有好奇心。
```

### 1.3 发布渠道

```
Rust 博客的发布渠道（按影响力排序）：

1. 个人博客（最重要——这是你的数字资产）
   → 用 GitHub Pages + Jekyll/Zola/mdBook
   → 域名：yourname.dev / yourname.rs
   → 长期来看，这是唯一完全属于你的渠道

2. This Week in Rust（TWiR）
   → https://this-week-in-rust.org
   → 在 GitHub 上提交你的博客链接给 TWiR 编辑
   → TWiR 每期都收录社区优秀文章（免费曝光）

3. Rust 官方博客
   → https://blog.rust-lang.org
   → 可以投稿（通过 rust-lang/blog.rust-lang.org 仓库）
   → 适合"年度回顾"、"feature 介绍"等正式内容

4. Reddit / Hacker News
   → r/rust（非常活跃，60K+ 成员）
   → 如果你写了一篇深度分析——发到 r/rust
   → 注意：标题党和无内容文章会被 downvote

5. 社交媒体（Twitter/X, Mastodon, LinkedIn）
   → 碎片化传播
   → 适合推广你的长篇文章
```

## 2. 回答技术问题

### 2.1 问答平台

```
┌──────────────┬─────────────────────────────────────────┐
│ 平台          │ 特点                                    │
├──────────────┼─────────────────────────────────────────┤
│ URLO          │ Rust 官方问答论坛                       │
│ (users.       │ → 问题质量高，有核心团队成员出没        │
│  rust-lang    │ → 适合"需要深度分析"的问题              │
│  .org)        │ → 回答得好会被标记为 solution           │
├──────────────┼─────────────────────────────────────────┤
│ Rust Discord  │ 实时聊天                                 │
│ / Zulip      │ → 快速回答"怎么做 X"                    │
│              │ → Zulip 是团队内部沟通工具（更技术）      │
│              │ → Discord 是社区交流（更 casual）         │
├──────────────┼─────────────────────────────────────────┤
│ StackOverflow │ 通用编程问答                             │
│              │ → 问题覆盖面广                           │
│              │ → 竞争激烈（已经有很多 Rust 答案）        │
├──────────────┼─────────────────────────────────────────┤
│ r/rust        │ Reddit 子版                             │
│              │ → 新闻 + 问题 + 讨论混合                 │
│              │ → 每周有 "Easy Questions" 主题帖          │
└──────────────┴─────────────────────────────────────────┘
```

### 2.2 好回答的特征

```markdown
1. 先理解再回答
   → 问清楚 XY 问题：用户是在问"怎么做 X"还是实际想做 Y？
   → "你想实现什么功能？可能有比 X 更好的方法。"

2. 给代码，不只给解释
   → 一个 5 行的完整示例 > 100 字的解释
   → 最好能直接复制运行的代码

3. 解释"为什么"
   → 不要只说"用 Rc<RefCell<T>>"
   → 要说"因为你需要在多个地方共享可变引用，而 borrow checker 不允许……"

4. 承认局限
   → "这个方案在 X 情况下会出问题——注意 Y"
   → 诚实 = 可信度

5. 链接到权威来源
   → The Book / The Reference / RFC / 标准库文档
   → 让提问者学会自己查找——授人以渔
```

## 3. 技术演讲

### 3.1 从 Meetup 到 Keynote 的路径

```
阶段 1: 本地 Rust Meetup（5-10 人）
  → 主题：你最近学到的 Rust 技巧
  → 时长：10-15 分钟（lightning talk）
  → 风险极低——听众是你的同城 Rust 爱好者
  → 如果你所在城市没有 Rust Meetup——组织一个！

阶段 2: 本地技术大会（50-100 人）
  → 主题：你在项目中用 Rust 解决的实际问题
  → 时长：20-30 分钟
  → 开始积累 speaker 经验

阶段 3: RustConf / RustNation / RustFest（500+ 人）
  → 主题：深度的技术分析（编译器、性能、系统编程）
  → 时长：30-45 分钟
  → 通过 CFP（Call for Proposals）提交
  → CFP 被拒是常态——每次被拒都是一次改进机会

阶段 4: 受邀 Keynote / 主题演讲
  → 受邀——不需要提交 CFP
  → 主题通常是你已经成名的领域
  → 这是"社区声誉"的最高形式之一
```

### 3.2 写一个好的 CFP

```markdown
# CFP 的要素

1. 标题（Title）
   → 具体、有诱惑力、不超过 10 词
   → 好: "How I Shrunk a Rust Binary from 15MB to 400KB"
   → 差: "Rust Performance Optimization"

2. 摘要（Abstract）—— 100-200 字
   → 问题 → 为什么重要 → 我的方案 → 听众能学到什么
   → 展示你的独特角度："我在真实项目中做了 X，踩了 Y 个坑"

3. 听众收获（Takeaways）—— 3 个要点
   → "听众离开时会知道……"
   → 具体 > 抽象
   → 好: "知道如何用 cargo-bloat 定位最大的依赖"
   → 差: "了解 Rust 编译优化"

4. 受众水平（Audience Level）
   → Beginner / Intermediate / Advanced
   → 诚实——不要谦虚也不要夸大

5. 关于你（Bio）
   → 2-3 句——你是谁，为什么适合讲这个主题
   → "我在 X 公司用 Rust 做了 Y 项目……"
```

## 4. 建立技术声誉

### 4.1 声誉的三个支柱

```
┌──────────────┬─────────────────────────────────────────┐
│ 支柱          │ 怎么建立                                │
├──────────────┼─────────────────────────────────────────┤
│ 一致性        │ 每周回答 3 个 URLO 问题                  │
│ (Consistency) │ 每月写 1 篇博客                         │
│              │ 每季做 1 次 Code Review（非自己项目）     │
│              │ → "这个人一直在做这件事"                 │
├──────────────┼─────────────────────────────────────────┤
│ 可见性        │ 写下来——不要只存在脑子里                │
│ (Visibility)  │ 博客 > 碎片化社交媒体                    │
│              │ GitHub 贡献记录 > 口头说"我做过 X"        │
│              │ → "别人能找到你的工作成果"               │
├──────────────┼─────────────────────────────────────────┤
│ 慷慨          │ 帮助新手——每个资深 Rustacean 都曾是新手  │
│ (Generosity)  │ Review 别人的 PR（即使没有权限）         │
│              │ 在 RFC 讨论中帮助提案者打磨 idea          │
│              │ → "和这个人合作很愉快"                   │
└──────────────┴─────────────────────────────────────────┘
```

### 4.2 声誉的复利效应

```
声誉不是"一天建成的"——
而是每天一小步的累积：

第 1 个月: 你回答了 5 个 URLO 问题
第 3 个月: 有人 @ 你说"上次你的回答帮我解决了问题"
第 6 个月: 你的博客开始出现在 TWiR 中
第 9 个月: 有人邀请你 review 他们的 RFC
第 12 个月: 你的 RustConf CFP 被接受了
第 18 个月: 你被邀请成为一个 crate 的 co-maintainer

关键：前 6 个月几乎没有可见的回报。
坚持的人在第 9-12 个月开始看到成果。
大部分人第 3 个月就放弃了——这就是你的机会。
```

## 5. Code Review 作为贡献

### 5.1 为什么 Review 是贡献？

```
Review 一个 PR 需要的技能：
  → 理解代码意图
  → 识别潜在的 edge case
  → 检查 API 设计是否一致
  → 验证测试是否充分

这比修一个 bug 更难——因为你需要：
  → 进入别人的思维（理解他们为什么这样写）
  → 看到他们没看到的（预测可能的问题）

好的 reviewer 和好的 contributor 同样稀缺。
当你开始做 review——你就不再是"消费者"，你是"守门人"。
```

### 5.2 写有建设性的 Review Comment

```markdown
差 Review:
  "This is wrong."           → 无用的。为什么 wrong？怎么改？
  "Use X instead."           → 有用的但命令式的。
  "Why did you do this?"      → 听起来像质疑能力。

好 Review:
  "I think this might panic if `list` is empty.
   Could we add a check here?"
  → 指出具体问题 + 建议方案 + 请求式语气

  "The naming `process_data` is a bit vague.
   Maybe `filter_valid_entries` would be clearer?"
  → 指出问题 + 提供替代方案

  "This looks great! One small thing:
   could you add a test for the empty case?"
  → 先认可 + 再建议。让人感到被尊重。

格式：认可 + 具体问题 + 建议方案
  → "Nice approach! I wonder if [edge case] would break this.
     Maybe we could [suggestion]?"
```
