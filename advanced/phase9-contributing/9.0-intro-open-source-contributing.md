# 9.0 开源贡献与社区参与概述

> 你花了 8 个 Phase 学习 Rust——类型系统、内存控制、异步编程、底层突破、元编程、工程实践、编译器内部、生态源码。现在，你需要换一个身份了。不是"Rust 学习者"，而是"Rust 贡献者"。Phase 9 不是教你更多 Rust 知识——Phase 7 和 Phase 8 已经给了你足够的技术武器。Phase 9 教的是：**如何把你的能力注入到 Rust 本身的演进中**。这不只是"提 PR"——这是理解一个语言的治理结构、找到你的位置、建立你的声誉、最终影响这门语言走向何方的完整路径。

---

## 1. 为什么贡献？

### 1.1 对你个人的价值

```
贡献 Rust 的三重回报：

1. 学习加速
   ─────────
   • 读别人的代码是最好的学习方式（Phase 8 的延续）
   • reviewer 的反馈是免费的顶级 mentorship
   • 修复 bug 比写新代码更考验理解深度

2. 技术声誉
   ─────────
   • 在 Rust 社区中，代码贡献是最硬的通货
   • "那个给 std 加了 is_sorted 的人"——你的贡献会成为你的名片
   • Rust 团队成员的招募主要来自活跃贡献者

3. 语言影响力
   ─────────
   • 参与 RFC 讨论，影响 Rust 的发展方向
   • 你用的每一个 Rust feature，背后都有人为它辩护过
   • 下一个改变 Rust 的提案，为什么不能是你的？
```

### 1.2 对社区的价值

```
开源项目的生命线不是钱——是人。

维护者 burnout 是开源最大的威胁：
  "issue 堆积 → 没人 review → 贡献者流失 → 维护者更累 → burnout"

每多一个活跃贡献者 → 维护者负担减轻一点 → 项目更健康一点
→ 你用的工具更不容易被 abandon

你的第一个 PR 可能只是修一个拼写错误，
但你的第十个 PR 可能就是 review 别人的代码了。
```

## 2. 贡献的 5 种形式

```
┌──────────────┬────────────────────────────────────────────┐
│ 形式          │ 具体做什么                                  │
├──────────────┼────────────────────────────────────────────┤
│ 代码          │ 修 bug、加 feature、优化性能               │
│              │ 这是最"可见"的贡献，但不是唯一的            │
├──────────────┼────────────────────────────────────────────┤
│ 文档          │ 改进 API 文档、写教程、翻译                │
│              │ 很多 crate 的文档比代码更需要帮助           │
├──────────────┼────────────────────────────────────────────┤
│ 测试          │ 加测试用例、修 flaky test、提高覆盖率      │
│              │ 写测试是理解代码最快的方式之一              │
├──────────────┼────────────────────────────────────────────┤
│ 设计讨论      │ 参与 RFC/pre-RFC、issue 讨论、API 设计    │
│              │ 好的设计 review 有时比代码贡献更有价值      │
├──────────────┼────────────────────────────────────────────┤
│ 社区建设      │ 回答问题、写博客、组织 meetup、mentoring  │
│              │ 每个资深 Rustacean 都曾是新手              │
└──────────────┴────────────────────────────────────────────┘
```

## 3. Rust 项目治理结构

### 3.1 谁决定 Rust 变成什么样？

```
Rust 项目的治理结构（简化）：

                  Rust Foundation（法律/财务）
                         │
                  ┌──────┴──────┐
                  │   Council   │（治理决策）
                  └──────┬──────┘
                         │
        ┌────────────────┼────────────────┐
        │                │                │
   Lang Team        Libs Team       Compiler Team
   （语言设计）       （标准库）        （编译器实现）
        │                │                │
   RFC 决策         API 审批         实现审核

   ┌────────┐    ┌──────────┐    ┌──────────────┐
   │ Clippy │    │  Cargo   │    │  Moderation   │
   │  Team  │    │   Team   │    │     Team      │
   └────────┘    └──────────┘    └──────────────┘

   ┌────────────┐  ┌────────────┐  ┌──────────────┐
   │ Async WG  │  │ Embedded WG│  │  WASM WG     │
   └────────────┘  └────────────┘  └──────────────┘
```

### 3.2 团队如何决策？

```
不同团队的决策机制不同：

Lang Team:
  - 通过 RFC 流程做语言设计决策
  - 需要团队成员达成共识（FCP）
  - 单个成员可以"check your box"（表示审查通过）
  - 被否决的提案不意味着"永远不行"——可以改进后重新提出

Libs Team:
  - 通过 libs-api 会议讨论 API 提案
  - 重视一致性（新 API 必须和已有 API 风格一致）
  - 极其保守（加了就不能删——所以加之前想清楚）

Compiler Team:
  - 通过 MCP（Major Change Proposal）管理大改动
  - 对性能退化零容忍（perf.rust-lang.org 持续监控）
  - 重视向后兼容（你不能让任何 stable 代码 break）

Clippy Team:
  - 相对轻量——lint 审核比语言特性快得多
  - 关注误报率（false positive rate）
  - 新人友好——有专门的 "good first issue" 标签
```

## 4. 贡献的难度阶梯

### 4.1 从易到难：你的第一年贡献路线图

```
阶段 1: 生态 Crate（第 1-3 个月）
  ├── 修拼写错误 / 改进文档
  ├── 修小 bug（good first issue）
  ├── 加测试
  └── 目标：熟悉 PR 流程和 review 文化

阶段 2: Clippy（第 3-6 个月）
  ├── 写新的 clippy lint
  ├── 改进已有 lint 的建议信息
  ├── 参与 lint 审核讨论
  └── 目标：理解编译器开发工具链

阶段 3: 标准库（第 6-9 个月）
  ├── 添加 Iterator 方法
  ├── 改进文档示例
  ├── 参与 libs-api 讨论
  └── 目标：建立 API 设计直觉

阶段 4: 编译器 / RFC（第 9-12 个月）
  ├── 修 ICE（内部编译器错误）
  ├── 改进诊断信息
  ├── 参与 RFC 讨论
  └── 目标：影响语言设计决策
```

### 4.2 不需要按顺序来

```
你完全可以跳过阶段 2 直接做阶段 4——如果那是你的兴趣所在。
但这个阶梯反映了大多数成功贡献者的路径：

"生态 crate → Clippy → std → rustc → RFC"

每一步都建立在前一步的"流程熟悉度"之上。
```

## 5. Phase 9 学习路线图

```
9.1 RFC 流程 → 理解 Rust 如何做决策（语言设计的第一课）
     ↓
9.2 rustc/Clippy → Phase 7 的实战延伸（从理论到贡献）
     ↓
9.3 标准库 → API 设计的实战（最保守的贡献目标）
     ↓
9.4 生态 Crate → 最常见的入门路径（最容易出成果）
     ↓
9.5 社区参与 → 代码之外的贡献（建立声誉）
     ↓
9.6 综合实战 → 完成一次真正的开源贡献（Phase 9 的毕业礼）
```

> **学习建议**：9.1 和 9.5 是"流程和人文"课——可以先读建立全局视野。9.2、9.3、9.4 是"实操"课——跟着动手。9.6 是最终考验——它不需要你学新东西，它需要你把前 5 课学到的东西组织成一次完整的行动。
