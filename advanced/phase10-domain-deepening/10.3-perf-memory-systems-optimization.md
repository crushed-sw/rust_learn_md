## 1. CPU Cache 架构：从第一性原理讲起

### 1.1 为什么需要 Cache：内存墙

CPU 和内存之间有一个 200x 的鸿沟。现代 CPU 一个周期大约 0.3ns（@3.3GHz），执行一条简单指令只需 1-2 个周期。但访问主内存（DRAM）需要 ~60-100ns——这意味着一个 L3 cache miss = CPU 白白等待约 200 个周期的指令。

Cache 的存在就是为了**用空间换时间**——把最经常访问的数据放在离 CPU 最近的地方。

```
┌──────────────────────────────────────────────────────────────┐
│        CPU → 内存 的访问延迟金字塔                              │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│                        ┌──────┐                               │
│                        │ CPU  │                               │
│                        │ 寄存器│  ~0 cycles                   │
│                        └──┬───┘                               │
│                           │                                    │
│                      ┌────▼────┐                              │
│                      │   L1    │  32KB,  ~1ns  (~3-4 cycles)  │
│                      │  cache  │  每核心私有                    │
│                      └────┬────┘                              │
│                           │                                    │
│                      ┌────▼────┐                              │
│                      │   L2    │  256KB-512KB, ~3-5ns (~12 cy) │
│                      │  cache  │  每核心私有                    │
│                      └────┬────┘                              │
│                           │                                    │
│                      ┌────▼────┐                              │
│                      │   L3    │  8MB-32MB,  ~12-20ns (~40cy) │
│                      │  cache  │  所有核心共享                  │
│                      │ (LLC)   │                               │
│                      └────┬────┘                              │
│                           │                                    │
│                      ┌────▼────┐                              │
│                      │  DRAM   │  8GB-256GB,  ~60-100ns        │
│                      │  主内存   │  (~200+ cycles)              │
│                      └─────────┘                              │
│                                                               │
│  L1 miss → L2 hit:  ~10 cycles 浪费                           │
│  L2 miss → L3 hit:  ~40 cycles 浪费                           │
│  L3 miss → DRAM:    ~200 cycles 浪费 ← 性能灾难               │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 Cache Line：数据传输的最小单位

CPU 从不单独读取一个字节——它每次从内存读取 **64 字节**（在 x86-64 上），称为一个 **cache line**。这意味着：

- 访问 `data[0]`（u64，8 字节）→ CPU 实际读取了 `data[0..8]`（64 字节，8 个 u64）
- 如果你紧接着访问 `data[1]` → 已在 cache 中，极快（~1ns）
- 如果你访问的是 `data[1000]` → 在另一个 cache line 上，可能需要重新从内存读取

```rust
/// Cache line 效果的直观演示
/// 两个遍历方式，算法复杂度相同但性能可以差 4-8 倍
fn row_major_traversal(matrix: &[[u64; 1024]; 1024]) -> u64 {
    // 按行遍历：访问的数据在内存中连续
    // data[0][0], data[0][1], data[0][2], ...
    // → 每次 cache miss 后可以连续命中 7 次（64B / 8B = 8 个 u64）
    let mut sum = 0u64;
    for row in 0..1024 {
        for col in 0..1024 {
            sum = sum.wrapping_add(matrix[row][col]);
        }
    }
    sum
}

fn column_major_traversal(matrix: &[[u64; 1024]; 1024]) -> u64 {
    // 按列遍历：访问的数据在内存中分散
    // data[0][0], data[1][0], data[2][0], ...
    // → 每次访问都可能 cache miss（因为相邻行在内存中相隔 8KB）
    let mut sum = 0u64;
    for col in 0..1024 {
        for row in 0..1024 {
            sum = sum.wrapping_add(matrix[row][col]);
        }
    }
    sum
}
// 两个函数都是 O(N²)，但在 1024×1024 的矩阵上 row_major 可以快 4-8 倍
```

## 2. Cache 感知的数据结构：从 AoS 到 SoA

### 2.1 AoS vs SoA：经典的选择

**AoS (Array of Structs)**：将所有字段打包在一个 struct 中，然后将 struct 放在数组里。这是 Rust 程序员的默认选择——很自然的 OOP 思维。

**SoA (Struct of Arrays)**：将每个字段拆成独立的数组。每个数组只包含一种类型的数据。

```
AoS 内存布局:                    SoA 内存布局:
┌────────────────────┐          ┌───────────┐
│ struct Particle {  │          │ x: [f32]  │ ← 64 字节全是 x 坐标
│   x: f32,          │          │ y: [f32]  │ ← 64 字节全是 y 坐标
│   y: f32,          │          │ vx: [f32] │ ← 64 字节全是 vx
│   vx: f32,         │          │ vy: [f32] │ ← 64 字节全是 vy
│   vy: f32,         │          │ alive: bool│ ← 独有字段
│   alive: bool,     │          └───────────┘
│ }                  │
│ vec<Particle>      │
└────────────────────┘

关键差异: 当只遍历 x 和 y 时:
AoS: 每个 cache line 包含 20B 有效数据 + 44B 浪费 (vx, vy, alive)
SoA: 每个 cache line 包含 64B 全部有效数据 (x 数组) —— 带宽效率 3.2x
```

```rust
// AoS vs SoA 的 Rust 实现对比

// === AoS 实现（默认选择，对 cache 不友好）===
#[repr(C)]
struct ParticleAoS {
    x: f32,
    y: f32,
    vx: f32,
    vy: f32,
    alive: bool,
    // padding: 3 bytes due to alignment
}

fn update_positions_aos(particles: &mut [ParticleAoS], dt: f32) {
    for p in particles.iter_mut() {
        if p.alive {
            // 遍历每颗粒子时，CPU 把整个 struct 加载到 cache
            // 但我们只需要 x, y, vx, vy——alive 和 padding 浪费了 cache 空间
            p.x += p.vx * dt;
            p.y += p.vy * dt;
        }
    }
}

// === SoA 实现（对 cache 友好，但代码不如 AoS 直观）===
struct ParticleSystem {
    x: Vec<f32>,
    y: Vec<f32>,
    vx: Vec<f32>,
    vy: Vec<f32>,
    alive: Vec<bool>,
}

impl ParticleSystem {
    fn update_positions_soa(&mut self, dt: f32) {
        // 热路径只访问 x, y, vx, vy 四个数组
        // 每个数组的 cache line 被 100% 有效利用
        for i in 0..self.x.len() {
            if self.alive[i] {
                // alive 的 cache miss 是额外代价，但 SoA 中 alive 数组独立——
                // 它的 miss 不会污染 x/y/vx/vy 的 cache
                self.x[i] += self.vx[i] * dt;
                self.y[i] += self.vy[i] * dt;
            }
        }
    }
}
```

### 2.2 Hot-Cold Splitting（热/冷字段分离）

当 struct 中有些字段被频繁访问（hot），有些几乎不碰（cold）时，将它们分开存储可以提升 cache 利用率：

```rust
// 热冷分离模式——把高频访问字段和低频字段分开
use std::sync::Arc;

/// 高频访问字段（hot）——每次查询都会碰到
#[repr(C)]
struct RecordHeader {
    id: u64,
    score: f64,         // 排序/比较频繁用到
    timestamp: u64,     // 过滤频繁用到
    // 这些字段放在一起，占据同一个 cache line
}

/// 低频访问字段（cold）——只在显示详情时用到
struct RecordPayload {
    description: String,   // 很少用到的长文本
    metadata: Vec<u8>,     // 很少用到的大二进制
    backup_ref: Arc<Vec<u8>>, // 更少用到
}

/// 组合起来：header 和 payload 分开存储
/// header 在最前面是因为其地址被频繁访问
struct Record {
    header: RecordHeader,       // 热数据
    payload: Option<Arc<RecordPayload>>, // 冷数据——Arc 避免大 clone
}

/// 遍历所有记录的 score 时，只有 header 被加载到 cache
/// Payload 完全不被碰——cache 中没有它的踪迹
fn find_top_scores(records: &[Record]) -> Vec<u64> {
    records.iter()
        .filter(|r| r.header.score > 0.9)
        .map(|r| r.header.id)
        .collect()
}
```

### 2.3 Benchmark 对比：AoS vs SoA

```rust
// Cargo.toml: criterion = "0.5"
use criterion::{black_box, criterion_group, criterion_main, Criterion};

const N: usize = 1_000_000;

fn bench_aos_vs_soa(c: &mut Criterion) {
    let mut group = c.benchmark_group("aos_vs_soa");
    group.bench_function("aos_update", |b| {
        let mut particles: Vec<ParticleAoS> = (0..N).map(|i| ParticleAoS {
            x: i as f32, y: 0.0, vx: 0.1, vy: 0.2, alive: true,
        }).collect();
        b.iter(|| update_positions_aos(black_box(&mut particles), black_box(0.016)));
    });
    group.bench_function("soa_update", |b| {
        let mut system = ParticleSystem {
            x: (0..N).map(|i| i as f32).collect(),
            y: vec![0.0; N], vx: vec![0.1; N], vy: vec![0.2; N], alive: vec![true; N],
        };
        b.iter(|| system.update_positions_soa(black_box(0.016)));
    });
    group.finish();
}
// 典型结果: AoS ~3.2ms/iter, SoA ~0.8ms/iter → SoA 快 ~4x（cache 带宽利用率差异）
```

## 3. Prefetch 指令：比 CPU 更早知道该读什么

### 3.1 硬件 Prefetcher 的盲区

现代 CPU 的硬件 prefetcher 非常擅长预测**顺序访问**模式——当你遍历一个 Vec 时，prefetcher 能提前几百个周期把后续 cache line 读入 L1/L2。但在以下场景中，prefetcher 完全失效：

1. **链表遍历**：每个节点的地址只在前一个节点里才知道
2. **哈希表查找**：key 的哈希值是随机的，访问模式不可预测
3. **树遍历**（B-Tree 除外）：二叉树的节点地址由指针决定
4. **大跨步访问**：跳过远超 cache line 大小的距离

在这些场景中，**手动 prefetch** 可以告诉 CPU 提前加载数据，减少 stall 周期。

### 3.2 std::intrinsics::prefetch 的使用

```rust
// 需要 nightly Rust: #![feature(core_intrinsics)]
use std::intrinsics::{prefetch_read_data, prefetch_write_data};

/// 链表遍历 + 手动 prefetch——mask hardware prefetcher 的盲区
struct Node {
    value: u64,
    next: Option<Box<Node>>,
}

// 链表遍历时硬件 prefetcher 无法预测下一个节点地址
// 手动 prefetch 可以让 CPU 提前开始加载下一个节点
fn linked_list_sum_with_prefetch(head: &Node) -> u64 {
    let mut sum = 0u64;
    let mut current: *const Node = head as *const Node;

    unsafe {
        // 预取第一个 next 节点（预热）
        if let Some(ref next) = (*current).next {
            prefetch_read_data(next as *const Node as *const i8, 3);
        }

        loop {
            // 读取当前值
            sum = sum.wrapping_add((*current).value);

            match &(*current).next {
                Some(next) => {
                    let next_ptr: *const Node = next.as_ref() as *const Node;
                    // 在访问当前节点的 next 之前，提前预取 "next的next"
                    // locality 参数 3 = 所有层级的 cache
                    if let Some(ref nn) = (*next_ptr).next {
                        prefetch_read_data(
                            nn.as_ref() as *const Node as *const i8,
                            3,
                        );
                    }
                    current = next_ptr;
                }
                None => break,
            }
        }
    }

    sum
}

// locality 参数的含义:
// 0: 仅 L1 cache（用完就丢的数据）
// 1: L1 + L2
// 2: L1 + L2 + L3
// 3: 所有层级（会被多次访问的数据）——最常用
```

### 3.3 Prefetch 的衡量标准：什么时候值得

Prefetch 不是免费的——它消耗内存带宽并占用 cache 空间。使用前需要确认：

```
Prefetch 适用条件检查表:
□ 数据访问模式是非顺序的（硬件 prefetcher 会 miss）
□ 在需要数据之前至少 100-200 cycles 就知道要访问什么地址
□ 预取的数据量在 L1 容量之内（过多 prefetch 会挤出有用的数据）
□ profiling 确认当前瓶颈确实是 cache miss（不是计算、不是分支预测）

不适用场景:
→ 遍历 Vec（硬件 prefetcher 已足够好）
→ 预取地址的计算本身比 cache miss 还贵
→ L1/L2 cache 已经很紧张的情况下
```

## 4. 热路径的分配器策略：Bump、Slab、Arena

### 4.1 为什么通用分配器在热路径上不够快

`jemalloc`、`mimalloc` 等通用分配器非常优秀——但它们要做很多热路径上不需要的事情：线程缓存（tcache）、大小分类（binning）、碎片合并（coalescing）。在每秒百万次分配的热路径上，这些操作是沉重的包袱。

专用分配器放弃通用性，换取极致速度。

```
分配器性能光谱（简化）:

通用分配器 (malloc/jemalloc/mimalloc)
  │
  ├── 优点: 通用、线程安全、内存利用率高
  ├── 缺点: 每次分配 ~20-50ns（最佳情况），有碎片
  │
  ▼
专用分配器
  │
  ├── Bump Allocator:   ~2ns/alloc,  不能单独释放，适合 per-request
  ├── Slab Allocator:   ~5ns/alloc,  只能分配固定大小，适合同质对象
  └── Arena Allocator:  ~10ns/alloc, 整批释放，适合阶段性计算
```

### 4.2 Bump Allocator：最快但最受限

```rust
use std::alloc::{alloc, Layout};
use std::sync::atomic::{AtomicUsize, Ordering};

/// Bump Allocator——~2ns/alloc。代价：不能单独释放，只能一起 reset。
struct BumpAllocator {
    memory: *mut u8,
    capacity: usize,
    offset: AtomicUsize, // 原子累加——无锁并发
}

impl BumpAllocator {
    fn new(capacity: usize) -> Self {
        let layout = Layout::from_size_align(capacity, 64).unwrap();
        let memory = unsafe { alloc(layout) };
        assert!(!memory.is_null(), "allocation failed");
        BumpAllocator { memory, capacity, offset: AtomicUsize::new(0) }
    }

    /// 分配 size 字节，对齐到 align。CAS 循环保证线程安全
    fn alloc(&self, size: usize, align: usize) -> Option<*mut u8> {
        let current = self.offset.load(Ordering::Relaxed);
        let aligned = (current + align - 1) & !(align - 1);
        if aligned + size > self.capacity { return None; }
        match self.offset.compare_exchange(
            current, aligned + size, Ordering::SeqCst, Ordering::Relaxed,
        ) {
            Ok(_) => Some(unsafe { self.memory.add(aligned) }),
            Err(_) => self.alloc(size, align), // CAS 失败——重试
        }
    }

    fn reset(&self) { self.offset.store(0, Ordering::SeqCst); }
}
```

### 4.3 Slab Allocator：同质对象的批量管理

```rust
/// Slab Allocator——为同类型的对象管理预分配内存，可单独释放和复用
struct SlabAllocator<T> {
    memory: Vec<T>,
    free_list: Vec<usize>, // 工业级实现用 free list 链表（在空闲 slot 内存储 next）
}

impl<T: Default + Clone> SlabAllocator<T> {
    fn new(capacity: usize) -> Self {
        SlabAllocator { memory: vec![T::default(); capacity], free_list: (0..capacity).collect() }
    }
    fn allocate(&mut self) -> Option<usize> { self.free_list.pop() } // ~5ns: O(1) push
    fn deallocate(&mut self, idx: usize) { self.free_list.push(idx); } // ~5ns: O(1) pop
    fn get_mut(&mut self, idx: usize) -> &mut T { &mut self.memory[idx] }
}
```

### 4.4 分配器决策指南

| 场景 | 推荐分配器 | Crate | 原因 |
|------|-----------|-------|------|
| Per-request 临时数据 | Bump | `bumpalo` | 最快，reset 即释放 |
| 同类型的大量对象 | Slab | `slab` | 固定高效，可复用 slot |
| 阶段性计算（分批释放）| Arena | `typed-arena` | 分组管理，整组释放 |
| 通用热路径 | tcmalloc 替代 | `mimalloc` | 多线程友好 |
| 自定义全局分配器 | 自定义 GlobalAlloc | - | 完全控制 |

## 5. 内存映射 I/O 与大页优化

### 5.1 mmap vs read/write

对于大文件的随机访问，`mmap` 提供了一种区别于 `read()` 的 I/O 模型：

```rust
use memmap2::Mmap;

/// mmap 零拷贝文件读取——直接映射内核 page cache，无需 copy_from_user
fn read_file_with_mmap(path: &str) -> std::io::Result<()> {
    let file = std::fs::File::open(path)?;
    let mmap = unsafe { Mmap::map(&file)? }; // 映射不实际读数据
    // 首次访问按页触发 page fault，后续命中 page cache
    let _sum: u64 = mmap.iter().take(1024 * 1024).map(|&b| b as u64).sum();
    Ok(())
}

/// O_DIRECT 直接 I/O——绕过 page cache（数据库等自己管理缓存的场景）
fn read_direct_io(path: &str) -> std::io::Result<()> {
    use std::os::unix::fs::OpenOptionsExt;
    let _file = std::fs::OpenOptions::new().read(true)
        .custom_flags(libc::O_DIRECT).open(path)?;
    // 缓冲区必须 page-aligned：memmap2 的匿名映射满足此要求
    let _buf = memmap2::MmapMut::map_anon(4096 * 1024)?;
    Ok(())
}
```

### 5.2 大页（Huge Pages）

默认的页面大小是 4KB。对于大内存工作负载（数据库、JVM、大型缓存），4KB 的页面粒度意味着 TLB（Translation Lookaside Buffer）条目不够用，导致频繁的 TLB miss——这会增加 ~20-50ns 的地址转换延迟。

Linux 支持两种大页：
- **2MB 页**（透明大页，THP）：内核自动使用，无需修改代码
- **1GB 页**：需要显式配置和代码修改

```rust
/// 使用 memmap2 分配 2MB 大页（透明大页）
/// 关键：分配 size 必须是 2MB 的倍数，且对齐到 2MB 边界
fn allocate_huge_page_aligned(size: usize) -> std::io::Result<memmap2::MmapMut> {
    let huge_page_size = 2 * 1024 * 1024; // 2MB
    // 对齐到 2MB 边界——THP 将自动生效
    let aligned_size = ((size + huge_page_size - 1) / huge_page_size) * huge_page_size;

    // 使用 mmap（而非 malloc）分配，对齐到 2MB
    // memmap2 的匿名映射保证页面对齐
    let mmap = memmap2::MmapOptions::new()
        .len(aligned_size)
        .map_anon()?;

    Ok(mmap)
}

// 通过 /proc 验证大页使用情况：
// $ cat /proc/meminfo | grep -i huge
// AnonHugePages:    204800 kB   ← 匿名透明大页已使用 200MB
// ShmemHugePages:        0 kB
// HugePages_Total:       0       ← 保留大页（需显式配置）
```

```bash
# 开启透明大页：
echo always > /sys/kernel/mm/transparent_hugepage/enabled

# 验证效果（perf）：
perf stat -e dTLB-load-misses,dTLB-loads ./your_binary
# TLB miss rate < 1%  → 良好
# TLB miss rate > 5%  → 考虑大页
```

## 6. Cache Line 对齐与虚假共享

### 6.1 虚假共享（False Sharing）：无声的性能杀手

虚假共享不在 profiler 里直接可见——它表现为"为什么这段代码比我预期的慢？"你会看到 CPU 使用率高但吞吐量上不去，cache miss 率很高但没有显式的锁。

**根本原因**：两个线程各自频繁写入不同的变量，但这两个变量碰巧在同一个 cache line 里。每次写入都会使对方的 cache line 失效——CPU 为此花费的周期全部浪费在 cache 一致性协议上。

```
┌──────────────────────────────────────────────────────────────┐
│              虚假共享的机制                                    │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Cache Line 0 (64 bytes):                                     │
│  ┌─────────────────┬──────────────────┐                      │
│  │  thread_0 写入   │  thread_1 写入    │                      │
│  │  counter_a (u64) │  counter_b (u64)  │                      │
│  └─────────────────┴──────────────────┘                      │
│                                                               │
│  线程 0 写入 counter_a → Core 0 的 cache line 变成 Modified   │
│  线程 1 写入 counter_b → Core 1 请求此 cache line             │
│  Core 0 必须 flush 到 L3/L2 → ~40 cycles 浪费                │
│  Core 1 拿到 cache line → 修改 counter_b                     │
│  Core 0 再次写 counter_a → 又要 flush ...                     │
│                                                               │
│  结果: 两个不相关的计数器，写入速度比独立时慢 50-100 倍        │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 解决方案：`#[repr(align(64))]` 和 `CachePadded`

```rust
// 方案 A: 使用 repr(align(64)) 手动对齐
// 确保每个原子变量独占一个 cache line（64 字节）

#[repr(align(64))]
struct AlignedCounter {
    value: std::sync::atomic::AtomicU64,
}

impl AlignedCounter {
    fn new(v: u64) -> Self {
        AlignedCounter {
            value: std::sync::atomic::AtomicU64::new(v),
        }
    }
}

// 现在即使两个 AlignedCounter 在数组里相邻，也不会虚假共享
let counters: Vec<AlignedCounter> = (0..8)
    .map(|i| AlignedCounter::new(i))
    .collect();
// 每个 counter 对齐到 64B 边界 → 各自独占 cache line
```

```rust
// 方案 B: 使用 crossbeam::utils::CachePadded（推荐）
// 这是工业级的实现，自动处理对齐和 padding

use crossbeam::utils::CachePadded;
use std::sync::atomic::AtomicU64;

struct WorkerState {
    // CachePadded 包装后，每个字段独占 64 字节的 cache line
    completed: CachePadded<AtomicU64>,
    in_flight: CachePadded<AtomicU64>,
    error_count: CachePadded<AtomicU64>,
}

// 即使两个 WorkerState 相邻分布，各自的 completed 也互不干扰
// 因为 CachePadded 保证它们不会落入同一个 cache line
```

### 6.3 虚假共享的 Benchmark 验证

```rust
use crossbeam::utils::CachePadded;
use std::sync::atomic::{AtomicU64, Ordering};
use std::thread;
use std::time::Instant;

const ITERATIONS: u64 = 10_000_000;
const NUM_THREADS: usize = 4;

/// 无 padding——4 个 AtomicU64 可能共享 cache line，虚假共享严重
fn bench_without_padding() {
    let counters: Vec<AtomicU64> = (0..NUM_THREADS).map(|_| AtomicU64::new(0)).collect();
    let start = Instant::now();
    thread::scope(|s| {
        for i in 0..NUM_THREADS {
            let c = &counters[i];
            s.spawn(move || { for _ in 0..ITERATIONS { c.store(1, Ordering::Relaxed); } });
        }
    });
    let ops = (NUM_THREADS as u64 * ITERATIONS) as f64 / start.elapsed().as_secs_f64();
    println!("无 padding:   {:.0} ops/s", ops);
}

/// 使用 CachePadded——每个 AtomicU64 独占 cache line，零虚假共享
fn bench_with_padding() {
    let counters: Vec<CachePadded<AtomicU64>> = (0..NUM_THREADS)
        .map(|_| CachePadded::new(AtomicU64::new(0))).collect();
    let start = Instant::now();
    thread::scope(|s| {
        for i in 0..NUM_THREADS {
            let c = &counters[i];
            s.spawn(move || { for _ in 0..ITERATIONS { c.store(1, Ordering::Relaxed); } });
        }
    });
    let ops = (NUM_THREADS as u64 * ITERATIONS) as f64 / start.elapsed().as_secs_f64();
    println!("With padding: {:.0} ops/s", ops);
}
// 典型输出（4 核，10M iterations each）:
// 无 padding:   ~17M ops/s ← 虚假共享 → 性能灾难
// With padding: ~1B ops/s  ← 无虚假共享 → 近线扩展
// 差异：~50x
```

### 6.4 虚假共享检测清单

设计多线程数据结构时，问自己：

```
□ 这个结构会被多个线程同时写入吗？
  → 如果只是读——无虚假共享风险
  → 如果只有一个线程写——无风险
  → 如果多线程同时写不同的字段 → 有风险

□ 被写入的字段在同一个 struct 内且总大小 < 128 字节吗？
  → 是 → 它们可能共享 cache line，有风险

□ 解决方式：
  → 方案 1: CachePadded 包装每个会被独立写入的字段
  → 方案 2: #[repr(align(64))] 独立分配每个字段
  → 方案 3: 重新设计——用 per-thread 数据结构替代共享结构
```
