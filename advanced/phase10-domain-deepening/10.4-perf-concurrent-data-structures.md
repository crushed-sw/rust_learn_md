# 10.4 并发数据结构：从 Crossbeam 到自定义

> **[Track A: 性能工程]** 当单线程优化达到极限后，并发是唯一出路。但并发编程的核心难点不在于"起多少线程"，而在于"线程之间如何高效、安全地共享数据"。本章深入 Rust 并发数据结构的生态——从 Crossbeam 的工业级 lock-free 实现，到你亲手打造一个 MPMC 队列，再到 loom 帮你找出那些肉眼看不见的并发 bug。

**前置知识：** 你应该已经学过 [[../phase3-async-concurrency/3.2-atomic-memory-ordering|3.2 原子操作与内存序]] 和 [[../phase3-async-concurrency/3.8-mpsc-oneshot-channels|3.8 通道]]，以及 [[../phase4-unsafe-ffi/4.10-custom-allocators|4.10 自定义分配器]]。本章假定你已理解 `Ordering::Acquire/Release` 的语义、CAS 的基本原理，以及 Rust 中 `unsafe` 的正确使用方式。

## 一、为什么需要并发数据结构

在深入具体数据结构之前，先理解"并发数据结构"解决的是什么问题。假设你要构建一个高性能的日志聚合器，多个生产者线程写入，一个消费者线程读取并聚合：

```rust
// ❌ 简单粗暴的加锁方案——伸缩性极差
use std::sync::Mutex;
use std::collections::VecDeque;

struct LogCollector {
    queue: Mutex<VecDeque<String>>,
}

impl LogCollector {
    fn push(&self, msg: String) {
        // 每次 push 都竞争同一把锁——8 核上也是串行
        self.queue.lock().unwrap().push_back(msg);
    }
}
```

这把锁成为瓶颈：
- **锁竞争（lock contention）**：一个线程持有锁时，其他线程都在排队。
- **缓存行乒乓（cache line ping-pong）**：锁变量在 CPU 核心之间被反复无效化。
- **上下文切换（context switch）**：等待锁的线程被操作系统挂起，唤醒代价高达几千个 CPU 周期。

**并发数据结构的解决思路**：

```
┌─────────────────────────────────────────────────────────────┐
│                    并发数据结构设计方法                        │
├─────────────────┬─────────────────┬─────────────────────────┤
│   策略           │   代表结构       │   核心技巧               │
├─────────────────┼─────────────────┼─────────────────────────┤
│ Lock-Free       │ Crossbeam SegQueue│ CAS 循环 (无锁)       │
│ Fine-Grained    │ DashMap           │ 分段锁 (多个细粒度锁)  │
│ Read-Optimized  │ evmap, left-right │ 读路径无锁，写路径隔离 │
│ Epoch-Based GC  │ crossbeam-epoch   │ 延迟回收，无引用计数   │
│ Wait-Free 边界  │ ArrayQueue        │ 有限容量但无需 CAS 循环 │
└─────────────────┴─────────────────┴─────────────────────────┘
```

## 二、Crossbeam 生态系统深入

Crossbeam 不是一个 crate，而是一个**生态**。`crossbeam` 元 crate 下包含六个子 crate，各自解决并发编程的一个侧面：

| Crate | 作用 | 使用场景 |
|-------|------|----------|
| `crossbeam-channel` | 多生产者多消费者通道 | 需要 select! 的复杂拓扑 |
| `crossbeam-deque` | 工作窃取双端队列 | rayon 的内部基础 |
| `crossbeam-epoch` | Epoch-based 内存回收 | 构建 lock-free 结构时的内存安全 |
| `crossbeam-queue` | Lock-free 队列 | 高性能 MPMC/SPSC |
| `crossbeam-skiplist` | 并发跳表 | 有序并发 map |
| `crossbeam-utils` | 缓存行对齐、Backoff 等 | 构建自定义并发原语 |

### 2.1 SegQueue —— 无锁 MPMC 队列

`SegQueue`（分段队列，Segmented Queue）是 Crossbeam 中最常用的高性能队列。它的核心设计是按生产者分段，每个生产者拥有自己的"本地槽"：

```rust
use crossbeam_queue::SegQueue;
use std::thread;
use std::sync::Arc;
use std::time::Instant;

fn main() {
    // SegQueue: 无界、多生产者、多消费者、无锁
    let queue: Arc<SegQueue<u64>> = Arc::new(SegQueue::new());

    let producer_count = 4;
    let consumer_count = 2;
    let items_per_producer = 1_000_000;

    let start = Instant::now();

    // 生产者线程：各自 push
    let mut producers = vec![];
    for p in 0..producer_count {
        let q = queue.clone();
        producers.push(thread::spawn(move || {
            for i in 0..items_per_producer {
                // push 是 lock-free，几乎无竞争
                // 因为每个生产者有自己的本地段
                q.push((p << 32) | i as u64);
            }
        }));
    }

    // 消费者线程：汇总计数
    let mut consumers = vec![];
    let counter = Arc::new(std::sync::atomic::AtomicU64::new(0));
    for _ in 0..consumer_count {
        let q = queue.clone();
        let c = counter.clone();
        consumers.push(thread::spawn(move || {
            let mut local_count = 0u64;
            loop {
                // pop 也是 lock-free
                if let Some(_val) = q.pop() {
                    local_count += 1;
                } else if c.load(std::sync::atomic::Ordering::SeqCst)
                    >= producer_count * items_per_producer
                {
                    break;
                }
            }
            c.fetch_add(local_count, std::sync::atomic::Ordering::SeqCst);
        }));
    }

    // 等待所有线程结束
    for p in producers {
        p.join().unwrap();
    }
    for c in consumers {
        c.join().unwrap();
    }

    println!(
        "处理 {} 条消息，耗时 {:?}，速率: {:.0} 条/秒",
        counter.load(std::sync::atomic::Ordering::SeqCst),
        start.elapsed(),
        counter.load(std::sync::atomic::Ordering::SeqCst) as f64
            / start.elapsed().as_secs_f64()
    );
}
```

**SegQueue 内部原理**：

```
┌──────────────────────────────────────────────────────────────┐
│                     SegQueue 内存布局                          │
│                                                               │
│  生产者 0 ──→ [slot0a][slot0b][slot0c]...  (本地段)           │
│  生产者 1 ──→ [slot1a][slot1b]...                             │
│  生产者 2 ──→ [slot2a][slot2b][slot2c][slot2d]...             │
│                     │                                         │
│                     ▼                                         │
│              ┌────────────┐                                   │
│              │  全局链表   │  ← 各生产者的段链接成单向链表      │
│              │  (segment) │                                   │
│              └─────┬──────┘                                   │
│                    ▼                                          │
│             消费者遍历链表, 逐个 pop                            │
└──────────────────────────────────────────────────────────────┘
```

关键特性：
1. **push 几乎无竞争**：因为每个生产者 push 到自己的段，只有段满了才需要分配新段（需要一次 CAS）。
2. **pop 需要遍历**：消费者顺序遍历各段，如果全部为空则返回 None。
3. **无界**：不像固定容量队列那样会满，代价是内存可以无限增长。

**何时使用 SegQueue**：
- 生产者数量很多（4+），且 push 频率极高。
- 容忍无界内存增长（或能确保消费速率 > 生产速率）。
- 不关心消息顺序（不同生产者的消息会交错）。

### 2.2 ArrayQueue —— 有界固定容量队列

当需要背压（backpressure）或内存可控时，`ArrayQueue` 是更好的选择：

```rust
use crossbeam_queue::ArrayQueue;
use std::thread;
use std::sync::Arc;

fn main() {
    // ArrayQueue: 有界、固定容量、MPMC、lock-free
    // 容量必须是 2 的幂（内部使用位掩码代替取模）
    let queue: Arc<ArrayQueue<i32>> = Arc::new(ArrayQueue::new(1024));
    //                        容量 = 1024 ──┘

    let q_prod = queue.clone();
    let producer = thread::spawn(move || {
        for i in 0..10_000 {
            // push 可能失败（队列满时返回 Err）
            while q_prod.push(i).is_err() {
                // 忙等待或 yield——实际应用中用 backoff
                std::thread::yield_now();
            }
        }
    });

    let q_cons = queue.clone();
    let consumer = thread::spawn(move || {
        let mut sum = 0i64;
        for _ in 0..10_000 {
            loop {
                if let Some(val) = q_cons.pop() {
                    sum += val as i64;
                    break;
                }
                std::thread::yield_now();
            }
        }
        sum
    });

    producer.join().unwrap();
    let result = consumer.join().unwrap();
    println!("总和: {}", result);
    // 0+1+2+...+9999 = 49995000
}
```

**ArrayQueue vs SegQueue 对照表**：

| 特性 | SegQueue | ArrayQueue |
|------|----------|------------|
| 容量 | 无界 | 有界（构造时指定） |
| 背压 | 无（可能 OOM） | 天然支持（push 返回 Err） |
| push 竞争 | 极低（每生产者私有段） | CAS 竞争（共享环形缓冲区） |
| pop 竞争 | 低 | CAS 竞争 |
| 缓存局部性 | 一般（链表遍历） | 好（连续数组） |
| 推荐场景 | 生产者多、push 密集 | 需要背压、内存可控 |

### 2.3 crossbeam::channel —— 带 select! 的多功能通道

相比于 `std::sync::mpsc`，`crossbeam::channel` 支持：
- **多接收者**（std::sync::mpsc 只有单个接收者）
- **select! 宏**：同时监听多个通道
- **Sender 可克隆**（mpsc 的 Sender 也可克隆）

```rust
use crossbeam_channel::{unbounded, select, tick, after};
use std::time::Duration;
use std::thread;

fn main() {
    let (tx1, rx1) = unbounded::<&str>();
    let (tx2, rx2) = unbounded::<&str>();
    let (tx_done, rx_done) = unbounded::<()>();

    // 模拟两个不同速率的消息源
    let t1 = tx1.clone();
    thread::spawn(move || {
        for i in 0..5 {
            thread::sleep(Duration::from_millis(100));
            t1.send("慢速消息").unwrap();
        }
    });

    let t2 = tx2.clone();
    thread::spawn(move || {
        for i in 0..10 {
            thread::sleep(Duration::from_millis(20));
            t2.send("快速消息").unwrap();
        }
    });

    // 超时定时器
    let timeout = after(Duration::from_secs(1));

    // select! 同时监听多个通道和超时
    let mut count_fast = 0;
    let mut count_slow = 0;

    loop {
        select! {
            recv(rx1) -> msg => {
                if let Ok(m) = msg {
                    count_slow += 1;
                    println!("  [慢速] {}", m);
                }
            }
            recv(rx2) -> msg => {
                if let Ok(m) = msg {
                    count_fast += 1;
                    println!("  [快速] {}", m);
                }
            }
            recv(timeout) -> _ => {
                println!("⏰ 1秒超时，停止接收");
                break;
            }
        }
    }

    println!(
        "统计: 快速消息 {} 条, 慢速消息 {} 条",
        count_fast, count_slow
    );
}
```

**select! 的执行模型**：
1. 每次调用 `select!`，内部检查所有通道的"准备情况"。
2. 如果恰好一个通道有数据，直接返回。
3. 如果多个通道同时有数据，**随机选择一个**（公平性保证）。
4. 如果全部为空，阻塞直到任意一个有数据。

**典型使用场景**：
- 网络服务器：同时监听新连接、已连接套接字、关闭信号。
- GUI 事件循环：用户输入 + 网络响应 + 定时刷新。
- 工作窃取调度器：本地队列 + 全局队列 + 其他线程的窃取。

## 三、Epoch-Based 内存回收（crossbeam-epoch）

### 3.1 问题：Lock-Free 结构中的内存回收

在 lock-free 数据结构中，一个线程可能正在读取一个节点，而另一个线程同时将其删除。传统的引用计数的开销太大了（`Arc` 的原子操作开销），而直接 `free` 会导致 use-after-free：

```rust
// ❌ 这个问题在 lock-free 结构中无处不在
// 线程 A: 正在读取 node->next
// 线程 B: 此时删除了 node
// 线程 A: 访问 node->next → use-after-free!
```

### 3.2 Epoch 算法原理

`crossbeam-epoch` 使用 **epoch-based garbage collection**（基于纪元的内存回收）：

```
时间轴 ──────────────────────────────────────────────►

epoch 1    │  epoch 2    │  epoch 3    │  epoch 1'   │
           │             │             │             │
线程 A [读] ├─────────────┤             │             │
           │  pin(ep1)   │             │             │
线程 B [写]│  [删除node] │             │             │
           │  加入retire  │             │             │
           │  列表(ep1)   │             │             │
线程 C [读]│             │ [读]        │             │
           │             │ pin(ep2)   │             │
                          │             │             │
当所有线程都离开 epoch 1 ──→ 回收 ep1 的 retired 列表 │
```

**三步走**：

```rust
use crossbeam_epoch::{Atomic, Owned, Shared, Guard, pin, Collector};
use std::sync::atomic::Ordering;

// 定义一个 lock-free 栈的节点
struct Node<T> {
    value: T,
    next: Atomic<Node<T>>, // epoch 管理的原子指针
}

impl<T> Node<T> {
    fn new(value: T) -> Owned<Self> {
        Owned::new(Node {
            value,
            next: Atomic::null(),
        })
    }
}

// Lock-free Treiber Stack
struct LockFreeStack<T> {
    head: Atomic<Node<T>>,
}

impl<T> LockFreeStack<T> {
    fn new() -> Self {
        LockFreeStack {
            head: Atomic::null(),
        }
    }

    fn push(&self, value: T) {
        let guard = pin(); // ① 进入当前 epoch，阻止 GC 回收
        let mut node = Node::new(value);

        loop {
            let head = self.head.load(Ordering::Relaxed, &guard);
            // ② 设置新节点的 next 指针
            node.next.store(head, Ordering::Relaxed);
            // ③ CAS 尝试将 head 指向新节点
            match self.head.compare_exchange(
                head,
                node,
                Ordering::Release, // 写入对其他线程可见
                Ordering::Relaxed,
                &guard,
            ) {
                Ok(_) => break, // 成功！node 的所有权转交给 Atomic
                Err(returned_node) => {
                    // 失败：取回所有权，重试
                    node = returned_node.new;
                }
            }
        }
        // guard 在此 drop，离开当前 epoch
    }

    fn pop(&self) -> Option<T> {
        let guard = pin();
        loop {
            let head = self.head.load(Ordering::Acquire, &guard);
            // head 是 Shared<'_, Node<T>>，类似 &Node<T>
            // 需要用 unsafe 解引用
            match unsafe { head.as_ref() } {
                None => return None,
                Some(node_ref) => {
                    let next = node_ref.next.load(Ordering::Relaxed, &guard);
                    match self.head.compare_exchange(
                        head,
                        next,
                        Ordering::Release,
                        Ordering::Relaxed,
                        &guard,
                    ) {
                        Ok(_) => {
                            // 安全读取值（在 epoch 保护期）
                            let value = unsafe {
                                std::ptr::read(&node_ref.value)
                            };
                            // 延迟释放 node：当所有线程离开当前 epoch 时
                            unsafe { guard.defer_destroy(head); }
                            return Some(value);
                        }
                        Err(_) => continue, // CAS 失败，重试
                    }
                }
            }
        }
    }
}
```

**关键 API 说明**：

| API | 作用 |
|-----|------|
| `pin()` | 固定当前线程到当前 epoch（返回 Guard） |
| `Guard` | RAII 守卫，drop 时自动离开 epoch |
| `Atomic<T>` | 带 epoch 管理的原子指针 |
| `Owned<T>` | 拥有所有权的指针（还未发布到 Atomic） |
| `Shared<'g, T>` | 从 Atomic 加载得到的安全引用（受 Guard 保护） |
| `defer_destroy(ptr)` | 将指针加入退役列表，等待安全时机回收 |

**Guard 的生命周期是关键**：

```rust
use crossbeam_epoch::pin;

// ✅ 正确：Guard 存活期间可以安全访问 Shared
fn safe_read(stack: &LockFreeStack<i32>) -> Option<i32> {
    let guard = pin();                // ← Guard 创建
    let head = stack.head.load(std::sync::atomic::Ordering::Acquire, &guard);
    let val = unsafe { head.as_ref() }.map(|n| n.value);
    // Guard 在函数返回时 drop ──→ 离开当前 epoch
    val // 返回的值已经复制到栈上，不再需要 Guard
}

// ❌ 危险：返回了依赖 Guard 的引用
// fn unsafe_read<'g>(stack: &LockFreeStack<i32>, guard: &'g Guard) -> &'g i32 {
//     // 如果 Guard 在外部被 drop，这个引用就悬垂了
//     todo!()
// }
```

### 3.3 何时用 Epoch vs Arc vs 手动 unsafe

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 高并发读的不可变数据 | `Arc<T>` | Arc 的 clone 是原子的，读取无需 pin |
| Lock-free 结构，pop 频繁 | `crossbeam-epoch` | 延迟释放比 Arc 的原子计数更轻量 |
| 单线程或低并发 | `Box<T>` + 手动控制 | 无原子操作开销 |
| 确定生命周期 | 借用 | 最轻量，编译器保证安全 |
| 读多写少，容忍短暂不一致 | `evmap` / `left-right` | 见下一节 |

## 四、从零构建 Lock-Free MPMC 队列

### 4.1 设计思路

目标：一个无锁的、有界的、多生产者多消费者的队列。核心数据结构是一个环形缓冲区 + 两个原子索引：

```
┌──────────────────────────────────────────────────────────┐
│             Lock-Free MPMC 环形缓冲区                      │
│                                                           │
│  buffer: [slot0][slot1][slot2]...[slotN-1]                │
│            ▲                       ▲                      │
│            │                       │                      │
│         head (原子)             tail (原子)                │
│      (消费者从这里取)        (生产者从这里放)               │
│                                                           │
│  满队列: (tail + 1) % capacity == head                     │
│  空队列: head == tail                                      │
└──────────────────────────────────────────────────────────┘
```

### 4.2 ABA 问题与 Tagged Pointers

在 lock-free 编程中，**ABA 问题**是最经典的陷阱：

```
时间轴：
1. 线程 A 读取 head = 0x1000 (值 A)
2. 线程 B pop 了 slot head → head → 0x1001 (A → B)
3. 线程 B 又 push 了一个新值，恰好分配到同一个地址 0x1000 (B → A)
4. 线程 A 执行 CAS，发现 head 还是 0x1000，以为没变化 → CAS 成功
   但实际上队列结构已经变了！
```

**解决方案：Tagged Pointer（标记指针）**——在指针的高位嵌入一个版本号：

```rust
use std::sync::atomic::{AtomicUsize, Ordering};

/// 带标记的 "指针" — 低 48 位存真实索引，高 16 位存版本号
/// 假设 buffer 不会超过 2^48 个元素（这在现实中完全够用）
#[derive(Debug, Clone, Copy)]
struct TaggedPtr {
    packed: usize,
}

impl TaggedPtr {
    const INDEX_MASK: usize = (1 << 48) - 1;  // 低 48 位用于索引
    const TAG_SHIFT: usize = 48;               // 高 16 位用于标记

    fn new(index: usize, tag: usize) -> Self {
        TaggedPtr {
            packed: (tag << Self::TAG_SHIFT) | (index & Self::INDEX_MASK),
        }
    }

    fn index(&self) -> usize {
        self.packed & Self::INDEX_MASK
    }

    fn tag(&self) -> usize {
        self.packed >> Self::TAG_SHIFT
    }

    fn next_index(&self, capacity: usize) -> Self {
        Self::new((self.index() + 1) % capacity, self.tag())
    }

    fn next_tag(&self) -> Self {
        Self::new(self.index(), self.tag().wrapping_add(1))
    }
}
```

### 4.3 完整实现

```rust
use std::sync::atomic::{AtomicUsize, Ordering};
use std::cell::UnsafeCell;
use std::mem::MaybeUninit;

/// 无锁 MPMC 有界队列
/// 容量 capacity 必须使得 buffer 的大小为 capacity + 1
/// （因为要区分空和满的状态）
struct LockFreeMpmc<T> {
    buffer: Box<[UnsafeCell<MaybeUninit<T>>]>,
    capacity: usize,            // 有效容量（比 buffer.len() 小 1）
    head: AtomicUsize,          // 消费者索引（带版本标记）
    tail: AtomicUsize,          // 生产者索引（带版本标记）
}

// 手动实现 Send 和 Sync
// 因为 UnsafeCell 不自动实现这些 trait
unsafe impl<T: Send> Send for LockFreeMpmc<T> {}
unsafe impl<T: Send> Sync for LockFreeMpmc<T> {}

impl<T> LockFreeMpmc<T> {
    /// 创建新队列。capacity 是有效容量，内部申请 capacity + 1 个槽位
    fn new(capacity: usize) -> Self {
        let buffer_size = capacity + 1; // 多一个槽位用于区分空/满
        let mut buffer = Vec::with_capacity(buffer_size);
        for _ in 0..buffer_size {
            buffer.push(UnsafeCell::new(MaybeUninit::uninit()));
        }
        LockFreeMpmc {
            buffer: buffer.into_boxed_slice(),
            capacity,
            head: AtomicUsize::new(0), // TaggedPtr { index: 0, tag: 0 }
            tail: AtomicUsize::new(0),
        }
    }

    /// 尝试入队。成功返回 Ok(())，满则返回 Err(value)
    fn push(&self, value: T) -> Result<(), T> {
        let mut tail = self.tail.load(Ordering::Relaxed);

        loop {
            let tp = TaggedPtr { packed: tail };
            let next = tp.next_index(self.buffer.len());

            // 检查是否已满：next index == head index
            let head = self.head.load(Ordering::Acquire);
            let hp = TaggedPtr { packed: head };

            if next.index() == hp.index() {
                // 队列满
                return Err(value);
            }

            // 尝试声明 tail 的下一个位置
            match self.tail.compare_exchange_weak(
                tail,
                TaggedPtr::new(next.index(), tp.tag()).packed,
                Ordering::AcqRel,
                Ordering::Relaxed,
            ) {
                Ok(_) => {
                    // 成功声明位置，写入数据
                    let slot = self.buffer[tp.index()].get();
                    unsafe {
                        (*slot).as_mut_ptr().write(value);
                    }

                    // 更新 tail 为带下一个 tag 的版本，通知消费者数据已就绪
                    // 这实际上更新了 "status field"
                    loop {
                        let current = self.tail.load(Ordering::Relaxed);
                        if current != TaggedPtr::new(next.index(), tp.tag()).packed {
                            // 有其他线程在修改，继续等待
                            std::hint::spin_loop();
                            continue;
                        }
                        // 生产者完成写入，更新为"已完成"标记
                        let done = TaggedPtr::new(next.index(), tp.tag().wrapping_add(1));
                        if self.tail.compare_exchange_weak(
                            current,
                            done.packed,
                            Ordering::Release,
                            Ordering::Relaxed,
                        ).is_ok() {
                            break;
                        }
                    }
                    return Ok(());
                }
                Err(actual) => {
                    // 其他线程抢先了，用最新值重试
                    tail = actual;
                }
            }
        }
    }

    /// 尝试出队。成功返回 Some(value)，空则返回 None
    fn pop(&self) -> Option<T> {
        let mut head = self.head.load(Ordering::Relaxed);

        loop {
            let hp = TaggedPtr { packed: head };

            // 检查是否为空：head index == tail index
            let tail = self.tail.load(Ordering::Acquire);
            let tp = TaggedPtr { packed: tail };

            if hp.index() == tp.index() {
                return None; // 队列空
            }

            // 读取数据
            let slot = self.buffer[hp.index()].get();
            let value = unsafe { (*slot).as_ptr().read() };

            // 尝试推进 head
            let next = hp.next_index(self.buffer.len());
            let new_head = TaggedPtr::new(next.index(), hp.tag().wrapping_add(1));

            match self.head.compare_exchange_weak(
                head,
                new_head.packed,
                Ordering::Release,
                Ordering::Relaxed,
            ) {
                Ok(_) => return Some(value),
                Err(actual) => {
                    head = actual;
                    // 注意：这里如果 CAS 失败，刚才读的 value 需要丢弃或重新放入
                    // 简化实现选择丢弃（实际使用中需要考虑）
                    unsafe { std::ptr::drop_in_place(&value as *const T as *mut T); }
                }
            }
        }
    }
}

impl<T> Drop for LockFreeMpmc<T> {
    fn drop(&mut self) {
        // 清空队列中剩余的元素
        while let Some(_val) = self.pop() {
            // drop val
        }
        // buffer 本身会被 Box<[_]> 的 Drop 自动释放
    }
}

// ─── 使用示例 ───
fn main() {
    use std::thread;
    use std::sync::Arc;

    let queue = Arc::new(LockFreeMpmc::<i32>::new(64));
    let mut handles = vec![];

    // 4 个生产者
    for t in 0..4 {
        let q = queue.clone();
        handles.push(thread::spawn(move || {
            for i in 0..1000 {
                loop {
                    if q.push(t * 1000 + i).is_ok() {
                        break;
                    }
                    std::hint::spin_loop();
                }
            }
        }));
    }

    // 2 个消费者
    let total = Arc::new(AtomicUsize::new(0));
    for _ in 0..2 {
        let q = queue.clone();
        let cnt = total.clone();
        handles.push(thread::spawn(move || {
            let mut local = 0;
            while local < 2000 {
                if q.pop().is_some() {
                    local += 1;
                } else {
                    std::hint::spin_loop();
                }
            }
            cnt.fetch_add(local, Ordering::SeqCst);
        }));
    }

    for h in handles {
        h.join().unwrap();
    }
    println!("处理总数: {}", total.load(Ordering::SeqCst));
}
```

### 4.4 内存顺序选择指南

| 操作 | 用什么 Ordering | 理由 |
|------|----------------|------|
| load head（pop） | Acquire | 需要读到最新的 tail 和最新写入的数据 |
| store/CAS head | Release | 释放槽位给生产者 |
| load tail（push） | Relaxed + 后续 Acquire | 先乐观读，发现冲突再 Acquire |
| CAS tail（push） | AcqRel | 成功 = 获得槽位所有权（acquire + release 双重语义） |
| 写入 slot（push） | 不需要原子 | 只有一个线程会写入该 slot |

## 五、读优化结构

### 5.1 evmap —— 最终一致的多读并写 Map

`evmap`（eventually consistent map）专为"读极多、写极少"的场景设计：

```rust
use evmap::{ReadHandle, WriteHandle};
use std::thread;

fn main() {
    // reader 和 writer 是两个"句柄"：写入者修改自己的副本，读取者读取另一个副本
    let (book_rx, mut book_tx) = evmap::new::<String, u64>();

    // 写入线程：只有一个写入者（设计约束）
    let writer = {
        let mut tx = book_tx.clone();
        thread::spawn(move || {
            for i in 0..10_000 {
                tx.update("users.active", i);
                tx.update("users.total", i * 2);
                // refresh 将写副本与读副本"交换"
                // 读取者此后能看到这批更新
                tx.refresh();
            }
        })
    };

    // 读线程：可以有很多
    let mut readers = vec![];
    for rid in 0..8 {
        let rx = book_rx.clone();
        readers.push(thread::spawn(move || {
            let mut count = 0usize;
            loop {
                // 读取者是 read handle，访问读副本
                // 读操作无锁！
                if let Some(active) = rx.get_and("users.active", |vals| {
                    vals.iter().copied().collect::<Vec<_>>()
                }) {
                    if active.len() >= 1 {
                        count += 1;
                    }
                }
                if count >= 10_000 {
                    break;
                }
            }
            println!("读者 {} 完成读取", rid);
        }));
    }

    writer.join().unwrap();
    for r in readers {
        r.join().unwrap();
    }
}
```

**evmap 内部结构**：

```
        ┌──────────────────┐          ┌──────────────────┐
        │  读副本 (Read)    │ ←─读取者在这里读（无锁）
        │  HashMap<...>    │
        └────────┬─────────┘
                 │
           [原子交换]
                 │
        ┌────────┴─────────┐
        │  写副本 (Write)   │ ←─写入者在这里写（独占）
        │  HashMap<...>    │
        └──────────────────┘
```

**限制与适用场景**：
- 只有一个 writer（不支持并发写入）。
- reader 看到的数据可能稍微滞后（最终一致性）。
- 适合配置中心、路由表、服务发现等场景。

### 5.2 left-right —— 双缓冲读写隔离

`left-right` 与 evmap 类似但更通用：

```rust
use left_right::{LeftRight, Absorb, WriteGuard, ReadGuard};
use std::collections::HashMap;

// 实现 Absorb trait 来定义"如何将写端的修改应用到读端"
#[derive(Clone)]
struct MyMap {
    data: HashMap<String, String>,
}

impl MyMap {
    fn new() -> Self {
        MyMap { data: HashMap::new() }
    }

    fn insert(&mut self, k: &str, v: &str) {
        self.data.insert(k.to_string(), v.to_string());
    }
}

// 核心 trait：定义如何"吸收"增量修改
impl Absorb<Op> for MyMap {
    fn absorb(&mut self, operation: &mut Op, _other: &Self) {
        match operation {
            Op::Insert(k, v) => {
                self.data.insert(k.clone(), v.clone());
            }
            Op::Remove(k) => {
                self.data.remove(k);
            }
        }
    }
}

// 定义操作枚举
#[derive(Clone)]
enum Op {
    Insert(String, String),
    Remove(String),
}

fn main() {
    // left-right 返回 (WriteHandle, ReadHandle)
    let (mut writer, reader) = LeftRight::<MyMap, Op>::new();
    //                                    ^^^^^^  ^^^^
    //                                    数据类型   操作类型

    // 写入线程：追加操作（不直接修改读端）
    let mut w = writer.clone();
    std::thread::spawn(move || {
        w.append(Op::Insert("name".into(), "Rust".into()));
        w.append(Op::Insert("version".into(), "1.85".into()));
        // publish 会触发 absorb 将操作应用到读端
        w.publish();
    }).join().unwrap();

    // 读取线程：永远无锁
    reader.enter().map(|guard: ReadGuard<MyMap>| {
        println!("map size: {}", guard.data.len());
        // guard 在此期间阻止写入者交换（但不会阻塞读）
    });
}
```

**left-right 与 evmap 对比**：

| 维度 | evmap | left-right |
|------|-------|------------|
| 写入策略 | 全量替换 | 增量操作（Absorb trait） |
| 交换频率 | 手动 refresh() | 手动 publish() |
| 内存开销 | 两份完整副本 | 一份 + 操作日志 |
| 适用 | 小数据集 + 批量更新 | 大数据集 + 频繁小更新 |

## 六、无锁 HashMap 对比

### 6.1 三大并发 HashMap

```rust
// ─── DashMap: 分段锁（最常用） ───
// 原理：将 HashMap 分成 N 个 shard（通常为 cpu_count * 4），
//       每个 shard 有独立的 RwLock。不同 key 大概率落在不同 shard。
use dashmap::DashMap;

let map = DashMap::<String, u64>::new();
map.insert("alice".to_string(), 100);
map.insert("bob".to_string(), 200);

// 并发读：如果 key 落在不同 shard，完全无锁
// 并发写：如果 key 落在不同 shard，无竞争
// 同一 shard 内的读写：受 RwLock 保护

// 获取值（返回引用，持有读锁直到 ref 被 drop）
if let Some(val) = map.get("alice") {
    println!("alice: {}", *val);
}

// 条件更新
map.entry("alice".to_string())
    .and_modify(|v| *v += 1)
    .or_insert(0);

// ─── flurry: Java ConcurrentHashMap 的 Rust 移植 ───
// 原理：基于 hash + CAS + 红黑树。读操作完全无锁！
use flurry::HashMap as FlurryMap;

let map = FlurryMap::<String, u64>::new();
let guard = flurry::epoch::pin(); // flurry 基于 crossbeam-epoch
map.insert("alice".to_string(), 100, &guard);
map.insert("bob".to_string(), 200, &guard);

// 读——无需锁，通过 epoch guard 保证安全
if let Some(val) = map.get("alice", &guard) {
    println!("alice: {}", *val);
}

// ─── scc: 可伸缩并发容器 ───
// 原理：类似 Java ConcurrentHashMap 的分段设计，但做了更多优化
use scc::HashMap as SccMap;

let map = SccMap::<String, u64>::new();
map.insert("alice".to_string(), 100);
map.insert("bob".to_string(), 200);

// upsert: 原子地插入或更新
map.upsert("alice".to_string(), || 1, |_, v| *v += 1);

// 并行遍历
map.for_each(|k, v| {
    println!("{} -> {}", k, v);
});
```

### 6.2 三者对比

```
┌────────────┬─────────────┬─────────────┬─────────────┐
│ 特性       │ DashMap     │ flurry      │ scc         │
├────────────┼─────────────┼─────────────┼─────────────┤
│ 读并发     │ RwLock 分段 │ 完全无锁    │ 极低竞争    │
│ 写并发     │ RwLock 分段 │ CAS 循环    │ 分段 + CAS  │
│ 内存顺序   │ 安全（编译  │ 需要 Guard  │ 安全        │
│            │  器保证）   │ 管理        │             │
│ 扩容       │ 单 shard 内 │ 渐进式      │ 渐进式      │
│ 性能热点   │ 热点 key 锁 │ CAS 冲突    │ shard 内    │
│ 学习曲线   │ ★☆☆ 最简单  │ ★★★ 最陡    │ ★★☆ 中等   │
│ API 风格   │ 像 HashMap  │ 像 Map API  │ 函数式      │
│ 适用场景   │ 通用        │ 极高读比例  │ 大容量 + 写 │
└────────────┴─────────────┴─────────────┴─────────────┘
```

**选择建议**：
- **默认首选 DashMap**：API 最友好，足够快，没有 hidden cost。
- **极致读性能用 flurry**：但需要管理 Guard，且写操作是 CAS 自旋。
- **大容量 + 需要批量操作用 scc**：其 `for_each` 和 `retain` 等批量 API 非常高效。

## 七、Loom：系统性并发测试

### 7.1 Loom 的原理

`loom` 是一个并发 bug 检测工具，它的原理类似 CDSChecker 或 Java 的 jcstress：

```
                   ┌──────────────────────┐
                   │  Loom 执行引擎        │
                   └──────────┬───────────┘
                              │
         ┌────────────────────┼────────────────────┐
         │                    │                    │
    ┌────▼─────┐      ┌──────▼──────┐     ┌───────▼──────┐
    │ 拦截 atom │      │ 管理线程    │      │ 穷举调度    │
    │ 操作      │      │ 调度        │      │ 排列        │
    └──────────┘      └─────────────┘     └──────────────┘
         │                                        │
    ┌────▼─────┐                           ┌─────▼──────┐
    │ 记录每   │                           │ 尝试所有   │
    │ 次内存   │                           │ 线程交错   │
    │ 访问     │                           │ 顺序       │
    └──────────┘                           └────────────┘
```

关键点：Loom **替换**了标准库的 `std::sync::atomic`，用自己的实现来记录每一次原子操作，然后系统性地排列所有可能的线程交错执行顺序。这比随机测试（如 `loom` 的 `fuzz` 模式）更系统，但也更慢。

### 7.2 用 Loom 测试 Lock-Free 栈

```rust
// 在 Cargo.toml 中：
// [dev-dependencies]
// loom = "0.7"

#[cfg(test)]
mod tests {
    use loom::sync::atomic::AtomicUsize;
    use loom::sync::Arc;
    use loom::thread;

    // 被测对象：一个"有 bug"的 lock-free 栈
    #[derive(Clone)]
    struct BuggyStack {
        head: Arc<AtomicUsize>, // 简化：只存值，不存节点指针
    }

    impl BuggyStack {
        fn new() -> Self {
            BuggyStack { head: Arc::new(AtomicUsize::new(usize::MAX)) }
        }

        fn push(&self, value: usize) {
            let mut head = self.head.load(loom::sync::atomic::Ordering::Relaxed);
            loop {
                let next = value;
                // ❌ BUG: Compare-exchange 的 success ordering 应该是 AcqRel
                match self.head.compare_exchange(
                    head,
                    next,
                    loom::sync::atomic::Ordering::Release,
                    loom::sync::atomic::Ordering::Relaxed,
                ) {
                    Ok(_) => break,
                    Err(actual) => head = actual,
                }
            }
        }

        fn pop(&self) -> Option<usize> {
            let mut head = self.head.load(loom::sync::atomic::Ordering::Acquire);
            loop {
                if head == usize::MAX {
                    return None;
                }
                let next = usize::MAX; // 简化：栈实际上只存一个值
                match self.head.compare_exchange(
                    head,
                    next,
                    loom::sync::atomic::Ordering::Release,
                    loom::sync::atomic::Ordering::Relaxed,
                ) {
                    Ok(_) => return Some(head),
                    Err(actual) => head = actual,
                }
            }
        }
    }

    #[test]
    fn test_concurrent_push_pop() {
        // loom::model 会穷举所有可能的线程交错
        loom::model(|| {
            let stack = Arc::new(BuggyStack::new());

            let s1 = stack.clone();
            let t1 = thread::spawn(move || {
                s1.push(42);
            });

            let s2 = stack.clone();
            let t2 = thread::spawn(move || {
                // 同时 push 和 pop 会导致什么？
                let val = s2.pop();
            });

            t1.join().unwrap();
            t2.join().unwrap();
        });
        // 如果代码有 bug（如上面的 内存顺序问题），
        // loom 会报告类似于：
        //
        // thread 'test_concurrent_push_pop' panicked at 'assertion failed:
        //   expected Ok(_), got Err(...)'
    }
}
```

### 7.3 Loom 模型的局限性

| 能检测 | 不能检测 |
|-------|---------|
| 原子操作顺序错误 | 数据竞争（编译器会报错） |
| 死锁（所有线程永远等待） | 活锁（线程在忙但无进展） |
| ABA 问题（配合模型检查） | 真实硬件内存模型的复杂性 |
| 丢失的唤醒/通知 | 外部 I/O 相关的并发 bug |

**实用建议**：
1. 对每个并发数据结构，写一个 `loom::model` 测试。
2. 线程数控制在 2-3 个，操作数控制在 2-3 次（状态空间随线程数和操作数指数级爆炸）。
3. 持续运行这些测试（CI 中积分），它们很慢但能捕获最微妙的 bug。

## 八、实战案例：高性能日志聚合器

将本章知识整合成一个实际的日志聚合器：

```rust
use crossbeam_queue::SegQueue;
use crossbeam_channel::{bounded, Receiver};
use dashmap::DashMap;
use std::sync::Arc;
use std::thread;
use std::time::{Duration, Instant};

/// 日志条目
#[derive(Debug, Clone)]
struct LogEntry {
    timestamp: u64,
    level: String,
    service: String,
    message: String,
}

/// 统计结果
#[derive(Debug, Default)]
struct Stats {
    count: u64,
    sum_latency: u64, // 用于计算平均延迟
}

fn main() {
    // ① 用 SegQueue 接收日志（多生产者、无锁）
    let log_queue: Arc<SegQueue<LogEntry>> = Arc::new(SegQueue::new());

    // ② 用 DashMap 存储聚合统计（多消费者、分段锁）
    let stats_map: Arc<DashMap<String, Stats>> = Arc::new(DashMap::new());

    // ③ 用 crossbeam channel 发送控制信号（select 能力）
    let (ctrl_tx, ctrl_rx) = bounded::<&str>(1);

    // ── 模拟 8 个日志生产者 ──
    let mut producers = vec![];
    for service_id in 0..8 {
        let q = log_queue.clone();
        producers.push(thread::spawn(move || {
            let service_name = format!("service-{}", service_id);
            for i in 0..100_000 {
                q.push(LogEntry {
                    timestamp: 1000 + i,
                    level: if i % 100 == 0 { "ERROR" } else { "INFO" }.into(),
                    service: service_name.clone(),
                    message: format!("event-{}", i),
                });
            }
        }));
    }

    // ── 消费者：从 SegQueue pop，聚合到 DashMap ──
    let consumer = {
        let q = log_queue.clone();
        let stats = stats_map.clone();
        let ctrl = ctrl_rx;
        let total_expected = 8 * 100_000u64;
        let consumed = Arc::new(std::sync::atomic::AtomicU64::new(0));
        let c = consumed.clone();
        thread::spawn(move || {
            let start = Instant::now();
            loop {
                // 非阻塞尝试 pop
                match q.pop() {
                    Some(entry) => {
                        // 按 service + level 聚合
                        let key = format!("{}/{}", entry.service, entry.level);
                        stats.entry(key)
                            .and_modify(|s| s.count += 1)
                            .or_insert(Stats { count: 1, sum_latency: 0 });
                        c.fetch_add(1, std::sync::atomic::Ordering::Relaxed);
                    }
                    None => {
                        // 队列空，检查是否应该退出
                        if c.load(std::sync::atomic::Ordering::SeqCst) >= total_expected {
                            break;
                        }
                        std::hint::spin_loop();
                    }
                }
            }
            println!("消费完成，耗时: {:?}", start.elapsed());
        })
    };

    // ── 等待全部完成 ──
    for p in producers {
        p.join().unwrap();
    }
    consumer.join().unwrap();

    // ── 输出统计 ──
    println!("\n=== 日志聚合统计 ===");
    for entry in stats_map.iter() {
        println!(
            "  {}: {} 条日志",
            entry.key(),
            entry.value().count
        );
    }
}
```

1. **Crossbeam 生态**：`SegQueue`（无锁 MPMC）、`ArrayQueue`（有界）、`crossbeam_channel`（select!）、`crossbeam_epoch`（GC for lock-free）分别解决什么问题？各自的适用场景是什么？
2. **Epoch-Based GC**：`pin()`、`Guard`、`defer_destroy()` 之间的关系是什么？为什么 `Guard` 存活期间可以安全解引用一个已被"退役"的指针？
3. **ABA 问题**：什么是 ABA 问题？Tagged Pointer 如何解决它？在你的 MPMC 实现中，TAG 放在哪个字段里？
4. **CAS 循环**：`compare_exchange_weak` 和 `compare_exchange` 的区别是什么？为什么 lock-free 结构中几乎总是用 `_weak`？
5. **读优化结构**：`evmap` 和 `left-right` 的核心区别是什么？为什么它们能做到"读路径永远不阻塞"？
6. **并发 HashMap 选型**：你的服务读 QPS 是写的 100 倍——选 DashMap 还是 flurry？你的服务有大量 batch insert（10 万条/批次）——选哪个？
