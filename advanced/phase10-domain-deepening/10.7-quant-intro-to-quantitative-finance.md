# 10.7 量化金融入门 —— 从零理解量化交易与 Rust 的角色

> **[Track B: 量化金融]** 本课是 Track B 的起点——我们不会假设你懂任何金融术语。你会了解量化金融的全景版图、核心玩家、职业角色，以及 Rust 在这个领域为何从"边缘尝试"变成了"主流选择"。同时，你会动手写第一个量化交易程序：一个简化但完整的限价订单簿（Limit Order Book）。

**领域背景：** 量化金融（Quantitative Finance）是用数学模型和计算机程序来做交易决策的领域。传统交易依赖交易员的直觉和经验——"我感觉市场要涨"。量化交易依赖数据和模型——"过去 10 年的数据显示，当 A 发生且 B 同时出现时，有 67% 的概率 C 会在接下来的 5 分钟内发生"。你的工作不是"预测市场"，而是找到统计上显著、经济上可行的规律，并用最可靠的代码执行它们。这个领域对 Rust 的接纳速度远超预期——因为当你的 bug 代价是数百万美元而非 500 错误页面时，Rust 的内存安全保证不再是一种"偏好"，而是一种"必需"。

## 3. 为什么 Rust 适合量化金融？

### 3.1 量化金融对系统的要求

```
量化交易系统的核心指标（按重要性排序）：

1. 确定性延迟（Deterministic Latency）
   → GC 停顿 = 不可接受。每次 GC pause 都是一次交易机会的损失。
   → Rust 的 ownership 系统 = 编译期确定的内存生命周期 = 零 GC 停顿。

2. 内存安全（Memory Safety）
   → segfault 在生产环境 = 数百万美元的损失。
   → 传统 C++ 方案：大量运行时检查，拖慢速度。
   → Rust：编译期保证，运行时零开销。

3. 类型安全（Type Safety）
   → 价格当成了数量？这是一个经典的 fat-finger 错误。
   → Rust 的 newtype 模式可以用类型系统消除这类错误：
     Price(u64) 和 Quantity(u64) 不能互相赋值——编译器帮你检查。

4. 并发安全（Concurrency Safety）
   → 市场数据处理管道天然多线程：UDP 收包、解析、订单簿更新、风控检查。
   → Rust 的 Send + Sync trait：编译期验证并发安全性。
   → C++：thread sanitizer 只能在运行时发现，且覆盖率依赖测试用例。

5. 零成本抽象（Zero-Cost Abstraction）
   → 你可以用 trait、泛型、enum 写出清晰的代码，
     编译后和手写的 C 代码一样快。
   → 不用在"可维护性"和"性能"之间做二选一。
```

### 3.2 业界趋势：Jane Street 的故事

```
Jane Street 是一家以 OCaml 闻名的做市商。
他们的 OCaml 技术栈曾经是业内独一无二的存在。

2015 → 核心系统全部 OCaml，每年 OCaml 编译器贡献几百万美元
2020 → 开始公开讨论 Rust 的潜力
       → OCaml 的 GC 是硬伤（虽然 tuned 到极致）
       → 高性能场景需要更多的内存布局控制
2022 → 内部开始用 Rust 构建新的交易所连接模块
       → 特别是 crypto 相关业务，性能要求更高
2024 → 公开招聘 Rust 工程师，目标是把性能敏感的路径迁移到 Rust
       → 保留 OCaml 做策略表达（DSL 层），Rust 做执行层

这个路径反映了整个行业的趋势：
  "策略层用生产率高的语言（Python/OCaml），
   执行层用性能好的语言（Rust/C++），
   Rust 正在从 C++ 手中抢走执行层。"
```

### 3.3 Rust vs C++ 在量化金融中的对比

| 维度 | C++ | Rust | 对量化金融的意义 |
|------|-----|------|------------------|
| 学习曲线 | 陡峭（20年历史包袱） | 陡峭（但设计一致） | 招新人更容易上手 Rust |

## 4. 量化金融核心词汇：从术语到 Rust 表示

### 4.1 市场数据术语

在写代码之前，你需要理解这些基础概念。每个概念配有 Rust 类型定义，让你看到"领域知识如何映射为类型安全的数据结构"。

```rust
// === 核心市场数据 ===

/// 价格：用定点小数表示，避免浮点精度问题
/// 在量化金融中，价格绝不能用 f64——0.1 + 0.2 的舍入误差
/// 在交易中就是"差了一分钱"
#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord)]
struct Price(i64);  // 隐式单位：分（cent）或 1/10000 美分

/// 数量：股票用整数（股），外汇/crypto 用定点小数
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
struct Quantity(u64);

/// Bid（买价）：买方愿意支付的最高价格
/// Ask/Offer（卖价）：卖方愿意接受的最低价格
/// Spread（价差）= Ask - Bid，做市商的盈利空间
///
/// 例：Bid = 100.00, Ask = 100.05, Spread = 0.05
///     → 你买入立刻卖出，亏 0.05（做市商赚走了）
#[derive(Debug, Clone)]
struct Quote {
    instrument_id: String,    // 品种标识符（如 "AAPL", "BTC-USD"）
    bid_price: Price,         // 买一价
    bid_size: Quantity,       // 买一量
    ask_price: Price,         // 卖一价
    ask_size: Quantity,       // 卖一量
    timestamp_ns: u64,        // 纳秒级时间戳（量化金融的标配精度）
}

/// 流动性：市场中可以快速成交的订单容量
/// 高流动性 → spread 小 → 交易成本低
/// 低流动性 → spread 大 → 适合做市商赚钱，但风险也大
#[derive(Debug, Clone)]
struct LiquidityMetrics {
    instrument_id: String,
    /// 深度：在 bid 价以下 N% 的累积订单量
    depth_bid_1pct: Quantity,
    /// 成交量加权平均价（Volume-Weighted Average Price）
    vwap: f64,  // VWAP 常用于衡量执行质量
    /// 过去 N 秒内的成交总量
    volume_1min: Quantity,
    /// 挂单队列失衡度（bid 总量 / ask 总量）
    book_imbalance: f64,
}

/// L1 / L2 / L3 市场数据
///
/// L1 (Level 1)：只展示最优买卖价和最近成交
///   → 用途：简单的价差监控、VWAP 计算
///   → 数据量小，延迟低
///
/// L2 (Level 2)：展示各价位的买卖深度
///   → 用途：订单簿分析、流动性建模、执行算法
///   → 数据量大，每个价位变化都会推送
///
/// L3 (Level 3)：展示每一笔订单（含 ID）
///   → 用途：精确的订单流分析、iceberg 订单检测
///   → 数据量极大，通常只有交易所内部使用
enum MarketDataLevel {
    L1,
    L2,
    L3,
}
```

### 4.2 交易执行术语

```rust
/// OHLCV：K 线（蜡烛图）数据
/// Open（开盘价）、High（最高价）、Low（最低价）、Close（收盘价）、Volume（成交量）
/// 这是所有技术分析的基础数据结构
#[derive(Debug, Clone)]
struct OHLCV {
    open: Price,
    high: Price,
    low: Price,
    close: Price,
    volume: Quantity,
    timestamp_ns: u64,
}

/// 滑点（Slippage）：预期成交价格 vs 实际成交价格的差异
/// 例：你挂 100.00 买入，但因为市场波动，实际成交在 100.03
///     → Slippage = 0.03
/// 原因：流动性不足、网络延迟、市场剧烈波动
#[derive(Debug, Clone)]
struct Slippage {
    expected_price: Price,
    actual_price: Price,
    /// 通常用 basis points (bps) 表示
    /// 1 bp = 0.01% = 万分之一
    slippage_bps: i64,
}

/// 成交率（Fill Ratio）：已成交数量 / 已发送数量
/// 低成交率 = 你的定价太差，没人愿意跟你交易
/// 高成交率但常被 adverse selection = 你的定价太好，被别人"薅羊毛"
#[derive(Debug, Clone)]
struct FillRatio {
    total_sent: u64,      // 总发送数量
    total_filled: u64,    // 总成交数量
    ratio: f64,           // total_filled / total_sent
}

/// TWAP（Time-Weighted Average Price）：时间加权平均价
/// 常用于大单拆分执行——避免一次性吃掉流动性
/// 算法：把一段时间分成 N 个时间片，每个时间片执行 1/N 的量
#[derive(Debug, Clone)]
struct Twap {
    period_secs: u64,      // 总时间段（秒）
    slices: u64,           // 时间片数量
    quantity_per_slice: Quantity,  // 每片执行量
}
```

### 4.3 订单簿术语

```rust
/// 订单簿（Order Book）：某个品种所有未成交订单的集合
/// 限价订单簿有两个方向：
///   Bid 侧：买方限价单，价格从高到低排列
///   Ask 侧：卖方限价单，价格从低到高排列
///
/// 订单簿是量化交易系统的核心数据结构——
/// 几乎每一笔交易决策都需要查询订单簿状态。

/// 订单方向
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum Side {
    Buy,   // 买入
    Sell,  // 卖出
}

/// 订单状态
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum OrderStatus {
    New,              // 新创建，尚未进入订单簿
    Open,             // 在订单簿中等待成交
    PartiallyFilled,  // 部分成交
    Filled,           // 完全成交
    Cancelled,        // 已取消
}

/// 限价单：以指定价格买入/卖出指定数量
/// 这是市场中最基本的订单类型
#[derive(Debug, Clone)]
struct LimitOrder {
    order_id: u64,
    side: Side,
    price: Price,
    quantity: Quantity,
    filled_quantity: Quantity,  // 已成交数量
    status: OrderStatus,
    timestamp_ns: u64,
}
```

## 5. 第一个量化程序：简化限价订单簿

### 5.1 设计思路

限价订单簿（Limit Order Book, LOB）是量化金融的 "Hello World"。每一个量化开发者都写过（或应该写过）一个 LOB。

我们的简化版本：
- 使用 `BTreeMap<Price, Vec<Order>>` 管理价格级别
- Bid 侧按价格从高到低排列（BTreeMap 默认升序，我们反转迭代）
- Ask 侧按价格从低到高排列
- 支持三个操作：add_order、cancel_order、match_orders
- 遵循价格-时间优先原则：价格优先，同价位的先到先匹配

### 5.2 完整实现

```rust
use std::collections::BTreeMap;

/// 价格：用 i64 表示，实际值为 Price/10000
/// 这样可以精确表示到 0.0001
#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord)]
struct Price(i64);

// 实现 Display 方便调试
impl std::fmt::Display for Price {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        write!(f, "{:.4}", self.0 as f64 / 10000.0)
    }
}

/// 数量：用 u64 表示，实际值为 Quantity/10000
#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord)]
struct Quantity(u64);

impl std::fmt::Display for Quantity {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        write!(f, "{:.4}", self.0 as f64 / 10000.0)
    }
}

/// 订单方向
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum Side { Buy, Sell }

/// 单笔订单
#[derive(Debug, Clone)]
struct Order {
    id: u64,           // 订单 ID（全局递增）
    side: Side,        // 买卖方向
    price: Price,      // 限价
    quantity: Quantity,// 剩余未成交数量
    timestamp: u64,    // 到达时间（用于时间优先排序）
}

impl Order {
    fn new(id: u64, side: Side, price: Price, quantity: Quantity, timestamp: u64) -> Self {
        Self { id, side, price, quantity, timestamp }
    }
}

/// 简化版限价订单簿
///
/// 数据结构：BTreeMap<Price, Vec<Order>>
/// - Bid 侧（买单）：key 为价格，value 为该价位的订单队列
/// - Ask 侧（卖单）：key 为价格，value 为该价位的订单队列
///
/// BTreeMap 的优点：
/// - 自动按 key 排序，查找最优价 O(log N)
/// - 遍历全部级别 O(N)
///
/// BTreeMap 的缺点（在生产环境需要优化）：
/// - 内存不连续，cache miss 较多
/// - 不适合超高频率（>100K ops/sec）——建议改用数组+索引
#[derive(Debug)]
struct OrderBook {
    instrument: String,            // 品种标识
    bids: BTreeMap<Price, Vec<Order>>,  // 买单簿（价格从高到低）
    asks: BTreeMap<Price, Vec<Order>>,  // 卖单簿（价格从低到高）
    next_order_id: u64,            // 订单 ID 生成器
    next_timestamp: u64,           // 模拟的时间戳
}

impl OrderBook {
    /// 创建新的订单簿
    fn new(instrument: &str) -> Self {
        Self {
            instrument: instrument.to_string(),
            bids: BTreeMap::new(),
            asks: BTreeMap::new(),
            next_order_id: 1,
            next_timestamp: 0,
        }
    }

    /// 添加限价单
    /// 订单到达后进入对应价位的队列末尾
    fn add_order(&mut self, side: Side, price: Price, quantity: Quantity) -> u64 {
        let id = self.next_order_id;
        self.next_order_id += 1;
        self.next_timestamp += 1;

        let order = Order::new(id, side, price, quantity, self.next_timestamp);

        let book_side = match side {
            Side::Buy => &mut self.bids,
            Side::Sell => &mut self.asks,
        };

        // 如果该价位已存在，追加到队列末尾
        // 如果该价位不存在，创建新的队列
        book_side.entry(price)
            .or_insert_with(Vec::new)
            .push(order);

        println!("[ADD]    Order#{} {} {}@{}", id,
            match side { Side::Buy => "BUY ", Side::Sell => "SELL" },
            quantity, price);

        id
    }

    /// 取消订单
    /// 从对应价位队列中移除指定 ID 的订单
    fn cancel_order(&mut self, order_id: u64) -> bool {
        // 检查买单簿
        for (price, orders) in self.bids.iter_mut() {
            if let Some(pos) = orders.iter().position(|o| o.id == order_id) {
                let order = orders.remove(pos);
                println!("[CANCEL] Order#{} {} {}@{}",
                    order.id,
                    match order.side { Side::Buy => "BUY ", Side::Sell => "SELL" },
                    order.quantity, order.price);
                // 如果该价位队列已空，移除价位键
                if orders.is_empty() {
                    // 注意：这里需要记下该 price，然后移除
                    // 但在迭代中不能直接移除，我们用个标记
                }
                return true;
            }
        }

        // 检查卖单簿
        for (price, orders) in self.asks.iter_mut() {
            if let Some(pos) = orders.iter().position(|o| o.id == order_id) {
                let order = orders.remove(pos);
                println!("[CANCEL] Order#{} {} {}@{}",
                    order.id,
                    match order.side { Side::Buy => "BUY ", Side::Sell => "SELL" },
                    order.quantity, order.price);
                return true;
            }
        }

        println!("[CANCEL] Order#{} not found", order_id);
        false
    }

    /// 撮合引擎：尝试匹配买卖单
    ///
    /// 撮合算法（价格-时间优先）：
    /// 1. 获取最优买价（bids 的最高价）和最优卖价（asks 的最低价）
    /// 2. 如果 best_bid >= best_ask，可以撮合
    ///    - 否则不能撮合（没有重叠）
    /// 3. 从先到达的订单开始匹配（时间优先）
    /// 4. 对交叉的最优价进行撮合
    /// 5. 如果一方订单被完全吃掉，继续下一笔
    fn match_orders(&mut self) -> Vec<Trade> {
        let mut trades = Vec::new();

        loop {
            // 获取最优买卖价
            let best_bid = match self.bids.keys().next_back() {
                Some(price) => *price,
                None => break, // 没有买单，无法撮合
            };
            let best_ask = match self.asks.keys().next() {
                Some(price) => *price,
                None => break, // 没有卖单，无法撮合
            };

            // 撮合条件：买价 >= 卖价
            if best_bid.0 < best_ask.0 {
                break; // 没有交叉，撮合结束
            }

            // 取出一方最优价位的队列（使用 time-priority 中的最早订单）
            // 成交价使用"被动方价格"（即先挂单的那一方）
            let (bid_ts, ask_ts) = {
                let bid_q = self.bids.get(&best_bid).unwrap();
                let ask_q = self.asks.get(&best_ask).unwrap();
                (bid_q.first().map(|o| o.timestamp).unwrap_or(0),
                 ask_q.first().map(|o| o.timestamp).unwrap_or(0))
            };

            // 确定成交价：被动方（先到者）的价格
            let trade_price = if bid_ts <= ask_ts {
                best_bid  // 买单先到，以买价成交
            } else {
                best_ask  // 卖单先到，以卖价成交
            };

            // 撮合操作
            let bid_order = self.bids.get_mut(&best_bid).unwrap().first_mut().unwrap();
            let ask_order = self.asks.get_mut(&best_ask).unwrap().first_mut().unwrap();

            let trade_qty = std::cmp::min(bid_order.quantity.0, ask_order.quantity.0);
            let trade_qty = Quantity(trade_qty);

            // 扣除数量
            bid_order.quantity.0 -= trade_qty.0;
            ask_order.quantity.0 -= trade_qty.0;

            trades.push(Trade {
                price: trade_price,
                quantity: trade_qty,
                bid_order_id: bid_order.id,
                ask_order_id: ask_order.id,
            });

            println!("[TRADE] {}@{} (Bid#{}, Ask#{})",
                trade_qty, trade_price, bid_order.id, ask_order.id);

            // 清理已完全成交的订单
            // 手动清理（简化版——生产环境需要用更好的数据结构）
        }

        // 清理所有已清空数量的订单
        self.cleanup_empty_orders();

        trades
    }

    /// 清理数量为 0 的订单
    fn cleanup_empty_orders(&mut self) {
        // 清理 bids
        let empty_bid_prices: Vec<Price> = self.bids.iter()
            .filter_map(|(price, orders)| {
                if orders.iter().all(|o| o.quantity.0 == 0) {
                    Some(*price)
                } else {
                    None
                }
            })
            .collect();
        for price in empty_bid_prices {
            self.bids.remove(&price);
        }

        // 手动移除已清空的订单（简化逻辑）
        for orders in self.bids.values_mut() {
            orders.retain(|o| o.quantity.0 > 0);
        }

        // 清理 asks
        let empty_ask_prices: Vec<Price> = self.asks.iter()
            .filter_map(|(price, orders)| {
                if orders.iter().all(|o| o.quantity.0 == 0) {
                    Some(*price)
                } else {
                    None
                }
            })
            .collect();
        for price in empty_ask_prices {
            self.asks.remove(&price);
        }

        for orders in self.asks.values_mut() {
            orders.retain(|o| o.quantity.0 > 0);
        }
    }

    /// 打印当前订单簿状态
    fn display(&self) {
        println!("\n┌──────────────────────────────────────────────┐");
        println!("│  Order Book: {:^30} │", self.instrument);
        println!("├──────────────────────────────────────────────┤");

        // 卖单：从高到低打印（逻辑上 ask 从低到高，但显示从高到低更直观）
        println!("│  ASK (Sell)  ─────────────────────────────────│");
        for (price, orders) in self.asks.iter().rev() {
            let total_qty: u64 = orders.iter().map(|o| o.quantity.0).sum();
            let count = orders.len();
            println!("│  {}  x {}  ({} orders)                 │",
                price, Quantity(total_qty), count);
        }

        println!("│  ─────────── SPREAD ────────────────           │");
        let best_bid = self.bids.keys().next_back().copied();
        let best_ask = self.asks.keys().next().copied();
        if let (Some(bid), Some(ask)) = (best_bid, best_ask) {
            let spread = Price(ask.0 - bid.0);
            println!("│  Spread: {} / {} = {}               │",
                spread, bid, spread.0 as f64 / bid.0 as f64 * 100.0);
        }

        // 买单：从高到低打印
        println!("│  BID (Buy)   ─────────────────────────────────│");
        for (price, orders) in self.bids.iter().rev() {
            let total_qty: u64 = orders.iter().map(|o| o.quantity.0).sum();
            let count = orders.len();
            println!("│  {}  x {}  ({} orders)                 │",
                price, Quantity(total_qty), count);
        }

        println!("└──────────────────────────────────────────────┘\n");
    }
}

/// 成交记录
#[derive(Debug, Clone)]
struct Trade {
    price: Price,
    quantity: Quantity,
    bid_order_id: u64,
    ask_order_id: u64,
}

// === 测试用例 ===
fn main() {
    let mut ob = OrderBook::new("AAPL");

    // 场景：构建一个简单的订单簿并观察撮合过程

    // 1. 添加一些买单
    //    有人愿意以 150.00 和 149.50 买入 AAPL
    ob.add_order(Side::Buy, Price(150_0000), Quantity(100_0000));  // 150.00 x 100
    ob.add_order(Side::Buy, Price(149_5000), Quantity(50_0000));   // 149.50 x 50

    // 2. 添加一些卖单
    //    有人愿意以 151.00 和 150.50 卖出 AAPL
    ob.add_order(Side::Sell, Price(151_0000), Quantity(80_0000));  // 151.00 x 80
    ob.add_order(Side::Sell, Price(150_5000), Quantity(60_0000));  // 150.50 x 60

    ob.display();

    // 3. 此时 Best Bid = 150.00, Best Ask = 150.50
    //    没有交叉（150.00 < 150.50），不能撮合
    let trades = ob.match_orders();
    println!("match_orders result: {} trades", trades.len()); // 预期 0

    // 4. 新增一个卖单，价格低于最优买价
    //    有人愿意以 149.80 卖出——这会和 150.00 的买单交叉！
    ob.add_order(Side::Sell, Price(149_8000), Quantity(80_0000));  // 149.80 x 80

    ob.display();

    // 5. 撮合！
    //    Best Bid = 150.00, Best Ask = 149.80 → 交叉，可以撮合
    let trades = ob.match_orders();
    println!("\n=== 撮合结果 ===");
    for t in &trades {
        println!("  Trade: {} @ {} (Bid#{}, Ask#{})",
            t.quantity, t.price, t.bid_order_id, t.ask_order_id);
    }

    ob.display();

    // 这个简化的订单簿展示了量化金融最核心的数据结构
    // 在后续课程中，我们会逐步完善它：
    // - 10.8：用真实的交易所数据填充订单簿
    // - 10.9：构建完整的撮合引擎（多种订单类型）
    // - 10.11：构建回测框架，用订单簿模拟交易
}
```

### 6.2 Rust 在量化技术栈中的切入点（深度标注）

```
每个量化公司对 Rust 的使用深度不同。从浅到深：

浅层使用（Low Commitment）：
  ├── 数据处理加速：用 Rust 写热点分析代码，Python 调用 via PyO3
  ├── 网络层重写：用 Rust 替换老旧的 C++ FIX 网关
  └── 工具链辅助：用 Rust 写 CLI 工具、部署脚本

中层使用（Moderate Commitment）：
  ├── 回测执行引擎：Rust 运行策略逻辑，Python 做分析和可视化
  ├── 实时风控服务：Rust 的确定性延迟保证风控检查的一致性
  └── 数据管线：Rust + Arrow/Parquet 做高效的 ETL

深层使用（Full Commitment）：
  ├── 撮合引擎：全 Rust 实现，零 allocation 热路径
  ├── FPGA 通信层：Rust 做软硬协同（控制 FPGA 的 Rust 驱动）
  └── 全栈量化系统：策略研究→回测→执行→监控，全部 Rust
```
