## 1. 从 Go 到 Rust：Discord Read States 迁移实录

### 1.1 问题背景

Discord 的 Read States 服务负责追踪每个用户对每条消息的"已读/未读"状态。这是一个读密集、写密集、并且延迟敏感的 OLTP 工作负载。在 2020 年前，该服务使用 Go 编写，核心数据结构是一个巨大的哈希表，将 `(channel_id, user_id)` 映射到最后一个已读消息 ID。

每次用户在频道中读到消息时，前端会发送一个 HTTP 请求更新 read state；每次用户切换频道时，也需要查询当前频道的 read state。这意味着 QPS 极高且延迟要求严格。

2020 年，Discord 团队发现该服务每隔几分钟就会出现一次**明显的延迟尖峰（latency spike）**。排查后确认原因是 Go GC（垃圾回收）导致的 stop-the-world 暂停——即使 Go 1.12+ 的 GC 已经大幅缩短了暂停时间，在内存压力大的场景下仍然会触发 ~10ms 级别的停顿。对于实时消息应用来说，这是不可接受的。

### 1.2 技术决策：用 Rust 重写

Discord 团队没有优化 Go 代码，而是直接选择用 Rust 重写。这在当时是一个冒险的决定——Rust 在 2020 年的成熟度远不如今天。但他们的判断很明确：

> "GC 消除本身就能带来 10x 的改善，任何 Go 层面的微优化都只是对症状的缓解，不去治疗病根。"

这个决策体现了一个重要的优化哲学：**先找到最大的瓶颈，再决定用什么层级的手段来解决它。** 瓶颈可能在一行代码里（micro），也可能在语言运行时里（architectural）。

### 1.3 关键优化点

重写后的 Rust 版本做了以下关键设计选择：

```rust
// Discord Read States 核心数据结构（简化示意）
use std::collections::HashMap;

/// 核心抽象：每个 (channel, user) 对追踪一条 read state
/// 注意：使用 u64 而非 String 作为 key 减少了哈希开销
type ChannelId = u64;
type UserId = u64;

/// Read State 服务的数据模型
struct ReadStateStore {
    // 主索引: (channel_id, user_id) -> 最后已读消息 ID
    // 在 Rust 版本中，没有 GC 暂停，内存布局完全可控
    states: HashMap<(ChannelId, UserId), ReadStateEntry>,
}

#[derive(Clone, Copy)] // Copy 语义避免了不必要的 clone
struct ReadStateEntry {
    last_message_id: u64,   // 最后已读消息 ID
    mention_count: u32,     // @提及 数量
    last_updated: u64,      // 最后更新时间戳（u64 避免 SystemTime 的开销）
}

impl ReadStateStore {
    /// 查询某个频道中用户的已读状态
    /// 注意：返回 Option<&T> 而非 clone，避免不必要的内存分配
    fn get_read_state(&self, channel_id: ChannelId, user_id: UserId) -> Option<&ReadStateEntry> {
        self.states.get(&(channel_id, user_id))
    }

    /// 更新已读状态
    fn update_read_state(&mut self, channel_id: ChannelId, user_id: UserId, new_msg_id: u64) {
        // entry API 减少了两次哈希查找（get + insert）
        self.states.entry((channel_id, user_id))
            .and_modify(|e| {
                e.last_message_id = new_msg_id;
                e.last_updated = now_timestamp();
            })
            .or_insert(ReadStateEntry {
                last_message_id: new_msg_id,
                mention_count: 0,
                last_updated: now_timestamp(),
            });
    }
}

// 假设已实现
fn now_timestamp() -> u64 { unimplemented!() }
```

### 1.5 核心教训

**GC 消除本身就是一种高杠杆优化。** 很多时候，开发者沉浸在函数级优化中（减少一次哈希查找、预分配 Vec），但 **语言运行时级别的瓶颈**（GC、JIT 编译预热、动态分派开销）才是更值得优先解决的大鱼。Rust 的优势不仅是"安全"，更是"可预测的性能"——你在写 HashMap 时就知道不会有 GC 打断你。

## 2. Cloudflare Pingora：用 Rust 替代 Nginx 的 HTTP 代理

### 2.1 为什么 Nginx 不再满足需求

Cloudflare 是全球最大的 CDN 之一。他们的边缘节点上运行着数千个 HTTP 代理实例，处理来自全球用户的流量。多年来，Nginx 一直是主力代理——但随着时间的推移，Nginx 的架构限制逐渐暴露：

1. **请求处理模型**：Nginx 使用 worker 进程模型（每个 worker 是一个独立进程），内存隔离好但通信成本高。
2. **共享状态困难**：连接池、速率限制计数器、缓存数据需要在 worker 之间共享，Nginx 依赖共享内存（shm）或外部服务，导致复杂度爆炸。
3. **扩展困难**：Nginx 的模块 API 使用 C 语言，编写安全的内存管理代码本身就是一种负担。
4. **连接重用不足**：每个请求都需要重新建立后端连接，在 TLS 握手密集的场景下开销巨大。

### 2.2 Pingora 的设计哲学

Pingora 的核心设计思想是**用共享内存代替进程隔离**，用 Rust 的类型系统保证在共享状态时不会产生 data race：

```
┌────────────────────────────────────────────────────────┐
│                  Pingora 架构（简化）                     │
├────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                 │
│  │ Thread 0 │  │ Thread 1 │  │ Thread 2 │  ...          │
│  │ (tokio)  │  │ (tokio)  │  │ (tokio)  │               │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘               │
│       │              │              │                    │
│       └──────────────┼──────────────┘                    │
│                      │                                   │
│          ┌───────────▼───────────┐                       │
│          │   共享连接池 (Arc)     │  ← Rust 保证安全共享  │
│          │   共享健康检查状态     │                       │
│          │   共享速率限制计数器   │                       │
│          └───────────────────────┘                       │
│                      │                                   │
│          ┌───────────▼───────────┐                       │
│          │   上游 (Origin) 服务器  │                      │
│          └───────────────────────┘                       │
│                                                         │
└────────────────────────────────────────────────────────┘
```

```rust
// Pingora 连接池核心逻辑（简化示意）
use std::sync::Arc;
use std::collections::HashMap;
use tokio::sync::Mutex; // tokio Mutex 不会阻塞整个线程

/// 连接池管理的核心抽象
/// 所有线程共享同一个连接池实例
struct ConnectionPool {
    // key: 上游服务器地址，value: 可用连接列表
    pools: Arc<Mutex<HashMap<String, Vec<PooledConnection>>>>,
}

struct PooledConnection {
    stream: tokio::net::TcpStream,
    last_used: std::time::Instant,
    // 在 Rust 中，TcpStream 的所有权明确——不会出现 C 中的 "谁负责关闭？" 问题
}

impl ConnectionPool {
    /// 获取一个到上游服务器的连接，优先复用已有连接
    /// 如果连接池中有空闲连接则直接复用，否则建立新连接
    async fn get_connection(&self, upstream: &str) -> PooledConnection {
        let mut pools = self.pools.lock().await;
        if let Some(list) = pools.get_mut(upstream) {
            if let Some(conn) = list.pop() {
                // 复用已有连接——节省了 TCP 握手和 TLS 握手的开销
                return conn;
            }
        }
        // 连接池空或没有到该 upstream 的连接：建立新连接
        drop(pools); // 显式释放锁，避免在建立连接时持有锁
        self.create_new_connection(upstream).await
    }

    /// 归还连接到连接池（而非直接关闭）
    async fn return_connection(&self, upstream: &str, conn: PooledConnection) {
        let mut pools = self.pools.lock().await;
        pools.entry(upstream.to_string())
            .or_insert_with(Vec::new)
            .push(conn);
    }

    // 假设已实现
    async fn create_new_connection(&self, _upstream: &str) -> PooledConnection { unimplemented!() }
}
```

### 2.3 零停机升级

Pingora 支持平滑重启（graceful restart），这也是 Rust 所有权模型的一个巧妙应用：

```rust
// Pingora 的平滑升级策略（概念示意）
use std::os::unix::net::UnixListener;
use std::net::TcpListener;
use std::os::unix::io::FromRawFd;

/// 零停机升级的核心思想：
/// 父进程将监听 socket 的文件描述符传递给子进程
/// 子进程接管后，父进程逐条完成现有请求再退出
struct GracefulRestart {
    // 监听 socket——这是必须无缝传递的资源
    listener: TcpListener,
}

impl GracefulRestart {
    /// 从父进程继承 socket fd 并转为 TcpListener
    /// 这是 Rust 中 unsafe 的经典使用场景——你承诺正确管理 fd 的生命周期
    unsafe fn inherit_from_parent(fd: std::os::unix::io::RawFd) -> Self {
        // 来自父进程的文件描述符
        // 使用 FromRawFd 时需要保证：
        // 1. fd 是有效的
        // 2. 你是唯一的所有者
        let listener = TcpListener::from_raw_fd(fd);
        GracefulRestart { listener }
    }

    /// 子进程启动后开始接受新连接
    async fn serve(self) {
        // 此时父进程仍在处理既有连接，但不再接受新连接
        // 新连接全部由子进程处理——实现零停机
        loop {
            let (stream, addr) = self.listener.accept().unwrap();
            tokio::spawn(async move {
                // 处理请求...
                drop(stream); // 连接处理完毕
            });
        }
    }
}
```

### 2.4 为什么选择 Rust 而不是继续用 C

Cloudflare 的选择揭示了 Rust 在网络服务领域的核心优势：

| 维度 | C (Nginx) | Rust (Pingora) |
|------|-----------|----------------|
| 内存安全 | 需开发者保证，UAF/double-free 常见 | 编译期保证 |
| 并发共享 | 需 shm 或锁，容易出错 | Arc + Mutex，类型系统护航 |
| 模块扩展 | C 模块 API，容易引入 bug | Cargo 生态，trait 抽象 |
| 学习成本 | "人人都知道 C"（但写对很难） | 学习曲线陡但犯错成本低 |
| 生产可靠性 | 依赖 code review 和测试 | 编译器在 CI 就拦截了大部分 bug |

## 3. AWS Firecracker：通过"删除代码"来优化

### 3.1 问题：微虚拟机的启动瓶颈

AWS Lambda 在每个请求到来时需要启动一个沙箱化的执行环境。传统的做法是启动一个完整虚拟机——但这太慢了（秒级）。AWS 需要一个能在 **125ms 以内**启动的 microVM。

Firecracker 是 AWS 用 Rust 编写的 microVM 管理器（VMM），基于 KVM。它的核心优化哲学是**极简主义**：

> "最好的优化不是让代码跑得更快，而是让代码根本不存在。"

### 3.2 通过缩减设备模型来加速启动

传统 VMM（如 QEMU）模拟了大量硬件设备：PCI 总线、USB 控制器、声卡、显卡、网络适配器、磁盘控制器、串口……每个设备都需要初始化，每个设备都有状态机，每个状态机都需要代码来管理。

Firecracker 的答案很简单——只模拟三个设备：

```
┌──────────────────────────────────────────────────────────┐
│           QEMU vs Firecracker 设备模型对比                  │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  QEMU 模拟的设备:                 Firecracker 模拟的设备:   │
│  ┌─────┐ ┌──────┐ ┌────┐         ┌───────────┐            │
│  │ PCI │ │ USB  │ │ VGA│         │ virtio-net│  网络      │
│  ├─────┤ ├──────┤ ├────┤         ├───────────┤            │
│  │ IDE │ │ SATA │ │AC97│         │ virtio-blk│  存储      │
│  ├─────┤ ├──────┤ ├────┤         ├───────────┤            │
│  │ PS2 │ │ 串口  │ │RTC │         │ serial    │  控制台    │
│  ├─────┤ ├──────┤ ├────┤         └───────────┘            │
│  │ ACPI│ │ SMBus│ │... │         共 ~3 个设备              │
│  └─────┘ └──────┘ └────┘                                  │
│  共 ~40+ 个设备                                           │
│                                                           │
│  启动时间: ~2s                 启动时间: <125ms            │
│  内存开销: ~100MB+             内存开销: <5MB              │
│  攻击面: 广泛                   攻击面: 极小                │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

```rust
// Firecracker 的极简设备模型（概念示意）
// 注意：以下不是 Firecracker 的真实源码，而是体现其设计思想的示意代码

/// 极简的虚拟设备模型——只有三个设备
struct MinimalDeviceModel {
    // 只模拟最基本的三个设备
    net: VirtioNetDevice,       // 网络
    block: VirtioBlockDevice,   // 块存储
    serial: SerialDevice,       // 控制台（用于日志输出）

    // 注：没有 USB、没有声卡、没有显卡、没有 PCI 总线——
    // 少了这些设备的初始化代码，启动时间自然大幅缩短
}

impl MinimalDeviceModel {
    /// 极快的初始化——因为几乎没有东西需要初始化
    fn init(&mut self) {
        // 传统 VMM 的设备初始化可能涉及：
        //   - PCI 枚举       (~50ms)
        //   - ACPI 表构建    (~30ms)
        //   - USB 控制器初始化 (~10ms)
        //   - VGA BIOS 加载  (~20ms)
        //   - ...
        //
        // Firecracker 全部跳过——因为没有这些设备。

        self.net.init();
        self.block.init();
        self.serial.init();
        // 初始化完成。因为项目少，启动时间极短。
    }
}
```

### 3.3 内存优化：Rust 的所有权保证零拷贝

Firecracker 使用 Rust 的另一个关键优势是内存效率。通过精确的所有权控制，Firecracker 避免了不必要的内存拷贝和分配：

```rust
/// Firecracker 的内存管理：使用 mmap + 精确所有权控制
/// 每个 microVM 的地址空间通过文件映射分配
use std::fs::OpenOptions;

struct GuestMemory {
    // VM 的物理内存映射到宿主机上的文件或匿名映射
    // Rust 的所有权保证：当 GuestMemory 被 drop 时，映射自动解除
    mapping: memmap2::MmapMut,
    size: usize,
}

impl GuestMemory {
    /// 为 microVM 分配内存区域
    /// 使用 mmap 而非 malloc——内存只在真正被访问时才分配物理页
    fn allocate(size_mb: usize) -> std::io::Result<Self> {
        let size = size_mb * 1024 * 1024;
        // memmap2 提供了安全的 mmap 封装
        let mapping = memmap2::MmapOptions::new()
            .len(size)
            .map_anon()?; // 匿名映射——不关联文件

        // 注意：此时并未真正分配物理内存——只是预留了虚拟地址空间
        // 物理内存会在第一次访问时按页分配（按需分页）
        Ok(GuestMemory { mapping, size })
    }
}
```

### 3.4 核心教训

AWS Firecracker 教会我们：**性能优化的最高形式是"不做这件事"。** 每次代码审查时都应该问自己：这段代码真的是必须的吗？少一个设备模型，就是少了数百行的初始化代码、测试代码、安全审计面。在性能的世界里，"删除代码"是一种被严重低估的优化手段。

## 4. Figma WASM 渲染器：跨语言优化边界

### 4.1 问题：浏览器中的设计工具

Figma 是一个基于浏览器的协作设计工具。它的核心挑战在于：在浏览器中渲染复杂的矢量图形需要极高的计算性能。JavaScript 做这件事太慢了，所以 Figma 选择将渲染引擎编译为 WebAssembly——而 WebAssembly 模块的编写语言是 Rust。

这是 Rust → WASM 的典型高性能场景：计算密集、类型安全要求高、内存管理必须显式。

### 4.2 WASM 性能模型：什么可以优化、什么不可以

WASM 的性能特征和原生 Rust 有微妙但重要的差异：

```
┌─────────────────────────────────────────────────────────────┐
│              Rust → WASM 性能边界                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  可以优化到接近原生的:        无法优化到接近原生的:            │
│  ┌────────────────────┐    ┌────────────────────────┐       │
│  │ 纯计算（数学、逻辑） │    │ SIMD（WASM SIMD 指令受限）│       │
│  │ 内存局部性好的代码   │    │ 内存分配（WASM 线性内存） │       │
│  │ 循环、分支          │    │ 多线程（SharedArrayBuffer）│       │
│  │ 函数调用（inlined）  │    │ 系统调用（无 OS 接口）    │       │
│  │ 结构体操作          │    │ 原生 SIMD  intrinsics    │       │
│  └────────────────────┘    │ 大内存页                  │       │
│                             │ 精确的 cache 控制         │       │
│  典型差距: 1.2x - 1.5x      └────────────────────────┘       │
│                             典型差距: 2x - 5x+               │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

```rust
// Figma 风格 WASM 渲染器核心逻辑（概念示意）

use wasm_bindgen::prelude::*;

/// 表示一个 2D 向量——在 WASM 中，这些纯计算操作几乎没有性能损失
#[wasm_bindgen]
#[derive(Clone, Copy)]
pub struct Vec2 {
    pub x: f32,
    pub y: f32,
}

/// 表示一个贝塞尔曲线段——Figma 的核心渲染原语
#[wasm_bindgen]
pub struct BezierSegment {
    p0: Vec2, // 起点
    p1: Vec2, // 控制点 1
    p2: Vec2, // 控制点 2
    p3: Vec2, // 终点
}

#[wasm_bindgen]
impl BezierSegment {
    /// 在参数 t (0.0..=1.0) 处求值
    /// 这是纯计算——WASM 中几乎无性能损失
    pub fn evaluate(&self, t: f32) -> Vec2 {
        let t2 = t * t;
        let t3 = t2 * t;
        let u = 1.0 - t;
        let u2 = u * u;
        let u3 = u2 * u;

        // 三次贝塞尔公式: B(t) = (1-t)^3*P0 + 3(1-t)^2*t*P1 + 3(1-t)*t^2*P2 + t^3*P3
        Vec2 {
            x: u3 * self.p0.x + 3.0 * u2 * t * self.p1.x + 3.0 * u * t2 * self.p2.x + t3 * self.p3.x,
            y: u3 * self.p0.y + 3.0 * u2 * t * self.p1.y + 3.0 * u * t2 * self.p2.y + t3 * self.p3.y,
        }
    }

    /// 将曲线展平为一组线段（用于光栅化）
    /// 在 WASM 中，内存分配（Vec push）的性能取决于 WASM 运行时的优化程度
    pub fn flatten(&self, tolerance: f32) -> Vec<Vec2> {
        let mut points = Vec::with_capacity(64); // 预分配减少 WASM 线性内存的重新分配
        self.recursive_flatten(0.0, 1.0, tolerance, &mut points);
        points
    }

    fn recursive_flatten(&self, t0: f32, t1: f32, tolerance: f32, out: &mut Vec<Vec2>) {
        // 递归细分——将曲线段拆成小段直到满足容差
        let mid = (t0 + t1) / 2.0;
        let p0 = self.evaluate(t0);
        let p1 = self.evaluate(t1);
        let pm = self.evaluate(mid);

        // 检查中点是否足够接近线段
        let dx = pm.x - (p0.x + p1.x) / 2.0;
        let dy = pm.y - (p0.y + p1.y) / 2.0;

        if dx * dx + dy * dy <= tolerance * tolerance {
            out.push(p1);
        } else {
            self.recursive_flatten(t0, mid, tolerance, out);
            self.recursive_flatten(mid, t1, tolerance, out);
        }
    }
}
```

### 4.3 WASM 特有问题：跨边界优化

Rust → WASM → JS → DOM 这个链条中，每一层边界都是潜在的优化盲区：

```rust
// 跨边界优化的反模式：频繁调用 JS←→WASM
#[wasm_bindgen]
pub fn render_frame_bad(canvas_width: u32, canvas_height: u32, elements: &[RenderElement]) {
    // 反模式：每个元素都单独调用 JS 进行绘制
    // 每次从 WASM 跨越到 JS 都有 ~100ns 的开销
    for elem in elements {
        draw_to_canvas_js(elem); // ← 跨边界调用！一次一个！
    }
}

// 跨边界优化的正确模式：批量处理
#[wasm_bindgen]
pub fn render_frame_good(canvas_width: u32, canvas_height: u32, elements: &[RenderElement]) {
    // 模式：在 WASM 侧完成所有计算，生成像素缓冲区
    let mut pixel_buffer = vec![0u8; (canvas_width * canvas_height * 4) as usize];

    // 所有计算在 WASM 内部完成，只有一次 JIT 编译的代码在运行
    for elem in elements {
        elem.rasterize_into(&mut pixel_buffer, canvas_width);
    }

    // 一次性将整个像素缓冲区传递到 JS——只跨越一次边界
    send_pixels_to_canvas_js(&pixel_buffer); // ← 一次跨边界调用
}

// 假设的辅助函数
#[wasm_bindgen]
extern "C" {
    fn draw_to_canvas_js(elem: &RenderElement);
    fn send_pixels_to_canvas_js(buf: &[u8]);
}

// 假设 RenderElement 已定义
struct RenderElement;
impl RenderElement {
    fn rasterize_into(&self, _buf: &mut [u8], _width: u32) {}
}
```

**WASM 优化的黄金法则：** 尽量在 WASM 内部完成所有计算，只在需要 DOM 交互时才跨越到 JS，并且尽可能批量跨越。

### 4.4 教训

Figma 的案例告诉我们，优化不是"让 Rust 代码更快"——Rust 本身的性能通常不是瓶颈。瓶颈出现在 **跨语言边界**（Rust→JS）、**跨运行时边界**（WASM→宿主）、以及**与你不完全控制的运行时交互**的地方。理解这些边界是高级性能工程的关键。

## 5. 可复用的优化模式：三级分类法

### 5.1 优化的三个层级

通过对以上四个案例的分析，我们可以提炼出一个**优化层级分类法**（Optimization Taxonomy）：

```
┌────────────────────────────────────────────────────────────────┐
│                    优化层级金字塔                               │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│                        ┌─────────┐                              │
│                        │ 架构级   │ ← 语言选择、GC vs 无GC       │
│                        │ 优化    │   共享内存 vs 进程隔离        │
│                        │ (10x+)  │   设备模型删减               │
│                        └────┬────┘                              │
│                             │                                   │
│                      ┌──────┴──────┐                            │
│                      │   算法级     │ ← 数据结构选择             │
│                      │   优化       │   算法替换                 │
│                      │   (2x-10x)   │   复杂度降低              │
│                      └──────┬──────┘                            │
│                             │                                   │
│                    ┌────────┴────────┐                          │
│                    │  实现级优化      │ ← 内存布局调整           │
│                    │  (5%-50%)       │   循环展开                │
│                    │                 │   分支预测 hints          │
│                    └─────────────────┘   SIMD 内联               │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

**层级对应表：**

| 层级 | 收益 | 风险 | 耗时 | 案例对应 |
|------|------|------|------|----------|
| 架构级 | 10x - 1000x | 高（需重写）| 周-月 | Discord Go→Rust, Firecracker 精简设备 |
| 算法级 | 2x - 10x | 中（换实现）| 天-周 | Pingora 连接池, 共享状态设计 |
| 实现级 | 5% - 50% | 低（局部改动）| 小时-天 | Figma WASM 跨边界批量化 |

### 5.2 从别人的优化报告中提炼方法

阅读外部优化报告时，问自己三个问题：

```
1. 这个项目解决了什么层级的瓶颈？
   → 如果只是 "换了 HashMap 的实现"（算法级），问：他们为什么不先考虑架构级？
   → 如果是 "从 Python 迁移到 Rust"（架构级），问：语言替换带来了哪些底层改善？

2. 优化前后测量了什么指标？用什么工具测量的？
   → 如果只有 "感觉变快了"，这不可靠
   → 如果用了火焰图、perf stat、latency histogram，值得参考

3. 这种优化在我的场景下适用吗？
   → Go→Rust 在 Discord 有效，在你的 CRUD API 上可能大材小用
   → 关键不是 "他们做了什么"，而是 "什么条件让他们成功了"
```

## 6. 阅读方法论：如何找到和分析性能资料

### 6.1 高效搜索关键词

当你遇到性能瓶颈时，使用以下搜索词比直接搜索 "Rust slow" 有效得多：

```
通用搜索词:
  "[crate name] performance case study"
  "[domain] rust optimization blog"
  "rust [problem] flamegraph analysis"
  "how [company] rewrote in rust"

技术搜索词:
  "cache miss reduction rust"
  "lock contention analysis rust"
  "allocation hot path rust"
  "io_uring benchmark rust 2024"

来源搜索:
  "site:reddit.com/r/rust performance tips"
  "site:discord.com blog rust"
  "site:infoq.com rust performance"
```

### 6.2 值得关注的性能相关博客和会议

| 来源 | 类型 | 特点 |
|------|------|------|
| "The Rust Performance Book" | 官方文档 | 系统性的性能优化指南 |
| fasterthanli.me | 个人博客 | 深度分析的 Rust 文章，干货极多 |
| Discord Blog | 公司博客 | 定期发布技术迁移文章，直面真实问题 |
| Cloudflare Blog | 公司博客 | 网络基础设施的性能优化实战 |
| RustConf talks | 会议视频 | 每年都有 sys/performance track |
| ripgrep blog (Andrew Gallant) | 作者博客 | rg 的性能优化决策过程 |
| println!() 论坛 "performance" tag | 社区 | 真实问题 + 社区解答 |

### 6.3 分析一篇性能报告的标准流程

当读到一篇性能优化文章时，按以下步骤分析：

```
步骤 1: 找出 "before" 数量
  → 优化前有多慢？在什么场景下慢？负载特征是什么？

步骤 2: 找出瓶颈的根因
  → 作者怎么定位的？用了什么工具？证据是什么？

步骤 3: 找出解决方案
  → 属于架构/算法/实现哪个层级？有没有更简单的方案？

步骤 4: 找出 "after" 数量
  → 优化后快了多少？用什么指标衡量？这些指标代表真实体验吗？

步骤 5: （关键）找出代价
  → 优化带来了什么新的复杂度？是否引入了新的限制？

步骤 6: 提炼可迁移的教训
  → 同样的方法论在我的项目里能用吗？如果不能，差在哪？
```
