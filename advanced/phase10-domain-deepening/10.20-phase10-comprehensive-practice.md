## 2. 选项 A: 性能优化实战

### 2.1 任务描述

找到一个真实的开源 Rust 项目，应用 10.6 的"优化 Kata"方法论，定位并修复一个真实的性能瓶颈，提交 PR。

### 2.2 推荐项目池

按难度和活跃度排序：

| 项目 | 类型 | 为什么适合 | 预期改进 |
|------|------|-----------|----------|
| `ripgrep` | 搜索工具 | 高度优化但仍有机会，作者乐于接受性能 PR | 5-20% |
| `csv` crate | CSV 解析 | 代码量小，容易理解，解析器优化经典 | 10-50% |
| `serde_json` | JSON 序列化 | 使用广泛，热点清晰，已有 benchmark 框架 | 5-30% |
| `pest` | 解析器 | 解析器生成器，通用优化空间大 | 10-40% |
| `rust-analyzer` | IDE 工具 | 很多性能 issue（启动速度、内存使用） | 10-50% |
| `cargo` | 构建工具 | 每个人都能受益，性能 issue 有标签 | 5-30% |

### 2.3 工作流程

```
Step 1: 选项目和 issue
  → 在 GitHub 上找一个标记为 "performance" 或 "optimization" 的 issue
  → 或者自己用 flamegraph 定位一个热点
  → 在 issue 下留言 "I'd like to work on this"

Step 2: 建立 baseline
  → 写 criterion benchmark（如果项目没有的话——这是一个很棒的贡献）
  → 记录 before 数据: throughput, latency p50/p99/p999, memory

Step 3: 定位瓶颈
  → cargo flamegraph → 找到最热的函数
  → perf stat → 确认问题类型（cache miss? branch miss? allocation?）
  → 形成假设: "慢在这里是因为 X"

Step 4: 实现修复
  → 最小化改动范围（遵循 PR 黄金法则: 小、聚焦、有测试）
  → 实现优化
  → 跑 benchmark → 确认改善

Step 5: 提交 PR
  → PR description: before/after benchmarks
  → 解释为什么这样改
  → 如果被 reviewer 要求修改 → 这正是学习的机会
```

### 2.4 产出要求

```markdown
# 产出清单

□ GitHub PR 链接（已提交，含 description 和 benchmarks）
□ Before/After 性能对比（截图或表格）
□ 使用的 profiling 工具和方法
□ 心得总结（至少 500 字）:
  - 怎么定位到瓶颈的？
  - 为什么选择这个优化方案？
  - 有尝试过但放弃的方案吗？
  - 从 reviewer 那里学到了什么？
□ （可选）被 merge 了？标注 merge commit
```

## 3. 选项 B: 量化交易系统

### 3.1 任务描述

基于 10.13 的微型交易系统架构，实现一个**新的交易策略**（不是均线回归），用至少 **1 年的历史数据回测**，写出完整的回测分析报告。

### 3.2 策略选项

选一个难度适合你的：

```
Level 1: 动量策略 (Momentum)
  → 信号: 过去 N 分钟涨幅超过 X% → 买入，跌破 Y% → 卖出
  → 参数: N (回溯窗口), X (入场阈值), Y (退出阈值)
  → 难度: 低, 概念直观

Level 2: 布林带策略 (Bollinger Bands)
  → 信号: 价格触及下轨 → 买入, 回归中轨 → 卖出, 触及上轨 → 做空
  → 参数: 窗口长度, 标准差倍数
  → 难度: 中, 需要在线计算均值和标准差

Level 3: 统计套利 (Pairs Trading)
  → 信号: 两个相关资产的价差偏离均值 → 做多弱者+做空强者, 价差回归 → 平仓
  → 参数: 协整检验窗口, 入场 Z-score, 出场 Z-score
  → 难度: 高, 需要协整检验和动态对冲比
```

### 3.3 数据来源

```
免费历史数据来源:
  → Yahoo Finance (yfinance crate 或手动下载 CSV)
  → Alpha Vantage (免费 API, 需要注册)
  → Binance public API (加密货币, 数据免费且丰富)
  → IEX Cloud (美股, 免费 tier)

推荐: 用 Binance 的加密货币数据
  → 数据免费、API 友好、交易对多、24/7 交易
  → 用 reqwest 调 REST API 下载 1 年的 1 分钟 K 线
```

### 3.4 回测报告模板

```markdown
# 回测报告: [策略名] on [交易品种]

## 策略描述
[一段话描述策略逻辑]

## 数据
- 交易品种: [e.g. BTC/USDT]
- 时间范围: [e.g. 2025-01-01 ~ 2025-12-31]
- 频率: [e.g. 1 分钟 K 线]
- 数据来源: [e.g. Binance API]

## 参数
- 参数 1: 值 (搜索范围: X-Y)
- 参数 2: 值 (搜索范围: X-Y)

## 回测结果
| 指标 | 值 | 基准 (买入持有) |
|------|------|------|
| 总收益率 | +X% | +Y% |
| 年化收益率 | X% | Y% |
| Sharpe Ratio | X | Y |
| Sortino Ratio | X | Y |
| Max Drawdown | -X% | -Y% |
| Win Rate | X% | Y% |
| Profit Factor | X | Y |
| 交易次数 | X | - |
| 平均持仓时间 | X 分钟 | - |

## 收益曲线
[ASCII 图或 matplotlib 截图]

## 分析
- 策略在什么市场环境下表现好？（趋势? 震荡?）
- 策略在什么市场环境下表现差？
- 参数稳定性如何？（不同参数组合的表现方差）
- 有哪些明显的过拟合风险？

## 改进方向
- [2-3 个具体的改进想法]
```


## 4. 选项 C: Web3 产品

### 4.1 任务描述

从以下三个产品创意中选择一个，用 Anchor 框架实现 Solana 程序，部署到 **devnet**，写一个简单的 Web 前端展示核心交互。

### 4.2 产品创意

```
创意 1: 简易 DEX (去中心化交易所)
  核心功能:
    → 创建流动性池 (token A + token B)
    → 添加流动性 (deposit both tokens, receive LP tokens)
    → 移除流动性 (burn LP tokens, receive underlying)
    → 兑换 (constant product AMM: x * y = k)
  技术要点: PDA, Token Program CPI, 滑点保护, 手续费计算
  难度: 中
  学习价值: 理解 AMM 的数学和 Solana 的 CPI 模式

创意 2: NFT 市场
  核心功能:
    → 上架 NFT (list for sale)
    → 购买 NFT (buy)
    → 取消上架 (cancel listing)
    → 更新价格 (update price)
    → 平台手续费 (marketplace fee, 例如 2.5%)
  技术要点: Metaplex Token Metadata, PDA escrow, 版税分配
  难度: 中
  学习价值: 理解 NFT 标准、托管模式、版税机制

创意 3: 预测市场
  核心功能:
    → 创建市场 (create market with question + outcome options)
    → 购买 outcome token (buy shares of an outcome)
    → 结算市场 (resolve market, distribute winnings)
    → 获取奖励 (claim winnings)
  技术要点: 状态管理、时间锁、Oracle 集成 (or 手动结算)
  难度: 高
  学习价值: 理解预测市场的经济模型和链上结算机制
```

### 4.3 技术栈

```
链上: Anchor (Solana program)
测试: solana-program-test + solana-test-validator
前端 (可选但加分): React + @solana/wallet-adapter + @project-serum/anchor
部署: solana program deploy (devnet)
```

### 4.4 开发流程

```
Step 1: 脚手架
  → anchor init my-project
  → 配置 Anchor.toml (cluster = devnet)
  → 创建程序入口 (lib.rs)

Step 2: 实现核心逻辑
  → 定义账户结构 (#[account])
  → 定义指令 (#[derive(Accounts)] for each instruction)
  → 实现处理器 (the actual logic)

Step 3: 写测试
  → TypeScript 或 Rust 测试
  → 测试每个指令的 happy path + error cases
  → anchor test -- --features testing

Step 4: 部署到 devnet
  → solana config set --url devnet
  → solana airdrop 2 (获取测试 SOL)
  → anchor build && anchor deploy

Step 5: (可选) 前端
  → React app 连接 Phantom/Solflare 钱包
  → 展示核心功能
  → 写 README 说明如何使用
```

### 4.5 产出要求

```
□ GitHub repo（必须，需要 showcase）
□ Anchor 程序部署到 devnet（provable via Solana Explorer）
□ 完整的测试覆盖（至少 happy path + 2 个 error cases）
□ README 包含:
  - 项目描述
  - 部署地址 (program ID)
  - 如何本地运行测试
  - 如何与部署的程序交互
□ （可选）简单的前端界面
□ 心得总结（至少 500 字）
```
