# 10.8 市场数据基础设施 —— 从交易所原始报文到可分析的订单簿

> **[Track B: 量化金融]** 数据是量化交易的第一法则——没有可靠的市场数据，再好的策略模型都是空中楼阁。本课从 Level 1/2/3 的定义出发，带你用 Rust 构建一条完整的市场数据管线：解析二进制协议报文（Nasdaq ITCH）、设计纳秒级时序存储、用零拷贝技术榨干解析性能、以及从增量行情重建实时订单簿。这是一条量化开发者每天都要面对的生产级数据通道。

**领域背景：** 全球交易所每秒产生数百万条市场数据报文。这些数据通过 UDP 组播从交易所机房以近乎光速推送到你的服务器。每条报文都有严格的结构——一个字节错了，整条订单簿就乱了。市场数据分为三个层次：Level 1 只告诉你要买多少钱、要卖多少钱、最后一笔成交价是多少（最轻量）；Level 2 告诉你每个价位的深度（适合分析流动性）；Level 3 告诉你每一笔挂单都是谁的（交易所内部使用）。你的任务是构建一条不丢包、不重序、纳秒级精度的数据采集管线——这是量化系统中最难写但最重要的基础设施。

## 1. 市场数据类型：L1、L2、L3 的 Rust 定义

### 1.1 L1（Level 1）：最优报价 + 最近成交

```
L1 数据量：每条品种每变化一次约 100 字节
典型更新频率：毫秒级（取决于交易所更新速率）
典型用途：行情展示、价差监控、简单执行策略

L1 只展示"一层皮"——你看不到市场的深度
但它的数据量最小、延迟最低
```

```rust
use chrono::{NaiveDateTime, Timelike};

/// Level 1 行情：最优买卖价 + 最近一笔成交
#[derive(Debug, Clone)]
struct Level1Quote {
    /// 品种代码，如 "AAPL"、"BTC-USD.P"
    symbol: String,
    /// 最优买价（最高的买入限价）
    bid_price: i64,        // 价格单位：1/10000 美分
    /// 最优买价对应的挂单数量
    bid_size: u64,
    /// 最优卖价（最低的卖出限价）
    ask_price: i64,
    /// 最优卖价对应的挂单数量
    ask_size: u64,
    /// 最后成交价
    last_trade_price: i64,
    /// 最后成交量
    last_trade_size: u64,
    /// 交易所生成的时间戳（纳秒精度）
    exchange_timestamp_ns: u64,
    /// 本地接收时间戳（用于计算网络延迟）
    local_timestamp_ns: u64,
    /// 交易日（即 YYYY-MM-DD，用于分区写入）
    trade_date: String,
}

impl Level1Quote {
    /// 计算价差（spread）——做市商盈利的核心指标
    fn spread(&self) -> i64 {
        self.ask_price - self.bid_price
    }

    /// 计算中间价——许多交易策略的锚点
    fn mid_price(&self) -> f64 {
        (self.bid_price as f64 + self.ask_price as f64) / 2.0
    }

    /// 计算价差百分比——用于比较不同价格品种的流动性成本
    fn spread_pct(&self) -> f64 {
        (self.ask_price - self.bid_price) as f64 / self.mid_price() * 100.0
    }

    /// 计算网络传输延迟（纳秒）
    fn network_latency_ns(&self) -> u64 {
        self.local_timestamp_ns.saturating_sub(self.exchange_timestamp_ns)
    }
}
```

### 1.2 L2（Level 2）：全深度订单簿快照

```
L2 数据量：每条品种每次变化约 50-200 条价位更新
典型更新频率：每变化一笔都推送（可能每秒数百次）
典型用途：流动性建模、算法执行、深度分析

L2 揭示市场的"剖面"——每个价位的挂单量
但 L2 不告诉你是谁挂的单（没有订单 ID）
```

```rust
use std::collections::BTreeMap;

/// L2 价格级别（订单簿的一个价位）
#[derive(Debug, Clone)]
struct PriceLevel {
    /// 价格
    price: i64,
    /// 该价位的总挂单量（比如价格 150.00 处有 5000 股挂单）
    total_quantity: u64,
    /// 该价位的订单数量（有多少人在这个价位排队）
    order_count: u32,
}

/// Level 2 订单簿快照：全深度
#[derive(Debug, Clone)]
struct Level2Book {
    symbol: String,
    /// 买单价位：价格从高到低排列
    /// BTreeMap 默认升序，迭代时用 .rev() 获取从高到低
    bids: BTreeMap<i64, PriceLevel>,
    /// 卖单价位：价格从低到高排列
    asks: BTreeMap<i64, PriceLevel>,
    /// 快照的时间戳
    sequence_number: u64,
    exchange_timestamp_ns: u64,
    local_timestamp_ns: u64,
}

impl Level2Book {
    /// 获取最优买卖价（L2 可以直接提取 L1 信息）
    fn best_bid(&self) -> Option<(&i64, &PriceLevel)> {
        // bids 是升序，最后一个 key 是最高价
        self.bids.iter().next_back()
    }

    fn best_ask(&self) -> Option<(&i64, &PriceLevel)> {
        // asks 是升序，第一个 key 是最低价
        self.asks.iter().next()
    }

    /// 计算 5 档深度（最优的 5 个价位总挂单量）
    /// 这是量化学者常用的流动性指标
    fn depth_5_levels(&self, side: &str) -> u64 {
        let book = match side {
            "bid" => &self.bids,
            "ask" => &self.asks,
            _ => return 0,
        };
        book.iter()
            .rev()  // 从最优价开始
            .take(5)
            .map(|(_, level)| level.total_quantity)
            .sum()
    }

    /// 计算订单簿不平衡度
    /// (买单深度 - 卖单深度) / (买单深度 + 卖单深度)
    /// 正值 = 买方力量强，负值 = 卖方力量强
    fn imbalance(&self, levels: usize) -> f64 {
        let bid_sum: u64 = self.bids.iter().rev().take(levels)
            .map(|(_, l)| l.total_quantity).sum();
        let ask_sum: u64 = self.asks.iter().take(levels)
            .map(|(_, l)| l.total_quantity).sum();
        if bid_sum + ask_sum == 0 {
            return 0.0;
        }
        (bid_sum as f64 - ask_sum as f64) / (bid_sum as f64 + ask_sum as f64)
    }
}
```

### 1.3 L3（Level 3）：每条订单可见

```
L3 数据量：极大（每个订单每条变化都推送）
典型用途：交易所内部（散户通常无法获取完整 L3）
关键信息：订单 ID 可见 → 可以追踪每个参与者的行为

L3 对量化开发者来说是一个"圣杯"级别的数据
因为它揭示了市场的微观博弈——谁在做什么
```

```rust
/// L3 订单级别的数据：每个订单的信息都是可见的
#[derive(Debug, Clone)]
struct Level3Order {
    /// 订单唯一 ID（在交易所范围内唯一）
    order_id: u64,
    /// 买卖方向
    side: Side,
    /// 限价
    price: i64,
    /// 初始数量
    original_quantity: u64,
    /// 剩余数量
    remaining_quantity: u64,
    /// 是否隐藏数量（iceberg 订单）
    is_hidden: bool,
    /// 该订单的"优先级"位置（在此价位排第几个）
    position: u32,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum Side { Buy, Sell }

/// L3 全订单级视图
#[derive(Debug, Clone)]
struct Level3Book {
    symbol: String,
    /// 所有活跃订单的完整视图
    orders: BTreeMap<u64, Level3Order>,  // order_id → order
    /// 按价位索引的订单（用于快速查询深度）
    bids_by_price: BTreeMap<i64, Vec<u64>>,  // price → [order_id]
    asks_by_price: BTreeMap<i64, Vec<u64>>,
}
```

## 2. Tick 数据库：纳秒级时序存储

### 2.1 为什么 A 股时间和 Unix 时间都需要支持？

```
量化金融中时间精度的演进：

1990s: 秒级 —— 足够了，交易频率按分钟计
2000s: 毫秒级 —— 算法交易兴起，1ms 的延迟差异变得重要
2010s: 微秒级 —— HFT 竞争，硬件加速和 co-location 成为标配
2020s: 纳秒级 —— 极端竞争，FPGA + 专线 + 距离优化

Rust 的 chrono crate 默认精度只到微秒
纳秒级的精度需要扩展存储能力
```

```rust
use chrono::{NaiveDateTime, NaiveDate, NaiveTime, Timelike};

/// 纳秒精度的时间戳
/// chrono 默认支持到微秒（nanosecond 字段值为 0-999,999,999
/// 但 chrono 内部存储的是 u32: 0..1_999_999_999）
#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord)]
struct NanoTimestamp {
    /// Unix 纪元以来的秒数
    seconds: i64,
    /// 纳秒部分（0..1_000_000_000）
    nanos: u32,
}

impl NanoTimestamp {
    /// 从 chrono NaiveDateTime 构造
    fn from_naive(dt: NaiveDateTime) -> Self {
        Self {
            seconds: dt.timestamp(),
            nanos: dt.nanosecond(),
        }
    }

    /// 转换为 NaiveDateTime（如果纳秒部分大于 999,999,999 则截断）
    fn to_naive(&self) -> NaiveDateTime {
        NaiveDateTime::from_timestamp_opt(self.seconds, self.nanos)
            .expect("无效的时间戳")
    }

    /// 两个时间戳之间的纳秒差值（用于计算延迟）
    fn nanos_between(&self, earlier: &NanoTimestamp) -> i64 {
        let sec_diff = (self.seconds - earlier.seconds) * 1_000_000_000;
        let nano_diff = self.nanos as i64 - earlier.nanos as i64;
        sec_diff + nano_diff
    }
}

/// 一条完整的 tick 数据记录
/// 用序列化的格式存储（适合 Parquet 列式存储）
#[derive(Debug, Clone)]
struct TickRecord {
    /// 品种 ID（用于分区）
    symbol_id: u32,
    /// 时间戳
    timestamp: NanoTimestamp,
    /// 价格（定点整数）
    price: i64,
    /// 数量
    quantity: u64,
    /// 交易类型：1=正常, 2=集合竞价, 3=取消等
    trade_type: u8,
    /// 行情序号（用 delta 编码可以极大压缩）
    sequence_number: u64,
    /// 买卖方向（买=1, 卖=-1, 未知=0）
    side: i8,
    /// 该 tick 之前的最佳买价
    bid_price_before: i64,
    /// 该 tick 之前的最佳卖价
    ask_price_before: i64,
}

/// Delta 编码工具：存储序列号之间的差值而非绝对值
/// 连续性的数字（如 sequence number, timestamp）delta 编码后
/// 通常只需要 1-2 字节（原本需要 8 字节）
/// 压缩率通常可达 80-90%
struct DeltaEncoder {
    prev_value: u64,
}

impl DeltaEncoder {
    fn new() -> Self {
        Self { prev_value: 0 }
    }

    /// 编码：返回当前值与上一个值的差值
    fn encode(&mut self, value: u64) -> i64 {
        let delta = value.wrapping_sub(self.prev_value) as i64;
        self.prev_value = value;
        delta
    }

    /// 解码：从差值恢复原始值
    fn decode(&mut self, delta: i64) -> u64 {
        let value = (self.prev_value as i64).wrapping_add(delta) as u64;
        self.prev_value = value;
        value
    }
}
```

## 3. 高性能市场数据解析：零拷贝与二进制协议

### 3.1 二进制协议 vs 文本协议

```
┌─────────────────┬────────────────────────┬──────────────────────┐
│ 维度             │ 二进制协议（ITCH/OUCH）│ 文本协议（JSON/CSV）  │
├─────────────────┼────────────────────────┼──────────────────────┤
│ 报文大小         │ 30-50 bytes/条         │ 200-500 bytes/条     │
│ 解析速度         │ 极快（字段固定偏移）    │ 较慢（需扫描+转换）  │
│ 人类可读         │ 否                     │ 是                  │
│ 带宽成本（每秒    │ 50 MBps               │ 250+ MBps           │
│   1M msgs/s）    │                        │                     │
│ 典型场景         │ 交易所推送/交易指令     │ 日志/API/WebSocket   │
└─────────────────┴────────────────────────┴──────────────────────┘

量化金融中，二进制协议是主流——没人愿意在纳秒级竞争中使用 JSON
```

### 3.2 用 rkyv 实现零拷贝反序列化

```rust
// rkyv 是一个零拷贝序列化框架
// 和 serde/bincode 的区别：
//   serde → 遍历所有字段，逐字段复制到新结构体（需要分配内存）
//   rkyv  → 直接从原始字节中"投影"出结构体引用（零分配）

// 这在实际市场数据处理中意味着什么？
//   每秒 100 万条报文，每条 150 bytes = 150 MB/s
//   如果用 serde 反序列化，每条都要 200ns → 200ms/s 的 CPU 时间 → 20% CPU
//   如果用 rkyv 零拷贝，每条只需要 20ns → 20ms/s → 2% CPU

use rkyv::{Archive, Deserialize, Serialize};

/// ITCH 协议中的"新增订单"消息（简化版）
/// 
/// ITCH 是 Nasdaq 的市场数据协议
/// 每类消息有固定的字节布局
#[derive(Archive, Deserialize, Serialize, Debug, PartialEq)]
#[archive(check_bytes)]  // 验证字节的完整性
struct ItchAddOrder {
    /// 消息类型：'A' 或 'F'（代表 Add Order）
    /// 字段长度：1 字节
    message_type: u8,
    /// 订单唯一 ID（Nasdaq 分配的，在当日唯一）
    /// 字段长度：8 字节，大端序
    order_reference_number: u64,
    /// 买卖方向：'B' = Buy, 'S' = Sell
    /// 字段长度：1 字节
    buy_sell_indicator: u8,
    /// 股票数量
    /// 字段长度：4 字节，大端序
    shares: u32,
    /// 股票代码（Nasdaq 用字母数字编码）
    /// 字段长度：8 字节，右填充空格
    stock: [u8; 8],
    /// 价格：隐含小数点（例如 1234500 表示 123.4500）
    /// 字段长度：4 字节，大端序
    price: u32,
}

/// 手工解析 ITCH 消息（不依赖框架，展示底层字节布局）
/// 在生产环境中，手工解析比框架更快（因为框架有验证开销）
struct ItchParser;

impl ItchParser {
    /// 解析 ITCH 新增订单消息（字节布局已知）
    /// add_order 消息的字节布局：
    ///   字节 0:     消息类型 'A'(0x41) 或 'F'(0x46)
    ///   字节 1-2:   股票定位号（stock locate）
    ///   字节 3-4:   追踪号（tracking number）
    ///   字节 5-10:  时间戳（6 字节，纳秒，从午夜起）
    ///   字节 11-18: 订单参考号（8 字节，大端序）
    ///   字节 19:    买卖方向（'B'或'S'）
    ///   字节 20-23: 数量（4 字节，大端序）
    ///   字节 24-31: 股票符号（8 字节，ASCII）
    ///   字节 32-35: 价格（4 字节，大端序）
    ///
    /// 总长度：36 字节
    fn parse_add_order(data: &[u8]) -> Option<ParsedAddOrder> {
        // 基本长度检查
        if data.len() < 36 {
            return None;
        }

        // 消息类型检查
        if data[0] != b'A' && data[0] != b'F' {
            return None;
        }

        // 直接通过偏移量读取字段（零拷贝，无需反序列化）
        let order_ref = u64::from_be_bytes([
            data[11], data[12], data[13], data[14],
            data[15], data[16], data[17], data[18],
        ]);

        let side = if data[19] == b'B' { Side::Buy } else { Side::Sell };

        let shares = u32::from_be_bytes([
            data[20], data[21], data[22], data[23],
        ]);

        let mut stock_bytes = [0u8; 8];
        stock_bytes.copy_from_slice(&data[24..32]);
        // 去掉右侧填充的空格
        let stock = std::str::from_utf8(&stock_bytes)
            .unwrap_or("UNKNOWN")
            .trim_end()
            .to_string();

        let price = u32::from_be_bytes([
            data[32], data[33], data[34], data[35],
        ]);

        Some(ParsedAddOrder {
            order_reference_number: order_ref,
            side,
            shares,
            stock,
            price,
            message_type: data[0],
        })
    }
}

/// 解析后的订单（与内存表示脱钩，所有权在自己手上）
#[derive(Debug, Clone)]
struct ParsedAddOrder {
    order_reference_number: u64,
    side: Side,
    shares: u32,
    stock: String,
    price: u32,
    message_type: u8,
}

#[cfg(test)]
mod itch_tests {
    use super::*;

    /// 构造一条 ITCH 消息并验证解析
    #[test]
    fn test_parse_add_order() {
        // 构造一条合法的 Add Order 消息
        let mut msg = vec![0u8; 36];
        msg[0] = b'A';                      // message_type
        msg[11..19].copy_from_slice(&1u64.to_be_bytes()); // order_ref = 1
        msg[19] = b'B';                     // side = Buy
        msg[20..24].copy_from_slice(&100u32.to_be_bytes()); // shares = 100
        msg[24..32].copy_from_slice(b"AAPL    ");     // stock = "AAPL"
        msg[32..36].copy_from_slice(&1500000u32.to_be_bytes()); // price = 150.0000

        let result = ItchParser::parse_add_order(&msg);
        assert!(result.is_some());
        let order = result.unwrap();
        assert_eq!(order.order_reference_number, 1);
        assert_eq!(order.side, Side::Buy);
        assert_eq!(order.shares, 100);
```

## 4. 数据归一化层：统一不同交易所的格式

### 4.1 归一化的价值

```
为什么需要归一化层？

交易所 A (Nasdaq)：ITCH 协议，price = u32 隐含 4 位小数
交易所 B (Binance)：WebSocket/JSON，price = "150.32" 字符串
交易所 C (CME)：FIX/FAST 协议，price = i64 带独立指数

如果每个策略都要理解三种格式 → 维护难度爆炸
归一化层的作用：将三种格式统一为内部格式 → 策略只需要处理一种
```

```rust
/// 内部统一的行情格式（所有交易所的数据都归一化为这个结构体）
#[derive(Debug, Clone)]
struct NormalizedQuote {
    /// 全局唯一品种 ID（由归一化层赋予，不是交易所的 symbol）
    instrument_id: u32,
    /// 品种符号（原始符号，如 "AAPL", "BTCUSDT"）
    symbol: String,
    /// 所属交易所的内部 ID
    exchange_id: u32,
    /// 买一价（统一单位：1/10000 美分或 1/10000 base 货币）
    bid_price: i64,
    ask_price: i64,
    bid_size: u64,
    ask_size: u64,
    /// 统一时间戳（UTC 纳秒）
    timestamp_ns: i64,
    /// 原始报文的序列号（用于去重和 gap 检测）
    sequence_number: u64,
    /// 原始报文类型（用于出问题时回溯）
    original_message_type: u16,
}

/// 归一化处理器：将不同交易所的原始数据转换为统一定义
trait ExchangeNormalizer {
    /// 交易所代码（如 "NASDAQ", "BINANCE", "CME"）

    /// 将原始字节转换为内部统一格式
    fn normalize(&self, raw_data: &[u8], local_ts_ns: i64)
        -> Result<NormalizedQuote, NormalizeError>;
}

    /// 报文格式不正确
    InvalidFormat(String),
    /// 不支持的消息类型（我们不需要它）
    UnsupportedMessageType(u8),
    /// 不是我们的品种（过滤掉不需要的数据）
    NotSubscribedSymbol(String),
    /// 字段解析失败
    ParseError(&'static str),
}

/// Nasdaq ITCH 归一化器（骨架示例）
struct ItchNormalizer;

impl ExchangeNormalizer for ItchNormalizer {
    fn exchange_code(&self) -> &'static str {
        "NASDAQ"

    fn normalize(&self, raw_data: &[u8], local_ts_ns: i64)
```

### 4.2 数据清洗：坏点检测

```rust
/// 坏点（bad tick）检测器
/// 
/// 坏点来源：
///   - 交易所错误（罕见但存在）
///   - 供应商编码错误
///   - 网络损坏（UDP 无校验的部分）
///   - 除权除息未调整
///
/// 如果坏点进了订单簿 → 策略可能看到"离谱"的价格 → 执行错误的交易
struct BadTickDetector {
    symbol: String,
    /// 上一个有效价格（用于检测异常跳变）
    last_valid_price: i64,
    /// 允许的最大涨跌幅（例如 20%）
    max_price_change_pct: f64,
    /// 价格的有效范围（例如 AAPL 不会在 $0.01 或 $1000000 交易）
    price_range_min: i64,
    price_range_max: i64,
    /// 被丢弃的坏点计数
    bad_tick_count: u64,
}

impl BadTickDetector {
    fn new(symbol: &str, reference_price: i64) -> Self {
        Self {
            symbol: symbol.to_string(),
            last_valid_price: reference_price,
            max_price_change_pct: 20.0,
            // 合理范围：参考价的 50% 到 200%
            price_range_min: reference_price / 2,
            price_range_max: reference_price * 2,
            bad_tick_count: 0,
        }
    }

    /// 检测是否为坏点
    fn is_bad_tick(&mut self, new_price: i64) -> bool {
        // 检查 1：价格是否在合理范围内
        if new_price < self.price_range_min || new_price > self.price_range_max {
            self.bad_tick_count += 1;
            return true;
        }

        // 检查 2：价格跳变是否过大
        if self.last_valid_price > 0 {
            let change_pct = (new_price - self.last_valid_price).abs() as f64
                / self.last_valid_price as f64 * 100.0;
            if change_pct > self.max_price_change_pct {
                self.bad_tick_count += 1;
                return true;
            }
        }

        // 通过所有检查，更新有效价格
        self.last_valid_price = new_price;
        false
    }
}

/// 除权除息调整
/// 
/// 当股票拆分（2:1 split → 价格减半）或发放股息时
/// 历史价格需要调整（backward adjustment）
/// 如果不调整，某些技术指标（如移动平均）会突然断裂
fn adjust_for_corporate_action(
    historical_prices: &mut [i64],
    split_ratio_num: i64,   // 分子（如 2:1 split → 2）
    split_ratio_den: i64,   // 分母（如 2:1 split → 1）
    dividend_per_share: i64, // 每股股息
) {
    for price in historical_prices.iter_mut() {
        // 先做拆分调整
        *price = *price * split_ratio_den / split_ratio_num;
        // 再减股息（股息会影响价格）
        *price = *price - dividend_per_share;
    }
}
```

## 5. 实时行情接收架构：UDP 组播与订单簿重建

### 5.1 UDP 组播接收

```
交易所市场数据结构（UDP 组播模式）：

┌──────────┐    UDP Multicast     ┌──────────────┐
│ Exchange │ ─────────────────→   │  Colo Server │
│ Matching │   (MoldUDP/NASDAQ)   │  (你的机器)  │
│ Engine   │                      │              │
└──────────┘                      └──────┬───────┘
                                         │
                                  ┌──────┴───────┐
                                  │ Market Data  │
                                  │ Feed Handler │
                                  └──────────────┘

UDP 的关键问题：
  1. 不保证送达 → 需要 sequence number gap 检测
  2. 不保证顺序 → 需要重新排序（虽然同机房通常有序）
  3. 不保证不重复 → 需要去重
```

```rust
use std::collections::VecDeque;
use std::net::UdpSocket;

/// UDP 组播市场数据接收器
struct MulticastFeedReceiver {
    /// 监听的 socket
    socket: UdpSocket,
    /// 当前期望的下一个序列号
    expected_seq: u64,
    /// 收到的最高序列号（可能会有乱序）
    max_seen_seq: u64,
    /// 乱序缓存：缓存那些比 expected_seq 大但还没到顺序的消息
    out_of_order_buffer: VecDeque<(u64, Vec<u8>)>,
    /// gap 检测：如果在 N 条消息内还没收到缺失的消息 → 触发重传
    gap_detection_threshold: u64,
}

impl MulticastFeedReceiver {
    fn new(multicast_addr: &str, interface_addr: &str) -> std::io::Result<Self> {
        let socket = UdpSocket::bind("0.0.0.0:0")?;
        // 加入组播组
        socket.join_multicast_v4(
            &multicast_addr.parse().unwrap(),
            &interface_addr.parse().unwrap(),
        )?;
        // 设置接收缓冲区（市场数据突发量大，需要较大的缓冲区）
        socket.set_read_buffer(64 * 1024 * 1024)?; // 64 MB
        // 设置非阻塞（我们不希望 recv 阻塞整个线程）
        socket.set_nonblocking(true)?;

        Ok(Self {
            socket,
            expected_seq: 1,  // 假设序列号从 1 开始
            max_seen_seq: 0,
            out_of_order_buffer: VecDeque::new(),
            gap_detection_threshold: 1000, // 1000 条消息后触发重传
        })
    }

    /// 处理接收到的消息
    fn process_message(&mut self, seq_num: u64, payload: &[u8]) -> ProcessResult {
        if seq_num < self.expected_seq {
            // 重复消息 → 丢弃
            return ProcessResult::Duplicate;
        }

        if seq_num > self.max_seen_seq {
            self.max_seen_seq = seq_num;
        }

        if seq_num == self.expected_seq {
            // 正好是下一条 → 直接处理
            self.expected_seq += 1;

            // 检查乱序缓存中是否有后续消息可用
            while let Some(front) = self.out_of_order_buffer.front() {
                if front.0 == self.expected_seq {
                    // 缓存中的消息可以"释放"了
                    // 实际处理这个释放的消息（在真实系统中这里会调用回调）
                    self.expected_seq += 1;
                    self.out_of_order_buffer.pop_front();
                } else {
                    break;
                }
            }

            ProcessResult::InOrder
        } else {
            // 乱序到达 → 缓存到乱序缓冲区
            self.out_of_order_buffer.push_back((seq_num, payload.to_vec()));
            // 保持乱序缓冲区有序（按序列号排序）
            // 简化：每次都排序（生产环境用 BTreeMap）
            self.out_of_order_buffer.make_contiguous()
                .sort_by_key(|(seq, _)| *seq);

            // Gap 检测
            if seq_num - self.expected_seq > self.gap_detection_threshold {
                // 触发重传请求（在实际系统中向 exchange/source 请求缺失的消息）
                return ProcessResult::GapDetected {
                    missing_from: self.expected_seq,
                    missing_to: seq_num - 1,
                };
            }

            ProcessResult::OutOfOrder
        }
    }
}

enum ProcessResult {
    InOrder,           // 有序到达，已处理
    OutOfOrder,        // 乱序到达，已缓存
    Duplicate,         // 重复消息，已丢弃
    GapDetected {      // 发现 gap，需要重传
        missing_from: u64,
        missing_to: u64,
    },
}
```

### 5.2 从增量更新重建订单簿

```
订单簿重建的核心挑战：

交易所不会每秒发一次"完整快照"
而是发增量更新：Order#123 added at 150.00, Order#456 cancelled...
你必须自己维护状态

如果丢包了 → 你的订单簿就不准了
如果启动时没有初始快照 → 你的订单簿就是空的

解决方案：
  1. 订阅快照通道 + 增量通道
  2. 从快照开始初始化订单簿
  3. 切换到增量更新维护订单簿
  4. 如果怀疑状态不一致 → 重新获取快照
```

```rust
/// 从增量更新重建订单簿的核心算法
struct OrderBookRebuilder {
    /// 维护的订单簿（BTreeMap: price → Level）
    bids: BTreeMap<i64, u64>,  // price → total_quantity
    asks: BTreeMap<i64, u64>,
    /// 当前快照序列号（只有当增量序列号 > 快照序列号时，增量才有效）
    snapshot_seq: u64,
    /// 最后处理的增量序列号
    last_seq: u64,
}

enum IncrementalUpdate {
    /// 新增订单：在指定价位增加指定数量
    AddOrder {
        price: i64,
        quantity: u64,
        side: Side,
        seq_num: u64,
    },
    /// 修改订单：在指定价位调整数量
    ModifyOrder {
        price: i64,
        new_quantity: u64,
        side: Side,
        seq_num: u64,
    },
    /// 取消订单：在指定位移除指定数量
    CancelOrder {
        price: i64,
        quantity: u64,
        side: Side,
        seq_num: u64,
    },
}

impl OrderBookRebuilder {
    fn new() -> Self {
        Self {
            bids: BTreeMap::new(),
            asks: BTreeMap::new(),
            snapshot_seq: 0,
            last_seq: 0,
        }
    }

    /// 从快照初始化订单簿
    fn apply_snapshot(&mut self, bids: BTreeMap<i64, u64>,
                       asks: BTreeMap<i64, u64>, seq_num: u64) {
        self.bids = bids;
        self.asks = asks;
        self.snapshot_seq = seq_num;
        self.last_seq = seq_num;
    }

    /// 应用一条增量更新
    fn apply_incremental(&mut self, update: IncrementalUpdate) -> bool {
        let IncrementalUpdate { price, quantity, side, seq_num } = update;

        // 只有当增量序列号 > 快照序列号时才应用
        if seq_num <= self.snapshot_seq || seq_num <= self.last_seq {
            return false; // 过期或重复的更新，忽略
        }

        let book = match side {
            Side::Buy => &mut self.bids,
            Side::Sell => &mut self.asks,
        };

        match update {
            IncrementalUpdate::AddOrder { .. } => {
                // 新增：累加到该价位
                *book.entry(price).or_insert(0) += quantity;
            }
            IncrementalUpdate::ModifyOrder { new_quantity, .. } => {
                // 修改：替换该价位的数量
                if new_quantity > 0 {
                    book.insert(price, new_quantity);
                } else {
                    book.remove(&price); // 数量为 0 → 删除该价位
                }
            }
            IncrementalUpdate::CancelOrder { .. } => {
                // 取消：从该价位减去指定数量
                book.entry(price).and_modify(|q| {
                    *q = q.saturating_sub(quantity);
                });
                // 如果数量归零，删除该价位
                if book.get(&price).map_or(true, |&q| q == 0) {
                    book.remove(&price);
                }
            }
        }

        self.last_seq = seq_num;
        true
    }

    /// 最佳买卖价
    fn best_bid_ask(&self) -> (Option<i64>, Option<i64>) {
        let bid = self.bids.iter().next_back().map(|(&p, _)| p);
        let ask = self.asks.iter().next().map(|(&p, _)| p);
        (bid, ask)
    }
}
```
