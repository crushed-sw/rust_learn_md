# 10.15 Solana 开发：高性能区块链与 Anchor 框架

> **[Track C: Web3]** Solana 是目前吞吐量最高的 L1 区块链（理论峰值 65,000 TPS），也是 Rust 在区块链领域的最强展示——从核心协议到智能合约（Solana 称为"程序"），整个技术栈都是 Rust。本节从 Solana 的独特架构开始，深入账户模型、Anchor 框架、SPL Token 标准，以及测试和高级模式。你会理解为什么 Solana 选择了"用 Rust 写一切"，以及 Anchor 如何把区块链开发的复杂度降低 10 倍。

**领域背景：** Solana 与以太坊的根本区别在于"扩展哲学"。以太坊走 Rollup 路线（L1 做安全层，L2 做执行层），而 Solana 走"单层极致优化"路线——用 Proof of History 作为去中心化时钟，用 Sealevel 实现交易并行执行，用 Gulf Stream 做无内存池交易转发。这套架构极度依赖 Rust 的零成本抽象和确定性——任何一点不确定性都会让 PoH 时钟碎裂，导致整个共识崩溃。

## 1. Solana 的独特架构

### 1.1 Proof of History（PoH）：去中心化时钟

PoH 是 Solana 最具原创性的创新——在区块链中嵌入一个可验证的时间流逝证明。

```
┌─────────────────────────────────────────────────────────────────┐
│  Proof of History：SHA-256 哈希链作为可验证延迟函数              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  hash1 = SHA256(seed)                                           │
│  hash2 = SHA256(hash1)                                          │
│  hash3 = SHA256(hash2)                                          │
│  ...                                                            │
│  hashN = SHA256(hashN-1)                                        │
│                                                                 │
│  关键性质：                                                       │
│  1. 串行性：每次哈希依赖前一次结果，不能并行加速                    │
│  2. 可验证：任何人都可以快速验证 hashN 是从 seed 串行演算来的       │
│  3. 时间锚定：如果在 hashK 位置记录了某个事件，可以证明该事件       │
│     发生在 "第K次哈希的时刻"                                     │
│                                                                 │
│  为什么 SHA-256？                                                │
│    → 单核最多 ~2M hashes/秒（硬件限制）                           │
│    → 即使有 100 台机器，每台也要串行运行自己的 PoH                 │
│    → PoH 不替代共识，它为 PBFT 提供可信的时间源                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```rust
// proof_of_history.rs —— PoH 的核心：SHA-256 可验证延迟函数
// 这个 ~30 行的代码就是 Solana 整个网络"时钟"的密码学基础

use sha2::{Sha256, Digest};

struct ProofOfHistory {
    current_hash: [u8; 32],
    tick_count: u64,
    events: Vec<(u64, Vec<u8>)>,  // (tick_index, event_data)
}

impl ProofOfHistory {
    fn new(seed: &[u8; 32]) -> Self {
        Self {
            current_hash: *seed,
            tick_count: 0,
            events: Vec::new(),
        }
    }

    /// 推进一步 PoH：SHA256(current_hash) → new_hash
    /// 这是不可并行化的操作——必须等前一步的哈希结果
    fn tick(&mut self) {
        let mut hasher = Sha256::new();
        hasher.update(&self.current_hash);
        self.current_hash = hasher.finalize().into();
        self.tick_count += 1;
    }

    /// 推进一步并将事件混入哈希（事件"锚定"到时间点）
    fn tick_with_event(&mut self, event_data: &[u8]) {
        let mut hasher = Sha256::new();
        hasher.update(&self.current_hash);
        hasher.update(event_data);  // 事件数据混入哈希流中
        self.current_hash = hasher.finalize().into();
        self.events.push((self.tick_count, event_data.to_vec()));
        self.tick_count += 1;
    }

    /// 运行 N 个 tick（批量推进）
    fn advance(&mut self, n: u64) {
        for _ in 0..n {
            self.tick();
        }
    }

    /// 验证：给定 (seed, N, 声称的 hashN)，验证是否正确
    fn verify(seed: &[u8; 32], ticks: u64, expected_hash: &[u8; 32]) -> bool {
        let mut hash = *seed;
        for _ in 0..ticks {
            let mut hasher = Sha256::new();
            hasher.update(&hash);
            hash = hasher.finalize().into();
        }
        &hash == expected_hash
    }
}
```

**PoH 的意义：** 在经典 PoW/PoS 中，节点需要大量通信来商定时间。"哪个区块在哪个时刻产生"需要投票决定。PoH 把这个时间同步问题变成了密码学问题——时钟内嵌在数据流中，任何节点看到一段 PoH 序列都能推断出"事件 A 在事件 B 之前发生"。这减少了 90% 的共识通信开销，是 Solana 能达到千级 TPS 的关键。

### 1.2 Tower BFT：PoH 增强的 PBFT

Tower BFT 是 PBFT（Practical Byzantine Fault Tolerance）的 Solana 变体。传统 PBFT 需要 O(N^2) 的消息量来同步时钟，Tower BFT 把 PoH 作为外部可信时间源，将通信复杂度降到 O(N)。

```
┌───────────────────────────────────────────────────────────────┐
│  Tower BFT 共识流程（简化）                                     │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  1. 领导者（Leader）产生 PoH 序列 → 它"在时钟上讲话"          │
│  2. 领导者将交易打包进 PoH 序列中 → 每个交易都有精确时间戳    │
│  3. 验证者（Validators）接收到 PoH 序列 + 交易                │
│     → 验证 PoH 序列是否正确（重新跑一遍 SHA-256 链）          │
│     → 验证交易签名和执行结果                                   │
│  4. 验证者对区块投票（用 PoH 中的 tick 作为超时时钟）         │
│     → 任何验证者在 Y 个 tick 内不投票 = 超时                  │
│  5. 需要 2/3 投票数才能最终确认                                 │
│                                                               │
│  关键：不需要协议层面的心跳——PoH 本身就是心跳                 │
│       这就是 Tower BFT "更快"的原因                           │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 1.3 Sealevel：并行交易执行引擎

以太坊的 EVM 串行执行交易（每笔交易可能修改任意状态变量，无法预知依赖关系）。Solana 的 Sealevel 让每笔交易声明"我要读写哪些账户"，然后像数据库的事务调度器一样并行执行不冲突的交易。

```
┌─────────────────────────────────────────────────────────────────┐
│  Sealevel 并行执行                                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Tx1: read(AccountA), write(AccountB)                           │
│  Tx2: read(AccountC), write(AccountD)                           │
│  Tx3: read(AccountB), write(AccountA)                           │
│  Tx4: read(AccountE), write(AccountE)                           │
│                                                                 │
│  依赖分析:                                                       │
│    Tx1 和 Tx3 → 冲突（都读写 AccountA 和 AccountB）→ 顺序执行    │
│    Tx1 和 Tx2 → 无冲突（读写的账户完全不重叠）→ 并行执行         │
│    Tx1 和 Tx4 → 无冲突 → 并行执行                                │
│                                                                 │
│  结果: Tx1+T2+T4 可以并行执行，只 T3 需要等 Tx1 之后             │
│  4 笔交易 → 2 轮（而非串行的 4 轮）                              │
│                                                                 │
│  这正是 Solana 远超 EVM 吞吐量的核心原因：                       │
│    不是 CPU 更快，而是"同时做不互相干扰的事"                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.4 Gulf Stream：无内存池转发

以太坊的经济模型中，用户交易先进入"内存池"（mempool，即 pending 交易池），矿工从内存池中选交易打包。这导致了 MEV 问题（见 10.16 第 5 节）。Solana 的 Gulf Stream 协议直接让交易生产者（用户钱包/RPC 节点）将交易转发给下一轮的领导者，而不等待当前区块。这同时提高了吞吐量并减少了 MEV 机会。

## 2. Solana 账户模型：万物皆账户

### 2.1 Account 数据结构

在 Solana 中，**一切**都是账户——你的钱包是账户，智能合约（程序）是账户，程序存储的数据也是账户。这和以太坊的"合约账户 + EOA"双模型全然不同。

```rust
// solana_account_model.rs —— Solana 账户的底层结构
// 注意：这是概念展示，实际 Solana 运行时用更紧凑的布局

/// Solana 的核心数据结构：Account
/// 链上所有信息都存储在不同类型的 Account 中
#[derive(Debug, Clone)]
struct SolanaAccount {
    /// 账户拥有的 lamports（1 SOL = 10^9 lamports）
    /// 每个账户都需要一定数量的 lamports 作为 rent（租金）
    lamports: u64,

    /// 账户数据（最大 10MB，但实际程序通常在 1KB~10KB）
    /// 这是通用的字节数组——具体含义取决于 account 的类型
    data: Vec<u8>,

    /// 管理此账户的程序（Program）的公钥
    /// 只有这个程序可以修改账户的 data 和扣减 lamports
    owner: [u8; 32],

    /// 该账户是否为可执行账户（即它是一个 Program）
    /// true → 这是一个链上程序（智能合约）
    /// false → 这是一个数据账户（钱包 or 程序存储）
    executable: bool,

    /// 本账户下个需要支付 rent 的 epoch
    /// rent_epoch 机制：每 epoch（约 2 天）收取一次租金
    /// 如果账户的 lamports >= 2 年租金，则被标记为 rent-exempt
    rent_epoch: u64,
}

// 四种核心账户类型：
//
// 1. 系统账户（System Account / 钱包账户）
//    owner = System Program, executable = false
//    data = 空，lamports = 用户余额
//    功能：持有 SOL，签署交易
//
// 2. 程序账户（Program Account / 智能合约）
//    owner = BPF Loader, executable = true
//    data = BPF 字节码（编译后的 Rust→BPF 代码）
//    lamports = 由部署者支付的 rent-exempt 金额
//
// 3. 程序数据账户（Program Derived Data Account）
//    owner = 某个 Program（非 BPF Loader）
//    data = 程序定义的数据结构（经过 Borsh 序列化）
//    lamports = 至少 rent-exempt 阈值
//    功能：程序用它存储持久化状态
//
// 4. Token 账户（SPL Token Account）
//    owner = Token Program
//    data = Borsh 序列化的 { mint, owner, amount }
//    功能：记录"某个地址在某个 mint 下有多少代币"
```

### 2.2 Rent（租金）与 Rent Exemption

Solana 的存储不是免费的——账户需要持有足够 SOL 来 cover 存储成本。这种设计防止了"状态膨胀攻击"（攻击者创建海量账户耗尽存储）。

```rust
// rent_exemption.rs —— 理解 Solana 的租金模型

/// 计算账户需要多少 lamports 才能获得 rent-exempt 资格
fn rent_exempt_minimum_balance(data_len: usize) -> u64 {
    // Solana 的 rent 计算公式（简化版）：
    //   rent = (data_len + ACCOUNT_STORAGE_OVERHEAD) * LAMPORTS_PER_BYTE_YEAR
    //         * EXEMPTION_THRESHOLD_YEARS
    //
    // ACCOUNT_STORAGE_OVERHEAD = 128 字节（账户元数据固定开销）
    // LAMPORTS_PER_BYTE_YEAR ≈ 3480 lamports/字节/年
    // EXEMPTION_THRESHOLD = 2 年（从 epoch 数换算）
    //
    // 结果大小参考（2024年实际值）：
    //   0 字节数据 → 约 0.0009 SOL
    //   10KB 数据  → 约 0.035 SOL
    //   1MB 数据   → 约 3.5 SOL
    //   10MB 数据  → 约 35 SOL (程序账户不需要这么多数据)

    let storage_overhead = 128u64;
    let lamports_per_byte_year: u64 = 3480;
    let exemption_years: u64 = 2;

    (data_len as u64 + storage_overhead) * lamports_per_byte_year * exemption_years
}

// Rent-exempt 账户：
//   → 余额 >= rent_exempt_minimum_balance(data.len())
//   → 永远不会被收取租金（即使 10 年不用也保持全额）
//   → 几乎所有 dApp 都设计为 rent-exempt（简化管理）

// 非 rent-exempt 账户（一般只有临时的系统账户）：
//   → 每个 epoch 被扣掉一部分 lamports
//   → 如果余额降到 0，账户被删除
//   → 支付 rent 的账户不能作为 program 的存储账户
```

### 2.3 Borsh 序列化

Solana 使用 Borsh（Binary Object Representation Serializer for Hashing）作为标准序列化格式。Borsh 是确定性的：同样的数据在任何平台、任何编译器版本下都产生相同的字节序列。这对区块链至关重要——验证者运行在不同机器上，必须对"这个状态是否等于那个哈希"达成完全一致。

```rust
// borsh_serialization.rs —— Borsh 序列化示例
use borsh::{BorshSerialize, BorshDeserialize};

/// 一个简单计数器程序的状态
#[derive(BorshSerialize, BorshDeserialize, Debug, Clone, PartialEq)]
struct CounterState {
    count: u64,
    owner: [u8; 32],       // 公钥 (32 字节)
    bump: u8,               // PDA bump seed
}

/// 一个投票程序的状态
#[derive(BorshSerialize, BorshDeserialize, Debug, Clone, PartialEq)]
struct VotingState {
    proposal: String,
    yes_votes: u64,
    no_votes: u64,
    is_active: bool,
}

fn borsh_demo() {
    // 序列化
    let counter = CounterState {
        count: 42,
        owner: [1u8; 32],
        bump: 255,
    };
    let serialized = borsh::to_vec(&counter).unwrap();
    println!("CounterState 序列化后: {} 字节", serialized.len());
    // 8 (u64) + 32 ([u8;32]) + 1 (u8) = 41 字节

    // 反序列化
    let deserialized: CounterState = borsh::from_slice(&serialized).unwrap();
    assert_eq!(deserialized, counter);
    println!("✓ Borsh 反序列化结果与原始数据一致");

    // 确定性验证
    let serialized_again = borsh::to_vec(&counter).unwrap();
    assert_eq!(serialized, serialized_again);
    println!("✓ Borsh 序列化是确定性的（两次结果完全相同）");
}

// 为什么不选 JSON？
//  → 字段顺序可能变化（虽然 serde 有 serialize_with 可以锁定）
//  → 浮点数表示可能不一致
//  → 编码开销大（用 UTF-8 表示二进制数据效率低）
//
// 为什么不选 Protobuf？
//  → protobuf 不保证确定性（map 字段的迭代顺序未定义）
//  → 需要 .proto 文件和代码生成
//  → Borsh 更轻量（只需 derive 宏）
```

### 2.4 Program Derived Addresses (PDAs)

PDA 是 Solana 最具特色的概念之一。它是"从程序 ID 和 seeds 确定性地推导出一个地址，且保证该地址没有对应的私钥"。

```
┌─────────────────────────────────────────────────────────────────┐
│  Program Derived Address (PDA)                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  PDA = hash256(program_id || seeds || bump_seed)                │
│                                                                 │
│  输入:                                                           │
│    program_id: 你的程序（合约）的公钥                             │
│    seeds: 任意数量的字节序列（如 b"vote", 用户公钥）              │
│    bump_seed: 一个 u8 值 (0-255)，用于将结果"推"出 ed25519 曲线 │
│                                                                 │
│  条件: 结果公钥不在 ed25519 曲线上                               │
│    → 这意味着 NOBODY 拥有私钥                                    │
│    → 只有该 program_id 对应的程序可以"签名"（通过 invoke_signed）  │
│                                                                 │
│  例子:                                                           │
│    VotingProgram 为每个用户创建投票记录                           │
│    PDA = find_pda(VotingProgram, [b"vote", user_pubkey])        │
│    每个用户有唯一且确定的投票账户地址                             │
│    只有 VotingProgram 可以修改该账户                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```rust
// pda_simulation.rs —— PDA 的查找和验证逻辑

// 模拟 Solana 的 find_program_address 方法
fn find_program_address(seeds: &[&[u8]], program_id: &[u8; 32]) -> ([u8; 32], u8) {
    // bump seed 从 255 开始递减
    for bump in (0u8..=255).rev() {
        let mut hasher = Sha256::new();
        // 将所有种子 + bump + program_id 合并哈希
        for seed in seeds {
            hasher.update(seed);
        }
        hasher.update(&[bump]);
        hasher.update(program_id);
        hasher.update(b"ProgramDerivedAddress");
        let hash: [u8; 32] = hasher.finalize().into();

        // 检查结果是否在 ed25519 曲线上
        // 在实际 Solana 中，使用 curve25519_dalek 做这个检查
        // 这里简化：假设 hash[0] 的最高位为 0 表示"在曲线外"（不精确）
        if !is_on_ed25519_curve(&hash) {
            return (hash, bump);
        }
    }
    panic!("超过 256 次尝试仍无法找到有效的 PDA（概率极低）");
}

// 模拟 ed25519 curve 检查
fn is_on_ed25519_curve(point: &[u8; 32]) -> bool {
    // 在真正的 Solana 实现中：
    // 1. 将 32 字节解释为ed25519 曲线上的 y 坐标
    // 2. 尝试计算对应的 x 坐标
    // 3. 如果 x 有解 → 该点在曲线上 → 有人可能拥有私钥 → 不是有效的 PDA
    // 这确保了 PDA 总是在曲线外 → 绝对没有私钥
    // 简化版：约 50% 的点在曲线外，所以平均 ~2 次尝试就能找到
    Sha256::digest(point)[0] & 1 == 1  // 不精确的简化
}
```

**PDA 解决的核心问题：** 在没有 PDA 之前，程序只能用"自己有一个私钥"的方式管理存储——这很危险（私钥泄露 = 所有用户资金被偷）。PDA 让程序"天生就没有私钥"，但可以通过 `invoke_signed` 以程序身份授权操作——这是 Solana 安全模型最优雅的部分。

## 3. Anchor 框架深度解析

Anchor 是 Solana 开发的"Rails/Spring/Django"——它消除了 90% 的模板代码，特别是安全检查（权限验证）的模板代码。在原生 Solana 开发中，你需要手动验证每个指令的调用者是否有权操作目标账户；Anchor 通过 `#[derive(Accounts)]` 自动生成这些检查。

### 3.1 入门：计数器程序

```rust
// counter_program.rs —— Anchor 框架下的计数器程序（完整可部署）
// 
// 部署流程：
//   anchor init counter
//   cd counter
//   anchor build    → 编译 Rust → BPF 字节码
//   anchor deploy   → 部署到 devnet/localnet
//   anchor test     → 运行 TypeScript 测试

use anchor_lang::prelude::*;

// 声明程序 ID（anchor build 时自动生成并替换）
declare_id!("Fg6PaFpoGXkYsidMpWTK6W2BeZ7FEfcYkg476zPFsLnS");

#[program]
pub mod counter {
    use super::*;

    /// 初始化计数器——创建新的计数器账户
    pub fn initialize(ctx: Context<Initialize>, initial_value: u64) -> Result<()> {
        let counter = &mut ctx.accounts.counter;
        counter.count = initial_value;
        counter.authority = ctx.accounts.user.key();  // 记录谁创建了它
        counter.bump = ctx.bumps.counter;             // 保存 bump seed 供后续使用
        msg!("计数器已初始化，初始值: {}", initial_value);
        Ok(())
    }

    /// 增加计数——只允许 authority 调用
    pub fn increment(ctx: Context<Update>) -> Result<()> {
        let counter = &mut ctx.accounts.counter;
        counter.count = counter.count.checked_add(1)
            .ok_or(ErrorCode::Overflow)?;
        msg!("计数增加到: {}", counter.count);
        Ok(())
    }

    /// 减少计数——只允许 authority 调用
    pub fn decrement(ctx: Context<Update>) -> Result<()> {
        let counter = &mut ctx.accounts.counter;
        counter.count = counter.count.checked_sub(1)
            .ok_or(ErrorCode::Underflow)?;
        msg!("计数减少到: {}", counter.count);
        Ok(())
    }

    /// 关闭计数器——退还 rent 给 authority
    pub fn close(ctx: Context<Close>) -> Result<()> {
        msg!("计数器已关闭，剩余 lamports 退还给 authority");
        Ok(())
    }
}

    pub user: Signer<'info>,
}
```

```rust
declare_id!("VotEproGramAddr111111111111111111111111111111");

pub mod voting {
    use super::*;

    /// 初始化投票——创建投票账户（PDA）
    pub fn create_vote(
        ctx: Context<CreateVote>,
        proposal: String,
        end_time: i64,  // Unix 时间戳，投票截止时间
    ) -> Result<()> {
        let vote_account = &mut ctx.accounts.vote_account;
        vote_account.creator = ctx.accounts.creator.key();
        vote_account.proposal = proposal;
        vote_account.yes_count = 0;
        vote_account.no_count = 0;
        vote_account.end_time = end_time;
        vote_account.is_active = true;
        vote_account.bump = ctx.bumps.vote_account;
        msg!("投票已创建");
        Ok(())
    }

    /// 投赞成票
    pub fn vote_yes(ctx: Context<Vote>) -> Result<()> {
        let vote = &mut ctx.accounts.vote_account;
        // 检查投票是否已结束
        let clock = Clock::get()?;
        require!(clock.unix_timestamp < vote.end_time, ErrorCode::VoteEnded);
        require!(vote.is_active, ErrorCode::VoteEnded);
        vote.yes_count = vote.yes_count.checked_add(1)
            .ok_or(ErrorCode::Overflow)?;
        msg!("赞成票 +1，当前: {}", vote.yes_count);
        Ok(())
    }

    /// 投反对票
    pub fn vote_no(ctx: Context<Vote>) -> Result<()> {
        let vote = &mut ctx.accounts.vote_account;
        let clock = Clock::get()?;
        require!(clock.unix_timestamp < vote.end_time, ErrorCode::VoteEnded);
        require!(vote.is_active, ErrorCode::VoteEnded);
        vote.no_count = vote.no_count.checked_add(1)
            .ok_or(ErrorCode::Overflow)?;
        msg!("反对票 +1，当前: {}", vote.no_count);
        Ok(())
    }

    /// 结束投票（只有创建者可以调用）
    pub fn end_vote(ctx: Context<EndVote>) -> Result<()> {
        let vote = &mut ctx.accounts.vote_account;
        require!(vote.is_active, ErrorCode::VoteEnded);
        require!(
            ctx.accounts.creator.key() == vote.creator,
            ErrorCode::NotCreator
        );
        vote.is_active = false;
        msg!("投票已结束。结果: 赞成 {} vs 反对 {}", vote.yes_count, vote.no_count);
        Ok(())
    }
}
```

### 3.3 CPI（Cross-Program Invocation）：程序调用程序

一个 Solana 程序可以调用另一个 Solana 程序——这就是可组合性（composability）。

```rust
// cpi_example.rs —— Cross-Program Invocation (CPI) 示例
// 场景：你的程序需要向用户支付 SPL Token（需要调用 Token Program）

use anchor_lang::prelude::*;
use anchor_spl::token::{self, Token, TokenAccount, Transfer};

#[program]
pub mod reward_distributor {
    use super::*;

    /// 发放奖励：从程序的代币账户向用户转账
    pub fn distribute_reward(ctx: Context<DistributeReward>, amount: u64) -> Result<()> {
        // 构建 CPI 上下文：调用 Token Program 的 transfer 指令
        let cpi_accounts = Transfer {
            from: ctx.accounts.program_token_account.to_account_info(),
            to: ctx.accounts.user_token_account.to_account_info(),
            authority: ctx.accounts.program_signer.to_account_info(),  // PDA 签名
        };

        // 创建 CPI 上下文（指定被调用程序）
        let cpi_ctx = CpiContext::new(
            ctx.accounts.token_program.to_account_info(),
            cpi_accounts,
        );

        // 执行 CPI：调用 Token Program 的 transfer
        token::transfer(cpi_ctx, amount)?;

        msg!("已从程序代币账户向用户转账 {}", amount);
        Ok(())
    }
}

#[derive(Accounts)]
pub struct DistributeReward<'info> {
    /// 程序的代币账户（from，需要为 PDA 的 seeds 签名）
    #[account(mut)]
    pub program_token_account: Account<'info, TokenAccount>,

    /// 用户的代币账户（to）
    #[account(mut)]
    pub user_token_account: Account<'info, TokenAccount>,

    /// 程序签名者 PDA（控制 program_token_account 的 authority）
    /// CHECK: 这是 PDA，不需要 data 反序列化
    #[account(
        seeds = [b"reward_vault"],
        bump,
    )]
    pub program_signer: AccountInfo<'info>,

    /// Token Program（被调用的程序）
    pub token_program: Program<'info, Token>,

    /// 用户（支付交易费）
    #[account(mut)]
    pub user: Signer<'info>,
}

// CPI 的精髓：
//   程序 A 在程序 B 的"管辖范围内"做操作
//   本例中：我们的 Reward 程序调用了 Token 程序来转账
//   因为 program_signer 是 PDA → Token 程序验证 authority = PDA → 允许操作
//   整个过程中 user 只需要签名一笔交易，即可触发跨程序的复杂操作
```

## 6. 高级模式与安全陷阱

### 6.1 PDA 作为程序状态

```rust
// pda_state_pattern.rs —— PDA 用于存储可升级程序的状态

// 关键设计：将状态和代码解耦
//   程序账户（executable）: 存储代码 → 可以升级
//   状态 PDA 账户: 存储数据 → 独立于代码版本

#[account]
pub struct ProgramState {
    /// 全局状态数据，独立于程序代码
    pub admin: Pubkey,            // 管理员公钥
    pub total_users: u64,
    pub paused: bool,
    pub upgrade_authority: Pubkey, // 谁有权升级程序
    pub bump: u8,
}

// 为什么这样设计？
//   → 程序升级时，BPF Loader 只替换程序账户的 data（代码）
//   → 状态 PDA 账户不受影响（它的 owner 还是你的程序，不受升级影响）
//   → 这也是为什么需要 upgrade_authority 字段：防止恶意升级
```

### 6.2 重初始化攻击（Reinitialization Attack）

这是 Solana 开发中最常见的安全漏洞之一。

```rust
// reinitialization_attack.rs —— 演示重初始化攻击及防御

// 漏洞代码（不要这样写！）
#[derive(Accounts)]
pub struct InsecureInit<'info> {
    #[account(
        init,  // ← 没有检查账户是否已经初始化
        payer = user,
        space = 8 + 100,
    )]
    pub state: Account<'info, ProgramState>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>,
}

// 攻击场景：
//   1. 用户正常调用 initialize → 创建 state 账户，存入数据
//   2. 攻击者再次调用 initialize（用不同的 user）
//   3. 因为是 init（而非 init_if_needed），Anchor 会尝试再次创建
//      → 如果 seed 相同 → 与已有账户碰撞失败（Anchor 保护）
//      → 如果 seed 不同 → 创建了新的 state 账户
//   关键在于：攻击者可以"覆盖"程序的全局状态
//
// 正确做法：区分 Init 和 Update 指令
#[derive(Accounts)]
pub struct SecureInit<'info> {
    #[account(
        init,
        payer = user,
        space = 8 + ProgramState::LEN,
        seeds = [b"global_state"],  // 固定的 seeds → PDA 地址固定
        bump                        // 只要 seeds 相同，任何人都无法创建第二个
    )]
    pub state: Account<'info, ProgramState>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>,
}

// 更强的保护：在 Init 指令中检查 data 是否为空
#[derive(Accounts)]
pub struct SafeInit<'info> {
    #[account(
        init,
        payer = user,
        space = 8 + ProgramState::LEN,
        seeds = [b"global_state"],
        bump
    )]
    pub state: Account<'info, ProgramState>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>,
}
```

### 6.3 可升级程序（BPF Upgradeable Loader）

Solana 程序部署后可以升级——这是通过 BPF Upgradeable Loader（一个特殊的链上程序）实现的。

```
┌─────────────────────────────────────────────────────────────────┐
│  程序升级流程（BPF Upgradeable Loader）                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Program Account                  Program Data Account           │
│  ┌─────────────────┐            ┌─────────────────────┐         │
│  │ executable: true │──指向──→  │ 程序字节码 (BPF)     │         │
│  │ owner: BPFLoader │           │ upgrade_authority    │         │
│  │ data: 很小        │           │ slot (版本号)         │         │
│  └─────────────────┘            └─────────────────────┘         │
│                                                                 │
│  升级步骤：                                                       │
│    1. 构建新版本的 BPF 字节码（anchor build）                    │
│    2. 用 upgrade_authority 密钥签名升级交易                       │
│    3. 将新字节码写入 Program Data Account                        │
│    4. 旧字节码被新字节码替换，但程序的状态 PDA 账户不受影响        │
│                                                                 │
│  安全考虑：                                                       │
│    → upgrade_authority 是最重要的密钥（可以替换你的程序逻辑）      │
│    → 生产级项目通常将 upgrade_authority 交给多重签名或 DAO         │
│    → 可以放弃升级权限（burn upgrade_authority）：                     │
│      solana program set-upgrade-authority <PROGRAM_ID> --final    │
│    → 放弃后程序变成不可变的（immutable）——像 Uniswap 那样         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
