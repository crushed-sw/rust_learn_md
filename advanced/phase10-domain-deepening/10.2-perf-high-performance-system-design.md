## 1. Thread-per-Core 架构：为什么共享是敌人

### 1.1 问题的根源

在多线程架构中，性能杀手不是"CPU 不够快"，而是 **"线程在等锁"**。当多个线程竞争同一份数据时，即使算法是 O(1) 的，锁争用也会让你退化到接近串行。更隐蔽的是：即使没有显式锁，**cache 一致性协议**也会在写共享数据时产生隐式的硬件锁。

```
┌──────────────────────────────────────────────────────────────┐
│        共享状态 vs 线程私有状态的性能差异                       │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  共享模型（糟糕）:              thread-per-core 模型（好）:     │
│  ┌────┐ ┌────┐ ┌────┐        ┌────┐ ┌────┐ ┌────┐           │
│  │ T0 │ │ T1 │ │ T2 │        │ T0 │ │ T1 │ │ T2 │           │
│  └──┬─┘ └──┬─┘ └──┬─┘        └──┬─┘ └──┬─┘ └──┬─┘           │
│     │      │      │             │      │      │               │
│     └──────┼──────┘             │      │      │               │
│     ┌──────▼──────┐        ┌────▼──┐ ┌──▼────┐ ┌──▼────┐     │
│     │  Shared     │        │ Data  │ │ Data  │ │ Data  │     │
│     │  State      │        │ Shard │ │ Shard │ │ Shard │     │
│     │  (锁争用！)  │        │  0    │ │  1    │ │  2    │     │
│     └─────────────┘        └───────┘ └───────┘ └───────┘     │
│                                                               │
│   每次访问都要抢锁             每个线程只访问自己的数据分区      │
│   cache line 频繁 invalidate   无锁、无 cache 一致性开销        │
└──────────────────────────────────────────────────────────────┘
```

Thread-per-core 的核心理念：**每个 CPU 核心拥有自己的数据分区、自己的队列、自己的工作集。线程之间不共享可变状态。**

### 1.2 CPU 亲和性：把线程"钉"在核上

线程迁移会丢失 L1/L2 cache 中的热数据。`core_affinity` crate 提供了绑定线程到特定核心的能力：

```rust
// Cargo.toml: core_affinity = "0.8"
use core_affinity::CoreId;

/// Thread-per-core 架构：将每个工作线程绑定到特定 CPU 核心
/// 每个核心拥有私有的数据分区——无锁、零争用
fn spawn_per_core_workers() -> Vec<std::thread::JoinHandle<()>> {
    let core_ids = core_affinity::get_core_ids().unwrap();
    let mut handles = Vec::new();

    for (i, core_id) in core_ids.iter().enumerate() {
        let core_id = *core_id;
        let handle = std::thread::spawn(move || {
            // 将当前线程的 CPU 亲和性设为此核心——OS 不会迁移它
            core_affinity::set_for_current(core_id);

            // 每个核心独立的本地数据——完全不与其他线程共享
            let mut local_data: Vec<u64> = vec![i as u64; 1024 * 1024];

            loop {
                for item in local_data.iter_mut() {
                    // 所有内存访问都是本地的——享受 L1/L2 cache
                    *item = item.wrapping_mul(1103515245).wrapping_add(12345);
                }
                // 实际应用中用 AtomicBool 检查退出信号
                break; // 示例中只执行一次
            }
        });
        handles.push(handle);
    }
    handles
}
```

### 1.3 数据分区策略

| 策略 | 原理 | 适用场景 |
|------|------|----------|
| 哈希分区 | `hash(key) % num_cores`，请求到对应核心 | 访问均匀的 KV 存储 |
| 范围分区 | `key in [A-M] -> core 0`，适合范围扫描 | 顺序访问需求 |
| 轮询分区 | 请求轮流发送到每个核心 | 无状态服务 |
| 自适应分区 | 根据负载动态调整 | 负载不均的场景 |

```rust
/// 哈希分区路由——将请求按 key 哈希发到正确核心
use std::hash::{Hash, Hasher};
use std::collections::hash_map::DefaultHasher;

struct HashRouter {
    core_channels: Vec<crossbeam::channel::Sender<Request>>,
    num_cores: usize,
}

struct Request { key: u64, payload: Vec<u8> }

impl HashRouter {
    fn route(&self, req: Request) {
        let mut h = DefaultHasher::new();
        req.key.hash(&mut h);
        let core_idx = (h.finish() as usize) % self.num_cores;
        self.core_channels[core_idx].send(req).unwrap();
    }
}
```

## 2. Run-to-Completion 模型：单线程事件循环

### 2.1 Work-Stealing vs Run-to-Completion

tokio 默认使用 work-stealing：多个 worker 共享任务队列，空闲 worker 从繁忙 worker "偷"任务。这在通用场景下极好——负载均衡自动。但在**极致延迟敏感**的场景下，任务迁移导致 cache miss，性能骤降。

Run-to-Completion 模型：**一个任务从开始到结束在同一个核心上执行，从不迁移。**

```
┌──────────────────────────────────────────────────────────────┐
│      Work-Stealing vs Run-to-Completion                       │
├──────────────────────────────────────────────────────────────┤
│  tokio Work-Stealing:           Run-to-Completion (io_uring): │
│  ┌─────┐ ┌─────┐ ┌─────┐       ┌─────┐ ┌─────┐ ┌─────┐      │
│  │Core0│ │Core1│ │Core2│       │Core0│ │Core1│ │Core2│      │
│  │ A,B │ │ ←偷A │ │ C,D │       │ A,B │ │ C,D │ │ E,F │      │
│  └─────┘ └─────┘ └─────┘       └─────┘ └─────┘ └─────┘      │
│  优点: 自动负载均衡            优点: 零 cache miss              │
│  缺点: 任务迁移 → cache miss   缺点: 负载不均需手动处理         │
│  适合: 通用 Web 服务          适合: 代理、DB、低延迟服务        │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 基于 io_uring 的单线程事件循环

io_uring 通过内核与用户态共享的环形缓冲区实现零系统调用 I/O，使单线程可以处理 10 万+ 并发连接。

```rust
/// 基于 io_uring 的单线程 Run-to-Completion 事件循环核心逻辑
/// 特点：无锁、热路径零分配、无任务迁移
use io_uring::{IoUring, opcode};

struct EventLoop {
    ring: IoUring,
}

impl EventLoop {
    fn new(entries: u32) -> std::io::Result<Self> {
        Ok(EventLoop { ring: IoUring::new(entries)? })
    }

    /// 核心循环——单线程，run-to-completion
    fn run(&mut self) -> std::io::Result<()> {
        loop {
            // 1. 提交 pending 的 I/O 操作（批量化）
            self.ring.submit()?;

            // 2. 等待至少一个 I/O 完成（<1μs 的典型等待时间）
            self.ring.submitter().submit_and_wait(1)?;

            // 3. 处理所有完成的 I/O（Completion Queue Entries）
            for cqe in self.ring.completion() {
                let user_data = cqe.user_data();
                let result = cqe.result();
                // 根据 user_data 找到连接上下文，提交下一个读/写
                // 全程零内存分配、零锁
                Self::handle_completion(user_data, result);
            }
        }
    }

    fn handle_completion(_user_data: u64, _result: i32) {
        // 处理完成事件，提交下一个 I/O
        unimplemented!()
    }
}
```

### 2.3 选择指南

| 场景 | 推荐模型 | 原因 |
|------|----------|------|
| HTTP API 服务 | work-stealing (tokio) | 负载均衡好，开发体验佳 |
| 数据库 / 缓存 | run-to-completion | 数据分区 + 本地处理 |
| HTTP 代理 / LB | run-to-completion | 延迟敏感，连接复用 |
| WebSocket 网关 | work-stealing | 长连接，任务多样 |
| 嵌入式 / IoT | run-to-completion | 确定性延迟 |

## 3. 分片 Map 模式：散开争用

### 3.1 RwLock<HashMap> 的瓶颈

当多线程频繁读写同一个 HashMap 时，即使使用 RwLock，高写比例下也会退化：

```rust
use std::sync::{Arc, RwLock};
use std::collections::HashMap;

// 反模式：全局一把大锁
type BigLockedMap = Arc<RwLock<HashMap<u64, u64>>>;

// 8 个线程对同一把锁竞争——吞吐约等于单线程
// 8 线程 × 1M insert → 全部串行化，锁开销远超业务逻辑
```

### 3.2 dashmap 的分片原理

`dashmap` 将 HashMap 分成 N 个分片（shard），每个分片有独立的 RwLock，大大降低锁争用：

```
insert(key="foo", value=42)
       │
       ▼
  hash("foo") → 分片索引 = hash % num_shards
       │
  ┌────┴────┬──────────┬──────────┬──────────┐
  │ Shard 0 │ Shard 1  │ Shard 2  │ Shard 3  │  ...
  │ RwLock  │ RwLock   │ RwLock   │ RwLock   │
  │ HashMap │ HashMap  │ HashMap  │ HashMap  │
  └─────────┴──────────┴──────────┴──────────┘

  不同 key 落在不同分片 → 不同线程可并行操作
  同一分片的冲突概率 ≈ 1/num_shards
```

```rust
use dashmap::DashMap;
use std::thread;

fn bench_dashmap() {
    // 默认 num_cpus * 4 个分片，每分片独立 RwLock
    let map = DashMap::new(); // DashMap 是 Sync 的，不需要 Arc

    let threads: Vec<_> = (0..8).map(|t| {
        thread::spawn(move || {
            for i in 0..1_000_000 {
                map.insert((t * 1_000_000 + i) as u64, i as u64 * 2);
            }
        })
    }).collect();

    for t in threads { t.join().unwrap(); }
    // 相比单一大锁，分片带来的加速比 ≈ num_shards / 2
}
```

### 3.3 手动实现分片 Map（理解原理）

```rust
use std::sync::RwLock;
use std::collections::HashMap;
use std::hash::{Hash, Hasher, DefaultHasher};

struct ShardedMap<K, V> {
    shards: Vec<RwLock<HashMap<K, V>>>,
    num_shards: usize,
}

impl<K, V> ShardedMap<K, V>
where K: Eq + Hash + Clone, V: Clone,
{
    fn new(shard_count: usize) -> Self {
        ShardedMap {
            shards: (0..shard_count).map(|_| RwLock::new(HashMap::new())).collect(),
            num_shards: shard_count,
        }
    }

    fn shard_of(&self, key: &K) -> usize {
        let mut h = DefaultHasher::new();
        key.hash(&mut h);
        (h.finish() as usize) % self.num_shards
    }

    fn insert(&self, key: K, value: V) {
        self.shards[self.shard_of(&key)].write().unwrap().insert(key, value);
    }

    fn get(&self, key: &K) -> Option<V> {
        self.shards[self.shard_of(key)].read().unwrap().get(key).cloned()
    }
}
```

### 3.4 分片效果分析

```
分片数量 vs 争用概率（8 线程）：

N = 1  （不分片）: 争用 100%，吞吐 ≈ 单线程
N = 8  （1×线程）: 冲突概率 ~1/8，吞吐 ~6× 单线程
N = 64 （8×线程）: 冲突概率 ~1/64，吞吐接近 8× 单线程

经验规则：
→ 分片数 = CPU 核心数 × 4~16 是良好起点
→ 过多分片不会提升性能（锁开销已可忽略），只在浪费内存
→ 热点 key 会突破分片保护——需要应用层热点检测
```

## 4. 对象池设计：分配 vs 复用的权衡

### 4.1 决策框架

对象池是"有成本的优化"，只在以下条件同时满足时才值得引入：

| 条件 | 不满足时的选择 |
|------|----------------|
| 分配频率 >10k/s | 直接分配即可，分配器很聪明 |
| 对象大小 >128 字节 | 栈分配或 smallvec |
| 生命周期短暂且可预测 | 管理复杂度 > 收益 |
| alloc 占 >5% CPU（profiling 确认）| 过早优化 |

### 4.2 deadpool 的使用

`deadpool` 是通用异步对象池，适合数据库连接、HTTP 客户端等"创建昂贵"的资源：

```rust
use deadpool::managed;

struct HeavyObject { data: Vec<u8> }

struct MyManager;
impl managed::Manager for MyManager {
    type Type = HeavyObject;
    type Error = Box<dyn std::error::Error + Send + Sync>;

    async fn create(&self) -> Result<Self::Type, Self::Error> {
        // 创建昂贵对象（建立连接、分配大缓冲区等）
        Ok(HeavyObject { data: vec![0u8; 1024 * 1024] })
    }

    async fn recycle(
        &self, _obj: &mut Self::Type, _metrics: &managed::Metrics,
    ) -> managed::RecycleResult<Self::Error> {
        // 归还前验证有效性——不健康则返回 Err 让它被丢弃
        Ok(())
    }
}

async fn example_deadpool() {
    let pool = managed::Pool::<MyManager>::builder(MyManager)
        .max_size(16)
        .build()
        .unwrap();

    let obj = pool.get().await.unwrap(); // 获取或创建
    // ... 使用 obj ...
    // obj 离开作用域 → 自动归还到池
}
```

### 4.3 无锁 MPSC 对象池——纳秒级

对于热路径上的缓冲区复用，`crossbeam::queue::ArrayQueue` 提供无锁的对象池：

```rust
use crossbeam::queue::ArrayQueue;

/// 字节缓冲区的无锁对象池——热路径零分配、零释放
struct BufferPool {
    free_buffers: ArrayQueue<Vec<u8>>,
    buffer_size: usize,
}

impl BufferPool {
    fn new(capacity: usize, buffer_size: usize) -> Self {
        let pool = ArrayQueue::new(capacity);
        // 预分配所有缓冲区
        for _ in 0..capacity {
            pool.push(vec![0u8; buffer_size]).unwrap();
        }
        BufferPool { free_buffers: pool, buffer_size }
    }

    /// 从池获取——lock-free
    fn acquire(&self) -> Option<Vec<u8>> {
        self.free_buffers.pop()
    }

    /// 归还到池——lock-free
    fn release(&self, mut buf: Vec<u8>) {
        buf.clear();
        // 池满则丢弃——让 Vec drop 释放内存
        let _ = self.free_buffers.push(buf);
    }
}

// 使用：热路径零分配
fn process(pool: &BufferPool, req: &[u8]) {
    let mut buf = pool.acquire().unwrap_or_else(|| vec![0u8; 4096]);
    buf.clear();
    buf.extend_from_slice(req);
    // ... 处理 ...
    pool.release(buf);
}
```

## 5. 批处理与合并：减少系统调用

### 5.1 系统调用的代价

用户态到内核态的切换约 100~500ns。每秒 100 万次 write() 时，syscall 开销可达 CPU 的 30%+。**批处理**将 N 次 syscall 合并为 1 次：

```
不批处理:                     批处理后:
 for item in items:            let batch = items.drain(..);
   write(fd, item);            writev(fd, batch);  // 一次 syscall
   ↑ N 次 syscall

N=10  → ~5x     N=100  → ~30x    N=1000 → ~200x
```

### 5.2 Vec::drain 批处理模式

```rust
use std::collections::VecDeque;

/// 基于环形缓冲区的批处理
struct Batcher<T> {
    buf: VecDeque<T>,
    batch_size: usize,
}

impl<T> Batcher<T> {
    fn new(batch_size: usize) -> Self {
        Batcher { buf: VecDeque::with_capacity(batch_size * 2), batch_size }
    }

    /// 添加元素——缓冲区满时返回整个批次
    fn push(&mut self, item: T) -> Option<Vec<T>> {
        self.buf.push_back(item);
        if self.buf.len() >= self.batch_size {
            Some(self.buf.drain(..).collect())
        } else {
            None
        }
    }

    /// 强制刷新剩余元素
    fn flush(&mut self) -> Vec<T> {
        self.buf.drain(..).collect()
    }
}
```

```rust
/// Vec::drain 实战：从队列批量取出并处理
struct WorkQueue { items: Vec<Vec<u8>> }

impl WorkQueue {
    fn process_batch_limited(&mut self, max: usize, f: &mut dyn FnMut(Vec<Vec<u8>>)) {
        let end = self.items.len().min(max);
        let batch: Vec<Vec<u8>> = self.items.drain(..end).collect();
        if !batch.is_empty() { f(batch); }
        // items 为空但 capacity 保留——后续 push 无分配
    }

    fn process_all(&mut self, f: &mut dyn FnMut(Vec<Vec<u8>>)) {
        let batch: Vec<Vec<u8>> = self.items.drain(..).collect();
        if !batch.is_empty() { f(batch); }
    }
}
```

## 6. Seastar/ScyllaDB 的 Shared-Nothing 哲学：迁移到 Rust

### 6.1 什么是 Shared-Nothing

Shared-Nothing 架构起源于分布式数据库（ScyllaDB），核心思想同样适用于单机多核：

> 每个线程/核心拥有独立的资源（内存、fd、数据结构），线程之间**不共享任何可变状态**。通信通过显式的消息传递。

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│   Shard 0    │  │   Shard 1    │  │   Shard N    │
│  CPU 0       │  │  CPU 1       │  │  CPU N       │
│  自己的内存池 │  │  自己的内存池 │  │  自己的内存池 │
│  自己的 fd   │  │  自己的 fd   │  │  自己的 fd   │
│  自己的锁     │  │  自己的锁     │  │  自己的锁     │
│  (几乎不用)   │  │  (几乎不用)   │  │  (几乎不用)   │
└──────┬───────┘  └──────┬───────┘  └──────┬───────┘
       └─────────┬───────┴─────────┬───────┘
                 ▼                 ▼
           消息传递（crossbeam channel / SPSC ring buffer）
```

### 6.2 Rust 中的 Shared-Nothing 实现

```rust
/// Seastar 风格的 shared-nothing shard
use crossbeam::channel;

struct Shard {
    id: usize,
    data: Vec<String>,
}

enum ShardMessage {
    Insert(String, String),
    Query { key: String, reply_to: channel::Sender<Option<String>> },
    Shutdown,
}

impl Shard {
    fn run(mut self, rx: channel::Receiver<ShardMessage>) {
        core_affinity::set_for_current(core_affinity::CoreId { id: self.id });

        loop {
            match rx.try_recv() {
                Ok(ShardMessage::Insert(k, v)) => self.data.push(format!("{}:{}", k, v)),
                Ok(ShardMessage::Query { key, reply_to }) => {
                    let result = self.data.iter().find(|d| d.starts_with(&key));
                    let _ = reply_to.send(result.cloned());
                }
                Ok(ShardMessage::Shutdown) => break,
                Err(channel::TryRecvError::Empty) => std::hint::spin_loop(),
                Err(channel::TryRecvError::Disconnected) => break,
            }
        }
    }
}
```

### 6.3 高性能 Rust 系统的"三位一体"

将前文的三个技术组合起来，形成完整的高性能体系：

```
┌──────────────────────────────────────────────────────────────┐
│     高性能 Rust 系统的"三位一体"                               │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. Thread-per-Core (CPU 亲和性)                              │
│     → 消除线程迁移、保证 cache locality                       │
│                                                               │
│  2. Run-to-Completion (io_uring + 单线程事件循环)              │
│     → 消除锁、消除任务迁移、减少 syscall                      │
│                                                               │
│  3. Shared-Nothing (显式数据所有权)                            │
│     → 消除共享可变状态、消除 cache 一致性开销                  │
│                                                               │
│  组合效果：                                                   │
│  → 零锁：所有数据私有或只读共享                               │
│  → 零任务迁移：任务在同一核心完成                             │
│  → 近乎无 syscall（io_uring 批量化）                           │
│  → 线可扩展：N 核心 ≈ N× 单核心吞吐                          │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## 7. 回顾

- [ ] 理解 Thread-per-Core 的核心理念：消除共享来消除争用，`core_affinity` 绑定线程到核心
- [ ] 区分 Work-Stealing (tokio) 和 Run-to-Completion (io_uring) 的场景和 trade-off
- [ ] 掌握分片 Map 原理：哈希分散争用，`dashmap` 是工业级实现
- [ ] 掌握对象池决策框架：仅在分配频率高、对象大、profiling 确认瓶颈时引入
- [ ] 理解批处理的杠杆效应：N 次 syscall 合并为 1 次，`Vec::drain` 是天然工具
- [ ] 理解 Shared-Nothing 哲学：消息传递替代共享内存，每个核心独立资源
- [ ] 知道如何组合 Thread-per-Core + Run-to-Completion + Shared-Nothing 构建可扩展系统
