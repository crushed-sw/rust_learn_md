# 10.6 端到端优化实战：优化 Kata ★ Track A Capstone

> **[Track A: 性能工程 — Capstone]** 这是 Track A 的综合实战。你已经学完了案例（10.1）、设计模式（10.2）、内存优化（10.3）、并发结构（10.4）和面试技巧（10.5）。现在，用一个可重复的"Kata"练习把这些技能串起来。本章会带你从一个 50 行的朴素 CSV 聚合器出发，经历五轮系统化优化，最终达到 10-15x 的性能提升——每一轮都有 benchmark 数据验证，形成一个可反复练习的优化套路。

## 〇、"Kata"概念

"Kata"（型）一词源自武术，指一套标准化的、可反复练习的动作序列。在编程领域，Code Kata 由 Dave Thomas 在 2007 年推广——一段可以反复练习的代码题，每次练习时都可以尝试不同的方法，逐步内化技能。

**为什么选择 CSV 聚合器？**
- 足够简单：5 分钟即可理解需求——读取 CSV 文件，计算每列的 sum、average、count。
- 足够复杂：涉及 I/O、字符串解析、浮点数运算、内存管理，每层都有优化空间。
- 可扩展：任何数据工程任务都可以套用类似的优化流程。

## 一、起点：朴素实现

### 1.1 需求定义

输入：一个 CSV 文件，第一行是列名，后续行是数值数据。
输出：每列的 sum、average、count。

```
输入示例 (data.csv):
id,temperature,humidity,pressure
1,23.5,45.2,1013.2
2,24.1,44.8,1012.8
3,22.9,46.1,1013.5
...

输出:
id: sum=..., avg=..., count=...
temperature: sum=..., avg=..., count=...
humidity: sum=..., avg=..., count=...
pressure: sum=..., avg=..., count=...
```

### 1.2 朴素实现（~50 行）

```rust
// kata_v0_naive.rs — 朴素实现，所有性能问题的起点

use std::fs;
use std::collections::HashMap;

#[derive(Debug)]
struct ColumnStats {
    sum: f64,
    count: u64,
}

fn main() {
    let path = "data.csv";

    // ── 第 1 步：一次性读取全部文件到内存 ──
    // 问题：大文件直接 OOM，且无法流式处理
    let content = fs::read_to_string(path).expect("无法读取文件");

    let mut lines = content.lines();

    // ── 第 2 步：解析 header ──
    let header_line = lines.next().expect("文件为空");
    let column_names: Vec<String> = header_line
        .split(',')
        .map(|s| s.trim().to_string()) // 每个列名分配新 String
        .collect();

    // ── 第 3 步：存储空间 ──
    // 问题：HashMap 的开销 + String key 的开销
    let mut stats_map: HashMap<String, ColumnStats> = HashMap::new();
    for name in &column_names {
        stats_map.insert(name.clone(), ColumnStats { sum: 0.0, count: 0 });
    }

    // ── 第 4 步：逐行解析 ──
    for line in lines {
        if line.trim().is_empty() {
            continue;
        }

        // 问题：每行 split 都在分配新的 String 集合
        let fields: Vec<&str> = line.split(',').collect();

        for (col_idx, field_str) in fields.iter().enumerate() {
            if col_idx >= column_names.len() {
                break;
            }

            // 问题：trim() 和 parse() 都涉及字符串处理开销
            let trimmed = field_str.trim();
            if let Ok(value) = trimmed.parse::<f64>() {
                let col_name = &column_names[col_idx];
                // 问题：每次查找都要 hash String key
                if let Some(stats) = stats_map.get_mut(col_name) {
                    stats.sum += value;
                    stats.count += 1;
                }
            }
        }
    }

    // ── 第 5 步：输出结果 ──
    for (name, stats) in &stats_map {
        let avg = if stats.count > 0 {
            stats.sum / stats.count as f64
        } else {
            0.0
        };
        println!(
            "{}: sum={:.2}, avg={:.2}, count={}",
            name, stats.sum, avg, stats.count
        );
    }
}

// ─── 性能问题清单 ───
// 1. read_to_string: 一次性加载全部文件到内存（内存峰值 = 文件大小）
// 2. HashMap<String, ...>: 每次访问都要 hash String key
// 3. line.split(',').collect(): 每行分配 Vec<&str>
// 4. trim().parse(): trim 可能重新分配，parse 使用标准库的通用算法
// 5. 单线程，完全没有利用多核
// 6. String 到处 clone，内存分配密集
```

```bash
#
```

**如何解读这些数据**：
1. **IPC = 0.66**：现代 CPU 的理想 IPC 是 2-4。这说明 CPU 大部分时间在等待 memory subsystem（cache miss 或内存延迟）。
2. **Cache miss rate = 22.7%**：HashMap 的随机访问模式 + String 的堆分配导致缓存行为极差。
3. **Branch miss rate = 4.2%**：`parse::<f64>()` 内部有大量分支（正负号、小数点、指数），分支预测器难以预测。
4. **Page faults**：频繁的堆分配触发了大量的 minor page faults（操作系统首次触摸这些页面时的开销）。

## 三、Round 1：数据结构优化

目标：消除不必要的内存分配，用索引替代 String key。

```rust
// kata_v1_structures.rs — Round 1: 数据结构优化

use std::io::{BufRead, BufReader};
use std::fs::File;

#[derive(Debug, Clone)]
struct ColumnStats {
    sum: f64,
    count: u64,
}

impl ColumnStats {
    fn new() -> Self {
        ColumnStats { sum: 0.0, count: 0 }
    }

    fn add(&mut self, value: f64) {
        self.sum += value;
        self.count += 1;
    }

    fn avg(&self) -> f64 {
        if self.count > 0 {
            self.sum / self.count as f64
        } else {
            0.0
        }
    }
}

/// Round 1 优化后的主处理函数
fn process(path: &str) -> (Vec<String>, Vec<ColumnStats>) {
    let file = File::open(path).expect("无法打开文件");

    // ✅ 优化 1: BufReader 流式读取（替代 read_to_string）
    let reader = BufReader::with_capacity(256 * 1024, file);

    let mut lines = reader.lines();

    // ✅ 优化 2: 解析 header 时用 with_capacity 预分配
    let header_line = lines.next().expect("文件为空").expect("读取失败");
    let column_names: Vec<String> = header_line
        .split(',')
        .map(|s| s.trim().to_string())
        .collect();

    let col_count = column_names.len();

    // ✅ 优化 3: 用 Vec<ColumnStats> 按索引访问（替代 HashMap<String, ...>）
    // 索引访问 O(1) + 缓存友好 + 无 hash 开销
    let mut stats: Vec<ColumnStats> = (0..col_count)
        .map(|_| ColumnStats::new())
        .collect();

    // ✅ 优化 4: 重用 line buffer
    let mut line_buf = String::with_capacity(4096);

    // ✅ 优化 5: 重用字段值缓冲区
    let mut values = vec![0.0f64; col_count];

    for line_result in lines {
        // ✅ 优化 6: 使用 read_line 风格，重用 buffer
        // 此处用标准 API，实际可以更底层
        let line_str = line_result.expect("读取行失败");
        if line_str.trim().is_empty() {
            continue;
        }

        let mut col_idx = 0;

        // ✅ 优化 7: 手动解析，避免 split 的 Vec 分配
        let mut field_start = 0usize;
        let bytes = line_str.as_bytes();
        let line_len = bytes.len();

        for (i, &byte) in bytes.iter().enumerate() {
            if byte == b',' || i == line_len - 1 {
                let field_end = if byte == b',' { i } else { i + 1 };
                if col_idx >= col_count {
                    break;
                }

                // ✅ 优化 8: 从 &[u8] 直接解析 f64
                // （仍在用标准库 parse，Round 4 会替换）
                let field_bytes = &bytes[field_start..field_end];
                let field_str = std::str::from_utf8(field_bytes)
                    .unwrap_or("");
                if let Ok(val) = field_str.trim().parse::<f64>() {
                    stats[col_idx].add(val);
                }

                col_idx += 1;
                field_start = i + 1;
            }
        }
    }

    (column_names, stats)
}

// ─── Round 1 结果 ───
fn round1_results() {
    println!("
    ┌──────────────────────────────────────────────────────────┐
    │                Round 1 — 数据结构优化                     │
    ├──────────────┬────────────┬────────────┬─────────────────┤
    │ 指标          │ V0 (Baseline)│ V1 (优化后) │ 改进           │
    ├──────────────┼────────────┼────────────┼─────────────────┤
    │ 处理时间      │ 8.2 秒     │ 4.1 秒     │ 2.0x            │
    │ 内存峰值      │ 450 MB     │ 180 MB     │ 2.5x ↓          │
    │ IPC           │ 0.66       │ 0.92       │ 39% ↑           │
    │ Cache Miss %  │ 22.7%      │ 14.1%      │ 38% ↓           │
    │ 内存分配次数  │ ~3.5M      │ ~120K      │ 29x ↓           │
    └──────────────┴────────────┴────────────┴─────────────────┘
    ");

    // 主要收益来源：
    // 1. 消除 HashMap String key → 索引访问
    // 2. BufReader 流式读取 → 减少内存峰值
    // 3. Vec 替代 HashMap → 缓存友好
    // 4. 减少 split 分配 → 更少 malloc/free
}
```

## 四、Round 2：并发化（rayon）

目标：利用多核并行处理 CSV 的不同分块。

```rust
// kata_v2_rayon.rs — Round 2: rayon 并行化

use std::io::{BufRead, BufReader};
use std::fs::File;
use rayon::prelude::*;

/// 分块信息：记录 CSV 中一块数据的起止位置
#[derive(Debug, Clone, Copy)]
struct Chunk {
    start_byte: usize,
    end_byte: usize,
}

/// 将文件按行边界分割成大致均匀的块
fn split_file_into_chunks(data: &[u8], chunk_count: usize) -> Vec<Chunk> {
    let total_bytes = data.len();
    let chunk_size = total_bytes / chunk_count;

    let mut chunks = Vec::with_capacity(chunk_count);
    let mut start = 0usize;

    // 跳过 header 行（找到第一个 \n）
    let header_end = data.iter()
        .position(|&b| b == b'\n')
        .map(|p| p + 1)
        .unwrap_or(0);
    start = header_end;

    for _ in 0..chunk_count {
        let mut end = (start + chunk_size).min(total_bytes);

        // 对齐到下一行的开始（找到 \n）
        if end < total_bytes {
            while end < total_bytes && data[end] != b'\n' {
                end += 1;
            }
            if end < total_bytes {
                end += 1; // 包含 \n
            }
        }

        if start < end {
            chunks.push(Chunk {
                start_byte: start,
                end_byte: end,
            });
        }
        start = end;
    }

    chunks
}

/// 处理一个数据块，返回该块的聚合统计
fn process_chunk(data: &[u8], col_count: usize) -> Vec<ColumnStats> {
    let mut stats = (0..col_count)
        .map(|_| ColumnStats::new())
        .collect::<Vec<_>>();

    let chunk_data = data;
    let mut pos = 0usize;

    while pos < chunk_data.len() {
        // 找到行尾
        let line_end = chunk_data[pos..].iter()
            .position(|&b| b == b'\n')
            .map(|p| pos + p)
            .unwrap_or(chunk_data.len());

        let line = &chunk_data[pos..line_end];
        pos = line_end + 1; // 跳过 \n

        if line.is_empty() {
            continue;
        }

        let mut col_idx = 0;
        let mut field_start = 0;

        for (i, &byte) in line.iter().enumerate() {
            if byte == b',' || i == line.len() - 1 {
                let field_end = if byte == b',' { i } else { i + 1 };
                if col_idx >= col_count {
                    break;
                }

                let field = &line[field_start..field_end];
                if let Ok(val_str) = std::str::from_utf8(field) {
                    if let Ok(val) = val_str.trim().parse::<f64>() {
                        stats[col_idx].add(val);
                    }
                }
                col_idx += 1;
                field_start = i + 1;
            }
        }
    }
    stats
}

/// Round 2 主函数
fn process(path: &str) -> (Vec<String>, Vec<ColumnStats>) {
    // ✅ 优化 1: mmap 文件（零拷贝 I/O）
    let file = File::open(path).expect("无法打开文件");
    let mmap = unsafe { memmap2::Mmap::map(&file).expect("mmap 失败") };
    let data: &[u8] = &mmap;

    // 解析 header
    let header_end = data.iter()
        .position(|&b| b == b'\n')
        .map(|p| p + 1)
        .unwrap_or(0);

    let header_str = std::str::from_utf8(&data[..header_end - 1]).unwrap();
    let column_names: Vec<String> = header_str
        .split(',')
        .map(|s| s.trim().to_string())
        .collect();
    let col_count = column_names.len();

    // ✅ 优化 2: 分块后交给 rayon 并行处理
    let num_threads = rayon::current_num_threads();
    let chunks = split_file_into_chunks(&data[header_end..], num_threads * 2);

    // ✅ 优化 3: fold + reduce 模式（无锁聚合）
    let final_stats: Vec<ColumnStats> = chunks
        .par_iter() // ← rayon 并行迭代器
        .map(|chunk| {
            let chunk_data = &data[header_end..][chunk.start_byte..chunk.end_byte];
            process_chunk(chunk_data, col_count)
        })
        .reduce(
            || (0..col_count).map(|_| ColumnStats::new()).collect(),
            |mut acc, chunk_stats| {
                for (a, c) in acc.iter_mut().zip(chunk_stats.iter()) {
                    a.sum += c.sum;
                    a.count += c.count;
                }
                acc
            },
        );

    (column_names, final_stats)
}

// ─── Round 2 结果 ───
fn round2_results() {
    println!("
    ┌──────────────────────────────────────────────────────────┐
    │                 Round 2 — rayon 并行化                    │
    ├──────────────┬────────────┬────────────┬─────────────────┤
    │ 指标          │ V1 (Round 1)│ V2 (优化后) │ 改进           │
    ├──────────────┼────────────┼────────────┼─────────────────┤
    │ 处理时间      │ 4.1 秒     │ 0.95 秒    │ 4.3x (8核)      │
    │ CPU 利用率    │ ~100%      │ ~650%      │ 6.5 核同时工作  │
    │ 并行效率      │ —          │ 81%        │ 接近线性扩展    │
    │ 内存峰值      │ 180 MB     │ 280 MB     │ mmap ~100MB     │
    └──────────────┴────────────┴────────────┴─────────────────┘
    ");
}
```

**为什么是 4.3x 而不是 8x？**
1. **阿姆达尔定律**：串行部分（header 解析、结果 merge）占 ~15% 时间。
2. **内存带宽瓶颈**：8 个核同时读内存，DDR4 的带宽成为限制。
3. **伪共享（false sharing）**：不同线程的 local_stats 可能落在同一缓存行。
4. **负载不均衡**：某些 chunk 的行比其他 chunk 长。

## 五、Round 3：I/O 优化

目标：消除 I/O 瓶颈，探究 BufReader vs mmap 的实际差异。

```rust
// kata_v3_io.rs — Round 3: I/O 路径优化

use std::io::{BufRead, BufReader, Read};
use std::fs::File;

/// I/O 策略对比：三种读取方式的基准测试
fn io_strategies(path: &str) {
    // ── 策略 A: 默认 BufReader (8KB buffer) ──
    fn read_with_default_bufreader(path: &str) -> Vec<u8> {
        let file = File::open(path).unwrap();
        let mut reader = BufReader::new(file);
        let mut data = Vec::new();
        reader.read_to_end(&mut data).unwrap();
        data
    }
    // 基准: ~120 MB/s（受限于小 buffer 导致的频繁系统调用）

    // ── 策略 B: BufReader with 256KB buffer ──
    fn read_with_large_bufreader(path: &str) -> Vec<u8> {
        let file = File::open(path).unwrap();
        // ✅ 自定义 buffer 大小
        let mut reader = BufReader::with_capacity(256 * 1024, file);
        //                                     ^^^^^^^^ 256KB buffer
        let mut data = Vec::new();
        reader.read_to_end(&mut data).unwrap();
        data
    }
    // 基准: ~380 MB/s（更大的 buffer = 更少的 read() 系统调用）

    // ── 策略 C: mmap (零拷贝) ──
    fn read_with_mmap(path: &str) -> &[u8] {
        // 注意：返回的是引用，不能独立于 mmap 对象
        // 实际使用中需要确保 mmap 对象的生命周期
        let file = File::open(path).unwrap();
        let mmap = unsafe {
            memmap2::Mmap::map(&file).unwrap()
        };
        // mmap 零拷贝——文件内容直接映射到虚拟内存
        // 优势：没有 user-space buffer 拷贝
        // 注意：页面错误（page fault）随访问延迟触发
        //       → 首次访问时可能变慢（冷启动效应）
        unsafe { std::slice::from_raw_parts(mmap.as_ptr(), mmap.len()) }
    }
    // 基准: ~800 MB/s（无拷贝 + 预读由 OS 管理）
    // 但注意：实际性能取决于 OS 的页面缓存状态

    // ── I/O 比较总结 ──
    println!("
    ┌──────────────────────────────────────────────────────────┐
    │             I/O 策略对比（100MB 文件单线程读取）          │
    ├─────────────────────┬──────────┬─────────────────────────┤
    │ 策略                 │ 吞吐量    │ 适用场景               │
    ├─────────────────────┼──────────┼─────────────────────────┤
    │ BufReader (8KB)      │ 120 MB/s │ 默认，适合小文件       │
```

## 六、Round 4：算法层面

目标：把标准库的 `parse::<f64>()` 替换为更快的浮点数解析，消除关键路径上的分支。

```rust
// kata_v4_algorithm.rs — Round 4: 算法优化

/// ✅ 优化 1: 使用 fast_float crate 替代 std::str::parse<f64>
/// fast_float 是 Daniel Lemire 的快速浮点数解析算法的 Rust 实现
/// 比标准库快 4-10x，因为使用了 SIMD 和查表法

use fast_float;

/// ✅ 优化 2: 查表法解析整数（无分支，适合整数列）
const DIGITS_LOOKUP: [u8; 256] = {
    let mut table = [255u8; 256];
    let mut i = 0u8;
    while (b'0' as usize + i as usize) <= b'9' as usize {
        table[b'0' as usize + i as usize] = i;
        i += 1;
    }
    table
};

/// 超快整数解析（无分支版本）
/// 适用于确定是整数的字段，比 parse::<i64>() 快 15-20x
#[inline(always)]
fn fast_parse_int(bytes: &[u8]) -> Option<i64> {
    // 处理符号
    let (sign, start) = if bytes.first() == Some(&b'-') {
        (-1i64, 1usize)
    } else {
        (1i64, 0usize)
    };

    let mut result: i64 = 0;
    for &b in &bytes[start..] {
        let digit = DIGITS_LOOKUP[b as usize];
        if digit == 255 {
            return None; // 非数字字符，这种列不适合整数解析
        }
        result = result.wrapping_mul(10).wrapping_add(digit as i64);
    }
    Some(result * sign)
}

/// ✅ 优化 3: 分支消除（branchless min/max）
/// 用于需要同时计算 min/max 的场景
#[inline(always)]
fn update_min_max_branchless(
    value: f64,
    current_min: &mut f64,
    current_max: &mut f64,
) {
    // 使用条件移动指令（cmov）替代分支跳转
    // 不会触发分支预测失败
    if value < *current_min {
        *current_min = value;
    } else if value > *current_max {
        // 注意：这里仍有分支，但在多数场景下
        // 实际使用 f64::min / f64::max 能达到类似效果
        *current_max = value;
    }
    // 注：Rust 标准库没有直接暴露 cmov intrinsic
    // 但现代编译器通常能自动将简单的 if 优化为 cmov
}

/// ✅ 优化 4: 预计算常量
/// 对于固定格式的数据，可以跳过解析
const DATE_PREFIX_LEN: usize = 10; // "2026-07-05"

/// 跳过固定前缀，只需要解析后面的时间部分
#[inline(always)]
fn skip_prefix_and_parse(bytes: &[u8]) -> Option<f64> {
    if bytes.len() <= DATE_PREFIX_LEN {
        return fast_float::parse(bytes);
    }
    // 跳过日期前缀，只解析数值部分
    let numeric_part = &bytes[DATE_PREFIX_LEN..];
    fast_float::parse(numeric_part)
}

/// Round 4: 整合后的解析函数
#[inline(always)]
fn parse_field_fast(bytes: &[u8]) -> Option<f64> {
    // fast_float 内部使用 SIMD（SSE4.2 / AVX2）
    fast_float::parse(bytes)
}

// ─── Round 4 结果 ───
fn round4_results() {
    println!("
    ┌──────────────────────────────────────────────────────────┐
    │                  Round 4 — 算法优化                       │
    ├──────────────┬────────────┬────────────┬─────────────────┤
    │ 指标          │ V3 (Round 3)│ V4 (优化后) │ 改进           │
    ├──────────────┼────────────┼────────────┼─────────────────┤
    │ 处理时间      │ 0.41 秒    │ 0.24 秒    │ 1.7x            │
    │ IPC           │ 1.31       │ 2.18       │ 66% ↑           │
    │ Branch Miss % │ 3.1%       │ 0.9%       │ 71% ↓           │
    │ 解析时间占比  │ 62%        │ 28%        │ 解析不再是瓶颈  │
    └──────────────┴────────────┴────────────┴─────────────────┘
    ");

    // fast_float 为什么快？
    // 1. SIMD: 一次比较 16/32 个字节
    // 2. 查表: 预计算的 ASCII → 数字映射
    // 3. 浮点数转换使用 Daniel Lemire 算法（避免大整数除法）
    // 4. 解析和转换合并为一个紧凑循环
}
```

## 七、Round 5：PGO + BOLT

### 7.1 Profile-Guided Optimization (PGO)

PGO 让编译器根据实际的运行 profile 来优化分支预测、内联决策和代码布局。

```bash
# ── Step 1: 编译 instrumented 版本 ──
RUSTFLAGS="-Cprofile-generate=/tmp/pgo-data" cargo build --release

# ── Step 2: 用代表性数据运行训练 ──
./target/release/kata_v4 data_train.csv

# ── Step 3: 合并 profile 数据 ──
llvm-profdata merge -o /tmp/pgo-data/merged.profdata /tmp/pgo-data/

# ── Step 4: 用 profile 重新编译（优化版） ──
RUSTFLAGS="-Cprofile-use=/tmp/pgo-data/merged.profdata" cargo build --release

# ── 结果: ──
# PGO 后的二进制更快，因为：
# 1. 热路径上的分支被优化为"likely taken"（减少分支预测失败）
# 2. 热函数被更激进地内联
# 3. 代码布局优化：热代码放在一起（更好的指令缓存局部性）
```

**PGO 原理图示**：

```
                 ┌──────────────────────┐
                 │   Instrumented Build │
                 │   (插入计数器)        │
                 └──────────┬───────────┘
                            │
                    运行训练数据
                            │
                 ┌──────────▼───────────┐
                 │   Profile Data       │
                 │   (分支频率、函数    │
                 │    调用次数)          │
                 └──────────┬───────────┘
                            │
                 ┌──────────▼───────────┐
                 │   Optimized Build    │
                 │   (根据 profile      │
                 │    重新决策)          │
                 └──────────────────────┘
```

**PGO 在 Rust 中的限制**：
1. 需要 nightly 编译器（部分 stable 支持有限）。
2. 构建时间增加 ~2x（instrument 阶段更慢）。
3. 只对"LLVM 能看到的"代码有效——对宏展开的代码影响取决于宏的生成模式。
4. `build.rs` 和 proc macro 不会被 instrumented。

### 7.2 BOLT (Binary Optimization and Layout Tool)

BOLT 是 Meta 开源的二进制后优化工具，在 PGO 的基础上进一步优化指令布局。

```bash
# ── Step 1: 先对二进制运行 perf record（收集硬件采样） ──
perf record -e cycles:u -o /tmp/perf.data \
    ./target/release/kata_v4 data_train.csv

# ── Step 2: 将 perf.data 转换为 BOLT 可读的格式 ──
perf2bolt -p /tmp/perf.data -o /tmp/perf.fdata \
    ./target/release/kata_v4

# ── Step 3: 运行 BOLT 优化 ──
llvm-bolt ./target/release/kata_v4 \
    -o ./target/release/kata_v4.bolt \
    -data /tmp/perf.fdata \
    -reorder-blocks=ext-tsp \          # 扩展 TSP 算法重排基本块
    -reorder-functions=hfsort+ \       # 函数排序（热函数放一起）
    -split-functions \                 # 拆分冷热函数
    -split-all-cold \                  # 将所有冷代码移到最后
    -dyno-stats \                      # 输出优化统计
    -lite=false
# 注：BOLT 需要二进制包含重定位信息（link with --emit-relocs）
```

**BOLT 做了什么**：

```
优化前二进制布局：              优化后二进制布局：
┌────────────────┐              ┌────────────────┐
│ main (部分热)   │              │ main (热部分)   │ ← 紧凑放在一起
│ utils (热)     │              │ process_chunk   │   指令 cache 友好
│ parse (冷)     │              │ parse (热部分)  │
│ main (冷部分)  │              ├────────────────┤ ← 缓存行边界
│ process_chunk  │              │ main (冷部分)   │ ← 冷代码挪到最后
│ parse (热部分) │              │ parse (冷部分)  │   不会污染 cache
│ ...            │              │ utils (热部分)  │
└────────────────┘              │ ...            │
                                └────────────────┘
```
