## 二、系统设计中的性能考量

### 2.1 案例一：设计 1M QPS 的 URL 短链接服务

这是一个经典系统设计题的 **Rust 性能视角**：

**需求分析**：
- 写入：每秒生成 1,000 个短链接
- 读取：每秒 1,000,000 次跳转（读写比 1000:1）
- 存储量：假设 100 亿条，每条约 1KB，共约 10TB
- 延迟目标：P99 < 5ms

**Rust 技术选型思路**：

```rust
// ── 第一步：Hash 函数选择 ──
// 
// 不选 SHA256（太长，碰撞极低但 URL 太长）
// 不选简单的 fnv（碰撞率太高）
// 
// 选择：xxhash + base62 编码

use xxhash_rust::xxh3::xxh3_64;

/// 生成短链接 ID
fn generate_short_id(url: &str, salt: u64) -> String {
    let hash = xxh3_64((url.to_owned() + &salt.to_string()).as_bytes());
    // 将 64 位 hash 转为 base62 (0-9a-zA-Z)
    base62_encode(hash)
}

/// base62 编码
fn base62_encode(mut num: u64) -> String {
    const CHARSET: &[u8] = b"0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz";
    if num == 0 {
        return "0".to_string();
    }
    let mut result = Vec::with_capacity(11); // 64位最多11个base62字符
    while num > 0 {
        result.push(CHARSET[(num % 62) as usize]);
        num /= 62;
    }
    result.reverse();
    String::from_utf8(result).unwrap()
}
```

**第二步：存储引擎选择**

```
┌──────────────────────────────────────────────────────────────┐
│                    存储架构                                    │
│                                                               │
│   写入请求 ──→ [内存写入缓冲区] ──→ [LSM-Tree 存储]           │
│                    │                    (如 Rust Sled 或       │
│                    │                     RocksDB via FFI)      │
│                    ▼                                           │
│   读取请求 ──→ [DashMap 热缓存] ──→ 命中返回                   │
│                    │                                           │
│                    └──→ 未命中 → 查存储 → 回填缓存             │
└──────────────────────────────────────────────────────────────┘
```

```rust
use dashmap::DashMap;
use lru::LruCache;
use std::sync::Mutex;
use std::num::NonZeroUsize;

/// 两层缓存架构
struct UrlCache {
    // L1: 热缓存，DashMap，读路径无锁（绝大多数读命中的地方）
    hot: DashMap<String, String>,       // short_id → long_url
    // L2: LRU 淘汰缓存，保护 L1 不被冷数据污染
    warm: Mutex<LruCache<String, String>>,
}

impl UrlCache {
    fn new(hot_capacity: usize, warm_capacity: usize) -> Self {
        UrlCache {
            hot: DashMap::with_capacity(hot_capacity),
            warm: Mutex::new(LruCache::new(
                NonZeroUsize::new(warm_capacity).unwrap()
            )),
        }
    }

    fn get(&self, short_id: &str) -> Option<String> {
        // ① 先查 L1 热缓存（无锁）
        if let Some(url) = self.hot.get(short_id) {
            return Some(url.clone());
        }
        // ② 再查 L2 温缓存（加锁，但访问频率低）
        let mut warm = self.warm.lock().unwrap();
        if let Some(url) = warm.get(short_id) {
            // 回填到 L1
            self.hot.insert(short_id.to_string(), url.clone());
            return Some(url.clone());
        }
        None
    }

    fn insert(&self, short_id: &str, long_url: &str) {
        self.hot.insert(short_id.to_string(), long_url.to_string());
    }
}
```

**第三步：限流器（Rate Limiter）**

```rust
use std::collections::VecDeque;
use std::time::{Duration, Instant};
use dashmap::DashMap;

/// 基于滑动窗口的分布式限流器（单机简化版）
struct RateLimiter {
    windows: DashMap<String, VecDeque<Instant>>,
    max_requests: usize,
    window_size: Duration,
}

impl RateLimiter {
    fn new(max_requests: usize, window_secs: u64) -> Self {
        RateLimiter {
            windows: DashMap::new(),
            max_requests,
            window_size: Duration::from_secs(window_secs),
        }
    }

    fn is_allowed(&self, client_id: &str) -> bool {
        let now = Instant::now();
        let cutoff = now - self.window_size;

        // entry API 减少一次 hash 查找
        let mut window = self.windows
            .entry(client_id.to_string())
            .or_insert_with(VecDeque::new);

        // 清理过期的时间戳
        while window.front().map_or(false, |t| *t < cutoff) {
            window.pop_front();
        }

        if window.len() >= self.max_requests {
            return false; // 超出限制
        }

        window.push_back(now);
        true
    }
}
```

**第四步：性能评估**

```
每秒 1M 读取的硬件估算：
  - 每次读取：1次 DashMap 查找 + 1次 memcpy (约 1KB)
  - 内存带宽：1M * 1KB = 1GB/s → DDR4 的理论带宽是 20-50GB/s → OK
  - CPU：1M * ~30ns (hashing + lookup) = 30ms CPU time per second → 单核远不够
  - 需要：30-40 个核用于哈希 + 8 个核用于 I/O → 一台 64 核机器足够
  - 加上冗余：2 台服务器 + 负载均衡 → 每台只需 0.5M QPS
```

### 2.2 案例二：实时分析管道

设计一个类 Kafka 的消息队列 + 实时聚合：

```rust
// ── 核心设计：分区 + 批量 + 物化视图 ──

use std::collections::HashMap;
use dashmap::DashMap;

/// 物化视图：预聚合的结果集
/// 原理：不存储原始数据，只存储聚合结果（count, sum, min, max）
#[derive(Debug, Clone)]
struct MaterializedView {
    // 维度：按分钟 + 事件类型聚合
    // key: "2026-07-05T14:30/click"
    // value: (count, sum_latency)
    aggregates: HashMap<String, (u64, f64)>,
}

impl MaterializedView {
    fn new() -> Self {
        MaterializedView { aggregates: HashMap::new() }
    }

    /// 吸收一条事件，更新聚合
    fn ingest(&mut self, timestamp: u64, event_type: &str, latency: f64) {
        let minute_bucket = timestamp / 60;
        let key = format!("{}/{}", minute_bucket, event_type);
        let entry = self.aggregates.entry(key).or_insert((0, 0.0));
        entry.0 += 1;       // count++
        entry.1 += latency; // sum_latency += latency
    }

    /// 查询某个时间窗口的聚合
    fn query(&self, start_minute: u64, end_minute: u64, event_type: &str)
        -> (u64, f64)
    {
        let mut total_count = 0u64;
        let mut total_latency = 0.0f64;
        for minute in start_minute..=end_minute {
            let key = format!("{}/{}", minute, event_type);
            if let Some((count, sum)) = self.aggregates.get(&key) {
                total_count += count;
                total_latency += sum;
            }
        }
        (total_count, total_latency)
    }
}

// ── 分区处理器：每个分区一个线程，独立聚合 ──
use std::sync::Arc;
use crossbeam_channel::{bounded, Sender};

struct PartitionProcessor {
    partition_id: usize,
    view: MaterializedView,
    // 接收事件
    receiver: crossbeam_channel::Receiver<RawEvent>,
    // 发送物化结果到查询层
    result_tx: Sender<MaterializedView>,
}

#[derive(Debug, Clone)]
struct RawEvent {
    timestamp: u64,
    event_type: String,
    latency: f64,
}

impl PartitionProcessor {
    fn run(mut self) {
        // 每 1000 条或每 100ms 刷新一次物化视图
        let mut batch_count = 0usize;
        let flush_interval = std::time::Duration::from_millis(100);
        let mut last_flush = std::time::Instant::now();

        loop {
            match self.receiver.recv_timeout(
                std::time::Duration::from_millis(10)
            ) {
                Ok(event) => {
                    self.view.ingest(
                        event.timestamp,
                        &event.event_type,
                        event.latency,
                    );
                    batch_count += 1;
                }
                Err(_) => {
                    // 超时，检查是否需要 flush
                    if batch_count >= 1000
                        || last_flush.elapsed() >= flush_interval
                    {
                        // 克隆物化视图发送到查询层
                        let snapshot = self.view.clone();
                        let _ = self.result_tx.try_send(snapshot);
                        batch_count = 0;
                        last_flush = std::time::Instant::now();
                    }
                }
            }
        }
    }
}
```

**性能分析要点（面试时说这些）**：
1. 分区（partition）避免了全局锁。
2. 物化视图（materialized view）把 O(n) 的聚合查询变成了 O(1)。
3. 批量 flush 减少了消息传递的开销。
4. 如果需要持久化，可以在分区内挂一个本地 WAL。

## 三、常见性能问答

### 3.1 "为什么这个 HashMap 这么慢？"

```rust
use std::collections::HashMap;

// ❌ 常见性能陷阱
fn slow_hashmap() {
    let mut map: HashMap<Vec<u8>, u64> = HashMap::new();

    // 问题 1：key 是 Vec<u8>，每次 hash 都要遍历整个 Vec
    // 问题 2：默认 hash 函数 SipHash-1-3 是密码学安全的，但不是最快的
    // 问题 3：没有预分配容量，会多次 resize

    for i in 0..1_000_000 {
        map.insert(vec![i as u8; 64], i);
        //           ^^^^^^^^^^^^^^ key 是 64 字节的 Vec
        //           每次 insert 都要 hash 这 64 字节！
    }
}

// ✅ 优化方案
use rustc_hash::FxHashMap; // 使用更快的 hash 函数（非密码学安全）
use std::hash::BuildHasherDefault;

fn fast_hashmap() {
    // 方案 A：换 hash 函数——FxHash 比 SipHash 快 3-5x
    let mut map: FxHashMap<u64, u64> = FxHashMap::default();

    // 方案 B：用整数 key 代替长字节数组
    // 如果必须用字节数组，考虑先 hash 成 u64 再用作 key

    // 方案 C：预分配容量
    let mut map: FxHashMap<u64, u64> =
        FxHashMap::with_capacity_and_hasher(
            1_000_000,
            BuildHasherDefault::default(),
        );

    for i in 0..1_000_000 {
        map.insert(i, i);
    }
}

// ── 面试时的回答模板 ──
// "HashMap 慢通常有三个原因：
//  1. hash 函数——SipHash 安全但慢，FxHash 快但不防 HashDoS
//  2. key 设计——长 key 导致 hash 计算开销大，考虑用 hash 值做 key
//  3. 扩容——频繁 resize 会触发内存分配和重新 hash，用 with_capacity 预分配"
```

### 3.2 "如何减少这个程序的内存使用？"

```rust
// ❌ 内存浪费的例子
enum Message {
    Text { content: String, sender: String, timestamp: u64 },
    Image { url: String, width: u32, height: u32, sender: String },
    Video { url: String, duration: u32, sender: String },
    Sticker { id: u64 },
}

// 每个 Message 的大小 = max(变体大小) + tag
// Text: 3 * 24 (String) + 8 (u64) = 80 bytes
// Image: 2 * 24 (String) + 2 * 4 (u32) = 56 bytes
// Sticker: 8 (u64) = 8 bytes
// → 每个 Message 至少 80 + 8 (discriminant) = 88 bytes
// 即使存 Sticker，也要占 88 bytes！

// ✅ 优化方案
// 方案 A：Box 大字段 —— 减少枚举整体大小
enum OptimizedMessage {
    Text {
        // 将大的数据放在堆上，枚举本身只存指针
        content: Box<str>,       // Box<str> = 16 bytes (ptr + len)
        sender: Box<str>,        // 比 String 少一个 capacity 字段
        timestamp: u64,
    },
    Image {
        url: Box<str>,
        dimensions: (u16, u16), // 实际场景下 u16 足够
        sender: Box<str>,
    },
    Sticker { id: u64 },
}
// 优化后：Text 变体大小显著减小
// Box<str> = 16 bytes (ptr + len) vs String = 24 bytes (ptr + len + cap)
// 并且常见的小变体不再为最大变体预留空间

// 方案 B：Arena 分配 —— 批量管理生命周期相同的对象
use typed_arena::Arena;

struct MessageArena<'a> {
    content: &'a str,
    sender: &'a str,
    timestamp: u64,
}

fn arena_example() {
    let arena = Arena::with_capacity(1024 * 1024); // 1MB arena

    let mut messages: Vec<MessageArena> = Vec::new();
    for i in 0..100_000 {
        let content = arena.alloc(format!("content-{}", i));
        let sender = arena.alloc("system".to_string());
        messages.push(MessageArena {
            content,
            sender,
            timestamp: i as u64,
        });
    }
    // 所有消息的生命周期受 arena 管理，一次性释放
    // 相比独立 String，减少了 100% 的 malloc/free 开销
    // 内存碎片更少，缓存局部性更好
}
```

**面试时补充要点**：
- "enum niche optimization"：`Option<Box<T>>` 和 `Box<T>` 大小相同，因为 Rust 用 null 指针表示 None。
- `Box<[T]>` vs `Box<Vec<T>>`：前者少一层间接访问，适合不需要 resize 的数组。
- "20 bytes rule"：小于 20 字节的值类型放栈上，大于的考虑引用计数或 arena。

### 3.3 "这个函数有高尾延迟（tail latency），怎么排查？"

```rust
// 面试官给你一段代码，问 P99 延迟为什么是 500ms 而 P50 只有 2ms

use std::sync::Mutex;
use std::collections::HashMap;

/// ❌ 有尾延迟问题的代码
fn process_request(
    cache: &Mutex<HashMap<String, Vec<u8>>>,
    key: &str,
    payload: Vec<u8>,
) -> Option<Vec<u8>> {
    // 关键嫌疑点：
    // 1. 锁竞争——Mutex 在被持有时，其他请求排队
    // 2. HashMap 的内部 resize——持有锁时触发容量扩容
    let mut map = cache.lock().unwrap();  // ← 获取锁

    if let Some(cached) = map.get(key) {
        return Some(cached.clone()); // ← clone 大 Vec 时仍持有锁
    }

    // 3. 大 payload 的 hash 计算——如果 key 很长
    // 4. insert 可能触发 resize，在持有锁时重新分配和移动所有元素
    map.insert(key.to_string(), payload); // ← resize 时所有线程等待

    None
}
```

**回答框架（STAR 方法改编）**：

```
步骤 1 — 收集数据（Measure）：
  "首先用火焰图和 perf stat 验证假设。
   我预期看到：
   - __lll_lock_wait 的调用栈占大量时间（锁竞争）
   - 页错误（page fault）很多（HashMap resize 时大量分配）
   - clone 调用栈很高（复制大 Vec）"

步骤 2 — 诊断根因（Diagnose）：
  "确认是这三个问题后，分别制定对策"

步骤 3 — 修复（Fix）：
  // → 用 read-heavy 优化的结构替代 Mutex<HashMap>
  // → 预分配 HashMap 容量
  // → 缓存值使用 Arc<[u8]> 避免 clone

✅ 优化后的代码：
use dashmap::DashMap;
use std::sync::Arc;

type CacheMap = DashMap<String, Arc<[u8]>>;

fn process_request_fast(
    cache: &CacheMap,
    key: &str,
    payload: Vec<u8>,
) -> Option<Arc<[u8]>> {
    // DashMap: 分段锁，不同 key 大概率无竞争
    // get() 返回引用，无需 clone
    if let Some(cached) = cache.get(key) {
        return Some(cached.clone()); // Arc::clone 只是原子加计数
    }

    // insert 只锁一个 shard
    let shared: Arc<[u8]> = payload.into();
    cache.insert(key.to_string(), shared);

    None
}

步骤 4 — 验证结果（Validate）：
  "修改后 P99 从 500ms 降到 12ms，P50 从 2ms 降到 0.3ms。
   原先的 Mutex 瓶颈被分段锁解决，Arc 避免了数据复制。"
```

## 四、白板优化：CSV 解析器逐步进化

面试中可能被要求口头优化一段代码。以下是一个标准模板——每一步都说清楚"做了什么、为什么、预期提升多少"。

### 4.1 起点：朴素实现

```rust
/// Round 0: 朴素 CSV 解析器
/// 目标：读取 CSV 文件，计算每列的数值总和
fn naive_csv_sum(path: &str) -> Vec<f64> {
    let content = std::fs::read_to_string(path).unwrap();
    //                                      ^ 一次性读入内存，大文件直接 OOM

    let mut lines = content.lines(); // 按 '\n' 分割，分配每行的 String
    let header = lines.next().unwrap();
    let col_count = header.split(',').count();

    let mut sums = vec![0.0f64; col_count];

    for line in lines {
        for (i, val_str) in line.split(',').enumerate() {
            // split 返回 &str 迭代器（没问题）
            // 但 parse 可能失败
            if let Ok(val) = val_str.trim().parse::<f64>() {
                sums[i] += val;
            }
        }
    }
    sums
}
// 问题：
// 1. read_to_string 一次性加载全部文件
// 2. lines() 返回每行的 String 引用，但底层 content 已经分配了全部内存
// 3. 每行 split 要扫描两次（先 lines 按 \n 分割，再 split 按 , 分割）
// 4. trim() 和 parse() 对每个字段都要做字符串处理

// 基准：处理 100MB CSV，约 8 秒，吞吐 12.5 MB/s
```

### 4.2 Round 1：预分配 + BufReader

```rust
use std::io::{BufReader, BufRead};
use std::fs::File;

/// Round 1: 流式读取 + 预分配
fn round1_csv_sum(path: &str) -> Vec<f64> {
    let file = File::open(path).unwrap();
```

### 4.3 Round 2：零拷贝 + 并发

```rust
use rayon::prelude::*;
use memmap2::Mmap;

/// Round 2: mmap + rayon 并发处理
fn round2_csv_sum(path: &str) -> Vec<f64> {
    let file = File::open(path).unwrap();
    // ✅ mmap：将文件直接映射到虚拟内存，零拷贝
    let mmap = unsafe { Mmap::map(&file).unwrap() };

    // ✅ 将字节数组转为 &[u8]
    let data: &[u8] = &mmap;

    // ✅ 按换行符找到所有行的起始位置
    let line_starts: Vec<usize> = memchr::memchr_iter(b'\n', data)
        .map(|pos| pos + 1) // 跳过 \n
        .collect();

    // 从第二行开始（第一行是 header），解析 CSV
    let header_end = line_starts.first().copied().unwrap_or(data.len());
    let col_count = data[..header_end].iter()
        .filter(|&&b| b == b',')
        .count() + 1;

    // ✅ rayon 并发处理：每个线程处理一批行
    let sums: Vec<f64> = line_starts
        .par_iter() // parallel iterator
        .skip(1)    // 跳过 header
        .fold(
            || vec![0.0f64; col_count],
            |mut local_sums, &line_start| {
                // 找到这一行的结束位置
                let line_end = data[line_start..]
                    .iter()
                    .position(|&b| b == b'\n')
                    .map(|p| line_start + p)
                    .unwrap_or(data.len());

                let line_data = &data[line_start..line_end];
                let mut col_idx = 0;
                let mut field_start = 0;

                for (i, &byte) in line_data.iter().enumerate() {
                    if byte == b',' {
                        let field = &line_data[field_start..i];
                        if let Ok(val) = fast_float::parse::<f64, _>(field) {
                            local_sums[col_idx] += val;
                        }
                        col_idx += 1;
                        field_start = i + 1;
                    }
                }
                // 处理最后一个字段
                let last_field = &line_data[field_start..];
                if let Ok(val) = fast_float::parse::<f64, _>(last_field) {
                    local_sums[col_idx] += val;
                }
                local_sums
            },
        )
        .reduce(
            || vec![0.0f64; col_count],
            |mut a, b| {
                for (a_val, b_val) in a.iter_mut().zip(b.iter()) {
                    *a_val += *b_val;
                }
                a
            },
        );

    sums
}

// - mmap: 零拷贝文件 I/O
// - 无字符串分配：直接从 &[u8] 解析
```

### 4.4 Round 3：SIMD + 分支消除

```rust
/// Round 3: SIMD 加速 + 查表法
/// 使用 lookup table 加速 ASCII 到数字的转换

// 预计算的查找表
const DIGIT_TABLE: [u8; 256] = {
    let mut table = [255u8; 256];
    table[b'0' as usize] = 0;
    table[b'1' as usize] = 1;
    table[b'2' as usize] = 2;
    table[b'3' as usize] = 3;
    table[b'4' as usize] = 4;
    table[b'5' as usize] = 5;
    table[b'6' as usize] = 6;
    table[b'7' as usize] = 7;
    table[b'8' as usize] = 8;
    table[b'9' as usize] = 9;
    table
};

/// 使用查表法快速解析 u64（无分支版本）
#[inline]
fn fast_parse_u64(bytes: &[u8]) -> Option<u64> {
    let mut result: u64 = 0;
    for &b in bytes {
        let digit = DIGIT_TABLE[b as usize];
        if digit == 255 {
            return None; // 非数字字符
        }
        result = result.wrapping_mul(10).wrapping_add(digit as u64);
    }
    Some(result)
}

/// 用条件移动（cmov）消除分支——避免分支预测失败
#[inline]
fn min_max_branchless(values: &[f64]) -> (f64, f64) {
    let mut min = f64::MAX;
    let mut max = f64::MIN;
    for &v in values {
        // 条件移动：无论 v 是否小于 min，都执行"移动"
        // CPU 不需要预测分支
        let is_less = v < min;
        // 利用 f64 的位模式：is_less 时选择 v，否则选 min
        min = if is_less { v } else { min };
        // 实际用 f64::min 或更底层的 intrinsics 更好
        max = if v > max { v } else { max };
    }
    (min, max)
}

// 基准总结：
fn round3_notes() {
    // Round 0: ~8.0 秒   (baseline naive)
    // Round 1: ~3.5 秒   (2.3x — 流式 I/O + fast_float)
    // Round 2: ~0.6 秒   (5.8x — mmap + rayon 8核)
    // Round 3: ~0.4 秒   (1.5x — 查表法 + 分支消除)
    // Total:   ~20x 提升
}
```

### 6.2 面试中的"权衡话术"

**经典问题**："你在做一个低延迟交易系统。每条消息的处理延迟必须 < 10 微秒（P999）。你会做哪些取舍？"

```text
回答模板：

"首先，我需要明确这个约束意味着什么：
 - P999 < 10us 意味着 GC 停顿是不可能的（排除 Java/Golang）
 - 动态内存分配也可能触发超过 10us 的延迟（malloc 的 tail latency）
 - 也就是说，我需要放弃动态分配和锁。

所以在设计中有这些取舍：

1. 【内存 vs 延迟】
   牺牲内存使用的灵活性来换取确定的延迟。
   我会用预分配的环形缓冲区，而不是 Vec 或 HashMap。
   Arena 分配可能可以，但 free 操作必须批量或延迟执行。

2. 【吞吐量 vs 延迟】
   可能牺牲一些吞吐量。例如 batch 处理能提高吞吐，
   但会增加单条消息的延迟。在这种场景下，逐条处理优先。

3. 【开发速度 vs 运行时速度】
   愿意花 3 倍的开发时间写 SIMD intrinsics 或手写汇编，
   因为这是系统的核心约束。但只在关键路径上做——
   非关键路径仍然用 safe Rust。

4. 【一致性 vs 延迟】
   如果这个系统需要多副本同步，我可能会选择
   最终一致性而不是强一致性，因为分布式共识
   的延迟通常在毫秒级，远超 10us 的预算。

量化的话：我可以接受多使用 2x 的内存和多花 3x 的开发时间，
来换取 P999 < 10us 的确定性延迟保证。"
```

### 6.3 反直觉的权衡

面试官喜欢听到"不按常理出牌"但有理有据的分析：

```rust
// ── 反直觉案例 1：unsafe 有时候"更安全" ──
// 在性能关键路径中，unsafe 的手动边界检查可以减少一次条件判断，
// 从而减少分支预测失败的概率。代价是放弃了编译器的自动检查，
// 但如果你有完善的 fuzzing 和模型检查（如 loom），风险是可控的。

// ── 反直觉案例 2：多分配有时候比少分配快 ──
// 假设你要合并两个 Vec：
fn concat_vecs(a: Vec<u8>, b: Vec<u8>) -> Vec<u8> {
    // 方案 A: 精确分配（2次 malloc）
    let mut result = Vec::with_capacity(a.len() + b.len());
    result.extend_from_slice(&a);
    result.extend_from_slice(&b);
    result
}

fn concat_vecs_allocator_friendly(a: Vec<u8>, b: Vec<u8>) -> Vec<u8> {
    // 方案 B: 先 push a，再在扩容时 push b（1次 malloc，但有一次 memcpy）
    // 在 jemalloc 或 mimalloc 下，方案 B 可能更快
    // 因为分配器对大块分配有优化，且减少了一次 malloc 的同步开销
    let mut result = Vec::with_capacity(a.len()); // 只预分配 a
    result.extend_from_slice(&a);                  // 填满
    // extend_from_slice(&b) 会触发扩容——一次大分配+一次 memcpy
    // vs 两次分配+两次 memcpy
    result.extend_from_slice(&b);
    result
}
// 结论：没有绝对正确的方案，必须 benchmark。

// ── 反直觉案例 3：HashMap 的 load_factor 不是越低越好 ──
// 很多人认为 load_factor 越低（更多空槽），查询越快。
// 但这忽略了缓存效应。load_factor 0.5 意味着 hash table
// 占用 2x 内存，cache miss rate 可能更高。
// 在 L3 cache 有限的系统上，load_factor 0.75-0.85 通常最优。
```
