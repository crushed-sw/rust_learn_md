# 10.9 订单簿撮合引擎 —— 实现一个完整的多类型限价单匹配系统

> **[Track B: 量化金融]** 撮合引擎（Matching Engine）是任何一个交易所、ATS（另类交易系统）或 ECN（电子通讯网络）的心脏。它的职责很简单：把愿意买的人和愿意卖的人配成对。但"简单"的背后是状态机、价格-时间优先算法、多种订单类型的语义差异、以及每秒钟数十万笔的低延迟要求。本课从数据结构出发，实现一个支持 6 种订单类型的撮合引擎，然后用 proptest 做随机化验证，用 criterion 做微基准测试。

**领域背景：** 撮合引擎的工作原理是"价格-时间优先"（Price-Time Priority）。价格优先：出价更高的买单优先成交，出价更低的卖单优先成交。时间优先：同一价格的订单，先到者优先成交。这两种规则的组合就是金融世界中最公平的交易规则——你不需要贿赂任何人，只需要出更好的价格或更早进入市场。但这只是基本逻辑——真实世界中的撮合引擎还需要处理市价单（立即成交）、冰山订单（隐藏大单）、止损单（触发条件）、以及各种不符合基本逻辑但交易所必须支持的订单类型。

## 1. 订单簿数据结构设计

### 1.1 从 BTreeMap 到有序价格链表

```
撮合引擎数据结构的演进（从小到大）：

v0.1: Vec<Order> 全扫描 → 学生项目，O(N) 撮合
v0.2: BTreeMap<Price, Vec<Order>> → 10.7 课的方案，入门可用
v0.3: 数组 + 索引 → 固定价格步长（tick size），O(1) 定位
v0.4: 侵入式双向链表 → 极致性能，O(1) 插入/删除/撮合

本课使用 v0.3 方案：
  价格被映射到数组索引（因为 tick size 是固定步长）
  例如 tick_size = 0.01, price 范围 [100.00, 200.00]
  → 10000 个可能的价位 → 数组大小 10000
  → 任何价格可以 O(1) 定位到索引
```

```rust
    /// 在订单簿中等待撮合
    Open,
    /// 部分成交，剩余数量仍在订单簿中
    PartiallyFilled,
    /// 完全成交，从订单簿中移除
    Filled,
    /// 已取消
    Cancelled,
    /// 被拒绝（例如 FOK 订单无法立即完全成交）
    Rejected,
}
```

### 1.3 订单生命周期状态机

```
OrderState 状态转换图：

    New ──────→ Open
                  │
                  ├────→ PartiallyFilled ──→ Filled
                  │                           ↑
                  │                           │
                  └────→ Cancelled            │
                  │                           │
                  └────→ Rejected             │
                  │                           │
                  PartiallyFilled ────────────┘
                         │
                         └────→ Cancelled

规则：
  - New → Open: 订单通过价格检查，进入订单簿排队
  - Open → PartiallyFilled: 撮合中部分成交
  - Open → Cancelled: 用户主动撤单
  - Open → Rejected: FOK 订单无法立即完全成交
  - PartiallyFilled → Filled: 剩余部分被撮合
  - PartiallyFilled → Cancelled: 剩余部分被撤单
```

### 1.4 订单与价格级别

```rust
/// 单个订单（冷热字段分离设计预留）
#[derive(Debug, Clone)]
struct Order {
    // === 热数据（撮合路径频繁访问）===
    id: u64,
    price: Price,
    remaining_qty: u64,    // 剩余未成交数量
    side: Side,
    state: OrderState,

    // === 温数据（偶尔访问）===
    order_type: OrderType,
    time_in_force: TimeInForce,

    // === 冷数据（撮合时不访问）===
    client_id: u64,
    original_qty: u64,     // 原始数量（用于成交率计算）
    created_at_ns: u64,
    display_qty: Option<u64>, // 冰山订单的显示数量
    stop_price: Option<Price>, // 止损触发价
}

/// 价格级别：一个价位上的所有订单队列
#[derive(Debug)]
struct PriceLevel {
    price: Price,
    /// 该价位所有订单的总数量（撮合引擎需要快速知道）
    total_quantity: u64,
    /// 订单队列：VecDeque 比 Vec 更适合"先进先出"
    /// push_back = 新订单进入队尾，pop_front = 成交取队头
    orders: VecDeque<Order>,
}

impl PriceLevel {
    fn new(price: Price) -> Self {
        Self {
            price,
            total_quantity: 0,
            orders: VecDeque::new(),
        }
    }

    /// 在队尾添加订单（时间优先：后来的排在后面）
    fn push_back(&mut self, order: Order) {
        self.total_quantity += order.remaining_qty;
        self.orders.push_back(order);
    }

    /// 从队头取出订单并减少数量
    /// 返回被完全消耗的订单列表
    fn consume_front(&mut self, mut qty_to_consume: u64) -> Vec<Order> {
        let mut filled_orders = Vec::new();

        while qty_to_consume > 0 {
            if let Some(mut order) = self.orders.pop_front() {
                let consumed = std::cmp::min(order.remaining_qty, qty_to_consume);
                order.remaining_qty -= consumed;
                self.total_quantity -= consumed;
                qty_to_consume -= consumed;

                if order.remaining_qty > 0 {
                    // 部分成交，放回队头
                    order.state = OrderState::PartiallyFilled;
                    self.orders.push_front(order);
                } else {
                    // 完全成交
                    order.state = OrderState::Filled;
                }
                filled_orders.push(order);
            } else {
                break; // 队列空了
            }
        }

        filled_orders
    }
}
```

## 2. 撮合引擎核心：价格-时间优先算法

### 2.1 撮合算法详解

```
算法输入：一个 incoming order（新到达的订单）
算法输出：一组成交（trades）+ 更新后的订单簿状态

价格-时间优先的撮合流程：

1. 确定 incoming order 的方向
   - 如果是买单：在卖单侧（asks）寻找对手方
   - 如果是卖单：在买单侧（bids）寻找对手方

2. 从对手方的最优价格开始遍历
   - 买单对手 → asks 的最低价（BTreeMap 第一个 key）
   - 卖单对手 → bids 的最高价（BTreeMap 最后一个 key）

3. 对于每一个对手方价位：
   a. 检查是否满足价格条件
      - 买单 incoming：incoming.price >= 对手方 price
      - 卖单 incoming：incoming.price <= 对手方 price
   b. 如果不满足 → 撮合结束（没有更多可对手方）
   c. 如果满足 → 从该价位的队列头部开始消耗订单
      - 成交价 = 被动方（对手方）的价格（价格优先）
      - 按 FIFO 顺序消耗（时间优先）
      - 每消耗一个对手订单，产生一笔 trade

4. incoming order 处理结果
   - 如果全部成交 → Filled
   - 如果部分成交 + 是限价单 → 剩余部分进入订单簿
   - 如果部分成交 + 是 IOC → 剩余部分取消
   - 如果部分成交 + 是 FOK → 全部取消（回滚已成交的）
```

### 2.2 撮合引擎实现

```rust
/// 成交记录
#[derive(Debug, Clone)]
struct Trade {
    trade_id: u64,
    price: Price,
    quantity: u64,
    /// 主动方订单 ID（incoming）
    aggressive_order_id: u64,
    /// 被动方订单 ID（resting，已在簿中）
    resting_order_id: u64,
    timestamp_ns: u64,
}

/// 撮合引擎
struct MatchingEngine {
    instrument: String,
    /// 买单簿：价格从高到低遍历
    /// BTreeMap 升序 → rev() 从高到低
    bids: BTreeMap<Price, PriceLevel>,
    /// 卖单簿：价格从低到高遍历
    asks: BTreeMap<Price, PriceLevel>,
    /// 按订单 ID 快速查找（用于撤单）
    order_index: BTreeMap<u64, (Side, Price)>,
    /// ID 生成器
    next_order_id: u64,
    next_trade_id: u64,
    /// 当前时间戳（纳秒，模拟用）
    current_time_ns: u64,
}

impl MatchingEngine {
    fn new(instrument: &str) -> Self {
        Self {
            instrument: instrument.to_string(),
            bids: BTreeMap::new(),
            asks: BTreeMap::new(),
            order_index: BTreeMap::new(),
            next_order_id: 1,
            next_trade_id: 1,
            current_time_ns: 0,
        }
    }

    /// 主入口：处理一个新订单
    fn process_order(&mut self, mut order: Order) -> Vec<Trade> {
        self.current_time_ns += 1;
        order.id = self.next_order_id;
        self.next_order_id += 1;
        order.created_at_ns = self.current_time_ns;

        // 1. 如果是触发型订单（Stop/Stop-Limit），检查触发条件
        match order.order_type {
            OrderType::Stop | OrderType::StopLimit => {
                // 止损单需要先检查是否触发
                // 简化处理：如果触发条件不满足，进入条件订单簿，不撮合
                // 生产系统中需要实时监控价格变化来触发
                // 此处我们假设用当前最优价判断是否触发
                if !self.is_stop_triggered(&order) {
                    // 未触发，放入条件订单簿（简化：直接拒绝）
                    order.state = OrderState::Rejected;
                    return Vec::new();
                }
                // 触发后，转为市价单或限价单
            }
            _ => {}
        }

        // 2. 确定匹配方向
        let opponent_side = match order.side {
            Side::Buy => &mut self.asks,   // 买单匹配卖单簿
            Side::Sell => &mut self.bids,  // 卖单匹配买单簿
        };
        let own_side = match order.side {
            Side::Buy => &mut self.bids,
            Side::Sell => &mut self.asks,
        };

        let mut remaining = order.remaining_qty;
        let mut trades = Vec::new();
        let mut all_or_none_failed = false;

        // 3. 遍历对手方价位（从最优价开始）
        //    对于买单，对手方是 asks → 从最低价开始（BTreeMap 第一个）
        //    对于卖单，对手方是 bids → 从最高价开始（BTreeMap 最后一个）
        let opponent_prices: Vec<Price> = match order.side {
            Side::Buy => opponent_side.keys().copied().collect(),
            Side::Sell => opponent_side.keys().rev().copied().collect(),
        };

        for &opp_price in &opponent_prices {
            // 检查价格条件：对于买单，incoming_price >= opp_price
            let can_match = match order.side {
                Side::Buy => order.price >= opp_price,
                Side::Sell => order.price <= opp_price,
            };

            if !can_match {
                break; // 价格条件不满足 → 结束撮合
            }

            // 从该价位消耗订单
            if let Some(opp_level) = opponent_side.get_mut(&opp_price) {
                let consumed = opp_level.consume_front(remaining);
                for filled_order in consumed {
                    let trade_qty = order.original_qty - filled_order.remaining_qty;
                    // 实际上应该用 original_qty 对比，简化处理
                    let trade_qty = filled_order.original_qty; // Use original for simplicity
                    // 修正：取的消耗数量 = original - remaining
                    // 这里简化，实际 consumed 的量由 consume_front 中的逻辑确定

                    if remaining > 0 {
                        let execute_qty = std::cmp::min(remaining, opp_level
                            .orders.front().map(|o| o.remaining_qty).unwrap_or(0));
                        // 简化：直接取 remaining 为成交数量
                        let actual_qty = std::cmp::min(remaining,
                            filled_order.original_qty);
                        remaining -= actual_qty;

                        trades.push(Trade {
                            trade_id: self.next_trade_id,
                            price: opp_price, // 被动方价格优先
                            quantity: actual_qty,
                            aggressive_order_id: order.id,
                            resting_order_id: filled_order.id,
                            timestamp_ns: self.current_time_ns,
                        });
                        self.next_trade_id += 1;
                    }
                }
            }

            if remaining == 0 {
                break; // 完全成交 → 结束撮合
            }
        }

        // 4. 处理 incoming order 的剩余部分
        if remaining > 0 {
            match order.time_in_force {
                TimeInForce::GTC => {
                    // GTC（Good-Till-Cancelled）：剩余部分进入订单簿
                    order.remaining_qty = remaining;
                    order.state = if remaining < order.original_qty {
                        OrderState::PartiallyFilled
                    } else {
                        OrderState::Open
                    };

                    let price = order.price;
                    self.order_index.insert(order.id, (order.side, order.price));

                    let book = match order.side {
                        Side::Buy => &mut self.bids,
                        Side::Sell => &mut self.asks,
                    };

                    book.entry(price)
                        .or_insert_with(|| PriceLevel::new(price))
                        .push_back(order);
                }
                TimeInForce::IOC => {
                    // IOC（Immediate-Or-Cancel）：剩余部分取消
                    order.state = OrderState::Cancelled;
                }
                TimeInForce::FOK => {
                    // FOK（Fill-Or-Kill）：没有完全成交 → 全部取消
                    // FOK 的语义是"要么全成交，要么全不成交"
                    // 但因为我们已经部分成交了，需要回滚（这是 FOK 的难点）
                    // 简化处理：在入口处先做预检查
                    all_or_none_failed = true;
                    order.state = OrderState::Rejected;
                    trades.clear(); // FOK 失败，回滚成交
                }
            }
        } else {
            // 完全成交
            order.state = OrderState::Filled;
        }

        trades
    }

    /// 取消指定订单
    fn cancel_order(&mut self, order_id: u64) -> bool {
        if let Some((side, price)) = self.order_index.remove(&order_id) {
            let book = match side {
                Side::Buy => &mut self.bids,
                Side::Sell => &mut self.asks,
            };
            if let Some(level) = book.get_mut(&price) {
                if let Some(pos) = level.orders.iter().position(|o| o.id == order_id) {
                    let mut order = level.orders.remove(pos).unwrap();
                    order.state = OrderState::Cancelled;
                    level.total_quantity -= order.remaining_qty;
                    // 如果该价位已空，移除价格级别
                    if level.orders.is_empty() {
                        book.remove(&price);
                    }
                    return true;
                }
            }
        }
        false
    }

    /// 检查 Stop 订单是否触发
    fn is_stop_triggered(&self, order: &Order) -> bool {
        let stop_price = match order.stop_price {
            Some(p) => p,
            None => return true, // 没有止损价 = 总是触发
        };
        let best_price = match order.side {
            Side::Buy => self.asks.keys().next().copied(),
            Side::Sell => self.bids.keys().next_back().copied(),
        };
        match (best_price, order.side) {
            (Some(bp), Side::Buy) => bp <= stop_price,
            (Some(bp), Side::Sell) => bp >= stop_price,
            _ => false,
        }
    }

    fn best_bid_ask(&self) -> (Option<Price>, Option<Price>) {
        let bid = self.bids.keys().next_back().copied();
        let ask = self.asks.keys().next().copied();
        (bid, ask)
    }
}
```

## 3. 订单类型及其语义

### 3.1 订单类型枚举

```rust
/// 订单类型
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum OrderType {
    /// 市价单（Market Order）：以当前最优价立即成交
    /// 无价格限制——如果流动性不足，可能以很差的价格成交
    /// 用途：需要立即执行的场景，不在乎成交价
    Market,

    /// 限价单（Limit Order）：以指定价格或更好的价格成交
    /// 最基础、最常用的订单类型
    Limit,

    /// 止损单（Stop Order）：当市场价格触及止损价时，转为市价单
    /// 用途：限制损失——"如果价格跌到 90，立刻卖出"
    /// 注意：触发后以市价执行，真实成交价可能更差
    Stop,

    /// 止损限价单（Stop-Limit）：当市场价格触及止损价时，转为限价单
    /// 相比 Stop，多了限价保护，防止市价滑点
    StopLimit,

    /// 冰山订单（Iceberg Order）：只显示一部分数量，隐藏其余
    /// 用途：大户隐藏真实意图，避免引起市场恐慌
    /// 例：display_qty = 100, total_qty = 10000
    ///     市场上只看到 100，成交后再自动补充 100
    Iceberg,

    /// 挂钩订单（Pegged Order）：价格跟随参考价动态调整
    /// 例：以买一的 99% 挂单 → 总是比最优买价低 1%
    Pegged,
}

/// 时效类型（Time in Force）
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum TimeInForce {
    /// Good-Till-Cancelled：一直有效直到成交或取消
    GTC,

    /// Immediate-Or-Cancel：立即成交（可部分），剩余取消
    IOC,

    /// Fill-Or-Kill：要么全部成交，要么完全拒绝
    FOK,
}
```

### 3.2 各订单类型的匹配逻辑

```rust
/// 每种订单类型的特殊处理逻辑
impl MatchingEngine {
    /// 处理市价单
    /// 市价单没有价格限制，会吃掉对手方簿中所有可用的流动性
    /// 这是最"危险"的订单类型——在低流动性市场中价差可能很大
    fn match_market_order(&mut self, order: &mut Order) -> Vec<Trade> {
        // 市价单：设置价格为极值，保证一定能匹配
        order.price = match order.side {
            Side::Buy => Price(i64::MAX),   // 买单 × 极高价
            Side::Sell => Price(0),          // 卖单 × 零价
        };
        // 然后走正常的撮合逻辑
        self.process_limit_order(order)
    }

    /// 处理限价单（标准撮合）
    fn process_limit_order(&mut self, order: &mut Order) -> Vec<Trade> {
        self.process_order(order.clone()) // 简化：复用主流程
    }

    /// 处理 FOK 订单
    /// FOK 的核心挑战：需要在匹配之前计算是否能完全成交
    /// 如果不能 → 直接拒绝，不产生任何成交
    /// 这需要一次"预匹配"（pre-match check）
    fn match_fok_order(&self, order: &Order) -> (bool, u64) {
        let opponent_side = match order.side {
            Side::Buy => &self.asks,
            Side::Sell => &self.bids,
        };

        let mut available_qty: u64 = 0;
        let opponent_prices: Vec<Price> = match order.side {
            Side::Buy => opponent_side.keys().copied().collect(),
            Side::Sell => opponent_side.keys().rev().copied().collect(),
        };

        for &opp_price in &opponent_prices {
            let can_match = match order.side {
                Side::Buy => order.price >= opp_price,
                Side::Sell => order.price <= opp_price,
            };
            if !can_match {
                break;
            }
            if let Some(level) = opponent_side.get(&opp_price) {
                available_qty += level.total_quantity;
                if available_qty >= order.remaining_qty {
                    return (true, order.remaining_qty); // 可以完全成交
                }
            }
        }

        (false, available_qty) // 不能完全成交
    }

    /// 冰山订单的显示数量计算
    /// 冰山订单在订单簿中只显示 display_qty
    /// 当 display_qty 被吃掉后，自动从隐藏部分补充
    /// 
    /// 行为示例：
    ///   total = 1000, display = 100
    ///   1. 簿中显示 qty=100
    ///   2. 市价单吃掉 60 → 簿中显示 qty=40（自动补充 60 到显示队列尾部）
    ///   3. 市价单再吃掉 40 → 簿中显示 qty=100（从隐藏部分补充）
    fn handle_iceberg_refill(&self, order: &mut Order) {
        if let Some(display_qty) = order.display_qty {
            if order.remaining_qty > 0 && order.remaining_qty < display_qty {
                // 显示数量被部分吃掉后，自动补充
                // 注意：补充的新数量排到队尾（时间优先被重置）
                // 这是 iceberg 订单的一个微妙特征
            }
        }
    }

    /// Pegged 订单的价格计算
    /// Pegged 订单的价格不是固定的，而是相对于参考价格动态计算
    ///
    /// 例：Buy Pegged at BestBid - 0.01
    ///     当前 BestBid = 150.00 → 挂单价 = 149.99
    ///     如果 BestBid 变为 150.50 → 挂单价 = 150.49
    ///
    /// 竞争优势：始终比当前最优价格"差一点"，等待被动成交
    fn compute_pegged_price(&self, pegged_order: &Order) -> Option<Price> {
        let (best_bid, best_ask) = self.best_bid_ask();
        match (pegged_order.side, best_bid, best_ask) {
            // Buy pegged: 挂单比最优买价稍微低一点（排队在别人后面）
            (Side::Buy, Some(bid), _) => {
                Some(Price(bid.0 - 100)) // 比最优买价低 0.01
            }
            // Sell pegged: 挂单比最优卖价稍微高一点（排队在别人后面）
            (Side::Sell, _, Some(ask)) => {
                Some(Price(ask.0 + 100)) // 比最优卖价高 0.01
            }
            _ => None, // 参考价格不可用
        }
    }
}
```

### 4.2 Arena 分配：消除 per-order 的堆分配

```rust
/// 订单 Arena：预分配一块内存，订单从中获取槽位
/// 
/// 关键洞察：
///   每秒 100K 个新订单 → 每秒 100K 次 malloc
///   → 每秒 100K 次 kernel 交互（mmap/brk）
///   → 争锁、碎片、cache thrash
///
/// Arena 替代方案：
///   预分配一个 [Order; 100K] 的大数组
///   每个新订单从数组取一个空闲槽位
///   → 零 syscall、良好局部性、可预测的延迟
struct OrderArena {
    /// 预分配的订单存储
    storage: Vec<Order>,
    /// 空闲槽位索引列表（被取消的订单释放的槽位）
    free_slots: Vec<usize>,
}

impl OrderArena {
    /// 创建一个能容纳 capacity 个订单的 arena
    fn new(capacity: usize) -> Self {
        // 预分配并填充默认值
        // 生产环境中这里会用 unsafe 避免初始化开销
        let storage = vec![
            Order {
                id: 0,
                price: Price(0),
                remaining_qty: 0,
                side: Side::Buy,
                state: OrderState::Cancelled,
                order_type: OrderType::Limit,
                time_in_force: TimeInForce::GTC,
                client_id: 0,
                original_qty: 0,
                created_at_ns: 0,
                display_qty: None,
                stop_price: None,
            };
            capacity
        ];

        Self {
            storage,
            free_slots: (0..capacity).collect(),
        }
    }

    /// 分配一个新订单槽位
    fn allocate(&mut self) -> Option<usize> {
        self.free_slots.pop()
    }

    /// 释放订单槽位
    fn deallocate(&mut self, index: usize) {
        // 标记为空闲（重置部分字段以帮助调试）
        self.storage[index].state = OrderState::Cancelled;
        self.free_slots.push(index);
    }

    /// 获取订单的引用
    fn get(&self, index: usize) -> &Order {
        &self.storage[index]
    }

    fn get_mut(&mut self, index: usize) -> &mut Order {
        &mut self.storage[index]
    }
}
```

### 4.3 Cache-Line 友好的数据布局

```rust
/// 热冷分离布局（Hot-Cold Split）
/// 
/// CPU 缓存行通常是 64 字节。
/// 撮合引擎在热路径上只需要访问：price、remaining_qty、side
/// → 3 × 8 bytes = 24 bytes
/// → 一个缓存行可以装 2.5 个热订单
/// 
/// 但标准布局把所有字段放在一起：
/// → order 结构体 ~80 bytes → 一个缓存行装不下 1 个
/// → 访问 hot 字段时，cold 字段被不需要地加载到缓存
/// → 浪费了宝贵的 cache 容量

/// 热数据（撮合热路径访问）
#[derive(Debug, Clone)]
struct OrderHot {
    price: Price,
    remaining_qty: u64,
    side: Side,
    state: OrderState,
}

/// 冷数据（撮合时不访问，只有管理操作时访问）
#[derive(Debug, Clone)]
struct OrderCold {
    client_id: u64,
    original_qty: u64,
    created_at_ns: u64,
    order_type: OrderType,
    time_in_force: TimeInForce,
    display_qty: Option<u64>,
    stop_price: Option<Price>,
}

/// 通过索引关联热冷数据
/// 热路径只需要扫描 hot_orders 数组
struct HotColdSplitBook {
    /// 索引到热数据（紧凑排列，cache 友好）
    hot_orders: Vec<OrderHot>,
    /// 索引到冷数据（热路径不访问）
    cold_orders: Vec<OrderCold>,
}

impl HotColdSplitBook {
    /// 遍历所有活跃订单的剩余数量（O(N) 但 cache 友好）
    fn total_hot_quantity(&self) -> u64 {
        // 这个循环几乎全部命中 L1 缓存
        // 因为 hot_orders 是紧凑的 Vec
        self.hot_orders.iter()
            .filter(|o| o.state == OrderState::Open 
                     || o.state == OrderState::PartiallyFilled)
            .map(|o| o.remaining_qty)
            .sum()
    }
}
```

## 5. 用 proptest 做随机化验证

### 5.1 订单簿不变量

```
在任何时刻，一个正确的订单簿必须满足以下不变量：

1. best_bid < best_ask（不能有交叉订单簿）
   → 如果 best_bid >= best_ask，说明撮合逻辑有 bug

2. 所有 quantity >= 0
   → 不能有负数量

3. 同一价位的订单按到达顺序排列（FIFO）
   → 时间戳递增

4. 成交量守恒：总成交 = 总被吃掉的数量
   → incoming.original_qty = Σtrade.qty + incoming.remaining_qty

5. 价格有效性：incoming 的成交价格都在其限价范围内
   → 买单成交价 <= incoming.price
   → 卖单成交价 >= incoming.price
```

### 6.2 延迟目标与测量结果解读

```
撮合引擎延迟基准（单核，同一 CPU）：

操作类型              | 简单实现(BTreeMap) | 优化后(数组+arena) | 目标
──────────────────────┼───────────────────┼───────────────────┼────────
LimitOrder(未撮合)     |    200-400 ns     |    50-100 ns      | <200 ns
MarketOrder(撮合10笔)  |    800-1200 ns    |    200-400 ns     | <500 ns
CancelOrder            |    100-200 ns     |    30-50 ns       | <100 ns
全簿遍历(1000价位)     |   5000-8000 ns    |   1000-2000 ns    | <5000 ns

关键指标：
  P50（中位数延迟）：代表"正常情况"下的延迟
  P99（99 百分位延迟）：代表"接近最差情况"的延迟
  P99.9（99.9 百分位延迟）：代表 tail latency

量化金融关注 P99.9：
  如果 99.9% 的请求在 10μs 内完成，那么 0.1% 的请求超过 10μs
  → 每 1000 次交易就有 1 次"慢"的
  → 一天成千上万比交易 → 那一次"慢"可能挨上市场波动 → 损失
```
