# 10.14 区块链基础：从密码学到共识

> **[Track C: Web3]** 本节从零开始构建区块链的底层积木——密码学原语、Merkle 树、账户模型、共识机制。Track C 不假设你有任何区块链知识：每个概念都从"为什么需要它"讲起，再用 Rust 实现。到本节结束时，你将拥有一个完全可运行的玩具区块链（~300 行 Rust 代码），它包含 PoW 挖矿、UTXO 管理和链验证。这不仅是为了理解区块链——更是为了理解为什么 Rust 是这个领域的主导语言。

**领域背景：** 区块链的本质是一个"不可篡改的分布式账本"——数据一旦写入就无法删除或修改，所有参与者拥有相同的副本。技术上，它由三个核心组件构成：密码学（保证数据完整性和身份认证）、数据结构（Merkle 树高效验证数据完整性）、共识机制（让互不信任的节点就"哪条链是正确的"达成一致）。这三者中的每一个，Rust 都有最佳的实现效率和安全保障。

## 1. 密码学原语：在 Rust 中从零实现

### 1.1 SHA-256：理解 Merkle-Damgard 构造

SHA-256 是比特币和大多数区块链的基石。它的核心是 Merkle-Damgard 构造：将任意长度的消息分成 512-bit 块，逐块处理，每块处理时更新一个 256-bit 的状态向量，最终输出 256-bit 的哈希。

```
┌─────────────────────────────────────────────────────────────────┐
│  Merkle-Damgard 构造（SHA-256）                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  消息 M → [填充到512-bit的倍数] → M₁│M₂│...│Mₙ                  │
│                                                                 │
│  IV(初始向量) ──→ [压缩函数f] ──→ H₁ ──→ [压缩函数f] ──→ ...    │
│                    ↑                ↑                           │
│                    M₁               M₂                          │
│                                                                 │
│  每轮: Hᵢ = f(Hᵢ₋₁, Mᵢ)，其中 H₀ = IV，输出 = Hₙ              │
│  如果f是抗碰撞的，则整个构造也是抗碰撞的                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

下面是从零实现 SHA-256 的核心代码（仅展示压缩函数和主循环，完整实现约 150 行）：

```rust
// sha256_core.rs —— 从零实现 SHA-256 的核心
// 理解区块链的密码学基础，从自己写哈希函数开始

/// SHA-256 初始哈希值（前8个质数的平方根的小数部分的前32位）
const INITIAL_HASH: [u32; 8] = [
    0x6a09e667, 0xbb67ae85, 0x3c6ef372, 0xa54ff53a,
    0x510e527f, 0x9b05688c, 0x1f83d9ab, 0x5be0cd19,
];

/// SHA-256 轮常量（前64个质数的立方根的小数部分的前32位）
const K: [u32; 64] = [

    0x3956c25b, 0x59f111f1, 0x923f82a4, 0xab1c5ed5,
    0xd807aa98, 0x12835b01, 0x243185be, 0x550c7dc3,
    0x72be5d74, 0x80deb1fe, 0x9bdc06a7, 0xc19bf174,
    0xe49b69c1, 0xefbe4786, 0x0fc19dc6, 0x240ca1cc,
    0x2de92c6f, 0x4a7484aa, 0x5cb0a9dc, 0x76f988da,
    0x983e5152, 0xa831c66d, 0xb00327c8, 0xbf597fc7,
    0xc6e00bf3, 0xd5a79147, 0x06ca6351, 0x14292967,
    0x27b70a85, 0x2e1b2138, 0x4d2c6dfc, 0x53380d13,
    0x650a7354, 0x766a0abb, 0x81c2c92e, 0x92722c85,
    0xa2bfe8a1, 0xa81a664b, 0xc24b8b70, 0xc76c51a3,
    0xd192e819, 0xd6990624, 0xf40e3585, 0x106aa070,
    0x19a4c116, 0x1e376c08, 0x2748774c, 0x34b0bcb5,
    0x391c0cb3, 0x4ed8aa4a, 0x5b9cca4f, 0x682e6ff3,
    0x748f82ee, 0x78a5636f, 0x84c87814, 0x8cc70208,
    0x90befffa, 0xa4506ceb, 0xbef9a3f7, 0xc67178f2,
];

// 右旋转（循环右移）
fn rotr(x: u32, n: u32) -> u32 {
    (x >> n) | (x << (32 - n))
}

// SHA-256 压缩函数：处理一个 512-bit（64字节）的块
fn compress(state: &mut [u32; 8], block: &[u8; 64]) {
    // 步骤1: 将64字节的块转换为16个32-bit大端字
    let mut w = [0u32; 64];
    for i in 0..16 {
        w[i] = u32::from_be_bytes([
            block[i * 4], block[i * 4 + 1], block[i * 4 + 2], block[i * 4 + 3],
        ]);
    }

    // 步骤2: 消息扩展——将16个字扩展为64个字
    for i in 16..64 {
        let s0 = rotr(w[i - 15], 7) ^ rotr(w[i - 15], 18) ^ (w[i - 15] >> 3);
        let s1 = rotr(w[i - 2], 17) ^ rotr(w[i - 2], 19) ^ (w[i - 2] >> 10);
        w[i] = w[i - 16].wrapping_add(s0).wrapping_add(w[i - 7]).wrapping_add(s1);
    }

    // 步骤3: 初始化工作变量为当前状态值
    let [mut a, mut b, mut c, mut d, mut e, mut f, mut g, mut h] = *state;

    // 步骤4: 64轮压缩——每轮混合状态和消息字
    // 这就是 Merkle-Damgard 构造的核心：f(state, message_block)
    for i in 0..64 {
        let s1 = rotr(e, 6) ^ rotr(e, 11) ^ rotr(e, 25);
        let ch = (e & f) ^ ((!e) & g);  // Choice: if e then f else g
        let temp1 = h.wrapping_add(s1)
            .wrapping_add(ch)
            .wrapping_add(K[i])
            .wrapping_add(w[i]);
        let s0 = rotr(a, 2) ^ rotr(a, 13) ^ rotr(a, 22);
        let maj = (a & b) ^ (a & c) ^ (b & c);  // Majority: 多数的bit
        let temp2 = s0.wrapping_add(maj);

        // 移位：每轮状态向下传递
        h = g;
        g = f;
        f = e;
        e = d.wrapping_add(temp1);
        d = c;
        c = b;
        b = a;
        a = temp1.wrapping_add(temp2);
    }

    // 步骤5: 将本轮结果加回状态（Davies-Meyer 构造）
    state[0] = state[0].wrapping_add(a);
    state[1] = state[1].wrapping_add(b);
    state[2] = state[2].wrapping_add(c);
    state[3] = state[3].wrapping_add(d);
    state[4] = state[4].wrapping_add(e);
    state[5] = state[5].wrapping_add(f);
    state[6] = state[6].wrapping_add(g);
    state[7] = state[7].wrapping_add(h);
}

/// SHA-256 主函数：计算任意字节序列的哈希
pub fn sha256(data: &[u8]) -> [u8; 32] {
    let mut state = INITIAL_HASH;
    let mut msg = data.to_vec();

    // Merkle-Damgard 填充规则：
    // 1. 追加一个 1-bit（0x80 字节）
    // 2. 填充 0-bits 直到消息长度 ≡ 448 mod 512
    // 3. 追加 64-bit 原始消息长度（大端序）
    let bit_len = (msg.len() as u64) * 8;
    msg.push(0x80u8);
    while (msg.len() % 64) != 56 {
        msg.push(0x00u8);
    }
    msg.extend_from_slice(&bit_len.to_be_bytes());

    // 逐块处理：对每个 512-bit 块调用压缩函数
    for chunk in msg.chunks(64) {
        let block: [u8; 64] = chunk.try_into().unwrap();
        compress(&mut state, &block);
    }

    // 将最终状态拼接为 256-bit 输出（8个32-bit字 → 32字节）
    let mut result = [0u8; 32];
    for i in 0..8 {
        result[i * 4..(i + 1) * 4].copy_from_slice(&state[i].to_be_bytes());
    }
    result
}

mod tests {

        let hash = sha256(b"");

            "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"

```

**关键理解：** SHA-256 的安全性来自两个数学性质：(1) 原像抗性——给定哈希值 H，无法在合理的计算量内找到 M 使 SHA256(M) = H；(2) 碰撞抗性——找不到 M1 != M2 使得 SHA256(M1) = SHA256(M2)。这正是区块链"不可篡改"的密码学基础——如果有人修改了区块中的任何数据，哈希值将完全不同。

### 1.2 ECDSA 与 Ed25519：数字签名

区块链中的"交易"本质上是一个数字签名——用私钥证明"我授权了这笔转账"。Rust 生态中有两组一流的签名 crate：

```rust
// digital_signatures.rs —— ECDSA 和 Ed25519 签名对比
// 两个主流椭圆曲线签名方案，在区块链中各有应用场景

use k256::ecdsa::{SigningKey, VerifyingKey, Signature, signature::Signer, signature::Verifier};
use ed25519_dalek::{SigningKey as EdSigningKey, VerifyingKey as EdVerifyingKey};

/// ECDSA (secp256k1 曲线) —— 比特币和以太坊使用的签名方案
fn ecdsa_demo() {
    // 步骤1: 生成密钥对（私钥 → 公钥，不能反向推导）
    let signing_key = SigningKey::random(&mut rand::thread_rng());
    let verifying_key = VerifyingKey::from(&signing_key);

    println!("=== ECDSA (secp256k1) ===");
    println!("私钥长度: 32 字节");
    println!("公钥长度: 33 字节 (压缩格式)");

    // 步骤2: 对消息签名
    let message = b"Alice pays Bob 10 BTC";
    let signature: Signature = signing_key.sign(message);
    println!("签名长度: {} 字节", signature.to_bytes().len());  // ~64-72字节 (DER编码)

    // 步骤3: 验证签名——任何人都可以用公钥验证
    assert!(verifying_key.verify(message, &signature).is_ok());
    println!("✓ 签名验证通过");

    // 步骤4: 篡改消息会被检测
    let tampered = b"Alice pays Bob 100 BTC";
    assert!(verifying_key.verify(tampered, &signature).is_err());
    println!("✓ 篡改后的消息验证失败（正确行为）");

    // 步骤5: 以太坊地址从公钥派生
    // 以太坊地址 = keccak256(公钥)的后20字节
    use sha3::{Keccak256, Digest};
    let public_key_bytes = verifying_key.to_encoded_point(false);
    // 取未压缩公钥的x,y坐标（去掉04前缀）
    let mut hasher = Keccak256::new();
    hasher.update(&public_key_bytes.as_bytes()[1..]);
    let hash = hasher.finalize();
    let eth_address = &hash[12..];  // 后20字节
    println!("以太坊地址: 0x{}", hex::encode(eth_address));
}

/// Ed25519 —— Solana、NEAR、Polkadot 使用的签名方案
/// 相比 ECDSA 的优势：固定大小签名（64字节）、批量验证、确定性签名
fn ed25519_demo() {
    // 步骤1: 生成密钥对
    let signing_key = EdSigningKey::generate(&mut rand::thread_rng());
    let verifying_key: EdVerifyingKey = (&signing_key).into();

    println!("\n=== Ed25519 (爱德华兹曲线) ===");
    println!("私钥长度: 32 字节");
    println!("公钥长度: 32 字节（单坐标，比 ECDSA 更短）");

    // 步骤2: 签名
    let message = b"Alice pays Bob 100 SOL";
    // Ed25519 不需要随机数 → 确定性签名，天然抵抗弱随机数攻击
    let signature = signing_key.sign(message);
    println!("签名长度: 64 字节（固定长度，无需 DER 编码）");

    // 步骤3: 验证
    assert!(verifying_key.verify(message, &signature).is_ok());
    println!("✓ 签名验证通过");

    // 步骤4: Ed25519 的批量验证优势
    // 可以一次性验证 N 个签名，性能远超逐个验证
    // 这是 Solana 等高性能链选择 Ed25519 的核心原因之一
    println!("✓ Ed25519 支持批量验证（1次椭圆曲线操作验证N个签名）");
}

fn main() {
    ecdsa_demo();
    ed25519_demo();
}
```

**为什么区块链使用不同的签名方案？** 比特币选择了 secp256k1 ECDSA（2009年最成熟的椭圆曲线方案）。Solana/NEAR/Polkadot 选择了 Ed25519（更现代、签名更短、批量验证性能更好、无随机数重用的安全隐患）。这不是"谁好谁坏"的问题——是工程上的权衡。

## 2. Merkle 树：密码学承诺与证明

### 2.1 从零实现 Merkle 树

Merkle 树是区块链的"数据指纹"：用一棵二叉树将大量数据（交易）压缩为单个 32 字节的根哈希（Merkle Root）。任何人只需要 O(log N) 个哈希值就能证明某个数据在树中——这就是"轻客户端"（SPV）的密码学基础。

```
┌─────────────────────────────────────────────────────────────────┐
│  Merkle 树结构                                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                      Root = H(H01│H23)                          │
│                     ┌───────┴───────┐                            │
│                  H01=H(L0│L1)    H23=H(L2│L3)                   │
│                 ┌───┴───┐       ┌───┴───┐                       │
│                L0      L1      L2      L3  ← 叶子               │
│              H(Tx0)  H(Tx1)  H(Tx2)  H(Tx3)                    │
│                                                                 │
│  Merkle 证明（如何证明 L2 在树中）：                              │
│    提供: [L2, H(L3), H(H(L0)│H(L1))]                           │
│    验证: H(H(H(L0)│H(L1))│H(L2│L3)) == Root？                  │
│    复杂度: O(log N) 个哈希 vs O(N) 全量                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```rust
// merkle_tree.rs —— 从零实现 Merkle 树（完整可运行）
use sha2::{Sha256, Digest};

/// Merkle 树节点：叶子节点存储原始数据，内部节点存储子节点哈希
#[derive(Debug, Clone)]
struct MerkleTree {
    /// 扁平数组存储所有节点（层次遍历序）
    /// leaves[0..n] 是叶子，内部节点在之后
    nodes: Vec<[u8; 32]>,
    leaf_count: usize,
}

impl MerkleTree {
    /// 从叶子数据构建 Merkle 树
    /// 如果叶子数不是 2 的幂，复制最后一个叶子补齐
    pub fn build(leaves: &[Vec<u8>]) -> Self {
        if leaves.is_empty() {
            return Self { nodes: vec![], leaf_count: 0 };
        }

        // 步骤1: 计算叶子哈希
        let leaf_hashes: Vec<[u8; 32]> = leaves
            .iter()
            .map(|data| {
                let mut hasher = Sha256::new();
                // 域分离：给叶子节点加前缀 0x00，防止与内部节点碰撞
                hasher.update(&[0x00u8]);
                hasher.update(data);
                hasher.finalize().into()
            })
            .collect();

        // 步骤2: 补齐到 2 的幂
        let n = leaf_hashes.len();
        let padded_size = n.next_power_of_two();
        let mut padded_leaves = leaf_hashes.clone();
        while padded_leaves.len() < padded_size {
            padded_leaves.push(leaf_hashes[n - 1]);  // 重复最后一个叶子
        }

        // 步骤3: 自底向上构建内部节点
        let total_nodes = 2 * padded_size - 1;
        // 为简单起见，我们使用索引方案：
        // 叶子节点索引: 0..padded_size
        // 内部节点索引: padded_size..2*padded_size-1
        let mut nodes = vec![[0u8; 32]; total_nodes];
        let offset = padded_size - 1;  // 内部节点数组的起始位置

        // 填充叶子（在叶子层的后半部分存储补齐后的叶子）
        // 使用更直观的方式：数组末尾是叶子，开头是内部节点
        // 对于满二叉树：nodes[0] = root, nodes[2*i+1] = 左子, nodes[2*i+2] = 右子
        let mut all_nodes = vec![[0u8; 32]; 2 * padded_size];
        for i in 0..padded_size {
            all_nodes[padded_size + i] = padded_leaves[i];
        }

        // 自底向上：对每个内部节点，H(左子 || 右子)
        let mut hasher = Sha256::new();
        for i in (0..padded_size).rev() {
            hasher = Sha256::new();
            // 域分离：给内部节点加前缀 0x01
            hasher.update(&[0x01u8]);
            // 如果 children 是空的（未补齐场景），需要特殊处理
            // 这里简化：对补齐的叶子不特殊处理
            hasher.update(&all_nodes[2 * i]);
            hasher.update(&all_nodes[2 * i + 1]);
            all_nodes[i] = hasher.finalize_reset().into();
        }

        // 移除索引 0 处未使用的 padding 空间
        // 上面的实现中 all_nodes[0..padded_size] 是内部节点
        // 但索引 0 对应的叶子没有被存储
        // 重新整理：让 nodes[0] 是 root
        let result_nodes = all_nodes[..padded_size + n].to_vec();

        Self {
            nodes: result_nodes,
            leaf_count: n,
        }
    }

    // 简化版实现：使用已有的 merkle 实现思路
    // （完整实现太冗长，这里使用更直接的写法）
    
    /// 简化版：直接构建而非复杂的索引管理
    pub fn build_simple(leaves: &[Vec<u8>]) -> Self {
        if leaves.is_empty() {
            return Self { nodes: vec![], leaf_count: 0 };
        }

        let leaf_hashes: Vec<[u8; 32]> = leaves
            .iter()
            .map(|data| {
                let mut h = Sha256::new();
                h.update(&[0x00]);
                h.update(data);
                h.finalize().into()
            })
            .collect();

        let n = leaf_hashes.len();
        let mut current_level = leaf_hashes.clone();

        // 补齐到偶数
        if current_level.len() % 2 == 1 {
            current_level.push(current_level.last().unwrap().clone());
        }

        let mut all_nodes: Vec<[u8; 32]> = current_level.clone();

        // 逐层构建：每层节点数是上一层的一半
        while current_level.len() > 1 {
            let mut next_level = Vec::with_capacity(current_level.len() / 2);
            for pair in current_level.chunks(2) {
                let mut h = Sha256::new();
                h.update(&[0x01]);
                h.update(&pair[0]);
                h.update(if pair.len() > 1 { &pair[1] } else { &pair[0] });
                next_level.push(h.finalize().into());
            }
            // 将这一层插入 all_nodes 头部（用于 Merkle 证明的兄弟节点查找）
            all_nodes.splice(0..0, next_level.clone());
            current_level = next_level;
        }

        Self {
            nodes: all_nodes,
            leaf_count: n,
        }
    }

    /// 获取 Merkle Root（树的根哈希）
    pub fn root(&self) -> [u8; 32] {
        self.nodes[0]  // 根节点永远是 nodes[0]
    }

    /// 为指定叶子生成 Merkle 证明
    /// 返回：(叶子哈希, 证明路径[(兄弟哈希, 是右兄弟?)])
    pub fn generate_proof(&self, leaf_index: usize) -> Vec<([u8; 32], bool)> {
        assert!(leaf_index < self.leaf_count, "叶子索引越界");

        // 在全节点列表中，叶子在末尾
        let total = self.nodes.len();
        let leaf_start = total - self.leaf_count;  // 叶子开始位置
        // 补齐后的叶子起始位置（考虑了补齐层）
        // 简化：从 bottom layer 向上
        let mut proof = Vec::new();
        let mut idx = total - self.leaf_count + leaf_index;
        
        while idx > 0 {
            let sibling_idx = if idx % 2 == 0 { idx - 1 } else { idx + 1 };
            if sibling_idx < self.nodes.len() {
                let is_right = idx % 2 == 0;  // 当前是左子(idx为奇)，兄弟是右子
                proof.push((self.nodes[sibling_idx], is_right));
            }
            idx = (idx - 1) / 2;  // 父节点
        }
        proof
    }

    /// 验证 Merkle 证明
    pub fn verify_proof(
        root: &[u8; 32],
        leaf_hash: &[u8; 32],
        proof: &[([u8; 32], bool)],
    ) -> bool {
        let mut current = *leaf_hash;
        for (sibling_hash, is_right) in proof {
            let mut h = Sha256::new();
            h.update(&[0x01u8]);
            if *is_right {
                h.update(&current);
                h.update(sibling_hash);
            } else {
                h.update(sibling_hash);
                h.update(&current);
            }
            current = h.finalize().into();
        }
        &current == root
    }
}

#[cfg(test)]
```

### 2.2 稀疏 Merkle 树（Sparse Merkle Tree）

以太坊的状态存储使用了一种特殊的 Merkle 树——稀疏 Merkle 树（Sparse Merkle Tree, SMT）。不同于普通 Merkle 树（叶子是交易的哈希列表），SMT 的每个叶子对应一个键值对。树的深度固定（通常 256 层，对应 256-bit 哈希空间），但绝大多数叶子是空的（默认值），所以叫"稀疏"。

```
┌─────────────────────────────────────────────────────────────────┐
│  稀疏 Merkle 树（SMT）：State = Key → Value 的密码学承诺         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Key 0x0000...0000 → Value(空)        Key 0xffff...ffff → V     │
│  ┌──────────────┐                     ┌─────────────┐           │
│  │ 叶子: 默认值  │                     │ V = balance  │           │
│  └──────┬───────┘                     └──────┬──────┘           │
│         └────────┬──────────────────────────┘                   │
│                  │                                              │
│               ┌──┴──┐                                          │
│               │ 内部 │ ← 大部分路径导向默认值，可压缩存储         │
│               └──┬──┘                                          │
│                  │  ← 到达根节点的路径                            │
│            ┌─────┴─────┐                                        │
│            │  Root Hash │ ← 所有状态的密码学承诺                  │
│            └───────────┘                                        │
│                                                                 │
│  关键性质：                                                       │
│  - 缺失证明：证明某个 key 不存在（其叶子是默认值）                  │
│  - 包含证明：证明某个 key 的值是 V（路径上所有兄弟哈希）            │
│  - 更新高效：O(log sparse_size)，但 256 层深度意味着 O(256)       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 3. 账户模型 vs UTXO 模型

这是区块链最基本的两种"记账方式"。

### 3.1 比特币的 UTXO 模型（无界面的状态）

UTXO (Unspent Transaction Output) 模型将"钱"建模为未花费的输出。每笔交易消费一些 UTXO，产生新的 UTXO。就像现金：你有一堆不同面额的钞票（UTXO），花掉时可能要找零。

```rust
// utxo_model.rs —— UTXO 模型的 Rust 表达

use std::collections::HashMap;

/// UTXO：一个"未花费的交易输出"
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
struct Utxo {
    txid: [u8; 32],    // 产生这个 UTXO 的交易 ID
    index: u32,         // 在这个交易中是第几个输出
    amount: u64,        // 金额（单位：聪，1 BTC = 10^8 聪）
    owner_pubkey: Vec<u8>,  // 谁可以花费它（公钥哈希）
}

/// 一笔比特币风格的交易
#[derive(Debug, Clone)]
struct Transaction {
    inputs: Vec<TxInput>,       // 消费哪些 UTXO
    outputs: Vec<TxOutput>,     // 产生哪些新 UTXO
    // 注意：比特币中没有"账户余额"这个概念！
    // "你有多少 BTC" = 遍历整个 UTXO 集，找你公钥对应的所有 UTXO 的总和
}

#[derive(Debug, Clone)]
struct TxInput {
    prev_txid: [u8; 32],   // 引用之前的交易
    prev_index: u32,       // 引用那个交易的第几个输出
    signature: Vec<u8>,    // 用 owner 的私钥对交易签名
    pubkey: Vec<u8>,       // 公钥（用于验证签名）
}

#[derive(Debug, Clone)]
struct TxOutput {
    amount: u64,
    owner_pubkey: Vec<u8>,  // 锁定脚本（简化版）
}

/// UTXO 集合：全局状态
struct UtxoSet {
    utxos: HashMap<(Vec<u8>, u32), Utxo>,  // (txid, index) → UTXO
}

impl UtxoSet {
    /// 验证一笔交易
    fn validate_transaction(&self, tx: &Transaction) -> Result<(), String> {
        let mut total_input = 0u64;
        let total_output: u64 = tx.outputs.iter().map(|o| o.amount).sum();

        for input in &tx.inputs {
            let key = (input.prev_txid.to_vec(), input.prev_index);
            match self.utxos.get(&key) {
                Some(utxo) => {
                    // 验证公钥匹配（简化：跳过实际的签名验证）
                    if utxo.owner_pubkey != input.pubkey {
                        return Err(format!("公钥不匹配"));
                    }
                    total_input += utxo.amount;
                }
                None => return Err(format!("UTXO 不存在或已花费")),
            }
        }

        if total_input < total_output {
            return Err(format!(
                "输入金额不足: {} < {} (差额为矿工费)",
                total_input, total_output
            ));
        }

        // total_input - total_output 是矿工费（不返回给任何人）
        Ok(())
    }

    /// 应用交易：移除旧的 UTXO，添加新的 UTXO
    fn apply_transaction(&mut self, tx: &Transaction) {
        // 步骤1: 移除被消费的 UTXO
        for input in &tx.inputs {
            let key = (input.prev_txid.to_vec(), input.prev_index);
            self.utxos.remove(&key);
        }

        // 步骤2: 添加新产生的 UTXO
        let txid: [u8; 32] = Sha256::digest(
            &bincode::serialize(tx).unwrap()
        ).into();
        
        for (i, output) in tx.outputs.iter().enumerate() {
            self.utxos.insert(
                (txid.to_vec(), i as u32),
                Utxo {
                    txid,
                    index: i as u32,
                    amount: output.amount,
                    owner_pubkey: output.owner_pubkey.clone(),
                },
            );
        }
    }
}

// 关键认知：
//  UTXO 模型的优势: 并行处理（不同 UTXO 互不依赖）、隐私（每个 UTXO 是独立的）
//  UTXO 模型的劣势: 编程复杂（智能合约非常难实现）
//  → 这就是以太坊选择账户模型的原因
```

### 3.2 以太坊的账户模型（状态机）

以太坊像银行账户一样管理状态：每个地址有一个余额（和其他数据），交易直接修改这些余额。这天然适合智能合约——合约就是"一段代码 + 一个账户"。

```rust
// account_model.rs —— 以太坊账户模型的 Rust 表达

use std::collections::HashMap;

/// 以太坊账户
#[derive(Debug, Clone)]
struct Account {
    nonce: u64,              // 交易计数器（防重放攻击）
    balance: u64,            // 余额（单位：wei, 1 ETH = 10^18 wei）
    storage_root: [u8; 32],  // 合约存储的 Merkle Trie 根（EOA 为空）
    code_hash: [u8; 32],     // 合约字节码的哈希（EOA 为 keccak256("")）
}

/// 全球状态：所有账户的映射
struct WorldState {
    // 在真正的以太坊中，这是一个 Merkle Patricia Trie
    // 这里简化为 HashMap 来表达概念
    accounts: HashMap<[u8; 20], Account>,  // 20字节地址 → 账户
}

impl WorldState {
    /// 执行一笔转账交易
    fn transfer(
        &mut self,
        from: &[u8; 20],
        to: &[u8; 20],
        amount: u64,
    ) -> Result<(), String> {
        // 步骤1: 检查发送方余额
        let sender = self.accounts.get(from)
            .ok_or("发送方不存在")?;
        if sender.balance < amount {
            return Err(format!("余额不足: {} < {}", sender.balance, amount));
        }

        // 步骤2: 更新 nonce（防重放）
        let sender = self.accounts.get_mut(from).unwrap();
        sender.nonce += 1;
        sender.balance -= amount;

        // 步骤3: 更新接收方余额（如果不存在则创建）
        let receiver = self.accounts.entry(*to).or_insert(Account {
            nonce: 0,
            balance: 0,
            storage_root: [0u8; 32],
            code_hash: keccak256(b""),
        });
        receiver.balance += amount;

        Ok(())
    }
}

// 两个模型的本质区别：
//
//  UTXO 模型:
//    → 状态被拆分成"硬币"颗粒
//    → 交易是"拆毁旧硬币，铸造新硬币"
//    → 验证只需检查: 旧硬币是否存在 + 签名是否匹配
//    → 天然适合简单转账，天然不适合合约
//
//  账户模型:
//    → 状态集中在"账户"中
//    → 交易是"修改账户状态"
//    → 验证需要: 余额检查 + 签名 + nonce
//    → 天然适合智能合约（合约 = 账户 + 代码）
//    → 但需要更高昂的状态存储成本
```

## 4. 共识机制：谁说了算？

### 4.1 工作量证明（Proof of Work）

PoW 的核心思想：通过计算难题（找 nonce 使得 hash 前 N 位为零）来证明你付出了计算资源。最长的有效链就是正确的链。

### 4.2 权益证明（Proof of Stake）

PoS 不再比算力，而是比"你押了多少币"。验证者质押代币，如果行为不端则被罚没（slashing）。以太坊 2022 年从 PoW 迁移到 PoS（The Merge），将能耗降低了 99.95%。

### 4.3 实现一个简单的 PoW 矿工

```rust
// pow_miner.rs —— 30行 Rust 实现工作量证明挖矿
use sha2::{Sha256, Digest};

/// 挖矿：找一个 nonce 使得 SHA256(block_data || nonce) 的前 difficulty 个比特为零
fn mine(block_data: &[u8], difficulty: usize) -> (u64, [u8; 32]) {
    // difficulty 表示前导零的比特数（不是字节数）
    // Bitcoin 的难度: ~70-80 比特（2024年），即前 ~9-10 字节为零
    let mut nonce: u64 = 0;
    loop {
        // 将 nonce 追加到区块数据后
        let mut data = block_data.to_vec();
        data.extend_from_slice(&nonce.to_le_bytes());
        let hash: [u8; 32] = Sha256::digest(&data).into();

        // 检查前导零比特数是否 >= difficulty
        if leading_zero_bits(&hash) >= difficulty {
            return (nonce, hash);
        }
        nonce = nonce.wrapping_add(1);

        // 每 100 万次打印进度（方便调试）
        if nonce % 1_000_000 == 0 {
            println!("  已尝试 {} nonce，当前哈希: {}", nonce, hex::encode(&hash[..8]));
        }
    }
}

/// 计算哈希值的前导零比特数
fn leading_zero_bits(hash: &[u8; 32]) -> usize {
    let mut count = 0usize;
    for &byte in hash.iter() {
        if byte == 0 {
            count += 8;
        } else {
            count += byte.leading_zeros() as usize;
            break;
        }
    }
    count
}

fn main() {
    let block_data = b"block:index=42|prev_hash=abc123|txs=[...]";

    println!("开始挖矿（难度: 20 前导零比特）...");
    let start = std::time::Instant::now();
    let (nonce, hash) = mine(block_data, 20);
    let elapsed = start.elapsed();

    println!("✓ 找到有效 nonce: {}", nonce);
    println!("✓ 区块哈希: {}", hex::encode(&hash));
    println!("✓ 前导零比特: {}", leading_zero_bits(&hash));
    println!("✓ 耗时: {:.2?} 秒", elapsed);
    // 难度 20 → 期望 2^20 ≈ 100万次尝试 → ~0.05 秒
    // 难度 30 → 期望 2^30 ≈ 10亿次尝试 → ~50 秒
    // Bitcoin 难度 70 → 期望 2^70 次尝试 → 全网 ~10分钟
}
```

## 6. 构建一个玩具区块链（完整实现）

这是本节的集大成——约 300 行 Rust 代码，构建一个包含 PoW 挖矿、UTXO 交易、链验证的玩具区块链。

```rust
// toy_blockchain.rs —— 用 Rust 构建一个可运行的区块链（完整实现）
//
// 运行方式: cargo run
// 功能: 创世块 → PoW 挖矿 → 交易上链 → 链验证

use sha2::{Sha256, Digest};
use std::time::{SystemTime, UNIX_EPOCH};

// ====================================================================
// 第1部分：数据结构定义
// ====================================================================

/// 区块：区块链的基本单元
#[derive(Debug, Clone)]
struct Block {
    index: u64,              // 区块高度（从0开始）
    timestamp: u64,          // Unix 时间戳（秒）
    prev_hash: [u8; 32],     // 前一个区块的哈希（形成链）
    transactions: Vec<Transaction>,  // 本区块包含的交易
    nonce: u64,              // PoW 挖矿找到的随机数
    hash: [u8; 32],          // 本区块的哈希（挖矿目标）
}

/// 交易：价值在地址间转移
#[derive(Debug, Clone)]
struct Transaction {
    from: String,            // 发送方（简化：直接用字符串表示地址）
    to: String,              // 接收方
    amount: u64,             // 金额
    fee: u64,                // 交易手续费（归矿工）
}

/// UTXO 条目
#[derive(Debug, Clone)]
struct UtxoEntry {
    tx_hash: [u8; 32],
    output_index: usize,
    amount: u64,
    owner: String,
}

/// 完整的区块链
struct Blockchain {
    chain: Vec<Block>,                      // 所有区块
    utxo_set: Vec<UtxoEntry>,              // 未花费交易输出集
    pending_transactions: Vec<Transaction>, // 待打包交易
    difficulty: usize,                      // 挖矿难度（前导零比特数）
    mining_reward: u64,                     // 区块奖励
}

// ====================================================================
// 第2部分：核心功能实现
// ====================================================================

impl Block {
    /// 计算区块的哈希值（用于 PoW 挖矿）
    fn compute_hash(
        index: u64,
        timestamp: u64,
        prev_hash: &[u8; 32],
        transactions: &[Transaction],
        nonce: u64,
    ) -> [u8; 32] {
        let mut hasher = Sha256::new();
        hasher.update(&index.to_le_bytes());
        hasher.update(&timestamp.to_le_bytes());
        hasher.update(prev_hash);
        for tx in transactions {
            hasher.update(tx.from.as_bytes());
            hasher.update(tx.to.as_bytes());
            hasher.update(&tx.amount.to_le_bytes());
            hasher.update(&tx.fee.to_le_bytes());
        }
        hasher.update(&nonce.to_le_bytes());
        hasher.finalize().into()
    }

    /// 获取当前 Unix 时间戳
    fn current_timestamp() -> u64 {
        SystemTime::now()
            .duration_since(UNIX_EPOCH)
            .unwrap()
            .as_secs()
    }
}

impl Blockchain {
    /// 创建新区块链（包含创世块）
    fn new(difficulty: usize, mining_reward: u64) -> Self {
        let genesis_tx = Transaction {
            from: "coinbase".to_string(),
            to: "genesis".to_string(),
            amount: 1000,
            fee: 0,
        };

        let timestamp = Block::current_timestamp();
        let prev_hash = [0u8; 32];  // 创世块没有前一个区块
        let (nonce, hash) = Self::mine_block(
            0,                          // index = 0
            timestamp,
            &prev_hash,
            &vec![genesis_tx.clone()],
            difficulty,
        );

        let genesis = Block {
            index: 0,
            timestamp,
            prev_hash,
            transactions: vec![genesis_tx.clone()],
            nonce,
            hash,
        };

        let genesis_utxo = UtxoEntry {
            tx_hash: genesis.hash,
            output_index: 0,
            amount: genesis_tx.amount,
            owner: genesis_tx.to.clone(),
        };

        Blockchain {
            chain: vec![genesis],
            utxo_set: vec![genesis_utxo],
            pending_transactions: Vec::new(),
            difficulty,
            mining_reward,
        }
    }

    /// PoW 挖矿：找一个 nonce 使得 hash 前 difficulty 位为零
    fn mine_block(
        index: u64,
        timestamp: u64,
        prev_hash: &[u8; 32],
        transactions: &[Transaction],
        difficulty: usize,
    ) -> (u64, [u8; 32]) {
        let mut nonce: u64 = 0;
        loop {
            let hash = Block::compute_hash(index, timestamp, prev_hash, transactions, nonce);
            if Self::leading_zero_bits(&hash) >= difficulty {
                return (nonce, hash);
            }
            nonce = nonce.wrapping_add(1);
        }
    }

    /// 计算前导零比特数
    fn leading_zero_bits(hash: &[u8; 32]) -> usize {
        let mut count = 0;
        for &byte in hash.iter() {
            if byte == 0 {
                count += 8;
            } else {
                count += byte.leading_zeros() as usize;
                break;
            }
        }
        count
    }

    /// 添加一笔待处理交易
    fn add_transaction(&mut self, tx: Transaction) -> Result<(), String> {
        // 验证交易：发送方有足够的 UTXO 吗？
        let sender_balance: u64 = self.utxo_set
            .iter()
            .filter(|u| u.owner == tx.from)
            .map(|u| u.amount)
            .sum();

        if tx.from != "coinbase" && sender_balance < tx.amount + tx.fee {
            return Err(format!(
                "余额不足: 需要 {} (金额 {} + 手续费 {})，但只有 {}",
                tx.amount + tx.fee, tx.amount, tx.fee, sender_balance
            ));
        }

        self.pending_transactions.push(tx);
        Ok(())
    }

    /// 挖一个新区块（打包 pending 交易）
    fn mine_pending_transactions(&mut self, miner_address: &str) -> Block {
        // 步骤1: 构建矿工奖励交易（coinbase 交易）
        let coinbase_tx = Transaction {
            from: "coinbase".to_string(),
            to: miner_address.to_string(),
            amount: self.mining_reward,
            fee: 0,
        };

        // 步骤2: 收集所有待打包交易 + coinbase 交易
        let mut block_txs = vec![coinbase_tx];
        block_txs.extend(self.pending_transactions.drain(..));

        // 步骤3: 挖矿
        let prev_block = self.chain.last().unwrap();
        let index = prev_block.index + 1;
        let timestamp = Block::current_timestamp();
        let (nonce, hash) = Self::mine_block(
            index,
            timestamp,
            &prev_block.hash,
            &block_txs,
            self.difficulty,
        );

        let block = Block {
            index,
            timestamp,
            prev_hash: prev_block.hash,
            transactions: block_txs.clone(),
            nonce,
            hash,
        };

        // 步骤4: 更新 UTXO 集（应用本区块的所有交易）
        self.apply_block(&block);

        // 步骤5: 将区块加入链
        self.chain.push(block.clone());
        block
    }

    /// 应用区块：更新 UTXO 集
    fn apply_block(&mut self, block: &Block) {
        for (i, tx) in block.transactions.iter().enumerate() {
            // 移除被消费的 UTXO（coinbase 除外）
            if tx.from != "coinbase" {
                let needed = tx.amount + tx.fee;
                let mut remaining = needed;
                // 收集发送方的 UTXO 直到凑够金额
                let mut to_remove = Vec::new();
                let mut idx = 0;
                for (j, utxo) in self.utxo_set.iter().enumerate() {
                    if utxo.owner == tx.from && remaining > 0 {
                        remaining = remaining.saturating_sub(utxo.amount);
                        to_remove.push(j);
                        idx += 1;
                        if remaining == 0 { break; }
                    }
                }
                // 从后往前删除（避免索引偏移）
                for &j in to_remove.iter().rev() {
                    self.utxo_set.remove(j);
                }
            }

            // 添加新产生的 UTXO（接收方）
            self.utxo_set.push(UtxoEntry {
                tx_hash: block.hash,
                output_index: i,
                amount: tx.amount,
                owner: tx.to.clone(),
            });
        }
    }

    /// 验证整条链的完整性
    fn validate_chain(&self) -> Result<(), String> {
        for i in 1..self.chain.len() {
            let current = &self.chain[i];
            let previous = &self.chain[i - 1];

            // 检查1: 当前区块的 prev_hash 是否等于前一区块的 hash
            if current.prev_hash != previous.hash {
                return Err(format!(
                    "区块 {} prev_hash 不匹配", current.index
                ));
            }

            // 检查2: 当前区块的哈希是否正确
            let recomputed_hash = Block::compute_hash(
                current.index,
                current.timestamp,
                &current.prev_hash,
                &current.transactions,
                current.nonce,
            );
            if recomputed_hash != current.hash {
                return Err(format!(
                    "区块 {} 哈希无效", current.index
                ));
            }

            // 检查3: 哈希是否满足难度要求
            if Self::leading_zero_bits(&current.hash) < self.difficulty {
                return Err(format!(
                    "区块 {} 不满足难度要求", current.index
                ));
            }

            // 检查4: 区块索引是否连续
            if current.index != previous.index + 1 {
                return Err(format!(
                    "区块 {} 索引不连续", current.index
                ));
            }
        }
        Ok(())
    }

    /// 查询某个地址的余额
    fn get_balance(&self, address: &str) -> u64 {
        self.utxo_set
            .iter()
            .filter(|u| u.owner == address)
            .map(|u| u.amount)
            .sum()
    }
}

fn main() {

    println!("╚═══════════════════════════════════════════════════════╝\n");

    // 初始化区块链（难度 16，区块奖励 50）
    let mut blockchain = Blockchain::new(16, 50);

    println!("[创世块] 区块 0 已创建");
    println!("  genesis 余额: {}\n", blockchain.get_balance("genesis"));

    // 添加几笔交易
    blockchain.add_transaction(Transaction {

        from: "genesis".to_string(),
        to: "bob".to_string(),
        amount: 200,

    }).unwrap();

    println!("[挖矿] 区块 1（矿工: miner1）");
    let block1 = blockchain.mine_pending_transactions("miner1");
    println!("  nonce: {}, hash: {}...", block1.nonce, hex::encode(&block1.hash[..8]));
    println!("  genesis 余额: {}", blockchain.get_balance("genesis"));
    println!("  alice 余额: {}", blockchain.get_balance("alice"));
    println!("  bob 余额: {}", blockchain.get_balance("bob"));
    println!("  miner1 余额: {}\n", blockchain.get_balance("miner1"));

    // Alice 转账给 Bob
    blockchain.add_transaction(Transaction {
        from: "alice".to_string(),
        to: "bob".to_string(),
        amount: 100,
        fee: 1,
    }).unwrap();

    blockchain.add_transaction(Transaction {
        from: "bob".to_string(),
        to: "charlie".to_string(),
        amount: 50,
        fee: 1,
    }).unwrap();

    println!("[挖矿] 区块 2（矿工: miner2）");
    let block2 = blockchain.mine_pending_transactions("miner2");
    println!("  nonce: {}, hash: {}...", block2.nonce, hex::encode(&block2.hash[..8]));
    println!("  alice 余额: {}", blockchain.get_balance("alice"));
    println!("  bob 余额: {}", blockchain.get_balance("bob"));
    println!("  charlie 余额: {}", blockchain.get_balance("charlie"));
    println!("  miner2 余额: {}\n", blockchain.get_balance("miner2"));

    // 验证整条链
    match blockchain.validate_chain() {
        Ok(()) => println!("✓ 整条链验证通过！区块 {} 个，所有哈希和链接正确",
                           blockchain.chain.len()),
        Err(e) => println!("✗ 链验证失败: {}", e),
    }

    // 展示整条链的结构
    println!("\n=== 完整区块链 ===");
    for block in &blockchain.chain {

                 block.index,
                 block.transactions.len(),
                 hex::encode(&block.hash[..8]),
                 block.nonce,
        );
    }
}
```

这个玩具区块链包含了一个真实区块链的所有核心组件：
1. **区块结构** — index, timestamp, prev_hash, transactions, nonce, hash
2. **工作量证明挖掘** — 找 nonce 使 hash 满足难度要求
3. **UTXO 管理** — 交易消耗 UTXO，产生新 UTXO，查询任意地址余额
4. **链验证** — 逐块确认哈希链接 + 工作量证明 + 索引连续性
5. **矿工奖励** — coinbase 交易在每个区块中创建新代币

**实验挑战（加深理解）：**
- 难度设为 20 再跑一次，观察挖矿时间的变化（约 2^4 = 16 倍耗时）
- 添加区块大小限制（每个区块最多 N 笔交易）
- 实现最长链规则：如果有两条分叉链，选累积工作量最大的那条
- 用 `HashMap` 替代 `Vec<UtxoEntry>` 提升余额查询效率（O(1) vs O(N)）
