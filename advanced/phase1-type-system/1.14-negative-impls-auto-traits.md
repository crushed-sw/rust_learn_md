# 1.14 Negative Impls & Auto Trait 控制——说"不"的权利

> Rust 的 `Send` 和 `Sync` 是自动的——编译器根据类型的字段自动推导。大多数时候这很方便：你不需要手动标注 `unsafe impl Send for MyType`。但有时"自动"是危险的——你包装了一个 `*mut c_void`，编译器知道这是指针所以认为它是 `!Send`；但你的 wrapper 在逻辑上是线程安全的。或者反过来——你有一个 `i32` 字段，编译器认为类型是 `Send`；但你通过 FFI 获取了这个指针，它必须绑定在原线程上。`impl !Send for MyType` 让你在需要的时候"踩下刹车"——它是 Rust 类型系统的安全阀。

## 1. Auto Trait 的自动推导机制

### 1.1 什么是 Auto Trait

```rust
// Auto Trait（自动 trait）是编译器自动为类型实现的 trait
// 目前 Rust 有三个 auto trait：
// - Send：可以安全地在线程间转移所有权
// - Sync：可以安全地在线程间共享引用
// - Unpin：不需要 Pin 保护（大部分类型都是 Unpin）
// （早期还有 Sized，但它规则不同——虽然也是"自动的"）

// auto trait 的核心特征：
// 你不需要写 impl Send for MyType {} —— 编译器自己推导
// 推导规则：如果所有字段都满足条件，类型就满足条件

struct MyType {
    x: i32,          // Send ✅
    y: String,       // Send ✅
    z: Vec<u8>,      // Send ✅
}
// → MyType 自动实现 Send ✅

struct MyOtherType {
    x: i32,                      // Send ✅
    y: std::rc::Rc<String>,      // Send ❌ — Rc<String> 不是 Send
}
// → MyOtherType 自动不实现 Send ❌
```

### 1.2 Auto Trait 的推导规则

```rust
// 结构化推导（structural derivation）：
// 一个类型的 auto trait 实现取决于其字段的 auto trait 实现

// 规则：
// 1. 结构体（struct）：
//    Send 如果所有字段都是 Send → 结构体是 Send
//    Sync 如果所有字段都是 Sync → 结构体是 Sync

// 2. 枚举（enum）：
//    所有变体的所有字段都满足 → 枚举满足

// 3. 联合体（union）—— 注意！
//    联合体**不会**自动推导 Send/Sync
//    因为编译器无法知道当前活跃的是哪个字段
//    你需要手动 unsafe impl

// 4. PhantomData：
//    PhantomData<T> 会"传染" T 的 auto trait 性质
//    如果 T: !Send，PhantomData<T> 也是 !Send
//    → 包含 PhantomData<T> 的结构体也会变成 !Send
```

## 2. Negative Impls：显式移除 Auto Trait

### 2.1 基本语法

```rust
// nightly 功能：
#![feature(negative_impls)]

struct MyWrapper {
    data: i32,  // i32 是 Send
}

// 显式声明：MyWrapper 不是 Send，即使字段都是 Send
impl !Send for MyWrapper {}
//  ^ 感叹号——否定实现

// 效果：
fn is_send<T: Send>() {}
// is_send::<MyWrapper>();  // ❌ 编译错误！
//                           // MyWrapper 不实现 Send

// 用法 1：阻止误用
let w = MyWrapper { data: 42 };
std::thread::spawn(move || {
    // ❌ 不能编译——w 不是 Send
    println!("{}", w.data);
});
```

### 2.2 基本用例：包装非线程安全的 FFI 类型

```rust
#![feature(negative_impls)]

// 场景：通过 FFI 获取的 C 指针，必须固定在线程上使用
mod ffi {
    pub struct NativeHandle {
        ptr: *mut std::ffi::c_void,  // 裸指针——编译器自动认为 !Send
    }
}

// 包装类型：
struct SafeWrapper {
    handle: ffi::NativeHandle,
    guard: std::sync::Mutex<()>,  // 添加了 Mutex 保护
}

// 问题：NativeHandle 是 !Send，但 Mutex<()> 是 Send
// SafeWrapper 是 Send 还是 !Send？
// 答：!Send——因为 NativeHandle 不满足 Send

// 如果你认为 SafeWrapper 实际上线程安全（因为有 Mutex 保护）：
unsafe impl Send for SafeWrapper {}
// unsafe impl —— 你在向编译器承诺"我知道怎么做才是安全的"
```

### 2.3 反面：包装类型应该是 !Send 但编译器认为它是 Send

```rust
#![feature(negative_impls)]

// 场景：你有一个结构体，所有字段都是 Send，但逻辑上不线程安全
struct ThreadLocalCache {
    // 所有字段都是 Send！
    id: u64,
    name: String,
    // 但是！这个缓存使用了 thread-local 的底层机制（通过 FFI）
    // 编译器不知道——编译器只看字段类型推导 auto trait
}

// ❌ 如果没有 negative impl，编译器会认为 ThreadLocalCache 是 Send
//    但把 ThreadLocalCache 传到其他线程会导致未定义行为
impl !Send for ThreadLocalCache {}
impl !Sync for ThreadLocalCache {}

// 现在：
std::thread::spawn(move || {
    let cache = ThreadLocalCache { id: 1, name: "t1".into() };
    // ❌ 编译错误！ThreadLocalCache 不是 Send
});
```

## 3. `PhantomData` 控制 Auto Trait

### 3.1 用 PhantomData 移除 Send/Sync（无需 nightly）

```rust
use std::marker::PhantomData;
use std::rc::Rc;

// 不需要 nightly negative_impls 的方法：
struct MyNonSendType<T> {
    data: T,
    _not_send: PhantomData<Rc<()>>,  // Rc<()> 不是 Send
    //          ^^^^^^^^^^^^^^^^^^^^
    //          编译器推导：PhantomData<Rc<()>> 是 !Send
    //          → MyNonSendType<T> 是 !Send（即使 T 是 Send）
}

// 使用：
fn require_send<T: Send>(_: T) {}

let x = MyNonSendType { data: 42, _not_send: PhantomData };
// require_send(x);  // ❌ 编译错误！MyNonSendType<i32> 不是 Send

// PhantomData 也支持 Sync：
struct MyNonSyncType<T> {
    data: T,
    _not_sync: PhantomData<std::cell::Cell<()>>,  // Cell<()> 不是 Sync
}
```

### 3.2 PhantomData 控制 Auto Trait 的完整矩阵

```rust
use std::marker::PhantomData;
use std::rc::Rc;
use std::cell::Cell;
use std::sync::atomic::AtomicBool;

// 需要 !Send —— 用 Rc（不是 Send）
struct NotSend<T> {
    data: T,
    _marker: PhantomData<Rc<()>>,
}

// 需要 !Sync —— 用 Cell 或 RefCell（不是 Sync）
struct NotSync<T> {
    data: T,
    _marker: PhantomData<Cell<()>>,
}

// 需要 !Send 且 !Sync —— 用 Rc<Cell<()>>（两者都不满足）
struct NeitherSendNorSync<T> {
    data: T,
    _marker: PhantomData<Rc<Cell<()>>>,
}

// 需要 Send 但 !Sync —— 用 AtomicBool + 手动 impl
struct SendButNotSync<T> {
    data: T,
    _marker: PhantomData<AtomicBool>,  // AtomicBool: Send + Sync
}
// 这不能直接用 PhantomData 表达"Send 但 !Sync"
// 需要 unsafe impl Send 或 negative impl !Sync
```

### 3.3 PhantomData vs Negative Impl——什么时候用哪个

```rust
// PhantomData 的优势：
// ✅ stable Rust 可用
// ✅ 语义清晰——字段直接出现在结构体定义中
// ✅ 无需 unsafe

// PhantomData 的劣势：
// ❌ 只能"否定"，不能"肯定"
//    → 如果 T: !Send，但你想让 MyType<T>: Send，只能用 unsafe impl
// ❌ 不能精细控制"为什么不是 Send/Sync"

// Negative Impl 的优势：
// ✅ 精确、显式——直接写在 impl 块中
// ✅ 可以附带文档注释解释为什么不是 Send/Sync
// ❌ 需要 nightly（feature gate）
// ❌ 不能"部分否定"——不能写 impl<T: !SomeTrait> !Send for MyWrapper<T>

// 决策树：
// 1. 如果你需要 !Send/!Sync 且不依赖泛型条件 → PhantomData（stable）
// 2. 如果你需要精确控制 + 文档说明 → Negative Impl（nightly）
// 3. 如果你需要条件性的否定（T 满足某个条件时否定）→ 用其他设计模式
```

## 4. `unsafe impl` —— 反向：手动添加 Auto Trait

### 4.1 基本语法

```rust
// unsafe impl Send/Sync：手动告诉编译器"我知道我的类型是线程安全的"

struct RawWrapper {
    ptr: *const u8,  // 裸指针——编译器认为：!Send, !Sync
}

// 如果你知道 ptr 指向的数据是只读的且生命周期足够长：
unsafe impl Send for RawWrapper {}
//   ^^^^^^ ——"我向编译器承诺这是安全的"

unsafe impl Sync for RawWrapper {}
//   ^^^^^^ ——"我向编译器承诺共享引用是安全的"
```

### 4.2 什么时候需要 `unsafe impl`

```rust
// 场景 1：包装裸指针，你保证它的线程安全性
struct ImmutableBuffer {
    data: *const u8,    // !Send（编译器默认）
    len: usize,
    // 你作为开发者知道：data 指向只读的静态数据
    // → 多个线程同时读取是安全的 → 应该是 Send + Sync
}

unsafe impl Send for ImmutableBuffer {}
unsafe impl Sync for ImmutableBuffer {}

// 场景 2：FFI 类型——你保证 C 库是线程安全的
mod ffi {
    // C 库提供的结构体：
    pub struct FfiConnection {
        handle: *mut std::ffi::c_void,
    }
    // 文档说：FfiConnection 是线程安全的（内部有锁）
    // 但 Rust 编译器不知道——它只看字段类型
}

// 你：
unsafe impl Send for ffi::FfiConnection {}
unsafe impl Sync for ffi::FfiConnection {}

// 场景 3：原子操作保护的内部可变性
use std::sync::atomic::{AtomicBool, Ordering};

struct MyCell<T> {
    value: std::cell::UnsafeCell<T>,  // UnsafeCell: !Sync
    lock: AtomicBool,                  // AtomicBool: Sync
    // 虽然 UnsafeCell 不是 Sync，但 lock 保证了只有一个线程能访问 value
}

// UnsafeCell 让 MyCell 自动 !Sync —— 但你知道这是安全的
unsafe impl<T: Send> Sync for MyCell<T> {}
// MyCell 是 Sync（如果 T 是 Send）——和 Mutex<T> 类似的设计
```

### 4.3 `unsafe impl` 的安全契约

```rust
// unsafe impl Send 的安全契约：
// "这个类型的值可以安全地在线程间转移所有权"
// 你需要确保：
// 1. 没有数据竞争
// 2. 没有 use-after-free（跨线程）
// 3. 没有逻辑上的线程绑定（如 thread-local 数据）

// unsafe impl Sync 的安全契约：
// "这个类型的共享引用可以安全地在线程间共享"
// 你需要确保：
// 1. 所有通过共享引用的访问都是线程安全的
// 2. 没有通过共享引用进行的非同步修改
// 3. 内部可变性有适当的同步机制保护

// ⚠️ 错误的 unsafe impl 会导致 UB！
// 编译器信任你的 unsafe impl——不安全的代码会破坏 Rust 的安全保证
```

## 5. 常见陷阱

### 陷阱 1：Negative Impl 的传染性

```rust
// Negative impl 只影响目标类型本身，不影响包含它的类型

struct Outer {
    inner: Inner,
}

impl !Send for Inner {}  // Inner 不是 Send

// Outer 是 Send 吗？
// → 规则：Outer 的 Send 取决于字段类型
// → Inner 不是 Send → Outer 不是 Send ✅ （这是预期的）
// 所以 negative impl 会"传染"到包含者

// 但：
impl !Send for Inner {}

unsafe impl Send for Outer {}  // 编译器不知道你这么做是否安全
// Outer 现在"假装"是 Send——但 inner 字段是 !Send
// 这个 unsafe impl 非常危险！你需要在运行时保证 inner 不被误用
```

### 陷阱 2：Negative Impl 无法 "有条件的否定"

```rust
// ❌ 不能写条件性的 negative impl：
// impl<T> !Send for MyType<T> where T: SomeTrait {}
// 这是一个尚未稳定的功能

// 目前只能写全局的 negative impl：
impl !Send for MyType<i32> {}  // 只对 i32 版本的 MyType 生效

// 或通过 PhantomData 间接控制：
struct MyType<T> {
    data: T,
    _marker: PhantomData<Rc<T>>,  // Rc<T> 不是 Send
    // 如果 T 是 Send，Rc<T> 仍然不是 Send → MyType<T> 是 !Send
    // 但这不能做到"如果 T: Something 则 !Send，否则 Send"
}
```

### 陷阱 3：多线程安全的"假阳性"——看似 Send 实则不安全

```rust
// 危险案例：所有字段都是 Send/Sync，但类型不是线程安全的

use std::cell::Cell;

// ❌ 危险——这段代码可以编译，但是 UB（未定义行为）：
struct ThreadUnsafe {
    counter: Cell<u64>,  // Cell 是 Send（不是 Sync）
}

// 为什么危险？
// Cell<u64> 是 Send——你可以在两个线程中各自拥有 Cell<u64>
// 这是安全的——每个线程有自己的 Cell

// 但你可能写了这样的代码：
let shared = &Cell::new(0u64);  // 单线程中使用
// 然后你用了 unsafe 把 shared 传到另一个线程...
// → Cell 没有原子性保证 → 数据竞争 → UB

// 教训：auto trait 的推导是"保守"的——它保证"如果推导为 Send，那么确实满足基本条件"
// 但 auto trait 不能保证你的"逻辑语义"正确——你需要自己确保这一点
```

## 7. 练习答案

### 练习 1 答案

```
(a) struct A { x: i32, y: String }
    i32: Send + Sync
    String: Send + Sync（String 拥有堆内存，但所有权模型保证了线程安全）
    → A 自动实现 Send ✅ 和 Sync ✅

(b) struct B { x: i32, y: *const u8 }
    i32: Send + Sync
    *const u8: !Send, !Sync（裸指针不保证任何线程安全）
    → B 自动 !Send ❌ 和 !Sync ❌

(c) struct C { x: i32, y: Rc<String> }
    i32: Send + Sync
    Rc<String>: !Send, !Sync（Rc 使用非原子的引用计数）
    → C 自动 !Send ❌ 和 !Sync ❌

(d) struct D { x: i32, y: Arc<String> }
    i32: Send + Sync
    Arc<String>: Send + Sync（如果 String: Send + Sync）
    → D 自动实现 Send ✅ 和 Sync ✅

(e) struct E { x: i32, _p: PhantomData<Rc<()>> }
    i32: Send + Sync
    PhantomData<Rc<()>>: !Send, !Sync（PhantomData 继承 Rc 的性质）
    → E 自动 !Send ❌ 和 !Sync ❌

(f) union F { x: i32, y: f64 }
    union 不自动推导 Send/Sync！
    → F 既不 Send 也不 Sync —— 需要手动 unsafe impl
    i32 和 f64 都是 Send/Sync，但编译器不推导 union
```

### 练习 2 答案

```rust
use std::marker::PhantomData;
use std::rc::Rc;
use std::cell::Cell;

// Rc<()> 不是 Send，Cell<()> 不是 Sync
// Rc<Cell<()>> 既不是 Send 也不是 Sync
struct ThreadBound<T> {
    inner: T,
    _marker: PhantomData<Rc<Cell<()>>>,
    // Rc<Cell<()>>: !Send + !Sync
    // PhantomData 继承这些性质
    // → ThreadBound<T>: !Send + !Sync（无论 T 是什么）
}

impl<T> ThreadBound<T> {
    fn new(inner: T) -> Self {
        ThreadBound { inner, _marker: PhantomData }
    }

    fn get(&self) -> &T { &self.inner }
    fn get_mut(&mut self) -> &mut T { &mut self.inner }
}

// 验证：
fn require_send<T: Send>(_: T) {}
fn require_sync<T: Sync>(_: T) {}

let tb = ThreadBound::new(42i32);
// require_send(tb);  // ❌ ThreadBound<i32> 不是 Send
// require_sync(&tb); // ❌ ThreadBound<i32> 不是 Sync
```

### 练习 3 答案

```rust
mod ffi {
    pub struct DatabaseHandle {
        ptr: *mut std::ffi::c_void,
    }

    // C 库文档：可以线程间移动（内部有锁）
    unsafe impl Send for DatabaseHandle {}
    // ^^^^^^ 手动承诺：移动到其他线程是安全的

    // C 库文档：不能在线程间共享
    // → 不需要 unsafe impl Sync —— 默认就是 !Sync
    //    （因为 *mut c_void 不是 Sync）
}
```

### 练习 4 答案

```rust
// 问题：Cache 的所有字段都是 Sync
// Vec<String>: Sync（如果 String: Sync）
// Instant: Sync
// → Cache 自动实现 Sync！
// → impl !Sync for Cache {} 被编译器拒绝：
//    "conflicting implementations of trait `Sync` for type `Cache`"
//    你不能"否定"一个已经被自动推导的 trait
//    （这等价于两个 impl 冲突——正向 impl（自动）和负向 impl 冲突）

// 修复方案 1：使用 PhantomData 阻止自动推导
struct Cache {
    data: Vec<String>,
    last_access: std::time::Instant,
    _not_sync: PhantomData<std::cell::Cell<()>>,  // Cell<()>: !Sync
}

// 修复方案 2：使用内部可变性（如果确实需要）
use std::cell::RefCell;

struct Cache {
    data: RefCell<Vec<String>>,    // RefCell: !Sync
    last_access: RefCell<std::time::Instant>,
}
// RefCell 不是 Sync → Cache 自动不是 Sync

// 关键理解：
// negative impl 不能"覆盖"已经自动满足的 trait！
// 你必须从源头上阻止它——让某个字段不满足条件
```

## 8. 深度扩展

### 8.1 C++ 的线程安全——全靠自觉 vs Rust 的类型级保证

```cpp
// C++ 的线程安全规则：基于"约定"和"文档"
class ConnectionCache {
    std::vector<std::string> data;
    std::chrono::steady_clock::time_point last_access;
public:
    // 文档："这个类不是线程安全的——不要跨线程共享"
    void update(const std::string& key) {
        // 没有任何编译期保护——调用方需要"记得"文档的警告
        data.push_back(key);
        last_access = std::chrono::steady_clock::now();
    }
};

// 调用方：
auto cache = std::make_shared<ConnectionCache>();
std::thread t1([cache]() {
    cache->update("thread1");  // 数据竞争！但编译器不会阻止
});
std::thread t2([cache]() {
    cache->update("thread2");  // 数据竞争！但编译器不会阻止
});

// C++ 的问题层级：
// 1. 没有类型级别的 Send/Sync 等价物
// 2. 文档/注释是唯一的"线程安全规范"
// 3. 静态分析工具（如 clang-tidy）只能检查有限的模式
// 4. ThreadSanitizer（TSan）是运行时检测——发现不了所有问题
// 5. 团队中新人不知道"这个类型不能跨线程"——直到在生产环境遇到数据竞争

// Rust 的编译期保证：
// - Send/Sync 在类型层面编码了线程安全性
// - 违反约束 → 编译错误，不是运行时 bug
// - 这是 Rust 的"无畏并发"的基石——编译器帮你检查
```

### 8.2 如果没有 Auto Trait 和 Negative Impl——所有 Send/Sync 都要手动标注

```rust
// 假设 Rust 没有 auto trait（自动推导）——你需要手动标注每个类型：

// 基础类型需要手动标注：
// unsafe impl Send for i32 {}
// unsafe impl Sync for i32 {}
// unsafe impl Send for String {}
// unsafe impl Sync for String {}
// ...数百个标准库类型的 impl

// 每个自定义类型也需要：
struct MyType {
    x: i32,
    y: String,
}
unsafe impl Send for MyType {}  // 手动标注——容易遗漏
unsafe impl Sync for MyType {}

// 问题：
// 1. 样板代码爆炸——每个类型都要写
// 2. 遗漏风险——忘记标注导致"看起来没问题"但实际上缺少保证
// 3. 重构风险——给 struct 加一个 !Send 字段后，忘了移除 Send impl
//     → 编译不过（unsafe impl 和字段的 !Send 冲突）——这倒是好的
// 4. 生态碎片——不同库的 Send/Sync 标注质量不一致

// Auto Trait 的设计哲学：
// "宁可保守地多给（Send/Sync），也不要让用户遗漏。"
// 编译器根据字段自动推导——正确率很高
// 错误的情况（应该 !Send 但推导为 Send）用 negative impl 或 PhantomData 纠正
```

### 8.3 OIBIT 的历史——Opt-In Builtin Trait

```
Send 和 Sync 并非一直是 auto trait。Rust 早期（1.0 之前）有一个
叫 OIBIT（Opt-In Builtin Trait）的机制：

- 类型必须显式"选择加入" Send/Sync（opt-in）
- 例如：unsafe impl Send for MyType {}
- 不标注 = 默认不是 Send/Sync

这个设计的问题：
1. 太保守——大部分类型确实应该是 Send/Sync
2. 样板代码巨大——每个类型都要标注
3. 库作者经常忘记标注——导致可用性差

所以 Rust 在 1.0 前切换到了"Auto Trait + Opt-Out"模型：
- 默认自动推导（auto）
- 如果需要阻止——用 PhantomData 或 negative impl 退出（opt-out）

这是"安全默认 + 显式退出"哲学的又一个体现：
默认给最安全的选项（Send/Sync），少数例外显式声明。
```

### 8.4 `PhantomData` 的深层设计——为什么不用一个 `#[not(Send)]` 属性

```rust
// 你可能想：为什么不用一个简单的属性？
// #[not_send]
// struct MyType { ... }

// 原因 1：泛型行为
struct Wrapper<T> {
    data: T,
}
// 如果 Wrapper<T> 应该 !Send 当 T: !Send —— 属性无法表达这个
// PhantomData<Rc<T>> 的行为了：Rc<T> 的 Send 性质随 T 变化

// 原因 2：正交性
// PhantomData 是一个通用的"类型占有标记"——它不只为 auto trait 设计
// 它还能标记生命周期关系、所有权关系
// 用一个统一的机制解决多个问题 = Rust 的"最小原语"哲学

// 原因 3：编译期零成本
// PhantomData 不占任何运行时空间
// 属性系统需要更复杂的编译器逻辑
// 而 PhantomData 只是"利用现有的类型推导规则"

// 原因 4：可见性
// PhantomData 在结构体定义中可见——你一眼就能看到
// 属性可能被忽视或被工具隐藏
```

### 8.5 Auto Trait 和 Trait Object 的交互

```rust
// auto trait 的性质影响 trait object 的创建：

trait MyTrait {}

struct MyStruct;
impl MyTrait for MyStruct {}

// 创建 trait object 时：
fn make_dyn() -> Box<dyn MyTrait + Send> {
    Box::new(MyStruct)
    // ✅ MyStruct 自动推导为 Send
}

fn make_dyn_non_send() -> Box<dyn MyTrait> {
    // 不带 Send 的 dyn Trait = dyn Trait + 'static
    // 任何类型都可以放进去——Send 不是必需的
    Box::new(MyStruct)
}

// 但对于 Send trait object：
struct NonSend {
    _rc: std::rc::Rc<()>,  // Rc 让 NonSend 是 !Send
}
impl MyTrait for NonSend {}

// fn make_dyn_send() -> Box<dyn MyTrait + Send> {
//     Box::new(NonSend)  // ❌ NonSend 不是 Send
// }

// auto trait 让你不需要在每个 impl 中考虑 Send/Sync
// 编译器自动帮你检查——类型安全扩展到 trait object
```

### 8.6 `Unpin` —— 第三个 Auto Trait

```rust
// Unpin 是第三个 auto trait，规则与 Send/Sync 类似：
// - 绝大多数类型自动实现 Unpin
// - !Unpin 的类型包括：包含自引用或内部指针的类型
// - 最常见的 !Unpin：通过 async 生成的 Future、 Generator

// Unpin 和 Send/Sync 的区别：
// Unpin 不是安全性相关的——它是"这个类型是否需要 Pin 保护"
// 手动实现 Unpin 不需要 unsafe —— 因为 Unpin 只是优化/便利

// 你可以用 negative impl 移除 Unpin：
#![feature(negative_impls)]

struct MustBePinned {
    _private: (),
}

impl !Unpin for MustBePinned {}
// 现在 MustBePinned 必须被 Pin 住才能安全使用

// 但实际上很少需要 negative impl Unpin
// 因为你需要 Pin 的类型通常自动就是 !Unpin
// （比如通过内部指针或 Generator）
```

### 8.7 实际应用案例：`MutexGuard<T>` 的 Auto Trait 设计

```rust
// MutexGuard<T> 的设计展示了 auto trait 的精妙之处：

// MutexGuard 是：
// - Send 如果 T: Send（因为 MutexGuard 含有一个对 Mutex 的引用）
//   ——但 MutexGuard 本身也是 Send（当 T 满足条件时）
//   等等，不对！MutexGuard 实际上是 !Send！
//
// 真实规则：
// MutexGuard<T>: Send   ✅（当 T: Send——标准库确保跨线程的锁释放安全性）
// MutexGuard<T>: Sync   ✅（当 T: Sync——锁守卫可以在线程间共享引用）
//
// 等等，似乎我记错了。让我澄清：
// std::sync::MutexGuard<'_, T>:
// - Send: 不实现（在大多数平台上）—— 防止跨线程释放锁
// - Sync: 实现（如果 T: Sync）—— 多个线程可以查看锁的内容
//
// 这体现了"用 auto trait 编码安全约束"的精妙：
// 锁必须在获取它的线程中释放 → MutexGuard 不是 Send
// 可以安全地从多个线程查看锁的内容 → MutexGuard 是 Sync（当 T: Sync）
```

### 8.8 Negative Impl 的未来——条件性否定和更精细的控制

```
目前的 negative impl 很"粗糙"——只能写全局否定：

impl !Send for MyType {}  // 无条件的

未来可能的功能（不稳定/讨论中）：

1. 条件性 negative impl：
   impl<T> !Send for Wrapper<T> where T: SomeCondition {}
   // 只有在 T 满足条件时才否定

2. 特化的 negative impl：
   impl !Send for Vec<SomeSpecificType> {}
   // 对特定实例否定

3. Auto Trait 的自定义规则：
   // 声明"我的类型从不实现 Send"
   // 阻止编译器自动推导

4. Trait 约束的反向传播：
   impl<T: Send> !Sync for MyType<T> {}
   // 即使字段都是 Sync，也否定 Sync

这些功能在复杂 FFI 绑定和异步运行时中特别有用——
但目前还没有明确的稳定化路线图。
```
