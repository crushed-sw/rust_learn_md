# 1.2 Trait 对象安全 (Object Safety)

> 上一课我们理解了 `dyn Trait` 靠 vtable 实现动态分发。这一课回答：
> **为什么有些 trait 不能创建 `dyn Trait`？编译器凭什么拒绝你？**

---

## 1. 从一个"为什么不行"开始

```rust
// 你肯定遇到过的编译错误：
let x: Box<dyn Clone> = Box::new(42i32);

// error[E0038]: the trait `Clone` cannot be made into an object
//   = note: the trait cannot be made into an object because
//           method `clone` references the `Self` type in its return type
```

`Clone` 这么基础的一个 trait，为什么不能创建 `dyn Clone`？反观 `Display` 就可以：

```rust
let x: Box<dyn Display> = Box::new(42i32);  // ✅ 完全没问题
let y: Box<dyn Debug> = Box::new(42i32);    // ✅ 也没问题
```

直觉上：`Clone::clone` 返回 `Self`，而 `Display::fmt` 只接收 `&self`。但深层次的原因是什么？

**核心答案**：编译器需要为 `dyn Trait` 构建 vtable。vtable 中的每个方法必须有确定的、可以塞进函数指针的签名。如果方法签名依赖 `Self`（一个在运行时才知道具体是什么的类型），编译器就"写不出"这个函数指针的签名。

---

## 2. 什么是对象安全？—— 一句话定义

> 一个 trait 是**对象安全 (Object-Safe)** 的，当且仅当编译器能为它构建一个完整的 vtable，使得 `dyn Trait` 可以通过 vtable 完成所有方法的动态分发。

反过来，如果一个 trait 中**任何一个方法**让编译器无法确定 vtable 条目，整个 trait 就不能用于 trait 对象。

---

## 3. 四条核心规则

### 规则 1：`Self: Sized` 不能是 trait 自身的 supertrait

```rust
// ❌ 不是对象安全的
trait Foo: Sized {
    fn method(&self);
}
// let x: &dyn Foo = ...; // 错误！

// 为什么？因为 `trait Foo: Sized` 相当于要求所有实现者都是 Sized
// 但 dyn Foo 本身不是 Sized——矛盾了
```

```rust
// ✅ 对象安全
trait Foo {
    fn method(&self);
}
// Foo 本身不要求 Sized，dyn Foo 可以存在
```

**但注意**：方法上带 `where Self: Sized` 不仅合法，而且是重要设计模式（见规则 2）。

### 规则 2：返回 `Self` 的方法必须带 `where Self: Sized`

这是最常见的不安全原因。回到 `Clone` 的例子：

```rust
// ❌ 不是对象安全的
trait Clone {
    fn clone(&self) -> Self;  // 返回 Self——不合法！
}

// 为什么？vtable 需要为 clone 存一个函数指针。
// 但 clone 的返回类型是 Self——在 dyn Clone 上下文中，Self 是未知的。
// vtable 条目需要具体的签名，而 fn clone(...) -> ??? 无法确定。

// 从调用方看：
// let x: Box<dyn Clone> = ...;
// let y = x.clone();  // y 的类型是什么？编译器不知道！
//                     // 返回的大小是 compile-time unknown
```

**对比**：

```rust
// ✅ 返回 &self 可以，因为 Self 不出现在返回类型
trait Display {
    fn fmt(&self, f: &mut Formatter) -> fmt::Result;  // ✅ 对象安全
}

// ✅ 返回 () 可以
trait Drop {
    fn drop(&mut self);  // ✅ 对象安全（特殊：编译器自动调用）
}
```

**修复手段**：如果你真的需要在 trait 中定义一个返回 `Self` 的方法，但又想让 trait 保持对象安全，用 `where Self: Sized`：

```rust
trait Animal {
    fn speak(&self) -> &str;  // ✅ 对象安全

    // ✅ 这个方法有 where Self: Sized，从 vtable 中排除
    fn clone_animal(&self) -> Self
    where
        Self: Sized;
}

// 编译器的处理：
// - Animal 是对象安全的
// - vtable 中只包含 speak，不包含 clone_animal
// - 通过 &dyn Animal 只能调用 speak
// - clone_animal 只能通过具体类型调用
```

### 规则 3：方法不能有泛型参数

```rust
// ❌ 不是对象安全的
trait Processor {
    fn process<T: Display>(&self, input: T);
    //       ^^^^^^^^^ 泛型参数——不合法！
}

// 为什么？泛型方法在编译时会被单态化——为每个 T 生成一份代码。
// 但 vtable 只有一个，需要一份固定的函数指针。泛型方法的"多个版本"
// 无法压缩进一个 vtable 条目。
```

```rust
// ✅ 把泛型移到 trait 级别
trait Processor<T: Display> {
    fn process(&self, input: T);  // ✅ 对象安全（如果 T 是具体类型）
}

// 但这样每个具体 T 都需要不同的 dyn Processor<T>
let p: Box<dyn Processor<String>> = ...;   // ✅
let p: Box<dyn Processor<i32>> = ...;      // ✅
// let p: Box<dyn Processor> = ...;        // ❌ T 没有指定
```

```rust
// ✅ 更好的方案：用 dyn Trait 代替泛型
trait Processor {
    fn process(&self, input: &dyn Display);  // ✅ 对象安全
    // 虽然损失了单态化的性能优化，但保证了对象安全
}
```

### 规则 4：关联常量必须有值

```rust
// ❌ 不是对象安全的（如果常量没有默认值）
trait Config {
    const MAX_SIZE: usize;  // 没有默认值——vtable 不知道该用什么值
}

// ✅ 有默认值的常量是安全的
trait Config {
    const MAX_SIZE: usize = 256;  // ✅ 对象安全
}
// 所有通过 dyn Config 访问的 MAX_SIZE 都返回 256
```

## 4. `Self: Sized` 技巧 —— 深度剖析

这是 Rust 设计中最优雅的模式之一：**让 trait 保持对象安全，同时允许非对象安全的方法存在**。

### 4.1 本质

```rust
trait MyTrait {
    // 对象安全的方法——会进入 vtable
    fn object_safe_method(&self);

    // 非对象安全的方法——被 where Self: Sized 排除出 vtable
    fn non_object_safe_method(self) -> Self
    where
        Self: Sized;
}
```

编译器看到 `where Self: Sized` 时，就"跳过"这个方法，不把它放进 vtable。这意味着：

```
通过 &dyn MyTrait 调用：
  ✅ object_safe_method()     —— 在 vtable 中，可以动态分发
  ❌ non_object_safe_method() —— 不在 vtable 中，编译错误

通过具体类型调用：
  ✅ object_safe_method()     —— 静态分发
  ✅ non_object_safe_method() —— 静态分发
```

### 4.2 标准库中的实战案例

**`Iterator` trait** —— 最经典的例子：

```rust
// 简化的 Iterator trait（标准库实际定义）
pub trait Iterator {
    type Item;

    // ✅ 对象安全：返回 Option<Item>，不涉及 Self
    fn next(&mut self) -> Option<Self::Item>;

    // ❌ 以下方法都不可用于 dyn Iterator，但 trait 本身是对象安全的

    // collect 返回具体的集合类型（依赖 Self 的大小）
    fn collect<B: FromIterator<Self::Item>>(self) -> B
    where
        Self: Sized,
    { ... }

    // 消耗迭代器的方法
    fn count(self) -> usize
    where
        Self: Sized,
    { ... }

    fn last(self) -> Option<Self::Item>
    where
        Self: Sized,
    { ... }

    fn fold<B, F>(self, init: B, f: F) -> B
    where
        Self: Sized,
        F: FnMut(B, Self::Item) -> B,
    { ... }

    // ... 几十个方法如 map, filter, take 等
}
```

**这意味着什么？**

```rust
fn process_dyn(iter: &mut dyn Iterator<Item = i32>) {
    // ✅ 可以：next 在 vtable 中
    let val = iter.next();

    // ❌ 不可以：collect 带 where Self: Sized
    // let vals: Vec<i32> = iter.collect();
}

fn process_static<I: Iterator<Item = i32>>(iter: I) {
    // ✅ 都可以
    let val = iter.next();     // 静态分发
    let vals = iter.collect(); // 静态分发
}
```

**`DoubleEndedIterator`** —— 更有趣的例子：

```rust
pub trait DoubleEndedIterator: Iterator {
    fn next_back(&mut self) -> Option<Self::Item>;

    // 所有消耗型方法都带 where Self: Sized
    fn nth_back(&mut self, n: usize) -> Option<Self::Item>
    where
        Self: Sized,
    { ... }

    fn rfold<B, F>(self, init: B, f: F) -> B
    where
        Self: Sized,
        F: FnMut(B, Self::Item) -> B,
    { ... }
}

// DoubleEndedIterator 是对象安全的！
let iter: &mut dyn DoubleEndedIterator<Item = i32> = ...;
iter.next_back();  // ✅
```

### 4.3 什么时候该加 `where Self: Sized`？

```
方法签名中出现以下情况 → 加 where Self: Sized：
1. 参数是 self（按值）而不是 &self
2. 返回类型包含 Self
3. 方法有泛型参数

加不加的判断标准：
→ 这个方法在 dyn Trait 上下文中能不能被"有意义地"调用？
→ 如果不能，加 where Self: Sized，保持 trait 对象安全
```

## 5. 可调度接收器 (Dispatchable Receivers)

并不是所有 `self` 形式都能放进 vtable。以下是**合法的 receiver 类型**：

| Receiver | 是否可调度 | 说明 |
|:---|:---:|:---|
| `&self` | ✅ | 最常见，不可变借用 |
| `&mut self` | ✅ | 可变借用 |
| `self: Box<Self>` | ✅ | 消耗 Box 中的 self |
| `self: Rc<Self>` | ✅ | 消耗 Rc 中的 self |
| `self: Arc<Self>` | ✅ | 消耗 Arc 中的 self |
| `self: Pin<Box<Self>>` | ✅ | Pin 包装的 Box |
| `self`（按值） | ❌ | `Self` 大小未知（在 dyn 上下文中） |

```rust
trait Example {
    fn by_ref(&self);           // ✅
    fn by_mut(&mut self);       // ✅
    fn by_box(self: Box<Self>); // ✅
    fn by_rc(self: Rc<Self>);   // ✅
    fn by_value(self);          // ❌ 不是对象安全（除非加 where Self: Sized）
}
```

**为什么按值 `self` 不合法？** 因为 `dyn Trait` 的大小在编译时未知。编译器需要知道方法的参数布局才能生成调用代码——`self` 在栈上占多少字节？不知道 → 无法编译。

## 6. 常见 Trait 的对象安全一览

### 6.1 对象安全的

| Trait | 原因 |
|:---|:---|
| `Display` | 只有 `&self`，返回 `fmt::Result` |
| `Debug` | 同上 |
| `Write` | 方法用 `&mut self`，不返回 Self |
| `Read` | 同上 |
| `Fn()` / `FnMut()` / `FnOnce()` | 通过编译器魔法实现（见 6.3） |
| `Drop` | 只有 `&mut self` |
| `Any` | 设计为 trait 对象使用 |

### 6.2 不是对象安全的

| Trait | 不安全的根本原因 | 修复 |
|:---|:---|:---|
| `Clone` | `clone` 返回 `Self` | 无法修复——设计上就不支持 dyn Clone |
| `Copy` | 是 `Clone` 的子 trait + marker | 无法修复 |
| `PartialEq` | `eq` 参数是 `&Self`（不是 `&self`） | 用 `dyn PartialEq<ConcreteType>` 或 enum |
| `Eq` | 是 `PartialEq` 的子 trait | 同上 |
| `Ord` | 方法参数含 `Self` | 同上 |
| `Hash` | 方法参数含 `Self` | 同上 |
| `Default` | 返回 `Self` | 无法修复 |
| `From<T>` | 返回 `Self` | 无法修复 |
| `Into<T>` | 返回 `T`（不涉及 Self）→ 实际上是安全的！ | — |

### 6.3 部分安全（带 `where Self: Sized` 的模式）

| Trait | 安全方法 | 不安全方法 |
|:---|:---|:---|
| `Iterator` | `next()` | `collect()`, `count()`, `fold()`, `last()`, `map()`, `filter()`... |
| `DoubleEndedIterator` | `next_back()` | `nth_back()`, `rfold()`... |
| `BufRead` | `fill_buf()`, `consume()` | `lines()`, `split()`... |

## 7. Supertrait 的传递性

如果 trait A 要求 `trait A: B`，而 B 不是对象安全的，那么 A 也不是对象安全的。

```rust
trait NotObjectSafe {
    fn clone_self(&self) -> Self;  // ❌ 返回 Self
}

trait AlsoNotSafe: NotObjectSafe {
    fn speak(&self);  // 这个方法自己是安全的...
}

// ❌ let x: &dyn AlsoNotSafe = ...;  // 不行！因为 NotObjectSafe 不是对象安全的
// 为什么？dyn AlsoNotSafe 的 vtable 需要同时包含 NotObjectSafe 的方法
// 而 NotObjectSafe 的方法无法放入 vtable
```

**反例**——安全的 supertrait 链：

```rust
trait Animal {
    fn speak(&self) -> &str;  // ✅ 对象安全
}

trait Pet: Animal {
    fn owner(&self) -> &str;  // ✅ 对象安全
}

// ✅ 两者都对象安全，dyn Pet 可用
let pet: &dyn Pet = ...;
pet.speak();  // 通过 vtable 调用 Animal::speak
pet.owner();  // 通过 vtable 调用 Pet::owner
```

## 8. 对象安全 vs 关联合类型

关联合类型（associated types）**不影响**对象安全，但会影响你如何使用 trait 对象：

```rust
trait Container {
    type Item;  // 关联合类型——不影响对象安全
    fn get(&self) -> Option<&Self::Item>;
}

// ✅ Container 是对象安全的
// 但在创建 trait 对象时，必须指定关联类型：
let c: Box<dyn Container<Item = i32>> = Box::new(vec![1, 2, 3]);
//                       ^^^^^^^^^^ 必须指定

// 如果关联类型有默认值，可以省略：
trait Container {
    type Item = u8;  // 有默认值
    fn get(&self) -> Option<&Self::Item>;
}
// let c: Box<dyn Container> = ...;  // Item 默认为 u8
```

## 9. 实战：当 trait 不是对象安全时怎么办

### 策略 1：用 enum 代替（enum dispatch）

```rust
// 问题：你要在同一个集合中存不同类型
trait Shape {
    fn area(&self) -> f64;
    fn clone_shape(&self) -> Box<dyn Shape>;  // ❌ 不是对象安全
}

// ❌ 不可行：Box<dyn Shape> 不行，因为 clone_shape 不是对象安全

// ✅ 方案：用 enum 枚举所有变体
enum AnyShape {
    Circle { radius: f64 },
    Rectangle { width: f64, height: f64 },
    Triangle { base: f64, height: f64 },
}

impl AnyShape {
    fn area(&self) -> f64 {
        match self {
            AnyShape::Circle { radius } => std::f64::consts::PI * radius * radius,
            AnyShape::Rectangle { width, height } => width * height,
            AnyShape::Triangle { base, height } => 0.5 * base * height,
        }
    }

    fn clone_shape(&self) -> Self {
        // enum 可以自动 derive Clone
    }
}

let shapes: Vec<AnyShape> = vec![
    AnyShape::Circle { radius: 5.0 },
    AnyShape::Rectangle { width: 3.0, height: 4.0 },
];
// ✅ 异质集合，且所有操作都是静态分发，甚至更快
```

### 策略 2：拆分 trait

```rust
// 问题：一个 trait 混合了对象安全和不安全的方法
trait DataProcessor {
    fn process(&self, data: &[u8]) -> Vec<u8>;    // ✅ 对象安全
    fn new() -> Self;                               // ❌ 返回 Self
}

// ✅ 方案：拆成两个 trait
trait Process {
    fn process(&self, data: &[u8]) -> Vec<u8>;
}

trait Construct: Process {
    fn new() -> Self
    where
        Self: Sized;
}

// Process 是对象安全的，可以用于 dyn
let processors: Vec<Box<dyn Process>> = vec![...];
```

### 策略 3：用函数指针/闭包替代

```rust
// 问题：需要用 trait 对象但 trait 不是对象安全的
// trait 只有一个方法且不返回 Self——但有其他问题

// ✅ 直接使用函数指针
type Validator = Box<dyn Fn(&str) -> bool>;

let validators: Vec<Validator> = vec![
    Box::new(|s: &str| s.len() > 3),
    Box::new(|s: &str| s.chars().all(|c| c.is_alphanumeric())),
];

for v in &validators {
    println!("{}", v("hello"));
}
```

### 策略 4：手动 vtable（高级）

当你需要极致控制，可以手写 vtable：

```rust
// 手动定义 vtable 结构
struct MyVTable {
    drop: unsafe fn(*mut ()),
    speak: unsafe fn(*const ()) -> String,
}

struct MyDynTrait {
    data: *mut (),
    vtable: &'static MyVTable,
}

// 为具体类型构造 vtable
impl MyDynTrait {
    fn new<T: Speak>(value: T) -> Self {
        // 常量 vtable
        static VTABLE: MyVTable = MyVTable {
            drop: |ptr| unsafe { drop(Box::from_raw(ptr as *mut T)) },
            speak: |ptr| unsafe { (*(ptr as *const T)).speak() },
        };

        MyDynTrait {
            data: Box::into_raw(Box::new(value)) as *mut (),
            vtable: &VTABLE,
        }
    }

    fn speak(&self) -> String {
        unsafe { (self.vtable.speak)(self.data) }
    }
}

// 注意：这种模式只适合极端场景，绝大多数情况下直接用 trait 对象
```

## 10. 设计哲学：Rust 为什么这样设计？

### 10.1 "保证 vtable 的确定性"

```
核心原则：vtable 中的每个条目必须有确定的函数签名。

编译器要求对象安全，是因为它在编译 trait 定义时就要生成 vtable 的类型布局。
如果 vtable 条目依赖运行时才知道的信息（Self 的大小、泛型参数的具体类型），
编译器无法生成。

对比 C++：
  C++ 的虚函数表中，返回协变类型是允许的（子类可以返回比父类更具体的类型）。
  Rust 禁止这一点——强制所有实现者返回完全相同的签名。
  
代价：灵活性降低（无法用 dyn Clone）
收益：vtable 布局在所有实现者之间完全一致，不需要复杂的运行时类型检查
```

### 10.2 "显式选择 vs 魔法行为"

```
决策：非对象安全的方法不自动排除，而是由开发者显式标注 where Self: Sized

如果编译器"聪明地"自动排除非对象安全的方法：

好处：
  ✅ 写起来更省事

坏处：
  ❌ 你可能不小心从 dyn 调用路径中失去了一个重要方法，且没有编译警告
  ❌ "这个方法在 dyn 上下文中不可用" 变成隐式行为，调试噩梦
  ❌ API 的契约变得模糊——看 trait 定义看不出哪些方法能通过 dyn 调用

Rust 的选择：显式 > 隐式。
where Self: Sized 让你一眼看出 "这个方法不参与动态分发"。
```

### 10.3 "不为你不用的特性付费"的另一个面目

```
为什么存在"不是对象安全的 trait"？

如果所有 trait 都必须对象安全：
  → Clone 必须移除 clone(&self) -> Self 方法
  → Iterator 的所有适配器方法不能存在
  → 大量零成本抽象的泛型模式无法表达

Rust 的选择：
  trait 首先服务于静态分发的泛型编程。
  对象安全是一个"可选特性"——不是所有 trait 都需要它。
  那些需要既有静态又有动态分发的 trait（如 Iterator），
  通过 where Self: Sized 实现了两者兼顾。
```

## 11. 编译器错误诊断

### 错误 1：`the trait X cannot be made into an object`

```rust
// 错误示例
trait Bad {
    fn clone_me(&self) -> Self;
}
let x: Box<dyn Bad> = ...;  // ❌

// 编译器输出：
// error[E0038]: the trait `Bad` cannot be made into an object
//   = note: for a trait to be "object safe" it needs to allow building a vtable
//           so that the trait object can access all methods with dynamic dispatch
//   = note: method `clone_me` references the `Self` type in this trait
```

**诊断步骤**：

```
1. 读 note：哪个方法出了问题？
2. 看那个方法的签名：
   - 返回了 Self？→ 加 where Self: Sized
   - 有泛型参数？→ 改为 dyn Trait 或移到 trait 级别
   - 用了 self（按值）？→ 改为 &self 或加 where Self: Sized
3. 检查 trait 本身是否要求 Self: Sized
```

### 错误 2：在 dyn 对象上调用了非对象安全的方法

```rust
trait MyIter {
    type Item;
    fn next(&mut self) -> Option<Self::Item>;
    fn collect(self) -> Vec<Self::Item>
    where
        Self: Sized;
}

fn use_iter(iter: &mut dyn MyIter<Item = i32>) {
    let val = iter.next();     // ✅
    // let vals = iter.collect(); // ❌
}

// 编译器输出：
// error[E0038]: the trait `MyIter` cannot be made into an object
//   (same error, because collect's presence doesn't remove it from diagnostics)
// 或者：
// error[E0599]: no method named `collect` found for mutable reference
//   `&mut dyn MyIter<Item = i32>` in the current scope
```

## 12. 举一反三

### 模式 1：对象安全的 trait 不一定设计给 dyn 用

```rust
// Iterator 是对象安全的，但实际项目中很少用 dyn Iterator
// 原因：next() 返回 Option<Item>，Item 通常是 Copy 的小类型，
// 但每次动态调用都有 vtable 开销，抵不过静态分发 + 内联的收益

// 经验：对象安全 ≠ 应该用 dyn
// 动态分发适合"调用频率低、实现类型多、方法体大"的场景
```

### 模式 2：不是对象安全 ≠ trait 有 bug

```rust
// Clone 不是对象安全——这是故意的设计，不是缺陷
// 你遇到的"不能 Box<dyn Clone>" 是在提醒你：
// "clone 的返回类型必须编译时确定——用具体类型或 enum"

// PartialEq 不是对象安全——同样故意
// 比较两个 dyn PartialEq 需要知道它们的动态类型相同，
// 这需要 RTTI（运行时类型信息），而 Rust 选择不支持
```

### 模式 3：对象安全规则在 trait 定义时检查

```rust
// 即使你永远不打算创建 dyn MyTrait，
// 编译器仍然在 trait 定义时检查对象安全性
// 这是为了确保 trait 的"语义一致性"

// 如果你确定某个 trait 不需要对象安全，有三条路：
// 1. 什么都不做——编译器不会因为你"没创建 dyn"就报错
// 2. 刻意让方法返回 Self——这会让 trait 明确不用于 dyn
// 3. 加 trait MyTrait: Sized——显式声明"我不支持 trait 对象"
```

### 练习 1 答案

```rust
trait A { fn foo(&self); }
// ✅ 对象安全 —— &self receiver，不返回 Self

trait B { fn foo(&self) -> Self; }
// ❌ 对象安全 —— 返回 Self，除非加 where Self: Sized

trait C { fn foo<T: Display>(&self, x: T); }
// ❌ 对象安全 —— 方法有泛型参数 T

trait D: Sized { fn foo(&self); }
// ❌ 对象安全 —— trait 本身要求 Self: Sized

trait E {
    fn foo(&self);
    fn bar(self) -> Self where Self: Sized;
}
// ✅ 对象安全 —— bar 有 where Self: Sized，被排除出 vtable

trait F {
    fn foo(&self);
    const N: usize;
}
// ❌ 对象安全 —— N 没有默认值

trait G {
    type Item;
    fn get(&self) -> &Self::Item;
}
// ✅ 对象安全 —— 关联类型不影响对象安全
// 但使用时需要指定关联类型：dyn G<Item = i32>
```

### 练习 2 答案

```rust
// ❌ 原始 trait（不是对象安全的）
// trait Repository {
//     fn find_by_id(&self, id: u64) -> Option<String>;  // ✅ 这个方法安全
//     fn create(data: String) -> Self;                    // ❌ 返回 Self，无 receiver
//     fn find_all() -> Vec<Self>;                         // ❌ 返回 Self，无 receiver
// }

// ✅ 修复方案：为静态方法加 where Self: Sized
trait Repository {
    /// 通过 ID 查找 —— 对象安全，可以通过 dyn Repository 调用
    fn find_by_id(&self, id: u64) -> Option<String>;

    /// 创建实例 —— 只在具体类型上可用
    fn create(data: String) -> Self
    where
        Self: Sized;

    /// 查找所有 —— 只在具体类型上可用
    fn find_all() -> Vec<Self>
    where
        Self: Sized;
}

// 使用示例：
struct UserRepo;

impl Repository for UserRepo {
    fn find_by_id(&self, id: u64) -> Option<String> {
        Some(format!("user_{}", id))
    }

    fn create(data: String) -> Self {
        UserRepo
    }

    fn find_all() -> Vec<Self> {
        vec![UserRepo]
    }
}

// dyn 调用（只能用 find_by_id）
fn lookup(repo: &dyn Repository, id: u64) -> Option<String> {
    repo.find_by_id(id)  // ✅
}

// 静态调用（所有方法都可用）
fn static_ops() {
    let repo = UserRepo::create("admin".into());
    let all = UserRepo::find_all();
}
```

### 练习 3 答案

```rust
trait Plugin {
    /// 插件名称 —— 对象安全
    fn name(&self) -> &str;

    /// 执行插件逻辑 —— 对象安全
    fn execute(&self, input: &str) -> String;
}

/// 插件工厂 —— 独立的 trait，不要求对象安全
trait PluginFactory {
    fn create(name: &str) -> Option<Self>
    where
        Self: Sized;
}

// 具体插件实现
struct UpperCasePlugin;
impl Plugin for UpperCasePlugin {
    fn name(&self) -> &str { "uppercase" }
    fn execute(&self, input: &str) -> String {
        input.to_uppercase()
    }
}

struct ReversePlugin;
impl Plugin for ReversePlugin {
    fn name(&self) -> &str { "reverse" }
    fn execute(&self, input: &str) -> String {
        input.chars().rev().collect()
    }
}

// 插件注册表 —— 使用 dyn Plugin
struct PluginRegistry {
    plugins: Vec<Box<dyn Plugin>>,
}

impl PluginRegistry {
    fn new() -> Self {
        PluginRegistry { plugins: Vec::new() }
    }

    fn register(&mut self, plugin: Box<dyn Plugin>) {
        self.plugins.push(plugin);
    }

    fn run_all(&self, input: &str) -> Vec<(String, String)> {
        self.plugins.iter()
            .map(|p| (p.name().to_string(), p.execute(input)))
            .collect()
    }
}

// 使用
fn main() {
    let mut registry = PluginRegistry::new();
    registry.register(Box::new(UpperCasePlugin));
    registry.register(Box::new(ReversePlugin));

    for (name, result) in registry.run_all("hello") {
        println!("[{}] -> {}", name, result);
    }
    // 输出：
    // [uppercase] -> HELLO
    // [reverse] -> olleh
}
```

### 练习 4 答案

**Q1: 为什么 `PartialEq` 不是对象安全的？这是缺陷还是设计选择？**

```
PartialEq 不是对象安全的因为 eq 方法的签名：

  fn eq(&self, other: &Self) -> bool;
  //              ^^^^^ 参数类型是 &Self

问题：当你通过 dyn PartialEq 比较两个值时，编译器需要知道：
  - 两个值的动态类型是否相同？
  - 如果不同，怎么比较？

C++ 的解决方案：RTTI（运行时类型信息）+ dynamic_cast
  → 运行时开销 + 可能抛出异常

Rust 的选择：不支持异构比较
  → "要比较两个值，它们必须在编译时确定是同一类型"
  → 这不是缺陷，而是"类型安全 > 运行时灵活性"的体现

如果你需要异构比较，用 enum 或 Any：
  let any1: Box<dyn Any> = ...;
  let any2: Box<dyn Any> = ...;
  if any1.type_id() == any2.type_id() {
      // 同类型，可以向下转型后比较
  }
```

**Q2: `dyn Fn(i32) -> i32` 为什么合法？Fn 不是有泛型吗？**

```
Fn / FnMut / FnOnce 是编译器特殊处理的 trait。

FnOnce 的定义（简化）：
  trait FnOnce(Args) {
      type Output;
      fn call_once(self, args: Args) -> Self::Output;
  }

问题：call_once 消耗 self（按值），按理说不是对象安全。

但编译器对这三个 trait 做了"魔法处理"：
  - dyn Fn(i32) -> i32 实际上是调用 call 方法（对 Fn trait）
  - Fn 继承 FnMut，FnMut 继承 FnOnce
  - call 方法用 &self，所以 Fn 是对象安全的
  - call_mut 用 &mut self，所以 FnMut 是对象安全的
  - call_once 用 self，但因为编译器魔法，可以正确分发

这是 Rust 中少数几个"编译器特殊处理"的 trait。
```

**Q3: 什么情况下选择手动 vtable 而不是 dyn Trait？**

```
1. 需要跨 FFI 边界传递 trait 对象
   dyn Trait 的 vtable 布局是 Rust 内部不稳定的
   手动 vtable 可以精确控制 ABI

2. 需要更紧凑的内存布局
   dyn Trait 总是胖指针（16 字节）
   手写可以实现"瘦指针 + 全局 vtable"（8 字节）

3. 需要自定义分发逻辑
   例如根据两个维度的 vtable 做多重分发

4. 嵌入/no_std 环境
   可能没有完整的标准库支持，手写更可控

5. 性能的极致控制
   避免编译器生成的 vtable 查找的间接层次

实际项目中：99% 的情况用 dyn Trait 就够了。
手写 vtable 通常是框架作者和系统编程场景的需要。
```

## 15. 深度扩展

### 15.1 编译器如何构建 vtable —— 完整的内存布局

上一课（1.1）讲了 vtable 的概念。这里深入到**对象安全性直接决定的 vtable 形状**。

```rust
trait Animal {
    fn speak(&self) -> String;
    fn eat(&mut self, food: &str);
}

struct Dog { name: String, age: u8 }

impl Animal for Dog {
    fn speak(&self) -> String { format!("{} says woof", self.name) }
    fn eat(&mut self, food: &str) { println!("{} eats {}", self.name, food); }
}

// 编译器为 Dog 生成的 vtable（数据在 .rodata 段）：
// static DOG_ANIMAL_VTABLE: AnimalVTable = AnimalVTable {
//     drop_in_place: <Dog as Drop>::drop_in_place,  // 或默认释放
//     size: 32,        // Dog 的大小（String=24 + u8=1，对齐到 32）
//     align: 8,        // Dog 的对齐
//     speak: <Dog as Animal>::speak,   // fn(*const Dog) -> String 的类型擦除版本
//     eat:   <Dog as Animal>::eat,     // fn(*mut Dog, &str) 的类型擦除版本
// };
```

**方法指针的类型擦除**：vtable 中所有方法签名都被擦除为裸指针参数。这是对象安全规则存在的根本原因——编译器必须能写出一个确定的、无泛型的函数指针类型。

```rust
// speak 的实际签名是 fn(&self) -> String
// 在 vtable 中变成：fn(*const ()) -> String
//   self: &Dog   →   *const Dog   →   *const ()（类型擦除）

// 这就是为什么"返回 Self"不行——返回类型 Self 被擦除后是 ()，
// 编译器无法在调用点重建具体类型
```

### 15.2 为什么 `fn(self)` 不能进 vtable？—— 从调用约定理解

```
按值 self 的问题：

fn consume(self) { ... }
// self 的类型是 Self（在 vtable 中是未知的）
// 参数传递时，编译器需要知道：
//   1. self 放在哪个寄存器/栈位置？
//   2. self 占多少字节？（32？0？100？）
//   3. 调用者如何准备参数？（memcpy 多少字节？）

如果返回 Self：
// fn clone(&self) -> Self
// 返回值也需要知道大小——调用者需要分配空间容纳返回值

// C++ 怎么解决的？C++ 虚函数不能返回"未知大小的类型"。
// C++ 的协变返回类型也要求返回的是指针或引用，不是值。

// Rust 的解决方案：用 where Self: Sized 排除这些方法
// 这样 vtable 只包含"签名确定"的方法
```

### 15.3 多重 trait 对象：`dyn TraitA + TraitB`

```rust
trait Speak {
    fn speak(&self) -> &str;
}

trait Walk {
    fn walk(&self);
}

// 可以组合多个 trait：
fn demo(x: &(dyn Speak + Walk)) {
    x.speak();
    x.walk();
}

// 内部表示：胖指针包含一个合并的 vtable
// &(dyn Speak + Walk) = {
//     data_ptr: *const (),
//     vtable_ptr: &CombinedVTable {
//         // 来自 Speak 的方法
//         speak: fn(*const ()) -> &str,
//         // 来自 Walk 的方法
//         walk: fn(*const ()),
//         // 布局信息（来自两个 trait 的大小/对齐取 max？不——实际只有一个 drop/size/align）
//         drop_in_place: fn(*mut ()),
//         size: usize,
//         align: usize,
//     }
// }
```

**关键约束**：
- 所有组合的 trait 都必须对象安全
- `dyn TraitA + TraitB + TraitC + AutoTrait` 是常见的（AutoTrait 如 Send、Sync 不需要 vtable 条目）
- 但不能 `dyn TraitA + TraitB` 其中 TraitA 和 TraitB 有同名方法——编译器会报歧义

```rust
trait A { fn foo(&self); }
trait B { fn foo(&self); }

// ❌ 歧义！dyn A + B 的 vtable 中应该放 A::foo 还是 B::foo？
// fn demo(x: &(dyn A + B)) { x.foo(); }  // 编译错误
```

### 15.4 对象安全与 RPITIT（Return Position Impl Trait In Trait）

Rust 1.75 稳定了 RPITIT——在 trait 中使用 `async fn` 或 `-> impl Trait`：

```rust
trait Service {
    async fn process(&self) -> String;
    // 等价于：
    // fn process(&self) -> impl Future<Output = String>;
}

// ❌ Service 不是对象安全的！
// 为什么？process 返回 impl Future，这是一个"存在类型"——
// 编译器不知道 Future 的具体大小，无法放进 vtable

// ✅ 如果要对象安全，用 Box：
trait Service {
    fn process(&self) -> Box<dyn Future<Output = String> + Send + '_>;
}
// 现在 Service 是对象安全的——Box 的大小固定（8字节）
```

### 15.5 Trait Upcasting —— `dyn SubTrait` 到 `dyn SuperTrait`

这是 Rust 正在稳定化的特性（1.76+ 逐渐完善）：

```rust
trait Animal { fn speak(&self) -> &str; }
trait Dog: Animal { fn wag_tail(&self); }

// #![feature(trait_upcasting)]  // 早期需要 nightly
fn demo(dog: &dyn Dog) {
    // 从 dyn Dog 上转型到 dyn Animal
    let animal: &dyn Animal = dog;  // 自动 upcast（1.76+ 部分稳定）
    animal.speak();
}

// 底层原理：DogVTable 的前 N 个条目就是 AnimalVTable 的布局
// 编译器保证 supertrait 的 vtable 条目排在前面
// 所以 upcast 只需取出 vtable_ptr 的前半部分——零开销！
```

### 15.6 `dyn Any` —— 有限的运行时类型信息

`Any` trait 是 Rust 中唯一的"官方 RTTI"：

```rust
use std::any::Any;

fn inspect(value: &dyn Any) {
    if let Some(s) = value.downcast_ref::<String>() {
        println!("It's a String: {}", s);
    } else if let Some(i) = value.downcast_ref::<i32>() {
        println!("It's an i32: {}", i);
    } else {
        println!("Unknown type: {:?}", value.type_id());
    }
}

inspect(&"hello".to_string());  // It's a String: hello
inspect(&42i32);                 // It's an i32: 42
```

**Any 为什么是对象安全的？**

```rust
// Any trait（简化）：
pub trait Any: 'static {
    fn type_id(&self) -> TypeId;  // ✅ 返回 TypeId，不涉及 Self
}
// type_id 实现在编译时生成，运行时返回一个固定值
// Any 没有返回 Self、没有泛型方法——完全对象安全
```

**Any 的限制**：只能 `downcast_ref` 到编译时已知的类型。你不能"枚举所有可能的类型"——这是 Rust 有意不支持完整 RTTI 的结果。

### 15.7 动态分发 vs 静态分发的性能实测

不只是理论——实际数字：

```rust
use std::time::Instant;

trait Calculator {
    fn compute(&self, x: i32) -> i32;
}

struct AddOne;
impl Calculator for AddOne {
    fn compute(&self, x: i32) -> i32 { x + 1 }
}

// 静态分发
fn static_dispatch(c: &AddOne, n: i32) -> i32 {
    (0..n).fold(0, |acc, x| acc + c.compute(x))
}

// 动态分发
fn dynamic_dispatch(c: &dyn Calculator, n: i32) -> i32 {
    (0..n).fold(0, |acc, x| acc + c.compute(x))
}

// 典型结果（release build，1000 万次调用）：
// static:  ~0.5ms   （compute 被内联，fold 循环完全展开）
// dynamic: ~8ms     （每次 compute 都是间接调用，无法内联）
// 差距约 16 倍——动态分发的开销在热路径上是真实的
```

**经验法则**：热循环中不要用 trait 对象。冷路径（如配置解析、插件加载）用 trait 对象是完全合理的。

### 15.8 "对象安全"不是质量的评判标准

```
一个常见的误解：对象安全的 trait = 好的 trait，不安全的 = 坏的。

真相：
- Clone 不是对象安全 → 故意设计，不是缺陷
- Iterator 是对象安全 → 但实际很少用 dyn Iterator（性能原因）
- Fn/FnMut 是对象安全 → 编译器特殊处理，常规 trait 做不到

设计 trait 时先想"这个 trait 的用途是什么"：
- 如果主要用于泛型/静态分发 → 不需要对象安全（Clone, Default）
- 如果需要异质集合/插件 → 需要对象安全（Plugin, EventHandler）
- 如果两者都需要 → 用 where Self: Sized 模式（Iterator）

让设计驱动规则，不要让规则驱动设计。
```

**本课核心知识链：**

1. **对象安全** = trait 能否用于 `dyn Trait` 的判断标准
2. **四条规则**：trait 不能要求 `Self: Sized` / 方法不能返回 `Self` / 方法不能有泛型 / 关联常量必须有值
3. **`where Self: Sized` 技巧** = 让非安全方法存在但不进 vtable，保持 trait 整体安全
4. **super trait 传递性** = 父 trait 不安全则子 trait 也不安全
5. **关联类型** = 不影响对象安全，但使用 `dyn` 时必须指定
6. **四种修复策略** = enum dispatch / 拆分 trait / 函数指针 / 手动 vtable
