# 1.5 转换类：From/Into、AsRef/AsMut、Borrow/ToOwned

> Rust 有 7 个核心转换 trait，但它们的职责边界经常被混淆。这一课清楚地划分：**什么情况下用哪个，以及为什么不能混用**。

---

## 1. 转换全景图——一张表看清所有关系

```
┌──────────────────────────────────────────────────────────────────┐
│                    Rust 类型转换体系                               │
├──────────────┬─────────────────────┬──────────────────────────────┤
│   Trait      │  方向               │  语义                        │
├──────────────┼─────────────────────┼──────────────────────────────┤
│ From<T>      │ T → Self           │ 所有权转移，不会失败           │
│ Into<T>      │ Self → T           │ From 的反转（自动获得）       │
│ TryFrom<T>   │ T → Result<Self,E> │ 所有权转移，可能失败           │
│ TryInto<T>   │ Self → Result<T,E> │ TryFrom 的反转                │
│ AsRef<T>     │ &Self → &T         │ 廉价引用转换                  │
│ AsMut<T>     │ &mut Self → &mut T │ 廉价可变引用转换              │
│ Borrow<T>    │ &Self → &T         │ 与 AsRef 类似但语义更强       │
│ ToOwned      │ &Self → Owned      │ Borrow 的反向（Clone 的泛化） │
└──────────────┴─────────────────────┴──────────────────────────────┘
```

## 2. From / Into：所有权的单向转移

### 2.1 为什么是两个 trait？

```rust
// 你只需要实现 From
impl From<i32> for MyNumber {
    fn from(value: i32) -> Self {
        MyNumber(value as f64)
    }
}

// Into 自动获得！标准库有 blanket impl：
// impl<T, U> Into<U> for T where U: From<T> { ... }
```

**设计上的优雅**：实现 `From`，`Into` 免费赠送。你永远**只实现 From，使用 Into**：

```rust
// 实现 From——你做的事
impl From<&str> for MyString {
    fn from(s: &str) -> Self {
        MyString(s.to_owned())
    }
}

// 使用 Into——调用方做的事
fn process(s: impl Into<MyString>) {
    let my_s: MyString = s.into();
    // ...
}

process("hello");           // &str → 自动调用 From<&str>
process(MyString::from("hello")); // 也可以直接传
```

### 2.2 From 的"传染性"——标准库为什么到处都是 From

```rust
// String 的 From 实现矩阵：
impl From<&str> for String { }        // 字符串切片 → String
impl From<&String> for String { }     // 引用 → 拥有（克隆）
impl From<String> for Box<str> { }    // String → Box<str>
impl From<String> for Vec<u8> { }     // String → 字节向量
impl From<String> for Box<dyn Error> { }  // String → trait object
impl From<String> for OsString { }    // 跨平台字符串
impl From<String> for PathBuf { }     // 路径
impl From<String> for Rc<str> { }     // 引用计数字符串
impl From<char> for String { }        // 单字符 → String
impl<'a> From<Cow<'a, str>> for String { }  // Cow → String
```

**观察到什么？** `From` 形成了一个**所有权转换网络**：任何类型只要能"变成"另一个类型（不丢失信息），就应该有 `From` 实现。

### 2.3 什么时候实现 From

```rust
/// ✅ 正确：无歧义的转换，不丢失信息
impl From<u16> for u32 {
    fn from(small: u16) -> u32 { small as u32 }
}

/// ✅ 正确：不同表示，保持语义
impl From<Color> for [u8; 3] {
    fn from(c: Color) -> [u8; 3] { [c.r, c.g, c.b] }
}

/// ❌ 错误：丢失信息的转换不应该用 From
impl From<i32> for u32 {
    fn from(x: i32) -> u32 { x as u32 }  // 负数会变成大正数
}
// 应该用 TryFrom 而不是 From！
```

### 2.4 TryFrom / TryInto：可能失败的转换

```rust
use std::convert::TryFrom;

impl TryFrom<i32> for u32 {
    type Error = &'static str;

    fn try_from(x: i32) -> Result<Self, Self::Error> {
        if x >= 0 {
            Ok(x as u32)
        } else {
            Err("negative number cannot be converted to u32")
        }
    }
}

// 使用
let n: i32 = -5;
match u32::try_from(n) {
    Ok(val) => println!("OK: {}", val),
    Err(e) => println!("Failed: {}", e),
}

// 更简洁：用 ? 运算符（配合 Result 上下文）
fn do_stuff(x: i32) -> Result<(), Box<dyn std::error::Error>> {
    let y: u32 = u32::try_from(x)?;  // 失败时自动传播错误
    Ok(())
}
```

### 2.5 From / Into 在泛型 API 中的威力

```rust
// ❌ 不够泛用的写法
fn save_data(data: &str, path: &str) { /* ... */ }

// ✅ 泛用的写法：接受任何能变成 String 和 PathBuf 的东西
fn save_data(
    data: impl Into<String>,
    path: impl Into<PathBuf>,
) -> std::io::Result<()> {
    let data: String = data.into();
    let path: PathBuf = path.into();
    std::fs::write(path, data)
}

// 调用方不需要手动转换
save_data("hello world", "/tmp/test.txt");  // &str → String, &str → PathBuf
save_data(String::from("hello"), PathBuf::from("/tmp/test.txt")); // 也行
```

## 3. AsRef / AsMut：廉价的引用转换

### 3.1 核心语义

```rust
// AsRef：我可以"作为"某种类型的引用被看待
pub trait AsRef<T: ?Sized> {
    fn as_ref(&self) -> &T;
}

// AsMut：可变版本
pub trait AsMut<T: ?Sized> {
    fn as_mut(&mut self) -> &mut T;
}
```

**关键约束**：`AsRef` 必须是**廉价的**（通常是直接返回内部引用，不分配内存）。

### 3.2 标准使用模式

```rust
// String 实现了 AsRef<str>、AsRef<[u8]>、AsRef<Path>、AsRef<OsStr>
fn open_file(path: impl AsRef<Path>) -> std::io::Result<File> {
    let path: &Path = path.as_ref();
    File::open(path)
}

// 这些都能用：
open_file("/etc/hosts");             // &str
open_file(String::from("/etc/hosts")); // String
open_file(Path::new("/etc/hosts"));   // &Path
open_file(PathBuf::from("/etc/hosts")); // PathBuf
open_file(OsStr::new("/etc/hosts"));  // &OsStr
```

### 3.3 什么时候实现 AsRef

```rust
struct User {
    name: String,
    email: String,
}

// ✅ 正确：User 的内部有 String，可以引用它的 inner
impl AsRef<str> for User {
    fn as_ref(&self) -> &str {
        &self.name
    }
}

// ✅ 正确：User 可以被视为 &[u8]（email 的字节表示）
//         但这也暴露了设计问题——哪个字段是"主要的"？
impl AsRef<[u8]> for User {
    fn as_ref(&self) -> &[u8] {
        self.email.as_bytes()
    }
}

// ❌ 不要这样：转换不廉价
impl AsRef<String> for User {
    fn as_ref(&self) -> &String {
        // 如果 User 存的是 String，这样是可以的
        &self.name  // ✅ 返回引用，廉价
    }
}

// ❌ 绝不要这样：分配了内存
impl AsRef<String> for User {
    fn as_ref(&self) -> &String {
        // 这种情况你返回不了 &String，你需要创建新的 String
        // → 不应该用 AsRef
    }
}
```

### 3.4 AsRef 的核心规则

```
AsRef 的黄金法则：
├── 转换必须是极其廉价的（O(1)，无分配）
├── 通常直接返回内部字段的引用
├── 一个类型可以实现多个 AsRef（比如对 str、[u8]、Path）
└── 如果转换昂贵或有失败可能，用 From / TryFrom，别用 AsRef
```

## 4. Borrow / ToOwned：比 AsRef 更强的语义

### 4.1 Borrow 和 AsRef 的本质区别

```rust
// AsRef：肤浅的引用转换，"我可以作为 T 被看待"
// Borrow：深层的等价关系，"我的引用和 T 的引用在 Hash/Eq/Ord 上一致"

pub trait Borrow<Borrowed: ?Sized> {
    fn borrow(&self) -> &Borrowed;
}
```

**关键差异示例**：

```rust
use std::borrow::Borrow;
use std::collections::HashMap;

#[derive(Eq, PartialEq, Hash)]
struct UserId(u64);

// 实现 Borrow——语义上"UserId 可以被 Borrow 为 u64"
impl Borrow<u64> for UserId {
    fn borrow(&self) -> &u64 {
        &self.0
    }
}

// 这就有用了：
let mut map: HashMap<UserId, String> = HashMap::new();
map.insert(UserId(42), "Alice".into());

// 可以用 &u64 查找，因为 HashMap 用 Borrow trait
assert_eq!(map.get(&42u64), Some(&"Alice".to_string()));
```

**如果改用 `AsRef<u64>` 能达到同样效果吗？**

**不能。** `HashMap::get` 用的是 `Borrow` trait，不是 `AsRef`。原因：

```rust
// HashMap::get 的签名（简化）：
impl<K, V> HashMap<K, V> {
    pub fn get<Q: ?Sized>(&self, k: &Q) -> Option<&V>
    where
        K: Borrow<Q>,     // ← 注意：是 Borrow，不是 AsRef
        Q: Hash + Eq,
    { /* ... */ }
}
```

### 4.2 Borrow 的额外要求——等价性

```rust
// Borrow 要求：borrow() 返回的引用在 Hash、Eq、Ord 上
// 必须和原值的行为一致。

// ❌ 违反 Borrow 契约的例子（虽然能编译）：
impl Borrow<i32> for UserId {
    fn borrow(&self) -> &i32 {
        // 强行返回一个和原值无关的 i32
        &42  // 这不可能编译通过（生命周期问题），但是思想实验
    }
}
```

**Borrow 的核心契约**：

```
如果 x.borrow() == y.borrow()，那么 x == y 必须也成立。
```

这意味着 `Borrow` 的转换必须保持"同一性"——不能随意映射。

### 4.3 ToOwned：Borrow 的反向

```rust
pub trait ToOwned {
    type Owned: Borrow<Self>;
    fn to_owned(&self) -> Self::Owned;
}

// 标准实现示例：
// str 实现 ToOwned<Owned = String>
// [T] 实现 ToOwned<Owned = Vec<T>>
// Path 实现 ToOwned<Owned = PathBuf>
// CStr 实现 ToOwned<Owned = CString>
```

**ToOwned 是 Clone 的泛化**：

```rust
// Clone：T → T（同一个类型）
// ToOwned：&T → T::Owned（可以从借用类型创建拥有类型）

let s: &str = "hello";
let owned: String = s.to_owned();  // ToOwned：&str → String

let s: &String = &String::from("hello");
let cloned: String = s.clone();    // Clone：&String → String
```

## 5. 六个 trait 的对比矩阵

| 维度 | From | Into | AsRef | AsMut | Borrow | ToOwned |
|:---|:---|:---|:---|:---|:---|:---|
| **方向** | T → Self | Self → T | &Self → &T | &mut Self → &mut T | &Self → &T | &Self → Owned |
| **所有权** | 获取所有权 | 获取所有权 | 借用 | 借用 | 借用 | 创建新所有者 |
| **成本** | 不限 | 不限 | 必须廉价 | 必须廉价 | 必须廉价 | 任意 |
| **可否失败** | 否 | 否 | 否 | 否 | 否 | 否 |
| **典型场景** | 构造函数 | API 入口 | 函数参数 | 函数参数 | HashMap 查找 | 借用到拥有 |
| **实现准则** | 单向实现 | 自动获得 | 多对多 | 多对多 | 保持等价性 | 两个方向结合 |
| **可派生** | #[derive] 可以 | — | — | — | — | — |

## 6. 实战决策树

遇到"一个类型变另一个类型"的需求时，按这个决策树选：

```
需要转换类型？
├── 所有权转移？
│   ├── 永远不会失败 → 实现 From
│   └── 可能失败 → 实现 TryFrom
│
├── 只是借用/引用？
│   ├── 廉价引用转换 → AsRef / AsMut
│   ├── 需要在 HashMap/BTreeMap 中查找 → Borrow
│   └── Borrow 的反向（借用到拥有） → ToOwned
│
└── 只是复制一份？ → Clone
```

## 7. 常见陷阱

### 陷阱 1：混淆 AsRef 和 From

```rust
// ❌ 错误：用 AsRef 做昂贵的转换
impl AsRef<String> for MyType {
    fn as_ref(&self) -> &String {
        // 你没法返回 &String 除非 MyType 内部已经存了一个 String
        // 如果在此处分配内存，你在返回一个悬垂引用！
    }
}

// ✅ 正确：用 From 做所有权转换
impl From<MyType> for String {
    fn from(mt: MyType) -> String {
        format!("{:?}", mt)  // 分配是 OK 的
    }
}
```

### 陷阱 2：为同一对类型同时实现 AsRef 和 Borrow

```rust
// ⚠️ 虽然技术上可以，但会产生歧义
impl AsRef<str> for UserId { /* ... */ }
impl Borrow<str> for UserId { /* ... */ }

// 调用时：
let uid = UserId(42);
let s: &str = uid.as_ref();       // AsRef
let s: &str = uid.borrow();        // Borrow
// 功能完全一样，容易混淆
```

**建议**：大多数情况下只实现 `AsRef`。只有当你需要在 `HashMap`/`BTreeMap` 中使用该类型作为 key 时，才实现 `Borrow`。

### 陷阱 3：忘记了 `impl From<T> for U` 会自动提供 `impl Into<U> for T`

```rust
// 你实现了这个：
impl From<&str> for MyString {
    fn from(s: &str) -> Self { MyString(s.to_owned()) }
}

// 这些就自动可用了（无需额外代码）：
let ms: MyString = "hello".into();         // 自动获得的 Into
let ms = MyString::from("hello");          // 你明确实现的 From
fn take_owned(s: impl Into<MyString>) { }  // 泛型 API
take_owned("hello");                       // 自动转换
```

### 陷阱 4：AsRef 的泛型二义性

```rust
struct Container {
    data: Vec<u8>,
}

impl AsRef<[u8]> for Container {
    fn as_ref(&self) -> &[u8] { &self.data }
}

// 问题：如果一个类型实现了多个 AsRef
impl AsRef<Vec<u8>> for Container {
    fn as_ref(&self) -> &Vec<u8> { &self.data }
}

fn process<T: AsRef<[u8]>>(data: T) {
    let slice: &[u8] = data.as_ref();  // 调用哪个？编译器可能报错
}

// ✅ 解决方法：不要为同一类型实现多个相互冲突的 AsRef
```

## 8. 设计哲学：Rust 的转换体系为什么这样设计

### 8.1 为什么需要一套"转换 trait"——C++ 的教训

如果你学过 C++，你一定见过这些：

```cpp
// C++ 的转换方式——隐式的、非统一的、容易出错的

// 1. 隐式转换构造函数（单参数构造函数默认允许隐式转换）
class String {
public:
    String(const char* s);  // 隐式！const char* → String 自动发生
};

void foo(String s) { }
foo("hello");  // OK——const char* 隐式转换为 String
// 问题：转换发生得太"安静"了，你可能完全没有意识到这里有一次堆分配

// 2. 转换运算符（隐式！）
class UserId {
    int id;
public:
    operator int() const { return id; }  // UserId → int 隐式转换
};

UserId uid{42};
int x = uid + 1;  // OK——隐式转换为 int 然后做加法
// 问题：UserId 和 int 的语义边界被打破了

// 3. 没有统一的"可能失败"的转换机制
// 只能用异常或者返回特殊值——没有类型系统层面的 TryFrom<T>
int parse_int(const std::string& s) {
    return std::stoi(s);  // 失败时抛异常——不是类型安全的
}
```

**C++ 的根本问题**：转换是**隐式的、分散的、没有统一接口的**。一个函数接受什么类型、可以产生什么类型——完全靠文档和约定。编译器不会帮你检查"这个转换失败了怎么办"。

**Rust 的方案**：

| C++ 问题 | Rust 解法 |
|:---|:---|
| 隐式转换不可见 | `From`/`Into` 需要显式调用 `.into()`——分配/转换在调用处可见 |
| 失败转换没有统一机制 | `TryFrom`/`TryInto` 返回 `Result`——`?` 运算符统一传播 |
| 引用转换和所有权转换混淆 | `AsRef`（廉价引用）vs `From`（所有权）——意图明确 |
| HashMap 查找的类型等价无法保证 | `Borrow` trait 约束等价性——`hash(a) == hash(b)` ↔ `a == b` |
| 没有从借用创建所有者的统一方法 | `ToOwned` —— `&str → String`, `&Path → PathBuf` 等统一 |

### 8.2 如果没有 Rust 的转换 trait 体系——用 C++ 的方式"手工实现"会怎样

**假设 Rust 没有 `From`/`Into`**，你只能这样写：

```rust
// 每个转换自己写一个构造函数
impl String {
    fn from_str(s: &str) -> String { /* ... */ }
    fn from_char(c: char) -> String { /* ... */ }
    fn from_cow(c: Cow<str>) -> String { /* ... */ }
    // 没有统一的 trait → 每一个转换函数名字可能不同
    // 泛型代码无法表达"任何可以变成 String 的类型"
}

// 泛型函数不好写：
// fn save<T: ???>(data: T) { let s: String = ???; }
// 没有 From trait，你无法表达"T 可以变成 String"
// 你只能：fn save(data: &str)、fn save(data: String)、fn save(data: &String)...
// 为每一种类型写一个重载 → 代码膨胀
```

**假设 Rust 没有 `AsRef`**：

```rust
// 文件操作只能接受 &Path
fn open_file(path: &Path) -> io::Result<File> { /* ... */ }

// 调用方必须手动转换：
let s = String::from("/etc/hosts");
let p = Path::new(&s);   // 手动的
let f = open_file(p);    // 麻烦但能工作

// 但这更加暴露：
let p = PathBuf::from("/etc/hosts");
// open_file(&p);  // ❌ &PathBuf 不能自动变 &Path... 等等，其实可以（Deref）
// 但如果 PathBuf::as_path() 没有统一的 trait 名字呢？
// 每个类型的方法名不同：PathBuf::as_path(), OsStr::as_os_str(), String::as_str()
// → 泛型代码无法抽象"可以变成 Path 引用的类型"
```

**假设 Rust 没有 `Borrow`**，HashMap 只能用具体类型查找：

```rust
let mut map: HashMap<UserId, Profile> = HashMap::new();
// 只能这样查找：
let uid = UserId(42);
let profile = map.get(&uid);  // 必须构造完整的 UserId！

// 不能：map.get(&42u64)  // ❌ 没有 Borrow<u64>
// 每次查找都要构造 UserId → 烦琐
```

**这就是 Rust 转换 trait 存在的意义**：它们把"类型之间的转换"变成了**类型系统的一等公民**，让泛型代码可以在编译期表达和利用转换关系。

### 8.3 设计决策总结

| 决策 | 原因 | C++ 对比 |
|:---|:---|:---|
| **From/Into 分离** | 你只实现 From，Into 自动获得。减少实现者的工作量 | C++ 的 `operator T()` 是单向的——是隐式的，也没有"自动反向" |
| **AsRef 廉价约束** | 不强制检查但通过惯例约束——违反会导致性能问题 | C++ 的 `const_cast`/`reinterpret_cast` 完全不区分代价 |
| **Borrow ≠ AsRef** | `HashMap` 需要等价性保证，`AsRef` 不提供这个 | C++ 的 `std::hash` + `operator==` 是两个独立的概念，没有 trait 绑定——容易不一致 |
| **TryFrom 独立 trait** | 不污染 `From` 的成功语义——明确分离正确和错误路径 | C++ 用异常或 `std::optional`——没有编译期区分"能失败"和"不能失败" |
| **ToOwned 泛化 Clone** | `Clone` 是 `T → T`，`ToOwned` 是 `&T → Owned`——更灵活 | C++ 没有统一概念——有时是拷贝构造，有时是特殊工厂方法 |

## 9. 练习

### 练习 1：为自定义类型实现转换

定义 `struct Celsius(f64)` 和 `struct Fahrenheit(f64)`，实现：

- `From<Celsius> for Fahrenheit`（公式：F = C * 9/5 + 32）
- `From<Fahrenheit> for Celsius`（公式：C = (F - 32) * 5/9）
- 验证 `.into()` 调用自动可用

## 10. 练习答案

### 练习 1 答案

```rust
struct Celsius(f64);
struct Fahrenheit(f64);

impl From<Celsius> for Fahrenheit {
    fn from(c: Celsius) -> Self {
        Fahrenheit(c.0 * 9.0 / 5.0 + 32.0)
    }
}

impl From<Fahrenheit> for Celsius {
    fn from(f: Fahrenheit) -> Self {
        Celsius((f.0 - 32.0) * 5.0 / 9.0)
    }
}

fn main() {
    let c = Celsius(100.0);
    let f: Fahrenheit = c.into();  // ✅ Into 自动可用
    println!("100°C = {}°F", f.0);

    let f = Fahrenheit(212.0);
    let c: Celsius = f.into();
    println!("212°F = {}°C", c.0);

    // 甚至可以直接用 From
    let c2 = Celsius::from(Fahrenheit(32.0));
    println!("32°F = {}°C", c2.0);  // 0°C
}
```

### 练习 2 答案

```rust
use std::path::{Path, PathBuf};
use std::fs;

struct Config {
    path: String,
    timeout: u64,
}

impl AsRef<Path> for Config {
    fn as_ref(&self) -> &Path {
        Path::new(&self.path)
    }
}

fn load_config(path: impl AsRef<Path>) -> std::io::Result<String> {
    let path = path.as_ref();
    fs::read_to_string(path)
}

fn main() -> std::io::Result<()> {
    let cfg = Config {
        path: String::from("/tmp/test.cfg"),
        timeout: 30,
    };

    // Config 可以直接作为 &Path 使用
    let content = load_config(&cfg)?;
    println!("Config: {}", content);
    Ok(())
}
```

### 练习 3 答案

```rust
use std::borrow::Borrow;
use std::collections::HashMap;

#[derive(Debug, PartialEq, Eq, Hash)]
struct Email(String);

impl Email {
    fn new(s: impl Into<String>) -> Option<Self> {
        let s: String = s.into();
        if s.contains('@') {
            Some(Email(s))
        } else {
            None
        }
    }
}

impl Borrow<str> for Email {
    fn borrow(&self) -> &str {
        &self.0
    }
}

#[derive(Debug)]
struct UserProfile {
    name: String,
    age: u8,
}

fn main() {
    let mut users: HashMap<Email, UserProfile> = HashMap::new();

    let alice_email = Email::new("alice@example.com").unwrap();
    users.insert(alice_email, UserProfile {
        name: "Alice".into(),
        age: 30,
    });

    // 用 &str 查找——Borrow trait 使得这个能工作！
    let key = "alice@example.com";  // &str
    match users.get(key) {           // key 自动被 borrow 为 &str
        Some(profile) => println!("Found: {:?}", profile),
        None => println!("Not found"),
    }
}
```

### 练习 4 答案

```rust
// From<T> 在 T = &str 时的实现：
impl<'a> From<&'a str> for Wrapper<&'a str> {
    fn from(t: &'a str) -> Self { Wrapper(t) }
}

// Into 的自动实现：
// impl<T, U> Into<U> for T where U: From<T> { ... }

// 问题：
let s = String::from("hello");
// let w: Wrapper<&str> = s.into();
//                          ^^^^^^^^
// Into 需要 U: From<T>，即 Wrapper<&str>: From<String>
// 但我们只实现了 From<T> for Wrapper<T>（T 相同）
// From<String> for Wrapper<&str> 不存在！
// String ≠ &str，所以泛型 T 必须一致！

// 正确做法：
let w: Wrapper<&str> = (&s).into();
// (&s) 是 &String，而 &String 可以自动解引用强制转换为 &str
// 所以 From<&str> for Wrapper<&str> 匹配上了
```

## 11. 深度扩展

### 11.1 From 的零成本抽象原理

Rust 的 `From` 在发布构建中编译成**和内联代码完全相同的机器码**：

```rust
// 这两段代码生成完全相同的汇编：

// 版本 1：显式转换
fn explicit(x: u32) -> u64 { x as u64 }

// 版本 2：用 From
fn via_from(x: u32) -> u64 { u64::from(x) }

// 版本 3：用 Into
fn via_into(x: u32) -> u64 { x.into() }
```

编译器因为单态化和内联，三个版本完全等价。`From`/`Into` 是零成本的类型安全包装。

### 11.2 AsRef 在 trait bound 上的"过度宽容"

```rust
// AsRef<Path> 看起来很方便，但有隐藏成本：
fn open_config(path: impl AsRef<Path>) -> std::io::Result<File> {
    File::open(path.as_ref())
}

// 实际上任何实现了 AsRef<Path> 的类型都会单态化出一个独立版本
// 如果调用方传了 5 种不同类型，就会生成 5 份代码（代码膨胀）
//
// 权衡：
// - 小型 crate / 应用程序：用 AsRef 没问题
// - 大型库：考虑用 &Path 然后让调用方手动转换
```

### 11.3 Blanket impl 如何让转换体系"自洽"

标准库利用 blanket impl 让转换网络自动闭合：

```rust
// 1. 反射性：每个类型都可以变成自己
impl<T> From<T> for T {
    fn from(t: T) -> T { t }
}
// 这意味着 u32::from(42u32) 是合法的（虽然是恒等操作）

// 2. 传递性（部分）：
// 如果你有 From<A> for B 和 From<B> for C，
// 不能自动得到 From<A> for C，但可以用 A → Into<B> → Into<C>
```

### 11.4 Cow（Clone-on-Write）与转换体系的深度集成

```rust
use std::borrow::Cow;

// Cow 的美妙之处在于它依赖 Borrow + ToOwned：
// pub enum Cow<'a, B: ?Sized + 'a>
// where
//     B: ToOwned,
// {
//     Borrowed(&'a B),
//     Owned(<B as ToOwned>::Owned),
// }

fn process(data: Cow<str>) -> String {
    // 如果传入的是 &str（Borrowed 状态），不分配内存
    // 如果传入的是 String（Owned 状态），不克隆
    // 只有在需要修改时，调用 to_mut() 才分配
    match data {
        Cow::Borrowed(s) => format!("readonly: {}", s),
        Cow::Owned(s) => format!("owned: {}", s),
    }
}

// 妙用：
let s1: Cow<str> = Cow::Borrowed("hello");     // 不分配
let s2: Cow<str> = Cow::Owned(String::from("hello")); // String 直接存入
// From<String> for Cow<'_, str> 也是通过 blanket impl 实现的
```

### 11.5 为什么没有 `AsOwned` trait

你可能会觉得不对称：

```
AsRef<str>  for String    ✅ 有
AsRef<Path> for PathBuf   ✅ 有
"反向"特质？              ❌ 没有 AsOwned
```

**原因**：`ToOwned` + `From` 已经覆盖了所有权反向转换的需求。增加 `AsOwned` 会让 API 冗余且更复杂。

### 11.6 `impl Into<String>` vs `impl AsRef<str>`——API 设计的选择

```rust
// 方式 A：获取所有权
fn process_a(s: impl Into<String>) {
    let s: String = s.into();
    // s 被拥有，可以修改，可以存起来，可以发到线程
}

// 方式 B：只需引用
fn process_b(s: impl AsRef<str>) {
    let s: &str = s.as_ref();
    // s 只是引用，不能修改，不能存到其他地方
}

// 选择标准：
// - 如果你需要拥有数据 → Into<String>
// - 如果你只需要读取数据 → AsRef<str>
// - 如果你的函数同时有读取和存储 → Into<String>
```

### 11.7 Deref 强制转换 vs AsRef 的区别

很多人混淆这两个：

```rust
use std::ops::Deref;

struct MyString(String);

// Deref：使得 &MyString 可以自动变成 &String
impl Deref for MyString {
    type Target = String;
    fn deref(&self) -> &String { &self.0 }
}

// AsRef：不会自动发生，需要显式调用 .as_ref()
impl AsRef<str> for MyString {
    fn as_ref(&self) -> &str { &self.0 }
}

// 差异：
let s = MyString(String::from("hello"));
let _: &String = &s;       // ✅ Deref：自动强制转换
let _: &str    = s.as_ref(); // ✅ AsRef：显式调用
// let _: &str = &s;        // ❌ Deref 不会链式强制：&MyString → &String，但不继续 → &str
```

**规则**：Deref 是"隐式的、有限链的"，AsRef 是"显式的、无限的"。

### 11.8 `From` 与错误处理的深度结合

Rust 错误处理的基础设施是用 `From` 搭建的：

```rust
// ? 运算符的魔法：
fn read_config() -> Result<Config, MyError> {
    let s = std::fs::read_to_string("config.toml")?;
    //                                         ^^
    // ? 展开为：
    // match std::fs::read_to_string("config.toml") {
    //     Ok(v) => v,
    //     Err(e) => return Err(MyError::from(e)),
    //                                     ^^^^
    //                                    From trait!
    // }

    Ok(toml::from_str(&s)?)
    //                   ^^ 同理
}

// 所以只要实现了 From<std::io::Error> for MyError，
// ? 运算符就能自动转换错误类型
```

这就是为什么 `thiserror` 和 `anyhow` 生态如此依赖 `From`——它是 Rust 错误处理的基石。
