# 1.13 `!` 永不返回类型——类型系统的"底"

> `()` 是"返回空"，`!` 是"不可能返回"。前者有一个值——`()` 本身；后者**没有值**——函数永远不会正常结束。这个差异不仅是对语义的精确编码，更是让 Rust 的类型推导系统能够"理解"不可达代码，让 `match` 表达式变得完整，让 `Result<T, !>` 成为零成本的 infallible 类型。`!` 很小，但它是 Rust 类型理论完备性的基石。

---

## 1. `!` vs `()` —— 根本区别

### 1.1 两者定义上的鸿沟

```rust
// () —— 单元类型（unit type）：有且仅有一个值
let x: () = ();  // ✅ () 类型的值就是 ()
// 函数"返回空"：
fn do_nothing() -> () {
    // 隐式返回 ()
}
// do_nothing 会正常结束并返回 ()

// ! —— 永不返回类型（never type）：没有任何值
// let x: ! = ???;   // ← 你无法构造一个 ! 类型的值！
//                     //   没有表达式能产生 ! 类型的值
fn never_returns() -> ! {
    panic!("I never come back");
    // 或者：
    // loop {}          ← 无限循环
    // std::process::exit(1)  ← 终止进程
}
// never_returns 永远不会"正常返回"——调用者等不到返回值
```

**直观理解**：

```rust
// 一个类型的"大小"可以用它有多少个值来衡量：
// bool       ：2 个值（true, false）
// u8         ：256 个值
// i32        ：2^32 个值
// ()         ：1 个值 —— ()
// !          ：0 个值 —— 不存在

// 这就是为什么 ! 被称为"底类型"（bottom type）：
// 在所有类型的"子类型"关系中，! 是"最小的"——它没有值
```

### 1.2 返回 `!` 的函数意味着什么

```rust
// 所有返回 ! 的表达式都是"发散"的——执行流不会从它们那里继续

fn divergent1() -> ! {
    panic!("boom");
    // panic! 本身返回 ! —— 它展开栈帧或终止进程，不会正常返回
}

fn divergent2() -> ! {
    loop {
        // 无限循环——永远不会退出
        std::thread::sleep(std::time::Duration::from_secs(1));
    }
}

fn divergent3() -> ! {
    std::process::exit(1);
    // 直接退出进程——不存在"返回"这个概念
}

fn divergent4() -> ! {
    // 所有匹配分支都返回 !
    match some_option() {
        Some(x) => loop { println!("{}", x); },
        None => panic!("no value"),
    }
}
```

## 2. `!` 的类型系统魔法——可以"转为"任何类型

### 2.1 核心规则：`!` 可以强制转换为任何类型

```rust
// 这是 ! 最强大的属性：它可以从任何位置被"当作"任何类型

fn always_panics() -> ! {
    panic!("never returns");
}

// ✅ ! 可以当作 i32：
let x: i32 = always_panics();
// 这里发生了什么？always_panics() 返回 !，但期望 i32
// 编译器说："! 可以被当作任何类型" — 因为这一行代码实际上永远执行不到

// ✅ ! 可以当作 String：
let y: String = always_panics();

// ✅ ! 可以当作 Vec<HashMap<String, Arc<Mutex<()>>>>：
let z: Vec<std::collections::HashMap<String, std::sync::Arc<std::sync::Mutex<()>>>> = always_panics();

// 原因：因为 always_panics() 不会返回，所以"返回值类型是什么"根本无关紧要
// 编译器利用这一点让 ! 回归到任何上下文中
```

### 2.2 `match` 与 `!`——不可达分支不需要返回值

```rust
// ! 让 match 表达式更灵活：

fn get_value() -> i32 { 42 }

let result: i32 = match get_value() {
    0 => 0,
    1 => 1,
    _ => panic!("unexpected value"),  // panic! 返回 !，但被当作 i32
    //                                // 因为 match 期望这个分支返回 i32
};

// 更实际的例子——从 Option 解包：
let value = match opt {
    Some(x) => x,
    None => panic!("expected Some"),  // ✅ ! 可以当作 x 的类型
    //     // 没有这行会编译错误：match 不是穷尽的
};

// 如果没有 !：
// None => ???   // 你无法写一个"不返回值"的分支
//               // 必须返回某种占位值（但这可能导致逻辑错误被隐藏）
```

### 2.3 `!` 参与类型推断

```rust
// ! 让一些看起来很奇怪的代码能够编译通过：

let x: i32 = loop {
    break 42;  // loop 表达式可以通过 break 返回一个值
};
assert_eq!(x, 42);
// 整个 loop {} 的类型是 break 后的值的类型

let forever: i32 = loop {
    // 没有 break —— loop 的类型被推断为 !
    // 但 ! 可以转为 i32，所以能赋值给 i32 变量
};
// (当然，这行代码永远不会执行完毕)
```

## 3. `Infallible` —— `!` 的稳定别名

### 3.1 为什么需要 `Infallible`

```rust
// ! 类型本身在 stable Rust 中不能直接使用：
// fn foo() -> ! { panic!() }  // ❌ stable 中不支持直接用 ! 写返回类型

// 但你可以用 std::convert::Infallible：
use std::convert::Infallible;

fn infallible_operation() -> Result<i32, Infallible> {
    Ok(42)
    // Infallible 表示"错误类型是不可能的"——这个操作不会失败
}

// Infallible 的定义：
pub enum Infallible {}
//          ^^ 空枚举——没有任何构造器
// 一个没有变体的枚举不可能被构造 → 等价于 !

// 所以：
// Result<T, Infallible> → "不可能出错" → 可以安全 unwrap
// Result<T, !>          → "不可能出错" → 同一个意思
```

### 3.2 `Infallible` 的实际应用

```rust
use std::convert::Infallible;
use std::str::FromStr;

// 对于总是成功的解析：
impl FromStr for AlwaysParsable {
    type Err = Infallible;
    fn from_str(s: &str) -> Result<Self, Infallible> {
        Ok(AlwaysParsable(s.to_string()))
    }
}

// TryFrom 也可以用 Infallible：
impl TryFrom<i32> for AlwaysParsable {
    type Error = Infallible;
    fn try_from(value: i32) -> Result<Self, Infallible> {
        Ok(AlwaysParsable(value.to_string()))
    }
}

// 由于 Infallible 从不被构造，你可以安全 unwrap：
let x: AlwaysParsable = "hello".parse().unwrap();
// unwrap 在 Result<T, Infallible> 上永远不 panic
// 因为 Err(Infallible) 不可能存在
```

### 3.3 `Infallible` 的 `Into` 转换

```rust
// 标准库为 Infallible 实现了：
impl<T> From<Infallible> for T {
    fn from(never: Infallible) -> T {
        match never {
            // 没有分支——因为 Infallible 没有变体
            // 这个 match 是"完备的"（exhaustive）——空匹配
        }
    }
}

// 这意味着：
// ✅ Infallible 可以转为任何类型
// let x: String = Infallible::from(???).into();  // 但实际上你无法创建 Infallible 值
```

## 4. 产生 `!` 的表达式

### 4.1 永不返回的内置宏

```rust
// 以下宏/函数都返回 !：

panic!("error message");        // → !  展开栈帧
todo!();                        // → !  等价于 panic!("not yet implemented")
unimplemented!();               // → !  等价于 panic!("not implemented")
unreachable!();                 // → !  告诉编译器"这行代码不可能被执行到"

// 编译期断言：
// compile_error!("message");   // 这不是 !——这是编译期错误
```

### 4.2 控制流转移

```rust
// return、break、continue 在类型系统中也被视为返回 !

fn process(flag: bool) -> i32 {
    let x = if flag {
        return 0;  // return 的类型是 !——它不会产生值
        //         // 但因为 ! 可以转为任何类型，可以放在 i32 的上下文中
    } else {
        42         // i32
    };
    // x 在这里一定是 42（flag = false 时）
    // 如果 flag = true，x = return 0——函数已经返回了
    x + 1
}

// 等价于：
fn process2(flag: bool) -> i32 {
    if flag { return 0; }
    42 + 1
}
```

### 4.3 无限循环

```rust
// 没有 break 的 loop 也是 !：
let forever: ! = loop {
    // 这个循环永不终止——\n    // 所以整个 loop 表达式的类型是 !
};

// 有 break 的 loop 则是 break 值的类型：
let x: i32 = loop {
    if condition() { break 42; }
    // break 42 的类型也是 !（break 不产生值）
    // 整个 loop 的类型是 i32（来自 break 42）
};
```

### 4.4 `continue` 也在类型层面的处理

```rust
// continue 和 break 的类型都是 !：
loop {
    let _x: i32 = if condition() {
        continue;  // continue → ! → 可以转为 i32
    } else {
        42
    };
    // ...
}
```

## 5. `!` 在泛型中的妙用

### 5.1 `Result<T, !>` —— 零成本的 infallible

```rust
// Result<T, !> 的大小只包含 T（而不是 T + 错误变体的判别）
// 因为 Err(!) 永远不可能被构造

// 这对于 infallible 操作非常有用：
fn always_succeeds() -> Result<i32, Infallible> {
    Ok(42)
}

// 可以安全地从 Result<T, Infallible> 提取 T：
let value = always_succeeds().unwrap();  // 永远不 panic
let value = always_succeeds().unwrap_or(0);  // 默认值从未使用
let value = always_succeeds().unwrap_or_else(|e| match e {});
//                                                      ^^ 空 match——e 是 Infallible

// ok_or 系列：
let x: Option<i32> = Some(42);
let r: Result<i32, Infallible> = x.ok_or(Infallible);
//                                    ^^^ 其实无法构造 Infallible
// 但 ok_or 的类型签名要求你传一个 Err 值——
// 由于 Infallible 不能构造，你只能在类型层面满足要求
```

### 5.2 `From<!> for T` —— 通用转换

```rust
// 标准库有：impl<T> From<!> for T
// 这意味着什么？

fn example() {
    // 假设你有一个函数返回 Result<T, E>
    // 其中 E 可以通过 ? 转换为外层函数错误类型
    fn inner() -> Result<i32, std::io::Error> {
        Ok(42)
    }

    fn outer() -> Result<i32, std::io::Error> {
        let x = inner()?;  // ? 用 From<io::Error> 转换
        Ok(x)
    }

    // 现在假设 inner 返回 Result<i32, Infallible>：
    fn inner_infallible() -> Result<i32, Infallible> {
        Ok(42)
    }

    fn outer_with_any_error<E: std::fmt::Debug>() -> Result<i32, E> {
        let x = inner_infallible()?;  // ? 用 From<Infallible> for E 转换
        // 而 From<Infallible> 对任何 E 都实现了！
        // → infallible 操作可以在任何错误类型中无缝使用 ?
        Ok(x)
    }
}
```

### 5.3 表达式类型统一

```rust
// ! 让 match/if 的分支类型统一变得灵活：

enum Command { Start, Stop, Restart }

fn handle(cmd: Command) -> usize {
    match cmd {
        Command::Start => 1,
        Command::Stop => 2,
        Command::Restart => std::process::exit(0),
        // exit 返回 ! → 可以被"当作"usize
        // 但注意：这行代码后面的所有代码都不会执行
    }
}

// 类似地：
fn get_port() -> u16 {
    match read_config() {
        Some(port) => port,
        None => {
            eprintln!("FATAL: no port configured");
            std::process::exit(1);
            // 编译器知道这里不会返回——不需要"假装"返回 u16
        }
    }
}
```

## 6. 常见陷阱

### 陷阱 1：`!` vs `()` 在函数签名中

```rust
// ❌ 错误：把发散函数标注为 -> ()
fn my_panic() -> () {
    panic!("error");  // panic! 返回 !，但函数签名说 ()
    // 编译器会接受这个吗？会！
    // 因为 ! 可以转为 ()——所以这段代码可以编译
    // 但签名 -> () 没有传达"这个函数不返回"的信息
}

// ✅ 更好：如果函数不会返回，用 -> !
fn my_panic_better() -> ! {
    panic!("error");
}
// 调用方可以通过类型签名知道这个函数不会返回
// 编译器也可以利用这个信息做更好的优化和死代码消除
```

### 陷阱 2：`Infallible` 不能直接替代 `!`

```rust
// Infallible 和 ! 有微妙区别：

// Infallible 是一个"正常"的类型（只是一个空枚举）
// ! 是一个"原生"类型（编译器内置，有特殊规则）

// 主要区别：
// 1. ! 可以隐式强制转换为任何类型（coercion）
//    Infallible 需要显式的 From/Into 转换
let x: i32 = panic!();  // ✅ ! 隐式转为 i32
// let x: i32 = return_infallible();  // ⚠️ 需要 From 转换而不是隐式 coercion

// 2. ! 可以用于返回类型（nightly）
fn never() -> ! { loop {} }  // nightly only
fn infallible() -> Infallible { loop {} }  // stable

// 3. ! 的类型推断更智能
let x = match opt {
    Some(v) => v,
    None => panic!("gone"),  // ! 隐式匹配
};
// 如果用 Infallible 替代可能需要更多标注
```

### 陷阱 3：`loop` 的类型会随 `break` 改变

```rust
// loop 的类型取决于 break 表达式：
let x = loop {
    break 42;  // loop 的类型是 i32
};
// x: i32

let y = loop {
    // 没有 break
};  // loop 的类型是 ! —— 但这个循环永远不结束
// y 的类型被推断为 !（但因为可以转为任何类型，赋值给什么都行）

let z: i32 = loop {
    if condition() { break 1; }
    // 这里有两条路径：一条 break 返回 i32，一条继续循环
    // 整个 loop 的类型是 i32
};
```

## 8. 练习答案

### 练习 1 答案

```rust
(a) ✅ 编译通过。exit() 返回 !，可以隐式转为 i32。
    当然，这一行实际上永远不会执行完毕。

(b) ✅ 编译通过。没有 break 的 loop 类型是 !，可以转为 i32。
    这也是无限循环。

(c) ❌ 不能编译（stable）。你不能声明 ! 类型的变量。
    nightly 中也需要特殊的 feature gate。

(d) ✅ 编译通过。panic!() 返回 !，! 可以转为 ()。
    注意：这个函数签名是 -> ()，但 panic 导致它根本不会执行完。

(e) ❌ 不能编译。return 的类型是 !，但 return 不能作为函数体的唯一表达式——
    函数体必须有一个 ! 类型的尾部表达式（如 loop {} 或 panic!()）
    空的 return; 实际上返回的是 ()，不是 !。

(f) ❌ 不能编译。loop { break; } 的类型是 ()（break 没有带值）
    但函数签名要求返回 !。
    实际上 loop { break; } 是一个只会执行一次的循环——等价于 ()。
```

### 练习 2 答案

```rust
// Result<T, Infallible> 的 unwrap 定义大致如下：
impl<T> Result<T, Infallible> {
    fn unwrap(self) -> T {
        match self {
            Ok(t) => t,
            Err(e) => match e {
                // Infallible 没有任何变体
                // → 这个分支是"不可达的"
                // → 编译器知道这里不需要返回 T
                // → 所以 unwrap 不需要 panic
            }
        }
    }
}

// 类型层面：
// Infallible 是空枚举 → 不存在 Infallible 类型的值
// → Err(Infallible) 不可能被构造
// → match 的 Err 分支不可能被触发
// → unwrap 是"安全的"——永远不会 fail

// 对于 !：同理，! 是无底类型（没有任何值）
// Result<T, !> 的 unwrap 也是安全的
```

### 练习 3 答案

```rust
use std::convert::Infallible;
use std::str::FromStr;

// 方案 1：定义一个包装类型，其 FromStr 的 Err = Infallible
struct AlwaysNumber(i32);

impl FromStr for AlwaysNumber {
    type Err = Infallible;
    fn from_str(s: &str) -> Result<Self, Infallible> {
        // 假设输入总是有效
        Ok(AlwaysNumber(s.parse().unwrap()))
    }
}

// 但这没有从根本上解决问题——还是用了 unwrap

// 方案 2：直接用类型系统表达"解析不会失败"
// 使用 ? 并利用 From<Infallible> for any Error：
fn parse_numbers_robust(input: &str) -> Result<Vec<i32>, Infallible> {
    input.split(',')
        .map(|s| s.trim().parse::<i32>())
        .collect::<Result<Vec<i32>, _>>()
}
// 如果 parse 返回 Err，? 会传播错误
// 如果所有解析都成功，返回 Ok

// 方案 3：用类型系统保证永远不会传入非法值
// 这就是 ! 的关键哲学——让不可能的情况在类型层面就被排除
```

### 练习 4 答案

```rust
// MyInfallible 和 std::convert::Infallible 定义相同（都是空枚举）
// 但它们不"等价"——它们是不同的类型

// 编译器不会自动为 MyInfallible 实现 From<MyInfallible> for T
// std::convert::Infallible 的特殊性来自标准库的显式实现：

// 标准库中：
impl<T> From<Infallible> for T {
    fn from(never: Infallible) -> T {
        match never {}  // 空 match
    }
}

// 你的 MyInfallible 没有这个实现——除非你自己写：
impl<T> From<MyInfallible> for T {
    fn from(never: MyInfallible) -> T {
        match never {}
    }
}

// 另一个区别：!（真的 never type）有编译器级别的隐式 coercion
// 而 Infallible 只是标准库中的普通类型——隐式转换是硬编码的编译器行为
```

## 9. 深度扩展

### 9.1 C++ 的 "无返回值" —— `[[noreturn]]`、`void` 和 `std::unreachable`

```cpp
// C++ 的"无返回"表示：多种机制，各有缺陷

// 方式 1：[[noreturn]] 属性（C++11）
[[noreturn]] void fatal_error(const char* msg) {
    std::cerr << msg << std::endl;
    std::abort();
}
// 问题：
// - [[noreturn]] 只是"属性"——不影响类型系统
// - 返回类型仍然是 void——类型签名不反映"永不返回"
// - 编译器用 [[noreturn]] 做警告/优化，但不能做类型层面的推理

// 方式 2：void 类型
void process() {
    if (error_condition) {
        fatal_error("something wrong");
        // fatal_error 是 [[noreturn]] void，但从类型系统看它"返回了 void"
        // 下面的代码在类型层面是可达的——编译器不会帮你发现死代码
    }
    // 这里继续执行
}

// 方式 3：C++23 std::unreachable()
if (value == 0) {
    std::unreachable();  // 标记不可达——但仍然是 void 类型
}
// 这仍然是"约定"而不是"类型保证"

// C++ 做不到的事：
// 1. 在类型层面表达"这个分支不会返回值"——void ≠ never
// 2. 让 match/if 的各分支类型统一时利用"无返回值"——C++ 没有 match 表达式
// 3. 实现 From<never> for T——C++ 没有类似的 trait 和类型转换系统
// 4. Result<T, never> 的零成本 infallible——C++ 的 std::expected<T, E>
//    即使 E 是"空类型"，仍然占用空间（因为 C++ 没有 niche 优化）
```

### 9.2 如果没有 `!`——`Result<T, !>` 只能用运行时方案替代

```rust
// 如果 Rust 没有 !，你如何表示 "infallible operation"？

// 方案 A：Result<T, ()>——语义错误
fn parse() -> Result<i32, ()> {
    Ok(42)
}
// 问题：Err(()) 是可能被构造的！
// 后续代码无法安全地 unwrap——因为 Err(()) 是一个真实的可能性

// 方案 B：自定义空错误类型 + 手动 From 实现
enum Void {}  // 空枚举

fn parse() -> Result<i32, Void> {
    Ok(42)
}

// 使用时：
let x = match parse() {
    Ok(v) => v,
    Err(e) => match e {},  // 空 match —— 手动写太啰嗦
};

// 方案 C：直接返回 T——但失去了 Result 的组合能力
fn parse() -> i32 {
    42
}
// 无法和 ? 操作符配合，无法统一错误处理流程

// ! 的存在让方案 B 变成了标准库的一部分（Infallible）
// 而且 From<Infallible> for T 让 ? 操作符无感处理 infallible
```

### 9.3 类型论视角：Bottom Type 和类型完备性

```
在形式类型论中，"底类型"（Bottom Type）⊥ 是每种类型的子类型。
这意味着 ⊥ 的值可以被用作任何类型——因为没有东西"是"⊥（没有 ⊥ 的值存在）。

Rust 的 ! 就是 ⊥。

类型层次（简化）：
        Any（概念上的"顶"类型，Rust 没有真正的 Top Type）
       /  |  \
    i32  String  Vec<u8>  ...
       \  |  /
        !（底类型——所有类型的子类型）

! 的语义规则：
1. ! ≤ T 对于所有类型 T（! 是任何 T 的子类型）
2. 如果 e: !，则 e 可以用在任何期望 T 的位置
3. ! 没有构造函数——不存在类型为 ! 的表达式（除了"发散表达式"）

为什么需要底类型？
- match 分支类型统一：if true { 42 } else { panic!() }
  第二个分支不产生值，但 match 要求所有分支类型兼容
  ! 可以被"当作"i32 使用 → 分支类型统一

- 类型推断：let x = if cond { 1 } else { loop {} };
  编译器需要 x 的类型——loop {} 的类型是 !，但 ! ≤ i32
  → 推断 x: i32

- 穷尽性检查：match 的所有分支必须覆盖所有情况
  ! 的分支意味着"不会从这里返回"——不需要提供值
```

### 9.4 `!` 的稳定化之路

```
Rust 的 ! 类型稳定化是一个漫长的故事：

2016: 内部引入 never type 作为编译器内部类型
2018: #![feature(never_type)] —— 实验性 feature gate
      fn foo() -> ! { loop {} }  只能在 nightly 使用

2018-2024: ! 的稳定化被多次延迟
      原因：! 和 Infallible 的交互、fallback 行为、
      隐式 coercion 的精确语义都需要仔细设计

当前状态（2024+）：
      - fn foo() -> ! 仍然 nightly-only
      - 但 ! 在 match 分支、loop {} 等位置隐式工作
      - Infallible 是 stable 的替代方案
      - 标准库正在逐步引入 ! 的支持

未来的 ! 稳定化计划：
      1. ! 可以出现在返回类型（最优先）
      2. ! 在泛型位置（如 Result<T, !>）
      3. 更精细的隐式转换规则
```

### 9.5 `Infallible` 和 `!` 的 "Niche 优化"

```rust
// Rust 的枚举布局优化可以"借用"不可能的值来节省空间

// Option<bool> 的大小：1 字节
// bool 是 1 字节（0 或 1），Option 用 2-255 的其他值表示 None
// 这就是 "Niche 优化"

// Infallible 参与 Niche 优化：
use std::mem::size_of;

// Result<i32, Infallible> 的大小 = i32 的大小（4 字节）
// 因为 Err 永远不会发生，编译器不需要为 Err 分配空间
assert_eq!(size_of::<Result<i32, Infallible>>(), size_of::<i32>());

// 对比 Result<i32, bool>：
assert_eq!(size_of::<Result<i32, bool>>(), 8);  // i32 + bool + 对齐

// Result<T, !>（nightly）同理 —— 0 开销的 infallible
// 这是 C++ 的 std::expected<T, E> 做不到的——
// C++ 没有 Niche 优化，std::expected<T, monostate> 仍然比 T 大
```

fn example(flag: bool) -> i32 {
    if flag {
        return 42;
    }
    // 编译器知道：如果 flag = true，已经 return 了
    // → 到达这里的只有 flag = false 的情况

    // panic!("unreachable");
    // 编译器知道这行代码永远不会执行
    // → 可以消除整个分支

fn cascade() -> i32 {

### 9.7 `!` 与 async 和 Future 的交互

```rust
// async 函数中的 !：

async fn fetch_or_panic(url: &str) -> String {
    match reqwest::get(url).await {
        Ok(resp) => resp.text().await.unwrap_or_else(|_| panic!("invalid text")),
        Err(e) => {
            eprintln!("fatal: {}", e);
            panic!("cannot fetch");
            // panic! 返回 ! —— 在这个 async 块中它被当作 String
        }
    }
}

// 等价于 Future 的 poll 返回 Poll<Result<String, Infallible>> 在某些路径上
// Infallible 优化让 Future 状态机更小
```

### 9.8 `!` 在设计模式中的应用——"永远不会"的类型表达

```rust
// 实际应用 1：不可能发生的错误
fn connect() -> Result<Connection, Infallible> {
    // 这个 API 的调用方知道 connect 不会返回 Err
    Ok(Connection::new())
}
// 类型签名就是文档——"这个函数不会失败"

// 实际应用 2：必须被覆盖的 match 分支
enum Event { Click, KeyPress(char), Resize(u32, u32) }

fn handle(event: Event) -> String {
    match event {
        Event::Click => "clicked".into(),
        Event::KeyPress(c) => format!("key: {}", c),
        Event::Resize(w, h) => format!("resized: {}x{}", w, h),
        // 如果添加新的 Event 变体，编译器会报错——match 不再穷尽
        // 不需要 default 分支或 unreachable!()
    }
}

// 实际应用 3：类型状态机的"死胡同"
struct DeadEnd;
impl Connection<DeadEnd> {
    fn this_can_never_be_called(&self) -> ! {
        // 这个方法永远不应该被调用
        // 类型系统保证了 DeadEnd 状态不可达
        unreachable!("DeadEnd state should never be constructed")
    }
}
```

## 10. 回顾

```
! 永不返回类型学习清单：

□ 理解 !（0 个值）和 ()（1 个值）的本质区别
□ 知道 ! 可以隐式强制转换为任何类型（底类型的属性）
□ 理解 panic!、loop{}、exit()、return、break、continue 都"返回" !
□ 掌握 Infallible 作为 ! 的 stable 替代——空枚举
□ 知道 Result<T, Infallible> 是零成本的 infallible 类型
□ 理解 From<Infallible> for T 让 infallible 操作无缝集成到 ? 链中
□ 了解 ! 在类型推断和死代码消除中的作用
□ 知道 C++ 的 [[noreturn]] 和 void 无法在类型层面表达"永不返回"
```

> **一句话总结**：`!`（never type）是 Rust 类型系统的"底"——它没有值，因此可以被当作任何类型使用。这让 `match` 的不可达分支不需要返回值，让 `Result<T, !>` 成为零成本的 infallible 类型，让编译器能精确地消除死代码。C++ 的 `[[noreturn]]` 只是编译提示（不影响类型系统），而 Rust 的 `!` 是真正融入类型推断的一等公民——类型论里的"底类型"在系统编程语言中的具体实现。
