# 1.1 Sized / ?Sized / DST / 胖指针 / vtable

> 理解"类型的大小"是一切 Rust 内存模型的基础。这一课回答一个本质问题：
> **编译器在编译时，凭什么知道一个类型占多少字节？当它不知道的时候怎么办？**

---

## 1. 从一个反直觉的问题开始

先看一段完全合法、你觉得理所当然的代码：

```rust
let x: i32 = 42;
let y: String = String::from("hello");
let z: Vec<u64> = vec![1, 2, 3];
```

你从来没想过这个问题：**编译器怎么知道要为 `x`、`y`、`z` 分配多少栈空间？**

答案是：`i32` 总是 4 字节，`String` 总是 24 字节（指针 + 容量 + 长度），`Vec<u64>` 也是 24 字节。编译时就知道——这些类型都是 **Sized**。

那你觉得下面这段代码为什么编译**不通过**？

```rust
let s: str = "hello";  // ❌ error[E0277]: the size for values of type `str`
                        //    cannot be known at compilation time
```

`"hello"` 总共 5 个字节的 UTF-8 数据，有什么"不知道"的？

**关键洞察**：`str` 本身不携带长度信息。编译器看到类型 `str` 时，只知道"这是一段 UTF-8 字节序列"，但不知道多长。长度是运行时信息，存在胖指针里。

```rust
// 这才是对的：
let s: &str = "hello";  // &str 是 Sized（16 字节：指针 + 长度）
```

这就是本课的核心：**Sized 判定 + 胖指针补偿 = Rust 在不牺牲灵活性的前提下保证编译时知道所有栈上数据的大小。**

---

## 2. Sized：Rust 最"安静"的 trait

### 2.1 它是 auto trait

```rust
// 你从来没手动写过，但每个具体类型都自动实现了 Sized
// 编译器内部等价于：
impl Sized for i32 {}       // 自动
impl Sized for String {}    // 自动
impl Sized for &str {}      // 自动（胖指针大小固定）
impl Sized for Box<dyn Fn()> {}  // 自动（Box 大小固定）
```

Sized 是一个 **auto trait**——编译器自动为所有"编译时大小已知"的类型实现它。你不能也没必要手动实现。

### 2.2 它是隐式默认约束

```rust
// 你写的：
fn foo<T>(x: T) { }

// 编译器实际看到的：
fn foo<T: Sized>(x: T) { }  // T 默认要求 Sized
```

**Rust 的设计选择**：泛型参数默认要求 `Sized`。为什么？

| 如果默认 Sized | 如果默认不 Sized |
|:---|:---|
| 参数可以直接放栈上 `fn foo(x: T)` | 参数必须通过引用 `fn foo(x: &T)` |
| 写法简洁，符合直觉 | 每次都要手动标注 |
| 特殊情况显式放宽 `T: ?Sized` | 绝大多数情况被迫写 `T: Sized` |

Rust 选择"大多数情况简洁，特殊情况显式"——这是整个语言设计的核心哲学。

```rust
// 这才是完整的写法——?Sized 显式解除默认约束
fn foo<T: ?Sized>(x: &T) { }  // T 可以是 str, [u8], dyn Trait...

// 等价写法（推荐）：
fn foo<T>(x: &T) { }  // &T 本身是 Sized 的，T 不需要是 Sized
                      // 因为此处 T 在 & 后面，约束自动放宽
```

### 2.3 编译器到底凭什么判断 Sized？

一个类型是 Sized 的，当且仅当它满足以下递归规则：

```
1. 所有原始类型是 Sized：i8 ~ i128, u8 ~ u128, f32, f64, bool, char
2. 所有引用是 Sized：&T, &mut T（它们是胖指针或瘦指针，大小固定）
3. 所有裸指针是 Sized：*const T, *mut T
4. fn 指针是 Sized：fn(i32) -> bool（8 字节）
5. 数组是 Sized：[T; N]（N * size_of::<T>()）
6. 结构体/枚举是 Sized：当且仅当所有字段都是 Sized
7. 元组是 Sized：当且仅当所有元素都是 Sized
8. Box<T> / Rc<T> / Arc<T> / Vec<T> / String 是 Sized（它们内部有堆指针，自身大小固定）
```

反过来说，**只有以下类型不是 Sized**：

| 类型 | 为什么不是 Sized |
|:---|:---|
| `str` | "一串 UTF-8 字节"，长度未知 |
| `[T]` | "一串 T"，长度未知 |
| `dyn Trait` | "某个实现了 Trait 的类型"，具体类型未知 |
| 包含 `?Sized` 字段的结构体 | 递归：字段不 Sized 则结构体也不 Sized |

## 3. ?Sized：不是"可选的 Sized"，是"放宽约束"

### 3.1 语法含义

`?Sized` 不是"Maybe Sized"——它读作 **"不要求 Sized"** 或 **"放宽 Sized 约束"**。

```rust
// "T 不一定需要是 Sized"
fn print_debug<T: ?Sized + Debug>(x: &T) {
    println!("{:?}", x);
}
```

`?` 在 trait bound 中特指"放宽"，类似于：
- `?Sized`：不要求 Sized
- `?Send`：不要求 Send（极少用）
- `?Sync`：不要求 Sync（极少用）

### 3.2 只能在特定位置使用

```rust
// ❌ 不能直接作为局部变量类型
fn bad() {
    let x: dyn Display = ...;  // 栈上变量必须 Sized
}

// ✅ 可以放在引用/指针后面
fn good() {
    let x: &dyn Display = &42;         // 胖指针（data ptr + vtable ptr）
    let y: Box<dyn Display> = Box::new(42);  // 堆上分配
    let z: Rc<dyn Display> = Rc::new(42);    // 引用计数包装
}
```

**Rust 的硬性约束**：栈上变量必须在编译时知道大小，因为编译器要计算栈帧布局。DST 只能通过一层间接访问（引用、Box、Rc 等），这些包装类型本身是 Sized 的。

### 3.3 结构体中的 ?Sized 字段

```rust
// 结构体最后一个字段可以是 ?Sized
struct MyWrapper<T: ?Sized> {
    header: u32,
    payload: T,  // 必须是最后一个字段
}

// 用法：
let sized_version: MyWrapper<String> = MyWrapper { header: 1, payload: "hi".into() };
// MyWrapper<String> 是 Sized 的（24 + 4 = 28，对齐到 32 字节）

let unsized_version: &MyWrapper<str> = ...;  
// MyWrapper<str> 是 !Sized，必须通过引用访问
// &MyWrapper<str> 是胖指针：{ ptr, len_of_payload }
```

**为什么必须是最后一个字段？** 因为对于 DST 变体，编译器无法知道 `payload` 后面还有什么——字段偏移量无法确定。只有最后一个字段可以是不定长的。

标准库中的例子：

```rust
// std::path::Path 就是这样一个包装——
// 它本质上是一个 newtype over OsStr，后者是 DST
pub struct Path {
    inner: OsStr,  // OsStr 是 DST，Path 也是 DST
}

// 所以 Path 总是通过引用使用：
let p: &Path = Path::new("/usr/bin/rustc");
```

## 4. DST：动态大小类型

### 4.1 三种 DST 的完整对比

```rust
use std::mem::size_of;

// ============ str ============
let s1: &str = "hello";           // 胖指针: { ptr, len=5 }
let s2: &str = "hello world";    // 胖指针: { ptr, len=11 }
// &str 本身都是 16 字节（64位平台），但指向的数据长度不同

// ============ [T] ============
let a1: &[i32] = &[1, 2, 3];     // 胖指针: { ptr, len=3 }
let a2: &[i32] = &[1, 2, 3, 4, 5]; // 胖指针: { ptr, len=5 }
// &[i32] 本身都是 16 字节

// ============ dyn Trait ============
trait Animal { fn speak(&self) -> &str; }
struct Dog;
impl Animal for Dog { fn speak(&self) -> &str { "woof" } }
struct Cat;
impl Animal for Cat { fn speak(&self) -> &str { "meow" } }

let d: &dyn Animal = &Dog;  // 胖指针: { data_ptr, vtable_ptr }
let c: &dyn Animal = &Cat;  // 胖指针: { data_ptr, vtable_ptr }
// &dyn Animal 本身都是 16 字节
```

**三种 DST 的胖指针结构**：

```
  &str / &[T]                     &dyn Trait
┌──────────────────────┐    ┌──────────────────────┐
│  data pointer  (8B)  │    │  data pointer  (8B)  │
│  length        (8B)  │    │  vtable pointer (8B) │
└──────────────────────┘    └──────────────────────┘
      16 字节                      16 字节
```

### 4.2 胖指针的"元数据"

胖指针 = **瘦指针 + 元数据**。元数据的内容取决于 DST 类型：

| DST | 元数据 | 胖指针大小 | 用途 |
|:---|:---|:---:|:---|
| `str` | `usize`（字节长度） | 16 B | 边界检查、遍历 |
| `[T]` | `usize`（元素个数） | 16 B | 边界检查、遍历 |
| `dyn Trait` | `&'static VTable` | 16 B | 方法分发、drop、size/align |
| 自定义 DST | 由尾部字段决定 | — | `std::ptr::metadata()` |

### 4.3 你什么时候会创建自定义 DST？

不常见，但理解它帮你看清全貌：

```rust
// 一个 ZST header + 一个 [u8] payload
// Packet<[u8]> 自身就是 DST
#[repr(C)]
struct Packet<T: ?Sized> {
    version: u8,
    flags: u8,
    payload: T,  // 最后一个字段是 DST
}

// Sized 版本：payload 编译时确定
let p1: Packet<[u8; 4]> = Packet {
    version: 1,
    flags: 0,
    payload: [10, 20, 30, 40],
};
// Packet<[u8; 4]> 是 Sized

// DST 版本：payload 可以任意长度
fn process(raw: &[u8]) -> &Packet<[u8]> {
    // unsafe: 从原始字节构造 DST 胖指针
    let header_size = std::mem::size_of::<Packet<()>>();  // 2 字节
    let payload_len = raw.len() - header_size;
    
    // 构造胖指针：{ ptr, metadata = payload_len }
    unsafe {
        std::ptr::from_raw_parts(
            raw.as_ptr() as *const (),
            payload_len,  // DST 的元数据
        )
    }
}

let data = [1u8, 0, 10, 20, 30, 40, 50];  // version=1, flags=0, payload=10,20,30,40,50
let packet: &Packet<[u8]> = process(&data);
assert_eq!(packet.version, 1);
assert_eq!(packet.payload.len(), 5);
```

`from_raw_parts` 让我们可以从零开始构造一个胖指针，但在日常开发中更常见的操作是**读取**已有胖指针的元数据。`std::ptr::metadata`（目前为 nightly 功能）正是 `from_raw_parts` 的反向操作：它接收一个胖指针引用，返回其中的"额外部分"——对于切片来说是长度，对于 trait object 来说是虚表指针。下面的例子展示了不同类型 DST 的元数据读取：

```rust
// 自定义 DST 的元数据获取（nightly）
#![feature(ptr_metadata)]
use std::ptr::metadata;

let arr: &[i32] = &[1, 2, 3];
let meta: usize = metadata(arr);  // = 3（长度）

let s: &str = "hello";
let meta: usize = metadata(s);   // = 5（字节长度）
```

## 5. vtable：动态分发的引擎

### 5.1 vtable 里面到底有什么？

```rust
trait Animal {
    fn speak(&self) -> String;
    fn eat(&self, food: &str);
}

// 编译器为 dyn Animal 生成的 vtable 结构大致是：
struct AnimalVTable {
    // 布局信息（每个 vtable 都有）
    drop_in_place: unsafe fn(*mut ()),  // 析构函数
    size: usize,                         // 具体类型的大小
    align: usize,                        // 具体类型的对齐
    
    // 方法指针（按 trait 方法声明顺序）
    speak: fn(&Dog) -> String,  // 注：实际签名是 fn(*mut ()) -> ...
    eat: fn(&Dog, &str),        //    用类型擦除的裸指针
}
```

验证 vtable 中存在 size/align：

```rust
use std::mem;

trait Foo {}
impl Foo for i32 {}
impl Foo for String {}

// dyn Foo 是 16 字节（data ptr + vtable ptr）
assert_eq!(mem::size_of::<&dyn Foo>(), 16);

// 但你不能直接访问 vtable——它是编译器内部实现细节
// 间接验证：通过 transmute 看看 size 是否正确
// （实际代码中不要这么做，仅供理解）
```

### 5.2 动态分发的过程

```rust
trait Animal {
    fn speak(&self) -> &str;
}

struct Dog;
impl Animal for Dog {
    fn speak(&self) -> &str { "woof" }
}

struct Cat;
impl Animal for Cat {
    fn speak(&self) -> &str { "meow" }
}

fn make_sound(animals: &[&dyn Animal]) {
    for animal in animals {
        // 这里发生了什么？
        println!("{}", animal.speak());
    }
}

fn main() {
    let dog = Dog;
    let cat = Cat;
    make_sound(&[&dog, &cat]);
}
```

`animal.speak()` 的底层执行流程：

```
1. animal 是 &dyn Animal（胖指针）
   ┌──────────────┐
   │ data_ptr ────┼──→ Dog 或 Cat 实例
   │ vtable_ptr ──┼──→ DogVTable 或 CatVTable
   └──────────────┘

2. 从 vtable 的固定偏移处读取 speak 的函数指针
   vtable.speak = Dog::speak  // 或 Cat::speak

3. 调用该函数指针，传入 data_ptr
   (vtable.speak)(data_ptr)
```

**关键效率代价**：

| | 静态分发 (`impl Animal`) | 动态分发 (`dyn Animal`) |
|:---|:---|:---|
| 调用开销 | 直接调用（可内联） | 两次间接：读 vtable → 跳转到函数指针（无法内联） |
| 二进制大小 | 每个具体类型生成一份代码（膨胀） | 一份代码处理所有类型（紧凑） |
| 单态化 | 是 | 否 |

### 5.3 vtable 是静态常量，不需要分配

很多人以为 vtable 在堆上——**不是**。vtable 是编译时生成的**静态常量**，放在只读数据段（`.rodata`）。

```rust
// 这个 vtable 在编译时就确定了，存在 .rodata 段
let animal: &dyn Animal = &Dog;
// vtable_ptr 指向这个静态常量：
//   static DOG_VTABLE: AnimalVTable = AnimalVTable {
//       drop_in_place: <Dog as Drop>::drop,  // 或默认的释放
//       size: 0,   // Dog 是 ZST
//       align: 1,
//       speak: Dog::speak,
//   };
```

### 5.4 什么时候用 dyn Trait vs impl Trait vs 泛型？

| 场景 | 推荐 | 原因 |
|:---|:---|:---|
| 编译时已知具体类型 | `fn foo<T: Trait>(x: T)` | 静态分发，零开销 |
| 运行时多种类型混合 | `fn foo(x: &dyn Trait)` | 同质容器 |
| 函数内部只用一次 | `fn foo(x: impl Trait)` | 语法糖，等于泛型 |
| 需要存储不同类型 | `Vec<Box<dyn Trait>>` | 异质集合 |
| trait 不是对象安全的 | 只能用泛型 | `Clone`、带 `Self: Sized` 的方法 |

```rust
// 场景 1：异质集合——必须用 dyn
trait UiWidget { fn render(&self); }
struct Button;
struct TextBox;
struct Slider;
impl UiWidget for Button { fn render(&self) {} }
impl UiWidget for TextBox { fn render(&self) {} }
impl UiWidget for Slider { fn render(&self) {} }

let widgets: Vec<Box<dyn UiWidget>> = vec![
    Box::new(Button),
    Box::new(TextBox),
    Box::new(Slider),
];
// 三种不同类型存在同一个 Vec 里——只有 dyn Trait 能做到

// 场景 2：性能敏感路径——用泛型
fn process<T: UiWidget>(widget: &T) {
    widget.render();  // 编译时确定调用哪个 render，可能内联
}

// 场景 3：需要 Clone——不能用 dyn（Clone 不是对象安全的）
// ❌ let x: Box<dyn Clone> = Box::new(42);  // 编译错误
// ✅ fn foo<T: Clone>(x: T) { x.clone(); }
```

## 6. 设计哲学：Rust 为什么这样设计？

### 6.1 "默认 Sized" 的权衡

```
决策：泛型参数默认要求 Sized
----------------------------------------------------------------
好处：
  ✅ 绝大多数代码更简洁，少写 T: Sized
  ✅ 函数参数可以直接放在栈上，不需要间接访问
  ✅ 编译器能更好地优化（静态分发、内联）
  ✅ 与 C/C++ 程序员的直觉一致

坏处：
  ❌ 需要动态大小类型时必须显式写 T: ?Sized
  ❌ 新手容易困惑为什么 fn foo(x: str) 不行
----------------------------------------------------------------
Rust 的选择：默认优化最常用路径，特殊情况显式标注。
这条哲学贯穿整个语言设计。
```

### 6.2 "DST 必须间接访问" 的权衡

```
决策：DST 不能直接作为栈上变量，必须通过引用/指针/智能指针
----------------------------------------------------------------
如果允许栈上 DST（像某些语言的可变长数组 VLA）：

好处：
  ✅ 写法更灵活

坏处：
  ❌ 编译器无法确定栈帧大小 → 无法生成高效的函数序言/尾声
  ❌ 栈溢出风险难以静态分析
  ❌ 不能 move DST 变量（大小未知，无法 memcpy）
  ❌ ABI 变得异常复杂
----------------------------------------------------------------
Rust 的选择：牺牲一点灵活性，换取编译时的确定性。
这也使得 Rust 的调用约定保持简洁，FFI 友好。
```

### 6.3 "胖指针" 这个设计的优雅之处

对比其他语言的方案：

| 语言 | str/[T] 的实现 | 代价 |
|:---|:---|:---|
| **C** | `char*` + 手动跟踪长度 / `\0` 终止 | 缓冲区溢出、O(n) 取长度、不安全 |
| **C++** | `std::string`（堆分配 + 长度字段）| 总是堆分配，无法引用子串 |
| **Go** | slice header（运行时结构体） | 运行时开销，GC 压力 |
| **Rust** | `&str` 胖指针（编译时类型系统） | 胖指针语义需要学习 |

Rust 的胖指针 = 把运行时元数据提升到类型系统层面，编译时就能做边界检查。

```rust
// C 的问题：
// char *s = "hello";
// s[100] = 'x';  // 编译通过！运行时才崩溃（甚至不崩溃）

// Rust 的方案：
let s: &str = "hello";
// s.as_bytes()[100];  // ❌ 编译时：index out of bounds
//                      // 或者运行时 panic 带清晰错误信息
```

### 6.4 trait 对象 vs C++ 虚函数

```
              Rust dyn Trait              |         C++ virtual
──────────────────────────────────────────┼──────────────────────────
vtable 存储    方法 + drop + size + align   |   仅方法（可能有 RTTI）
胖/瘦指针      胖指针（data + vtable）       |   瘦指针（vtable 存在对象内部）
多重 trait    不支持（只能 dyn 一个 trait）  |   支持多重继承
包装方式      &dyn, Box<dyn>, Arc<dyn>     |   裸指针或智能指针
分离关注点    trait 对象和数据指针分离       |   vtable 指针嵌入对象
```

**Rust 的设计选择**：把 vtable 从对象中**剥离**。这意味着：

```rust
// Rust：一个普通 struct 不携带 vtable
struct Point { x: f64, y: f64 }
// Point 大小 = 16 字节（两个 f64），没有额外开销

// 只有当你创建 &dyn Trait 时，才生成胖指针（额外的 8 字节 vtable 指针）
let p: &dyn Debug = &Point { x: 1.0, y: 2.0 };
// 但原始 Point 不受影响

// C++：如果有虚函数，每个对象都携带 vtable 指针（+8 字节）
// class Point { virtual ... };  → sizeof(Point) = 24
```

这是 Rust "零成本抽象" 的又一个例子：**不为你不用的特性付费**。

## 7. 举一反三：通用模式提取

### 模式 1：编译时已知 → 编译时未知的转换

`Sized → ?Sized` 的模式在 Rust 中很常见：

```rust
// String → str
let owned: String = "hello".to_string();
let borrowed: &str = &owned;        // &String → &str（Deref 强制转换）

// Vec<T> → &[T]
let owned: Vec<i32> = vec![1, 2, 3];
let borrowed: &[i32] = &owned;      // &Vec<i32> → &[i32]

// Box<T> → &T
let owned: Box<i32> = Box::new(42);
let borrowed: &i32 = &owned;        // &Box<i32> → &i32

// PathBuf → Path
let owned: PathBuf = PathBuf::from("/usr/bin");
let borrowed: &Path = &owned;       // &PathBuf → &Path
```

这就是 Rust 的 **"拥有-借用" 对偶**模式：一个 Sized 的拥有者 + 一个对应的 DST 借用形式。

### 模式 2：?Sized 不意味着"总是 DST"

```rust
trait MyTrait {}

fn flexible<T: ?Sized + MyTrait>(x: &T) {
    // T 可以是 Sized（如 i32）也可以是 !Sized（如 dyn MyTrait）
}

// 两种用法都合法：
flexible(&42i32);                    // T = i32（Sized）
flexible(&42i32 as &dyn MyTrait);    // T = dyn MyTrait（!Sized）
```

`?Sized` 放宽约束，允许更多类型——它不是"要求 DST"，而是"不拒绝 DST"。

### 模式 3：胖指针是 Sized 的

```rust
use std::mem::size_of;

// 不管指向什么，指针本身的大小是固定的
assert_eq!(size_of::<&str>(), 16);           // 64 位平台
assert_eq!(size_of::<&[u8]>(), 16);
assert_eq!(size_of::<&dyn std::fmt::Display>(), 16);
assert_eq!(size_of::<Box<str>>(), 16);       // Box<str> = 胖指针
assert_eq!(size_of::<Box<[u8]>>(), 16);
assert_eq!(size_of::<Box<dyn std::fmt::Display>>(), 16);

// 瘦指针对比
assert_eq!(size_of::<&i32>(), 8);            // 普通引用 = 瘦指针
assert_eq!(size_of::<Box<i32>>(), 8);        // Box<i32> = 瘦指针
```

这个性质使得你可以：
- 把 `Box<dyn Trait>` 放在 `Vec` 里（每个元素都是 16 字节）
- 在函数间传递时只是拷贝两个 `usize`

## 8. 常见陷阱

### 陷阱 1：`impl Trait` ≠ `dyn Trait`

```rust
// 完全不同！
fn by_impl(x: &impl Display) { }   // 泛型语法糖，编译时单态化
fn by_dyn(x: &dyn Display) { }     // 动态分发，运行时查 vtable

// by_impl 等价于：
// fn by_impl<T: Display>(x: &T) { }
```

### 陷阱 2：`Box<dyn Trait>` 不能自动转成 `&dyn Trait`

```rust
// ✅ Box<T> → &T（Deref）
let b: Box<i32> = Box::new(42);
let r: &i32 = &b;

// ❌ Box<dyn Trait> → &dyn Trait 不是自动的
let b: Box<dyn Display> = Box::new(42);
// let r: &dyn Display = &b;  // &b 的类型是 &Box<dyn Display>，不是 &dyn Display

// ✅ 正确做法：
let r: &dyn Display = &*b;  // 先解引用得到 dyn Display，再取引用
```

### 陷阱 3：DST 不能作为函数返回值（裸的）

```rust
// ❌ 不能
fn bad() -> str { "hello" }       // str 不是 Sized

// ❌ 也不能
fn bad() -> dyn Display { 42 }    // dyn Display 不是 Sized

// ✅ 必须包装
fn good() -> &'static str { "hello" }
fn good() -> Box<dyn Display> { Box::new(42) }
```

## 10. 深度扩展

### 10.1 Unsized Coercion（DST 强制转换）—— 你不知道的隐式转换

Rust 有一类隐式转换是很多人没意识到的——**从 Sized 引用到 DST 引用的自动转换**。

```rust
// 你每天都在用但可能没意识到的转换：

// [T; 3] → [T]（固定数组 → 切片）
let arr: [i32; 3] = [1, 2, 3];
let slice: &[i32] = &arr;    // &[i32; 3] 自动转成 &[i32]
//                    ^^^^ 这是 unsized coercion

// String → str
let s: String = "hello".into();
let s_ref: &str = &s;        // &String 自动转成 &str

// 它甚至支持多重包装：
let b: Box<[i32; 3]> = Box::new([1, 2, 3]);
let b_slice: Box<[i32]> = b; // Box<[i32; 3]> 自动转成 Box<[i32]>
//                         ^ 注意：这是所有权转移，不是引用！
```

**Unsized Coercion 的规则**：

```
转换发生在以下类型之间：
  &T     → &U        当 T: Unsize<U>
  &mut T → &mut U    当 T: Unsize<U>
  Box<T> → Box<U>    当 T: Unsize<U>
  *const T → *const U  当 T: Unsize<U>
  *mut T → *mut U      当 T: Unsize<U>
  Rc<T> → Rc<U>       当 T: Unsize<U>
  Arc<T> → Arc<U>     当 T: Unsize<U>
```

`Unsize<U>` 这个 trait 是编译器内部实现，你无法手动实现。只有这些转换是合法的：

| T | U | 说明 |
|:---|:---|:---|
| `[T; N]` | `[T]` | 固定数组 → 切片 |
| `T`（具体类型） | `dyn Trait`（T 实现了 Trait） | 具体类型 → trait 对象 |
| `Foo<..., T, ...>` | `Foo<..., U, ...>`（T: `Unsize<U>`） | 结构体的最后一个字段 |
| `(..., T)` | `(..., U)`（T: `Unsize<U>`） | 元组的最后一个元素 |

**这个机制为什么重要？** 它使得 Rust 在保持严格类型系统的前提下，仍然有流畅的人机工程学：

```rust
// 没有 unsized coercion 的世界（想像一下）：
let s: String = "hello".into();
// let s_ref: &str = &s;  // ❌ 类型不匹配！
let s_ref: &str = &s as &str;  // 必须手动转换——恶心

// 没有 unsized coercion 的世界（函数签名必须这样写）：
fn takes_slice(arr: &[i32; 3]) { }  // 只能接受固定长度！

// 现实世界中（有 unsized coercion）：
fn takes_slice(arr: &[i32]) { }
takes_slice(&[1, 2, 3]);    // &[i32; 3] → &[i32]，自动转换
takes_slice(&[1, 2, 3, 4]); // &[i32; 4] → &[i32]，自动转换
```

### 10.2 `Sized` 在 trait 中的特殊角色 —— `Self: Sized` 技巧

这是一个极其重要的设计模式——在 trait 中，你可以把某些方法标记为要求 `Self: Sized`：

```rust
trait Animal {
    fn speak(&self) -> &str;

    // 这个方法要求 Self: Sized——什么意思？
    fn into_owned(self) -> Box<dyn Animal>
    where
        Self: Sized,
    {
        Box::new(self)
    }
}

struct Dog;
impl Animal for Dog {
    fn speak(&self) -> &str { "woof" }
    // into_owned 有默认实现，不需要手动写
}

// Sized 版本：
let dog = Dog;
let owned: Box<dyn Animal> = dog.into_owned();  // ✅ Dog 是 Sized

// DST 版本：
let dog_ref: &dyn Animal = &Dog;
// dog_ref.into_owned();  // ❌ dyn Animal 不是 Sized，所以这个方法不可用！
```

**为什么需要这个技巧？** 因为 `self`（按值）的类型是 Self，而 Self 默认要求 Sized。如果 Self 是 `dyn Animal`，`self` 就无法放在栈上。

```rust
// 这个技巧在标准库中广泛使用：
// std::clone::Clone
pub trait Clone: Sized {  // Clone 要求 Self: Sized
    fn clone(&self) -> Self;
}

// 这就是为什么 Clone 不是对象安全的：
// let x: Box<dyn Clone> = Box::new(42);  // ❌ 编译错误
// 因为 Clone::clone 返回 Self，而 dyn Clone 中的 Self 不是 Sized

// 另一个变体：用 where Self: Sized 限制特定方法
trait Iterator {
    type Item;
    fn next(&mut self) -> Option<Self::Item>;

    // collect 需要知道输出类型的大小，所以要求 Self: Sized
    fn collect<B: FromIterator<Self::Item>>(self) -> B
    where
        Self: Sized,  // ← 就是这个
    { ... }
}

// 所以你不能在 dyn Iterator 上调用 collect
let iter: &mut dyn Iterator<Item = i32> = &mut vec![1, 2, 3].into_iter();
// iter.collect::<Vec<_>>();  // ❌ dyn Iterator 不是 Sized
```

### 10.3 `DispatchFromDyn` —— 什么样的智能指针可以包装 trait 对象？

这是一个不稳定特性，但理解它帮你理解 Rust 类型系统的深层约束：

```rust
// 为什么 Box<dyn Trait> 可以工作？
// 因为 Box<T> 实现了 DispatchFromDyn<T>

// 为什么 Rc<dyn Trait> 可以工作？
// 因为 Rc<T> 实现了 DispatchFromDyn<T>

// 为什么 &dyn Trait 可以工作？
// 因为 &T 实现了 DispatchFromDyn<T>

// DispatchFromDyn 的含义：这个指针类型的布局，在 T 是 Sized vs T 是 !Sized 时，
// 只差在元数据上，数据指针不变。

// 现在假设你自定义了一个智能指针：
use std::ptr::NonNull;

struct MyUniquePtr<T: ?Sized> {
    ptr: NonNull<T>,
}

// 你可能想让 MyUniquePtr<dyn Trait> 工作——
// 但目前这需要 unstable 的 DispatchFromDyn 特性：
// #![feature(unsize, dispatch_from_dyn)]
// unsafe impl<T: ?Sized, U: ?Sized> DispatchFromDyn<MyUniquePtr<U>> for MyUniquePtr<T>
// where
//     T: Unsize<U>,
// {}
```

### 10.4 ZST 与 Sized 的关系 —— 占 0 字节的 "有大小" 类型

零大小类型（ZST）是一个有趣的矛盾体：它们**是** Sized，但占 0 字节。

```rust
use std::mem::size_of;

struct Unit;              // 单元结构体
struct Empty {}           // 空结构体（同上）
enum Void {}              // 空枚举（无法构造！）

assert_eq!(size_of::<Unit>(), 0);
assert_eq!(size_of::<()>(), 0);
assert_eq!(size_of::<[u8; 0]>(), 0);     // 零长度数组
assert_eq!(size_of::<Vec<()>>(), 24);    // Vec 本身不是 ZST

// ZST 仍然是 Sized——编译器知道它的大小（0 字节）
fn takes_sized<T: Sized>(x: T) {}

takes_sized(());       // ✅ () 是 Sized
takes_sized(Unit);     // ✅ Unit 是 Sized

// 但 ZST 有一些反直觉的行为：
let v = vec![(); 100];  // 100 个 () —— 但实际堆分配是 0 字节！
// Vec<()> 的容量是 usize::MAX（因为 size_of::<()> == 0 时除法有问题，所以特殊处理）
```

**PhantomData 就是利用 ZST 实现零成本类型标记的经典案例**：

```rust
use std::marker::PhantomData;

// 用类型参数区分不同单位，但运行时零开销
struct Meters(f64);
struct Kilometers(f64);

// PhantomData<T> 是 ZST，让编译器认为你"拥有" T
struct MyWrapper<T> {
    value: i32,
    _phantom: PhantomData<T>,  // 0 字节，只影响类型系统
}
```

### 10.5 Extern Types —— FFI 中的未知大小类型

Rust 现在可以声明"我不知道这个类型多大"——主要用于 C ABI：

```rust
// #![feature(extern_types)]  // nightly required

extern "C" {
    // 一个不透明的 C 类型，Rust 不知道它的大小
    // type OpaqueCStruct;
}

// 你只能通过指针使用它：
// fn use_it(x: &OpaqueCStruct) { }
// fn use_it_box(x: Box<OpaqueCStruct>) { }  // 甚至 Box<OpaqueCStruct> 都不行！

// 这和不透明类型的区别：
//   struct Opaque;    ← Rust 内部定义，Sized（0 字节或未知）
//   extern type Opaque;  ← FFI 定义，!Sized（大小真的未知）
```

### 10.6 `std::mem::size_of_val` —— 运行时获取 DST 的大小

`size_of_val` 能从胖指针的元数据中提取大小信息：

```rust
use std::mem::size_of_val;

let s: &str = "hello";
assert_eq!(size_of_val(s), 5);   // 不是 16！是 str 的实际大小

let arr: &[i32] = &[1, 2, 3, 4, 5];
assert_eq!(size_of_val(arr), 20); // 5 个 i32 × 4 字节 = 20

// 底层原理：size_of_val 读取胖指针的元数据字段
// 对于 &str / &[T]：元数据 = 长度，size = 长度 × 元素大小
// 对于 &dyn Trait：元数据 = vtable_ptr，size = vtable.size

// 你可以利用这个做泛型编程：
fn debug_size<T: ?Sized>(x: &T) {
    println!("size_of_val = {}", size_of_val(x));
}

debug_size(&42i32);            // size_of_val = 4
debug_size(&"hello world");    // size_of_val = 11
debug_size(&[1u8, 2, 3, 4]);   // size_of_val = 4
```

### 10.7 Enum 布局优化 —— DST 思维的自然延伸

同样是对"大小"的利用，Rust 编译器会优化 enum 的布局。这个和 Sized/DST 同属"编译器如何巧用类型信息"的大主题：

```rust
use std::mem::size_of;

// Option<&T> 利用了"引用不能为 0"这个非法值
assert_eq!(size_of::<Option<&i32>>(), 8);   // 和 &i32 一样大！
// 内部：None 用全 0 表示，Some(ptr) 用 ptr 表示

// Option<bool> 有优化吗？（bool 只有 0 和 1，但占 1 字节）
assert_eq!(size_of::<Option<bool>>(), 1);   // 编译器用 2-255 的值表示 None

// Option<NonZeroU32> 利用 0 作为 None
use std::num::NonZeroU32;
assert_eq!(size_of::<Option<NonZeroU32>>(), 4);  // 和 NonZeroU32 一样大！

// 但 Option<u32> 不行（因为 0 是合法值，没法区分）
assert_eq!(size_of::<Option<u32>>(), 8);  // 需要额外的判别字节（对齐到 8）
```

这是 Rust 的 **niche 优化**——编译器寻找类型的非法值来表示 enum 的变体。这和 DST 的关系是：两者都体现了 Rust 编译器对"类型大小/布局"这一信息的极致利用。

### 10.8 `CoerceUnsized` —— 自定义智能指针的 DST 强制转换

你自己的智能指针如何支持 unsized coercion？需要一个 unstable trait：

```rust
// #![feature(coerce_unsized)]

use std::ops::CoerceUnsized;
use std::marker::Unsize;

struct MyBox<T: ?Sized> {
    ptr: *mut T,
}

// 让 MyBox 支持 unsized coercion
// impl<T: ?Sized, U: ?Sized> CoerceUnsized<MyBox<U>> for MyBox<T>
// where
//     T: Unsize<U>,
// {}

// 有了这个，你就能写：
// let b: MyBox<[i32; 3]> = MyBox { ptr: ... };
// let b_dst: MyBox<[i32]> = b;  // 自动转换！(nightly)
```

### 练习 1 答案：分析类型大小

```rust
use std::mem::size_of;

// 64 位平台答案：
assert_eq!(size_of::<i32>(),        4);   // 基本类型
assert_eq!(size_of::<&i32>(),       8);   // 瘦指针（只有地址）
assert_eq!(size_of::<&[i32]>(),     16);  // 胖指针（地址 + 长度）
assert_eq!(size_of::<&str>(),       16);  // 胖指针（地址 + 长度）
assert_eq!(size_of::<&dyn Fn()>(),  16);  // 胖指针（地址 + vtable）
assert_eq!(size_of::<Box<i32>>(),   8);   // 瘦指针（Box 只存堆地址）
assert_eq!(size_of::<Box<[i32]>>(), 16);  // 胖指针（堆地址 + 长度）
assert_eq!(size_of::<Box<str>>(),   16);  // 胖指针（堆地址 + 长度）
assert_eq!(size_of::<String>(),     24);  // ptr(8) + len(8) + cap(8) = 24
assert_eq!(size_of::<Vec<i32>>(),   24);  // ptr(8) + len(8) + cap(8) = 24
assert_eq!(size_of::<&String>(),    8);   // 瘦指针（&T 指向 String 本身，String 是 Sized）
assert_eq!(size_of::<&&str>(),      8);   // 瘦指针（&&str = &(&str)，&str 是 Sized，所以 &(&str) 是瘦指针）
```

**关键理解点**：
- `Box<i32>` = 8 字节（瘦），`Box<[i32]>` = 16 字节（胖）。同样的 Box，取决于它包装的是 Sized 还是 DST。
- `&String` = 8 字节（瘦），因为 `String` 自身是 Sized，`&String` 就是普通的瘦引用。
- `&&str` = 8 字节（瘦），因为 `&str` 是 Sized（16 字节），所以指向 Sized 类型的引用就是瘦指针。

### 练习 2 答案：设计处理 DST 的函数

```rust
// 定义一个 trait，用于返回"可打印的长度"
trait HasLength {
    fn length(&self) -> usize;
}

// 为所有需要的类型实现
impl HasLength for str {
    fn length(&self) -> usize { self.len() }
}

impl HasLength for [i32] {
    fn length(&self) -> usize { self.len() }
}

// 注意：String 和 Vec<i32> 是 Sized，也需要显式实现
impl HasLength for String {
    fn length(&self) -> usize { self.len() }
}

impl HasLength for Vec<i32> {
    fn length(&self) -> usize { self.len() }
}

// 关键函数：接受异质集合
fn print_lengths(items: &[&dyn HasLength]) {
    for (i, item) in items.iter().enumerate() {
        println!("元素 {} 的长度: {}", i, item.length());
    }
}

// 使用
fn main() {
    let s = String::from("hello");
    let v = vec![1, 2, 3, 4];

    // 注意这里用了 unsized coercion：
    // &s → &str（String 自动转成 str 引用）
    // &v → &[i32]（Vec<i32> 自动转成 [i32] 引用）
    print_lengths(&[
        &s as &str,       // String → str（unsized coercion）
        &v as &[i32],     // Vec<i32> → [i32]（unsized coercion）
        "world",          // &str 本身已经是 DST 引用
        &[10, 20, 30],    // &[i32] 本身已经是 DST 引用
    ]);
}

// 输出：
// 元素 0 的长度: 5
// 元素 1 的长度: 4
// 元素 2 的长度: 5
// 元素 3 的长度: 3
```

**设计要点**：
1. trait `HasLength` 定义统一接口
2. 为 DST 类型（`str`, `[i32]`）实现 trait——注意 impl 的是 `str` 和 `[i32]`，不是 `&str` 和 `&[i32]`
3. 用 `&dyn HasLength` 做异质集合
4. 利用 unsized coercion 将 `&String` 自动转换为 `&str`

### 练习 3 答案：自定义 DST

```rust
use std::mem;

// 与 C flexible array member 兼容的结构
#[repr(C)]
struct Message<T: ?Sized> {
    msg_type: u16,
    flags: u16,
    payload: T,  // C flexible array member 的 Rust 对应
}

impl Message<[u8]> {
    /// 从原始字节构造 Message DST 引用
    ///
    /// # Safety
    ///
    /// raw 的前 4 字节必须是合法的 msg_type + flags
    pub unsafe fn from_raw(raw: &[u8]) -> &Self {
        let header_size = mem::size_of::<Message<()>>();  // 4 字节
        assert!(raw.len() >= header_size, "数据太短");

        let payload_len = raw.len() - header_size;
        // 构造 DST 胖指针：ptr + metadata（payload_len）
        &*std::ptr::from_raw_parts(
            raw.as_ptr() as *const (),
            payload_len,
        )
    }
}

// 使用示例
fn main() {
    // 模拟网络收到的字节：msg_type=1, flags=0, payload=[10, 20, 30, 40, 50]
    let raw_data = [0u8, 1, 0, 0, 10, 20, 30, 40, 50];
    //               msg_type=0x0100 (big-endian), flags=0

    let msg: &Message<[u8]> = unsafe { Message::from_raw(&raw_data) };

    // 验证 header
    assert_eq!(msg.msg_type, 0x0100);
    assert_eq!(msg.flags, 0);

    // 验证 payload（DST 字段）
    assert_eq!(msg.payload.len(), 5);
    assert_eq!(msg.payload, &[10, 20, 30, 40, 50]);
}

// 进阶：用同样的模式定义一个字符串消息头
#[repr(C)]
struct StringMessage<T: ?Sized> {
    opcode: u8,
    body: T,
}

impl StringMessage<str> {
    pub unsafe fn from_raw(raw: &[u8]) -> &Self {
        let header_size = mem::size_of::<StringMessage<()>>(); // 1 字节（但可能对齐到更大）
        let str_len = raw.len() - header_size;

        let str_ptr = raw.as_ptr().add(header_size);
        let str_slice = std::slice::from_raw_parts(str_ptr, str_len);
        let body: &str = std::str::from_utf8(str_slice).unwrap();

        // 构造 DST 胖指针：ptr + metadata（body 的长度）
        &*std::ptr::from_raw_parts(
            raw.as_ptr() as *const (),
            body.len(),  // 元数据是 str 的长度（和 [u8] 一样）
        )
    }
}
```

### 练习 4 答案：解释题

**Q1: 为什么 String 是 Sized，str 不是？两者的设计意图有何不同？**

| 维度 | String | str |
|:---|:---|:---|
| **内存位置** | 栈上 24 字节 (ptr/len/cap)，数据在堆上 | 本身不占独立空间，是一段数据的"视图" |
| **设计意图** | 拥有者 (owner)，管理内存生命周期，可变长可修改 | 借用者 (borrower)，不管理内存，不可变只读 |
| **为什么 Sized / !Sized** | 固定 24 字节，编译器能确定栈上布局 | 长度在运行时才知道，但长度存在引用/指针中而非类型本身 |

这个设计使得：

- String 可以按值传递、move、作为 Vec 的元素
- &str 可以作为任意字符串数据的零拷贝视图（子串、字面量、文件映射）
- 避免了 C++ std::string_view 需要单独定义一套类型的冗余

**Q2: 为什么 `Vec<T>` 是 Sized，`[T]` 不是？**

和 String vs str 完全一样的道理：

- `Vec<T>` = `{ ptr: *mut T, len: usize, cap: usize }` —— 固定 24 字节
- `[T]` = 编译时长度未知的 T 的连续序列

`Vec<T>` 是拥有者，`[T]` 是视图。这种"拥有者 Sized + 视图 DST"的对偶设计是 Rust 核心模式之一。

其他例子：PathBuf / Path, OsString / OsStr, CString / CStr

**Q3: 在什么场景下你会主动选择 dyn Trait 而不是泛型？**

**场景 1: 异质集合** —— 需要把不同类型放在同一个容器里
`Vec<Box<dyn EventHandler>>` 可以混合 MouseHandler、KeyHandler

**场景 2: 减小二进制体积** —— 泛型为每个具体类型生成代码（代码膨胀），dyn Trait 只生成一份代码。适合有几十种实现类型但调用不频繁的场景

**场景 3: 插件/动态加载** —— 编译时不知道具体类型，从配置文件或动态库加载实现
`let handler: Box<dyn Plugin> = load_plugin("my_plugin.so")?;`

**场景 4: 递归类型** —— enum 递归需要 Box 间接，但有时 dyn Trait 更简洁
`enum Expr { Add(Box<Expr>, Box<Expr>), ... }` vs 用 trait 对象

**场景 5: 减少编译时间** —— 泛型导致大量单态化导致编译慢，dyn Trait 只编译一次

## 12. 编译器错误诊断指南 —— 当 Sized 约束出问题时

实际编码中，Sized 相关的编译错误是最常见的困惑来源之一。这里教你怎么一眼看懂、一秒修复。

### 错误 1：`the size for values of type X cannot be known at compilation time`

```rust
// ❌ 错误代码
fn process(data: [u8]) { }     // 或 data: str, data: dyn Trait

// 编译器输出：
// error[E0277]: the size for values of type `[u8]` cannot be known
//               at compilation time
//   = help: the trait `Sized` is not implemented for `[u8]`
//   = note: all local variables must have a statically known size
//   = help: unsized locals are gated as an unstable feature
```

**修复策略（按推荐顺序）**：

```rust
// 方案 1：改成引用（最常用）
fn process(data: &[u8]) { }

// 方案 2：用 Box 获取所有权
fn process(data: Box<[u8]>) { }

// 方案 3：使用 Sized 版本的类型
fn process(data: Vec<u8>) { }     // Vec<u8> 是 Sized
fn process(data: String) { }      // String 是 Sized
```

### 错误 2：泛型函数中 `T` 不能用于 DST

```rust
// ❌ 错误代码
fn first<T: Default>(slice: &[T]) -> T {
    if slice.is_empty() { T::default() }
    else { slice[0] }  // 试图 move 出一个元素
}

// 编译器输出：
// error[E0277]: the size for values of type `T` cannot be known
//   = note: `T` doesn't implement `std::marker::Sized`

// 🤔 为什么？因为 `slice[0]` 试图按值返回一个元素，
// 如果 T = [u8]，那返回值的大小是不确定的
```

**修复**：

```rust
// 修复 1：显式加 Sized 约束（如果没有 DST 需求）
fn first<T: Default + Sized>(slice: &[T]) -> T { ... }

// 修复 2：返回引用而不是值
fn first<T>(slice: &[T]) -> &T { &slice[0] }

// 修复 3：Clone 而不是 move
fn first<T: Clone>(slice: &[T]) -> T { slice[0].clone() }
```

### 错误 3：`?Sized` 位置不对

```rust
// ❌ 位置 1：局部变量不能用 ?Sized
fn bad() {
    let x: dyn Display = todo!();     // ❌
}

// ❌ 位置 2：函数返回值不能用 ?Sized（除非用引用/Box）
fn bad() -> dyn Display { todo!() }  // ❌

// ❌ 位置 3：不是最后一个字段
struct Bad<T: ?Sized> {
    a: T,       // ← 如果 T 是 DST，b 的偏移无法确定
    b: u32,     // ❌ T 必须是最后一个字段
}
```

### 错误 4：在 trait 对象上调用了需要 Sized 的方法

```rust
trait Foo {
    fn method1(&self);
    fn method2(self) where Self: Sized;  // 需要 Self: Sized
}

fn use_dyn(x: &dyn Foo) {
    x.method1();  // ✅ 只用了 &self
    // x.method2();  // ❌ method2 需要 Self: Sized，但 dyn Foo 不是 Sized
}
```

**理解错误信息的套路**：

```
所有 Sized 相关错误的根因只有 3 种：

1. 把 DST 放在需要 Sized 的位置（最常见）
   → 修复：加 & / Box / 改成 Sized 类型

2. 泛型参数隐式要求 Sized，但传入了 DST
   → 修复：加 T: ?Sized 放宽约束

3. 试图 move 一个 DST 的值
   → 修复：通过引用操作，或 clone
```

## 13. Sized 与 Move 语义

这是一个很多人没意识到的事实：**你只能 move Sized 类型的值**。

```rust
// ✅ 可以 move 的（都是 Sized）
let a: i32 = 42;           // move 到新位置
let b: String = a.to_string();  // move
let c: Vec<u32> = vec![1, 2];   // move

// ❌ 不能 move 的
// let d: str = *"hello";        // 无法编译：str 是 DST
// let e: [u8] = *b"hello";      // 无法编译：[u8] 是 DST

// 为什么？因为 move = memcpy，编译器必须知道拷贝多少字节
// 对于 DST，字节数在编译时未知 → 无法 memcpy → 无法 move
```

这个约束解释了很多设计决策：

```rust
// 为什么 Box<dyn Trait> 存在？
// 因为 dyn Trait 不能 move，所以要用 Box 把 DST 放在堆上
// Box 本身只在栈上存指针（8/16 字节），可以 move

// 为什么 &dyn Trait 可以 move？
// 因为 &dyn Trait 是胖指针（16 字节），是 Sized 的
// move 的是胖指针本身，不是指向的数据

// 为什么 fn foo(x: &str) 可以但 fn foo(x: str) 不行？
// &str 是 Sized，作为函数参数时编译器知道要拷贝 16 字节
// str 是 DST，编译器不知道拷贝多少
```

**Move 和 DST 的完整对照表**：

| 操作 | Sized 类型 | DST（通过胖指针） |
|:---|:---|:---|
| 局部变量 `let x: T` | ✅ | ❌ |
| 函数参数 `fn f(x: T)` | ✅ | ❌ |
| 函数返回值 `fn f() -> T` | ✅ | ❌（需要用 `Box`/`&`） |
| 函数参数 `fn f(x: &T)` | ✅ | ✅ |
| 放入 Vec `Vec<T>` | ✅ | ❌（用 `Vec<Box<T>>`） |
| Clone | ✅ | ❌（`dyn Clone` 不合法） |
| Copy | ✅ | ❌ |

## 14. Sized 与 Rust 的高级特性交互

### 14.1 闭包

```rust
// 每个闭包都有唯一的、编译时生成的匿名类型
let f = |x: i32| x + 1;
// f 的类型是编译器生成的，类似 Closure123——它是 Sized 的

// 所以你可以：
let f: Box<dyn Fn(i32) -> i32> = Box::new(|x| x + 1);
//           ^^^^^^^^^^^^^^^^  Fn trait 派生自 FnOnce

// 但你不能：
// fn return_closure() -> dyn Fn(i32) -> i32 { |x| x + 1 }  // ❌ 不 Sized
fn return_closure() -> Box<dyn Fn(i32) -> i32> { Box::new(|x| x + 1) }  // ✅
```

### 14.2 async 块

```rust
// async 块编译为匿名的 Future 类型——它是 Sized 的
let fut = async { 42 };
// fut 的类型是编译器生成的，类似 AsyncBlock123——它是 Sized

// 但是！async 块内部被编译为自引用状态机，需要 Pin
// 这就是为什么 tokio::spawn 需要 Future + Send + 'static
// Sized 是隐式满足的，你不需要关心
```

### 14.3 `impl Trait` 与 Sized

```rust
// impl Trait 作为返回值时，具体类型是 Sized 的
fn make_iter() -> impl Iterator<Item = i32> {
    vec![1, 2, 3].into_iter()  // 具体类型是 std::vec::IntoIter<i32>，Sized
}

// 但你不能用 impl Trait 替代 dyn Trait 做异质集合
// fn bad() -> Vec<impl Display> { vec![1, "hello"] }  // ❌ 所有元素必须是同一类型
fn good() -> Vec<Box<dyn Display>> { vec![Box::new(1), Box::new("hello")] }  // ✅
```

## 15. std 中的完整 DST 配对目录

Rust 标准库中反复出现"拥有者 (Sized) ↔ 视图 (DST)"的设计模式，全部列举如下：

| 拥有者 (Sized) | 视图 (!Sized) | 说明 |
|:---|:---|:---|
| `String` | `str` | UTF-8 字符串 |
| `Vec<T>` | `[T]` | 泛型切片 |
| `PathBuf` | `Path` | 文件系统路径 |
| `OsString` | `OsStr` | OS 原生字符串（Unix bytes / Windows WTF-16） |
| `CString` | `CStr` | C 风格 null 结尾字符串 |

```rust
use std::path::{PathBuf, Path};
use std::ffi::{CString, CStr, OsString, OsStr};

// 所有配对遵循相同的 Deref 模式：

// String → str
let s: String = "hello".into();
let r: &str = &s;           // Deref<Target = str>

// Vec<T> → [T]
let v: Vec<i32> = vec![1, 2, 3];
let r: &[i32] = &v;          // Deref<Target = [i32]>

// PathBuf → Path
let p: PathBuf = PathBuf::from("/tmp");
let r: &Path = &p;           // Deref<Target = Path>

// OsString → OsStr
let o: OsString = OsString::from("hello");
let r: &OsStr = &o;          // Deref<Target = OsStr>

// CString → CStr
let c: CString = CString::new("hello").unwrap();
let r: &CStr = &c;           // Deref<Target = CStr>
```

**为什么标准库用 DST 而不是全用泛型？**

```rust
// 如果 str 不存在，每个接受字符串的函数都需要两个版本：
fn process_owned(s: String) { }
fn process_borrowed(s: &String) { }  // 冗余！

// 有了 str，只需要一个：
fn process(s: &str) { }
// String 通过 Deref 自动转成 &str——一个函数覆盖两种情况

// 这就是 DST 视图的人体工程学价值：
// 一次定义，同时服务拥有者和借用者
```

## 16. 思维模型：遇到 Sized/DST 问题时的决策框架

当你写代码遇到类型大小问题时，按这个决策树走：

> **我要操作的数据，编译时知道大小吗？**
>
> **→ 知道（Sized）**
> - 数据在栈上？→ `fn f(x: T)` 按值传递
> - 数据在堆上？→ `fn f(x: Box<T>)` 堆上分配
> - 想动态分发？→ `fn f(x: &dyn Trait)` 胖指针（16 字节）
>
> **→ 不知道（DST）**
> - 要拥有数据？→ `Box<dyn Trait>` / `Box<[T]>` / `Box<str>`（胖指针 Box）
> - 只需要借用？→ `&dyn Trait` / `&[T]` / `&str`（胖指针引用）
> - 泛型代码同时支持 Sized 和 DST？→ `fn f<T: ?Sized + Trait>(x: &T)` 放宽约束
> - 结构体中存 DST？→ `struct S<T: ?Sized> { ..., last: T }` 尾部字段

## 17. 回顾

**本课核心知识链：**

1. **Sized** = 编译时大小已知的类型（默认约束）
2. ↓ **?Sized** = 在泛型中"放宽"这个要求
3. ↓ **DST** = 运行时长才能确定大小的类型（`str`, `[T]`, `dyn Trait`）
4. ↓ **胖指针** = 补偿机制：瘦指针 + 元数据，保证指针本身是 Sized
5. ↓ **vtable** = `dyn Trait` 的元数据，包含 size / align / drop / 方法指针
