# 1.6 比较类：PartialEq / Eq / PartialOrd / Ord / Hash

> 看起来最简单的一组 trait——"不就是 == 和 < 吗？"但 Rust 的比较体系是最容易被误解的。`f32` 为什么不是 `Eq`？`Ord` 和 `PartialOrd` 到底区别在哪？`Hash` 和 `Eq` 之间有什么隐藏契约？这一课把所有暗坑照亮。

---

## 1. 从"浮点数能不能比较"这个问题开始

```rust
let a = 0.1_f64 + 0.2_f64;
let b = 0.3_f64;

println!("{}", a == b);  // false!
```

**为什么？** 因为浮点数运算的精度问题。但更深层的问题是：

```rust
let x: f64 = f64::NAN;
println!("{}", x == x);  // false!
```

**NaN 不等于任何值，包括它自己。** 这就叫"部分等价"（partial equivalence）——不是所有值之间都有明确的相等关系。

Rust 的类型系统**精确地**模拟了这一点：

```rust
// f64 实现的是 PartialEq（部分等价），不是 Eq（全等价）
impl PartialEq for f64 { }  // ✅
// impl Eq for f64 { }      // ❌ f64 永远不会实现 Eq
```

---

## 2. 比较 trait 的层级结构

```
PartialEq  ←  Eq         ("部分等于" ← "完全等于")
    │
PartialOrd ←  Ord        ("部分排序" ← "全排序")
    │
   Hash                  (哈希——依赖 Eq)

图例：A ← B 表示 B 是 A 的 sub-trait（B 比 A 更强）
```

**关键含义**：如果你实现 `Eq`，你必须先实现 `PartialEq`。如果你实现 `Ord`，你必须先实现 `PartialOrd` 和 `Eq`。

---

## 3. PartialEq：部分等价

### 3.1 定义

```rust
pub trait PartialEq<Rhs: ?Sized = Self> {
    /// 这个方法必须满足：
    /// - 对称性：a == b → b == a
    /// - 传递性：a == b && b == c → a == c
    fn eq(&self, other: &Rhs) -> bool;

    fn ne(&self, other: &Rhs) -> bool {
        !self.eq(other)  // 默认实现
    }
}
```

**注意 `Rhs` 参数**：`PartialEq` 允许你定义**跨类型比较**。`Rhs` 默认是 `Self`。

### 3.2 跨类型比较

```rust
// 你可以让 String 和 &str 互相比较
// 标准库已经做了：impl PartialEq<&str> for String

let s = String::from("hello");
assert!(s == "hello");   // String 和 &str 比较
assert!("hello" == s);   // 反过来也行（因为标准库还实现了
                         // PartialEq<String> for &str）

// 但这种通常不合理：
#[derive(PartialEq)]
struct Meters(f64);

#[derive(PartialEq)]
struct Feet(f64);

impl PartialEq<Feet> for Meters {
    fn eq(&self, other: &Feet) -> bool {
        (self.0 - other.0 * 0.3048).abs() < 0.001
    }
}

// 现在可以跨类型比较
let m = Meters(1.0);
let f = Feet(3.28);
assert!(m == f);  // ✅ 但你确定希望这样？
```

**设计建议**：跨类型的 `PartialEq` 极少使用。大多数情况下你应该保持 `Rhs = Self`。

### 3.3 浮点数的 PartialEq

```rust
// f64 的 PartialEq 特化处理了 NaN：
assert!(f64::NAN != f64::NAN);  // NaN 不等于任何值，包括自己

// 如果你想检查是不是 NaN：
assert!(f64::NAN.is_nan());
// 不能写：f64::NAN == f64::NAN

// 这给集合带来的问题：
let mut set = std::collections::HashSet::new();
set.insert(f64::NAN);
set.insert(f64::NAN);
assert_eq!(set.len(), 2);  // 两个 NaN 被认为不相等，所以都存进去了！
```

## 4. Eq：完全等价

### 4.1 定义

```rust
pub trait Eq: PartialEq<Self> {
    // 这个 trait 是"标记"trait——没有方法
    // 它只增加了一个语义承诺：反射性
    //
    // 反射性要求：对任意 a，a == a 必须是 true
}
```

`Eq` 唯一的作用是给 `PartialEq` 增加一个**语义保证**：

```
PartialEq：a == a 可能返回 false（如果我包含 NaN）
Eq：       a == a 一定返回 true
```

### 4.2 为什么需要标记 trait

编译器不能自动验证"a == a 一定为 true"，所以 `Eq` 是一个**信任契约**：

```rust
// ✅ i32 是 Eq——显然 a == a 总是 true
assert_eq!(5, 5);

// ❌ f64 不是 Eq——f64::NAN == f64::NAN 是 false
// 所以任何包含 f64 的结构体也默认不能 derive(Eq)

#[derive(PartialEq, Eq)]  // ❌ 编译错误！
struct Measurement {
    value: f64,  // f64 不是 Eq → Measurement 也不能是 Eq
}

// ✅ 解决办法：用 OrderedFloat 或手动实现
use ordered_float::OrderedFloat;

#[derive(PartialEq, Eq)]
struct Measurement {
    value: OrderedFloat<f64>,  // OrderedFloat 把 NaN 也看作相等
}
```

## 5. PartialOrd / Ord：排序

### 5.1 层级定义

```rust
pub trait PartialOrd<Rhs: ?Sized = Self>: PartialEq<Rhs> {
    fn partial_cmp(&self, other: &Rhs) -> Option<Ordering>;

    fn lt(&self, other: &Rhs) -> bool { /* 默认实现 */ }
    fn le(&self, other: &Rhs) -> bool { /* 默认实现 */ }
    fn gt(&self, other: &Rhs) -> bool { /* 默认实现 */ }
    fn ge(&self, other: &Rhs) -> bool { /* 默认实现 */ }
}

pub trait Ord: Eq + PartialOrd<Self> {
    fn cmp(&self, other: &Self) -> Ordering;

    fn max(self, other: Self) -> Self { /* 默认实现 */ }
    fn min(self, other: Self) -> Self { /* 默认实现 */ }
    fn clamp(self, min: Self, max: Self) -> Self { /* 默认实现 */ }
}
```

### 5.2 关键区别：`partial_cmp` 返回 `Option`，`cmp` 返回裸值

```rust
use std::cmp::Ordering;

// PartialOrd：可能不可比较
let result: Option<Ordering> = 1.0_f64.partial_cmp(&f64::NAN);
assert_eq!(result, None);  // NaN 无法和任何东西比较！

// Ord：保证可比较
let result: Ordering = 5_i32.cmp(&10);
assert_eq!(result, Ordering::Less);  // 永远能出结果
```

### 5.3 f64 的排序困境

```rust
let mut v = vec![1.0, f64::NAN, 3.0, f64::NAN, 2.0];

// ❌ v.sort()：f64 没有实现 Ord，所以 sort() 不能用
// v.sort();  // 编译错误！

// ✅ 使用 sort_by 配合 partial_cmp
v.sort_by(|a, b| a.partial_cmp(b).unwrap_or(Ordering::Equal));

// 更好的方法：用 total_cmp（Rust 1.62+）
v.sort_by(|a, b| a.total_cmp(b));
// total_cmp 定义了 NaN 的完整顺序（NaN > 无穷大 > 正常数 > ...）
```

### 5.4 为什么 Ord 要求 Eq

```rust
// 没有 Eq 的 Ord 会导致逻辑矛盾：
// 如果 Ord 允许 a.cmp(b) 返回 Ordering::Equal，
// 但 a == b 返回 false（因为 Eq 不满足）...
//
// 这是语义上的矛盾："排序说相等，但等价判断说不相等"
// Rust 通过 trait bound 在类型层面防止了这个矛盾
```

## 6. Hash：哈希

### 6.1 定义

```rust
pub trait Hash {
    fn hash<H: Hasher>(&self, state: &mut H);

    // 关联方法（1.70+）
    fn hash_slice<H: Hasher>(data: &[Self], state: &mut H)
    where Self: Sized
    { /* 默认实现 */ }
}
```

### 6.2 Hash 和 Eq 的关键契约

**这是 Rust 中最重要但没写出来的契约之一：**

```
如果 a == b，那么 hash(a) == hash(b) 必须成立。

逆命题不成立：
hash(a) == hash(b) 不能推出 a == b（哈希冲突）
```

违反这个契约的后果：

```rust
use std::collections::HashSet;

// 假设我们错误地实现了 Hash（不看 value 字段）
#[derive(Debug)]
struct BadKey {
    id: u64,
    value: String,
}

impl PartialEq for BadKey {
    fn eq(&self, other: &Self) -> bool {
        self.id == other.id && self.value == other.value  // 两个字段
    }
}

impl Eq for BadKey { }

impl std::hash::Hash for BadKey {
    fn hash<H: std::hash::Hasher>(&self, state: &mut H) {
        self.id.hash(state);
        // ❌ BUG：忘了 hash value 字段！
    }
}

// 后果：
let mut set = HashSet::new();
set.insert(BadKey { id: 1, value: "a".into() });
set.insert(BadKey { id: 1, value: "b".into() });

// 根据 Eq，两个 key 不同（value 不同），
// 但根据 Hash，它们的 hash 相同（id 相同），
// HashSet 认为它们在同一桶，可能覆盖或丢失！
// → 未定义行为级的数据损坏
```

**记住**：如果你手动实现 `Hash`，必须保证和 `PartialEq` 用相同的字段子集。

### 6.3 为什么 derive(Hash) 是安全的

```rust
#[derive(PartialEq, Eq, Hash)]
struct User {
    id: u64,
    name: String,
}
// derive 保证：Hash 和 PartialEq 使用相同的字段组合
// 绝不会出现上述的不匹配
```

## 7. 比较 trait 的自动实现策略

### 7.1 derive 的条件传递

```rust
#[derive(PartialEq)]  // ✅ 所有字段都实现了 PartialEq → 可以 derive
struct AllGood {
    a: i32,
    b: String,
}

#[derive(PartialEq)]  // ❌ f64 实现了 PartialEq，所以可以 derive PartialEq
struct WithFloat {
    value: f64,       // ✅ PartialEq 可以 derive
}

#[derive(Eq)]         // ❌ f64 不是 Eq，所以整个 struct 不能 derive Eq
struct WithFloat {
    value: f64,
}
// error[E0277]: the trait bound `f64: Eq` is not satisfied
```

### 7.2 手动实现必须遵守的规则

如果你手动实现任何一个比较 trait，必须遵守以下规则：

```
PartialEq:
├── 对称性：a == b → b == a
├── 传递性：a == b 且 b == c → a == c
└── 一致性：多次调用 eq 对同一对值结果相同（不可变性）

Eq（额外）：
└── 反射性：a == a 总是 true

Ord（额外）：
├── 反对称性：如果 a < b，那么 b > a
├── 传递性：a < b 且 b < c → a < c
├── 全序性：对任意 a, b，要么 a < b，要么 a == b，要么 a > b
└── 与 Eq 一致：a == b ↔ cmp 返回 Equal

Hash（额外）：
└── 与 Eq 一致：a == b → hash(a) == hash(b)
```

### 7.3 常见打破规则的行为

```rust
// 反例 1：破坏对称性
impl PartialEq for Weird {
    fn eq(&self, other: &Self) -> bool {
        self.id == 0 || other.id == 0 || self.id == other.id
    }
    // Weird { id: 0 } == Weird { id: 5 } 是 true
    // 但 Weird { id: 5 } == Weird { id: 0 } 也是 true（因为 other.id == 0）
    // 这个例子里对称性恰好满足，
    // 但传递性严重违反：0==1 且 0==2，但 1!=2
}

// 反例 2：Hash 与 Eq 不一致
// （见 6.2 的 BadKey 例子）

// 反例 3：Ord 与 PartialEq 不一致
impl Ord for BadOrd {
    fn cmp(&self, other: &Self) -> Ordering {
        Ordering::Equal  // 永远返回 Equal
    }
}
// 但如果 PartialEq::eq 返回 false（字段不同），
// Ord 返回 Equal 而 PartialEq 说不相等 → 矛盾！
```

## 8. 实战：各种数据结构的 trait 需求

| 数据结构 | PartialEq | Eq | PartialOrd | Ord | Hash |
|:---|:---:|:---:|:---:|:---:|:---:|
| `Vec::contains` | ✅ | — | — | — | — |
| `HashSet` | ✅ | ✅ | — | — | ✅ |
| `HashMap` key | ✅ | ✅ | — | — | ✅ |
| `BTreeSet` | ✅ | ✅ | — | ✅ | — |
| `BTreeMap` key | ✅ | ✅ | — | ✅ | — |
| `Vec::sort` | — | — | — | ✅ | — |
| `Vec::sort_by` | — | — | ✅ | — | — |
| `BinaryHeap` (max) | — | — | — | ✅ | — |
| `==` 运算符 | ✅ | — | — | — | — |
| `<` 运算符 | ✅ | — | ✅ | — | — |

**经验法则**：
- 只需要查找/包含检查 → `PartialEq + Eq`
- 需要排序 → `Ord`（或 `PartialOrd` for sort_by）
- 需要哈希表 → `PartialEq + Eq + Hash`
- 需要有序表 → `PartialEq + Eq + Ord`

## 9. 设计哲学：为什么是 Partial + Full 两层

### 9.1 C++ 的"单层"做法——坏在哪

C++ 只有一个 `operator==`，所有类型都实现它。但**没有区分"总是有意义"和"有时没意义"**：

```cpp
#include <cmath>
#include <set>

// C++ 中，浮点数有 operator==
double a = 0.1 + 0.2;
double b = 0.3;
if (a == b) { /* 通常不成立——精度问题 */ }

// 更严重的问题：NaN
double nan = std::nan("");
if (nan == nan) {  // false! IEEE 754 标准说 NaN != NaN
    // 这永远不会执行
}

// 但 C++ 的容器不关心：
std::set<double> s;
s.insert(nan);
s.insert(nan);
// s.size() == 2! 因为 NaN != NaN，两个 NaN 被认为"不同"
// → 你的 set 里有一堆"看不见"的重复元素

// 而且 C++ 编译器不会帮你发现这个问题——
// 没有任何编译期警告、没有类型层面的约束
```

**C++ 没有 `PartialOrd` vs `Ord` 的区分**：

```cpp
std::vector<double> v = {1.0, std::nan(""), 3.0};
std::sort(v.begin(), v.end());
// 结果？未定义！
// C++ 的 sort 假定 operator< 是 strict weak ordering
// 但 NaN 不是——NaN < 1.0 是 false，1.0 < NaN 也是 false
// → sort 行为未定义，可能崩溃、可能产生乱序结果

// Rust 直接禁止了这行：
// v.sort();  // ❌ f64 没有实现 Ord，编译器阻止你犯错
```

### 9.2 如果 Rust 只有单层 Equality trait 会怎样

假设 Rust 只有一个 `Eq` trait（像 C++ 的 `operator==` 一样），没有 `PartialEq`：

```rust
// 只有一个 Eq trait，f64 必须实现它
impl Eq for f64 { }  // 假设可以

// 那么编译器就会允许：
let mut set: HashSet<f64> = HashSet::new();
set.insert(f64::NAN);
set.insert(f64::NAN);
// HashSet 依赖 Eq 的反射性（a == a）
// 但 f64::NAN 不满足——f64 违反 Eq 契约
// → 程序静默错误——HashSet 里有两个"应该相同"的 NaN

// Rust 的解决：f64 只实现 PartialEq，不实现 Eq
// 所以 HashSet<f64> 根本无法编译：
// error[E0599]: the method `insert` exists but the trait bound `f64: Eq` is not satisfied
// 编译器替你阻止了这个语义错误
```

### 9.3 多语言对比表

| 语言 | 做法 | NaN 在集合中的行为 | 类型安全 |
|:---|:---|:---|:---:|
| **Rust** | `PartialEq` + `Eq` 两层 | `f64` 不能用于 `HashSet` key（因为不是 `Eq`） | ✅ 编译期 |
| **C++** | 单一的 `operator==` | `std::set<double>` 可以塞 NaN——每个 NaN 都"不同"，静默错误 | ❌ 无保护 |
| **Haskell** | `Eq` typeclass | NaN 是 `Eq` 实例，但 `compare NaN NaN = GT`——违反直觉 | ⚠️ 类型层面通过了 |
| **Java** | `equals()` 方法 | `Double.NaN.equals(Double.NaN)` → `true`！与 IEEE 754 矛盾 | ⚠️ 语义不一致 |
| **Go** | `==` 运算符 | `math.NaN() == math.NaN()` → `false`，但 `map` 用 `==` 比较 key——NaN 可以不同 | ❌ 运行时坑 |

**Rust 的核心设计选择**：在类型层面上建模数学事实——有些等价关系是"部分定义"的。这让编译器（而不是程序员的记忆力）来保证安全。

### 9.4 Rust 的 PartialOrd / Ord 同理

```
C++ 的 sort 假设 operator< 是全序（total order）
→ 但浮点数的 < 不是全序（NaN 问题）
→ 编译器不会告诉你 → 运行时 UB

Rust 的 sort 要求 Ord（全序）
→ f64 没有实现 Ord
→ 编译器阻止你 sort 浮点数（除非你用 total_cmp 等明确处理 NaN 的方式）
→ 编译期安全
```

这多出来的一个 trait 层级，在"写代码时"是多敲几个字，但在"debug 时"可能节省数小时。

## 11. 练习答案

### 练习 1 答案

```rust
use std::cmp::Ordering;

#[derive(Debug, Hash)]
struct Version {
    major: u32,
    minor: u32,
    patch: u32,
}

impl PartialEq for Version {
    fn eq(&self, other: &Self) -> bool {
        self.major == other.major
            && self.minor == other.minor
            && self.patch == other.patch
    }
}

impl Eq for Version { }

impl PartialOrd for Version {
    fn partial_cmp(&self, other: &Self) -> Option<Ordering> {
        Some(self.cmp(other))  // 委托给 Ord
    }
}

impl Ord for Version {
    fn cmp(&self, other: &Self) -> Ordering {
        self.major.cmp(&other.major)
            .then(self.minor.cmp(&other.minor))
            .then(self.patch.cmp(&other.patch))
    }
}

fn main() {
    let v1 = Version { major: 1, minor: 2, patch: 3 };
    let v2 = Version { major: 1, minor: 3, patch: 0 };
    let v3 = Version { major: 1, minor: 2, patch: 3 };

    assert!(v1 < v2);         // Ord
    assert!(v1 == v3);        // PartialEq
    assert_ne!(v1, v2);

    // Hash 一致性验证
    use std::collections::hash_map::DefaultHasher;
    use std::hash::{Hash, Hasher};

    let mut h1 = DefaultHasher::new();
    let mut h2 = DefaultHasher::new();
    v1.hash(&mut h1);
    v3.hash(&mut h2);
    assert_eq!(h1.finish(), h2.finish());
}
```

### 练习 3 答案

```rust
use std::hash::{Hash, Hasher};

struct Point(f64, f64);

impl PartialEq for Point {
    fn eq(&self, other: &Self) -> bool {
        // 处理 NaN：两个 NaN 被认为相等（跟 f64 的默认行为不同！）
        if self.0.is_nan() && other.0.is_nan() && self.1.is_nan() && other.1.is_nan() {
            return true;
        }
        self.0 == other.0 && self.1 == other.1
    }
}

impl Eq for Point { }

impl Hash for Point {
    fn hash<H: Hasher>(&self, state: &mut H) {
        // 把 f64 转换成 u64 的位表示来 hash
        // NaN 的多种位表示会被统一处理
        let hash_x = if self.0.is_nan() {
            0u64  // 所有 NaN 得到相同的 hash
        } else {
            self.0.to_bits()
        };

        let hash_y = if self.1.is_nan() {
            0u64
        } else {
            self.1.to_bits()
        };

        hash_x.hash(state);
        hash_y.hash(state);
    }
}

// 验证一致性
fn main() {
    let p1 = Point(f64::NAN, f64::NAN);
    let p2 = Point(f64::NAN, f64::NAN);

    assert_eq!(p1, p2);  // PartialEq 说相等

    use std::collections::hash_map::DefaultHasher;
    let mut h1 = DefaultHasher::new();
    let mut h2 = DefaultHasher::new();
    p1.hash(&mut h1);
    p2.hash(&mut h2);
    assert_eq!(h1.finish(), h2.finish());  // Hash 也相等 ✅
}
```

### 练习 4 答案

**BTreeMap 基于 B-Tree 数据结构**：节点中的 key 按序排列，查找时使用二分查找（比较 key），所以只需要 `Ord`。不需要哈希任何东西。

**HashMap 基于哈希表**：key 被哈希到一个桶中，查找时先哈希定位桶，然后用 `Eq` 在桶内精确匹配。哈希函数不保证唯一性（哈希冲突），所以同时需要 `Hash` 和 `Eq`。

```
HashMap 查找：hash(key) → bucket → 遍历桶内 → eq(key, candidate)
BTreeMap 查找：root → cmp(key, node_key) → left/right → ... → eq(key, found)
```

两者的时间和空间复杂度也不同：
- HashMap：O(1) 平均查找，但需要更多内存（负载因子 ~0.7）
- BTreeMap：O(log n) 查找，但内存更紧凑，且 key 自动排序

## 12. 深度扩展

### 12.1 derive 宏生成 PartialEq 的实际代码

```rust
// 你写的：
#[derive(PartialEq)]
struct Point { x: i32, y: i32 }

// derive 宏生成的（近似）：
impl PartialEq for Point {
    fn eq(&self, other: &Point) -> bool {
        self.x == other.x && self.y == other.y
    }
    fn ne(&self, other: &Point) -> bool {
        self.x != other.x || self.y != other.y
    }
}

// 对于有大量字段的 struct，derive 生成的代码
// 会做短路求值（&& 的短路特性），先比较"便宜的"字段
```

`ne` 的默认实现是 `!eq()`，但 derive 宏会生成独立的 `ne` 实现以避免双重否定（性能微观优化）。

### 12.2 Rust 1.62+ 的 `total_cmp` 方法

```rust
// 传统方式：sort_by + partial_cmp
v.sort_by(|a, b| a.partial_cmp(b).unwrap_or(Ordering::Equal));

// Rust 1.62+ 方式：total_cmp
v.sort_by(|a, b| a.total_cmp(b));
// 或者直接：
v.sort_by(f64::total_cmp);  // 更简洁！

// total_cmp 定义了完整排序：
// -NaN < -inf < -1.0 < -0.0 < 0.0 < 1.0 < inf < NaN
// 注意：-0.0 < 0.0，NaN 在最后
// 这满足 Ord 的所有要求（反射性、传递性、反对称性）
```

### 12.3 Hash 的 Dos 防护：RandomState

HashMap 默认使用 `RandomState` 而不是简单的哈希函数：

```rust
// 默认的 HashMap：
let map: HashMap<String, i32> = HashMap::new();
// 等价于：
let map: HashMap<String, i32, RandomState> = HashMap::new();

// 为什么用 RandomState？
// 1. 每次程序运行，哈希种子都不同
// 2. 防止攻击者利用哈希冲突发起 DoS 攻击
//    （如果种子固定，攻击者可以构造大量相同哈希的 key）

// 如果你想自己提供种子（比如为了可复现的行为）：
use std::collections::hash_map::DefaultHasher;
let map: HashMap<String, i32, std::collections::hash_map::BuildHasherDefault<DefaultHasher>>
    = HashMap::default();
```

### 12.4 f32/f64 二进制语义 vs 数学语义

```rust
// 浮点数的"特殊值"全景图：
//  正零 (+0.0)  → 0x0000_0000_0000_0000
//  负零 (-0.0)  → 0x8000_0000_0000_0000
//  正无穷       → 0x7FF0_0000_0000_0000
//  负无穷       → 0xFFF0_0000_0000_0000
//  NaN          → 指数全 1，尾数非零（多种位表示）

// PartialEq 的数学语义：
assert!(0.0 == -0.0);       // 数学上相等
assert!(!f64::NAN.is_nan() || f64::NAN != f64::NAN);  // NaN 数学上不定义相等

// 位级别比较（用于排序）：
assert_ne!(0.0_f64.to_bits(), (-0.0_f64).to_bits());  // 位不同
assert!(!f64::NAN.to_bits() != f64::NAN.to_bits());    // NaN 的位可能相同
```

这就是 `total_cmp` 要解决的问题——它用 IEEE 754 的 totalOrder 语义，处理了位级别和数学级别的不一致。

### 12.5 PartialOrd 和 Ord 在泛型中的传播

```rust
// 比较的泛型约束传播——"缺一个全链崩溃"
fn find_max<T: Ord>(slice: &[T]) -> Option<&T> {
    slice.iter().max()  // max() 需要 Ord
}

struct Team {
    members: Vec<String>,
    score: f64,  // ← f64 不是 Ord！
}

// find_max(&teams)   ❌ Team 没有实现 Ord（因为 f64 没有）

// 解决：用 sort_by 或为 Team 手动实现 Ord（在比较 score 时处理 NaN）
```

### 12.6 为什么 Rust 的 BTreeMap 用 Ord 而不是 Hash

这是一个精妙的设计选择：

```
BTreeMap (基于 B-Tree)         HashMap (基于哈希表)
──────────────────────────────────────────────────
内存：连续分配，缓存友好         内存：桶分散，对缓存不友好
插入：O(log n)                  插入：O(1) 平均
遍历：按键顺序，O(n)            遍历：无序，O(n)
Key 要求：Ord                   Key 要求：Hash + Eq

Rust 提供了两种选择，各有用武之地：
- 需要按序遍历 → BTreeMap
- 需要最快随机访问 → HashMap
- Key 难以实现 Hash → BTreeMap
- Key 难以实现 Ord → HashMap
```

### 12.7 `#[derive]` 的可控性：排除字段

```rust
// 有时候你希望 PartialEq 只看部分字段：
#[derive(PartialEq, Eq, Hash)]
struct CacheEntry {
    key: String,
    value: String,
    #[derive(PartialEq = "ignore")]  // 这只是伪代码——实际不支持！
    last_accessed: std::time::Instant,
}

// 实际上你需要手动实现：
struct CacheEntry {
    key: String,
    value: String,
    last_accessed: std::time::Instant,
}

impl PartialEq for CacheEntry {
    fn eq(&self, other: &Self) -> bool {
        self.key == other.key  // 只看 key，忽略 last_accessed
    }
}

impl Eq for CacheEntry { }

impl Hash for CacheEntry {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.key.hash(state);  // 同样只 hash key
    }
}
```

### 12.8 标准库的比较 trait 实现策略

标准库中，比较 trait 的 blanket impl 很少，但有一个非常重要的模式：

```rust
// 比较的"穿透"模式：容器类型的 PartialEq 依赖于元素类型的 PartialEq
impl<T: PartialEq> PartialEq for Vec<T> {
    fn eq(&self, other: &Vec<T>) -> bool {
        self.len() == other.len()
            && self.iter().zip(other).all(|(a, b)| a == b)
    }
}

// 但这个 blanket impl 不存在——为什么？
// impl<T: PartialEq, U: PartialEq> PartialEq<Vec<U>> for Vec<T> { }
// 因为它要求两个 Vec 的元素类型可以化约到同一个类型
// 而 Rust 不允许"为外部类型在外部的 trait bound 上这样做"

// 但标准库在特定类型上做了跨类型比较：
impl<T: PartialEq<U>, U, A: Allocator> PartialEq<Vec<U, A>> for VecDeque<T, A> { }
// Vec 和 VecDeque 可以互相比较！
```

## 13. 回顾

```
比较 trait 学习清单：

□ 理解 PartialEq（部分等价）和 Eq（完全等价）的语义差异
□ 知道为什么 f64 不是 Eq（NaN 破坏反射性）
□ 理解 PartialOrd（可能不可比较）和 Ord（全序）的区别
□ 掌握 Hash 和 Eq 的一致性契约（a == b → hash(a) == hash(b)）
□ 能识别违法比较 trait 契约的代码
□ 知道 BTreeMap vs HashMap 的 trait 需求差异
□ 掌握浮点数的 total_cmp 方法（Rust 1.62+）
□ 知道 derive 宏在比较 trait 上的安全性保证
```

> **一句话总结**：Rust 的比较体系用 `Partial` 和"完全"两层分离，在类型层面精确地模拟了"部分定义"的数学现实——浮点数 NaN 是这种设计最生动的注脚。理解这些 trait 的契约，你就能写出在集合、排序、去重等场景下绝不出现静默 bug 的代码。
