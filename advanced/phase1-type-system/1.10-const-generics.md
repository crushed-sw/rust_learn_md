# 1.10 Const Generics + 编译期计算

> 泛型可以是类型（`Vec<T>`），可以是生命周期（`&'a str`）。但从 Rust 1.51 开始，泛型还可以是**值**——编译期已知的常量。Const Generics 将编译期计算提升到了类型层面，开启了零成本的数组抽象、类型级整数运算和编译期验证的大门。

---

## 1. 问题：为什么需要 Const Generics

### 1.1 一个经典困境

```rust
// 你想写一个固定大小的数组运算：
fn dot_product(a: &[f64], b: &[f64]) -> f64 {
    assert_eq!(a.len(), b.len());
    a.iter().zip(b).map(|(x, y)| x * y).sum()
}

// 问题：长度检查在运行时！
// 如果长度不匹配，程序 panic——糟糕的用户体验
// 而且长度在编译期就已知了，为什么不能编译期检查？
```

### 1.2 Const Generics 的解法

```rust
// 长度成为编译期已知的常量——编译期就能检查！
fn dot_product<const N: usize>(a: &[f64; N], b: &[f64; N]) -> f64 {
    a.iter().zip(b).map(|(x, y)| x * y).sum()
}

fn main() {
    let a = [1.0, 2.0, 3.0];  // [f64; 3]
    let b = [4.0, 5.0, 6.0];  // [f64; 3]
    println!("{}", dot_product(&a, &b));  // ✅ 32.0

    let c = [1.0, 2.0];  // [f64; 2]
    // dot_product(&a, &c);  // ❌ 编译错误！长度不匹配
    //                        // expected `&[f64; 3]`, found `&[f64; 2]`
}
```

## 2. Const Generics 基础语法

### 2.1 声明与使用

```rust
// 语法：const 参数名: 类型
// 目前支持的类型：整数（有符号/无符号）、bool、char

// 基本形式：
fn make_array<const N: usize>(value: f64) -> [f64; N] {
    [value; N]
}

let arr: [f64; 5] = make_array(3.14);
assert_eq!(arr, [3.14; 5]);

// 多个 const 泛型：
fn make_matrix<const ROWS: usize, const COLS: usize>(
    value: f64
) -> [[f64; COLS]; ROWS] {
    [[value; COLS]; ROWS]
}

let matrix: [[f64; 3]; 2] = make_matrix(0.0);
```

### 2.2 在结构体中使用

```rust
// 编译期固定大小的 Vector
struct Vector<T, const N: usize> {
    data: [T; N],
}

impl<T: Default + Copy, const N: usize> Vector<T, N> {
    fn new() -> Self {
        Vector {
            data: [T::default(); N],
        }
    }

    fn from_array(arr: [T; N]) -> Self {
        Vector { data: arr }
    }
}

// 使用时指定具体大小
let v3: Vector<f64, 3> = Vector::new();
let v4 = Vector::from_array([1.0, 2.0, 3.0, 4.0]);  // N = 4 自动推断
```

### 2.3 在 trait 中使用

```rust
// trait 也可以带 const 泛型
trait FixedSize {
    const SIZE: usize;

    fn as_array(&self) -> &[u8; Self::SIZE];
}

// 为特定大小实现
struct Header {
    data: [u8; 16],
}

impl FixedSize for Header {
    const SIZE: usize = 16;

    fn as_array(&self) -> &[u8; 16] {
        &self.data
    }
}
```

## 3. Const Generics 的编译期计算

### 3.1 编译期表达式

```rust
// const 泛型参数可以在编译期参与运算
fn concat_arrays<const N: usize, const M: usize>(
    a: [i32; N],
    b: [i32; M],
) -> [i32; N + M] {
    //              ^^^^^ 编译期加法！
    let mut result = [0; N + M];
    let mut i = 0;
    while i < N {
        result[i] = a[i];
        i += 1;
    }
    while i < N + M {
        result[i] = b[i - N];
        i += 1;
    }
    result
}

fn main() {
    let a = [1, 2];
    let b = [3, 4, 5];
    let c: [i32; 5] = concat_arrays(a, b);
    assert_eq!(c, [1, 2, 3, 4, 5]);
}
```

### 3.2 编译期断言

```rust
// 在编译期验证不变量
fn safe_divide<const DIVISOR: i32>(value: i32) -> i32 {
    // 编译期断言——如果 DIVISOR == 0，编译失败！
    const { assert!(DIVISOR != 0, "Divisor must not be zero!"); }
    value / DIVISOR
}

fn main() {
    let x = safe_divide::<5>(100);    // ✅ 100 / 5 = 20
    // let y = safe_divide::<0>(100); // ❌ 编译错误！assertion failed
}
```

### 3.3 编译期布尔逻辑

```rust
// 编译期条件编译
fn read_bytes<const BIG_ENDIAN: bool>(data: &[u8]) -> u32 {
    if BIG_ENDIAN {
        u32::from_be_bytes([data[0], data[1], data[2], data[3]])
    } else {
        u32::from_le_bytes([data[0], data[1], data[2], data[3]])
    }
}

// 编译器会优化掉死分支——零运行时开销
let be: u32 = read_bytes::<true>(&[0x12, 0x34, 0x56, 0x78]);
let le: u32 = read_bytes::<false>(&[0x12, 0x34, 0x56, 0x78]);
```

## 4. 核心使用场景

### 4.1 场景 1：编译期大小保证的矩阵运算

```rust
use std::ops::{Add, Mul};

struct Matrix<T, const ROWS: usize, const COLS: usize> {
    data: [[T; COLS]; ROWS],
}

// 矩阵加法——大小在编译期验证
impl<T: Add<Output = T> + Copy, const R: usize, const C: usize>
    Add for Matrix<T, R, C>
{
    type Output = Matrix<T, R, C>;

    fn add(self, rhs: Self) -> Self::Output {
        let mut result = [[self.data[0][0]; C]; R];
        for i in 0..R {
            for j in 0..C {
                result[i][j] = self.data[i][j] + rhs.data[i][j];
            }
        }
        Matrix { data: result }
    }
}

// 矩阵乘法——维度在编译期验证
impl<T: Add<Output = T> + Mul<Output = T> + Copy + Default,
     const R: usize, const C: usize, const K: usize>
    Mul<Matrix<T, K, C>> for Matrix<T, R, K>
{
    type Output = Matrix<T, R, C>;

    fn mul(self, rhs: Matrix<T, K, C>) -> Self::Output {
        let mut result = [[T::default(); C]; R];
        for i in 0..R {
            for j in 0..C {
                for k in 0..K {
                    result[i][j] = result[i][j] + self.data[i][k] * rhs.data[k][j];
                }
            }
        }
        Matrix { data: result }
    }
}

fn main() {
    let a = Matrix::<i32, 2, 3> { data: [[1, 2, 3], [4, 5, 6]] };
    let b = Matrix::<i32, 3, 2> { data: [[7, 8], [9, 10], [11, 12]] };
    let c: Matrix<i32, 2, 2> = a * b;  // ✅ (2×3) * (3×2) = (2×2)
    // let d = a * a;  // ❌ 编译错误！维度不匹配！
}
```

### 4.2 场景 2：编译期验证的协议解析

```rust
// 解析固定格式的消息头
struct Message<const VERSION: u8, const TYPE: u8> {
    payload: Vec<u8>,
}

impl<const VERSION: u8, const TYPE: u8> Message<VERSION, TYPE> {
    fn new(payload: Vec<u8>) -> Self {
        // VERSION 和 TYPE 在编译期已知——
        // 序列化/反序列化时不需要运行时传递这些值
        Message { payload }
    }

    fn version(&self) -> u8 { VERSION }
    fn msg_type(&self) -> u8 { TYPE }
}

// 不同的消息类型是不同的类型：
type V1Request = Message<1, 0x01>;
type V1Response = Message<1, 0x02>;
type V2Request = Message<2, 0x01>;
// 类型安全！不可能混淆不同版本/类型的消息
```

### 4.3 场景 3：编译期大小的 Buffer

```rust
// 栈上分配的固定大小 buffer——无堆分配
struct StackBuffer<const CAPACITY: usize> {
    data: [u8; CAPACITY],
    len: usize,
}

impl<const CAPACITY: usize> StackBuffer<CAPACITY> {
    fn new() -> Self {
        StackBuffer {
            data: [0; CAPACITY],
            len: 0,
        }
    }

    fn push(&mut self, byte: u8) -> Result<(), &'static str> {
        if self.len >= CAPACITY {
            return Err("buffer full");
        }
        self.data[self.len] = byte;
        self.len += 1;
        Ok(())
    }

    fn as_slice(&self) -> &[u8] {
        &self.data[..self.len]
    }
}

// 使用：完全在栈上，无动态分配
let mut buf = StackBuffer::<1024>::new();
buf.push(b'H').unwrap();
buf.push(b'i').unwrap();
println!("{:?}", buf.as_slice());  // [72, 105]
```

## 5. Const Generics 的当前限制

### 5.1 支持的类型有限

```rust
// ✅ 支持的 const 泛型类型（Rust 1.85）：
// i8, i16, i32, i64, i128, isize
// u8, u16, u32, u64, u128, usize
// bool
// char

// ❌ 不支持的：
// f32, f64（浮点数）
// &str, String（字符串）
// 自定义类型（即使 derive(PartialEq, Eq)）
// const fn 返回值（实验性）

// ❌ 浮点数不能用：
// struct FloatVector<const N: f64> { }  // 编译错误！
```

### 5.2 不能使用泛型常量表达式（部分场景）

```rust
// ✅ 简单的算术表达式可以：
fn foo<const N: usize>() -> [i32; N + 1] { }

// ✅ const 块（Rust 1.79+）可以：
fn bar<const N: usize>() -> [i32; { N * 2 + 1 }] { }

// ❌ 复杂的泛型常量表达式受限：
// fn baz<const N: usize, const M: usize>() -> [i32; N * M] { }
// 某些版本中 N * M 需要 const 块：{ N * M }

// ✅ 使用 const 块解决：
fn baz<const N: usize, const M: usize>() -> [i32; { N * M }] { }
```

### 5.3 不能在 function pointer 位置使用

```rust
// ❌ const 泛型不能出现在函数指针中：
// type MyFn = fn<const N: usize>(input: [i32; N]) -> i32;  // 不支持

// ✅ 只能用泛型函数：
fn process<const N: usize>(input: [i32; N]) -> i32 {
    input.iter().sum()
}
```

## 6. 设计哲学：Const Generics 解决的核心矛盾

### 6.1 C++ 的对应方案及问题

C++ 有直接的对应物——非类型模板参数（non-type template parameters）：

```cpp
// C++：非类型模板参数
template <typename T, int N>
struct Array {
    T data[N];
    constexpr int size() const { return N; }
};

Array<int, 5> arr;  // 等价于 Rust 的 [i32; 5]

// 还可以做编译期计算：
template <int N, int M>
Array<int, N + M> concat(const Array<int, N>& a, const Array<int, M>& b) {
    Array<int, N + M> result;
    // ...
    return result;
}
```

**C++ 的问题不是"能不能做"——而是"做得不干净"**：

```cpp
// 问题 1：int 作为模板参数的语义模糊
template <int N>   // N 可以是负数、可以是零——但数组大小不能是负数！
void foo() {
    int arr[N];    // 如果 N <= 0，行为未定义或编译器扩展
}
// Rust：const N: usize —— usize 天然非负，编译器阻止不合理场景

// 问题 2：错误信息。C++ 的模板错误是出了名的难以阅读：
// template argument deduction/substitution failed:
//   200 lines of template instantiation backtrace...
// Rust：error[E0308]: mismatched types: expected `[f64; 3]`, found `[f64; 2]`
//       ——直接、清晰

// 问题 3：什么能做非类型模板参数？
// C++：整数类型、枚举、指针、（C++20）浮点数、类类型——但各编译器支持不同
// Rust：明确的列表（整数、bool、char）——行为统一
```

### 6.2 C++ 的 `std::array` vs Rust 的 `[T; N]`：两种哲学

```cpp
// C++：std::array 是一个库类型（用模板实现的 wrapper）
std::array<int, 5> arr = {1, 2, 3, 4, 5};
// 优势：有自己的 .size()、.begin()、.end() 等方法
// 劣势：和 C 数组 (int[5]) 不是同一个类型，需要转换

// Rust：[i32; 5] 是编译器内置的原生类型
let arr: [i32; 5] = [1, 2, 3, 4, 5];
// 优势：和切片 [i32] 自然关联，&[i32; 5] 自动强制转换为 &[i32]
// 劣势：数组方法较少（但可以通过切片获得）
```

这两种设计反映了不同的哲学：
- C++：尽可能用库解决问题（`std::array` 是库，不是语言特性）
- Rust：把"编译期已知大小"提升为语言的一等公民（`[T; N]` 是编译器原生类型）

### 6.3 如果没有 Const Generics —— Rust 1.51 之前的世界

Rust 的 const generics 在 2021 年 3 月（1.51）才稳定。在此之前，生态用了 **typenum** + **generic-array** 做类型级别的整数运算：

```rust
// typenum crate：用类型系统模拟整数
use typenum::{U1, U2, U3, U5, U8};
use generic_array::GenericArray;

// "U3 + U5 = U8" 是类型级别的计算（编译器推导）
type Sum = <U3 as std::ops::Add<U5>>::Output;  // = U8

// GenericArray 用类型参数表示数组大小
let arr: GenericArray<i32, U5> = GenericArray::from([1, 2, 3, 4, 5]);

// 这种做法的痛苦：
// 1. 每个数字需要单独的类型（U0, U1, U2, ..., U255, ...）——类型爆炸
// 2. trait 约束极其复杂：where T: Add<U3, Output = U5>
// 3. 编译时间非常长（类型计算在编译期跑，大整数很慢）
// 4. 错误信息不可读——"expected U8, found <U3 as Add<U5>>::Output"
// 5. 只能到几百——typenum 不可能定义 U10000 类型
```

**Const Generics 解决了 typenum 的所有这些问题**：

```rust
// 现在——直接、自然、快速
fn concat<const N: usize, const M: usize>(a: [u8; N], b: [u8; M]) -> [u8; N + M] {
    let mut result = [0u8; N + M];  // { N + M } ——自然
    result[..N].copy_from_slice(&a);
    result[N..].copy_from_slice(&b);
    result
}
// N 和 M 可以是任何 usize 值——不需要为每个数字定义类型
// 编译速度：类型级别的简单整数运算 vs typenum 的 trait 解析——百倍差距
```

### 6.4 设计决策总结

| 决策 | Rust | C++ | 评判 |
|:---|:---|:---|:---|
| 支持的类型 | 整数、bool、char | 整数、枚举、指针、（C++20 浮点/类）| Rust 保守但安全；C++ 功能强但不统一 |
| 编译期表达式 | `N + M`（简单算术） | `N + M`（constexpr 更强大）| C++ constexpr 可以做几乎任何事，Rust 还在追赶 |
| 浮点支持 | ❌（稳定性考虑） | ✅（C++20，各编译器行为微妙不同）| Rust 选择保守——NaN 问题没有满意的类型层面解决方法 |
| 自定义类型 | ❌（未来可能） | ✅（C++20） | C++ 领先——但"什么类型可以做模板参数"的规则极其复杂 |
| 与 trait 集成 | ✅ 自然——const 泛型就是类型的一部分 | ⚠️ 模板和 concepts 是两个体系 | Rust 的 trait + const generics 更统一 |

### 6.5 核心哲学

```
"如果某个值在编译期就知道，那就让编译器利用它。"
这消除了大量不必要的运行时检查，把错误从"运行 panic"
提升为"编译错误"。

C++ 的哲学是"模板可以做任何事"——功能强大但语义边界模糊。
Rust 的哲学是"const generics 做该做的事"——功能克制但语义清晰。

两种哲学的选择：当你写系统软件时，你宁愿"少一些功能但保证正确"，
而不是"无所不能但可能在某个角落藏着 UB"。
```

## 7. 常见陷阱

### 陷阱 1：const 泛型不等于运行时变量

```rust
// ❌ 不能从运行时输入构造 const 泛型：
fn create_array(size: usize) {
    // let arr: [i32; size] = [0; size];  // ❌ size 是运行时值
    // 编译器不知道 size 是多少——无法确定 arr 的类型
}

// ✅ 必须是编译期已知的常量：
fn create_array<const N: usize>() -> [i32; N] {
    [0; N]
}
let arr: [i32; 5] = create_array();  // N = 5 在编译期确定
```

### 陷阱 2：const 泛型的单态化爆炸

```rust
// 每个不同的 N 值会生成一份独立的代码！
fn foo<const N: usize>() -> [i32; N] {
    [0; N]
}

// foo::<1>()、foo::<2>()、... foo::<1000>()
// → 生成 1000 份代码 → 二进制膨胀！

// 缓解方案：把与 N 无关的逻辑提取到非泛型函数
fn foo<const N: usize>() -> [i32; N] {
    let arr = [0; N];
    do_something_with_slice(&arr);  // 提取到非泛型函数
    arr
}

fn do_something_with_slice(data: &[i32]) {
    // 这里是大小无关的逻辑——不会单态化爆炸
}
```

### 陷阱 3：类型推断的 const 泛型限制

```rust
// ✅ 可以从上下文推断：
let arr: [i32; 3] = [1, 2, 3];
let v = Vector::from_array(arr);  // N = 3 自动推断

// ⚠️ 但有时需要 turbofish：
let v = Vector::<i32, 3>::new();
//               ^^^^^ 需要显式指定 N
```

## 9. 练习答案

### 练习 1 答案

```rust
fn transmute_array<const N: usize, const M: usize>(bytes: [u8; N]) -> [u32; M] {
    // 编译期断言
    const { assert!(N == M * 4, "Size mismatch: input must be 4x output"); }

    let mut result = [0u32; M];
    for i in 0..M {
        let idx = i * 4;
        result[i] = u32::from_le_bytes([
            bytes[idx],
            bytes[idx + 1],
            bytes[idx + 2],
            bytes[idx + 3],
        ]);
    }
    result
}

fn main() {
    let bytes = [0x01, 0x00, 0x00, 0x00, 0x02, 0x00, 0x00, 0x00];
    let ints: [u32; 2] = transmute_array(bytes);
    assert_eq!(ints, [1, 2]);

    // let bad: [u32; 1] = transmute_array([0u8; 3]);  // ❌ 编译错误！
}
```

### 练习 2 答案

```rust
struct Packet<const ID: u8, const PAYLOAD_SIZE: usize> {
    payload: [u8; PAYLOAD_SIZE],
}

impl<const ID: u8, const PAYLOAD_SIZE: usize> Packet<ID, PAYLOAD_SIZE> {
    fn new(payload: [u8; PAYLOAD_SIZE]) -> Self {
        Packet { payload }
    }

    fn serialize(&self) -> [u8; { PAYLOAD_SIZE + 1 }] {
        let mut result = [0u8; { PAYLOAD_SIZE + 1 }];
        result[0] = ID;
        result[1..].copy_from_slice(&self.payload);
        result
    }

    fn deserialize(data: &[u8]) -> Option<Self> {
        if data.len() != PAYLOAD_SIZE + 1 {
            return None;
        }
        if data[0] != ID {
            return None;
        }
        let mut payload = [0u8; PAYLOAD_SIZE];
        payload.copy_from_slice(&data[1..]);
        Some(Packet { payload })
    }
}

// 类型安全的协议定义
type Heartbeat = Packet<0x01, 0>;
type DataPacket = Packet<0x02, 64>;
```

### 练习 3 答案

```rust
use std::ops::Index;

struct StackVec<T, const CAP: usize> {
    data: [std::mem::MaybeUninit<T>; CAP],
    len: usize,
}

impl<T, const CAP: usize> StackVec<T, CAP> {
    fn new() -> Self {
        StackVec {
            data: unsafe { std::mem::MaybeUninit::uninit().assume_init() },
            len: 0,
        }
    }

    fn push(&mut self, value: T) -> Result<(), T> {
        if self.len >= CAP {
            return Err(value);
        }
        self.data[self.len].write(value);
        self.len += 1;
        Ok(())
    }

    fn pop(&mut self) -> Option<T> {
        if self.len == 0 {
            return None;
        }
        self.len -= 1;
        Some(unsafe { self.data[self.len].assume_init_read() })
    }

    fn len(&self) -> usize { self.len }
    fn is_empty(&self) -> bool { self.len == 0 }
    fn as_slice(&self) -> &[T] {
        unsafe {
            std::slice::from_raw_parts(
                self.data.as_ptr() as *const T,
                self.len,
            )
        }
    }
}

impl<T, const CAP: usize> Index<usize> for StackVec<T, CAP> {
    type Output = T;

    fn index(&self, index: usize) -> &T {
        assert!(index < self.len, "index out of bounds");
        unsafe { self.data[index].assume_init_ref() }
    }
}

impl<T, const CAP: usize> Drop for StackVec<T, CAP> {
    fn drop(&mut self) {
        for i in 0..self.len {
            unsafe { self.data[i].assume_init_drop() };
        }
    }
}
```

### 练习 4 答案

**浮点数不能作为 const 泛型参数的核心原因：**

1. **NaN 的等价性问题**：`NaN != NaN`。如果 `f32::NAN` 可以做 const 泛型参数，那么 `Foo::<f32::NAN>` 和 `Foo::<f32::NAN>` 是同一个类型吗？类型等价性要求"相同的参数 → 相同的类型"，但 NaN 不满足 `==` 的等价性。

2. **多表示性**：`0.0_f64` 和 `-0.0_f64` 在浮点语义上是不同的（`0.0 == -0.0` 但位模式不同）。类型等价性应该基于位模式还是数学相等？两难。

3. **精度问题**：`0.1 + 0.2 != 0.3` 在浮点运算中。这会导致 `Foo::<0.3>` 和 `Foo::<0.1 + 0.2>` 是不同类型——极端反直觉。

4. **编译期确定性**：浮点运算在不同平台上的结果可能有微小差异（特别是超越函数）。Rust 要求编译期计算是确定性的——浮点不满足。

```rust
// 如果浮点支持了——这些歧义怎么处理？
// Foo::<0.0> 和 Foo::<-0.0> 是同一个类型吗？
// Foo::<f64::NAN> 和 Foo::<f64::NAN> 是同一个类型吗？
// Foo::<0.1 + 0.2> 和 Foo::<0.3> 是同一个类型吗？
```

### 10.1 单态化——Const Generics 的底层实现

### 10.2 `const` 块与编译期计算（Rust 1.79+）

```rust
// const 块允许在类型位置进行编译期计算：
fn create_buffer<const N: usize>() -> [u8; { N * 1024 }] {
    [0; { N * 1024 }]
}

// 编译期条件：
fn maybe_include_checksum<const INCLUDE_CHECKSUM: bool>(data: &[u8]) -> Vec<u8> {
    let mut result = data.to_vec();
    if INCLUDE_CHECKSUM {
        let checksum = data.iter().fold(0u8, |a, b| a.wrapping_add(*b));
        result.push(checksum);
    }
    result  // 编译器优化：死分支消除——零运行时开销
}
```

### 10.3 类型级别的 Peano 自然数

**Peano 自然数**是一种用"后继"（successor）来定义自然数的数学方法：0 是自然数；如果 n 是自然数，那么 S(n)（读作"n 的后继"）也是自然数。用这种方式，1 = S(0)，2 = S(S(0))，以此类推。

在 Rust 的类型系统中，我们可以用**零大小的结构体 + const generics**来模拟这种编码。核心思路：
- `Zero` 代表 0——一个空结构体
- `Succ<N>` 代表"上一个数加 1"——N 是一个 const generic
- `Nat` trait 定义了从类型中提取数值的 `VALUE` 常量
- `Add` trait 在类型层面实现了加法

**这有什么用？** 在你无法通过普通泛型表达"这个函数的返回值长度等于两个输入长度之和"时，类型级别的算术让你可以用类型标记长度关系。比如：合并两个数组后，返回类型是 `[T; N+M]`——编译器可以验证这一点。

下面的代码展示了这个模式：

```rust
// 用 const generics 在类型层面编码自然数
struct Succ<const N: usize>;
struct Zero;

trait Nat {
    const VALUE: usize;
}

impl Nat for Zero {
    const VALUE: usize = 0;
}

impl<const N: usize> Nat for Succ<N> {
    const VALUE: usize = N + 1;
}

// 类型级别的加法：
trait Add<Other> {
    type Output;
}

impl<const N: usize, const M: usize> Add<Succ<M>> for Succ<N>
where
    Succ<{ N + M }>: Nat,
{
    type Output = Succ<{ N + M }>;
}

// 编译期验证：
fn test_type_level_math() {
    // 编译器知道 Succ<Succ<Zero>>::VALUE = 2
    let _: Succ<Succ<Zero>>;
}
```
