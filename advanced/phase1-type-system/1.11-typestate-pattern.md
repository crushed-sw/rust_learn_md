# 1.11 Typestate 模式——用类型编码状态

> "这个连接还没认证，你不能调用 `send_data`。"——如果能用**类型**而不是断言来表达这条规则，编译器就会在你犯错之前阻止你。Typestate 模式正是做这件事：把对象的状态编码到类型参数中，让非法的状态转换变成**类型错误**。这是 Rust 类型系统最优雅的应用之一——零运行时开销的状态机验证。

---

## 1. 问题：状态验证的三种方式

### 1.1 方式 A：运行时检查（最弱）

```rust
enum ConnectionState {
    Unauthenticated,
    Authenticated,
}

struct Connection {
    state: ConnectionState,
    token: Option<String>,
}

impl Connection {
    fn authenticate(&mut self, password: &str) -> Result<(), String> {
        if password == "secret" {
            self.state = ConnectionState::Authenticated;
            self.token = Some("token_abc".into());
            Ok(())
        } else {
            Err("wrong password".into())
        }
    }

    fn send_data(&self, data: &[u8]) -> Result<(), String> {
        match self.state {
            ConnectionState::Authenticated => {
                // 发送数据...
                Ok(())
            }
            ConnectionState::Unauthenticated => {
                Err("not authenticated!".into())
            }
        }
    }
}

// 问题：send_data 在未认证时也可以"调用"——只是运行时返回 Err
let conn = Connection { state: ConnectionState::Unauthenticated, token: None };
// conn.send_data(b"hello").unwrap();  // panic 在运行时！
```

### 1.2 方式 B：Typestate 模式（类型层面）

```rust
use std::marker::PhantomData;

// 状态类型——零大小，只存在于编译期
struct Unauthenticated;
struct Authenticated { token: String }

// 核心：用泛型参数标记状态
struct Connection<State> {
    server_addr: String,
    _state: PhantomData<State>,
}

// 为"未认证"状态实现的方法
impl Connection<Unauthenticated> {
    fn new(addr: &str) -> Self {
        Connection {
            server_addr: addr.into(),
            _state: PhantomData,
        }
    }

    fn authenticate(self, password: &str) -> Result<Connection<Authenticated>, String> {
        if password == "secret" {
            Ok(Connection {
                server_addr: self.server_addr,
                _state: PhantomData,
            })
        } else {
            Err("wrong password".into())
        }
    }
}

// 只为"已认证"状态实现的方法
impl Connection<Authenticated> {
    fn send_data(&self, data: &[u8]) {
        // 编译期保证：只有已认证的连接才能走到这里
        println!("Sending {} bytes to {}", data.len(), self.server_addr);
    }

    fn logout(self) -> Connection<Unauthenticated> {
        Connection {
            server_addr: self.server_addr,
            _state: PhantomData,
        }
    }
}

// ✅ 使用：
let conn = Connection::new("192.168.1.1:8080");
// conn.send_data(b"hello");  // ❌ 编译错误！
//   no method named `send_data` found for struct `Connection<Unauthenticated>`

let conn = conn.authenticate("secret").unwrap();
conn.send_data(b"hello");  // ✅ 编译通过——已认证
let conn = conn.logout();
// conn.send_data(b"hello");  // ❌ 又编译错误了——已登出
```

**核心优势**：错误从"运行时崩溃"变成了"编译不过"。这不是 lint 警告——是类型系统的硬性约束。

## 2. Typestate 模式的核心机制

### 2.1 状态作为零大小的类型标记

```rust
// 状态类型通常不持有数据——它们只做类型标记
struct Draft;
struct Published;
struct Archived;

// 如果有状态关联数据，放在泛型结构体中
struct Document<State> {
    content: String,
    author: String,
    state_data: State,  // 状态数据存在这里而不是 PhantomData
}
```

### 2.2 状态转换消耗 self

```rust
impl Document<Draft> {
    fn new(content: String, author: String) -> Self {
        Document { content, author, state_data: Draft }
    }

    // ✅ consume self → 旧状态消失，新状态诞生
    fn publish(self) -> Document<Published> {
        Document {
            content: self.content,
            author: self.author,
            state_data: Published {
                published_at: std::time::SystemTime::now(),
            },
        }
    }
}

struct Published {
    published_at: std::time::SystemTime,
}

impl Document<Published> {
    // consume self → 旧状态消失
    fn archive(self) -> Document<Archived> {
        Document {
            content: self.content,
            author: self.author,
            state_data: Archived {
                archived_at: std::time::SystemTime::now(),
            },
        }
    }
}

struct Archived {
    archived_at: std::time::SystemTime,
}
```

**为什么必须 consume self？**

```rust
// 如果不用 self 而用 &mut self：
impl Document<Draft> {
    fn publish(&mut self) {
        // 如果改成 &mut self...类型怎么变？
        // Document<Draft> 不能原地变成 Document<Published>
        // ——它们是不同类型！内存布局可能不同（Published 多了字段）
        // 所以必须消耗旧值、构造新值
    }
}
```

### 2.3 共享方法——跨状态的 impl

```rust
// 所有状态都可以调用的方法——用泛型 impl
impl<State> Document<State> {
    fn author(&self) -> &str {
        &self.author
    }

    fn content(&self) -> &str {
        &self.content
    }
}

let draft = Document::new("hello".into(), "Alice".into());
println!("by {}", draft.author());  // ✅ 草稿也能查作者
let published = draft.publish();
println!("by {}", published.author());  // ✅ 已发布也能查
```

## 3. PhantomData 在 Typestate 中的精确角色

### 3.1 什么时候需要 PhantomData

```rust
// 情况 1：状态数据直接存在结构体中——不需要 PhantomData
struct Document<State> {
    content: String,
    state_data: State,  // State 被实际使用，编译器知道 T 被占有了
}

// 情况 2：状态只是标记，没有状态数据——需要 PhantomData
struct Connection<State> {
    addr: String,
    _marker: PhantomData<State>,  // "假装"占有 State
    //                            // 没有这一行，编译报错：
    //                            // parameter `State` is never used
}

// 情况 3：状态在不同变体中表达——需要 PhantomData
enum ProtocolState<S> {
    Idle(PhantomData<S>),
    Connecting { attempts: u32, _marker: PhantomData<S> },
    Connected { session_id: u64, _marker: PhantomData<S> },
}
```

### 3.2 PhantomData 不占空间

```rust
use std::mem;

struct Connection<State> {
    addr: String,
    _marker: PhantomData<State>,
}

// PhantomData 是零大小类型（ZST）
assert_eq!(mem::size_of::<Connection<()>>(), mem::size_of::<String>());
// Connection 的大小只包含 String，没有 PhantomData 的开销

// ✅ 所以 Typestate 是真正的"零运行时开销抽象"
```

## 4. Builder 模式——Typestate 的杀手级应用

### 4.1 传统 Builder 的问题

```rust
// 传统 Builder（运行时检查）：
struct Config {
    host: String,
    port: u16,
    username: String,
    password: String,
}

struct ConfigBuilder {
    host: Option<String>,
    port: Option<u16>,
    username: Option<String>,
    password: Option<String>,
}

impl ConfigBuilder {
    fn new() -> Self {
        ConfigBuilder { host: None, port: None, username: None, password: None }
    }

    fn host(mut self, h: &str) -> Self { self.host = Some(h.into()); self }
    fn port(mut self, p: u16) -> Self { self.port = Some(p); self }
    fn username(mut self, u: &str) -> Self { self.username = Some(u.into()); self }
    fn password(mut self, p: &str) -> Self { self.password = Some(p.into()); self }

    fn build(self) -> Result<Config, &'static str> {
        // 运行时检查：所有字段都填了吗？
        Ok(Config {
            host: self.host.ok_or("host missing")?,
            port: self.port.ok_or("port missing")?,
            username: self.username.ok_or("username missing")?,
            password: self.password.ok_or("password missing")?,
        })
    }
}
```

### 4.2 Typestate Builder——编译期保证完整性

```rust
// 每个"缺失的字段"是一个状态类型
struct MissingHost;
struct MissingPort;
struct MissingUsername;
struct MissingPassword;

// Builder 有两个泛型参数：P（port 有没有）、H（host 有没有）
struct ConfigBuilder<H, P, U, W> {
    host: Option<String>,
    port: Option<u16>,
    username: Option<String>,
    password: Option<String>,
    _state: PhantomData<(H, P, U, W)>,
}

// 初始状态——所有字段都缺失
impl ConfigBuilder<MissingHost, MissingPort, MissingUsername, MissingPassword> {
    fn new() -> Self {
        ConfigBuilder {
            host: None, port: None, username: None, password: None,
            _state: PhantomData,
        }
    }
}

// 设置 host 后——H 从 MissingHost → 具体类型
struct HostSet;

impl<P, U, W> ConfigBuilder<MissingHost, P, U, W> {
    fn host(mut self, h: &str) -> ConfigBuilder<HostSet, P, U, W> {
        self.host = Some(h.into());
        ConfigBuilder {
            host: self.host,
            port: self.port,
            username: self.username,
            password: self.password,
            _state: PhantomData,
        }
    }
}

struct PortSet;

impl<H, U, W> ConfigBuilder<H, MissingPort, U, W> {
    fn port(mut self, p: u16) -> ConfigBuilder<H, PortSet, U, W> {
        self.port = Some(p);
        ConfigBuilder {
            host: self.host,
            port: self.port,
            username: self.username,
            password: self.password,
            _state: PhantomData,
        }
    }
}

struct UsernameSet;

impl<H, P, W> ConfigBuilder<H, P, MissingUsername, W> {
    fn username(mut self, u: &str) -> ConfigBuilder<H, P, UsernameSet, W> {
        self.username = Some(u.into());
        ConfigBuilder {
            host: self.host,
            port: self.port,
            username: self.username,
            password: self.password,
            _state: PhantomData,
        }
    }
}

struct PasswordSet;

impl<H, P, U> ConfigBuilder<H, P, U, MissingPassword> {
    fn password(mut self, p: &str) -> ConfigBuilder<H, P, U, PasswordSet> {
        self.password = Some(p.into());
        ConfigBuilder {
            host: self.host,
            port: self.port,
            username: self.username,
            password: self.password,
            _state: PhantomData,
        }
    }
}

// build 方法：只有所有字段都"Set"才能调用
impl ConfigBuilder<HostSet, PortSet, UsernameSet, PasswordSet> {
    fn build(self) -> Config {
        Config {
            host: self.host.unwrap(),
            port: self.port.unwrap(),
            username: self.username.unwrap(),
            password: self.password.unwrap(),
        }
    }
}

struct Config {
    host: String,
    port: u16,
    username: String,
    password: String,
}

// ✅ 使用：
let config = ConfigBuilder::new()
    .host("example.com")
    .port(8080)
    .username("admin")
    .password("secret")
    .build();  // ✅ 编译通过

// let config = ConfigBuilder::new()
//     .host("example.com")
//     // .port(8080)  ← 忘了！
//     .username("admin")
//     .password("secret")
//     .build();  // ❌ 编译错误：no method `build`——
//                //    port 字段还是 MissingPort，build 不可用！
```

### 4.3 Builder 的简化版本——泛型布尔标记

```rust
// 对于简单场景，用布尔类型参数就够：
struct Server<const AUTHENTICATED: bool> {
    addr: String,
}

impl Server<false> {
    fn new(addr: &str) -> Self { Server { addr: addr.into() } }
    fn authenticate(self, token: &str) -> Server<true> {
        println!("auth with {}", token);
        Server { addr: self.addr }
    }
}

impl Server<true> {
    fn request(&self, path: &str) -> String {
        format!("GET {}{}", self.addr, path)

}

let s = Server::new("http://api.example.com");
// s.request("/users");  // ❌ 编译错误
let s = s.authenticate("bearer xxx");
s.request("/users");     // ✅ 编译通过
```

## 5. 协议状态机——Typestate 的高级应用

### 5.1 TCP 连接的状态建模

```rust
use std::marker::PhantomData;

// 状态定义
struct Closed;
struct Listening;
struct SynSent;
struct Established;
struct FinWait1;
struct FinWait2;
struct TimeWait;

struct TcpConnection<State> {
    socket_fd: i32,
    _state: PhantomData<State>,
}

// Closed → Listen
impl TcpConnection<Closed> {
    fn new() -> Self {
        TcpConnection { socket_fd: -1, _state: PhantomData }
    }

    fn listen(self, port: u16) -> TcpConnection<Listening> {
        // bind + listen syscall
        TcpConnection { socket_fd: 3, _state: PhantomData }
    }
}

// Listening → SynSent
impl TcpConnection<Listening> {
    fn connect(self, addr: &str) -> TcpConnection<SynSent> {
        // send SYN
        TcpConnection { socket_fd: self.socket_fd, _state: PhantomData }
    }
}

// SynSent → Established
impl TcpConnection<SynSent> {
    fn complete_handshake(self) -> TcpConnection<Established> {
        TcpConnection { socket_fd: self.socket_fd, _state: PhantomData }
    }
}

// Established → FinWait1
impl TcpConnection<Established> {
    fn send_data(&self, data: &[u8]) { /* ... */ }
    fn recv_data(&self) -> Vec<u8> { /* ... */ vec![] }
    fn close(self) -> TcpConnection<FinWait1> {
        TcpConnection { socket_fd: self.socket_fd, _state: PhantomData }
    }
}

// 只有在 Established 状态下才能收发数据
// 编译期保证：
// - 不能在 Closed 状态下 send_data
// - 不能在 FinWait1 状态下 recv_data
// - 不能跳过 Listen 直接进入 Established
```

### 5.2 整个转换图变成了类型图

```
     ┌─────────┐
     │  Closed  │
     └────┬─────┘
          │ listen()
     ┌────▼─────┐
     │Listening │
     └────┬─────┘
          │ connect()
     ┌────▼─────┐          编译期保证了：
     │ SynSent  │          没有"非法边"
     └────┬─────┘
          │ complete_handshake()
     ┌────▼──────┐
     │Established│ ← 只有这里能 send_data/recv_data
     └────┬──────┘
          │ close()
     ┌────▼─────┐
     │FinWait1  │
     └──────────┘
```

## 6. 常见陷阱

### 陷阱 1：泛型爆炸

```rust
// ❌ 太多泛型参数让类型签名难以阅读
struct Document<Visibility, Review, Publish, Archive> {
    // ...4 个泛型参数，而且它们之间还有依赖关系
}

// ✅ 用模块化的状态类型
enum DocumentState {
    Draft(DraftData),
    InReview(ReviewData),
    Published(PublishData),
    Archived(ArchiveData),
}

// 如果状态之间的转换不是简单的线性流程，
// 枚举可能比 typestate 更合适
```

### 陷阱 2：过度设计

```rust
// ❌ 为一个简单的布尔标志用 Typestate
struct Connection<State> { /* ... */ }
// 当你只有一个 Authenticated/Unauthenticated 两种状态时，
// 用两个独立的 struct 可能更清晰：
struct UnauthenticatedConnection { /* ... */ }
struct AuthenticatedConnection { /* ... */ }
```

### 陷阱 3：状态数据的移动

```rust
// ❌ 容易在状态转换时丢失数据
impl Document<Draft> {
    fn publish(self) -> Document<Published> {
        Document {
            content: self.content,
            author: self.author,
            // 忘了复制 comments 字段？编译器不会提醒你
            state_data: Published { published_at: now() },
        }
    }
}

// ✅ 使用结构体更新语法减少遗漏风险：
fn publish(self) -> Document<Published> {
    Document {
        state_data: Published { published_at: now() },
        ..self  // ❌ 也不行——因为类型不同（Document<Draft> vs Document<Published>）
    }
    // 实际上你需要手动搬运每个字段
    // 这是 typestate 的一个固有痛点
}
```

### 陷阱 4：和枚举的取舍

```rust
// ❌ 不适合 Typestate 的情况：
// - 状态数很多（>5-6 种）
// - 状态转换不构成清晰的 DAG（有循环）
// - 需要在运行时动态决定下一个状态
// - 需要存储"当前状态未知"的对象

// ✅ 适合 Typestate 的情况：
// - 状态数少（2-4 种）
// - 状态转换是单向的、线性的
// - 状态在编译期就能确定
// - Builder、Protocol、生命周期阶段
```

## 7. 练习

### 练习 1：实现 HTTP 请求的 Typestate Builder

为 HTTP 请求实现一个 typestate builder，要求：
- 必须设置 method、url 后才能设置 body
- 必须设置 method、url 后才能发送请求
- body 是可选的

```rust
// 目标 API：
let response = HttpRequest::new()
    .method("POST")
    .url("https://api.example.com/users")
    .body(r#"{"name": "Alice"}"#)
    .send();
```

### 练习 2：文件操作的 Typestate

为文件操作建模 typestate：

```
Closed → open() → Opened → read()/write() → Opened
Opened → close() → Closed
```

- `Closed` 状态只能 open
- `Opened` 状态可以 read/write/close
- 不允许对 Closed 文件 read/write

### 练习 3：Typestate vs Enum 选择

以下场景，判断应该用 Typestate 还是 Enum，并说明理由：

(a) 数据库连接池：Idle → InUse → Idle（循环）
(b) 订单：Created → Paid → Shipped → Delivered（单向）
(c) 游戏角色：Idle → Running → Jumping → Attacking（任意跳转）
(d) 安装向导：第1步 → 第2步 → 第3步 → 完成（单向线性）

### 练习 4：给 Typestate 添加共享数据

在上面的 `Document<State>` 示例中，如何让所有状态共享日志器（logger）而不需要在每次转换时手动搬运？

## 8. 练习答案

### 练习 1 答案

```rust
use std::marker::PhantomData;

struct NoMethod;
struct NoUrl;
struct MethodSet { method: String }
struct UrlSet { method: String, url: String }

struct HttpRequest<M, U> {
    method: Option<String>,
    url: Option<String>,
    body: Option<String>,
    _state: PhantomData<(M, U)>,
}

impl HttpRequest<NoMethod, NoUrl> {
    fn new() -> Self {
        HttpRequest {
            method: None, url: None, body: None,
            _state: PhantomData,
        }
    }
}

impl<U> HttpRequest<NoMethod, U> {
    fn method(self, m: &str) -> HttpRequest<MethodSet, U> {
        HttpRequest {
            method: Some(m.into()),
            url: self.url,
            body: self.body,
            _state: PhantomData,
        }
    }
}

impl<M> HttpRequest<M, NoUrl> {
    fn url(self, u: &str) -> HttpRequest<M, UrlSet> {
        HttpRequest {
            method: self.method,
            url: Some(u.into()),
            body: self.body,
            _state: PhantomData,
        }
    }
}

// body 只在 method 和 url 都设置后才能调用
impl HttpRequest<MethodSet, UrlSet> {
    fn body(mut self, b: &str) -> Self {
        self.body = Some(b.into());
        self
    }

    fn send(self) -> String {
        format!("{} {} body={:?}",
            self.method.unwrap(), self.url.unwrap(), self.body)
    }
}

// 使用：
let resp = HttpRequest::new()
    .method("POST")
    .url("https://api.example.com/users")
    .body(r#"{"name": "Alice"}"#)
    .send();
```

### 练习 2 答案

```rust
use std::marker::PhantomData;

struct Closed;
struct Opened { fd: i32 }

struct File<State> {
    path: String,
    state: State,
}

impl File<Closed> {
    fn new(path: &str) -> Self {
        File { path: path.into(), state: Closed }
    }

    fn open(self) -> Result<File<Opened>, std::io::Error> {
        // 模拟打开文件
        Ok(File {
            path: self.path,
            state: Opened { fd: 42 },
        })
    }
}

impl File<Opened> {
    fn read(&self, buf: &mut [u8]) -> Result<usize, std::io::Error> {
        println!("reading from fd {}", self.state.fd);
        Ok(0)
    }

    fn write(&self, data: &[u8]) -> Result<usize, std::io::Error> {
        println!("writing to fd {}", self.state.fd);
        Ok(data.len())
    }

    fn close(self) -> File<Closed> {
        println!("closing fd {}", self.state.fd);
        File { path: self.path, state: Closed }
    }
}
```

### 练习 3 答案

```
(a) 数据库连接池 → ❌ 不适合 Typestate
    原因：Idle ↔ InUse 是循环的（借出、归还），Typestate 擅长单向转换。
    每次 borrow 都要通过类型变化会很繁琐。
    建议：用 Enum + 运行时检查。

(b) 订单状态 → ✅ 适合 Typestate
    原因：单向流程，永不回头。用户不会把 Shipped 改回 Paid。
    编译器可以保证业务规则不被违反。

(c) 游戏角色 → ❌ 不适合 Typestate
    原因：状态可以在任意之间跳转（Idle→Running→Jumping→Attacking→Idle...）
    Typestate 的 consume-self 模式在这里会让代码非常冗长。
    建议：用 Enum + match。

(d) 安装向导 → ✅ 适合 Typestate
    原因：严格的线性流程，步骤不可跳过。
    编译期保证用户不会跳过第 2 步直接到第 4 步。
```

### 练习 4 答案

```rust
// 方案 1：把共享数据放在最外层，不参与 typestate
struct DocumentManager<State> {
    logger: Logger,             // 共享——不参与状态转换
    document: Document<State>,  // 状态相关的部分
}

// 方案 2：用 Arc 共享（如果多线程）
use std::sync::Arc;

struct Document<State> {
    content: String,
    logger: Arc<Logger>,  // Arc 是 Clone 的，不需要搬运所有权
    state_data: State,
}

// 方案 3：把共享数据放在跨状态的 impl 中
// 但这需要共享数据不被状态转换消耗——实际上方案 1 最实用
```

## 9. 深度扩展

### 9.1 C++ 的状态管理模式——为什么 Typestate 在 Rust 中如此自然

```cpp
// C++ 的状态管理：三种典型方式

// 方式 1：Enum + Runtime Check（最常见）
enum class ConnectionState { Unauthenticated, Authenticated };

class Connection {
    ConnectionState state;
public:
    void send_data(const std::vector<uint8_t>& data) {
        if (state != ConnectionState::Authenticated) {
            throw std::runtime_error("not authenticated!");
        }
        // 发送数据...
    }
};
// 问题：send_data 总是可以调用，错误发现延迟到运行时/测试/生产环境

// 方式 2：继承 + dynamic_cast（运行时类型转换）
class ConnectionBase { /* ... */ };
class UnauthenticatedConnection : public ConnectionBase { /* ... */ };
class AuthenticatedConnection : public ConnectionBase {
public:
    void send_data(const std::vector<uint8_t>& data) { /* ... */ }
};

void process(ConnectionBase* conn) {
    // 必须在运行时检查：
    auto* auth = dynamic_cast<AuthenticatedConnection*>(conn);
    if (auth) { auth->send_data(data); }
}
// 问题：dynamic_cast 有运行时开销，而且不能保证"一定转换成功"

// 方式 3：CRTP + 模板（最接近 Rust 的 Typestate，但体验差很多）
template <typename State>
class Connection { /* ... */ };

class Unauthenticated {};
class Authenticated {};

template <>
class Connection<Unauthenticated> {
public:
    Connection<Authenticated> authenticate(const std::string& pw);
};

template <>
class Connection<Authenticated> {
public:
    void send_data(const std::vector<uint8_t>& data);
};

// 这看起来像 Rust 的做法，但 C++ 的问题：
// 1. 模板特化需要重复定义整个类
// 2. 没有所有权系统——无法"消耗"旧对象来创建新对象
// 3. 移动语义可以模拟 consume，但无法保证旧对象不可用
// 4. 没有 PhantomData 等价物——C++ 不允许未使用的模板参数
//    除非用 [[no_unique_address]] 技巧

// C++ 的独特痛点："use after move" 不是编译错误
Connection<Unauthenticated> conn;
auto conn2 = conn.authenticate("pw");
// conn.send_data(data);  // C++ 中 conn 仍然"存在"
//    虽然有移动语义，编译器不禁止你使用移动后的对象
//    （只是约定——实际上很多项目用静态分析工具来检查）
```

**本质区别**：

| 维度 | C++ | Rust |
|------|-----|------|
| 状态验证时机 | 运行时（enum）或模板特化（笨重） | 编译期，轻量 |
| 旧状态失效 | 无保证（最多 lint） | 类型系统保证（consume self） |
| 零成本 | 运行时版本有开销 | 完全零运行时开销 |
| 语法开销 | 模板特化非常冗长 | 每个状态一个 impl 块，清晰 |
| PhantomData | 需要 `[[no_unique_address]]` 技巧 | 原生支持，简洁 |

### 9.2 如果没有 Typestate——只用运行时检查会怎样

```rust
// 如果没有 typestate，所有状态验证都是运行时检查：

struct Connection {
    state: State,
    // ...fields
}

impl Connection {
    fn send_data(&self, data: &[u8]) {
        assert!(self.state == State::Authenticated,
                "BUG: send_data called on unauthenticated connection");
        // ...
    }
}

// 这个问题有多严重？
// 1. 测试覆盖率：你需要给每个"非法状态转换"写测试
// 2. 生产环境：如果某个分支漏了检查 → panic 或更糟
// 3. 重构风险：加一个新状态后，所有 match 都需要更新
//    但编译器不会帮你找到遗漏的地方
// 4. 文档与实际行为脱节：文档说"只能在 Authenticated 后调用"
//    但代码里只是一个 panic——维护者可能不知道
```

**Typestate 把"协议"写进了类型签名**——它既是文档（一眼就能看出状态转换），又是编译器验证的规范。

### 9.3 `consume self` vs `&mut self`——为什么不能"原地变类型"

```rust
// 初学者常问：为什么认证方法要 consume self？
// 为什么不能"原地"改变类型？

// ❌ 假想的语法（Rust 不支持，也不应该支持）：
impl Connection<Unauthenticated> {
    fn authenticate(&mut self) /* 魔法——self 的类型变成 Connection<Authenticated> */ {
        // 问题 1：self 占用的栈/堆空间可能不够放 Authenticated 的额外字段
        // 问题 2：如果有其他引用指向这个 self 怎么办？它们还是 Uauthenticated 类型
        // 问题 3：这和 Rust 的"一个值一个类型"原则冲突
    }
}

// Rust 的选择：类型变化 → 值变化 → 新值（可能复用旧值的内存，但类型变了）
// 这就是为什么 consume self 是最自然的实现方式
//
// 而且编译器可以优化掉"假"的移动——如果新旧类型布局相同，
// LLVM 可能直接在原地构造新值（但这是优化，不是语义保证）
```

### 9.4 嵌套 Typestate——多层状态组合

```rust

struct ProtocolHttp;
struct ProtocolWebSocket;

struct Unencrypted;
struct Encrypted;

// 组合状态：
struct Connection<Proto, Auth, Enc> {
    socket: i32,
    _p: PhantomData<Proto>,

// 这在实际中用得很少，因为：

// 实践中，如果确实需要多层状态，考虑：
// - 每层状态用独立的 typestate struct 套娃
```


### 9.6 用宏简化 Typestate Builder

手写 typestate builder 需要为每个状态组合写 `impl` 块——随着字段增多，状态组合呈指数增长。**声明宏**可以自动生成这些样板代码。下面这个 `typestate_builder!` 宏展示了核心思路：

1. 宏接受字段列表和必填字段列表
2. 为每个必填字段生成一个**状态标记类型**（如 `struct Name; struct Age;`）
3. 为每个状态泛型组合生成对应的 **setter 方法**——该方法消费当前状态、设置字段值、返回下一个状态
4. `MissingField` 和 `Set` 是用来区分"已设置"和"未设置"的两个状态标记

这个宏的核心技巧在于 `$($other:ident,)*` 模式——它能匹配泛型参数列表中"除了当前字段之外的"所有其他状态参数，从而在任意顺序的设置下生成正确的转换。

```rust
// 手写 typestate builder 很冗长——宏可以大幅简化：

macro_rules! typestate_builder {
    // 定义一个 builder 的字段和必填/可选标记
    (
        $builder:ident {
            $( $field:ident : $ty:ty ),+ $(,)?
        }
        required { $( $req:ident ),+ }
    ) => {
        // 生成状态类型
        $( struct $req; )+

        struct $builder< $( $req, )+ > {
            $( $field: Option<$ty>, )+
            _state: std::marker::PhantomData<( $( $req, )+ )>,
        }

        // 为每个必填字段生成 setter
        $(
            impl< $( $other:ident, )* > $builder< $($other,)* MissingField, $($other2:ident,)* > {
                fn $field(mut self, v: $ty) -> $builder< $($other,)* Set, $($other2,)* > {
                    self.$field = Some(v);
                    $builder {
                        $field: self.$field,
                        $( $other: self.$other, )*
                        _state: std::marker::PhantomData,
                    }
                }
            }
        )+
    };
}
// 注意：这个宏只是示意——生产级的 typestate builder 宏比这复杂得多
// 实践中，用 `typed-builder` 或 `derive_builder` crate 更可靠
```

### 9.7 Typestate 与 Rust 的所有权系统——双剑合璧

```rust
// Typestate 和所有权系统的结合产生了 Rust 独有的能力：

// 1. 状态转换 + 所有权移动 = 旧状态不可访问
//    这不是"约定"——是编译器强制规则
let draft = Document::new("hello".into(), "Alice".into());
let published = draft.publish();
// draft.author();  // ❌ use of moved value: `draft`
//    C++ 做不到这一点——移动后的对象仍然"存在"

// 2. 借用 + Typestate = 共享只读访问
let doc = Document::new("hello".into(), "Alice".into());
let doc_ref = &doc;  // 共享引用——可以读取
doc_ref.author();     // ✅
// let doc = doc.publish();  // ❌ 不能通过共享引用修改所有权
//    必须等 doc_ref 不再使用后才能 publish

// 3. Typestate 的生命周期 = 编译期状态机的生命周期
//    当值被 drop 时，状态自然消失——不需要"反注册"步骤
```

### 9.8 Typestate 的局限性——什么时候不该用

```rust
// 局限性 1：泛型参数传染
// 一旦一个结构体使用了 Typestate，所有包含它的结构体也必须泛型化
struct Server<S: ConnectionState> {
    conn: Connection<S>,
}
// Server 现在也有了泛型参数 S——传染到整个系统

// 局限性 2：无法表达"或"关系
// 你想说"这个函数接受 Authenticated 或 Admin 状态"——Typestate 做不到
// 需要 trait 抽象：
trait CanSendData {}
impl CanSendData for Authenticated {}
impl CanSendData for Admin {}

fn send_data<S: CanSendData>(conn: &Connection<S>, data: &[u8]) { /* ... */ }

// 局限性 3：序列化困难
// Connection<Unauthenticated> 和 Connection<Authenticated> 是不同的类型
// 无法放入同一个 Vec（除非用 trait object 或 enum）

// 局限性 4：编译时间
// 大量 Typestate 泛型 → 大量单态化 → 编译时间增加
// 这是泛型的一般问题，但 Typestate 让泛型参数数量更容易膨胀

// 决策框架：
// ✅ 用 Typestate：状态数 ≤4，流程线性/单向，编译期已知
// ⚠️ 用 Trait 抽象：状态数 >4 或有"或"关系
// ✅ 用 Enum：状态数多、循环、运行时决定
```

> **一句话总结**：Typestate 模式把"运行时 if/panic"变成了"编译不过"，它利用 Rust 的泛型系统和所有权语义，将对象的状态机编码到类型参数中——`consume self → 返回新类型` 的模式既表达了状态转换，又保证了旧状态不可访问。C++ 可以用模板特化模拟类似效果，但缺少所有权系统意味着"移动后的对象仍可被误用"——Typestate + Rust 的所有权系统才是真正的双剑合璧。
