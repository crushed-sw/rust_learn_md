## 1. Clone vs Copy：Rust 最容易被误会的两个 trait

### 1.1 一句话区分

```rust
// Clone：显式的、可能昂贵的复制
// Copy：隐式的、保证廉价的按位复制

let x = String::from("hello");
let y = x.clone();  // Clone：显式调用，分配了新的堆内存

let a: i32 = 42;
let b = a;  // Copy：隐式的，只是复制 4 个字节
// a 仍然可用！
```

### 1.2 为什么 String 是 Clone 但不是 Copy

```rust
// String 的内存布局：
// Stack: { ptr, len, cap }  →  Heap: "hello"
//
// 如果 String 是 Copy：
let s1 = String::from("hello");
let s2 = s1;  // 如果 Copy：按位复制 {ptr, len, cap}
// s1 和 s2 的 ptr 指向同一块堆内存！
// → 离开作用域时，两个 String 都会尝试释放同一块内存
// → double free → 安全漏洞

// 所以 String 必须是 Clone 而不是 Copy：
// Clone 让你显式执行深拷贝（分配新的堆内存）
let s1 = String::from("hello");
let s2 = s1.clone();  // 深拷贝：新的堆内存 "hello"
// s1 和 s2 各管各的，没有 double free 问题
```

### 1.3 Copy trait 的本质

```rust
// Copy 的定义：一个标记 trait，继承自 Clone
pub trait Copy: Clone {
    // 没有方法——只有语义保证
}

// Copy 的条件：类型的所有字段都是 Copy
// 编译器检查：
// - 不能实现 Drop（有析构逻辑的类型不能按位复制）
// - 所有字段都必须是 Copy
// - 没有资源所有权（堆内存、文件句柄、锁等）
```

### 1.4 什么类型是 Copy，什么不是

```rust
// ✅ Copy 类型（纯数据）
i8, i16, i32, i64, i128, isize,
u8, u16, u32, u64, u128, usize,
f32, f64,
bool, char,
&T（共享引用——注意！），
[T; N]（如果 T: Copy），
函数指针 fn(...) -> ...，
Option<T>（如果 T: Copy），
()

// ❌ 不是 Copy 的类型（拥有资源）
String, Vec<T>, Box<T>,
&T 的独占引用 &mut T（因为只有一个可变引用）
File, TcpStream, MutexGuard<T>,
Rc<T>, Arc<T>

// 🔍 特别说明：&T 是 Copy
let s = String::from("hello");
let r1: &String = &s;
let r2: &String = r1;  // Copy！r1 和 r2 都是指向 s 的共享引用
println!("{}", r1);     // r1 仍然可用
```

### 1.5 自定义 Copy vs Clone

```rust
// ✅ 可以同时是 Clone 和 Copy（纯数据）
#[derive(Clone, Copy)]
struct Point {
    x: f64,
    y: f64,
}
// 注意 #[derive] 的顺序：Copy 依赖 Clone，所以 Clone 在前或同时

// ✅ 只能是 Clone（拥有资源）
#[derive(Clone)]
struct Buffer {
    data: Vec<u8>,
}
// Buffer 不能是 Copy，因为它拥有 Vec（拥有堆内存）

// ❌ 不能是 Copy（有 Drop）
struct Resource {
    handle: i32,
}
impl Drop for Resource {
    fn drop(&mut self) {
        // 释放资源
    }
}
// 任何实现了 Drop 的类型自动不能是 Copy
```

### 1.6 Clone 的代价

```rust
// Clone 不是零成本的——它可能很昂贵：
let v = vec![1u8; 1_000_000];  // 1MB 数据
let v2 = v.clone();             // 分配新的 1MB 并逐字节复制

// 这解释了为什么 Rust 不隐式 Clone：
// 隐式 clone 会导致性能问题被隐藏
// 显式的 .clone() 调用让你意识到"这里发生了分配"
```

## 2. Display vs Debug：给人看 vs 给开发者看

### 2.1 定义与区别

`Debug` 和 `Display` 的函数签名**几乎一模一样**——都是 `fn fmt(&self, f: &mut Formatter<'_>) -> fmt::Result`。那么区别在哪里？区别在于**语义约定和编译器/宏的支持**：

- **`Debug`**：通过 `#[derive(Debug)]` 自动生成，输出的是"给开发者看"的内部表示。它不保证格式稳定——不同版本的编译器可能生成不同的输出。你用 `{:?}` 来格式化它。
- **`Display`**：必须**手动实现**，因为它代表"给最终用户看"的、面向人类的输出。`println!("{}", x)` 使用 Display。没有 `#[derive(Display)]`，因为编译器不可能知道你想怎么呈现给用户。

简单说：`Debug` = "这个类型的内部长什么样"（机器/开发者视角），`Display` = "这个类型的理想人类可读形式"（用户视角）。下面看它们的方法签名：

```rust
// Debug：给开发者看的，用于调试输出
pub trait Debug {
    fn fmt(&self, f: &mut Formatter<'_>) -> fmt::Result;
}

// Display：给最终用户看的，用于面向用户的输出
pub trait Display {
    fn fmt(&self, f: &mut Formatter<'_>) -> fmt::Result;
}
```

### 2.2 使用场景

```rust
#[derive(Debug)]
struct User {
    id: u64,
    name: String,
    email: String,
}

impl std::fmt::Display for User {
    fn fmt(&self, f: &mut std::fmt::Formatter) -> std::fmt::Result {
        write!(f, "{} <{}>", self.name, self.email)
    }
}

// Debug 输出：
let user = User { id: 1, name: "Alice".into(), email: "alice@example.com".into() };
println!("{:?}", user);
// 输出：User { id: 1, name: "Alice", email: "alice@example.com" }

// Display 输出：
println!("{}", user);
// 输出：Alice <alice@example.com>

// 规则：
// { }    → Display   （面向用户）
// {:?}   → Debug     （面向开发者）
// {:#?}  → 美化 Debug（多行，缩进）
```

### 2.3 什么时候实现哪个

```rust
// ✅ 总是实现 Debug（至少用 #[derive(Debug)]）
// Rust 的惯例：每个公开类型都应该实现 Debug
#[derive(Debug)]
pub struct Config { /* ... */ }

// ✅ 如果你的类型有"面向用户的字符串表示"，实现 Display
//     比如：错误类型、URL、路径、用户名等
impl Display for Config {
    fn fmt(&self, f: &mut Formatter) -> fmt::Result {
        write!(f, "path={}, timeout={}", self.path, self.timeout)
    }
}

// 何时 NOT 实现 Display：
// - 没有单一的"自然"字符串表示
// - 类型是内部的、抽象的（但至少实现 Debug）
```

### 2.4 Debug 的 derive 宏

```rust
#[derive(Debug)]
struct Node {
    value: i32,
    children: Vec<Node>,
}

// derive(Debug) 等价于（简化）：
impl fmt::Debug for Node {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        f.debug_struct("Node")
            .field("value", &self.value)
            .field("children", &self.children)
            .finish()
    }
}
```

## 3. Send 和 Sync：Rust 并发安全的基石

### 3.1 一句话定义

```rust
// Send：类型的所有权可以安全地转移到另一个线程
// Sync：类型的共享引用可以安全地在多个线程间共享

// 都是"自动 trait"（auto trait）——编译器自动推导
// 都是"不安全 trait"（unsafe trait）——手动实现需要 unsafe
pub unsafe auto trait Send { }
pub unsafe auto trait Sync { }
```

### 3.2 自动推导规则

```rust
// Send 的自动推导：
// 一个类型是 Send，当且仅当它的所有字段都是 Send

struct MyType {
    a: i32,           // Send ✅
    b: String,        // Send ✅
    c: Vec<u32>,      // Send ✅
}
// → MyType 自动是 Send ✅

struct NotSend {
    a: i32,
    b: std::rc::Rc<String>,  // Rc 不是 Send ❌
}
// → NotSend 自动不是 Send ❌
```

### 3.3 典型类型的 Send/Sync 状态

```rust
//      类型          Send    Sync    原因
//  ────────────────────────────────────────────
//   i32              ✅      ✅      纯数据
//   String           ✅      ✅      拥有堆数据，但独占
//   Vec<T>           ✅(T)   ✅(T)   取决于 T
//   &T               ✅(T)   ✅(T)   共享引用
//   &mut T           ✅(T)   ❌      独占引用不能共享
//   Box<T>           ✅(T)   ✅(T)   取决于 T
//   Rc<T>            ❌      ❌      引用计数不是原子的
//   Arc<T>           ✅(T)   ✅(T)   原子引用计数
//   Mutex<T>         ✅(T)   ✅(T)   互斥锁保护
//   RefCell<T>       ✅(T)   ❌      !Sync——运行时借用检查
//   Cell<T>          ✅(T)   ❌(T)   非线程安全的内部可变性
//   *const T         ❌      ❌      裸指针不安全
//   *mut T           ❌      ❌      裸指针不安全
//   MutexGuard<T>    ❌(T)   ❌      锁守卫不 Send（防止跨线程释放）
```

### 3.4 `Rc<T>` 不是 Send 的深层原因

```rust
use std::rc::Rc;

let rc = Rc::new(42);
let rc2 = rc.clone();  // 引用计数变为 2

// Rc 的引用计数操作不是原子的：
// 两个线程同时 clone → 引用计数可能从 2 变成 3 而不是 4
// → 引用计数不正确 → use-after-free

// Arc<T> 解决了这个问题——引用计数的增减用原子操作
use std::sync::Arc;
let arc = Arc::new(42);
let arc2 = arc.clone();  // 引用计数的增加是原子的
```

### 3.5 `RefCell<T>` 是 Send 但不是 Sync

```rust
use std::cell::RefCell;

let cell = RefCell::new(42);

// Send ✅：可以把 RefCell 的所有权转移到另一个线程
std::thread::spawn(move || {
    *cell.borrow_mut() = 100;  // OK：只有这个线程有所有权
});

// Sync ❌：不能从多个线程同时借用一个 RefCell
// RefCell 的借用检查是运行时的、非原子操作：
// let b1 = cell.borrow();     // 线程 1
// let b2 = cell.borrow_mut(); // 线程 2 → 数据竞争！
```

### 3.6 手动干预 Send/Sync

```rust
use std::marker::PhantomData;

// 方式 1：用 PhantomData 阻止自动推导
struct MyWrapper<T> {
    data: T,
    _not_sync: PhantomData<std::cell::Cell<()>>,  // Cell<()> 不是 Sync
}
// MyWrapper 现在不是 Sync 了（即使 T 是 Sync）

// 方式 2：unsafe 手动实现（极少使用，需要你保证正确性）
struct MyUnsafeType {
    raw: *const u8,
}

unsafe impl Send for MyUnsafeType { }
unsafe impl Sync for MyUnsafeType { }
// ⚠️ 这相当于向编译器承诺："相信我，这个类型是线程安全的"
// 但编译器不会检查——责任全在你
```

## 4. 六个 trait 的对比矩阵

| Trait | 类别 | 自动？ | 标记？ | 核心语义 |
|:---|:---|:---:|:---:|:---|
| **Clone** | 显式复制 | 否 | 否 | `.clone()` 创建独立的副本 |
| **Copy** | 隐式复制 | 否 | ✅ | 按位复制，赋值后原变量仍可用 |
| **Display** | 格式化 | 否 | 否 | 面向用户的格式化输出 `{}` |
| **Debug** | 格式化 | 否 | 否 | 面向开发者的调试输出 `{:?}` |
| **Send** | 并发安全 | ✅ | ✅ | 所有权可以跨线程转移 |
| **Sync** | 并发安全 | ✅ | ✅ | 共享引用可以跨线程使用 |

## 5. Copy 与 Move 的交互——最容易踩的坑

### 5.1 Copy 类型不会触发 Move

```rust
// Copy 类型：赋值后原变量还在
let x: i32 = 42;
let y = x;    // x 被 COPY（不是 move）
println!("{}", x);  // ✅ x 仍然可用

// 非 Copy 类型：赋值后原变量被 move
let s = String::from("hello");
let t = s;    // s 被 MOVE
// println!("{}", s);  // ❌ s 已被 move
```

### 5.2 元组和数组的 Copy 传递

```rust
// 元组的 Copy 性取决于所有元素：
let t1: (i32, i32) = (1, 2);  // ✅ 两个元素都是 Copy
let t2 = t1;
println!("{:?}", t1);  // ✅ t1 还在

let t3: (i32, String) = (1, String::from("hello"));  // ❌ String 不是 Copy
let t4 = t3;
// println!("{:?}", t3);  // ❌ t3 已被 move

// 数组同理：
let a1: [i32; 3] = [1, 2, 3];  // ✅
let a2 = a1;
println!("{:?}", a1);  // ✅

let a3: [String; 2] = [String::from("a"), String::from("b")];  // ❌
let a4 = a3;  // a3 被 move
```

### 5.3 泛型中的 Copy 约束

```rust
// ❌ 不够泛用的写法
fn duplicate_vec(v: &Vec<i32>) -> Vec<i32> {
    v.clone()
}

// ✅ 泛用写法：用 Clone 约束
fn duplicate<T: Clone>(v: &[T]) -> Vec<T> {
    v.iter().cloned().collect()
}

// ✅ 更精确：用 Copy 约束（如果 T 是 Copy，用 *x 比 .clone() 更高效）
fn duplicate_copy<T: Copy>(v: &[T]) -> Vec<T> {
    v.iter().copied().collect()  // copied() 需要 T: Copy
}
```

## 6. 设计哲学：自动 trait 的巧思

`Send` 和 `Sync` 是 Rust 并发模型中最精妙的设计之一：

```
自动推导（auto trait）：
├── 编译器自动为类型推导 Send/Sync
├── 推导规则简单：所有字段满足 → 类型满足
├── 出现不满足的字段（如 Rc、RefCell、裸指针）→ 类型自动不满足
└── 这种"传染性"保证了线程安全的封闭性

手动干预：
├── unsafe impl Send for MyType { }：向编译器承诺
├── PhantomData：阻止自动推导
└── 两种方式都不常用——大多数情况下自动推导就对了

其他语言对比：
├── C++：完全靠程序员自觉 + 文档（"这个类是线程安全的"）
├── Java：synchronized 关键字 + volatile（语法层面，可绕过）
├── Go："Share memory by communicating"（通过 channel 规范，非类型系统）
└── Rust：Send/Sync 在类型层面保证了线程安全（编译器强制检查）
```

### 6.1 如果没有 Send/Sync —— C++ 的"全靠自觉"模式

在 C++ 中，线程安全完全是程序员的个人责任：

```cpp
#include <thread>
#include <string>
#include <vector>

// 这个类线程安全吗？——不知道，看文档（如果有的话）
class UserCache {
    std::vector<std::string> users;
    // 没有 mutex、没有 atomic、没有类型层面的标记
public:
    void add_user(const std::string& name) {
        users.push_back(name);  // ❌ 如果多线程调用 → data race → UB
    }
};

// 编译完全通过：
UserCache cache1;
UserCache cache2;
std::thread t1([&] { cache1.add_user("alice"); });
std::thread t2([&] { cache2.add_user("bob"); });
// 即使 cache1 和 cache2 是不同的对象，编译器不会报错
// 但 std::vector 可能有全局状态（比如内存分配器）——这仍然可能是 data race

// 在 C++ 中，你需要手动做的一切：
// 1. 阅读 std::vector 的文档来确认它"线程安全"——但文档可能不完整
// 2. 自己加 std::mutex
// 3. 自己确保锁的粒度正确
// 4. 自己确保没有死锁
// 5. 用 sanitizer 在运行时检测 data race——可能漏掉

// 而在 Rust 中：
// 如果 UserCache 的所有字段都是 Send，它就是 Send（自动）
// 如果某个字段不是 Send（如 Rc），整个类型就不是 Send（自动）
// → 当你试图把它发到另一个线程，编译器就会报错
// 根本不需要到运行时才发现问题
```

### 6.2 为什么 Send/Sync 用"自动推导"而不是"手动实现"

```
如果 Send/Sync 像 Clone 一样需要手动实现：

好处：
  ✅ 程序员"意识到"线程安全性（每次 impl 都要想一下）

坏处（远大于好处）：
  ❌ 99% 的类型天然是 Send/Sync（纯数据的、拥有所有权的类型）
     → 手动 impl 成为无意义的样板代码
  ❌ 忘记 impl Send 会导致"明明是线程安全的类型却发不到线程"
     → 反向误报，让人关掉检查
  ❌ 从 C++ 的教训：如果安全机制带来大量样板，程序员就会绕过去

Rust 的选择：自动推导 + 传染性
  - 大多数类型自动得到正确的 Send/Sync 标记
  - 只有真正不安全的类型（Rc, RefCell, 裸指针）才需要特殊处理
  - 这保证了"默认安全"——你不需要做任何事就有安全保障
```

这是 Rust "安全应该是默认的"哲学的完美体现——跟 C++ 的"安全需要你自己保证"形成鲜明对比。

## 8. 练习答案

### 练习 1 答案

```rust
(a) &String         → ✅ Copy（共享引用总是 Copy）
(b) &mut i32        → ❌ 不是 Copy（独占引用不是 Copy，因为只能有一个）
(c) Box<i32>        → ❌ 不是 Copy（拥有堆内存，需要 Drop）
(d) Option<&str>    → ✅ Copy（&str 是 Copy + 引用的 Copy，Option<Copy> 也是 Copy）
(e) *const i32      → ✅ Copy！（裸指针是 Copy——编译器不管理它们）
(f) [i32; 100]      → ✅ Copy（数组类型，所有元素都是 Copy）
```

**重点解释 (e)**：

```rust
// 裸指针是 Copy！但解引用它们需要 unsafe：
let x: i32 = 42;
let p: *const i32 = &x;
let q = p;        // ✅ p 被 Copy，p 仍然可用
// unsafe { println!("{}", *p); }  // ✅ p 指向的地址没变
// 裸指针是 Copy 因为它们的语义就是"任意的内存地址"
// 编译器不追踪它们的生命周期——责任在程序员
```

### 练习 2 答案

```rust
use std::fmt;

struct Url {
    scheme: String,
    host: String,
    port: u16,
    path: String,
}

impl fmt::Display for Url {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "{}://{}:{}{}", self.scheme, self.host, self.port, self.path)
    }
}

impl fmt::Debug for Url {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        f.debug_struct("Url")
            .field("scheme", &self.scheme)
            .field("host", &self.host)
            .field("port", &self.port)
            .field("path", &self.path)
            .finish()
    }
}

fn main() {
    let url = Url {
        scheme: "https".into(),
        host: "example.com".into(),
        port: 443,
        path: "/api/v1".into(),
    };

    println!("Display: {}", url);   // https://example.com:443/api/v1
    println!("Debug: {:?}", url);
    // Debug: Url { scheme: "https", host: "example.com", port: 443, path: "/api/v1" }
    println!("Pretty: {:#?}", url);
}
```

### 练习 3 答案

```rust
struct A { x: i32, y: String }
// Send ✅ （i32✅, String✅）
// Sync ✅ （i32✅, String✅）

struct B { x: i32, y: std::rc::Rc<String> }
// Send ❌ （Rc<String> 不是 Send）
// Sync ❌ （Rc<String> 不是 Sync）

struct C { x: i32, y: std::sync::Arc<String> }
// Send ✅ （i32✅, Arc<String>✅）
// Sync ✅ （i32✅, Arc<String>✅）

struct D { x: i32, y: std::cell::RefCell<String> }
// Send ✅ （i32✅, RefCell<String>✅ — RefCell 是 Send）
// Sync ❌ （RefCell<String> 不是 Sync — 非原子的运行时借用检查）

struct E { x: i32, y: *const u8 }
// Send ❌ （*const u8 不是 Send — 裸指针编译器不追踪所有权）
// Sync ❌ （*const u8 不是 Sync — 裸指针编译器不保证线程安全）
```

### 练习 4 答案

```rust
// 如果 MutexGuard<T> 实现了 Send...（伪场景）

use std::sync::Mutex;

let m = Mutex::new(42);

// 线程 1：获取锁
let guard = m.lock().unwrap();  // MutexGuard

// 如果 MutexGuard 是 Send：
std::thread::spawn(move || {
    // guard 被移动到线程 2
    // 但线程 1 并没有释放锁！
    // 线程 2 现在持有 MutexGuard，但不知道线程 1 是否已经离开了临界区
});
// 线程 1 结束，guard 的生命周期...但 MutexGuard 在另一个线程！

// 更严重的问题：MutexGuard 的 Drop 释放锁
// 如果线程 2 释放了锁（Drop MutexGuard），
// 而线程 1 的栈上还有一些局部变量依赖锁保护...
// → 未定义行为

// 所以 MutexGuard<T> 被设计为 !Send：
// - 锁必须在获取它的线程中释放
// - 这保证了临界区的完整性
```

### 9.1 `#[derive(Copy)]` 的内部机制

```rust
// derive(Copy) 宏实际做的事情：
// 1. 检查所有字段都实现了 Copy
// 2. 检查类型没有实现 Drop
// 3. 生成 impl Copy for Type { }

// 第 2 点比较微妙：不能同时 derive(Copy) 和实现 Drop
#[derive(Clone, Copy)]
struct Watchdog {
    timer: u64,
    restart_count: u64,
}

// impl Drop for Watchdog {  // ❌ 编译错误！
//     fn drop(&mut self) {
//         println!("Watchdog stopped after {} restarts", self.restart_count);
//     }
// }
// error[E0184]: the trait `Copy` cannot be implemented for this type;
//               the type has a destructor

// 原因：如果 Copy 类型有了 Drop，赋值时按位复制了但 Drop 逻辑
// 可能期待所有权转移而不是复制——语义矛盾
```

### 9.2 Clone 和 Copy 在 trait bound 上的精确使用

```rust
// 选择 Clone 还是 Copy 作为 trait bound：

// ✅ 用 Clone：更宽松，接受更多类型
fn first_duplicate<T: Clone + PartialEq>(slice: &[T]) -> Option<T> {
    for i in 0..slice.len() {
        for j in (i + 1)..slice.len() {
            if slice[i] == slice[j] {
                return Some(slice[i].clone());
            }
        }
    }
    None
}

// ✅ 用 Copy：更严格，但调用方不需要 .clone()
fn sum_elements<T: Copy + std::ops::Add<Output = T>>(slice: &[T]) -> T
where
    T: Default,
{
    slice.iter().copied().fold(T::default(), |a, b| a + b)
}

// 准测：如果函数内部只需要"读取"值（不需要修改所有权），
// 用 Copy 约束可以避免 .clone() 调用。
// 如果类型可能不是 Copy（如 String），就用 Clone 约束。
```

### 9.4 Send/Sync 的"负向实现"（Negative impls）

**负向实现**（negative impl）是 Rust 的一个 nightly 特性，让你可以**主动声明**某个类型不实现某个 auto trait。正常情况下，编译器会自动为类型推导 `Send` 和 `Sync`。但有时你需要阻止这种行为：

- **FFI 场景**：包装了一个 C 库的数据结构，而 C 库内部不是线程安全的——你需要标记 wrapper 为 `!Send` 来防止 Rust 代码把它传到其他线程。
- **unsafe 抽象边界**：你实现了一个"看起来像线程安全"的类型，但实际上依赖特定线程的上下文——用 `!Send` 在类型层面明确这个限制。
- **比 `PhantomData<*const ()>` 更直接**：用 `PhantomData` 来阻止 auto trait 虽然稳定，但语义上是"间接的"——负向实现直接表达了"这个类型不满足 Send/Sync"。

注意：这是 **nightly-only** 功能（需要 `#![feature(negative_impls)]`）。在 stable Rust 中，你需要用 `PhantomData<*const ()>` 或不包含 `Send` 字段来间接达到同样的效果。

```rust
// nightly 功能：手动声明某个类型不是 Send/Sync
#![feature(negative_impls)]

struct MyRcWrapper<T> {
    inner: T,
    // MyRcWrapper 包装 T，即使 T 是 Send，wrapper 也不是 Send
}

impl<T> !Send for MyRcWrapper<T> { }
impl<T> !Sync for MyRcWrapper<T> { }

// 这在 FFI 和 unsafe 场景下非常有用——
// 你可以"阻止"编译器自动推导 Send/Sync
```

### 9.5 为什么 `&T` 是 Copy 而 `&mut T` 不是

这回到了 Rust 的引用规则：

```rust
// &T（共享引用）：可以有多个，都是 Copy
// 这很安全——多个只读视角，不会冲突
let x = 42;
let r1 = &x;
let r2 = r1;       // Copy：r1 被复制，r1 和 r2 都指向 x
println!("{} {}", r1, r2);  // 两个都可以用

// &mut T（独占引用）：只有一个，不能 Copy
// 如果 &mut T 是 Copy，就会有两个可变引用指向同一数据
let mut x = 42;
let r1 = &mut x;
let r2 = r1;  // Move（不是 Copy）：r1 被移动，不能再使用
// *r1 = 100;  // ❌ r1 已被移动
*r2 = 100;     // ✅ 只有 r2 可用
```

**设计逻辑**：`Copy` 的底层语义是"赋值后原变量仍可使用"。如果允许 `&mut T` 是 Copy，就等于允许同时存在多个可变引用——这违反了 Rust 的核心安全保证。

### 9.6 `Display` 的 `Formatter` 能力

```rust
use std::fmt;

impl fmt::Display for MyType {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        // Formatter 提供了很多元信息：
        let width = f.width();      // 用户指定的宽度 {:<10}
        let precision = f.precision(); // 用户指定的精度 {:.2}
        let align = f.align();      // 对齐方式 {:<}, {:>}, {:^}
        let fill = f.fill();        // 填充字符 {:_>10} → '_'

        // 根据这些信息调整输出
        match width {
            Some(w) => write!(f, "{:>width$}", self.name, width = w),
            None => write!(f, "{}", self.name),
        }
    }
}
```

### 9.7 `MutexGuard<T>` 和 `RefMut<T>` 的 Send/Sync 设计模式

```rust
// MutexGuard<T> 和 RefMut<T> 都实现了 Deref 和 DerefMut，
// 但它们的 Send/Sync 性质不同：

// RefMut<T>（RefCell 的借用）：不是 Sync，不是 Send（大部分情况）
// MutexGuard<T>：是 Sync（如果 T 是 Sync），不是 Send

// 这种设计反映了两个锁的语义差异：
// - RefCell: 单线程运行时借用检查 → 不支持 Sync
// - Mutex: 多线程互斥 → 支持 Sync（多个线程可以"看到"锁守卫）
//   但不支持 Send（锁必须在获取它的线程中释放）

// 这个设计防止了：
let m = std::sync::Mutex::new(42);
let guard = m.lock().unwrap();
std::thread::spawn(move || {
    // ❌ guard 不能 move 到其他线程
    // 因为 MutexGuard 不是 Send
});  // 编译错误
```

### 9.8 PhantomData 家族：不仅仅是占有

```rust
use std::marker::PhantomData;
use std::rc::Rc;

// PhantomData 的三种主要用途：

// 1. 占有类型（像拥有了 T，但没有存储 T）
struct MyPtr<T> {
    ptr: *const T,
    _marker: PhantomData<T>,  // 告诉编译器"我们逻辑上拥有 T"
}
// 这样 MyPtr<T> 的 Send/Sync 取决于 T 的 Send/Sync

// 2. 阻止 Send/Sync 自动推导
struct ThreadLocal<T> {
    data: T,
    _not_send: PhantomData<Rc<()>>,
}
// Rc<()> 不是 Send，所以 ThreadLocal<T> 也不是 Send

// 3. 标记生命周期关系（不需要所有权）
struct Ref<'a, T: 'a> {
    ptr: *const T,
    _lifetime: PhantomData<&'a T>,  // 绑定生命周期，但没有实际存储引用
}
```
