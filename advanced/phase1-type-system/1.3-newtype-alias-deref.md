# 1.3 newtype 模式 & 类型别名 & Deref 链

> 类型系统的三个"包装工具"——一个给你类型安全，一个给你别名便利，一个给你自动转换。三者看起来都能"把一个类型变成另一个"，但设计意图完全不同。用错了会导致 API 灾难。

---

## 1. 从一个经典 Bug 开始

```rust
// 这是同一个公司的两个团队写的代码，然后合并了：

// 团队 A：用 f64 表示距离（单位：米）
fn launch_rocket(distance: f64) {
    println!("发射！距离 {} 米", distance);
}

// 团队 B：用 f64 表示距离（单位：公里）
fn main() {
    let distance_to_target = 150.0;  // 150 公里
    launch_rocket(distance_to_target); // 💥 把 150 公里当 150 米用了
    // 火箭在 150 米处炸了
}
```

两个 `f64`，类型系统看不出任何区别。这就是 **原始类型迷恋 (Primitive Obsession)**——用一个通用类型表达有明确语义的值。

**newtype 模式** 就是解药：

```rust
// ✅ 类型系统会阻止你犯错
#[derive(Debug, Clone, Copy)]
struct Meters(f64);

#[derive(Debug, Clone, Copy)]
struct Kilometers(f64);

fn launch_rocket(distance: Meters) {
    println!("发射！距离 {} 米", distance.0);
}

fn main() {
    let target = Kilometers(150.0);
    // launch_rocket(target);  // ❌ 编译错误！类型不匹配！
    //                    ^^^^^^ expected `Meters`, found `Kilometers`

    // ✅ 必须显式转换，强迫你思考
    let target_meters = Meters(target.0 * 1000.0);
    launch_rocket(target_meters);
}
```

## 2. newtype 模式

### 2.1 定义

newtype = **单字段元组结构体**，包装一个已有类型：

```rust
// 三种等价写法（通常用第一种）
struct MyInt(i32);            // 元组结构体——推荐
struct MyInt { value: i32 }   // 命名字段结构体——当你需要多字段时

// 本质：MyInt 和 i32 是完全不同的类型，编译器不会混淆
let a: MyInt = MyInt(42);
let b: i32 = 42;
// let c: i32 = a;  // ❌ 类型不匹配！
```

### 2.2 零成本抽象保证

newtype 的运行时开销 = **零**。编译后，`Meters(f64)` 就是 `f64` 本身：

```rust
use std::mem::size_of;

struct Meters(f64);
struct Kilometers(f64);

// 编译后完全一样
assert_eq!(size_of::<Meters>(), 8);
assert_eq!(size_of::<f64>(), 8);
assert_eq!(size_of::<Kilometers>(), 8);

// 函数调用也没有额外开销——参数直接传 f64
fn foo(m: Meters) { /* m.0 在寄存器中，编译器完全优化掉包装 */ }
```

**什么时候需要 `#[repr(transparent)]`？**

```rust
// 默认情况（repr(Rust)）：编译器可能重排字段，但对于单字段结构体通常没问题

// FFI 场景中：必须标记，保证 ABI 与内部类型完全一致
#[repr(transparent)]
struct FileDescriptor(i32);
// 确保 FileDescriptor 的 ABI 与 i32 完全一致——可以安全传给 C
```

### 2.3 newtype 的四大用途

#### 用途 1：类型安全（单位、标识符）

```rust
// 防止混淆不同类型的 ID
struct UserId(u64);
struct OrderId(u64);
struct ProductId(u64);

fn get_user(id: UserId) -> Option<User> { ... }
fn get_order(id: OrderId) -> Option<Order> { ... }

// get_user(OrderId(1));  // ❌ 编译错误——永远不会混淆了
```

以上示例展示了 newtype 在"区分语义相同但含义不同的 ID"上的威力。但 newtype 的另一大用途是**在类型系统层面保证数据合法性**。通过将验证逻辑放在构造函数中，你可以确保：一旦某个类型的值存在，它就一定满足某些不变量。例如 `Port(u16)` 保证端口号永远不会是零——这种保证完全不需要运行时检查，类型本身就提供了安全网：

```rust
// 非零保证
use std::num::NonZeroU32;

struct Port(u16);
impl Port {
    fn new(p: u16) -> Option<Self> {
        if p == 0 { None } else { Some(Port(p)) }
    }
}
// Port(0) 永远不会存在（通过构造函数保证）
```

#### 用途 2：孤儿规则的 workaround

这是 newtype 最强大的用法之一。回忆孤儿规则：你只能在 trait 的 crate 或类型的 crate 中为类型实现 trait。

```rust
// 问题：你想为 Vec<u64> 实现 Display，但：
// impl Display for Vec<u64> { ... }  // ❌ 孤儿规则阻止！
// Display 在 std，Vec 在 std，u64 在 std——都不是你的 crate

// ✅ newtype 方案：
struct MyVec(Vec<u64>);

impl fmt::Display for MyVec {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "[{}]", self.0.iter()
            .map(|n| n.to_string())
            .collect::<Vec<_>>()
            .join(", "))
    }
}

let v = MyVec(vec![1, 2, 3]);
println!("{}", v);  // [1, 2, 3]
```

#### 用途 3：语义约束（验证、单位转换）

```rust
// 保证 Email 永远是有效的
#[derive(Debug, Clone)]
struct Email(String);

impl Email {
    pub fn new(raw: &str) -> Result<Self, &'static str> {
        if raw.contains('@') && raw.contains('.') {
            Ok(Email(raw.to_string()))
        } else {
            Err("Invalid email format")
        }
    }
}

// 任何接受 Email 的函数都不需要再验证
fn send_email(to: &Email, subject: &str) {
    // to 肯定是有效的 email——构造函数已经保证了
}

// ✨ "parse, don't validate" —— 让类型系统替你保证不变量
```

#### 用途 4：封装与 API 演进

```rust
// 初版：直接暴露内部类型
pub struct Config {
    pub timeout: u64,  // 暴露了内部表示
}

// 后来：timeout 要改为 Duration，所有使用方都要改——破坏性变更

// ✅ 从一开始就用 newtype：
pub struct Timeout(u64);

impl Timeout {
    pub fn as_secs(&self) -> u64 { self.0 }
    pub fn as_millis(&self) -> u64 { self.0 * 1000 }
}

// 将来改成 Duration 时，只需修改 Timeout 内部，API 不变
pub struct Timeout(Duration);
// as_secs 和 as_millis 签名不变，所有调用方不需要改
```

### 2.4 要不要为新类型实现 Deref？

**这可能是 newtype 模式中最重要的设计决策**。

```rust
struct Meters(f64);

// 方案 A：不实现 Deref（推荐）
impl Meters {
    fn value(&self) -> f64 { self.0 }
}

// 方案 B：实现 Deref（谨慎！）
impl Deref for Meters {
    type Target = f64;
    fn deref(&self) -> &f64 { &self.0 }
}
```

| | 不实现 Deref | 实现 Deref |
|:---|:---|:---|
| 使用方式 | `m.value()` 显式提取 | 自动解引用，`*m` 或 `m.abs()` 直接可用 |
| 类型安全 | 不会意外混用 | 任何接受 `&f64` 的地方都能传入 `&Meters` |
| API 清晰度 | 调用方明确知道在做什么 | 隐式行为，可能掩盖 bug |
| 推荐场景 | 语义上不同的类型（单位、ID） | "智能指针"型包装 |

**黄金法则**：

```
Deref 是为"智能指针"设计的，不是为"类型转换"设计的。

✅ 应该实现 Deref 的：
  - Box<T> → T（智能指针）
  - Rc<T> → T（引用计数指针）
  - Arc<T> → T（原子引用计数指针）
  - String → str（拥有者 → DST 视图）

❌ 不应该实现 Deref 的：
  - Meters → f64（语义不同）
  - Email → String（可以破坏不变量）
  - UserId → u64（语义不同）
```

**为什么 `Email` 不该实现 `Deref<Target = String>`？**

```rust
// 如果 Email 实现了 Deref<Target = String>：
struct Email(String);
impl Deref for Email {
    type Target = String;
    fn deref(&self) -> &String { &self.0 }
}

// 那么你可以在 Email 上调用 String 的所有方法：
let email = Email::new("a@b.com").unwrap();
email.clear();        // 💥 String::clear()——把 email 清空了！
email.push_str("x");  // 💥 String::push_str()——把 email 改烂了！
// Email 的"有效 email"不变量被彻底破坏
```

## 3. 类型别名 (Type Alias)

### 3.1 类型别名 vs newtype

这是最容易混淆的概念对：

```rust
// 类型别名：完全透明，类型相同
type Meters = f64;
let a: Meters = 5.0;
let b: f64 = a;  // ✅ 可以互相赋值——它们是同一个类型

// newtype：完全隔离，类型不同
struct Meters(f64);
let a = Meters(5.0);
// let b: f64 = a;  // ❌ 类型不匹配
let b: f64 = a.0;   // ✅ 必须显式提取
```

| | 类型别名 | newtype |
|:---|:---|:---|
| 运行时代价 | 无 | 无 |
| 类型安全 | 无（透明） | 完全隔离 |
| 编译器检查 | 不区分 | 严格区分 |
| trait 实现 | 不能额外实现 trait | 可以自由实现 trait |
| 孤儿规则 | 受约束 | 可以绕过 |
| 主要用途 | 减少样板、提升可读性 | 类型安全、trait 实现 |

### 3.2 什么时候用类型别名

#### 场景 1：简化复杂泛型签名

```rust
// 没有别名：签名爆炸
fn process(
    data: Arc<Mutex<HashMap<String, Vec<Box<dyn Fn(i32) -> String>>>>>,
) { ... }

// ✅ 有别名：一眼看懂
type Callback = dyn Fn(i32) -> String;
type CallbackVec = Vec<Box<Callback>>;
type DataStore = Arc<Mutex<HashMap<String, CallbackVec>>>;

fn process(data: DataStore) { ... }
```

#### 场景 2：给外部类型起语义化的本地名字

```rust
// 你的系统里所有 ID 都用 u64 存储，但不想引入 newtype 的摩擦
type UserId = u64;
type OrderId = u64;

fn get_user(id: UserId) { ... }
// 注意：这只是文档性质——编译器不会阻止你传 OrderId 给 get_user
```

#### 场景 3：平台相关的类型统一

```rust
// 条件编译 + 类型别名
#[cfg(target_arch = "x86_64")]
type Word = u64;

#[cfg(target_arch = "x86")]
type Word = u32;

// 后续代码只需用 Word，不需要关心平台
fn read_word() -> Word { ... }
```

### 3.3 类型别名 vs `impl Trait`

```rust
// 类型别名：暴露具体类型，但不重复写
type MyIter = std::vec::IntoIter<i32>;
fn make_iter() -> MyIter { vec![1, 2, 3].into_iter() }

// impl Trait：隐藏具体类型
fn make_iter() -> impl Iterator<Item = i32> { vec![1, 2, 3].into_iter() }

// 区别：
// - 类型别名适合"我知道这是什么类型，但我不想写两遍"
// - impl Trait 适合"我不希望你依赖这个具体类型"（API 稳定性）
```

## 4. Deref 强制解引用 (Deref Coercion)

### 4.1 机制：编译器在幕后做了什么

```rust
use std::ops::Deref;

struct MyBox<T>(T);

impl<T> Deref for MyBox<T> {
    type Target = T;
    fn deref(&self) -> &T { &self.0 }
}

fn takes_str(s: &str) {
    println!("{}", s);
}

fn main() {
    let s = MyBox(String::from("hello"));

    // 你写的：
    takes_str(&s);

    // 编译器实际做的（Deref 链）：
    // &s          → &MyBox<String>       （取引用）
    // *(&s)       → &String              （Deref: &MyBox<String> → &String）
    // *(*&s)      → &str                 （再 Deref: &String → &str）
    // 最终：&s 强制转换为 &str
}
```

**Deref Coercion 的触发条件**：

```rust
// 1. &T → &U       当 T: Deref<Target = U>
// 2. &mut T → &mut U  当 T: DerefMut<Target = U>
// 3. &mut T → &U   当 T: Deref<Target = U>（可变转不可变）
//                  注意：&T → &mut U 永远不会发生

// 示例：
let s: String = "hello".into();
let r: &str = &s;          // 规则 1: &String → &str
let b: Box<String> = Box::new("hello".into());
let r: &str = &b;          // 规则 1 链: &Box<String> → &String → &str

let mut v: Vec<i32> = vec![1, 2, 3];
let r: &mut [i32] = &mut v;  // 规则 2: &mut Vec<i32> → &mut [i32]
let r: &[i32] = &mut v;      // 规则 3: &mut Vec<i32> → &[i32]
```

### 4.2 方法解析与 Deref

```rust
let s = String::from("hello");

// String 上直接定义的方法
s.len();     // String::len（存疑——实际 String 没有 len，是 str 的）
//            实际上 String 通过 Deref 找到了 str::len，所以它"看起来"有 len

// 方法查找过程：
// 1. 直接在 String 类型上查找 len → 未找到
// 2. Deref 到 &str → 在 str 上查找 len → 找到！
// 3. 调用
// 这个过程对调用者完全透明
```

```rust
// Deref 在方法查找时可能产生歧义
use std::ops::Deref;

struct A;
impl A { fn foo(&self) { println!("A::foo"); } }

struct B(A);
impl Deref for B {
    type Target = A;
    fn deref(&self) -> &A { &self.0 }
}
impl B { fn foo(&self) { println!("B::foo"); } }

let b = B(A);
b.foo();  // 输出什么？

// 答案：B::foo
// 方法查找优先查找自身 impl，没找到才走 Deref 链
```

### 4.3 DerefMut

```rust
use std::ops::{Deref, DerefMut};

struct MyBox<T>(T);

impl<T> Deref for MyBox<T> {
    type Target = T;
    fn deref(&self) -> &T { &self.0 }
}

impl<T> DerefMut for MyBox<T> {
    fn deref_mut(&mut self) -> &mut T { &mut self.0 }
}

let mut b = MyBox(42i32);
*b = 100;  // DerefMut 允许你通过 MyBox 修改内部值
assert_eq!(*b, 100);

// 注意：不能只实现 DerefMut 不实现 Deref
// 编译器要求：DerefMut: Deref（因为 &mut T 可以当 &T 用）
```

### 4.4 完整的标准库 Deref 图谱

| 类型 | Deref Target | 用途 |
|:---|:---|:---|
| `String` | `str` | 让 `&String` 可以当 `&str` 用 |
| `Vec<T>` | `[T]` | 让 `&Vec<T>` 可以当 `&[T]` 用 |
| `Box<T>` | `T` | 让 `&Box<T>` 可以当 `&T` 用 |
| `Rc<T>` | `T` | 让 `&Rc<T>` 可以当 `&T` 用 |
| `Arc<T>` | `T` | 同上 + 线程安全 |
| `PathBuf` | `Path` | 让 `&PathBuf` 可以当 `&Path` 用 |
| `OsString` | `OsStr` | 同上模式 |
| `CString` | `CStr` | 同上模式 |
| `Cow<'a, B>` | `B` | `Cow<str>` → `&str`，`Cow<[T]>` → `&[T]` |
| `MutexGuard<'a, T>` | `T` | 让你直接调用被保护数据的方法 |
| `Ref<'a, T>` | `T` | 同上（RefCell 的借用） |
| `RefMut<'a, T>` | `T` | 同上 |

```rust
// 这套体系的好处：
// 你写 fn process(data: &str) 时，以下全都能传：
process("literal");            // &str
process(&String::from("x"));  // &String → &str (Deref)
process(&*Box::new("x"));     // &Box<str> → &str (Deref)
process(&*Rc::new("x"));      // &Rc<str> → &str (Deref)
process(&*Cow::from("x"));    // &Cow<str> → &str (Deref)
// 一个函数签名，覆盖了六种调用方式
```

## 5. Deref 的正确与错误用法

### 5.1 错误用法：用 Deref 做"类型继承"

```rust
// ❌ 反模式：模仿 C++ 的 is-a 继承
struct Animal {
    name: String,
}

struct Dog {
    animal: Animal,
    breed: String,
}

impl Deref for Dog {
    type Target = Animal;
    fn deref(&self) -> &Animal { &self.animal }
}

// 这看起来很"方便"：
let dog = Dog { animal: Animal { name: "Rex".into() }, breed: "Husky".into() };
println!("{}", dog.name);  // 通过 Deref 访问 animal.name

// 但问题来了：
fn register(animal: &Animal) { /* 注册到动物数据库 */ }
register(&dog);  // Dog 悄悄变成了 Animal——它本来是 Dog！

// 而且一个类型只能实现一次 Deref——如果 Dog 还"继承"了 Canine 呢？
// 只能用组合（dog.animal.name），这是 Rust 推荐的方式
```

### 5.2 正确用法 vs 错误用法的判断标准

```
Deref 应该实现，当且仅当：
  ✅ 你的类型是一个"指针"或"容器"，它的核心职责是提供对另一类型值的访问
  ✅ 你的类型和被包装类型之间有"is-a-pointer-to"或"is-a-view-of"的关系
  ✅ Deref 后得到的方法集对调用方在语义上完全合理

Deref 不应该实现，当：
  ❌ 只是"方便"——为了少打几个字母
  ❌ 被包装类型的方法会破坏你的类型的不变量（如 Email → String）
  ❌ 你的类型和被包装类型是"has-a"关系而非"pointer-to"关系
  ❌ 你需要多种不同的"视图"（Deref 只能有一个 Target）
```

### 5.3 不用 Deref 时，用什么？

```rust
struct Meters(f64);

// 方案 1：显式方法（最推荐）
impl Meters {
    fn value(&self) -> f64 { self.0 }
    fn as_f64(&self) -> &f64 { &self.0 }
}

// 方案 2：AsRef（让泛型代码能接受你的类型）
impl AsRef<f64> for Meters {
    fn as_ref(&self) -> &f64 { &self.0 }
}
// 好处：fn measure<T: AsRef<f64>>(x: T) 可以接受 Meters

// 方案 3：From/Into 双向转换
impl From<f64> for Meters {
    fn from(v: f64) -> Self { Meters(v) }
}
impl From<Meters> for f64 {
    fn from(m: Meters) -> Self { m.0 }
}
// 好处：.into() 语法糖，显式但不啰嗦
```

## 6. 三者对比：何时用什么

| 场景 | 推荐 | 原因 |
|:---|:---|:---|
| 防止混淆 `Meters` 和 `Kilometers` | newtype | 类型隔离 |
| 为外部类型实现外部 trait | newtype | 绕过孤儿规则 |
| 保证值满足约束（Email、Port） | newtype | 封装构造逻辑 |
| 简化 `Arc<Mutex<HashMap<...>>>` 签名 | 类型别名 | 纯可读性，无需类型隔离 |
| 平台相关类型统一 | 类型别名 | 零成本条件编译 |
| 让 `&MySmartPtr<T>` 能当 `&T` 用 | Deref | 智能指针语义 |
| 让 `&String` 能当 `&str` 传给任何函数 | Deref | 视图/拥有者关系 |
| 给字段起一个更语义化的名字 | 方法或 AsRef | 不要滥用 Deref |

## 7. 设计哲学

### 7.1 "显式 > 隐式"，但代价要合理

```
newtype：完全显式。.0 访问，::new() 构造。
类型别名：完全隐式。同一个类型，编译器不区分。
Deref：半隐式。自动转换，但有明确的语义限定。

Rust 的设计倾向：
  类型安全场景：newtype（哪怕多敲几个字符）
  便捷性场景：Deref（在语义合理的前提下）
  可读性场景：类型别名（当类型已经很明确时）
```

### 7.2 C++ 的"is-a 继承" vs Rust 的 newtype —— 两条完全不同的路

如果你学过 C++，你可能习惯用**继承**来解决"A 就是一种 B"的问题：

```cpp
// C++：用继承表达"UserId 就是一种 int"
class UserId : public int {  // ❌ 实际上不能继承基本类型，这只是示例
    // ...
};

// 更常见的模式：用私有继承 + using 声明
class UserId : private std::string {  // UserId "用" string 实现
public:
    using std::string::size;          // 暴露部分接口
    // 但 string 的其他方法（如 operator+=）也被暴露了？
    // C++ 的 using 声明很难精确控制
};

// 或者根本不封装，直接用 typedef/using：
using UserId = int;               // 和 int 完全一样！
using Meters = double;
using Feet = double;

// 然后：
void launch_rocket(Meters distance);
void walk(Feet distance);

launch_rocket(Feet{10.0});  // ❌ 编译通过！逻辑上不应该
// 因为 Meters 和 Feet 都是 double，类型系统不分
```

**C++ 的根本困境**：你要么选择完全相同的类型（`typedef` = 零安全），要么选择继承（语义混乱，而且 `int` 这种基本类型无法继承），要么写一个完整的 wrapper class（工作量大，且丢失了原始类型的所有运算符）。

**Rust 的 newtype 给出了第三条路**：

```rust
// Rust: newtype = C++ typedef 的便利 + 继承的类型安全 + 零运行时开销
struct Meters(f64);
struct Feet(f64);

// 编译器保证：
launch_rocket(Meters(10.0));       // ✅
// launch_rocket(Feet(10.0));      // ❌ 编译错误——类型不匹配！
// walk(10.0);                      // ❌ 裸 f64 也不能用

// 但如果你需要，所有原始操作都可以显式转发或 derive
#[derive(Debug, Clone, Copy, PartialEq, PartialOrd, Add, Mul)]
struct Meters(f64);

let a = Meters(5.0);
let b = Meters(3.0);
let c = a + b;  // 编译通过！但 a + 3.0 不行（需要显式转换）
// C++ 的隐式转换不会在这个层次生效——你必须显式决定什么操作是合法的
```

**对比总结**：

| 需求 | C++ 做法 | Rust 做法 | 优劣 |
|:---|:---|:---|:---|
| "别名" | `using X = Y;` | `type X = Y;` | 相同——完全透明 |
| "新类型" | 继承或 wrapper class | `struct X(Y);`（newtype） | Rust 零成本，C++ 有 vtable 或 wrapper 开销 |
| "is-a"关系 | `class Dog : public Animal` | 不直接支持——用 trait + 组合 | C++ 有菱形继承问题，Rust 没有 |
| 自动转换 | 隐式构造函数 / `operator T()` | `Deref` / `From` | C++ 可能静默转换，Rust 显式调用或有限自动 |

### 7.3 如果没有 newtype，你只能用类型别名——C++ 的血泪教训

```
一个真实案例（简化）：

一家航天公司的两个团队：
- 导航团队：用 double 表示角度（单位：度）
- 控制团队：用 double 表示角度（单位：弧度）

导航团队写：void adjust_course(double angle_degrees);
控制团队写：void apply_thrust(double angle_radians);

一次集成：adjust_course(0.785);  // 0.785 度
           apply_thrust(0.785);  // 也传了 0.785，但控制团队以为这是弧度！
           // → 火箭偏航

如果有 newtype：
struct Degrees(f64);
struct Radians(f64);

fn adjust_course(angle: Degrees) { }
fn apply_thrust(angle: Radians) { }

adjust_course(Degrees(0.785));  // ✅ 明确
apply_thrust(Radians(0.785));   // ✅ 明确
// adjust_course(Radians(0.785));  // ❌ 编译错误——救了火箭
```

### 7.4 Deref 的单一职责

```
一个类型只能实现一次 Deref（只有一个 Target）。
这个限制是有意为之——它强制你思考：

"我的类型到底'是'什么？它本质上是哪种指针/视图？"

C++ 的做法：一个类可以有多个 operator T() —— 隐式转换可以往多个方向
→ 歧义：编译器不知道该选哪个转换路径
→ Rust 的选择：一个类型只有一个 Deref Target → 没有歧义

如果答案是"它既像 A 又像 B"——那你应该用组合 + 显式方法，
而不是试图用一个 Deref 覆盖所有场景。

这是 Rust 对抗 C++ 多重继承混乱的设计选择。
C++ 的多重继承可以产生两个基类有同名方法的"钻石问题"——
Rust 的 Deref 是单向单层的，从根源上防止了这种歧义。
```

### 7.5 newtype ≠ 运行时开销

```
你有多少次因为"不想多写代码"而没有用 newtype？
结果是：运行时花更多时间 debug 类型混淆的 bug。

newtype 的零成本保证意味着：
  写代码时多敲 3 个字符（.0） → 省下数小时的调试时间
  编译时多一个类型检查 → 运行时没有任何额外开销

C++ 的 wrapper class 可能有额外开销（额外的构造函数、可能的虚函数表）；
Rust 的 newtype 通过 #[repr(transparent)] 保证内存布局完全等同原始类型。

这是 Rust 最基本的价值主张：用编译时的严格换取运行时的安全。
```

```rust
use std::fmt;
use std::ops::Deref;

#[derive(Debug, Clone)]
struct NonEmptyString(String);

impl NonEmptyString {
    pub fn new(s: impl Into<String>) -> Result<Self, &'static str> {
        let s: String = s.into();
        if s.is_empty() {
            Err("String must not be empty")
        } else {
            Ok(NonEmptyString(s))
        }
    }

    pub fn as_str(&self) -> &str {
        &self.0
    }
}

impl fmt::Display for NonEmptyString {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "{}", self.0)
    }
}

// ❌ 不实现 Deref
// 理由：Deref<Target = String> 会让调用方可以调用 String 的修改方法
// （如 .clear(), .push_str()），破坏"非空"不变量。
// 只提供只读的 as_str() 方法。

// From<String> 应该返回 Result，因为可能失败：
impl TryFrom<String> for NonEmptyString {
    type Error = &'static str;

    fn try_from(s: String) -> Result<Self, Self::Error> {
        NonEmptyString::new(s)
    }
}

// 使用示例
fn main() {
    let s = NonEmptyString::new("hello").unwrap();
    println!("{}", s);                     // Display
    println!("内容：{}", s.as_str());       // 只读访问

    // let s = s.clear();  // ❌ 无法访问 String 的修改方法
    // 不变量得到保证！

    let from_string: NonEmptyString = "world".to_string().try_into().unwrap();
    println!("{}", from_string);
}
```

## 12. 深度扩展

### 12.1 `#[repr(transparent)]` —— 不仅仅是 FFI

很多人以为 `#[repr(transparent)]` 只用于 FFI。其实它在纯 Rust 代码中也有重要用途：

```rust
// 默认 repr(Rust)：编译器可能优化单字段结构体的布局
// 实际上对于单字段元组结构体，repr(Rust) 已经保证与内部类型相同布局
// 但你不能依赖这个保证——它只是"当前实现"

// #[repr(transparent)]：硬性保证——ABI 与内部类型完全一致
#[repr(transparent)]
struct Wrapper<T>(T);

// 什么时候必须用？
// 1. 包含 PhantomData 的 newtype（PhantomData 可能影响 repr(Rust) 布局）
// 2. 跨 FFI 边界传递
// 3. unsafe 代码中依赖精确布局（如 transmute、裸指针操作）
// 4. 嵌套包装（Wrapper<Wrapper<T>> 的布局需要保证）
```

这里有一个特别微妙的情况：`PhantomData`。它虽然是零大小类型（ZST），但在 `repr(Rust)` 下，编译器并不保证带有 `PhantomData` 字段的结构体与去掉该字段的结构体拥有相同的 ABI 布局。这意味着如果你的 newtype 依赖 `transmute` 或裸指针操作（例如在 unsafe 代码中传递结构体给 FFI 函数），你就不能信任默认布局。`#[repr(transparent)]` 强制编译器忽略所有 ZST 字段的影响，让你可以安全地在 newtype 上叠加 `PhantomData` 而不改变底层表示：

```rust
use std::marker::PhantomData;
use std::mem;

// 没有 #[repr(transparent)]：
struct Tagged<T> {
    value: u32,
    _tag: PhantomData<T>,
}
// 在 repr(Rust) 下，PhantomData 是 ZST 但编译器不保证与 u32 布局相同
// 实际上当前编译器会优化成与 u32 相同，但你不能依赖这个

// 有 #[repr(transparent)]：硬性保证
#[repr(transparent)]
struct Tagged<T> {
    value: u32,
    _tag: PhantomData<T>,
}
// 编译器被迫保证 Tagged<T> 的 ABI 与 u32 完全一致

assert_eq!(mem::size_of::<Tagged<String>>(), 4);  // 有 #[repr(transparent)] 保证
```

### 12.2 孤儿规则的细化 —— 为什么 newtype 是唯一的"合法绕过"

```rust
// 孤儿规则的精确表述：
// impl Trait for Type 合法，当且仅当：
//   Trait 在当前 crate 中定义，或者 Type 在当前 crate 中定义

// 注意：Type 指的是"最外层类型"
// 所以 impl Display for Vec<u64> 是 ❌（Display 和 Vec 都在 std）
// 但 impl Display for MyVec 是 ✅（MyVec 在当前 crate）

// newtype 通过创建一个新的"最外层类型"绕过了规则
// 这是有意设计的——编译器信任你最外层的类型代表你的语义

// 但孤儿规则有一个微妙的例外——blanket impl：
impl<T: Display> ToString for T { ... }  // 标准库中的 blanket impl

// 你不能写：impl Display for MyVec
// 如果 MyVec 内部是 Vec<u64>，而标准库有
// impl<T: Display> ToString for T
// 这不会冲突——因为 blanket impl 不覆盖你为 MyVec 实现的 Display
// 编译器能区分"具体类型的 impl"和"blanket impl"
```

### 12.3 Deref 在模式匹配中的隐式行为

Deref 不仅仅影响 `.` 操作符——它还影响 `match` 和 `if let`：

```rust
use std::ops::Deref;

struct MyString(String);

impl Deref for MyString {
    type Target = String;
    fn deref(&self) -> &String { &self.0 }
}

fn demo(x: &MyString) {
    // match 中的自动解引用：
    match x.as_str() {
        "hello" => println!("matched!"),  // 正常工作
        _ => {}
    }

    // 等价于编译器做了 Deref 后调用 String 的方法：
    // match (*(*x)).as_str() { ... }
    // x: &MyString → Deref → &String → .as_str() → &str

    // 但这可能导致意外：
    let s = MyString("hello".into());

    // 你期望匹配 MyString 的某种模式，但实际上 Deref 到了 String：
    match &s {
        // 这里 s 被 Deref 到 String，然后匹配 String 的方法返回值
        _ => {}
    }
}
```

**规则**：`match` 和 `if let` 会对 scrutinee（被匹配的值）应用 Deref 链，就像方法调用一样。这通常不是问题，但如果你有**同名方法**在不同 Deref 层级上，就会产生歧义。

### 12.4 `Borrow` vs `AsRef` vs `Deref` —— 三种"借用"的微妙区别

这三种 trait 看起来都能"从 A 获取对 B 的引用"，但语义完全不同：

```rust
use std::borrow::Borrow;

// Deref：隐式、自动、“指针语义”
// 编译器自动插入。用于智能指针和拥有者→视图的关系。
// 一个类型只能有一个 Deref Target。

// AsRef：显式、手动、"轻量级引用转换"
// 调用方显式写 .as_ref()。一个类型可以有多个 AsRef 实现。
impl AsRef<str> for String { ... }
impl AsRef<[u8]> for String { ... }  // ✅ 可以同时有多个

// Borrow：语义强约束、"等价借用"
// Borrow<T> 要求：被借用的值与原始值有相同的 Hash、Eq、Ord 结果
// 这是 Borrow 与 AsRef 的最大区别
```

```rust
use std::borrow::Borrow;
use std::collections::HashMap;

// HashMap::get 接受 impl Borrow<Q>，不是 AsRef
let mut map: HashMap<String, i32> = HashMap::new();
map.insert("hello".to_string(), 42);

// ✅ 可以用 &str 查找，因为 String: Borrow<str>
let val = map.get("hello");  // "hello" 是 &str

// 为什么不用 AsRef？因为 AsRef 不保证 Hash/Eq 一致性
// 如果 AsRef<str> 返回的值与原 String 有不同的 hash，HashMap 会找不到
```

```rust
// 选择指南：
// ┌──────────┬──────────────────────────────────────┐
// │ 场景      │ 用什么                               │
// ├──────────┼──────────────────────────────────────┤
// │ 智能指针  │ Deref（编译器自动，一个 Target）       │
// │ 轻量转换  │ AsRef（显式调用，可以有多个实现）       │
// │ HashMap key│ Borrow（保证 Hash/Eq 一致性）         │
// │ 类型转换  │ From/Into（所有权转换）               │
// │ 视图      │ Deref（拥有者 → DST，如 String → str） │
// └──────────┴──────────────────────────────────────┘
```

### 12.5 newtype 模式与 unsafe —— 裸指针包装

newtype 在 unsafe 代码中扮演关键角色——把裸指针包装成"语义类型"：

```rust
use std::ptr::NonNull;

// ❌ 裸指针满天飞：
fn process(buf: *mut u8, len: usize) { ... }

// ✅ newtype 包装：
#[repr(transparent)]
struct Buffer {
    ptr: NonNull<u8>,
}

struct BufferMut {
    ptr: NonNull<u8>,
    len: usize,
}
// NonNull<T> 是标准库提供的"保证非空"的 newtype

// 好处：
// - Buffer 和 BufferMut 是不同的类型，不会混淆
// - NonNull 提供了 niche 优化：Option<NonNull<T>> 与 NonNull<T> 大小相同
// - 所有 unsafe 操作都集中在 Buffer 的方法中，调用方代码更安全
```

### 12.6 Deref 链与编译器优化边界

Deref 链对编译器优化的影响：

```rust
fn through_deref(b: &Box<String>) -> usize {
    b.len()  // 编译器的优化路径：
    // 1. b 是 &Box<String> → Deref → &String → Deref → &str → .len()
    // 2. 编译器尝试"看穿" Deref：
    //    - 如果 Deref::deref 被标注 #[inline]（标准库的都标注了）
    //    - 编译器可以内联 Deref 调用，直接计算最终地址
    //    - 优化后等价于：(*b.ptr).len()
}

// 但多层 Deref 链可能阻碍逃逸分析：
fn may_leak(b: &Box<String>) -> &str {
    &b[..]  // 返回对 Box 内部数据的引用
    // 编译器需要证明这个引用在 Box 被 drop 前不会悬垂
    // Deref 链增加了分析的复杂度
}

// 实际影响：对于标准库的 Deref 实现，编译器优化得很好
// 自定义 Deref 如果体量大或不标注 #[inline]，可能留下优化障碍
```

### 12.7 newtype 与 Zerocopy / bytemuck —— 安全 transmute 生态

newtype 和 `#[repr(transparent)]` 是实现安全位级转换的基础：

```rust
// bytemuck crate 提供了安全 transmute 的 trait：
// use bytemuck::{Pod, Zeroable};

// Pod = "Plain Old Data"——可以安全 transmute 的类型
// Zeroable = 全零位模式是合法的值

#[derive(Clone, Copy)]
#[repr(transparent)]
struct Color(u32);

// unsafe impl Pod for Color {}  // bytemuck 提供的 derive 或 manual
// unsafe impl Zeroable for Color {}

// 现在可以安全地从 &[u32] 转换为 &[Color]：
// let pixels: &[Color] = bytemuck::cast_slice(raw_u32_data);
// 零拷贝，零开销，编译时保证安全性——newtype + repr(transparent) 的威力
```

### 12.8 类型别名与"类型标记"模式 —— 当 newtype 太重时

```rust
// 类型别名本身不提供类型安全，但可以配合"标记"使用：
use std::marker::PhantomData;

// 轻量级标记类型——零成本的类型区分
struct Tagged<T, Tag> {
    value: T,
    _tag: PhantomData<Tag>,
}

// 定义标记（ZST）
struct UserIdTag;
struct OrderIdTag;

type UserId = Tagged<u64, UserIdTag>;
type OrderId = Tagged<u64, OrderIdTag>;

// 两个 UserId 无法混用
fn get_user(id: UserId) { }
// get_user(OrderId { value: 1, _tag: PhantomData });  // ❌ 类型不匹配！
```

那么这种"标记类型"模式与经典 newtype 相比到底有什么优劣？下面的对比帮助你根据实际场景做出选择。核心权衡在于：标记模式统一了 trait 实现（一次 `derive` 到处可用），但多了一层 `PhantomData` 的间接性；经典 newtype 更直接但需要为每个新类型重复写 boilerplate code：

```rust
// 与纯 newtype 对比：
struct UserId(u64);           // 方式 1：经典 newtype（需要 derive 各种 trait）
type UserId = Tagged<u64, UserIdTag>;  // 方式 2：标记模式（Tagged 上统一 derive）

// 方式 2 的优势：所有"标记类型"共享 Tagged 的 trait 实现
// 方式 2 的劣势：多一层 PhantomData 字段（repr(transparent) 消除）
// 选择取决于你需要多少种标记类型及其 trait 需求
```

## 13. 回顾

**本课三条知识链：**

```
newtype (struct Wrapper(T))
├── 零成本，类型完全隔离
├── 四大用途：类型安全 / 孤儿规则 / 约束保证 / API 演进
└── 黄金法则：不要随意实现 Deref

类型别名 (type Alias = T)
├── 零成本，类型完全透明
├── 三大用途：简化签名 / 语义命名 / 平台统一
└── 注意：编译器不区分别名和原类型

Deref 强制解引用
├── 编译器自动插入解引用操作
├── 方法查找沿 Deref 链进行
├── 正确用法：智能指针、拥有者→视图
└── 错误用法：类型继承、便利性转换
```

```
三者选择速查：

需要类型隔离？          → newtype
需要简化复杂签名？      → 类型别名
需要"当 T 用"的自动转换？→ Deref（且语义合理）
需要透明重命名？        → 类型别名
需要绕过孤儿规则？      → newtype
需要保护不变量？        → newtype（不实现 Deref）
```
