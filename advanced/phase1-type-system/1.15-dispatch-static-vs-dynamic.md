# 1.15 静态分发 vs 动态分发——编译期决策的显式选择

> 在 C++ 里，模板永远是静态分发，虚函数永远是动态分发——你没办法选择"这个模板参数我只想擦除成一个虚表"。Python 里所有调用都是动态分发，你没法说"这段代码我希望编译时完全展开以提升性能"（PyPy 的 JIT 做了一些，但不是语言层面的语义）。Rust 给了你三个按钮：`fn foo<T: Trait>(t: T)`（静态分发，单态化，每个具体类型生成一份代码）、`fn foo(t: &dyn Trait)`（动态分发，虚表间接调用，类型擦除）、`fn foo(t: impl Trait)`（静态分发但类型隐藏）。你能在同一个语言里精确控制"这个函数的调用开销是零抽象还是有一个间接跳转"——这既是性能工程师的武器，也是 API 设计师的选择。

## 1. 泛型与单态化——静态分发的完整机制

### 1.1 泛型编译的底层过程

```rust
// 你写的是这个：
fn print_value<T: std::fmt::Display>(value: T) {
    println!("value = {}", value);
}

fn main() {
    print_value(42i32);       // 调用 1
    print_value("hello");     // 调用 2
    print_value(3.14f64);     // 调用 3
}

// 编译器实际生成的（概念上）：
// 编译器扫描所有调用点，为每个具体类型生成一份独立的函数副本

fn print_value_i32(value: i32) {
    println!("value = {}", value);
}

fn print_value_str(value: &str) {
    println!("value = {}", value);
}

fn print_value_f64(value: f64) {
    println!("value = {}", value);
}

// main() 中的调用会被替换为：
// print_value_i32(42i32);
// print_value_str("hello");
// print_value_f64(3.14f64);

// 这个过程叫"单态化"（monomorphization）——
// 从 "mono"（单） + "morph"（形态）——为每个具体类型生成"单一形态"的代码
```

### 1.2 单态化对性能的影响——零成本抽象的基石

```rust
use std::time::Instant;

// 泛型版本——看似有"抽象开销"
fn max_of<T: PartialOrd>(a: T, b: T) -> T {
    if a > b { a } else { b }
}

// 手动特化版本——没有抽象
fn max_of_i32(a: i32, b: i32) -> i32 {
    if a > b { a } else { b }
}

fn main() {
    let x: i32 = 1;
    let y: i32 = 2;

    // 生成的机器码完全一样
    let m1 = max_of(x, y);     // 泛型版本
    let m2 = max_of_i32(x, y); // 手动版本
    // LLVM 会把 max_of 的 i32 单态化版本和 max_of_i32 优化到相同指令

    // 这就是"零成本抽象"：
    // 泛型代码在编译后和手写特化代码完全一样
    // 没有虚表查找，没有间接调用，没有装箱
}
```

### 1.3 单态化的代价——代码膨胀

```rust
// 单态化的代价：代码膨胀（code bloat）

fn process<T: std::fmt::Debug>(items: &[T]) {
    for item in items {
        println!("{:?}", item);
    }
}

fn main() {
    // 每次用不同类型调用 process，编译器生成一份新代码
    process(&[1i32, 2, 3]);         // 生成 process<i32>     ——第一份副本
    process(&[1i64, 2, 3]);         // 生成 process<i64>     ——第二份副本
    process(&["a", "b", "c"]);      // 生成 process<&str>    ——第三份副本
    process(&[1u8, 2, 3]);          // 生成 process<u8>      ——第四份副本
    process(&[1u16, 2, 3]);         // 生成 process<u16>     ——第五份副本

    // 如果你用了 10 种不同类型调用同一个泛型函数：
    // → 编译器生成 10 份几乎相同的机器码
    // → 每一份都"特化"到具体类型——调用 println!("{:?}", &i32) 而非泛型的 Debug
    // → 好处：每份都是最优的（零抽象）
    // → 坏处：最终的二进制文件变大了

    // 极端案例：
    // fn foo<T: Trait>(t: T) {} + 50 个实现 Trait 的类 + 每个类都调用 foo
    // = 50 份 foo 的机器码 —— 即使它们只有 Debug print 的行不一样
}

// 什么时候代码膨胀是问题：
// 1. 嵌入式系统：flash 空间有限（几百 KB 的二进制）
// 2. WebAssembly：下载大小直接影响加载时间
// 3. 大泛型函数：函数体很长（>100 行）+ 使用很多具体类型
// 4. 多层泛型：A<B<C<D>>> —— 每一层展开产生指数级代码
```

### 1.4 LLVM 对单态化代码的优化能力

```rust
// 单态化不仅"展开"代码，还让 LLVM 可以做跨函数的深度优化

fn compute<T: std::ops::Add<Output = T> + std::ops::Mul<Output = T>>(a: T, b: T, c: T) -> T {
    let sum = a + b;
    let product = sum * c;
    product
}

fn main() {
    let result = compute(2i32, 3, 4);
    // 单态化版本：
    // fn compute_i32(a: i32, b: i32, c: i32) -> i32 {
    //     let sum = a + b;           // LLVM 看到的：i32 加法指令
    //     let product = sum * c;     // LLVM 看到的：i32 乘法指令
    //     product
    // }
    // 
    // LLVM 能做：
    // 1. 内联 compute_i32 到 main（因为函数体不大）
    // 2. 常量折叠：(2 + 3) * 4 = 20
    // 3. 消除 dead code
    // → 最终机器码可能是：mov eax, 20; ret
    // → 整个泛型函数在编译时被完全求值！

    println!("{}", result); // 打印 20——运行时成本几乎为 0
}

// 对比：如果用的是 trait object（动态分发），LLVM 没法内联——
// 因为具体类型在编译时未知，虚表调用是一个间接跳转
// LLVM 无法"穿透"虚表去优化调用目标和调用方之间的交互
```

## 2. Trait Object 与动态分发——运行时的灵活性

### 2.1 虚表（vtable）的内部结构

```rust
trait Animal {
    fn speak(&self) -> String;
    fn default_action(&self) -> String {
        format!("{} is resting...", self.speak())
    }
}

struct Dog;
struct Cat;

impl Animal for Dog {
    fn speak(&self) -> String { "Woof!".into() }
}

impl Animal for Cat {
    fn speak(&self) -> String { "Meow!".into() }
}

// Trait object 的内存布局：
// 
// &dyn Animal 是一个胖指针（两个 usize）：
// ┌──────────────────────────────────────────────┐
// │  ptr_to_data  │  ptr_to_vtable               │
// │  (指向 Dog/Cat 的数据)  │  (指向虚表)        │
// └──────────────────────────────────────────────┘
//
// 虚表的结构（每个实现 Trait 的类型各有一份）：
// ┌─────────────────────────────────┐
// │  drop_in_place   (析构函数指针)  │  ← offset 0
// │  size            (类型大小)      │  ← offset 1
// │  align           (对齐)          │  ← offset 2
// │  speak           (方法指针)      │  ← offset 3
// │  default_action  (方法指针)      │  ← offset 4
// │  ...                            │
// └─────────────────────────────────┘
//
// 调用 t.speak() 变成：
// ((*t.vtable)[speak_offset])(t.data_ptr)
// 即：从虚表中取出 speak 的函数指针，然后调用它

fn demo_vtable() {
    let dog = Dog;
    let cat = Cat;

    let animals: Vec<&dyn Animal> = vec![&dog, &cat];

    for animal in animals {
        // 每次调用都是：读 vtable → 取函数指针 → 间接调用
        println!("{}", animal.speak());
    }
}
```

### 2.2 动态分发的性能代价——微观测量

```rust
use std::time::Instant;

trait Calculator {
    fn compute(&self, x: i64) -> i64;
}

struct Add10;
impl Calculator for Add10 {
    fn compute(&self, x: i64) -> i64 { x + 10 }
}

struct Mul10;
impl Calculator for Mul10 {
    fn compute(&self, x: i64) -> i64 { x * 10 }
}

// 静态分发版本
fn sum_static<C: Calculator>(calc: &C, values: &[i64]) -> i64 {
    values.iter().map(|v| calc.compute(*v)).sum()
}

// 动态分发版本
fn sum_dynamic(calc: &dyn Calculator, values: &[i64]) -> i64 {
    values.iter().map(|v| calc.compute(*v)).sum()
}

// 微观性能对比：
// ┌─────────────────┬──────────────┬─────────────────────┐
// │ 版本            │ 调用方式     │ 每次迭代开销        │
// ├─────────────────┼──────────────┼─────────────────────┤
// │ static (泛型)   │ 直接调用     │ ~0 overhead         │
// │ static (impl T) │ 直接调用     │ ~0 overhead         │
// │ dynamic (&dyn)  │ vtable 间接  │ ~1-3ns (取决于 CPU) │
// └─────────────────┴──────────────┴─────────────────────┘
//
// 1-3ns 看起来可以忽略——但这是在 100 万次迭代的循环中
// 累计下来就是毫秒级的差异
// 在热路径（hot path）中，累积效应不可忽视
```

### 2.3 动态分发的关键优势——异构集合

```rust
trait Widget {
    fn render(&self) -> String;
    fn id(&self) -> u64;
}

struct Button { id: u64, label: String }
struct TextBox { id: u64, text: String }
struct Slider { id: u64, value: f64 }

impl Widget for Button {
    fn render(&self) -> String { format!("[Button: {}]", self.label) }
    fn id(&self) -> u64 { self.id }
}
impl Widget for TextBox {
    fn render(&self) -> String { format!("[TextBox: {}]", self.text) }
    fn id(&self) -> u64 { self.id }
}
impl Widget for Slider {
    fn render(&self) -> String { format!("[Slider: {}]", self.value) }
    fn id(&self) -> u64 { self.id }
}

// 异构集合——这是泛型做不到的：
fn render_all() {
    let widgets: Vec<Box<dyn Widget>> = vec![
        Box::new(Button { id: 1, label: "OK".into() }),
        Box::new(TextBox { id: 2, text: "Hello".into() }),
        Box::new(Slider { id: 3, value: 0.75 }),
    ];

    // 用泛型的话：
    // Vec<T: Widget> —— 所有元素必须是同一类型
    // 你不能把 Button、TextBox、Slider 放在同一个 Vec 里

    for widget in &widgets {
        println!("widget #{} → {}", widget.id(), widget.render());
    }
}
```

### 2.4 动态分发解决递归类型问题

```rust
// 递归类型必须用间接引用——Box<dyn Trait> 是解决方案之一

// ❌ 编译错误：递归类型有无限大小
// struct ListNode {
//     value: i32,
//     next: Option<ListNode>,  // 编译错误！ListNode 包含自身
// }

// ✅ 用 Box 引入间接引用
struct ListNode {
    value: i32,
    next: Option<Box<ListNode>>,  // Box<ListNode> 的大小 = usize（指针）
}

// ✅ 用 trait object：
trait Node {
    fn value(&self) -> i32;
    fn next(&self) -> Option<&dyn Node>;
}

struct IntNode {
    value: i32,
    next: Option<Box<dyn Node>>,
}

impl Node for IntNode {
    fn value(&self) -> i32 { self.value }
    fn next(&self) -> Option<&dyn Node> {
        self.next.as_deref()
    }
}

// trait object 版递归的额外好处：
// 链表中可以混合不同类型的节点
struct StringNode {
    value: String,
    next: Option<Box<dyn Node>>,
}
impl Node for StringNode {
    fn value(&self) -> i32 { self.value.len() as i32 }
    fn next(&self) -> Option<&dyn Node> {
        self.next.as_deref()
    }
}
```

## 3. `impl Trait`——第三条路：静态分发 + 类型隐藏

### 3.1 `impl Trait` 在参数位置——"我不关心具体类型"

```rust
trait Speaker {
    fn speak(&self) -> &str;
}

struct Dog;
impl Speaker for Dog { fn speak(&self) -> &str { "Woof" } }

struct Cat;
impl Speaker for Cat { fn speak(&self) -> &str { "Meow" } }

// 三种写法——三种分发策略：

// 写法 1：泛型（静态分发，类型暴露）
fn announce1<T: Speaker>(speaker: &T) {
    println!("{}", speaker.speak());
}
// 调用方可以知道 T 的具体类型——类型是透明的

// 写法 2：impl Trait（静态分发，类型隐藏）
fn announce2(speaker: &impl Speaker) {
    println!("{}", speaker.speak());
}
// 调用方不知道具体类型——但编译器仍然是静态分发
// impl Trait 在参数位置是泛型的语法糖

// 写法 3：trait object（动态分发，类型隐藏）
fn announce3(speaker: &dyn Speaker) {
    println!("{}", speaker.speak());
}
// 运行时虚表调用——具体类型被擦除

fn main() {
    let dog = Dog;
    let cat = Cat;

    announce1(&dog);   // 编译为 announce1::<Dog>(&dog)
    announce2(&dog);   // 编译为 announce2::<Dog>(&dog) ——和上面一样
    announce3(&dog);   // 编译为 vtable 间接调用
}
```

### 3.2 `impl Trait` vs 泛型——API 层面的关键区别

```rust
// 关键区别：impl Trait 隐藏类型，泛型暴露类型

// 泛型版本——类型暴露给调用方：
fn make_identity<T>(value: T) -> T {
    value  // 调用方知道 T = i32
}

let x: i32 = make_identity(42);
// make_identity::<i32>(42) —— T = i32 对调用方可见

// impl Trait 版本——类型对调用方隐藏：
fn make_speaker() -> impl Speaker {
    // 调用方只知道返回值实现了 Speaker
    // 不知道具体是 Dog 还是 Cat
    Dog
}

let s = make_speaker();
// s 的类型是 "impl Speaker" ——编译器内部知道是 Dog
// 但代码中不能写成 let s: Dog = make_speaker(); ——编译错误
// 也不能匹配：match make_speaker() { Dog => ... } ——编译错误

// API 设计的含义：
// - 用泛型：调用方可以依赖具体类型（适合库的内部 API）
// - 用 impl Trait：调用方只能依赖 trait 接口（适合公共 API——封装性更好）
```

### 3.3 返回值 `impl Trait` 的限制——只能返回单一类型

```rust
trait Shape {
    fn area(&self) -> f64;
}
struct Circle { radius: f64 }
impl Shape for Circle { fn area(&self) -> f64 { std::f64::consts::PI * self.radius * self.radius } }
struct Rectangle { width: f64, height: f64 }
impl Shape for Rectangle { fn area(&self) -> f64 { self.width * self.height } }

// ❌ 编译错误！impl Trait 在返回值位置要求：所有代码路径返回同一具体类型
// fn make_shape(kind: &str) -> impl Shape {
//     match kind {
//         "circle" => Circle { radius: 1.0 },
//         "rect" => Rectangle { width: 2.0, height: 3.0 },
//         _ => Circle { radius: 0.5 },
//     }
// }
// 错误：match arms have incompatible types
//   expected `Circle`, found `Rectangle`

// ✅ 如果需要返回不同类型 → 必须用 trait object：
fn make_shape(kind: &str) -> Box<dyn Shape> {
    match kind {
        "circle" => Box::new(Circle { radius: 1.0 }),
        "rect" => Box::new(Rectangle { width: 2.0, height: 3.0 }),
        _ => Box::new(Circle { radius: 0.5 }),
    }
}
```

## 4. 决策框架——何时用哪种分发

### 4.1 四维决策矩阵

```rust
// 选择分发策略时，考虑四个维度：

// 维度 1：性能需求
// 热路径（hot path） → 静态分发（泛型 / impl Trait）
// 冷路径（cold path） → 动态分发也可以
// 微秒级敏感的循环内部 → 必须静态分发

// 维度 2：灵活性需求
// 需要异构集合 → 动态分发（Box<dyn Trait>）
// 需要递归类型 → 动态分发
// 需要运行时决定具体类型 → 动态分发
// 编译时已知所有类型 → 静态分发

// 维度 3：二进制大小
// 目标文件大小敏感（WASM、嵌入式） → 考虑动态分发减少代码膨胀
// 二进制大小不敏感（桌面/服务器） → 静态分发通常更好

// 维度 4：API 设计
// 公共 API 需要稳定性（不暴露类型） → impl Trait（返回值）/ trait object
// 内部 API 需要灵活性 → 泛型（调用方知道具体类型）
// 库的公共接口不希望泛型参数"传染"调用方 → trait object

// 总结成一个表：
// ┌──────────────────┬──────────┬───────────┬─────────────────────┐
// │ 场景             │ 泛型     │ impl Tr.  │ Box<dyn Trait>      │
// ├──────────────────┼──────────┼───────────┼─────────────────────┤
// │ 零成本抽象       │ ✅       │ ✅        │ ❌ (虚表开销)       │
// │ 异构集合         │ ❌       │ ❌        │ ✅                  │
// │ 递归类型         │ ❌       │ ❌        │ ✅                  │
// │ 隐藏具体类型     │ ❌       │ ✅        │ ✅                  │
// │ 多态返回值       │ ❌       │ ❌ (单类型)│ ✅                  │
// │ 编译期类型检查   │ ✅       │ ✅        │ ✅ (部分退化)       │
// │ 栈分配           │ ✅       │ ✅        │ ❌ (Box 是堆分配)    │
// └──────────────────┴──────────┴───────────┴─────────────────────┘
```

### 4.2 减少二进制膨胀——用动态分发"浓缩"冷路径

```rust
// 实际策略：用动态分发包装泛型代码

// 核心逻辑——用泛型保证性能（热路径在函数内部）
fn process_items_hot_path<T: std::fmt::Debug>(items: &[T]) -> Vec<String> {
    items.iter().map(|x| format!("{:?}", x)).collect()
}

// 包装层——用 trait object 减少代码生成
fn process_items_boxed(items: &[Box<dyn std::fmt::Debug>]) -> Vec<String> {
    items.iter().map(|x| format!("{:?}", x)).collect()
}

// 如果你有 50 种类型要处理：
// 方案 A（全泛型）：生成 50 份 process_items_hot_path 的代码
// 方案 B（trait object）：只生成 1 份 process_items_boxed 的代码
//   （代价：每个元素的调用是虚表间接调用，且每个值需要在堆上分配 Box）

// 这是经典的"外包"策略：
// 主逻辑用 trait object（一份代码） + 把个别热函数"特化"成泛型版本
fn architecture_example() {
    // 主循环用 trait object——代码量小
    let widgets: Vec<Box<dyn Widget>> = /* ... */;

    for widget in &widgets {
        widget.render();  // 虚表调用——但 render 不是热路径
    }

    // 渲染循环用静态分发
    fn draw_widget<W: Widget>(widget: &W) {
        // 这是热路径——需要最高的性能
        // 对每个 W 类型生成独立的单态化代码
    }
}
```

### 4.3 库设计中的选择——泛型参数的"传染性"

```rust
// 泛型的"传染性"：一旦用了泛型，调用方也必须变成泛型

// 库 A（用 trait object）——用户可以轻松使用
pub struct Logger {
    writer: Box<dyn std::io::Write>,  // trait object：不需要泛型参数
}

impl Logger {
    pub fn new(writer: Box<dyn std::io::Write>) -> Self {
        Logger { writer }
    }

    pub fn log(&mut self, msg: &str) {
        writeln!(self.writer, "LOG: {}", msg).unwrap();
    }
}

// 用户代码——干净，没有泛型参数
fn use_logger() {
    let logger = Logger::new(Box::new(std::io::stdout()));
    // logger 的类型是 Logger——不需要泛型参数
}

// 库 B（用泛型）——用户必须携带泛型参数
pub struct GenericLogger<W: std::io::Write> {
    writer: W,
}

impl<W: std::io::Write> GenericLogger<W> {
    pub fn new(writer: W) -> Self {
        GenericLogger { writer }
    }
}

// 用户代码——泛型参数向上传播
fn use_generic_logger<W: std::io::Write>(writer: W) {
    let logger = GenericLogger::new(writer);
    // use_generic_logger 也必须是泛型——或者调用方已经知道 W 的具体类型
    // 泛型"传染"到调用方的整个链条
}

// 选择指南：
// - 库的公共类型（struct/enum）→ 考虑 trait object 减少泛型传染
// - 库的内部实现 → 泛型最大化性能
// - 函数参数 → impl Trait 在性能和封装间取得平衡
```

## 5. 常见陷阱

### 陷阱 1："泛型总是比 trait object 快"——不一定

```rust
// 直觉：静态分发 > 动态分发（性能）
// 实际：也取决于上下文

// 场景：大函数 + 多具体类型 = 代码膨胀 → i-cache miss
// i-cache（指令缓存）miss 的代价可能超过虚表调用的开销

// 场景：只调用一次的函数
// 静态分发：生成代码 → 给 i-cache 带来压力
// 动态分发：虚表调用 → 少生成代码 → i-cache 更友好
// 在冷路径上，动态分发可能"总体更快"——因为缓存更热

// 正确的思维：
// "泛型给你零成本抽象的潜力——但不保证在所有场景下都是最快的"
// 用性能测量工具（perf、flamegraph）验证，不要凭直觉做选择
```

### 陷阱 2：`Box<dyn Trait>` 不能直接用于需要 `Sized` 的地方

```rust
trait MyTrait {}

// ❌ 很多标准库操作要求 Sized
fn take_ownership<T: MyTrait>(t: T) {}  // ✅ T 是 Sized（默认）

// take_ownership(my_boxed_trait_object);  // ❌ dyn MyTrait 不是 Sized

// 解决办法：用泛型包装
fn take_boxed(b: Box<dyn MyTrait>) {}  // ✅ Box<dyn MyTrait> 是 Sized

// 教训：dyn Trait 本身不是 Sized——不能放在栈上，
// 必须通过引用或 Box 使用
```

### 陷阱 3：`impl Trait` 在返回值位置时，不同代码路径必须返回同一类型

```rust
trait Animal { fn name(&self) -> &str; }

struct Dog;
impl Animal for Dog { fn name(&self) -> &str { "dog" } }

struct Cat;
impl Animal for Cat { fn name(&self) -> &str { "cat" } }

// ❌ 编译错误——两个分支返回不同类型
// fn pick_animal(choose_cat: bool) -> impl Animal {
//     if choose_cat {
//         Cat    // 返回 Cat 类型
//     } else {
//         Dog    // 返回 Dog 类型——和 Cat 不是同一类型！
//     }
// }

// ✅ 返回 trait object
fn pick_animal(choose_cat: bool) -> Box<dyn Animal> {
    if choose_cat {
        Box::new(Cat)
    } else {
        Box::new(Dog)
    }
}
```

### 陷阱 4：`Box<dyn Trait>` 移出值时的障碍——`Sized` 限制

```rust
trait Value { fn get(&self) -> i32; }

struct IntValue(i32);
impl Value for IntValue { fn get(&self) -> i32 { self.0 } }

fn consume(v: Box<dyn Value>) {
    // ❌ 不能从 Box<dyn Value> 中移出值：
    // let inner: dyn Value = *v;  // dyn Value 不是 Sized——不能在栈上存在
    // 
    // ✅ 你只能通过引用使用：
    let val: i32 = v.get();  // 通过 trait 方法访问

    // 如果想恢复具体类型——需要用 Any 做 downcast
    use std::any::Any;
    fn downcast_example(v: Box<dyn Any>) {
        if let Some(int_val) = v.downcast_ref::<IntValue>() {
            println!("downcasted: {}", int_val.0);
        }
    }
    // 但这要求 trait 继承 Any
}
```

## 7. 练习答案

### 练习 1 答案

```
(a) 游戏渲染管线 → 泛型（静态分发）
   理由：每帧数千次调用的热路径，需要零抽象开销。
   LLVM 可以内联和深度优化单态化后的代码。

(b) 插件系统 → Box<dyn Trait>（动态分发）
   理由：运行时加载插件意味着编译时不知道具体类型。
   异构集合是动态分发的核心用例。

(c) 公共库 API → impl Trait（静态分发但类型隐藏）
   理由：不暴露内部类型（封装性好），同时保持静态分发的性能。
   这是 impl Trait 在返回值位置的经典用例。

(d) 日志库 → Box<dyn Trait>（动态分发）
   理由：运行时选择输出目标，需要异构能力。
   日志不是热路径（相对于业务逻辑），虚表开销可接受。

(e) WASM 项目 → 优先 trait object，热路径单独特化
   理由：需要控制二进制大小。
   将冷路径合并为动态分发减少代码生成量，
   热路径才用泛型特化。
```

### 练习 2 答案

```rust
trait Format {
    fn parse(&self, data: &str) -> Vec<String>;
    fn name(&self) -> &str;
}

struct JsonFormat;
impl Format for JsonFormat {
    fn parse(&self, data: &str) -> Vec<String> {
        vec![data.to_string()]
    }
    fn name(&self) -> &str { "json" }
}

struct CsvFormat;
impl Format for CsvFormat {
    fn parse(&self, data: &str) -> Vec<String> {
        data.split(',').map(|s| s.to_string()).collect()
    }
    fn name(&self) -> &str { "csv" }
}

// 重构版本——去除了泛型参数
struct FileProcessor {
    formats: Vec<Box<dyn Format>>,
    processed_count: usize,
}

impl FileProcessor {
    fn new() -> Self {
        FileProcessor {
            formats: Vec::new(),
            processed_count: 0,
        }
    }

    fn add_format(&mut self, format: Box<dyn Format>) {
        self.formats.push(format);
    }

    fn process_with(&mut self, format_name: &str, data: &str) -> Option<Vec<String>> {
        self.processed_count += 1;
        self.formats
            .iter()
            .find(|f| f.name() == format_name)
            .map(|f| f.parse(data))
    }
}

// 使用示例：
fn demo() {
    let mut processor = FileProcessor::new();
    processor.add_format(Box::new(JsonFormat));
    processor.add_format(Box::new(CsvFormat));

    let result = processor.process_with("csv", "a,b,c");
    println!("{:?}", result);  // Some(["a", "b", "c"])
}

// 关键变化：
// 1. FileProcessor 不再是泛型——去除了 <F: Format>
// 2. formats 字段从 F 变为 Vec<Box<dyn Format>>
// 3. 失去了编译时类型检查（format_name 是字符串查找——运行时错误可能）
// 4. 增加了灵活性——可以在运行时动态添加新格式
```

### 练习 3 答案

```rust
// 包装结构体，组合两个闭包
struct Chained<F, G> {
    f: F,
    g: G,
}

impl<F, G, T> FnOnce<(T,)> for Chained<F, G>
where
    F: FnOnce(T) -> T,
    G: FnOnce(T) -> T,
{
    type Output = T;
    extern "rust-call" fn call_once(self, args: (T,)) -> T {
        (self.g)((self.f)(args.0))
    }
}

impl<F, G, T> FnMut<(T,)> for Chained<F, G>
where
    F: FnMut(T) -> T,
    G: FnMut(T) -> T,
{
    extern "rust-call" fn call_mut(&mut self, args: (T,)) -> T {
        (self.g)((self.f)(args.0))
    }
}

impl<F, G, T> Fn<(T,)> for Chained<F, G>
where
    F: Fn(T) -> T,
    G: Fn(T) -> T,
{
    extern "rust-call" fn call(&self, args: (T,)) -> T {
        (self.g)((self.f)(args.0))
    }
}

// 这是 unstable 的 API——在实际项目中，更实际的做法是：

fn chain_operations<F, G, T>(f: F, g: G) -> impl Fn(T) -> T
where
    F: Fn(T) -> T + 'static,
    G: Fn(T) -> T + 'static,
    T: 'static,
{
    // 直接用闭包组合——Rust 会把它变成一个匿名类型
    move |x: T| g(f(x))
    // 返回类型是 "impl Fn(T) -> T" ——具体类型被隐藏
}

// 使用：
fn add_one(x: i32) -> i32 { x + 1 }
fn times_two(x: i32) -> i32 { x * 2 }

fn demo() {
    let combined = chain_operations(add_one, times_two);
    // combined 的类型是 "impl Fn(i32) -> i32"
    // 外部代码无法知道具体的匿名类型

    assert_eq!(combined(5), 12);  // (5 + 1) * 2 = 12
}
```

### 练习 4 答案

```rust
use std::time::Instant;

trait Operate {
    fn operate(&self, x: i64) -> i64;
}

struct Add1;
struct Mul2;
impl Operate for Add1 { fn operate(&self, x: i64) -> i64 { x + 1 } }
impl Operate for Mul2 { fn operate(&self, x: i64) -> i64 { x * 2 } }

// 版本 A：泛型（静态分发）
fn sum_with_generic<O: Operate>(op: &O, data: &[i64]) -> i64 {
    let mut sum = 0;
    for &v in data {
        sum += op.operate(v);
    }
    sum
}

// 版本 B：&dyn Trait（动态分发）
fn sum_with_dyn_ref(op: &dyn Operate, data: &[i64]) -> i64 {
    let mut sum = 0;
    for &v in data {
        sum += op.operate(v);
    }
    sum
}

// 版本 C：Box<dyn Trait>（动态分发 + 堆分配）
fn sum_with_dyn_box(op: &Box<dyn Operate>, data: &[i64]) -> i64 {
    let mut sum = 0;
    for &v in data {
        sum += op.operate(v);
    }
    sum
}

fn benchmark() {
    const N: usize = 1_000_000;
    const WARMUP: usize = 10;
    const ITERATIONS: usize = 100;

    let data: Vec<i64> = (0..N as i64).collect();

    let op = Add1;
    let boxed_op: Box<dyn Operate> = Box::new(Add1);

    // Warmup
    for _ in 0..WARMUP {
        sum_with_generic(&op, &data);
        sum_with_dyn_ref(&op, &data);
        sum_with_dyn_box(&boxed_op, &data);
    }

    // Benchmark: 泛型
    let start = Instant::now();
    for _ in 0..ITERATIONS {
        sum_with_generic(&op, &data);
    }
    let elapsed_generic = start.elapsed();

    // Benchmark: &dyn Trait
    let start = Instant::now();
    for _ in 0..ITERATIONS {
        sum_with_dyn_ref(&op, &data);
    }
    let elapsed_dyn_ref = start.elapsed();

    // Benchmark: Box<dyn Trait>
    let start = Instant::now();
    for _ in 0..ITERATIONS {
        sum_with_dyn_box(&boxed_op, &data);
    }
    let elapsed_dyn_box = start.elapsed();

    println!("泛型 (static dispatch):     {:?}", elapsed_generic);
    println!("&dyn Trait (dynamic ref):   {:?}", elapsed_dyn_ref);
    println!("Box<dyn Trait> (dynamic):   {:?}", elapsed_dyn_box);

    // 典型结果（因硬件而异）：
    // 泛型:      ~12ms
    // &dyn:      ~18ms（虚表调用的额外开销 ~1.5x）
    // Box<dyn>:  ~18ms（和 &dyn 相似——因为 operate 不涉及 Box 操作）
    //
    // 关键发现：
    // 1. 动态分发的开销是真实存在的（~50% 慢）
    // 2. 但在大多数业务代码中，这种差异被 I/O、网络、数据库操作淹没
    // 3. 只有在紧密循环中，静态分发的优势才显著
}
```

### 8.1 C++ 对比——模板 vs 虚函数，以及 Rust 的中间地带

```cpp
// C++ 中，分发策略由"语言机制"决定，而非"选择"：

// 1. 模板 = 永远静态分发
template <typename T>
T max_of(T a, T b) {
    return a > b ? a : b;
}
// 每个 T 生成一份代码——无法"选"用虚表版本
// 如果想减少代码膨胀：没办法（除非用类型擦除技术如 std::function）

// 2. 虚函数 = 永远动态分发
class Animal {
public:
    virtual std::string speak() = 0;
};
// 每个虚函数调用都通过 vtable ——无法"选"用直接调用
// 即使编译时知道具体类型（devirtualization 有时能优化，但不可靠）

// 3. CRTP（Curiously Recurring Template Pattern）——模拟静态多态
template <typename Derived>
class AnimalBase {
public:
    void make_sound() {
        static_cast<Derived*>(this)->speak_impl();
    }
};
// 这是一个 hack——用模板模拟静态多态
// 但代码可读性差，错误信息晦涩

// Rust 让你在这三者之间做"显式选择"：
// - fn foo<T: Trait>(t: T)     → C++ 模板（静态）
// - fn foo(t: &dyn Trait)      → C++ 虚函数（动态）
// - fn foo(t: impl Trait)      → C++ 没有直接等价物（静态分发 + 类型隐藏）

// 关键差异：
// C++ 的模板和虚函数是"不同的世界"——互不兼容
// Rust 的泛型和 trait object 共享同一套 trait 定义——无缝切换
// 你可以从泛型代码快速重构到 trait object 代码，因为 trait 定义不变
```

### 8.2 如果没有 trait object——如何实现异构集合

```rust
// 假设 Rust 没有 trait object——你该如何处理异构集合？

// 方案 1：枚举（enum）——每个变体手动处理
enum Widget {
    Button { id: u64, label: String },
    TextBox { id: u64, text: String },
    Slider { id: u64, value: f64 },
}

impl Widget {
    fn render(&self) -> String {
        match self {
            Widget::Button { id, label } => format!("[Button-{}: {}]", id, label),
            Widget::TextBox { id, text } => format!("[TextBox-{}: {}]", id, text),
            Widget::Slider { id, value } => format!("[Slider-{}: {}]", id, value),
        }
    }
}

// 优点：栈分配、无虚表开销、模式匹配穷尽
// 毛病：
// 1. 每添加一种新 Widget，必须修改 enum 定义和所有 match 语句
// 2. 库用户不能定义新的 Widget 变体——enum 是封闭的
// 3. 代码重复——每个变体都要写一遍 render 的分支

// 方案 2：手动虚表——自己实现类似 trait object 的东西
struct ManualVTable {
    render_fn: fn(*const ()) -> String,
    drop_fn: fn(*mut ()),
    data: *mut (),
}
// 这基本上就是编译器为 dyn Trait 生成的代码
// ——重新发明轮子，而且更危险（没有编译器帮你验证类型安全）

// 方案 3：Any + downcast
use std::any::Any;

fn render_widget(widget: &dyn Any) -> Option<String> {
    if let Some(btn) = widget.downcast_ref::<Button>() {
        Some(format!("[Button: {}]", btn.label))
    } else if let Some(tb) = widget.downcast_ref::<TextBox>() {
        Some(format!("[TextBox: {}]", tb.text))
    } else {
        None  // 未知类型——编译期不保证安全性
    }
}
// 没有任何编译期保证——新增类型后编译器不会提醒你更新此函数

// 结论：trait object 是异构集合问题的最优解
// 它提供了类型安全的动态多态——既不是"裸虚表"也不"只有枚举"
```

### 8.3 单态化的内部机制——编译器如何生成代码

```
Monomorphization 的完整流程（以 rustc 为例）：

1. 收集阶段（Collection）
   rustc 扫描整个 crate，找到所有泛型函数/类型的"使用点"
   fn foo<T: Debug>(x: T) {...}
   使用点：foo(42i32)、foo("hello")、foo(3.14f64)
   收集结果：[foo<i32>, foo<&str>, foo<f64>]

2. 实例化阶段（Instantiation）
   为每个收集到的具体类型组合生成代码副本：
   - 将参数 T 替换为 i32
   - 展开 impl Debug for i32 的具体方法
   - 替换方法调用为直接调用（去掉 trait 层）

3. 优化阶段（Optimization）
   LLVM 后端处理单态化后的代码：
   - 内联（inlining）：小函数直接展开到调用点
   - 常量折叠（constant folding）：编译时能算出结果就直接替换
   - 死代码消除（dead code elimination）：未用到的分支删除
   - 循环优化（loop optimization）：因为类型已知，可以针对具体类型优化

4. 链接阶段（Linking）
   如果有完全相同的单态化副本（不同 crate 中），链接器合并它们（COMDAT）

关键点：
- 单态化是"全自动"的——你不需要标注 #[specialize]（不像 C++ 的显式模板实例化）
- 编译器只为你实际用到的类型组合生成代码——不是为所有可能的类型
- LLVM 的后端优化是静态分发性能优势的关键——生成代码的质量取决于 LLVM
```

### 8.4 何时编译器可以"去虚拟化"（devirtualization）

```rust
// Devirtualization：编译器在编译时知道 trait object 的具体类型，
// 从而把虚表调用优化为直接调用

fn make_dog() -> Box<dyn Animal> {
    Box::new(Dog)
}

fn call_speak() {
    let animal = make_dog();
    // 理论上，编译器可以追踪到这里 animal 一定是 Dog
    // 所以 animal.speak() 可以"去虚拟化"为 Dog::speak()
    animal.speak();
}

// 但目前（2026 年），rustc/LLVM 的 devirtualization 能力有限：
// - 在同一个函数内，如果具体类型可以静态推断 → 可能去虚拟化
// - 跨函数时（如上例 make_dog → call_speak）→ 通常不去虚拟化
// - 链接时优化（LTO）能提高去虚拟化成功率

// 实际建议：
// 不要依赖编译器的去虚拟化优化——把它当作"惊喜"而非"保证"
// 如果需要确定性性能 → 选择静态分发
// 如果选择 trait object → 接受虚表调用的性能代价
```

### 8.5 静态分发与编译器缓存——增量编译视角

```
增量编译场景下的分发选择：

泛型（静态分发）的编译成本：
- 修改泛型函数体 → 所有使用该函数的单态化实例都要重新编译
- crate A 定义 fn foo<T: Trait>(t: T) { ... 长函数 ... }
- crate B 使用 foo<i32>, foo<String>, foo<MyType>
- 修改 foo 的函数体 → B 中所有 3 个实例重新编译
- 如果 T 有 50 种实例 → 50 次重新编译

trait object（动态分发）的编译成本：
- 修改函数体 → 只有 1 个版本需要重新编译
- 但添加新的 impl Trait → 已有的代码不受影响（因为 vtable 在实现侧生成）

选择指南：
- 在开发迭代快的阶段（频繁修改函数体）→ trait object 减少编译时间
- 在稳定阶段（函数体很少变）→ 泛型获得运行性能
- 大型项目（100+ crate）→ trait object 可以显著缩短增量编译时间

这是"编译时间 vs 运行时间"权衡在分发选择上的具体体现
```

### 8.6 `impl Trait` 的设计哲学——"需要知道的"最小化

```rust
// impl Trait 体现了 Rust 的"最小权限"哲学——
// 调用方只需要知道"最少必要信息"

// 传统泛型签名：
fn process<T: Iterator<Item = String>>(iter: T) -> Vec<String> {
    iter.collect()
}
// 调用方知道：
// 1. T 实现了 Iterator<Item = String>
// 2. T 的具体类型（通过 turbofish 可见、可通过任何渠道依赖）
// → 调用方可能依赖 T 的隐式性质（如 T: Clone）——即使函数不需要

// impl Trait 签名：
fn process(iter: impl Iterator<Item = String>) -> Vec<String> {
    iter.collect()
}
// 调用方知道：
// 1. iter 实现了 Iterator<Item = String>
// 2. 不知道具体类型
// → 调用方无法依赖具体的 T——只能用 Iterator 的方法

// 这带来两个好处：
// 1. API 稳定性：以后可以把返回类型从 Vec<...> 换成别的 Iterator 实现
//    调用方不会因此而编译失败——因为他们从来不知道具体类型
// 2. 减少误用：调用方无法"意外"依赖泛型参数的具体类型特征
//
// 这是"封装"（encapsulation）在类型层面的体现——
// 不仅是"隐藏字段"，更是"隐藏具体类型"
```

### 8.7 实际案例——Tokio 的 `Service` trait 和分发选择

```rust
// 异步生态中的分发选择——以 tower::Service 为例

// tower::Service trait（简化版）：
trait Service<Request> {
    type Response;
    type Error;
    type Future: Future<Output = Result<Self::Response, Self::Error>>;

    fn call(&self, req: Request) -> Self::Future;
}

// 链式组合用泛型（静态分发堆叠）：
// ServiceBuilder 串联多个中间件——每个中间件是一个泛型层
// 最终编译为深度的单态化调用链——所有层都内联优化
// 这是"编译时组合"的极致——零运行时开销

// 但是：
// - 编译时间非常长（每层中间件 x 每个具体类型 = 大量代码生成）
// - 类型非常复杂（编译错误信息像"模板圣代"）
// - 二进制膨胀明显

// 动态分发替代方案（tower::util::BoxService）：
// 用 Box<dyn Service<...>> 包装所有中间件
// 编译时间短，类型签名简洁
// 但每次调用多一层虚表间接引用

// Tokio 的教训：
// 在当前生态中，"全泛型"方式在编译时间上已经成为了实际问题
// 社区在探索更好的平衡（如 async fn in trait 稳定后简化类型）
```

### 8.8 未来——`async fn` 对分发选择的影响

```rust
// async fn 返回的具体类型是编译器生成的匿名 Future
// 这让"隐藏返回类型"变得特别重要

// 没有 impl Trait 时，async fn 的返回类型不可命名：
async fn fetch_data(url: &str) -> String {
    // 这个 Future 的类型是编译器生成的匿名类型
    // 无法在代码中显式写出
    reqwest::get(url).await.unwrap().text().await.unwrap()
}

// 其他函数接收这个 Future 时：
fn process_future(fut: impl Future<Output = String>) {
    // impl Trait 让接收"匿名 Future"成为可能
    // 如果用泛型：fn process_future<F: Future<Output = String>>(fut: F)
    // 效果一样——但 impl Trait 更简洁
}

// 当需要存储 future 时：
struct TaskQueue {
    // tasks: Vec<???>  —— 每个 async fn 返回不同的匿名类型
    tasks: Vec<Box<dyn Future<Output = ()> + Send>>,
    // 只能用 trait object ——不同类型的 Future 放入同一集合
}

// async 生态中分发选择的特殊性：
// - async fn 天然倾向 impl Trait（因为返回类型不可命名）
// - 异构 Future 集合必须用动态分发
// - 但动态分发和 async 结合有额外限制：
//   Box<dyn Future> 需要 Future 是 Unpin（或者手动 pin）
```

## 9. 回顾

```
静态 vs 动态分发学习清单：

□ 理解单态化（monomorphization）的工作机制——编译期为每个具体类型生成代码副本
□ 知道虚表（vtable）的内部结构——胖指针 = 数据指针 + 虚表指针
□ 能区分三种分发方式：泛型（静态）、impl Trait（静态+隐藏）、trait object（动态）
□ 理解"零成本抽象"——泛型代码编译后和手写特化代码性能相同
□ 知道代码膨胀的代价——i-cache miss 可能抵消静态分发的性能优势
□ 能选择正确的分发策略：热路径 → 静态，异构集合 → 动态，公共 API → impl Trait
□ 理解泛型的"传染性"——泛型参数会沿着调用链向上传播
□ 会测量实际的性能差异（benchmark），而不依赖直觉做选择
□ 知道 trait object 的内存是堆分配的（Box<dyn>）或通过引用访问（&dyn）
□ 理解 Rust 的分发选择是"显式的"——不同于 C++ 的隐含选择（模板 vs 虚函数）
```

> **一句话总结**：Rust 提供了从编译期零成本（泛型/impl Trait）到运行时灵活性（trait object）的完整分发谱系——你有权在同一个语言里为每个函数做出精确选择。C++ 把模板和虚函数放在截然不同的轨道上（互不兼容），Rust 通过共享 trait 定义让静态和动态分发无缝切换。`impl Trait` 是第三条路——保持静态分发的性能，同时隐藏具体类型以保护 API 稳定性。决策的核心是四个维度（性能、灵活性、二进制大小、API 设计）的权衡——没有"永远正确"的默认选择。
