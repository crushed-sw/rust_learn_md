# 7.0 编译器内部概述 —— 打开黑盒

> 前六个 Phase 你一直在用编译器——你写了代码，`cargo build` 吐出一个二进制。编译器对你来说是一个黑盒：`src/main.rs` 进去，可执行文件出来。你信任它——它检查类型、它报告生命周期错误、它优化代码——但你并不知道它在做什么。Phase 7 要改变这件事。你不一定需要成为编译器开发者，但理解编译器内部能让你：对借用检查器的"为什么报错"有第一性原理的理解，对性能优化有编译器层面的直觉，对 Rust 语言的设计决策有更深层的认同——因为你知道实现代价。

---

## 1. 为什么要理解编译器内部？

### 1.1 三个核心收益

```
┌─────────────────────────────────────────────────────────┐
│  收益 1: 更好的错误信息解读                                │
│  ─────────────────────────────────────                   │
│  "borrow checker says lifetime doesn't live long enough" │
│  如果你理解 NLL 的 region inference，就知道为什么它         │
│  "不够聪明"——它不是不够聪明，它是保守的近似。               │
├─────────────────────────────────────────────────────────┤
│  收益 2: 编译器层面的性能直觉                              │
│  ─────────────────────────────────────                   │
│  内联真的发生了吗？为什么这个泛型让二进制膨胀了 2MB？        │
│  理解 MIR 优化和单态化后，这些问题有清晰的答案。             │
├─────────────────────────────────────────────────────────┤
│  收益 3: 语言设计的深层理解                                │
│  ─────────────────────────────────────                   │
│  为什么 ? operator 只能用在返回 Result 的函数中？           │
│  为什么 async fn 不能在 trait 中（直到 1.75）？             │
│  这些限制的根本原因是编译器的内部实现，不是"设计师忘了"。     │
└─────────────────────────────────────────────────────────┘
```

### 1.2 谁需要理解编译器内部？

| 角色 | 需要的深度 | 关注点 |
|------|----------|--------|
| 应用开发者 | 知道阶段名称 + 会用调试 flag | 更好的错误理解 + 编译时间优化 |
| 库作者 | 理解 MIR 优化 + 单态化 | 设计零成本抽象 + 控制二进制大小 |
| 性能工程师 | 理解优化 pass + LLVM 集成 | PGO 微调 + 内联指导 |
| 语言贡献者 | 全部 | 添加特性 + 修复 bug |
| **你（资深学习者）** | **能看懂编译管线，理解关键算法** | **本 Phase 的目标** |

## 2. 编译管线全景：从源码到二进制

### 2.1 一个请求走过 rustc 的完整旅程

```
                                编译过程
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  源码     │   │  Token   │   │   AST    │   │   HIR    │   │   THIR   │
│ .rs 文件  │──▶│  流       │──▶│  抽象     │──▶│  高级     │──▶│  类型化   │
│          │   │           │   │  语法树   │   │  中间表示  │   │  中间表示  │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
   ↑ lexer        ↑ parser     ↑ ast_lowering ↑ typeck      ↑ thir_build

                                继续 →
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│   MIR    │   │  MIR     │   │  LLVM IR │   │  机器码   │
│  中级     │──▶│  优化后   │──▶│          │──▶│  .o /     │
│  中间表示  │   │          │   │          │   │  binary   │
└──────────┘   └──────────┘   └──────────┘   └──────────┘
   ↑ mir_build    ↑ mir_opts    ↑ codegen      ↑ LLVM
```

**每一层的核心区别**：

| 层 | 全称 | 特点 | 代码位置 |
|----|------|------|---------|
| Token Stream | — | 语言表面结构：标识符、关键字、标点 | `rustc_lexer` |
| AST | Abstract Syntax Tree | 完整语法树，包含所有原始语法信息 | `rustc_ast` |
| HIR | High-level IR | AST 脱糖后：`for` → `loop`、`?` → `match` | `rustc_hir` |
| THIR | Typed HIR | HIR + 类型信息（类型检查完成后） | `rustc_mir_build` |
| MIR | Mid-level IR | 简化控制流图，只有约 9 种语句 | `rustc_mir` |
| LLVM IR | — | LLVM 自己的 IR，交给 LLVM 优化 | `rustc_codegen_llvm` |

### 2.2 为什么这么多中间表示？

```rust
// 看一个例子，理解为什么需要多层 IR
fn example(x: Option<i32>) -> i32 {
    if let Some(v) = x {  // ← AST: 完整的 if let 语法
        v * 2
    } else {
        0
    }
}

// ━━━ 在 HIR 中，if let 被脱糖 ━━━
fn example(x: Option<i32>) -> i32 {
    match x {            // ← HIR: if let → match
        Some(v) => v * 2,
        None => 0,
    }
}

// ━━━ 在 MIR 中，变成控制流图 ━━━
// bb0: {
//     _2 = discriminant(_1);  // 检查 Option 的判别值
//     switchInt(_2) -> [0: bb2, 1: bb1];  // None → bb2, Some → bb1
// }
// bb1: {
//     _0 = ((_1 as Some).0: i32) * const 2_i32;  // 提取 Some 的值并计算
//     return;
// }
// bb2: {
//     _0 = const 0_i32;
//     return;
// }
```

每一层抽象承担不同职责：
- **AST**：保留所有语法细节（适合 IDE、格式化工具）
- **HIR**：语义层面的 AST，脱糖后（适合类型检查、借用检查）
- **MIR**：简化到控制流图（适合数据流分析、借用检查、优化）

## 3. rustc 的模块组织

### 3.1 核心 crate 地图

```
rustc 的 crate 依赖关系（简化版）

rustc_driver (入口：main → 编排整个编译流程)
    │
    ├── rustc_interface (编译器的高级 API，被 rustc_driver 和 RLS/rust-analyzer 调用)
    │     │
    │     ├── rustc_lexer      (词法分析：源码 → Token 流)
    │     ├── rustc_parse      (语法分析：Token 流 → AST)
    │     ├── rustc_ast        (AST 的类型定义)
    │     ├── rustc_hir        (HIR 的类型定义)
    │     ├── rustc_ast_lowering (AST → HIR 的转换)
    │     ├── rustc_mir_build  (HIR/THIR → MIR)
    │     ├── rustc_mir        (MIR 的类型定义 + 分析)
    │     ├── rustc_mir_transform (MIR 优化 pass)
    │     ├── rustc_borrowck   (借用检查，消费 MIR)
    │     ├── rustc_trait_selection (trait 解析)
    │     ├── rustc_typeck     (类型检查)
    │     ├── rustc_infer      (类型推导)
    │     ├── rustc_codegen_ssa    (代码生成的抽象层)
    │     ├── rustc_codegen_llvm   (LLVM 代码生成后端)
    │     ├── rustc_metadata   (crate 元数据读写：.rmeta 文件)
    │     ├── rustc_resolve    (名称解析)
    │     └── rustc_errors     (错误诊断)
    │
    ├── rustc_middle (所有中间表示的"中枢"：Ty、Substs、Instance 等核心类型)
    ├── rustc_span   (源码位置：Span、SourceFile、SourceMap)
    └── rustc_data_structures (内部数据结构：图、位集、工作队列)
```

### 3.2 把 rustc 当库来用

```rust
// 你可以把 rustc 作为库来编程——但 rustc 的 API 不稳定
// 大多数情况下，你用的是 stable 的工具：

// 1. 查看编译各阶段输出
// cargo rustc -- -Z time-passes        → 各 pass 耗时
// cargo rustc -- -Z unpretty=hir       → 打印 HIR
// cargo rustc -- -Z unpretty=mir       → 打印 MIR
// cargo rustc -- -Z dump-mir=all       → dump 所有 MIR 阶段

// 2. 程序化分析（syn + quote）
// 如果想在稳定通道上分析 Rust 代码，用 syn crate（实现了自己的 parser）
use syn::{parse_file, Item};

let code = "fn main() { let x = 42; }";
let ast = parse_file(code).unwrap();
// ↑ syn 重新实现了 parser，独立于 rustc，API 稳定
```

### 4.2 三个"不要"

1. **不要试图读完 rustc 源码**：rustc 超过 100 万行 Rust 代码。本 Phase 的目标是建立**概念地图**——知道这些东西在哪，以后需要时能找到。
2. **不要在 stable 编译器上尝试 compiler internals API**：rustc 的内部 API 不稳定（每 6 周发布一版）。用 `rustup override set nightly` 和 `#![feature(rustc_private)]` 来实验。
3. **不要被术语淹没**：MIR、HIR、THIR、AST——这些缩写看起来很吓人，但每个都只是"文本到二进制"路径上的一站。

## 5. 先热身：看一个真实函数的编译之旅

```rust
// 一个简单的函数
pub fn is_positive(x: i32) -> bool {
    if x > 0 {
        true
    } else {
        false
    }
}
```

```bash
# ━━━ 看 AST（语法树）━━━
cargo rustc -- -Z unpretty=ast-tree
# 输出大致是：
# FnDecl: is_positive
#   Param: x: i32
#   Body:
#     If:
#       BinaryOp >: x, 0
#       Block: true
#       Block: false

# ━━━ 看 HIR（脱糖后）━━━
cargo rustc -- -Z unpretty=hir-tree
# AST 和 HIR 在这个简单例子里几乎一样
# 但如果用了 ? operator、for loop、async，区别就大了

# ━━━ 看 MIR（控制流图）━━━
cargo rustc -- -Z unpretty=mir
# bb0: {
#     _2 = Gt(_1, const 0_i32);  // x > 0
#     switchInt(_2) -> [false: bb2, otherwise: bb1];
# }
# bb1: {
#     _0 = const true;
#     return;
# }
# bb2: {
#     _0 = const false;
#     return;
# }

# ━━━ 看 MIR 优化后 ━━
cargo rustc -- -Z unpretty=mir-optimized
# bb0: {
#     _0 = Gt(_1, const 0_i32);  // 直接返回比较结果！不再有分支
#     return;
# }
# ↑ 编译器发现 if true { true } else { false } 等价于 x > 0
```

从这个热身你已经看到：
- **AST/HIR** 保留了 `if` 的语义结构
- **MIR** 变成了底层控制流（basic block + switch）
- **MIR 优化后** 消除了冗余分支，直接返回比较结果

这就是编译器内部的核心故事——**每一层都在"理解"代码，然后"简化"它，直到 LLVM 能把它变成机器码。**
