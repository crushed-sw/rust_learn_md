# 7.9 Edition 机制与 Feature Gates

> Rust 有一个独特的设计：每三年发布一个新的 Edition（2015 → 2018 → 2021 → 2024）。Edition 不是语言的"新版本"——不同 Edition 的代码可以无缝链接在一起。这很反直觉：大多数语言（Python 2 vs 3、C++98 vs 11 vs 17）的"版本"意味着 Breaking Changes。Rust 怎么做到了"同一个编译器、不同默认行为"？另外，`#![feature(...)]` 这个在 nightly 中见过的属性在编译器内部是怎么工作的？本节将 Edition 和 Feature Gates 放在一起——因为它们都是"编译器版本管理"的两面。

---

## 1. Edition 的反直觉哲学

### 1.1 核心原则：同一个编译器

```
大多数语言：
  Python 2.7 编译器 ≠ Python 3.9 编译器
  C++98 编译器    ≠ C++17 编译器
  → 往上升级意味着切换编译器

Rust：
  同一个 rustc 同时支持 2015, 2018, 2021, 2024 Editions
  → 升级 Edition 只是改变了"默认行为"的 flag
  → 不同 Edition 的 crate 可以用 cargo 无缝链接
```

### 1.2 Edition 可以改变什么？

```
Edition 可以改变的（不影响互操作）：
  ✓ 关键字是否保留（2018: async/await 变成关键字）
  ✓ 语法歧义的解析方式（2021: 闭包捕获规则）
  ✓ 默认 trait 导入（2018: impl Trait 语法、dyn Trait）
  ✓ 标准 prelude 的内容（2015: 少很多）

Edition 不能改变的：
  ✗ 类型系统（不能添加/删除 trait）
  ✗ 内存模型（不能改变引用规则）
  ✗ ABI（不同 Edition 的 crate 必须能链接）
  ✗ 标准库 API（不能用 Edition 来废弃 API——用 #[deprecated]）
```

## 2. 每个 Edition 的关键变化

### 2.1 2015 → 2018

```rust
// ━━━ 变化 1: dyn Trait 语法 ━━━
// 2015: trait object 不强制写 dyn
let x: Box<Read> = ...;          // 只是警告
// 2018: 必须写 dyn（不写是错误）
let x: Box<dyn Read> = ...;      // ✅

// ━━━ 变化 2: async/await 变成关键字 ━━━
// 2015: async 可以是变量名
let async = 42;                  // ✅
// 2018: async 是关键字
let async = 42;                  // ❌ 编译错误
let async_ = 42;                 // ✅ 用 raw identifier 转义
let r#async = 42;                // ✅ 或 raw identifier

// ━━━ 变化 3: impl Trait ━━━
// 2015: 不支持
// fn foo() -> impl Iterator<Item=i32> { ... }  // ❌
// 2018: 支持
fn foo() -> impl Iterator<Item=i32> { ... }  // ✅

// ━━━ 变化 4: use 路径变化 ━━━
// 2015: use 路径默认为 crate root
use foo::bar;     // 指的是 crate::foo::bar
// 2018: use 路径默认为 crate root + extern crate 不再需要
use foo::bar;     // 还是 crate::foo::bar，但 extern crate 可省略
```

### 2.2 2018 → 2021

```rust
// ━━━ 变化 1: 闭包捕获规则改变（Disjoint Capture）━━━
struct Point { x: i32, y: i32 }
let mut p = Point { x: 0, y: 0 };
let mut c = || {
    p.x += 1;    // ← 只捕获 p.x
};
// 2018: c 捕获整个 p → 不能同时访问 p.y
// let y = &p.y;  // ❌ 2018: p 已经被 c 借用了
// 2021: c 只捕获 p.x → 可以访问 p.y
let y = &p.y;     // ✅ 2021

// ━━━ 变化 2: arrays IntoIterator ━━━
// 2021 之前：for i in [1, 2, 3] {}  // 通过 slice 的 IntoIterator
// 2021 之后：for i in [1, 2, 3] {}  // 通过 array 的 IntoIterator（不丢失所有权）

// ━━━ 变化 3: panic! 宏的一致性 ━━━
// 2021 前：panic!("{}", x) 和 panic!("{x}") 行为不一致
// 2021 后：统一了格式字符串的解析
```

### 2.3 2021 → 2024

```rust
// ━━━ 变化 1: RPIT 的生命周期捕获改变 ━━━
// 2021: impl Trait 会捕获所有泛型的生命周期
fn foo<'a>(x: &'a str) -> impl Use<Output=&'a str> { x }
// 2024: impl Trait 默认不捕获（需要显式 use<'a>）

// ━━━ 变化 2: unsafe 相关的收紧 ━━━
// 2024: unsafe 函数体和 unsafe 操作需要显式的 unsafe {} 块
// unsafe fn foo() {
//     *raw_ptr = 42;  // ❌ 2024: 需要 unsafe {} 块
// }
unsafe fn foo() {
    unsafe { *raw_ptr = 42; }  // ✅ 2024
}

// ━━━ 变化 3: match 的穷举检查改进 ━━━
// 2024: 对 match 表达式的穷举检查更严格
```

## 3. Edition 的技术实现

### 3.1 crate-level attribute

```rust
// edition 通过 Cargo.toml 或 crate-level attribute 指定

// Cargo.toml
[package]
edition = "2021"

// 编译器的内部表示（自动注入）：
// #![crate_edition = "2021"]
```

### 3.2 编译器如何按 Edition 选择行为

```rust
// rustc 内部，Edition 敏感的地方

// 解析器（parser）
match crate_edition {
    Edition::Edition2015 => {
        // async 允许作为标识符
    }
    Edition::Edition2018 | Edition::Edition2021 | Edition::Edition2024 => {
        // async 是关键字
    }
}

// 类型检查（typeck）
match crate_edition {
    Edition::Edition2021 | Edition::Edition2024 => {
        // 使用 2021 的闭包捕获规则（disjoint capture）
    }
    _ => {
        // 使用 2018 的规则
    }
}
```

### 3.3 cargo fix --edition

```bash
# 迁移到新 Edition 的工具

# Step 1: 修改 Cargo.toml 的 edition
edition = "2024"

# Step 2: cargo check → 看到 edition 相关的 lint
cargo check    # warning: `async` is a keyword in Edition 2018

# Step 3: cargo fix 自动修复
cargo fix --edition
# → 自动将变量名 `async` 改为 `r#async` 或 `async_`

# Step 4: cargo fix --edition-idioms
cargo fix --edition-idioms
# → 应用新 Edition 的 idiomatic 写法（如添加 dyn）
```

## 4. Feature Gates

### 4.1 `#![feature(...)]` 的内部

```rust
// 你在 nightly 上写：
#![feature(generic_const_exprs)]
//                 ^^^^^^^^^^^^^^^^^
//                 feature gate 的名字

// 编译器内部发生了什么：
// 1. 解析器遇到 #![feature(...)]
// 2. 检查当前编译器版本是 nightly（stable/beta 拒绝 feature gates）
// 3. 在全局 feature_gates 集合中添加这个 feature
// 4. 之后编译器在任何地方检查：
//    if feature_gates.contains("generic_const_exprs") {
//        // 允许使用 const generic 表达式
//    } else {
//        // 报错：feature `generic_const_exprs` is not yet stable
//    }
```

### 4.2 Feature Gate 的生命周期

```
Phase 1: Accepted RFC
  社区接受了这个提案 → 开始实现

Phase 2: Implementation + Feature Gate
  - 在 compiler/rustc_feature/src/feature_gate.rs 中注册
  - 代码标记为 #![feature(xxx)]
  - 只在 nightly 可用

Phase 3: Testing & Feedback
  - Nightly 用户试用
  - 在 rust-lang/rust issues 中收集反馈
  - 修复 bug、调整 API

Phase 4: Stabilization Report
  - 提议移除 feature gate
  - 编译器团队讨论
  - FCP → 批准

Phase 5: Stable
  - 移除 feature gate
  - 在 stable 中可用
  - #![feature(xxx)] 变成 no-op（已稳定）
```

### 4.3 追踪 Feature 的稳定化进度

```bash
# 方法 1: Can I Use Rust Yet
# https://caniuse.rs/  ← 查询 feature 在哪个版本稳定

# 方法 2: GitHub tracking issue
# 每个 feature 有一个 tracking issue
# 例如: https://github.com/rust-lang/rust/issues/44580 (const generics)

# 方法 3: Rust 发布博客
# 每 6 周的 release blog 中有 "Stabilized APIs" 部分
```

### 4.4 在生产中使用 Nightly Features 的风险

```rust
// ━━━ 风险管理策略 ━━━

// 策略 1: 用 cfg 开关
#[cfg(feature = "nightly")]
#![feature(xxx)]

#[cfg(not(feature = "nightly"))]
// 使用 stable 替代方案

// 策略 2: CI 中用固定版本的 nightly
// .github/workflows/ci.yml
// - uses: actions-rs/toolchain@v1
//   with:
//     toolchain: nightly-2024-01-15  ← 固定日期

// 策略 3: 只在内部工具中使用 nightly features
// 库的公共 API 只用 stable 特性
// 内部实现可以用 nightly（通过 cfg flag 保护）
```
