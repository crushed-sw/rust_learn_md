# 7.6 类型推导算法 —— 编译器如何"猜出"你的类型

> Rust 让你省去了大量类型标注——`let x = 42` 不需要写 `let x: i32 = 42`，`v.iter().collect()` 不需要 turbofish。但你有没有想过编译器到底怎么"猜"出来的？答案是 **Hindley-Milner 类型推导** 的一个扩展版本——比 HM 更复杂，因为要处理 subtyping（生命周期）、trait bound、impl Trait 等 Rust 特有的东西。本节不要求你手写 unification 算法，但会让你理解编译器在"猜类型"时的思考过程——这样你就不会再对类型推导失败的错误信息感到困惑了。

---

## 1. 类型推导 = 信息传播

### 1.1 不是"猜"，是"传播约束"

```rust
let x = 42;
// 编译器不是"猜" x 是 i32——而是：
// 1. 整数 literal 42 的默认类型是 i32（在没有其他约束时）
// 2. 如果后续代码要求 x 是 u64，编译器会调整

let y: u64 = 42;      // 明确标注 y 是 u64
let z = x + y;         // ❌ 类型不匹配：x 被认为是 i32，y 是 u64
// 但：
let x = 42u64;        // 明确标注
let z = x + y;         // ✅ 都是 u64
```

### 1.2 类型信息在程序中流动

```
约束传播的两个方向：

向下传播（从声明到使用）
  let x: Vec<i32> = Vec::new();
//       ^^^^^^^^ 类型标注向下
//                x 被约束为 Vec<i32>

向上传播（从使用反推到声明）
  let mut data = Vec::new();   // data 的类型未知
  data.push(42i32);            // ← 从 push(42i32) 推导:
                               //    data 必须是 Vec<i32>
//               data 现在被约束为 Vec<i32>

双向传播
  let numbers: Vec<_> = vec![1, 2, 3].iter().map(|x| x * 2).collect();
//            ^^^^^^^  "_" 表示"从右边推导"
// vec![1,2,3] → Vec<i32>
// .iter() → Iter<i32>
// .map(|x| x * 2) → 返回 i32 的迭代器
// .collect() → 因为左边是 Vec<_>，所以 collect 返回 Vec<i32>
// 所以 _ = i32
```

## 2. Hindley-Milner 的 Rust 变体

### 2.1 经典 HM：两步法

经典 Hindley-Milner 类型推导分为两步：

```
Step 1: 约束生成 (Constraint Generation)
  遍历表达式树 → 为每个子表达式生成类型变量 + 约束

Step 2: 约束求解 (Unification)
  求解所有约束 → 找到类型变量的最一般替换（most general unifier）
```

```rust
// 经典 HM 处理的约束：
//   type_var_0 = type_var_1      (两个类型变量相等)
//   type_var_0 = i32             (类型变量等于具体类型)
//   type_var_0 = type_var_1 → type_var_2  (函数类型)

fn apply(f: ???, x: ???) -> ??? {
    f(x)
}
// 约束：
//   f: ?a → ?b
//   x: ?a
//   返回值: ?b
// 如果调用 apply(|n| n + 1, 42)：
//   ?a = i32 (从 42 推导)
//   |n| n + 1: i32 → i32
//   ?b = i32 (从闭包返回类型)
// 所以 apply: (i32 → i32, i32) → i32
```

### 2.2 Rust 的额外复杂度

```rust
// Rust 的 HM 扩展需要处理：

// 1. Subtyping（因为生命周期）
fn foo<'a>(x: &'a i32, y: &'a i32) { }
// &'long i32 是 &'short i32 的子类型（subtype）
// 'long 比 'short 长 → &'long i32 可以用在需要 &'short i32 的地方

// 2. Trait bound
fn bar<T: Display>(x: T) { }
// 不仅需要 T = 具体类型，还需要 T: Display

// 3. impl Trait
fn baz() -> impl Iterator<Item = i32> { }
// 返回类型是"某个实现了 Iterator<Item=i32> 的具体类型"——existential type

// 4. 方法调用 + Deref 链
let s = String::from("hi");
s.len();  // 编译器需要：在 String、&String、&str、...中找 len()
```

## 3. Unification（合一）算法的直观理解

### 3.1 类型变量和约束

```rust
// 编译器看到的类型推导过程（简化）
let mut v = Vec::new();    // v: ?T（类型变量）
v.push(42i32);             // 约束: ?T = Vec<i32>（因为 push 的参数必须匹配）
v.push(100);               // 约束: i32 = i32（100 是 i32, v 期望 i32）
// 求解: ?T = Vec<i32> ✓
```

### 3.2 合一失败的例子

```rust
let mut v = Vec::new();    // v: ?T
v.push(42i32);             // 约束: ?T = Vec<i32>
v.push("hello");           // 约束: i32 = &str ← 冲突！
// ❌ 合一失败 → 类型错误
```

### 3.3 合一递归

```rust
// 嵌套类型的合一
let mut v = Vec::new();       // v: ?T
v.push(vec![1, 2, 3]);        // 约束: ?T = Vec<?U>
                               // 且 ?U = i32 (从 vec![1,2,3])
// 求解: ?T = Vec<Vec<i32>> → v: Vec<Vec<i32>>
```

## 4. 什么时候你需要帮编译器？

### 4.1 Turbofish 的场景

```rust
// ━━━ 场景 1: collect() 不知道收集成什么 ━━━
let items: Vec<_> = (0..10).collect();  // ✅ 注解了 Vec<_>
// let items = (0..10).collect();       // ❌ 编译器不知道你想收集成 Vec、HashMap...

// ━━━ 场景 2: parse() 不知道解析成什么 ━━━
let x: i32 = "42".parse().unwrap();     // ✅ 注解了 i32
let x = "42".parse::<i32>().unwrap();   // ✅ turbofish
// let x = "42".parse().unwrap();       // ❌ 不知道解析成什么

// ━━━ 场景 3: 多个 impl 提供同一个方法 ━━━
trait A { fn method(&self); }
trait B { fn method(&self); }
impl A for MyType { fn method(&self) {} }
impl B for MyType { fn method(&self) {} }

let x = MyType;
// x.method();  // ❌ 歧义：A::method 还是 B::method？
<MyType as A>::method(&x);  // ✅ 完全限定语法
```

### 4.2 类型标注不宜过多

```rust
// ❌ 过度标注：每个变量都写类型
let x: i32 = 42;
let y: i32 = x + 1;
let z: String = y.to_string();

// ✅ 适当标注：只在"编译器需要帮助"的地方写类型
let x = 42;
let y = x + 1;
let z: String = y.to_string();  // 这里 to_string() 可以是多种类型
```

## 5. 类型推导失败时的错误报告

### 5.1 编译器的策略

```rust
// 当类型推导失败时，编译器不立即报错
// 而是继续推导——收集"矛盾点"，然后在最后报出最相关的错误

fn example() {
    let mut v = Vec::new();
    v.push(42);
    v.push("hello");  // ← 这里产生了矛盾
    // 编译器报错：
    // error[E0308]: mismatched types
    //   expected `i32`, found `&str`
    // note: expected because this value was pushed first
}
```

### 5.2 理解"推导链"

```
当你看到类型推导错误的诊断时，注意 "note:" 行：

error[E0282]: type annotations needed
   --> src/main.rs:3:9
    |
3   |     let x = Vec::new();
    |         ^ cannot infer type for type parameter `T`
    |
    = note: type must be known at this point

这告诉你：编译器试了，但信息不够。需要你加注解。
```

## 6. Rust 类型推导的已知局限

```rust
// ━━━ 局限 1: 不能推导递归函数的返回类型 ━━━
// fn recursive(n: u32) -> _ {  // ❌ 不能推导自己的返回类型
//     if n == 0 { 0 } else { n + recursive(n - 1) }
// }
fn recursive(n: u32) -> u32 {  // ✅ 必须标注
    if n == 0 { 0 } else { n + recursive(n - 1) }
}

// ━━━ 局限 2: 闭包的参数类型必须能推导 ━━━
let f = |x| x + 1;     // ✅ 从 x + 1 推导出 x: i32（默认整数类型）
// let g = |x| x;      // ❌ 没有任何信息：x 是什么类型？

// ━━━ 局限 3: 收集器的类型歧义 ━━━
let v: Vec<_> = [1, 2, 3].iter().map(|x| x * 2).collect();
// 你需要 Vec<_> 或 collect::<Vec<_>>()
```
