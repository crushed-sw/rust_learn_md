# 7.1 rustc 架构：前端到后端全景

> 7.0 你看到了编译管线的全景图——六层抽象从源码到机器码。本节将每一层拆开：词法分析器如何吃进字符吐出 Token？Parser 如何把 Token 变成语法树？HIR 脱糖做了什么？MIR 为什么只有 9 种语句？LLVM IR 又是什么？你会亲手用 `-Z unpretty` 和 `-Z dump-mir` 看到每一步的输出，建立从 `.rs` 到可执行文件的完整心智模型。

---

## 1. 第一站：词法分析（Lexer）

### 1.1 Token 是什么？

```rust
// 这是你写的代码
let x: i32 = 42;

// 这是 lexer 看到的 Token 流：
// ┌─────────┬──────┬────┬──────┬─────┬──────┬───┬─────┬───┐
// │ let     │ ws   │ x  │ ws   │ :   │ ws   │ i │ 32  │ ; │
// │ keyword │      │ id │      │colon│      │ id│ int │semi│
// └─────────┴──────┴────┴──────┴─────┴──────┴───┴─────┴───┘
```

**rustc_lexer 的特点**：
- 完整手写（不依赖 lexer generator）——对 Rust 的复杂词法规则（如 raw string literal `r#"..."#`、lifetime `'a`、整数后缀 `42u8`）有精确控制
- 不分配字符串：Token 是源码的切片 `&str`，零拷贝
- 按需生成：不是一次性生成全部 Token，而是迭代器模式的惰性生成

### 1.2 词法歧义：Rust lexer 最难的地方

```rust
// 歧义 1：<< 是"左移运算符"还是"嵌套泛型的闭合尖括号"？
let x: Vec<Vec<i32>> = vec![];  // >> 被解析为两个 > 而不是右移
// 这需要 parser 向 lexer 反馈（"turbofish 歧义"）

// 歧义 2：'a 是生命周期还是 char literal？
fn foo<'a>() { let c = 'a'; }
//         ↑ 生命周期               ↑ char literal
// lexer 把两个都标记为 lifetime，parser 根据上下文区分

// 歧义 3：.. 是什么？
match x {
    1..=5 => {}    // 范围模式 1..=5（三个 token：1 ..= 5）
}
let r = 1..5;     // 范围表达式（两个 token：1 ..5）
```

## 2. 第二站：语法分析（Parser）→ AST

### 2.1 从 Token 到语法树

```bash
# 看某个函数的 AST
cargo rustc -- -Z unpretty=ast-tree
```

```rust
// 输入
pub fn greet(name: &str) -> String {
    format!("Hello, {}!", name)
}

// Parser 构建的 AST（简化表示）：
// Item::Fn {
//   vis: Visibility::Public,
//   ident: "greet",
//   generics: Generics { params: [] },
//   decl: FnDecl {
//     inputs: [Param { ident: "name", ty: &str }],
//     output: Return(String),
//   },
//   body: Block {
//     stmts: [
//       Expr::Call {
//         func: Expr::Path("format"),
//         args: ["Hello, {}!", name],
//       }
//     ]
//   }
// }
```

### 2.2 AST 保留了"一切"

AST 不做语义分析，只做语法结构识别。这意味着：
- 宏调用在 AST 中是未展开的：`println!("{}", x)` 在 AST 中是一个 `MacCall` 节点
- `if let` 和 `match` 是不同的 AST 节点（到 HIR 才统一）
- AST 包含了所有注释和空白（以 token 间隔的形式）

## 3. 第三站：AST → HIR（脱糖）

### 3.1 HIR 的核心工作：消除语法糖

```rust
// ━━━ 语法糖 1: if let → match ━━━
// 你写的
if let Some(x) = option { process(x) }

// HIR 中变成
match option {
    Some(x) => process(x),
    _ => {}
}

// ━━━ 语法糖 2: for → loop + match ━━━
// 你写的
for item in vec.iter() { process(item); }

// HIR 中变成
{
    let mut _iter = std::iter::IntoIterator::into_iter(vec.iter());
    loop {
        match _iter.next() {
            Some(item) => process(item),
            None => break,
        }
    }
}

// ━━━ 语法糖 3: ? operator → match ━━━
// 你写的
let value = fallible_op()?;

// HIR 中变成
let value = match fallible_op() {
    Ok(v) => v,
    Err(e) => return Err(e.into()),
};

// ━━━ 语法糖 4: 闭包 → 匿名 struct ━━━
// 你写的
let f = |x| x + 1;

// HIR 中变成（简化）
// struct Closure_xxx { ... }  ← 捕获的变量作为字段
// impl Fn(i32) -> i32 for Closure_xxx { ... }
```

### 3.2 HIR 的另一个关键工作：名称解析

```rust
// 在 AST 中，所有的名字都只是字符串
// 在 HIR 中，每个名字都绑定到了它引用的定义

use std::collections::HashMap;  // ← 这个 use 被解析
let map: HashMap<String, i32> = HashMap::new();
//        ↑ HIR 知道这个 HashMap 指向 std::collections::HashMap
//        不是你自己定义的同名类型
```

HIR 中的每个标识符都带有一个 `HirId`——在编译器中唯一标识一个定义的 ID。

## 4. 第四站：类型检查 → THIR

### 4.1 THIR = Typed HIR

```rust
// HIR（类型检查前）：只知道语法结构
fn add(x, y) {
    x + y
}

// THIR（类型检查后）：每个表达式都带类型
fn add(x: i32, y: i32) -> i32 {
    (x + y): i32  // ← 每个子表达式都标注了类型
}
```

类型检查在这个阶段做了：
- 推导所有变量的类型
- 检查 trait bound 是否满足
- 检查方法调用的 receiver 类型
- 生成类型推导的约束并求解

## 5. 第五站：THIR → MIR

### 5.1 MIR：简化到极致

MIR（Mid-level IR）是 rustc 最重要的中间表示——借用检查、优化、代码生成都基于它。MIR 被刻意设计得非常简单：

```
MIR 只有 9 种语句（Statement）：
  Assign         let _0 = ...;
  FakeRead       标记读取（借用检查用）
  SetDiscriminant  设置 enum 变体
  StorageLive    标记变量开始存活
  StorageDead    标记变量结束存活
  Deinit         标记变量被"解初始化"
  Retag          Stacked Borrows 的重标记（别名分析）
  AscribeUserType  用户标注的类型（调试用）
  Coverage       代码覆盖率桩
  Intrinsic      内建函数调用
  ConstEvalCounter  常量求值限制

MIR 只有 6 种终结器（Terminator）：
  Goto           无条件跳转
  SwitchInt      整数匹配跳转
  Return         函数返回
  Unreachable    不可达代码
  Call           函数调用
  Assert         断言
  Yield          generator yield
  GeneratorDrop  generator drop
  FalseEdge      虚假边（借用检查用）
  FalseUnwind    虚假 unwind（借用检查用）
  InlineAsm      内联汇编
```

### 5.2 一个函数从 Rust 到 MIR

```rust
// 源码
fn abs(x: i32) -> i32 {
    if x >= 0 { x } else { -x }
}
```

```bash
cargo rustc -- -Z unpretty=mir
```

```
// MIR 输出（简化）
fn abs(_1: i32) -> i32 {
    debug x => _1;                    // _1 是参数 x
    let mut _0: i32;                  // _0 是返回值

    bb0: {
        StorageLive(_2);              // 临时变量 _2 存活开始
        _2 = Ge(_1, const 0_i32);     // _2 = x >= 0
        switchInt(_2) -> [false: bb2, otherwise: bb1];
    }

    bb1: {                            // then 分支 (x >= 0)
        _0 = _1;                      // 返回 x
        goto -> bb3;
    }

    bb2: {                            // else 分支 (x < 0)
        _0 = Neg(_1);                 // 返回 -x
        goto -> bb3;
    }

    bb3: {
        StorageDead(_2);              // 临时变量 _2 存活结束
        return;
    }
}
```

**关键观察**：
- 每个 basic block 都有一个标签（`bb0`, `bb1`, ...）
- 每个 basic block 以终结器结束（`switchInt`, `goto`, `return`）
- 所有临时变量有显式的存活范围（`StorageLive` / `StorageDead`）
- 没有 `if`——被 `switchInt` + basic block 替代

## 6. 第六站：MIR 优化

在 MIR 上运行的优化 pass（详见 7.5）：

```bash
# 看 MIR 优化前后的对比
cargo rustc -- -Z unpretty=mir         # 优化前
cargo rustc -- -Z unpretty=mir-optimized  # 优化后
```

MIR 优化器发现 `if x >= 0 { x } else { -x }` 中，`bb1` 和 `bb3` 可以合并，`bb2` 后也可以直接返回——消除不必要的 basic block。

## 7. 第七站：LLVM IR 生成

### 7.1 MIR → LLVM IR

```bash
# 看 LLVM IR
cargo rustc -- --emit=llvm-ir
```

```llvm
; abs 函数在 LLVM IR 中（简化）
define i32 @abs(i32 %x) {
start:
  %0 = icmp sge i32 %x, 0        ; x >= 0 ?
  br i1 %0, label %then, label %else

then:
  ret i32 %x                      ; 返回 x

else:
  %1 = sub i32 0, %x              ; 0 - x = -x
  ret i32 %1                      ; 返回 -x
}
```

LLVM 还会进一步优化（常量折叠、指令合并等），但核心结构已经定型。

### 7.2 代码生成的后端

rustc 默认使用 LLVM 作为代码生成后端，但也有实验性替代：

```bash
# LLVM 后端（默认）
cargo build --release

# Cranelift 后端（实验性，debug 构建更快）
cargo build -Z build-std --release
# 或配合 rustup component add rustc-codegen-cranelift-preview

# GCC 后端（rustc_codegen_gcc，实验性）
# 目标：让 Rust 可以用 GCC 编译
```

## 8. 编译耗时分析

### 8.1 看每个阶段花了多少时间

```bash
cargo rustc -- -Z time-passes
```

输出示例：
```
time:   0.003; rss:   42MB  parsing
time:   0.012; rss:   48MB  expansion          ← 宏展开
time:   0.025; rss:   62MB  typeck             ← 类型检查（常是瓶颈）
time:   0.018; rss:   59MB  borrow_checking
time:   0.015; rss:   61MB  mir_borrowck
time:   0.032; rss:   75MB  mir_optimizations
time:   0.045; rss:   89MB  codegen_to_LLVM_IR
time:   0.120; rss:  110MB  LLVM_passes         ← LLVM 通常是最大头
time:   0.008; rss:   85MB  linking
```

**关键洞察**：LLVM 优化通常占总编译时间的 50-60%，类型检查占 20-30%。这就是为什么 `opt-level=0` 的 debug 构建比 release 快很多——大部分时间省在了 LLVM 优化上。
