# 7.5 MIR 优化全景

> 很多人以为 Rust 的优化全靠 LLVM——"rustc 只是把代码翻译成 LLVM IR，然后 LLVM 做所有优化"。这是错的。rustc 在 MIR 层面有约 30 个优化 pass，在把代码交给 LLVM 之前就做了大量的化简。更重要的是，其中很多优化是**Rust 特有的**——利用所有权、借用、enum 判别值等信息做 LLVM 做不到的优化。比如：当编译器知道 `&mut T` 是唯一的引用时，它可以做更激进的数据流优化。本课深入 MIR 优化层——每个 pass 做什么、对什么代码有效、如何用 `-Z dump-mir` 观察每一步。

---

## 1. MIR：为什么是优化的最佳位置？

### 1.1 三层的对比

```
AST/HIR 层：太高层
  ✗ 包含太多语法细节
  ✗ 控制流不显式（if let、for 等语法糖还在）
  ✗ 不适合做数据流分析

MIR 层：恰到好处  ← 重点优化层
  ✓ 简化到 basic block + 控制流图
  ✓ 只有 ~9 种语句, ~6 种终结器
  ✓ 所有临时变量显式存在（StorageLive/StorageDead）
  ✓ Rust 类型信息完整（enum 判别值、引用唯一性）
  ✓ 可以做 Rust 特有的优化

LLVM IR 层：太底层
  ✓ 能做与语言无关的通用优化（指令选择、寄存器分配）
  ✗ 丢失了 Rust 的类型系统信息（不知道 &T 是共享引用还是可变引用）
  ✗ 不能做 Rust 特有的优化
```

### 1.2 MIR 优化的三个阶段

```
built MIR (构造时 MIR)
  │ ← 刚从 HIR/THIR 翻译过来的原始 MIR
  │
  ▼ 第 1 层: 清理 pass (Cleanup)
  │   SimplifyCfg, SimplifyLocals, ...
  │
  ▼ 第 2 层: 主要优化 pass
  │   Inline, ConstProp, InstCombine, ...
  │
  ▼ 第 3 层: 代码生成准备 pass
  │   CopyProp, DeadStoreElimination, ...
  │
  ▼
optimized MIR (优化后 MIR) → 送给 LLVM
```

## 2. 核心 MIR 优化 Pass 详解

### 2.1 SimplifyCfg —— 清理控制流图

```rust
// SimplifyCfg 做什么：消除不可达 basic block、合并空 block

// 优化前 MIR:
// bb0: { _0 = const 42_i32; goto -> bb1; }
// bb1: { goto -> bb2; }         ← 空 block，无用
// bb2: { return; }

// 优化后 MIR:
// bb0: { _0 = const 42_i32; return; }
```

### 2.2 Inline —— 内联函数调用

```rust
// MIR 层面的内联：把函数体嵌入调用点
fn wrapper(x: i32) -> i32 {
    add_one(x)  // ← 调用被内联
}
fn add_one(x: i32) -> i32 { x + 1 }

// 内联后 (wrapper 的 MIR):
fn wrapper(x: i32) -> i32 {
    // add_one 的函数体被直接嵌入
    _0 = Add(_1, const 1_i32);  // x + 1
    return;
}
```

**MIR 内联 vs LLVM 内联**：
- MIR 内联能看到 Rust 类型信息（enum 判别值、引用关系）
- LLVM 内联做后期的更精细决策（调用频率、指令数量等）
- 两者都做，互补

### 2.3 ConstProp (常量传播)

```rust
// 优化前:
let x = 42;
let y = x + 1;
let z = y * 2;
println!("{}", z);

// ConstProp 后:
// let x = 42;           ← 可能被消除（如果 x 不再被使用）
// let y = 43;           ← x + 1 → 43
// let z = 86;           ← y * 2 → 86
println!("{}", 86);     // ← 直接使用常量 86
```

### 2.4 CopyProp (拷贝传播)

```rust
// 优化前:
let tmp = x;
let result = tmp + tmp;

// CopyProp 后:
let result = x + x;  // ← tmp 被替换为 x，tmp 变量消除
```

### 2.5 DeadStoreElimination (死存储消除)

```rust
// 优化前:
let mut counter = 0;
counter = 1;     // ← 死存储：counter = 0 被覆盖，0 从未被读取
counter = 2;     // ← counter = 1 也是死存储
println!("{}", counter);  // 只用到 2

// 优化后:
let counter = 2;  // 只有最后一次赋值被保留
println!("{}", counter);
```

### 2.6 MatchBranchSimplification

```rust
// 利用 enum 判别值的已知信息简化 match

fn process(x: Option<i32>) -> i32 {
    match x {
        Some(v) => v * 2,
        None => 0,
    }
}

// 如果 context 已知 x 是 Some，match 被简化为：
// _0 = (x as Some).0 * const 2_i32;  // 直接提取值
```

这得益于 MIR 中保留了 enum 的完整类型信息——LLVM 已经看不到 `Option<i32>` 了（被编译为结构体+tag）。

## 3. Rust 特有的 MIR 优化

### 3.1 基于所有权的优化

```rust
// 编译器知道：&mut T 是唯一的引用
// → 某些原本"需要保守"的操作可以更激进

fn unique_ref_opt(x: &mut i32) -> i32 {
    *x = 10;
    let y = *x;   // ← 编译器知道：在 *x = 10 和这里之间
    *x = 20;      //    没有其他引用能修改 x
    y             //    所以 y 一定是 10！
}
// → 编译器可以优化为: fn unique_ref_opt(x: &mut i32) -> i32 { *x = 20; 10 }
```

### 3.2 基于 enum niche 的优化

```rust
// Option<&T> 使用 niche 优化：None 用全 0 表示
// MIR 可以利用这个信息优化 match

fn get_first(v: &[i32]) -> Option<&i32> {
    v.first()  // 底层是用指针是否为 null 决定 None/Some
}

// MIR 优化器知道 Option<&T> 的判别值是嵌入在指针里的
// → 消除了显式的 tag 检查，直接检查指针是否为 null
```

## 4. 用 `-Z dump-mir` 观察优化过程

### 4.1 导出所有 MIR 优化阶段

```bash
# 导出所有函数的 MIR，每个 pass 前后各一份
cargo rustc -- -Z dump-mir=all

# 输出目录: mir_dump/
# 每个函数有一系列文件：
#   my_function.built.mir         ← 刚构建的 MIR
#   my_function.SimplifyCfg.after.mir
#   my_function.Inline.after.mir
#   my_function.ConstProp.after.mir
#   ...
#   my_function.optimized.mir     ← 最终优化后的 MIR
```

### 4.2 只看优化前后对比

```bash
# 快速对比优化效果
cargo rustc -- -Z unpretty=mir         # 优化前
cargo rustc -- -Z unpretty=mir-optimized  # 优化后

# 用 diff 工具对比
diff <(cargo rustc -- -Z unpretty=mir 2>&1) \
     <(cargo rustc -- -Z unpretty=mir-optimized 2>&1)
```

### 4.3 看特定函数

```bash
# 只看某个函数的 MIR
cargo rustc -- -Z unpretty=mir -Z mir-opt-level=2

# 看某个函数在特定 pass 后的状态
cargo rustc -- -Z dump-mir=Inline  # 只看 Inline pass
```

## 5. MIR 优化的局限

### 5.1 什么留给 LLVM？

| 优化类型 | MIR 能做 | LLVM 能做 |
|---------|---------|----------|
| 死代码消除 | ✓ | ✓ |
| 内联 | ✓ (Rust 类型感知) | ✓ (profile 感知) |
| 常量传播 | ✓ | ✓ |
| 向量化 (SIMD) | ✗ | ✓ |
| 指令选择 | ✗ | ✓ |
| 寄存器分配 | ✗ | ✓ |
| 循环展开 | 基础 | ✓ (更复杂) |
| 分支预测优化 | ✗ | ✓ (PGO 数据) |

**关键原则**：MIR 做 **Rust 特有的、需要类型信息** 的优化；LLVM 做 **通用的、底层指令级别的** 优化。

### 5.2 循环优化：MIR 的短板

```rust
// 这个循环 MIR 优化器基本不动它
fn sum_vec(v: &[i32]) -> i32 {
    let mut sum = 0;
    for &item in v {
        sum += item;
    }
    sum
}
// → 循环展开、向量化都留给 LLVM
// → MIR 只确保 for 脱糖后的 loop 结构正确
```
