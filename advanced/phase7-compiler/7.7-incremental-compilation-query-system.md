# 7.7 增量编译：Query System

> 改了一行代码，`cargo build` 重新编译了整个 crate——这是每个 Rust 开发者都经历过的沮丧。但 rustc 其实有一个聪明的机制来避免不必要的重编译：**Query System**。它把编译的每个任务（类型检查一个函数、解析一个模块、推导一个类型）都建模为可缓存的 query——当输入没变时，直接返回缓存结果。本节揭秘 query system 的工作原理：dep graph 如何追踪依赖、什么场景对增量编译友好（或不友好）、如何用 `-Z self-profile` 诊断增量编译的效率。

---

## 1. 为什么需要增量编译？

### 1.1 编译的"粒度"问题

```
非增量编译：
  改一行代码 → 整个 crate 重新编译
  (Type checking × 500 functions, borrow checking × 全部...)

增量编译：
  改一行代码 → 只重新编译受影响的函数
  (Type checking × 1 function, borrow checking × 相关函数)
```

### 1.2 什么决定了"受影响"？

```rust
// 你改了函数 A 的返回类型
fn a() -> i32 { 42 }       // 改之前
fn a() -> String { "42".into() }  // 改之后

fn b() {
    let x = a();  // ← b 的代码没改，但 a() 的返回类型变了
    // b 需要重新类型检查！因为 x 的类型从 i32 变成了 String
}

fn c() {
    let y = 42;   // ← 完全不依赖 a()，不需要重新编译
}
```

**dep graph**（依赖图）就是用来追踪这种依赖关系的。当 `a()` 改变时，dep graph 告诉我们 `b()` 需要重编译，`c()` 不需要。

## 2. Query System 架构

### 2.1 一切皆 Query

```rust
// 在 rustc 内部，编译任务被组织为 query：

// ━━━ Query 的示例 ━━━
// typeck_query(def_id) → 返回该函数的类型检查结果
// mir_built_query(def_id) → 返回该函数的 MIR
// borrowck_query(def_id) → 返回该函数的借用检查结果
// expand_crate_query(krate) → 返回整个 crate 的宏展开结果

// 每个 query 有一个 key（如 def_id）和一个 value（计算结果）
```

### 2.2 Query 的缓存策略

```
query("typeck", fn_main):
  ┌─────────────────────────────────────────────┐
  │ 1. 检查缓存：key = (typeck, fn_main)         │
  │    - 如果缓存存在，且输入没有改变 → 直接返回   │
  │    - 如果缓存不存在或输入已改变 → 继续        │
  │                                             │
  │ 2. 执行 query                                │
  │    - 在 dep graph 中记录"谁调用了谁"          │
  │    - 如果执行中调用了其他 query，记录 edge     │
  │                                             │
  │ 3. 缓存结果                                  │
  │    - 存入磁盘（增量编译缓存）                  │
  │    - 标记为 current                          │
  └─────────────────────────────────────────────┘
```

### 2.3 Dep Graph 的边

```rust
// dep graph 记录两种边：

// ━━━ 1. 直接依赖（调用关系）━━━
// 当 typeck(fn_main) 执行过程中调用了 typeck(fn_helper)
// → dep graph 记录: typeck(fn_main) depends on typeck(fn_helper)
// → 当 fn_helper 改变时，fn_main 的缓存被 invalidate

// ━━━ 2. 读取依赖（数据访问）━━━
// 当 borrowck(fn_main) 读取了 typeck(fn_main) 的结果
// → dep graph 记录: borrowck(fn_main) depends on typeck(fn_main)
// → 当 typeck(fn_main) 被重新计算时，borrowck(fn_main) 也被 invalidate
```

## 3. 增量编译的"杀手"：什么让它失效？

### 3.1 杀手 1：改变泛型函数的定义

```rust
// crate A: 定义泛型函数
pub fn generic<T: Display>(x: T) { ... }

// 如果改了 generic 的定义 → 所有使用 generic 的下游 crate 都要重编译
// 因为单态化（monomorphization）发生在下游 crate，
// 而下游 crate 需要重新读取 generic 的泛型 MIR
```

### 3.2 杀手 2：过程宏

```rust
// proc-macro crate 的任何改变 → 所有使用该宏的 crate 都要重编译
// 因为宏展开了才知道生成的代码是什么

#[derive(MyMacro)]  // ← 这个宏的 crate 变了
struct MyStruct;     // → MyStruct 需要重新展开 → 重新编译
```

### 3.3 杀手 3：`build.rs` 的输出改变

```rust
// build.rs 生成代码或设置环境变量
fn main() {
    println!("cargo:rustc-env=BUILD_TIME={}", chrono::Utc::now());
    // ↑ 每次 build 这个值都不同 → 每个依赖 BUILD_TIME 的 crate 都被 invalidate
}
```

### 3.4 杀手 4：大范围的 cfg 属性

```rust
// 如果 cfg 条件改变了，大量代码可能被 include/exclude
#[cfg(feature = "expensive")]
mod big_module {  // ← feature flag 改变 → 整个模块在/不在编译范围内
    // ...
}
```

## 4. 增量编译的配置

### 4.1 Cargo 配置

```toml
# Cargo.toml
[profile.dev]
incremental = true  # 默认开启（debug build）

[profile.release]
incremental = false  # 默认关闭（release build）
# 但有些项目在 release CI 中也开启增量以加速迭代
incremental = true
```

### 4.2 增量编译缓存的位置

```
增量编译缓存存储在 target/ 下：

target/debug/incremental/
  └── <crate_name>-<hash>/
      ├── query-cache.bin    ← query 结果的磁盘缓存
      └── dep-graph.bin      ← 依赖图

每次 cargo build 读取这些文件，
只在输入改变时重新计算。
```

## 5. `-Z self-profile` —— 诊断增量编译

### 5.1 生成 profile

```bash
# 开启 self-profile（nightly）
cargo rustc -- -Z self-profile

# 生成的文件：<crate>-<timestamp>.mm_profdata
# 可以用 crox 或 summarize 工具分析
```

### 5.2 分析增量编译效率

```bash
# 安装分析工具
cargo install summarize

# 跑两次编译，比较增量效果
# 第一次：全量编译
cargo clean && cargo build

# 第二次：不改代码，再编译（应该全部命中缓存）
touch src/lib.rs  # 修改时间戳但不改内容
cargo build

# 用 summarize 分析 self-profile 数据
summarize summarize <crate>-*.mm_profdata
```

输出示例：
```
Query               | Calls | Cache hits | Cache miss %
typeck              |  1234 |       1200 |         2.7%
borrowck            |   450 |        445 |         1.1%
expand_crate        |     1 |          0 |       100.0%  ← 缓存没命中！
```

### 5.3 为什么 expand_crate 缓存总是不命中？

```rust
// 宏展开（expand_crate）是增量编译最薄弱的环节
// 原因：
// 1. 宏展开的依赖非常广——只要任何宏定义改变，都要重新展开
// 2. proc-macro 的展开结果不在缓存中（proc-macro 是外部进程）

// 优化建议：
// - 减少过程宏的使用（特别是在 crate 的核心路径上）
// - 用声明宏（macro_rules!）替代简单的过程宏
```

## 6. 让增量编译更友好

### 6.1 最小化公开 API 的变化

```rust
// ❌ 改了公开函数的签名 → 所有下游 crate 要重编译
pub fn api_v1(x: i32) -> String { ... }
pub fn api_v2(x: i32, default: String) -> String { ... }

// ✅ 添加新函数，旧函数转发给新函数
pub fn api_v1(x: i32) -> String {
    api_v2(x, "".into())  // ← 旧 API 不变，下游不用重编译
}
pub fn api_v2(x: i32, default: String) -> String { ... }
```

### 6.2 内联代码放在"叶子模块"

```rust
// ❌ 在 crate 的顶层模块写大量 #[inline] 代码
// → 任何修改都会 invalidate 整个 crate

// ✅ 把可内联的代码放在"叶子模块"
mod internal {
    #[inline]
    pub fn commonly_called() { ... }
}
```

### 6.3 控制 crate 体积

```
过大的 crate → 增量编译效果差
因为：一个文件改了一行 → 整个 crate 的缓存都要重新验证

解决方案：
  - 拆分成多个小 crate（workspace）
  - 改动只影响所在 crate，其他 crate 不受影响
```
