# 7.8 编译器开发流程与 MCP

> 前七节课你在"读"编译器——理解它的架构和算法。这节课是"做"编译器——了解如何构建 rustc、如何添加一个最小的语言特性、如何走通 MCP（Major Change Proposal）流程。你不一定会成为编译器贡献者，但了解这个流程有三个好处：第一，你知道一个语言特性从 idea 到稳定需要经历什么；第二，你能更好地评估 nightly features 的成熟度；第三，当你想修一个 clippy lint 或者改进错误信息时，你知道从哪开始。

---

## 1. 构建 rustc

### 1.1 获取源码

```bash
# clone rustc 仓库（约 500MB depth=1，约 2GB+ 完整）
git clone https://github.com/rust-lang/rust.git --depth 1
cd rust

# 看目录结构
ls
# compiler/    ← rustc 源码（最有用的部分）
# library/     ← 标准库源码
# src/         ← 构建系统和其他工具
# tests/       ← 测试套件
# Cargo.toml   ← 是的，rustc 用 Cargo 管理！
```

### 1.2 ./x.py —— rustc 的构建系统

```bash
# rustc 不用 cargo build（cargo 本身是用 rustc 编译的，循环依赖！）
# 而是用 ./x.py（基于 Python + bootstrap）

# ━━━ 基本命令 ━━━

# 设置构建环境（下载依赖、配置工具链）
./x.py setup

# 构建 stage1 的编译器（最小可用版本）
./x.py build

# 构建完整的编译器
./x.py build --stage 2

# 运行编译器测试
./x.py test

# 用刚构建的编译器编译你的代码
./x.py run --stage 1 -- test.rs
```

### 1.3 stage0 / stage1 / stage2 是什么？

```
rustc 的 bootstrap 过程（三个阶段）：

stage0:
  你系统上已经安装的 rustc（stable/nightly）
  → 用来编译 stage1 的 rustc

stage1:
  用 stage0 编译出的新版 rustc
  → 包含了你的修改
  → 最小功能（可能有 bug），但可以编译标准库

stage2:
  用 stage1 编译标准库和完整的 rustc
  → 这是"真正的"新编译器
  → 通过了测试，可以发布

日常开发中你通常只需要 stage1（快），
跑完整测试时才需要 stage2。
```

```bash
# 1. Fork rust-lang/rust
# 2. 创建分支
```

## 3. MCP（Major Change Proposal）流程

### 3.1 什么需要走 MCP？

```
需要 MCP 的变更：
  ✓ 改变稳定的 API（需要 RFC）
  ✓ 引入新的编译器 flag
  ✓ 改变编译器内部架构
  ✓ 改变错误信息的分类
  ✓ 添加新的 lint（特别是 warn-by-default 的）

不需要 MCP 的变更：
  ✗ 修复 bug
  ✗ 改进错误信息（不改分类）
  ✗ 性能优化（不改变行为）
  ✗ 文档改进
```

### 3.2 MCP 的完整流程

```
Step 1: 问题识别
  你发现一个编译器可以改进的地方

Step 2: 非正式讨论
  在 Zulip（rust-lang.zulipchat.com）上讨论可行性
  确认不是已有方案 / 已知限制

Step 3: 提交 MCP
  在 rust-lang/rust 的 MCP 频道（GitHub issue）提交提案
  填写 MCP 模板：动机、方案、替代方案、影响评估

Step 4: 讨论期
  社区成员（特别是编译器团队成员）讨论
  可能持续数天到数周

Step 5: FCP（Final Comment Period）
  如果讨论基本达成共识 → 一名团队成员发起 FCP
  10 天（通常）等待最终意见

Step 6: 决策
  FCP 结束 → 批准、拒绝或要求修改
  批准后 → 进入实现阶段

Step 7: 实现 + 稳定化
  写代码 → 并通过 stabilization report
```

### 3.3 实际案例：`impl Trait` 在 trait 中的稳定化

```
MCP 的典型节奏（简化）：

2021: 提出 "async fn in trait" 的 MCP
      → 讨论方案：RPITIT vs #[async_trait]
2022: 实现 RPITIT（初始版本）
      → 标记为 #![feature(return_position_impl_trait_in_trait)]
2023: 收集反馈 → 修复 bug → 扩展支持
2024.01: Stabilization report → FCP → 批准
2024.02: Rust 1.75 稳定 RPITIT
```

**关键洞察**：从 idea 到 stable，通常需要 **1.5-3 年**。这就是为什么 nightly features 需要谨慎使用——即使在 nightly 上工作正常，稳定化过程中可能会有 breaking changes。

## 4. 如何找到"第一个编译器贡献"

### 4.1 Good First Issues

```bash
# 在 GitHub 上搜索
# 标签: E-easy, E-mentor, good-first-issue

# clippy 的 good first issue 通常最容易入手
# 因为：
# - clippy 是独立仓库，构建更快
# - lint 的逻辑通常独立，不容易破坏其他功能
# - clippy 团队对新手友好，reviewer 会提供很多指导
```

### 4.2 改进错误信息

```
最容易的贡献类型：改进诊断信息

1. 找到一条不够好的错误信息
2. 在 tests/ui/ 中添加或修改测试用例
3. 修改错误信息的文本模板
4. 更新 .stderr 文件（预期输出）

不需要理解编译器的核心逻辑，只需要：
- 理解这个错误什么时候触发
- 写出更清晰的指导信息
```

### 4.3 添加或改进测试用例

```
编译器测试在 tests/ui/ 目录下

# 运行已有测试
./x.py test tests/ui/borrowck/

# 添加新测试
# 创建 tests/ui/my-test.rs
# 如果预期编译失败，创建 tests/ui/my-test.stderr

# 跑你的测试
./x.py test tests/ui/my-test.rs
```
