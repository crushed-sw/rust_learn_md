# 7.2 借用检查器：NLL 原理与 Polonius 改进

> 借用检查器（Borrow Checker）是 Rust 最独特的组件——没有其他主流语言在编译器里做这件事。它对每一个 Rust 程序员来说都不陌生，因为每个人都跟它的错误信息打过交道：`does not live long enough`、`cannot borrow X as mutable more than once`、`cannot move out of borrowed content`。但借用检查器到底在做什么？它为什么有时候"不够聪明"？NLL（Non-Lexical Lifetimes）比旧的词法生命周期强在哪里？Polonius 又准备怎么改进？这一课给你答案。

---

## 1. 借用检查器的核心问题

### 1.1 一句话总结

> **借用检查器要证明：程序中每条引用（`&T`、`&mut T`）的生命周期，不超过它指向的数据的生命周期。**

形式化地说：
```
对于程序中每一个引用 r: &'a T，都要保证：
  'a（引用的生命期） ⊆ 'data（被引用数据的生命期）
```

### 1.2 三个基本规则

```rust
// 规则 1: 共享借用（&T）——可以同时有多个
let x = 42;
let r1 = &x;
let r2 = &x;    // ✅ 多个共享借用可以共存
println!("{} {}", r1, r2);

// 规则 2: 可变借用（&mut T）——同一时间只能有一个
let mut x = 42;
let r1 = &mut x;
// let r2 = &mut x;  // ❌ 不能同时有两个可变借用
*r1 += 1;

// 规则 3: 共享借用和可变借用不能同时存在
let mut x = 42;
let r1 = &x;
// let r2 = &mut x;  // ❌ 已经有了共享借用
println!("{}", r1);
```

这三条规则你在 Phase 2 就学过了。借用检查器的工作，就是**自动检查所有代码路径上是否满足这三条规则**。听起来简单，但在有循环、分支、函数调用的真实代码中，这是一件非常复杂的事。

## 2. NLL（Non-Lexical Lifetimes）的原理

### 2.1 旧版借用检查器（AST 级别，Rust 2015 及之前）

```rust
// Rust 2015 的借用检查器基于 AST（词法作用域）
fn old_problem() {
    let mut v = vec![1, 2, 3];
    let r = &mut v;   // ← 借用从这里开始
    r.push(4);
    // ... 其他代码 ...
    println!("{:?}", v);  // ❌ 2015: 编译错误！r 的词法作用域还没结束
    // 但在 2018+ NLL 中：✅ r 的最后一次使用是 r.push(4)
}
```

在旧版中，借用的生命周期是**词法作用域**——从变量定义到变量离开作用域。这意味着 `r` 一直存活到 `old_problem` 函数末尾，即使它在 `r.push(4)` 后就不再使用了。

### 2.2 NLL：基于 MIR 的借用检查

```rust
// NLL 不关心变量的"词法作用域"，只关心"实际被使用的位置"

fn nll_works() {
    let mut v = vec![1, 2, 3];
    let r = &mut v;
    r.push(4);
    // ↑ r 的最后一次使用在这
    // ↓ 从这里开始，r 不再存活！可以借用了
    println!("{:?}", v);  // ✅ NLL 允许
}
```

NLL 的核心改进在于：

```
旧版: 借用的生命期 = 变量定义 → 变量离开作用域（词法的）
NLL:  借用的生命期 = 变量定义 → 最后一次使用（基于控制流的）
```

### 2.3 NLL 的实现：MIR + 约束求解

NLL 在 MIR 上工作，因为它需要控制流信息。步骤：

```
Step 1: 在 MIR 上计算 liveness（哪些变量在哪个点"存活"）
Step 2: 为每条借用生成 region constraint
Step 3: 求解 region constraint —— 是否存在有效的生命周期解？
Step 4: 如果有解 → 代码通过；如果无解 → 报告错误
```

```rust
// 看一个 MIR + NLL 的例子
fn example(x: &i32, y: &i32) -> &i32 {
    if *x > 0 { x } else { y }
}

// MIR（简化，关注借用）：
// bb0: switchInt(*_1 > 0) → [false: bb2, true: bb1]
// bb1: _0 = _1; goto bb3    // 返回 x
// bb2: _0 = _2; goto bb3    // 返回 y
// bb3: return

// NLL 生成的 region constraint：
// 'x (参数 x 的生命期) 必须 :> 'return (返回值的生命期)
// 'y (参数 y 的生命期) 必须 :> 'return
// 因为返回的引用可能是 x 也可能是 y

// 函数的签名正是 fn example<'a>(x: &'a i32, y: &'a i32) -> &'a i32
// ↑ 这就是 NLL 推导出的最精确的生命周期约束
```

## 3. 借用检查的关键概念

### 3.1 Loan（借出）

```rust
let mut x = 42;
let r = &mut x;   // ← loan: x 的 .0 字段被可变借出给 r
//          loan 记录了：借了什么、借的类型（共享/可变）、借给谁
```

编译器内部，每次 `&x` 或 `&mut x` 操作都会创建一个 **loan**。Loan 记录：
- **借出路径**（borrowed place）：如 `x`、`x.field`、`(*ptr).field`
- **可变性**：`MutBorrow` 或 `SharedBorrow`
- **借入方**：接收这个借用的变量

### 3.2 Region（生命周期区域）

在 NLL 中，region 不再是"从某行到某行"，而是**MIR 中 basic block 的子集**——从 loan 生效到 loan 失效的 MIR 路径。

```
region('r) = { 程序中变量 r 可能被使用的所有 MIR 位置的集合 }
```

### 3.3 Liveness（存活分析）

```rust
fn liveness_example() {
    let mut v = vec![1];
    // point A: v is live (还没被使用，但"需要活着")
    v.push(2);
    // point B: v is live
    let x = v.pop();
    // point C: v is live (pop 了但可能还用)
    drop(v);
    // point D: v is NOT live (已经 drop 了，再访问是 UB)
}
```

借用检查器需要知道每个变量在每个 MIR 位置上是否"存活"——这样才能判断引用是否在指向有效的内存。

### 3.4 Subset Relation（子集约束）

```
// 这个签名隐含了约束
fn foo<'a, 'b: 'a>(x: &'a i32, y: &'b i32) -> &'a i32 { ... }
//               ^^^^ 'b: 'a 意思是 'b 必须至少和 'a 一样长
//                    这是一个 outlives constraint
```

NLL 生成的约束都是 "outlives" 约束：`region_a: region_b`（a 活得比 b 长）。约束求解器检查所有这些约束是否有循环（有循环 = 无解 = 编译错误）。

## 4. Polonius：用逻辑编程重写借用检查

### 4.1 NLL 的局限性

```rust
// ━━━ NLL 仍然"不够聪明"的例子 ━━━
fn nll_limitation(map: &mut HashMap<i32, String>) -> Option<&String> {
    if map.contains_key(&0) {      // ← 第 1 次借用（共享）
        // map 的共享借用在 contains_key 返回后就结束了
        Some(&map[&0])              // ← 第 2 次借用：&map[&0]
        // ❌ NLL 报错！它认为 map 在这整个 if 块中都被借用了
        // 但实际上，contains_key 的借用在 if 条件之后就释放了
    } else {
        None
    }
}
```

```rust
// ━━━ 另一个 NLL 的盲点：跨循环的借用推理 ━━━
fn nll_loop_problem() {
    let mut v = vec![1, 2, 3];
    let r = &v[0];     // ← 借用 v
    loop {
        println!("{}", r);
        if v.len() > 5 { break; }  // ❌ NLL: 不能借用 v，r 还活着
        // NLL 不够聪明：它不知道 println!("{}", r) 不需要 r 之后再次使用 r
    }
}
```

### 4.2 Polonius 的核心思想

Polonius 的核心创新：**把借用检查表述为 Datalog（逻辑编程）规则**，而不是命令式的约束求解。

```prolog
% 用 Datalog（类似 Prolog）表述借用检查规则

% 规则 1: 如果一个 loan 是有效的，且借出的路径被访问，则 loan 必须存活
% "如果在 point P，loan L 借出了 path X，且 program 访问了 X，
%  那么 L 在 P 必须是存活的"

loan_live_at(Loan, Point) :-
    loan_issued_at(Loan, IssuePoint),
    place_accessed_at(Path, Point),
    loan_borrows(Loan, Path),
    reachable(IssuePoint, Point),
    !loan_killed_between(Loan, IssuePoint, Point).

% 规则 2: 如果一个 loan 是活的且是可变借出，则不能有冲突的借用
error_double_mut_borrow(L1, L2, Point) :-
    loan_live_at(L1, Point),
    loan_live_at(L2, Point),
    L1 != L2,
    loan_is_mut(L1),
    loan_borrows(L1, Path),
    loan_borrows(L2, Path).

% ... 更多规则覆盖所有借用冲突情况
```

**Datalog 的优势**：
1. **声明式**：规则描述"什么构成错误"，不是"怎么检查"。比命令式代码更易理解和验证。
2. **增量求解**：当代码小改时，不必重新求解整个借用关系——只需重算受影响的部分。
3. **更容易扩展**：添加新的借用规则只需加新的 Datalog 规则，不修改核心算法。

### 4.3 Polonius 的状态

```rust
// Polonius 目前仍在开发中（截止 2024 年）
// 你可以用 nightly 体验：
// RUSTFLAGS="-Z polonius" cargo build
```

Polonius 能解决的 NLL 盲点：

```rust
// Polonius 能正确处理这个 NLL 拒绝的代码
fn polonius_fix(map: &mut HashMap<i32, String>) -> Option<&String> {
    if map.contains_key(&0) {
        Some(&map[&0])  // ✅ Polonius 知道 contains_key 的借用已结束
    } else {
        None
    }
}
```


### 5.2 常见"为什么借用检查器拒绝我"的答案

```rust
// 场景 1: 跨循环借用
fn problem1() {
    let mut v = vec![1, 2, 3];
    for item in &v {        // ← 共享借用 v
        if *item == 2 {
            v.push(4);      // ❌ 可变借用 v → 冲突！
        }
    }
}

// 解决：用索引而不是迭代器
fn solution1() {
    let mut v = vec![1, 2, 3];
    for i in 0..v.len() {  // ← 没有借用 v 的迭代器
        if v[i] == 2 {
            v.push(4);      // ✅ OK（但要注意重新分配的影响）
        }
    }
}

// 场景 2: 函数调用隐藏了借用
fn problem2() {
    let mut v = vec![1, 2, 3];
    let r = &mut v;
    takes_vec(&v);  // ❌ r 的可变借用还活着
    r.push(4);
}
fn takes_vec(_: &Vec<i32>) {}

// 解决：缩小可变借用的范围
fn solution2() {
    let mut v = vec![1, 2, 3];
    {
        let r = &mut v;
        r.push(4);
    }  // ← r 的借用在这里结束
    takes_vec(&v);  // ✅ OK
}
```
