# 7.3 Trait 解析与 Chalk

> 你写了 `v.sort()`，编译器要做一件比你想象中复杂得多的事情：它要找到"该用哪个 `sort` 方法"——这涉及在上下文中搜索所有可能的 trait impl，检查哪个最匹配，在可能有多个候选项时选择一个，在都不匹配时给出有用的错误。这就是 trait resolution（trait 解析）。本课将 trait solver 的工作原理拆开：candidate assembly → overlap check → confirmation → specialization。然后介绍 Chalk——用逻辑编程重写 trait 解析的下一代方案。

---

## 1. 一个 `v.sort()` 背后的计算

### 1.1 你写的一行 vs 编译器做的事

```rust
let mut v = vec![3, 1, 2];
v.sort();  // ← 就这一行
```

编译器要回答的问题：

```
1. v 的类型是什么？ → Vec<i32>
2. Vec<i32> 有一个叫 sort 的方法吗？
3. 搜素所有 impl 块（包括当前 crate 和所有依赖）
4. 找到了：impl<T: Ord> [T] { fn sort(&mut self) { ... } }
5. Vec<i32> 能 deref 到 [i32]（通过 Deref） ✓
6. i32 实现了 Ord 吗？ → ✓（std 给所有整数类型实现了 Ord）
7. 所以 v.sort() 是合法的调用
```

这个搜索过程就是 **trait resolution**（或叫 **method resolution**）。

### 1.2 五种"找一个 impl"的场景

```rust
// 场景 1: 显式 trait 方法调用
let x: i32 = 42;
let y = x.clone();       // i32: Clone？→ ✓

// 场景 2: 隐式 trait bound 检查
fn process<T: Display>(x: T) { ... }
process(42);              // i32: Display？→ ✓
process(vec![1,2,3]);     // Vec<i32>: Display？→ ✗（编译错误）

// 场景 3: 方法调用（通过 Deref 链）
let s = String::from("hi");
let len = s.len();        // String → &str → str::len()

// 场景 4: trait object 转换
let x: Box<dyn Display> = Box::new(42);
                           // i32: Display？→ ✓，可以转为 trait object

// 场景 5: blanket impl 匹配
let v = vec![1, 2, 3];
let cloned = v.clone();   // Vec<i32>: Clone？→ ✓
                           // 因为 blanket impl: impl<T: Clone> Clone for Vec<T>
                           // 检查 i32: Clone？→ ✓
```

## 2. Trait Solver 的两大阶段

### 2.1 Candidate Assembly（候选收集）

```rust
// 当编译器需要知道 i32: Ord 时，它先收集所有可能满足的 impl
// ━━━ 候选来源 ━━━

// 来源 1: 直接 impl
impl Ord for i32 { ... }  // ← 候选 1（这是一个固有 impl）

// 来源 2: blanket impl
impl<T: Ord> Ord for &T { ... }  // ← 候选 2（对引用类型的 blanket impl）

// 来源 3: trait alias / supertrait
// trait Super: Ord { ... }
// → 如果 T: Super，则 T: Ord ← 候选 3

// 来源 4: where clause（调用上下文中的约束）
// fn foo<T: Ord>(x: T) {}  ← T: Ord 直接在 where clause 中给出
```

对于 `i32: Ord`，候选 1 匹配；候选 2 不匹配（`i32` 不是引用类型）。

对于 `&i32: Ord`，候选 1 不匹配（`impl Ord for &i32` 不存在）；候选 2 需要检查 `i32: Ord` → ✓。

### 2.2 Confirmation（确认阶段）

找到候选后，编译器要确认它**确实是**我们要的那个：

```rust
// 如果只有一个候选匹配 → 直接使用
// 如果有多个候选匹配 → 需要选择"最具体"的一个

trait Foo { fn foo(&self) -> &str; }

impl Foo for i32 { fn foo(&self) -> &str { "i32" } }
impl<T> Foo for T { fn foo(&self) -> &str { "any T" } }

let x: i32 = 42;
println!("{}", x.foo());  // 两个候选都匹配！
// impl Foo for i32    ← 更具体（只匹配 i32）
// impl<T> Foo for T   ← 更泛化（匹配任何 T）
// 编译器选择更具体的：impl Foo for i32 → 输出 "i32"
```

## 3. Overlap Check 与孤儿规则

### 3.1 为什么需要 overlap check？

```rust
// ❌ 这两个 impl 互相重叠，编译器会报错
impl<T> Trait for T { ... }      // 覆盖所有类型
impl Trait for String { ... }    // 也覆盖 String
// error[E0119]: conflicting implementations of trait `Trait` for type `String`

// ❌ 即使当前的 T 不包含 String，编译器也拒绝
// 因为将来可能有 crate 同时使用 String + 这个 impl
```

**孤儿规则（Orphan Rule）**的存在就是为了防止 overlap：

```
impl Trait for Type 必须满足：
  - Trait 在你的 crate 中定义，或
  - Type 在你的 crate 中定义
（至少一个在当前 crate 中）
```

### 3.2 模糊 overlaps 与 `negative_impls`

```rust
// 即使是孤儿规则保护下，overlap check 仍有模糊地带
#![feature(negative_impls)]

struct MyStruct;

// 显式声明 MyStruct 不实现某个 trait
impl !Send for MyStruct {}  // negative impl（nightly）
```

## 4. Specialization（特殊化）——不稳定特性

### 4.1 问题：想要一个"更具体的 impl"但被拒绝

```rust
// 你想要：对所有类型提供一个默认实现，但对 String 提供特殊实现
trait Describe {
    fn describe(&self) -> String;
}

impl<T> Describe for T {
    fn describe(&self) -> String { "some type".to_string() }
}

// ❌ 错误！conflicting implementations
// impl Describe for String {
//     fn describe(&self) -> String { format!("String: {}", self) }
// }
```

### 4.2 Specialization 的解决方案

```rust
#![feature(min_specialization)]  // nightly only

// default impl：可以被特殊化覆盖
impl<T> Describe for T {
    default fn describe(&self) -> String { "some type".to_string() }
    //  ↑ default 关键字允许被覆盖
}

// specializing impl：覆盖 default impl
impl Describe for String {
    fn describe(&self) -> String { format!("String: {}", self) }
}

let x = 42;
println!("{}", x.describe());         // "some type"
println!("{}", "hi".describe());      // "String: hi"
```

**Specialization 为什么还没稳定**：
1. 在泛型上下文中，当编译器不知道具体类型时，它不知道该选哪个 impl
2. 与 lifetimes / borrow checking 的交互很复杂
3. `default` 关键字的设计仍在讨论中

## 5. Chalk：用逻辑编程重写 Trait 系统

### 5.1 当前 trait solver 的问题

当前 trait solver 是用 Rust 写的命令式代码，有几个问题：
- **不可组合**：每个新的 trait 特性（GAT、specialization）都增加复杂性
- **不总是能推导出正确结果**：有些合法的 trait 组合会超时或给出模糊结果
- **难以形式化验证**：无法证明 trait solver 是正确的

### 5.2 Chalk 的核心思想

Chalk 把 trait 解析表达为**逻辑编程**（类似 Prolog/Datalog）：

```prolog
% Chalk 风格：trait impl 被翻译为逻辑子句

% impl<T: Clone> Clone for Vec<T> {}
% 翻译为逻辑规则：
% Vec<T>: Clone :- T: Clone.
%   ↑ 结论      ↑ 前提条件

% impl Ord for i32 {}
% 翻译为事实（无条件）：
% i32: Ord.

% trait PartialOrd: PartialEq {}
% 翻译为：
% T: PartialEq :- T: PartialOrd.
% ↑ 如果 T 是 PartialOrd，那么 T 也是 PartialEq
```

### 5.3 Chalk 的推理过程

```
问题：Vec<i32>: Clone ?

Step 1: 查找匹配的规则
  候选: Vec<T>: Clone :- T: Clone.
  代入 T = i32 → 需要证明 i32: Clone

Step 2: 查找 i32: Clone
  候选: i32: Clone.  ← 无条件事实
  ✓ 成立！

Step 3: 回溯
  Vec<i32>: Clone ✓ 成立！
```

### 5.4 Chalk 的状态

```bash
# Chalk 可以通过命令行独立测试
cargo install chalk-cli

# 在 chalk 中探索 trait 解析
echo "exists<T> { T: Clone }" | chalk-cli
# → 结果：Ambiguous; suggested substitution T = i32
#   （因为不知道 T 是什么，但 i32 是一个可能的答案）
```

在 rustc 中，Chalk 仍处于实验阶段：

```bash
# 尝试 chalk 作为 trait solver（nightly）
cargo build -Z chalk
```

## 6. 关联类型在 Trait Solver 中的解析

```rust
trait Iterator {
    type Item;  // ← 关联类型
    fn next(&mut self) -> Option<Self::Item>;
}

let mut v = vec![1, 2, 3];
let first = v.iter().next();
// first 的类型是什么？
// 编译器需要解析：
//   v.iter() → std::slice::Iter<i32>
//   Iter<i32>: Iterator → ✓
//   <Iter<i32> as Iterator>::Item → &i32
//   next() → Option<&i32>
```

关联类型的解析比方法调用更复杂，因为：
1. 编译器要找到"唯一确定的"关联类型投影
2. 如果 `<T as Trait>::Assoc` 不能被唯一确定，需要报错
