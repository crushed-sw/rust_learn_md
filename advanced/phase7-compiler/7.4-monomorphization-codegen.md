# 7.4 单态化与代码生成

> 泛型是 Rust 零成本抽象的核心——你在 Phase 1 学过，泛型不引入运行时开销，因为编译器为每个具体类型生成独立的代码副本。这就是**单态化（Monomorphization）**。但"零成本"是运行时的——编译时的成本是真实存在的：代码膨胀（code bloat）、更长的编译时间、更大的二进制文件。这节深入单态化的内部机制：编译器怎么决定"该为哪些类型生成代码"？同一个泛型函数在同一个 crate 内被多次用同一个类型调用——会生成多份副本吗？`cargo bloat` 和 `cargo llvm-lines` 如何帮你找到膨胀的源头？

---

## 1. 单态化的本质

### 1.1 从泛型代码到具体代码

```rust
// 你写的泛型代码
fn first<T>(slice: &[T]) -> Option<&T> {
    if slice.is_empty() { None } else { Some(&slice[0]) }
}

// 编译器实际生成的（简化）
fn first_i32(slice: &[i32]) -> Option<&i32> {  // ← T = i32
    if slice.is_empty() { None } else { Some(&slice[0]) }
}
fn first_string(slice: &[String]) -> Option<&String> {  // ← T = String
    if slice.is_empty() { None } else { Some(&slice[0]) }
}
fn first_f64(slice: &[f64]) -> Option<&f64> {  // ← T = f64
    if slice.is_empty() { None } else { Some(&slice[0]) }
}
// ... 为每个你用过的具体类型生成一份
```

每一个 `T` 的具体类型组合都产生一份独立的机器码。**零运行时开销**——因为生成的代码就是针对具体类型的手写代码。

### 1.2 单态化的时机

```
编译管线中的单态化位置：

HIR (泛型代码存在)
   │
   ▼ typeck (类型检查，但不生成具体版本)
MIR (仍是泛型的)
   │
   ▼ monomorphization_collector (收集所有需要的实例)
   │
   ▼ codegen (为每个实例生成 LLVM IR)
LLVM IR (具体类型版本)
   │
   ▼ LLVM 优化 (可能合并相同的函数)
机器码
```

关键：单态化发生在 **MIR → LLVM IR** 阶段。在此之前，泛型代码以泛型形式存在。

## 2. 单态化收集器

### 2.1 编译器怎么知道"该为哪些类型生成代码"？

```rust
// crate A: 定义泛型函数
pub fn generic_func<T: Display>(x: T) {
    println!("{}", x);
}

// crate B: 使用泛型函数
fn main() {
    generic_func(42);           // ← 需要 T = i32
    generic_func("hello");      // ← 需要 T = &str
}
```

编译 crate B 时，单态化收集器发现 `generic_func` 被调用了两次（`i32` 和 `&str`），于是生成两份代码。crate A 本身不包含任何 `generic_func` 的具体版本——只是在 `.rmeta` 中保存了泛型的 MIR。

**收集器的规则**：
1. 对每个函数调用，收集所有的类型参数组合
2. 如果同一个类型组合被多次调用，只生成一份
3. 跨越 crate 边界时，下游 crate 也会生成自己的实例（除非有 `#[inline]`）

### 2.2 跨 crate 的单态化共享

```rust
// ━━━ 场景 1: 非 inline 函数 ━━━
// crate A
pub fn foo<T: Display>(x: T) { println!("{}", x); }
// A 不包含任何 foo 的具体版本——只有泛型 MIR

// crate B
fn use_foo() { foo(42); }
// B 生成 foo::<i32> 的代码

// crate C
fn also_use_foo() { foo(42); }
// C 也生成 foo::<i32> 的代码！
// ↑ 两份 foo::<i32> 在两个 crate 中，链接器会去重

// ━━━ 场景 2: #[inline] 函数 ━━━
// crate A
#[inline]
pub fn bar<T: Display>(x: T) { println!("{}", x); }
// 标记 #[inline] → 下游 crate 在调用点内联
// 可能完全消除 bar::<i32> 的独立函数体
```

## 3. 二进制膨胀的量化

### 3.1 `cargo bloat` —— 找到最大的函数

```bash
cargo install cargo-bloat

# 找出最大的函数（按代码大小）
cargo bloat --release --crates
# 输出：
#  File  .text     Size        Crate Name
#  0.2%  24.1%  123.4KiB    my_crate super_generic_parser::<TypeA>
#  0.1%  12.3%   63.2KiB    my_crate super_generic_parser::<TypeB>
#  0.1%  10.1%   51.8KiB    my_crate super_generic_parser::<TypeC>
# ...
# ↑ 同一个泛型函数为 3 个类型各生成了 50-120KB 的代码

# 看具体某个函数的膨胀
cargo bloat --release --filter super_generic_parser
```

### 3.2 `cargo llvm-lines` —— 找到 LLVM IR 行数最多的函数

```bash
cargo install cargo-llvm-lines

cargo llvm-lines --release
# 输出：
#  Lines                Copies              Function name
#  -----                ------              -------------
#  123456               12                  my_crate::large_generic (T: all)  
#   89234                3                  std::collections::hash_map::HashMap::insert
#   45123                8                  serde::de::Deserialize::deserialize
# ...
# Lines: 该函数的所有实例（所有类型组合）生成的 LLVM IR 总行数
# Copies: 该函数被单态化了多少次（多少个不同的类型组合）
```

## 4. 减少单态化膨胀的策略

### 4.1 抽出非泛型内部函数

```rust
// ━━━ 膨胀的写法 ━━━
pub fn process<T: AsRef<str>>(input: T) -> Result<Output> {
    let s = input.as_ref();
    // ... 200 行对 &str 的操作，不需要泛型 ...
    parse(s)?;
    validate(s)?;
    transform(s)?;
    Ok(Output::new(s))
}
// ↑ process::<String>, process::<&str>, process::<Cow<str>>
//   三个版本各包含 200 行 &str 操作 → 600 行 LLVM IR = 膨胀！

// ━━━ 减少膨胀的写法 ━━━
pub fn process<T: AsRef<str>>(input: T) -> Result<Output> {
    let s = input.as_ref();
    process_str(s)  // ← 委托给非泛型内部函数
}

fn process_str(s: &str) -> Result<Output> {
    // 200 行对 &str 的操作 → 只有一份代码
    parse(s)?;
    validate(s)?;
    transform(s)?;
    Ok(Output::new(s))
}
// ↑ process::<T> 只有一行（调用 as_ref() → 调用 process_str）
//   process_str 只有一份 → 大幅减少 LLVM IR
```

**这个模式在标准库中广泛使用**：`str::to_lowercase` 内部先 `as_str()` 然后委托给非泛型函数。

### 4.2 用 trait object 替代静态分发（在合适的地方）

```rust
// ━━━ 静态分发：每个类型一份代码 ━━━
fn render_all<T: Render>(items: &[T]) -> String {
    items.iter().map(|item| item.render()).collect()
}
// render_all::<Button>, render_all::<TextBox>, render_all::<Image>
// → 三份代码

// ━━━ 动态分发：只有一份代码 ━━━
fn render_all_dyn(items: &[&dyn Render]) -> String {
    items.iter().map(|item| item.render()).collect()
}
// → 一份代码 + 每个调用有 vtable 间接调用开销

// 决策：当 T 的类型数量 ≤ 3 → 静态分发；≥ 10 → 考虑动态分发减少膨胀
```

### 4.3 考虑使用特征约束的门面模式

```rust
// 当泛型函数体很大，但泛型参数只影响入口和出口时
pub fn parse_and_process<T: Deserialize + Serialize>(raw: &str) -> Result<Vec<u8>> {
    let parsed: T = serde_json::from_str(raw)?;        // ← T 只在这里出现
    let processed = complex_operation(&parsed)?;         // ← 与 T 无关的操作
    let result: Vec<u8> = serde_json::to_vec(&processed)?; // ← T 只在这里出现

    // 重构：
    let intermediate = serde_json::from_str::<serde_json::Value>(raw)?;
    // ↑ 用 Value 替代泛型 T
    let processed = complex_operation(&intermediate)?;
    Ok(serde_json::to_vec(&processed)?)
}
```

## 5. 单态化与编译时间

### 5.1 单态化是编译时间的隐形杀手

```
一个典型的 Rust 项目编译时间构成：
  ├── 类型检查:      ~15%
  ├── 借用检查:      ~10%
  ├── MIR 优化:      ~10%
  ├── 单态化收集:     ~5%
  └── LLVM 代码生成:  ~60%    ← 单态化后的代码都在这里！
       ├── 每个泛型实例 → 独立的 LLVM 函数
       └── 实例数 × 函数大小 → LLVM 工作量
```

**关键结论**：减少单态化实例数可以直接减少 LLVM 编译时间（通常占总编译时间的 50-60%）。

### 5.2 诊断编译时间中的单态化成本

```bash
# 看 LLVM 花费的时间
cargo rustc -- -Z time-passes | grep -i "llvm"
# codegen_to_LLVM_IR: 2.3s
# LLVM_passes: 8.5s         ← 大头
# 如果这个数字很大，可能是单态化膨胀

# 结合 cargo-llvm-lines 找具体的膨胀源
# 减少那个函数的实例数 → 减少 LLVM_passes 时间
```
