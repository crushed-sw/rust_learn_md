## 二、第一课：ripgrep 源码深潜

> **为什么值得学**：ripgrep 是 Rust 社区公认的**代码质量标杆**。作者 BurntSushi（Andrew Gallant）用 3.5 万行 Rust 构建了地球上最快的文本搜索工具。读 rg 的源码 = 上一堂"Rust 工程最佳实践"的大师课。
>
> **核心理念**："Make it correct first, then make it fast. But also, don't make it slow in the first place."
>

#### 2.1 设计哲学

ripgrep 的设计哲学可以概括为 **六个原则**：

```
原则 1: "工具应该做正确的事，用户不应该操心细节"
  → 自动尊重 .gitignore / .ignore / .rgignore
  → 自动跳过二进制文件
  → 自动检测终端是否支持颜色
  → 自动选择最合适的搜索策略（内存映射 vs 缓冲读取）
  → 你不知道这些"细节"时，rg 仍然做正确的事

原则 2: "速度不是目标——速度是正确设计的自然结果"
  → rg 没有为了速度牺牲正确性
  → 它快是因为：
      a) 用 Rust 的 regex crate（SIMD 加速的正则引擎）
      b) 用 memory-mapped I/O（让 OS 管理页面缓存）
      c) 用 ignore crate 智能跳过不关心的文件（省掉了大量 syscall）
  → 作者名言："I spent far more time optimizing for compile time 
     than I did optimizing for search speed."

原则 3: "每个 crate 解决一个独立问题"
  → ripgrep 不是一个巨大的单体——它是 12 个独立 crate 的组合
  → grep-searcher: 搜索策略
  → grep-regex: 正则匹配
  → grep-matcher: 匹配器 trait（你可以接入自己的匹配器）
  → grep-printer: 输出格式化
  → ignore: 目录遍历 + gitignore 规则
  → globset: glob 模式匹配
  → 每个 crate 可以独立使用 → 极高的复用价值

原则 4: "API 是给程序员用的，CLI 是给所有人用的"
  → rg 提供两种界面：命令行（给终端用户）+ 库 API（给 Rust 程序员）
  → grep-searcher 可以作为库被其他工具集成
  → "Library-first" 设计：CLI 只是一个薄壳

原则 5: "二进制搜索是最可靠的速度保证"
  → rg 用 memchr 先找"字面量前缀"，再对候选做正则匹配
  → 这是 rg 比 grep/ag/the_silver_searcher 的关键差异之一
  → 不是在正则内部优化，而是避免在大部分文本上运行正则

原则 6: "简单的事情应该简单，复杂的事情应该可能"
  → rg 'pattern'              → 搜索当前目录，这就是你 90% 的用法
  → rg -t rust 'fn main'      → 搜索 Rust 文件
  → rg -l 'TODO'              → 只列出文件名
  → rg --type-add 'foo:*.foo' → 扩展自定义文件类型
  → 默认行为是"最通用的有用行为"，而不是"最灵活的行为"
```

#### 2.2 学会使用 rg

##### 日常高频命令

```bash
# === 基本搜索 ===
rg 'pattern'                    # 递归搜索当前目录
rg 'pattern' file.txt           # 搜索指定文件
rg 'pattern' src/               # 搜索指定目录

# === 文件过滤 ===
rg -t rust 'fn'                 # 只搜索 Rust 文件
rg -T lock 'serde'              # 排除 lock 文件
rg -g '*.md' '## '              # 只搜索 markdown（glob 模式）
rg -g '!*.json' 'key'           # 排除 JSON

# === 输出控制 ===
rg -l 'TODO'                    # 只列出文件名
rg -c 'error'                   # 显示匹配计数
rg --files-without-match 'test' # 列出没有匹配的文件
rg -A 3 -B 2 'fn main'         # 上下文：After 3 行, Before 2 行
rg --no-heading 'pat'           # 不显示文件名

# === 高级功能 ===
rg --json 'pattern'             # JSON 输出（给工具链使用）
rg -F 'literal.string'          # 固定字符串搜索（不用正则）
rg -P '(?i)pattern'             # PCRE2 正则（Perl 兼容）
rg --multiline 'struct\{[^}]*field'  # 多行匹配
rg -U 'line1\nline2'            # 多行匹配（跨换行符）

# === 替换（rg 不支持原地替换，用管道） ===
rg -l 'old' | xargs sed -i 's/old/new/g'  # 替换所有匹配文件
```

##### 在你的 Rust 项目中使用 ripgrep 的库

```rust
// 使用 grep-searcher 在你的工具中嵌入搜索能力
// Cargo.toml:
// [dependencies]
// grep = { package = "grep-searcher", version = "0.1" }
// grep-regex = "0.1"
// ignore = "0.4"

use grep_regex::RegexMatcher;
use grep_searcher::Searcher;
use ignore::Walk;

fn search_dir(pattern: &str, dir: &str) -> anyhow::Result<()> {
    let matcher = RegexMatcher::new(pattern)?;
    let mut searcher = Searcher::new();

    for result in Walk::new(dir) {
        let entry = result?;
        if entry.file_type().map_or(false, |ft| ft.is_file()) {
            searcher.search_path(
                &matcher,
                entry.path(),
                grep_searcher::sinks::UTF8(|line_num, line| {
                    println!("{}:{}:{}", entry.path().display(), line_num, line);
                    Ok(true)  // return false to stop searching this file
                }),
            )?;
        }
    }
    Ok(())
}
```

#### 2.3 手写 mini_ripgrep

> 分 6 步，从 10 行的 grep 到支持 .gitignore 的并行搜索工具。

##### Step 1: 最朴素的 grep（10 行）

```rust
// mini_rg_step1: 在单个文件中搜索字符串
use std::fs;
use std::env;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let args: Vec<String> = env::args().collect();
    let pattern = &args[1];
    let filename = &args[2];

    let contents = fs::read_to_string(filename)?;
    for (line_no, line) in contents.lines().enumerate() {
        if line.contains(pattern) {
            println!("{}:{}:{}", filename, line_no + 1, line);
        }
    }
    Ok(())
}
// 用法: cargo run -- 'pattern' file.txt
```

**验证点**：想想你从"10 行 grep"到"地球最快搜索工具"之间差了什么？带着这个差距感读后面的步骤。

##### Step 2: 正则 + 递归目录

```rust
// mini_rg_step2: 加正则 + walkdir 递归遍历
// Cargo.toml: regex = "1", walkdir = "2"
use regex::Regex;
use walkdir::WalkDir;
use std::{env, fs};

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let args: Vec<String> = env::args().collect();
    let pattern = Regex::new(&args[1])?;
    let dir = args.get(2).map(|s| s.as_str()).unwrap_or(".");

    for entry in WalkDir::new(dir) {
        let entry = entry?;
        if !entry.file_type().is_file() { continue; }
        let path = entry.path();

        let contents = match fs::read_to_string(path) {
            Ok(c) => c,
            Err(_) => continue,  // 跳过二进制/不可读文件
        };

        for (line_no, line) in contents.lines().enumerate() {
            if pattern.is_match(line) {
                println!("{}:{}:{}", path.display(), line_no + 1, line);
            }
        }
    }
    Ok(())
}
```

**关键观察**：这个版本每次 `fs::read_to_string` 都分配一个 `String`。真正的 rg 用 **memory-mapped I/O**（`memmap2` crate）避免了这件事——让 OS 决定哪些页面需要加载到内存。

##### Step 3: memory-mapped I/O

```rust
// mini_rg_step3: memory-mapped I/O 替代 read_to_string
// Cargo.toml: memmap2 = "0.9"
use regex::bytes::Regex as BytesRegex; // 注意：bytes::Regex
use memmap2::Mmap;
use std::{env, fs::File};

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let args: Vec<String> = env::args().collect();
    let pattern = BytesRegex::new(&args[1])?;
    let filepath = &args[2];

    let file = File::open(filepath)?;
    let mmap = unsafe { Mmap::map(&file)? };
    // mmap 不会立即把文件加载到内存——OS 按需加载页面

    for (line_no, line) in mmap.split(|&b| b == b'\n').enumerate() {
        if pattern.is_match(line) {
            let line_str = String::from_utf8_lossy(line);
            println!("{}:{}:{}", filepath, line_no + 1, line_str);
        }
    }
    Ok(())
}
```

**关键收获**：
- mmap 避免了用户态的缓冲区管理和 `read()` 系统调用
- OS 页面缓存：多个进程共享同一文件的缓存
- 大文件不会被完全加载到内存——OS 只加载被访问的页面

##### Step 4: 字面量前缀优化

这是 rg 最核心的优化——**先用简单搜索缩小候选，再用正则验证**：

```rust
// mini_rg_step4: 提取正则的字面量前缀加速搜索
use regex::bytes::Regex as BytesRegex;
use memmap2::Mmap;
use std::{env, fs::File};

fn find_literal_prefix(re: &str) -> Option<&str> {
    // 简化版：提取正则中最先出现的纯文本部分
    // 实际 rg 用 regex_syntax crate 做这件事
    if re.starts_with('^') {
        return Some(&re[1..]);  // 过于简化——仅示意
    }
    // 找第一个不是元字符的连续文本
    let meta_chars = |c: char| ".^$*+?()[]{}|\\".contains(c);
    let start = re.find(|c: char| c.is_alphanumeric() && !meta_chars(c))?;
    let end = re[start..].find(meta_chars).unwrap_or(re.len() - start);
    Some(&re[start..start + end])
}

fn search_file(pattern: &BytesRegex, filepath: &str) -> Result<(), Box<dyn std::error::Error>> {
    let file = File::open(filepath)?;
    let mmap = unsafe { Mmap::map(&file)? };

    // 关键优化：先找字面量前缀
    let literal_prefix = find_literal_prefix(pattern.as_str());
    
    if let Some(prefix) = literal_prefix {
        // Fast path: 用 memchr 找字面量前缀位置
        // 只在命中的行上跑完整的正则
        for (line_no, line) in mmap.split(|&b| b == b'\n').enumerate() {
            if line.windows(prefix.len()).any(|w| w == prefix.as_bytes()) {
                // 候选项——现在才跑正则
                if pattern.is_match(line) {
                    let line_str = String::from_utf8_lossy(line);
                    println!("{}:{}:{}", filepath, line_no + 1, line_str);
                }
            }
        }
    } else {
        // Slow path: 没有字面量前缀（例如纯元字符正则），每行都跑正则
        for (line_no, line) in mmap.split(|&b| b == b'\n').enumerate() {
            if pattern.is_match(line) {
                let line_str = String::from_utf8_lossy(line);
                println!("{}:{}:{}", filepath, line_no + 1, line_str);
            }
        }
    }
    Ok(())
}
```

**为什么这很重要**：
- `memchr::memmem::find` 用 SIMD（AVX2/SSE4.2）一次比较 32 字节
- 正则引擎再快也比不过 SIMD 字节搜索
- 对大多数搜索场景，90%+ 的行在"字面量前缀"阶段就被跳过了

##### Step 5: 并行搜索

```rust
// mini_rg_step5: rayon 并行搜索多个文件
// Cargo.toml: rayon = "1"
use regex::bytes::Regex as BytesRegex;
use memmap2::Mmap;
use std::{env, fs::File, sync::Mutex};

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let args: Vec<String> = env::args().collect();
    let pattern = BytesRegex::new(&args[1])?;

    use walkdir::WalkDir;
    let files: Vec<_> = WalkDir::new(".")
        .into_iter()
        .filter_map(|e| e.ok())
        .filter(|e| e.file_type().is_file())
        .map(|e| e.path().to_path_buf())
        .collect();

    use rayon::prelude::*;
    let output = Mutex::new(Vec::new());

    files.par_iter().for_each(|path| {
        if let Ok(file) = File::open(path) {
            if let Ok(mmap) = unsafe { Mmap::map(&file) } {
                for (line_no, line) in mmap.split(|&b| b == b'\n').enumerate() {
                    if pattern.is_match(line) {
                        let line_str = String::from_utf8_lossy(line);
                        let mut out = output.lock().unwrap();
                        out.push(format!(
                            "{}:{}:{}",
                            path.display(),
                            line_no + 1,
                            line_str
                        ));
                    }
                }
            }
        }
    });

    // 结果排序输出（因为并行执行，顺序是乱的）
    output.lock().unwrap().sort();
    for line in output.lock().unwrap().iter() {
        println!("{}", line);
    }

    Ok(())
}
```

**真实 rg 的并行策略更精细**：
- rg 用 `ignore` crate 的 `WalkParallel`——目录遍历本身也是并行的
- 使用无锁 MPSC channel 而非 `Mutex<Vec>`
- interleaved 输出：一边搜索一边打印，不等全部搜索完

##### Step 6: .gitignore 支持

```rust
// mini_rg_step6: 尊重 .gitignore
// Cargo.toml: ignore = "0.4"
use ignore::WalkBuilder;
use regex::bytes::Regex as BytesRegex;
use memmap2::Mmap;
use std::{env, fs::File};

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let args: Vec<String> = env::args().collect();
    let pattern = BytesRegex::new(&args[1])?;

    // ignore crate 自动处理 .gitignore, .ignore, .rgignore
    let walker = WalkBuilder::new("./")
        .hidden(false)           // 搜索隐藏文件（默认跳过）
        .git_ignore(true)        // 尊重 .gitignore
        .git_global(true)        // 尊重全局 gitignore
        .git_exclude(true)       // 尊重 .git/info/exclude
        .ignore(true)            // 尊重 .ignore
        .parents(true)           // 搜索父目录的 ignore 文件
        .build();

    for result in walker {
        let entry = match result {
            Ok(e) => e,
            Err(err) => {
                eprintln!("ERROR: {}", err);
                continue;
            }
        };

        // ignore crate 已经过滤了——这里拿到的都是"应该搜索"的文件
        if !entry.file_type().map_or(false, |ft| ft.is_file()) {
            continue;
        }

        let path = entry.path();
        let file = match File::open(path) {
            Ok(f) => f,
            Err(_) => continue,
        };

        let mmap = match unsafe { Mmap::map(&file) } {
            Ok(m) => m,
            Err(_) => continue,
        };

        for (line_no, line) in mmap.split(|&b| b == b'\n').enumerate() {
            if pattern.is_match(line) {
                let line_str = String::from_utf8_lossy(line);
                println!("{}:{}:{}", path.display(), line_no + 1, line_str);
            }
        }
    }

    Ok(())
}
```

**mini_rg 演进总结**：

| Step 2 | 正则 + 递归遍历 | `regex`, `walkdir` |
| Step 3 | memory-mapped I/O | `memmap2`, OS 页面缓存 |
| Step 4 | 字面量前缀优化 | `memchr`, SIMD, 快速路径/慢速路径 |
| Step 5 | 并行搜索 | `rayon`, `Mutex`, 无锁 channel |
| Step 6 | .gitignore 支持 | `ignore` crate, WalkBuilder |

```rust
fn search_path_impl(
    &mut self,
    matcher: &impl Matcher,
    path: &Path,
) -> Result<(), Error> {
    // 1. 打开文件 + mmap
    let file = File::open(path)?;
    let mmap = unsafe { Mmap::map(&file)? };
    let haystack = &mmap[..];

    // 2. 检测二进制文件（看前 N 字节是否有 NUL）
    if self.binary_detection == BinaryDetection::Convert {
        if is_binary(haystack) {
            return binary_bypass(...);
        }
    }

    // 3. 检查 BOM（Byte Order Mark）
    let (encoding, start) = detect_encoding(haystack);

    // 4. 在 haystack[start..] 中搜索匹配
    let mut last_match_end = 0;
    let mut mat = Match::default();
    
    while matcher.find_at(haystack, last_match_end)?.is_some() {
        mat = ...;  // 获取匹配位置
        // 处理行号、上下文、输出
        self.sink.matched_line(...);
        last_match_end = mat.end();
    }

    Ok(())
}
```

## 三、第二课：tracing 源码深潜

> **学习路径**："mini 版理解设计 → 读源码印证直觉 → 能参与贡献"。包含：设计哲学、使用指南、mini 版实现、源码导读、贡献指南。
>
> **为什么值得学**：tracing 是 Rust 异步生态的**可观测性基石**。tokio、h2、tonic、linkerd 都在用它。读 tracing 源码 = 学会"如何用 trait 和泛型实现零成本抽象的可组合系统"。它的 Subscriber/Layer 设计模式是可以迁移到任何需要"数据收集 vs 数据消费解耦"的场景。
>
> 与你的 [[advanced/phase6-engineering/6.4-observability-tracing|6.4 可观测性]] 和 [[advanced/phase8-crate-source/8.3-tracing-infrastructure-design|8.3 tracing 基础设施设计]] 课程衔接。

#### 3.1 设计哲学

tracing 的设计哲学中最核心的一条：

```
核心原则: "Instrumentation（打点）和 Consumption（消费）完全解耦"

你的代码只负责"发出"span 和 event。
怎么记录、怎么过滤、怎么格式化、发送到哪里——你的代码完全不知道，也不应该知道。

这带来了三个"免费"的好处：
  1. 同一套打点代码，测试时输出到 stdout，生产时输出到 OpenTelemetry
  2. 库作者可以打点而不担心"用户想要什么输出格式"
  3. 运行时决定哪些 span 被记录（按级别、按目标、按 span 字段动态过滤）
```

更具体的设计原则：

```
原则 1: "Core 是最小可行的抽象"
  → tracing-core 定义了 Span、Event、Subscriber、Metadata、Field 五个东西
  → 零依赖！（除了 std）
  → 库作者只依赖 tracing-core 就能打点——不依赖 tracing-subscriber

原则 2: "Subscriber 是消费者，Layer 是组合器"
  → Subscriber: 接收 span/event 数据的"桶"
  → Layer: 可组合的处理逻辑（过滤、格式化、采样...）
  → 一个 Subscriber = Registry + 多个 Layer 的组合
  → 类比: Layer 是 tower 中的 Middleware，洋葱模型

原则 3: "编译时零成本"
  → `trace!(...)` 宏在编译时检查 `enabled!(target: ..., level: ...)`
  → 如果当前没有 subscriber 对该 level/target 感兴趣——根本不构造消息
  → 连字符串格式化都不会发生

原则 4: "Span 是核心原语，Event 是瞬时 snapshot"
  → Span: 代表一段时间的"上下文"（进入→离开）
  → Event: 代表一个时刻的"事件"（发生了一件事）
  → Span 可以嵌套（父 span → 子 span），形成树状结构
  → 每个 span/event 携带结构化字段（key-value pairs），不是字符串

原则 5: "异步安全是设计目标，不是事后补丁"
  → Span 通过 `tracing::Span::current()` 跨 .await 传递
  → tokio 的 task 自动继承当前 span
  → Future 被 poll 时自动进入正确的 span 上下文
```

```rust
// mini_trace_step2: 定义 Subscriber trait
use std::any::{Any, TypeId};
// 接上面的代码...

/// Subscriber：消费 span/event 的抽象（核心接口！）
pub trait Subscriber {
    /// 当新 span 被创建时调用
    fn on_new_span(&self, span: &Span) -> usize;  // 返回 span id
    
    /// 当 span 被进入时调用
    fn on_enter(&self, span_id: usize);
    
    /// 当 span 被退出时调用
    fn on_exit(&self, span_id: usize);
    
    /// 当 event 被记录时调用
    fn on_event(&self, event: &Event);
    
    /// 某个 level/target 的事件是否应该被记录？
    fn enabled(&self, metadata: &Metadata) -> bool;
}

// === 全局 subscriber（简化版——真实 tracer 用 thread-local） ===
use std::sync::RwLock;
static SUBSCRIBER: RwLock<Option<Box<dyn Subscriber + Send + Sync>>> = RwLock::new(None);

pub fn set_subscriber<S: Subscriber + Send + Sync + 'static>(subscriber: S) {
    let mut guard = SUBSCRIBER.write().unwrap();
    *guard = Some(Box::new(subscriber));
}

pub fn with_subscriber<T>(f: impl FnOnce(&dyn Subscriber) -> T) -> Option<T> {
    let guard = SUBSCRIBER.read().unwrap();
    guard.as_ref().map(|s| f(s.as_ref()))
}
```

```rust
// mini_trace_step3: 实现 ConsoleSubscriber
// 接上面的代码...

use std::sync::Mutex;

pub struct ConsoleSubscriber {
    spans: Mutex<Vec<Span>>,
}

impl ConsoleSubscriber {
    pub fn new() -> Self {
        ConsoleSubscriber {
            spans: Mutex::new(Vec::new()),
        }
    }

    fn indent_for(&self, span_id: usize) -> String {
        let spans = self.spans.lock().unwrap();
        let mut depth = 0;
        let mut current = Some(span_id);
        while let Some(id) = current {
            depth += 1;
            current = spans[id].parent;
        }
        "  ".repeat(depth)
    }
}

impl Subscriber for ConsoleSubscriber {
    fn on_new_span(&self, span: &Span) -> usize {
        let mut spans = self.spans.lock().unwrap();
        let id = spans.len();
        spans.push(span.clone());
        
        println!(
            "{}{{ {}",
            self.indent_for(id),
            span.metadata.name
        );
        id
    }

    fn on_enter(&self, _span_id: usize) {
        // 简化实现
    }

    fn on_exit(&self, span_id: usize) {
        println!("{}}}", self.indent_for(span_id));
    }

    fn on_event(&self, event: &Event) {
        let indent = match event.parent {
            Some(id) => self.indent_for(id),
            None => String::new(),
        };
        
        let level_str = match event.metadata.level {
            Level::Error => "ERROR",
            Level::Warn  => "WARN",
            Level::Info  => "INFO",
            Level::Debug => "DEBUG",
            Level::Trace => "TRACE",
        };
        
        print!("{}[{}] {}", indent, level_str, event.metadata.name);
        for (key, value) in &event.metadata.fields {
            print!(" {}={:?}", key, value);
        }
        println!();
    }

    fn enabled(&self, _metadata: &Metadata) -> bool {
        true  // 简化：不过滤，全输出
    }
}
```

```rust
// mini_trace_step4: 实现 info! / error! / span! 等宏

// 全局 span 栈（简化版——真实版本用 thread-local）
use std::cell::RefCell;
thread_local! {
    static SPAN_STACK: RefCell<Vec<usize>> = RefCell::new(Vec::new());
}

/// 进入一个 span
pub fn enter_span(span_id: usize) {
    with_subscriber(|s| s.on_enter(span_id));
    SPAN_STACK.with(|stack| stack.borrow_mut().push(span_id));
}

/// 退出当前 span
pub fn exit_span() {
    let span_id = SPAN_STACK.with(|stack| stack.borrow_mut().pop());
    if let Some(id) = span_id {
        with_subscriber(|s| s.on_exit(id));
    }
}

/// 当前 span id
pub fn current_span_id() -> Option<usize> {
    SPAN_STACK.with(|stack| stack.borrow().last().copied())
}

// === 宏 ===
#[macro_export]
macro_rules! info {
    ($($arg:tt)*) => {
        let event = $crate::Event::new(&format!($($arg)*), $crate::Level::Info);
        $crate::with_subscriber(|s| s.on_event(&event));
    };
}

#[macro_export]
macro_rules! error {
    ($($arg:tt)*) => {
        let event = $crate::Event::new(&format!($($arg)*), $crate::Level::Error);
        $crate::with_subscriber(|s| s.on_event(&event));
    };
}

#[macro_export]

    ($name:expr, $body:block) => {{

        $crate::enter_span(span_id);
        let result = $body;
        $crate::exit_span();
        result

}
```

```rust
// mini_trace_step5: Layer 组合模式

/// Layer trait：每个 Layer 可以处理 span/event，也可以包装下一个 Layer
pub trait Layer {
    fn on_new_span(&self, span: &Span) {}
    fn on_event(&self, event: &Event) {}
    fn enabled(&self, metadata: &Metadata) -> bool { true }
}

/// Filter trait：过滤条件
pub trait Filter {
    fn enabled(&self, metadata: &Metadata) -> bool;
}

// 一个"按级别过滤"的 Filter
pub struct LevelFilter {
    pub max_level: Level,
}

impl Filter for LevelFilter {
    fn enabled(&self, metadata: &Metadata) -> bool {
        metadata.level >= self.max_level
    }
}

// 一个"写入 stdout"的 Layer
pub struct StdoutLayer;

impl Layer for StdoutLayer {
    fn on_event(&self, event: &Event) {
        println!(
            "[{}] {} at {}",
            format!("{:?}", event.metadata.level),
            event.metadata.name,
            event.metadata.target
        );
    }
}

// === 组合：Layered Subscriber ===
// 这是真实 tracing-subscriber::Registry 的简化版
pub struct LayeredSubscriber<L: Layer, F: Filter> {
    layer: L,
    filter: F,
}

impl<L: Layer, F: Filter> LayeredSubscriber<L, F> {
    pub fn new(layer: L, filter: F) -> Self {
        LayeredSubscriber { layer, filter }
    }
}

impl<L: Layer, F: Filter> Subscriber for LayeredSubscriber<L, F> {
    fn on_new_span(&self, span: &Span) -> usize {
        self.layer.on_new_span(span);
        0
    }

    fn on_event(&self, event: &Event) {
        if self.filter.enabled(&event.metadata) {
            self.layer.on_event(event);
        }
    }

    fn on_enter(&self, _span_id: usize) {}
    fn on_exit(&self, _span_id: usize) {}

    fn enabled(&self, metadata: &Metadata) -> bool {
        self.filter.enabled(metadata)
    }
}

// 使用示例：
// let subscriber = LayeredSubscriber::new(
//     StdoutLayer,
//     LevelFilter { max_level: Level::Info }
// );
// set_subscriber(subscriber);
```

```
读 Day 1: 理解核心（tracing-core）

    核心方法:
      - enabled(&self, metadata: &Metadata) -> bool
      - new_span(&self, span: &Attributes) -> SpanId

      - event(&self, event: &Event)

      - exit(&self, span: &Id)
    问自己: 为什么 new_span / enter / exit 是分开的？
    答: 因为 span 的创建和进入是两个独立事件。
        span 创建时立即记录字段，进入时才建立父子关系。

  → tracing-core/src/dispatcher.rs  ← Dispatch（全局 subscriber 分发器）
    问自己: 为什么需要 Dispatch 而不是直接调用 Subscriber？

        1) interest cache —— 快速判断"这个 target+level 是否有人关心"
        2) default subscriber —— 如果没设置 subscriber，用 no-op
        3) thread-local override —— 临时替换 subscriber（用于测试）

  → tracing-core/src/callsite.rs    ← Interest cache 的实现

    每个 trace!() 调用点在编译时注册为一个 Callsite
    Callsite 缓存了 subscriber 对它的"兴趣"

读 Day 2: 理解宏和用户 API（tracing）

          // 这才构造 Event 并调用 dispatch.event()

    问自己: enabled! 怎么知道是否 enabled？
    答: interest cache ——如果之前所有 subscriber 都说"不感兴趣"，
        编译时注册的 callsite 缓存为 "never"，宏直接跳过。

    注意 Span::enter() 和 Span::in_scope() 的区别
```

```rust
pub enum Interest {
    Never,      // 永远不关心 → 宏直接跳过
    Sometimes,  // 有时关心 → 每次调用时检查
    Always,     // 总是关心 → 跳过 enabled 检查
```

## 四、从零造轮子：mini_serde & mini_tokio

> 有了 ripgrep 和 tracing 的源码阅读经验，你对"好代码长什么样"已经有了标准。现在是时候拿起键盘，从零构建 Rust 生态中最核心的两个组件——序列化框架和异步运行时。

### 1. mini_serde

> **为什么值得学**：serde 是 Rust 生态的基石。理解 serde = 理解 Rust 的 trait 系统、零成本抽象、过程宏三者的极限配合。

#### 核心学习点

| 模块 | 学什么 | Rust 知识点 |
|:---|:---|:---|
| **数据模型** | serde 定义了 29 种"中间类型"（bool/i8-i64/u8-u64/f32/f64/char/str/bytes/some/none/unit/seq/map...），所有格式统一映射到这个模型 | Trait 设计、`Visitor` 模式 |
| **Serialize trait** | `fn serialize<S: Serializer>(&self, s: S) -> Result<S::Ok, S::Error>` | 泛型约束、关联类型 |
| **Deserialize trait** | `fn deserialize<D: Deserializer<'de>>(d: D) -> Result<Self, D::Error>` | 生命周期、GAT |
| **Serializer/Deserializer trait** | 每个格式实现一组方法（serialize_bool, serialize_i32...） | 巨型 trait 设计、代码组织 |
| **derive 宏** | `#[derive(Serialize)]` — 自动生成对每个字段的序列化调用 | 过程宏、syn/quote |
| **零成本抽象** | 序列化不是运行时反射，编译期就生成了所有代码 | 单态化、内联 |

#### 实现路径（建议分 5 步）

```
Step 1: 手写 Serialize/Deserialize trait，不用宏
  ├── 定义数据模型（中间类型枚举）
  ├── 实现 Serializer trait（JSON 格式）
  └── 手动为 struct 写 impl Serialize

Step 2: 实现 Deserializer
  ├── 词法分析（JSON tokenizer）
  ├── 实现 Deserializer trait
  └── 手动为 struct 写 impl Deserialize

Step 3: 写 derive 宏
  ├── #[derive(MySerialize)] 过程宏
  ├── #[derive(MyDeserialize)] 过程宏
  └── 处理：命名字段、元组结构体、枚举

Step 4: 支持更多格式
  ├── 抽象格式层，添加 MessagePack 或 YAML
  └── 验证数据模型的通用性

Step 5: 进阶特性
  ├── #[serde(rename = "...")] 属性
  ├── #[serde(skip)] 跳过字段
  └── 泛型结构体的序列化（带 trait bound）
```

#### 关键考验

- derive 宏中处理**泛型参数**和 **where 子句**的正确传递
- 反序列化时**生命周期标注**的复杂性（`Deserialize<'de>`）
- `Serializer` trait 的 30+ 方法如何组织，避免调用方的心智负担

### 2. mini_tokio

> **为什么值得学**：tokio 是 Rust 异步生态的绝对核心。自己实现一个迷你版 = 彻底搞懂 async/await 到底是怎么回事。结合你在 tracing 深潜中学到的可观测性模式，你可以在 mini_tokio 中打点，让调度器的内部状态"可视化"。

#### 核心学习点

| 模块 | 学什么 | Rust 知识点 |
|:---|:---|:---|
| **Future trait** | `poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<T>` | Pin、自引用、状态机 |
| **Waker 机制** | 运行时如何"唤醒"暂停的 Future | `RawWaker`、`Arc`、vtable |
| **Reactor (I/O 驱动)** | epoll/kqueue/IOCP 封装 | 系统调用、mio crate、事件循环 |
| **调度器** | work-stealing 多线程调度 | `Arc`、`Mutex`、无锁队列 |
| **spawn** | 将 Future 提交到运行时 | 类型擦除、`Send` bound |
| **Timer** | `sleep()` 的实现 | 时间轮/二叉堆、epoll timeout |
| **select!** | 同时等待多个 Future | 分支 Future、取消安全 |

#### 实现路径（建议分 6 步）

```
Step 1: 单线程 Future 执行器
  ├── 定义 Task：Box<dyn Future<Output = ()> + Send>
  ├── 实现 run()：循环 poll 所有 task
  └── 验证：手动实现 Future trait 的简单例子

Step 2: Waker 与 Reactor
  ├── 封装 epoll（用 mio crate）
  ├── 实现 AsyncTcpStream（基于 mio + Waker）
  └── 验证：异步 TCP echo server

Step 3: spawn 与 多任务
  ├── 任务队列（VecDeque 或 channel）
  ├── spawn() 将 Future 放入队列
  └── 主循环：从队列取 task → poll → 完成则移除

Step 4: 多线程 work-stealing
  ├── 每个 worker 有自己的任务队列
  ├── 空闲 worker 从其他 worker "偷"任务
  └── 用 crossbeam::deque 或手写无锁队列

Step 5: Timer 实现
  ├── 用二叉堆管理定时任务
  ├── epoll_wait timeout = 最近定时任务的时间
  └── 实现 sleep(duration)

Step 6: select! 宏
  ├── 用声明宏或过程宏实现 select!
  └── 处理分支完成/取消的语义
```

#### 关键考验

- **Waker 的 vtable 构建**：`RawWaker::new(ptr, &VTABLE)` — 这是 unsafe 的核心
- **多线程调度器**的竞态条件
- **IO 边界**：epoll 返回就绪事件 → 找到对应 Waker → wake → 重新调度
- **取消安全**：select! 中被丢弃的分支会导致什么后果

#### 与真实 tokio 的差距（不必实现但需理解）

| 真实 tokio 有 | mini 版本 |
|:---|:---|
| 多层 I/O 抽象（Resource → Driver → Reactor） | 直接封 mio |
| 协作式调度 + LIFO slot 优化 | 简单 FIFO |
| io_uring 支持 | 仅 epoll |
| 完善的 `JoinHandle` + 取消 | 简化版 |
| tracing 集成 | 无（你可以自己加上！） |

> **为什么值得做**：这是对 unsafe Rust + 操作系统接口的综合考验。短小精悍，但学习密度极高。

#### 学习价值

- `GlobalAlloc` trait 实现
- 指针运算（`add`, `offset`, `align_offset`）
- 对齐 (`Layout::align`, `#[repr(align)]`)
- 内存映射 (`mmap`, `VirtualAlloc`)
- Arena / Bump / Buddy / Slab 分配策略
- 与 `Vec` / `Box` / `String` 的集成验证

#### 实现路径

```
Bump Allocator (1 周)
  ├── 最简单的：一个 cursor，只分配不释放
  └── 验证：用 #[global_allocator] 替换全局分配器

Buddy Allocator (2 周)
  ├── 二叉树管理空闲块
  └── 分裂与合并

Slab Allocator (1-2 周)
  ├── 固定大小对象池
  └── 与全局分配器分层使用
```

## 七、推荐学习顺序

核心思路：**先通过读源码建立品味 → 再自己造轮子**。按以下路径：

```
Phase 1-10 完成 → 进入实战项目阶段

  第一课: ripgrep 源码深潜（2-4 周）
    → 学习"什么是工业级 Rust 代码"
    → 手写 mini_rg（10行 → mmap → SIMD → 并行 → .gitignore）
    → 带着 mini_rg 的经验读 rg 源码
    → 建立性能工程的直觉

  第二课: tracing 源码深潜（2-3 周）
    → 学习"如何用 trait 实现可组合架构"
    → 手写 mini_tracing（Span/Event → Subscriber → Layer 组合）
    → 带着 mini_tracing 的经验读 tracing 源码
    → 理解 Subscriber/Layer/Interest Cache 设计

  第三课: mini_serde（3-5 周）
    → 过程宏 + trait 设计的终极训练
    → 现在你有了 rg 的"代码品味"，写的 serde 会不一样

  第四课: mini_tokio（4-8 周）
    → 异步运行时内核
    → 用上 tracing 打点来"看见"调度器内部状态

  ─── 以下选做 ───

  ├── mini HTTP 框架（天然延伸，用上 tokio + tracing）
  ├── mini 数据库引擎（最全面的系统编程练习）
  ├── mini 内存分配器（短平快，提升 unsafe 功底）
  ├── mini 编程语言（编译原理 + 模式匹配的极致）
  └── lock-free 数据结构（并发内功）
```

**推荐的 7 项核心组合**（选 5-7 个做深）：

| # | 项目 | 理由 |
|:--|:---|:---|
| 1 | **ripgrep 深潜** | **工程级代码品味 + 性能工程实战** |
| 2 | **tracing 深潜** | **可组合架构模式 + 结构化可观测性** |
| 3 | mini_serde | 过程宏 + trait 设计 |
| 4 | mini_tokio | 异步运行时内核 |
| 5 | mini 数据库 | 系统编程集大成 |
| 6 | mini 语言 | 编译原理 + 类型系统 |
| 7 | mini 分配器 | 短但极度深入的 unsafe 训练 |

## 八、快速决策

如果时间有限，只能选 **4 个**：

| 组合 | 覆盖的核心能力 |
|:---|:---|
| **ripgrep** + **tracing** + mini_serde + mini_tokio | 工程品味 + 可观测性 + Trait + 异步 —— **必选组合，这四个做完你已经脱胎换骨** |
| 再加 mini 数据库 | 以上全部 + 存储引擎 —— 适合后端/基础设施方向 |
| 再加 mini 编程语言 | 以上全部 + 编译原理 —— 适合语言/工具方向 |

> 💬 **我的推荐**：**ripgrep 深潜 → tracing 深潜 → mini_serde → mini_tokio → mini 数据库 → mini 语言**。
> 顺序很重要——先读好代码建立标准，再自己写。这 6 个做完，你在 Rust 生态里几乎没有看不懂的代码了。
>
> 前两个是"源码深潜"（不用从零造轮子，2-4 周 + 2-3 周），
> 后四个是"从零造轮子"（有了前两个的标准，质量会高很多）。
